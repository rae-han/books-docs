# Chapter 3: Other Engineers and Code Contracts (다른 개발자와 코드 계약)

## 핵심 질문

내 코드는 정말 나만의 것인가? 다른 개발자는 내 코드의 사용법을 어떻게 알아내며, 그 방법 중 무엇이 가장 믿을 만한가? 코드 계약이란 무엇이고, "명백한 항목"과 "세부 조항"은 어떻게 다른가? 세부 조항에 의존하지 않고 계약을 전달하려면 어떻게 해야 하며, 세부 조항을 피할 수 없을 때 체크와 어서션은 어떤 역할을 하는가?

## 0. 이 장에서 다루는 것 · 코드 표기 규약

- 다른 개발자들이 코드와 어떻게 상호작용하는지
- 코드 계약과 그 세부 조항
- 세부 조항을 최소화하는 것이 어떻게 오용·예측 이탈을 예방하는지
- 세부 조항을 피할 수 없을 때 체크와 어서션을 어떻게 쓰는지

소프트웨어를 만들고 유지보수하는 일은 대개 팀 단위의 작업이다. 내가 작성한 코드를 다른 개발자가 만지고, 반대로 다른 개발자의 코드를 내가 만진다. 1장의 6대 핵심 요소 중 **예측 가능한 코드**와 **오용하기 어려운 코드**는 바로 이 "다른 사람이 내 코드와 상호작용할 때 잘못될 수 있는 일"과 직결된다. 이 장은 코드의 중요한 세부사항을 다른 개발자에게 **전달하는 여러 기법**을 신뢰도 관점에서 다룬다.

## 1. 자신의 코드와 다른 개발자의 코드

내가 작성한 코드는 **독립적으로 존재하지 않는다.** 다른 개발자의 코드 위에 쌓이고, 다른 개발자는 다시 내 코드 위에 또 다른 계층을 쌓는다. 2장의 `TextSummarizer`를 예로 들면, 내가 기사 요약용으로 만든 코드를 다른 개발자가 **댓글 요약**이나 **읽기 시간 추정**에 재사용할 수 있다.

또한 요구사항은 항상 변한다. 코드베이스는 여러 개발자가 끊임없이 수정하는 "활발하고 바쁜 장소"다.

> 대규모 파티를 열 때 근사한 유리그릇을 꺼내지 않고, 경기장의 장벽을 금속으로 만들어 땅에 볼트로 고정하는 데는 다 이유가 있다. 부서지기 쉬운 물건은 바쁜 장소에 둘 수 없다.

코드를 작성할 때 고려할 세 가지가 있다.

### 1.1 자신에게 분명하다고 해서 다른 사람에게도 분명한 것은 아니다

문제를 며칠씩 붙들고 있으면 내 로직에 너무 익숙해져 모든 것이 자명해 보인다. 하지만 다른 개발자는 그 문제를 고민할 시간을 갖지 못한 상태다. 내게 자명했던 것이 그들에겐 전혀 그렇지 않다. 코드가 무엇을 하고 왜 그렇게 하는지 전달하는 것이 유용하다 (단, 이는 주석을 많이 쓰라는 뜻이 아니라 **코드 자체로 설명되게** 하라는 뜻이다).

### 1.2 다른 개발자는 무의식중에 내 코드를 망가뜨릴 수 있다

내 코드는 끊임없이 변하는 코드 위에 놓여 있고, 내 코드에 의존하는 코드도 계속 작성된다. 다른 개발자는 내 코드가 왜 존재하고 무슨 일을 하는지 모른 채 수정하다가 의도치 않게 코드를 망가뜨리거나 오용할 수 있다.

> **핵심 통찰**: 다른 개발자의 변경이 내 코드를 망가뜨렸을 때 **가장 신뢰할 수 있는 방어선은 두 가지 — 코드가 컴파일되지 않게 하거나, 테스트가 실패하게 하는 것**이다. 병합 전에 이 중 하나가 일어나면 문제 있는 코드는 코드베이스에 병합되지 못한다. 고품질 코드의 많은 고려사항이 궁극적으로 이 두 가지를 겨냥한다.

### 1.3 시간이 지나면 자신의 코드를 기억하지 못한다

지금은 세부사항이 생생하지만 시간이 지나면 잊는다. 1~2년 전 내가 작성한 코드를 다시 보는 일은 **다른 사람의 코드를 보는 것과 크게 다르지 않다.**

> **핵심 통찰**: 이 책의 모든 내용에서 **"미래의 나"는 본질적으로 다른 개발자**라고 간주해야 한다. 배경지식이 거의 없는 사람에게도 이해하기 쉽고, 잘 돌아가던 코드에 버그를 넣기 어려운 코드를 만드는 것은 남에게 베푸는 호의이자 미래의 나에게 주는 선물이다.

## 2. 다른 사람들은 내 코드의 사용법을 어떻게 아는가

다른 개발자가 내 코드를 쓰려면 어떤 함수를 언제 호출하는지, 클래스가 무엇을 나타내는지, 어떤 인수를 넣는지, 무엇을 반환하는지 파악해야 한다. 이를 알아내는 방법은 다섯 가지이며, **신뢰도가 서로 다르다.**

1. **이름 확인** — 함수·클래스·열거형 등의 이름을 본다. 가장 흔히 쓰는 방법. 이름은 책의 목차와 같아서 `removeEntry()`를 `addEntry()`와 혼동할 수 없다.
2. **데이터 유형 확인** — 매개변수·반환값의 타입을 본다. **가장 신뢰할 만한 방법.** 정적 타입 언어에서는 타입을 틀리면 컴파일조차 안 되므로, 타입 시스템은 오용을 원천 차단하는 좋은 수단이다.
3. **문서 읽기** — 주석·자바독·README 등. 유용하지만 신뢰도가 낮다: (a) 읽는다는 보장이 없고, (b) 읽어도 오해할 수 있으며, (c) 업데이트가 제때 안 되어 내용이 어긋난다.
4. **직접 물어보기** — 팀이라면 물어볼 수 있지만, 질문 응대에 시간이 들고, 작성자가 휴가·퇴사하거나 1년 뒤엔 본인도 잊어버려 제한적이다.
5. **코드 살펴보기** — 가장 확실하지만 가장 비실용적이다. 의존하는 수백 개 코드의 구현을 다 읽어야 한다면 적당한 기능 하나에도 수십만 줄을 읽어야 하고, 이는 추상화 계층의 이점을 정면으로 부정한다.

> **핵심 통찰**: 다섯 가지 중 **처음 셋(이름·타입·문서)만 실용적이고, 그중에서도 이름과 타입이 문서보다 훨씬 신뢰할 만하다.** 따라서 코드 사용법을 전달하는 가장 좋은 방법은 **이름을 잘 짓고 타입 시스템을 활용하는 것**이다.

## 3. 코드 계약

**계약에 의한 프로그래밍(*Programming by Contract*)** 또는 **계약에 의한 디자인(*Design by Contract - 버트랜드 마이어가 1980년대에 도입, Eiffel 언어의 중심 개념*)**은 코드 간 상호작용을 계약처럼 보는 원칙이다. 호출하는 코드는 특정 요건을 충족해야 하고, 호출되는 코드는 원하는 값을 반환하거나 상태를 수정한다. 코드 계약의 용어는 세 범주로 나뉜다.

- **선결 조건(*precondition*)**: 코드를 호출하기 전에 사실이어야 하는 것 (시스템 상태, 입력 등)
- **사후 조건(*postcondition*)**: 코드가 호출된 후에 사실이어야 하는 것 (새 상태, 반환값 등)
- **불변 사항(*invariant*)**: 호출 전후로 변하지 않아야 하는 것

이 용어를 몰라도, 입력 매개변수를 받고 값을 반환하거나 상태를 수정하는 함수를 작성하면 이미 계약을 맺는 것이다.

### 3.1 계약의 세부 조항

현실의 계약이 **명백한 항목**과 깨알 같은 **세부 조항(*small print*)**으로 나뉘듯 코드 계약도 그렇다.

> **비유 — 전기 스쿠터 렌털**: 예약 화면에는 "시간당 $10"이 명백히 보이지만(명백한 항목), 약관 링크 뒤에는 "충돌 시 수리비 부담", "시 경계 넘으면 $100 벌금", "시속 30마일 초과 시 모터 손상으로 $300 벌금(속도 제한 장치가 없어 초과하기 쉬움)" 같은 세부 조항이 숨어 있다. 세 번째 조항은 대부분 미처 생각 못 해 뜻밖의 벌금을 물게 된다.

코드 계약에서 무엇이 명백한 항목이고 무엇이 세부 조항인가:

- **명백한 항목**: 함수·클래스 이름, 인자 유형, 반환 유형, 검사 예외(*checked exception - 자바에만 있으며, 호출 코드가 처리하지 않으면 컴파일되지 않는다*). 호출 측이 무시할 수 없다.
- **세부 조항**: 주석·문서, 비검사 예외(*unchecked exception - 일반적으로 "예외"라 하면 이것을 의미*). 꼼꼼히 읽어야 알 수 있지만 잘 읽지 않는다.

> **핵심 통찰**: 코드 계약에서 조건을 **명백하게** 하는 것이 세부 조항에 의존하는 것보다 훨씬 낫다. 사람들은 세부 조항을 잘 읽지 않고, 읽어도 대충 훑어 오해하며, 문서는 업데이트가 밀려 부정확해지기 때문이다.

예제 3.1은 세부 조항이 많은 나쁜 예다. `UserSettings`를 쓰려면 `loadSettings()` → `init()`을 **정확한 순서**로 호출해야 하고, `getUiColor()`의 널 반환값이 "색상 미선택"인지 "초기화 안 됨"인지 이중 의미를 갖는다 — 모두 주석(세부 조항)으로만 전달된다.

  if (chosenColor === null) {
    // null이면 기본 색상. 그러나 '색상 미선택'과 '유효하지 않은 상태'가 구분되지 않는다.

### 3.2 세부 조항에 너무 의존하지 말라

세부 조항은 쉽게 간과되므로 코드 계약을 전달하는 신뢰할 만한 방법이 아니다. 스쿠터 예에서 **시속 30마일을 넘기는 것 자체를 불가능하게(속도 제한 장치)** 만들면 세부 조항이 필요 없어지듯, 코드도 **잘못된 일을 처음부터 불가능하게** 만드는 것이 좋다. 목표는 코드가 오용되거나 잘못 설정되면 **컴파일조차 안 되게** 하는 것이다.

예제 3.3은 `UserSettings`를 **정적 팩토리 함수 `create()` + 프라이빗 생성자**로 바꿔, 완전히 초기화된 유효한 인스턴스만 얻을 수 있게 한다. `loadSettings()`·`init()`은 프라이빗이 되어 외부에서 잘못 호출할 수 없고, 인스턴스가 항상 유효하므로 `getUiColor()`의 널은 이제 "색상 미선택" 하나만 의미한다.

<details>
<summary>의사코드 (원서) — 예제 3.3 세부 조항이 거의 없는 코드 (좋은 예)</summary>

```java
class UserSettings {
  // 생성자가 프라이빗이므로 다른 개발자는 생성자 대신 create()를 쓸 수밖에 없다.
  private UserSettings() { ... }

  // UserSettings 인스턴스를 만드는 유일한 방법. 로딩 실패 시 널을 반환한다.
  // 이렇게 함으로써 누구도 유효하지 않은 상태의 인스턴스를 가질 수 없다.
  static UserSettings? create(File location) {
    UserSettings settings = new UserSettings();
    if (!settings.loadSettings(location)) {
      return null;
    }
    settings.init();
    return settings;
  }

  // 상태를 변경하는 함수는 다 프라이빗이다.
  private Boolean loadSettings(File location) { ... }
  private void init() { ... }

  // 색상을 선택하지 않은 경우에만 널을 반환한다.
  Color? getUiColor() { ... }
}
```

</details>

```typescript
class UserSettings {
  // TS: private 생성자로 외부 인스턴스화를 차단 → create()만 사용 가능
  private constructor() { /* ... */ }

  /** UserSettings를 만드는 유일한 방법. 로딩 실패 시 null. 유효하지 않은 인스턴스는 존재할 수 없다. */
  static create(location: File): UserSettings | null {
    const settings = new UserSettings();
    if (!settings.loadSettings(location)) {
      return null;
    }
    settings.init();
    return settings;
  }

  // 상태를 변경하는 함수는 모두 private
  private loadSettings(location: File): boolean { /* ... */ }
  private init(): void { /* ... */ }

  /** 색상을 선택하지 않은 경우에만 null을 반환한다. */
  getUiColor(): Color | null { /* ... */ }
}
```

> **핵심 통찰**: 이 기법은 **상태(*state*)와 가변성(*mutability - 생성 후 상태를 바꿀 수 있으면 가변, 없으면 불변*)이 클래스 외부로 노출되는 것을 없앤다.** `new UserSettings()`는 프라이빗 생성자 탓에 컴파일되지 않으므로, 잘못된 상태의 인스턴스를 만드는 것이 애초에 불가능하다. (상태·가변성 최소화는 7장에서 자세히 다룬다.) 다만 로딩 실패 시 널만 반환하면 실패 이유를 알 수 없어 디버깅이 어렵다 — 이 한계는 4장(오류)에서 다룬다.

## 4. 체크 및 어서션

컴파일러로 계약을 강제할 실질적 방법이 없을 때는 **런타임 검사**로 확인할 수 있다. 컴파일 타임 확인만큼 강력하진 않지만(실행 중에야 위반이 드러남), 아예 확인하지 않는 것보다는 낫다.

### 4.1 체크

**체크(*check*)**는 코드 계약이 준수되었는지 확인하는 추가 로직으로, 위반 시 놓칠 수 없는 명백한 오류를 낸다. **신속한 실패(*fail fast*)**와 밀접하다(4장). 전제 조건 검사(입력·초기화·상태)와 사후 상태 검사(반환값·상태)로 나뉜다.

> **비유 — 스쿠터 안전장치**: 체크는 시속 30마일에 도달하면 스쿠터를 완전히 정지시키는 안전장치와 같다. 모터 손상은 막지만 갑자기 멈춰 짜증나거나 위험할 수 있다. 애초에 초과가 불가능한 속도 제한 장치(컴파일 타임 차단)가 최상이다.

<details>
<summary>의사코드 (원서) — 예제 3.4 계약 시행을 위한 체크 사용</summary>

```java
class UserSettings {
  ...
  void init() {
    // 클래스가 유효하지 않은 방법으로 사용되면 예외가 발생한다.
    if (!haveSettingsBeenLoaded()) {
      throw new StateException("Settings not loaded");
    }
    ...
  }

  Color? getUiColor() {
    if (!hasBeenInitialized()) {
      throw new StateException("Settings not initialized");
    }
    ...
  }
}
```

</details>

```typescript
class UserSettings {
  init(): void {
    // 클래스가 유효하지 않은 방식으로 사용되면 예외가 발생한다.
    if (!this.haveSettingsBeenLoaded()) {
      throw new StateError("Settings not loaded");
    }
    // ...
  }

  getUiColor(): Color | null {
    if (!this.hasBeenInitialized()) {
      throw new StateError("Settings not initialized");
    }
    // ...
  }
}
```

체크의 기대는 오용이 **개발·테스트 단계에서 발견·수정**되는 것이지만, 효과가 보장되진 않는다 — 테스트가 그 조건을 건드리지 않으면 배포 후에야 드러나고, 예외가 상위에서 조용히 처리·로깅되면 아무도 알아차리지 못할 수 있다.

> **참고 — 퍼즈 테스트(fuzz test)**: 버그나 잘못된 설정을 드러낼 입력값(예: 임의의 문자열 다수)을 생성해 장애·예외가 나는지 확인하는 테스트다. 퍼즈 테스트는 오류·예외 발생에 의존하므로, 체크·어서션과 **함께** 쓰면 미묘한 버그 발견 확률이 높아진다.

### 4.2 어서션

**어서션(*assertion*)**은 많은 언어가 언어 차원에서 지원하며 체크와 매우 유사하다. 핵심 차이는 **배포 빌드에서는 보통 컴파일에서 제외**된다는 점이다 — (1) 성능(검사에 CPU 사이클 소모), (2) 오류 발생률(가용성이 더 중요한 시스템의 절충) 때문이다. 배포 빌드에도 어서션을 남기는 팀도 많으며, 그 경우 체크와 다를 바 없다.

<details>
<summary>의사코드 (원서) — 예제 3.5 계약 시행을 위한 어서션 사용</summary>

```java
class UserSettings {
  Color? getUiColor() {
    // 클래스가 유효하지 않은 방식으로 사용되면 어서션이 에러나 예외를 발생시킨다.
    assert(hasBeenInitialized(), "UserSettings가 초기화되지 않음");
    ...
  }
}
```

</details>

```typescript
class UserSettings {
  getUiColor(): Color | null {
    // 유효하지 않은 방식으로 사용되면 에러를 발생시킨다.
    // (TS/JS에는 표준 assert가 없어 직접 구현하거나 node:assert 등을 쓴다.)
    assert(this.hasBeenInitialized(), "UserSettings가 초기화되지 않음");
    // ...
  }
}
```

체크와 어서션 모두, 계약에 세부 조항이 있을 때 유용하다. 하지만 **애초에 세부 조항을 피하는 것이 더 바람직하다.** 코드에 체크가 많다면 세부 조항 자체를 없앨 수 있는지 고민해 봐야 한다는 신호다.

## 5. 코드 품질 6대 요소 연결

이 장의 코드 계약·세부 조항 최소화 기법은 6대 요소 중 다음 둘의 **근본 기술**이다(1장에서 명시).

| 6대 요소 | 이 장이 기여하는가? | 근거 |
|---|---|---|
| 예측 가능 | ✅ (근본) | 명백한 계약은 널의 이중 의미 같은 놀라움을 없앤다 |
| 오용 어렵게 | ✅ (근본) | 잘못된 상태·순서를 컴파일 타임에 불가능하게 만든다 |
| 읽기 쉬움 | (간접) | 좋은 이름·타입은 계약을 드러내 가독성도 높인다 |
| 모듈화 | (간접) | 계약이 명확할수록 계층 경계가 뚜렷해진다 |
| 재사용·일반화 | (간접) | 세부 조항이 적은 코드일수록 안전하게 재사용된다 |
| 테스트 용이 | (간접) | 체크·어서션·퍼즈 테스트가 계약 위반을 드러낸다 |

> **핵심 통찰**: 계약을 전달하는 수단은 **신뢰도 순서**가 있다 — (1) 컴파일 타임 강제(타입·프라이빗 생성자·정적 팩토리) > (2) 런타임 체크·어서션 > (3) 세부 조항(주석·문서). 위쪽일수록 좋고, 세부 조항 의존은 최후의 수단이다. 이 장의 도구는 6장(예측 가능)·7장(오용 방지)에서 더 구체적인 기법으로 확장된다.

## 요약

- 코드베이스는 계속 변하고, 일반적으로 여러 개발자에 의해 변경된다. "미래의 나"도 다른 개발자로 간주해야 한다.
- 다른 개발자가 어떻게 코드를 해석·오용할 수 있을지 생각하고, 그 가능성을 최소화하거나 아예 불가능하게 만드는 방식으로 작성하는 것이 유용하다.
- 코드 사용법을 전달하는 다섯 가지 방법 중 **이름·데이터 유형이 가장 신뢰할 만하고**, 문서·직접 질문·코드 읽기는 신뢰도가 낮다.
- 코드를 작성하면 항상 일종의 **코드 계약**이 만들어진다. 여기엔 명백한 항목과 세부 조항이 있다.
- 세부 조항은 계약 전달의 신뢰할 만한 방법이 아니다. **명백한 항목으로 전달하는 것이 더 낫다.**
- 컴파일러로 계약을 확인하는 것이 가장 신뢰할 수 있다. 불가능하면 **체크·어서션**으로 런타임에 확인하되, 세부 조항 자체를 없앨 수 있는지 먼저 고민한다.

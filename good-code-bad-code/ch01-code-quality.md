# Chapter 1: Code Quality (코드 품질)

## 핵심 질문

코드 품질이란 정확히 무엇이며 왜 중요한가? 고품질 코드가 달성하려는 목표는 무엇인가? 일상의 코딩에서 품질을 높이기 위해 적용할 수 있는 구체적인 전략은 무엇인가? 고품질 코드를 작성하면 정말 개발 일정이 지연되는가?

---

## 0. 이 장에서 다루는 것

- 코드 품질이 중요한 이유
- 고품질 코드가 이루고자 하는 **네 가지 목표**
- 고품질 코드 작성을 위한 높은 수준의 **여섯 가지 전략**(코드 품질의 6대 핵심 요소)
- 고품질 코드 작성이 어떻게 중장기적으로 시간과 노력을 절약하는가

소프트웨어 공학은 엄밀한 과학이 아니며, 절대적인 규칙 몇 가지로 요약할 수 없다. 모든 프로젝트는 서로 다르고, 거의 모든 경우에 트레이드오프가 존재한다. 따라서 좋은 코드를 작성하려면 규칙을 맹목적으로 따르기보다 **당면한 상황에 합리적인 판단력을 적용**하고, 특정 행동 방식의 결과가 어떻게 될지 주의 깊게 생각해야 한다.

> **핵심 통찰**: 코드 품질은 "사소하고 별로 중요하지 않은 세세한 충고"처럼 들리지만 실제로는 그렇지 않다. 코드 품질의 모든 원칙은 단 하나의 실질적 목표에 근거를 둔다 — **더 나은 소프트웨어를 만드는 것**. 소프트웨어가 하는 모든 일은 결국 누군가가 그것을 실현하기 위해 코드를 작성했기 때문에 일어난다.

---

## 1. 코드는 어떻게 소프트웨어가 되는가

코드는 작성하자마자 실행되는 소프트웨어가 되는 것이 아니라, 여러 단계의 검사와 테스트(소프트웨어 개발 및 배포 프로세스)를 거친다. 핵심 용어는 다음과 같다.

- **코드베이스(codebase)**: 소프트웨어를 빌드할 수 있는 코드가 저장된 저장소. 보통 깃·서브버전·퍼포스 같은 형상 관리 시스템으로 관리된다.
- **코드 제출(submitting code)**: '코드 커밋', '풀 요청 병합'이라고도 한다. 로컬 복사본에서 변경한 코드를 메인 코드베이스에 반영하는 것.
- **코드 검토(code review)**: 코드베이스에 제출하기 전 다른 엔지니어가 변경 내용을 검토하는 것. 원고 교정과 비슷하며, 작성자가 놓친 문제를 발견한다.
- **제출 전 검사(pre-submit check)**: '병합 전 훅/점검', '커밋 전 점검'이라고도 한다. 테스트가 실패하거나 컴파일되지 않으면 병합을 차단한다.
- **배포(release)**: 코드베이스의 스냅숏을 기반으로 빌드하여 품질 보증 검사 후 실제 환경에 내보내는 것.
- **프로덕션(production)**: 테스트 환경이 아닌 실제 서비스되는 환경. 소프트웨어가 비즈니스 작업을 수행하면 "프로덕션 환경에서 실행된다"고 말한다.

전형적인 흐름: (1) 개발자가 로컬 복사본에서 코드를 변경 → (2) 코드 검토를 위해 병합 요청 → (3) 다른 개발자가 검토·제안 → (4) 합의되면 코드베이스에 병합 → (5) 코드베이스에서 주기적으로 배포 → (6) 테스트 실패·컴파일 실패 시 병합/배포를 차단. 세부 사항과 자동화 수준은 조직·팀마다 다르다.

## 2. 코드 품질의 네 가지 목표

코드를 고품질/저품질로 나누는 것은 주관적이므로, 한발 물러서서 **"코드를 통해 정말로 달성하려는 것이 무엇인가"**를 생각하면 객관적이 된다. 저자는 코드를 작성할 때 다음 네 가지 상위 목표를 달성하려 한다.

### 2.1 코드는 작동해야 한다

코드를 작성하는 이유는 문제 해결(기능 구현, 버그 수정, 작업 수행)이다. 첫 번째 목표는 애초 작성된 목적대로 동작하는 것이며, 이는 버그가 없어야 함을 의미한다. "작동한다"를 정의할 때는 **모든 요구사항**을 파악해야 한다 — 성능(지연 시간·CPU 사용량), 개인정보 보호, 보안 같은 고려사항도 요구사항의 일부다.

### 2.2 코드는 작동이 멈춰서는 안 된다

코드 작동은 매우 일시적일 수 있다. 코드는 고립된 환경에서 홀로 실행되지 않으므로 주변 상황이 바뀌면 멈출 수 있다.

- 의존하는 다른 코드가 수정·변경될 수 있다
- 새로운 기능이 필요할 때 코드를 수정해야 할 수 있다
- 해결하려는 문제 자체가 시간이 지나며 변한다(소비자 선호, 비즈니스 요구, 기술)

> **핵심 통찰**: 당장 돌아가는 코드를 만들기는 쉽지만, **변화하는 환경과 요구사항에도 계속 작동하는 코드**를 만드는 것은 훨씬 어렵다. "나중에 테스트 몇 개 추가하면 되겠지"라고 가정하는 것은 효과적인 접근법이 아니다 — 코딩의 모든 단계에서 고려해야 한다.

### 2.3 코드는 변경된 요구사항에 적응할 수 있어야 한다

한 번 작성하고 다시는 수정되지 않는 코드는 거의 없다. 요구사항이 어떻게 변할지 정확히 예측할 수는 없지만, 변한다는 사실 자체를 무시해서는 안 된다. 두 극단을 보자.

- **시나리오 A**: 모든 잠재적 미래 변화를 예측해 대비하려 한다 → 개발 기간이 폭증(3개월 소프트웨어를 1년 넘게 붙듦), 예측이 틀리면 시간 낭비
- **시나리오 B**: 변화 가능성을 완전히 무시하고 현재 요구사항만 충족 → 조금만 지나면 유효하지 않은 가정이 곳곳에 생기고, 처음부터 다시 짜야 함

두 극단 모두 나쁜 결과를 낳는다. 최적점은 프로젝트와 조직 문화에 달려 있으며, 이 책은 그 중간을 위한 실용적 기술들을 다룬다.

### 2.4 코드는 이미 존재하는 기능을 중복 구현해서는 안 된다

큰 문제를 작은 하위 문제로 나누다 보면, 많은 하위 문제는 이미 다른 사람이 해결해 두었다(예: 파일에서 바이트 로드, 바이트 분석해 이미지 생성). 기존 코드를 재사용하면 좋은 이유:

1. **시간과 노력을 절약한다**: 언어가 지원하는 기능을 쓰면 몇 줄·몇 분이면 되지만, 직접 다시 작성하면 며칠이 걸린다
2. **버그 가능성을 줄인다**: 기존 코드는 이미 철저히 테스트되고 실제 환경에서 검증되었다
3. **기존 전문 지식을 활용한다**: 해당 코드를 관리하는 팀의 전문성과 향후 업데이트를 그대로 받는다
4. **코드가 이해하기 쉽다**: 표준화된 방식은 다른 개발자가 이미 접해봤을 가능성이 크다

이 개념은 양방향이다. 남이 이미 작성했으면 그것을 쓰고, 내가 작성했으면 **남이 다시 짜지 않도록 재사용하기 쉬운 방식으로 구성**해야 한다.

## 3. 코드 품질의 여섯 가지 핵심 요소

네 가지 목표는 "무엇을 성취하려는가"에 초점을 맞추지만, 일상 코딩에 대한 구체적 조언은 아니다. 이 책은 목표에 부합하는 코드를 작성하기 위한 여섯 가지 전략(코드 품질의 6대 핵심 요소)을 중심으로 진행된다.

### 3.1 코드는 읽기 쉬워야 한다

> **비유 — 초콜릿 브라우니 레시피**: 그릇을 A, 냄비를 B라 부르고 재료·분량·오븐 예열이 뒤죽박죽 섞인 레시피는 무엇을 만드는지조차 파악하기 어렵다. 형편없이 짜인 코드를 읽는 경험이 바로 이렇다.

코드를 읽을 때 우리는 (1) 코드가 하는 일, (2) 어떻게 그 일을 하는지, (3) 무엇을 필요로 하는지(입력·상태), (4) 실행 결과물을 이해하려 애쓴다. 가독성이 낮으면 이해에 시간이 오래 걸리고, 기능을 오해하거나 중요한 세부사항을 놓쳐 **코드 검토 중 버그를 못 잡거나 수정 시 새 버그를 유입**시킨다. → **2장(추상화 계층)**, **5장(가독성 기법)**에서 다룬다.

### 3.2 코드는 예측 가능해야 한다

> **비유 — 피자 vs 마르가리타 칵테일**: 전화 걸기 앱이 "통화 중이면 자동으로 다음 식당에 전화"하는 '똑똑한' 기능을 추가하자, 사용자는 피자집에 주문했다고 믿었지만 실제로는 멕시코 식당에 마르가리타 칵테일을 주문하게 된다. 좋은 의도였지만 사용자의 **정신 모델(mental model)**을 깨뜨렸다.

코드 사용자는 이름·데이터 유형·관행 같은 단서로 코드의 정신 모델을 구축한다. 코드가 이 모델과 어긋나는 일을 하면 버그가 아무도 모르게 유입되고, 나중에 전혀 상관없어 보이는 곳에서 이상 현상이 터진다(때로는 데이터 손상 같은 재앙). → **3장(코드 계약)**, **4장(오류)**, **6장(예측 벗어나는 코드 피하기)**에서 다룬다.

### 3.3 코드를 오용하기 어렵게 만들라

> **비유 — TV 뒷면 소켓**: 제조사는 전원 코드를 HDMI 소켓에 꽂지 못하도록 소켓 모양을 다르게 만든다. 모든 소켓이 같은 모양이면 잘못 꽂아 TV가 폭발할 수도 있다.

내 코드는 다른 코드에 의해 호출되며, 호출 시 특정 인수·상태를 기대한다(TV 뒷면과 비슷). 잘못된 것이 "꽂히면" 시스템이 멈추거나 DB가 손상될 수 있다. 오용을 어렵거나 불가능하게 만들면 코드가 계속 잘 작동할 가능성이 극대화된다. → **3장(코드 계약)**, **7장(오용하기 어려운 코드)**에서 다룬다.

### 3.4 코드를 모듈화하라

> **비유 — 조립 장난감 vs 꿰맨 인형**: 머리·팔·손·다리를 못과 구멍(잘 정의된 인터페이스)으로 끼운 장난감은 손 하나를 쉽게 교체할 수 있다. 반면 20개 실로 얽혀 꿰매진 인형은 손을 바꾸려면 전체를 뜯어야 한다.

모듈화란 구성 요소가 **독립적으로 교환·교체될 수 있음**을 의미한다. 핵심은 인터페이스가 잘 정의되어 구성 요소 간 상호작용 지점이 최소화되는 것이다. 한 기능을 바꿔도 다른 부분을 건드릴 필요가 없어 요구사항 변화에 쉽게 적응하고, 이해·추론이 쉬워진다. → **2장(추상화 계층)**, **8장(모듈화 기법)**에서 다룬다.

### 3.5 코드를 재사용 가능하고 일반화할 수 있게 작성하라

> **비유 — 핸드 드릴**: 드릴은 벽·바닥·천장 어디든 구멍을 뚫을 수 있어 **재사용 가능**하고(같은 문제, 다른 상황), 구멍 뚫기뿐 아니라 나사도 박을 수 있어 **일반화**되어 있다(미묘하게 다른 문제들). 방향별 전용 드릴 4개를 따로 사는 대신 하나면 된다.

- **재사용성(reusability)**: 같은 문제를 여러 다른 상황에서 해결
- **일반화성(generalizability)**: 개념적으로 유사하지만 미묘하게 다른 문제들을 해결

> **핵심 통찰**: 우리는 코드를 많이 작성한 것에 대한 대가를 받는 것이 아니라 **문제를 해결한 것**에 대한 대가를 받는다. 코드는 수단일 뿐이며, 코드 라인이 적을수록 좋다(버그가 생길 여지가 줄어든다). → **9장(재사용·일반화 기법)**에서 다룬다.

### 3.6 테스트가 용이한 코드를 작성하고 제대로 테스트하라

이 요소에는 두 가지 개념이 있다.

- **테스트(test)**: 코드/소프트웨어를 실제로 돌려보고 정상 작동을 확인하는 것. 수준별로 **단위 테스트(unit test, 개별 함수·클래스)**, **통합 테스트(integration test, 구성 요소 연결)**, **종단 간(E2E) 테스트(전체 워크플로)**가 있다. 이 책은 일상 코딩에서 가장 유용한 **단위 테스트**만 자세히 다룬다
- **테스트 용이성(testability)**: 테스트 대상이 되는 '실제' 코드가 얼마나 테스트하기 적합한지. 모듈화와 깊은 관련이 있다

> **비유 — 비상 브레이크 시스템**: 모듈화가 안 되어 있으면 실제 차에 설치해 보행자 도로에서 운전하며 테스트해야 한다(비싸고 위험). 별개의 모듈이면 녹화 영상으로 신호 출력만 확인하면 되어 수천 시나리오를 싸고 안전하게 테스트할 수 있다.

> **핵심 통찰**: 테스트는 코드를 다 작성한 후가 아니라 **작성의 모든 단계에서** 고려해야 한다. "코드를 작성하기 전에 테스트부터 작성하라"는 것이 TDD(*Test-Driven Development - 테스트 주도 개발*)의 관행이며, 10장(10.5절)에서 다룬다. → **10·11장**이 테스트 전용이지만, 테스트는 책 전체에 걸쳐 다뤄진다.

## 4. 고품질 코드 작성은 일정을 지연시키는가?

단기적으로는 **그렇다** — 머릿속에 떠오르는 것을 바로 코딩하는 것보다 조금 더 많은 생각과 노력이 든다. 하지만 한 번 쓰고 버릴 유틸리티가 아닌 중요한 소프트웨어라면, **중장기적으로는 개발 시간을 단축**해 준다.

> **비유 — 선반 설치**: 벽 기둥에 브래킷을 달아 선반을 장착하는 '적절한 방법'(30분)과 접착제로 붙이는 '임시변통'(10분)이 있다. 접착제는 20분을 아끼는 듯하지만, 석고가 갈라져 선반이 떨어지면 벽을 다시 꾸며야 하고(몇 시간~며칠), 방을 다시 꾸밀 때도 골칫거리가 된다. 임시변통으로 시작하면 또 다른 임시변통을 부른다.

> 서두르지 않으면 더 빠르다.

이 말은 무언가를 충분히 생각하지 않고 성급하게 행동하면 오히려 전반적인 속도가 늦어진다는 뜻으로, 고품질 코드가 왜 시간을 절약하는지를 한 문장으로 요약한다.

## 5. 코드 품질 6대 요소 연결 (책 전체 지도)

이 책의 나머지 장은 6대 핵심 요소를 구체적 기법으로 풀어낸다. 각 요소가 앞의 네 가지 목표 중 무엇에 기여하며 어디서 다뤄지는지 정리하면 다음과 같다.

| 6대 핵심 요소 | 주로 기여하는 목표 | 다루는 장 |
|---|---|---|
| 1. 읽기 쉬운 코드 | 작동·적응 | 2장(추상화 계층), 5장 |
| 2. 예측 가능한 코드 | 작동·멈추지 않음 | 3장(계약), 4장(오류), 6장 |
| 3. 오용하기 어려운 코드 | 작동·멈추지 않음 | 3장(계약), 7장 |
| 4. 모듈화된 코드 | 적응·멈추지 않음 | 2장(추상화 계층), 8장 |
| 5. 재사용·일반화 가능한 코드 | 중복 구현 방지·적응 | 9장 |
| 6. 테스트가 용이한 코드 | 작동·멈추지 않음 | 10·11장(+책 전체) |

> **핵심 통찰**: 6대 요소는 독립적이지 않고 서로 얽혀 있다. 특히 **모듈화**는 재사용성·일반화성·테스트 용이성을 동시에 끌어올리는 핵심 지렛대이며, **코드 계약(3장)**은 예측 가능성과 오용 방지 둘 다의 근본 기술이다.

## 요약

- 좋은 소프트웨어를 만들려면 고품질 코드를 작성해야 한다. 코드는 여러 단계의 검사·테스트(수동·자동)를 거쳐 프로덕션에 배포된다.
- 고품질 코드의 **네 가지 목표**: (1) 작동해야 한다, (2) 작동이 멈춰서는 안 된다, (3) 변경된 요구사항에 적응해야 한다, (4) 기존 기능을 중복 구현해서는 안 된다.
- 목표를 달성하기 위한 **여섯 가지 핵심 요소**: 읽기 쉬움 / 예측 가능 / 오용 어렵게 / 모듈화 / 재사용·일반화 / 테스트 용이.
- 각 요소는 브라우니 레시피·피자/칵테일·TV 소켓·조립 장난감·핸드 드릴·비상 브레이크 같은 일상 비유로 설명된다.
- 테스트는 코드 작성의 마지막이 아니라 모든 단계에서 고려해야 한다.
- 고품질 코드는 단기적으로는 시간이 더 걸리지만 중장기적으로는 개발 시간을 단축한다 — "서두르지 않으면 더 빠르다".

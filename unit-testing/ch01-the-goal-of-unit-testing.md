# Chapter 1: The Goal of Unit Testing (단위 테스트의 목표)

## 핵심 질문

단위 테스트를 왜 하는가? 테스트의 양이 아닌 **질**이 중요하다면, 좋은 테스트와 나쁜 테스트를 어떻게 구분하는가?

---

## 1. 단위 테스트의 현재 상태

### 1.1 논쟁의 변화

단위 테스트를 둘러싼 논쟁은 시대에 따라 변해왔다:

| 시기 | 논쟁의 초점 |
|------|------------|
| 과거 | 단위 테스트를 **해야 하는가?** |
| 현재 | **좋은** 단위 테스트를 어떻게 작성하는가? |

"단위 테스트를 작성해야 하는가?"라는 질문은 이제 논쟁 대상이 아니다. 대부분의 개발자와 기업이 단위 테스트의 필요성에 동의한다. 하지만 문제는 **단위 테스트를 작성하는 것만으로는 충분하지 않다**는 것이다.

### 1.2 나쁜 테스트의 비용

많은 프로젝트에서 테스트 스위트(*Test Suite — 프로젝트에 속한 모든 테스트의 집합*)가 존재하지만 기대한 만큼의 이점을 얻지 못한다. Khorikov는 이 현상을 다음과 같이 진단한다:

- 테스트가 있지만 **버그는 여전히** 프로덕션에서 발견된다
- 테스트가 있지만 **리팩터링이 두렵다** — 코드를 바꾸면 테스트가 깨질까 봐
- 테스트가 있지만 **유지보수 비용이 높다** — 코드 변경마다 테스트도 대량으로 수정해야 한다

> **핵심 통찰**: 테스트는 **자산(asset)**이 아니라 **부채(liability)**가 될 수 있다. 모든 코드는 유지보수 비용을 수반한다. 테스트 코드도 예외가 아니다. 테스트가 가치를 제공하지 못하면서 비용만 발생시킨다면, 그 테스트는 프로젝트에 해를 끼치고 있는 것이다.

---

## 2. 단위 테스트의 목표: 소프트웨어 프로젝트의 지속 가능한 성장

### 2.1 지속 가능한 성장이란?

프로젝트 초기에는 테스트 없이도 빠르게 개발할 수 있다. 하지만 프로젝트가 성장하면서 코드베이스가 커지면, 테스트 없는 프로젝트는 점점 느려진다. 새 기능 추가에 점점 더 많은 시간이 걸리고, 버그 수정이 새로운 버그를 만들어낸다.

이 현상을 Khorikov는 **소프트웨어 엔트로피(software entropy)**라고 부른다:

```
개발 속도
    ↑
    │  ╲
    │   ╲  테스트 없는 프로젝트
    │    ╲
    │     ╲___________
    │
    │  ─────────────── 테스트가 있는 프로젝트
    │
    └──────────────────→ 시간
```

테스트 없는 프로젝트는 초기에 빠르게 시작하지만 **급격히 감속**한다. 좋은 테스트가 있는 프로젝트는 초기에는 약간 느리지만(테스트 작성 비용) **일정한 속도를 유지**한다.

### 2.2 안전망(Safety Net)으로서의 테스트

테스트가 지속 가능한 성장을 가능하게 하는 메커니즘은 **안전망** 역할이다:

1. **회귀(regression) 방지**: 이미 동작하던 기능이 새 변경으로 깨지는 것을 즉시 감지한다
2. **리팩터링 자신감**: 테스트가 뒷받침되면, 내부 구조를 과감하게 개선할 수 있다
3. **살아있는 문서**: 테스트는 시스템이 어떻게 동작하는지를 보여주는 실행 가능한 명세다

> **핵심 통찰**: 단위 테스트의 목표는 "버그를 잡는 것"이 아니다. 진짜 목표는 소프트웨어 프로젝트의 **지속 가능한 성장을 가능하게 하는 것(enable sustainable growth)**이다. 이 미묘한 차이가 중요하다. 버그를 잡는 것은 수단이지 목적이 아니다.

## 3. 좋은 테스트 스위트와 나쁜 테스트 스위트

### 3.1 좋은 테스트 스위트의 특성

좋은 테스트 스위트에는 다음 특성이 있다:

- **개발 사이클에 통합되어 있다** — 테스트를 가끔 실행하는 것이 아니라, 코드를 변경할 때마다 실행한다
- **코드베이스의 가장 중요한 부분만 대상으로 한다** — 모든 코드를 동일한 수준으로 테스트하지 않는다
- **최소한의 유지보수 비용으로 최대한의 가치를 제공한다** — 적은 수의 고품질 테스트가 많은 수의 저품질 테스트보다 낫다

### 3.2 나쁜 테스트의 징후

| 징후 | 문제 |
|------|------|
| 프로덕션 코드 한 줄 수정에 테스트 수십 개가 깨진다 | 테스트가 구현 세부사항에 결합되어 있다 |
| 테스트가 통과해도 프로덕션에서 버그가 나온다 | 테스트가 의미 있는 시나리오를 검증하지 않는다 |
| 테스트를 이해하는 데 몇 분이 걸린다 | 테스트의 가독성이 낮다 |
| 테스트 실행에 너무 오래 걸린다 | 피드백 루프가 느려져 자주 실행하지 않게 된다 |

## 4. 커버리지 지표의 함정

### 4.1 코드 커버리지(Code Coverage)

코드 커버리지는 가장 널리 사용되는 테스트 품질 지표다. 계산 공식은 다음과 같다:

```
코드 커버리지 = 실행된 코드 라인 수 / 전체 코드 라인 수
```

예를 들어:

```csharp
public static bool IsStringLong(string input)
{
    if (input.Length > 5)    // ← 실행됨
        return true;         // ← 실행됨
    return false;            // ← 실행 안 됨
}

[Fact]
public void Test()
{
    bool result = IsStringLong("abc def");
    Assert.True(result);
}
```

이 테스트의 코드 커버리지는 2/3 = **66.7%**이다. 3줄 중 2줄만 실행되었기 때문이다.

### 4.2 분기 커버리지(Branch Coverage)

분기 커버리지는 코드의 분기(if/else, switch 등)를 기준으로 측정한다:

```
분기 커버리지 = 통과한 분기 수 / 전체 분기 수
```

위 예제에서 `if`문에는 두 개의 분기(true, false)가 있고, 테스트는 true 분기만 통과했으므로 분기 커버리지는 1/2 = **50%**이다.

### 4.3 커버리지 지표가 신뢰할 수 없는 이유

Khorikov는 커버리지 지표를 **목표로 삼아서는 안 된다**고 강력히 주장한다. 이유는 두 가지다:

**이유 1: 커버리지를 쉽게 속일 수 있다**

코드를 리팩터링하면 커버리지가 올라갈 수 있다 — 테스트의 품질이 전혀 변하지 않았는데도:

```csharp
// 리팩터링 전 (코드 커버리지 66.7%)
public static bool IsStringLong(string input)
{
    if (input.Length > 5)
        return true;
    return false;
}

// 리팩터링 후 (코드 커버리지 100%)
public static bool IsStringLong(string input)
{
    return input.Length > 5;
}
```

같은 테스트인데 코드가 한 줄이 되면서 커버리지가 100%로 올랐다. 테스트의 실제 검증 능력은 변하지 않았다.

**이유 2: 실행과 검증은 다르다**

커버리지는 코드가 "실행되었는지"만 측정하지, 실행 결과가 "검증되었는지"는 측정하지 못한다:

```csharp
[Fact]
public void Test()
{
    bool result = IsStringLong("abc def");
    // Assert 없음 — 아무것도 검증하지 않지만 커버리지는 100%
}
```

이 테스트는 코드 커버리지 100%를 달성하지만, **아무것도 검증하지 않는다**. 어떤 버그도 잡을 수 없다.

> **핵심 통찰**: 커버리지 지표는 좋은 테스트 스위트의 **필요조건**이지 **충분조건**이 아니다. 커버리지가 낮으면 문제가 있다고 말할 수 있지만, 커버리지가 높다고 테스트가 좋다고 말할 수는 없다. Khorikov는 "커버리지 지표를 **최소 기준(threshold)**으로는 사용하되, **목표(goal)**로는 삼지 말라"고 조언한다.

### 4.4 커버리지 목표를 강제하면 안 되는 이유

팀에서 "커버리지 90% 이상"을 강제하면 어떤 일이 벌어지는가?

1. 개발자들이 커버리지를 올리기 위해 **의미 없는 테스트**를 작성한다
2. Assert 없는 테스트, trivial getter/setter 테스트 등이 늘어난다
3. 테스트 스위트의 유지보수 비용은 올라가지만 가치는 올라가지 않는다
4. 결과적으로 테스트에 대한 **팀의 신뢰가 하락**한다

## 5. 무엇이 성공적인 테스트 스위트를 만드는가?

### 5.1 가치와 비용의 균형

모든 테스트에는 비용이 수반된다:

| 비용 항목 | 설명 |
|----------|------|
| **작성 비용** | 테스트를 처음 만드는 데 드는 시간 |
| **실행 비용** | 테스트를 실행하는 데 걸리는 시간 |
| **유지보수 비용** | 프로덕션 코드 변경 시 테스트를 수정하는 비용 |
| **이해 비용** | 테스트를 읽고 이해하는 데 드는 시간 |

테스트의 가치가 이 비용의 합을 초과해야만, 그 테스트는 프로젝트에 기여하는 것이다.

### 5.2 가장 중요한 부분에 집중하기

모든 코드가 동일한 중요도를 갖지 않는다. Khorikov는 테스트 노력을 다음과 같이 집중할 것을 권한다:

```
테스트 가치가 높은 코드:
  ✓ 비즈니스 로직 (도메인 모델)
  ✓ 복잡한 알고리즘
  ✓ 여러 곳에서 사용되는 핵심 유틸리티

테스트 가치가 낮은 코드:
  ✗ 단순한 CRUD 연산
  ✗ 외부 라이브러리의 래퍼(wrapper)
  ✗ 글루 코드(glue code) — 컴포넌트를 연결만 하는 코드
```

> **핵심 통찰**: "이 코드를 테스트해야 하는가?"가 아니라, "이 코드를 테스트하는 데 드는 비용 대비 얼마나 가치가 있는가?"를 물어야 한다. 테스트 작성은 투자이며, 모든 투자는 ROI(투자 수익률)를 따져야 한다.

## 요약

- 단위 테스트의 목표는 "버그를 잡는 것"이 아니라 **소프트웨어 프로젝트의 지속 가능한 성장을 가능하게 하는 것**이다.
- 테스트는 **안전망** 역할을 하여 회귀를 방지하고, 리팩터링에 대한 자신감을 주며, 코드의 동작을 문서화한다.
- 나쁜 테스트는 자산이 아니라 **부채**다. 가치 없는 테스트는 유지보수 비용만 발생시킨다.
- 코드 커버리지는 좋은 테스트의 **필요조건이지 충분조건이 아니다**. 커버리지를 목표로 삼으면 의미 없는 테스트가 양산된다.
- 좋은 테스트 스위트는 **비용 대비 가치가 높은 테스트**로 구성되며, 비즈니스 로직처럼 중요한 코드에 집중한다.
- 테스트 작성은 투자다. **모든 테스트의 ROI**를 따져야 한다.

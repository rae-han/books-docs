# Chapter 4: The Four Pillars of a Good Unit Test (좋은 단위 테스트의 네 가지 기둥)

## 핵심 질문

좋은 단위 테스트와 나쁜 단위 테스트를 구분하는 **객관적인 기준**은 무엇인가? 네 가지 기둥 중 어떤 것이 가장 중요하며, 왜 모든 기둥을 동시에 최대화할 수 없는가?

---

## 1. 네 가지 기둥 개요

Khorikov는 좋은 단위 테스트를 판별하는 네 가지 기둥을 제시한다:

| 기둥 | 핵심 질문 |
|------|----------|
| **1. 회귀 방지 (Protection against regressions)** | 이 테스트가 버그를 얼마나 잘 찾아내는가? |
| **2. 리팩터링 내성 (Resistance to refactoring)** | 리팩터링 시 거짓 양성(false positive)을 얼마나 적게 만드는가? |
| **3. 빠른 피드백 (Fast feedback)** | 얼마나 빨리 실행되는가? |
| **4. 유지보수성 (Maintainability)** | 테스트를 이해하고 수정하기 얼마나 쉬운가? |

> **핵심 통찰**: 이 네 가지 기둥은 Khorikov가 책 전체에서 반복적으로 사용하는 **핵심 프레임워크**다. 모든 테스트 관련 결정 — Mock 사용 여부, 테스트 스타일 선택, 통합 테스트 범위 등 — 이 네 가지 기둥에 비추어 평가된다.

---

## 2. 첫 번째 기둥: 회귀 방지

### 2.1 회귀란?

회귀(*Regression — 이전에 올바르게 동작하던 기능이 코드 변경 후 더 이상 동작하지 않는 것*)는 소프트웨어에서 가장 흔한 문제 중 하나다. 코드가 많아질수록 회귀의 가능성이 높아진다.

### 2.2 회귀 방지 능력을 높이는 요인

테스트의 회귀 방지 능력은 다음 세 가지에 의해 결정된다:

1. **테스트가 실행하는 코드의 양**: 더 많은 코드를 실행할수록 회귀를 발견할 가능성이 높다
2. **코드의 복잡도**: 복잡한 비즈니스 로직의 회귀가 단순한 getter의 회귀보다 중요하다
3. **코드의 도메인 유의성(*Domain significance — 코드가 비즈니스 문제 영역과 얼마나 직접적으로 관련되어 있는지를 나타내는 척도*)**: 도메인 로직의 버그는 유틸리티 코드의 버그보다 비즈니스에 훨씬 큰 영향을 미친다

```
회귀 방지 능력 = 실행 코드 양 × 코드 복잡도 × 도메인 유의성
```

### 2.3 trivial 코드를 테스트하지 말라

```csharp
// 테스트 가치가 낮은 코드
public string Name { get; set; }

// 테스트 가치가 높은 코드
public decimal CalculateDiscount(Customer customer, Product product)
{
    decimal discount = customer.Tier switch
    {
        CustomerTier.Gold => 0.1m,
        CustomerTier.Silver => 0.05m,
        _ => 0
    };

    if (product.IsOnSale)
        discount += 0.15m;

    return Math.Min(discount, 0.25m);  // 최대 25% 할인
}
```

getter/setter, 단순 위임, 보일러플레이트 코드를 테스트하는 것은 회귀 방지에 기여하지 않으면서 유지보수 비용만 늘린다.

## 3. 두 번째 기둥: 리팩터링 내성

### 3.1 거짓 양성(False Positive)의 문제

리팩터링 내성은 **거짓 양성을 만들어내지 않는 정도**를 의미한다.

| 용어 | 의미 | 결과 |
|------|------|------|
| **거짓 음성 (False Negative)** | 버그가 있는데 테스트가 통과한다 | 회귀 방지 실패 |
| **거짓 양성 (False Positive)** | 버그가 없는데 테스트가 실패한다 | 리팩터링 내성 실패 |

거짓 양성은 왜 해로운가?

1. **신뢰 저하**: 테스트가 자주 거짓으로 실패하면, 팀은 테스트 실패를 무시하기 시작한다 ("또 그 테스트겠지...")
2. **리팩터링 포기**: 코드를 개선하고 싶어도 "테스트가 깨질까 봐" 리팩터링을 포기한다
3. **생산성 하락**: 거짓 양성을 조사하고 테스트를 수정하는 데 시간을 낭비한다

> **핵심 통찰**: 거짓 양성은 **"양치기 소년" 효과**를 만든다. 테스트가 자주 거짓으로 실패하면, 진짜 실패(회귀)도 무시하게 된다. 이는 테스트 스위트의 가치를 근본적으로 훼손한다.

### 3.2 거짓 양성의 원인: 구현 세부사항에 결합

거짓 양성은 테스트가 **구현 세부사항(implementation detail)**에 결합될 때 발생한다:

```csharp
// 구현 세부사항에 결합된 테스트 → 거짓 양성 발생
[Fact]
public void Message_is_sent_via_grpc()
{
    var messageBus = new Mock<IMessageBus>();
    var sut = new OrderService(messageBus.Object);

    sut.PlaceOrder(new Order());

    messageBus.Verify(                               // "gRPC로 보냈는가?"를 검증
        x => x.SendViaGrpc(It.IsAny<OrderPlaced>()),
        Times.Once);
}
```

나중에 gRPC를 HTTP로 변경하면, 동작은 동일한데(주문 이벤트 발송) 테스트가 깨진다.

### 3.3 해결: 최종 결과(Observable Behavior)를 검증하라

```csharp
// 최종 결과를 검증하는 테스트 → 리팩터링에 강함
[Fact]
public void Placing_an_order_sends_order_placed_event()
{
    var messageBus = new FakeMessageBus();
    var sut = new OrderService(messageBus);

    sut.PlaceOrder(new Order());

    Assert.Contains(messageBus.SentMessages,           // "이벤트가 발송되었는가?"를 검증
        m => m is OrderPlaced);
}
```

gRPC든 HTTP든 RabbitMQ든 — 이벤트만 발송되면 테스트는 통과한다.

## 4. 세 번째 기둥: 빠른 피드백

테스트가 느리면 자주 실행하지 않게 되고, 자주 실행하지 않으면 피드백이 늦어진다. 피드백이 늦으면 버그를 늦게 발견하고, 늦게 발견한 버그는 수정 비용이 높다.

```
빠른 피드백의 선순환:
테스트 빠름 → 자주 실행 → 빠른 회귀 발견 → 낮은 수정 비용

느린 피드백의 악순환:
테스트 느림 → 가끔 실행 → 늦은 회귀 발견 → 높은 수정 비용
```

단위 테스트는 본질적으로 빠르다(밀리초 단위). 통합 테스트나 E2E 테스트는 느리다(초~분 단위). 이것이 테스트 피라미드에서 단위 테스트가 가장 많아야 하는 이유 중 하나다.

## 5. 네 번째 기둥: 유지보수성

유지보수성은 두 가지 요소로 구성된다:

| 요소 | 설명 |
|------|------|
| **이해 난이도** | 테스트가 얼마나 읽기 쉬운가? 테스트가 클수록 이해하기 어렵다 |
| **실행 난이도** | 테스트를 실행하기 위해 얼마나 많은 인프라가 필요한가? (DB, 외부 서비스 등) |

유지보수성은 Chapter 3에서 다룬 AAA 패턴, 네이밍 규칙, 팩토리 메서드 등의 관행과 직접 연결된다.

---

## 6. 네 가지 기둥 사이의 관계

### 6.1 이상적인 테스트는 존재하지 않는다

네 가지 기둥을 **모두 최대화하는 테스트는 불가능**하다. 처음 세 가지 기둥 — 회귀 방지, 리팩터링 내성, 빠른 피드백 — 사이에는 본질적인 **트레이드오프**가 있다:

```
                 회귀 방지
                    △
                   / \
                  /   \
                 /     \
                /   ×   \     ← 세 꼭짓점을 동시에 달성할 수 없다
               /         \
              /           \
빠른 피드백 ◁─────────────▷ 리팩터링 내성
```

| 테스트 유형 | 회귀 방지 | 리팩터링 내성 | 빠른 피드백 |
|------------|----------|-------------|-----------|
| **E2E 테스트** | 높음 | 높음 | **낮음** |
| **Mock 과다 단위 테스트** | 높음 | **낮음** | 높음 |
| **Trivial 테스트** | **낮음** | 높음 | 높음 |

### 6.2 리팩터링 내성은 타협 불가

Khorikov는 네 가지 기둥 중 **리팩터링 내성은 양보할 수 없다**고 주장한다. 이유:

- 리팩터링 내성은 **이진적(binary)**이다 — 있거나 없거나. 중간이 없다
- 회귀 방지와 빠른 피드백은 **연속적(continuous)**이다 — 조금 더/덜 할 수 있다
- 거짓 양성은 **누적**된다 — 시간이 지날수록 테스트 스위트에 대한 신뢰를 점점 더 깎아낸다

따라서 좋은 테스트 전략은:

1. **리팩터링 내성을 최대화**하고 (구현 세부사항이 아닌 동작을 검증)
2. 회귀 방지와 빠른 피드백 사이에서 **상황에 맞게 균형**을 잡는다

> **핵심 통찰**: 리팩터링 내성은 **타협 불가능한 기둥**이다. 테스트가 구현 세부사항에 결합되어 거짓 양성을 만들어내면, 다른 기둥이 아무리 좋아도 그 테스트는 장기적으로 프로젝트에 해를 끼친다. 이것이 이 책의 **가장 중요한 메시지** 중 하나다.

## 7. 테스트 피라미드 재해석

### 7.1 전통적인 테스트 피라미드

```
        △  E2E 테스트 (소수)
       ╱ ╲
      ╱   ╲  통합 테스트 (중간)
     ╱     ╲
    ╱       ╲  단위 테스트 (다수)
   ━━━━━━━━━━
```

### 7.2 네 가지 기둥으로 본 피라미드

| 테스트 유형 | 회귀 방지 | 리팩터링 내성 | 빠른 피드백 | 유지보수성 |
|------------|----------|-------------|-----------|----------|
| **단위 테스트** | 중간 | 높음 | **높음** | 높음 |
| **통합 테스트** | 높음 | 높음 | 중간 | 중간 |
| **E2E 테스트** | **높음** | 높음 | 낮음 | 낮음 |

- **단위 테스트**는 빠른 피드백이 강점. 회귀 방지는 범위가 좁아 상대적으로 제한적
- **E2E 테스트**는 회귀 방지가 강점. 하지만 느리고 유지보수가 어려움
- **통합 테스트**는 균형 잡힌 위치

피라미드 형태인 이유는, **빠른 피드백이 높은 테스트(단위)를 많이** 두고, **회귀 방지가 높지만 느린 테스트(E2E)를 적게** 두어 전체적으로 네 가지 기둥의 균형을 맞추기 위함이다.

## 8. 블랙박스 테스트 vs 화이트박스 테스트

### 8.1 정의

| 접근법 | 관점 | 검증 대상 |
|--------|------|----------|
| **블랙박스 테스트** | SUT의 내부를 모른다고 가정 | 입력과 출력만 검증 |
| **화이트박스 테스트** | SUT의 내부 구조를 알고 테스트 | 분기, 경로, 내부 상태 검증 |

### 8.2 리팩터링 내성과의 관계

| 접근법 | 회귀 방지 | 리팩터링 내성 |
|--------|----------|-------------|
| **블랙박스** | 상대적으로 낮음 | **높음** |
| **화이트박스** | **높음** | 낮음 |

Khorikov는 **블랙박스 테스트를 기본**으로 하되, 회귀 방지를 위해 **테스트를 작성할 때 화이트박스 분석을 참고**하는 접근을 권장한다:

- **테스트 작성 시**: 코드 내부를 분석하여 중요한 시나리오를 식별한다 (화이트박스 분석)
- **테스트 자체는**: 최종 결과만 검증한다 (블랙박스 실행)

> **핵심 통찰**: "화이트박스로 분석하고, 블랙박스로 테스트하라." 코드 내부를 이해하는 것과 테스트가 코드 내부에 결합되는 것은 다른 문제다. 전자는 좋은 테스트 시나리오를 발견하게 해주고, 후자는 테스트를 취약하게 만든다.

---

## 요약

- 좋은 단위 테스트의 네 가지 기둥: **회귀 방지, 리팩터링 내성, 빠른 피드백, 유지보수성**.
- 처음 세 기둥 사이에는 **트레이드오프**가 있어 모두 동시에 최대화할 수 없다.
- 네 기둥 중 **리팩터링 내성은 타협 불가능**하다. 거짓 양성(코드는 정상인데 테스트 실패)은 테스트 스위트의 신뢰를 무너뜨린다.
- 거짓 양성의 원인은 테스트가 **구현 세부사항에 결합**되는 것이다. 최종 결과(동작)를 검증하라.
- 테스트 피라미드는 네 기둥의 **균형을 맞추기 위한 전략**이다. 빠른 단위 테스트를 많이, 느리지만 범위 넓은 E2E 테스트를 적게.
- **화이트박스로 분석하고, 블랙박스로 테스트하라**.

# Chapter 1: What Is Functional Programming? (함수형 프로그래밍이란)

## 핵심 질문

함수형 프로그래밍은 무엇을 지향하는 패러다임인가? 순수 함수·일급 함수·참조 투명성은 서로 어떤 관계이며, 현대 언어들이 함수형을 포함한 멀티패러다임으로 수렴한 이유는 무엇인가?

---

과거에는 프로그래밍 언어를 주로 함수형, 객체지향, 절차적 언어로 나눴다. 그러나 오늘날에는 자바, C#, 스칼라, 타입스크립트, 자바스크립트, 코틀린, 스위프트 등 대부분의 주요 언어가 두 패러다임을 모두 갖춘 멀티패러다임 언어로 자리잡았다. 이 장에서는 패러다임의 지도 위에서 함수형 프로그래밍이 어디에 위치하는지 확인하고, 함수형 프로그래밍을 이루는 핵심 개념 — 순수 함수, 부수효과, 일급 함수, 참조 투명성 — 을 코드로 확인한다.

---

## 1. 프로그래밍 패러다임의 지도

프로그래밍 패러다임은 크게 명령형과 선언형으로 나뉜다.

- **명령형 프로그래밍(*Imperative Programming*)**: 무엇(What)을 할 것인지 나타내기보다, 어떻게(How) 할 것인지 설명하는 방식
  1. **절차지향 프로그래밍**: 수행되어야 할 순차적인 처리 과정을 포함하는 방식
  2. **객체지향 프로그래밍**: 객체들의 집합으로 프로그램의 상호작용을 표현
- **선언형 프로그래밍(*Declarative Programming*)**: 무엇(What)을 할 것인지 설명하는 방식
  - **함수형 프로그래밍**: 순수 함수를 조합하여 소프트웨어를 만드는 방식

함수형 프로그래밍은 거의 모든 것을 순수 함수로 나누어 문제를 해결하는 기법으로, 작은 문제를 해결하기 위한 함수를 작성하여 가독성을 높이고 유지보수를 용이하게 해준다. 『클린 코드』의 저자 로버트 C. 마틴은 함수형 프로그래밍을 **대입문이 없는 프로그래밍**이라 정의했다.

```javascript
let a = [1, 2, 3, 4, 5];

// 명령형 프로그래밍의 예시
for (let i = 0; i < a.length; i++) {
  console.log(a[i],i);
} // i에 값을 대입한다

// 함수형 프로그래밍의 예시
a.forEach((v, i) => console.log(v, i)); 
// 1. 대입문을 사용하지 않는다.
// 2. 작은 문제를 해결하기 위한 함수를 작성한다
// -> 무엇을(What)에 포커스를 두고 출력을 하는 함수를 파라미터로 넘겼다.
```

명령형 예시는 인덱스 `i`에 값을 대입해 가며 **어떻게** 순회할지를 직접 기술하고, 함수형 예시는 각 요소로 **무엇을** 할지 담은 함수를 전달한다.

## 2. 멀티패러다임으로 수렴하는 현대 언어

함수형과 객체지향 기능을 모두 갖춘 하이브리드 언어가 등장하면서, 이제는 자바, C#, 스칼라, 타입스크립트, 자바스크립트, 코틀린, 스위프트 등 대부분의 주요 프로그래밍 언어가 멀티패러다임 언어가 되었다. 멀티패러다임 언어는 클래스, 상속, 인터페이스 같은 객체지향 기능뿐 아니라 일급 함수, 리스트 프로세싱, 지연 평가, 비동기/동시성 처리 등 함수형 프로그래밍 기능도 폭넓게 지원한다. 또한 유연한 타입 시스템, 타입 추론, 제네릭 등을 제공하며 심지어 정적 타입과 동적 타입을 혼합하는 것도 가능하다.

오라일리 미디어의 부사장이자 여러 기술 서적의 저자인 마이크 루키데스는 이렇게 말했다.

> 우리는 함수형, 객체지향, 명령형 패러다임을 제공하는 멀티패러다임 언어를 효율적으로 활용하는 법을 배워야 한다. 많은 툴이 개선되었고 이제는 어떻게 잘 사용할지를 배울 차례다. 그리고 이제는 오랫동안 각기 고유했고 심지어 갈등 속에 있던 패러다임들을 섞어야 할 때다.<br>— 마이크 루키데스(Mike Loukides)

이러한 환경이 갖춰진 지금, 어느 한쪽 패러다임만 사용하는 것은 큰 기회를 놓치는 것일 수 있다. 해결해야 하는 문제 중에는 함수형으로 푸는 것이 우세한 경우가 있고 객체지향으로 푸는 것이 더 우세한 경우도 있다. 프로그래머는 더 이상 함수형이 좋은지 객체지향이 좋은지 논쟁할 필요가 없다. 언어를 만들고 표준을 정의하는 단체와 기업들이 이미 하나의 언어와 플랫폼에서 여러 패러다임을 지원하도록 만들었고, 우리는 하나의 프로젝트, 심지어 하나의 함수 안에서도 여러 패러다임을 조합하여 프로그래밍할 수 있게 되었다.

이 가이드가 함수형 프로그래밍에서 출발해 후반부(Part 5)에서 객체지향과의 조합으로 나아가는 이유가 여기에 있다. 함수형은 대체재가 아니라 멀티패러다임의 한 축이다.

## 3. 함수형 프로그래밍의 정의

> 함수형 프로그래밍은 부수효과가 없는 순수 함수를 일급 객체로 간주하여 파라미터로 넘기거나 반환값으로 사용할 수 있으며, 참조 투명성을 지킬 수 있다.

이 정의에 등장하는 개념을 하나씩 풀어 보자.

### 3.1 부수효과 (Side Effect)

부수효과란 다음과 같은 변화, 또는 변화가 발생하는 작업을 의미한다.

- 변수의 값이 변경됨
- 자료구조를 제자리에서 수정함
- 객체의 필드값을 설정함
- 예외나 오류가 발생하며 실행이 중단됨
- 콘솔 또는 파일 I/O가 발생함

### 3.2 순수 함수 (Pure Function)

이런 부수효과들을 제거한 함수를 순수 함수라 부르며, 함수형 프로그래밍에서 사용하는 함수는 이런 순수 함수들이다.

- 메모리 또는 I/O 관점에서 부수효과가 없는 함수
- 함수의 실행이 외부에 영향을 끼치지 않는 함수

순수 함수를 이용하면 얻을 수 있는 효과는 다음과 같다.

- 함수 자체가 독립적이며 부수효과가 없기 때문에 스레드 안전성(*Thread Safety - 여러 스레드가 동시에 접근해도 프로그램 실행에 문제가 없는 성질*)을 보장받을 수 있다.
- 언제 실행됐는지 평가 시점이 중요하지 않고, 항상 같은 값을 리턴하므로 안전하고 다루기 쉽다.
- 스레드 안전성을 보장받아 병렬 처리를 동기화 없이 진행할 수 있다.
- 순수 함수가 아닌 함수보다 강력한 조합성을 갖는다.

### 3.3 일급 객체, 일급 함수, 고차 함수

일급 객체란 다음이 가능한 객체를 의미한다.

- 변수나 데이터 구조 안에 담을 수 있다.
- 파라미터로 전달할 수 있고, 인자(argument)로 전달받을 수 있다.
- 반환값으로 사용할 수 있다.
- 할당에 사용된 이름과 무관하게 고유한 구별이 가능하다.

이렇게 함수를 값처럼 다루어 변수에 담거나 다른 함수의 인자로 전달하거나 함수의 반환값으로 사용할 수 있는 함수를 **일급 함수(*First-class Function*)**라고 한다. 그리고 일급 함수의 특성을 활용해, 하나 이상의 함수를 인자로 받거나 함수를 반환하는 함수를 **고차 함수(*Higher-order Function*)**라고 한다. 일급 함수와 고차 함수는 함수형 프로그래밍 패러다임의 핵심적인 구성 요소로서 로직을 더욱 쉽게 모듈화하거나 조합할 수 있게 돕는다.

자바스크립트에서 함수는 일급 함수이면서 일급 객체이고, 일급 객체이기 때문에 다른 일반적인 객체에 적용 가능한 연산을 모두 지원한다. 그런 이유로 고차 함수를 만들 수 있고, 콜백을 사용할 수 있다.

### 3.4 참조 투명성 (Referential Transparency)

참조 투명성이란 다음과 같다.

- 동일한 인자에 대해 항상 동일한 결과를 반환해야 한다.
- 참조 투명성을 통해 기존의 값은 변경되지 않고 유지된다(*Immutable Data*).

참조 투명성을 지키기 위해 부수효과가 없는 순수 함수를 사용하는 것이 아니라, **부수효과가 없는 순수 함수를 사용하기 때문에 참조 투명성을 지킬 수 있다.**

참조에 투명하다는 것은 말 그대로 함수를 실행하여도 어떠한 상태의 변화 없이 항상 동일한 결과를 반환하여, 항상 동일하게(투명하게) 실행 결과를 참조(예측)할 수 있다는 것을 의미한다. 즉 어떤 함수 f에 인자 x를 넣고 실행하면 f는 입력된 인자에만 의존하므로 항상 f(x)라는 동일한 결과를 얻는다. 부작용을 제거하여 프로그램의 동작을 이해하고 예측하기 쉽게 만드는 것이 함수형 프로그래밍의 핵심 동기 중 하나다.

이런 성질은 병렬 처리 환경에서 개발할 때 경쟁 상태(*Race Condition - 여러 실행 흐름이 공유 자원에 동시에 접근해 결과가 실행 순서에 따라 달라지는 상태*)에 대한 비용을 줄여준다. 함수형 프로그래밍에서는 값의 대입 없이 항상 동일한 실행에 대해 동일한 결과를 반환하기 때문이다.

> **핵심 통찰**: 함수형 프로그래밍은 순수 함수를 통해 부수효과를 지양하고, 조합성을 강조하여 모듈화 수준을 높이는 프로그래밍 패러다임이다.

## 4. 평가, 일급, 고차 함수 — 코드로 보기

### 4.1 평가

평가(*Evaluation*)란 코드가 계산되어 값을 만드는 것이다.

```typescript
1; // 1
1 + 2; // 3
[1, 2]; // [1, 2]
[1, 2 + 3]; // [1, 5]
```

"함수를 값으로 다룬다", "평가를 미룬다" 같은 이후 장의 표현들은 모두 이 평가 개념 위에 서 있다.

### 4.2 일급

일급이란 값으로 다룰 수 있다는 뜻이다.

- 값으로 다룰 수 있다.
- 변수에 담을 수 있다.
- 함수의 인자로 사용될 수 있다.
- 함수의 결과로 사용될 수 있다.

일급 함수는 함수를 값으로 다룰 수 있다는 것이며, 조합성과 추상화의 도구가 된다.

### 4.3 고차 함수의 두 형태

고차 함수는 함수를 값으로 다루는 함수다. 크게 두 형태로 나뉜다.

**첫째, 함수를 인자로 받아서 실행하는 함수.** 함수를 값으로 받고 다른 값을 받아 안에서 함수에 인자를 적용하는 방식으로, 응용형 프로그래밍(*Applicative Programming - 함수를 인자로 받아 원하는 시점에 적용하는 프로그래밍 방식*)이라고도 한다.

```typescript
const apply10 = f => f(10);
const add5 = num => num + 5;

log(apply10(add5)) // 15

const times = (f, n) => {
  let i = -1;
  while( ++i < n) f(i)
}

times(log, 3) // 0 1 2
times((v) => log(v + 10), 3); // 10 11 12
```

**둘째, 함수를 만들어 리턴하는 함수(클로저를 만들어 리턴하는 함수).** 클로저는 함수가 만들어질 때의 환경(아래 코드의 `x`)과 함수 자체 객체를 통칭하는 용어다. 아래 코드에서 `addMake`는 클로저를 리턴한다.

```typescript
const addMake = x => y => x + y;
const add10 = addMake(10);

log(add10(5)) // 15
log(add10(10)) // 20
```

## 5. 순수 함수 판별 연습

어떤 함수가 순수한지 아닌지를 판별하는 감각은 이후 모든 장의 기반이 된다. 여섯 가지 예시로 연습해 보자.

**예시 1 — 순수 함수.**

```javascript
const add = (a, b) => {
	return a+b;
}
// 동일한 인자에 항상 동일한 결과가 나오고 부수효과가 없으므로 순수함수이다.
```

**예시 2 — 외부 변수를 읽는 함수.**

```javascript
let c = 10;
const add = (a, b) => {
  return a + b + c;
}
// c값이 변하면 a, b 인자가 같더라도 결과 값이 달라질 수 있다.
// 다만 c 값이 상수 값으로 고정되어 있다면 이 함수도 순수함수이다.
```

**예시 3 — 외부 상태를 변경하는 함수.**

```javascript
const add = (a, b) => {
  c = b;
  return a + b;
}
// 동일한 인자에 항상 동일한 결과가 나오지만,
// 함수 실행 전후로 외부 상태인 c 값이 달라지므로 순수함수가 아니다.
```

**예시 4 — 인자를 직접 변경하는 함수와 불변성.**

```javascript
let obj = { val: 10 };
const add = (obj, b) => {
  obj.val += b;
}

// 인자로 들어온 값의 상태를 직접 변경하고, 새로운 값을 리턴하지도 않는다.
// 함수형 프로그래밍에서는 원본 값을 두고 새로운 값을 만들어 원하는 부분에 새로운 값을 대치시킴으로 변경한다.
// 이를 불변성 유지라고 한다.

// 원본을 직접 변경
let arr1 = [1, 2, 3]
arr1.push(4);

// 새로운 값을 생성
let arr2 = [1, 2, 3]
arr2 = arr2.concat(4);
```

**예시 5 — 클로저와 순수 함수.**

```javascript
const addMaker = (a) => {
  return function(b) { // 클로저
    return a + b; 
  }
};
// 함수 내부에서만 a를 사용하며, a 값을 변경시키지 않는다. 참조만 할뿐 변경을 하지 않기 때문에 이 함수는 항상 동일한 값을 가리키고 있는 a와 함께 b를 더하는 순수 함수가 된다.

const add10 = addMaker(10);
console.log(add10(20)); // add10은 언제 평가해도 항상 동일한 값을 리턴시킨다
```

> 클로저는 **함수와 그 함수가 선언된 렉시컬 환경과의 조합**이다.<br>A closure is the combination of a function and the lexical environment within which that function was declared.

> **참고**: 자바스크립트 엔진은 함수를 어디서 호출했는지가 아니라 **함수를 어디에 정의했는지에 따라 상위 스코프를 결정**한다. 이를 렉시컬 스코프(정적 스코프)라 한다.

**예시 6 — 함수형 프로그래밍의 전형적인 형태.**

```javascript
const f4 = (f1, f2, f3) => {
  return f3(f1() + f2());
}

const result1 = f4(
  () => 2,
  () => 1,
  n => n*n
);
console.log(result1) // 9
```

- 함수형 프로그래밍 패러다임은 위와 같은 형태를 띠게 된다.
- 원하는 인자를 받아둔 함수에 함수를 인자로 받아서, 받은 함수의 로직을 원하는 시점에 평가하는 것이다. 즉 함수가 함수를 인자로 받아 그 함수의 로직대로 원하는 시점에 평가하고, 원하는 인자를 받아둔 함수에 적용하면서 로직을 완성해 나간다.
- 비동기나 동시성이 일어나는 병렬 처리를 동기화 없이 처리할 수 있다.

## 6. 함수형 사고방식 — 함수가 먼저다

> 함수형 사고방식은 문제의 해결 방법을 동사(함수)들로 구성(조합)하는 것<br>— 마이클 포거스(Michael Fogus)

함수형 프로그래밍은 애플리케이션, 함수의 구성요소, 더 나아가 언어 자체를 함수처럼 여기도록 만들고, 이러한 함수 개념을 가장 우선순위에 놓는다. 단순하게는 객체가 먼저 나오면 객체지향, 함수가 먼저 나오면 함수 지향 프로그래밍이다.

```javascript
person.moveLeft();
person.moveRight();
man.moveLeft();
man.moveRight();
// -> 객체지향에서는 데이터를 먼저 디자인하고 그 데이터에 맞는 메서드를 만든다

moveLeft(person);
moveRight(person);
moveLeft(man);
moveRight({ x: 5, y: 2 });
// -> 함수를 먼저 만들고, 그 함수에 맞게 데이터 세트를 구성한다
```

### 6.1 세 패러다임의 로직 구성 방식

절차지향 프로그래밍에서는 위에서 아래로 내려가면서 특정 변수의 값을 변경해 나가는 식으로 로직을 만든다. 객체지향 프로그래밍에서는 객체들을 만들어 놓고 객체들 간의 협업을 통해 로직을 만든다. 이벤트 등으로 서로를 연결한 후 상태의 변화를 감지하여 스스로 자신이 가진 값을 변경하거나, 상대의 메서드를 직접 실행하여 상태를 변경하는 식으로 프로그래밍한다.

함수형 프로그래밍에서는 항상 동일하게 동작하는 함수를 만들고 보조 함수를 조합하는 식으로 로직을 완성한다. 내부에서 관리하는 상태를 따로 두지 않고 넘겨진 인자에만 의존한다. 동일한 인자가 들어오면 항상 동일한 값을 리턴하도록 한다. 보조 함수 역시 인자이며, 보조 함수에서도 상태를 변경하지 않으면 보조 함수를 받은 함수는 항상 동일한 결과를 만드는 함수가 된다.

### 6.2 함수형과 객체지향은 대척점이 아니다

객체지향적으로 작성된 코드에서도 이전 객체와 같은 상태를 지닌 새 객체를 만드는 식으로 부수효과를 줄일 수 있다. 그러나 무수히 많고 각기 다른 종류로 나누어진 객체들을 복사하는 식으로 다루는 것은 운용도 어렵고 객체지향과 어울리지 않는다. 자신의 상태를 메서드를 통해 변경하는 것은 객체지향의 단점이 아니라 객체지향의 방법론 그 자체다. 반면 함수형 프로그래밍은 부수효과를 최소화하는 것이 목표에 가깝다. 이것은 단점이냐 장점이냐의 이야기가 아니라 **지향점의 차이**에 대한 것이다.

많은 사람들이 함수형 프로그래밍은 객체지향과 완전한 대척점에 있다고 생각하거나 그런 주장을 하기도 한다. 이것은 오해다. 결국에는 함께 동작해야 한다. 현대 프로그래밍에서 다루는 값은 대부분 객체이므로 함수형 프로그래밍에서도 결국 객체를 다뤄야 한다. 다만 기능 확장을 객체의 확장으로 풀어가느냐 함수 확장으로 풀어가느냐의 차이다. 객체를 확장하느냐 객체를 다루는 함수를 늘리느냐의 차이이며, **추상화의 단위가 클래스이냐 함수이냐의 차이**다.

이 관점은 2장(멀티패러다임으로 수렴하는 언어들)과 맞닿아 있고, Part 5의 멀티패러다임 설계(구조 문제는 OOP로, 로직 문제는 FP로)로 이어진다.

### 6.3 딥 다이브: 불변성에 관한 토론

원 노트에 남아 있던, `unshift`를 감싼 `prepend` 함수가 순수한가에 대한 토론이다. 부수효과·참조 투명성·불변성이 실제 논쟁에서 어떻게 쓰이는지 보여준다.

> 불변성의 경우에는 한 개 뎁스까지만 변경될 거라 불변성 유지를 위해서라기에는 애매한 부분이라고 생각합니다.<br><br>— 한 개 뎁스까지만 변경될 거라 "불변성 유지를 위해서라기에는 애매하다"는 것이 왜인지 여쭤봐도 될까요?<br><br>— 저는 `unshift` 같은 경우에는 여러 부수효과 케이스 중 '자료구조를 제자리에서 수정함'에 해당한다고 생각합니다. 그렇게 되면 `unshift`가 들어 있는 `prepend` 함수는 같은 인자일 경우 항상 같은 반환 값이 나오므로 순수한 함수라고 생각할 수 있으나, `unshift` 자체가 데이터 처리의 참조 투명성을 지키지 않는, 순수하지 않은 함수입니다. 순수한 함수는 결과에 대한 추론이 가능하고 테스트도 쉽고, 동시성 프로그래밍에서는 공유 자원이 변경될 걱정이 없어 자연스럽게 참조 투명성을 지킬 수 있다고 생각합니다. 하지만 모든 프로그램에서 부수효과가 없을 수는 없습니다. 보통 이런 순수하지 못한 작업을 모듈화하여 분리하는 방법으로 접근하는데, 그 모듈이 `prepend` 함수라고 하면 바로 납득했을 것 같습니다. 리액트에서 오류에 닫힌 코드를 위해 값과 함수를 매 렌더링마다 다시 만들게 하는 방법을 쓸 수도 있고, 성능이나 특수한 상황으로 `useMemo`나 `useCallback`을 쓸 수도 있습니다. 그래서 불변성을 지켜보며 같이 생각해 보면 좋겠다는 의미였습니다.

요지는 이렇다. (1) 순수함의 판별은 "반환값이 같은가"만이 아니라 **데이터를 제자리에서 수정하는가**까지 포함한다. (2) 모든 프로그램에서 부수효과를 없앨 수는 없으므로, 순수하지 못한 작업은 **모듈로 격리**하는 것이 함수형의 실전 전략이다.

## 7. 함수형 프로그래밍이 앞으로 각광받는 이유

- 좋아지는 하드웨어 성능
- 좋아지는 컴파일러
- 좋아지는 분산/리액티브 환경
- 동시성 + 병렬성 관련 기술
- 함수형 프로그래밍 기술

하드웨어와 실행 환경이 병렬·분산으로 갈수록, 상태 공유 없이 조합되는 순수 함수의 가치는 커진다.

## 요약

- 프로그래밍 패러다임은 **명령형**(절차지향·객체지향)과 **선언형**(함수형)으로 나뉘며, 함수형 프로그래밍은 "무엇을 할 것인지"를 순수 함수의 조합으로 표현한다.
- 현대 주요 언어들은 함수형과 객체지향을 모두 갖춘 **멀티패러다임 언어**로 수렴했고, 하나의 함수 안에서도 패러다임을 조합할 수 있다.
- **순수 함수**는 부수효과(변수 변경, 제자리 수정, I/O 등)가 없는 함수이며, 스레드 안전성·평가 시점 자유·강력한 조합성을 제공한다.
- **일급 함수**는 함수를 값으로 다룰 수 있다는 성질이고, 이를 바탕으로 함수를 인자로 받거나 반환하는 **고차 함수**를 만들 수 있다.
- **참조 투명성**은 동일 인자에 동일 결과를 보장하는 성질로, 순수 함수를 사용하기 때문에 얻어지는 결과다.
- 함수형과 객체지향은 대척점이 아니라 **추상화의 단위(함수 vs 클래스)가 다른 지향점의 차이**이며, 결국 함께 동작한다.

- **3장**: "함수를 만들어 리턴하는 함수"(4.3절)가 커링과 함수 합성으로 발전한다.
- **5장**: 일급 함수가 객체지향의 반복자 패턴과 만나 멀티패러다임의 교차점을 이루는 지점을 다룬다.
- **13장**: 참조 투명성이 깨지는 현실(외부 상태, 비동기)에서 합성을 안전하게 만드는 도구로 모나드가 등장한다.

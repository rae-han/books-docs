# Chapter 9: S3 - 객체 스토리지: Simple Storage Service

## 핵심 질문

파일을 저장하고 서빙하는 가장 기본적인 방법인 S3는 어떻게 동작하는가? 정적 웹사이트 호스팅, 프론트엔드에서의 직접 업로드, 스토리지 클래스별 비용 최적화, 버전 관리와 수명 주기 정책을 어떻게 설계해야 하는가?

## 1. S3란

### 1.1 객체 스토리지의 개념

S3(*Simple Storage Service - AWS의 대표적인 객체 스토리지 서비스. 사실상 무제한의 데이터를 저장할 수 있으며, 내구성 99.999999999%(일레븐 나인)를 보장한다*)는 AWS에서 가장 오래되고 가장 널리 사용되는 서비스 중 하나이다. 이미지, 동영상, 로그 파일, 백업, 정적 웹사이트 에셋 등 거의 모든 종류의 파일을 저장할 수 있다.

S3를 이해하려면 먼저 스토리지의 세 가지 유형을 구분해야 한다.

```
스토리지 유형 비교:

블록 스토리지 (EBS)          파일 스토리지 (EFS)          객체 스토리지 (S3)
┌─────────────────┐       ┌─────────────────┐       ┌─────────────────┐
│  ┌───┬───┬───┐  │       │  /home/          │       │  버킷            │
│  │ B │ B │ B │  │       │  ├── user/       │       │  ├── img/a.png   │
│  │ L │ L │ L │  │       │  │   └── data/   │       │  ├── logs/b.log  │
│  │ K │ K │ K │  │       │  └── shared/     │       │  └── backup.zip  │
│  └───┴───┴───┘  │       │      └── docs/   │       │                  │
│  고정 크기 블록    │       │  계층적 디렉터리    │       │  플랫 구조 + 키    │
└─────────────────┘       └─────────────────┘       └─────────────────┘

용도: 하드디스크/SSD         용도: 공유 파일 시스템         용도: 대량 비정형 데이터
예: EC2 루트 볼륨             예: 여러 서버가 공유하는 NAS    예: 이미지, 로그, 백업
수정: 블록 단위 수정 가능      수정: 파일 단위 수정 가능       수정: 전체 교체만 가능
```

| 특성 | 블록 스토리지 (EBS) | 파일 스토리지 (EFS) | 객체 스토리지 (S3) |
|------|-------------------|-------------------|-------------------|
| **구조** | 고정 크기 블록 | 계층적 디렉터리/파일 | 플랫 키-값 쌍 |
| **수정** | 블록 단위 부분 수정 | 파일 단위 수정 | 전체 객체 교체만 가능 |
| **접근 방식** | OS에 마운트 | NFS/SMB 마운트 | HTTP API |
| **확장성** | 볼륨 크기 수동 조정 | 자동 확장 | 사실상 무제한 |
| **지연 시간** | 매우 낮음 (ms 이하) | 낮음 (ms) | 보통 (수십 ms) |
| **비용** | 높음 | 중간 | 낮음 |
| **주요 AWS 서비스** | EBS | EFS | S3 |

### 1.2 S3의 핵심 개념

```
S3 구조:

┌──────────────── S3 (글로벌 네임스페이스) ────────────────┐
│                                                         │
│  ┌──── 버킷 (my-app-assets) ────────────────────────┐   │
│  │  리전: ap-northeast-2 (서울)                      │   │
│  │                                                   │   │
│  │  객체 1: images/logo.png                          │   │
│  │    ├── 키(Key): images/logo.png                   │   │
│  │    ├── 값(Value): 바이너리 데이터 (최대 5TB)        │   │
│  │    ├── 메타데이터: Content-Type, Cache-Control 등   │   │
│  │    └── 버전 ID: (버전 관리 활성화 시)               │   │
│  │                                                   │   │
│  │  객체 2: docs/readme.pdf                          │   │
│  │  객체 3: backups/2024-01-15.tar.gz                │   │
│  └───────────────────────────────────────────────────┘   │
│                                                         │
│  ┌──── 버킷 (my-app-logs) ──────────────────────────┐   │
│  │  리전: ap-northeast-2 (서울)                      │   │
│  │  ...                                              │   │
│  └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**핵심 용어:**

| 용어 | 설명 |
|------|------|
| **버킷(Bucket)** | 객체를 담는 최상위 컨테이너. 이름이 **전 세계적으로 고유**해야 한다 |
| **객체(Object)** | S3에 저장되는 하나의 파일. 키 + 값 + 메타데이터로 구성 |
| **키(Key)** | 버킷 내에서 객체를 식별하는 고유 경로. `images/logo.png` 같은 형태 |
| **값(Value)** | 실제 파일 데이터. 단일 객체 최대 5TB |
| **메타데이터** | 객체에 대한 부가 정보. `Content-Type`, `Cache-Control` 등 |
| **리전(Region)** | 버킷이 물리적으로 저장되는 AWS 리전 |

> **핵심 통찰**: S3에는 실제로 "폴더"가 존재하지 않는다. `images/logo.png`는 폴더 안의 파일이 아니라 **`images/logo.png`라는 키를 가진 하나의 객체**이다. AWS 콘솔에서 폴더처럼 보이는 것은 키의 `/` 구분자를 기준으로 시각적으로 그룹화한 것일 뿐이다. 이 사실을 이해하면 S3의 퍼포먼스 특성과 접근 패턴을 더 잘 이해할 수 있다.

## 2. 버킷 생성과 설정

### 2.1 버킷 이름 규칙

S3 버킷 이름은 **전 세계 모든 AWS 계정에서 고유**해야 한다. 다른 계정의 다른 리전에 이미 같은 이름의 버킷이 있으면 생성할 수 없다.

**이름 규칙:**

- 3~63자 사이
- 소문자, 숫자, 하이픈(`-`)만 사용 가능
- 문자 또는 숫자로 시작
- IP 주소 형태(`192.168.1.1`) 불가
- `xn--`으로 시작 불가, `-s3alias`로 끝나면 안 됨

```
좋은 버킷 이름 예시:
  my-app-assets-prod          ← 앱 이름 + 용도 + 환경
  company-frontend-2024       ← 회사 + 용도 + 연도
  my-org-backup-ap-northeast  ← 조직 + 용도 + 리전 힌트

나쁜 버킷 이름 예시:
  MyBucket                    ← 대문자 불가
  my bucket                   ← 공백 불가
  assets                      ← 너무 일반적, 이미 사용 중일 가능성 높음
```

### 2.2 리전 선택

버킷은 특정 리전에 생성되며, 객체는 명시적으로 복제하지 않는 한 해당 리전을 벗어나지 않는다. 리전 선택 기준:

- **사용자와의 물리적 거리**: 한국 사용자 대상이면 `ap-northeast-2`(서울)
- **법적 규정**: 데이터가 특정 국가를 벗어나면 안 되는 경우
- **다른 AWS 서비스와의 근접성**: 같은 리전의 EC2에서 S3에 접근하면 지연 시간이 낮고 데이터 전송 비용이 없다

### 2.3 버킷 생성

```bash
# 기본 버킷 생성 (서울 리전)
aws s3 mb s3://my-app-assets-prod --region ap-northeast-2

# 또는 s3api로 상세 옵션과 함께 생성
aws s3api create-bucket \
  --bucket my-app-assets-prod \
  --region ap-northeast-2 \
  --create-bucket-configuration LocationConstraint=ap-northeast-2

# 버킷 목록 확인
aws s3 ls
```

> **실무 팁**: 버킷 이름에 환경(`dev`, `staging`, `prod`)과 용도(`assets`, `logs`, `backup`)를 포함하는 네이밍 컨벤션을 팀에서 정하라. 예: `{조직}-{서비스}-{용도}-{환경}` → `mycompany-api-uploads-prod`. AWS 계정을 환경별로 분리하는 경우에는 환경 접미사를 생략할 수도 있다.

## 3. 객체 업로드와 다운로드

### 3.1 AWS CLI로 파일 관리

```bash
# 단일 파일 업로드
aws s3 cp ./build/index.html s3://my-app-assets-prod/index.html

# Content-Type 명시 (HTML 파일을 텍스트로 서빙하지 않도록)
aws s3 cp ./build/index.html s3://my-app-assets-prod/index.html \
  --content-type "text/html"

# 디렉터리 전체 업로드 (재귀)
aws s3 cp ./build/ s3://my-app-assets-prod/ --recursive

# 동기화 (변경된 파일만 업로드, 삭제된 파일은 S3에서도 삭제)
aws s3 sync ./build/ s3://my-app-assets-prod/ --delete

# 파일 다운로드
aws s3 cp s3://my-app-assets-prod/data.json ./local-data.json

# S3에서 S3로 복사 (버킷 간 이동)
aws s3 cp s3://source-bucket/file.zip s3://dest-bucket/file.zip

# 객체 목록 조회
aws s3 ls s3://my-app-assets-prod/
aws s3 ls s3://my-app-assets-prod/images/ --recursive

# 객체 삭제
aws s3 rm s3://my-app-assets-prod/old-file.txt

# 디렉터리(접두사) 전체 삭제
aws s3 rm s3://my-app-assets-prod/temp/ --recursive
```

### 3.2 AWS SDK (TypeScript)로 파일 관리

```typescript
import {
  S3Client,
  PutObjectCommand,
  GetObjectCommand,
  DeleteObjectCommand,
  ListObjectsV2Command,
} from "@aws-sdk/client-s3";
import { readFile } from "fs/promises";

const s3 = new S3Client({ region: "ap-northeast-2" });
const BUCKET = "my-app-assets-prod";

// 파일 업로드
async function uploadFile(key: string, filePath: string) {
  const body = await readFile(filePath);
  await s3.send(
    new PutObjectCommand({
      Bucket: BUCKET,
      Key: key,
      Body: body,
      ContentType: "image/png", // MIME 타입 지정
    })
  );
  console.log(`Uploaded: s3://${BUCKET}/${key}`);
}

// 파일 다운로드
async function downloadFile(key: string): Promise<string> {
  const response = await s3.send(
    new GetObjectCommand({
      Bucket: BUCKET,
      Key: key,
    })
  );
  return await response.Body!.transformToString();
}

// 객체 목록 조회 (페이지네이션 포함)
async function listObjects(prefix: string) {
  const response = await s3.send(
    new ListObjectsV2Command({
      Bucket: BUCKET,
      Prefix: prefix,
      MaxKeys: 100,
    })
  );
  return response.Contents ?? [];
}

// 객체 삭제
async function deleteFile(key: string) {
  await s3.send(
    new DeleteObjectCommand({
      Bucket: BUCKET,
      Key: key,
    })
  );
}
```

### 3.3 메타데이터와 태그

객체를 업로드할 때 메타데이터와 태그를 지정할 수 있다. 메타데이터는 HTTP 헤더로 반환되며, 태그는 비용 할당이나 수명 주기 정책에 활용된다.

```bash
# 메타데이터와 태그를 포함하여 업로드
aws s3 cp ./image.png s3://my-app-assets-prod/images/hero.png \
  --content-type "image/png" \
  --cache-control "public, max-age=31536000, immutable" \
  --metadata '{"uploaded-by":"ci-pipeline","version":"1.2.3"}' \
  --tagging "environment=prod&team=frontend"
```

> **실무 팁**: 정적 에셋(이미지, CSS, JS)을 S3에 업로드할 때는 반드시 `Cache-Control` 헤더를 설정하라. 파일명에 해시를 포함하는 빌드 도구(Webpack, Vite)를 사용한다면 `max-age=31536000, immutable`로 설정하여 브라우저 캐시를 최대한 활용할 수 있다. `index.html`처럼 파일명이 고정된 엔트리 포인트에는 `no-cache`를 설정하여 항상 최신 버전을 가져오게 한다.

## 4. 정적 웹사이트 호스팅

### 4.1 정적 웹사이트 호스팅이란

S3는 버킷 자체를 **HTTP 웹 서버로 동작**시킬 수 있다. React, Next.js(정적 내보내기), Vue 등의 SPA(*Single Page Application - 서버 없이 브라우저에서 실행되는 프론트엔드 애플리케이션*)를 별도의 서버 없이 S3만으로 서빙할 수 있다.

```
정적 웹사이트 호스팅 아키텍처:

사용자 브라우저
    │
    ↓
S3 버킷 (정적 웹사이트 호스팅 활성화)
    │
    ├── index.html       ← 인덱스 문서
    ├── error.html       ← 에러 문서 (404)
    ├── static/
    │   ├── css/main.abc123.css
    │   └── js/bundle.def456.js
    └── images/
        └── logo.png

엔드포인트:
http://my-app-assets-prod.s3-website.ap-northeast-2.amazonaws.com
```

### 4.2 React/Next.js 정적 빌드 배포

**Next.js 정적 내보내기 (Static Export):**

```typescript
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "export", // 정적 HTML로 내보내기
  // trailingSlash: true, // /about → /about/index.html (선택)
};

export default nextConfig;
```

```bash
# 빌드
npm run build
# → out/ 디렉터리에 정적 파일 생성

# S3에 배포
aws s3 sync out/ s3://my-app-assets-prod/ --delete

# index.html은 캐시하지 않음
aws s3 cp out/index.html s3://my-app-assets-prod/index.html \
  --cache-control "no-cache" \
  --content-type "text/html"

# 정적 에셋은 장기 캐시
aws s3 sync out/_next/ s3://my-app-assets-prod/_next/ \
  --cache-control "public, max-age=31536000, immutable"
```

### 4.3 정적 웹사이트 호스팅 설정

```bash
# 1. 정적 웹사이트 호스팅 활성화
aws s3 website s3://my-app-assets-prod/ \
  --index-document index.html \
  --error-document error.html

# 2. Block Public Access 비활성화 (정적 웹사이트에 필요)
aws s3api put-public-access-block \
  --bucket my-app-assets-prod \
  --public-access-block-configuration \
    BlockPublicAcls=false,IgnorePublicAcls=false,BlockPublicPolicy=false,RestrictPublicBuckets=false

# 3. 버킷 정책으로 퍼블릭 읽기 허용
aws s3api put-bucket-policy \
  --bucket my-app-assets-prod \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Sid": "PublicReadGetObject",
        "Effect": "Allow",
        "Principal": "*",
        "Action": "s3:GetObject",
        "Resource": "arn:aws:s3:::my-app-assets-prod/*"
      }
    ]
  }'
```

> **핵심 통찰**: S3 정적 웹사이트 호스팅을 단독으로 사용하면 **HTTP만 지원**된다(HTTPS 불가). 프로덕션에서는 S3 앞에 **CloudFront(CDN)**를 두어 HTTPS를 제공하고, 전 세계 엣지 로케이션에서 캐시하여 성능을 높이는 구성이 사실상 표준이다. 이 구성은 Chapter 10에서 상세히 다룬다.

### 4.4 SPA 라우팅 처리

React Router나 Next.js App Router를 사용하는 SPA는 `/about`, `/dashboard` 같은 클라이언트 사이드 경로를 사용한다. 이 경로로 직접 접근하면 S3는 해당 키의 객체를 찾지 못해 404를 반환한다.

**해결 방법: 에러 문서를 index.html로 설정**

```bash
# 에러 문서를 index.html로 지정
aws s3 website s3://my-app-assets-prod/ \
  --index-document index.html \
  --error-document index.html
```

이렇게 하면 S3가 객체를 찾지 못했을 때 `index.html`을 반환하고, React Router가 클라이언트에서 올바른 컴포넌트를 렌더링한다. 다만 이 경우 HTTP 상태 코드가 403 또는 404로 반환되므로, CloudFront의 커스텀 에러 응답을 사용하여 200으로 변환하는 것이 바람직하다.

## 5. 버킷 정책과 접근 제어

### 5.1 접근 제어 계층

S3의 접근 제어는 여러 계층으로 구성된다. 모든 계층에서 허용되어야 최종적으로 접근이 가능하다.

```
S3 접근 제어 계층:

┌─── Block Public Access (계정/버킷 수준) ───┐
│  모든 퍼블릭 접근을 차단하는 최상위 스위치     │
│  (기본값: 모두 차단)                         │
│                                             │
│  ┌─── 버킷 정책 (Bucket Policy) ──────┐     │
│  │  "누가 이 버킷에 무엇을 할 수 있는가"  │     │
│  │  JSON 형태의 리소스 기반 정책         │     │
│  │                                     │     │
│  │  ┌─── IAM 정책 ──────────────┐     │     │
│  │  │  IAM 사용자/역할에 부여된     │     │     │
│  │  │  자격 증명 기반 정책         │     │     │
│  │  └───────────────────────────┘     │     │
│  └─────────────────────────────────────┘     │
└─────────────────────────────────────────────┘
```

### 5.2 Block Public Access

2018년 S3 버킷 설정 실수로 인한 대규모 데이터 유출 사고 이후 AWS가 도입한 안전장치이다. **기본적으로 모든 퍼블릭 접근이 차단**되며, 의도적으로 퍼블릭 접근이 필요한 경우에만 비활성화한다.

```bash
# Block Public Access 상태 확인
aws s3api get-public-access-block --bucket my-app-assets-prod

# 모든 퍼블릭 접근 차단 (기본값, 권장)
aws s3api put-public-access-block \
  --bucket my-app-assets-prod \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

| 설정 | 설명 |
|------|------|
| **BlockPublicAcls** | 퍼블릭 ACL 추가를 차단 |
| **IgnorePublicAcls** | 기존 퍼블릭 ACL을 무시 |
| **BlockPublicPolicy** | 퍼블릭 버킷 정책 추가를 차단 |
| **RestrictPublicBuckets** | 퍼블릭 정책이 있는 버킷의 퍼블릭/크로스 계정 접근을 제한 |

### 5.3 버킷 정책 (Bucket Policy)

버킷 정책(*Bucket Policy - 버킷에 직접 부여하는 리소스 기반 정책. JSON 형태로, 버킷과 객체에 대한 접근 권한을 세밀하게 제어한다*)은 IAM 정책(Chapter 3)과 동일한 JSON 구조를 사용하지만, **버킷에 부착**된다는 점이 다르다.

**예시 1: 특정 IP에서만 접근 허용**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowFromSpecificIP",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-app-assets-prod/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": ["203.0.113.0/24"]
        }
      }
    }
  ]
}
```

**예시 2: CloudFront OAC에서만 접근 허용 (프로덕션 권장)**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontOAC",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-app-assets-prod/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E1234567890"
        }
      }
    }
  ]
}
```

**예시 3: HTTPS 접근만 허용**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyNonSSL",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-app-assets-prod",
        "arn:aws:s3:::my-app-assets-prod/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

### 5.4 ACL vs 버킷 정책

ACL(*Access Control List - S3의 레거시 접근 제어 메커니즘. 버킷 정책보다 기능이 제한적이며, AWS는 버킷 정책 사용을 권장한다*)은 S3 초기부터 존재하던 접근 제어 방식이다.

| 특성 | 버킷 정책 | ACL |
|------|----------|-----|
| **형식** | JSON (IAM 정책과 동일) | XML |
| **세밀함** | 조건, IP, 경로 등 세밀한 제어 | 단순 읽기/쓰기 수준 |
| **AWS 권장** | 권장 | 비권장 (레거시) |
| **기본 설정** | - | 2023년 4월부터 새 버킷은 ACL 비활성화가 기본 |

```bash
# 버킷 소유권 설정 (ACL 비활성화 — 권장)
aws s3api put-bucket-ownership-controls \
  --bucket my-app-assets-prod \
  --ownership-controls '{
    "Rules": [{"ObjectOwnership": "BucketOwnerEnforced"}]
  }'
```

## 6. 버전 관리 (Versioning)

### 6.1 버전 관리란

버전 관리(*Versioning - S3 객체의 모든 변경 이력을 보존하는 기능. 동일한 키에 새 파일을 업로드하면 이전 버전이 삭제되지 않고 보관된다*)를 활성화하면, 같은 키에 파일을 덮어쓰거나 삭제해도 이전 버전이 모두 보존된다.

```
버전 관리 동작:

키: images/logo.png

V1 (10:00 업로드) ─── 원본 로고
V2 (14:00 업로드) ─── 수정된 로고 (V1은 보존)
V3 (16:00 업로드) ─── 최종 로고 (V1, V2 모두 보존)

삭제 시:
V3 위에 "삭제 마커" 추가 ─── 외부에서는 삭제된 것처럼 보임
                             V1, V2, V3 모두 여전히 보존
```

### 6.2 버전 관리 설정

```bash
# 버전 관리 활성화
aws s3api put-bucket-versioning \
  --bucket my-app-assets-prod \
  --versioning-configuration Status=Enabled

# 버전 관리 상태 확인
aws s3api get-bucket-versioning --bucket my-app-assets-prod

# 모든 버전 조회
aws s3api list-object-versions \
  --bucket my-app-assets-prod \
  --prefix images/logo.png

# 특정 버전 다운로드
aws s3api get-object \
  --bucket my-app-assets-prod \
  --key images/logo.png \
  --version-id "abc123xyz" \
  restored-logo.png

# 특정 버전 영구 삭제
aws s3api delete-object \
  --bucket my-app-assets-prod \
  --key images/logo.png \
  --version-id "abc123xyz"
```

### 6.3 삭제 마커 (Delete Marker)

버전 관리가 활성화된 버킷에서 객체를 삭제하면, 실제로 데이터가 삭제되지 않는다. 대신 **삭제 마커(*Delete Marker - 버전 관리된 객체가 삭제되었음을 나타내는 특수 마커. 삭제 마커를 제거하면 객체가 복원된다*)** 가 추가된다.

```bash
# 삭제 (실제로는 삭제 마커 추가)
aws s3 rm s3://my-app-assets-prod/images/logo.png

# 삭제 마커 확인
aws s3api list-object-versions \
  --bucket my-app-assets-prod \
  --prefix images/logo.png \
  --query 'DeleteMarkers'

# 복원: 삭제 마커를 삭제하면 이전 버전이 복원됨
aws s3api delete-object \
  --bucket my-app-assets-prod \
  --key images/logo.png \
  --version-id "DELETE_MARKER_VERSION_ID"
```

> **비용 주의**: 버전 관리를 활성화하면 **모든 이전 버전이 스토리지 비용을 차지**한다. 10MB 파일을 매일 업데이트하면 한 달에 300MB의 이전 버전이 쌓인다. 수명 주기 정책(섹션 7)으로 오래된 버전을 자동 삭제하거나 저렴한 스토리지 클래스로 이동시켜야 한다.

## 7. 수명 주기 정책 (Lifecycle Policy)

### 7.1 수명 주기 정책이란

수명 주기 정책(*Lifecycle Policy - 객체의 나이에 따라 스토리지 클래스를 자동으로 전환하거나 삭제하는 규칙*)은 시간이 지남에 따라 객체를 자동으로 관리한다. 비용 최적화의 핵심 도구이다.

```
수명 주기 정책 예시:

Day 0          Day 30          Day 90          Day 365
  │              │               │                │
  ↓              ↓               ↓                ↓
Standard ──→ Standard-IA ──→ Glacier ──→ 자동 삭제
(빈번 접근)    (가끔 접근)      (아카이브)     (보존 기한 만료)
```

### 7.2 정책 설정

```bash
# 수명 주기 정책 설정
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-app-assets-prod \
  --lifecycle-configuration '{
    "Rules": [
      {
        "ID": "MoveLogsToIA",
        "Filter": { "Prefix": "logs/" },
        "Status": "Enabled",
        "Transitions": [
          {
            "Days": 30,
            "StorageClass": "STANDARD_IA"
          },
          {
            "Days": 90,
            "StorageClass": "GLACIER"
          }
        ],
        "Expiration": {
          "Days": 365
        }
      },
      {
        "ID": "CleanupOldVersions",
        "Filter": { "Prefix": "" },
        "Status": "Enabled",
        "NoncurrentVersionTransitions": [
          {
            "NoncurrentDays": 30,
            "StorageClass": "STANDARD_IA"
          }
        ],
        "NoncurrentVersionExpiration": {
          "NoncurrentDays": 90
        }
      },
      {
        "ID": "AbortIncompleteMultipartUpload",
        "Filter": { "Prefix": "" },
        "Status": "Enabled",
        "AbortIncompleteMultipartUpload": {
          "DaysAfterInitiation": 7
        }
      }
    ]
  }'
```

**각 규칙의 의미:**

| 규칙 | 동작 |
|------|------|
| **MoveLogsToIA** | `logs/` 접두사 객체를 30일 후 Standard-IA, 90일 후 Glacier로 이동, 365일 후 삭제 |
| **CleanupOldVersions** | 비현재 버전을 30일 후 Standard-IA로 이동, 90일 후 삭제 |
| **AbortIncompleteMultipartUpload** | 7일 이상 완료되지 않은 멀티파트 업로드를 자동 정리 |

> **실무 팁**: **AbortIncompleteMultipartUpload** 규칙은 모든 버킷에 기본으로 설정하라. 실패한 멀티파트 업로드의 파트 조각이 영원히 남아 스토리지 비용이 발생하는 것을 방지한다. 이 비용은 눈에 잘 띄지 않아 놓치기 쉽다.

## 8. 스토리지 클래스

### 8.1 스토리지 클래스 비교

S3는 데이터의 접근 빈도와 내구성 요구사항에 따라 여러 스토리지 클래스를 제공한다. 접근 빈도가 낮을수록 저장 비용이 싸지만, 검색 비용이 추가되거나 접근 지연 시간이 늘어난다.

| 스토리지 클래스 | 저장 비용(/GB/월) | 최소 보관 기간 | 검색 비용 | 가용성 | 사용 사례 |
|---------------|-----------------|-------------|----------|--------|----------|
| **Standard** | ~$0.025 | 없음 | 없음 | 99.99% | 빈번히 접근하는 데이터 |
| **Intelligent-Tiering** | ~$0.025 + 모니터링 비용 | 없음 | 없음 | 99.9% | 접근 패턴을 예측할 수 없는 데이터 |
| **Standard-IA** | ~$0.0138 | 30일 | GB당 과금 | 99.9% | 가끔 접근하는 데이터 (백업, 로그) |
| **One Zone-IA** | ~$0.011 | 30일 | GB당 과금 | 99.5% | 재생성 가능한 데이터, AZ 손실 허용 |
| **Glacier Instant** | ~$0.005 | 90일 | GB당 과금 | 99.9% | 분기 1회 접근, 즉시 검색 필요 |
| **Glacier Flexible** | ~$0.0045 | 90일 | 분~시간 대기 | 99.99% | 연 1~2회 접근, 수 분~수 시간 대기 허용 |
| **Glacier Deep Archive** | ~$0.002 | 180일 | 12~48시간 대기 | 99.99% | 규정 준수용 장기 보관 (7~10년) |

```
스토리지 클래스별 비용 vs 접근성:

비용(저장)  높음 ┤ Standard
               │      Intelligent-Tiering
               │           Standard-IA
               │                One Zone-IA
               │                     Glacier Instant
               │                          Glacier Flexible
          낮음 ┤                               Deep Archive
               └──────────────────────────────────────────→
              즉시 접근                              수 시간 대기
                           접근 지연 시간
```

### 8.2 Intelligent-Tiering

접근 패턴을 예측하기 어렵다면 **Intelligent-Tiering**이 좋은 선택이다. AWS가 자동으로 객체를 모니터링하여 최적의 티어로 이동시킨다.

```
Intelligent-Tiering 자동 계층화:

빈번한 접근 (Frequent)     ← 30일 내 접근된 객체
       │
       │ 30일 미접근
       ↓
가끔 접근 (Infrequent)     ← 30~90일 미접근 객체
       │
       │ 90일 미접근
       ↓
아카이브 접근 (Archive)     ← 90일+ 미접근 (선택적 활성화)
       │
       │ 180일 미접근
       ↓
딥 아카이브 (Deep Archive)  ← 180일+ 미접근 (선택적 활성화)
```

```bash
# Intelligent-Tiering으로 업로드
aws s3 cp ./data.json s3://my-app-assets-prod/data.json \
  --storage-class INTELLIGENT_TIERING

# 기존 객체의 스토리지 클래스 변경
aws s3 cp s3://my-app-assets-prod/old-data.json \
  s3://my-app-assets-prod/old-data.json \
  --storage-class STANDARD_IA
```

### 8.3 스토리지 클래스 선택 가이드

```
"이 데이터를 얼마나 자주 접근하는가?"

  매일/매시간 접근
    └──→ Standard

  접근 빈도를 모르겠다
    └──→ Intelligent-Tiering

  월 1~2회 접근
    └──→ Standard-IA
          └── 재생성 가능? → One Zone-IA (더 저렴)

  분기~연 1회 접근
    └──→ Glacier Instant (즉시 필요)
         Glacier Flexible (몇 시간 대기 가능)

  거의 접근 안 함, 규정 때문에 보관
    └──→ Glacier Deep Archive
```

## 9. S3 이벤트 알림

### 9.1 이벤트 알림이란

S3 버킷에서 객체가 생성, 삭제, 복원될 때 **자동으로 다른 AWS 서비스를 트리거**할 수 있다. 대표적인 대상은 Lambda, SQS, SNS이다.

```
S3 이벤트 알림 아키텍처:

사용자가 이미지 업로드
    │
    ↓
S3 버킷 (uploads/)
    │
    ├── s3:ObjectCreated:* 이벤트
    │
    ├──→ Lambda 함수: 이미지 리사이징
    │     └──→ S3 (thumbnails/) 에 썸네일 저장
    │
    ├──→ SQS 큐: 처리 대기열에 추가
    │     └──→ 워커가 메타데이터 추출
    │
    └──→ SNS 토픽: 관리자에게 알림
```

### 9.2 Lambda 트리거 설정

```bash
# S3 이벤트 알림 설정 (Lambda 트리거)
aws s3api put-bucket-notification-configuration \
  --bucket my-app-uploads \
  --notification-configuration '{
    "LambdaFunctionConfigurations": [
      {
        "Id": "ImageUploadTrigger",
        "LambdaFunctionArn": "arn:aws:lambda:ap-northeast-2:123456789012:function:resize-image",
        "Events": ["s3:ObjectCreated:*"],
        "Filter": {
          "Key": {
            "FilterRules": [
              { "Name": "prefix", "Value": "uploads/" },
              { "Name": "suffix", "Value": ".jpg" }
            ]
          }
        }
      }
    ]
  }'
```

**Lambda 함수 예시 (이미지 업로드 처리):**

```typescript

const s3 = new S3Client({});

export const handler = async (event: S3Event) => {
  for (const record of event.Records) {
    const bucket = record.s3.bucket.name;
    const key = decodeURIComponent(record.s3.object.key.replace(/\+/g, " "));

    console.log(`New object: s3://${bucket}/${key}`);

    // 원본 객체 가져오기
    const original = await s3.send(
      new GetObjectCommand({ Bucket: bucket, Key: key })
    );

    // 여기서 이미지 리사이징 로직 수행...
    // (sharp 라이브러리 등 사용)

    // 썸네일 저장
    const thumbnailKey = key.replace("uploads/", "thumbnails/");
    await s3.send(
      new PutObjectCommand({
        Bucket: bucket,
        Key: thumbnailKey,
        Body: processedImage,
        ContentType: "image/jpeg",
      })
    );
  }
};
```

## 10. 멀티파트 업로드

### 10.1 멀티파트 업로드란

멀티파트 업로드(*Multipart Upload - 대용량 파일을 여러 조각으로 분할하여 병렬로 업로드하는 기법. 실패한 파트만 재전송할 수 있어 네트워크 효율이 높다*)는 대용량 파일을 작은 파트로 나누어 병렬 업로드하는 방식이다.

```
일반 업로드 (PUT):          멀티파트 업로드:

┌─────────────────┐         ┌──── Part 1 (5MB) ────→ ┐
│  1GB 파일       │         ├──── Part 2 (5MB) ────→ │
│  한 번에 전송    │         ├──── Part 3 (5MB) ────→ ├→ S3가 조합
│  실패 시 처음부터 │         ├──── ...            ──→ │
└─────────────────┘         └──── Part N (5MB) ────→ ┘
                              병렬 전송, 실패한 파트만 재전송
```

**권장 기준:**

- **100MB 이상**: 멀티파트 업로드 권장
- **5GB 이상**: 멀티파트 업로드 필수 (단일 PUT 최대 5GB)

### 10.2 AWS CLI의 자동 멀티파트

AWS CLI의 `s3 cp`는 파일 크기에 따라 자동으로 멀티파트 업로드를 사용한다.

```bash
# AWS CLI는 기본적으로 8MB 이상 파일을 멀티파트로 업로드
aws s3 cp large-file.zip s3://my-app-assets-prod/backups/

# 멀티파트 임계값과 파트 크기 조정
aws configure set default.s3.multipart_threshold 64MB
aws configure set default.s3.multipart_chunksize 16MB
```

### 10.3 SDK로 멀티파트 업로드

```typescript
import { S3Client } from "@aws-sdk/client-s3";
import { Upload } from "@aws-sdk/lib-storage";
import { createReadStream } from "fs";

const s3 = new S3Client({ region: "ap-northeast-2" });

async function uploadLargeFile(filePath: string, key: string) {
  const upload = new Upload({
    client: s3,
    params: {
      Bucket: "my-app-assets-prod",
      Key: key,
      Body: createReadStream(filePath),
    },
    // 파트 크기: 10MB, 동시 업로드: 4개
    partSize: 10 * 1024 * 1024,
    queueSize: 4,
  });

  // 진행 상황 모니터링
  upload.on("httpUploadProgress", (progress) => {
    const percent = Math.round(
      ((progress.loaded ?? 0) / (progress.total ?? 1)) * 100
    );
    console.log(`Upload progress: ${percent}%`);
  });

  await upload.done();
  console.log("Upload complete");
}
```

## 11. Presigned URL (서명된 URL)

### 11.1 Presigned URL이란

Presigned URL(*Presigned URL - 임시 접근 권한이 포함된 URL. AWS 자격 증명이 없는 클라이언트도 이 URL을 통해 S3에 직접 접근할 수 있다*)은 프론트엔드에서 S3에 직접 파일을 업로드하거나 비공개 파일을 다운로드할 때 사용하는 핵심 패턴이다.

```
Presigned URL 업로드 패턴:

1. 클라이언트 ──→ 백엔드 API: "이미지를 업로드하고 싶습니다"
2. 백엔드 API ──→ S3: presigned URL 생성 요청
3. 백엔드 API ──→ 클라이언트: presigned URL 반환
4. 클라이언트 ──→ S3: presigned URL로 직접 업로드 (백엔드를 거치지 않음)

┌──────────┐  ① POST /upload-url  ┌──────────┐
│          │ ──────────────────→  │          │
│ 클라이언트 │                     │ 백엔드    │
│ (React)  │  ③ presigned URL    │ (Node.js) │
│          │ ←──────────────────  │          │
└──────────┘                     └──────────┘
     │                                │
     │ ④ PUT (presigned URL로       ② getSignedUrl()
     │    파일 직접 업로드)              │
     ↓                                ↓
┌────────────────────────────────────────┐
│                S3 버킷                  │
└────────────────────────────────────────┘
```

### 11.2 이 패턴의 장점

| 장점 | 설명 |
|------|------|
| **서버 부하 감소** | 대용량 파일이 백엔드 서버를 거치지 않으므로 대역폭과 CPU를 절약 |
| **확장성** | 파일 업로드 트래픽이 S3로 직접 가므로 백엔드 스케일링 부담 감소 |
| **보안** | URL에 만료 시간이 있고, 특정 키/Content-Type만 허용 가능 |
| **비용** | 데이터가 백엔드를 거치지 않아 네트워크 비용 절감 |

### 11.3 백엔드에서 Presigned URL 생성

```typescript
import { S3Client, PutObjectCommand, GetObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
import { randomUUID } from "crypto";

const s3 = new S3Client({ region: "ap-northeast-2" });
const BUCKET = "my-app-uploads";

// 업로드용 presigned URL 생성
async function createUploadUrl(
  contentType: string,
  userId: string
): Promise<{ uploadUrl: string; key: string }> {
  const key = `uploads/${userId}/${randomUUID()}`;

  const command = new PutObjectCommand({
    Bucket: BUCKET,
    Key: key,
    ContentType: contentType,
  });

  const uploadUrl = await getSignedUrl(s3, command, {
    expiresIn: 300, // 5분 후 만료
  });

  return { uploadUrl, key };
}

// 다운로드용 presigned URL 생성
async function createDownloadUrl(key: string): Promise<string> {
  const command = new GetObjectCommand({
    Bucket: BUCKET,
    Key: key,
  });

  return await getSignedUrl(s3, command, {
    expiresIn: 3600, // 1시간 후 만료
  });
}
```

### 11.4 프론트엔드에서 Presigned URL로 업로드

```typescript
// React 컴포넌트
async function handleFileUpload(file: File) {
  // 1. 백엔드에서 presigned URL 받기
  const response = await fetch("/api/upload-url", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ contentType: file.type }),
  });
  const { uploadUrl, key } = await response.json();

  // 2. S3에 직접 업로드
  await fetch(uploadUrl, {
    method: "PUT",
    body: file,
    headers: {
      "Content-Type": file.type,
    },
  });

  console.log(`Uploaded to: ${key}`);
}
```

> **핵심 통찰**: Presigned URL 패턴은 **파일 업로드의 사실상 표준**이다. 백엔드에서 파일을 `multipart/form-data`로 받아 S3에 전달하는 방식은 서버의 메모리와 대역폭을 낭비한다. 특히 이미지나 동영상 같은 대용량 파일을 다루는 서비스에서 presigned URL은 선택이 아닌 필수이다. 백엔드는 "URL 발급"만 담당하고, 실제 데이터 전송은 클라이언트와 S3가 직접 한다.

## 12. CORS 설정

### 12.1 CORS가 필요한 이유

브라우저에서 S3에 직접 접근할 때(presigned URL 업로드, 정적 에셋 로딩 등), S3 버킷의 도메인(`*.s3.amazonaws.com`)은 프론트엔드 도메인(`app.example.com`)과 다르다. 브라우저의 동일 출처 정책(*Same-Origin Policy - 보안을 위해 다른 도메인의 리소스 접근을 차단하는 브라우저 정책*)에 의해 요청이 차단된다.

```
CORS 에러 시나리오:

브라우저 (https://app.example.com)
    │
    │ PUT https://my-bucket.s3.ap-northeast-2.amazonaws.com/...
    │
    └──→ 🚫 CORS 에러!
         "Access to XMLHttpRequest at 'https://my-bucket.s3...'
          from origin 'https://app.example.com' has been blocked
          by CORS policy"
```

### 12.2 S3 CORS 설정

```bash
# CORS 설정
aws s3api put-bucket-cors \
  --bucket my-app-uploads \
  --cors-configuration '{
    "CORSRules": [
      {
        "AllowedOrigins": ["https://app.example.com", "http://localhost:3000"],
        "AllowedMethods": ["GET", "PUT", "POST"],
        "AllowedHeaders": ["*"],
        "ExposeHeaders": ["ETag", "x-amz-request-id"],
        "MaxAgeSeconds": 3600
      }
    ]
  }'
```

**각 필드의 의미:**

| 필드 | 설명 |
|------|------|
| **AllowedOrigins** | 접근을 허용할 도메인 목록. 개발 환경(`localhost`)도 포함 |
| **AllowedMethods** | 허용할 HTTP 메서드 |
| **AllowedHeaders** | 허용할 요청 헤더 |
| **ExposeHeaders** | 브라우저에서 읽을 수 있도록 노출할 응답 헤더 |
| **MaxAgeSeconds** | 브라우저가 preflight 결과를 캐시하는 시간 |

```bash
# CORS 설정 확인
aws s3api get-bucket-cors --bucket my-app-uploads
```

> **실무 팁**: CORS 디버깅에서 가장 많이 겪는 실수는 **프로덕션 배포 후 `AllowedOrigins`에 실제 도메인을 추가하지 않는 것**이다. 로컬 개발 시 `http://localhost:3000`만 넣어두고, 배포 후 `https://app.example.com`을 추가하지 않으면 프로덕션에서 CORS 에러가 발생한다. CI/CD 파이프라인에서 환경별 CORS 설정을 자동화하는 것이 좋다.

## 13. S3 + CloudFront 연동 개요

### 13.1 왜 CloudFront와 함께 사용하는가

S3 단독 사용의 한계:

- **HTTPS 미지원**: S3 정적 웹사이트 엔드포인트는 HTTP만 지원
- **지연 시간**: 서울 리전의 S3에 미국 사용자가 접근하면 느림
- **비용**: S3에서 직접 데이터를 서빙하면 데이터 전송 비용이 CloudFront보다 비쌈

```
S3 단독:                           S3 + CloudFront:

사용자 (미국)                      사용자 (미국)
    │                                  │
    │ HTTP (수백 ms)                   │ HTTPS (수십 ms)
    │                                  ↓
    │                             CloudFront 엣지 (미국)
    │                                  │ 캐시 히트: 즉시 응답
    │                                  │ 캐시 미스: 오리진에 요청
    ↓                                  ↓
S3 버킷 (서울)                    S3 버킷 (서울)
```

### 13.2 OAC (Origin Access Control)

OAC(*Origin Access Control - CloudFront만 S3 버킷에 접근할 수 있도록 제한하는 설정. 사용자가 S3 URL을 직접 접근하는 것을 차단한다*)를 사용하면 S3 버킷을 퍼블릭으로 열 필요가 없다. CloudFront를 통해서만 접근할 수 있으므로 보안이 강화된다.

```
OAC 아키텍처:

사용자
  │
  ├──→ CloudFront (https://d111.cloudfront.net) ──→ S3 버킷 ✅
  │     (OAC 자격 증명으로 접근)
  │
  └──→ S3 직접 접근 (https://bucket.s3.amazonaws.com) ──→ 🚫 403 Forbidden
       (Block Public Access 활성화)
```

**장점 요약:**

| 항목 | S3 단독 (퍼블릭) | S3 + CloudFront + OAC |
|------|-----------------|----------------------|
| **HTTPS** | 불가 | 가능 (ACM 인증서) |
| **전 세계 캐시** | 불가 | 엣지 로케이션에서 캐시 |
| **퍼블릭 접근** | 필요 | 불필요 (보안 강화) |
| **비용** | S3 데이터 전송 비용 | CloudFront 전송 비용 (더 저렴) |
| **커스텀 도메인** | 제한적 | 완전 지원 |

CloudFront와의 상세 연동 설정(배포 생성, 캐시 정책, 커스텀 에러 응답 등)은 **Chapter 10**에서 다룬다.

## 자주 하는 실수

| 실수 | 설명 | 올바른 접근 |
|------|------|------------|
| **버킷을 무분별하게 퍼블릭으로 열기** | Block Public Access를 끄고 `Principal: "*"`로 모든 접근 허용 | CloudFront + OAC를 사용하여 퍼블릭 접근 없이 콘텐츠 서빙 |
| **Content-Type 미설정** | HTML 파일이 다운로드되거나, JSON이 텍스트로 반환됨 | 업로드 시 `--content-type`을 명시적으로 지정 |
| **Cache-Control 미설정** | CDN과 브라우저가 파일을 적절히 캐시하지 못함 | 해시된 에셋은 `immutable`, 엔트리 포인트는 `no-cache` |
| **버전 관리 없이 중요 데이터 저장** | 실수로 덮어쓰거나 삭제하면 복구 불가 | 중요 버킷에는 버전 관리를 활성화 |
| **수명 주기 정책 미설정** | 오래된 로그/버전이 쌓여 스토리지 비용 증가 | 수명 주기 정책으로 자동 이동/삭제 설정 |
| **CORS 설정 누락** | 브라우저에서 S3 직접 접근 시 CORS 에러 | `AllowedOrigins`에 프론트엔드 도메인 추가 |
| **멀티파트 업로드 미완료 방치** | 실패한 업로드 파트가 스토리지 비용 발생 | AbortIncompleteMultipartUpload 수명 주기 규칙 설정 |
| **Presigned URL 만료 시간 너무 길게** | URL이 유출되면 오랜 기간 무단 접근 가능 | 업로드는 5~15분, 다운로드는 1시간 이내로 설정 |
| **S3 정적 호스팅으로 프로덕션 서비스** | HTTP만 지원, 캐시 없음, 느린 전 세계 접근 | 프로덕션에서는 반드시 CloudFront를 앞에 배치 |
| **모든 데이터를 Standard에 저장** | 접근하지 않는 데이터도 높은 비용 지불 | Intelligent-Tiering 또는 수명 주기 정책으로 비용 최적화 |

## 요약

- **S3**는 객체 스토리지 서비스로, 사실상 무제한의 데이터를 저장할 수 있으며 99.999999999%의 내구성을 보장한다. 블록/파일 스토리지와 달리 HTTP API를 통해 접근하며, 부분 수정 없이 전체 객체를 교체한다.
- **버킷**은 객체를 담는 최상위 컨테이너이며, 이름이 전 세계적으로 고유해야 한다. 버킷은 특정 리전에 생성되며 객체는 해당 리전을 벗어나지 않는다.
- **객체**는 키(경로) + 값(데이터) + 메타데이터로 구성된다. S3에 "폴더"는 존재하지 않으며, 키의 `/` 구분자로 시각적으로 그룹화할 뿐이다.
- **정적 웹사이트 호스팅**으로 React/Next.js SPA를 서빙할 수 있지만, HTTPS를 지원하지 않으므로 프로덕션에서는 CloudFront를 앞에 배치해야 한다.
- **접근 제어**는 Block Public Access(최상위 안전장치) + 버킷 정책(세밀한 제어) + IAM 정책의 조합으로 이루어진다. ACL은 레거시이므로 버킷 정책을 사용하라.
- **버전 관리**를 활성화하면 객체를 덮어쓰거나 삭제해도 이전 버전이 보존된다. 삭제 마커를 제거하면 객체를 복원할 수 있다.
- **수명 주기 정책**으로 오래된 객체를 저렴한 스토리지 클래스로 자동 이동하거나 삭제할 수 있다. AbortIncompleteMultipartUpload 규칙은 모든 버킷에 설정하라.
- **스토리지 클래스**: Standard(기본), Standard-IA(가끔 접근), Glacier(아카이브), Deep Archive(장기 보관), Intelligent-Tiering(자동 최적화)을 접근 빈도에 따라 선택한다.
- **S3 이벤트 알림**으로 객체 생성/삭제 시 Lambda, SQS, SNS를 자동 트리거할 수 있다. 이미지 리사이징, 메타데이터 추출 등에 활용한다.
- **멀티파트 업로드**는 대용량 파일(100MB+)을 병렬로 분할 업로드하는 방식이다. AWS CLI는 자동으로 처리하며, SDK에서는 `@aws-sdk/lib-storage`의 `Upload` 클래스를 사용한다.
- **Presigned URL**은 프론트엔드에서 S3에 직접 업로드/다운로드하는 표준 패턴이다. 백엔드는 URL 발급만 담당하고, 실제 데이터 전송은 클라이언트와 S3가 직접 한다.
- **CORS 설정**은 브라우저에서 S3에 직접 접근할 때 필수이다. `AllowedOrigins`에 프론트엔드 도메인을 포함해야 한다.
- **S3 + CloudFront + OAC** 조합이 프로덕션 표준이다. HTTPS, 전 세계 캐시, 퍼블릭 접근 차단을 모두 제공한다.

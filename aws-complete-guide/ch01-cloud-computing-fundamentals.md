# Chapter 1: AWS와 클라우드 컴퓨팅: Cloud Computing Fundamentals

## 핵심 질문

클라우드 컴퓨팅은 기존 온프레미스 인프라와 무엇이 다르며, AWS는 어떤 서비스 모델과 설계 원칙 위에 구축되어 있는가?

---

## 1. 클라우드 컴퓨팅이란

### 1.1 가장 단순한 정의

클라우드 컴퓨팅(*Cloud Computing - 인터넷을 통해 서버, 스토리지, 데이터베이스, 네트워킹 등의 컴퓨팅 리소스를 필요한 만큼 빌려 쓰는 모델*)은 한 문장으로 요약하면 **"다른 사람의 컴퓨터를 빌려 쓰는 것"**이다. 물론 이 "다른 사람의 컴퓨터"는 전 세계에 분산된 수백만 대의 서버로 구성된 거대한 데이터센터이며, 이를 체계적으로 관리하고 서비스로 제공하는 것이 클라우드 제공자(AWS, Azure, GCP 등)의 역할이다.

NIST(미국 국립표준기술연구소)의 공식 정의에 따르면, 클라우드 컴퓨팅은 다섯 가지 핵심 특성을 갖는다:

| 특성 | 설명 | 예시 |
|------|------|------|
| **온디맨드 셀프서비스** | 사람의 개입 없이 필요할 때 즉시 리소스를 프로비저닝 | AWS 콘솔에서 클릭 몇 번으로 서버 생성 |
| **광범위한 네트워크 접근** | 인터넷을 통해 어디서든 접근 가능 | 노트북, 모바일, API 등 다양한 방식으로 접근 |
| **리소스 풀링** | 여러 고객이 물리 리소스를 공유(멀티테넌트) | 하나의 물리 서버에 여러 고객의 VM이 실행 |
| **신속한 탄력성** | 수요에 따라 자동으로 리소스를 늘리거나 줄임 | 블랙프라이데이에 서버 100대 → 평소 10대 |
| **측정 가능한 서비스** | 사용한 만큼만 과금(Pay-as-you-go) | EC2 인스턴스를 3시간 쓰면 3시간분만 지불 |

### 1.2 온프레미스의 문제점

클라우드가 왜 필요한지 이해하려면, 먼저 전통적인 **온프레미스(*On-Premises - 기업이 자체 시설에 서버와 네트워크 장비를 직접 구축하고 운영하는 방식*)** 환경의 문제를 알아야 한다.

**문제 1: 용량 예측의 딜레마**

온프레미스에서는 서버를 미리 구매해야 한다. 트래픽이 얼마나 올지 예측해서 그에 맞는 장비를 주문하는데, 이 예측이 정확할 수 없다.

```
트래픽 예측 시나리오:

                  ┌─── 실제 트래픽
                  │    ┌─── 구매한 용량
    과잉 투자     │    │
    ──────────    │    │     ┌──── 리소스 낭비
              ████│████│█████│████
              ████│    │     │
              ████│    │     │
              ████│    │     │
    ──────────────┼────┼─────┼──────── 시간

    과소 투자     │    │
    ──────────    │    │     ┌──── 장애 발생!
              ████│    │     │
              ████│    │█████│████
              ████│████│█████│████████
              ████│████│█████│████████
    ──────────────┼────┼─────┼──────── 시간
```

- **과잉 투자**: 서버 10대를 샀는데 실제로는 3대만 필요 → 7대분의 비용이 낭비
- **과소 투자**: 서버 3대를 샀는데 트래픽 폭증 → 서비스 장애, 추가 서버 도착까지 2~4주 대기

**문제 2: 초기 비용과 리드 타임**

서버 한 대의 가격은 수백만~수천만 원이며, 주문 후 도착까지 수 주가 걸린다. 스타트업이 아이디어를 검증하기 위해 서버 인프라에 수천만 원을 먼저 투자하는 것은 합리적이지 않다.

**문제 3: 운영 부담**

서버를 직접 운영하면 하드웨어 교체, OS 패치, 보안 업데이트, 물리적 보안, 전력 관리, 냉각 시스템 등을 모두 직접 처리해야 한다. 이런 작업은 비즈니스 가치를 직접 만들지 않는 **"미분화된 무거운 작업(undifferentiated heavy lifting)"**이다.

> **핵심 통찰**: 클라우드의 핵심 가치는 "비용 절감"이 아니라 **"민첩성(agility)"**이다. 아이디어가 떠오르면 몇 분 안에 인프라를 구축하고, 실패하면 즉시 리소스를 반납할 수 있다. 이 속도가 비즈니스 경쟁력으로 직결된다.

## 2. 클라우드 서비스 모델

클라우드 서비스는 제공자가 관리하는 범위에 따라 세 가지 모델로 나뉜다. 이 분류를 이해하면 AWS의 수많은 서비스가 각각 어떤 레벨에서 동작하는지 파악할 수 있다.

### 2.1 IaaS, PaaS, SaaS

```
책임 범위 비교:

              On-Premises    IaaS         PaaS         SaaS
              ──────────    ──────       ──────       ──────
애플리케이션   [  사용자  ]  [  사용자  ]  [  사용자  ]  [제공자]
데이터         [  사용자  ]  [  사용자  ]  [  사용자  ]  [제공자]
런타임         [  사용자  ]  [  사용자  ]  [  제공자  ]  [제공자]
미들웨어       [  사용자  ]  [  사용자  ]  [  제공자  ]  [제공자]
OS            [  사용자  ]  [  사용자  ]  [  제공자  ]  [제공자]
가상화         [  사용자  ]  [  제공자  ]  [  제공자  ]  [제공자]
서버           [  사용자  ]  [  제공자  ]  [  제공자  ]  [제공자]
스토리지       [  사용자  ]  [  제공자  ]  [  제공자  ]  [제공자]
네트워킹       [  사용자  ]  [  제공자  ]  [  제공자  ]  [제공자]
```

**IaaS (Infrastructure as a Service)**

가상 서버, 네트워크, 스토리지 등 **인프라 자체**를 빌려 쓴다. OS 설치부터 애플리케이션 배포까지 모두 사용자가 관리한다.

```bash
# IaaS 예시: EC2 인스턴스에 직접 Node.js 설치 및 배포
ssh ec2-user@my-instance.amazonaws.com
sudo yum install -y nodejs
git clone https://github.com/my-app.git
cd my-app && npm install && npm start
```

- AWS 서비스: **EC2**, VPC, EBS
- 비유: 빈 사무실을 임대하는 것. 인테리어(OS, 런타임)는 직접 해야 한다.

**PaaS (Platform as a Service)**

OS, 런타임, 미들웨어까지 제공자가 관리한다. 사용자는 **코드만 배포**하면 된다.

```bash
# PaaS 예시: Elastic Beanstalk에 배포
eb init my-app --platform node.js
eb create production
eb deploy
# OS, Node.js 설치, 로드밸런서 설정 등은 Beanstalk이 자동 처리
```

- AWS 서비스: **Elastic Beanstalk**, **Amplify**, Lambda (FaaS)
- 비유: 풀옵션 사무실을 임대하는 것. 책상과 컴퓨터(런타임)가 이미 세팅되어 있다.

**SaaS (Software as a Service)**

완성된 소프트웨어를 서비스로 사용한다. 사용자는 인프라에 대해 전혀 신경 쓸 필요가 없다.

- AWS 서비스: **Amazon WorkMail**(이메일), **Amazon Chime**(화상회의)
- 일반 예시: Gmail, Slack, Notion
- 비유: 공유 오피스(WeWork)를 이용하는 것. 모든 것이 준비되어 있고, 들어가서 일만 하면 된다.

### 2.2 서비스 모델과 트레이드오프

| 모델 | 자유도 | 운영 부담 | 적합한 상황 |
|------|--------|-----------|------------|
| **IaaS** | 높음 | 높음 | 세밀한 인프라 제어가 필요할 때 |
| **PaaS** | 중간 | 낮음 | 빠른 개발/배포가 중요할 때 |
| **SaaS** | 낮음 | 없음 | 표준적인 기능을 바로 사용할 때 |

> **실무 팁**: 프론트엔드 개발자가 가장 자주 접하는 AWS 서비스는 PaaS 레벨이다. **Amplify**로 Next.js를 배포하거나, **S3 + CloudFront**로 정적 사이트를 호스팅하는 것이 대표적이다. IaaS(EC2)를 직접 다루는 것은 DevOps/인프라 팀의 영역인 경우가 많지만, 구조를 이해하면 협업이 훨씬 수월해진다.

## 3. AWS의 역사와 시장 위치

### 3.1 탄생 배경

AWS는 2006년에 **S3(스토리지)**와 **SQS(메시지 큐)**를 시작으로 서비스를 출시했다. 같은 해 EC2(가상 서버)가 베타로 공개되었다. Amazon이 자사 이커머스 플랫폼을 운영하면서 축적한 인프라 운영 노하우를 외부에 서비스로 제공한 것이 시작이다.

| 연도 | 이정표 |
|------|--------|
| **2006** | S3, SQS, EC2(베타) 출시 — 클라우드 시대의 개막 |
| **2009** | VPC 출시 — 격리된 네트워크 환경 제공 |
| **2012** | DynamoDB 출시 — NoSQL 서비스 본격화 |
| **2014** | Lambda 출시 — 서버리스 컴퓨팅의 시작 |
| **2017** | Fargate 출시 — 서버리스 컨테이너 실행 |
| **2018** | CDK 출시 — 코드로 인프라를 정의하는 IaC 도구 |
| **2023~** | 생성형 AI 서비스(Bedrock 등) 본격 확장 |

### 3.2 시장 점유율

2024년 기준, 글로벌 클라우드 인프라 시장에서 AWS는 약 **31%**의 점유율로 1위를 유지하고 있다. Microsoft Azure가 약 25%, Google Cloud Platform(GCP)이 약 11%로 뒤를 잇는다.

| 제공자 | 점유율(2024) | 강점 |
|--------|-------------|------|
| **AWS** | ~31% | 가장 넓은 서비스 범위, 성숙한 에코시스템 |
| **Azure** | ~25% | Microsoft 생태계 통합, 엔터프라이즈 강세 |
| **GCP** | ~11% | 데이터/AI/ML, Kubernetes(GKE) |

AWS를 먼저 배우면 좋은 이유는 단순히 점유율이 높아서만이 아니다. AWS에서 익힌 개념(VPC, IAM, 오토스케일링 등)은 Azure나 GCP에서도 **이름만 다를 뿐 거의 동일하게 적용**된다. 클라우드의 기본기를 다지기에 가장 적합한 플랫폼이다.

## 4. AWS 서비스 카테고리

AWS는 200개 이상의 서비스를 제공하지만, 실제로 대부분의 프로젝트에서 사용하는 핵심 서비스는 20개 미만이다. 서비스를 카테고리별로 정리하면 전체 그림이 보인다.

### 4.1 핵심 서비스 맵

```
┌─────────────────────────────────────────────────────────┐
│                    AWS 핵심 서비스 맵                       │
├──────────┬──────────┬──────────┬──────────┬──────────────┤
│  보안     │  컴퓨팅   │  스토리지  │ 데이터베이스│  네트워킹     │
│          │          │          │          │              │
│ IAM      │ EC2      │ S3       │ RDS      │ VPC          │
│ WAF      │ Lambda   │ EBS      │ DynamoDB │ Route 53     │
│ KMS      │ ECS      │ EFS      │ ElastiC. │ CloudFront   │
│ Secrets  │ Fargate  │ Glacier  │ Aurora   │ API Gateway  │
│ Manager  │          │          │          │ ELB          │
├──────────┴──────────┴──────────┴──────────┴──────────────┤
│                      배포/운영                             │
│                                                          │
│  Amplify · CodePipeline · CodeBuild · CloudWatch · SNS   │
└──────────────────────────────────────────────────────────┘
```

### 4.2 카테고리별 핵심 서비스

**컴퓨팅 (Compute)**

코드를 실행하는 서비스들이다. "어디에서 내 애플리케이션이 돌아가는가?"에 대한 답이다.

| 서비스 | 설명 | 사용 사례 |
|--------|------|----------|
| **EC2** | 가상 서버 (IaaS) | 커스텀 서버 환경, 레거시 앱 호스팅 |
| **Lambda** | 서버리스 함수 (FaaS) | API 엔드포인트, 이벤트 처리, 이미지 리사이징 |
| **ECS/Fargate** | 컨테이너 오케스트레이션 | Docker 컨테이너 운영, 마이크로서비스 |
| **Elastic Beanstalk** | 관리형 앱 플랫폼 (PaaS) | 빠른 배포, 인프라 관리 최소화 |

**스토리지 (Storage)**

데이터를 저장하는 서비스들이다.

| 서비스 | 설명 | 사용 사례 |
|--------|------|----------|
| **S3** | 객체 스토리지 | 정적 파일 호스팅, 이미지/동영상 저장, 백업 |
| **EBS** | 블록 스토리지 | EC2 인스턴스의 디스크 (SSD/HDD) |
| **EFS** | 파일 시스템 | 여러 EC2 인스턴스가 공유하는 파일 스토리지 |
| **S3 Glacier** | 아카이브 스토리지 | 장기 백업, 규정 준수용 데이터 보관 |

**네트워킹 (Networking)**

리소스 간의 네트워크 연결과 외부 트래픽 관리를 담당한다.

| 서비스 | 설명 | 사용 사례 |
|--------|------|----------|
| **VPC** | 가상 네트워크 | 리소스를 배치할 격리된 네트워크 환경 |
| **Route 53** | DNS 서비스 | 도메인 등록 및 DNS 라우팅 |
| **CloudFront** | CDN | 전 세계 엣지에서 콘텐츠 캐싱/배포 |
| **ELB** | 로드밸런서 | 트래픽을 여러 서버로 분산 |
| **API Gateway** | API 관리 | REST/WebSocket API 생성 및 관리 |

**데이터베이스 (Database)**

| 서비스 | 타입 | 사용 사례 |
|--------|------|----------|
| **RDS** | 관계형 DB (MySQL, PostgreSQL 등) | 전통적인 트랜잭션 데이터 |
| **Aurora** | 고성능 관계형 DB | RDS의 5배 성능이 필요한 워크로드 |
| **DynamoDB** | NoSQL (키-값/문서) | 대규모 읽기/쓰기, 밀리초 지연 |
| **ElastiCache** | 인메모리 캐시 (Redis/Memcached) | 세션 스토어, 캐싱 계층 |

**보안 (Security)**

| 서비스 | 설명 | 사용 사례 |
|--------|------|----------|
| **IAM** | 접근 제어 | 사용자/서비스 권한 관리 |
| **WAF** | 웹 방화벽 | SQL 인젝션, XSS 공격 차단 |
| **KMS** | 키 관리 | 데이터 암호화 키 관리 |
| **Secrets Manager** | 비밀 관리 | DB 비밀번호, API 키 안전 저장 |
| **ACM** | SSL/TLS 인증서 | HTTPS 인증서 무료 발급 및 관리 |

## 5. Well-Architected Framework

AWS는 **Well-Architected Framework**이라는 설계 원칙을 제공한다. 이는 AWS에서 시스템을 설계할 때 고려해야 할 여섯 가지 핵심 축(*Pillar*)을 정의한 것이다. 모든 AWS 서비스를 배울 때 이 프레임워크를 기준으로 생각하면 올바른 설계 결정을 내릴 수 있다.

### 5.1 여섯 가지 축

| 축 (Pillar) | 핵심 질문 | 예시 |
|-------------|----------|------|
| **운영 우수성** (Operational Excellence) | 시스템을 어떻게 운영하고 모니터링하는가? | CloudWatch 알람, 자동화된 배포 |
| **보안** (Security) | 데이터와 시스템을 어떻게 보호하는가? | IAM 최소 권한, 데이터 암호화, WAF |
| **안정성** (Reliability) | 장애가 발생해도 서비스가 계속 동작하는가? | Multi-AZ 배포, Auto Scaling |
| **성능 효율성** (Performance Efficiency) | 리소스를 효율적으로 사용하고 있는가? | 적절한 인스턴스 타입 선택, CloudFront 캐싱 |
| **비용 최적화** (Cost Optimization) | 불필요한 비용이 발생하고 있지 않은가? | 예약 인스턴스, 사용하지 않는 리소스 정리 |
| **지속 가능성** (Sustainability) | 환경에 미치는 영향을 최소화하고 있는가? | 효율적인 리소스 사용, 적절한 리전 선택 |

### 5.2 프론트엔드 개발자에게 특히 중요한 축

여섯 가지 모두 중요하지만, 프론트엔드 개발자가 일상적으로 가장 많이 접하는 세 가지 축이 있다:

1. **성능 효율성**: CloudFront CDN 설정, 이미지 최적화 전략, 캐시 정책 설계가 사용자 경험에 직결된다.
2. **보안**: `NEXT_PUBLIC_*` 환경 변수는 브라우저에 노출되므로, 민감한 정보(API 키 등)가 클라이언트에 포함되지 않도록 주의해야 한다. Secrets Manager와의 연동이 여기에 해당한다.
3. **비용 최적화**: S3 스토리지 클래스 선택, CloudFront 캐시 히트율 관리, Lambda 실행 시간 최적화 등이 비용에 직접 영향을 미친다.

> **비용 주의**: AWS는 **무료 체험 티어(Free Tier)**를 제공하지만, 한도를 초과하면 과금이 시작된다. 학습 중 예상치 못한 과금을 방지하려면 **빌링 알림(Billing Alert)**을 반드시 설정하라(Chapter 2에서 다룸).

## 6. 프론트엔드 개발자와 AWS

### 6.1 왜 프론트엔드 개발자가 AWS를 알아야 하는가

프론트엔드 개발은 더 이상 브라우저 안에서만 끝나지 않는다. 현대의 프론트엔드 애플리케이션은 서버 사이드 렌더링(SSR), 엣지 컴퓨팅, CDN 배포, 서버리스 API 등 인프라와 밀접하게 연결되어 있다.

```
현대 프론트엔드 애플리케이션의 인프라 의존성:

브라우저 ← CloudFront(CDN) ← S3(정적 파일)
    │                           │
    │                    Lambda@Edge(SSR)
    │
    └──→ API Gateway ──→ Lambda ──→ DynamoDB
                                    RDS
```

**구체적으로 알아야 하는 이유:**

1. **배포**: Next.js를 Amplify에 배포하거나, 정적 사이트를 S3 + CloudFront로 배포할 때 AWS 지식이 필요하다.
2. **성능**: CloudFront 캐시 정책을 이해하면 이미지/에셋의 로딩 속도를 극적으로 개선할 수 있다.
3. **비용**: 잘못된 S3 설정이나 CloudFront 구성이 불필요한 비용을 발생시킬 수 있다.
4. **협업**: 백엔드/DevOps 팀과 소통할 때 "VPC 안에서 Lambda가 RDS에 접근하는데 NAT Gateway가 필요하다"는 대화를 이해할 수 있어야 한다.
5. **풀스택**: Next.js의 API Routes, Server Actions 등을 사용하면 백엔드 로직도 작성하게 되는데, 이때 AWS 서비스와의 연동이 필수적이다.

### 6.2 이 가이드의 학습 경로

이 가이드는 기초부터 순서대로 읽어도 좋고, 필요한 챕터만 골라 읽어도 괜찮도록 구성했다. 다만 몇 가지 의존 관계가 있다:

```
필수 선수 지식 흐름:

Ch 1 (기초) ──→ Ch 2 (인프라/계정) ──→ Ch 3 (IAM)
                                        │
                    ┌───────────────────┤
                    ↓                   ↓
                Ch 4 (VPC)          Ch 9 (S3)
                    │                   │
            ┌───────┼───────┐     Ch 10 (CloudFront)
            ↓       ↓       ↓
        Ch 6(EC2) Ch 7(λ) Ch 8(ECS)   Ch 13 (Amplify)
```

- **Part 1 (Ch 1-3)은 필수**: 나머지 모든 챕터의 기초가 된다.
- **Part 2 (VPC, Route 53)**: EC2, RDS 등 VPC 안에서 동작하는 서비스를 다루기 전에 읽어야 한다.
- **Part 4 (S3, CloudFront)와 Part 6 (Amplify)**: 프론트엔드 개발자라면 이 부분을 먼저 읽어도 좋다. Part 1만 완료하면 바로 진입 가능하다.

## AWS CLI 레퍼런스

이 챕터에서는 특정 CLI 명령어를 사용하지 않지만, 앞으로의 실습을 위해 AWS CLI의 기본 구조를 미리 이해해두면 좋다.

```bash
# AWS CLI 기본 구조
aws <서비스> <명령> [옵션]

# 예시
aws s3 ls                          # S3 버킷 목록 조회
aws ec2 describe-instances         # EC2 인스턴스 목록 조회
aws iam list-users                 # IAM 사용자 목록 조회
aws lambda list-functions          # Lambda 함수 목록 조회

# 출력 형식 지정
aws s3 ls --output json            # JSON 형식
aws s3 ls --output table           # 테이블 형식
aws s3 ls --output text            # 텍스트 형식

# 리전 지정
aws ec2 describe-instances --region ap-northeast-2  # 서울 리전
```

AWS CLI 설치와 설정은 Chapter 2에서 자세히 다룬다.

## 요약

- **클라우드 컴퓨팅**은 컴퓨팅 리소스를 인터넷을 통해 필요한 만큼 빌려 쓰는 모델이다. 핵심 가치는 비용 절감보다 **민첩성(agility)**에 있다.
- 온프레미스의 세 가지 근본 문제는 **용량 예측의 딜레마**, **높은 초기 비용과 리드 타임**, **운영 부담**이다.
- 클라우드 서비스 모델은 **IaaS**(EC2), **PaaS**(Amplify, Lambda), **SaaS**(Gmail)로 나뉘며, 위로 갈수록 자유도는 높아지지만 운영 부담도 증가한다.
- AWS는 2006년 S3/EC2로 시작하여 현재 200개 이상의 서비스를 제공하는 **시장 점유율 1위** 클라우드 플랫폼이다.
- 핵심 서비스는 **컴퓨팅**(EC2, Lambda, ECS), **스토리지**(S3), **네트워킹**(VPC, CloudFront, Route 53), **데이터베이스**(RDS, DynamoDB), **보안**(IAM)으로 분류된다.
- **Well-Architected Framework**의 여섯 가지 축(운영 우수성, 보안, 안정성, 성능 효율성, 비용 최적화, 지속 가능성)은 모든 설계 결정의 기준이 된다.
- 현대 프론트엔드 개발은 CDN, 서버리스, SSR 등 인프라와 밀접하므로, AWS 지식은 **선택이 아닌 필수**다.

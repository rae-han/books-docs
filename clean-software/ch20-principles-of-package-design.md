## 1. 패키지를 이용한 설계

UML(*Unified Modeling Language - 객체지향 모델링 표준 표기법*)에서 패키지는 클래스들을 담는 그릇으로 쓰인다. 클래스를 패키지로 묶어 놓으면 더 높은 추상화 차원에서 설계에 대해 논의할 수 있다. 패키지는 소프트웨어 개발이나 배포를 관리하기 위해서도 쓰인다.

클래스가 다른 클래스에 의존 관계를 갖는 경우는 흔하며, 이런 의존 관계가 패키지 경계를 넘어서는 일도 많다. 따라서 패키지 사이에도 의존 관계가 생기기 마련이다. **패키지 사이의 관계는 높은 차원의 애플리케이션 내부 조직에 대한 표현이며, 이것을 관리하는 것이 우리의 역할이다.**

### 1.1 패키지를 둘러싼 다섯 가지 의문

1. 클래스를 패키지로 할당할 때 따를 원칙들은 무엇인가?
2. 어떤 설계 원칙들이 패키지 사이의 의존 관계를 지배하는가?
3. 클래스보다 패키지를 먼저 설계해야 하는가(하향식), 아니면 클래스를 먼저 설계해야 하는가(상향식)?
4. 패키지는 물리적으로 어떻게 표현해야 하는가? C++, 자바, 개발 환경에서는 각각 어떻게 하는가?
5. 패키지를 만들었다면, 이 패키지들을 어떤 목적으로 사용해야 하는가?

이 장에서는 패키지 작성과 상호관계, 그리고 패키지 사용을 지배하는 **여섯 가지 설계 원칙**을 제시한다.

### 1.2 여섯 가지 원칙 개요

| 분류 | 원칙 | 약어 | 다루는 대상 |
|------|------|------|-------------|
| **패키지 응집도** | 재사용-릴리즈 등가 원칙 | REP | 어떤 클래스가 같은 재사용 단위에 속하는가 |
| **패키지 응집도** | 공통 폐쇄 원칙 | CCP | 어떤 클래스가 같은 변경 이유를 갖는가 |
| **패키지 응집도** | 공통 재사용 원칙 | CRP | 어떤 클래스를 묶지 말아야 하는가 |
| **패키지 결합도** | 비순환 의존성 원칙 | ADP | 패키지 간 의존성 그래프의 모양 |
| **패키지 결합도** | 안정 의존성 원칙 | SDP | 의존의 방향 |
| **패키지 결합도** | 안정 추상화 원칙 | SAP | 안정성과 추상성의 관계 |

> **핵심 통찰**: 처음 세 원칙은 **클래스를 패키지로 분류하는 방법**을 다루며 상향식 접근을 따른다. 마지막 세 원칙은 **패키지 사이의 관계**를 다룬다. 그리고 마지막 두 원칙은 의존성 관리(*DM - Dependency Management*) 측정법까지 제공한다.

## 2. 재사용-릴리즈 등가 원칙 (REP: Reuse-Release Equivalence Principle)

> **재사용의 단위가 릴리즈의 단위다.**

### 2.1 재사용자가 작성자에게 기대하는 것

라이브러리를 재사용하려는 사람이 작성자에게 바라는 것은 단지 좋은 문서, 동작하는 코드, 잘 명세화된 인터페이스만이 아니다. 그것보다 훨씬 더 정치적이고 사무적인 보장이 필요하다.

| 재사용자가 요구하는 것 | 작성자가 제공해야 하는 것 |
|------------------------|--------------------------|
| **유지보수 보장** | 작성자가 직접 코드를 유지보수해 줄 것 |
| **변경 통보** | 인터페이스나 기능 변경 전 미리 알려줄 것 |
| **버전 선택권** | 새 버전을 거부할 권리 |
| **이전 버전 지원** | 일정 기간(3개월~1년) 옛 버전 계속 사용 가능 |

이런 요구를 충족시키려면 작성자는 자신의 소프트웨어를 **재사용 가능한 패키지로 조직한 다음 릴리즈 번호를 붙이고 계속 추적**해야 한다.

### 2.2 원칙의 의미

REP에 따르면 재사용 단위(예: 패키지)는 릴리즈 단위보다 작을 수 없다. 재사용하는 모든 것은 반드시 릴리즈된 다음 추적되어야 한다.

> **핵심 통찰**: 개발자가 그냥 클래스 하나 달랑 만들고 재사용 가능하다고 주장하는 것은 현실적이지 않다. **잠재적인 재사용자들에게 필요한 통보, 안전성, 지원에 대한 보장을 제공하는 추적 시스템이 먼저 있은 다음에야 재사용성이라는 말을 할 수 있다.**

### 2.3 패키지 내부 구조에 주는 시사점

- 패키지에 재사용될 소프트웨어가 들어 있다면, 그 패키지에는 **재사용을 목적으로 설계되지 않은 소프트웨어가 들어 있어서는 안 된다.**
- 패키지의 모든 클래스가 재사용 가능하든지, 모두 그렇지 않든지 해야 한다.
- 재사용자의 관점도 고려해야 한다 — 컨테이너 클래스 라이브러리와 금융 프레임워크는 둘 다 재사용 가능하지만, 한 패키지 안에 넣으면 안 된다. **패키지 안의 모든 클래스는 동일한 재사용자를 대상으로 해야 한다.**

> **Uncle Bob의 경험**: 정치적인 원인이 소프트웨어 분할에 영향을 미친다는 점이 마음에 걸릴 수도 있다. 하지만 소프트웨어는 순수한 수학적 규칙에 따라 구조를 잡을 수 있는 존재가 아니다. 소프트웨어는 사람의 노력을 뒷받침하는, 사람이 만드는 제품이다. 곧 소프트웨어는 사람이 만들고 사람이 사용한다. 만약 소프트웨어가 재사용될 예정이라면, 그 목적을 가진 사람이 편리하게 느낄 방식으로 분할되어야 한다.

## 3. 공통 재사용 원칙 (CRP: Common-Reuse Principle)

> **패키지 안의 클래스들은 함께 재사용되어야 한다. 어떤 패키지의 클래스 하나를 재사용한다면 나머지도 모두 재사용한다.**

### 3.1 함께 협력하는 클래스는 같은 패키지에

CRP에 따르면 자주 함께 재사용되는 클래스들은 동일한 패키지에 속해야 한다. 단독으로 재사용되는 클래스는 거의 없으며, 재사용 가능한 클래스들은 같은 추상적 범주에 속한 다른 클래스들과 협력하기 때문이다.

대표적인 예가 **컨테이너 클래스와 그에 딸린 반복자(iterator)**다. 이들은 서로 단단히 결합되어 있어 함께 재사용된다. 따라서 같은 패키지에 들어 있어야 한다.

<details>
<summary>원문 Java 코드</summary>

```java
// Java - 함께 재사용되는 컨테이너와 반복자
package com.example.collections;

public class LinkedList<T> {
    // ...
    public Iterator<T> iterator() { /* ... */ }
}

public class Iterator<T> {
    // LinkedList와 협력하는 반복자
}
```

</details>

```typescript
// TypeScript - 함께 재사용되는 컨테이너와 반복자
// package: collections

export class LinkedList<T> {
    iterator(): Iterator<T> {
        return new Iterator<T>(this);
    }
}

export class Iterator<T> {
    constructor(private list: LinkedList<T>) {}
    // LinkedList와 협력하는 반복자
}
```

### 3.2 CRP의 반대 측면: 묶지 말아야 할 클래스

CRP는 단지 "어떤 클래스를 같은 패키지에 넣어야 하는지"만 말하는 것이 아니다. **"어떤 클래스를 같은 패키지에 넣지 않아야 하는지"**도 말해 준다.

패키지가 다른 패키지를 사용하면 둘 사이에 의존 관계가 생긴다. 첫 번째 패키지가 두 번째 패키지에서 사용하는 클래스가 오직 하나뿐이라도, 의존 관계의 강약에는 변화가 없다.

```
PackageA  ─────depends on─────▶  PackageB
                                  ├── ClassX (사용함)
                                  ├── ClassY (사용 안 함)
                                  └── ClassZ (사용 안 함)
```

PackageB에서 ClassY가 변경되어 새 릴리즈가 나오면, PackageA는 ClassY를 전혀 사용하지 않더라도 **재검증하고 다시 릴리즈해야 한다.** 특히 패키지가 JAR이나 DLL 같은 물리적 형태로 나타나는 경우 더 심각하다.

> **핵심 통찰**: 어떤 패키지에 의존한다면 그 패키지의 모든 클래스에 의존하는지 확실히 해 두어야 한다. **CRP에 따르면 클래스 관계로 서로 단단히 묶여 있지 않은 클래스들은 같은 패키지에 넣지 말아야 한다.** 그렇지 않으면 필요 이상으로 재검증과 재배포해야 하는 일이 많아져 상당한 노력을 낭비하게 된다.

## 4. 공통 폐쇄 원칙 (CCP: Common-Closure Principle)

> **같은 패키지 안의 클래스들은 동일한 종류의 변화에 모두 폐쇄적이어야 한다. 패키지에 어떤 변화가 영향을 미친다면, 그 변화는 그 패키지의 모든 클래스에 영향을 미쳐야 하고 다른 패키지에는 영향을 미치지 않아야 한다.**

### 4.1 SRP의 패키지 버전

CCP는 대상이 패키지인 단일 책임 원칙(SRP)이다. 클래스를 변경할 이유가 여러 가지면 안 된다는 SRP와 마찬가지로, 이 원칙은 패키지를 변경할 이유도 여러 가지면 안 된다고 말한다.

| 원칙 | 대상 | 변경 이유 |
|------|------|-----------|
| **SRP** | 클래스 | 한 클래스는 단 하나의 변경 이유만 |
| **CCP** | 패키지 | 한 패키지는 단 하나의 변경 이유만 |

### 4.2 유지보수성 > 재사용성

> **핵심 통찰**: 대부분의 애플리케이션에서 재사용성보다는 유지보수성이 더 중요하다. 애플리케이션 코드를 꼭 변경해야 한다면, 여러 패키지를 고치는 것보다 패키지 하나만 고치고 싶을 것이다. 단일 패키지에만 변경의 초점이 맞춰져 있다면, 재배포할 필요가 있는 패키지도 그것뿐이다.

CCP는 **"동일한 이유로 변할 것 같은 클래스들을 한 장소에 모아 놓으라"**고 권장한다. 클래스 2개가 물리적으로나 개념적으로나 단단히 결합되어 있어서 언제나 함께 변경해야 한다면, 이들은 같은 패키지에 속한다.

### 4.3 OCP와의 관계

CCP는 개방 폐쇄 원칙(OCP)과도 밀접한 관련이 있다. OCP에 따르면 클래스는 변경에는 폐쇄되어 있되 확장에는 개방되어 있어야 한다. 하지만 **100% 폐쇄는 이룰 수 없다.** 폐쇄에 대해서는 전략적으로 생각해야 한다.

CCP는 **특정 종류의 변화에 개방되어 있는 클래스들을 같은 패키지 안에 몰아넣어** 이 전략을 더욱 확대한다. 따라서 요구 사항에 변화가 오더라도 그에 따라 변경할 패키지 수를 최소화할 수 있다.

## 5. 패키지 응집도의 균형

과거에는 응집도에 대한 생각이 단순했다 — "어떤 모듈이 단 하나의 기능만 수행하면 응집도가 높다"고 봤다. 하지만 패키지 응집도에 대한 세 원칙은 더 다양한 종류의 응집도가 있음을 보여 준다.

| 원칙 | 강조점 | 단계 |
|------|--------|------|
| **REP** | 재사용 가능성 | 재사용자 관점 |
| **CRP** | 패키지 분리 (배제) | 의존성 최소화 |
| **CCP** | 유지보수성 (포함) | 변경 영향 최소화 |

> **Uncle Bob의 경험**: 같은 패키지에 넣을 클래스를 결정하려면 재사용성과 개발 용이성에 관련된 **상충하는 힘들**을 모두 검토해야 한다. 게다가 이 균형은 거의 언제나 동적이다 — **오늘은 좋았던 클래스 분할이 내년에는 그렇지 않을 수도 있다.** 패키지 구성은 프로젝트의 초점이 개발 용이성에서 재사용성으로 옮겨감에 따라 시간이 흐르면서 조금씩 흔들리며 진화하기 마련이다.

## 6. 비순환 의존성 원칙 (ADP: Acyclic-Dependencies Principle)

> **패키지 의존성 그래프에서 순환을 허용하지 말라.**

### 6.1 "다음 날 아침 증후군"

하루 종일 일해서 어떤 것을 작동하게 만들어 놓고, 다음 날 다시 와 보니 또 작동하지 않는 경험을 해 본 적이 있는가? 어젯밤 누군가가 여러분이 의존하는 어떤 것을 변경했기 때문이다.

다음 두 가지 해결 방법이 발전해 왔다:

| 방법 | 설명 | 한계 |
|------|------|------|
| **주간 빌드** (Weekly Build) | 평일 4일 독립 개발, 금요일에 통합 | 프로젝트가 커질수록 통합 부담 증가 |
| **ADP** | 패키지 단위 릴리즈 + 비순환 의존성 | 의존성 구조를 적극 관리해야 함 |

### 6.2 패키지 단위 작업

해결 방법은 개발 환경을 **릴리즈로 만들 수 있는 패키지로 분할**하는 것이다.

- 패키지가 작업의 단위가 된다
- 개발자나 팀은 패키지를 체크아웃해서 작업한다
- 동작하면 릴리즈 번호를 붙여 다른 팀이 사용할 수 있게 한다
- 다른 팀은 새 릴리즈를 즉시 채택할지 결정할 수 있다 — **준비가 되지 않았다면 옛 릴리즈 계속 사용**
- 통합이 점진적으로 일어난다 — 단일 통합 시점이 없다

### 6.3 비순환 방향 그래프 (DAG)

이 방법이 동작하려면 패키지 의존성 구조에 순환이 있어서는 안 된다. 이상적인 구조는 **DAG(*Directed Acyclic Graph - 비순환 방향 그래프*)** 다.

```
                    ┌──────────────────┐
                    │  MyApplication   │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
       ┌──────────┐   ┌──────────┐   ┌──────────┐
       │ MyTasks  │   │MyDialogs │   │ Database │
       └────┬─────┘   └────┬─────┘   └──────────┘
            │              │
            ▼              ▼
       ┌──────────┐   ┌─────────────┐
       │  Tasks   │   │MessageWindow│
       └────┬─────┘   └──────┬──────┘
            │                │
            ▼                ▼
       ┌──────────┐   ┌──────────┐
       │TaskWindow│   │ Windows  │
       └────┬─────┘   └────┬─────┘
            │              │
            └──────┬───────┘
                   ▼
            ┌──────────┐
            │ Windows  │
            └──────────┘
```

#### DAG의 장점

| 장점 | 설명 |
|------|------|
| **영향 추적** | MyDialogs 변경 시 화살표를 거꾸로 따라가면 MyTasks, MyApplication만 영향 받음을 즉시 알 수 있음 |
| **독립 테스트** | MyDialogs를 테스트할 때 사용하는 Windows 패키지와 함께 컴파일·링크만 하면 됨 |
| **명확한 빌드 순서** | 상향식 — Windows → MessageWindow/MyDialogs → Task → TaskWindow/Database → MyTasks → MyApplication |

### 6.4 순환이 생기면 일어나는 일

새 요구 사항으로 MyDialogs의 클래스가 MyApplication의 클래스를 사용하게 되었다고 가정해 보자.

```
              ┌──────────────────┐
              │  MyApplication   │ ◀──────┐
              └────────┬─────────┘        │
                       │                   │ 순환!
              ┌────────┴────────┐         │
              ▼                 ▼          │
        ┌──────────┐     ┌──────────┐    │
        │ MyTasks  │     │MyDialogs │────┘
        └──────────┘     └──────────┘
```

순환이 생기자마자 문제가 생긴다:

| 문제 | 결과 |
|------|------|
| **사실상 거대한 패키지화** | MyApplication, MyTasks, MyDialogs는 항상 동시에 릴리즈해야 함 |
| **테스트 부담 폭증** | MyDialogs 하나 테스트하려고 전체 빌드를 해야 함 |
| **빌드 순서 결정 불가** | 올바른 빌드 순서가 존재할 수 없음 |
| **다음 날 아침 증후군 부활** | 서로의 일에 간섭하게 됨 |

### 6.5 순환을 끊는 두 가지 방법

#### 방법 1: 의존 관계 역전 원칙(DIP) 적용

MyDialogs가 필요한 인터페이스를 갖고 있는 **추상 기반 클래스**를 만들고, MyApplication이 이 추상 클래스를 상속하게 만든다.

```
          ┌──────────────────┐
          │  MyApplication   │
          └────────┬─────────┘
                   │ implements
                   ▼
          ┌──────────────────┐
          │  «interface»     │
          │     IServer      │
          └──────────────────┘
                   ▲
                   │ uses
          ┌────────┴─────────┐
          │     MyDialogs    │
          └──────────────────┘
```

<details>
<summary>원문 Java 코드</summary>

```java
// Java - DIP로 순환 끊기
// package: mydialogs
public interface IServer {
    void doSomething();
}

public class Dialog {
    private IServer server;
    public Dialog(IServer server) {
        this.server = server;
    }
}

// package: myapplication
import mydialogs.IServer;

public class Application implements IServer {
    public void doSomething() { /* ... */ }
}
```

</details>

```typescript
// TypeScript - DIP로 순환 끊기
// package: mydialogs
export interface IServer {
    doSomething(): void;
}

export class Dialog {
    constructor(private server: IServer) {}
}

// package: myapplication
import { IServer } from 'mydialogs';

export class Application implements IServer {
    doSomething(): void {
        // ...
    }
}
```

> **핵심 통찰**: 인터페이스 이름을 서버(MyApplication)가 아니라 **클라이언트(MyDialogs)를 따라** 지었다는 점에 주목하자. **인터페이스는 클라이언트에게 속한 것**이라는 규칙의 또 다른 적용 예다.

#### 방법 2: 공통 의존 대상이 되는 새 패키지 생성

MyDialogs와 MyApplication이 둘 다 의존하는 새로운 패키지를 만들고, 두 패키지가 모두 의존하는 클래스들을 이 새 패키지로 옮긴다.

```
          ┌──────────────────┐
          │  MyApplication   │
          └────────┬─────────┘
                   │
          ┌────────┴─────────┐
          │     MyDialogs    │
          └────────┬─────────┘
                   │
                   ▼
          ┌──────────────────┐
          │   aNewPackage    │
          └──────────────────┘
```

### 6.6 패키지 구성의 흔들림

두 번째 해결 방법에는 **요구 사항 변경이라는 현실 앞에서 패키지 구조가 쉽게 바뀐다**는 의미가 들어 있다. 애플리케이션이 성장하면서 패키지 의존성 구조도 흔들리며 성장한다.

---

## 7. 하향식 설계의 문제

> **핵심 통찰**: 패키지 구조는 **하향식으로 설계할 수 없다.** 시스템 설계 초기에 패키지 구조가 포함되지 않는다는 뜻이다. 실제로 패키지 구조는 시스템이 성장하고 변화하면서 같이 진화한다.

직관과 어긋날 수 있다 — 패키지처럼 큰 단위의 분해를 보면 시스템의 기능을 나타내야만 한다는 느낌을 받는다. 하지만 사실 패키지 의존 관계 다이어그램은 **애플리케이션의 기능을 기술하는 일과는 거의 관계가 없다.** 이 다이어그램은 **애플리케이션의 빌드 용이성을 보여 주는 일종의 지도**다.

### 7.1 패키지 구조의 진화 순서

| 단계 | 적용 원칙 | 초점 |
|------|-----------|------|
| **1. 초기** | 패키지 의존성 없음 | 클래스 설계 시작 |
| **2. 성장기** | SRP, CCP | 함께 변경되는 클래스 묶기 |
| **3. 재사용 검토** | CRP | 재사용 가능 요소 분리 |
| **4. 안정화** | ADP | 순환 탐지 → 새 패키지 생성으로 끊기 |

빌드할 소프트웨어가 없으므로 빌드 지도도 필요 없다. **패키지 의존 관계는 시스템의 논리적 설계와 함께 성장하고 진화해야 한다.**

## 8. 안정된 의존 관계 원칙 (SDP: Stable-Dependencies Principle)

> **의존은 안정적인 쪽으로 향해야 한다.**

### 8.1 변동성과 안정성의 비대칭

설계는 완전히 정적일 수 없다. CCP를 지키면 특정 변화에 쉽게 반응할 수 있는 패키지를 만들 수 있는데, 이 패키지들은 **쉽게 변하도록 의도적으로 설계된 패키지**다.

> **핵심 통찰**: 쉽게 바뀔 것이라고 예상되는 패키지들이 바뀌기 어려운 패키지들의 의존 대상이 되어서는 안 된다! 이렇게 되면 **쉽게 바뀔 패키지들도 바꾸기 어렵게 되어 버린다.** 모듈의 소스 코드 한 줄 건드리지 않았는데도 갑자기 그 모듈을 변경하기가 어려워지는 것이 소프트웨어의 기묘함이다.

### 8.2 안정성의 정의

웹스터 사전에 따르면 "어떤 것이 쉽게 움직이지 않는다면 그것은 안정적이다." **안정성은 변화의 빈도가 아니라, 변화를 만들기 위해 필요한 일의 양과 관련**되어 있다.

```
             3개 패키지가 의존              X 자신은
                  ↓                       어디에도 의존 안 함
          ┌──────────────────┐
   ───▶   │                  │
   ───▶   │       X          │  ← 안정적, 독립적
   ───▶   │   (책임이 큼)    │  ← 변경하지 말아야 할 이유 3개
          └──────────────────┘


   ┌──────┐
   │  Y   │ ← 책임 없음 (의존하는 자 없음)
   └──┬───┘   의존적 (외부 영향 3개)
      │       → 불안정
      ├──▶ ┌──────┐
      ├──▶ ├──────┤
      └──▶ └──────┘
```

| 용어 | 정의 |
|------|------|
| **책임이 있다(Responsible)** | 다른 패키지들이 이 패키지에 의존한다 |
| **독립적이다(Independent)** | 이 패키지가 다른 패키지에 의존하지 않는다 |
| **안정적이다(Stable)** | 책임이 있고 독립적이다 → 변경하기 어렵다 |
| **의존적이다(Dependent)** | 다른 패키지에 의존한다 |
| **불안정하다(Instable)** | 책임이 없고 의존적이다 → 변경하기 쉽다 |

### 8.3 안정성 측정법

패키지에 들어오거나 나가는 의존 관계 화살표의 수로 **위치적 안정성**을 계산할 수 있다.

| 측정값 | 영문 / 약어 | 의미 |
|--------|------------|------|
| **들어오는 결합** | Afferent Couplings, **Ca** | 패키지 내부 클래스에 의존하는 외부 클래스의 수 |
| **나가는 결합** | Efferent Couplings, **Ce** | 패키지 내부 클래스가 의존하는 외부 클래스의 수 |
| **불안정성** | Instability, **I** | I = Ce / (Ca + Ce) |

- I 값의 범위는 [0, 1]이다
- **I = 0** → 가장 안정적인 패키지 (Ca > 0, Ce = 0)
- **I = 1** → 가장 불안정한 패키지 (Ca = 0, Ce > 0)

#### 측정 예시

```
       PackageA              PackageB
       ┌─────┐               ┌─────┐
       │  q  │──────┐        │  u  │
       └─────┘      │        └──┬──┘
                    │           │
                    ▼           ▼
       PackageC  ┌─────┐    ┌─────┐
                 │  r  │    │  s  │ ← (PackageC 내부)
                 └──┬──┘    └─────┘
                    │
                    ▼
                 ┌─────┐
                 │  t  │ ← (PackageD 내부, PackageC가 의존)
                 └─────┘
```

PackageC의 안정성을 측정해 보자:
- PackageC 내부에 의존하는 외부 클래스 수: **Ca = 3** (q, u, …)
- PackageC 내부가 의존하는 외부 클래스 수: **Ce = 1** (t)
- **I = 1 / (3 + 1) = 0.25** → 비교적 안정적

> **핵심 통찰**: C++에서는 이런 의존 관계가 보통 `#include`로 나타난다. 자바에서는 `import` 문과 패키지 이름까지 완전히 포함한 이름을 세어서 계산한다. **SDP에 따르면 어떤 패키지의 I 측정값은 그 패키지가 의존하는 다른 패키지들의 I 값보다 반드시 커야 한다** — 즉, 의존 관계 방향으로 I가 줄어들어야 한다.

### 8.4 모든 패키지가 안정적일 필요는 없다

만약 시스템의 모든 패키지가 안정적이라면 시스템은 변경할 수 없게 된다. 이상적인 패키지 구성은 **불안정한 패키지가 위쪽, 안정적인 패키지가 아래쪽**에 있어 위쪽에서 아래쪽으로 의존하는 모양이다.

```
    ┌─────────┐     ┌─────────┐
    │  I = 1  │     │  I = 1  │  ← 불안정 (위)
    │Instable │     │Instable │
    └────┬────┘     └────┬────┘
         │               │
         └───────┬───────┘
                 ▼
            ┌─────────┐
            │  I = 0  │  ← 안정 (아래)
            │ Stable  │
            └─────────┘
```

> **핵심 통찰**: 불안정한 패키지를 다이어그램 위에 그리는 습관은 매우 유용하다. **위쪽으로 향하는 화살표는 모두 SDP를 위반**하는 것이기 때문이다.

### 8.5 SDP 위반 사례와 수정

```
     [위반 상황]                [DIP로 수정]
                                 
  Stable                       Stable
   ┌──┐                          ┌──┐
   │  │ U                        │  │ U
   └─┬┘                          └─┬┘
     │                             │ uses
     ▼ (위반!)                     ▼
   ┌──┐                       ┌──────────┐
   │  │ V                     │«interface»│
   └─┬┘                       │    IU    │  ← UInterface 패키지
     │                        └────▲─────┘
     ▼                             │ implements
  Flexible                         │
                              ┌────┴────┐
                              │ Flexible │
                              │    C     │ (V는 IU를 통해 C 사용)
                              └──────────┘
```

> **Uncle Bob의 경험**: Stable이 Flexible에 의존하면 Flexible은 더 이상 변경하기 쉬운 패키지가 아니게 된다. DIP를 사용해서 IU라는 인터페이스 패키지(UInterface)를 만들어 두 패키지가 모두 UInterface에 의존하게 만든다. UInterface는 매우 안정적이며(I = 0), Flexible은 자신이 필요한 불안정성을 다시 찾는다(I = 1).

## 9. 안정된 추상화 원칙 (SAP: Stable-Abstractions Principle)

> **패키지는 자신이 안정적인 만큼 추상적이기도 해야 한다.**

### 9.1 안정성과 추상성의 관계

높은 차원의 설계는 자주 바뀌면 안 된다. 따라서 이런 설계는 안정적인 패키지(I = 0)에 들어가야 한다. 하지만 안정적이기만 하면 변경할 수 없으므로 설계의 유연성이 떨어진다.

이 딜레마의 답은 **OCP**에서 찾을 수 있다 — 변경하지 않고도 확장할 수 있을 정도로 유연한 클래스를 만드는 것이 답이다. 그 답은 **추상 클래스(*Abstract Class - 인스턴스화할 수 없고 적어도 하나 이상의 추상 메서드를 가진 클래스*)** 다.

| 패키지 종류 | 안정성 | 추상성 | 권장 구성 |
|-------------|--------|--------|-----------|
| **안정적** | I = 0 | 높아야 함 | 추상 클래스 위주 |
| **불안정** | I = 1 | 낮아야 함 | 구체 클래스 위주 |

### 9.2 SDP + SAP = 패키지판 DIP

| 원칙 | 말하는 것 |
|------|-----------|
| **SDP** | 의존 관계의 방향이 안정성의 증가 방향과 같아야 한다 |
| **SAP** | 안정성은 추상성을 내포한다 |
| **합치면** | **의존 관계는 추상성의 방향으로 흘러야 한다** |

DIP는 클래스 대상이라 흑백이 분명하지만(추상이거나 아니거나), SDP+SAP는 패키지 대상이므로 **부분적으로 추상적이거나 부분적으로 안정적인 것**도 허용한다.

### 9.3 추상성 측정법

| 측정값 | 정의 |
|--------|------|
| **N** | 패키지 안에 들어 있는 클래스 수 |
| **Na** | 패키지 안에 들어 있는 추상 클래스 수 |
| **A** | 추상성 = Na / N |

- A 값의 범위는 [0, 1]이다
- **A = 0** → 추상 클래스가 하나도 없음
- **A = 1** → 패키지에 추상 클래스만 있음

## 10. 주 계열 (Main Sequence)

A를 Y축, I를 X축으로 삼은 그래프를 그려 보자.

### 10.1 A/I 그래프와 배제 구역

```
       A
       │
   1.0 ●  (0, 1)
       │ \ 쓸모없는 지역
       │  \  (Zone of Uselessness)
       │   \
       │    \
       │     \  주 계열
       │      \  (Main Sequence)
       │       \
       │        \
       │         \
       │          \
       │  고통의   \
       │  지역      \
       │ (Zone of    \
       │  Pain)       \
   0.0 ●─────────────●(1, 0)
       (0,0)          1.0    I
```

| 위치 | 명칭 | 의미 |
|------|------|------|
| **(0, 0) 근처** | **고통의 지역** (Zone of Pain) | 매우 안정적이고 매우 구체적 → 경직됨, 확장 불가 |
| **(1, 1) 근처** | **쓸모없는 지역** (Zone of Uselessness) | 매우 추상적이지만 의존하는 자가 없음 → 쓸모없음 |
| **(0, 1) ~ (1, 0) 직선** | **주 계열** (Main Sequence) | 두 배제 지역에서 가장 멀리 떨어진 이상적 위치 |

### 10.2 고통의 지역에 들어갈 수도 있는 패키지

> **Uncle Bob의 경험**: 고통의 지역에 들어갈 수밖에 없는 경우도 있다. **데이터베이스 스키마**가 대표적인 예 — 쉽게 바뀌기로 악명 높고, 극단적으로 구체적이며, 굉장히 의존도 많이 받는다. 이 점이 객체지향 애플리케이션과 데이터베이스 사이 인터페이스가 굉장히 어렵고 일반적으로 스키마 갱신이 고통스러운 이유다. **구체적인 유틸리티 라이브러리**(예: string 패키지)도 (0,0)에 있을 수 있지만, 별로 바뀔 가능성이 없으므로 큰 해는 없다.

### 10.3 주 계열의 의미

> **핵심 통찰**: 주 계열 위에 있는 패키지는 안정성에 비해 너무 추상적이지도 않고, 추상성에 비해 너무 불안정하지도 않다. 이 패키지는 쓸모없지도 않고 특별히 고통스럽지도 않다. **자신이 추상적인 정도만큼 의존의 대상이 되며, 자신이 구체적인 정도만큼 다른 패키지에 의존한다.**

가장 바람직한 패키지 위치는 주 계열의 양 끝점(0,1) 또는 (1,0) 중 하나다. 하지만 경험상 절반 이하만 그런 이상적 특징을 갖는다. 나머지는 주 계열 위나 가까이에 있을 때 가장 좋다.

### 10.4 주 계열로부터의 거리

| 측정값 | 정의 | 범위 |
|--------|------|------|
| **D** | 거리 = \|A + I - 1\| / √2 | [0, ~0.707] |
| **D'** | 정규화된 거리 = \|A + I - 1\| | [0, 1] |

- **D' = 0** → 패키지가 주 계열 바로 위에 있음
- **D' = 1** → 패키지가 주 계열에서 가장 멀리 떨어진 위치

이 측정값을 활용하면:
- 0에 가깝지 않은 D 값을 가진 패키지를 다시 검사해서 재구성 가능
- 설계 전체의 평균과 편차를 계산해서 통계적 분석 가능
- 시간 흐름에 따른 D' 변화 추적으로 의존성 악화 감지 가능

```
   D'
   │
0.2│
   │                            ● R2.1 ← 관리 한계 초과!
   │
0.1├──────────────────────관리 한계 (D' = 0.1)
   │   ●        ●
   │R1.0       R1.2    ●
   │       ●           R2.0
   │      R1.1
0.0└─────────────────────────────────────▶
     R1.0   R1.1   R1.2   R2.0   R2.1   릴리즈
```

## 11. 결론

> **핵심 통찰**: 이 장에서 설명한 의존성 관리 측정법들은 **이상적인 의존성과 추상성의 패턴에 어떤 설계가 얼마나 일치하는지**를 측정한다. 측정값은 신이 아니다 — 단지 임의로 만든 어떤 기준에 따라 측정해 본 값일 뿐이다. 본 책에서 선택한 기준이 특정 애플리케이션에만 적합하고 다른 것에는 부적합한 경우도 있을 수 있다.

여섯 가지 원칙은 다음과 같이 통합된다:

```
     [패키지 응집도]              [패키지 결합도]
     ┌─────────┐                 ┌─────────┐
     │   REP   │ 재사용=릴리즈    │   ADP   │ 순환 없음
     ├─────────┤                 ├─────────┤
     │   CRP   │ 분리 (배제)      │   SDP   │ 안정성 방향
     ├─────────┤                 ├─────────┤
     │   CCP   │ 묶음 (포함)      │   SAP   │ 추상성 = 안정성
     └─────────┘                 └─────────┘
          │                            │
          ▼                            ▼
    클래스 → 패키지            패키지 사이 의존성 그래프
```

## 요약

- **패키지(Package)**는 클래스보다 큰 조직화 단위이며, 동시에 **릴리즈 단위**이기도 하다
- 패키지 설계의 6원칙은 응집도 3개 + 결합도 3개로 구성된다

### 패키지 응집도 (클래스를 패키지에 어떻게 묶을 것인가)

| 원칙 | 한 줄 요약 |
|------|------------|
| **REP** (재사용-릴리즈 등가) | 재사용의 단위가 릴리즈의 단위다. 릴리즈 추적 없이 재사용성을 주장할 수 없다 |
| **CRP** (공통 재사용) | 함께 재사용되는 클래스만 같은 패키지에 — 강하게 묶이지 않은 클래스는 같이 넣지 말라 |
| **CCP** (공통 폐쇄) | 같은 이유로 변경되는 클래스를 한 패키지에 — SRP의 패키지 버전 |

### 패키지 결합도 (패키지 사이 관계를 어떻게 만들 것인가)

| 원칙 | 한 줄 요약 |
|------|------------|
| **ADP** (비순환 의존성) | 의존성 그래프는 DAG여야 한다 — 순환은 DIP나 새 패키지로 끊는다 |
| **SDP** (안정된 의존 관계) | 의존은 안정적인 쪽으로 향해야 한다 — I = Ce / (Ca + Ce) |
| **SAP** (안정된 추상화) | 안정적인 패키지는 추상적이어야 한다 — A = Na / N |

### 측정법 (DM Metrics)

- **Ca** (들어오는 결합) / **Ce** (나가는 결합) → **I = Ce / (Ca + Ce)** (불안정성)
- **N** (전체 클래스) / **Na** (추상 클래스) → **A = Na / N** (추상성)
- **주 계열**: (0,1) ~ (1,0)을 잇는 직선이 이상 영역
- **고통의 지역** (0,0): 구체적이면서 안정적 → 경직됨
- **쓸모없는 지역** (1,1): 추상적이면서 누구도 의존하지 않음 → 무의미
- **D' = \|A + I - 1\|** → 주 계열로부터의 거리 (0에 가까울수록 좋음)

### 패키지 설계는 하향식이 아니다

- 패키지 구조는 시스템 기능을 표현하지 않는다 — **빌드 용이성의 지도**다
- 처음부터 패키지 구조를 설계할 수 없다 — 클래스 설계와 함께 진화한다
- 진화 순서: **SRP·CCP → CRP → ADP**
- 측정값은 절대 기준이 아니라 **패턴 일치도를 평가하는 도구**일 뿐이다

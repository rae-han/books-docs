# Chapter 8: The Single-Responsibility Principle (단일 책임 원칙 / SRP)

## 핵심 질문

한 클래스가 여러 가지 일을 동시에 하면 무엇이 문제인가? "책임"이란 무엇이며, 책임을 어떻게 식별하고 분리해야 하는가? 그리고 — 책임을 분리하는 것이 좋다면, 왜 항상 분리하지는 않는가?

> 신비스러운 비밀들을 드러낸 것에 대한 책임은 다른 사람이 아닌 바로 부처 스스로 가져야 한다…<br>— E. Cobham Brewer, 『Dictionary of Phrase and Fable』(1898)

---

## 1. SRP의 정의

> **단일 책임 원칙 (Single-Responsibility Principle)**<br>**한 클래스는 단 한 가지의 변경 이유만을 가져야 한다.**

SRP는 톰 드마르코(Tom DeMarco)와 메이릴 페이지존스(Meilir Page-Jones)의 연구에서 설명된 **응집도(*cohesion - 모듈 요소 간의 기능적인 연관성*)** 개념에서 유래했다. 다만 본 책에서는 응집도를 "기능적 연관"보다는 **"모듈/클래스의 변경을 야기하는 응집력"**으로 의미를 조금 바꿔 사용한다.

### 1.1 볼링 게임 예제 (6장과의 연결)

6장에서 다룬 볼링 게임의 `Game` 클래스는 개발 과정 대부분에서 두 가지 책임을 맡고 있었다:

1. **현재 프레임을 기억하는 것** — 게임 상태 관리
2. **스코어를 계산하는 것** — 점수 계산 로직

결국 이 두 책임은 두 개의 클래스로 분리되었다:
- `Game` — 현재 프레임을 기억하는 책임
- `Scorer` — 스코어를 계산하는 책임

> **핵심 통찰**: 두 책임을 별개의 클래스로 분리한 이유는, **각 책임이 변경의 축**이기 때문이다. 요구 사항이 변경될 때, 변경은 클래스 안에서의 책임 변경을 통해 명백해진다. 한 클래스가 하나 이상의 책임을 맡는다면, 그 클래스를 변경할 하나 이상의 이유가 있다는 뜻이다.

### 1.2 결합된 책임의 위험

만약 한 클래스가 하나 이상의 책임을 맡는다면, 그 책임들은 **결합(coupling)**된다. 그 결과:

- 한 책임에 대한 변경이 다른 책임을 충족시키는 능력을 떨어뜨릴 수 있다
- 변경을 했을 때 예상치 못한 방식으로 잘못 동작하는 **취약한 설계**를 유발한다

## 2. Rectangle 예제 — 두 책임 결합의 폐해

### 2.1 나쁜 설계: 한 클래스에 두 책임

```
┌──────────────────────────┐         ┌──────────────────────────┐
│ Computational            │         │ Graphical                │
│ Geometry Application     │         │ Application              │
└────────────┬─────────────┘         └────────────┬─────────────┘
             │                                    │
             └─────────────┬──────────────────────┘
                           ▼
                   ┌──────────────┐
                   │  Rectangle   │
                   ├──────────────┤
                   │ + draw()     │  ◀── GUI에 의존
                   │ + area()     │
                   └──────────────┘
                           │
                           ▼
                         GUI
```

<details>
<summary>원문 Java 코드</summary>

```java
// Java
public class Rectangle {
    public void draw() {
        // GUI 라이브러리를 사용해 화면에 그린다
    }

    public double area() {
        // 넓이 계산
    }
}
```

</details>

```typescript
// TypeScript
class Rectangle {
    draw(): void {
        // GUI 라이브러리를 사용해 화면에 그린다
    }

    area(): number {
        // 넓이 계산
        return 0;
    }
}
```

### 2.2 두 책임이 결합된 결과

`Rectangle`은 서로 다른 두 애플리케이션에서 사용된다:

| 애플리케이션 | 사용 목적 |
|--------------|-----------|
| **Computational Geometry Application** | 수학적 계산만 사용. 화면에 그리지 않음 |
| **Graphical Application** | 계산 + 화면에 그림 |

이 설계는 SRP를 위반한다 — `Rectangle`은 두 가지 책임을 맡고 있다:
1. 직사각형 모양의 **수학적 모델 제공**
2. GUI에 **직사각형 그리기**

#### 위반의 구체적 폐해

**문제 1: 불필요한 의존성**
- 계산 기하학 애플리케이션에 **GUI를 포함시켜야 한다**
- C++라면 GUI가 링킹(*linking - 컴파일된 객체 파일들을 하나의 실행 파일로 결합*)되어 링킹 시간, 컴파일 시간, 메모리 영역을 소비
- Java라면 GUI를 위한 `.class` 파일들이 타깃 플랫폼에 배포되어야 함

**문제 2: 변경의 전염**
- `GraphicalApplication`의 변경이 `Rectangle`의 변경을 유발하면
- 그 변경 때문에 `ComputationalGeometryApplication`을 **재빌드, 재테스트, 재배포해야 할지도 모른다**
- 이를 잊으면 애플리케이션이 예상치 못한 방식으로 동작할 수 있다

### 2.3 좋은 설계: 책임 분리

```
┌──────────────────────────┐         ┌──────────────────────────┐
│ Computational            │         │ Graphical                │
│ Geometry Application     │         │ Application              │
└────────────┬─────────────┘         └────────────┬─────────────┘
             │                                    │
             ▼                                    ▼
   ┌─────────────────────┐             ┌──────────────────┐
   │ GeometricRectangle  │◀────────────│    Rectangle     │
   ├─────────────────────┤             ├──────────────────┤
   │ + area(): double    │             │ + draw()         │
   └─────────────────────┘             └────────┬─────────┘
                                                ▼
                                               GUI
```

이제 `GeometricRectangle`은 순수한 수학적 모델이고, `Rectangle`은 그것을 사용해 그리기만 한다. **직사각형이 그려지는 방식이 변경되어도 `ComputationalGeometryApplication`에 영향을 주지 않는다.**

## 3. 책임이란 무엇인가?

### 3.1 책임 = 변경을 위한 이유

SRP의 맥락에서 **책임(responsibility)**은 **변경을 위한 이유(a reason for change)**로 정의된다.

> 만약 여러분이 한 클래스를 변경하기 위한 **한 가지 이상의 이유**를 생각할 수 있다면, 그 클래스는 한 가지 이상의 책임을 맡고 있는 것이다.

때로 이것은 알아내기가 어렵다 — 우리는 책임을 묶어서 생각하는 데 익숙해져 있기 때문이다.

### 3.2 Modem 인터페이스 예제

다음 `Modem` 인터페이스를 보자:

<details>
<summary>원문 Java 코드</summary>

```java
// Java - 목록 8-1: Modem.java (SRP 위반)
interface Modem {
    public void dial(String pno);
    public void hangup();
    public void send(char c);
    public char recv();
}
```

</details>

```typescript
// TypeScript - SRP 위반
interface Modem {
    dial(pno: string): void;
    hangup(): void;
    send(c: string): void;
    recv(): string;
}
```

대부분의 사람은 이 인터페이스가 **타당해 보인다**고 동의할 것이다. 4개의 함수는 분명히 "모뎀에 속한 기능"을 표현한다.

그러나 여기에는 **두 개의 책임**이 보인다:

| 메서드 | 책임 |
|--------|------|
| `dial`, `hangup` | **연결 관리** (Connection Management) |
| `send`, `recv` | **데이터 통신** (Data Communication) |

### 3.3 분리해야 할까? — 변경의 축에 달려있다

> **핵심 통찰**: 이 두 책임이 분리되어야 할지는 **애플리케이션이 어떻게 바뀌느냐**에 달려있다. 만약 연결 함수의 시그너처에 영향을 주는 방식으로 애플리케이션이 바뀐다면, `send`와 `recv`를 호출하는 클래스는 더 자주 재컴파일·재배포되어야 할 것이고, 이 설계는 **경직성(rigidity)의 악취**를 풍기게 된다.

#### 분리된 설계

```
┌────────────────────────┐         ┌────────────────────────┐
│  «interface»           │         │  «interface»           │
│  DataChannel           │         │  Connection            │
├────────────────────────┤         ├────────────────────────┤
│ + send(c: char)        │         │ + dial(pno: String)    │
│ + recv(): char         │         │ + hangup()             │
└──────────┬─────────────┘         └──────────┬─────────────┘
           │                                   │
           └──────────────┬────────────────────┘
                          ▼
                ┌─────────────────────┐
                │ ModemImplementation │
                └─────────────────────┘
```

<details>
<summary>원문 Java 코드</summary>

```java
// Java - 분리된 Modem 인터페이스
interface DataChannel {
    public void send(char c);
    public char recv();
}

interface Connection {
    public void dial(String pno);
    public void hangup();
}

class ModemImplementation implements DataChannel, Connection {
    // 구현
}
```

</details>

```typescript
// TypeScript - 분리된 인터페이스
interface DataChannel {
    send(c: string): void;
    recv(): string;
}

interface Connection {
    dial(pno: string): void;
    hangup(): void;
}

class ModemImplementation implements DataChannel, Connection {
    dial(pno: string): void { /* ... */ }
    hangup(): void { /* ... */ }
    send(c: string): void { /* ... */ }
    recv(): string { return ""; }
}
```

### 3.4 책임 분리의 균형

> **핵심 통찰**: 변경의 축은 **변경이 실제로 일어날 때만** 변경의 축이다. 아무 증상도 없는데 SRP나 다른 원칙을 적용하는 것은 현명하지 못하다. 분리할 필요가 없다면, **불필요한 복잡성의 악취**를 풍기게 된다.

| 상황 | 권장 행동 |
|------|-----------|
| 두 책임이 서로 다른 시간에 변경된다 | **분리** — SRP 적용 |
| 두 책임이 동일한 시간에만 변경된다 | **결합 유지** — 분리하면 복잡성만 늘어남 |

## 4. 결합된 책임 분리하기

`ModemImplementation` 클래스를 보면, 두 책임을 **결합된 상태로 놔뒀다**는 점에 주목하자. 왜인가?

### 4.1 필요악으로서의 결합

> **Uncle Bob의 경험**: 모든 의존성은 필요악일 수 있다. 하드웨어나 OS의 세부적인 사항과 관련된 이유로 인해, 오히려 책임이 결합되도록 만드는 경우가 종종 있다. 하지만 **인터페이스는 분리**하여 애플리케이션의 나머지 부분에 한해 개념을 분리했다.

### 4.2 결합 격리의 핵심: 의존성 방향

`ModemImplementation`은 어울리지 않는 요소(*kludge - 임시방편적이고 어색한 해결책*)나 혹(hack)으로 볼 수도 있다. 그러나 **모든 의존성 흐름이 여기서 나간다**는 점이 핵심이다.

```
   DataChannel   Connection         ◀── 클라이언트가 의존
        ▲             ▲
        │             │
        └─────┬───────┘
              │
   ModemImplementation                ◀── 이 클래스에 아무도 의존하지 않음
              │
              ▼
            main                      ◀── 생성만 알면 됨
```

- 아무것도 `ModemImplementation`에 의존하지 않는다
- `main` 외에 어느 것도 이것이 존재한다는 사실을 알 필요가 없다
- **담을 세우고 그 뒤에 보기 싫은 비트를 집어넣은 것**이다
- 이 보기 싫은 부분이 애플리케이션의 나머지를 오염시킬 필요가 없다

## 5. Persistence 예제 — 흔한 SRP 위반

### 5.1 결합된 영속성

```
┌─────────────────────┐         ┌─────────────────────┐
│      Employee       │────────▶│    Persistence      │
├─────────────────────┤         │    Subsystem        │
│ + CalculatePay()    │         └─────────────────────┘
│ + Store()           │
└─────────────────────┘
```

`Employee` 클래스는 두 책임을 결합하고 있다:

| 책임 | 변경 빈도 | 변경 이유 |
|------|-----------|-----------|
| **업무 규칙** (`CalculatePay`) | 자주 | 비즈니스 정책 변화 |
| **영속성 제어** (`Store`) | 드물게 | 데이터베이스, ORM 변화 |

> **핵심 통찰**: 이 두 책임은 거의 어울리지 않는다. **변경 빈도가 다르고, 변경 이유도 완전히 다르다.** 업무 규칙과 영속성 서브시스템을 묶는 것은 문제를 부르는 것이나 다름없다.

### 5.2 해결: TDD가 자연스럽게 강제한다

> **Uncle Bob의 경험**: 다행히도 4장에서 본 바와 같이 **테스트 주도 개발(TDD) 방식은 보통 설계에서 악취가 나기 한참 전에 이 두 책임이 분리되도록 만든다.** 그러나 테스트가 분리를 강제하지 않았고, 경직성과 취약성의 악취가 강해진 경우에는, 이 설계에 **퍼사드(FACADE) 패턴**이나 **프록시(PROXY) 패턴**을 사용해서 두 책임이 분리되도록 리팩토링해야 한다.

## 6. 자주 하는 실수

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| **God Class** — 한 클래스가 도메인의 모든 책임을 떠안음 | 변경의 모든 축이 한 곳에 모임 | 책임별로 클래스 분리 |
| **이른 분리** — 변경 증상이 없는데 SRP 적용 | 불필요한 복잡성, 클래스 폭증 | 변경의 축이 드러날 때까지 기다림 |
| **데이터+영속성 결합** — 도메인 객체가 자기 자신을 DB에 저장 | 업무 규칙 변경과 DB 변경이 충돌 | Repository 패턴 등으로 분리 |
| **데이터+UI 결합** — 도메인 객체가 자기 자신을 그림 | 비즈니스 로직과 표현 로직이 함께 변함 | View/Renderer 분리 |
| **인터페이스 미분리** — 구현은 결합되어도 인터페이스까지 결합 | 클라이언트가 불필요한 책임 모두에 의존 | 인터페이스를 책임별로 분리 (ISP와 연결) |

## 7. 설계 원칙

| 원칙 | 설명 |
|------|------|
| **변경의 축이 책임이다** | 책임은 "기능적 묶음"이 아니라 "함께 변하는 것" |
| **변경이 실제로 일어날 때만 분리하라** | 가설적 변경에 대비한 SRP 적용은 불필요한 복잡성을 낳음 |
| **인터페이스는 분리, 구현은 결합 허용** | 하드웨어/OS 제약으로 구현이 결합되어도, 인터페이스는 항상 분리 |
| **의존성 방향이 핵심이다** | 보기 싫은 결합은 격리하되, 다른 코드가 그것에 의존하지 않도록 |

## 8. 결론

> **핵심 통찰**: SRP는 **가장 간단한 원칙 중 하나임과 동시에 제대로 적용하기 가장 어려운 원칙 중 하나**이기도 하다. 책임들을 결합하는 것은 너무나 자연스럽게 해버리고 마는 일이다. 이런 책임을 찾고 하나씩 분리하는 것이 소프트웨어 설계에서 실제로 하는 일의 **대부분**이다.

이후에 논할 나머지 SOLID 원칙들도 어떤 식으로든 이 문제로 돌아오게 된다.

## 요약

- **SRP의 정의**: 한 클래스는 단 한 가지의 변경 이유만을 가져야 한다
- **책임 = 변경의 축** — 함께 변경되는 것이 한 책임이고, 따로 변경되는 것이 다른 책임이다
- **결합의 폐해**:
  - 한 책임 변경이 다른 책임 동작을 깨뜨릴 수 있음
  - 불필요한 의존성으로 빌드/배포 부담
  - 변경의 전염
- **분리 판단 기준**: 변경의 축이 실제로 드러날 때 분리. 가설적 분리는 불필요한 복잡성
- **인터페이스는 항상 분리, 구현은 결합 허용** — `ModemImplementation` 패턴
- **TDD는 자연스럽게 SRP를 강제**한다 — 테스트가 책임을 자연스럽게 분리시킴

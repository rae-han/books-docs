# Appendix A: UML Notation I — CGI Example (UML 표기법 I: CGI 예제)

## 핵심 질문

분석과 설계에 모두 쓸 수 있는 단일 표기법이란 가능한가? UML(*Unified Modeling Language - 객체지향 분석과 설계를 위한 통합 표기법*)의 어떤 부분이 분석에 적합하고, 어떤 부분이 설계에 적합한가? 도메인 모델, 유스케이스, 클래스 다이어그램, 시퀀스 다이어그램은 각각 무엇을 표현하며 어떻게 함께 쓰이는가?

---

## 1. UML의 등장 배경

소프트웨어의 분석과 설계는 표기법을 절실히 필요로 한다. 흐름도, 데이터 흐름도, 개체(*entity - 시스템에서 다루는 개념적 단위*) 관계 다이어그램 등 다양한 표기법이 제안되어 왔다. 객체지향 프로그래밍의 출현 이후에는 표기법이 폭발적으로 증가했다.

대표적인 객체지향 표기법:

| 표기법 | 제안자 | 특징 |
|--------|--------|------|
| Booch94 | Grady Booch | 강력한 **설계** 표기법으로 환영받음 |
| OMT (Object Modeling Technique) | Rumbaugh 등 | 강력한 **분석** 표기법으로 여겨짐 |
| RDD (Responsibility Driven Design) | Wirfs-Brock 등 | 책임 주도적 설계 |
| Coad/Yourdon | Peter Coad, Ed Yourdon | 분석 중심 |

> **핵심 통찰**: 1980년대 후반에는 "분석과 설계를 같은 표기법으로 표현할 수 있다"는 점이 객체지향의 장점으로 여겨졌다. 그러나 시간이 흐르면서 분석가는 OMT를, 설계자는 Booch94를 선호하게 되었다. **하나의 표기법만으로는 부족하다**는 사실이 드러난 것이다. UML은 단일 표기법이면서도 분석과 설계 모두에 적용할 수 있는 넓은 응용성을 제공한다.

이 부록은 축소된 사례 연구(교육 과정 등록 시스템)를 통해 UML의 다양한 측면을 보여준다. 분석을 먼저 하고 설계를 나중에 하는 순서에는 특별한 의미가 없으며, 실제 프로젝트에서는 추상화의 모든 단계가 **동시적으로** 진행된다.

## 2. 교육 과정 등록 시스템: 문제 설명

객체지향 분석과 설계 전문 교육 과정을 제공하는 회사를 위한 시스템을 만든다고 가정한다. 이 회사는 수업과 등록한 학생을 기록하는 시스템을 원한다.

### 2.1 요구사항 요약

- 사용자는 수강 가능한 강좌 메뉴를 보고 등록할 강좌를 선택할 수 있다
- 선택 후, 사용자는 폼에 이름·전화번호·팩스번호·이메일 주소를 입력한다
- 지불 방식 선택: **수표 / 구매주문서 / 신용카드**
  - 수표: 수표번호 입력
  - 신용카드: 카드번호·유효기간·카드상 이름 입력
  - 구매주문서: PO 번호·회사명·회계부 담당자 이름과 전화번호 입력
- 등록 완료 후 요약 화면 표시 → 출력 → 서명 → 등록센터로 팩스 송부
- 등록 정보 요약본을 등록센터 직원과 사용자에게 이메일 전송
- 각 강좌는 최대 학생 수가 정해져 있으며, 꽉 차면 자동으로 '마감' 표시
- 등록센터 직원은 강좌 등록자에게 일괄 이메일 전송 가능
- 등록센터 직원은 출석/수강료 지불 상태를 조회·변경 가능

## 3. 액터와 유스케이스

요구사항 분석의 한 작업은 액터(*actor - 시스템과 대화하는 외부 엔티티, 보통 사용자 역할*)와 유스케이스(*use case - 한 액터의 시점에서 본 시스템과의 대화 명세*)를 식별하는 것이다.

### 3.1 액터 식별

이 시스템의 모든 액터는 사람이다:

| 액터 | 역할 |
|------|------|
| **수강신청인 (Enroller)** | 학생을 강좌에 등록시킨다. 강좌 선택, 학생 정보 입력, 결제 방식 선택 |
| **수강등록 직원 (Enrollment Clerk)** | 등록 통보 이메일 수신, 학생에게 공지 이메일 발송, 출석·결제 보고 |
| **학생 (Student)** | 등록 확인 이메일 수신, 등록한 강좌에 출석 |

### 3.2 유스케이스 표기법

액터는 사람 모양의 막대 그림으로, 유스케이스는 타원으로 표현하고 둘은 데이터 흐름 방향을 나타내는 선으로 묶인다.

```
    유스케이스 #1: 강좌 메뉴 보기
    
      ┌─┐
      │ │           ┌──────────────────┐
     ─┼─  ────────▶│  강좌 메뉴 보기  │
      │ │           └──────────────────┘
     / \
    수강신청인
```

### 3.3 주요 유스케이스

**유스케이스 #1: 강좌 메뉴 보기** — 수강신청인은 강좌 카탈로그에서 현재 수강할 수 있는 강좌 목록을 요청한다. 시스템은 이름·시간·장소·수강료를 표시하고, 최대 학생 수와 마감 여부도 보여준다.

**유스케이스 #2: 강좌 등록하기** — 수강신청인은 먼저 강좌 메뉴를 본다(유스케이스 #1을 `<<include>>`). 메뉴에서 강좌를 선택하면 시스템은 학생 정보와 결제 방식을 묻는다. 폼 제출 후 학생과 등록 직원에게 확인 이메일이 발송된다.

확장 위치(*extension point - 다른 유스케이스가 이 유스케이스를 확장하는 지점*)에서 결제 방식별로 분기된다:

- **유스케이스 #2.1: 구매주문서로 지불하기** — PO 번호, 회사명, 담당자 정보 입력
- **유스케이스 #2.2: 수표로 지불하기** — 수표번호 입력
- **유스케이스 #2.3: 카드로 지불하기** — 카드번호·유효기간·이름 입력

**유스케이스 #3: 학생들에게 이메일 보내기** — 수강등록 직원이 강좌를 선택하고 메시지를 입력하면, 시스템은 그 강좌 등록자 전원에게 메시지를 발송한다.

**유스케이스 #4: 출석 기록하기** — 직원이 강좌와 학생을 선택하면 시스템은 출석/지불 상태를 표시하고, 직원이 정보를 변경할 수 있게 한다.

**유스케이스 #5: 학생 상태 보기** — 직원이 관심 있는 학생을 선택하면 시스템은 출석과 납부 상태를 보고서 형태로 보여준다. 확장 유스케이스(#5.1 한 강좌의 학생들 선택, #5.2 미납 학생들 선택)가 존재한다.

### 3.4 `<<include>>` vs `<<extend>>`

```
   ┌───────────────┐                    ┌───────────────┐
   │  강좌 메뉴    │                    │  강좌 등록    │
   │     보기      │◀ - - - - - - - - - │     하기      │
   └───────────────┘   «include»        └───────────────┘
                                                ▲
                                                │ «extend»
                                                │
                                        ┌───────────────┐
                                        │ 카드로 지불   │
                                        │     하기      │
                                        └───────────────┘
```

| 관계 | 의미 | 목적 | 화살표 방향 |
|------|------|------|------------|
| `<<include>>` | 포함하는 유스케이스가 **포함되는 유스케이스를 참조** | 중복 제거. 공통 부분을 한 곳으로 모음 | 포함되는 쪽을 가리킴 |
| `<<extend>>` | 확장하는 유스케이스가 **확장되는 유스케이스의 문맥에 삽입됨** (확장되는 쪽은 모름) | 변경 관리. 변하는 부분과 변하지 않는 부분 분리 | 확장**되는** 쪽을 가리킴 |

> **핵심 통찰**: `<<include>>`의 목표는 **중복 제거**, `<<extend>>`의 목표는 **변경 관리**다. 새 지불 방법 옵션이 추가되어도 기존 유스케이스를 수정하지 않고 새 확장 유스케이스만 추가하면 된다.

### 3.5 시스템 경계 다이어그램

전체 유스케이스 집합은 시스템 경계를 표현하는 직사각형으로 묶어 한눈에 볼 수 있다.

```
                  ┌───────── 교육 과정 등록 시스템 ──────────┐
                  │                                            │
                  │    ┌────────────┐                          │
                  │    │ 강좌 메뉴  │                          │
                  │    │   보기     │◀───┐                     │
                  │    └────────────┘    │ «include»           │
                  │            ▲         │                     │
   ┌─┐            │            │   ┌────────────┐              │
  ─┼─ ───────────▶│            └───│ 강좌 등록  │              │
   │ │            │                │   하기     │              │
  / \             │                └────────────┘              │
 수강신청인       │                                            │
                  │    ┌────────────┐                          │
                  │    │ 이메일     │                          │
                  │ ┌─▶│ 보내기     │                          │
                  │ │  └────────────┘                          │
   ┌─┐            │ │                                          │
  ─┼─ ────────────┤ │  ┌────────────┐         ┌─┐              │
   │ │            │ ├─▶│ 출석 기록  │         │ │              │
  / \             │ │  └────────────┘        ─┼─               │
 수강등록직원     │ │                         │ │              │
                  │ │  ┌────────────┐        / \               │
                  │ └─▶│ 학생 상태  │◀───── 학생                │
                  │    │   보기     │                          │
                  │    └────────────┘                          │
                  └────────────────────────────────────────────┘
                       그림 A-1 시스템 경계 다이어그램
```

이 다이어그램은 소프트웨어 구조 다이어그램이 **아니다**. 시스템의 어떤 소프트웨어 구성요소에 대한 정보도 주지 않는다. 분석가와 이해 당사자 간의 **의사소통**을 위한 것이다.

> **핵심 통찰**: 마틴의 문서화 제1법칙 — **"그 필요가 급박하고 중요하지 않다면 아무 문서도 만들지 마라."** 이 다이어그램은 유용할 수도 있지만 종종 불필요할 때도 있다. 필요할 때 그리고, 그렇지 않다면 필요해질 때까지 기다려라.

## 4. 도메인 모델

도메인 모델(*domain model - 유스케이스에 나오는 용어들의 정의를 도와주는 다이어그램 집합*)은 문제의 핵심 객체들과 상호관계를 보여준다.

> **핵심 통찰**: 도메인 모델의 객체는 **반드시 소프트웨어의 객체지향 설계와 대응하는 것이 아니다.** 도메인 모델은 사람들이 결정을 기록하고 의사소통하는 것을 돕기 위한 기술 도구다. 미숙한 프로젝트에서는 도메인 모델을 코드가 될 준비가 되어 있는 것으로 착각하여 설계 과정을 생략하는 경향이 있다.

### 4.1 `<<type>>` 스테레오타입과 구름 아이콘

UML은 도메인 모델의 클래스에 `<<type>>` 스테레오타입(*stereotype - UML 요소의 특별한 종류를 지정하는 표기*)을 사용해 **개념적 엔티티**임을 표시한다. `<<type>>`은 다른 `<<type>>`과의 연계, 오퍼레이션, 속성을 가질 수 있지만 직접적으로 코드에 대응되지는 않는다.

설계/구현용 UML 클래스와 시각적으로 구분하기 위해, `<<type>>` 클래스에는 **구름 아이콘**을 사용하는 것이 권장된다.

```
            ╭────────────────╮
           ╱  Course Catalog  ╲
          (                    )
           ╲   AddCourse()    ╱
            ╲  RemoveCourse()╱
             ╰───────┬──────╯
                     │ offers
                     │ 0..n
                     ▼
            ╭────────────────╮
           ╱     Course       ╲
          (                    )
           ╲  fee: Currency   ╱
            ╲ enrollmentLimit╱
             ╰────────────────╯
            그림 A-3 강좌 카탈로그 도메인 모델
```

`Course Catalog`가 `AddCourse`, `RemoveCourse` 오퍼레이션을 갖는 것은 "강좌를 추가하고 삭제할 수 있어야 한다"는 **책임**을 표현한다. 실제 클래스의 멤버 함수 명세가 아니라 개념일 뿐이다.

### 4.2 도메인 모델 완성

강좌 하나가 여러 시간·장소에서 개설될 수 있고 교사도 다를 수 있다. 따라서 두 엔티티가 필요하다:

- **Course**: 강좌 그 자체(날짜·위치·교사 제외)
- **Session**: 특정 날짜·위치·교사를 가진 강좌의 한 회

```
   ╭───────────────────╮
  ╱  Course Catalog    ╲
 (   version: date      )
  ╲  AddCourse()       ╱
   ╲ RemoveCourse()   ╱
    ╰───────┬────────╯
            │ offers
            │ 0..n
            ▼
   ╭───────────────────╮          ╭──────────────────────╮
  ╱     Course          ╲ teaches╱  Session Schedule     ╲
 (    fee: currency      )◀──────(                        )
  ╲  enrollmentLimit    ╱        ╲   AddSession()        ╱
   ╰────────────────────╯         ╲  RemoveSession()    ╱
                                   ╰─────────┬─────────╯
                                             │ schedules
                                             │ 0..n
                                             ▼
                                  ╭──────────────────────╮
                                  ╱      Session          ╲
                                 (    session date         )
                                  ╲   current enrollment  ╱
                                   ╲     instructor      ╱
                                    ╰─────────┬─────────╯
                                              │ enrolled in
                                              │ 0..n
                                              ▼
                                  ╭──────────────────────╮
                                  ╱       Student         ╲
                                 (      paid: bool         )
                                  ╲    attended: bool     ╱
                                   ╰──────────────────────╯
                            그림 A-4 강좌와 세션
```

### 4.3 연관과 다수성

엔티티들을 연결하는 선을 **연관(*association - 두 클래스의 인스턴스가 메시지를 주고받을 수 있는 관계*)** 이라 부른다. 연관의 끝에 붙는 **다수성(*multiplicity - 연관에 참여하는 엔티티의 개수*)** 아이콘:

| 표기 | 의미 |
|------|------|
| `0..*` | 0에서 다수까지 |
| `*` | 0에서 다수까지 (같은 의미) |
| `1..*` | 1에서 다수까지 |
| `0..1` | 0이나 1 |
| `6` | 정확히 6 |
| `3..6` | 3에서 6 사이 |
| `3,5` | 3이나 5 |

연관은 방향을 겨냥하는 화살표가 없다면 **양방향**이다. 화살표의 존재는 지시하는 방향으로만 정보를 제한한다.

## 5. 아키텍처: CGI 웹 애플리케이션

이제 소프트웨어 설계의 문제로 내려와본다. 아키텍처는 애플리케이션의 뼈대를 이루는 소프트웨어 구조를 표현하며, 코드와 밀접하게 대응된다.

### 5.1 플랫폼 결정

| 옵션 | 설명 |
|------|------|
| 웹 기반 CGI 애플리케이션 | 웹 브라우저로 폼 접근, 웹 서버에서 CGI 스크립트 실행 |
| 데이터베이스 애플리케이션 | 관계형 DB + 폼 패키지 + 4GL |
| 비주얼 XXX | 비주얼 프로그래밍 언어 도구 |

이 예제에서는 **웹 기반 CGI**(*Common Gateway Interface - 웹 서버가 외부 프로그램을 실행해 동적 콘텐츠를 생성하는 표준 방식*)를 선택한다.

### 5.2 컴포넌트 다이어그램

```
       ┌────────────────┐                    ┌────────────────┐
       │  «cgi»         │ - - - - - - - - ─▶│    «html»      │
       │ Session Menu   │                    │  Session Menu  │
       │   Generator    │                    │                │
       └────────────────┘                    └────────────────┘
         그림 A-5 세션 메뉴 아키텍처
```

스테레오타입 `<<cgi>>`, `<<html>>`로 컴포넌트 종류를 명시한다. 점선 화살표는 **의존성 관계**다 — 어떤 컴포넌트가 다른 컴포넌트를 알고 있음을 표시한다. 세션 메뉴 생성기는 세션 메뉴 페이지를 생성하므로 그것을 알지만, 그 반대는 아니다.

```
                               ┌────────┐    ┌────────┐
```

### 5.4 HTML 템플릿으로 유연성 향상

CGI 프로그램이 HTML을 직접 생성하면 페이지 수정마다 프로그램을 다시 빌드해야 한다. 대신 CGI 프로그램이 **HTML 템플릿**을 읽고 특별한 기호를 생성된 코드로 대체하도록 한다.

```
       ┌──────┐                ┌──────┐
       │  WT  │                │  WT  │
       │ Menu │                │ Form │
       └──┬───┘                └──┬───┘
          ▲                       ▲
          │ (의존)                │
   ┌──────┴────┐    ┌────────┐  ┌─┴──────────┐
   │   CGI     │───▶│   W    │  │   CGI      │
   │ Menu Gen  │    │  Menu  │  │ Form Gen   │
   └───────────┘    └────────┘  └────────────┘
       그림 A-7 등록 컴포넌트 다이어그램에 HTML 템플릿 추가하기
```

> **핵심 통찰**: 명세/인스턴스 이분법 — 생성된 웹 페이지의 이름에는 **밑줄**이 쳐져 있다. UML에서 인스턴스에는 밑줄을 친다는 규정이 있기 때문이다. 밑줄이 없는 요소는 손으로 작성된 명세이고, 밑줄이 있는 요소는 명세에서 만들어 내는 어떤 과정의 산물이다.

### 5.5 HTMLTemplate 설계

```
   ┌────────────────────────────────────┐
   │          HTMLTemplate              │
   ├────────────────────────────────────┤
   │ - itsFile : string                 │
   ├────────────────────────────────────┤
   │ + Insert(name: string,             │
   │          replacement: string)      │
   │ + Generate() : string              │
   └─────────────────┬──────────────────┘
                     │
                     ◆ (합성)
                     │
   ┌─────────────────┴──────────────────┐
   │     map<string, string>            │
   └────────────────────────────────────┘
            그림 A-8 HTMLTemplate 설계
```

이 다이어그램은 **합성(*composition - 전체가 부분의 생명주기에 책임을 지는 강한 연관*)** 관계를 보여준다. 검은 마름모는 `HTMLTemplate`이 `map` 클래스의 생명주기를 책임짐을 나타낸다.

<details>
<summary>원문 C++ 코드</summary>

```cpp
// C++
HTMLTemplate myPage("mypage.html");
myPage.insert("header", "<h1> this is a header </h1>\n");
cout << myPage.Generate();
```

</details>

```typescript
// TypeScript
class HTMLTemplate {
  private readonly insertions = new Map<string, string>();

  constructor(private readonly itsFile: string) {}

  insert(name: string, replacement: string): void {
    this.insertions.set(name, replacement);
  }

  generate(): string {
    let html = readFileSync(this.itsFile, "utf-8");
    for (const [name, value] of this.insertions) {
      html = html.replaceAll(`<insert ${name}>`, value);
    }
    return html;
  }
}

const myPage = new HTMLTemplate("mypage.html");
myPage.insert("header", "<h1> this is a header </h1>\n");
console.log(myPage.generate());
```

## 6. UML 클래스 표기법과 의미 개관

UML에서 한 클래스는 3개 구획으로 나뉜 직사각형으로 그린다:

```
   ┌─────────────────────────────────┐
   │  «stereotype»                   │  ◀── 이름 구획
   │  Name                           │      (스테레오타입은 길러멧으로 둘러쌈)
   │  {boolProperty,                 │      (프로퍼티는 중괄호로 둘러쌈)
   │   valueProperty=x}              │
   ├─────────────────────────────────┤
   │  attribute: type                │  ◀── 속성 구획
   ├─────────────────────────────────┤
   │  operation()                    │  ◀── 오퍼레이션 구획
   └─────────────────────────────────┘
```

### 6.1 스테레오타입

스테레오타입은 UML 클래스의 종류를 나타내는 이름이다. 길러멧(*guillemet - 프랑스어 인용부호 «»*)으로 둘러싸인다.

| 스테레오타입 | 의미 |
|--------------|------|
| `<<implementation class>>` (기본) | 소프트웨어 클래스. 속성=멤버 변수, 오퍼레이션=멤버 함수 |
| `<<type>>` | 개념적 엔티티. 속성=논리적 정보, 오퍼레이션=책임 |
| `<<interface>>` | 완전 추상 클래스. 이름을 이탤릭체로 표기 |

### 6.2 캡슐화 지정자

속성과 오퍼레이션은 다음 접근 지정자로 꾸밀 수 있다:

| 기호 | 의미 |
|------|------|
| `+` | 공용 (Public) |
| `-` | 전용 (Private) |
| `#` | 보호 (Protected) |

속성의 자료형은 콜론으로 구분된다: `count: int`. 함수 인자도 마찬가지: `SetName(name: string)`. 반환형은 인자 목록 뒤 콜론으로 지정: `Distance(from: Point): float`.

### 6.3 연관, 집합, 합성

```
   ┌──────────┐              ┌──────────┐
   │ Employer │──────────────│ Employee │   연관 (Association)
   └──────────┘              └──────────┘   - 양방향, 메시지 전송 가능

   ┌──────────┐              ┌──────────┐
   │  Modem   │─────────────▶│  Dialer  │   방향이 있는 연관 (Navigable)
   └──────────┘              └──────────┘   - 단방향 가시성

   ┌──────────┐◇             ┌──────────┐
   │  Family  │──────────────│  Person  │   집합 (Aggregation)
   └──────────┘              └──────────┘   - '전체/부분' 관계 (암시적)

   ┌──────────┐◆             ┌────────────────┐
   │ Receiver │──────────────│ Message Buffer │   합성 (Composition)
   └──────────┘              └────────────────┘   - 전체가 부분의 생명주기 책임
```

| 관계 | 표기 | 의미 |
|------|------|------|
| 연관 | 단순 선 | 인스턴스 간 메시지 전송 가능 |
| 방향이 있는 연관 | 화살표 | 한 방향으로만 알고 있음 |
| 집합 (Aggregation) | 흰 마름모 | "전체/부분", 의미적 차이 없음 (암시적) |
| 합성 (Composition) | 검은 마름모 | "전체"가 "부분"의 생명주기 책임 |

## 7. 패키지와 데이터베이스 인터페이스 레이어

각 CGI 프로그램은 강좌·반·학생 등의 데이터에 접근해야 한다. 데이터베이스 형식의 변경이 애플리케이션에 영향을 주지 않도록 **DIL(*Database Interface Layer - 애플리케이션과 데이터베이스 사이의 추상화 레이어*)** 을 끼워 넣는다.

### 7.1 패키지 다이어그램

```
   ┌──────────────────────┐
   │   ┌─┐                │
   │   └─┘ Training       │
   │       Application    │
   └──────────┬───────────┘
              │ «import»
              ▼
   ┌──────────────────────┐
   │   ┌─┐                │
   │   └─┘ Training DIL   │
   └──────────┬───────────┘
              │ «import»
              ▼
   ┌──────────────────────┐
   │   ┌─┐                │
   │   └─┘ Training       │
   │       Database       │
   └──────────────────────┘
   그림 A-9 데이터베이스 인터페이스 레이어의 의존성 특징
```

DIL은 애플리케이션과 데이터베이스 모두에 의존한다. 애플리케이션과 데이터베이스는 서로에 대해 **아무것도 알지 못한다**. 따라서 한쪽을 바꾸지 않고도 다른 쪽을 교체할 수 있다.

> **핵심 통찰**: 이는 **의존성 역전 원칙(DIP)** 의 적용이다. 데이터베이스가 애플리케이션에 의존하는 것이 아니라, 둘 다 추상에 의존한다.

### 7.2 패키지 종류

| 스테레오타입 | 의미 |
|--------------|------|
| `<<package>>` (기본) | 형상 관리·릴리즈의 물리적 단위 (자바 패키지, JAR 등) |
| `<<subsystem>>` | 모델 구성요소 + 동작까지 지정하는 논리적 단위 |

패키지 간 관계는 두 가지로 표현된다:

```
       Imported                  Abstract
          ▲                          △
          │ «import»                 │
          │                          │
       Importing                Implementing
```

- **임포트 의존관계** (점선 화살표): 임포트하는 패키지가 임포트되는 패키지의 모든 공용 요소에 접근 가능
- **일반화** (빈 삼각형): 추상 패키지의 모든 공용/보호 요소에 가시성을 가짐

### 7.3 인터페이스와 실현 관계

```
                  ┌───────────────────────────┐
                  │  «interface»              │
                  │   Session Schedule        │   ◀── 이탤릭체 = 추상
                  ├───────────────────────────┤
                  │ + GetSessionIterator():   │
                  │     Iterator<Session>     │
                  └─────────────▲─────────────┘
                                △ (점선 + 빈 삼각형 = realize)
                                │
                  ┌─────────────┴─────────────┐
                  │  DIL::SessionSchedule     │
                  └───────────────────────────┘
   그림 A-10 교육 과정 애플리케이션 패키지의 데이터베이스 인터페이스
```

| 표기 | 의미 |
|------|------|
| 이탤릭체 클래스명/오퍼레이션 | 추상 (abstract) |
| 점선 + 빈 삼각형 화살표 | 실현 관계 (realize) — 자바 `implements`, C++ 상속에 대응 |
| 이중 콜론 `DIL::Session` | 패키지 내 클래스 (네임스페이스) |

## 8. 시퀀스 다이어그램

`SessionMenuGenerator`는 세션 스케줄의 HTML 표현을 만들어내야 한다. 이를 묘사하는 동적 모델이 **시퀀스 다이어그램**이다.

### 8.1 다이어그램 구조

```
   SessionMenu       :SessionSchedule    :HTMLTemplate
    Generator
       │                    │                   │
       │  «create»          │                   │
       ├───────────────────────────────────────▶│ ●
       │                    │                   │ │
       │  GetSessionIterator│                   │ │
       ├───────────────────▶│                   │ │
       │◀ ── iterator<Sess>─┤                   │ │
       │                    │                   │ │
       │ ┌─────────────────────────────────────┐│ │ 활동
       │ │ loop [각 Session]                   ││ │
       │ │  GetNext  ▶                         ││ │
       │ │  GetCourse▶                         ││ │
       │ │  GetTime, GetLocation               ││ │
       │ │  insert("Schedule", lineItem) ─────▶││ │
       │ └─────────────────────────────────────┘│ │
       │                                        │ │
       │  Generate                              │ │
       ├───────────────────────────────────────▶│ │
       │◀──────────────────────────────────────┤  │
       │                                        ✕ (소멸)
       │  Destroy                                  
       ✕                                          
       그림 A-11 SessionMenuGenerator 시퀀스 다이어그램
```

### 8.2 시퀀스 다이어그램 표기법

| 요소 | 의미 |
|------|------|
| 직사각형 (밑줄 친 이름) | 객체 (클래스가 아님). 형식: `이름:클래스` 또는 `:클래스` |
| 점선 (객체 아래) | 수명선 (lifeline) — 객체의 생명주기 |
| 화살표 | 메시지. 시간은 위에서 아래로 진행 |
| 끝에 원이 붙은 짧은 화살표 | 데이터 토큰 — 매개변수 또는 반환값 |
| 굵은 직사각형 (수명선 위) | 활동 (activation) — 메소드 실행 기간 |
| 수명선 끝 `✕` | 객체 소멸 |
| 굵은 선 직사각형 박스 | 루프 (완료 기준은 박스 아래) |

> **핵심 통찰**: 객체 이름은 콜론으로 구분된 두 요소로 이루어진다. 콜론 앞은 객체의 단순 이름, 뒤는 **클래스 또는 객체가 구현하는 인터페이스 이름**이다. 시퀀스 다이어그램에서 객체가 인터페이스에서 인스턴스로 만들어진 것처럼 보일 수 있는데, 이는 그 객체의 클래스가 해당 인터페이스를 구현하거나 거기서 파생되었음을 의미할 뿐이다.

<details>
<summary>원문 C++ 코드 (개념적 흐름)</summary>

```cpp
// C++
void SessionMenuGenerator::Run() {
    HTMLTemplate* myPage = new HTMLTemplate("sessionMenu.html");
    Iterator<Session>* it = sessionSchedule.GetSessionIterator();
    while (it->HasNext()) {
        Session* s = it->GetNext();
        Course* c = s->GetCourse();
        string lineItem = formatLine(
            c->GetName(), c->GetCost(),
            s->GetTime(), s->GetLocation()
        );
        myPage->insert("Schedule", lineItem);
    }
    cout << myPage->Generate();
    delete myPage;
    delete it;
}
```

</details>

```typescript
// TypeScript
class SessionMenuGenerator implements CGIProgram {

  }
}
```

## 9. 정적 모델과 스테레오타입 연관

`SessionMenuGenerator`의 동적 모델(시퀀스 다이어그램)이 정적 모델을 내포한다.

### 9.1 정적 모델

```
                                          «instantiates»
   ┌──────────────────┐ ─────────────────────────────────▶ ┌──────────────┐
   │ SessionMenu      │                                     │ HTMLTemplate │
   │ Generator        │                                     └──────────────┘
   └─────────┬────────┘
             │ «parameter»                       
             ▼                                   
   ┌──────────────────┐    ┌──────────────────┐  ┌────────────┐
   │ SessionSchedule  │    │ Iterator<Session>│  │  Session   │
   │  {singleton}     │    └──────────────────┘  └────────────┘
   └──────────────────┘
                                                 ┌────────────┐
                                                 │  Course    │
                                                 └────────────┘
            그림 A-12 세션 메뉴 생성기 애플리케이션의 정적 모델
```

### 9.2 주요 스테레오타입

| 스테레오타입 | 의미 |
|--------------|------|
| `{singleton}` 프로퍼티 | 애플리케이션 내 단 하나의 인스턴스만 존재 (싱글톤 패턴) |
| `<<instantiates>>` | 한 클래스가 다른 클래스의 인스턴스를 생성함 |
| `<<create>>` | 인스턴스 생성 관계 |
| `<<parameter>>` | 메소드 인자나 반환값으로 알게 됨 (인스턴스 변수 아님) |
| `<<function>>` | 자유로운 전역 함수 |

> **핵심 통찰**: 연관에 화살표만 그리지 않고 **스테레오타입**으로 수식하면 "이 클래스들이 어떻게 서로를 알게 되는지"의 의도까지 표현할 수 있다. `<<parameter>>` 연관은 객체가 메소드 인자/반환값으로만 일시적으로 알게 되며, **인스턴스 변수를 통한 영속 관계가 아님**을 나타낸다.

## 10. CGIProgram 인터페이스와 링킹 시 다형성

`main()`은 어떻게 적절한 `CGIProgram`의 파생 객체를 만들어낼까? `CGIProgram`의 파생 클래스가 많기 때문에 여러 `main()` 함수를 만드는 것은 비효율적이다.

### 10.1 링킹 시 다형성 해법

`main()`을 `cgiProgram.cc`에 한 번만 구현하고, 그 안에서 `CreateCGI`라는 **전역 함수**를 선언만 한다(구현하지 않음). 각 파생 클래스 구현 파일(예: `sessionMenuGenerator.cc`)이 `CreateCGI`를 구현한다.

```
   ┌─────────────────────────────────┐    ┌──────────────────────────┐
   │       CGIProgram                │    │  «function»              │
   ├─────────────────────────────────┤    │   main()                 │
   │ # GetValue(fieldName: string)   │    │  {file=cgiProgram.cc}    │
   │     : string                    │    └──────────────────────────┘
   │ + CGIProgram(data: stream)      │
   │ + Run()                         │
   └─────────────────┬───────────────┘
                     △
                     │
   ┌─────────────────┴───────────────┐
   │   SessionMenuGenerator          │  «instantiates»  ┌──────────────────────────┐
   └─────────────────────────────────┘─────────────────▶│ «function»               │
                                                        │  CreateCGI(stream)       │
                                                        │ {file=sessionMenuGen.cc} │
                                                        └──────────────────────────┘
       그림 A-13 CGI 프로그램의 아키텍처
```

### 10.2 프로퍼티

`{file=sessionMenuGenerator.cc}` 같은 프로퍼티는 함수가 구현된 **파일을 지정**한다. 프로퍼티는 중괄호로 둘러싸이고 콤마로 구분된 `이름=값` 쌍 목록이다. 등호가 생략되면 불리언 타입으로 간주되어 `true` 값을 받는다.

```typescript
// TypeScript — CGIProgram 추상 클래스
abstract class CGIProgram {
  constructor(protected readonly data: NodeJS.ReadableStream) {}

  protected getValue(fieldName: string): string {
    // 표준 입력에서 폼 필드 값 파싱
    return parseField(this.data, fieldName);
  }

  abstract run(): void;
}

// 파생 클래스 — sessionMenuGenerator.ts
class SessionMenuGenerator extends CGIProgram {
  run(): void {
    const course = this.getValue("course");
    // ...
  }
}

// 공통 main — cgiProgram.ts
declare function createCGI(stream: NodeJS.ReadableStream): CGIProgram;

function main(): void {
  const program = createCGI(process.stdin);
  program.run();
}
```

## 요약

- **UML은 단일 표기법이지만 분석과 설계 모두에 적용 가능**하다. 어떤 부분은 분석용, 어떤 부분은 설계용이다.
- **유스케이스 다이어그램**은 액터와 시스템의 대화를 명세한다. `<<include>>`(중복 제거)와 `<<extend>>`(변경 관리)로 유스케이스를 구조화한다.
- **도메인 모델**은 `<<type>>` 스테레오타입(구름 아이콘)을 사용해 **개념적 엔티티**를 표현한다. 소프트웨어 설계와 직접 대응되지 않는다.
- **클래스 다이어그램**은 속성·오퍼레이션 구획, 캡슐화 지정자(`+/-/#`), 스테레오타입과 프로퍼티로 구성된다.
- **연관 관계**: 단순 연관 → 방향이 있는 연관 → 집합(흰 마름모) → 합성(검은 마름모). 합성은 생명주기 책임까지 포함한다.
- **컴포넌트 다이어그램**은 물리적 소프트웨어 컴포넌트(CGI, HTML, HTML 템플릿)와 의존성을 보여준다. 인스턴스(런타임 산물)는 **밑줄**로 표시한다.
- **패키지 다이어그램**은 임포트 의존관계와 일반화 관계로 패키지를 묶는다. `<<package>>`(릴리즈 단위)와 `<<subsystem>>`(논리적 동작 단위)이 구분된다.
- **인터페이스**는 이탤릭체 이름과 점선+빈 삼각형(실현 관계)으로 표기한다. 이중 콜론(`DIL::Session`)은 패키지 네임스페이스다.
- **시퀀스 다이어그램**은 객체(밑줄), 수명선(점선), 메시지(화살표), 활동(굵은 직사각형), 루프 박스로 객체 간 메시지 흐름을 시간 순으로 보여준다.
- **정적 모델은 동적 모델에 의해 내포된다.** 시퀀스 다이어그램이 정적 모델을 드러내고, `<<parameter>>`/`<<create>>`/`<<instantiates>>` 스테레오타입으로 관계의 성격을 명시한다.
- **DIL 패턴**은 의존성 역전 원칙(DIP)의 실제 적용이다. 애플리케이션과 데이터베이스가 모두 추상에 의존하므로 서로 독립적으로 교체될 수 있다.
- **마틴의 문서화 제1법칙**: "급박하고 중요하지 않다면 아무 문서도 만들지 마라." UML 다이어그램은 의사소통의 도구이지 필수 산출물이 아니다.

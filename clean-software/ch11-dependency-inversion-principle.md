# Chapter 11: The Dependency-Inversion Principle (DIP / 의존 관계 역전 원칙)

## 핵심 질문

왜 "역전(*inversion - 차례나 위치 따위를 서로 뒤바꿈*)"이라는 단어를 썼는가? 전통적인 절차적 설계에서 상위 모듈이 하위 모듈에 의존하는 것이 왜 문제인가? 어떻게 추상화에 의존하게 만들어 — 정책이 세부 사항으로부터 독립되도록 — 의존성의 방향을 뒤집을 수 있는가? 그리고 객체지향 설계의 본질은 결국 이 의존성 역전인가?

> 더는 국가의 중요한 일들이 인간의 나약함을 흔들지도 모르는 무수한 가능성에 휘둘리지 않게 해야 한다.<br>— 토머스 눈 탈파우드(Thomas Noon Talfourd) 경, 1795~1854

---

## 1. DIP의 정의

> **의존 관계 역전 원칙 (Dependency-Inversion Principle)**<br>**a. 상위 수준의 모듈은 하위 수준의 모듈에 의존해서는 안 된다. 둘 모두 추상화에 의존해야 한다.**<br>**b. 추상화는 구체적인 사항에 의존해서는 안 된다. 구체적인 사항은 추상화에 의존해야 한다.**

### 1.1 왜 "역전"인가?

수년 동안 많은 사람들이 왜 이 원칙의 이름에 "역전"이란 단어를 사용했는지 질문해 왔다. 그 이유는, 구조적 분석 설계(*structured analysis and design - 절차적·하향식 설계 방법론*) 같은 좀 더 전통적인 소프트웨어 개발 방법에서는 다음과 같은 경향이 있었기 때문이다:

- 소프트웨어 구조에서 **상위 수준의 모듈이 하위 수준의 모듈에 의존**한다
- **정책이 구체적인 것에 의존**한다
- 상위 수준 모듈이 하위 수준 모듈을 호출하는 방법을 묘사하는 서브프로그램(*subprogram - 절차/함수 단위의 호출 가능한 코드 블록*)의 계층 구조를 정의한다

> **핵심 통찰**: 잘 설계된 객체지향 프로그램의 의존성 구조는 **전통적인 절차적 방법에 의해 일반적으로 만들어진 의존성 구조가 역전된 것**이다.

### 1.2 상위 수준이 하위 수준에 의존하면 무엇이 잘못되는가?

어떤 애플리케이션의 **중요한 정책 의사결정과 업무 모델을 포함하고 있는 것은 상위 수준의 모듈**이며, 이 모듈은 애플리케이션의 본질을 담고 있다. 그런데 이런 모듈이 하위 수준 모듈에 의존할 때:

- 하위 수준 모듈의 변경이 상위 수준 모듈에 **직접적인 영향**을 미친다
- 결과적으로 상위 수준의 모듈이 변경되게 된다

> **핵심 통찰**: 이런 상황은 말도 안 된다. 하위 수준의 구체적인 모듈에 영향을 주어야 하는 것은 **정책을 결정하는 상위 수준의 모듈**이다. 업무 규칙을 포함하는 상위 수준의 모듈은 구체적인 구현을 포함한 모듈에 우선하면서 동시에 독립적이어야 한다.

### 1.3 재사용성과 DIP

우리가 재사용하고 싶어 하는 것은 **정책을 결정하는 상위 수준의 모듈**이다. 그러나:

| 의존 방향 | 결과 |
|-----------|------|
| 상위 → 하위 (전통적) | 상위 모듈을 다른 문맥에서 재사용하기 매우 어려움 |
| 둘 다 → 추상화 (DIP) | 상위 모듈이 하위 모듈과 독립적이어서 간단히 재사용 가능 |

이 원칙은 **프레임워크 설계의 핵심**에 있다.

## 2. 레이어 나누기

### 2.1 미숙한 레이어 나누기

부치(Booch)는 "잘 구조화된 모든 객체지향 아키텍처는 레이어를 분명하게 정의했다. 여기서 각 레이어는 잘 정의되고 제어되는 인터페이스를 통해 일관된 서비스의 집합을 제공한다."고 말했다. 이 말을 그대로 고지식하게 해석한 설계자는 다음과 같은 구조를 만들게 된다:

```
┌──────────────────────┐
│   Policy Layer       │   ◀── 상위 수준 정책
└──────────┬───────────┘
           │ 의존
           ▼
┌──────────────────────┐
│  Mechanism Layer     │   ◀── 중간 메커니즘
└──────────┬───────────┘
           │ 의존
           ▼
┌──────────────────────┐
│   Utility Layer      │   ◀── 하위 유틸리티
└──────────────────────┘
```

**그림 11-1: 미숙한 레이어 나누기 계획**

얼핏 보기에는 괜찮아 보이지만, `Policy` 레이어가 아래 `Utility` 레이어의 모든 변화에 민감하다는 특성이 함정으로 숨어 있다. 의존성은 **이행적(*transitive - A→B, B→C 이면 A→C 가 성립함*)**이다. 따라서 `Policy` 레이어는 `Utility` 레이어에 이행적으로 의존하게 된다.

### 2.2 역전된 레이어

좀 더 적절한 모델은 각 상위 수준 레이어가 **그것이 필요로 하는 서비스에 대한 추상 인터페이스를 선언**하고, 하위 수준의 레이어가 이 추상 인터페이스로부터 실체화되는 것이다.

```
┌─────────────────────────────────────────┐
│            Policy Layer                 │
│                                         │
│   ┌──────────────────────────────────┐  │
│   │ «interface»                      │  │
│   │ PolicyServiceInterface           │  │
│   └────────────▲─────────────────────┘  │
└────────────────┼────────────────────────┘
                 │ 구현
┌────────────────┼────────────────────────┐
│         Mechanism Layer                 │
│                │                        │
│   ┌────────────┴─────────────────────┐  │
│   │           Mechanism              │  │
│   └──────────────────────────────────┘  │
│                                         │
│   ┌──────────────────────────────────┐  │
│   │ «interface»                      │  │
│   │ MechanismServiceInterface        │  │
│   └────────────▲─────────────────────┘  │
└────────────────┼────────────────────────┘
                 │ 구현
┌────────────────┼────────────────────────┐
│           Utility Layer                 │
│                │                        │
│   ┌────────────┴─────────────────────┐  │
│   │            Utility               │  │
│   └──────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**그림 11-2: 역전된 레이어**

각 상위 수준 클래스는 추상 인터페이스를 통해 다음 하위 수준의 레이어를 사용한다. 따라서:

- 상위 레이어는 **하위 레이어에 의존하지 않는다**
- 반대로, 하위 레이어가 **상위 레이어에 선언된 추상 서비스 인터페이스에 의존**한다
- `Policy Layer`의 `Utility Layer`에 대한 이행적 의존성만 끊어지는 것이 아니라, `Mechanism Layer`에 대한 직접적 의존성도 끊어진다

### 2.3 소유권의 역전

여기서의 역전은 의존성에 대해서만이 아니라, **인터페이스 소유권**에 대한 것도 의미한다.

| 관점 | 일반적 사고 | DIP 적용 |
|------|-------------|----------|
| 인터페이스 소유자 | 유틸리티 라이브러리가 자신의 인터페이스를 소유 | **클라이언트가 추상 인터페이스를 소유** |
| 서버 위치 | 라이브러리가 클라이언트보다 먼저 정해짐 | 서버가 클라이언트의 인터페이스에서 파생 |

> **핵심 통찰**: 이것은 "연락드리겠습니다."라는 **할리우드(Hollywood) 원칙**으로도 알려져 있다. 하위 수준의 모듈은 상위 수준의 모듈 안에 선언되어 호출되는 인터페이스의 구현을 제공한다.

이런 소유권의 역전을 사용하면:

- `Policy Layer`는 `Mechanism Layer`나 `Utility Layer`의 어떤 변경에도 **영향을 받지 않는다**
- `Policy Layer`는 `PolicyServiceInterface`에 맞는 하위 수준 모듈을 정의하는 어떤 문맥에서든 **재사용될 수 있다**
- 결과적으로 좀 더 유연하고, 튼튼하고, 이동이 쉬운 구조가 만들어진다

## 3. 추상화에 의존하자

다소 고지식하지만, 그래도 상당히 강력한 DIP의 해석은 "추상화에 의존하자"라는 간단한 **휴리스틱(*heuristic - 정확하진 않지만 빠른 의사결정을 돕는 경험적 접근법*)**이다.

### 3.1 휴리스틱의 세 가지 규칙

- 어떤 변수도 **구체 클래스(*concrete class - 인스턴스를 만들 수 있는 클래스*)**에 대한 포인터나 참조 값을 가져선 안 된다
- 어떤 클래스도 구체 클래스에서 파생되어서는 안 된다
- 어떤 메소드도 그 기반 클래스에서 구현된 메소드를 오버라이드해서는 안 된다

### 3.2 이 휴리스틱의 예외

물론 이 휴리스틱은 모든 프로그램에서 종종 한 번 이상 위반된다. 어느 것인가는 구체 클래스의 인스턴스를 생성해야 하고, 그런 일을 하는 모듈은 이 클래스에 의존해야 한다.

게다가, 구체적이긴 하지만 **비휘발적(*non-volatile - 자주 변경되지 않는*)인 클래스**에는 이 휴리스틱을 적용할 이유가 없어 보인다.

| 클래스 종류 | 휴리스틱 적용 여부 | 예시 |
|-------------|-------------------|------|
| 비휘발적 구체 클래스 | 적용 불필요 | Java의 `String` 클래스 |
| 휘발적 구체 클래스 | **반드시 추상화 뒤에 숨길 것** | 애플리케이션 코드의 대부분 |

대부분의 시스템에서 어떤 문자열을 묘사하는 클래스는 구체적이다. Java의 `String`은 휘발적이지 않다. 그러므로 이것에 직접 의존하는 것은 전혀 해가 되지 않는다.

그러나 우리가 애플리케이션 프로그램의 일부로 작성하는 대부분의 구체 클래스는 휘발적이다. 우리가 직접적으로 의존하지 않기를 원하는 구체 클래스가 바로 이것들이다.

> **핵심 통찰**: 이 휴리스틱이 다소 고지식한 이유는, 휘발적인 클래스의 인터페이스가 변경되면 분명히 그 변경이 추상 인터페이스로 전파되기 때문이다. 그러나 **클라이언트 클래스가 자신이 필요로 하는 서비스 인터페이스를 선언한다**는 장기적 관점을 택한다면, 이 인터페이스가 변경되는 경우는 오직 클라이언트가 변경을 필요로 할 때가 된다.

## 4. 간단한 예 — Button과 Lamp

### 4.1 미숙한 모델

`Button` 객체는 외부 환경을 감지한다. `Poll` 메시지를 받으면 사용자가 그것을 눌렀는지 판단한다. 지각 메커니즘이 무엇이든 상관없다 — GUI 상의 버튼 아이콘, 물리적 버튼, 주택 보안 시스템의 움직임 탐지기일 수도 있다.

`Lamp` 객체는 외부 환경에 영향을 미친다. `TurnOn` 메시지를 받으면 어떤 종류의 조명을 밝히고, `TurnOff` 메시지를 받으면 그 조명을 끈다. 물리적 메커니즘은 중요하지 않다.

```
┌──────────────┐         ┌──────────────┐
│    Button    │────────▶│     Lamp     │
├──────────────┤         ├──────────────┤
│ + poll()     │         │ + turnOn()   │
│              │         │ + turnOff()  │
└──────────────┘         └──────────────┘
```

**그림 11-3: Button과 Lamp의 미숙한 모델**

<details>
<summary>원문 Java 코드 (목록 11-1: Button.java)</summary>

```java
// Java - DIP 위반
public class Button {
    private Lamp itsLamp;

    public void poll() {
        if (/* 어떤 조건치 */) {
            itsLamp.turnOn();
        }
    }
}
```

</details>

```typescript
// TypeScript - DIP 위반
class Lamp {
    turnOn(): void { /* ... */ }
    turnOff(): void { /* ... */ }
}

class Button {
    private itsLamp: Lamp;

    constructor(lamp: Lamp) {
        this.itsLamp = lamp;
    }

    poll(): void {
        if (/* 어떤 조건치 */ true) {
            this.itsLamp.turnOn();
        }
    }
}
```

### 4.2 왜 이것이 미숙한가?

`Button` 클래스가 `Lamp` 클래스에 직접 의존하고 있다는 점에 주목하자. 이런 의존성은:

- `Button`이 `Lamp`에 대한 변경에 **영향을 받을 것임을 의미**한다
- `Button`이 `Motor` 객체를 제어할 수 있게 **재사용하는 것도 불가능**하다
- 이 설계에서 `Button` 객체는 `Lamp` 객체를, 그리고 **오직 `Lamp` 객체만을** 제어한다

이 해결책은 DIP를 위반한다:

- 애플리케이션의 **상위 수준 정책**(버튼이 눌리면 무언가를 켜고 끈다)이 **하위 수준 구현**(`Lamp`)에서 분리되어 있지 않다
- 추상화가 구체적인 것에서 분리되어 있지 않다

### 4.3 내재하는 추상화를 찾아서

> **핵심 통찰**: 무엇이 상위 수준의 정책인가? 그것은 **애플리케이션에 내재하는 추상화**이자, 구체적인 것이 변경되더라도 바뀌지 않는 진실이다. 시스템 안의 시스템이며, 메타포(*metaphor - 비유; 코드의 본질을 표현하는 개념적 모델*)다.

`Button`/`Lamp` 예에서 내재하는 추상화는 **"사용자로부터 켜고 끄는 동작을 탐지해 그 동작을 대상 객체에 전해주는 것"**이다.

- 사용자의 동작을 탐지하기 위해 어떤 메커니즘이 사용되는가? **관계없다**
- 대상 객체는 무엇인가? **관계없다**
- 이것은 추상화에 영향을 주지 않는 구체적 내용들이다

### 4.4 의존성 역전 적용

```
┌──────────────┐         ┌──────────────────────┐
│    Button    │────────▶│    «interface»       │
├──────────────┤         │    ButtonServer      │
│ + poll()     │         ├──────────────────────┤
│              │         │ + turnOn()           │
│              │         │ + turnOff()          │
└──────────────┘         └──────────▲───────────┘
                                    │ 구현
                         ┌──────────┴───────────┐
                         │        Lamp          │
                         └──────────────────────┘
```

**그림 11-4: Lamp에 적용한 의존성 역전**

`Button`은 `ButtonServer`라는 것과 어떤 관계를 가진다. `ButtonServer`는 `Button`이 어떤 것을 켜거나 끄기 위해 사용할 수 있는 추상 메소드를 제공하고, `Lamp`는 `ButtonServer` 인터페이스를 구현한다. 따라서 **`Lamp`는 이제 의존을 당하는 게 아니라, 반대로 의존하게 된다.**

<details>
<summary>원문 Java 코드</summary>

```java
// Java - DIP 적용
public interface ButtonServer {
    void turnOn();
    void turnOff();
}

public class Button {
    private ButtonServer itsServer;

    public Button(ButtonServer server) {
        this.itsServer = server;
    }

    public void poll() {
        if (/* 어떤 조건치 */) {
            itsServer.turnOn();
        }
    }
}

public class Lamp implements ButtonServer {
    public void turnOn() { /* ... */ }
    public void turnOff() { /* ... */ }
}
```

</details>

```typescript
// TypeScript - DIP 적용
interface ButtonServer {
    turnOn(): void;
    turnOff(): void;
}

class Button {
    private itsServer: ButtonServer;

    constructor(server: ButtonServer) {
        this.itsServer = server;
    }

    poll(): void {
        if (/* 어떤 조건치 */ true) {
            this.itsServer.turnOn();
        }
    }
}

class Lamp implements ButtonServer {
    turnOn(): void { /* ... */ }
    turnOff(): void { /* ... */ }
}
```

이제 `Button`은 `ButtonServer` 인터페이스를 구현하려는 **어떤 장치든 제어**할 수 있다. 이것은 상당한 유연성을 가져다줄 뿐만 아니라, `Button` 객체가 아직 만들어지지 않은 객체조차 제어할 수 있음을 의미한다.

### 4.5 한 번 더 정제하기 — 이름 분리

그러나 이 해결책은 `Button`에 의해 제어되길 원하는 객체에 제약을 걸게 된다. 이런 객체는 `ButtonServer` 인터페이스를 구현해야 하는데, `Lamp` 같은 객체는 `Button`이 아닌 `Switch`나 다른 객체에 의해 제어되기를 원할 수도 있기 때문에 아주 안타까운 일이다.

> **핵심 통찰**: 의존성의 방향을 역전시키고 `Lamp`가 의존을 당하는 대신 의존하게 만듦으로써, 우리는 `Lamp`가 다른 구체적인 `Button`에 의존하게 만들었다. 그렇다면 다른 것에는 어떨까?

해결책:

- `ButtonServer`의 이름을 좀 더 일반적인 **`SwitchableDevice`** 같은 것으로 변경
- `Button`과 `SwitchableDevice`가 개별적인 라이브러리에 존재함을 확실히 함
- `SwitchableDevice`를 사용하는 것이 곧 `Button`을 사용하는 것임을 의미하지 않게 만듦

따라서 의존성은 **이름에만 존재**한다.

## 5. 용광로 사례

좀 더 흥미로운 사례를 보자. 어떤 용광로의 조절기를 제어하는 소프트웨어를 생각해 보자. 이 소프트웨어는 IO 채널에서 현재 온도를 읽고 다른 IO 채널에 명령어를 전송하여 용광로를 켜거나 끈다.

### 5.1 절차적 구현 — 구체적인 것에 직접 의존

<details>
<summary>원문 C 코드 (목록 11-2: 자동온도조절기의 간단한 알고리즘)</summary>

```c
// C - 절차적, DIP 위반
#define THERMOMETER 0x86
#define FURNACE     0x87
#define ENGAGE      1
#define DISENGAGE   0

void Regulate(double minTemp, double maxTemp) {
    for (;;) {
        while (in(THERMOMETER) > minTemp) {
            wait(1);
        }
        out(FURNACE, ENGAGE);
        while (in(THERMOMETER) < maxTemp) {
            wait(1);
        }
        out(FURNACE, DISENGAGE);
    }
}
```

</details>

```typescript
// TypeScript - 절차적, DIP 위반
const THERMOMETER = 0x86;
const FURNACE = 0x87;
const ENGAGE = 1;
const DISENGAGE = 0;

declare function inPort(port: number): number;
declare function outPort(port: number, value: number): void;
declare function wait(ms: number): void;

function regulate(minTemp: number, maxTemp: number): void {
    for (;;) {
        while (inPort(THERMOMETER) > minTemp) {
            wait(1);
        }
        outPort(FURNACE, ENGAGE);
        while (inPort(THERMOMETER) < maxTemp) {
            wait(1);
        }
        outPort(FURNACE, DISENGAGE);
    }
}
```

이 알고리즘의 상위 수준 목적은 분명하지만, 이 코드는 많은 하위 수준의 구체적인 내용으로 어지럽혀져 있다. 이 코드는 **다른 제어 하드웨어에서는 절대 재사용할 수 없다.**

### 5.2 일반적인 조절기 — 의존성 역전

```
                ┌─────────────────────┐
                │   «function»        │
                │   Regulate          │
                └──┬───────────────┬──┘
                   │ «parameter»   │ «parameter»
                   ▼               ▼
       ┌────────────────────┐   ┌────────────────────┐
       │  «interface»       │   │  «interface»       │
       │  Thermometer       │   │  Heater            │
       ├────────────────────┤   ├────────────────────┤
       │ + read()           │   │ + engage()         │
       │                    │   │ + disengage()      │
       └─────────▲──────────┘   └─────────▲──────────┘
                 │ 구현                   │ 구현
       ┌─────────┴──────────┐   ┌─────────┴──────────┐
       │  IO Channel        │   │  IO Channel        │
       │  Thermometer       │   │  Heater            │
       └────────────────────┘   └────────────────────┘
```

**그림 11-5: 일반적인 조절기**

여기서는 조절 함수가 2개의 인자를 받는데, 둘 다 인터페이스다.

- `Thermometer` 인터페이스는 읽힐 수 있다
- `Heater` 인터페이스는 동작하게 되거나 멈추게 될 수 있다
- `Regulate` 알고리즘이 필요로 하는 것은 이게 전부다

<details>
<summary>원문 C++ 코드 (목록 11-3: 일반적인 조절기)</summary>

```cpp
// C++ - 동적 다형성
class Thermometer {
public:
    virtual double Read() = 0;
};

class Heater {
public:
    virtual void Engage() = 0;
    virtual void Disengage() = 0;
};

void Regulate(Thermometer& t, Heater& h, double minTemp, double maxTemp) {
    for (;;) {
        while (t.Read() > minTemp) {
            wait(1);
        }
        h.Engage();
        while (t.Read() < maxTemp) {
            wait(1);
        }
        h.Disengage();
    }
}
```

</details>

```typescript
// TypeScript - 동적 다형성
interface Thermometer {
    read(): number;
}

interface Heater {
    engage(): void;
    disengage(): void;
}

declare function wait(ms: number): void;

function regulate(t: Thermometer, h: Heater, minTemp: number, maxTemp: number): void {
    for (;;) {
        while (t.read() > minTemp) {
            wait(1);
        }
        h.engage();
        while (t.read() < maxTemp) {
            wait(1);
        }
        h.disengage();
    }
}
```

여기서는 상위 수준의 조절 정책이 자동 온도 조절기나 용광로의 구체적인 사항에 의존하지 않도록 의존성을 역전시켰다. **이 알고리즘은 제대로 재사용할 수 있다.**

### 5.3 인터페이스의 소유자

이 경우, **아무도 인터페이스를 소유하지 않는다.** 이 인터페이스가 다른 여러 개의 클라이언트에 의해 사용될 수 있고, 여러 개의 서버에 의해 구현될 수 있는 흥미로운 상황이다.

> **핵심 통찰**: 이 인터페이스는 어떤 그룹에도 속하지 않은 채로 혼자 운영될 필요가 있다. C++에서는 이것을 개별적인 네임스페이스(*namespace - 이름 충돌을 피하기 위해 코드를 묶어두는 단위*)와 라이브러리에 넣을 테고, Java에서는 개별적인 패키지에 넣을 것이다.

스몰토크(Smalltalk), 파이썬(Python), 루비(Ruby) 같은 동적 언어에서는 인터페이스가 명시적인 소스 코드처럼 존재하지는 않는다.

## 6. 동적 다형성과 정적 다형성

우리는 **동적 다형성(*dynamic polymorphism - 런타임에 메소드 디스패치가 결정되는 다형성, 즉 추상 클래스나 인터페이스 기반*)**을 이용해서 의존성의 역전을 해결했고, `Regulate`를 일반적인 것으로 만들었다. 그러나 다른 방법도 있다 — C++의 템플릿(*template - 컴파일 타임에 타입을 매개변수로 받아 코드를 생성하는 기능*)이 제공하는 **정적 다형성(static polymorphism)**을 사용할 수도 있다.

<details>
<summary>원문 C++ 코드 (목록 11-4: 템플릿 기반)</summary>

```cpp
// C++ - 정적 다형성 (템플릿)
template <typename THERMOMETER, typename HEATER>
void Regulate(THERMOMETER& t, HEATER& h, double minTemp, double maxTemp) {
    for (;;) {
        while (t.Read() > minTemp) {
            wait(1);
        }
        h.Engage();
        while (t.Read() < maxTemp) {
            wait(1);
        }
        h.Disengage();
    }
}
```

</details>

```typescript
// TypeScript - 정적 다형성 (제네릭 + 구조적 타입)
interface HasRead {
    read(): number;
}

interface HasEngageDisengage {
    engage(): void;
    disengage(): void;
}

declare function wait(ms: number): void;

function regulate<T extends HasRead, H extends HasEngageDisengage>(
    t: T,
    h: H,
    minTemp: number,
    maxTemp: number,
): void {
    for (;;) {
        while (t.read() > minTemp) {
            wait(1);
        }
        h.engage();
        while (t.read() < maxTemp) {
            wait(1);
        }
        h.disengage();
    }
}
```

이렇게 하면 동적 다형성의 부하(또는 유연성) 없이도 똑같은 의존성 역전을 이룰 수 있다. C++에서 `Read`, `Engage`, `Disengage` 메소드는 모두 비가상(*non-virtual - 런타임 디스패치를 사용하지 않는 일반 메소드*) 메소드일 수도 있다. 게다가, 이 메소드를 선언하는 어떤 클래스라도 템플릿에서 사용될 수 있다. 이들은 같은 기반 클래스를 상속할 필요가 없다.

### 6.1 두 방식의 트레이드오프

| 비교 | 동적 다형성 | 정적 다형성 |
|------|-------------|-------------|
| 디스패치 시점 | 런타임 | 컴파일 타임 |
| 형 변경 | 런타임에 가능 | **런타임에 바뀔 수 없음** |
| 새 종류 추가 | 재컴파일 불필요 (DI/플러그인) | **재컴파일과 재배포 필요** |
| 성능 | 가상 호출 부하 있음 | 부하 없음, 인라이닝 가능 |
| 인터페이스 강제 | 명시적 인터페이스/상속 필요 | 구조적 매칭 (덕 타이핑) |

> **핵심 통찰**: 정적 다형성은 소스 코드의 의존성을 깔끔하게 끊어주지만, 동적 다형성만큼 많은 문제를 해결해 주지는 않는다. 따라서 **속도가 아주 절실히 필요한 게 아니라면, 동적 다형성이 더 나은 선택**일 것이다.

## 7. 자주 하는 실수

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| **상위 모듈이 하위 모듈을 직접 `new`로 생성** | 컴파일 시점부터 구체 클래스에 결박됨 | 생성자/팩토리 주입으로 추상 타입을 받음 |
| **추상화가 구체적인 것에 의존** | 인터페이스가 특정 구현의 타입(파일 핸들, DB 커넥션 등)을 노출 | 인터페이스에서 구현 세부 타입을 제거하고 도메인 언어로 표현 |
| **하위 모듈이 자신의 인터페이스를 소유** | 클라이언트가 라이브러리에 종속, 라이브러리 교체 어려움 | **클라이언트가 필요한 인터페이스를 선언** (할리우드 원칙) |
| **휘발성 구체 클래스에 직접 의존** | 구체 클래스 변경이 클라이언트로 전파 | 추상 클래스/인터페이스 뒤에 숨김 |
| **비휘발성 클래스까지 모두 추상화** | 불필요한 추상화 폭증, `String` 같은 안정 클래스를 인터페이스로 감쌈 | 자주 바뀌지 않는 구체 클래스는 직접 의존 허용 |
| **인터페이스 이름이 특정 클라이언트에 종속** | `ButtonServer`처럼 한 클라이언트에 종속된 이름이 다른 사용처를 막음 | `SwitchableDevice`처럼 도메인-중립 이름으로 변경 |
| **이행적 의존을 무시한 레이어링** | 상위 레이어가 중간 레이어를 통해 하위 레이어에 이행적으로 의존 | 각 레이어가 자신의 서비스 인터페이스를 선언 (그림 11-2) |

## 8. 설계 원칙

| 원칙 | 설명 |
|------|------|
| **둘 다 추상화에 의존** | 상위 모듈도, 하위 모듈도 추상화에 의존. 상위가 하위에 직접 의존 금지 |
| **추상화는 세부 사항에 의존 금지** | 인터페이스에 구현 세부 타입이 새어나오지 않아야 함 |
| **할리우드 원칙** | "연락드리겠습니다" — 하위 모듈이 상위 모듈이 선언한 인터페이스의 구현을 제공 |
| **클라이언트가 인터페이스를 소유** | 인터페이스 소유권은 라이브러리/서버가 아닌 **클라이언트** 쪽에 둔다 |
| **휘발성 기반의 추상화 결정** | 비휘발성 구체 클래스(`String` 등)는 직접 의존 허용, 휘발성 구체 클래스만 추상화 뒤로 숨김 |
| **재사용 가능한 것은 상위 정책** | 상위 정책이 하위 세부와 독립적이어야 다양한 문맥에서 재사용 가능 |
| **객체지향의 증명은 의존성 역전** | 의존성이 역전되어 있으면 객체지향 설계, 아니면 절차적 설계 |

## 9. 결론

전통적인 절차지향 프로그래밍 방식은 **정책이 구체적인 것에 의존하는 의존성 구조**를 만든다. 이런 경우 정책은 구체적인 사항의 변경에 따라 같이 변하기 때문에 불행한 일이다.

객체지향 프로그래밍은 이런 의존성 구조를 역전시켜:

- 구체적인 사항과 정책이 모두 추상화에 의존하고
- 대개 그 클라이언트가 서비스 인터페이스를 소유하게 만든다

> **핵심 통찰**: 좋은 객체지향 설계의 증명이 바로 이와 같은 **의존성의 역전**이다. 프로그램이 어떤 언어로 작성되었는가는 상관없다. 프로그램의 의존성이 역전되어 있다면 이것은 객체지향 설계이며, 의존성이 역전되어 있지 않다면 절차적 설계다.

> **Uncle Bob의 경험**: 의존성 역전의 원칙은 객체지향 기술에서 당연하게 요구되는 많은 이점 뒤에 있는 하위 수준에서의 기본적인 메커니즘이다. **재사용 가능한 프레임워크를 만들기 위해서는 이것의 적절한 응용이 필수적**이다. 이 원칙은 또한 변경에 탄력적인 코드를 작성하는 데 있어 결정적으로 중요하다. 추상화와 구체적 사항이 서로 분리되어 있기 때문에, 이 코드는 유지보수하기가 훨씬 쉽다.

## 요약

- **DIP의 정의**:
  - (a) 상위 모듈은 하위 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다
  - (b) 추상화는 세부 사항에 의존해서는 안 된다. 세부 사항이 추상화에 의존해야 한다
- **"역전"의 의미**: 전통적인 절차적 설계의 의존 방향(상위→하위, 정책→구체)을 뒤집어, 둘 다 추상화를 향하게 함
- **소유권의 역전 (할리우드 원칙)**: 클라이언트가 자신이 필요로 하는 인터페이스를 선언하고, 서버(하위 모듈)가 그 인터페이스를 구현
- **레이어링의 함정**: 단순 계층 의존은 이행적 의존을 낳음 → 각 레이어가 자신의 서비스 인터페이스를 선언하는 방식으로 역전
- **"추상화에 의존하자" 휴리스틱**: 구체 클래스 참조 금지, 구체 클래스 상속 금지, 구현된 메소드 오버라이드 금지
- **휘발성 vs 비휘발성**: 자주 바뀌지 않는 구체 클래스(`String` 등)는 직접 의존 허용, 휘발적 구체 클래스만 추상화 뒤로 숨김
- **Button-Lamp 예제**: `Button → Lamp` (DIP 위반) → `Button → ButtonServer ← Lamp` (DIP 적용) → `SwitchableDevice`로 이름까지 분리
- **용광로 예제**: 절차적 코드(`in/out` 직접 호출)는 재사용 불가 → `Thermometer`, `Heater` 인터페이스로 추상화하면 재사용 가능
- **동적 vs 정적 다형성**: 일반적으로는 동적 다형성이 더 나은 선택. 속도가 절실히 필요할 때만 정적(템플릿/제네릭) 다형성 고려
- **객체지향의 본질**: 의존성이 역전되어 있으면 객체지향 설계, 아니면 절차적 설계 — 언어와 무관

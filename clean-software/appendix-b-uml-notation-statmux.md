# Appendix B: UML Notation II — Statmux (UML 표기법 II: 통계적 멀티플렉서)

## 핵심 질문

UML의 동적 모델링 기법(*Dynamic Modeling - 시간에 따른 시스템 상태 변화와 객체 간 상호작용을 표현하는 기법*)은 어떻게 실시간 멀티스레드 시스템을 설계하고 검토하는 데 활용되는가? 상태 기계 다이어그램(*State Machine Diagram - 상태와 이벤트에 의한 전이로 시스템 동작을 모델링하는 도구*), 활동 다이어그램, 협동 다이어그램, 메시지 시퀀스 차트는 각각 어떤 문제를 풀기 위한 표기법이며 어떤 한계를 갖는가?

> 다이어그램이 내 의도를 잘 표현할 수도 있지만, 별도의 설명은 도움이 된다. 다이어그램 자체만으로 이해하기는 거의 어렵다.<br>— Robert C. Martin

---

## 1. 통계적 멀티플렉서(Statmux) 사례

부록 B는 통계적 멀티플렉서(*Statistical Multiplexer / Statmux - 하나의 전송선을 통해 복수의 직렬 데이터 스트림을 전송할 수 있게 해 주는 장치*)라는 1980년대 통신 장비를 사례로 UML의 동적 표기법을 설명한다. 정적인 클래스 다이어그램만으로는 잡아낼 수 없는 *동시성*, *상태 전이*, *경쟁 상태*와 같은 문제를 다루기 위해 어떤 표기법을 어떻게 결합하는지를 보여준다.

### 1.1 통계적 멀티플렉서란

56K 모뎀 하나와 직렬 포트 16개가 있는 장치를 생각해 보자. 두 스태트먹스가 전화선으로 연결되면, 문자가 한 장치의 1번 포트에서 다른 장치의 1번 포트로 전송된다. 모뎀 하나로 동시에 16개의 전이중(*Full Duplex - 동시에 양방향 전송이 가능한 통신 방식*) 통신 세션을 지원하는 구조다.

```
  +---------+                                +---------+
  | 프린터  |---+                       +----| 모니터  |
  +---------+   |                       |    +---------+
                |                       |
  +---------+   |   +------+    56K    +------+
  | 모니터  |---+---|statmux|=========|statmux|
  +---------+   |   +------+   임대선  +------+
        ...     |       (16ch)              |
  +---------+   |                            |
  | VAX     |---+                            |
  +---------+                                |
   (디트로이트)                          (시카고)
```

16개 채널 전체를 동시에 돌리면 56K를 나눠 갖게 되지만, 단말기와 프린터는 보통 사용률(*Duty Cycle - 장치가 실제로 작동하는 시간 비율*)이 10% 미만이다. 따라서 선이 공유되어 있어도 각 사용자는 통계적으로 56K에 가까운 성능을 경험한다.

### 1.2 소프트웨어 환경과 실시간 제약

스태트먹스 내부의 소프트웨어는 모뎀과 16개의 직렬 포트 사이에 위치한다.

```
  +--------+    +--------+    +--------+         +--------+
  | port 1 |    | port 2 |    | port 3 |   ...   |port 16 |
  +---+----+    +---+----+    +---+----+         +----+---+
      |             |             |                   |
      |             |  +-------+  |                   |
      +-------------+--|타이머|--+-------------------+
                       +---+---+
                           |
                  +--------+--------+
                  | 스태트먹스 SW   |
                  +--------+--------+
                           |
                       +---+---+
                       | 모뎀  |
                       +-------+
```

각 직렬 포트는 송신 준비 인터럽트와 수신 인터럽트 2개를 생성한다. 모뎀도 비슷한 인터럽트를 생성하므로 총 34개의 인터럽트 소스가 있고, 모뎀 인터럽트는 직렬 포트 인터럽트보다 우선순위가 높다. 추가로 밀리초 단위로 발화하는 타이머가 있어 이벤트 스케줄링이 가능하다.

> **핵심 통찰**: 초당 인터럽트 수는 5,600(직렬) + 1,000(타이머) ≈ 191,400으로 계산된다. 즉 인터럽트당 5.2µs 이상을 쓸 수 없다. 다행히 시스템 최대 지속 처리율은 초당 11,200문자(문자 간격 ≈ 90µs)이므로, ISR 외부 코드는 프로세서의 94% 이상을 자유롭게 쓸 수 있다.

## 2. 상태 기계 다이어그램 — 링 버퍼

입력 인터럽트 서비스 루틴(*ISR: Interrupt Service Routine - 인터럽트 발생 시 즉시 실행되는 함수*)은 하드웨어로부터 문자를 받아 한가할 때 처리할 수 있는 곳에 저장해야 한다. 이를 위해 **링 버퍼(*Ring Buffer - 고정 크기 배열을 순환적으로 사용하는 큐 자료구조*)** 가 사용된다.

### 2.1 클래스 다이어그램과 새로운 스테레오타입

```
       +-------------------+
       |     <<ISR>>       |
       |   Input ISR       |
       +-------------------+
                |
                | calls
                v
       +-------------------+        +-------------------+
       |   <<struct>>      |        |   <<interface>>   |
       |   RingBuffer      |<-------|       Ring        |
       +-------------------+        +-------------------+
       | first: char*      |        | + Get(): char     |
       | last:  char*      |        | + Put(char)       |
       | count: int        |        +-------------------+
       | buf:   char[N]    |                  ^
       +-------------------+                  | realize
                ^                             |
                |                +-------------------+
                |                |   ISRRingBuffer   |
                |                +-------------------+
                |                |  <<reentrant>>    |
                +----------------|  + Get(): char    |
                                 |  + Put(char)      |
                                 +-------------------+

       <<reentrant>>
       Put(RingBuffer*, char)
       {deadline=2µs, assembly, inline}
```

- `<<ISR>>` 스테레오타입 — 이 클래스가 인터럽트 서비스 루틴임을 표시한다. 이름 없는 단일 메소드만 가지며, 인터럽트 발생 시에만 실행된다.
- `<<struct>>` — 메소드가 없는 자료구조임을 표시한다. 어셈블리 함수가 직접 접근하기 위함이다.
- `<<reentrant>>` — 인터럽트에 대해 안전(재진입성*Reentrancy - 다른 실행 컨텍스트에서 동시에 호출되어도 일관되게 동작하는 성질*)함을 표시한다.
- `{deadline=2µs, assembly, inline}` — 함수의 비기능적 제약(*Property - 함수의 시간/공간/구현 제약을 표시하는 UML 태그*).

> **핵심 통찰**: 스테레오타입(*Stereotype - UML 모델 요소에 추가 의미를 부여하는 확장 메커니즘*)은 표준 UML이 잡지 못하는 영역(실시간 마감, 어셈블리 구현 등)을 도메인 어휘로 끌어들이는 메커니즘이다. 클래스의 목록 부분에 스테레오타입을 두면 그 아래 요소들이 모두 그 스테레오타입을 따른다.

### 2.2 상태 다이어그램으로 본 링 버퍼 동작

```
          +-------------------- Puttable ---------------------+
          |                                                   |
          |    *  Empty            Partial          Full      |
          |    -> [count==0] <----------> [count==N]          |
          |                                                   |
          +---------------------------------------------------+
                                  |
                  +--------------- Gettable -----------------+

   Put(c)/                       Get()/
     buf[last]=c;                  count--;
     last=(last+1)%N;              int f=first;
     count++;                      first=(first+1)%N;
                                   return buf[f];

   Empty: Get()/Underflow
   Full:  Put()/Overflow
```

- **세 가지 상태**: `Empty`(비어 있음), `Partial`(부분적으로 참), `Full`(가득 참).
- **상위 상태(*Superstate - 여러 하위 상태를 감싸는 상태*)**: `Puttable`(Empty + Partial), `Gettable`(Partial + Full).
- **보존 조건(*Guard Condition - 전이가 시작되기 위한 불리언 단언*)**: `[count==0]`, `[count==N]` — count 변수의 값에 따라 Partial과 Empty/Full 사이를 전이한다.

### 2.3 상태와 내부 전이

UML에서 상태는 가장자리가 둥근 직사각형으로 표현되며 두 구획을 가질 수 있다.

```
   +-----------------------------+
   |        State                |    <- 이름 구획
   +-----------------------------+
   | entry / myEntryFunction     |
   | exit  / myExitFunction      |    <- 내부 전이 구획
   | myEvent / myAction          |
   +-----------------------------+

   * --- initial --->  State   --- final ---> (●)
```

- **entry**: 상태에 진입할 때 실행.
- **exit**: 상태를 빠져나갈 때 실행(다시 같은 상태로 돌아오는 전이라도 발생).
- **내부 전이**: `이벤트이름 / 동작` 형태로 표기. 상태에 머무르면서 동작만 수행한다.
- **시작 의사 상태**(검은색 원) → 상태 기계 실행 시작점.
- **종료 의사 상태**(이중 원) → 도달 시 상태 기계 종료.

### 2.4 상태 간 전이

```
                event[guard]/action
   state1   ----------------------->   state2
```

전이는 이벤트가 발생하고 보존 조건이 참인 경우에만 발동된다. 시퀀스는 다음과 같다.

1. `state1` 탈출 → exit 동작 실행.
2. 전이 중 동작 실행.
3. `state2` 진입 → entry 동작 실행.

### 2.5 중첩된 상태

```
         A
          \
           Y
            \
   +---------v----- S ------------+
   |  +--+    W      +--+         |  Z
   |  | B |<-------->| C |--------|----> D
   |  +--+    X      +--+         |
   |   ^                          |
   |   |  S-Initial               |
   |   |  (default)               |
   +-Q-+--------------------------+
       |
       v
       (final)
```

상위 상태 S의 하위 상태로 B와 C가 있다. 다이어그램이 보여주는 동작 규칙은 다음과 같다.

- 전이 `Y`(외부 → S 테두리) → S의 시작 의사 상태를 통해 기본 하위 상태(C)로 진입. S와 C의 entry 동작이 차례로 실행된다.
- 전이 `W`, `X` → 하위 상태 사이 이동. S는 빠져나가지 않으므로 S.exit는 실행되지 않는다.
- 전이 `Z`(상위 상태 테두리 밖에서 시작) → B/C 어느 상태에 있든 D로 이동. 하위 상태 exit → S.exit → D.entry 순서.
- 전이 `T`(히스토리 기록기*History Marker - 마지막으로 활성화되었던 하위 상태를 기억하는 의사 상태*) → S에 다시 들어갈 때 마지막으로 활성화되었던 하위 상태로 복귀.
- 전이 `Q`(종료 의사 상태로) → S 종료 및 히스토리 정보 초기화.

> **핵심 통찰**: 일련의 이벤트 `Y → Z → T`는 기계를 하위 상태 C에 두지만, `R → T`나 `Y → W → Q → R → T`는 기계를 B에 둔다. 히스토리 기록기는 "이전에 어디 있었는지"를 기억함으로써 일시 정지 후 재개 같은 패턴을 단순하게 표현한다.

## 3. 출력 ISR 상태 기계

출력 ISR은 입력 ISR과 비슷하지만 한 가지 다른 점이 있다. 직렬 포트가 다음 문자를 받을 준비가 되어 인터럽트를 생성하더라도 링 버퍼가 비어 있으면 ISR은 아무 일도 하지 않는다. 직렬 포트는 새 문자가 전송되기 전까지 다시 신호를 보내지 않으므로, **새 문자가 도착했을 때 출력 인터럽트를 다시 시작할 전략**이 필요하다.

### 3.1 구조 — ISRRingBuffer에서 OutputISR로의 `<<calls>>` 의존성

```
       +-------------------+
       |     <<ISR>>       |
       |    Output ISR     |
       +-------------------+
                ^
                |  <<calls>>  (인위적 인터럽트)
                |
       +-------------------+
       |   ISRRingBuffer   |
       |  <<reentrant>>    |
       | + Get(): char     |
       | + Put(char)       |
       +-------------------+
```

ISRRingBuffer가 마치 인터럽트가 도착한 것처럼 OutputISR을 호출할 수 있게 한다.

### 3.2 출력 링 버퍼 상태 기계

입력 링 버퍼와 비교했을 때 차이점은 다음과 같다.

```
            Put(c)/buf[last]=c; last=(last+1)%N; count++;
                                      |
   *  Empty  -----------------------> Partial  [count==N]  Full
                                       ^                    |
   Get()/Underflow                     |                    |
                                       +------ [count==0]---+
                                          (Get() 처리 후)

   +-- Gettable: entry / OutputISR.interrupt --+
   |   (Empty → Partial 진입 시)              |
   +------------------------------------------+

   Get() / count--; int f=first; first=(first+1)%N; return buf[f];
   Full: Put(c)/Overflow
```

- `Puttable` 상위 상태가 없어지고 `Put` 전이가 2개로 분리되었다.
- Empty → Partial 진입 시 `Gettable` 상위 상태의 entry 동작에서 OutputISR에 대한 **인위적 인터럽트**를 발생시켜 송신을 재개한다.
- 두 번째 Put 전이는 Partial의 내부 전이다.

> **실무 팁**: 인위적 인터럽트 생성 메커니즘은 플랫폼에 강하게 의존한다. 어떤 기계에서는 ISR을 단순 함수처럼 호출할 수 있지만, 어떤 기계에서는 별도의 프로토콜이 필요하다. 이런 결합은 스테레오타입과 프로퍼티로 명시적으로 표시해야 후속 유지보수가 가능하다.

## 4. 활동 다이어그램 — 통신 프로토콜

두 스태트먹스는 모뎀을 통해 통신하며, 전송선은 *불완전*하다고 가정한다. 패킷이 소실/왜곡될 수 있으므로 **전송 프로토콜**이 필요하다. 책에서는 파이프라인과 피기백(*Piggyback - 정보 프레임에 ACK을 함께 실어 보내는 기법*)을 사용하는 **슬라이딩 윈도 프로토콜**(*Sliding Window Protocol - 윈도 크기만큼 확인 응답 없이 패킷을 연속 전송할 수 있는 프로토콜*)이 채택된다.

### 4.1 세 개의 독립 스레드

활동 다이어그램(*Activity Diagram - 상태 전이 다이어그램, 흐름도, 페트리 네트를 합친 이벤트 주도 멀티스레드 알고리즘 표기법*)으로 다음 세 스레드를 모델링한다.

```
            * (Initial)
            |
   +--------+----------+----------+
   |        |          |          |
   v        v          v          v
   (변수 초기화: W=N/2, SendLimit=W-1, S=0, A=N-1, E=0,
                RCVD[0..W-1]=false, RCVD[W..N-1]=true)
            |
            ===== (분기 - asynchronization bar) =====
            |          |          |
            v          v          v
        Sending    Receiving    Timer
        Thread     Thread       Thread
```

| 스레드 | 역할 |
|---|---|
| **Sending Thread** | 큐에 들어온 패킷을 전송. 확인 응답을 받지 않은 패킷이 W개 쌓일 때까지 전송 후 대기. |
| **Receiving Thread** | 패킷/응답을 수신, 검증, 처리. 중복 패킷은 폐기. |
| **Timer Thread** | Timeout(K)에 응답해 K 패킷을 재전송. AckTime에 응답해 응답 프레임 전송. |

### 4.2 전송 스레드의 활동

```
   +---- (loop) -----------------------------+
   |                                         |
   |   <S == SendLimit?>                     |
   |       |        |                        |
   |      Yes       No                       |
   |       |        |                        |
   |     (wait)    P[S] = ReadQueue()        |
   |               Send(P[S], S, A)          |
   |               Set Timeout for S         |
   |               Reset AckTime             |
   |               S = (S+1) % N             |
   |                                         |
   +-----------------------------------------+
```

- `S` — 다음 전송할 패킷의 식별 번호(0..N).
- `SendLimit = (Frame.Ack + W) % N` — 송신자가 사용할 수 있는 식별 번호 공간의 상한. 수신자가 응답한 위치에서 윈도 크기 W만큼 앞으로 설정된다.
- 각 패킷은 자신의 식별 번호 S와 피기백 응답 A(마지막으로 받은 패킷의 식별 번호)와 함께 전송된다.

### 4.3 수신 스레드와 타이밍 스레드

수신 스레드는 다음 동작을 수행한다.

1. 프레임 도착 대기.
2. `SendLimit = (Frame.Ack + W) % N`로 갱신, `Frame.Ack`의 타임아웃 종료.
3. 프레임이 패킷을 포함하면, RCVD 배열에서 중복 여부 확인.
4. 새 패킷이면 `RCVD[PS] = true`, `R[PS] = Frame.Packet`.
5. `PS == E`이면 별도 스레드로 `Process(R[E])` 호출, `RCVD[(E+W)%N] = false`, `A = E`, `E = (E+1) % N`.

타이밍 스레드는 두 종류의 이벤트를 처리한다.

- **`Timeout(K)`**: 패킷 K의 응답이 도착하지 않음 → K 재전송, 타이머 재시작.
- **`AckTime`**: X 밀리초 동안 패킷이 전송되지 않음 → 응답 프레임만 단독 전송.

### 4.4 활동 다이어그램의 표기 요소

```
       동작 상태(Action State)
       +------------------+
       (  procedure statement )    <- 양옆이 둥글고 위/아래는 평평
       +------------------+
              |
              v
          ┌───◇───┐
          │ 조건  │   <- 결정(decision) 마름모꼴
          └─Yes/No┘
              |
   비동기화 막대(asynchronization bar)
   ===========================
       |        |        |
       v        v        v
       (목적 상태들 - 동시에 진입)
```

- **동작 상태**(*Action State - 진입 동작이 완료되는 즉시 자동 탈출되는 상태*) — 진입 동작 완료 시 자동 탈출. 나가는 전이는 이벤트 라벨이 없고, 보호된 전이(*Guarded Transition - 보존 조건이 참일 때만 시작 가능한 전이*)들의 합집합은 항상 참이어야 한다.
- **결정(decision)** — 마름모꼴 아이콘. 모든 나가는 보존 조건의 불리언 합은 참.
- **복합 전이(*Complex Transition - 다수의 제어 흐름 스레드가 나뉘고 합쳐지는 전이*)** — 검은색 비동기화 막대로 표기. 원천 상태가 더 많으면 스레드 결합(join), 목적 상태가 더 많으면 스레드 분기(fork).

> **핵심 통찰**: "다이어그램이 맞는지 어떻게 알 수 있을까? 알 수 없다." 활동 다이어그램은 보통 코드처럼 직접 테스트할 수 없다. 알고리즘이 정말 맞는지를 알기 위해서는 코드가 나올 때까지 기다려야 한다. 다이어그램은 좋은 *교육* 도구지만 설계의 *유일한* 명세가 되기에는 충분히 표현적이고 정확하지 않다.

## 5. 객체 다이어그램과 능동적 객체

세 스레드가 공유 변수를 사용하므로, 그 스레드가 실행하는 함수는 같은 클래스의 메소드다. 그러나 오늘날 스레드 시스템은 스레드를 객체와 동일시한다 — UML에서는 이를 **능동적 객체(*Active Object - 단일 실행 스레드에 대한 책임을 지는 객체*)** 라고 한다.

### 5.1 객체 다이어그램

```
       +--------------------+         +--------+
       | : Communications   |<--------| :Thread|  itsSendingThread
       |   Protocol         |         +--------+
       | -W:int             |              ^
       | -SendLimit:int     |              | <<friend>>
       | -S:int             |              |
       | -RCVD:int[N]       |     +-------------------+
       | -P:PacketPtr[N]    |---->| sra:SenderRunAdapter|
       | -R:PacketPtr[N]    |     | Runnable + Run()  |
       +--------------------+     +-------------------+
              ^   ^
              |   |
              |   +-----------+
              |   <<friend>>  |
              |               |
       +-----------------+    |
       |:ReceiverRunAdapter|  |
       | Runnable +Run() |    |
       +-----------------+    |
              ^               |
              |               |
       +--------+         +--------+
       | :Thread|         | :Timer |
       +--------+         +--------+
       itsReceivingThread     ^
                              | <<friend>>
                       +------+------+
                       | :TimeoutAdapter|
                       | :AckTimeAdapter|
                       | Runnable+Run() |
                       +----------------+
```

- 객체 다이어그램(*Object Diagram - 특정 시점의 객체 인스턴스와 링크 관계를 묘사하는 다이어그램*)은 *특정 시각*에 객체들 사이의 정적 관계를 보여준다. 클래스 다이어그램과 달리 *런타임* 관계를 보여준다.
- `<<friend>>` 관계는 어댑터가 호출하는 메소드들이 CommunicationsProtocol에 대해 private이기를 원하기 때문에 만들었다. 어댑터 외의 것이 이 메소드를 호출하는 것은 원치 않는다.
- 능동적 객체는 굵은 테두리선으로 표시된다.

### 5.2 인터페이스 막대사탕

인터페이스는 `<<interface>>` 스테레오타입이나 막대사탕 아이콘으로 표현된다.

```
   스테레오타입 표기법                 막대사탕 표기법

   +--------------+                  Client ---o Server
   | <<interface>>|                                ^
   |   Server     |                                |
   +--------------+                          ServerImp
        ^   ^
        |   |
   Client   ServerImp
```

두 다이어그램은 같은 의미다. `Client`는 `Server` 인터페이스를 이용하며, `ServerImp`가 `Server` 인터페이스를 구현한다.

## 6. 협동 다이어그램 — 초기화 과정

협동 다이어그램(*Collaboration Diagram - 객체 간 메시지 흐름을 순서 번호와 함께 보여주는 다이어그램*)은 객체 다이어그램과 비슷하지만 시간에 따라 객체 간에 전송되는 메시지를 인자/반환값과 함께 보여준다.

### 6.1 표기 규칙

```
                  1.2.3a: message(arg)
   objectA  ----------------------------> objectB
                  <-------- return
```

- **순서 번호** — 점으로 구분된 숫자 목록 + 선택적 스레드 식별자. 메시지 1이 호출한 두 메시지는 1.1, 1.2가 된다.
- **스레드 식별자** — 메시지 1.2가 새 스레드 `a`를 생성하면 그 첫 메시지는 1.2.1a.
- **동기 메시지** — 검은색 채워진 화살표. 호출한 모든 메시지가 반환되어야 반환.
- **비동기 메시지** — 막대 모양 화살표. 새 제어 스레드 생성 후 즉시 반환.

### 6.2 CommunicationsProtocol 초기화

```
   : Communications
     Protocol
        |
        | 1: Initialize()
        v
   +-------------------+
   | 1.1: Create       |  ----> :SendingThread (:Thread)
   | 1.2: Create       |  ----> sra:SenderRunAdapter
   | 1.3: SetRunnable(sra)
   | 1.4: Start()      |        (1.4 비동기 → 분기)
   +-------------------+
                              1.4.1a: Run
                              1.4.1.1a: RunSendingThread()
   +-------------------+
   | 1.5: Create       |  ----> :ReceivingThread (:Thread)
   | 1.6: Create       |  ----> rra:ReceiverRunAdapter
   | 1.7: SetRunnable(rra)
   | 1.8: Start()      |        (1.8 비동기 → 분기)
   +-------------------+
                              1.8.1b: Run
                              1.8.1.1b: RunReceivingThread()
   +-------------------+
   | 1.9:  Create      |  ----> toa:TimeoutAdapter
   | 1.10: Create      |  ----> ata:AckTimeAdapter
   | 1.11: Register(toa)|----> :Timer
   | 1.12: Register(ata)|----> :Timer
   +-------------------+
```

1.4가 비동기 호출이므로 메인 흐름은 1.5~1.8까지 진행되어 ReceivingThread를 만든다. 그 사이 별도 스레드 `a`가 시작되어 SenderRunAdapter.Run을 실행하고, 어댑터는 다시 1.4.1.1a: RunSendingThread 메시지를 CommunicationsProtocol에 보낸다.

## 7. 메시지 시퀀스 차트 — 경쟁 상태 식별

경쟁 상태(*Race Condition - 별개의 두 이벤트의 순서가 예측될 수 없어 시스템 상태가 그 순서에 좌우되는 상황*)는 프로그래머가 잡기 어렵고 일시적이며 진단이 어려운 에러를 만든다. 메시지 시퀀스 차트(*Message Sequence Chart - 메시지 화살표가 아래로 기울어 전송/수신 사이에 시간이 흐를 수 있음을 표현한 시퀀스 다이어그램*)는 이런 상황을 발견하고 문서화하는 데 사용된다.

### 7.1 정상 흐름 — 패킷에 대한 응답

```
   :Timer   localSender   localReceiver   remoteSender   remoteReceiver
     |         |               |                |               |
     |Set Timeout for S        |                |               |
     |<--------|  Packet(S)    |                |               |
     |         |---------------------------------------------->|
     |         |               |                | ACK(S)        |
     |         |<-----------------------------|                 |
     |Kill Timeout for S       |                |               |
     |<--------|               |                |               |
```

### 7.2 응답 소실 — 재전송

```
   :Timer   localSender                                  remoteReceiver
     |         |                                              |
     |Set Timeout for S    Packet(S)                          |
     |<--------|--------------------------------------------->| (소실)
     |   너무 많은 시간                                       |
     |Timeout  |                                              |
     |-------->|                                              |
     |         |  Packet(S) (재전송)                          |
     |         |--------------------------------------------->|
```

### 7.3 ACK / 재전송 경쟁 상태

```
   :Timer   localSender                                  remoteReceiver
     |         |                                              |
     |Set Timeout for S    Packet(S)                          |
     |<--------|--------------------------------------------->|
     |   너무 많은 시간            ACK(S)  ◀─┐                |
     |Timeout  |◀──────────────────────────|<─────────────────|
     |-------->|  (먼저 도착)              |   교차            |
     |         |  Packet(S) (재전송)        |                  |
     |         |--------------------------------------------->|
     |Kill Timeout for S      (ACK 늦게 도착)                  |
     |<--------|                                              |
```

- 패킷 S는 제대로 전송되었고 ACK도 응답으로 전송되었다.
- 하지만 ACK이 타임아웃 이후에 도착했다. 따라서 패킷은 ACK을 받았음에도 재전송된다.
- 교차된 선이 이 경쟁을 표현한다.

> **핵심 통찰**: 그림 B-7의 슬라이딩 윈도 알고리즘은 이 경쟁을 훌륭하게 처리한다. 원격 수신자는 두 번째로 도착한 패킷 S가 중복되었음을 RCVD 배열로 인식하고 그것을 폐기한다. 메시지 시퀀스 차트의 주된 용도는 이런 *현장을 엉망으로 만들기 전에 잡아내는* 것이다.

## 8. 동시성 표기 정리

동시성과 관련된 UML 표기 요소를 정리하면 다음과 같다.

| 표기 | 의미 |
|---|---|
| 검은색 비동기화 막대 | 스레드 분기(fork)/결합(join) |
| 막대 모양 화살표 메시지 | 비동기 호출(새 스레드 생성) |
| 검은색 채워진 화살표 메시지 | 동기 호출 |
| 굵은 테두리 객체 | 능동적 객체(스레드 보유) |
| 메시지 순서 번호 뒤의 알파벳 | 스레드 식별자 (예: 1.4.1a) |
| 메시지 시퀀스 차트의 기울어진 화살표 | 전송/수신 사이의 시간 경과 |

> **핵심 통찰**: UML의 동적 표기법은 *언어*이지 *증명*이 아니다. 같은 시스템을 상태 기계, 활동 다이어그램, 협동 다이어그램, 메시지 시퀀스 차트로 여러 각도에서 표현해도 알고리즘이 옳다는 보장은 코드를 작성하기 전까지 얻을 수 없다. 그러나 *경쟁 상태처럼 코드에서는 잘 보이지 않는 결함을 미리 잡아내는 데*는 매우 유용하다.

## 9. TypeScript 예제 — 링 버퍼와 상태

상태 기계 다이어그램이 보여준 링 버퍼 동작을 코드로 옮기면 다음과 같다.

<details>
<summary>Java 원본 스타일</summary>

```java
public class ISRRingBuffer implements Ring {
  private final char[] buf;
  private int first = 0;
  private int last = 0;
  private int count = 0;

  public ISRRingBuffer(int n) {
    this.buf = new char[n];
  }

  public synchronized void Put(char c) {
    if (count == buf.length) {
      throw new RuntimeException("Overflow");
    }
    buf[last] = c;
    last = (last + 1) % buf.length;
    count++;
  }

  public synchronized char Get() {
    if (count == 0) {
      throw new RuntimeException("Underflow");
    }
    int f = first;
    first = (first + 1) % buf.length;
    count--;
    return buf[f];
  }
}
```

</details>

```typescript
/** 링 버퍼의 상태. */
type RingState = "Empty" | "Partial" | "Full";

/** 링 버퍼 — 상태 기계 모델을 그대로 따른다. */
class ISRRingBuffer {
  private readonly buf: string[];
  private first = 0;
  private last = 0;
  private count = 0;

  constructor(private readonly n: number) {
    this.buf = new Array<string>(n);
  }

  /** 현재 상태를 derive — count로부터 계산. */
  state(): RingState {
    if (this.count === 0) {
      return "Empty";
    }
    if (this.count === this.n) {
      return "Full";
    }
    return "Partial";
  }

  /** Put 전이 — Full 상태에서는 Overflow. */
  put(c: string): void {
    if (this.state() === "Full") {
      throw new Error("Overflow");
    }
    this.buf[this.last] = c;
    this.last = (this.last + 1) % this.n;
    this.count++;
  }

  /** Get 전이 — Empty 상태에서는 Underflow. */
  get(): string {
    if (this.state() === "Empty") {
      throw new Error("Underflow");
    }
    const f = this.first;
    this.first = (this.first + 1) % this.n;
    this.count--;
    return this.buf[f];
  }
}
```

> **핵심 통찰**: 상태 다이어그램의 보존 조건(`[count==0]`, `[count==N]`)은 코드의 `if` 분기가 된다. 다이어그램이 표현한 entry/exit 동작이 있다면 객체에서 상태 변경 직후 호출되는 메소드로 옮겨진다. 다이어그램은 *코드 구조를 안내*하지만 *코드를 대체하지는 않는다*.

## 요약

- **통계적 멀티플렉서(Statmux)** 사례는 실시간/멀티스레드 시스템에 UML 동적 표기법을 적용하는 방식을 보여준다.
- **상태 기계 다이어그램**은 entry/exit 동작, 내부 전이, 보존 조건, 중첩 상태, 히스토리 기록기 등을 활용해 객체의 생명주기를 모델링한다.
- **활동 다이어그램**은 동작 상태, 결정, 비동기화 막대(fork/join)를 통해 이벤트 주도 멀티스레드 알고리즘을 표현한다. 슬라이딩 윈도 프로토콜의 세 스레드(전송/수신/타이머)를 하나의 다이어그램으로 통합한다.
- **객체 다이어그램**은 특정 시점의 객체 인스턴스 관계를 보여주며, **능동적 객체**는 굵은 테두리로 표시되어 스레드 보유를 나타낸다.
- **협동 다이어그램**은 점 기반 순서 번호와 스레드 식별자로 메시지 시퀀스와 깊이를 표현하며, 비동기 메시지로 인한 스레드 분기를 자연스럽게 묘사한다.
- **메시지 시퀀스 차트**는 화살표 기울기를 통해 시간 경과를 표현하며, **경쟁 상태**를 발견하고 문서화하는 데 적합하다.
- **스테레오타입**(`<<ISR>>`, `<<struct>>`, `<<reentrant>>`, `<<friend>>`, `<<calls>>`)과 **프로퍼티**(`{deadline=2µs, assembly, inline}`)는 표준 UML이 잡지 못하는 도메인 제약을 표현하는 확장 메커니즘이다.
- 다이어그램은 직접 테스트할 수 없고 알고리즘의 정확성을 보장하지 못한다. 다이어그램은 *교육*과 *경쟁 상태 발견*에는 매우 유용하지만, 설계의 *유일한 명세*가 되기에는 부족하다.

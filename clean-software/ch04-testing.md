# Chapter 4: Testing (테스트)

## 핵심 질문

테스트를 작성하는 일은 단순한 "검증" 작업인가, 아니면 그 이상의 무엇인가? 코드보다 테스트를 먼저 작성하면 설계에 어떤 영향을 주는가? 단위 테스트와 인수 테스트는 어떻게 다르며, 각각이 시스템 구조에 어떤 압박을 가하는가? 그리고 — 왜 테스트가 결국 좋은 설계를 강제하게 되는가?

> 불은 황금을 시험하고, 역경은 강한 사람을 시험한다.<br>— 세네카(Seneca), 기원전 3세기 ~ 서기 65년

---

## 1. 테스트는 검증이 아니라 설계 활동이다

하나의 단위 테스트(*Unit Test - 한 함수나 클래스 같은 시스템의 작은 구성 요소의 동작을 검증하는 자동화된 테스트*)를 작성하는 일은 **단순한 검증이라기보다 설계의 문제**다. 또한 **단순한 검증이라기보다 문서화의 문제**이기도 하다. 단위 테스트를 작성하는 일은 상당한 숫자의 피드백 루프(*feedback loop - 결과를 다시 입력으로 받아 행동을 조정하는 순환 구조*)를 마무리짓는 일이고, 이 중 최소한 하나는 기능의 검증에 속한다.

> **핵심 통찰**: 테스트는 "코드를 검증하는 부수적 작업"이 아니다. 테스트를 작성하는 행위 자체가 **설계 활동이며 문서화 활동**이다. 검증은 그중 하나의 부산물일 뿐이다.

## 2. 테스트 주도 개발 (Test-Driven Development)

### 2.1 사고 실험: 테스트를 먼저 작성한다면?

프로그램을 설계하기 전에 먼저 테스트를 설계하면 어떨까? 어떤 함수가 존재하지 않으면 실패하는 테스트를 만든 다음, 프로그램에서 그 함수를 구현하면 어떨까? 아예 한 줄의 코드도 없어서 실패하는 테스트를 만든 다음에야 프로그램에 그 코드를 추가하는 것은 어떨까?

처음에 어떤 기능성의 존재 여부를 검사하는 테스트를 작성한 후에 단계적으로 그 기능성을 추가해 나가는 것은 어떨까? 이런 방식이 개발하던 소프트웨어의 설계에 주는 효과는 무엇일까? 이 같은 포괄적인 테스트 집합의 존재에서 이끌어낼 수 있는 이점은 무엇일까?

이러한 사고 실험에서 출발하는 개발 방식이 **테스트 주도 개발(TDD: Test-Driven Development)**이다.

### 2.2 TDD가 가져오는 네 가지 효과

#### 효과 1: 모든 함수에 대한 안전망

가장 명백한 효과는 프로그램의 **모든 단일 함수가 그 동작을 검증하는 테스트를 갖게 된다**는 것이다. 이 테스트 집합은 그 이후의 개발을 위한 뒷받침이 된다:

- 프로그래머가 기존 기능을 부주의하게 망가뜨릴 때마다 그 사실을 알려준다
- 프로그래머는 중요한 것을 망가뜨릴 염려 없이 함수를 추가하거나 구조를 바꿀 수 있다
- 테스트가 프로그램이 아직 제대로 동작하고 있음을 알려주므로, 훨씬 자유롭게 수정과 개선이 가능하다

#### 효과 2: 호출자 관점의 설계

명백하진 않지만 더 중요한 효과는, 테스트를 먼저 작성할 경우 **프로그래머가 다른 관점에서 문제를 해결**할 수 있다는 것이다. 프로그래머는 작성할 프로그램을 그 **프로그램의 호출자 관점**에서 봐야 한다. 따라서 함수만큼이나 **인터페이스에도 바로 관심**을 가져야 한다.

> **핵심 통찰**: 테스트를 먼저 작성함으로써, 프로그래머는 **편리하게 호출할 수 있는 소프트웨어**를 설계할 수 있다. 코드를 먼저 짜면 구현의 편의를 위한 인터페이스가 나오기 쉽지만, 테스트를 먼저 짜면 사용의 편의를 위한 인터페이스가 나온다.

#### 효과 3: 테스트 가능한 설계 강제

테스트를 먼저 작성함으로써, 프로그래머는 자신이 반드시 **테스트 가능한 프로그램을 설계하도록 강제**할 수 있다. 호출 가능하고 테스트 가능한 프로그램을 설계하는 일은 매우 중요하다.

소프트웨어가 호출 가능하고 테스트 가능해지려면 **주위 환경에서 분리**되어야 한다. 따라서 테스트를 먼저 작성한다는 건 프로그래머가 **소프트웨어를 다른 환경과 분리하도록 강제**하는 것이다.

#### 효과 4: 살아있는 문서

테스트를 먼저 작성할 때의 또 다른 효과는 테스트가 **문서화의 귀중한 한 형태**로 기능할 수 있다는 것이다. 어떤 함수를 호출하거나 어떤 객체를 생성하는 방법을 알고 싶을 때, 그것을 보여주는 테스트가 있다.

| 일반 문서 | 테스트로서의 문서 |
|-----------|-------------------|
| 시간이 지나면 코드와 어긋난다 | 컴파일 및 실행 가능 |
| 거짓을 담을 수 있다 | 거짓을 보여줄 수 없다 |
| 갱신이 미뤄지기 쉽다 | 항상 최근의 상태를 반영 |

> **핵심 통찰**: 테스트는 다른 프로그래머가 그 코드를 사용하는 방법을 알게 도와주는 **예제 집합**으로서 역할을 한다. 이 문서화는 컴파일 및 실행이 가능하며, 항상 최근의 상태를 반영하고, 거짓을 보여줄 수도 없다.

## 3. 테스트 우선 방식 설계의 예: 움퍼스 사냥 게임

### 3.1 게임 설정

Uncle Bob은 재미로 <몬스터 사냥하기(Hunt the Wumpus)>의 변형 버전을 만들어 봤다. 이 프로그램은 플레이어가 동굴을 돌아다니면서 움퍼스(Wumpus)에게 잡혀먹히기 전에 먼저 움퍼스를 처치하는 간단한 어드벤처 게임이다.

- 동굴은 통로로 서로 연결되어 있는 방의 집합으로 구성
- 각 방에는 동서남북으로 향하는 통로가 있을 수 있다
- 플레이어는 컴퓨터에게 어느 방향으로 갈지를 알려서 움직인다

### 3.2 첫 번째 테스트 (목록 4-1)

다음 테스트는 새 `WumpusGame`을 생성하고, 동쪽 통로를 통해 방 4를 방 5와 연결하고, 플레이어를 방 4에 넣고, 동쪽으로 이동하라는 명령을 내린 다음, 플레이어가 방 5에 있는지 확인한다.

<details>
<summary>원문 Java 코드</summary>

```java
// Java - 목록 4-1: testMove
public void testMove() {
    WumpusGame g = new WumpusGame();
    g.connect(4, 5, "E");
    g.setPlayerRoom(4);
    g.east();
    assertEquals(5, g.getPlayerRoom());
}
```

</details>

```typescript
// TypeScript
function testMove(): void {
    const g = new WumpusGame();
    g.connect(4, 5, "E");
    g.setPlayerRoom(4);
    g.east();
    assertEquals(5, g.getPlayerRoom());
}
```

이 코드는 모두 `WumpusGame`의 어떤 부분보다도 **먼저** 작성한 것이다.

### 3.3 계획된 프로그래밍 (Intentional Programming)

Uncle Bob은 워드 커닝햄(Ward Cunningham)의 조언을 받아들여, **내용만 전달하는 방식**으로 테스트를 작성했다. 이 테스트가 내포하는 구조에 맞는 코드를 작성하기만 하면 이 테스트를 동작하게 할 수 있다고 믿었다.

> **계획된 프로그래밍(Intentional Programming)**<br>프로그래머는 자신의 의도를 구현하기 전에, 먼저 그 의도를 **가능한 한 단순하고 읽기 편하게 만들어 테스트로 제시**한다. 그리고 이 단순성과 명쾌함이 바로 프로그램의 좋은 구조를 의미한다고 믿는다.

### 3.4 흥미로운 설계 의사결정: Room 클래스는 필요한가?

계획된 프로그래밍은 곧 흥미로운 설계 의사결정의 문제로 이끌었다. **이 테스트는 `Room` 클래스를 사용하지 않는다.** 한 방을 다른 방과 연결하는 동작이 의도를 그대로 전달한다.

| 직관적 접근 | 테스트가 드러낸 설계 |
|-------------|----------------------|
| "방"이 핵심 개념이므로 `Room` 클래스 필요 | 방은 그냥 정수로 표현 |
| 게임이 방 사이 이동에 관한 것 | **연결(connection)**이 더 주된 개념 |

이것은 직관적이지 않을 수도 있다. 결국, 이 프로그램은 전부 방에 관련된 것처럼 보일 수도 있다. 그러나 Uncle Bob은 **움퍼스 게임에 있어 연결의 개념이 방의 개념보다 더 주된 것**이라고 주장한다.

> **핵심 통찰**: 요점은 어느 설계가 옳은가가 아니라, 이 테스트가 **아주 이른 시기에 주요한 설계 이슈를 명백히 한다**는 것이다. 테스트를 먼저 작성하는 것은 **설계 의사결정의 차이를 식별**하는 것이다.

### 3.5 컴파일 가능한 문서로서의 테스트

나중에 두 방을 연결하는 방식이나 특정한 방향으로 움직이는 방식을 알고 싶다면, 모호한 용어를 사용하지 않고도 이 테스트를 통해 그것이 어떤 방식으로 이루어지는지 보여줄 수 있다. 이렇게 테스트는 **프로그램을 설명하는 컴파일 가능하고 실행 가능한 문서**가 된다.

## 4. 테스트 분리 — Mock Object 패턴

### 4.1 급여 관리 시스템의 예 (그림 4-1)

급여 관리(Payroll) 애플리케이션의 결합된 모델을 보자:

```
                ┌──────────────────┐
                │   CheckWriter    │
                ├──────────────────┤
                │ + writeCheck()   │
                └────────▲─────────┘
                         │
            ┌────────────┴────────────┐
            │        Payroll          │
            └────────────┬────────────┘
                         │
                         ▼
                ┌──────────────────┐
                │     Employee     │
                ├──────────────────┤
                │ + calculatePay() │
                │ + postPayment()  │
                └──────────────────┘
                         │
                         ▼
                ┌──────────────────┐
                │ EmployeeDatabase │
                ├──────────────────┤
                │ + getEmployee()  │
                │ + putEmployee()  │
                └──────────────────┘
```

`Payroll` 클래스는 `EmployeeDatabase` 클래스를 이용해 `Employee` 객체를 꺼낸다. `Employee`에 자신의 임금을 계산하도록 요청하고, `CheckWriter` 객체에 그 임금을 넘겨주어 수표를 만든다. 마지막으로, `Employee` 객체에 임금을 지급하고 그 객체를 데이터베이스에 다시 기록한다.

### 4.2 테스트 작성의 곤란함

아직 어떤 코드도 작성하지 않았다고 가정하자. 이제 `Payroll` 객체의 행위를 명시하는 테스트를 작성해야 한다. 그런데 테스트 작성과 관련하여 여러 가지 문제가 있다:

| 문제 | 구체적인 곤란 |
|------|---------------|
| **어떤 데이터베이스를 사용할 것인가?** | `Payroll`을 테스트하기도 전에 완벽한 기능을 갖춘 데이터베이스를 작성해야 하는가? 그 데이터베이스에 어떤 데이터를 로드해야 하는가? |
| **수표 출력을 어떻게 검증하나?** | 프린터에서 출력된 수표를 보고 그 액수를 확인하는 자동화 테스트를 작성하는 것은 불가능하다 |

### 4.3 해결책: Mock Object 패턴

이 문제의 해결책은 **의사 객체(*Mock Object - 실제 객체를 흉내내는 테스트 전용 객체*) 패턴**을 이용하는 것이다. `Payroll`의 모든 관련 요소 사이에 인터페이스를 추가하고 이 인터페이스를 구현하는 **테스트 스텁(*stub - 테스트에서 특정 입력에 대해 정해진 응답을 돌려주도록 만든 가짜 구현*)**을 생성할 수 있다.

#### 분리된 Payroll 구조 (그림 4-2)

```
       ┌───────────────────┐     ┌──────────────────┐
       │  «interface»      │     │  «interface»     │
       │   CheckWriter     │     │     Employee     │
       │ + writeCheck()    │     │ + calculatePay() │
       │                   │     │ + postPayment()  │
       └─────────▲─────────┘     └─────────▲────────┘
                 │                         │
       ┌─────────┴────┐          ┌─────────┴─────────┐
       │ MockCheck    │          │   MockEmployee    │
       │   Writer     │          │                   │
       └──────────────┘          └───────────────────┘
                 ▲                         ▲
                 │                         │
                 │      ┌──────────┐       │
                 └──────│ Payroll  │───────┤
                        └──────────┘       │
                              │            │
                              ▼            │
                  ┌──────────────────────┐ │
                  │  «interface»         │ │
                  │  EmployeeDatabase    │◀┘
                  │  + getEmployee()     │
                  │  + putEmployee()     │
                  └─────────▲────────────┘
                            │
                  ┌─────────┴──────────┐
                  │ MockEmployeeData   │   ◀── PayrollTest가 질의
                  │       base         │
                  └────────────────────┘
```

### 4.4 분리된 Payroll 테스트 (목록 4-2)

이 테스트는 적절한 의사 객체를 생성하고, 그것을 `Payroll` 객체에 넘겨준다. 그리고 그 `Payroll` 객체에게 모든 직원의 임금을 지불하라고 한 다음, 의사 객체에 모든 수표가 올바르게 작성됐는지, 모든 임금이 올바르게 지급됐는지 검증해 달라는 요청을 한다.

<details>
<summary>원문 Java 코드</summary>

```java
// Java - 목록 4-2: testPayroll
public void testPayroll() {
    MockEmployeeDatabase db = new MockEmployeeDatabase();
    MockCheckWriter w = new MockCheckWriter();
    Payroll p = new Payroll(db, w);
    p.payEmployees();
    assertTrue(w.checksWereWrittenCorrectly());
    assertTrue(db.paymentsWerePostedCorrectly());
}
```

</details>

```typescript
// TypeScript
function testPayroll(): void {
    const db = new MockEmployeeDatabase();
    const w = new MockCheckWriter();
    const p = new Payroll(db, w);
    p.payEmployees();
    assertTrue(w.checksWereWrittenCorrectly());
    assertTrue(db.paymentsWerePostedCorrectly());
}
```

물론 이 테스트가 검사하는 것은 **`Payroll`이 전부 옳은 데이터로 옳은 함수를 호출하는지의 여부**다. 사실 수표가 제대로 작성됐는지를 검사하는 것은 아니다. 데이터베이스가 제대로 갱신됐는지도 검사하지 않는다. 그보다는, **`Payroll` 클래스가 분리된 것처럼 동작하는지 여부**를 검사한다.

### 4.5 MockEmployee는 왜 필요한가?

`MockEmployee`가 무엇을 위한 것인지 궁금할지도 모른다. 의사 객체 대신 실제 `Employee` 클래스를 쓸 수 있을 것처럼 보인다. 만약 그게 가능했다면, 아무 망설임 없이 사용했을 것이다.

> **Uncle Bob의 경험**: 이 경우, 나는 `Payroll`의 함수를 검사할 때 필요한 정도보다 `Employee` 클래스가 더 복잡하다고 가정했다. 의사 객체를 쓸지, 진짜 객체를 쓸지는 **그 객체의 복잡도와 테스트 격리 필요성을 저울질**해서 결정한다.

### 4.6 운 좋게 얻은 분리

`Payroll`을 다른 객체와 분리하는 것은 바람직한 일이다:

- 테스트가 가능해진다
- 애플리케이션 확장 측면에서 다른 데이터베이스와 수표 기록기로 교체할 수 있게 해준다
- 이 분리 작업이 **테스트에 대한 필요에서 촉발**되었다는 점이 흥미롭다

> **핵심 통찰**: 테스트에서 모듈 분리에 대한 필요성은 프로그래머가 **프로그램 전체 구조에 이득이 되는 방식으로 분리 작업을 하도록 강제**한다. **코드보다 테스트를 먼저 작성하면 설계가 개선된다.**

이 책의 많은 부분은 의존성을 관리하기 위한 설계 원칙을 다룬다. 이 원칙은 클래스와 패키지를 주위 환경으로부터 분리하기 위한 가이드라인과 테크닉을 제공한다. 이를 **단위 테스트 전략의 일부**로 이용하면 가장 이롭다.

---

## 5. 인수 테스트 (Acceptance Test)

### 5.1 단위 테스트만으로는 불충분하다

단위 테스트는 필수적이지만 검증으로서는 불충분하다.

| 구분 | 단위 테스트 | 인수 테스트 |
|------|-------------|-------------|
| 검증 대상 | 시스템의 작은 구성 요소 | 시스템 전체로서의 동작 |
| 테스트 종류 | **화이트박스 테스트**(*white-box test - 테스트하는 모듈의 내부 구조를 알고 그것에 의존하는 테스트*) | **블랙박스 테스트**(*black-box test - 테스트하는 모듈의 내부 구조를 모르고 그것에 의존하지 않는 테스트*) |
| 작성자 | 프로그래머 | 고객, 또는 고객과 연결된 기술 인력 (QA 등) |
| 검증 관점 | 개별 메커니즘이 기대대로 동작하는가 | **고객의 요구사항이 충족되는가** |
| 문서 역할 | 시스템 **내부 요소**를 위한 문서 | 시스템 **기능 요소**를 위한 문서 |

### 5.2 인수 테스트는 궁극적 문서다

> **핵심 통찰**: 인수 테스트는 **기능 요소의 궁극적인 문서화 형태**다. 고객이 어떤 기능 요소가 제대로 되어 있는지 검증하는 인수 테스트를 작성하기만 하면, 프로그래머는 이 인수 테스트를 보고 정확하게 그 기능 요소를 이해할 수 있다.

따라서 단위 테스트가 시스템 내부 요소를 위한 컴파일 가능하고 실행 가능한 문서로서의 역할을 수행하는 것과 마찬가지로, 인수 테스트는 시스템의 기능 요소를 위한 컴파일 가능하고 실행 가능한 문서로서의 역할을 수행한다.

### 5.3 인수 테스트는 아키텍처에 영향을 준다

인수 테스트를 먼저 작성하는 것은 **시스템의 아키텍처에 큰 영향**을 준다. 시스템을 테스트 가능하게 만들려면, 시스템은 **상위 아키텍처 수준에서 주위 환경으로부터 분리**되어야 한다.

예: 사용자 인터페이스(UI: User Interface)는 인수 테스트가 UI를 통하지 않고도 업무 규칙에 접근할 수 있는 것과 같은 식으로 **업무 규칙으로부터 분리**되어야 한다.

### 5.4 수동 인수 테스트의 함정

프로젝트의 초기 반복에는 인수 테스트를 수동으로 하려는 충동이 생긴다. 그러나 이것은 현명한 선택이 못 된다.

> **Uncle Bob의 경험**: 인수 테스트의 자동화 필요성 때문에 생기는 **분리 작업을 위한 압박**을 그 반복으로부터 빼앗기 때문에 현명하지 못한 선택이다. 인수 테스트를 자동화해야 한다는 사실을 잘 이해하고 제일 처음의 반복을 시작하면, 아키텍처 사이의 균형(trade-off)을 완전히 다르게 만들 수 있다.

| 단위 테스트 | 인수 테스트 |
|-------------|-------------|
| 프로그래머로 하여금 **작은 단위에서 뛰어난 설계 의사결정**을 할 수 있게 만든다 | 프로그래머로 하여금 **큰 단위에서 뛰어난 아키텍처 의사결정**을 할 수 있게 해준다 |

### 5.5 인수 테스트 프레임워크의 점진적 구축

인수 테스트 프레임워크(framework — 다양한 분야의 관리 툴을 하나로 통합할 수 있는 일종의 플랫폼)를 만드는 일이 기운 빠지는 작업처럼 보일지도 모른다. 그러나:

- 단 한 번의 반복 동안에 작업할 기능 요소를 택하고
- 그 몇 개의 인수 테스트에 필요한 프레임워크 부분만을 만든다면

작성하기가 그다지 어렵지 않음을 알게 될 것이다. 또한 이런 노력이 그만큼 가치가 있다는 사실도 알게 될 것이다.

## 6. 인수 테스트의 예: 급여 관리

### 6.1 첫 번째 반복의 요구

급여 관리 애플리케이션의 첫 번째 반복에서:

- 데이터베이스에 직원을 추가하고 삭제할 수 있어야 한다
- 현재 데이터베이스에 있는 직원의 임금을 지급 수표로 만들 수 있어야 한다
- 다행히도 여기서는 **월급을 주는 직원만** 다루면 되며, 그 밖의 직원에 대한 처리는 이후의 반복까지 보류한다

아직 코드는 작성하지 않았고, 설계에도 시간을 들이지 않았다. 바로 지금이 인수 테스트에 대해 생각할 적기다.

### 6.2 단순한 텍스트 스크립트

Uncle Bob은 작성하기 편하고, 변경하기 쉬운 인수 테스트를 만들려고 한다. 형상 관리 툴(*configuration-management tool - 코드/문서의 버전과 변경 이력을 관리하는 도구*)에 인수 테스트를 포함시키고 저장하여 원할 때마다 실행하기를 원한다. 그러므로 **인수 테스트를 단순한 텍스트 파일로 작성**하는 생각은 일리가 있다.

다음은 인수 테스트 스크립트의 한 예다:

```
AddEmp 1429 "Robert Martin" 3215.88
Payday
Verify Paycheck EmpId 1429 GrossPay 3215.88
```

이 예에서는:

1. 직원 번호 1429를 데이터베이스에 추가한다. 이 직원의 이름은 "Robert Martin"이고, 월급은 3215.88달러다
2. 시스템에 오늘이 월급날이고 모든 직원에게 월급을 주어야 한다는 사실을 알려준다
3. 1429번 직원의 지급 수표가 GrossPay(총액) 필드 3215.88달러로 생성됐는지 확인한다

분명히, 이런 종류의 스크립트는 **고객이 작성하기에 아주 쉽다**. 또한 이런 스크립트에는 새로운 기능성을 추가하기도 쉽다.

### 6.3 스크립트가 내포하는 아키텍처 정보

이 스크립트의 처음 두 라인은 급여 관리 애플리케이션의 기능으로서, **payroll 트랜잭션**(*transaction - 시스템 상태를 변경하는 하나의 작업 단위*)이라 부를 수 있다. 이것은 급여 관리 시스템의 사용자가 기대하는 기능이다.

그러나 **`Verify` 라인은 급여 관리 시스템의 사용자가 기대한 트랜잭션이 아닌, 인수 테스트에 특화된 지시어(directive)**다.

| 라인 | 종류 |
|------|------|
| `AddEmp 1429 "Robert Martin" 3215.88` | 시스템의 실제 기능 (payroll 트랜잭션) |
| `Payday` | 시스템의 실제 기능 (payroll 트랜잭션) |
| `Verify Paycheck EmpId 1429 GrossPay 3215.88` | 인수 테스트 지시어 |

따라서 이 인수 테스트 프레임워크는 **payroll 트랜잭션을 인수 테스트 지시어와 분리**해서 이 텍스트 파일을 해석해야 한다. 급여 관리 애플리케이션에 payroll 트랜잭션을 보내고 난 후, 데이터를 검증하기 위해 인수 테스트 지시어를 사용해서 질의를 해야 한다.

### 6.4 아키텍처에 가해지는 긴장

이것은 이미 급여 관리 프로그램에 **아키텍처와 관련된 긴장(stress)**을 주게 된다. 급여 관리 프로그램은:

- **사용자로부터 직접 입력**을 받아야 하고
- **인수 테스트 프레임워크로부터도 입력**을 받아야 한다

프로그래머는 가능한 한 빨리 이 두 가지 입력 경로를 하나로 모으고 싶어 한다. 그래서 급여 관리 프로그램은 하나 이상의 소스에서 오는 `AddEmp`와 `Payday` 형태의 **트랜잭션을 처리할 수 있는 트랜잭션 처리기**가 필요할 것이다. 이 트랜잭션을 위한 **일반적인 폼(form)을 찾아서 특화된 코드의 양을 최소로 유지**하고 싶은 것이다.

### 6.5 해결책: XML 트랜잭션

한 가지 해결책은 급여 관리 애플리케이션에 **XML로 된 트랜잭션을 보내는 것**이다. 인수 테스트 프레임워크는 물론 XML을 생성할 수 있을 것이고, 급여 관리 시스템의 UI도 XML을 생성할 수 있음 직하다.

```xml
<AddEmp PayType="Salaried">
    <EmpId>1429</EmpId>
    <Name>Robert Martin</Name>
    <Salary>3215.88</Salary>
</AddEmp>
```

이 트랜잭션은 **서브루틴 호출, 소켓(socket), 심지어 배치(batch) 입력 파일**을 통해서도 급여 관리 애플리케이션에 입력될 수 있다. 어떤 것을 다른 것으로 바꾸는 건 개발 과정에서는 정말 평범한 일이다. 따라서 초기 반복에서는 파일로부터 트랜잭션을 읽어오기로 결정하고, 훨씬 나중에 API나 소켓으로 옮겨가도 괜찮다.

### 6.6 출력도 XML로

`Verify` 지시자를 어떻게 실행할 수 있을까? 분명히 급여 관리 애플리케이션이 생성한 데이터에 접근할 수 있는 방법이 있어야 한다. 그러나 인수 테스트 프레임워크가 출력된 수표에 쓰인 것을 읽으려고 시도하는 것은 원하지 않는다.

차선책: **급여 관리 애플리케이션이 지급 수표를 XML 형식으로 생성**하게 할 수 있다. 그러면 인수 테스트 프레임워크는 이 XML을 받아 적절한 데이터를 얻기 위해 질의할 수 있다. XML에서 수표를 출력하는 마지막 단계는 수동 인수 테스트로도 충분히 감당할 수 있을 만큼 평범한 일이 된다.

```xml
<Paycheck>
    <EmpId>1429</EmpId>
    <Name>Robert Martin</Name>
    <GrossPay>3215.88</GrossPay>
</Paycheck>
```

### 6.7 전체 흐름

```
[인수 테스트 텍스트 파일]
        │
        ▼
[인수 테스트 프레임워크]
   - 텍스트 → XML 트랜잭션 번역
   - 파일에 기록
        │
        ▼
[급여 관리 프로그램 실행]
   - 파일에서 XML 트랜잭션 읽기
   - 처리 후 XML 지급 수표 출력
        │
        ▼
[인수 테스트 프레임워크]
   - XML 출력 결과 읽기
   - Verify 지시자 실행
```

### 6.8 운 좋게 얻은 아키텍처

급여 관리 시스템 아키텍처의 인수 테스트가 주는 압박에 주목하자. **테스트를 먼저 생각했다**는 바로 그 사실이 프로그래머를 **XML 형식의 입력과 출력이란 개념**으로 아주 빠르게 연결시켰다.

이 아키텍처는:

- **트랜잭션 소스**를 급여 관리 애플리케이션으로부터 분리한다
- **급료 지급 수표 출력 메커니즘**을 급여 관리 애플리케이션으로부터 분리한다

이것은 아키텍처와 관련해 아주 바람직한 의사결정이다.

> **핵심 통찰**: 단위 테스트가 작은 단위의 분리를 강제하는 것처럼, 인수 테스트는 **큰 단위의 분리(아키텍처 수준의 분리)를 강제**한다. "테스트 가능성"이라는 압박이 자연스럽게 좋은 아키텍처로 안내한다.

## 7. 테스트의 빈도와 시스템 퇴보 방지

일련의 테스트를 실행하는 것이 간단할수록:

- 테스트는 좀 더 자주 실행되며
- 테스트를 많이 실행할수록 이 테스트에 어긋나는 것들을 좀 더 빨리 찾게 된다

만약 모든 테스트를 하루에 몇 번 정도 실행할 수 있다면, 시스템에 문제가 발생한 채 몇 분 이상 있는 일이 없을 것이다.

> **핵심 통찰**: 이것은 합리적인 목표다. 프로그래머는 그저 **시스템이 퇴보하지 않게만** 하면 된다. 일단 어떤 수준에서 제대로 동작하기만 하면, 절대 그보다 낮은 수준으로 퇴보하지는 않는다.

## 8. 결론

검증은 테스트 작성이 주는 이점 중 **하나일 뿐**이다.

### 8.1 모호하지 않은 문서

단위 테스트와 인수 테스트 모두 컴파일 및 실행 가능한 문서화의 한 형태이므로, **정확하고 신뢰성이 있다**. 게다가 이 테스트는 그것을 보는 사람이 읽을 수 있도록 모호하지 않은 언어로 작성된다.

| 테스트 종류 | 작성 언어 | 독자 |
|-------------|-----------|------|
| 단위 테스트 | 프로그래밍 언어 | 프로그래머 |
| 인수 테스트 | 고객이 설계한 (스크립트) 언어 | 고객도 읽을 수 있음 |

### 8.2 아키텍처와 설계에 미치는 효과

아마 이 모든 테스트 작업에서 **가장 중요한 이점은 아키텍처와 설계에 미치는 효과**일 것이다.

> **핵심 통찰**: 어떤 모듈이나 애플리케이션을 테스트 가능하게 만들려면 주위 환경으로부터 분리해야 하는데, 좀 더 테스트 가능해질수록 더욱 주위 환경으로부터 분리된다. 포괄적인 인수 테스트와 단위 테스트를 고려하는 활동은 **소프트웨어의 구조에 아주 긍정적인 영향**을 미친다.

## 요약

- **테스트는 검증이 아니라 설계 활동이다** — 단순한 검증이라기보다 설계와 문서화의 문제다
- **TDD의 네 가지 효과**:
  1. 모든 함수에 대한 안전망
  2. **호출자 관점**에서 설계하게 됨 (사용 편의를 위한 인터페이스)
  3. **테스트 가능한 설계** 강제 — 즉, 주위 환경과의 분리 강제
  4. 컴파일 가능하고 거짓을 담지 못하는 **살아있는 문서**
- **계획된 프로그래밍(Intentional Programming)** — 의도를 단순하게 테스트로 표현하고 그 구조에 맞게 코드를 작성. 단순성과 명쾌함이 좋은 구조를 낳는다
- **Mock Object 패턴**으로 테스트 격리 — 데이터베이스, 출력 장치 같은 외부 의존성을 인터페이스 뒤로 분리하고 테스트 스텁으로 대체
- **운 좋게 얻은 분리** — 테스트 가능성에서 출발한 분리 작업이 **프로그램 전체 구조에 이득이 되는 분리**가 된다
- **단위 테스트(화이트박스) vs 인수 테스트(블랙박스)**:
  - 단위 테스트는 작은 단위의 설계 의사결정에 영향
  - 인수 테스트는 큰 단위의 **아키텍처 의사결정**에 영향
- **인수 테스트는 자동화되어야 한다** — 수동 인수 테스트는 분리 압박을 잃어버려 아키텍처가 망가진다
- **운 좋게 얻은 아키텍처** — 인수 테스트가 강제한 입출력 분리가 좋은 아키텍처로 이어진다 (급여 관리 + XML 트랜잭션 예)
- **시스템은 퇴보하지 않게만 하면 된다** — 자주 실행하는 테스트가 시스템 품질의 하한선을 지킨다
- 테스트의 가장 큰 이점은 검증이 아니라 **아키텍처와 설계, 그리고 문서화**다

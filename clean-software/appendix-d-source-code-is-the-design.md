# Appendix D: Source Code Is the Design (소스 코드가 곧 설계다)

## 핵심 질문

소프트웨어 프로젝트에서 진짜 "공학 설계 문서"라고 부를 만한 것은 무엇인가? 다리나 비행기를 설계하는 엔지니어와, 코드를 작성하는 프로그래머는 같은 일을 하는가, 다른 일을 하는가? 만약 소스 코드가 설계라면, 우리가 "설계 단계"와 "구현 단계"를 분리하려는 시도는 왜 번번이 실패하는가?

> 본 부록은 잭 리브스(*Jack Reeves - 1992년 「C++ Journal」에 'What Is Software Design?'을 기고한 소프트웨어 엔지니어*)가 1992년에 쓴 에세이 「소프트웨어 설계란 무엇인가?(What Is Software Design?)」와, 그가 약 10년 뒤(2002년)에 덧붙인 후기를 옮긴 것이다. 본 책 Chapter 7 「애자일 설계란?」의 출발점이 된 핵심 사상의 원전이다.

---

## 1. 잭 리브스가 던진 질문 — 어떤 문서가 진짜 공학 문서인가?

리브스는 한 패널 토론에서 "소프트웨어 개발자가 엔지니어인가?"라는 질문을 받고 답을 고민하다가, 오래전 잡지에서 읽은 한 문장을 떠올렸다.

> 공학 프로세스의 최종 결과물은 "문서"다. 엔지니어는 물건이 아니라 문서를 만든다. 다른 사람들이 그 문서를 받아 실제 물건을 만든다.

이 정의를 받아들이면 자연스러운 후속 질문이 생긴다. **"보통 소프트웨어 프로젝트가 만들어내는 모든 문서 중에서, 진짜로 공학 문서로 여길 만한 것은 무엇인가?"**

리브스의 답은 단호하다.

> **핵심 통찰**: 진짜로 공학 설계의 기준을 만족시키는 소프트웨어 문서는 **오직 소스 코드 목록(source code listing) 하나뿐**이다. UML 다이어그램, 구조 차트, 부치(Booch) 다이어그램, PDL(*Program Design Language - 의사 코드 스타일의 설계 표기법*) 등은 설계의 일부이거나 메모일 수 있지만, 실제 공학 설계 문서는 아니다.

이 한 줄짜리 가정이 모든 것을 뒤집는다.

## 2. 하드웨어 공학과 소프트웨어 공학의 대응 관계 — 그동안 틀린 짝을 짓고 있었다

전통적인 공학 활동의 구조는 다음과 같다:

| 단계 | 하는 일 | 비용 |
|------|---------|------|
| 설계(Design) | 엔지니어가 어떤 것을 만들지 문서로 명세 | **매우 비쌈** |
| 제작(Construction) | 제작 노동자가 그 문서대로 실제 물건을 빌드 | 비용이 들지만 자동화 가능 |

소프트웨어 업계는 오랫동안 이 구조를 그대로 가져와서 다음과 같이 대응시켜 왔다:

- **설계(Design)** ≈ "소프트웨어 설계 문서" (요구 명세, 구조 차트, UML 등)
- **제작(Construction)** ≈ 프로그래밍(코딩)

> **핵심 통찰**: 이 대응이 잘못되었다. 소프트웨어에서 "제작"에 해당하는 것은 **컴파일과 링킹**이다. 컴파일러와 링커가 설계 문서(소스 코드)를 받아서 실제 동작하는 산출물(실행 파일)을 만들어낸다. 그리고 이 "제작"은 거의 공짜다 — 빌드 환경만 갖춰져 있다면 컴퓨터 한 대와 컴파일러 하나면 끝난다.

올바른 대응은 다음과 같다:

| 하드웨어 공학 | 소프트웨어 공학 |
|---------------|-----------------|
| 청사진(Blueprint) | **소스 코드** |
| 공장 / 건설 현장 | 컴파일러 + 링커 |
| 설계 검증 (풍동 시험, 시뮬레이션, 견본 제작) | **빌드 / 테스트 주기** |
| 시제품(Prototype) | 빠른 프로토타이핑, 빌드 / 테스트 사이클 |

이 재대응이 가져오는 충격적인 결론이 몇 개 있다.

---

## 3. 충격적 결론 1 — 소프트웨어 "제작"은 거의 공짜다

소프트웨어 제작 비용이 0에 가깝다는 사실은 너무 당연해서 대부분의 소프트웨어 조직에 **완벽한 맹점**이 되어 왔다.

- 50,000라인 C++ 프로젝트를 컴파일하는 데 시간이 영원처럼 느껴질 수도 있다.
- 그러나 동등한 복잡성의 하드웨어 시스템을 **물리적으로 빌드**하는 데 걸리는 시간과 비용을 생각하면, 이는 무료에 가깝다.
- 빌드한 결과물이 마음에 들지 않아 폐기해도, 소비된 자원은 완벽하게 회수된다.

이로부터 따라오는 첫 번째 뜻밖의 결론:

> **핵심 통찰**: **다른 어떤 검증 방법보다도, 그냥 설계를 빌드하고 테스트해 보는 것이 가장 싸고 빠르다.** 빌드 횟수는 상관없다. 정형 검증(*Formal Verification - 수학적 증명으로 프로그램 정확성을 입증하는 방법*) 같은 엄밀한 방법이 소프트웨어 산업에서 큰 관심을 끌지 못한 이유가 여기에 있다 — 증명하는 것보다 빌드해서 테스트하는 편이 훨씬 싸다.

---

## 4. 충격적 결론 2 — 소프트웨어 "설계"는 믿을 수 없을 만큼 비싸다

복잡한 상용 소프트웨어 시스템은 일반적으로 수십만~수백만 라인의 코드로 구성된다. 그리고 그 모든 라인이 **설계**다.

| 측면 | 소프트웨어 설계 |
|------|-----------------|
| 복잡성 | 일반 상용 SW: 10만~100만 라인. 게다가 계속 진화 |
| 변경 빈도 | 명세가 설계 중에도 빠르고 자주 바뀜 |
| 의존성 | 하드웨어, 라이브러리, OS, 외부 API 등 수많은 외부 요인에 의존 |
| 검증 | 정형 증명 거의 불가 → 빌드/테스트로 검증 |
| 본질 | **사회적/유기적 시스템과 더 유사** (단순한 청사진보다 훨씬 복잡) |

소프트웨어 설계는 **만들기는 (물리적으로는) 쉽지만, 제대로 만들기는 극도로 어렵다**. 50~100라인짜리 모듈 하나를 "작성"하는 데는 며칠이면 충분하지만, "완전히 디버깅된 상태로 만드는 것"은 또 다른 문제다 — 그것은 사실 설계를 다듬는 일이다.

---

## 5. 충격적 결론 3 — 코딩도, 테스트도, 디버깅도 모두 "설계"다

전통적인 소프트웨어 개발 프로세스는 단계를 분리한 칸막이를 만들려고 한다:

```
[요구사항] → [최상위 설계] → [상세 설계] → [코딩] → [테스트] → [완료]
   ↑ 동결           ↑ 동결        ↑ 동결       ↑ 노동       ↑ 검수
```

리브스가 보기에 이 그림은 **현실을 부정**한다. 진짜 그림은 다음과 같다:

```
[요구사항] ↔ [최상위 설계] ↔ [상세 설계] ↔ [코딩] ↔ [테스트] ↔ [디버깅]
   ←──────────── 모두 동등하게 "설계 활동" ────────────→
                        ↑ 모두 서로 영향을 주고받음
```

> **핵심 통찰**: 소프트웨어 개발의 압도적인 문제는 **모든 것이 설계 프로세스의 일부**라는 점이다. 코딩도 설계이고, 테스트와 디버깅도 설계의 일부이며, 우리가 보통 "소프트웨어 설계"라고 부르는 것도 설계의 일부다. 설계의 어떤 측면이 상세화 프로세스에서 빠진 채로 남아있다면, 최종 설계가 엉터리이거나 동작하지 않아도 놀랍지 않다.

테스트가 "공학"으로 여겨지지 않는 이유는, 단지 소프트웨어 업계가 **코드를 설계로 받아들이기를 거부**하기 때문이지, 다른 공학과 본질적인 차이가 있어서가 아니다. 하드웨어 엔지니어가 풍동에서 모델을 테스트하는 일과, 소프트웨어 엔지니어가 빌드/테스트 주기를 돌리는 일은 **같은 종류의 활동**이다 — 설계를 검증하고 다듬는 일.

## 6. 계층 사이에 위계는 없다 — 모든 측면이 똑같이 치명적

흔히 사람들은 "최상위 설계는 중요하고, 모듈 내부 알고리즘은 사소하다"고 생각한다. 리브스는 단호하게 반박한다.

> **핵심 통찰**: 소프트웨어 설계의 서로 다른 측면들 사이에는 **중요성의 계층 구조가 없다**. 최하위 모듈 수준에서의 잘못된 설계는 최상위 수준의 착오만큼이나 치명적이다. 소프트웨어 설계는 모든 측면에서 완벽하고 올바른 것이어야 한다. 그렇지 않으면 그 설계 위에 만들어진 모든 것이 잘못된 것이 된다.

이는 "프로그래머가 알고리즘만 정확히 짜면 된다"는 안일한 분업관과 정면으로 충돌한다. 모듈 내부의 한 알고리즘 선택이 전체 시스템의 성공/실패를 결정할 수 있다.

### 6.1 머피의 법칙과 파급 효과

리브스가 자기 경험으로 든 예시:

> 최근에 일한 프로젝트에서 모듈 A 내부와 다른 모듈 B 간의 타이밍 의존성이 발견됐다. 불행하게도, A의 내부는 추상화 뒤에 숨겨져 있어서 정상적인 순서로 B 호출과 연동할 방법이 없었다. 문제가 발견됐을 때는 A의 추상화를 변경하기에 너무 늦어 있었다. 그 뒤에 벌어진 일은 A의 내부 설계에 적용된 점점 더 복잡해지는 "픽스(fix)들"이었다. 인스톨 버전 1을 완성하기 전에, 설계가 망가졌다는 공통적 느낌이 퍼져 있었다. 모든 새 픽스는 더 오래된 픽스를 망가뜨리곤 했다. 결국 나와 동료들은 설계 변경을 주장했지만, 관리자가 동의하게 하기 위해 무급 초과 근무에 자원해야만 했다.

이것이 "보통의 소프트웨어 개발 프로젝트"의 모습이다. 한 계층의 양상은 다른 계층에 파문을 일으킨다.

## 7. 도구와 표기법은 설계가 아니다 — 무엇이 살아남았나

리브스는 소프트웨어 업계가 시도해 온 도구·표기법을 회고하며 흥미로운 관찰을 한다.

| 도구 / 기법 | 결과 |
|-------------|------|
| 구조화 프로그래밍, 파스칼(Pascal) | **성공** (대중화됨) |
| 객체지향, C++ | **성공** (당시 폭발적 인기) |
| CASE 툴(*Computer-Aided Software Engineering - 다이어그램·코드 생성을 자동화하는 통합 도구*) | 일부 인기, 보편화 실패 |
| 구조 차트 | 보편화 실패 |
| 워너-오(Warnier-Orr) 다이어그램 | 실패 |
| 부치(Booch) 다이어그램, 객체 다이어그램 | 실패 |
| PDL | 그나마 널리 퍼진 유일한 "설계 표기법" |

성공한 것들의 공통점: **모두 프로그래밍 언어이거나, 프로그래밍 언어로 옮길 수 있는 것**. 실패한 것들의 공통점: **모두 그림이고, 실제 소프트웨어 설계가 아니었다**.

> **핵심 통찰**: 소프트웨어 업계의 집단적 잠재의식은 이미 알고 있다 — **프로그래밍 언어의 발전이 다른 어떤 도구의 발전보다 압도적으로 중요하다**. 더 표현적인 프로그래밍 언어를 쓸 수 있다면 개발자들은 그것을 받아들인다. 그것이 진짜 설계 도구이기 때문이다.

리브스가 1992년에 C++를 주목한 이유가 여기에 있다. C++는 상위 수준의 설계 정보(추상화, 다형성, 캡슐화)를 코드에 **직접 표현**할 수 있게 해 주는 언어였다.

## 8. 코드 작성 전의 문서 vs. 코드 작성 후의 문서

리브스는 모든 프로그래머가 경험적으로 알고 있는 사실을 짚는다.

> 모든 프로그래머는 코드 작성 **이전**이 아니라 **이후**에 소프트웨어 설계 문서를 작성하는 편이 훨씬 정확한 문서를 만들어낸다는 사실을 알고 있다.

이유는 명백하다. 코드에 반영된 최종 설계는 빌드/테스트 주기에서 다듬어진 **단 하나의 설계**다. 처음 그린 설계가 이 주기 동안 바뀌지 않았을 확률은 모듈 수와 프로그래머 수에 반비례하며, 빠르게 0에 수렴한다.

이 사실은 흔히 "프로그래머가 게을러서 문서를 안 쓴다"로 오해되지만, 실제로는 **문서가 코드보다 정확할 수 없다**는 구조적 진실이다.

## 9. 빠른 프로토타이핑·나선형 개발의 정체

리브스는 폭포수 모델(*Waterfall - 분석→설계→구현→테스트가 일방적으로 진행되는 전통적 모델*)에서 나선형 개발(*Spiral - 위험 분석과 점진적 개발을 결합한 모델*), 빠른 프로토타이핑으로 진화해 온 흐름을 다음과 같이 해석한다.

> **핵심 통찰**: "위험 완화", "단축된 출하 시간" 같은 정당화는 표면적 이유다. 본질적으로 이런 기법들은 **생명주기에서 코딩을 더 일찍 시작하는 것에 대한 변명**이다. 그리고 이는 옳다 — 빌드/테스트 주기는 설계를 검증하는 핵심 활동이므로, 일찍 시작할수록 좋다.

이 통찰은 1992년에 나왔지만, 2001년의 애자일 선언(*Agile Manifesto - 동작하는 소프트웨어, 변화에의 반응 등을 강조한 4가지 가치 선언*)과 정확히 같은 방향을 가리킨다.

## 10. 보조 문서(architecture documentation)의 역할

소스 코드가 설계라면, 그 외의 모든 문서(UML 다이어그램, 아키텍처 텍스트, 사용자 매뉴얼)는 **보조 문서**다. 그러나 보조 문서가 무용한 것은 아니다. 리브스는 두 가지 핵심 용도를 짚는다.

| 용도 | 내용 |
|------|------|
| 1. 문제 도메인 정보 보존 | 소프트웨어 설계는 문제 도메인의 개념을 모델화한다. 모델화되지 않은 도메인 지식은 어딘가에 기록되어야 한다 — 나중에 모델을 변경할 때 참조하기 위해 |
| 2. 설계에서 추출하기 어려운 측면 문서화 | 핵심 클래스 관계, 아키텍처 의도 등 — 그림으로 묘사하는 편이 더 좋은 부분 |

다만 보조 문서는 **소스 코드를 대체하지 않으며**, **소스 코드 작성 이전이 아니라 동시에 또는 직후**에 만드는 편이 정확하다.

## 11. 2002년 후기 — 10년 뒤의 보강

리브스는 2002년에 자신의 1992년 글을 돌아보며 후기를 덧붙였다. 핵심 요점:

### 11.1 자신의 주장에 더 확신을 갖게 됨

10년 사이에 그의 신념을 강화한 사건들:

- **객체지향 언어의 인기 확산**: C++뿐 아니라 Java, C#, Python 등 다수 등장. UML 같은 객체지향 표기법도 자리 잡음
- **리팩토링(*Refactoring - 외부 동작을 바꾸지 않으면서 내부 구조를 개선하는 활동*) 개념의 등장**: "설계의 모든 측면은 유연해야 하고, 확인하면서 변경할 수 있어야 한다"는 그의 주장과 정확히 대응
- **애자일 개발 방식, 특히 XP(*eXtreme Programming - 짧은 주기, 페어 프로그래밍, TDD 등을 강조하는 애자일 방법론*)**: "소스 코드가 소프트웨어 개발의 가장 중요한 산출물"이라는 인식을 공유

### 11.2 더 강조하고 싶어진 부분 — 아키텍처의 중요성

> 1992년의 나는 조금 순진했다. 빌드/테스트 주기가 아키텍처 문제를 밝힐 수도 있지만, 보통은 요구사항 변경에 의해 더 많은 문제가 드러난다. 그리고 일단 프로젝트가 아키텍처 주위에 상당한 양의 코드를 만들고 나면, 아키텍처를 변경하는 일은 프로젝트를 폐기하고 다시 시작하는 것과 다름없다. 따라서 **초반에 제대로 만드는 것이 중요하다**.

이 후기에서 그는 **소프트웨어 디자인 패턴**(*Design Pattern - 자주 등장하는 설계 문제에 대한 검증된 해결책의 카탈로그*)이 이 문제의 한 해답이라고 본다. Robert C. Martin이 이 부록을 본 책에 실은 이유가 여기에 명확히 드러난다 — 본 책의 나머지 부분(SOLID 원칙, 패키지 설계 원칙, 디자인 패턴)은 모두 "초반에 아키텍처를 제대로 만들기 위한 도구"를 다룬다.

### 11.3 보조 문서에 대한 입장 변화

1992년에는 "소스 코드에서 자동으로 보조 문서를 추출하는 도구"가 등장하기를 희망했지만, 2002년에는 그 희망을 거의 포기했다.

> 좋은 객체지향 아키텍처는 보통 몇 개의 다이어그램과 몇십 장의 텍스트로 기술할 수 있다. 이 다이어그램과 텍스트는 설계의 **핵심 클래스와 관계**에 집중해야 한다. 소프트웨어 도구가 소스 코드의 구체적 부분들 덩어리에서 이런 중요한 면을 추출해 낼 만큼 똑똑해질 현실적 희망은 없다. 사람이 작성하고 유지해야 한다.

이는 본 책 Chapter 1에서 마틴이 제시한 "마틴의 문서화 제1법칙(*필요가 급박하고 중요하지 않다면 아무 문서도 만들지 마라*)"과 같은 결론에 도달한다 — 문서는 짧고 요약적이며 핵심에 집중해야 한다.

## 12. 리브스 에세이의 12가지 요약 명제

리브스가 본인의 글 끝에 정리한 명제 목록을 옮긴다. 본 책 전체의 정신적 토대다.

1. **실제 소프트웨어는 컴퓨터에서 실행된다.** 자기 매체에 저장된 1과 0의 연속이다. C++(혹은 그 밖의 언어)로 된 프로그램 목록이 아니다.
2. **프로그램 목록은 소프트웨어 설계를 표현하는 문서다.** 컴파일러와 링커가 실제 소프트웨어를 빌드한다.
3. **실제 소프트웨어를 만드는 데는 비용이 적게 든다.** 컴퓨터가 빨라지면서 더 싸지고 있다.
4. **실제 소프트웨어를 설계하는 데는 믿을 수 없을 만큼 비용이 많이 든다.** 소프트웨어가 복잡하고, 모든 단계가 설계 프로세스의 일부이기 때문이다.
5. **프로그래밍은 설계 활동이다.** 좋은 프로세스는 이를 인식하고 적절할 때 코딩을 망설이지 않는다.
6. **코딩은 보통 믿는 것보다 더 적절하다.** 설계를 코드로 나타내는 일은 실수와 추가 설계 필요성을 드러낸다. 일찍 할수록 설계가 좋아진다.
7. **정형 공학 검증 방법은 실제 개발에서 잘 안 쓰인다.** 증명보다 빌드/테스트가 더 싸고 쉽기 때문이다.
8. **테스트와 디버깅은 설계 활동이다.** 다른 공학의 검증·상세화 프로세스에 해당한다. 좋은 프로세스는 이를 생략하지 않는다.
9. **그 밖의 설계 활동이 있다.** 최상위 수준 설계, 모듈 설계, 아키텍처 설계 등. 좋은 프로세스는 이를 의도적으로 포함시킨다.
10. **모든 설계 활동은 상호작용한다.** 좋은 프로세스는 때로 완전히 송두리째 설계가 바뀌는 것도 허용한다.
11. **여러 표기법은 잠재적으로 유용하다** — 보조 문서·도구로서. 이것들이 소프트웨어 설계는 아니다.
12. **소프트웨어 개발은 아직 수공업에 가깝다.** 설계를 검증·개선하는 프로세스에서의 엄밀성 부족 때문이다.

마지막으로 그는 한 마디를 덧붙인다: 소프트웨어 개발의 진짜 진보는 **프로그래밍 언어의 진보**에 의존한다.

## 13. 이 부록이 본 책에서 차지하는 위치

본 부록은 본 책의 결론을 우회적으로 다시 확인시켜 준다.

| 본 책의 주장 | 잭 리브스의 대응 |
|--------------|------------------|
| 애자일은 "최소한의 사전 설계 + 지속적인 리팩토링" | 코드가 설계이며, 모든 단계가 설계의 일부 |
| SOLID 원칙, 디자인 패턴은 "코드 안에 표현되는 설계" | 진짜 설계 문서는 소스 코드뿐 |
| 빌드/테스트 주기, TDD가 핵심 | 빌드/테스트는 설계 검증의 본질 |
| UML은 보조 도구, 코드보다 우선하지 않음 | UML은 보조 문서, 진짜 설계가 아님 |
| 짧고 요약적인 아키텍처 문서 | 핵심 클래스/관계만 다이어그램으로 기록 |

본 책 Chapter 7 「애자일 설계란?」의 도입부에서 마틴은 이 에세이를 인용하며 "설계는 코드에서 떠나지 않는다"는 자신의 입장을 밝혔다. 본 부록은 그 입장의 **원전**이자 **근거**다.

## 요약

- **공학의 최종 산출물은 문서다.** 소프트웨어 프로젝트에서 진짜 공학 문서로 부를 만한 것은 **소스 코드 목록 단 하나**뿐이다
- **하드웨어 공학과의 올바른 대응**: 청사진 = 소스 코드 / 공장 = 컴파일러+링커 / 풍동 시험 = 빌드/테스트 주기
- **소프트웨어 "제작"은 거의 공짜다** — 컴파일과 링킹이 제작에 해당하기 때문이다. 그래서 "그냥 빌드해서 테스트해 보는 것"이 가장 싸고 빠른 검증 방법이 된다
- **소프트웨어 "설계"는 믿을 수 없을 만큼 비싸다** — 코드의 모든 라인이 설계이고, 코딩·테스트·디버깅 모두가 설계 활동이기 때문이다
- **단계 사이에 위계가 없다** — 최하위 모듈의 알고리즘 선택은 최상위 아키텍처 결정만큼 치명적이다
- **도구·표기법은 설계가 아니다** — 살아남은 것은 모두 프로그래밍 언어(또는 그 변형), 그림 표기법은 보조 도구일 뿐
- **코드 작성 이후의 문서가 코드 작성 이전의 문서보다 정확하다** — 빌드/테스트 주기를 거친 단 하나의 설계가 코드에 반영되기 때문
- **빠른 프로토타이핑·나선형·애자일은 모두 "코딩을 일찍 시작하기 위한 방법"** — 이는 옳다. 빌드/테스트 주기가 설계 검증의 본질이기 때문
- **보조 문서(아키텍처 다이어그램, 도메인 정보)는 필요하다** — 단, 짧고 핵심에 집중해야 하며 사람이 직접 작성·유지해야 한다
- **2002년 후기의 보강**: 객체지향 언어 확산, 리팩토링, 애자일, XP가 모두 이 주장을 강화. 다만 아키텍처는 초반에 "어느 정도 제대로" 잡아야 하며, 이를 위한 도구가 **디자인 패턴**이다
- 본 책 전체(SOLID, 패키지 원칙, 디자인 패턴)는 이 부록의 명제 위에 서 있다 — **코드가 설계라면, 좋은 코드를 만드는 원칙이 곧 좋은 설계의 원칙이다**

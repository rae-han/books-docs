# 헤드 퍼스트 디자인 패턴 (Head First Design Patterns, 2nd Edition)

> *Head First Design Patterns* (2nd Edition, Eric Freeman & Elisabeth Robson with Kathy Sierra & Bert Bates, O'Reilly, 2021)
> 한국어판: 『헤드 퍼스트 디자인 패턴 (개정판)』 (서환수 옮김, 한빛미디어, 2022)

GoF 디자인 패턴을 Head First 시리즈 특유의 **시각적이고 대화체인 방식**으로 풀어낸 입문서. 오리 시뮬레이터·스타버즈 커피·피자 가게·뽑기 기계 같은 친근한 예제로 14가지 핵심 패턴과 9가지 객체지향 디자인 원칙을 학습한다.

## 개요

Head First 시리즈는 인지과학·학습 이론에 기반한다 — 그냥 읽지 말고 연습문제를 직접 풀고, 코드를 타이핑하고, "뇌 단련" 질문에 스스로 답하며 능동적으로 학습하도록 설계됐다. 핵심 메시지는 두 가지다: **패턴은 코드가 아니라 경험의 재사용**이며, 개발자 사이의 **공유 어휘**라는 것이 패턴의 가장 큰 가치라는 것.

이 노트는 원서 Java 예제를 **TypeScript 기본(펼침) + Java `<details>` 접기**로 병기하고, JS/TS 고유 관점(내장 `Iterator`·`Proxy`·프로토타입·판별 유니언·ES 모듈=싱글턴 등)을 적극 반영했다. 본문은 학습에 유용한 요소(패턴 정의·원칙 콜아웃·Q&A·연습문제)만 선별하고, 원서의 재미 요소(낱말 퀴즈·방구석 토크·5분 드라마)는 각 챕터 끝 `## 보너스` 절에 모아 보존했다.

## 학습 가이드

1. **Ch1~4는 순서대로** — 원칙 1~6이 여기서 도입되고, 이후 모든 패턴 논의의 어휘가 된다
2. **Ch5~12는 관심 패턴부터 골라 읽어도 무방** — 각 장이 독립적. 단 Ch12(복합·MVC)는 Ch2(옵저버)·Ch1(전략)·Ch9(컴포지트) 이후에
3. **Ch13은 마무리 필독** — 패턴을 '언제 쓰지 말아야 하는가'까지 포함한 메타 관점
4. **헷갈리는 패턴은 비교 절 활용**:
   - **전략 vs 상태** (구조 동일): 클라이언트가 알고리즘 선택 vs Context가 상태에 따라 전환 → [Ch10](ch10-state-pattern.md)
   - **데코레이터 vs 어댑터 vs 퍼사드 vs 프록시** (모두 "감싸기"): 기능 추가 / 인터페이스 변환 / 단순화 / 접근 제어 → [Ch7](ch07-adapter-facade-pattern.md)·[Ch11](ch11-proxy-pattern.md)
   - **팩토리 메서드 vs 추상 팩토리**: 상속(단일 제품) vs 구성(제품군) → [Ch4](ch04-factory-pattern.md)
   - **템플릿 메서드 vs 전략**: 상속(골격 고정) vs 구성(알고리즘 교체) → [Ch8](ch08-template-method-pattern.md)

## 핵심 개념 맵

- **패턴 = 공유 어휘**: 코드 재사용이 아니라 경험의 재사용 — "옵저버로 갑시다" 한마디의 힘 (Ch1·13)
- **원칙이 패턴을 낳는다**: 캡슐화(1)·인터페이스 지향(2)·구성(3)이 전략 패턴으로, DIP(6)가 팩토리로 — 9원칙이 14패턴의 뿌리
- **상속보다 구성**: 데코레이터·전략·상태가 모두 이 원칙의 변주 (Ch1·3·10)
- **감싸기 4형제**: 데코레이터(기능)·어댑터(변환)·퍼사드(단순화)·프록시(접근 제어) — 목적으로 구분한다
- **MVC = 복합 패턴**: 옵저버+전략+컴포지트 — React·Vue 등 현대 프런트엔드의 뿌리 (Ch12)
- **JS/TS에는 언어에 흡수된 패턴이 많다**: 반복자(`Symbol.iterator`)·프록시(`Proxy`)·싱글턴(ES 모듈)·커맨드(일급 함수)

## 9가지 객체지향 디자인 원칙 (이 책의 시그니처)

각 원칙은 챕터가 진행되며 도구상자에 하나씩 쌓인다. 모든 챕터에 `> **디자인 원칙 N**` 콜아웃으로 등장하고, 각 챕터 노트에 "디자인 원칙 정리(누적)" 섹션이 있다.

1. 애플리케이션에서 **달라지는 부분을 찾아내 캡슐화**하고, 달라지지 않는 부분과 분리한다. *(Ch1)*
2. **구현보다는 인터페이스**(상위 형식)에 맞춰서 프로그래밍한다. *(Ch1)*
3. **상속보다는 구성**을 활용한다. *(Ch1)*
4. 상호작용하는 객체 사이에서는 가능하면 **느슨한 결합**을 사용한다. *(Ch2)*
5. 클래스는 **확장에는 열려 있고 변경에는 닫혀 있어야** 한다 (OCP). *(Ch3)*
6. **추상화에 의존**하고 구상 클래스에 의존하지 않는다 (DIP). *(Ch4)*
7. **진짜 절친에게만** 이야기한다 (최소 지식 원칙 / 데메테르의 법칙). *(Ch7)*
8. **먼저 연락하지 마세요, 저희가 연락 드리겠습니다** (할리우드 원칙). *(Ch8)*
9. 어떤 클래스가 **바뀌는 이유는 하나뿐**이어야 한다 (단일 책임 원칙). *(Ch9)*

## TS/JS 개발자를 위한 관점

원서는 Java 기반이지만, 여러 패턴이 JS/TS에서는 **언어 차원에 내장**되어 있거나 프레임워크가 흡수했다. 각 챕터 노트에 이를 반영했다.

- **반복자**(Ch9): `Symbol.iterator`·제너레이터·`for...of` 내장.
- **프록시**(Ch11): `Proxy`/`Reflect` 내장 (Vue 반응성·MobX의 기반).
- **싱글턴**(Ch5): 단일 스레드라 동기화 불필요, ES 모듈 export가 곧 싱글턴.
- **옵저버 + 전략 + 컴포지트 = MVC**(Ch12): React·Vue 등 현대 프런트엔드의 뿌리.
- **커맨드**(Ch6): 함수 일급 객체 → 함수가 곧 커맨드(람다).
- **책임 연쇄**(Ch14): Express/Koa 미들웨어. **빌더**(Ch14): fluent API(쿼리 빌더). **프로토타입**(Ch14): JS 자체가 프로토타입 언어(`structuredClone`).

# Head First Design Patterns (헤드 퍼스트 디자인 패턴)

**에릭 프리먼 · 엘리자베스 롭슨 · 케이시 시에라 · 버트 베이츠** 지음 / 서환수 옮김 / 한빛미디어 (개정판, 2022)
원서: *Head First Design Patterns, 2nd Edition* (O'Reilly, 2021)

GoF 디자인 패턴을 Head First 시리즈 특유의 **시각적이고 대화체인 방식**으로 풀어낸 입문서다. 오리 시뮬레이터·스타버즈 커피·피자 가게·뽑기 기계 같은 친근한 예제로 14가지 핵심 패턴과 9가지 객체지향 디자인 원칙을 학습한다.

---

## 이 노트에 대하여

- **언어**: 한국어, 서술형 "-다" 체.
- **코드 병기**: 원서 예제 언어는 **Java**다. 이 노트는 **TypeScript를 기본(펼침)** 으로 쓰고, 원서 **Java 코드를 `<details>` 로 접어** 병기한다. 사용자가 TS 개발자이므로 이해를 돕고, JS/TS 고유의 관점(내장 `Iterator`·`Proxy`·프로토타입·판별 유니언 등)을 적극 반영했다.
- **본문 구성(①번 선별)**: 학습에 유용한 요소만 본문에 녹였다 — 패턴 정의 박스, **디자인 원칙 콜아웃(누적)**, 무엇이든 물어보세요(Q&A), 핵심 통찰, 연습문제+해답.
- **보너스 섹션**: 각 챕터 맨 뒤 `## 보너스`에 원서의 재미·복습 요소(낱말 퀴즈, 뇌 단련, 방구석 토크, 5분 드라마, 코드 자석 등)를 모아 보존했다. 본문 학습 흐름은 깔끔하게 유지하고, 게임·복습 요소는 끝에 별도로 배치한 것.

## Head First 학습법 (들어가며 요약)

Head First 시리즈는 인지과학·학습 이론에 기반해 만들어졌다. "들어가며(Chapter 0)"의 핵심 조언:

- **대상 독자**: 자바(또는 객체지향 언어)의 기초(클래스·상속·다형성·인터페이스)를 이미 아는 사람. 패턴을 처음 배우는 사람.
- **능동적 학습**: 그냥 읽지 말고, **연습문제를 직접 풀고**, 코드를 타이핑하고, "뇌 단련" 질문에 스스로 답해 보라. 뇌는 능동적으로 씨름한 것을 오래 기억한다.
- **메타인지**: "내가 지금 무엇을, 왜 배우고 있는가"를 의식하라. 패턴을 **머릿속에 넣은 뒤** 애플리케이션에 어떻게 적용할지 파악하는 것이 핵심이다.
- **패턴은 코드가 아니라 경험의 재사용**이다. 그리고 개발자 사이의 **공유 어휘(vocabulary)** 다 — 이것이 패턴의 가장 큰 가치다.

## 9가지 객체지향 디자인 원칙 (이 책의 시그니처)

각 원칙은 챕터가 진행되며 도구상자에 하나씩 쌓인다. 모든 챕터에 `> **디자인 원칙 N**` 콜아웃으로 등장하고, 각 챕터 노트에 "디자인 원칙 정리(누적)" 섹션이 있다.

1. 애플리케이션에서 **달라지는 부분을 찾아내 캡슐화**하고, 달라지지 않는 부분과 분리한다. *(Ch1)*
2. **구현보다는 인터페이스**(상위 형식)에 맞춰서 프로그래밍한다. *(Ch1)*
3. **상속보다는 구성**을 활용한다. *(Ch1)*
4. 상호작용하는 객체 사이에서는 가능하면 **느슨한 결합**을 사용한다. *(Ch2)*
5. 클래스는 **확장에는 열려 있고 변경에는 닫혀 있어야** 한다 (OCP). *(Ch3)*
6. **추상화에 의존**하고 구상 클래스에 의존하지 않는다 (DIP). *(Ch4)*
7. **진짜 절친에게만** 이야기한다 (최소 지식 원칙 / 데메테르의 법칙). *(Ch7)*
8. **먼저 연락하지 마세요, 저희가 연락 드리겠습니다** (할리우드 원칙). *(Ch8)*
9. 어떤 클래스가 **바뀌는 이유는 하나뿐**이어야 한다 (단일 책임 원칙). *(Ch9)*

## 헷갈리기 쉬운 패턴 비교

패턴 학습에서 가장 자주 혼동되는 조합들:

- **전략 vs 상태** (구조 동일): 전략=클라이언트가 알고리즘 선택 / 상태=Context가 내부 상태에 따라 전환. → [Ch10](ch10-state-pattern.md)
- **데코레이터 vs 어댑터 vs 퍼사드 vs 프록시** (모두 "감싸기"): 데코레이터=기능 추가 / 어댑터=인터페이스 변환 / 퍼사드=인터페이스 단순화 / 프록시=접근 제어. → [Ch7](ch07-adapter-facade-pattern.md), [Ch11](ch11-proxy-pattern.md)
- **팩토리 메서드 vs 추상 팩토리**: 상속(단일 제품) vs 구성(제품군). → [Ch4](ch04-factory-pattern.md)
- **템플릿 메서드 vs 전략**: 상속(골격 고정) vs 구성(알고리즘 교체). 팩토리 메서드는 템플릿 메서드의 특수형. → [Ch8](ch08-template-method-pattern.md)

## TS/JS 개발자를 위한 관점

원서는 Java 기반이지만, 여러 패턴이 JS/TS에서는 **언어 차원에 내장**되어 있거나 프레임워크가 흡수했다. 각 챕터 노트에 이를 반영했다.

- **반복자**(Ch9): `Symbol.iterator`·제너레이터·`for...of` 내장.
- **프록시**(Ch11): `Proxy`/`Reflect` 내장 (Vue 반응성·MobX의 기반).
- **싱글턴**(Ch5): 단일 스레드라 동기화 불필요, ES 모듈 export가 곧 싱글턴.
- **옵저버 + 전략 + 컴포지트 = MVC**(Ch12): React·Vue 등 현대 프런트엔드의 뿌리.
- **커맨드**(Ch6): 함수 일급 객체 → 함수가 곧 커맨드(람다).
- **책임 연쇄**(Ch14): Express/Koa 미들웨어. **빌더**(Ch14): fluent API(쿼리 빌더). **프로토타입**(Ch14): JS 자체가 프로토타입 언어(`structuredClone`).

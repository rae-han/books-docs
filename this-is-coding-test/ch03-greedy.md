# Chapter 03: 그리디 (Greedy)

> "현재 상황에서 가장 좋아 보이는 것"을 매 단계 선택해서 답을 구하는 알고리즘 유형. 코딩 테스트에서 가장 자주 나오는 첫 관문이다.

## 핵심 질문

- 매 단계 **국소 최적(local optimum)** 을 선택했을 때, 정말 **전역 최적(global optimum)** 에 도달하는가?
- "이 문제는 그리디로 풀 수 있다"는 것을 어떻게 판단하는가?
- 그리디로 풀 수 없는 반례는 어떻게 식별하는가?

## 1. 그리디란 무엇인가

그리디 알고리즘은 **각 단계마다 그 시점에서 가장 좋아 보이는 선택**을 하는 방식이다. "탐욕법"이라고도 부른다.

특징:
- **단순함**: 정렬, 우선순위 큐, 큰 단위 우선 등 직관적인 규칙으로 접근
- **빠름**: 대부분 O(N log N) 이하
- **위험성**: 모든 문제에 적용되지 않는다. **정당성(correctness)** 검증이 필수

> 다익스트라 최단 경로 알고리즘(Ch 09), 크루스칼 최소 신장 트리(Ch 10)도 본질은 그리디다.

## 2. 도입 예제 — 거스름돈 문제

**문제**: 점원이 손님에게 거슬러 줘야 할 돈이 N원일 때, 동전 개수가 최소가 되도록 거슬러 주려면? (단, 동전은 500/100/50/10원만 있음)

### 그리디 아이디어

**가장 큰 단위의 화폐부터** 차례로 거슬러 준다.

예: N = 1,260원
1. 500원 × 2 = 1,000원 (남은 돈 260원)
2. 100원 × 2 = 200원 (남은 돈 60원)
3. 50원 × 1 = 50원 (남은 돈 10원)
4. 10원 × 1 = 10원 (남은 돈 0원)

→ 총 6개

### 코드

```python
# Python (책 원본)
n = 1260
count = 0
coin_types = [500, 100, 50, 10]
for coin in coin_types:
    count += n // coin
    n %= coin
print(count)
```

```ts
// TypeScript
let n = 1260;
let count = 0;
const coinTypes = [500, 100, 50, 10];
for (const coin of coinTypes) {
  count += Math.floor(n / coin);
  n %= coin;
}
console.log(count);
```

**시간복잡도**: O(K), K = 화폐 종류 수 (N과 무관!)

### 정당성 — 왜 큰 단위부터가 정답인가?

**가지고 있는 동전 단위가 항상 작은 단위의 배수**이기 때문이다 (500 = 100×5, 100 = 50×2, 50 = 10×5).

> **반례**: 동전이 500/400/100원인 경우, 800원 거스름돈을 그리디로 풀면 500+100+100+100 = 4개. 하지만 최적해는 400+400 = 2개. **그리디는 항상 옳은 게 아니다 — 정당성 검증이 핵심.**

## 3. 그리디로 풀 수 있는지 판단하는 법

| 신호 | 그리디 가능성 |
|------|---------------|
| "최소/최대" + "단순한 규칙" 키워드 | 높음 |
| 정렬한 뒤 한 방향으로 훑으면 될 듯 | 높음 |
| 큰 단위가 작은 단위의 배수 | 매우 높음 |
| 선택이 미래에 영향을 미치는 복잡한 의존성 | 낮음 — DP 의심 |
| 모든 경우를 살펴봐야 할 것 같음 | 낮음 — 완전 탐색/백트래킹 |

> **현장 팁**: 풀이가 떠오르면 **5~10초 안에 반례를 만들어보라**. 못 만들면 그리디로 풀고 정당성을 머릿속에 정리해 둔다.

## 4. 실전 문제 1 — 큰 수의 법칙

| 항목 | 값 |
|------|-----|
| 난이도 | ★ 하 |
| 풀이 시간 | 30분 |
| 시간 제한 | 1초 |
| 메모리 | 128MB |
| 기출 | 2019 국가교육기관 |

### 문제 요약

배열의 수들을 M번 더해 가장 큰 수를 만들되, **연속해서 K번 초과로 같은 인덱스를 더할 수 없다**.

예: `[2, 4, 5, 4, 6]`, M=8, K=3 → `6+6+6+5 + 6+6+6+5` = **46**

### 핵심 아이디어

가장 큰 수와 두 번째로 큰 수만 알면 된다. 가장 큰 수를 K번, 두 번째로 큰 수를 1번씩 반복.

### 단순 풀이 (M이 작을 때)

```ts
const [n, m, k] = "5 8 3".split(" ").map(Number);
const data = "2 4 5 4 6".split(" ").map(Number);
data.sort((a, b) => a - b);
const first = data[n - 1];        // 가장 큰 수
const second = data[n - 2];       // 두 번째로 큰 수

let result = 0;
let remaining = m;
while (remaining > 0) {
  for (let i = 0; i < k && remaining > 0; i++) {
    result += first;
    remaining--;
  }
  if (remaining > 0) {
    result += second;
    remaining--;
  }
}
console.log(result);  // 46
```

**시간복잡도**: O(M)

### 최적화 — 수학적 풀이 (M이 클 때)

반복 패턴: `[first × K, second × 1]` 의 길이 (K+1)인 수열이 반복.

- 가장 큰 수가 더해지는 횟수 = `Math.floor(M / (K+1)) * K + M % (K+1)`
- 두 번째로 큰 수가 더해지는 횟수 = M − 위 값

```ts
const [n, m, k] = "5 8 3".split(" ").map(Number);
const data = "2 4 5 4 6".split(" ").map(Number);
data.sort((a, b) => a - b);
const first = data[n - 1];
const second = data[n - 2];

const firstCount = Math.floor(m / (k + 1)) * k + (m % (k + 1));
const secondCount = m - firstCount;

console.log(first * firstCount + second * secondCount);  // 46
```

**시간복잡도**: O(N log N) — 정렬이 지배적, 이후는 O(1)

::: callout {icon="⚠️" color="gray_bg"}
**함정**: M이 100억 이상이면 단순 풀이는 시간 초과. **N, M, K의 입력 범위를 항상 먼저 보고** 단순/최적화 갈림길을 정할 것.
:::

## 5. 실전 문제 2 — 숫자 카드 게임

| 항목 | 값 |
|------|-----|
| 난이도 | ★ 하 |
| 시간 제한 | 1초 |
| 메모리 | 128MB |
| 기출 | 2019 국가교육기관 |

### 문제 요약

N×M 카드판에서 (1) 행을 고른 뒤 (2) 그 행의 **최솟값**을 뽑는다. 최종적으로 **가장 큰** 수를 뽑으려면?

### 핵심 아이디어

**각 행의 최솟값들 중 최댓값** = 답. 즉 `max(min(row) for each row)`.

### 코드

```ts
// 입력: lines[0] = "3 3", lines[1..3] = "3 1 2", "4 1 4", "2 2 2"
const lines = ["3 3", "3 1 2", "4 1 4", "2 2 2"];
const [n, m] = lines[0].split(" ").map(Number);

let result = 0;
for (let i = 1; i <= n; i++) {
  const row = lines[i].split(" ").map(Number);
  const minOfRow = Math.min(...row);
  result = Math.max(result, minOfRow);
}
console.log(result);  // 2
```

**시간복잡도**: O(N × M)

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 문제 설명이 길어도 핵심은 단 한 줄 — `max-of-min`. 그리디 문제는 **문제를 한 문장으로 압축**하는 연습이 중요하다.
:::

## 6. 실전 문제 3 — 1이 될 때까지

| 항목 | 값 |
|------|-----|
| 난이도 | ★ 하 |
| 시간 제한 | 1초 |
| 메모리 | 128MB |
| 기출 | 2018 E기업 알고리즘 대회 |

### 문제 요약

수 N이 1이 될 때까지 다음 두 연산 중 하나를 선택해서 반복:
1. N에서 1을 뺀다
2. N을 K로 나눈다 (K로 나누어 떨어질 때만)

**최소 횟수**를 구하라. 예: N=25, K=5 → 답 2 (25 → 5 → 1).

### 핵심 아이디어

**나누기가 빼기보다 항상 빠르게 N을 줄인다** (K ≥ 2이므로). 따라서:
- 가능하면 K로 나눈다
- 나누어 떨어지지 않을 때만 1을 뺀다

### 단순 풀이

```ts
let [n, k] = "25 5".split(" ").map(Number);
let result = 0;

while (n >= k) {
  while (n % k !== 0) {
    n -= 1;
    result++;
  }
  n = Math.floor(n / k);
  result++;
}
while (n > 1) {
  n -= 1;
  result++;
}
console.log(result);  // 2
```

**시간복잡도**: O(log N) — N이 K로 계속 나뉘므로

### 최적화 — N이 100억일 때

`while (n % k !== 0) n -= 1;` 부분이 N이 매우 클 때 느릴 수 있다 (1씩 빼지 말고 한 번에).

```ts
let [n, k] = "25 5".split(" ").map(Number);
let result = 0;

while (true) {
  // n을 k의 배수까지 한 번에 깎기
  const target = Math.floor(n / k) * k;
  result += n - target;
  n = target;

  // 더 이상 나눌 수 없으면 종료
  if (n < k) {
    break;
  }

  // k로 한 번 나누기
  result++;
  n = Math.floor(n / k);
}

// 남은 부분 1씩 빼기
result += n - 1;
console.log(result);
```

**시간복잡도**: O(log N) — 나누기 횟수 자체가 로그 스케일

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: "1씩 빼기"를 보면 **수학적으로 한 번에 처리**할 수 있는지 의심하라. `n -= n % k` 한 줄로 끝난다.
:::

## 자주 하는 실수

| 실수 | 원인 | 해결 |
|------|------|------|
| 정당성 검증 없이 그리디 답안 제출 | 반례를 안 떠올림 | 5~10초 안에 반례 시도 |
| 정렬 시 `arr.sort()` 사용 | 문자열 정렬됨 | `(a,b) => a-b` 명시 |
| `Math.min(...hugeArr)` 사용 | 스택 오버플로 | `arr.reduce(Math.min, Infinity)` 또는 for 루프 |
| 입력 범위 안 보고 단순 풀이 작성 | M, N이 큰 경우 시간 초과 | 입력 조건 먼저 보고 풀이 선택 |
| `n / k` 그대로 사용 | 실수 결과로 누적 오차 | `Math.floor(n / k)` 또는 `Math.trunc` |

## 요약

- 그리디 = 매 단계 국소 최적 선택. 단순하지만 **정당성 검증 필수**.
- 거스름돈처럼 **단위가 배수**일 때 안전. 800원/(500,400,100)처럼 배수 아닌 경우는 DP로.
- 큰 수의 법칙 → **반복 패턴 인식** + 수학으로 O(1) 처리
- 숫자 카드 게임 → `max-of-min` 한 줄
- 1이 될 때까지 → 가능한 한 **나누기 우선**, 1씩 빼기는 마지막
- 입력 범위(N, M, K)를 항상 먼저 본다 — 단순 풀이 vs 최적화 분기
- 다익스트라(Ch 09), 크루스칼(Ch 10)도 본질은 그리디 — 이 챕터의 사고방식이 후속 챕터에 계속 쓰인다

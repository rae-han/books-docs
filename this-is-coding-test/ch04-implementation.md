# Chapter 04: 구현 (Implementation)

> 머릿속 알고리즘을 정확하고 빠르게 소스코드로 옮기는 유형. "풀이는 떠오르는데 코드로 옮기기 어려운" 문제를 말하며, 흔히 완전 탐색과 시뮬레이션을 묶어 부른다.

## 핵심 질문

- 구현 유형은 왜 특별한 알고리즘 지식보다 **꼼꼼한 코드 작성력**을 요구하는가?
- 2차원 격자에서 상하좌우 방향 이동을 어떻게 **방향 벡터(dx/dy)** 로 우아하게 처리하는가?
- 완전 탐색(brute force)을 언제 써도 되는가? 시간복잡도 한계를 어떻게 가늠하는가?

## 1. 구현이란 무엇인가

구현(*Implementation - 머릿속의 알고리즘을 소스코드로 바꾸는 과정*)은 사실 모든 문제 풀이의 마지막 단계다. 그래서 알고리즘 교재에서는 별도 유형으로 잘 다루지 않지만, **취업용 코딩 테스트에서는 구현 자체가 중심이 되는 문제**가 자주 출제된다.

이 책에서는 두 가지를 묶어 "구현" 유형으로 다룬다.

| 하위 유형 | 의미 |
|------|------|
| 완전 탐색(*Brute Force*) | 가능한 모든 경우의 수를 빠짐없이 계산 |
| 시뮬레이션(*Simulation*) | 문제에서 제시한 절차를 한 단계씩 그대로 수행 |

> 구현 유형은 "풀이를 떠올리기는 쉽지만 소스코드로 옮기기 어려운" 문제다. 알고리즘 대회에서 "구현이 빠른 사람"을 두고 **피지컬이 좋다**고 표현하는데, 문법 숙련도와 코드 작성 속도가 곧 경쟁력이 된다.

까다로운 구현 문제의 전형:
- 알고리즘은 간단한데 코드가 지나치게 길어지는 문제
- 특정 소수점 자리까지 출력해야 하는 문제
- 문자열을 한 문자 단위로 끊어 파싱해야 하는 문제
- **사소한 조건 설정이 많은** 문제

## 2. 구현 문제의 핵심 무기 — 방향 벡터(dx/dy)

2차원 격자에서 "상하좌우로 이동" 같은 요구는 구현 문제에 거의 빠지지 않는다. 이때 **이동량을 배열 두 개(dx, dy)로 분리**해 두면, 반복문 하나로 모든 방향을 깔끔하게 처리할 수 있다.

```ts
// 현재 위치 (x, y)에서 네 방향(동/서/남/북 등)을 차례로 탐색
const dx = [-1, 1, 0, 0]; // 행(row) 변화량
const dy = [0, 0, -1, 1]; // 열(column) 변화량

for (let dir = 0; dir < 4; dir++) {
  const nx = x + dx[dir];
  const ny = y + dy[dir];
  // nx, ny가 격자 범위 안인지 검사 후 처리
}
```

> **핵심 통찰**: dx/dy 패턴은 상하좌우·시각 같은 단순 시뮬레이션부터 DFS/BFS(Ch 05)의 그래프 탐색까지 동일하게 재사용된다. 한 번 손에 익히면 격자 문제의 절반은 끝난 셈이다.

## 3. 시간·메모리 제약 감각

구현에 들어가기 전에 "이 풀이가 제한 안에 들어오는가"를 먼저 추산해야 한다.

| 항목 | 기준 |
|------|------|
| 연산 횟수 (파이썬 3.7) | 1초에 약 2,000만 번 |
| 연산 횟수 (PyPy3 / Node.js) | 1초에 약 1억 번 이상 |
| 완전 탐색 적용 한계 | 확인할 데이터가 **100만 개 이하**일 때 안전 |
| 1초 + 데이터 100만 개 | 보통 O(N log N) 이내 알고리즘 요구 |

> **실무 팁**: TypeScript(Node.js)는 파이썬보다 빠르므로, 책의 "파이썬 기준 2,000만 연산" 가정보다 여유가 있다. 다만 **정수 안전 범위**는 다르다. JS의 `number`는 IEEE 754 배정밀도라 안전 정수가 `2^53 − 1`(약 9,007조)까지다. 그보다 큰 정수는 `BigInt`(`123n`)를 써야 한다. 파이썬은 정수 크기 제한이 없다는 점과 대비된다.

## 4. 도입 예제 1 — 상하좌우 (시뮬레이션)

| 항목 | 값 |
|------|-----|
| 난이도 | ★ 하 |
| 풀이 시간 | 15분 |
| 시간 제한 | 1초 |
| 메모리 | 128MB |

### 문제 요약

N×N 공간에서 시작점 (1, 1)에 선 여행가가 계획서(`L`, `R`, `U`, `D`)대로 이동한다. **공간을 벗어나는 이동은 무시**한다. 최종 도착 좌표를 출력하라.

예: N=5, 계획서 `R R R U D D` → 최종 좌표 `(3, 4)`

### 핵심 아이디어

명령 문자별로 이동량을 정의해 두고, 한 번에 하나씩 시뮬레이션한다. 이동 후 좌표가 범위를 벗어나면 그 명령은 건너뛴다.

### 코드

```python
# Python (책 원본)
n = int(input())
x, y = 1, 1
plans = input().split()

# L, R, U, D에 따른 이동 방향
dx = [0, 0, -1, 1]
dy = [-1, 1, 0, 0]
move_types = ['L', 'R', 'U', 'D']

for plan in plans:
    for i in range(len(move_types)):
        if plan == move_types[i]:
            nx = x + dx[i]
            ny = y + dy[i]
    if nx < 1 or ny < 1 or nx > n or ny > n:  # 공간을 벗어나면 무시
        continue
    x, y = nx, ny

print(x, y)
```

```ts
// TypeScript
const n = 5;
const plans = "R R R U D D".split(" ");
let x = 1;
let y = 1;

// L, R, U, D에 따른 이동 방향
const dx = [0, 0, -1, 1];
const dy = [-1, 1, 0, 0];
const moveTypes = ["L", "R", "U", "D"];

for (const plan of plans) {
  let nx = x;
  let ny = y;
  for (let i = 0; i < moveTypes.length; i++) {
    if (plan === moveTypes[i]) {
      nx = x + dx[i];
      ny = y + dy[i];
    }
  }
  // 공간을 벗어나는 경우 무시
  if (nx < 1 || ny < 1 || nx > n || ny > n) {
    continue;
  }
  x = nx;
  y = ny;
}
console.log(x, y); // 3 4
```

**시간복잡도**: O(이동 횟수) — 매우 넉넉하다.

> **함정**: 파이썬은 `nx`를 `if` 블록 안에서만 할당해도 동작하지만, TypeScript에서는 블록 스코프 때문에 반복문 밖에서 쓸 수 없다. **루프 진입 시 `nx = x; ny = y;`로 초기화**해 두는 편이 안전하다.

## 5. 도입 예제 2 — 시각 (완전 탐색)

| 항목 | 값 |
|------|-----|
| 난이도 | ★ 하 |
| 풀이 시간 | 15분 |
| 시간 제한 | 2초 |
| 메모리 | 128MB |

### 문제 요약

00시 00분 00초부터 **N시 59분 59초**까지의 모든 시각 중, 숫자 **3이 하나라도 포함된** 시각의 개수를 구하라.

예: N=5 → `11475`

### 핵심 아이디어

하루는 86,400초뿐이라 **모든 시각을 그냥 다 세도** 시간 제한에 걸리지 않는다. 전형적인 완전 탐색이다. 시·분·초를 문자열로 이어 붙여 `'3'` 포함 여부만 검사하면 된다.

### 코드

```ts
const h = 5;
let count = 0;

for (let i = 0; i <= h; i++) {
  for (let j = 0; j < 60; j++) {
    for (let k = 0; k < 60; k++) {
      // 매 시각을 문자열로 이어 붙여 '3' 포함 여부 검사
      if (`${i}${j}${k}`.includes("3")) {
        count++;
      }
    }
  }
}
console.log(count); // 11475
```

**시간복잡도**: O(24 × 60 × 60) = 상수에 가까운 86,400회.

> **핵심 통찰**: 파이썬의 `'3' in str(i)+str(j)+str(k)` 는 TS에서 **템플릿 리터럴 + `String.includes()`** 로 1:1 대응된다. "경우의 수가 작다(보통 100만 이하)"는 신호를 보면 영리한 알고리즘 대신 **완전 탐색**을 먼저 의심하라.

## 6. 실전 문제 1 — 왕실의 나이트

| 항목 | 값 |
|------|-----|
| 난이도 | ★ 하 |
| 풀이 시간 | 20분 |
| 시간 제한 | 1초 |
| 메모리 | 128MB |

### 문제 요약

8×8 체스판 위 나이트가 **L자(8방향)** 로만 이동할 수 있다. 판을 벗어나지 않는 이동 경우의 수를 출력하라. 위치는 `a1`처럼 **열(a~h) + 행(1~8)** 문자열로 입력된다.

예: `a1` → `2`

### 핵심 아이디어

나이트의 8가지 이동을 `steps` 배열로 정의하고, 각 이동 후 좌표가 1~8 범위인지만 검사한다. 상하좌우 예제와 본질이 같되, 방향 벡터가 8개일 뿐이다.

### 코드

```ts
const inputData = "a1";
// 열(문자) → 1~8 숫자로 변환, 행(숫자)은 그대로
const row = Number(inputData[1]);
const column = inputData.charCodeAt(0) - "a".charCodeAt(0) + 1;

// 나이트가 이동할 수 있는 8가지 방향
const steps: [number, number][] = [
  [-2, -1], [-1, -2], [1, -2], [2, -1],
  [2, 1], [1, 2], [-1, 2], [-2, 1],
];

let result = 0;
for (const [dr, dc] of steps) {
  const nextRow = row + dr;
  const nextColumn = column + dc;
  // 8×8 좌표 평면 안에 있으면 카운트
  if (nextRow >= 1 && nextRow <= 8 && nextColumn >= 1 && nextColumn <= 8) {
    result++;
  }
}
console.log(result); // 2
```

**시간복잡도**: O(8) — 방향 수만큼만 검사.

> **핵심 통찰**: 파이썬의 `ord('a')` 는 TS에서 **`"a".charCodeAt(0)`** 로 옮긴다. 문자 좌표(`a`~`h`)를 숫자(`1`~`8`)로 정규화하면 이후 계산이 단순해진다. 입력 파싱을 깔끔히 정규화하는 것이 구현 문제의 절반이다.

## 7. 실전 문제 2 — 게임 개발

| 항목 | 값 |
|------|-----|
| 난이도 | ★★ 중 |
| 풀이 시간 | 40분 |
| 시간 제한 | 1초 |
| 메모리 | 128MB |
| 기출 | 삼성전자 SW 역량테스트 유형 |

### 문제 요약

N×M 맵(0: 육지, 1: 바다)에서 캐릭터가 방향(0:북, 1:동, 2:남, 3:서)을 갖고 다음 매뉴얼대로 움직인다.

1. 현재 방향 기준 **왼쪽(반시계 90도)** 부터 차례로 갈 곳을 정한다.
2. 왼쪽에 **안 가본 육지**가 있으면 회전 후 한 칸 전진. 없으면 회전만 하고 1단계 반복.
3. 네 방향 모두 가봤거나 바다면, 방향을 유지한 채 **한 칸 뒤로** 간다. 뒤도 바다면 멈춘다.

이동을 마쳤을 때 **방문한 칸의 수**를 출력하라.

### 핵심 아이디어

전형적인 시뮬레이션이다. 별도 알고리즘 없이 **매뉴얼을 오류 없이 그대로 코드로 옮기는** 것이 관건이다. 두 가지 장치가 핵심이다.

- **방향 배열 dx/dy**: 북·동·남·서 순서로 정의해, `direction` 인덱스로 현재 전진 방향을 얻는다.
- **왼쪽 회전 `turnLeft`**: `direction`을 1 줄이고, `-1`이 되면 `3`(서쪽)으로 되돌린다.

### 코드

```ts
const input = `4 4
1 1 0
1 1 1 1
1 0 0 1
1 1 0 1
1 1 1 1`;

const lines = input.split("\n");
const [n, m] = lines[0].split(" ").map(Number);
let [x, y, direction] = lines[1].split(" ").map(Number);

// 방문 기록 맵 (0으로 초기화) — 파이썬의 [[0]*m for _ in range(n)] 대응
const visited = Array.from({ length: n }, () => new Array(m).fill(0));
visited[x][y] = 1; // 현재 좌표 방문 처리

// 전체 맵 정보 (0: 육지, 1: 바다)
const board = lines.slice(2, 2 + n).map((line) => line.split(" ").map(Number));

// 북, 동, 남, 서 방향
const dx = [-1, 0, 1, 0];
const dy = [0, 1, 0, -1];

// 왼쪽으로 회전 (반시계). direction은 클로저로 공유 — 파이썬 global 대응
function turnLeft() {
  direction -= 1;
  if (direction === -1) {
    direction = 3;
  }
}

let count = 1;
let turnTime = 0;
while (true) {
  turnLeft();
  const nx = x + dx[direction];
  const ny = y + dy[direction];

  // 회전 후 정면에 안 가본 육지가 있으면 전진
  if (visited[nx][ny] === 0 && board[nx][ny] === 0) {
    visited[nx][ny] = 1;
    x = nx;
    y = ny;
    count++;
    turnTime = 0;
    continue;
  }
  // 가본 칸이거나 바다인 경우
  turnTime++;

  // 네 방향 모두 갈 수 없는 경우
  if (turnTime === 4) {
    const bx = x - dx[direction];
    const by = y - dy[direction];
    // 뒤로 갈 수 있으면 이동
    if (board[bx][by] === 0) {
      x = bx;
      y = by;
    } else {
      // 뒤가 바다로 막혀 있으면 멈춤
      break;
    }
    turnTime = 0;
  }
}
console.log(count); // 3
```

**시간복잡도**: O(N × M) — 각 칸을 사실상 한 번씩 방문.

> **함정**: 파이썬은 `global direction`으로 함수가 전역 변수를 수정했다. TS에서는 **`turnLeft`가 모듈 최상위의 `let direction`을 클로저로 공유**하므로 별도 키워드 없이 수정된다. 단, `direction`을 함수 인자로 받는 식으로 바꾸면 값 복사가 되어 동작이 깨지니 주의.

## 자주 하는 실수

| 실수 | 원인 | 해결 |
|------|------|------|
| `nx`를 `if` 안에서만 선언 | 파이썬 습관 (블록 스코프 없음) | 루프 진입 시 `let nx = x` 로 초기화 |
| 2차원 배열을 `new Array(n).fill(new Array(m))` 로 생성 | 같은 배열 참조가 n번 복사됨 | `Array.from({length:n}, () => new Array(m).fill(0))` |
| 행/열(x/y)과 좌표 입력 순서 혼동 | 문제마다 (행,열) vs (열,행) 다름 | 입력을 받자마자 의미를 주석으로 고정 |
| `direction`을 인자로 전달해 회전 | 값 복사라 원본 안 바뀜 | 클로저 공유 변수로 두거나 반환값 사용 |
| 범위 검사 누락 | 격자 밖 인덱스 접근 | 이동 직후 `0 <= nx < n` 먼저 검사 |

## 패턴 체크리스트

- [ ] 2차원 격자 이동인가? → `dx`/`dy` 방향 배열을 먼저 정의했는가?
- [ ] 경우의 수가 100만 이하인가? → 완전 탐색으로 충분한가?
- [ ] 문자열·좌표 입력을 숫자로 **정규화**(`charCodeAt`, `map(Number)`)했는가?
- [ ] 회전·방향 전환 로직을 별도 함수로 분리했는가?
- [ ] 격자 범위 검사(`0 <= nx < n && 0 <= ny < m`)를 이동 전에 넣었는가?

## 요약

- 구현 = 머릿속 알고리즘을 코드로 옮기는 능력. **완전 탐색 + 시뮬레이션**을 묶은 유형이다.
- 격자 이동의 만능 도구는 **방향 벡터 `dx`/`dy`** — 상하좌우 4방향, 나이트 8방향 모두 같은 패턴.
- 완전 탐색은 **확인할 경우의 수가 100만 이하**일 때 안심하고 쓴다 (시각 문제: 86,400가지).
- 파이썬 → TS 변환 핵심: `ord()` → `charCodeAt()`, `'x' in s` → `s.includes('x')`, `[[0]*m for _]` → `Array.from`, `global` → 클로저 공유.
- 게임 개발처럼 **매뉴얼이 긴 시뮬레이션**은 알고리즘이 아니라 **꼼꼼함**으로 푼다. 방향·회전 로직을 함수로 분리해 실수를 줄인다.

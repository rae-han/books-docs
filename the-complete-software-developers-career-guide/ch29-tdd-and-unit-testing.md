# Chapter 29: 테스트 주도 개발과 단위 테스트 (TDD and Unit Testing)

## 핵심 질문

단위 테스트는 왜 "진짜 테스트가 아니다"라고 하는가? 단위 테스트와 TDD의 진짜 가치는 무엇이며, 어떻게 오용되는가?

> 3,000개의 단위 테스트가 있다는 프로젝트를 맡았다. IDE로 열어 돌려 보니 모든 테스트가 녹색(통과)이었다. 의심이 들어 열어 보니 — **단언문(assert)이 하나도 없었다.** 실패할 것 같은 단언문은 주석 처리돼 있었다. 3,000개의 테스트가 실제로는 **아무것도 테스트하지 않았다.** 단위 테스트를 "작성하는 일"과 "이해하는 일" 사이엔 엄청난 간극이 있다.

## 1. 단위 테스트란 무엇인가

코드를 가능한 한 **작은 '단위'로 실행**해 보는 테스트를 작성하는 것이다(객체지향에서는 대개 **클래스** 단위). 애플리케이션과 같은 언어로, 다른 코드를 테스트하는 코드다.

> **핵심 통찰**: 단위 테스트는 사실 **진짜 테스트가 아니다.** 실행해서 안 되는 코드를 찾는 게 아니라, **작성하는 도중에** 코드가 무슨 일을 하는지 밝히고 설계 문제를 알게 된다. 단위 테스트는 코드가 지켜야 할 **아주 낮은 수준의 절대적 요구사항(명세)**이라고 생각하면 된다.

### 단위 테스트로 오해하는 것

- **통합 테스트(integration test)**: 클래스보다 큰 단위를 테스트하면 사실 통합 테스트다(많은 개발자가 단위 테스트라 주장하지만).
- **단언문 없는 테스트**: 그냥 아무것도 아니다. **모든 테스트의 마지막에는 통과 여부를 결정하는 확인 장치(단언문)가 있어야 한다.** 항상 통과하는 테스트와 항상 실패하는 테스트, 둘 다 무용지물이다.

## 2. 단위 테스트의 가치

단위 테스트를 하는 이유는 크게 **두 가지**다.

- **① 코드 설계 개선**: 단위를 고립시키려다 보면 설계 문제가 드러난다. 의존성 때문에 고립이 어려우면 **결합이 너무 강한 것**이고, 기본 기능이 여러 단위에 퍼져 있으면 **응집력이 부족한 것**이다. 단위 테스트를 짜려는데 "무슨 역할인지 생각나지 않아" 못 짜기도 한다. 이런 의미에서 단위 테스트는 테스트라기보다 **평가 활동**에 가깝다.
- **② 자동 회귀 테스트**: 코드를 변경해도 기능이 망가지지 않게 하는 저수준 명세가 된다.

> **핵심 통찰**: 단, 회귀 테스트는 **고수준 블랙박스 테스트가 훨씬 효과적**이다. 단위 테스트는 내부 구조를 테스트하므로, 겉 동작이 같아도 구조가 바뀌면 테스트를 수정·재작성해야 한다.

## 3. TDD란 무엇인가

**TDD(Test-Driven Development)**의 기본 개념은 **코드를 작성하기 전에 테스트부터 작성**해, 그 테스트를 코드가 해야 할 일을 정의하는 **명세**로 쓰는 것이다. 기존의 "코드 먼저, 테스트 나중" 방식을 완전히 뒤집는다.

과정은 **빨강 → 초록 → 리팩터**로 반복된다.

1. **빨강**: 기능을 정의하는 단위 테스트를 먼저 작성한다(아직 코드가 없어 실패).
2. **초록**: 그 테스트를 통과할 만큼만 코드를 작성한다.
3. **리팩터**: 코드를 정리해 더 간단명료하게 만든다.

### TDD의 목적

> **핵심 통찰**: TDD의 가장 큰 가치는 **테스트를 명세로 쓸 수 있게** 해주는 것이다. **테스트는 거짓말을 하지 않는다** — "이렇게 작동해야 한다"고 2주간 노력했더니 "사실 다르게 작동해야 한다"고 말을 바꾸지 않는다. 무언가 구현하기 전에 **무엇을 구현할지 제대로 이해**하는 과정이 선행되므로, 테스트할 것이 떠오르지 않으면 아직 대상을 이해하지 못한 것이다.

또한 TDD는 **테스트를 통과할 코드만** 쓰게 해 불필요한 코드를 줄인다. 이는 "나중에도 필요하지 않을 거야"라는 **YAGNI(You Aren't Gonna Need It)** 원칙을 지키게 해준다.

## 4. TDD 작업 흐름 — 로그인 예시

`Login` 기능을 TDD로 구현하는 흐름이다.

1. `"로그인 정보가 없으면 로그인이 되지 않는다"`라는 이름의 단위 테스트를 만든다. `Login` 클래스 인스턴스를 만들고 빈 사용자명·패스워드로 호출한 뒤, **로그인되지 않았음을 단언**한다.
2. 아직 `Login` 클래스가 없어 컴파일조차 안 된다 → 빈 클래스·메서드를 추가해 **컴파일은 되지만 실패**하게 한다.
3. 테스트를 겨우 통과할 만큼만 구현한다(사용자는 항상 로그인하지 않은 상태 반환) → **통과**.
4. 다음 테스트(`"유효한 사용자명·비밀번호면 로그인한다"`)를 작성 → 실패 → 통과할 코드 작성 → 통과.
5. 더 떠오르는 테스트 케이스가 없을 때까지 **테스트 → 코드 → 리팩터**를 반복한다.

의존성 때문에 클래스를 고립시키기 어려우면 TDD 고수들은 **모의 객체(mock)**를 써서 의존 기능을 흉내 낸다.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "TDD 없이 단위 테스트만 써도 되나요? 이미 작성된 코드에 나중에 단위 테스트를 만들어도 될까요?"
>
> **존**: **왜 그렇게 하려는지 자문**해 보라. "모범 사례니까", "코드라면 마땅히 테스트가 있어야 하니까"라는 이유로 만들지 마라. 단위 테스트의 목적(설계 개선)을 이해한다면, **이미 작성된 코드의 단위 테스트는 그 목적을 심각하게 훼손**한다는 걸 알 것이다. 물론 회귀 테스트로 유효할 때도 있다. **"그렇게 해야 한다니까" 무작정 따르지 말고 항상 실용적으로** 판단하라.

## 실천 항목

- [ ] 단위 테스트에 **단언문**이 있는지 확인한다("항상 통과"는 무의미).
- [ ] 새 기능을 **빨강→초록→리팩터** 사이클로 구현해 본다.
- [ ] 단위 테스트가 잘 안 짜이면 **설계(결합·응집) 문제**를 의심한다.
- [ ] 기존 코드에 테스트를 추가하기 전 **"왜 하는지"**를 실용적으로 자문한다.

## 요약

- 단위 테스트는 **가장 작은 단위(클래스)를 고립시켜** 테스트하는 코드이며, **단언문이 없으면 아무것도 아니다.**
- 진짜 가치는 **① 코드 설계 개선 ② 자동 회귀 테스트**다 — 테스트라기보다 **평가 활동**에 가깝다.
- **TDD**는 코드 전에 테스트를 써서 **명세**로 삼는 방식(빨강→초록→리팩터)으로, YAGNI를 지키게 한다.
- 무엇이든 **"모범 사례니까"가 아니라 목적을 이해하고 실용적으로** 적용해야 한다.

# Chapter 3: 당신이 배워야 하는 기술 (The Skills You Need to Know)

## 핵심 질문

소프트웨어 개발자가 되려면 압도적으로 많아 보이는 기술 중 무엇을, 어디서부터 배워야 하는가? 신입이 "배울 게 너무 많다"는 부담에 짓눌리지 않으려면 어떤 지도를 그려야 하는가?

## 1. 기술과 소프트 스킬, 둘 다 필요하다

손메즈는 개발자라면 기술뿐 아니라 소프트 스킬도 길러야 한다는 의견을 전폭 지지한다(그 주제로 『소프트 스킬』을 썼다). 그래도 **기술이 중요하다는 사실은 누구도 부인할 수 없다.**

> **핵심 통찰**: 코드를 작성하고 소프트웨어를 개발할 능력이 없다면 온갖 소프트 스킬도 큰 쓸모가 없다. 좋은 관리자나 코치는 될 수 있어도 **개발자**가 되긴 어렵다.

이 장은 필요한 기술을 **모조리 나열**하지 않는다. 핵심 기술만 선별해 개괄하고, 각 기술의 깊이 있는 내용은 3부 "소프트웨어 개발에 대해 알아야 할 것"에서 한 장씩 다룬다.

## 2. 꼭 배워야 하는 핵심 기술 12가지

### 언어와 코드 역량

- **① 프로그래밍 언어 한 가지 익히기**: 안정장치 삼아 여러 언어를 한꺼번에 배우려는 초보가 많지만, 오히려 혼란만 가중된다. **진짜 자신 있는 언어 하나**에 집중하라. "어떤 개발자가 될지 구체적 목표를 세우라"는 원칙이 여기서도 적용된다.
- **② 코드를 구조화하는 법**: 주석을 줄줄이 달지 않아도 **코드만으로 의사 전달이 되도록** 잘 작성하는 능력. 개발자는 새 코드 작성보다 기존 코드 **유지보수**에 훨씬 많은 시간을 쓰므로 중요하다. 초보라도 깔끔한 코드를 쓰면 면접관은 경력이 풍부하다고 추측한다. 이는 **장인의 표식**이다. (참고서: 스티브 맥코넬 『Code Complete』)
- **③ 객체지향 설계**: 복잡한 프로그램을 개별 클래스나 객체로 나눠, 각각이 기능을 캡슐화(*encapsulation - 데이터와 동작을 하나로 묶어 감추는 것*)하고 특정 역할·책임을 수행하게 하는 설계법. 인기 언어와 설계 패턴이 여전히 객체지향의 영향을 강하게 받으므로, 다형성(*polymorphism*)·캡슐화·클래스·상속 개념을 이해해 두어야 한다.
- **④ 알고리즘과 데이터 구조**: 컴퓨터 공학 교과의 핵심. 알고리즘을 잘 알면 남이 며칠 걸릴 문제를 한 시간에 풀기도 한다. 알아야 할 데이터 구조는 배열·벡터, 연결 리스트, 스택, 큐, 트리, 해시, 세트 등이다. 마이크로소프트·구글 면접을 통과하려면 반드시 잘 알아야 한다. (참고서: 게일 라크만 맥도웰 『코딩 인터뷰 완전 분석』)

> 나는 프로그래밍을 거의 독학으로 배워서 초기에는 데이터 구조와 알고리즘을 잘 몰랐다. 탑코더(TopCoder)에서 다른 이들과 경쟁해 보기 전까지는 이게 중요한 줄도 몰랐다. 지식을 익혀 문제 해결에 활용하기 시작하자, 감도 안 오던 문제를 아주 쉽고 재미있게 풀 수 있었다. 나는 이 분야가 소프트웨어 개발에서 **가장 재미있는 영역**이라고 생각한다.

### 플랫폼과 도구

- **⑤ 개발 플랫폼과 관련 기술**: 여기서 플랫폼이란 대체로 운영체제(OS) 또는 개발 환경을 뜻한다. 회사는 보통 **플랫폼이나 기술**을 기준으로 개발자를 고용한다(예: iOS 개발자). 언어를 고르면 플랫폼도 정해질 것 같지만 그렇지 않다 — C#만으로도 윈도우·맥·iOS·안드로이드·리눅스·임베디드를 다룰 수 있다. **언어만 고르지 말고 플랫폼도 정하라.**
- **⑥ 프레임워크와 스택**: **프레임워크**는 특정 플랫폼에서 개발할 때 쓰는 라이브러리 세트다(예: .NET). **스택**은 애플리케이션 하나를 완성하는 데 함께 쓰이는 기술 세트다. 예컨대 MEAN 스택은 MongoDB(데이터베이스)·Express.js(Node.js 웹 프레임워크)·AngularJS(프런트엔드 UI)·Node.js(런타임)의 조합이다. 스택을 안다는 건 애플리케이션을 **처음부터 끝까지** 개발할 능력을 갖췄다는 뜻이다.
- **⑦ 데이터베이스 기본 지식**: 관계형 데이터베이스(*relational database*)를 익숙하게 다루고 문서 데이터베이스(*document database*)를 기초적으로 이해해야 한다. 최소한 데이터베이스 작동 방식, 단순 쿼리, 삽입·수정·삭제, 데이터셋 조인(join) 정도는 알아야 한다. DBA가 따로 있어도 개발자는 데이터베이스와 연동되는 코드를 작성할 수 있어야 한다.
- **⑧ 소스 제어**: 모든 개발 프로젝트에 필수. 변경 이력을 보존하고, 여러 개발자가 동시에 작업한 뒤 변경사항을 병합할 수 있게 해준다. 최소 한 가지 이상의 소스 제어 시스템을 능숙하게 쓸 수 있어야 한다.
- **⑨ 빌드와 배포**: 요즘은 자동 빌드·배포 시스템을 쓴다. 빌드 시스템은 코드를 컴파일하고 (정교하다면) 단위 테스트·코드 품질 체크까지 수행한다. 배포 시스템은 프로덕션·테스트 환경에 코드를 배포한다. 전문가 수준까지는 아니어도 기본 작동 방식은 이해해야 한다. (이 책임은 데브옵스 DevOps 영역으로 본다.)

### 개발 프로세스

- **⑩ 테스트**: 코드 품질이 이제 **팀 전체의 책임**이 되면서 개발자와 테스터가 긴밀히 일한다. 화이트박스/블랙박스 테스트, 단위 테스트, 경계 조건, 테스트 자동화, 인수(acceptance) 테스트 같은 기본 용어를 알아야 한다. 좋은 개발자는 남에게 넘기기 전에 자기 코드를 테스트한다.
- **⑪ 디버깅**: 개발자로 일하는 시간의 상당 부분이 "왜 코드가 작동하지 않는지" 고민하는 데 쓰인다. TDD 같은 방법론으로 줄일 순 있지만, 자신이나 남이 짠 코드를 디버깅하는 법은 **반드시** 배워야 한다. 마구잡이보다 효과적인 방법을 제대로 익혀라.
- **⑫ 방법론**: 대부분의 팀은 방법론을 설정하고 (적어도 따르는 척은) 한다. 가장 흔한 두 가지는 폭포수(waterfall)와 애자일(agile)이다. 애자일 팀에 잘 적응하려면 알아야 할 관행과 의례가 있다.

> **존의 조언**: 팀이 표방하는 개발 방법론을 실제로 그대로 따를 거라 기대하지 마라. "매일 서서 회의한다"는 이유만으로 스크럼(Scrum)을 쓴다고 말하는 사람이 많다. 비난이 아니라 그냥 현실이 그렇다.

## 3. 부담 내려놓기

여기까지 많은 주제를 대부분 수박 겉핥기로 훑었다. 부담스럽거나 대부분 이해되지 않아도 **아직은 정상**이다.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "이 책에 링크가 꽤 많고, 거기에 당신이 만든 제품을 많이 홍보하는 것 같은데 어찌 된 영문이죠?"
>
> **존**: 맞다, 링크가 많다. 하지만 전부 열어 볼 필요는 없다. 대부분 내가 과거에 만든 연관 콘텐츠(블로그·유튜브)를 더 알고 싶은 사람을 위해 연결한 것이다. 내 제품을 홍보하는 것도 맞다 — 책은 저렴해서 돈보다 다른 목적이 있을 때 쓴다. 하지만 '스팸'처럼 느끼지 않았으면 한다. 다른 제품을 꼭 살 필요는 없고, 이 책은 그 자체로 충분한 가치가 있다.

## 실천 항목

- [ ] 위 12가지 기술을 놓고 **내가 이미 아는 것 / 모르는 것**을 체크한다(지식의 빈틈 파악).
- [ ] 언어 하나를 정해 "여러 개 기웃거리기"를 멈춘다.
- [ ] 3부에서 각 기술을 깊이 볼 것이므로, 지금은 **"무엇이 있는지"**만 지도에 표시해 둔다.
- [ ] 『Code Complete』·『코딩 인터뷰 완전 분석』을 읽을 목록에 추가한다.

## 요약

- 기술과 소프트 스킬 둘 다 필요하지만, **개발 능력 없이는** 소프트 스킬도 무력하다.
- 핵심 기술 12가지: 언어 하나·코드 구조화·객체지향 설계·알고리즘/자료구조 / 플랫폼·프레임워크와 스택·데이터베이스·소스 제어·빌드와 배포 / 테스트·디버깅·방법론.
- 이 장의 목표는 통달이 아니라 **지형 파악** — 무엇을 모르는지 아는 것이다. 깊이는 3부에서 채운다.
- 지금 부담스럽고 이해 안 되는 건 **정상**이다.

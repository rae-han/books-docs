# Chapter 28: 테스트와 QA 기초 (Testing and QA Basics)

## 핵심 질문

소프트웨어 테스트의 진짜 목표는 무엇인가? 개발자가 알아야 할 테스트 유형과 절차, 그리고 테스트에서 개발자의 역할은 무엇인가?

> 소프트웨어 개발 초창기에 나는 HP에서 테스터로 일하며 신제품 프린터 인쇄본과 구형 '마스터' 인쇄본을 비교했다. 이 경험을 통해 **대부분의 개발자가 테스트가 실제로 어떻게 이뤄지는지 모른다**는 것, 그리고 테스트 지식이 개발자의 경력 발전에도 도움이 된다는 것을 깨달았다.

## 1. 테스트의 핵심 목표 — 위험부담 줄이기

> **핵심 통찰**: 테스트의 핵심 목표는 **버그를 전부 찾는 것도, 명세 충족을 완벽히 확인하는 것도 아니다**(둘 다 달성 불가능). 소프트웨어를 쓸 **고객에게 크고 부정적인 영향을 미칠 위험 요소를 줄이는 것**이 테스트의 목표다.

그래서 테스트는 **영향(심각성)과 빈도**로 우선순위를 정한다. 회계 소프트웨어가 1,000달러 입력마다 1~2초 멈추는 버그는 **자주 발생하지만 영향이 작고**, 1,000번째 데이터 저장마다 오류가 나는 버그는 **드물지만 영향이 크다**. 가장 큰 위험 영역부터 테스트하고, 결함은 심각성에 따라 고치거나 기록으로 남긴다.

## 2. 일반적인 테스트 유형

- **블랙박스 테스트**: 소프트웨어를 **내부가 안 보이는 검은 상자**로 보고 입력과 출력만 신경 쓴다. 편견이 적어 가장 흔히 쓰인다.
- **화이트박스 테스트**: 반대로 **내부 구조·소스 코드를 알고** 진행한다. 예컨대 계산식이 값에 따라 둘로 나뉘어 있으면 양쪽을 겨냥한 테스트를 만들 수 있다(블랙박스로는 알 수 없다).
- **인수 테스트**: 소프트웨어가 **고객의 요구·기대에 맞게** 만들어졌는지 시스템 전체를 점검한다.
- **자동 테스트**: 실행·검증이 **자동으로** 이뤄지는 테스트. 수동 반복은 지루하고 오류가 나기 쉬워 점점 늘어나는 추세다.
- **회귀 테스트**: 시스템이 **과거에 작동하던 방식 그대로** 작동하는지 확인한다. 애자일에서 신기능이 기존 기능을 망가뜨릴 위험이 크므로 중요하다.
- **기능 테스트**: 시스템이 **기능적으로 해야 할 일**을 하는지에 초점을 맞춘다(성능·보안·사용성 등과 대비).
- **탐색적 테스트**: 테스트 케이스 없이 애플리케이션을 탐색하며 **예상 못한 문제**를 찾는다. 합리적 케이스로는 못 찾을 버그를 발견하기도 한다.
- **그 외**: 부하·성능·회복·보안·스트레스·사용성 테스트 등.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "블랙박스 테스트와 기능 테스트가 비슷해 헷갈려요. 회귀 테스트는 다 자동 테스트 아닌가요?"
>
> **존**: (조용히) 여러 테스팅 용어가 사실 **같은 것을 가리킨다.** **블랙박스/화이트박스는 테스트를 "수행하는 방식"**이고, **기능 테스트는 "무엇을 테스트하는지"**다. 그래서 기능 테스트는 보통 블랙박스 방식으로 한다. **자동/회귀도 상위 개념**이다 — 회귀 테스트는 "기능이 회귀하지 않게" 확인하는 개념이고, 자동 테스트는 그 용도에 잘 맞는 "방식"이라 자동 테스트를 회귀 테스트로 볼 수도 있다. 개발자는 **큰 그림과 "테스트의 목적은 위험을 줄이는 것"**만 이해하면 충분하다.

## 3. 테스트 절차

테스트는 대개 **계획 → 설계 → 작성·실행 → 결과 기록**으로 진행된다.

1. **테스트 계획**: 어떤 방식·전략·유형으로, 어떤 기능을, 어떤 일정에 테스트할지 정한다.
2. **테스트 설계**: 요구사항·기능을 기반으로 실행 가능한 테스트 케이스 목록을 만든다.
3. **작성·실행**: 테스트를 만들고 실행한다.
4. **결과 기록**: 버그·결함은 버그 추적 시스템에 기록하고 우선순위를 매겨 개발자에게 보낸다. 수정 후 다시 테스트하며, 품질 기준에 이를 때까지 반복한다.

### 애자일 팀의 테스트

애자일 팀은 매주 새 기능을 만들므로 **표준 테스트 절차를 그대로 쓰면 문제**가 생긴다. 절차를 더 작은 단계로 나누고 피드백을 자주 받아야 하며, **각 기능을 미니 프로젝트로 보고 코드 작성 전부터 테스트를 실행**하는 게 이상적이다. 짧은 주기 탓에 **회귀 테스트가 중요해져 자동화**가 핵심이 된다.

## 4. 테스트, 당신 그리고 개발자

> **핵심 통찰**: 개발자는 자신이 만든 코드의 품질에 **그 누구보다 책임감**을 느껴야 한다. "QA가 내 버그를 찾아줄 것"이라 생각하지 마라 — **버그는 늦게 발견될수록 고치는 비용이 커진다.**

개발자가 체크인 전에 직접 찾은 버그는 한 시간이면 고치지만, 놓치면 다음을 거친다.

- 테스터가 테스트를 실행·재실행해 버그를 확인 → 버그 추적 시스템에 저장
- 관리자가 심각성 판단 후 개발자에게 할당 → 개발자는 재현 시도(내 컴퓨터에선 정상)
- 테스터가 재현해 상세 기술 → 개발자가 겨우 재현·수정 → 테스터가 재확인

> **존의 조언**: 코드를 체크인하기 전 **10분만 투자해 테스트**해 보라. 모든 버그를 찾진 못해도 10%만 찾아내도 여러 사람의 긴 시간을 절약한다.

## 실천 항목

- [ ] 내 기능을 **영향·빈도**로 나눠 어디를 먼저 테스트할지 우선순위를 정한다.
- [ ] 코드를 **체크인하기 전 10분** 직접 테스트하는 습관을 들인다.
- [ ] 애자일 주기라면 **회귀 테스트를 자동화**한다.
- [ ] 테스트 용어(블랙박스·기능·회귀·자동)의 **관계**를 정리해 둔다.

## 요약

- 테스트의 목표는 완벽이 아니라 **고객에게 큰 영향을 줄 위험을 줄이는 것**이다(영향×빈도로 우선순위).
- 유형: **블랙박스/화이트박스(방식), 기능/인수(대상), 자동/회귀(용도)** 등 — 상당수 용어가 겹친다.
- 절차는 **계획→설계→작성·실행→기록**이며, 애자일에서는 잘게 나누고 자동화해야 한다.
- 개발자도 **자기 코드 품질에 책임**을 지고, 체크인 전 스스로 테스트해 비용을 줄여야 한다.

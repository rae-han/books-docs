# Chapter 22: 웹 개발이란 무엇인가? (What Is Web Development?)

## 핵심 질문

웹 개발은 왜 데스크톱·모바일 개발보다 까다로운가? 웹은 어떻게 작동하며, 개발자가 최소한 알아야 할 핵심 기술은 무엇인가?

> **핵심 통찰**: 웹 개발은 **런타임 환경에 대한 통제력이 떨어진다.** 데스크톱·모바일은 OS·언어가 명확하지만, 웹에는 다섯 강자(IE·엣지·크롬·사파리·파이어폭스) + 모바일 브라우저 + 콘솔 등 **버전마다 다른** 환경이 존재해 까다롭다. 그럼에도 오늘날 개발자 대부분은 웹 개발자다 — **웹은 개발 플랫폼계의 킹콩**이다.

## 1. 웹의 작동 방식

웹의 기본 기술은 변하지 않았다 — **웹 브라우저에서 작동하는 애플리케이션을 만드는 것**이 웹 개발이다.

1. 웹 브라우저가 **HTTP(HyperText Transfer Protocol)**로 웹 서버에 특정 리소스(URI) 요청을 보낸다.
2. 웹 서버가 HTML·CSS·자바스크립트로 응답한다.
3. 브라우저가 응답을 구문 분석·렌더링해 사용자에게 보여준다.

> **핵심 통찰**: 웹의 바탕인 **HTTP에는 상태(state)가 없다.** 데스크톱 앱은 메모리에 상태를 저장하지만, 웹 앱은 동작마다 서버에 요청을 보내야 하고 그 사이 **상태 관리와 사용자 추적** 방법을 따로 찾아야 한다. 이 점이 웹 개발을 다른 개발과 크게 다르게 만든다.

## 2. 웹의 간략사

웹 기술이 왜 지금처럼 복잡해졌는지는 진화 과정을 보면 이해된다.

- **정적 HTML**: 초창기엔 하이퍼링크로 연결된 정적 페이지뿐이었다. "웹 개발자"가 아니라 "웹마스터"만 있었다.
- **CGI**: 최초의 웹 애플리케이션 기술. 브라우저가 보낸 데이터(쿼리 문자열)로 조건부 HTML을 생성했지만 어려웠다.
- **초기 프레임워크(ColdFusion·ASP)**: 특별한 태그·마크업으로 동적 HTML 생성을 쉽게 했다(템플릿 언어 역할).
- **자바스크립트 + CSS**: 페이지를 인터랙티브하게, 스타일을 분리해 관리하기 쉽게.
- **AJAX → SPA**: 새로고침 없이 동적으로 업데이트되는 **SPA(Single Page Application)**가 탄생했다.

지금은 웹이 점점 데스크톱 애플리케이션을, 브라우저가 운영체제를 닮아가고 있다(예: 크롬 OS).

## 3. 주요 웹 개발 기술

- **HTML**: 웹의 주춧돌. 웹페이지의 **내용과 구조**를 만든다(`<img>`, `<h1>`, `<article>` 등의 태그).
- **CSS**: 내용(HTML)과 **스타일**을 분리한다. CSS 파일 하나만 바꾸면 모든 버튼 색이 한꺼번에 바뀐다. 단, 변수·함수·캡슐화를 지원하지 않아 코드가 반복되기 쉽다.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "CSS 전처리기는 뭔가요?"
>
> **존**: CSS 코드의 반복을 줄여주는 도구다. CSS를 쓸 때도 일반 코드처럼 **변수를 할당하고 반복문을 쓰고 재사용**할 수 있게 해준다. 큰 애플리케이션의 CSS 유지보수가 쉬워지고 시간이 많이 절약된다.

- **자바스크립트**: 웹 브라우저에서 바로 실행되는 동적 언어로, **DOM(Document Object Model)**을 조작해 콘텐츠·CSS를 추가·삭제·변경한다. Node.js를 쓰면 서버에서도 실행된다.

## 4. 서버 측 렌더링 vs 클라이언트 측 렌더링

- **서버 측 렌더링**: 웹페이지가 **서버 로직**에서 생성돼 HTML·CSS·JS가 브라우저로 전송된다. 가장 원시적인 모델(ASP.NET·PHP가 대체로 이 방식).
- **클라이언트 측 렌더링**: 자바스크립트가 **브라우저에서** 콘텐츠를 만든다. 새 페이지마다 서버에서 받을 필요가 없어 더 자연스럽고, 이런 앱을 **SPA**라 부른다.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "자바스크립트 말고 웹 개발을 할 수 있는 다른 언어는 없나요?"
>
> **존**: 대안은 굉장히 많다 — 커피스크립트, **타입스크립트(TypeScript)**, 바벨, 엘름, 다트, 클로저스크립트 등. 하지만 대부분 브라우저에서 실행하기 전에 **결국 자바스크립트로 컴파일해야 한다**는 사실을 기억하라.

## 5. API

> **핵심 통찰**: 오늘날 웹 개발의 상당 부분은 **API(특히 웹 API)**를 다루는 일이다. 클라이언트 측 렌더링 애플리케이션은 브라우저에서 실행되므로 **API를 통해 서버와 통신**해 데이터를 주고받는다. SPA는 대개 API로 데이터를 주고받는다.

## 실천 항목

- [ ] 브라우저 개발자 도구로 실제 페이지의 **HTTP 요청/응답**을 관찰한다.
- [ ] **HTML(구조)·CSS(스타일)·자바스크립트(동작)**의 역할 분담을 직접 예제로 확인한다.
- [ ] 관심 있는 웹 앱이 **서버 측/클라이언트 측 렌더링(SPA)** 중 무엇인지 판별해 본다.
- [ ] 간단한 **웹 API 호출**(fetch 등)로 데이터를 받아 화면에 표시해 본다.

## 요약

- 웹 개발은 **런타임(브라우저) 환경을 통제하기 어렵다**는 점에서 까다롭지만, 여전히 가장 지배적인 플랫폼이다.
- 브라우저 ↔ **HTTP(무상태)** ↔ 서버 구조이며, 상태 관리가 웹 개발의 핵심 난제다.
- 핵심 기술은 **HTML(구조)·CSS(스타일)·자바스크립트(동작)**이고, 렌더링은 **서버 측/클라이언트 측(SPA)**으로 나뉜다.
- 자바스크립트 대안(타입스크립트 등)도 결국 **자바스크립트로 컴파일**되며, 오늘날 웹은 **API 프로그래밍**이 큰 비중을 차지한다.

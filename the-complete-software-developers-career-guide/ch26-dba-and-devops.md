# Chapter 26: DBA와 데브옵스 (DBAs and DevOps)

## 핵심 질문

코드 작성이 끝난 뒤 데이터 저장·빌드·테스트·배포를 담당하는 DBA와 데브옵스는 어떤 역할을 하며, 개발자는 이들에 대해 무엇을 알아야 하는가?

> **핵심 통찰**: 소프트웨어 개발이 점점 **팀 단위**로, 특히 애자일 환경에서 이뤄지면서 개발자에게도 코드 작성 외의 여러 역할이 요구된다. DBA·데이터베이스 전문 팀이 없는 스타트업·작은 회사에서는 더욱 그렇다.

## 1. DBA (데이터베이스 관리자)

DBA의 주요 업무는 데이터베이스를 **설치·유지보수·보호·최적화·감시**하는 것이다. 데이터베이스는 시간이 지날수록 커지므로 하드웨어 선택·업그레이드, 정기 백업, 재난 대비, 성능 분석·인덱싱이 필요하다 — **데이터베이스에는 양육의 손길이 필요하다.**

모든 팀에 DBA가 있는 건 아니고 그 업무를 개발자에게 맡기는 조직이 많으므로, DBA가 될 필요는 없어도 다음 기본은 알아두면 좋다.

- 데이터베이스 설치·설정, 백업·복구
- 테이블·스키마 만들기, 저장 프로시저 만들기
- 테이블 인덱스 만들기와 작동법
- 조회·삽입·갱신 등 기초 SQL 작성, 테이블 조인(join)

### 대화: 〈잠깐만요, 존!〉

> **독자**: "스키마·저장 프로시저·테이블 조인이 뭔지도 몰라요!"
>
> **존**: 진정하라. 알아두면 좋은 개념을 소개한 것뿐이다.
> - **스키마**: 데이터베이스의 **청사진**. 어떤 테이블로 나눌지, 데이터를 어떤 양식으로 저장할지 정의한다. 스키마가 있으면 똑같은 구조의 빈 데이터베이스를 만들 수 있다.
> - **저장 프로시저**: 데이터베이스의 **함수/메서드**. 데이터베이스에서 바로 실행돼 특정 명령을 수행하는 SQL 코드로, 일반적인 작업을 논리적으로 정리하는 방법이다.
> - **테이블 조인**: 여러 테이블의 데이터를 **조합**하는 것. 예컨대 '고객' 테이블과 '주문' 테이블을 조인하면 특정 고객의 데이터와 그 고객의 모든 주문을 한 번에 가져올 수 있다.

## 2. 데브옵스 — 새로운 직군

데브옵스(DevOps)는 **개발(Development)과 운영(Operations)의 복합체**다. 아직 명확히 정의되지 않았지만, "코드를 빌드·테스트·배포하고 서비스를 운영하는 데 필요한 일이면 뭐든 한다"는 데에는 대체로 동의한다.

### 기존 방식의 문제

데브옵스 이전엔 **개발팀과 운영팀(IT)**이 분리돼 있었다. 개발자는 코드를 작성하고 운영팀은 배포하며, 서로 무슨 일을 하는지 잘 몰라 배포 때마다 서로를 비난했다. **6개월에 한 번 배포하던 시절**엔 견딜 만했지만, **애자일로 주에 여러 번 배포**하게 되면서 이 단순 구조로는 부족해졌다.

> **핵심 통찰**: 데브옵스는 코드를 **개발에서 생산까지 관리하는 종합적인 공정**이다. 절차를 여러 단계로 쪼개면 서로 적대하기 쉽지만, 데브옵스는 그 모든 과정을 아우른다. 이는 "소프트웨어가 완성되는 과정"에 대한 사고방식의 전면적 변화를 보여준다.

### 개발자에게 의미하는 것

> **존의 조언**: 소프트웨어 개발자는 엄밀히 말해 **데브옵스에 속한다.** "내 코드를 넘겼으니 그다음은 남의 문제"라고 생각하던 시절은 지났다. 개발자가 알아야 할 것:
> - **소스 제어**로 코드를 통합하고, IDE·로컬 빌드를 다룰 것
> - **지속적 통합·빌드 서버**의 기초
> - **테스트** 기초와 자동 테스트가 빌드·배포에 어떻게 적용되는지
> - 애플리케이션 **패키징·배포**(빌드→스테이징/프로덕션 이관, 환경 설정 관리)
> - 배포 후 **모니터링**

이 모든 걸 전문가 수준으로 알 필요는 없다. 야구팀 투수가 모든 포지션을 꿰뚫진 못해도 각 역할을 알아야 전략에 의견을 낼 수 있는 것과 같다.

## 실천 항목

- [ ] 데이터베이스 **설치·백업·기초 SQL·조인**을 직접 해본다(DBA가 없어도 대응 가능하게).
- [ ] 스키마·저장 프로시저·조인 등 **DB 핵심 개념**을 정리한다.
- [ ] 내 코드가 **빌드→테스트→배포**되는 전체 파이프라인을 한 번 따라가 본다.
- [ ] 배포 후 **모니터링** 지표를 확인하는 법을 익힌다.

## 요약

- 팀 단위 개발이 늘면서 개발자도 **DBA·데브옵스** 영역을 이해해야 한다.
- **DBA**는 데이터베이스를 양육(설치·백업·최적화·감시)한다 — 기본 SQL·스키마·조인은 개발자도 알아야 한다.
- **데브옵스**는 개발↔운영의 벽을 허물고 코드를 **개발에서 생산까지** 아우르는 공정이며, 애자일의 잦은 배포가 이를 촉발했다.
- 개발자는 이제 **소스 제어·CI·테스트·배포·모니터링**을 두루 이해해야 한다.

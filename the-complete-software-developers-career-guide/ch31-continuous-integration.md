# Chapter 31: 지속적 통합 (Continuous Integration)

## 핵심 질문

지속적 통합(CI)은 어떤 문제를 해결하며, 개발자가 코드를 체크인할 때마다 자동으로 무슨 일이 일어나는가?

> **핵심 통찰**: 손메즈는 새 팀에 배정되면 CI가 없을 경우 **가장 먼저 그것부터 설정**할 거라 단언한다. CI는 ① 느리고 고통스러운 빌드·테스트·패키징을 **자동화**하고, ② 자주 병합시켜 **병합 지옥을 막고**, ③ 체크인 후 몇 분 내로 문제를 알려주는 **빠른 피드백**을 준다. 피드백 주기가 빠를수록 소프트웨어는 빨리 진화하고 품질은 개선된다.

## 1. CI의 진화 과정

- **과거의 로컬 빌드**: 개발자마다 자기 컴퓨터에서 각기 다른 방식으로 빌드해, **같은 코드로 다른 결과물**이 나오기 일쑤였다(라이브러리 버전·폴더 구조 차이). 누가 컴파일도 안 되는 코드를 체크인해도 아무도 몰랐다.
- **빌드 서버 등장**: 환경을 제대로 갖춘 **중앙 빌드 서버**를 도입했다. **주간 빌드 → 야간 빌드**로 발전하며 문제를 더 일찍 발견하게 됐고, 빌드 프로세스 자동화(Makefile·Ant 등)가 촉진됐다.
- **마침내 CI로**: 애자일의 짧은 피드백 주기 요구로, **누군가 코드를 체크인할 때마다** 빌드하는 CI가 등장했다. 단순 컴파일을 넘어 **단위 테스트·정적 코드 분석**까지 동시에 실행하는 수준으로 진화했다.

## 2. CI 작업 흐름 (샘플)

새 코드가 체크인될 때마다 CI 서버가 다음을 자동 수행한다.

1. **코드 체크인**: (메인 저장소에 넣기 전 로컬에서 빌드·단위 테스트를 실행했어야 한다.)
2. **새 빌드 시작**: CI 소프트웨어가 소스 제어 브랜치의 변화를 감지해 빌드를 시작한다.
3. **코드 체크아웃**: 최신 수정사항을 모두 가져온다.
4. **코드 컴파일**: 외부 라이브러리 등을 링크해 컴파일한다. 실패하면 "**빌드가 깨졌다**"고 하며 거기서 멈춘다.
5. **정적 분석기 실행**: 코드 품질을 측정한다.
6. **단위 테스트 실행**: 통과하지 못하면 보통 **빌드 자체가 실패**하게 한다.
7. **결과 보고**: 성공/실패, 소요 시간, 품질·테스트 정보를 이메일·웹으로 보고한다.
8. **패키징 → (선택) 배포**: 배포 가능한 형태로 묶고, **지속적 배포(Continuous Deployment)**를 하는 팀은 특정 환경(때론 프로덕션)에 바로 배포한다.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "정적 코드 분석기가 뭐고, 그게 측정하는 코드 품질 기준은 뭔가요?"
>
> **존**: 소스 코드를 실행하지 않고 살펴보며 **발생할 것 같은 에러·모범 사례 위반**을 찾는 도구다. 예컨대 자바의 PMD는 사용되지 않는 변수·불필요한 객체 생성 등을 찾는다. **순환 복잡도(cyclomatic complexity), 유지보수성 인덱스, 상속의 깊이** 같은 코드 정보를 제공하는 분석기도 있어, 잠재적 문제를 예방적으로 고치는 데 쓴다.

> **핵심 통찰**: 단위 테스트 실패를 무시하는 건 **미끄러운 비탈길**이다. 한번 무너지면 "단위 테스트는 원래 실패하는 것"이라 여기게 되고, 그러면 단위 테스트가 무의미해진다. 그래서 **테스트 실패 시 빌드를 실패**시키는 것을 강력히 권한다.

## 3. CI 서버·소프트웨어

CI 소프트웨어는 위 작업 흐름을 자동화해 준다. 집필 시점 기준 주요 제품:

- **젠킨스(Jenkins)**: 손메즈가 즐겨 쓰는 CI 소프트웨어. 원래 자바용이었지만 설치가 쉽고 **플러그인이 매우 많아** 널리 쓰인다.
- **허드슨(Hudson)**: 젠킨스가 갈라져 나온 원조. 지금은 젠킨스가 더 낫다는 평.
- **트래비스 CI(Travis CI)**: 설치가 아니라 **가입**하는 호스팅(서비스형). 깃허브 프로젝트에서 쉽게 쓸 수 있어 인기.
- **TFS(Team Foundation Server)**: 마이크로소프트 도구 업체용.
- **팀시티(TeamCity)**: 젯브레인스의 라이선스 제품. .NET 팀이 많이 쓴다.

## 실천 항목

- [ ] 내 프로젝트에 **자동 빌드 + CI**가 있는지 확인하고, 없으면 도입을 제안한다.
- [ ] 메인 저장소에 체크인하기 전 **로컬 빌드·단위 테스트**를 실행한다.
- [ ] CI에 **단위 테스트·정적 분석**을 붙이고, 테스트 실패 시 빌드가 실패하도록 설정한다.
- [ ] 깃허브 프로젝트라면 **트래비스 CI** 같은 서비스형 CI를 붙여 본다.

## 요약

- CI는 **자동화·병합 지옥 방지·빠른 피드백**을 제공하는, 애자일의 핵심 요소다.
- 진화: **로컬 빌드 → 빌드 서버(주간/야간) → 체크인마다 빌드하는 CI**.
- 작업 흐름: **체크인 → 빌드 → 컴파일 → 정적 분석 → 단위 테스트 → 보고 → 패키징 → (선택) 배포**.
- 젠킨스·트래비스 CI 등 CI 소프트웨어가 이를 자동화하며, **테스트 실패는 빌드 실패로** 다뤄야 한다.

# Chapter 33: 코드 유지보수 (Maintaining Code)

## 핵심 질문

개발자는 왜 새 코드 작성보다 유지보수에 더 많은 시간을 쓰는가? 유지보수하기 쉬운 코드는 어떻게 작성하며, 물려받은 코드는 어떻게 관리하는가?

> **핵심 통찰**: 막상 개발자가 되면 새 코드를 쓰는 시간보다 **코드를 유지보수하는 시간이 훨씬 길다.** 세상에는 새 소프트웨어보다 오래된 소프트웨어가 항상 더 많고, 소프트웨어는 **살아 숨 쉬는 유기체**처럼 끊임없이 성장·변화하며 천천히 죽어간다. 아무리 아름답게 묘사된 자리라도 기존 시스템 유지보수는 피할 수 없다.

## 1. 훌륭한 개발자는 유지보수하기 좋은 코드를 만든다

> **존의 조언**: 내가 프로그래머를 판단하는 **단 하나의 기준은 "얼마나 유지보수하기 쉬운 코드를 작성하느냐"**다. 훌륭한 개발자는 자기 코드가 유지보수 단계에 가장 오래 머문다는 것, 오래 살아남는 코드가 가치 있다는 것을 알기에, **명확하고 느슨하게 결합된, 설명이 없어도 이해되는** 코드를 만든다.

## 2. 보이스카우트 규칙

> **핵심 통찰**: **"캠핑한 장소를 처음 왔을 때보다 더 깨끗하게 만들어 두고 떠나라."** 소프트웨어에도 적용된다 — **받은 코드보다 더 나은 코드로 만들어 두고 떠나라.**

- 버그 수정·기능 추가 시 코드를 **약간 더 낫게** 만든다.
- 단위 테스트를 추가해 뒷사람에게 더 튼튼한 코드를 남긴다.
- 변수 이름을 바꿔 의미를 명확히 하고, 기능을 메서드로 묶어 이해도를 높인다.
- 리팩토링으로 더 깔끔하고 단순하게 만든다.

이 규칙을 따르면 코드는 점점 나아지고, 적어도 **엔트로피(무질서)가 늘어나는 속도**가 크게 줄어든다.

## 3. 가독성이 가장 중요하다

> **핵심 통찰**: 유지보수 용이성에 가장 큰 영향을 미치는 요소는 **가독성**이다. **코드를 작성하는 시간보다 읽는 시간이 훨씬 길기** 때문이다. 간결한 것도 좋지만 **너무 짧고 기발한 코드는 재앙**으로 이어진다. 알아보기 쉬운 코드가 유지보수하기 쉽다.

## 4. 코드 리팩토링

**리팩토링은 기존 기능은 그대로 두고 코드 설계를 개선**하는 과정이다. "기능을 그대로 두는" 부분이 핵심이다 — 기능까지 바꾸면 새 버그가 생기기 쉽다. **테스트 없이는 기능이 바뀌었는지 확인하기 어려우므로, 리팩토링과 단위 테스트는 함께 간다.**

### 대화: 〈잠깐만요, 존!〉

> **독자**: "단위 테스트도 유지보수를 신경 써야 하나요?"
>
> **존**: (마음을 담아) **그렇다!** 단위 테스트도 코드이므로 유지보수해야 한다 — 오히려 더 신경 써야 한다. ① 실패했을 때 무엇을 확인하려던 건지 30분씩 씨름하고 싶은 사람은 없고, ② 코드가 바뀌면 테스트도 바꿔야 하며, ③ 단위 테스트는 반복 코드가 많아 유지보수가 특히 까다롭다. **프로젝트가 단위 테스트를 포기하는 이유는 하나 — 유지보수가 안 되기 때문이다.**

## 5. 자동화·주석·커밋 메시지

- **자동화는 필수**: 자동 빌드·CI·자동 테스트를 갖추면 수정·리팩토링을 **두려움 없이** 할 수 있고, 이것이 코드를 더 낫게 만든다.
- **주석을 쓸 거라면 잘 써라**: 손메즈는 주석보다 **주석 없이도 이해되는 명확한 코드**를 선호한다("수수께끼 같은 주석은 수수께끼 같은 코드만큼, 때론 더 나쁘다"). 쓴다면 설명이 필요한 부분을 명확히 설명하라.
- **커밋 메시지도 명확하게**: 커밋 메시지는 코드가 **왜 그렇게 변했는지**를 알려주므로, 까다로운 버그를 이해하는 데 큰 도움이 된다.

### 대화: 〈잠깐만요, 존!〉

> **독자**: "우리 팀에는 코드 스타일 가이드가 있어요. 따라야 할까요?"
>
> **존**: 따라야 한다. **완벽을 기하는 것보다 통일성을 이루는 게 우선**이다. 가이드의 양식이 가독성 최고가 아니더라도, 코드가 통일되면 **전체 코드베이스의 가독성이 크게 개선**된다. 자존심을 조금 덜고 규칙을 따르되, 진짜 동의할 수 없으면 합리적으로 입증해 가이드를 수정할 방법을 찾아라.

## 6. 추천 자료

유지보수하기 좋은 코드를 배울 수 있는 고전들이다.

- 로버트 마틴 『Clean Code』 — 이해하기 쉬운 코드 작성·설계·리팩토링
- 스티브 맥코넬 『Code Complete』 — 가독성 높은 코드의 저수준 구조
- 마이클 페더스 『레거시 코드 활용 전략』 — 기존/레거시 코드 다루기의 고전
- 마틴 파울러 『리팩토링』 — 기능을 유지하며 코드를 재구성하는 방법

## 실천 항목

- [ ] 코드를 만질 때마다 **보이스카우트 규칙**(받은 것보다 더 낫게)을 적용한다.
- [ ] "기발함"보다 **가독성**을 우선해 코드를 작성한다.
- [ ] 리팩토링 전후로 **단위 테스트**를 돌려 기능이 유지되는지 확인한다.
- [ ] 팀 **스타일 가이드**를 따르고, **명확한 커밋 메시지**를 남긴다.
- [ ] 『Clean Code』·『리팩토링』 등을 읽는다.

## 요약

- 개발자의 시간 대부분은 **유지보수**에 쓰이며, 훌륭한 개발자는 **유지보수하기 좋은 코드**를 만든다.
- **보이스카우트 규칙**(받은 것보다 더 낫게)과 **가독성 우선**이 핵심 원칙이다.
- **리팩토링**(기능 유지, 설계 개선)은 **단위 테스트와 함께**하고, 단위 테스트 자체도 유지보수해야 한다.
- **자동화·명확한 주석·좋은 커밋 메시지·스타일 가이드 준수**가 유지보수를 돕는다.

# Chapter 7: Technical Practices (기술적 실행 관례)

## 핵심 질문

기술적 실행 관례(*Technical Practices - TDD, 페어 프로그래밍, 리팩터링 등 개발자의 일상 작업 방식*)가 비즈니스에 어떤 가치를 주는가? 애자일 절차만으로도 이미 충분하지 않은가? 왜 스크럼(*Scrum - 반복적 개발과 스탠딩업 미팅을 기반으로 하는 애자일 프레임워크*)은 쉽게 팔리는데 TDD와 페어 프로그래밍은 쉽게 거부당하는가? "우리 회사에서는 어차피 그런 것 못 한다"라는 항변에는 어떻게 대응해야 하는가?

---

## 1. 올바른 일 vs 올바른 실행 — 애자일의 공백

애자일 방법론의 절차 중심적인 부분들은 비즈니스 목표에 맞게 가고 있는지 정기적으로 살펴볼 수 있는 피드백 루프(*Feedback Loop - 결과를 확인하고 다음 행동에 반영하는 순환 구조*)를 만들어준다. 애자일은 변화와 싸우는 대신 변화 자체를 내재화한다.

애자일 방법론은 우리가 "올바른 일(Right Things)"을 하고 있는지 점검하도록 돕는다. 최소 시장 충족 기능(*MMF - Minimum Marketable Features, 시장에 내놓을 수 있는 최소 기능 단위*), 백로그 관리, 스탠딩업 미팅, 번-다운 차트, 사용자 스토리, 크로스 펑셔널 팀 등이 이 범주에 속한다.

문제는 이런 절차 관례들이 **애플리케이션의 품질 상태를 알려주지 않는다**는 점이다. 코드베이스의 품질이 비즈니스 발목을 잡을 정도로 나빠졌음을 깨달았을 때는 이미 늦다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 나쁜 코드는 암과도 같다. 처음에는 발견하기 어렵지만 문제로 드러난 이후에는 대응하기 어렵다. 애자일과 소프트웨어 장인정신은 겹치는 부분이 있지만, 소프트웨어 장인정신은 **기술적 실행 관례**에 집중함으로써 코드 품질에 대한 빠른 피드백 루프를 제공해 애자일을 보완한다. 기술적 실행 관례는 우리가 일을 **올바르게(Right)** 하고 있는지 알려준다.
:::

| 구분 | 무엇을 알려주는가 |
|---|---|
| 애자일 절차 관례 | 올바른 일(Right Things)을 하고 있는가 |
| 기술적 실행 관례 | 일을 올바르게(Right) 하고 있는가 |

## 2. 상황 논리와 XP의 역사

XP(*Extreme Programming - 켄트 벡이 정립한 애자일 방법론의 하나, TDD/페어 프로그래밍/지속적 통합 등을 포함*) 실행 관례를 왜 이용하지 않느냐고 물으면 많은 개발자가 이렇게 답한다.

> 당신은 우리 회사를 모릅니다. 같이 일하는 사람들이 어떤지 몰라서 그래요. 이런 것들은 우리 회사에서 제대로 될 수 없어요. 소프트웨어 장인정신에 공감하지만, 우리 팀은 그렇지 않습니다.

각 회사의 상황은 다르다. 하지만 여러 회사에서 일해본 경험으로 볼 때 **공통으로 안고 있는 문제들이 많다**. 개발 역량 부족, 느린 대응 속도, 관료주의 등이 그것이다. 그리고 개별 상황과 독립적인 것들도 많다.

- 고객 만족에 신경 쓰지 않는 회사가 정말 있는가?
- 투자 대비 이익을 기대하지 않는 회사가 있는가?
- 요구사항을 만족시키지 못하는 소프트웨어에 돈을 지불하려는 회사가 있는가?

### 2.1 XP의 짧은 역사

1996년, 켄트 벡(*Kent Beck - TDD와 XP의 창시자, JUnit 개발자*)은 XP의 기원이 된 실행 관례 묶음을 발표했다. 이후 크라이슬러사의 급여 지급 시스템(C3) 프로젝트의 리더로 일하며 관례들을 다듬었다. C3에는 마틴 파울러(*Martin Fowler - 리팩터링 및 소프트웨어 아키텍처 저술가*), 론 제프리스, 체트 헨드릭슨, 돈 웰스 등 유명한 애자일 지지자들이 참여했다.

C3 프로젝트는 XP 실행 관례를 사용해 성공했고, 이후 돈 웰스가 포드사로 옮겨 동일한 실행 관례로 다시 성공하자 특이 사례로 치부하기 어려워졌다. XP 실행 관례 도입 후 양산 제품 버그는 1/3로 줄고, 디버깅 시간은 제로에 근접했으며, 생산성은 10배까지 증대했다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 2001년 애자일 서밋 이후 XP도 애자일의 하나로 분류되지만, 애자일을 도입한 회사 중 XP 실행 관례를 활용하는 곳은 매우 드물다. 애자일 코치와 컨설팅 회사 다수가 XP 실행 관례를 가르칠 만한 역량이 부족하기 때문이다.
:::

## 3. 실행 관례와 가치 — 어떻게 설득할 것인가

XP 실행 관례는 도입만으로 저절로 성과를 내지 않는다. **매일같이 습관처럼 해야 하는 것**이다. TDD를 한다면 하거나 하지 않거나 둘 중 하나다. "어떤 때는 TDD를 한다"는 말은 실제로 하지 않는다는 뜻이다.

관리자에게 스크럼의 가치를 납득시키기는 쉽지만 페어 프로그래밍이나 TDD의 가치를 이해시키기는 어렵다. "그런 것들이 비즈니스적으로 어떤 가치가 있나?"라는 질문이 되돌아온다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 기술적 실행 관례 자체를 직접적으로 팔려고 들면 아무런 의미가 없다. 도입 자체가 아니라 **현재 일하는 방식과 비교해서 그것이 가져올 이익**에 집중해야 한다. 빠른 피드백 루프, 지식 공유, 줄어드는 버그, 자동화되고 빨라지는 릴리즈 — 이것이 기술적 실행 관례가 주는 비즈니스 가치다.
:::

## 4. 주요 XP 실행 관례들

### 4.1 자동화된 테스트

자동화된 테스트는 클릭 한 번으로 전체 시스템을 몇 분 만에 검증할 수 있게 해준다. 코드 수정 후 몇 분 만에 안심하고 릴리즈에 반영할 수 있고, QA 검증을 피할 수 있다면 며칠에서 몇 주의 시간을 벌 수 있다.

| 구분 | 피드백 주기 |
|---|---|
| 수동 QA 회귀 테스트 | 며칠 ~ 몇 주 |
| 자동화 테스트 스위트 | 몇 분 |

### 4.2 테스트 먼저(Test First)

테스트 코드를 먼저 작성하면 여러 장점이 있다.

- 아이디어를 정리하고 한 번에 하나에만 집중할 수 있다
- 모듈/클래스/함수를 구체적으로 정의하도록 강제한다
- 딱 필요한 만큼만 코딩하도록 유도하여 오버 엔지니어링(*Over-engineering - 필요 이상으로 복잡하게 설계하는 것*)을 줄여준다
- 테스트 코드는 잘 정리된 요구사항 역할도 한다

### 4.3 테스트 주도 개발(TDD)

TDD는 이름에 "테스트"가 들어가 있지만 사실 **설계에 대한 실행 관례**다. 테스트가 코딩 방향을 주도하면 복잡한 코드를 작성하는 것 자체가 어려워진다. 그 이유는 두 가지다.

1. 정확히 요구사항만큼만 만족시키는, 즉 테스트로 규정된 부분만 작성하게 된다. 그래서 BDUF(*Big Design Up Front - 처음부터 큰 설계를 하는 방식*)를 피할 수 있다
2. 코드가 복잡하고 방대하면 테스트 자체가 어렵기 때문에, 테스트 하기 어려운 코드가 곧 설계 재검토와 리팩터링의 원인이 된다

TDD는 정기적인 설계 리뷰 미팅보다 훨씬 빠르고 객관적이다. 설계 리뷰는 필요한 것이지만 큰 변화 앞에서 몇 차례 정도가 적당하고, 자주 열면 참여자의 기여 욕구로 오히려 오버 엔지니어링이 되기 쉽다. TDD는 코드 한 줄마다 유용한 피드백을 준다.

TDD는 코드에 대한 살아 움직이는 문서 역할을 하며, 부가 효과로 회귀 테스트 역할도 해준다.

### 4.4 지속적 통합(Continuous Integration)

팀 하나는 보통 4~10명으로 구성된다. 큰 프로젝트는 여러 팀이 여러 국가와 시간대에 흩어져 있다. 어떻게 서로의 발을 밟지 않고 일할 수 있을까?

지속적 통합은 TDD와 함께 수행되어 피드백 루프를 몇 분으로 줄인다. 개발자가 코드를 올릴 때마다 전체 테스트 스위트가 실행되고, 실패하면 이메일이 모든 팀에 전달된다.

이 실행 관례는 "일단 멈추고 버그부터 수정한다"는 태도가 필요하다. 팀원들은 하던 일을 멈추고 마지막 변경점에서 문제 부분을 수정하는 데 집중해야 한다. 그 대신 시스템이 항상 배포 가능한 상태로 유지되고 버그가 누적되지 않는다.

### 4.5 페어 프로그래밍

코드 리뷰(*Code Review - 다른 개발자가 작성한 코드를 검토하고 피드백을 주는 절차*)는 흔히 사용되지만 피드백 주기가 문제다.

| 방식 | 피드백 주기 |
|---|---|
| 코드 리뷰 (주 1회) | 1주일 — 이미 그 코드에 의존한 다른 코드가 쌓여 있을 수 있다 |
| 코드 리뷰 (격주) | 2주일 |
| 페어 프로그래밍 | 즉시 (코드가 작성되는 순간) |

페어 프로그래밍을 하면 "4개의 눈"으로 즉시 검증된다. 변수 이름을 짓는 순간 다른 개발자가 "notLoggedIn 대신 guest로 바꾸면 어떨까요?", "이 메서드는 한 가지 일이 너무 많습니다. private 메서드로 분할하면 어떨까요?"라고 말할 수 있다.

같은 페어끼리 너무 오래 있으면 효과가 적으므로 하루이틀 단위로 페어를 바꾼다. 그러면 시스템에 대한 이해도와 스킬이 팀 차원에서 누적된다.

### 4.6 리팩터링

"엉망인 코드가 많을수록 엉망인 코드가 늘어나는 속도도 빨라진다." 이것은 깨진 유리창 법칙(*Broken Windows Theory - 사소한 방치가 더 큰 문제를 부른다는 이론*)으로도 알려져 있다.

지속적으로 리팩터링(*Refactoring - 외부 동작을 유지한 채 내부 구조를 개선하는 작업*)하면 이 위험이 줄어든다. 프로젝트 시작 단계부터 이 실행 관례를 받아들이면 코드가 점진적으로 좋아지고, 리팩터링에 큰 시간을 들일 일이 없어진다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 실용주의 관념 없이 리팩터링하는 것은 위험하다. 몇 년 동안 바뀐 적 없는 부분을 리팩터링하는 것은 의미가 없다. 리팩터링은 더 자주 변경되는 부분을 대상으로, 보이스카웃 규칙에 따라 해당 부분을 이해하고 변경할 필요가 있을 때 적용해야 한다.
:::

## 5. 실용주의와 책임감

기술, 방법론, 실행 관례는 계속 진화한다. 오늘날 훌륭한 것으로 여겨지는 것이 미래에도 그렇다는 보장은 없다. 어쩌면 수년 뒤에는 소프트웨어 장인정신이 장려하는 실행 관례 목록이 완전히 달라질 수도 있다.

**실용주의는 소프트웨어 장인이 가져야 하는 최선의 역량 중 하나다.** 누군가가 이야기했기 때문에, 또는 도입을 위한 도입을 해서는 안 된다. 지속적으로 더 나은 방법을 찾고 고객을 만족시켜야 한다. 결론이 TDD 도입이라면 그렇게 한다. 언제든 더 나은 가치를 주는 다른 것이 있다면 그것을 도입한다.

### 5.1 실행 관례를 거부할 때

이러한 실행 관례들의 가치를 설명해도 여전히 많은 사람들이 받아들이기를 거부한다. "그런 것들 없이도 좋은 소프트웨어를 개발할 수 있다"는 말이 반복된다.

그러한 항변을 들으면 다음과 같이 되물어야 한다.

> 이러한 가치와 최소한 동등한 수준의 가치를 만들어내기 위해 당신은(혹은 우리는) 무엇을 하고 있습니까? 이러한 실행 관례보다 더 나은 방법이 있습니까?

의사결정에 책임감을 가져야 한다. 관리자 역시 팀이 특정 실행 관례를 따르지 못하도록 할 때는 그에 대한 책임감이 있어야 한다. 이러한 결정을 기록하고 문제가 있을 때 에스컬레이션(*Escalation - 상급자나 상위 조직으로 이슈를 올려 판단을 요청하는 것*)해야 한다.

## 요약

- 애자일 절차 관례는 "올바른 일"을 하는지 알려주고, 기술적 실행 관례는 "일을 올바르게" 하는지 알려준다. 둘 다 필요하다
- 코드 품질이 나빠지고 있음은 개발자에게만 보인다. 나쁜 코드는 암과 같아서, 뒤늦게 발견되면 대응이 불가능하다
- XP 실행 관례는 크라이슬러 C3, 포드 프로젝트에서 실증되었다. 버그는 1/3, 디버깅은 제로, 생산성은 10배까지 향상되었다
- 실행 관례는 습관처럼 매일 해야 한다. "때때로 TDD를 한다"는 것은 하지 않는 것과 같다
- 관리자를 설득할 때는 실행 관례 자체를 팔지 말고, 빠른 피드백 루프/자동화된 QA/짧아지는 릴리즈 주기 등의 **비즈니스 가치**를 강조한다
- 주요 XP 실행 관례: 자동화된 테스트, 테스트 먼저, TDD, 지속적 통합, 페어 프로그래밍, 리팩터링
- TDD는 테스트가 아니라 **설계에 대한 실행 관례**다. 오버 엔지니어링을 억제하고 살아 있는 문서 역할을 한다
- 실용주의를 잃지 말라. 실행 관례는 도입 자체가 목적이 아니라 가치를 만들기 위한 수단이다. 더 나은 방법이 나타나면 언제든 바꿔야 한다
- 실행 관례 도입을 거부하는 사람에게는 "그렇다면 동등한 가치를 만들 다른 방법이 있느냐"고 되묻고, 결정에 대한 책임을 명확히 해야 한다

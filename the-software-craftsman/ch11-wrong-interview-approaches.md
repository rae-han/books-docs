# Chapter 11: Wrong Interview Approaches (잘못된 면접 방식)

## 핵심 질문

소프트웨어 장인을 유인하고 싶은 회사가 절대 하지 말아야 할 면접 방식은 무엇인가? 왜 두뇌 장난 같은 수수께끼 문제나 종이에 코드 쓰기, 알고리즘 문제 위주 면접이 좋은 개발자를 오히려 쫓아내는가? 면접관의 태도와 질문 유형이 어떻게 회사 문화를 드러내는가?

---

## 1. 채용 절차는 회사 문화의 거울이다

회사의 채용 절차를 하나하나 뜯어보면 그 회사가 추구하는 **가치와 문화**를 엿볼 수 있다. 경험 많고 재능 있는 개발자들은 자신이 일할 회사를 신중하게 가려서 선택한다. 이들은 금전적 보상 수준보다 다음을 더 우선한다:

- 자율성(*autonomy - 스스로 결정하고 통제할 권한*)
- 배움
- 목적 의식
- 생산적 파트너십
- 열정적인 사람들
- 좋은 업무 환경

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 어떤 채용 절차를 거치는지, 특히 면접에서 어떤 사람을 만나는지는 지원자가 그 회사에 들어갈 것이냐 말 것이냐를 결정짓는 매우 중요한 요소다.
:::

소프트웨어 장인을 유인하려는 회사라면 다음 면접 방식들을 **의식적으로 배제**해야 한다.

---

## 2. 안티패턴 1 — 똑똑한 척하는 면접관

면접관이 지원자보다 똑똑하거나 우월해 보이고 싶어 해서는 안 된다. 사사로운 즐거움을 위해 지원자를 힘든 상황으로 몰아붙이고, 자신의 직함·권한·지식으로 지원자를 압도하려 해서는 안 된다.

**거만하고 오만한 태도는 금물이다.** 상황에 맞지도 않고 배배 꼬인 질문으로 똑똑한 척하는 순간, 경험 있는 개발자는 면접관의 성향을 즉시 알아채고 같이 일하기 싫다는 결론을 내린다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 채용은 파트너가 될 사람을 찾는 과정이다. 고분고분하게 시키는 대로만 하지 않고, 팀이 더 나아지도록 도와줄 수 있는 사람을 뽑아야 한다. 지원자 앞에서 겸손하고 정직해야 하며, 면접은 취조가 아니라 존중하는 누군가와의 유익한 기술 토론이 되어야 한다.
:::

---

## 3. 안티패턴 2 — 수수께끼식 질문

직무와 전혀 관계없는 바보 같은 질문은 하지 말아야 한다.

- "비행기에 골프공이 몇 개나 들어갈까요?"
- "정육점 점원의 키가 180cm이고 280mm 스니커즈를 신었습니다. 몸무게는?"

이런 두뇌 장난 같은 질문에 답할 수 있느냐 없느냐는 **좋은 코드 작성, 팀 플레이, 프로페셔널 태도**와 전혀 관계없다. 완전히 시간 낭비다. 구글의 면접관들도 이런 질문이 잘못됐다는 것을 깨닫고 그만둔 지 오래되었다.

---

## 4. 안티패턴 3 — 답을 모르는 질문을 한다

면접관으로서 어떤 질문에 어떤 답변이 나와야 하는지 잘 모르겠다면, 채용 직무와 관련해서는 그다지 중요한 질문이 아닐 가능성이 높다. 해당 직무를 잘 알고 있다면 무엇이 중요하고 중요하지 않은지 이미 알고 있을 것이다.

> 팀 동료에게 흔히 하지 않는 질문이라면, 팀 동료가 짜증을 낼 질문이라면, 지원자에게도 삼가야 한다.

---

## 5. 안티패턴 4 — 지원자를 바보로 만든다

저자가 대규모 통신 회사 면접을 볼 때의 이야기다. 이전에 다른 통신 회사에서 근무한 경험이 있고, 사내 고참 개발자가 저자를 추천한 터라 면접관들이 큰 관심을 보였다.

부서 담당 임원과 수석 아키텍트가 함께 참석했다. 임원은 일반 질문으로 시작했고, 대화는 자연스럽게 애자일·자율성·협업으로 수렴했다. 임원은 15분 동안 수차례 아키텍트를 돌아보며 "**바로 이것이 제가 항상 얘기하려던 점입니다. 이런 것들이 마음에 듭니다.**"라고 말했다.

그런데 수석 아키텍트는 심기가 불편해 보였다. 그는 저자가 이전에 설계했던 아키텍처를 화이트보드에 그려달라고 요청했다. 저자가 설명을 마치자 아키텍트는 임원을 잠깐 쳐다보고는 저자를 향해 무시하는 말투로 이렇게 말했다.

> "너무 단순하네요. 그건 제대로 된 실제 아키텍처가 아닙니다."

경멸하는 태도가 확실히 느껴졌다. 저자는 심호흡을 하고 마음을 가라앉힌 뒤 답했다.

> "감사합니다. 저는 그 말씀을 큰 칭찬이라고 생각하겠습니다. 이 단순한 아키텍처가 **3개 대륙에 걸쳐 2천만 명이 넘는 가입자**를 서비스하고 있습니다. 이보다 더 복잡하게 만들지 않고서도 해냈다는 것을 기쁘게 생각합니다."

이 말을 하기 전까지 아키텍트는 저자를 그저 좋아하지 않는 수준이었겠지만, 이제는 저자를 증오하고 있을 것이 뻔했다. 임원이 면접을 되살리려 애썼지만 파트너십은 물 건너갔다는 것이 분명했다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 지원자를 바보로 만들려는 시도는 회사의 문화를 그대로 노출한다. 실력 있는 지원자는 즉시 알아채고 즉시 이별을 통보한다.
:::

---

## 6. 안티패턴 5 — 인터넷 접속을 막는다

코딩 면접을 할 때 인터넷 접속을 막는 회사들이 있다. 인터넷 없이 지원자의 진짜 실력을 알 수 있다는 이유다.

그러나 실제 업무에서 인터넷 없이 코딩하는가? 코딩에 필요한 모든 것을 머릿속에 외우고 있는가? **아니다.**

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 솔루션을 탐색하고 더 나은 문제 대응 방법을 찾는 것은 소프트웨어 개발자의 핵심 능력이다. 인터넷 검색 때문에 코딩 면접이 변별력을 잃을 수 있다면 **면접 과제 자체에 문제가 있다**.
:::

---

## 7. 안티패턴 6 — 종이나 화이트보드에 코드 작성

지원자에게 종이나 화이트보드에 코드를 작성하게 하는 것은 참으로 바보 같은 방법이다. 면접관 스스로도 할 수 없는 일이나 실제 업무 현장에서 부딪히지 않을 상황을 지원자에게 요구해서는 안 된다.

| 실제 업무 vs 종이 면접 | |
|---|---|
| 실제 업무에서 종이에 테스트 코드를 작성하는가? | 아니오 |
| 종이 코드를 리팩터링하기 위해 종이를 찢어 재조합하는가? | 아니오 |
| IDE의 자동완성/리팩터링 도구를 못 쓴다면? | 개발자의 실제 도구 숙련도를 볼 수 없다 |

**자신의 도구와 기술을 마스터하고 테스트·리팩터링으로 잘 작성된 코드를 만드는 개발자를 찾아야 한다.** 종이나 화이트보드에 불편하게 끄적인 코드가 아니라, 실제 도구로 생산된 실제 코드를 평가해야 한다.

> 고등학교 교사가 학생들에게 알고리즘의 의사코드(*pseudocode - 실제 실행되지 않는 알고리즘 서술*)를 작성하라고 하는 것과, 프로페셔널 소프트웨어 개발자를 채용하는 것은 다르다.

---

## 8. 안티패턴 7 — 알고리즘 문제

면접용 코딩 문제로 무엇이 적합할까? 많은 면접관이 **알고리즘 연습 문제**를 선택한다. "지원자의 문제 해결 능력을 봐야 한다"는 논리다.

물론 틀린 이야기는 아니지만, 알고리즘 문제 대신 회사의 실제 프로젝트와 가까운 다른 연습 문제로도 문제 해결 능력을 평가할 수 있다.

### 8.1 실제 시스템의 문제는 알고리즘이 아니다

여러 시스템의 문제들 중 거의 대부분은 알고리즘이 어떻게 작성되었느냐와는 관계가 없었다. 실제 문제는 다음과 같은 것들이다:

- 테스트가 덜 되었거나 좋은 테스트 방법이 준비되지 못함
- 잘못된 설계
- 떨어지는 응집성(*Cohesion - 하나의 모듈이 얼마나 단일 목적을 가지는가*)
- 깊은 종속성
- 새 기능 추가 시 부족했던 리팩터링
- 지속적인 요구사항 변경
- 도메인 모델(*Domain Model - 비즈니스 개념을 코드로 표현한 모델*)이 꼼꼼하지 못함

알고리즘은 절차적·함수적 형태로 구현된다. 알고리즘을 만들 때 비즈니스 도메인 모델이나 클래스를 만들지는 않는다. **시스템의 주요 문제가 알고리즘이 아니라면 코딩 면접도 알고리즘 문제 대신 실제 문제에 가까운 과제를 제시해야 한다.**

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 요지는 알고리즘 문제를 내야 하느냐의 여부가 아니라, **프로젝트를 위해 실제 필요한 역량이 무엇인지, 무엇이 가장 가치 있는지**를 코딩 문제에 잘 반영해야 한다는 것이다. 애플리케이션이 온통 알고리즘이라면 당연히 알고리즘 역량을 평가해야 한다.
:::

## 9. 안티패턴 8 — 전화 면접

전화 면접은 사전 선별이 부족할 때만 필요하다. 많은 회사가 오프라인 면접 전에 전화 면접을 하는데, 지원자가 최소한의 채용 요건을 갖추었는지 확인하는 목적이다.

### 9.1 전화 면접의 문제점

| 문제 | 이유 |
|---|---|
| 기계적으로 흐르기 쉬움 | 많은 사람 중 선별 목적이라 급하게 진행 |
| 인간미 없음 | 딱딱한 질문·답변으로 흐름 |
| 언어 장벽 | 모국어가 아닌 경우 전화로 뉘앙스 전달 불가 |
| 몸짓·표정 없음 | 종이·화이트보드 없이 의도 표현 어려움 |
| 즉각적 이해 확인 불가 | 상대가 제대로 알아들었는지 알 수 없음 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 해외 지원자처럼 어쩔 수 없을 때만 전화 면접을 사용하자. **오프라인에서 얼굴을 마주 보는 면접이 가장 바람직하다.**
:::

## 10. 좋은 면접 vs 나쁜 면접 비교

지금까지 살펴본 안티패턴을 정리하면 다음과 같다.

### 나쁜 면접

```
[똑똑한 척 하는 면접관] + [수수께끼 질문] + [답 모르는 질문]
     ↓
[종이 코드 작성] + [인터넷 차단] + [알고리즘 문제]
     ↓
[전화 면접으로 대충 선별]
     ↓
훌륭한 지원자는 손절
```

### 좋은 면접

```
[겸손하고 정직한 면접관] + [실제 도구로 페어 프로그래밍]
     ↓
[회사의 실제 문제와 가까운 코딩 과제] + [자유로운 인터넷 검색]
     ↓
[대면 대화 + 자유토론]
     ↓
서로가 파트너십을 확인
```

## 요약

- 채용 절차는 회사의 가치와 문화를 그대로 노출한다. 좋은 개발자는 절차만 봐도 회사의 정체를 판단할 수 있다
- **똑똑한 척하는 면접관**은 지원자를 즉시 잃는다. 면접은 취조가 아니라 존중하는 기술 토론이어야 한다
- **수수께끼 질문·두뇌 장난**은 좋은 코드나 팀 플레이 능력과 무관하다. 완전한 시간 낭비다
- **답을 모르는 질문을 하지 말라**. 팀 동료에게 하지 않을 질문이면 지원자에게도 하지 말라
- 저자가 "너무 단순하네요"라는 아키텍트의 말에 "3개 대륙 2천만 명을 서비스한다"고 답한 사례처럼, **지원자를 바보로 만들려는 시도는 실력자를 잃는 지름길**이다
- **인터넷 접속 차단**은 무의미하다. 검색 능력은 개발자의 핵심 역량이다. 검색으로 뚫린다면 과제가 잘못됐다
- **종이·화이트보드 코딩**은 실무와 무관하다. 실제 도구로 만든 실제 코드를 평가해야 한다
- **알고리즘 문제**를 무조건 배제하라는 것이 아니라, 프로젝트에 진짜 필요한 역량을 반영한 문제를 내라는 뜻이다. 대부분 시스템의 실제 문제는 알고리즘이 아니라 설계·테스트·응집성·도메인 모델링에 있다
- **전화 면접**은 해외 지원자처럼 어쩔 수 없을 때만 사용한다. 얼굴을 마주 보는 것이 최선이다
- 좋은 개발자를 원한다면 **좋은 개발자가 원하는 방식**으로 면접하라. 면접은 지원자가 회사를 면접하는 자리이기도 하다

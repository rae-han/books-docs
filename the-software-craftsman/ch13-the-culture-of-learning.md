# Chapter 13: The Culture of Learning (배움의 문화)

## 핵심 질문

개발자들의 태도와 업무 방식을 바꾸는 진짜 동력은 무엇인가? 관리자가 "이제부터 모든 팀은 TDD를 해야 합니다"라고 통보하면 왜 변화는 일어나지 않는가? 배움의 문화를 만드는 것은 관리자와 애자일 코치의 일인가? 관리자의 지원이 없어도 개발자 스스로 학습 환경을 만들 수 있는가? 그리고 아무도 참여하지 않는다면 그때는 어떻게 해야 하는가?

---

## 1. 잘못된 방향의 동기 부여 — 부도덕한 프로젝트 사례

저자는 컨설팅 회사에 근무하던 시절 어느 유명 투자은행의 이상한 요청을 받았다. "우리 레거시 코드베이스에 단위 테스트를 역으로 끼워 맞춰 만들어 줄 개발자가 필요합니다."

그 프로젝트에는 여러 이상한 점이 있었다:

- 코드가 회사 바깥으로 나오는 것을 금기시하는 금융권임에도 컨설팅 회사 사무실에서 작업하기를 원했다
- 레거시 코드를 만든 원 개발자와의 커뮤니케이션이 계약 조건에서 배제되어 있었다
- 목표는 오직 "테스트 커버리지 70% 달성"이었다

몇 달 뒤 프로젝트가 끝나고서야 진상이 드러났다. 고객사 고위 기술 담당의 그해 팀 목표에 "테스트 커버리지 70% 달성"이 있었고, **팀 보너스가 그 지표에 직접 연동**되어 있었다. 목표 달성 부분을 그냥 아웃소싱했던 것이다.

> 그때 우리는 부도덕한 일을 했던 것이다. 잘못된 테스트는 아예 테스트가 없는 것보다 못하다. 비즈니스 맥락을 전혀 모르는 외부 개발자들이 자기 마음대로 이해한 수준에서 검증했으니, 클래스에 버그가 있더라도 그것이 버그인지 원래 의도인지 몰라 그저 그 행동에 끼워 맞춰 테스트를 만들었다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 사람들에게 새로운 절차나 실행 관례를 강제한다고 조직이 변화하지는 않는다. 배움의 문화(*Culture of Learning - 조직 구성원 스스로 배우고 공유하려는 열정이 유지되는 환경*)를 만들어야 한다. 지표에 보너스를 연동하는 순간 사람들은 그 지표를 채우는 편법을 찾는다.
:::

## 2. 왜 배움의 문화가 필요한가

**롤 모델이 없으면 주니어는 잘못된 커리어 관을 학습한다.** 자기 일에 불평만 쏟아내고 동기부여가 되어 있지 않은 선배 개발자만 본 주니어들은 소프트웨어 개발이 신나는 커리어가 아니라고 믿게 된다. 그들은 어떻게 하면 더 빨리 관리자나 아키텍트가 될지에 열중하며 그것이 자연스러운 커리어 개발이라고 확신한다.

반대로 배움과 자기 계발에 열정이 있고 자기 일에 애정이 있는 선배 개발자와 일하면 어떤가? 그런 환경은 소프트웨어 장인이 되고자 하는 열망을 불어넣고 개발 자체를 즐기게 만든다. 이렇게 동기 부여가 된 사람들은 항상 회사 바깥세상에 관심을 갖기 때문에 회사는 외부의 혁신과 효율을 지속적으로 수혈받을 수 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 관리자는 개발자에게 무엇을 언제 어떻게 하라고 명령할 것이 아니라 권한을 위임하고, 개발자 스스로 배움의 문화를 만들어 갈 수 있도록 북돋워야 한다. 명령이 되는 순간 흥미는 사라진다.
:::

## 3. 배움의 문화를 만드는 구체적 활동들

### 3.1 활동 카탈로그

| 활동 | 방식 | 필요한 것 |
|---|---|---|
| **북 클럽** | 책 한 권을 골라 주 1회 점심시간에 토론 | 관심 있는 동료 한 명 |
| **테크 런치(*Tech Lunch - 점심시간 기술 난상 토론회*)** | 준비 없이 그 자리에서 주제를 던져 토론 | 회의실 하나 |
| **그룹 토론회** | 화이트보드를 둘로 나눠 짧은 발표 + 그룹 토론으로 진행 | 포스트잇, 펜, 진행 시간 관리 |
| **업무 교환** | 팀끼리 개발자를 스왑하여 페어 프로그래밍 | 페어 프로그래밍 문화 |
| **얼마 동안만 업무 교환** | 몇 시간~며칠 다른 팀 개발자 옆에서 페어 | 개발자 간 합의만 있으면 됨 |
| **그룹 코드 리뷰** | 특정 코드에 대해 개선점을 논쟁 | 퍼실리테이터, 비난 금지 규칙 |
| **코딩 실습(*Coding Kata - 반복 연습용 프로그래밍 훈련*)** | 카타를 골라 페어 TDD로 풀기 | 퍼실리테이터, 2시간 |
| **내부 학습 모임** | 두 사람으로 시작하는 정기 모임 | 정기적 리듬 |
| **펫 프로젝트(*Pet Project - 개인이 재미 삼아 하는 자율 프로젝트*)** | 근무 외 시간이든 근무 시간이든 자율 진행 | 회사의 허용 분위기 |
| **외부 커뮤니티 참여** | 지역 사용자 그룹, 콘퍼런스, 오픈소스 | 동료와 함께 참석 |

### 3.2 그룹 토론회 진행법

1. 화이트보드를 둘로 나눈다: 한쪽은 **짧은 주제 발표(5분 + 2분 Q&A)**, 다른 한쪽은 **그룹 토론 후보**
2. 참여자들이 포스트잇에 발표 주제 작성
3. 5분 발표 → 시간 초과된 Q&A는 그룹 토론 대상으로 옮김
4. 그룹 토론 후보 낭독 → 비슷한 것 병합
5. 손을 들어 투표(가능하면 1인 1표)
6. 다득표 주제 45분 진행 후 계속할지 재확인

득표가 비슷하다면 회의실을 둘로 나누거나, 재투표하거나, 시간을 앞뒤로 나눠 진행한다.

### 3.3 그룹 코드 리뷰 원칙

- 주석은 **주관적/개인적이지 않아야** 한다
- 누가 코드를 작성했는지는 중요하지 않다
- **커밋 히스토리를 뒤지지 않는다**: 비난할 사람을 찾기 위해 과거를 파헤치지 말고 미래에 집중
- 주석은 반드시 **객관적/생산적**이어야 하고 "왜 엉망인지"가 아닌 "어떻게 개선할지"에 집중
- **퍼실리테이터**의 도움: 모두에게 발언권을 주고 비난 상황을 방지
- 큰 진전을 기대하지 않는다

### 3.4 좋은 코딩 실습의 지침

- 퍼실리테이터는 **연습 문제를 사전에 두 번 이상** 풀어봐야 한다
- 페어 프로그래밍으로 풀되, 서로 처음 페어링하는 조합을 만든다
- 별도 지정이 없다면 **TDD를 기본 원칙**으로 지킨다
- 모든 개발자가 같은 문제를 풀어 해답을 비교한다
- 퍼실리테이터는 **정답이나 힌트를 주지 않고 비판적 질문**을 던진다
- 실습 후 리뷰 시간을 반드시 갖는다
- 페어 파트너에게 감사를 표한다
- 실습에 2시간을 할당하되 **1시간은 근무시간 + 1시간은 개인시간** — 회사와 개인 모두 배움에 투자한다

> 사용할 기술은 자유롭게 선택하게 하라. 언어나 프레임워크를 지정하면 편하지만, 그 기술에 관심 없는 개발자는 아예 참여하지 않는다.

## 4. 아무도 참여하지 않는다면 — 조언 모음

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 조직 전체 또는 부서/팀 전체를 한 번에 변화시키려는 시도의 대다수는 실패한다. 몇몇 사람만이라도 자부심과 즐거움, 열정을 되찾도록 돕는 것으로 충분하다.
:::

| 조언 | 요지 |
|---|---|
| **모범을 보여라** | "팀의 다른 개발자들이 관심 없어서 나도 안 한다"는 변명이다. 스스로 하지 않으면서 남을 설득할 수 없다 |
| **관심 있는 사람에게 집중하라** | 변화를 수용하는 사람과 페어 프로그래밍, 코드 리뷰, 토론을 즐기다 보면 함께하려는 사람이 늘어난다 |
| **강제하지 마라** | 참여 싫다는 사람을 억지로 데려오면 자발적 참여자의 흥미까지 꺾인다 |
| **모두를 변화시키려 들지 말라** | 서너 달 노력해도 아무도 관심 없다면 만족하자. 소수만으로도 큰 진보가 가능하다 |
| **모임 약속을 제때 하라** | 모두의 일정을 조율하기보다 "매주 수요일 12시" 같은 고정 시간을 공표하는 편이 낫다 |
| **허락을 구하지 마라** | 점심시간, 이른 아침, 늦은 오후의 빈 회의실이면 충분하다. 책임감 있게 행동하기만 하면 관리자가 불평할 이유가 없다 |
| **투덜대지 마라** | 사무실이 부적합하면 카페, 인터넷이 안 되면 노트북 준비, 다른 지역이면 그 지역 모임 |
| **리듬을 만들라** | 언제 어디서 열리는지 모두 알면 참여가 쉬워진다. 당신이 없을 때 대타를 준비해 계속 굴러가게 하라 |

> 개방된 커뮤니티의 원칙 중 하나는 "들어오는 사람이 초대받은 사람이다"라는 태도다.

---

## 5. 관리자에 대한 오해 풀기

배움의 문화를 만드는 것은 **관리자나 애자일 코치의 전담 업무가 아니다**. 관리자로부터의 지원이 없더라도 개발자들 스스로 서로 배우고 공유하는 환경을 얼마든지 만들 수 있다.

시간이 없다면 점심시간이나 업무 전 시간을, 장소가 없다면 카페를 이용한다. 매일이 어려우면 주 1회로 시작한다. 배움의 문화를 만드는 것은 쉽고 비용도 거의 들지 않는다. 열정적인 개발자에게 좋은 환경을 제공하기만 하면 그들 스스로 무엇을 배우고 공유할지 방법을 찾는다.

## 요약

- **강제된 변화는 실패한다**: 보너스를 특정 지표에 연동하면 개발자는 지표를 채우는 편법을 찾는다. 테스트 커버리지 70% 아웃소싱 사례는 그 극단이다
- **롤 모델의 부재가 커리어 관을 왜곡한다**: 열정 없는 선배만 본 주니어는 관리자/아키텍트가 되는 것을 자연스러운 커리어라고 착각한다
- **관리자는 명령이 아닌 위임을 해야 한다**: 언제, 무엇을, 어떻게 배울지 개발자 스스로 정하게 두라
- **활동은 다양하다**: 북 클럽, 테크 런치, 그룹 토론, 업무 교환, 그룹 코드 리뷰, 코딩 실습, 내부 학습 모임, 펫 프로젝트, 외부 커뮤니티 참여
- **코딩 실습의 핵심**: 문제 풀이가 아니라 훈련. 퍼실리테이터 사전 준비 + TDD 기본 + 같은 문제 공유 + 리뷰 시간 + 근무/개인 시간 반반 투자
- **아무도 참여하지 않을 때의 원칙**: 모범을 보이고, 관심 있는 사람에게 집중하고, 강제하지 않고, 모두를 바꾸려 하지 말고, 정기적 리듬을 만들어라
- **허락을 구하지 마라**: 배움의 문화는 관리자 지원 없이도 시작할 수 있다. 카페, 점심시간, 회의실 하나면 충분하다
- **"들어오는 사람이 초대받은 사람"**: 나가는 사람에게 실망하거나 화내지 말고 애초의 동기를 꾸준히 유지하라

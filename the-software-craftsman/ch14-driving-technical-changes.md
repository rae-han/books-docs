# Chapter 14: Driving Technical Changes (기술적 변화의 실행)

## 핵심 질문

팀에 TDD, 페어 프로그래밍, 새로운 프레임워크 같은 기술적 실행 관례를 어떻게 전파할 수 있는가? 왜 상사보다 실무 개발자를 설득하는 것이 더 어려운가? 어떤 종류의 회의론자를 어떻게 다루어야 하는가? 상사를 설득할 수 없다면 어떻게 해야 하는가? "상아탑 아키텍트"처럼 코딩하지 않는 결정권자의 방해를 어떻게 우회할 것인가?

---

## 1. 회의론(*Skepticism - 새로운 아이디어에 대한 반발/의심*)의 종류

테렌스 라이언(*Terrence Ryan - 『Driving Technical Change』 저자*)의 분류에 저자의 관찰을 더한 회의론자 유형이다.

### 1.1 라이언의 분류

| 유형 | 특징 |
|---|---|
| **무지(Uninformed)** | 새로운 도구/실행 관례의 존재조차 모른다. 두려움에 검토 없이 거부 |
| **대중(Herd)** | 자신감이 부족해 리더에게 결정을 맡긴다. 압박이 오면 쉽게 수용 |
| **냉소주의(Cynic)** | 논쟁을 즐기고 사소한 것으로 꼬투리를 잡는다. 새로운 것이 권위를 위협한다고 느낀다 |
| **트라우마(Burned)** | 과거 실패 경험이 있어 반복하고 싶어 하지 않는다 |
| **너무 바쁜(Time crunched)** | 장기 비용을 못 보고 근시안적 판단. 올바르게 할 시간은 없지만 같은 일을 반복할 시간은 있다 |
| **상사(Boss)** | 기술 배경이 없거나, 있어도 "내가 개발할 땐 TDD 없이 잘했다"고 반박 |
| **몰상식(Irrational)** | 최악의 타입. 논리적 근거 없이 이 문제에서 저 문제로 옮겨 다닌다. 그냥 받아들이기 싫을 뿐 |

### 1.2 저자의 추가 분류

| 유형 | 특징 |
|---|---|
| **무념무상** | 뭐가 되든 상관하지 않는다. 좋은 아이디어를 대충 구현해 나쁜 아이디어로 만든다 |
| **피해망상** | 회사가 자신에게 피해를 준다고 생각. 프로젝트를 일부러 망친다 |
| **무능력** | 제안의 본질을 파악할 인지 능력이 부족하다. 왜곡된 사실에 근거한 설익은 아이디어 |
| **상아탑 아키텍트(*Ivory Tower Architect - 코드를 쓰지 않으면서 개발팀에 지시하는 아키텍트*)** | 모든 것을 안다고 생각. 수년간 상용 코드 한 줄도 안 쓴 채 파워포인트로 아키텍처를 정한다 |
| **좌불안석** | 대체될까 두려워 새 기술 도입을 위협으로 본다. 사내 유일 IT 담당인 경우가 많다 |
| **팬 보이** | 특정 도구/언어/프레임워크에 광적으로 전념. 자신이 아는 그것만이 최고의 해답이라 믿는다 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 사람들은 바보가 아니다. 서로 다른 이유와 의견을 가질 뿐이다. 그들의 의견은 과거 경험, 당신이 모르는 정보, 자신감 부족, 또는 두려움에 기반한다. 각 회의론 유형별로 대응 전략이 달라야 한다.
:::

## 2. 변화를 시도하기 전의 준비

**용기**가 가장 중요하다. 언성 높은 논쟁을 두려워해서는 안 된다. 그 밖에도 챙길 것이 있다.

- **단순함을 추구하라**: 아이디어는 명료해야 한다. 예제를 이용하라
- **상대방의 언어로 말하라**: 관리자에게는 소스코드가 아니라 "유지보수 비용 절감", "릴리즈 주기 단축"
- **말할 내용을 스스로 이해하라**: 반대 의견을 미리 예상하고 답을 준비. 제안의 결점도 먼저 밝혀라
- **상대방을 존중하라**: 무례하면 방어적으로 만들어 설득이 불가능해진다
- **경청하는 법을 배워라**: 같은 문제라도 관점은 서로 다르다

---

## 3. 기술적 변화를 시작하는 방법

### 3.1 신뢰를 쌓으라

품질 좋은 소프트웨어를 **지속적으로** 딜리버리하는 것이 신뢰의 기본이다. 다만 "품질"의 의미는 상대에 따라 다르다.

| 상대 | 품질의 의미 |
|---|---|
| 동료 개발자 | 좋은 설계, 명확한 네이밍, 의미 있는 테스트, 중복 없는 코드 |
| 제품 관리자 | 요구사항 충족, 버그 없음, 일정 준수 |
| 최종 고객 | 상용 환경에서 돈을 벌거나 절약해주고 매출을 지켜주는 것 |

### 3.2 전문성을 확보하라

새로운 기술을 제안하기 전에 **본인 스스로 충분히 이해**하고 다른 사람에게 가르쳐 줄 수 있어야 한다. 전문성이 축적되면 다양한 회의론에 각각의 무기가 생긴다.

- '무지'에게는 지식을 준다
- '대중'은 당신을 함께할 사람으로 여긴다
- '냉소주의자'는 자신감과 지식 앞에 힘을 못 쓴다
- '트라우마'가 있는 사람의 아픔을 이해하게 된다
- '상아탑 아키텍트', '팬 보이'와의 논쟁에 무기가 된다

### 3.3 모범을 보여 사람들을 이끌라

TDD처럼 태도까지 바꿔야 하는 실행 관례는 **직접 보여주며 따라 하게 하는 것**이 최선이다. 옆에 앉아 함께 코드를 작성하면 처음 해보는 것에 대한 두려움이 사라진다.

### 3.4 신중하게 싸울 곳을 정하라

한 번에 모든 이슈, 조직 전체를 상대할 수는 없다. 저자는 투자은행에서 다음 순서로 진행했다:

1. **채용 절차** 개선 → 새로 합류할 사람이 사전에 원하는 문화와 맞도록
2. **TDD와 페어 프로그래밍** 도입 (1과 병렬)
3. 블랙박스 테스팅 프레임워크 → 오픈 소스 솔루션 교체
4. 아키텍처/디자인 개선

### 3.5 점진적으로 반복, 관찰, 수용하라

큰 충돌을 피하는 가장 쉬운 방법은 **점진적 변화**를 제안하는 것이다.

> "이번 스프린트에서만 이 접근을 시도해봅시다. 결과가 나쁘면 되돌리면 됩니다. 한 번 시도할 뿐 결정되는 것은 없습니다."

작업 주기의 경계를 활용해 팀 차원에서 채택 여부를 결정하게 만든다. 모두가 의사결정 과정에 참여하고 있다고 느끼면 저항이 협조로 바뀐다.

## 4. 두려움과 자신감 부족

두려움이 있는 데다가 자신감이 부족하면 회사는 관료적이고 정치적으로 변한다. "그게 잘못된 건 알지만 상사에게 문제 제기할 수는 없다. 그는 내 인사권자다"라는 태도는 개선의 싹을 자른다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 무능한 사람만이 일자리를 불안해한다. 소프트웨어 장인은 자신의 일과 커리어를 스스로 통제한다. 승진을 위한 정치 게임에 매달릴 이유도, 일자리 하나에 목을 맬 이유도 없다. 준비하고, 공부하고, 최고의 개발자가 되면 두려움은 사라진다.
:::

---

## 5. 상사를 설득하는 방법 — "설득할 수 없다"

**용서를 구하는 것이 허락을 구하기보다 쉽다.** 관리자들은 TDD/페어 프로그래밍/CI를 하든 말든 관심이 없다. 그들은 일정 준수, 예산, 버그 없는 결과에만 관심 있다.

만약 개발자가 관리자에게 "이 코드는 꼭 동작해야 하나요? 시스템에 손대면 버그가 생겨도 괜찮나요?"라고 물으면 어떤 관리자가 "그래도 됩니다"라고 답할까?

**단위 테스트, 리팩터링은 코딩 업무에 당연히 내재된 사항**이지 별개의 작업이 아니다. 상세 구현을 관리자에게 일일이 요청할 필요가 없다.

> 무슨 일을 하든 고객에게 가치를 전달해야 한다. 진척 상황은 정직하고 투명하게 말한다. 단, 상세 구현을 일일이 노출하면 관리자가 세세하게 지시하고 싶게 만든다.

## 6. 팀에 TDD를 수용시키는 방법

큰 충돌 없이 도입하는 최고의 방법은 **도움을 요청**하는 것이다.

> "이번 스토리를 도와줄 수 있어요? 한두 시간 TDD로 구현해봅시다. 재미로 한 번 해봐요."

**핑퐁 코딩(*Ping-Pong Coding - TDD의 페어링 방식으로 한 사람은 테스트, 다른 사람은 구현을 작성하며 매 테스트 통과 시 역할 교대*)**을 활용해 붉은 마크가 초록색으로 바뀔 때의 성취감을 공유한다.

한 가지 주의: 페어링을 하려면 **먼저 자신이 TDD에 능숙**해야 한다. 어설픈 TDD 소개는 "TDD는 어렵고 느려서 가치가 없다"는 확신만 심어주는 최악의 상황을 만든다.

## 7. 회의론 유형별 대응

| 유형 | 대응 |
|---|---|
| 무지 | 지식을 준다 |
| 대중 | 방향을 제시한다 |
| 냉소주의 | 자신감과 깊은 지식으로 납득시킨다 |
| 트라우마 | 이해하고 어디가 잘못됐는지 짚어준다 |
| 너무 바쁜 | 학습 곡선을 짧게 하는 방법을 제공 |
| 상사 | 상사의 언어(생산성, 비용 절감, 매출 증대)로 말한다 |
| 몰상식 | **무시한다**. 논리적 논쟁을 원하는 사람에 집중 |
| 무념무상/무능력/팬 보이 | 롤 모델이 되어 이끈다 |
| 좌불안석 | "당신의 자리를 위협하지 않는다"는 것을 명확히 |
| 상아탑 아키텍트 | 그의 결정에 책임을 지도록 만든다 |
| 피해망상 | 개인적으로 해결이 어렵다. 팀 융합을 도우면서 필요시 에스컬레이션 |

## 8. 상아탑 아키텍트 사례 — 권한과 책임

저자는 큰 조직에서 상아탑 아키텍트와 극단적인 대화를 나눈 적이 있다. 아키텍트는 "누가 이 웹 프레임워크 사용을 허가했나?"고 물었고, 저자는 프로젝트에 가장 적합해서 팀이 선택했다고 답했다.

핵심은 이 순간이었다:

***저자**: "결정 권한은 당신에게 있지만 그에 대한 책임은 개발팀이 진다. 프로젝트가 잘되면 당신 공, 잘못되면 개발팀 책임. 그런 식으로 일할 수는 없다. 두 가지 중 선택하라. (1) 개발팀이 모든 결정과 책임을 지거나, (2) 당신이 모든 기술적 의사결정의 책임자이니 문제가 발생하면 가장 먼저 당신에게 연락하라고 하겠다."*

***아키텍트**: "모든 결정을 문서화하고 나에게 공유해주세요."* (그는 나를 죽이고 싶은 감정을 숨기려 애쓰며 말했다)

이 논쟁 이후 팀은 외부 영향 없이 자유롭게 의사결정했고, 몇 달 뒤 그 아키텍트는 다른 부서로 이동했다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 진정한 프로페셔널은 **권한에는 항상 책임이 따른다**는 것을 이해한다. 권한을 갖고 싶다면 책임질 준비를 해야 하고, 이미 책임이 주어져 있다면 관련 의사결정 권한도 가져야 한다. 상아탑 아키텍트를 피하고 싶다면 그의 결정에 책임을 지도록 만들어라.
:::

---

## 9. 이 모든 것을 다 챙겨야 하는가

소프트웨어 장인은 기술적 역량뿐 아니라 **주니어의 롤 모델 역할**도 할 수 있어야 한다. 정직하고 투명하며, 고객을 속이거나 문제를 숨기지 않는다. 사람들이 듣고 싶은 말이 아니라 진심을 말하며, 프로젝트를 위해 싸우기를 주저하지 않는다.

잘못된 조직에서 완벽한 솔루션을 딜리버리하려면 단지 코드 잘 쓰는 것만으로는 부족하다.

## 요약

- **회의론에는 여러 유형**이 있다: 무지, 대중, 냉소주의, 트라우마, 너무 바쁨, 상사, 몰상식 + 무념무상, 피해망상, 무능력, 상아탑 아키텍트, 좌불안석, 팬 보이. 각각 다른 접근이 필요하다
- **준비물은 용기 + 단순함 + 상대의 언어 + 스스로의 이해 + 존중 + 경청**
- **신뢰는 지속적 딜리버리에서 나온다**. 다만 "품질"의 정의는 개발자/PM/고객마다 다르다
- **점진적 변화를 제안하라**: "한 번 시도할 뿐 결정되는 것은 없습니다"로 저항을 낮춘다
- **상사는 설득할 수 없다** — 그냥 해라. 테스트/리팩터링은 코딩 업무의 내재된 부분이지 별도 작업이 아니다
- **TDD 전파는 도움 요청으로 시작**한다. 재미로 페어링, 핑퐁 코딩으로 성취감 공유. 단, 스스로 능숙해진 후에
- **몰상식은 무시하라**. 논리 논쟁의 대상이 아니다
- **상아탑 아키텍트는 책임을 지게 하라**: 권한과 책임은 분리될 수 없다
- **두려움을 버려라**: 무능한 사람만이 일자리를 불안해한다. 준비하고 공부해 최고가 되면 정치 게임에 몰두할 필요가 없다

# Chapter 6: Working Software (동작하는 소프트웨어)

## 핵심 질문

애자일 선언(*Agile Manifesto - 2001년 발표된 애자일 소프트웨어 개발의 4가지 가치와 12가지 원칙*)이 강조하는 "동작하는 소프트웨어"는 그 자체로 충분한가? 손대기 두려운 레거시 코드도 결국 "동작"만 한다면 동작하는 소프트웨어라고 부를 수 있는가? 왜 나쁜 코드는 개발자에게만 보이고 관리자와 비즈니스에는 보이지 않는가? 개발자에게 "단위 테스트를 작성할 시간이 없다"라는 말이 성립할 수 있는가?

---

## 1. 동작하는 소프트웨어만으로는 부족하다

소프트웨어 장인정신 운동을 마음에 들어하지 않는 사람들이 가장 먼저 이야기하는 반론은, 잘 작성되고 깨끗한 코드만으로는 프로젝트를 성공시킬 수 없다는 것이다. 물론 맞는 이야기다. 소프트웨어 프로젝트가 실패하는 원인은 잘못된 비즈니스 의사결정, 더 우월한 경쟁자, 잘못된 프로젝트 관리, 높은 비용, 늦은 시장 대응, 파트너십 부재, 기술적 한계, 시스템 통합 오류 등 매우 다양하다. 그러다 보니 상대적으로 덜 중요해 보이는 코드 품질에는 조직 차원의 주의를 기울이지 않는 경향이 있다.

애자일 선언에서는 광범위한 문서보다 동작하는 소프트웨어를 강조하고, 두 가지를 추가로 언급한다.

- 동작하는 소프트웨어를 자주, 몇 주나 2개월 단위의 짧은 주기로 전달한다
- 동작하는 소프트웨어는 진척도를 가늠하는 주된 기준이다

이는 훌륭한 원칙이다. 문제는 시간이 흐르면서 이 개념이 "고품질의 동작하는 소프트웨어"로 옮겨가지 않고, 여전히 수준 이하의 코드가 넘쳐난다는 점이다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 매일같이 부딪히는 엉망진창인 레거시 코드들도 분명 "동작하는 소프트웨어"의 범주에 포함된다. 그러나 손을 대는 것조차 무서운 코드, 작은 수정 하나마다 방대한 수동 테스트가 뒤따르는 코드가 "동작하는 소프트웨어로서 충분한 조건"을 갖췄다고 말할 수 있는가? 이런 상황은 품질 개념을 잃어버렸거나 그저 체념하고 받아들이는 것이다.
:::

## 2. 정원 돌보기 — 코드는 유기물이다

> 프로그래밍은 집을 짓는다기보다는 정원을 돌보는 것에 더 가깝다.
> — 『실용주의 프로그래머』

코드는 기계장치라기보다는 유기물이다. 1년 내내 정원이 아름다우려면 지속적인 돌봄이 필요하다. 토양을 손보고, 잡초를 뽑고, 정기적으로 물을 주고, 죽은 화초를 없애고, 새로운 화초를 심고, 화초들이 건강하게 잘 자라도록 가지치기를 해야 한다. 짧은 기간이라도 돌보기를 게을리하면 다시 아름답게 가꾸는 데 훨씬 더 많은 노력이 들어간다.

코드도 마찬가지다. 정기적으로 살피지 않으면 변화가 있을 때마다, 새 기능을 추가할 때마다 상태가 나빠진다. 잘못된 설계, 테스트 부족, 프로그래밍 언어나 도구의 미숙한 활용도 코드를 더 빨리 썩게 만든다. 점점 다른 파트의 코드까지 오염되고 결국 유지보수 비용은 감당할 수 없을 만큼 커진다.

---

## 3. 보이지 않는 위협 — 서서히 진행되는 침식

프로젝트를 시작할 때는 모든 것이 훌륭하다. 얽매일 레거시 코드가 없어 개발자는 걱정 없이 새 기능을 개발할 수 있고, 테스터도 회귀(*Regression - 기존에 잘 동작하던 기능이 변경 후 망가지는 현상*) 테스트 부담이 없다. 관리자에게도 진척도가 빨라 보인다.

이력서 키워드와 희망 급여만으로 채용된 그저 그런 개발자들, 개발을 공장 라인처럼 취급하는 환경이라면 시간이 갈수록 상황은 악화된다. 원인 파악도 안 된 버그들이 생겨나고 기능 구현과 테스트에 걸리는 시간이 점점 더 길어진다. 이 모든 것이 몇 개월 혹은 몇 년에 걸쳐 아주 천천히 진행되기 때문에 관리자는 눈치채지 못한다.

| 시점 | 같은 규모의 기능 구현 시간 |
|---|---|
| 프로젝트 초기 (깨끗한 코드) | 짧다 |
| 중후반 (나쁜 코드 누적) | 훨씬 길어진다 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 나쁜 코드가 가장 위험한 이유는 개발자 외에는 아무도 그것을 볼 수 없기 때문이다. 관리자나 비즈니스 담당이 문제를 인지했을 때는 이미 늦다. 코드 품질을 돌보는 책임은 오직 개발자에게만 있다.
:::

### 3.1 인질이 된 비즈니스

기능을 수정하거나 새로 추가할 때 코드베이스에 너무 많은 시간을 들여야 한다면, 개발자들이 기존 코드에 손대는 것을 두려워한다면 즉시 대응해야 한다. 소프트웨어가 비즈니스의 발목을 잡는 상황은 매우 위험하다. **비즈니스의 기민함과 투자 효율을 위해 다른 무엇보다도 코드의 품질이 가장 중요하다.**

---

## 4. 시간에 대한 잘못된 인식

아침에 일어나면서 "오늘은 하루를 망쳐야겠다. 최악의 코드를 짜서 상사와 팀을 골탕 먹이겠어"라고 마음먹는 사람은 없다. 그럼에도 왜 많은 애자일 프로젝트가 실패할까? 개발자들은 시간에 대해 잘못된 개념을 가진 경우가 많다.

> 항상 업무와 일정의 압박 속에 있다는 생각을 가지고 있다. 언제나 급하게 일해야 한다고 느낀다. 비즈니스팀이 일정에 관해 압박할 때도 있지만 스스로 쫓기는 경우가 대부분이다.

빨리 하는 것과 허술한 것은 다르다. 며칠 더 늦게 안정적인 솔루션을 전달하기보다 버그가 있더라도 일정 안에 전달하는 편이 낫다고 느끼는 것은 잘못된 인식이다.

### 4.1 저자의 경험 — 시간이 없어서 테스트를 못 한다는 팀

저자가 대기업의 한 팀에 합류했을 때 이야기다. 자바 기반 프로젝트에 70여 개의 Maven 프로젝트와 모듈이 얽혀 있었다. 개발자 한 명과 페어 프로그래밍을 하면서 그가 어떻게 일하는지 지켜보았는데, 그 절차는 이랬다.

1. 로컬에서 코드를 일부 수정한다
2. 애플리케이션을 패키징한다
3. 테스트 시스템에 복사한다
4. 테스트 시스템에서 배포한다
5. 큐에 메시지를 넣고 로그를 살펴본다
6. 다시 1로 돌아간다

로컬에서 애플리케이션 구동 자체가 불가능해서 이 무한 루프를 반복하고 있었다. 저자가 "이제 원인이 밝혀졌으니 단위 테스트를 만드는 게 낫지 않겠느냐"고 물었을 때 그 개발자는 "이 부분은 정말 중요하기 때문에 빨리 끝내야 해서 테스트 코드를 작성할 시간이 없다"고 답했다. 이틀 동안의 삽질 끝에 밝혀진 원인은 단위 테스트로 즉시 잡을 수 있었을 사소한 논리 오류였다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: "테스트를 작성할 시간이 없다"라고 말하는 개발자는 정작 그로 인해 낭비되는 훨씬 더 많은 시간을 계산하지 못한다. 로컬-테스트 시스템을 오가는 수동 디버깅, 다른 개발자의 대기 시간, QA 반복 등을 모두 합치면 단위 테스트를 작성하는 시간을 훨씬 넘어선다.
:::

## 5. 단위 테스트는 별개의 업무가 아니다

개발 업무를 작은 단위로 나눌 때 "단위 테스트 구현"을 별도 작업으로 정의하는 개발자가 많다. 이는 반드시 피해야 한다.

### 5.1 이유 1 — 테스트는 이미 코딩 방식에 녹아 있다

테스트하지 않았다면 코드 작성을 완료했다고 할 수 없다. 단위 테스트는 코드가 제대로 구현되었는지 확인하는 가장 좋은 방법이며, 기능 구현 완료의 정의(*Definition of Done - 팀이 합의한 작업 완료 조건*)에 반드시 포함되어야 한다.

### 5.2 이유 2 — 별도 작업으로 올리면 제거되기 쉽다

공식 작업 일정표에 올려놓는 순간, 제품 오너가 "가치 없다"고 판단하여 삭제할 수 있는 권리를 부여한 것과 같다. 경험이 부족한 제품 오너에게 단위 테스트는 낭비로 비칠 수 있고, 자신의 역할에 충실하려는 선한 의도로 이 작업을 없앨 가능성이 높다.

### 5.3 QA 전담팀은 안티패턴

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 테스터가 버그를 발견하는 것은 개발자로서 대단히 수치스러운 일이다. 테스트 담당자의 역할은 개발자가 예상하지 못한 인간의 행동을 반영해 문제를 찾아내는 데 있지, 자동화 가능한 것을 수동으로 검사하는 데 있지 않다.
:::

1990년대만 해도 디버깅 스킬이 채용 면접의 주요 주제였다. 지금은 세상이 달라졌다. 코드 수준에서 디버깅을 해야 하는 상황 자체가 대단히 당혹스럽고 불편하다. 저자는 TDD(*Test-Driven Development - 테스트를 먼저 작성한 뒤 구현하는 개발 방법론*)에 익숙해진 이후로 코드 디버깅을 해야 할 상황이 손에 꼽을 정도로 드물었다.

### 5.4 자기중심적 개발자

"내가 짠 코드는 내가 잘 안다. 문제 없으니 테스트는 필요 없다"고 주장하는 개발자는 자기중심적이다. 소프트웨어 프로젝트는 팀워크다. 자신에게 쉬운 것이 팀 내 다른 개발자에게는 난해할 수 있다. 시스템이 커질수록 개인의 이기적 행동으로 인해 프로젝트 관계자 모두가 피해를 입는다.

## 6. 레거시 코드 — 태도의 변화

> 태도는 큰 차이를 가져올 수 있는 작은 요소다.
> — 윈스턴 처칠

소프트웨어 장인정신 토론회에서 그린필드(*Greenfield - 백지 상태에서 시작하는 신규 프로젝트*) 프로젝트를 선호하는 사람이 있냐는 질문에 거의 모두가 손을 들었다. 반면 레거시 코드를 다루는 프로젝트를 선호하는 사람은 아무도 없었다.

레거시 코드로 일할 때는 두 가지 관점 중 하나를 선택할 수 있다. 하나는 지금 하고 있는 일을 고통스러운 전생의 업보로 보는 것이고, 다른 하나는 재미있고 도전적인 문제로 바라보는 것이다. 우리는 후자를 선택해야 한다.

### 6.1 직소 퍼즐로서의 레거시 코드

레거시 코드로 일하는 것은 거대한 직소 퍼즐을 푸는 것과 비슷하다.

| 단계 | 접근 방식 |
|---|---|
| 1단계 | 모든 조각을 한꺼번에 맞추려 들지 않는다 |
| 2단계 | 색상이나 패턴 기준으로 조각들을 그룹으로 분리한다 |
| 3단계 | 작은 그룹부터 하나씩 맞춘다 |
| 4단계 | 맞춰진 그룹들이 늘어날수록 남은 조각을 맞추기가 쉬워진다 |

코드로 옮기면 이런 과정이다. 테스트 코드를 점진적으로 작성하며 이해도를 높이고, 메서드와 클래스를 정리하고, 변수명을 적합하게 바꾼다. 종속성이 해소되고 역할이 정의되면 새로운 프레임워크를 도입하거나 라이브러리 버전을 업그레이드할 수도 있다.

### 6.2 보이스카웃 규칙

"처음 발견했을 때보다 더 깨끗하게 남겨두라"는 보이스카웃 규칙을 지속적으로 적용해야 한다. 남이 작성한 코드를 엉망이라고 비웃기는 쉽다. 하지만 "나라면 더 잘 만들 수 있는가?"를 스스로에게 물어야 한다.

코드 리팩터링은 중독될 정도로 재미있을 수 있지만, 우리는 대가를 받는 프로페셔널이라는 사실을 잊어서는 안 된다. 소프트웨어를 깨끗하게 할수록 고객은 그 소프트웨어를 통해 더 오랫동안 이익을 얻고, 애플리케이션 수명이 늘어난다는 것은 고객의 투자 이익을 극대화한다는 의미다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 그린필드 프로젝트이든 브라운필드(*Brownfield - 레거시 코드가 존재하는 기존 프로젝트*) 프로젝트이든 상관하지 않게 될 것이다. 일을 즐길 수 있느냐 없느냐는 우리의 태도에 달려 있다.
:::

## 요약

- "동작하는 소프트웨어"만으로는 부족하다. 손대기 두려운 레거시 코드도 동작은 하지만, 그것이 충분한 조건을 갖췄다고 볼 수는 없다
- 코드는 정원과 같아서 꾸준한 돌봄이 필요하다. 방치할수록 되살리는 데 드는 비용이 기하급수적으로 커진다
- 나쁜 코드는 개발자 외에는 보이지 않는 서서히 진행되는 위협이다. 관리자가 눈치챘을 때는 이미 늦다
- "시간이 없다"는 인식은 대부분 잘못이다. 단위 테스트를 안 만들어서 절약한 시간보다, 수동 디버깅과 QA 반복으로 잃는 시간이 훨씬 크다
- 단위 테스트는 별개의 업무가 아니라 코딩의 일부다. 별도 항목으로 일정표에 올리면 제품 오너에 의해 삭제되기 쉽다
- QA 전담팀은 안티패턴이다. 테스터가 자동화 가능한 것을 수동으로 검사하며 시간을 낭비해서는 안 된다
- 레거시 코드는 태도의 문제다. 고통스러운 업보가 아니라 재미있는 도전으로 바라보고, 보이스카웃 규칙을 적용하며 점진적으로 개선해야 한다
- 도끼날을 정기적으로 가는 시간은 낭비가 아니다. 그것이야말로 시간을 절약하고 끊임없이 빨리 움직일 수 있는 최선의 방법이다

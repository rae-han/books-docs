# Chapter 27: Coding (코딩)

## 핵심 질문

> "코드(code)"라는 단어가 모스 부호에서 프로그래밍으로 의미가 확장된 여정에서, 기계어, 어셈블리, 고급 언어는 각각 어떤 역할을 하며, 추상화의 계층은 왜 필수적인가?

## 2. 기계어 (Machine Language)

### 2.1 정의

**기계어(machine code)**는 CPU가 직접 이해하고 실행하는 **이진 명령어**다.

```
10110000 01100001
│        │
│        └── 오퍼랜드: 0x61 = 97 = 'a'
└── 오피코드: MOV AL (AL 레지스터에 즉시값 이동)
```

### 2.2 기계어의 문제

```
10110000 01100001 10001000 00000101 01000000
11100100 01100001 00000000 00000000 11001101
00100001
```

이것이 무슨 프로그램인지 **사람은 알 수 없다**. 기계어는:
- 읽기 극히 어려움
- 쓰기 극히 어려움
- 디버깅 극히 어려움
- CPU 아키텍처마다 완전히 다름

---

## 3. 어셈블리 언어 (Assembly Language)

### 3.1 기계어의 인간 번역판

기계어의 이진 코드를 사람이 읽을 수 있는 **니모닉(mnemonic)**으로 대체한 것:

| 기계어 | 어셈블리 | 의미 |
|--------|----------|------|
| 10110000 01100001 | MOV AL, 97 | AL에 97 저장 |
| 00000101 00000011 | ADD AL, 3 | AL에 3 더하기 |
| 11101000 xxxx | CALL func | 함수 호출 |
| 11001101 00100001 | INT 21h | 인터럽트 호출 |

### 3.2 어셈블러 (Assembler)

어셈블리 언어 → 기계어로 변환하는 프로그램:

```
어셈블리 소스 (.asm)  →  [어셈블러]  →  기계어 (.exe/.bin)
```

어셈블리와 기계어는 **1:1 대응**이다. 어셈블리 한 줄 = 기계어 한 명령어.

### 3.3 어셈블리의 한계

- 여전히 하드웨어에 종속: CPU가 다르면 어셈블리도 다름
- 낮은 추상화: 레지스터, 메모리 주소를 직접 관리
- 생산성이 낮음: 간단한 작업도 많은 줄의 코드

## 4. 고급 프로그래밍 언어

### 4.1 왜 고급 언어인가?

```
어셈블리로 "a = b + c":          C 언어로:
    MOV  AX, [b_addr]            a = b + c;
    ADD  AX, [c_addr]
    MOV  [a_addr], AX
```

고급 언어는 하드웨어 세부사항을 숨기고 **인간의 사고방식에 가까운 표현**을 제공한다.

### 4.2 주요 프로그래밍 언어의 역사

| 언어 | 연도 | 특징 | 핵심 기여 |
|------|------|------|-----------|
| **FORTRAN** | 1957 | 최초의 성공적 고급 언어 | 과학 계산, 컴파일러의 실용성 증명 |
| **COBOL** | 1959 | 영어에 가까운 문법 | 비즈니스 프로그래밍 |
| **BASIC** | 1964 | 간단한 문법, 대화형 | 프로그래밍 교육의 대중화 |
| **C** | 1972 | 저수준 접근 + 고수준 구조 | Unix, 시스템 프로그래밍 |
| **C++** | 1983 | C + 객체지향 | 대규모 소프트웨어 |
| **Java** | 1995 | "한 번 쓰면 어디서든 실행" | 플랫폼 독립성, 가상 머신 |
| **Python** | 1991 | 간결한 문법, 다목적 | 스크립팅, AI/ML, 교육 |
| **JavaScript** | 1995 | 웹 브라우저용 | 웹 동적 기능 |

### 4.3 고급 언어의 핵심 구성 요소

| 구성 요소 | 고급 언어 | 기계어 대응 (Ch 24) |
|-----------|-----------|---------------------|
| **변수** | `int x = 5;` | 메모리 주소에 값 저장 |
| **조건문** | `if (x > 0) {...}` | CMP + 조건부 JUMP |
| **반복문** | `while (x > 0) {...}` | CMP + 조건부 JUMP (역방향) |
| **함수** | `int add(int a, int b)` | CALL + RET + 스택 |
| **배열** | `arr[i]` | 기저 주소 + 오프셋 |

## 5. 컴파일러와 인터프리터

### 5.1 컴파일러 (Compiler)

소스 코드 **전체**를 한 번에 기계어로 변환:

```
소스 코드 (.c)  →  [컴파일러]  →  기계어 (.exe)  →  실행
```

- 변환 시간이 걸리지만, **실행 속도가 빠름**
- 대표 언어: C, C++, Rust, Go

### 5.2 인터프리터 (Interpreter)

소스 코드를 **한 줄씩** 읽고 즉시 실행:

```
소스 코드 (.py)  →  [인터프리터]  →  즉시 실행 (줄 단위)
```

- 변환 없이 바로 실행 가능하지만, **실행 속도가 느림**
- 대표 언어: Python, Ruby, JavaScript (초기)

### 5.3 하이브리드 방식

현대 많은 언어는 두 방식을 혼합:

```
소스 코드  →  [컴파일러]  →  바이트코드  →  [가상 머신/JIT]  →  기계어  →  실행
```

- Java: .java → .class(바이트코드) → JVM(가상 머신)
- Python: .py → .pyc(바이트코드) → PVM
- JavaScript: 소스 → JIT 컴파일러 → 최적화된 기계어

## 6. 추상화의 계층

이 책 전체를 관통하는 핵심 주제:

```
인간의 의도: "웹 페이지에 'Hello'를 표시하라"
    ↓
고급 언어:  console.log("Hello");
    ↓
바이트코드/IR:  LOAD_CONST "Hello" / CALL print
    ↓
어셈블리:  MOV RDI, hello_str / CALL printf
    ↓
기계어:   48 89 C7 E8 xx xx xx xx
    ↓
CPU 제어 신호: PC_out, MEM_read, IR_in, ALU_add...
    ↓
논리 게이트: AND, OR, NOT, XOR
    ↓
트랜지스터: 전압으로 전류 제어
    ↓
전자: 도체를 따라 흐르는 전자
```

> **핵심 통찰**: 이 책의 여정은 전자의 흐름(Ch 4)에서 시작하여 프로그래밍 언어(이 챕터)에 도달했다. 각 층은 아래 층의 복잡성을 숨기고 위 층에 단순한 인터페이스를 제공한다. 이 **추상화 계층(layers of abstraction)**이 인간이 수십억 개의 트랜지스터를 다룰 수 있게 해주는 핵심 원리다.

## 요약

- **기계어**: CPU가 직접 실행하는 이진 명령어. 사람이 읽기/쓰기 극히 어려움
- **어셈블리**: 기계어의 니모닉 표현. 1:1 대응. 어셈블러가 기계어로 변환
- **고급 언어**: 인간의 사고에 가까운 표현. 변수, 조건문, 반복문, 함수 등
- **컴파일러**: 소스 전체를 기계어로 변환 (빠른 실행). **인터프리터**: 한 줄씩 실행 (느리지만 편리)
- "코드"의 의미: 모스 부호부터 프로그래밍까지, 모두 **정보의 형태를 변환하는 체계**

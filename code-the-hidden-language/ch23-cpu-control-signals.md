# Chapter 23: CPU Control Signals (CPU 제어 신호)

## 핵심 질문

> ALU, 레지스터, 메모리, 버스를 올바른 순서로 동작시키는 "지휘자"는 무엇이며, 명령어가 실제로 실행되는 과정은 어떤 단계를 거치는가?

---

## 1. 제어 장치 (Control Unit) — CPU의 지휘자

지금까지 만든 모든 부품이 모여 있지만, 누가 "지금 이 레지스터를 버스에 올려라", "ALU에 덧셈을 시켜라", "저 레지스터에 저장하라"고 지시하는가?

**제어 장치(Control Unit)**가 그 역할을 한다. 클럭의 각 박자에 맞춰 적절한 **제어 신호(control signal)**를 생성하여 CPU의 모든 부품을 조율한다.

> **Petzold의 비유**: 제어 장치는 오케스트라의 지휘자다. 악기(ALU, 레지스터, 메모리)는 모두 준비되어 있지만, 지휘자가 "지금 바이올린 연주!", "이제 드럼!"이라고 지시해야 음악(계산)이 된다.

---

## 2. 명령어 사이클 (Instruction Cycle)

### 2.1 페치-디코드-실행 (Fetch-Decode-Execute)

모든 명령어는 세 단계를 거친다:

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  Fetch  │────→│ Decode  │────→│ Execute │
│ (가져옴) │     │ (해석)   │     │ (실행)   │
└─────────┘     └─────────┘     └─────────┘
     ↑                                │
     └────────────────────────────────┘
              (다음 명령어로 반복)
```

**1단계 — Fetch (가져오기):**
```
1. PC의 값을 주소 버스에 올림
2. 메모리 읽기 신호 활성화
3. 메모리에서 명령어가 데이터 버스에 나옴
4. 데이터 버스의 값을 IR(명령어 레지스터)에 저장
5. PC = PC + 1 (다음 명령어를 가리킴)
```

**2단계 — Decode (해석):**
```
1. IR의 오피코드 부분을 분석
2. 어떤 연산인지 결정 (ADD? LOAD? STORE? JUMP?)
3. 필요한 제어 신호를 생성
```

**3단계 — Execute (실행):**
```
(연산에 따라 다른 동작)
- ADD: ALU에서 덧셈 수행, 결과를 누산기에 저장
- LOAD: 메모리에서 데이터를 읽어 레지스터에 저장
- STORE: 레지스터의 데이터를 메모리에 기록
- JUMP: PC를 새 주소로 변경
```

### 2.2 클럭 사이클 단위의 실행

각 단계는 하나 이상의 **클럭 사이클**이 필요하다. 간단한 명령어는 3-4 사이클, 복잡한 명령어는 더 많은 사이클이 필요할 수 있다.

---

## 3. 제어 신호의 상세

### 3.1 제어 신호 목록 (예시)

| 제어 신호 | 역할 |
|-----------|------|
| PC_out | PC의 값을 주소 버스에 올림 |
| MAR_in | 주소 버스의 값을 MAR에 저장 |
| MEM_read | 메모리 읽기 |
| MEM_write | 메모리 쓰기 |
| IR_in | 데이터 버스의 값을 IR에 저장 |
| A_in | 데이터 버스의 값을 누산기에 저장 |
| A_out | 누산기의 값을 데이터 버스에 올림 |
| ALU_add | ALU에 덧셈 연산 지시 |
| ALU_sub | ALU에 뺄셈 연산 지시 |
| PC_inc | PC 증가 |
| PC_in | 데이터 버스의 값을 PC에 저장 (점프) |

### 3.2 ADD 명령어의 마이크로 연산

"ADD [addr]" (메모리 주소 addr의 값을 누산기에 더하라):

| 클럭 사이클 | 활성화 신호 | 동작 |
|------------|------------|------|
| T₁ | PC_out, MAR_in | PC → 주소 버스 → MAR |
| T₂ | MEM_read, IR_in, PC_inc | 메모리[MAR] → IR, PC+1 |
| T₃ | IR_addr_out, MAR_in | IR의 주소 부분 → MAR |
| T₄ | MEM_read, ALU_add, A_in | 메모리[MAR] → ALU, A+메모리값 → A |

---

## 4. 명령어 형식 (Instruction Format)

### 4.1 구조

```
┌────────────────┬────────────────┐
│   오피코드      │   오퍼랜드      │
│   (연산 종류)   │   (연산 대상)   │
│   4~8비트       │   8~24비트      │
└────────────────┴────────────────┘
```

### 4.2 오퍼랜드의 종류 (주소 지정 모드)

| 모드 | 의미 | 예시 |
|------|------|------|
| **즉시값** | 오퍼랜드 자체가 데이터 | ADD #5 (5를 더해라) |
| **직접 주소** | 오퍼랜드가 메모리 주소 | ADD [42] (주소 42의 값을 더해라) |
| **레지스터** | 오퍼랜드가 레지스터 번호 | ADD B (레지스터 B의 값을 더해라) |

---

## 5. 간단한 명령어 집합 설계

8비트 오피코드 + 8비트 오퍼랜드 = 16비트 명령어:

| 오피코드 | 니모닉 | 의미 |
|----------|--------|------|
| 00000001 | LOAD addr | 메모리[addr] → A |
| 00000010 | STORE addr | A → 메모리[addr] |
| 00000011 | ADD addr | A + 메모리[addr] → A |
| 00000100 | SUB addr | A - 메모리[addr] → A |
| 00000101 | JUMP addr | PC ← addr |
| 00000110 | JZ addr | Z=1이면 PC ← addr |
| 00000111 | JNZ addr | Z=0이면 PC ← addr |
| 11111111 | HALT | 실행 중지 |

### 프로그램 예제: 5 + 3

```
주소 00: LOAD  10     ; 메모리[10]의 값(=5)을 A에 로드
주소 02: ADD   11     ; 메모리[11]의 값(=3)을 A에 더함
주소 04: STORE 12     ; A의 값(=8)을 메모리[12]에 저장
주소 06: HALT         ; 종료
...
주소 10: 05           ; 데이터: 5
주소 11: 03           ; 데이터: 3
주소 12: 00           ; 결과가 저장될 위치
```

> **핵심 통찰**: 이 간단한 프로그램은 **명령어(코드)와 데이터가 같은 메모리에 공존**하는 것을 보여준다. 주소 00-06에는 명령어가, 주소 10-12에는 데이터가 있다. CPU는 PC가 가리키는 곳의 바이트를 명령어로 해석하고, 명령어가 가리키는 곳의 바이트를 데이터로 해석한다. 같은 비트 패턴이 맥락에 따라 다르게 사용되는 것이다.

---

## 요약

- **제어 장치**: 클럭에 맞춰 제어 신호를 생성하여 CPU의 모든 부품을 조율하는 "지휘자"
- **명령어 사이클**: Fetch(가져오기) → Decode(해석) → Execute(실행) → 반복
- **제어 신호**: PC_out, MEM_read, ALU_add 등 각 부품의 동작을 켜고 끄는 신호
- **명령어 형식**: 오피코드(무엇을) + 오퍼랜드(무엇에)
- **주소 지정 모드**: 즉시값, 직접 주소, 레지스터 등 오퍼랜드를 해석하는 방법

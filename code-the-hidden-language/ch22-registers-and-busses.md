# Chapter 22: Registers and Busses (레지스터와 버스)

## 핵심 질문

> CPU 내부의 데이터는 어디에 임시 저장되며, 레지스터, 메모리, ALU 사이의 데이터 이동은 어떻게 이루어지는가?

---

## 1. 레지스터 (Register)

### 1.1 레지스터란?

**레지스터**는 CPU **내부**에 있는 초고속 소용량 저장소다. 플립플롭(Ch 17)으로 구성된 n비트 저장 장치.

| 비교 | 레지스터 | RAM |
|------|----------|-----|
| 위치 | CPU 칩 내부 | CPU 외부 |
| 속도 | **극히 빠름** (1 클럭 사이클) | 상대적으로 느림 |
| 용량 | 매우 적음 (수십 개) | 매우 큼 (GB~TB) |
| 용도 | 연산 중간값, 주소, 상태 | 프로그램과 데이터 저장 |

### 1.2 주요 레지스터

| 레지스터 | 약어 | 역할 |
|----------|------|------|
| **누산기** | A / ACC | ALU 연산 결과 저장 (Ch 20) |
| **범용 레지스터** | B, C, D, E, H, L 등 | 계산 중간값, 임시 데이터 |
| **프로그램 카운터** | PC | **다음 실행할 명령어의 메모리 주소** |
| **명령어 레지스터** | IR | 현재 실행 중인 명령어 |
| **스택 포인터** | SP | 스택의 꼭대기 주소 (Ch 24) |
| **플래그 레지스터** | FLAGS / PSW | ALU 플래그: Z, C, V, N (Ch 21) |

### 1.3 프로그램 카운터 (Program Counter)

PC는 CPU에서 가장 중요한 레지스터 중 하나다:

```
PC = 0: 메모리[0]의 명령어 실행
         → PC = 1
PC = 1: 메모리[1]의 명령어 실행
         → PC = 2
PC = 2: 메모리[2]의 명령어 실행
         → PC = 3
...
```

평상시에는 매 클럭마다 PC가 1씩 증가하여 순차적 실행. 점프 명령어(Ch 24)가 PC를 임의의 주소로 변경하면 실행 흐름이 바뀐다.

## 2. 버스 (Bus)

### 2.1 버스란?

**버스(bus)**는 여러 장치가 **공유하는 데이터 전송 경로**다. 병렬 전선의 묶음.

> **Petzold의 비유**: 버스는 마을의 **공용 도로**와 같다. 모든 건물(장치)이 같은 도로에 연결되어 있지만, 한 번에 한 차(데이터)만 도로를 사용할 수 있다. 교통 정리(제어 신호)가 필수다.

### 2.2 세 가지 버스

```
┌─────────┐    ┌───────────────────────────────────┐    ┌─────────┐
│         │    │                                   │    │         │
│   CPU   │←──→│          데이터 버스 (양방향)        │←──→│   RAM   │
│         │    │                                   │    │         │
│         │───→│          주소 버스 (CPU→메모리)      │───→│         │
│         │    │                                   │    │         │
│         │←──→│          제어 버스                  │←──→│         │
│         │    │   (읽기/쓰기, 인터럽트 등)            │    │         │
└─────────┘    └───────────────────────────────────┘    └─────────┘
```

| 버스 | 방향 | 폭 (예) | 역할 |
|------|------|---------|------|
| **데이터 버스** | 양방향 | 8/16/32/64비트 | 데이터를 이동 |
| **주소 버스** | CPU → 메모리 | 16/32/64비트 | 메모리 위치를 지정 |
| **제어 버스** | 양방향 | 다양 | 읽기/쓰기, 클럭 등 제어 신호 |

## 3. 3-상태 버퍼 (Tri-State Buffer)

### 3.1 문제

여러 장치가 하나의 버스를 공유한다면, 두 장치가 동시에 데이터를 올리면 충돌한다 (0과 1이 합선).

### 3.2 해결: 3-상태 출력

일반 게이트의 출력: 0 또는 1 (2가지)
3-상태 버퍼의 출력: **0, 1, 또는 Hi-Z** (3가지)

| Enable | 입력 | 출력 |
|--------|------|------|
| 0 | × | **Hi-Z** (고임피던스 = 연결 끊김) |
| 1 | 0 | 0 |
| 1 | 1 | 1 |

**Hi-Z (High Impedance)** = 전기적으로 "연결이 끊어진" 상태. 버스에 아무런 영향을 미치지 않음.

### 3.3 버스 동작

```
한 번에 하나의 장치만 Enable=1 (버스에 데이터를 올림)
나머지 모든 장치는 Enable=0 (Hi-Z = 버스에서 분리)
```

> **핵심 통찰**: 3-상태 버퍼는 "이 장치가 지금 말할 차례인가?"를 결정하는 스위치다. 한 번에 한 장치만 버스에 데이터를 올릴 수 있고, 나머지는 "침묵"해야 한다. 이 규칙을 어기면 버스 충돌이 발생하여 데이터가 깨진다.

## 4. 데이터 흐름 예시

### ADD 명령어 실행

"메모리 주소 42의 값을 누산기에 더하라" (ADD [42]):

```
단계 1: PC의 값(명령어 주소)을 주소 버스에 올림
단계 2: 메모리에서 명령어를 데이터 버스로 읽음 → IR에 저장
단계 3: IR을 디코드: "ADD [42]"
단계 4: 42를 주소 버스에 올림
단계 5: 메모리[42]의 값을 데이터 버스로 읽음
단계 6: 읽은 값과 누산기의 값을 ALU에서 더함
단계 7: ALU 결과를 누산기에 저장
단계 8: PC = PC + 1 (또는 +2, 명령어 길이에 따라)
```

## 요약

- **레지스터**: CPU 내부의 초고속 저장소. 누산기, PC, IR, 플래그 레지스터 등
- **프로그램 카운터(PC)**: 다음 실행할 명령어의 주소를 가리킴. 매 클럭마다 증가
- **버스**: 데이터 버스, 주소 버스, 제어 버스 — 장치 간 공유 전송 경로
- **3-상태 버퍼**: Hi-Z 상태로 버스 충돌 방지. 한 번에 하나의 장치만 버스 사용
- CPU = **레지스터 + ALU + 버스 + 제어 로직**의 통합

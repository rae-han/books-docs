# Chapter 13: From ASCII to Unicode (ASCII에서 유니코드까지)

## 핵심 질문

> 컴퓨터는 문자를 어떻게 숫자로 표현하며, 전 세계 모든 문자를 하나의 체계로 통합하는 것은 어떻게 가능해졌는가?

---

## 1. 문자 인코딩의 근본 문제

컴퓨터는 숫자(비트)만 처리할 수 있다. 문자를 다루려면 **각 문자에 고유한 숫자를 할당**해야 한다. 이것이 **문자 인코딩(character encoding)**이다.

Chapter 3의 점자(6비트, 64가지)가 이미 이 문제를 다루었다. 이제 컴퓨터 세계에서 같은 문제가 어떻게 해결되었는지 본다.

핵심 질문: **어떤 숫자에 어떤 문자를 할당할 것인가?**

---

## 2. ASCII — 미국 표준 코드

### 2.1 ASCII란?

**ASCII (American Standard Code for Information Interchange)**는 1963년에 제정된 문자 인코딩 표준이다.

- **7비트 코드**: 2⁷ = 128가지 문자
- 0~127까지의 코드 포인트

### 2.2 ASCII 코드표

**제어 문자 (0-31, 127):**

| 코드 | 16진 | 약어 | 의미 |
|------|------|------|------|
| 0 | 00 | NUL | 널 (문자열 종료) |
| 7 | 07 | BEL | 벨 소리 |
| 8 | 08 | BS | 백스페이스 |
| 9 | 09 | HT | 수평 탭 |
| 10 | 0A | LF | 줄바꿈 (Line Feed) |
| 13 | 0D | CR | 캐리지 리턴 |
| 27 | 1B | ESC | 이스케이프 |
| 32 | 20 | SP | 공백 (스페이스) |
| 127 | 7F | DEL | 삭제 |

**출력 가능 문자 (32-126):**

| 범위 | 내용 | 예시 |
|------|------|------|
| 32 | 공백 | ' ' |
| 48-57 | 숫자 | '0'-'9' |
| 65-90 | 영문 대문자 | 'A'-'Z' |
| 97-122 | 영문 소문자 | 'a'-'z' |
| 기타 | 기호, 구두점 | '!', '@', '#', '+', '=' 등 |

### 2.3 ASCII의 영리한 설계

ASCII 코드 값의 배치에는 의도적인 설계가 숨어 있다:

**대소문자 관계:**
```
'A' = 65 = 1000001₂
'a' = 97 = 1100001₂
                ↑
          차이: 비트 5 (= 32)
```

대문자와 소문자의 차이가 정확히 **32 (= 비트 5 하나)**다. 따라서:
- 대문자 → 소문자: 비트 5를 1로 설정 (OR 0x20)
- 소문자 → 대문자: 비트 5를 0으로 설정 (AND 0xDF)

**숫자 문자:**
```
'0' = 48 = 0110000₂
'5' = 53 = 0110101₂
```

숫자 문자의 하위 4비트가 **숫자 값 자체**다! '5'의 하위 니블 = 0101₂ = 5₁₀

> **핵심 통찰**: ASCII의 코드 배치는 무작위가 아니라 **비트 조작에 최적화된 설계**다. 대소문자 변환, 숫자 문자→숫자 값 변환이 단순한 비트 연산으로 가능하다. Chapter 3의 점자도 규칙적 패턴을 가졌듯, 좋은 코드 설계는 항상 체계적이다.

## 3. 8비트 확장과 코드 페이지 문제

### 3.1 ASCII의 한계

ASCII는 7비트(128문자)로 **영어만** 제대로 표현할 수 있다. 프랑스어의 é, 독일어의 ü, 스페인어의 ñ 같은 문자가 빠져 있다.

### 3.2 확장 ASCII (8비트)

8비트 바이트의 나머지 1비트(128~255)를 활용하여 추가 문자를 배정:

- **Latin-1 (ISO 8859-1)**: 서유럽 언어 문자
- **Windows-1252**: Microsoft의 서유럽 확장
- **ISO 8859-5**: 키릴 문자 (러시아어 등)
- **EUC-KR / CP949**: 한국어

### 3.3 코드 페이지 재앙

문제: **같은 코드 값이 지역마다 다른 문자를 나타낸다!**

| 코드 값 0xC0 | Latin-1 | ISO 8859-5 | Windows-1252 |
|-------------|---------|------------|--------------|
| 해석 | À | Р (러시아 R) | À |

결과:
- 한국어 문서를 일본어 인코딩으로 열면 → 깨짐
- 러시아어 이메일을 서유럽 인코딩으로 열면 → 깨짐
- 다국어 문서는 아예 불가능

> **Petzold의 비유**: 코드 페이지 문제는 마치 모든 나라가 같은 전화번호를 다른 사람에게 할당하는 것과 같다. 미국에서 555-1234로 전화하면 피자 가게인데, 프랑스에서 같은 번호로 전화하면 병원이 된다. 혼돈이다.

## 4. Unicode — 전 세계 문자의 통합

### 4.1 Unicode의 목표

**하나의 코드 체계로 전 세계 모든 문자를 표현하자.**

- 1991년에 Unicode Consortium이 설립
- 현재 버전: 150,000개 이상의 문자를 포함
- 라틴, 키릴, 한글, 한자, 아랍, 히브리, 데바나가리, 이모지 등 모든 문자 체계

### 4.2 코드 포인트 (Code Point)

Unicode에서 각 문자에 할당된 고유 번호를 **코드 포인트**라 한다:

| 문자 | 코드 포인트 | 설명 |
|------|-------------|------|
| A | U+0041 | 영문 대문자 A |
| a | U+0061 | 영문 소문자 a |
| 가 | U+AC00 | 한글 "가" |
| 漢 | U+6F22 | 한자 "한" |
| π | U+03C0 | 그리스 문자 파이 |
| 😀 | U+1F600 | 이모지 |

`U+` 뒤의 숫자가 16진수 코드 포인트다.

### 4.3 Unicode 평면 (Planes)

Unicode는 코드 포인트를 **평면(plane)**으로 조직화한다:

| 평면 | 범위 | 이름 | 내용 |
|------|------|------|------|
| 0 | U+0000 ~ U+FFFF | BMP (Basic Multilingual Plane) | 대부분의 현대 문자 |
| 1 | U+10000 ~ U+1FFFF | SMP (Supplementary Multilingual Plane) | 역사적 문자, 이모지 |
| 2 | U+20000 ~ U+2FFFF | SIP | 추가 한자 |
| 3-16 | ... | ... | 예약/미사용 |

총 17개 평면 × 65,536 코드 포인트 = 1,114,112개 가능

## 5. UTF-8 — Unicode의 실용적 인코딩

### 5.1 문제

Unicode 코드 포인트가 최대 U+10FFFF (21비트)이므로, 매 문자를 4바이트로 저장하면 낭비가 심하다. 영어 텍스트에서 대부분의 문자는 U+007F 이하(7비트)인데 항상 4바이트를 쓸 필요가 없다.

### 5.2 UTF-8의 가변 길이 인코딩

**UTF-8**은 가변 길이 인코딩으로, 문자에 따라 1~4바이트를 사용한다:

| 코드 포인트 범위 | 바이트 수 | 바이트 패턴 | 대상 |
|------------------|-----------|-------------|------|
| U+0000 ~ U+007F | 1바이트 | `0xxxxxxx` | ASCII (영어, 숫자) |
| U+0080 ~ U+07FF | 2바이트 | `110xxxxx 10xxxxxx` | 라틴 확장, 그리스, 키릴 |
| U+0800 ~ U+FFFF | 3바이트 | `1110xxxx 10xxxxxx 10xxxxxx` | **한글**, 한자, 일본어 |
| U+10000 ~ U+10FFFF | 4바이트 | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` | 이모지, 역사적 문자 |

### 5.3 UTF-8 인코딩 예제

**예제 1**: 'A' (U+0041)
```
코드 포인트: 0041 = 1000001₂
범위: U+0000~U+007F → 1바이트
인코딩: 01000001
결과: 0x41 (ASCII와 동일!)
```

**예제 2**: '가' (U+AC00)
```
코드 포인트: AC00 = 1010 1100 0000 0000₂
범위: U+0800~U+FFFF → 3바이트
패턴: 1110xxxx 10xxxxxx 10xxxxxx
비트 분배: 1010 | 110000 | 000000
인코딩: 11101010 10110000 10000000
결과: 0xEA 0xB0 0x80
```

### 5.4 UTF-8의 장점

| 장점 | 설명 |
|------|------|
| **ASCII 호환** | 영어 텍스트는 1바이트 = ASCII와 동일 |
| **효율적** | 자주 쓰는 문자에 적은 바이트 |
| **자기 동기화** | 바이트 중간에서 시작해도 문자 경계를 찾을 수 있음 |
| **정렬 유지** | UTF-8 바이트 순서 = Unicode 코드 포인트 순서 |

> **핵심 통찰**: UTF-8은 Chapter 1의 모스 부호와 같은 **가변 길이 코드**다. 자주 쓰는 문자(영어)에 짧은 코드(1바이트)를, 덜 쓰는 문자에 긴 코드를 할당한다. 모스 부호에서 E=·, Q=——·—인 것과 같은 원리다. 그러나 UTF-8은 모스 부호와 달리 구분자가 필요 없다 — 바이트의 상위 비트 패턴으로 길이를 즉시 알 수 있기 때문이다.

## 6. 다른 Unicode 인코딩

| 인코딩 | 단위 크기 | 특징 | 사용처 |
|--------|-----------|------|--------|
| **UTF-8** | 1~4바이트 | ASCII 호환, 웹 표준 | 웹, Linux, macOS |
| **UTF-16** | 2 또는 4바이트 | BMP는 2바이트 | Windows, Java, JavaScript |
| **UTF-32** | 항상 4바이트 | 고정 길이, 단순 | 내부 처리용 |

## 단계별 구축

| 단계 | 코드 체계 | 비트 수 | 문자 수 | 한계 |
|------|-----------|---------|---------|------|
| 1 | 점자 (Ch 3) | 6비트 | 64 | 이스케이프 코드 필요 |
| 2 | ASCII | 7비트 | 128 | 영어만 |
| 3 | 확장 ASCII | 8비트 | 256 | 코드 페이지 충돌 |
| 4 | **Unicode + UTF-8** | 가변(8~32비트) | 1,114,112 | 전 세계 통합 |

## 요약

- **ASCII**: 7비트, 128문자의 미국 표준. 대소문자 차이=32, 숫자 문자의 하위 니블=숫자 값
- **코드 페이지 문제**: 8비트 확장이 지역마다 달라 호환성 재앙 발생
- **Unicode**: 전 세계 모든 문자에 고유한 코드 포인트(U+XXXX)를 할당하는 통합 표준
- **UTF-8**: Unicode의 가변 길이 인코딩. ASCII 호환, 1~4바이트, 웹의 사실상 표준
- 코드 설계의 원리(가변 길이, 이스케이프 코드, 효율성)는 모스 부호(Ch 1)에서 점자(Ch 3)를 거쳐 Unicode까지 일관되게 반복된다

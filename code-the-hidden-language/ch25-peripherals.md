# Chapter 25: Peripherals (주변 장치)

## 핵심 질문

> CPU와 메모리만으로는 외부 세계와 소통할 수 없다. 키보드, 디스플레이, 저장 장치 같은 주변 장치는 어떻게 CPU와 통신하며, 인터럽트는 왜 필요한가?

---

## 1. 입출력(I/O)의 필요성

지금까지 만든 컴퓨터: CPU(ALU + 레지스터 + 제어 장치) + RAM. 계산은 할 수 있지만:
- 사용자가 데이터를 입력할 방법이 없다
- 결과를 사용자에게 보여줄 방법이 없다
- 프로그램과 데이터를 영구 저장할 방법이 없다

**주변 장치(peripheral)**가 이 간극을 메운다.

---

## 2. 입력 장치

### 2.1 키보드

| 구성 요소 | 역할 |
|-----------|------|
| **키 매트릭스** | 행과 열의 교차점에 키 배치. 어떤 키가 눌렸는지 감지 |
| **키 디바운싱** | 기계적 스위치의 떨림(바운스)을 필터링 |
| **스캔 코드** | 각 키에 할당된 고유 번호. ASCII와는 다름 |
| **키보드 컨트롤러** | 스캔 코드를 처리하여 CPU에 전달 |

키를 누르면:
```
키 눌림 → 스캔 코드 생성 → 키보드 컨트롤러 → CPU에 전달 → 소프트웨어가 ASCII로 변환
```

### 2.2 마우스, 터치스크린

- **마우스**: 이동량(상대 좌표)과 버튼 상태를 전달
- **터치스크린**: 터치 위치(절대 좌표)를 전달

---

## 3. 출력 장치

### 3.1 디스플레이

디스플레이의 핵심 원리: **비디오 메모리(video RAM / frame buffer)**

```
비디오 RAM의 각 바이트 ──→ 화면의 각 픽셀 색상
```

| 항목 | 값 | 계산 |
|------|-----|------|
| 해상도 | 1920 × 1080 | 2,073,600 픽셀 |
| 색 깊이 | 24비트 (트루컬러) | R:8, G:8, B:8 |
| 프레임 버퍼 크기 | | 1920×1080×3 ≈ 6 MB |
| 초당 갱신 | 60 Hz | 초당 60번 화면 그리기 |

**문자 모드 vs 그래픽 모드:**

| 모드 | 단위 | 비디오 RAM 내용 |
|------|------|-----------------|
| 문자 모드 | 문자 | ASCII 코드 + 색상 속성 |
| 그래픽 모드 | 픽셀 | 각 픽셀의 RGB 값 |

## 4. I/O 방식

CPU가 주변 장치와 통신하는 방법에는 두 가지가 있다:

### 4.1 포트 매핑 I/O (Port-Mapped I/O)

- 별도의 IN/OUT 명령어를 사용
- I/O 장치에 고유한 **포트 번호** 할당
- 메모리 주소 공간과 분리됨

```
OUT 0x60, A     ; 누산기 값을 포트 0x60에 출력
IN  A, 0x61     ; 포트 0x61에서 값을 읽어 누산기에 저장
```

### 4.2 메모리 매핑 I/O (Memory-Mapped I/O)

- 특정 메모리 주소를 I/O 장치에 할당
- 일반 LOAD/STORE 명령어로 I/O 수행
- 별도의 I/O 명령어 불필요

```
STORE A, 0xB800    ; 주소 0xB800에 쓰면 → 화면에 출력
LOAD  A, 0xFF00    ; 주소 0xFF00을 읽으면 → 키보드 입력
```

| 비교 | 포트 매핑 I/O | 메모리 매핑 I/O |
|------|---------------|-----------------|
| 명령어 | 별도 IN/OUT | 기존 LOAD/STORE |
| 주소 공간 | 분리 | 통합 |
| 대표 아키텍처 | x86 | ARM, 대부분의 현대 CPU |

## 5. 저장 장치

### 5.1 자기 디스크 (Hard Disk Drive, HDD)

- 회전하는 자기 원판(platter) + 읽기/쓰기 헤드
- 트랙(track), 섹터(sector) 단위로 데이터 조직화
- 순차 읽기는 빠르지만 랜덤 액세스는 느림 (헤드 이동 필요)

### 5.2 SSD (Solid State Drive)

- 플래시 메모리 기반 — **가동 부품 없음**
- 랜덤 액세스가 HDD보다 훨씬 빠름
- 쓰기 횟수 제한이 있지만 현대 SSD는 충분히 오래 사용 가능

### 5.3 메모리 계층 구조

```
속도 빠름                              용량 큼
←──────────────────────────────────────────→
레지스터 > 캐시(SRAM) > RAM(DRAM) > SSD > HDD
 수 ns      수 ns       ~100 ns    ~μs   ~ms
 수 바이트   수 MB       수 GB      수 TB  수 TB
```

> **핵심 통찰**: 컴퓨터의 메모리는 속도와 용량의 트레이드오프로 구성된 **계층 구조**다. 빠르고 비싼 메모리(레지스터, 캐시)는 적게, 느리고 저렴한 메모리(RAM, 디스크)는 많이 둔다. 이 계층 구조 덕분에 빠른 속도와 큰 용량을 동시에 (어느 정도) 달성할 수 있다.

## 6. 인터럽트 (Interrupt)

### 6.1 폴링의 문제

CPU가 키보드 입력을 확인하려면?

**폴링(polling)**: 주기적으로 "키가 눌렸나?" 확인
```
loop:
  IN  A, keyboard_status
  JZ  loop           ; 키 안 눌렸으면 계속 확인
  IN  A, keyboard_data  ; 키 눌렸으면 읽기
```

문제: CPU가 키보드 확인에 대부분의 시간을 소비. 다른 작업을 할 수 없다.

### 6.2 인터럽트

**인터럽트(interrupt)**: 장치가 CPU에게 "처리할 일이 있다"고 **능동적으로 알림**.

```
CPU가 프로그램 실행 중...
    ↓
키보드에서 인터럽트 신호 발생!
    ↓
CPU가 현재 상태(PC, 레지스터, 플래그)를 스택에 저장
    ↓
인터럽트 서비스 루틴(ISR)으로 점프
    ↓
ISR: 키보드 데이터 읽기, 처리
    ↓
IRET (인터럽트 리턴): 스택에서 상태 복원, 원래 프로그램으로 복귀
```

> **Petzold의 비유**: 폴링은 5분마다 현관 문을 열어 택배가 왔나 확인하는 것이고, 인터럽트는 초인종이 울릴 때만 문을 여는 것이다. 초인종(인터럽트)이 훨씬 효율적이다.

---

## 요약

- **키보드**: 키 매트릭스 → 스캔 코드 → CPU → ASCII 변환
- **디스플레이**: 비디오 RAM의 값이 화면 픽셀에 대응
- **I/O 방식**: 포트 매핑(별도 IN/OUT) vs 메모리 매핑(LOAD/STORE)
- **저장 장치**: HDD(자기) → SSD(플래시). 메모리 계층 구조로 속도-용량 최적화
- **인터럽트**: 장치가 CPU에 능동적으로 알림. 폴링보다 효율적. 스택으로 상태 저장/복원

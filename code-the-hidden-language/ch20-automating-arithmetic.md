## 2. 축적기 (Accumulator)

### 2.1 아이디어

가산기의 출력을 래치에 저장하고, 그 래치의 출력을 가산기의 한 입력에 다시 연결한다:

```
                    ┌─────────────┐
                    │             │
데이터 입력 ───→ [8비트 가산기] ───→ [8비트 래치] ───→ 출력 (결과)
                    ↑               │
                    └───────────────┘
                    (래치 출력 → 가산기 입력)

                    ↑ 클럭 (래치에)
```

### 2.2 동작

1. 래치 초기값: 0
2. 클럭 펄스 1: 데이터=5 → 0+5=5 → 래치에 5 저장
3. 클럭 펄스 2: 데이터=3 → 5+3=8 → 래치에 8 저장
4. 클럭 펄스 3: 데이터=7 → 8+7=15 → 래치에 15 저장

결과: 5+3+7 = 15. 클럭이 뛸 때마다 새 값을 누적하여 더한다!

이 래치를 **축적기(accumulator)**라 부른다 — 값을 축적하기 때문이다.

> **핵심 통찰**: 축적기는 **메모리(래치)와 연산(가산기)의 결합**이다. 이 단순한 피드백 구조가 자동 계산의 출발점이며, 이후 CPU 설계의 핵심 개념(누산기 레지스터)이 된다.

## 3. 메모리에서 데이터 공급하기

축적기의 "데이터 입력"을 사람이 수동으로 제공하면 자동화가 아니다. 데이터를 **메모리에 미리 저장**해두고 자동으로 하나씩 읽으면 된다:

```
[RAM]
주소 0: 5
주소 1: 3
주소 2: 7
  ↓ (자동으로 주소 증가)
[축적기] → 결과: 15
```

주소를 자동으로 증가시키는 장치가 필요하다 → **카운터**(Chapter 18)를 사용!

```
[클럭] → [카운터] → [RAM 주소] → [RAM 데이터 출력] → [축적기]
```

카운터가 클럭마다 주소를 0, 1, 2, 3... 으로 증가시키면, RAM에서 데이터가 순서대로 나와 축적기에 더해진다.

## 4. 명령어와 데이터의 분리

### 4.1 한계

위 구조는 "더하기"만 할 수 있다. 뺄셈, 논리 연산, 데이터 이동 등 다른 작업은?

### 4.2 오피코드(Opcode) — 연산 종류를 지정하는 코드

메모리에 **데이터(data)**뿐만 아니라 **명령어(instruction)**도 함께 저장한다. 명령어 = "무엇을 할지" 알려주는 코드.

```
메모리 내용:
주소 0: [LOAD 5]     ← "5를 축적기에 넣어라"
주소 1: [ADD 3]      ← "3을 더해라"
주소 2: [ADD 7]      ← "7을 더해라"
주소 3: [STORE 100]  ← "결과를 주소 100에 저장해라"
주소 4: [HALT]       ← "멈춰라"
```

각 명령어는 두 부분으로 구성:

```
┌──────────┬──────────┐
│ 오피코드  │ 오퍼랜드  │
│ (무엇을)  │ (무엇에)  │
└──────────┴──────────┘
```

- **오피코드(opcode)**: 연산의 종류 (LOAD, ADD, SUB, STORE, HALT 등)
- **오퍼랜드(operand)**: 연산의 대상 (숫자 값, 메모리 주소 등)

### 4.3 기본 명령어 집합 예시

| 오피코드 | 이진 코드 | 의미 |
|----------|-----------|------|
| LOAD | 0001 | 오퍼랜드 값을 축적기에 넣음 |
| STORE | 0010 | 축적기 값을 메모리에 저장 |
| ADD | 0011 | 오퍼랜드를 축적기에 더함 |
| SUB | 0100 | 오퍼랜드를 축적기에서 뺌 |
| HALT | 1111 | 실행 중지 |

## 5. 프로그램 내장 방식 (Stored-Program Concept)

### 5.1 혁명적 아이디어

> **명령어와 데이터를 같은 메모리에 저장한다.**

이전의 계산 기계(ENIAC 등)는 프로그램을 전선 연결이나 스위치 설정으로 구현했다 — 프로그램을 바꾸려면 물리적으로 재배선해야 했다.

프로그램 내장 방식에서는:
- 명령어도 숫자(비트 패턴)로 표현됨
- 메모리에 저장되므로 **소프트웨어만 바꾸면** 다른 작업 수행 가능
- 프로그램이 자기 자신을 수정할 수도 있음

### 5.2 폰 노이만 아키텍처 (Von Neumann Architecture)

1945년, 수학자 **John von Neumann**이 EDVAC 보고서에서 이 개념을 정리했다:

```
┌─────────────────────────────────┐
│           메모리 (RAM)            │
│  (명령어 + 데이터 모두 저장)       │
└───────────┬─────────────────────┘
            │ (주소 버스, 데이터 버스)
    ┌───────┴───────┐
    │     CPU       │
    │  ┌─────────┐  │
    │  │ 제어장치  │  │ ← 명령어를 해석하고 실행 순서 제어
    │  └─────────┘  │
    │  ┌─────────┐  │
    │  │  ALU    │  │ ← 산술/논리 연산
    │  └─────────┘  │
    │  ┌─────────┐  │
    │  │ 레지스터 │  │ ← 임시 데이터 저장
    │  └─────────┘  │
    └───────────────┘
            │
    ┌───────┴───────┐
    │   입출력 장치   │
    └───────────────┘
```

> **핵심 통찰**: 프로그램 내장 방식은 **"범용 기계(universal machine)"**를 가능하게 했다. 같은 하드웨어가 소프트웨어에 따라 계산기, 워드프로세서, 게임, 웹 브라우저가 될 수 있다. 하드웨어를 바꾸지 않고 소프트웨어만 바꾸면 된다. 이것이 컴퓨터가 모든 다른 기계와 다른 근본적인 이유다.

## 6. 자동 실행 과정

```
1. 프로그램 카운터(PC) = 0에서 시작
2. 메모리[PC]에서 명령어를 읽음 (페치)
3. 명령어를 해석 (디코드)
4. 연산을 수행 (실행)
5. PC = PC + 1 (다음 명령어로)
6. 2번으로 돌아감 (HALT까지 반복)
```

이것이 **페치-디코드-실행 사이클(fetch-decode-execute cycle)**이며, Chapter 23에서 상세히 다룬다.

## 요약

- **축적기(accumulator)**: 가산기 + 래치의 피드백 구조. 클럭마다 값을 누적
- **자동 데이터 공급**: 카운터로 메모리 주소를 증가시켜 데이터를 순서대로 읽음
- **오피코드(opcode)**: 연산의 종류를 지정하는 코드. 명령어 = 오피코드 + 오퍼랜드
- **프로그램 내장 방식**: 명령어와 데이터를 같은 메모리에 저장 — 소프트웨어만 바꾸면 다른 기능 수행
- **폰 노이만 아키텍처**: CPU(제어장치 + ALU + 레지스터) + 메모리 + I/O의 구조

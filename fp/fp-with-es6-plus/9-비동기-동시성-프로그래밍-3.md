# 9. 비동기/동시성 프로그래밍 3

## 1. async/await
async/await 는 비동기 상황을 보다 동기적인 코드로 다루기 위한 또 하나의 방법이다. Haskell의 do notation과도 비슷하다.
비동기적으로 일어나는 일들을 문장으로 다루려고 할 때 동기적인 문장으로 다룰 수 있게 해준다.
async 를 사용한 함수는 내부에서 await를 사용할 수 있는데, await 는 포로미스가 결과를 만들 때가지 실행을 멈추고 다 만들어 졌을 때 다음 흐름을 이어나가게 한다.
결국은 promise 기반이기 때문에 promise를 잘 알아야 한다.
await로 멈출 함수가 promise를 return 해야 하기 때문이다. 즉 결국 어디선가는 promise가 있어야 한다.
```typescript
const delay500 = (a, time = 500) =>
  new Promise((resolve) => setTimeout(() => resolve(a), time));

async function f1() {
  const a = await delay500(10);
  const b = await delay500(20);

  console.log(a + b); // 30
}
```
또 하나의 특징은 a, b가 동기적으로 떨어지기 때문에 저 값을 이용해 평가한 값을 반환하면 그 값을 알수 있을거라 생각할수 있다. 하지만 아니다. 어디까지나 저 블록 안에서만 그렇고 async를 선언하게 되면 promise를 리턴하는 함수가 된다. 심지어 안에 await가 없어도 마찬가지다.
```typescript
async function f1() {
  const a = await delay500(10);
  const b = await delay500(20);
  
  return a + b;
}

console.log(f1());
```
다음과 같이 확인할 수 있다.
```typescript
async function f1() {
  const a = await delay500(10);
  const b = await delay500(20);

  return a + b;
}

f1().then(console.log);
(async () => {
	console.log(await f1());
})()
```
이게 가능한 이유가 f1 같은 함수를 실행했을 때 즉시 값으로 promise가 들어오기 때문에 promise로 평가되는 값을 await 가 평가하는 것이다 라고 생각하면 좋다.
당연하게도 위 코드와 비슷한 흐름으로 resolve 메서드를 사용한 코드도 await로 꺼내올 수 있다.
```typescript
const p = Promise.resolve(10);
(async () => {
  console.log(await p);
})();
```

## 2. Array.prototype.map을 안쓰는 이유
다음과 같은 함수가 있다고 해보자.
```typescript
export const delay = (a, time = 100) =>
  new Promise((resolve) =>
    setTimeout(() => {
      resolve(a);
    }, time),
  );

function fn() {
  const list = [1, 2, 3, 4];

  const res = list.map((a) => delay(a * a));

  console.log(res);
}
fn();
```
```typescript
[
  Promise { <pending> },
  Promise { <pending> },
  Promise { <pending> },
  Promise { <pending> }
]
```
위 함수의 res에는 Promise 객체가 찍힐 것이고 결국 이 값은 실제 평가된 값이 아니기 때문에 연산(합치기) 할수 없다. delay 함수가 promise를 리턴 하므로 여기에 await를 붙여보자.
```typescript
function fn() {
  const list = [1, 2, 3, 4];

  const res = list.map(async (a) => await delay(a * a));

  console.log(res);
}
fn();
```
async 함수는 항상 Promise를 반환하므로 똑같이 Promise를 반환할 것이다. 그렇다면 다시 async await 키워드를 사용해 보자.
```typescript
async function fn() {
  const list = [1, 2, 3, 4];

  const res = await list.map(async (a) => await delay(a * a));

  console.log(res);
}
fn();
```
당연하게도 map을 한 결과는 배열이지 Promise가 아니므로 비동기 상황을 제어해 줄수 없다.
이때 우리가 만들었던 map을 사용하면 비동기에 대한 대응 로직이 있기 때문에 잘 제어할 것이다.
```typescript
async function fn() {
  const list = [1, 2, 3, 4];

  const res = await map((a) => delay(a * a), list);

  console.log(res);
}
fn();
```
```typescript
// 핵심 코드
const go1 = (a, f) => (a instanceof Promise ? a.then(f) : f(a));
```
두 코드를 비교해보면 더 이해하기 쉽다.
```typescript
async function fn1() {
  const list = [1, 2, 3, 4];
  const response = list.map(async (a) => await delay(a * a)); // 1
  console.log(response); // [Promise { <pending> }, Promise { <pending> }, Promise { <pending> }, Promise { <pending> }]
  const result = await response;
  console.log(result);
}
fn1();

async function fn2() {
  const list = [1, 2, 3, 4];
  const response = map((a) => delay(a * a), list); // 2
  console.log(response); // Promise { <pending> }
  const result = await response;
  console.log(result);
}
fn2();
```
2번 위치에서는 `Promise { <pending> }` 로 Array로 만들수 있는 Promise가 들어왔고 1번 위치에서는 저 위치에 Promise를 준 것이 아니라 Promise 들이 들어있는 배열을 주었기 때문에 await로 풀어낼수 없다.

## 3. async/await로 비동기 제어가 가능한데 왜 파이프라인이 필요한가?
async/await가 해결하고자 하는 문제와 파이프라인이나 체이닝이 해결하고자 하는 문제는 서로 다르다.
async/await 같은 경우 haskell에서도 denotation이 있고 그런 비동기적인 상황이 일어나는 것을 제어하기 위한 함수합성 기법이 따로 있다. 그런데 이것들이 따로 있는 이유는 해결하고자 하는 목적이 다르기 때문이다.
async/await 가 해결하고자 하는 부분은 원래 표현식으로 갇혀있는 Promise.then.then.then 이런 식으로 로직을 작성하는 게 어렵다보니 문장으로 다루기 위해 async/await 키워드를 통해 특정 부분을 함수 체인이 아닌 문장형으로 다루기 위한 목적을 가진 것이 async/await이다.
파이프라인은 map, filter, reduce를 연속적으로 iterable 중심으로 사고하면서 프로그래밍 해 명령형 프로그래밍을 하지 않고 더 안전하게 함수 합성을 하기 위한 것이 목적이다.
즉 파이프라인은 함수 합성수를 합성하는 것이 목적이고, async/await는 함수 합성이 아니라 풀어 놓으려고 하는 목적을 가지고 있다.
파이프라인은 어떤 코드를 리스트로 다루면서 연속적인 함수 실행과 함수 합성을 통해 더 효과적으로 함수들을 조합하고 테스트하기 쉽고 유지 보수하기 쉽게 만드는데 목적이 있다.
async/await는 비동기 상황을 동기적인 문장으로 풀어서 코딩을 하고 싶을 때 사용하는 기법이다.
헷갈려할수 있는 부분은 파이프라인으로 코딩을 했을 때도 동기적인 코드와 비동기적인 코드의 모습이 동일하기 때문에 비동기적인 상황을 동기적으로 표현하기 위해서라 생각할 수 있다. 하지만 비동기적인 코드를 동기적으로 표현하는 것의 목적을 가지고 있는 것이 아닌 안전하게 비동기 상황을 끝까지 놓치지 않고 잘 연결하거나 동기든 비동기든 상관없이 합수 합성을 안전하게 하기 위한 목적이 있다.
다음과 같은 코드가 있다 해보자.
```typescript
function f3() {
  return go(
    [1, 2, 3, 4, 5, 6, 7, 8],
    L.map((a) => delay(a * a)),
    L.filter((a) => a % 2),
    L.map((a) => delay(a + 1)),
    take(3),
    reduce((a, b) => delay(a + b)),
  );
}

go(f3(), console.log); // 38
```
이 코드가 해결하고자 했던 것은 async/await를 쓰지 않는 것이 아니라 복잡한 for문, if문 혹은 반복문을 빠져나가는 로직 같은 것을 쉽고 안전하게 코딩하기 위한 것이다. 즉 명령형적인 코드를 해결하기 위한 것이다.
이걸 명령형으로 작성하고 async/await로 풀었을 때는 아래와 같이 코드가 작성될 것이다.
```typescript
async function f4(list) {
  let temp = [];
  for (const a of list) {
    const b = await delay(a * a);
    if (await delay(b % 2)) {
      const c = await delay(b + 1);
      temp.push(c);
      if (temp.length == 3) break;
    }
  }
  let res = temp[0],
    i = 0;
  while (++i < temp.length) {
    res = await delay(res + temp[i]);
  }
  return res;
}
```
위 코드를 보면 알겠지만 async/await 는 사실상 명령형으로 어떤 비동기적인 상황을 문장으로 풀어서 해결하기 위해 사용된다. 그러니까 어떤 함수 안쪽에서 문장을 사용하려고 할 때 쓰이는 것이 await이다.
극단적인 방법으로 delay 함수를 즉시 값이 떨어지는 함수로 변경한다고 해보자.
```typescript
export const delay = (a) => a;
```
위의 f3 함수는 이와 상관 없이 값이 도출되지만 f4 같은 경우에는 promise 객체가 도출된다. 비동기가 일어나지 않더라도 이미 async 함수이기 때문에 비동기 상황이 발생하는 것이다.
즉 f4 함수는 비동기가 일어나지 않는다면 await 키워드를 없애는 씩으로 리팩토링 해야한다.
이렇단 이야기는 논리적으로 봤을 때 두 개가 해결하고자 하는 문제가 다른 것이다. f3은 async/await 키워드를 안쓰기위해 코딩한 것이 아니기 때문에 변할 것이 없는 것이고, f4는 async/await를 통해 해결하고자 하는 문제가 있기 때문에 키워드가 필요 없어지는 것이다.
다시 돌아와서 두 코드 모두 비동기로 동작 한다면 두 코드의 시간 복잡도는 같다. f3 함수의 map, filter, map, take 함수 조합은 f4 함수의 for문과 같다. 무슨 이야기냐면 f4함수는 for문 안에서 break 문을 사용하면서 담아야하는 만큼 값을 담도록 최적화하면서 순회하고 있는데 이건 f3함수도 마찬가지다. f3 함수에서 4개의 고차함수가 하는 일을 f4함수에서는 복잡하게 조건문을 사용해가며 구현 됐다.
이렇게 같은 동작을 다르게 구현했을 때 f3 함수가 전체 함수가 어떤 동작을 하는지 파악하는 시간도 빠르고 테스트도 쉬울 것이다.
또한 f3 함수의 take에 병렬 평가(C.take)를를 적용해 최대한 빠르게 실행될 수 있도록 처리할 수 있다. f4 함수에서 병렬 평가를 하기위해 바꾸는 일은 굉장이 어렵다. f4 함수에 적용 하려면 for 문 사이사이마다 로직을 뜯어 배열에 모아서 Promise.all을 통해 평가를 풀어서 가는 식으로 수정해야 한다.

## 4. async/await 와 파이프라인을 같이 사용하기도 할까?
물론이다. 하스켈에서도 아래 스타일과 같은 코드들이 있다.
```typescript
async function f5(list) {
  const r1 = await go(
    list,
    L.map((a) => delay(a * a)),
    L.filter((a) => delay(a % 2)),
    L.map((a) => delay(a + 1)),
    C.take(2),
    reduce((a, b) => delay(a + b)),
  );

  const r2 = await go(
    list,
    L.map((a) => delay(a * a)),
    L.filter((a) => delay(a % 2)),
    reduce((a, b) => delay(a + b)),
  );

  return r1 + r2;
}

go(f5([1, 2, 3, 4, 5, 6, 7, 8]), (a) => console.log(a, 'f5'));
```
위와 같이 비동기적으로 일어나는 코드를 파이프라인으로 선언을 하여 첫 결과를 꺼내놓고 그 다음 값을 꺼내는 식으로 처리할 수도 있다. 만약 여기에 또 딜레이가 필요하면 아래와 같은 코드도 나올수 있다.
```typescript
async function f5(list) {
  const r1 = await go(
    list,
    L.map((a) => delay(a * a)),
    L.filter((a) => delay(a % 2)),
    L.map((a) => delay(a + 1)),
    C.take(2),
    reduce((a, b) => delay(a + b)),
  );

  const r2 = await go(
    list,
    L.map((a) => delay(a * a)),
    L.filter((a) => delay(a % 2)),
    reduce((a, b) => delay(a + b)),
  );

  const r3 = await delay(r1 + r2);
  return r3;
}

go(f5([1, 2, 3, 4, 5, 6, 7, 8]), (a) => console.log(a, 'f5'));
```
그런데 사실상 위 코드는 delay를 해줄 필요가 없다. 그래서 r3로 또 다른 처리를 하는 로직을 만들었다하면 다음과 같이 파이프라인과 async/await를 같이 사용할 수 있다.
```typescript
async function f5(list) {
  const r1 = await go(
    list,
    L.map((a) => delay(a * a)),
    L.filter((a) => delay(a % 2)),
    L.map((a) => delay(a + 1)),
    C.take(2),
    reduce((a, b) => delay(a + b)),
  );

  const r2 = await go(
    list,
    L.map((a) => delay(a * a)),
    L.filter((a) => delay(a % 2)),
    reduce((a, b) => delay(a + b)),
  );

  const r3 = await delay(r1 + r2);

  return r3 + 10;
}

go(f5([1, 2, 3, 4, 5, 6, 7, 8]), (a) => console.log(a, 'f5'));
```

## 5. 동기 상황에서 에러 핸들링은 어떻게 해야할까?
에러 핸들링을 하는 방법은 언어마다 약간씩 다른 특징을 가지고 있고 선호하는 방식도 있고 상황에 따라 다를수 있다.
예를 들어 자바스크립트는 아래처럼 빈 배열을 순회 한다면 그대로 흘려보낸다.
```typescript
function f7(list) {
  return list
    .map((a) => a + 10)
    .filter((a) => a % 2)
    .slice(0, 2);
}

console.log(f7([]));
```
만약 아무 값을 안넣어 undefined로 오게 되면 기본 매개변수를 주는 식으로 흘려 보낼수도 있다.
```typescript
function f7(list = []) {
  return list
    .map((a) => a + 10)
    .filter((a) => a % 2)
    .slice(0, 2);
}

console.log(f7());
```
null 같이 생뚱맞은 값이 올수 있는 상황이라면 다음과 같이 처리할 수도 있다.
```typescript
function f7(list) {
  return (list || [])
    .map((a) => a + 10)
    .filter((a) => a % 2)
    .slice(0, 2);
}

console.log(f7(null));
```
일반적으로 자바스크립트에서 할 수 있는 가장 간편하고 간단한 핸들링은 비슷한 값으로 흘려 보내거나 에러를 일으키는 것이다. 보통 문제가 생겼다면 list나 이걸 가공하는 가정에서 문제가 생긴건데 위와 같이 최대한 비슷한 값을 흘려보내는 식으로 흘려 보내는 것이다. 이걸 try-catch로 처리한다면 아래와 같이 될 것이다.
```typescript
function f7(list = []) {
  try {
    return list
      .map((a) => a + 10)
      .filter((a) => a % 2)
      .slice(0, 2);
  } catch (error) {
    return [];
  }
}

console.log(f7(null));
```
위 코드에서 map의 보조함수가 잘못 됐더라도 똑같이 비슷한 값인 빈 배열을 흘려 보낼 것이다. 그리고 catch에서 어떤 에러인지도 확인 가능하다.
```typescript
function f7(list = []) {
  try {
    return list
      .map((a) => asdf)
      .filter((a) => a % 2)
      .slice(0, 2);
  } catch (error) {
	  console.log(error)
    return [];
  }
}

console.log(f7(null));
```
```typescript
function f7(list = []) {
  try {
    return list
      .map((a) => JSON.parse(a))
      .filter((a) => a % 2)

  } catch (error) {

  }
}

console.log(f7(['1', '2', '3', '4'])); // [1, 3]

```

## 6. 비동기 상황에서 에러 핸들링은 어떻게 해야할까?
비동기 상황에서는 에러 핸들링이 좀 더 복잡하다. 아래와 같이 순회 중간에 비동기가 있는 코드가 있다는 가정을 해보자.
```typescript
function f8(list = []) {
  try {
    return list
      .map(
        (a) =>
          new Promise((resolve) => {
            asdf;
          }),
      )
      .filter((a) => a % 2)
      .slice(0, 2);
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}

console.log(f8(['1', '2', '3', '4']));
```
이 케이스에서 map 해서 에러가 나도 catch 문에서 에러 핸들링이 되지 않는다. 만약 빈 배열이  출력 된다면 그건 catch에서 걸린게 아니라 map 에서 filter까지 엉뚱한 결과가 내려 가서 빈배열이 나오는 케이스 일 것이다.
다시 아래와 같이 정상적인 코드가 있다고 해도 에러 핸들링이 안되는 것은 마찬가지 일 것이다.
```typescript
function f8(list = []) {
  try {
    return list
      .map(
        (a) =>
          new Promise((resolve) => {
            resolve(JSON.parse(a));
          }),
      )
      .filter((a) => a % 2)
      .slice(0, 2);
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}

console.log(f8(['1', '2', '3', '4']));
```
위 코드에서  `(resolve) => { resolve(JSON.parse(a)); }` 이 스코프에 에러가 났다면 사용한 프로미스는 reject가 돼야한다. 이건 무슨 말이냐면 아래 프로미스는 rejected 프로미스가 돼야한다.
```typescript
const p = new Promise((resolve) => {
  resolve(JSON.parse('{'));
});
console.log(p); // Promise { <rejected> ReferenceError: asdf is not defined }
```
정상적인 코드가 와도 마찬가지이다.
```typescript
const p = new Promise((resolve) => {
  resolve(JSON.parse('{'));
});
console.log(p); // Promise { <rejected> SyntaxError: Expected property name or '}' in JSON at position 1 (line 1 column 2) }
```
하지만 둘 다 catch 에서 에러 메시지가 잡히지 않는다. 안되는 이유는 catch 로 흐름이 들어가지 않기 때문이다.
안되는 이유는 map, filter 같은 함수들이 프로미스를 잘 제어해 줄 수 있는 상태의 함수들이 아니기 때문이다.
저 부분에서 잘 제어하기 위해서 async/await를 하더라도 여전히 핸들링은 되지 않는다.
```typescript
async function f9(list = []) {
  try {
    return await list
      .map(
        async (a) =>
          await new Promise((resolve) => {
            resolve(JSON.parse(a));
          }),
      )
      .filter((a) => a % 2)
      .slice(0, 2);
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}

console.log(f9(['1', '2', '3', '4'])); // Promise { <pending> }
```
그리고 Promise 객체로 떨어지는 객체 부분에서 then-catch 문을 써서 핸들링을 해주려해도 핸들링이 되지 않는다.
```typescript
console.log(f9(['1', '2', '3', '4']).then(console.log).catch(console.log));
```

## 7. 동기/비동기 에러 핸들링에서의 파이프라인의 이점은?
비동기 에러 핸들링에서 파이프라인의 이점을 확인을 한다면 아래와 같은 코드가 될 것이다.
```typescript
async function f10(list) {
  try {
    return go(
      list,
      map((a) => new Promise((resolve) => resolve(JSON.parse(a)))),
      filter((a) => a % 2),
      take(2),
    );
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}

console.log(
  f10(['1', '2', '3', '?'])
    .then(console.log)
    .catch((error) => console.log('에러 핸들링 하겠다', error)),
);
```
이때 에러는 함수 실행부의 catch 인 에러 핸들링 하겠다 로 떨어진다. 물론 저렇게 한다고 해서 함수 구현부 문 안의 catch인 error!! 는 여전히 출력 안 될 것이다.
왜냐면 저 부분의 catch로 가려면 저 catch 의 try문 안이 다음과 같이 Promise.reject로 평가 돼야 한다.
```typescript
async function f10(list) {
  try {
    return await Promise.reject('error~~!!');
  } catch (error) {
    console.log('error!!', error); // error!! error~~!!
    return [];
  }
}

console.log(f10(['1', '2', '3', '?']));
```
위 코드처럼 try 안에서 reject로 평가해야 catch 안으로 떨어진다.
참고로 Promise.reject 이기 때문에 반드시 await를 붙여 줘야한다.
그렇단 이야기는 아까 첫 코드에서 await만 걸어주면 함수 내부 catch에서 에러 핸들링을 해줄수 있을 것이다.
```typescript
async function f10(list) {
  try {
    return await go(
      list,
      map((a) => new Promise((resolve) => resolve(JSON.parse(a)))),
      filter((a) => a % 2),
      take(2),
    );
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}
```
그리고 위 코드는 catch에서 에러핸들링을 해준 후 빈 배열 기본 값을 리턴 해주고 있으므로 실행부에서는 then에 해당하는 값이 흘러 들어올 것이다.
```typescript
console.log(f10(['1', '2', '3', '?']).then(console.log)); // []
```
그러니까 async/await를 통해 Promise 안 reject에 들어있는 에러 까지 꺼내올 수 있다. 그게 무슨 말인진 아래 코드를 보면 더 이해하기 쉽다.
```typescript
async function f10(list) {
  try {
    const res = go(
      list,
      map((a) => new Promise((resolve) => resolve(JSON.parse(a)))),
      filter((a) => a % 2),
      take(2),
    );

    return await res;
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}
```
위 코드에서 res가 Promise 여야 한단 뜻이다. 또 어떤 Promise여야 하냐면 안에 있는 일이 다 트래킹이 되고 함수합성이 연속적으로 잘 되어있고 잘 결론지어져 있는 promise로 가야지 전달이 된단 이야기이다.
여담으로 다른 재밌는건 아래 코드로 확인 가능하다.
```typescript
async function f10(list) {
  try {
    return await go(
      list,
      L.map((a) => new Promise((resolve) => resolve(JSON.parse(a)))),
      L.filter((a) => a % 2),
      take(2),
    );
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}

console.log(f10(['1', '2', '3', '?']).then(console.log));
```
위 코드에서 map, filter 에 지연성을 준다면 에러가 오기 전에 평가가 끝나기 때문에 문제 없이 값을 꺼내올 것이다. 물론 이건 개발자가 선택할 수 있다.
다시 돌아와서 결국 함수 구현부에서 reject를 통해 에러를 catch 했고 그 안에서는 빈 배열을 리턴하기 때문에 실행부에서는 더 이상 에러가 나오지 않는다.
```typescript
async function f10(list) {
  try {
    return await go(
      list,
      map((a) => new Promise((resolve) => resolve(JSON.parse(a)))),
      filter((a) => a % 2),
      take(2),
    );
  } catch (error) {
    console.log('error!!', error);
    return [];
  }
}

console.log(
  f10(['1', '2', '3', '?'])
    .then((a) => console.log('then', a)) // []
    .catch((e) => console.log('catch', e)), // 아무런 로그가 찍히지 않음.
);
```
결국 에러핸들링을 잘 하려면 await 에 Promise를 잘 걸어줘야하고 Promise를 await에 잘 걸어주려면 함수 안쪽에서 나는 에러들이 중간에 에러가 나도 전체 프로미스에게 크레이슬리 방식으로 에러를 잘 안전하게 합성해서 뒤로 연결해서 보낼수 있도록 준비가 돼 있어야 한다. 그래야 안전한 합성과 안전한 에러 핸들링이 가능하다.
하지만 그 부분들이 명령형적으로 작성할 때는 함수 체인적으로 하지 않기때문에 필요한 지점에 문제들을 걸어줘야 한다.

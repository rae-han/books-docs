## 1. map, L.map, take
기존 map 함수에 Promise 값이 들어오면 정상적인 연산이 되지 않는다.
```typescript
go(
  [Promise.resolve(1), Promise.resolve(2), Promise.resolve(3)],
  L.map((a) => a + 10),
  take(2),
  console.log,
); // [ '[object Promise]10', '[object Promise]10' ]
```
이때는 promise를 판별하기 위해 만들었던 go1 함수를 사용하면 된다.
```typescript
const go1 = (a, f) => (a instanceof Promise ? a.then(f) : f(a));
```
**(기존)**
```typescript
L.map = curry(function* (fn, iter) {
  for (const a of iter) {
    yield fn(a);
  }
});
```
**(go1 적용)**
```typescript
L.map = curry(function* (fn, iter) {
  for (const a of iter) {
    yield go1(a, fn);
  }
});
```
이렇게하면 promise 체인으로 연결 해 어떤 값이 될 예정인 값으로 만들어 줄수 있다.
이제 take 함수에서 promise로 들어있는 값을 꺼낼 수 있도록 동작하면 된다. 중간 까지 만들어 본 코드는 다음과 같다.
```typescript
export const take = curry((l, iter) => {
  let res = [];

  iter = iter[Symbol.iterator]();
  let cur;
  while (!(cur = iter.next()).done) {
    const a = cur.value;

    if (a instanceof Promise) {
      return a.then((a) => { // 1
        res.push(a);

        if (res.length >= l) {
          return res;
        }

        // 이 부분에서 다시 while문을 돌려줘야 한다.
      });
    }

    res.push(a);

    if (res.length >= l) {
      return res;
    }
  }

  return res;
});
```
값이 Promise 라면 then으로 꺼내주고 값이 부족하다면 다시 while 문으로 순회를 해야한다. 하지만 코드에서 이미 Promise 값을 꺼낼 때 return 을 해버려서(1) 함수를 빠져나갈 상황이 돼 다시 돌아가기 어렵다.
reduce 함수를 만들때와 마찬가지로 재귀할 부분을 일단 자르자.
```typescript
export const take = curry((l, iter) => {
  let res = [];
  iter = iter[Symbol.iterator]();

  // 자름
});
```
자른 부분은 함수로 만들어 즉시 실행 해준다.
```typescript
export const take = curry((l, iter) => {
  let res = [];
  iter = iter[Symbol.iterator]();

  return (function () {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;

      if (a instanceof Promise) {
        return a.then((a) => {
          res.push(a);

          if (res.length >= l) {
            return res;
          }

          // 이 부분에서 다시 while문을 돌려줘야 한다.
        });
      }

      res.push(a);

      if (res.length >= l) {
        return res;
      }
    }

    return res;
  })();
});
```
그 다음 해당 함수에 이름을 지어줘 유명 함수로 만들어 준다. 그리고 다시 시작 시킨다.
```typescript
export const take = curry((l, iter) => {
  let res = [];
  iter = iter[Symbol.iterator]();

  return (function recur() {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;

      if (a instanceof Promise) {
        return a.then((a) => { // 4
          res.push(a); // 3

          if (res.length >= l) { // 1
            return res;
          }

          return recur(); // 2
        });
      }

      res.push(a);

      if (res.length >= l) {
        return res;
      }
    }

    return res;
  })();
});
```
코드 정리를 하자면 1, 2 번 부분을 합칠 수 있다.
```typescript
return res.length >= l ? res : recur();
```
합친 코드와 3 번 부분 또한 아래와 같이 합칠 수 있다.
```typescript
return (res.push(a), res).length >= l ? res : recur();
```
이렇게 하면 코드가 한줄이 됐기 때문에 문장으로 두지 않고 다음과 같이 만들 수 있다.
```typescript
return a.then((a) => ((res.push(a), res).length >= l ? res : recur()));
```
완성된 코드는 다음과 같다.
```typescript
export const take = curry((l, iter) => {
  let res = [];
  iter = iter[Symbol.iterator]();

  return (function recur() {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;

      if (a instanceof Promise) {
        return a.then((a) => ((res.push(a), res).length >= l ? res : recur()));
      }

      res.push(a);

      if (res.length >= l) {
        return res;
      }
    }

    return res;
  })();
});
```
위와 같이 코드를 바꿨을 때 아래와 같이 Promise 가 L.map 의 매퍼 함수로 들어가면 어떻게 될까??
```typescript
go(
  [1, 2, 3],
  L.map((a) => Promise.resolve(a + 10)),
  take(2),
  console.log,
);
```
이 경우에는 L.map을 감싼 go1 함수를 보면 실제 들어오는 값은 promise가 아니기 때문에 f(a)가 실행되게 된다.
```typescript
const go1 = (a, f) => (a instanceof Promise ? a.then(f) : f(a));
```
하지만 결국 f(a) 이 promise를 반환하고 promise로 평가 되므로 take에서 promise 값을 풀어내 정상 동작 할 것이다.
또한 takeAll 또한 take로 만든 함수이기 때문에 정상 동작 할 것이다.
```typescript
go(
  [1, 2, 3],
  L.map((a) => Promise.resolve(a + 10)),
  takeAll,
  console.log,
);
```
참고로 중간에 map을 지연성이 있는 L.map으로 구현했었다.
```typescript
const map = curry(pipe(L.map, takeAll));
```
L.map과 takeAll의 조합이므로 다음과 같이 코드를 바꿔줄 수도 있다.
```typescript
go(
  [1, 2, 3],
  map((a) => a + 10),
  console.log,
);
```

## 2. filter, L.filter, nop, take
filter 함수에서 지연 평가와 비동기 동시성 즉 promise를 함께 지원하려면 크레이슬리 컴포지션을 활용해야 한다.
아래 코드는 동작하지 않는다.
```typescript
go(
  [1, 2, 3, 4, 5],
  L.map((a) => Promise.resolve(a * a)),
  L.filter((a) => a % 2),
  take(2),
  console.log,
);
```
이유는 L.map 을 통과할 때 promise가 L.filter 함수의 인자로 넘어오기 때문이다. L.filter로 들어오는 값이 프로미스를 해결한 채로 들어오도록 해야한다.
**(기존)**
```typescript
L.filter = curry(function* (fn, iter) {
  for (const a of iter) {
    if (fn(a)) yield a;
  }
});
```
**(go1 적용)**
```typescript
L.filter = curry(function* (fn, iter) {
  for (const a of iter) {
    const b = go1(a, fn);
    if (b) yield a;
  }
});
```
이렇게 하면 홀수 두 개를 가져오는 정상 동작이 아닌 [1, 4] 가 출력될 것이다. 이유는 b 값으로 promise 객체가 들어와 항상 true가 된다.
아래와 같이 수정해 보자.
```typescript
const nop = Symbol('nop');
L.filter = curry(function* (fn, iter) {
  for (const a of iter) {
    const b = go1(a, fn);

    if (b instanceof Promise) {
      yield b.then((b) => {
        return b ? a : Promise.reject(nop);
      });
    } else if (b) {
      yield a;
    }
  }
});
```
먼저 b가 promise인지 아닌지 판단하여 promise라면 값을 풀어주고 아니라면 그대로 반환해준다.
promise 케이스에서 푼 값이 참이라면 a를 리턴해준다. 여기서 a는 풀어지지 않은 promise 이겠지만 다른 곳에서 풀어 쓸 것이기 때문에 그대로 보내도 된다.
promise 케이스에서 푼 값이 거짓이라면 다음 로직을 실행 시키는 것이 아니라 아무 일도 일어나지 않게 해야한다. 이때 크레이슬리 컴포지션을 활용 해야한다. Promise.reject을 해주면 된다. reject를 한다면 이후 값은 다음 코드로 흘러가지 않게 된다. 다만 개발자가 아무 일도 하지 않길 바라는 reject인지 또는 정말 에러가 난 reject인지를 구별해야 하기 때문에 Symbol('nop')라는 구분자를 만들어, catch에서 꺼냈을 때 nop이 왔을 때 아무 일도 하지 않도록 처리해 주면 된다.
다만 저 코드만 있다면 Symbol('nop') 에러를 던질 것이다.
take 함수에서 저 에러를 그냥 통과하게 해주려면 아래와 같은 코드를 더하면 된다.
```typescript
export const take = curry((l, iter) => {
  let res = [];
  iter = iter[Symbol.iterator]();

  return (function recur() {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;

      if (a instanceof Promise) {
        return a
          .then((a) => {
            res.push(a);

            return res.length >= l ? res : recur();
          })
          .catch((e) => (e === nop ? recur() : Promise.reject(e)));
      }

      res.push(a);

      if (res.length >= l) {
        return res;
      }
    }

    return res;
  })();
});
```
catch 하여 error가 nop이면 다시 복하고 아니라면 reject 하는 식이다. 추가로 take 함수에서 catch 해 결정할 수 있다.
```typescript
export const take = curry((l, iter) => {
  let res = [];
  iter = iter[Symbol.iterator]();

  return (function recur() {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;

      if (a instanceof Promise) {
        return a
          .then((a) => {
            res.push(a);

            return res.length >= l ? res : recur();
          })
          .catch((e) => (e === nop ? recur() : Promise.reject(e)));
      }

      res.push(a);

      if (res.length >= l) {
        return res;
      }
    }

    return res;
  })();
});
```
이렇게 하면 then 이 아무리 많아도 catch로 가게 된다.

## 3. reduce 에서 nop 지원
기존 reduce 함수로 아래와 같은 로직을 실행 시키면 promise를 지원하지 않는다.
```typescript
go(
  [1, 2, 3, 4],
  L.map((a) => Promise.resolve(a * a)),
  L.filter((a) => Promise.resolve(a % 2)),
  reduce(add),
  console.log,
); // 1[object Promise][object Promise][object Promise]
```
1에 어떤 객체인지 보여주는 스트링이 붙은 모양이 된다.
이유를 알기 위해 기존 reduce 함수 코드를 보자.
```typescript
export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    iter = acc[Symbol.iterator]();
    acc = iter.next().value;
  } else {
    iter = iter[Symbol.iterator]();
  }

  return go1(acc, function recur(acc) {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;
      acc = fn(acc, a); // 1
      if (acc instanceof Promise) {
        return acc.then(recur);
      }
    }
    return acc;
  });
});
```
첫 조건 문에서 acc에 1이 들어 갔겠지만 fn(acc, a)(1) 를 실행할 때 a 가 promise 상태로 있기 때문에 이어 붙이게 된다. 한마디로 첫 값(acc)을 꺼내는 것 뿐 아니라 iter로 들어오는 값도 Promise 일 수 있다.즉 아래 부분의 코드만 집중 적으로 보면 된다.
```typescript
// ...
const a = cur.value;
acc = fn(acc, a);
// ...
```
이때 저 부분에서 해결하고자 하는 일을 함수로 따로 격리해주면 코드를 정리할 때도 쉽고 이전 상태와 분리되며 문제가 쉬워져 코드를 좀 더 쉽게 작성 가능할 것이다.
```typescript
// 추가된 코드 1.
const reduceF = (acc, a, f) =>
  a instanceof Promise ? a.then((a) => f(acc, a)) : f(acc, a);

export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    iter = acc[Symbol.iterator]();
    acc = iter.next().value;
  } else {
    iter = iter[Symbol.iterator]();
  }

  return go1(acc, function recur(acc) {
    let cur;
    while (!(cur = iter.next()).done) {
	    // const a = cur.value;
	    // acc = f(acc, a);
	    // 추가된 코드 2
      acc = reduceF(acc, cur.value, fn);
      // 참고로 recur(acc) 에서 acc는 아래 recurtive를 통해 항상 promise를 푼 값을 받기 때문에, a만 체크해 주면 된다.
      if (acc instanceof Promise) {
        return acc.then(recur);
      }
    }
    return acc;
  });
});
```
이제 roduceF 에 에러 처리를 하게 되면 아래와 같다. 아래 방법은 catch를 사용하는 것이다.
```typescript
const reduceF = (acc, a, f) =>
  a instanceof Promise
    ? a
        .then((a) => f(acc, a))
        .catch({
          /* 에러 로직 */
        })
    : f(acc, a);
```
위와 같은 방법도 있지만  then의 2번째 인자를 이용해 reject를 전달하는 방법도 잇다.
```typescript
const nop = Symbol('nop');
// ... 
const reduceF = (acc, a, f) =>
  a instanceof Promise
    ? a.then(
        (a) => f(acc, a),
        (e) => (e === nop ? acc : Promise.reject(e)),
      )
    : f(acc, a);
```
에러가 nop 인지 확인 후 만약 nop이라면 의도된 예외로 acc를 그대로 전달해 현재까지의 acc(누적 값)을 반환하고 의도한 nop이 아니라 또 다른 에러라면 Promise.reject 로 에러를 아래로 전달한다.
- **에러가 'nop'이면** : 이건 "의도된 예외"로, 그냥 무시하고 현재까지의 누적값(acc)을 그대로 반환 → 이후 함수합성(체인, reduce 등)을 계속한다.
- **에러가 'nop'이 아니면** : 의도한 게 아니므로, 더이상 함수합성을 하지 않고(더이상 then 등을 타지 않고), **`Promise.reject`**로 에러를 아래로 전달한다(즉시 종료, 에러 전파).
	- 비유적으로 "throw"와 동일한 효과.

조금 더 하면 좋은 부분이 있는데 아래와 같은 코드는 좀 더 다양한 경우의 수를 합성해 주는 함수가 될 수 있도록 문장이 아니라 표현식으로 정리해주면 좋다.
```typescript
export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    iter = acc[Symbol.iterator]();
    acc = iter.next().value;
  } else {
    iter = iter[Symbol.iterator]();
  }
  // ...
}
```
`acc = iter.next().value;` 이 부분에서 acc.next()를 꺼냈을 때, next() 하는 부분도 비동기가 일어난다면 안전하게 합성을 하기 위해 Promise를 잘 다루는 식으로 코드를 작성하면 좋다.
위 문장으로 된 코드들을 표현식 만으로 코딩하여 다형성을 잘 지원하고 안전하게 에러를 흘려보내기 위함이다.
애플리케이션 레벨의 코드에서는 이 정도 코드도 추상화 레벨이 높은 코드가 아니기 때문에 이렇게까지 처리할 필요는 없지만, 애플리케이션 레벨의 코드에서도 회사 내부에 많이 재사용 되거나 라이브러리로 만들었다면 비동기적인 상황을 처리해 줄 필요가 있다.
저 코드를 분석해보면 하는 일은 iterator를 만들고 next를 통해 결과를 꺼낸 후, reduce 이후 일들을 하는 코드이다. 이 부분을 좀 더 함수 합성적으로 작성하면 head를 뽑고 head가 뽑혀진 나머지 iterator로 reduce를 하는 코드이다.
head를 뽑는 코드만 따로 뽑아 만든다면 코드를 좀 더 간결하게 표현할 수 있다.
```typescript
export const head = (iter) => go1(take(1, iter), ([a]) => a);
```
이터레이터를 받아 take(1) 로 하나만 뽑은 후 구조분해를 해 head 값을 리턴하는 식으로 먼저 작성한다. 그리고 코드 1의 코드를 바꾸기 전에 head로 리턴된 값을 reduce로 그대로 전달하면 되므로 head로 리턴된 값이 그대로 acc가 되고 iterator를 리턴하는 식으로 작성할 수 있다.
```typescript
export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    return reduce(fn, head((iter = acc[Symbol.iterator]())), iter);
  } else {
    iter = iter[Symbol.iterator]();
  }
  // ...
}
```
위와 같은 코드라면 else 절 도 필요 없을 것이다.
```typescript
export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    return reduce(fn, head((iter = acc[Symbol.iterator]())), iter);
  }

  iter = iter[Symbol.iterator]();

  // ...
});
```

## 4. 지연 평가 + Promise의 효율성
이 전 코드들로 인해 지연 평가와 즉시 평가를 위한 map, filter, reduce 함수에서 동기, 비동기 상황을 다 지원할 수 있게 됐다. 이렇게 되면 많은 장점과 효율성이 생긴다. 아래와 같은 코드가 있다고 가정해 보자.
```typescript
go(
  [1, 2, 3, 4, 5, 6, 7, 8],
  L.map(
    (a) => new Promise((resolve) => setTimeout(() => resolve(a * a), 1000)),
  ),
  L.filter(
    (a) => new Promise((resolve) => setTimeout(() => resolve(a % 2), 1000)),
  ),
  take(2),
  console.log,
);
```
이때 1부터 8까지의 값을 전체 순회 하는게 아닌 1부터 3까지의 값을 순회(2는 filter에서는 실행 map은 실행하지 않는다)하므로 필요한 상황에만 로직을 실행 시키기 때문에 더 효율적인 코드 동작 및 작성이 가능한 상태가 됐다.

## 5. 지연된 함수열을 병렬적으로 평가하기 - C.reduce, C.take - 1
이제 병렬적인 로직에 대해 다뤄보자.
JavaScript는 브라우저와 Node.js 환경에서 비동기 I/O 방식으로 동작한다. 이는 싱글 스레드 기반에서 I/O 작업을 동기적으로 처리하는 대신, 비동기적으로 처리하여 CPU 활용도를 극대화하는 현대적인 프로그래밍 패러다임이다.
JavaScript는 하나의 메인 스레드에서 실행되지만, 이벤트 루프를 통해 비동기 작업을 효율적으로 관리한다. 파일 읽기, 네트워크 요청, 데이터베이스 쿼리와 같은 I/O 작업이 완료될 때까지 기다리지 않고, 그 시간 동안 다른 작업을 처리하여 CPU가 놀지 않도록 한다.
자바스크립트가 싱글 스레드로 돌아가고 있기 때문에 병렬적인 프로그래밍을 할 일이 없다 오해가 있을수 있다. 사실 자바스크립트는 어떤 로직을 제어하는 것을 싱글 스레드로 비동기 적으로 할 뿐이지 얼마든 병렬적인 처리는 필요할 수 있다.
예를 들어 PostgreSQL 같은 DB에 날리는 쿼리 같은 것을 병렬적으로 출발 시켜 결과를 한 번에 얻어온다거나  NoSQL DB를 쓰는 Redis 같은 경우 여러 개의 키를 가지고 여러 개의 결과를 한 번에 얻어 처리를 한다던지, 이미지 처리 같은 것들을 Node.js에서 이미지 처리할 때는 실제로 Node.js가 직접 처리하는 것이 아니라 네트워크나 기타 I/O로 작업을 보내고 대기를 하고 시점들을 다루는 일을 Node.js가 하는 것이기 때문에 어떤 처리들을 동시에 출발 시켰다가 또 하나의 로직으로 귀결시키는 로직은 개발자가 자바스크립트에서도 잘 다룰 필요가 있다.
아래와 같은 코드가 있다고 하자.
```typescript
const delay1000 = (a) =>
  new Promise((resolve) =>
    setTimeout(() => {
      console.log('hi! concurrent.');
      resolve(a);
    }, 500),
  );

go(
  [1, 2, 3, 4, 5],
  L.map((a) => delay1000(a * a)),
  L.filter((a) => a % 2),
  reduce(add),
  console.log,
);
```
이 코드는 지연성이 있는 코드이기 때문에 reduce 함수 전 까지는 함수들이 대기만 하고 있다가 reduce를 만나고 값이 필요해 질 때 순회 하면서 필요한 함수의 실행이 세로로 이뤄질 것이다.
만약 아래와 같은 코드가 있다면 이건 값을 하나씩 꺼내오는 것이 아닌 전체 값을 map에서 순회를 다 하고 filter로 넘어가는 식으로 평가가 이뤄질 것이다.
```typescript
go(
  [1, 2, 3, 4, 5],
  map((a) => delay1000(a * a)),
  filter((a) => a % 2),
  reduce(add),
  console.log,
);
```

지금은 reduce가 값을 하나씩 기다리고 있지만 만약 동시에 출발 시킨 후 값을 더할수 있다면 특정 상황에 부하는 생길수 있겠지만 더 빠르게 최종 결과를 만들수 있는 로직을 구현할 수 있을 것이다.
이때 필요한게 Concurrency하게 동작하는 reduce 함수이다. 함수형 프로그래밍에서는 상태나 외부 값에 의존하지 않기 때문에 각 라인이 개별적으로 동작해도 정상적으로 동작하는 만큼 병렬 코드를 만드는 것이 단순해 진다.
```typescript
C.reduce = (fn, acc, iter) =>
  iter ? reduce(fn, acc, [...iter]) : reduce(fn, [...acc]);
```
reduce로 해당하는 코드를 그대로 넘기는데 iter의 여부에 따라 iter 혹은 (실제로인 iter인)acc를 즉시 소비하여 넘겨준다.
추가로 커링 가능하도록 처리해 준다.
```typescript
C.reduce = curry((fn, acc, iter) =>
  iter ? reduce(fn, acc, [...iter]) : reduce(fn, [...acc]),
);
```
```typescript
console.time('')
go(
  [1, 2, 3, 4, 5],
  L.map((a) => delay1000(a * a)),
  L.filter((a) => a % 2),
  C.reduce(add),
  console.log,
  (_) => console.timeEnd(''),
); // 약 0.5~1초 뒤 전체 평가 됨.
```
이제 이 reduce 함수는 acc, iter 가 들어 왔을 때, 비동기가 일어나는 것을 제어하지 않고 대기된 함수들(L.map, L,filter)을 다 실행 시킨다. 실행 시킨 후 다시 한번 reduce에서 순회를 하면서 값을 꺼낸다.
기존의 reduce는 하나씩 순회 하며 첫 번째 값인 1을 가지고 대기열에 있는 모든 결과를 마친 후 2로 넘어가는데, C.reduce는 모든 함수(L.map, L.filter)들을 실행해 놓고 그 다음에 비동기 제어를 해서 앞에서부터(1부터) 누적 시킨다.
이렇게 흐름을 이어나가면 동기적으로 평가할 때는 값이 추가될 때마다 그 값만큼 딜레이 되는 반면, delay 함수의 작업을 현재 Node.js 환경에서 일어나는 것이 아니라 외부 환경으로 보내져 그 환경에서 병렬적으로 각각 처리해도 되는 상황이고 해당하는 작업이 부하가 아니라 좀 더 효율적인 상황이면 C.reduce를 사용해 주는 것만으로도 효율적인 프로그래밍을 할 수 있다.

## 6. 지연된 함수열을 병렬적으로 평가하기 - C.reduce, C.take - 2
위 코드에서 아래와 같이 추가적인 평가를 하면 문제 없이 동작해 값은 평가 되지만 콘솔에 에러가 나타날 것이다.
```typescript
go(
  [1, 2, 3, 4, 5],
  L.map((a) => delay1000(a * a)),
  L.filter((a) => a % 2),
  L.map((a) => delay1000(a * a)),
  C.reduce(add),
  console.log,
);
```
이유는 catch 되지 않은 부분이 있기 때문이다(`Uncaught (in promise) Symbol(nop)`).
이건 Javascript의 특성인데 콜스택에 Promise.reject로 평가되는 코드가 있으면 저렇게 출력하게 돼 있다. 이 값을 나중에 캐치를 해도 이미 출력된다.
예를 들어 브라우저 콘솔에서 아래와 같은 코드를 입력하면 Promise {<rejected>: "hi"} 와 에러가 출력될 것이다.
```typescript
const p = Promise.reject('hi');
```
그 후 원하는 시점에 catch를 해서 해결 한다고 해도 이미 출력된 에러 로그 자체는 어떻게 할 수 없다.
```typescript
p.catch(a => console.log('해결: ', a)); // 이 시점은 이미 에러 로그가 찍힌 후다.
```
개발자가 인지하고 있는 에러는 실행하고 끝나기 전에 catch를 해서 잘 정리해 줄 것이기 때문에 nop 에러가 콜스택에서 찍힐 필요는 없다고 Promise에 알려주는 정도의 코드를 넣어주면 좋을 것이다.
즉 알고 있어~ 그 부분은 다음 콜스택에서 비동기적으로 그 에러를 캐치를 할거야. 라고 알려주는 셈이다.
```typescript
C.reduce = curry((fn, acc, iter) => {
  return iter ? reduce(fn, acc, [...iter]) : reduce(fn, [...acc]);
});
```
위 코드에서 `[...iter]` 부분이 Promise reject이 평가될수 있는 상황이기 때문에 Promise reject이 일어나기 전에 catch가 달려 있으면 에러가 출력 안되지만 에러가 Promise reject가 된 후 catch를 해서 쓰게 되면 정확하게는 에러 캐치는 되지만 필요 없는 로그가 찍히게 되는 것이다.
이를테면 아래처럼 iter가 있다면 iter를 없다면 실제 이터레이터인 acc를 모두 병렬 적으로 평가해서 그대로 전달 할 것이다.
```typescript
C.reduce = curry((fn, acc, iter) => {
  const iter2 = iter ? [...iter] : [...acc];

  return iter ? reduce(fn, acc, iter2) : reduce(fn, acc, iter2);
});
```
그리고 iter2에 있는 값들을 순회하면서 임시적으로 캐치를 해 두는 것이다. 예를 들어 Promise가 있다고 가정 하고 그 값을 catch해서 아무일도 하지 않도록 하는 것이다.
```typescript
C.reduce = curry((fn, acc, iter) => {
  const iter2 = iter ? [...iter] : [...acc];
  iter2.forEach((a) => a.catch(function () {}));

  return iter ? reduce(fn, acc, iter2) : reduce(fn, iter2);
});
```
이렇게 해 두면 정상 동작만 하고 실제로 캐치 에러를 품지(뿜지?) 않는다.
여기서 중요한 점은 만약 캐치가 된 프로미스를 전달 한다면 다시 캐치를 할 수가 없다. 그래서 처리를 못한다. 그래서 캐치를 한 프로미스를 전달 하는 것이 아니라 캐치를 하지 않은 프로미스를 전달을 하되 임시적으로 앞에서 미리 캐치를 해두기만 하는 것이다.
예를 들어 p라는 값이 있는데 Promise.reject를 해 리젝트를 했다고 보고, p라는 값에서 catch를 해서 아무 일도 안일어 나게 catch를 했다고 보자.
```typescript
let p = Promise.reject('hi');
p = p.catch(a => `${a} catch`); // 이 코드로 인해 Uncaught (in promise) hi 오류가 출력되지 않음.
/// Promise {<fulfilled>: 'hi catch'}
```
코드를 평가하면서 p에 그대로 캐치까지 하는 상황이 된단 이야기는 아래와 같은 이야기이다.
```typescript
let p = Promise.reject('hi').catch(a => `${a} catch`);
```
p라는 값 자체가 catch 까지 한 프로미스가 되는 것이다. catch 메서드까지 적용된 promise가 되기 때문에 다시 캐치를 할 수 없다. 그래서 아래와 같은 코드를 다시 실행시켜도 catch를 한 것이 출력 되는 것이 아니라 메서드에 있던 promise를 그대로 리턴할 뿐이고 원래의 catch가 실행될 뿐이다.
```typescript
p.catch(a => `bye catch`);
// Promise {<fulfilled>: 'hi catch'} (p에 원래 붙어 있던 catch가 실행)
```
그래서 catch 만 걸어두고 같은 콜스택에서 캐치까지만 해 둬 에러가 뿜어지지 않게만 해두는 것이다. 그렇게 되면 p는 reject만 돼 있을 뿐 실제로 catch를 하지 않은 값이기 때문에 원하는 시점에 캐치를 할 수 있는 기법이다.
```typescript
let p = Promise.reject('hi');
p.catch(a => `${a} catch`); // p에 다시 할당하지 않음.
// Promise {<fulfilled>: 'hi catch'}

p.catch(a => `${a} re catch`);
Promise {<fulfilled>: 'hi re catch'}원
```
이제 코드 정리를 해보자. 우선 `function () {}` 같은 코드는 실제로 선언을 해서 많이 쓴다. 아무 일도 하지 않는 함수로 만들어두면 좋다.
```typescript
function noop() {}

C.reduce = curry((fn, acc, iter) => {
  const iter2 = iter ? [...iter] : [...acc];
  iter2.forEach((a) => a.catch(noop));

  return iter ? reduce(fn, acc, iter2) : reduce(fn, iter2);
});
```
또한 `iter2.forEach((a) => a.catch(noop));` 같은 코드도 아래와 같이 만들어주면 좋다.
```typescript
function noop() {}
const catchNoop = (arr) =>
  arr.forEach((a) => (a instanceof Promise ? a.catch(noop) : a));

C.reduce = curry((fn, acc, iter) => {
  const iter2 = iter ? [...iter] : [...acc];
  catchNoop(iter2);

  return iter ? reduce(fn, acc, iter2) : reduce(fn, iter2);
});
```
`catchNoop` 은 arr를 리턴해주도록 다음과 같이 콤마 연산자를 사용 가능하다.
```typescript
const catchNoop = (arr) => (
  arr.forEach((a) => (a instanceof Promise ? a.catch(noop) : a)), arr
);
```
그럼 코드는 아래와 같이 더 간결해 질 것이다.
```typescript
C.reduce = curry((fn, acc, iter) => {
  const iter2 = catchNoop(iter ? [...iter] : [...acc]);

  return iter ? reduce(fn, acc, iter2) : reduce(fn, iter2);
});
```
정리가 다 끝난 코드는 이후 함수 대기여링 얼마나 많든 상관 없이 병렬적으로 실행하게 된다. 원래는 Promise가 있는 구간에는 거기에 값이 들어가는 만큼 추가된 시간이 걸리겠지만 위 코드는 병렬적으로 실행할 것이다.
물론 자바스크립트 실행 환경은 싱글스레드지만 외부 네트워크 I/O를 한다던가 이미지 편집을 하는 이미지 매직이라든지 어떤 코드들에게 명령을 전달 할 때 제어만 병렬적으로 하겠다 라는 식으로 코드의 효율성을 높일 수 있다.
이 말은 즉 take 함수 같은 경우도 똑같은 방식으로 전체 대기열을 병렬적으로 실행하게 할 수 있다.
```typescript
C.take = curry((l, iter) => {
  iter = catchNoop([...iter]);
  return take(l, iter);
});
```
```typescript
C.take = curry((l, iter) => take(l, catchNoop([...iter])));
```
take와 C.take 함수의 실행을 비교해보면 좋을 것이다.

**C.reduce 사용**
```typescript
go(
  [1, 2, 3, 4, 5, 6, 7, 8, 9],
  L.map((a) => delay1000(a * a)),
  L.filter((a) => a % 2),
  L.map((a) => delay1000(a * a)),
  C.take(2),
  C.reduce(add),
  console.log,
);
```
**reduce 사용**
```typescript
go(
  [1, 2, 3, 4, 5, 6, 7, 8, 9],
  L.map((a) => delay1000(a * a)),
  L.filter((a) => a % 2),
  L.map((a) => delay1000(a * a)),
  C.take(2),
  reduce(add),
  console.log,
);
```
이렇게하면 C.reduce는 최대한 많은 자원을 써서 빠르게 결과를 꺼내는 식이 되고, reduce는 명령 자체를 덜 실행시켜 부하를 줄이는 방향으로 최적화 가능하다.
이렇게 선택적으로 선언적으로 하나씩 또는 병렬적으로 하겠다 관리를 이터러블 중심적으로 사고하면 가능해진다 또한 이런 병렬 프로그래밍을 안전하고 쉽고 선언적으로 해결 가능하다.

## 7. 즉시 병렬적으로 평가하기 - C.map, C.filter
지금까지는 reduce나 take 같이 마지막에 평가, 결론을 짓는 함수에서 그 앞에 있는 함수열(대기열)을 병렬적으로 실행할지, 동기적으로 실행할 지  다루는 식이었다. 이걸 C.map, C.filter 를 통해 특정 함수 라인만 병렬적으로 평가할 수 있다.
이때 C.takeAll을 만들어 둔다면 구현하기 더 간단할 것이다. C.takeAll을 C.take를 통해 간단히 구현 가능하다.
```typescript
C.takeAll = C.take(Infinity);
```
이 후 기존 L.map, L.filter 함수와 기존에 takeAll로 평가 하던 것을 C.takeAll 로 평가하면 병렬적으로 실행되는 C.map, C.filter 함수가 만들어 질 것이다.
```typescript
C.map = curry(pipe(L.map, C.takeAll));
C.filter = curry(pipe(L.filter, C.takeAll));
```
그 후 위 함수를 실행시켜 보면 잘 실행할 것이다.
```typescript
C.map((a) => delay1000(a * a), [1, 2, 3]).then(console.log);
```

## 8. 즉시, 지연, Promise, 병렬적 조합하기
지금까지 즉시 평가되는 함수, 지연 평가되는 함수, 그리고 두 곳에서 Promise를 잘 지원하는 코드들, 병렬적으로 실행하고 다루는 코드들을 작성했다. 이런 코드들을 조합해 원하는 평가 전략을 세우는 식으로 코딩을 할수 있다.

## 9. 코드 정리
우리가 만든 C.reduce의 모양은 아래와 같다.
```typescript
C.reduce = curry((fn, acc, iter) => {
  const iter2 = catchNoop(iter ? [...iter] : [...acc]);

  return iter ? reduce(fn, acc, iter2) : reduce(fn, iter2);
});
```
이렇게 조건을 두 번 나누는 것보다 함수를 연속적으로 실행하는 식으로 표현을 하는 것이 대부분 표현이 더 좋고 깔끔하다. 또한 이렇게 하면 return을 없앨수 도 있다.
```typescript
C.reduce = curry((fn, acc, iter) =>
  iter
    ? reduce(fn, acc, catchNoop([...iter]))
    : reduce(fn, catchNoop([...acc])),
);
```
그리고 catchNoop을 C.reduce, C.take에서만 사용 한다면 아래와 같이 정리 가능하다.
```typescript
function noop() {}
const catchNoop = (arr) => (
  arr.forEach((a) => (a instanceof Promise ? a.catch(noop) : a)), arr
);

C.reduce = curry((fn, acc, iter) =>
  iter
    ? reduce(fn, acc, catchNoop([...iter]))
    : reduce(fn, catchNoop([...acc])),
);
C.take = curry((l, iter) => take(l, catchNoop([...iter])));
```
```typescript
function noop() {}
const catchNoop = ([...arr]) => (
  arr.forEach((a) => (a instanceof Promise ? a.catch(noop) : a)), arr
);

C.reduce = curry((fn, acc, iter) =>
  iter ? reduce(fn, acc, catchNoop(iter)) : reduce(fn, catchNoop(acc)),
);
C.take = curry((l, iter) => take(l, catchNoop(iter)));
```
그리고 C.reduce를 아래와 같이 reduce 하나만 있도록 정리도 가능한데, 필요 없는 연산이 있어보여 그냥 두는게 더 낫다.
```typescript
C.reduce = curry((fn, acc, iter) =>
  reduce(fn, ...(iter ? [acc, catchNoop(iter)] : [catchNoop(acc)])),
);
```

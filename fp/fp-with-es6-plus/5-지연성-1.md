# 5. 지연성 1

## 1. lazy range
숫자를 받아 그 숫자 크기의 배열을 리턴하는 함수를 만든다면 아래와 같이 만들수 있을 것이다.
```typescript
const range = (length) => {
	let i = -1;
	const result = [];

	while (++i < length) {
		result.push(i);
	}

	return result;
};
```
또한 저 함수를 이용해 합을 구한다면 아래와 같이 작성 가능하다.
```typescript
const list = range(5);
console.log(reduce(add, list)); // 10
```
저 range 함수를 아래와 같이 generator 를 사용해 느긋한 함수로 만들수 있다.
```typescript
const L = {};

L.range = function* (length) {
	let i = -1;

	while (++i < length) {
		yield i;
	}
}
```
결과 값은 같다.
```typescript
const list = L.range(5);
console.log(reduce(add, list)); // 10
```
하지만 실제로 양쪽의 list를 확인하면 위는 배열이 아래는 이터레이터로 다를 것이다.
- `[ 0, 1, 2, 3, 4 ]`
- `Object [Generator] {}`

하지만 둘 다 같은 결과를 만든 이유는 reduce 함수가 이터러블 프로토콜을 따르기 때문에 이터레이터도 평가할 수 있기 때문이다.
또한 위의 `range` 함수는 list 단계에서 range의 코드가 완전히 평가 되었지만, `L.range` 함수는 약간 다르다. 실제 평가를 일으키는 reduce 함수 부분은 주석 처리하고 while 문에서 로그를 찍어보면 알수 있는데, 앞의 range 함수는 인덱스인 i가 출력 되지만 L.range 함수는 i가 출력되지 않는다. 즉 함수의 어떤 부분도 실행되지 않았다.
코드가 평가되는 시점은 이터레이터의 내부를 순회할 때 마다 하나씩 평가된다. 즉 next() 를 하기 전에는 어떤 코드도 동작하지 않는다. 즉 L.range 같은 경우는 배열로 바로 평가되는게 아니라 reduce 함수 안에서 해당 값이 필요해 질 때 평가가 이뤄진다.
또 첫 번째 range 함수는 Array를 다 만든 후 배열로 전달이 돼 동작하고, L.range 함수는 Array를 다 만들지 않고 하나씩 값을 꺼내기만 한다.
사실 생략된 부분이 있는데, reduce 함수 안에 들어 갔을 때 안쪽에서 이터레이터를 만든다. 즉 range 함수는 Array를 만들어 reduce 에서 이터레이터를 만들고 그 다음 순회를 한다. L.range 함수는 실행 됐을 때 이터레이터를 만들고 이터레이터가 곧 자기자신을 반환하는 이터러블이고 해당 함수를 실행하면 이미 만들어진 이터레이터를 그냥 리턴만 하고 그 다음에 순회를 하기 때문에 좀 더 효율적이다.

## 2. range와 L.range 테스트
두 코드를 테스트 하기 위해 우선 아래와 같은 테스트 함수를 작성해 보자.
```typescript
const test = (name, time, fn) => {
	console.time(name);
	while (time--) {
		fn();
	}
	console.timeEnd(name);
};
```
그리고 아래와 같이 두 함수를 반복한다.
```typescript
test('range', 10, () => reduce(add, range(1000000)));
test('L.range', 10, () => reduce(add, L.range(1000000)));
```
환경에 따라 다르겠지만 L.range 함수가 평균 30% 정도 더 빠른 것을 알수 있다.

## 3. take
take는 **이터러블(iterable)에서 특정 개수의 요소를 추출**하는 유틸리티 함수이다.
구현을 하게 되면 아래와 같다.
```typescript
const take = (l, iter) => {
	let res = [];

	for (const a of iter) {
		res.push(a);

		if (res.length >= l) {
			return res;
		}
	}

	return res;
};
```
이터러블을 받아 순회하여 값을 만드는 단순한 로직을 가지고 있다. 그리고 앞서 만들었던 지연성을 가진 L.range 함수도 사용 가능하다. 전혀 다른 함수지만 모두 이터러블 프로토콜을 따르고 있기 때문이다.
아래 코드에서 효율성을 보자.
```typescript
console.log(take(5, range(100))); // [0, 1, 2, 3, 4]
console.log(take(5, L.range(100))); // [0, 1, 2, 3, 4]
```
첫 번째 코드는 전체 100개 값을 만들고 순회하지만, 두 번째 코드는 필요한 값만 꺼내 쓰기에 효율적이다. 또한 아래의 경우 무한 수열을 만들어도 문제 없다.
또한 take 역시 마찬가지로 curry 함수를 적용 가능하다.
정리하면 generator, iterator 를 리턴하는 함수를 실행 했을 때, 해당 연산이 안에서 이뤄지지 않고 미루다가 필요할 때 값을 꺼내쓰는 기법이다.

## 4. 제너레이터/이터레이터 프로토콜로 구현하는 지연 평가
**지연 평가(Lazy Evaluation)**는 "게으른 평가"라고도 하지만, 실제로는 필요할 때까지 계산을 미루고, 정말 필요할 때만 계산하는 영리한 평가 방식이다. 즉, 값을 미리 계산해두지 않고, 실제로 그 값이 필요해질 때 코드를 실행해서 값을 만들어낸다. 이런 방식은 불필요한 계산과 메모리 사용을 줄이고, 무한한 데이터 구조도 다룰 수 있게 해준다
그런 이유로 제때 계산법(call-by-need)이라 표현되기도 한다.
이터러블 프로토콜의 등장 전에는 공식적인 구현은 어려웠지만 이터레이터, 제너레이터 그리고 이터러블 프로토콜을 기반으로 공식적으로 코드를 값으로 다룰수 있게 되었다.
**ES6 이전**에는 자바스크립트에서 지연 평가를 위한 공식적인 프로토콜(규약)이 없어 공식적인 구현은 어려웠다. 하지만 **ES6 이후**에 제너레이터와 이터레이터, 그리고 이터러블 프로토콜이 도입되면서, 언어 차원에서 지연 평가를 표준적으로 지원하게 됐다. 그래서 코드를 값처럼 다루고, 안전하게 조합할 수 있게 된 것이다.

## 5. L.map
앞서 만들었던 map 함수를 지연성을 가진 함수로 구현해보자. 제너레이터와 이터레이터 프로토콜을 기반으로 필요할 때만 값을 변환하는 방식으로 구현하면 아래와 같다.
```typescript
L.map = function* (f, iter) {
	for (const a of iter) {
		yield f(a);
	}
};
```
위 함수는 제너레이터 프로토콜을 따르고 있고 그 반환 값(value, done 프로퍼티)은 이터레이터 프로토콜 사양을 충족한다.

## 6. L.filter
마찬가지로 filter 함수를 제너레이터와 이터레이터 프로토콜을 따르는 지연성을 가지는 함수로 구현해보자.
```typescript
L.filter = function* (f, iter) {
	for (const a of iter) {
		if (f(a)) {
			yield a;
		}
	}
};
```

## 7. 즉시 평가되는 함수와 평가 순서
즉시 평가되는 함수로 예제 코드를 구현하면 아래와 같다.
```typescript
go(
	range(10),
	map((n) => n + 1),
	filter((n) => n % 2),
	take(2),
	console.log,
);
```
참고로 기존 구현 함수를 이용하는 것이 아니라 for…of 는 하는 일이 숨겨져 있기 때문에 break point 를 찍어보며 확인하기 전에 아래와 같이 for…of 가 하는 일을 명령형으로 바꿔보자.
참고로 map, filter, reduce 함수의 `for (const a of iter) {` 부분을 아래와 같이 바꿔주기만 하면 된다.
```typescript
//...
  iter = iter[Symbol.iterator]();
  let cur;
  while (!(cur = iter.next()).done) {
    const a = cur.value
    // ...
```
```typescript
export const map = curry((fn, iter) => {
	let result = [];

	iter = iter[Symbol.iterator]();
	let cur;
	while (!(cur = iter.next()).done) {
		const a = cur.value;
		result.push(fn(a));
	}

	return result;
});
```
```typescript
export const filter = curry((fn, iter) => {
	let result = [];

	iter = iter[Symbol.iterator]();
	let cur;
	while (!(cur = iter.next()).done) {
		const a = cur.value;
		if (fn(a)) {
			result.push(a);
		}
	}

	return result;
});
```
```typescript
export const reduce = curry((fn, acc, iter) => {
	if (!iter) {
		iter = acc[Symbol.iterator]();
		acc = iter.next().value;
	}

	iter = iter[Symbol.iterator]();
	let cur;
	while (!(cur = iter.next()).done) {
		const a = cur.value;
		acc = fn(acc, a);
	}
	return acc;
});
```
```typescript
export const take = curry((l, iter) => {
	let res = [];

	iter = iter[Symbol.iterator]();
	let cur;
	while (!(cur = iter.next()).done) {
		const a = cur.value;
		res.push(a);

		if (res.length >= l) {
			return res;
		}
	}

	return res;
});
```
브라우저에서 break point 를 찍어볼건데 take 는 i, 무슨 값을 넣는지, 반환 값을 확인한다.
map 함수 같은 경우 결과 값을 모으는 부분, 이터레이터가 어떻게 변하는지(cur = iter.next()).done 부분), 값을 넣는 부분에서 a 값 그리고 반환하는 부분을 확인한다.
참고로 map 함수 같은 경우 배열을 받게 될 것이고 [Symbol.iterator]()로 인해 Array Iterator 가 된다. 그리고 앞서 받은 iter 값을 모두 순회하면서 값을 반환해준다.

## 8. 지연 평가 되는 함수의 평가 순서
지연 평가되는 함수로 예제 코드를 구현하면 아래와 같다.
```typescript
go(
	L.range(10),
	L.map((n) => n + 1),
	L.filter((n) => n % 2),
	take(2),
	console.log,
);
```
위 함수를 실행하고 전 예제와 같이 break point를 찍는다면 어디에 가장 먼저 들어가게 될까?? take 함수로 들어가게 될 것이다(번외로 실행 자체는 위에서부터 아래로 될 것이다.).
take 함수의 iterator 매개변수 값에는 배열 같은 값이 아니라 이터레이터(`iter: Object [Generator] {}`)가 들어오게 된다. 자바스크립트에서 well-formed iterator는 자기자신이 이터레이터면서 Symbol.iterator 값을 가지고, 또 Symbol.iterator을 실행한 반환 값은 자기자신을 리턴하기 때문에 take 함수의 `iter[Symbol.iterator]()` 를 실행 시켜도 iter는 여전히 이터레이터이다.
그래서 배열 같은 이터러블 객체를 받아도 이터레이터를 만들 수 있고, L.range 함수의 제너레이터 함수가 만든 이터레이터도 이터레이터로 만들 수 있다.
이런식으로 다형성이 잘 유지할 수 있게 만들어져 있다.
다음 순서는 take 함수 내부에서 `iter.next())` 를 만난면 다음 코드를 실행 시키지 않고 L.filter 함수로 갈 것이다. 이유는 L.range, L.map, L.filter 함수의 평가를 미뤄뒀다가 값이 필요해졌는데, take 함수에 있는 iter 이터레이터는 L.filter 함수가 반환한 이터레이터이기 때문에 L.filter로 간 것이다. 그래서 L.filter 다음에는 L.map, L.range 순서로 break point 가 찍히고 range
즉시 평가되는 함수의 경우 모든 값을 다 넘기고 실행돼 가로로 값을 만드는 것이라면, 지연 평가 되는 함수는 그때그때 값이 필요할 때만 만들어 세로로 값을 만드는 것 처럼 보인다.

## 9. map, filter 계열 함수들의 결합 법칙
**map, filter 계열 함수**는 이터러블 프로토콜을 따를 때, 함수들의 조합 순서와 방식에 따라 결과가 달라지지 않는 결합 법칙을 가진다.
이 결합 법칙은 **이터러블 프로토콜을 따르는 map, filter**에서만 성립한다.(즉, Array의 map, filter 같은 함수는 즉시 평가이기 때문에 해당되지 않는다.)
지연 평가(Lazy Evaluation)를 사용할 때 이 결합 법칙이 더욱 자연스럽게 적용된다. 각 값이 필요할 때마다, 필요한 연산만 순차적으로(세로로) 적용되기 때문이다.

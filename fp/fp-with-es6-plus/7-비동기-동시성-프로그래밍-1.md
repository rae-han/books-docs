# 7. 비동기/동시성 프로그래밍 1

## 1. callback과 Promise
자바스크립트에서 비동기 동시성 프로그래밍을 하는 방법은 콜백 패턴과 프로미스 기반으로 한 메소드 체인을 통해 함수를 합성하는 방법과 async/await와 함께 사용하는 프로미스 패턴 두 가지가 있다.

### 1.1 callback 패턴
콜백 패턴에서 의도적인 비동기 상황을 만들어 특정 시간 이후에 전달하는 방식으로 작성한다면 아래와 같다.
```typescript
const add10 = (a, callback) => {
	setTimeout(() => callback(a + 10), 100);
};

add10(1, console.log); // 0.1초 뒤에 11 출력
```

### 1.2 promise 패턴
프로미스 패턴 같은 경우 아래와 같다.
```typescript
const add10 = (a) => {
	return new Promise((resolve) => setTimeout(() => resolve(a + 10), 100));
};

add10(1).then(console.log);
```
콜백 패턴과 비교해서 가장 상징적으로 차이나는 부분은 return이다.

### 1.3 callback 패턴과 promise 패턴의 차이
callback 패턴과 promise 패턴의 눈에 띄는 차이는 연속적인 실행을 할 때 차이가 난다.
```typescript
add10(1, (res) => {
	add10(res, (res) => {
		add10(res, console.log);
	});
});
```
```typescript
add10(1)
	.then(add10)
	.then(add10)
	.then(console.log);
```
콜백 패턴은 콜백 헬이 생기는 반면 프로미스 패턴은 좀 더 읽기 편해보인다.

## 2. 비동기를 값으로 만드는 Promise
사실 콜백과 프로미스의 모양이나 어떻게 결과를 꺼내보냐(then을 이용한)가 아니다.
가장 중요한 차이는 프로미스 패턴에선 promise라는 클래스를 통해 만들어진 인스턴스를 반환하는데, promise 인스턴스는 대기와 성공, 실패를 다루는 1급 값으로 이뤄져 있다.
대기와 그 다임 일을 끝내는 것들을 어떤 값을 코드나 컨텍스트로만 다루는 것이 아니라 어떤 값을 만든다 다 라는 점에서 콜백과 가장 큰 차이를 보인다.
콜백 패턴에서는 비동기적인 상황을 다루는 것이 코드로만 표현되고 있고, 프로미스 패턴에서는 비동기 상황에 대한 값을 만들어 리턴을 하고 있다는 점이 가장 크고 중요한 차이이다.
```typescript
const callback = add10(1, console.log);
const promise = add10(1).then(console.log);
```
즉 위 코드에서 callback은 undefined가 promise는 promise 객체가 나오는 부분이 가장 큰 차이다.
콜백에서는 반환 값이 중요하지 않고 setTimeout이 일어난다는 코드적인 상황 그리고 끝났을 때 다시 어떤 함수를 실행해 주고 있는 컨텍스트만 남아있다.
반면 프로미스 같은 경우 코드를 평가했을 때 즉시 프로미스가 리턴되는 것이 가장 큰 차이이다. 그리고 리 리턴된 값으로 내가 원하는 일을 다룰스 있다는 점이 다르다.
즉 이 결과에 따라 이후에 내가 하고 싶은 일을 추가적으로 할수 있다.
```typescript
const a = add10(1);
const b = a.then((v) => v + 5);
const c = b.then((v) => v + 5);
const d = c.then((v) => v + 5);
d.then(console.log);
```
예를 들어 위와 같이 어떤 결과를 통해 then 이후 내가 하고 싶은 일을 계속 추가적으로 해줄수 있다.
즉 비동기로 일어난 상황에 대해 값으로 다룰수 있고, 값으로 다룰수 있다는 것은 그것이 1급이라는 것이다. 즉 비동기 상황이 코드로서 다뤄지지 않고 값으로 다뤄지고 있다.

## 3. 값으로서의 Promise 활용
비동기 상황이 값이라는 것은 함수에게 전달할 수 있고 프로미스인지 아닌지도 확인할 수 있다.
만약 아래와 같은 go1 이라는 함수가 있다 해보자
```typescript
const go1 = (a, f) => f(a);
const add5 = a => a + 5;

console.log(go1(10, add5); // 15
```
값과 함수를 받아 값을 함수의 인자로 넣어 평가하는 함수이다. go1 함수가 잘 동작하려면 f 함수가 동기적으로 동작해야하고, a라는 값 역시 동기적으로 바로 값을 알수 있는 값이여야 한다.
```typescript
console.log(go1(Promise.resolve(10), add5)); // [object Promise]5
```
위 코드처럼 Promise가 들어가면 정상작동 하지 않는다. 아래도 마찮가지다.
```typescript
const delay100 = a => new Promise(resolve => setTimeout(() => resolve(a), 100));

console.log(go1(delay100(10), add5)); // [object Promise]5
```
이때 go1에서 동기, 비동기 코드가 모두 정상작동 되게 하려면 아래와 같이 구현하는 방법이 있다.
```typescript
const go1 = (a, f) => a instanceof Promise ? a.then(f) : f(a);
const add5 = a => a + 5;

const delay100 = a => new Promise(resolve => setTimeout(() => resolve(a), 100));

console.log(go1(delay100(10), add5)); // Promise { <pending> }
```
a 값이 프로미스인지 아닌지 확인한 후 분기를 한다.
그리고 아래와 같이 작성하면 값도 꺼내볼 수 있다.
```typescript
go1(delay100(10), add5).then(console.log);
```
그래도 기존의 동기 함수와 프로미스 함수가 표현 방법이 다른게 신경 쓰인다면 아래와 같이 똑같게 표현할 수 있다. 값 부분을 제외하면 완전히 같다.
```typescript
go1(go1(10, add5),console.log);
go1(go1(delay100(10), add5),console.log);
```
위와 같이 go1 함수를 통해 1급으로 다뤄질 수 있는 인자의 값이 비동기 상황인지 확인 가능하다. 그리고 즉시 결과를 만들거나 결과를 만들어 갈수 있다.
즉시 결과를 만든다면 위 코드 기준 10은 cosnole.log 함수까지 실행될 것다.
delay100을 쓴 함수는 반환 값이 `Promise { <pending> }` 이 되고 Promise를 계속 이어줄수 있어 결과의 상황을 일급 값으로 만들어 작업을 연결해 나갈수 있다는 점이 Promise의 가장 중요한 특징이다.

## 4. 합성 관점에서의 Promise와 모나드
프로미스는 비동기 상황에서 함수 합성을 안전하게 하기위한 도구이다.
비동기 값을 가지고 연속적인 함수 실행을 안전하게 함수 합성을 안전하게 하는 모나드라고 설명할 수 있다.
모나드를 알려면 일단 함수 합성에 대해서 알아야 한다.
f.g 라는 함수 합성이 있다고 하자. 이건 f(g(x)) 와 같다. 즉 x라는 인자를 전달 받았을 때 g 함수를 실행하고 그 결과를 그대로 f에게 전달하는 것이다.  이렇게 연속적으로 함수가 실행되는 함수 합성을 할 때 상황에 따라 안전하게 합성하도록 모나드라는 개념이 있다.
그런 구현체 중 비동기 상황을 안전하게 합성하는 것이 프로미스이다.
자바스크립트는 사실 동적 타입 언어이고 타입을 중심으로 사고하며 프로그래밍하는 언어는 아니기 때문에 monad나 대수구조 타입 같은 것들이 잘 묻어나지 않는 경향이 있다.
마찬가지로 자바스크립트와 비슷한 경향을 가진 Clojure, 리습, 스킴 같은 언어에서도 모나드라는 개념을 중시하거나 이야기하지 않는다.
실제로 모나드라는 값이나 객체가 없기 때문이다.
모나드는 결국 함수 합성을 안전하게 하기 위한 도구이다.
모나드는 값이 들어있는 박스라 생각하면 된다. 그 컨테이너(박스)에 값이 들어있고 이 값을 통해 함수 합성들을 안전하게 해나가는 것이다.
아래 코드를 예시로 보자.
```typescript
const g = a => a + 1;
const f = a => a * a;

console.log(f(g(1))); // 4
console.log(f(g())); // NaN
```
첫 번째 출력은 g와 f를 합성한 함수에 1을 전달해 원하는 결과를 잘 얻어냈다.
하지만 두 번째 출력처럼 빈 값을 넣었을 때 비정상적인 값이 만들어지거나 실제 로그를 찍는 효과가 잘못된 값이지만 일어났다.
즉 첫 번째는 1이라는 인자 값을 가지고 g함수를 실행하고 결과를 f에게 전달한 후 실행 후 반환되는 결과가 유의미한 값이라 로그에 전달 됐을 때 사용자 화면에 출력된다. 이렇게 연속적으로 이어져야되고 그것이 정상 동작이다.
하지만 g에게 전달되는 값이 어떤 값이 올지 알수 없는 현실 프로그래밍 세상에서는 잘못 됐을 때 로그를 출력하고 싶지 않고 외부에 영향을 주고 싶지 않은 효과라도 흘러가서 문제를 일이킨다.
이런 상황에서 어떤 값이 들어올지 알수 없는 상황에서 함수 합성을 안전하게 할수 있는 아이디어가 모나드이다.
```typescript
const g = a => a + 1;
const f = a => a * a;

const result = [1].map(g).map(f);
console.log(result); // 4
```
위에서 [1] 이 모나드이다. monad는 [1] 처럼 박스([])를 가지고 있고 박스 안에 실제 어떤 효과 또는 연산에 필요한 재로들을 가지고 있다. 그리고 함수 합성을 할 때, 이 박스가 가지고 있는 메소드를 통해 함수 합성을 한다. 위에선 map으로 g, f를 연속적으로 실행 하도록 합성 했다.
하지만 Array는 필요한 값이 아니다. Array는 개발자가 효과를 만들거나 값을 다룰 때 사용하는 도구이지 사용자 화면에 실제 노출되는 결론은 아니다. 함수를 합성 했을 뿐이다.
forEach 같은 함수를 통해 로그를 출력하거나 HTML을 집어 넣어 외부 세상에 효과를 만들어 낼 수 있다.
```typescript
[1].map(g).map(f).forEach(r => console.log(r)); // 4
```
이때 박스가 비어있다면 아무 일도 일어나지 않는다. 즉 함수 합성을 안전하게 하고 있단 뜻이다.
```typescript
[].map(g).map(f).forEach(r => console.log(r)); 
```
monad 형태의 Array로 함수를 합성하면 잘못된 값이 왔을 때 효과를 일으키지 않는 방식으로 다룰수 있다. 즉 부수효과가 일어나기 전에 함수들을 안전하게 합성해 두는 기법이다.
그럼 Promise는 어떤 함수 합성을 하는 것일까? 일단 생긴 건 비슷한데, resolve를 통해 값을 만들고, then을 통해 값을 합성한다.
```typescript
Array.of(1).map(g).map(f).forEach(r => console.log(r)); 
Promise.resolve(1).then(g).then(f).then(r => console.log(r))
```
프로미스는 비동기적으로 일어나는 상황을 안전하게 합성하기 위한 도구이다. 다만 성질은 조금 다른데 값이 있거나 없거나 하는 상황에서 안전한 합성을 하려는 것이 아니고, 비동기 상황에서의 합성을 안전하게 하는 도구이다.
```typescript
Array.of().map(g).map(f).forEach(r => console.log(r)); // 출력 없음
Promise.resolve().then(g).then(f).then(r => console.log(r)) // NaN
```
즉 어떤 시점 이후에 실행이 된다고 값을 알 수 있는 효과를 가진 모나드(`new Promise(resolve => setTimeout(() => resolve(1), 100))`)로 보면 함수 합성 하는 시점을 안전하게 만드는 성질을 가진 모나드이다.
```typescript
new Promise(resolve => setTimeout(() => resolve(1), 100)).then(g).then(f).then(r => console.log(r)); // 4
```
즉 Array가 값이 없을 수도 있고 filter 함수로 값이 좁혀질 수도 있는 상황에서 연속적으로 안전하게 실행하고 합성한다면, 프로미스는 합성 관점에서는 비동기 상황 즉 딜레이가 필요한 상황에서도 함수를 적절한 시점에 평가해 합성시키기 위한 도구이다.

## 5. Kleisli Composition 관점에서의 Promise
프로미스는 크레이슬리 컴포지션(Kleisli Composition 또는 Kleisli Arrow)을 지원하는 도구이다.
클레이슬리 함수 합성 방법은 오류가 있을 수 있는 상황에서의 함수 합성을 안전하게 하는 규칙이라 보면 간단하다.
수학적 프로그래밍을 하게 되면 안전하고 정확한 상수와 변수를 통해 합성이 이뤄지고 평가되며 결과를 만들지만, 현실 프로그래밍에서는 상태라는 것이 효과(Effect)가 있어 외부 세상에 의존을 할 수 밖에 없다. 이때 에러가 났을 때(잘못된 인자, 외부 상태에 의해 결과를 전달 불가능한 상황) 해결하기 위한 함수 합성이라 볼 수 있다.
예를 들어 f(g(x)) 처럼 함수가 합성 됐을 때 동일한 x라는 정의역이라면 f(g(x)) = f(g(x)) 가 성립 돼야한다. 하지만 g가 외부 세계의 값(Date 같은)을 바라보고 있다면 성립될 수 없기 때문에 순수한 함수형 프로그래밍을 할 수 없다고 볼 수 있다.
이런 상황에서 특정한 규칙을 만들어 합성을 안전하게 만들고 좀 더 수학적이다 라고 바라볼 수 있게 만드는 함수 합성이 Kleisli Composition 이다.
예를 들어 f(g(x)) 의 g 함수에서 아무런 에러 없이 동작하면 f(g(x)) = f(g(x)) 이지만 만약 g에서 에러가 난 경우 f(g(x)) 와 g(x) 를 한 결과를 같게 만드는 식으로 합성하는 것이다.
즉 g에서 바라보는 어떤 값에 의해 문제가 생긴다면 g 함수에서 return 한 값이 f를 합성해도 마치 합성하지 않은 것처럼 f(g(x)) = g(x) 이 되는 것이다.
코드를 통해 살펴보자.
```typescript
const users = [
  { id: 1, name: 'AA' },
  { id: 2, name: 'BB' },
  { id: 3, name: 'CC' },
];

const getUserById = (id) =>
  find((user) => user.id === id, users);
const getName = ({ name }) => name;

const f = getName;
const g = getUserById;

const fg = (id) => f(g(id));

console.log(fg(2) === fg(2)); // true
```
user.id 로 값을 찾는 g 함수와 name을 꺼내는 f 함수를 조합해 같은 인자를 전달하면 두 결과는 같을 것이다. 그런데 실세계의 프로그래밍에서는 users 의 상태가 변할 수 있다.
```typescript
const before = fg(2);

users.pop();
users.pop();

const after = fg(2);

console.log(before == after);
```
정확히 위 코드는 비교까지 가지도 못하고 after 값을 만드는 동안 에러가 날 것이다. 즉 아예 결과를 만들지 못하는 상황이다. f, g 함수는 안전하게 코드가 동작하는 함수지만 합성한 상황에서 위험한 상황이 생길 수 있다.
상황에 따라 에러가 날 수 있기 때문에 합성하면서 에러가 나지 않게 하는 것이 Kleisli Arrow라 볼 수 있다. 그렇게 함수를 합성 하려면 어떻게 해야할까?? 아래와 같이 코드를 작성해보자.
```typescript
const fg = (id) =>
  Promise.resolve(id).then(g).then(f);

fg(2).then(console.log);

// 아래 코드가 순차적으로 실행 되도록 해야 함. 브라우저에서 확인하면 좋을 듯.

users.pop();
users.pop();

fg(2).then(console.log);
```
사실 위 코드도 여전히 같은 에러가 날 것이다. 이때 g를 만들때 아래와 같이 할 수 있다.
```typescript
const getUserById = (id) =>
  find((user) => user.id === id, users) || Promise.reject('없어요!!');
```
Promise.reject 이라는 값을 return 하도록 했다. 그렇다면 에러는 `The promise rejected with the reason "없어요!!".` 로 출력된다. 즉 Promise 객체에 rejected 상태가 들어있는 값이다. 이렇게 되면 g만 단독으로 실행 시켰을 때와 fg 합성 합수를 실행 시켰을 때 출력되는 결과가 같을 것이다.
이때 fg 함수를 합성할 때 catch를 쓰면 어떻게 될까??
```typescript
const fg = (id) =>
  Promise.resolve(id)
    .then(g)
    .then(f)
    .catch((e) => e);
```
그렇다면 아래 두 결과는 같을 것이다.
```typescript
users.pop();
users.pop();

fg(2).then(console.log); // 없어요!!
g(2).then(console.log).catch(console.log); // 없어요!!
```

## 6. go, pipe, reduce 에서 비동기 제어
go, pipe, reduce 같은 함수들도 어떤 함수를 연속으로 실행하는 함수 합성에 대한 함수이다. 이 함수들에서도 promise의 비동기를 값으로 다루는 성징을 이용해 비동기 상황에 놓여도 잘 대응하도록 하던가 중간에 리젝트나 에러나 일어 났을 때 맨 뒤로 보내는 기법을 적용 가능하다.
만약 아래와 같은 함수가 있다고 해보자.
```typescript
go(
  1,
  (a) => a + 10,
  (a) => a + 100,
  (a) => a + 1000,
  console.log,
); // 11111
```
만약 특정 지점에 비동기 상황이 일어나 프로미스를 리턴하는 함수가 합성 되면 비정상적인 동작이 일어날 것이다.
```typescript
go(
  1,
  (a) => a + 10,
  (a) => Promise.resolve(a + 100),
  (a) => a + 1000,
  (a) => a + 10000,
  console.log,
); // [object Promise]100010000
```
이때 올바르게 동작하도록 하려면 `3. 값으로서의 Promise 활용` 의 go1 함수와 같은 기법을 사용하면 될 것이다. 참고로 go, pipe 함수는 reduce 함수를 이용해 구현 했기 때문에 reduce 함수만 잘 고쳐주면 나머지도 정상적으로 작동할 것이다.
```typescript
export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    iter = acc[Symbol.iterator]();
    acc = iter.next().value;
  }

  iter = iter[Symbol.iterator]();
  let cur;
  while (!(cur = iter.next()).done) {
    const a = cur.value;
    // acc = fn(acc, a); // 기존 코드
    acc = acc instanceof Promise ? acc.then((acc) => fn(acc, a)) : fn(acc, a);
  }
  return acc;
});
```
간단하게 위 코드의 수정 정도 만으로도 해결은 가능하다. 하지만 좋은 코드는 아니라고 볼수 있는데, Promise를 한번 만나면 Promise 체인에 계속 함수들을 합성해 연속적으로 비동기가 일어난다(+1000, +10000 코드).
하지만 개발자가 비동기가 아닌 코드에서는 동기적으로 즉시 하나의 콜스택에서 실행되기를 바랬다면, 그런 일을 처리하지 못하고 코드에 불필요한 로드가 생겨 성능 저하가 일어난다.
아래와 같이 수정하면 해결 된다.
```typescript
export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    iter = acc[Symbol.iterator]();
    acc = iter.next().value;
  } else {
    iter = iter[Symbol.iterator]();
  }

  return (function recur(acc) {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;
      acc = fn(acc, a); // 1
      if (acc instanceof Promise) { // 2
        return acc.then(recur); // 3
      }
    }
    return acc;
  })(acc);
});
```
우선 재귀를 할수 있도록 유명 함수인 recur를 만들어준다. 그리고 우선 함수를 적용 `acc = fn(acc, a);` 해보고 이게 promise라면 then을 해주고 recur을 실행 해준다.
위코드에서 a + 10 은 동기이기 때문에 1번인 `acc = fn(acc, a);` 까지 적용하고 다음 단계로 넘어간다. 그 다음 Promise.resolve(a + 100) 을 적용하면 비동기이기 대문에 2번 조건 문으로 들어간다. 그 후 3번에서 비동기 처리를 하고 현재 while문은 return 으로 끝내는 대신, 처리를 한 결과 값과 아직 진행되지 않은 값(go나 pipe에선 함수)을 다시 recur함수 내 while 문에서 반복하며 처리한다.
이렇게 하면 동기인 부분은 하나의 콜스택에서, 비동기는 다른 콜스택에서 처리한다.
```typescript
go(
  1,
  (a) => a + 10, // 동기로 처리
  (a) => Promise.resolve(a + 100),
  (a) => a + 1000, // 동기로 아래 코드와
  (a) => a + 10000, // 하나의 콜스택에서 처리
  console.log,
);
```
다만 이때도 아래와 같은 코드는 잘못된 결과를 만들 것이다.
```typescript
go(
  Promise.resolve(1),
  (a) => a + 10,
  (a) => Promise.resolve(a + 100),
  (a) => a + 1000,
  (a) => a + 10000,
  console.log,
);
```
이유는 첫 값은 promise 인지 확인을 안하고 바로 함수로 합성하기 때문이다. 이때 위에서 만들었던 go1 함수를 이용해 첫 acc가 promise면 풀어서 전달하고 아니라면 그냥 전달을 하는 식으로 해결 가능하다.
```typescript
const lift = (a, f) => (a instanceof Promise ? a.then(f) : f(a));
const go1 = lift;

export const reduce = curry((fn, acc, iter) => {
  if (!iter) {
    iter = acc[Symbol.iterator]();
    acc = iter.next().value;
  } else {
    iter = iter[Symbol.iterator]();
  }

  return go1(acc, function recur(acc) {
    let cur;
    while (!(cur = iter.next()).done) {
      const a = cur.value;
      acc = fn(acc, a);
      if (acc instanceof Promise) {
        return acc.then(recur);
      }
    }
    return acc;
  });
});
```
또한 아래처럼 reject 하는 코드가 있어도 그 다음 코드는 실행하지 않고 끝나게 된다.
```typescript
go(
  Promise.resolve(1),
  (a) => a + 10,
  (a) => Promise.reject('error~~!!'),
  (a) => a + 1000,
  (a) => a + 10000,
  console.log,
);
```
그래서 만약 catch 를 사용하게 되면 해당 에러를 처리할수 있다(코드로 처리 가능하단 말인가?)
```typescript
go(
  Promise.resolve(1),
  (a) => a + 10,
  (a) => Promise.reject('error~~!!'),
  (a) => a + 1000,
  (a) => a + 10000,
  console.log,
).catch(console.log); // error~~!!
```
이렇게 되면 reduce 함수와 파생된 함수의 경우 promise를 만났을 때 다형적으로 대응 해준다. 또한 프로미스를 직접 값으로 다루며 함수 합성도 안전하게 처리해 준다.
이를 통해 promise를 then.then.then 을 연속으로 사용해 콜백 지옥을 해결하려는 용도가 아닌 promise 라는 값을 가지고 내가 원하는 로직을 사용하거나 원하는 시점에 원하는 방식으로 적절한 시점에 받아 둔 함수를 실행하는 고차 함수를 만드는 식으로 응용을 다양하게 할 수 있다.

## 7. promise.then 의 중요한 규칙
프로미스에는 중요한 규칙이 있는데 then 메소드를 통해 결과를 꺼내면 반드시 프로미스가 아니라는 것이다.
```typescript
Promise.resolve(Promise.resolve(Promise.resolve(1))).then(console.log);
```
위 처럼 프로미스가 중첩돼 있어도 단 한 번의 then으로 안에 있는 결과를 꺼내 볼 수 있다.
이 점은 프로미스 체인이 연속으로 대기가 걸려 있어도 내가 원하는 곳에서 한 번에 then으로 원하는 결과를 받을수 있단 뜻이고, 자바스크립트에서 언어와 개발자가 서로 소통하는 데 있어 중요한 법칙이다.
즉 아래도 연속으로 resolve를 해도 한 번에 then으로 꺼내 볼 수 있다.
```typescript
new Promise((resolve) => resolve(new Promise((resolve) => resolve(1)))).then(
  console.log,
);
```
즉 then을 했을 때 해당 promise만 값으로 만들어 안에 있는 프로미스를 리턴하는 것이 아닌 아무리 깊게 프로미스를 대기 걸었어도(체이닝을 많이 했어도) 반드시 프로미스 안쪽에 있는 값을 한 번에 꺼낸다.

## 정리 문답
**여러 비동기 작업을 순차적으로 처리하기 위해 콜백 패턴을 깊게 중첩하여 사용할 때 흔히 발생하는 문제점은 무엇일까요?**
코드 구조가 복잡해지는 콜백 지옥

**Promise가 콜백 패턴과 비교하여 비동기 작업의 순차적 연결(chaining)을 용이하게 만드는 핵심적인 이유는 무엇인가요?**
Promise는 비동기 작업 자체를 나타내는 객체를 반환하고, `.then()` 메서드로 다음 작업을 연결할 수 있습니다.

**Promises가 비동기 상황을 '일급 값(first-class value)'으로 취급한다는 것의 의미는 무엇일까요?**
비동기 작업의 상태나 결과를 표현하는 Promise 객체를 변수에 할당하거나 함수의 인자로 전달하는 등 값처럼 다룰 수 있다는 의미입니다.

**함수 합성과 모나드(Monad) 관점에서 Promise는 어떤 도구로 설명될 수 있나요?**
비동기 상황에서도 함수들을 안전하게 연속적으로 합성(compose)할 수 있도록 돕는 도구

**Kleisli 합성 관점에서 Promise는 어떤 점에 기여하며 안전한 함수 합성을 돕는다고 볼 수 있나요?**
함수 실행 중 발생할 수 있는 오류나 실패를 `.catch()` 등으로 안전하게 처리하고 합성을 이어가거나 멈추는 것

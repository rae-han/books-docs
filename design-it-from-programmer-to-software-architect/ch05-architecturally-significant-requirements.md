# Chapter 5: Architecturally Significant Requirements (아키텍처 핵심 요구사항 알아내기)

## 핵심 질문

모든 요구사항이 아키텍처에 똑같이 중요한가, 아니면 그중 일부만이 아키텍처의 방향을 결정짓는가? 어떤 요구사항이 "아키텍처적으로 중요"하다고 판단할 수 있는가? 품질 속성 같은 추상적 단어("가용성", "확장성")를 어떻게 측정 가능한 형태로 다듬는가? 이해관계자가 요구사항을 말로 다 표현하지 못할 때, 아키텍트는 어떻게 숨은 요구사항을 이끌어내는가?

---

## 1. 아키텍처 핵심 요구사항이란

**아키텍처 핵심 요구사항(*Architecturally Significant Requirements, ASR - 아키텍처를 선택하거나 구성할 때 큰 영향을 미치는 요구사항*)** 은 시스템에 요청되는 수많은 요구사항 중에서도 **아키텍처의 방향을 결정짓는** 것들을 말한다. 어떤 요구사항이 중요한지 선별하는 일 자체가 아키텍트의 책무다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 모든 아키텍처 설계는 "누가·무엇을·왜 하는지"를 논의하는 데서 시작한다. 4장에서 다룬 이해관계자가 "누가"라면, 이번 장의 ASR은 **"무엇을"** 이다. 요구사항 자체가 곧 아키텍처를 결정하는 것이 아니라, **요구사항 중에서 아키텍처에 영향을 주는 부분**만이 아키텍처를 결정한다.
:::

ASR은 크게 네 가지로 구분된다.

| 카테고리 | 설명 |
|---|---|
| **제약(*Constraint - 협상 불가능하며 변경할 수 없는 결정사항*)** | 이미 결정되어 바꿀 수 없는 사항. 대체로 위에서 주어짐 |
| **품질 속성(*Quality Attribute - 시스템 외부에서 관측되는 "~성"*)** | 시스템이 특정 상황에서 어떻게 동작하는가에 대한 특성 (확장성·가용성·성능 등) |
| **영향력 있는 기능 요구사항(*Influential Functional Requirement - 아키텍처를 뒤흔들 수 있는 특정 기능*)** | 대부분의 기능은 아키텍처에 별 영향이 없지만 일부 기능은 아키텍처를 결정지음 |
| **기타 영향 요인** | 시간, 지식, 경험, 능력, 내부 규정, 개인의 개성 등 |

---

## 2. 제약으로 설계 선택지 줄이기

**제약**은 이미 정해져서 협상 불가능한 설계상의 의사결정을 의미한다. 아키텍트 스스로 선택해서 만드는 제약도 있고, 외부에서 주어지는 제약도 있다.

> 제약은 선택지를 줄이지만, 잘 선택된 제약은 문제를 단순하게 만들고 이로써 만족스러운 아키텍처가 나오기도 한다. 하지만 때로는 요구사항을 도저히 충족시키지 못하는 지옥을 만들기도 한다.

제약은 크게 **기술적 제약**과 **비즈니스적 제약**으로 나뉜다.

| 기술적 제약 | 비즈니스적 제약 |
|---|---|
| 개발 언어(예: JVM 기반이면 뭐든 상관없음) | 팀 구성(예: 팀 A가 컴포넌트 X를 만든다) |
| OS·플랫폼(예: 윈도우·리눅스·BeOS에서 구동돼야 함) | 일정·예산(예: 7월 전시회에 출시, 개발비 80억 원 이내) |
| 사용 가능한 컴포넌트(예: 기존 DB2를 활용해야 함) | 법률·계약(예: 라이선스상 일일 처리량 최대 5GB) |

### 2.1 제약을 표로 정리하기

제약을 파악하려면 **간단한 문장으로 설계 결정과 그 주체를 설명**한다. 각 제약에 대해 "누가 정했는지·기술적인지 비즈니스적인지·왜 그런지"를 함께 적으면 검토와 재확인에 유용하다.

### 2.2 제약을 받아들일 때 보수적이 되라

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 반드시 하지 않으면 실패하는 일과, 그냥 하면 좋은 일 사이에는 큰 차이가 있다. **제약을 확정하는 순간 협상도 되돌리기도 불가능**해지므로, 받아들이기 전에 "정말 제약인지" 다시 물어야 한다.
:::

시스템이 커질수록 우리가 스스로 만든 제약과 외부에서 주어진 제약을 구분하기가 점점 어려워진다. 시간이 지나면서 초기의 설계 선택이 미래의 아키텍트에게 제약처럼 굳어버릴 수도 있다. 새로운 제약이 등장할 때마다 **"이것이 스스로 만든 제약인지, 스스로를 위해 선택한 제약인지"** 를 조심스럽게 구분해야 한다.

## 3. 품질 속성 정의하기

품질 속성은 소프트웨어 시스템의 외부에서 볼 수 있는 특성과 그 시스템에 기대하는 동작을 설명한다. **"~성"** 으로 표현되는 특성이라 **품질 요구사항(*Quality Requirement*)** 이라고도 부른다.

『소프트웨어 아키텍처 이론과 실제』[BCK12]에서 다루는 일반적인 품질 속성은 다음과 같다.

| 설계 시점의 요소 | 실행 시점의 요소 | 추상적인 요소 |
|---|---|---|
| 변경 가능성(modifiability) | 가용성(availability) | 관리 용이성(manageability) |
| 유지보수성(maintainability) | 신뢰성(reliability) | 정비성(serviceability) |
| 재사용성(reusability) | 성능(performance) | 단순성(simplicity) |
| 테스트 가능성(testability) | 확장성(scalability) | 교육 편의성(learnability) |
| 구축 편의성(buildability) | 보안(security) | |

아키텍처에 관한 결정이 이뤄지면 품질 속성 중 하나는 반드시 촉진되거나 억제된다. 대부분의 결정은 **어떤 품질 속성을 촉진하는 동시에 다른 것을 억제한다.** 이 트레이드오프를 다루는 것이 아키텍처 설계 시간의 대부분을 차지한다.

> **품질 속성은 기능 요구사항이 아닌가요?**
>
> 전통적으로는 **기능 요구사항(*Functional Requirement*)** 과 **비기능 요구사항(*Non-functional Requirement*)** 으로 나눠 왔고, 이 분류상 품질 속성이나 제약은 비기능에 해당한다. 그러나 품질 속성은 "특정 기능이 어떻게 동작하는지"에 대한 시나리오이므로 완전히 비기능이라 부르기도 어색하다. 이 책은 **기능·제약·품질 속성** 세 부류로 나눈다 — 각기 설계에 미치는 힘의 방향이 다르기 때문이다.

### 3.1 품질 속성 시나리오 6요소

품질 속성은 단어일 뿐이다. "확장성", "가용성", "성능"만으로는 아무 의미가 없다. **품질 속성 시나리오(*Quality Attribute Scenario - 특정 상황에서 소프트웨어가 어떻게 동작하는지 설명하는 6요소 문서*)** 를 만들면 품질 속성을 명확하게 정의할 수 있다.

품질 속성 시나리오는 카네기 멜론 대학의 SEI(*Software Engineering Institute*)가 제시한 프레임워크로, 다음 6가지 요소로 구성된다.

| 요소 | 설명 | 예시 |
|---|---|---|
| **자극(stimulus)** | 시나리오를 시작하는 이벤트/신호 | 노드가 접속 불능이 되도록 트래픽이 몰림 |
| **자극원(source)** | 자극을 발생시키는 사람·시스템 | 사용자, 외부 시스템, 시스템 컴포넌트 |
| **산출물(artifact)** | 시나리오가 적용되는 시스템의 일부 | 시스템 전체, 특정 컴포넌트 |
| **주변 상황(environment)** | 시나리오 실행 시의 실행환경 | 보통 상태, 피크 트래픽, 재해 상태 |
| **반응(response)** | 자극 때문에 시스템이 하는 동작 | 실패 로그 기록, 캐시된 정보로 응답 |
| **반응 측정(response measure)** | 성공 기준 (구체적·측정 가능해야 함) | 99% 이상 응답, 3초 이내, 초당 100건 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 품질 속성 시나리오가 기능 요구사항과 다른 결정적인 지점은 **반응 측정**이다. "올바르게 반응하기"만으로는 부족하고 **"어떻게 반응하는지"** 를 측정 가능한 값으로 정의해야 한다. 측정 불가능한 시나리오는 검증할 수도, 설계 완성도를 평가할 수도 없다.
:::

### 3.2 원시 시나리오로 먼저 시작하기

**원시 시나리오(*Raw Scenario - 6요소를 모두 정교하게 다듬기 전에 만드는 간단한 설명*)** 는 품질 속성 시나리오를 정교하게 다듬기 전에 거치는 초벌 문서다.

제트 추진 연구소(JPL)의 화성 로버 사례가 좋은 예다[WFD16].

> **화성 로버 이동성 시나리오 (원시)**
> 프로세서와 플랫폼은 여러 프로젝트를 거치며 조금씩 변경될 수 있다. 어느 프로세서와 플랫폼을 선택하든 애플리케이션에 미치는 영향을 최소화하는 방향으로 시스템을 최적화한다.

이 원시 시나리오를 6요소로 다듬으면 이렇게 된다.

| 요소 | 내용 |
|---|---|
| 자극원 | 테스트 팀 |
| 자극 | 새로운 하드웨어 플랫폼으로 시나리오 실행 |
| 산출물 | 주행 제어 시스템 |
| 반응 | 테스트 결과 산출 |
| 반응 측정 | 코드 변경 없이 모든 테스트 통과 |

6가지를 모두 정의할 필요는 없다. 자극·자극원·반응·반응 측정 정도만 정의하는 경우도 많다. 실행환경이 특수한 경우에만 주변 상황을 추가한다.

### 3.3 측정 가능한 반응 만들기

반응 측정을 작성할 때는 자신의 경험에 따라 잠재적 값을 먼저 추정한 뒤, **허수아비 오류(*Straw-man - 일부를 전체인 것처럼 과장하는 오류. 여기서는 대화의 시작점으로 던지는 임시안*)** 를 사용해 이해관계자와 대화를 시작하면 좋다. "9개월이 걸리면 어떤가요? 6개월은요?" 식으로 물어보면 이해관계자가 실제로 원하는 값이 드러난다.

측정하기 좋을수록 테스트도 수월해진다. **시나리오로 테스트를 작성할 수 없다면, 그 시나리오는 명료하거나 측정 가능하지 않다.**

> 저자는 한때 전 세계 12개 군사기지를 연결해 모의 전쟁을 하는 시뮬레이션 테스트베드를 만들었다. 제한 시간 내에 모든 기지가 동일한 시뮬레이션 데이터를 수신해야 했는데, 몇 번의 시도 끝에 이 요구를 만족시키려면 **빛의 속도보다 빠르게 데이터를 전송해야 한다는 결론**이 나왔다. 이 반응 측정은 물리 법칙상 실현 불가능이었다 — 양자 네트워크가 실현되는 날에 다시 검토하는 게 낫겠다.

측정 기준이 현실과 동떨어져 있다면 시나리오 자체를 재검토해야 한다.

## 4. 영향력 있는 기능 요구사항 찾아내기

유스케이스나 사용자 스토리로 도출된 기능 요구사항은 소프트웨어의 동작을 정의하지만, 그중 대부분은 아키텍처와 무관하다. 어떤 기능 요구사항이 **아키텍처의 의사결정을 주도할 정도로 중요하다면** 이를 "영향력 있는 기능 요구사항"이라 부른다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 영향력 있는 기능 요구사항은 **아키텍처 킬러(*Architecture Killer*)** 라고도 불린다. 아키텍처가 높은 가치와 높은 우선순위를 가진 기능 중 **하나라도 소화하지 못하면 아키텍처를 해체하고 다시 만들어야 한다.**
:::

### 4.1 영향력 있는 기능을 가려내는 방법

저자가 사용하는 방법은 다음과 같다.

1. 먼저 현재의 생각을 요약한 **추상적인 수준의 아키텍처를 간단히 작성**한다.
2. 비슷한 문제끼리 묶어 **일반적인 요구사항을 분류**한다.
3. 앞서 그린 아키텍처로 각 분류를 해결할 수 있는지 살펴본다. **요구사항이 모호하고 해결 방법이 즉시 떠오르지 않는다면**, 그 요구사항은 아키텍처에 중요한 영향을 미칠 가능성이 높다.

영향력 있는 기능을 좁혀나갈 때 고려할 점.

- **하나의 아키텍처 요소로 구현할 수 있는 공통 요구사항을 묶기**: 예를 들어 데이터 불변성을 다루는 요구사항끼리, 사용자 인터랙션 요구사항끼리 묶는다
- **구현하기 어려운 요구사항을 찾기**: 이런 요구사항이 아키텍처에 큰 영향을 준다
- **가치가 높고 우선순위도 높은 기능**을 파악하기

### 4.2 계산기 예시 — 기록 기능이 바꿔놓는 것

간단한 계산기를 예로 들어보자. 두 개의 숫자를 더하는 기능은 아키텍처에 거의 영향을 미치지 않는다. 그런데 마케팅 팀이 새로운 기능을 던진다 — **"사용자가 전화기를 분실했을 때도 기존 계산 기록을 볼 수 있게 해주세요."**

기록 기능? 문제없다. 로컬 데이터베이스에 저장하면 된다. 그런데 **전화기를 잃어버려도** 접근 가능해야 한다면 이야기가 달라진다.

- 원격 데이터베이스 서버가 필요해진다
- 오프라인 대응은 어떻게 할지
- 가용성·확장성 목표는
- 호스팅 비용은
- 스키마 변경 시 애플리케이션 동기화는

단순해 보이는 기능 요청 하나가 계산기의 복잡도를 **기하급수적으로 높였다.** 기록 기능은 기존의 아키텍처와 요구사항만으로는 충족하지 못하므로 새로운 방향으로 발전해야 한다 — 이것이 영향력 있는 기능 요구사항이다.

## 5. 아키텍처에 영향을 미치는 다른 요소

ASR 외에도 아키텍처에 직·간접적으로 영향을 미치는 요소들이 있다.

- **최신 유행 기술**: 새로운 하드웨어·소프트웨어·설계 패러다임. 다만 대부분은 오래된 아이디어에 좋은 포장만 씌운 경우가 많다
- **아키텍트 자신의 역량, 기술, 지식**: 아는 것이 곧 상상력의 경계. "가진 도구가 망치뿐이라면 눈에는 못만 보인다"
- **팀의 역량, 기술, 지식**: 팀이 다룰 수 없는 기술은 아키텍처에 넣지 못한다
- **팀 구성**: 아래 콘웨이 법칙 참조
- **비즈니스 목표**

### 5.1 콘웨이 법칙

**콘웨이 법칙(*Conway's Law - 조직의 커뮤니케이션 구조가 곧 소프트웨어 아키텍처의 한계가 된다는 법칙*)** 은 1967년 멜빈 콘웨이가 제창했고, 프레더릭 브룩스가 『맨먼스 미신』[Bro95]에서 인용하며 널리 알려졌다.

> "설계 조직의 커뮤니케이션 구조는 곧 제품을 설계할 때의 한계를 정의한다."

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 팀이 셋이면 컴포넌트도 결국 세 개로 만들어진다. 사람 사이의 의사소통 경계가 아키텍처의 컴포넌트 경계로 나타난다. 콘웨이 법칙은 **양방향**으로 작용한다 — 아키텍처 위에 그려진 커뮤니케이션 경로는 팀을 구성하는 방식에도 영향을 미친다. 최고의 소프트웨어를 설계하고 싶다면 **팀을 재정비해야 한다.**
:::

## 6. 필요한 정보에 깊이 들어가기

ASR은 이곳저곳에 숨어 있다. 사용자 스토리에서, 매니저의 요청 속에서, 원하는 바는 있지만 어떻게 설명해야 할지 모르는 이해관계자들의 상상 속에서 발견된다.

**제품 백로그는 ASR의 보물창고**다. 거의 모든 기능 요구사항은 품질 속성을 암시적으로 묘사한다. 사용자 사례만으로도 응답 시간·확장성·오류 처리 방법을 뽑아낼 수 있는 경우가 많다.

ASR을 더 찾아낼 수 있는 활동들.

- **활동 3 GQM 접근법**: 목표(Goal) → 질문(Question) → 메트릭(Metric)을 구체화해 비즈니스 목표와 반응 측정치를 연결
- **활동 4 이해관계자 인터뷰**: 기술 이해관계자들과 대화할 때 특히 효과적
- **활동 5 가정 나열하기**: 암묵적으로 가정하고 있는 사항을 명시적으로 목록화
- **활동 7 미니 품질 속성 워크숍**: 우선순위가 높은 품질 속성 시나리오를 빠르게 정의
- **활동 9 허수아비 반응**: 이해관계자의 실제 반응 기준을 이끌어내는 대화 기법
- **활동 24 인셉션 덱**: 새 프로젝트의 킥오프 체크리스트로 사용

> **직접 해보기**: 지금 진행 중인 프로젝트의 제품 백로그에서 사용자 스토리 5개를 골라보자. 각각에서 "암시적으로 요구되는 품질 속성"(응답 시간, 오류 처리, 확장성 등)이 몇 가지 드러나는지 목록화해보자.

## 7. ASR 워크북 만들기

ASR이 나올 때마다 노트에 기록하고 이를 **ASR 워크북**이라 부른다. 초반에는 워크북을 빠르게 갱신하지만, 아키텍처가 통합되기 시작하면 편집은 줄고 참고 자료로서의 성격이 강해진다. 개발이 완료될 즈음이면 실행 가능한 테스트와 소스 코드가 워크북 일부를 대체하지만, 여전히 **변화의 역사책**으로 남는다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: ASR 워크북은 개발자·테스터·아키텍트 모두에게 프로젝트의 맥락과 정보를 제공한다. **ASR을 이해하는 사람이 많아질수록 아키텍트를 감독할 일이 줄어든다** — 즉 좋은 워크북은 아키텍트의 부담을 줄인다.
:::

### 7.1 ASR 워크북 목차 예시

이 목차를 요구사항 도출 체크리스트로도 활용할 수 있다.

- **목적과 범위**
  - 사용자 범위
  - 비즈니스 콘텍스트
- **이해관계자**
- **비즈니스 목표**
- **아키텍처 핵심 요구사항**
  - 기술적 제약
  - 비즈니스적 제약
  - 품질 속성에 관한 요구사항
  - 최상위 시나리오
  - 영향력 있는 기능 요구사항
  - 최우선 사용자 또는 페르소나
  - 유스케이스 또는 사용자 스토리
- **부록 A**: 용어집
- **부록 B**: 품질 속성의 위계와 분류

> **직접 해보기**: 위 목차를 그대로 복사해 현재 프로젝트의 빈 문서를 만들어보자. 각 섹션을 한 문장으로라도 채워보면 어느 영역이 비어 있는지가 드러난다.

### 7.2 능동적 듣기 — 이해관계자에게서 진짜 요구를 이끌어내기

이해관계자의 말을 그대로 받아 적어도 구현 가능한 요구사항으로 이어지지 않는다. 인간의 언어는 복잡하고 사회문화적 통념이 담겨 있어, 프로그래밍 언어와는 다르다.

> 사람 A가 B에게 목표나 문제를 설명한다. A가 설명을 끝마칠 때까지 B는 아무 말도 하면 안 된다. 설명이 끝나면 B는 이해를 높이기 위한 **질문만** 할 수 있다 — 피드백이나 비판은 안 된다. 이후 역할을 바꾼다.

능동적으로 듣는 자세를 가지려면 상대의 말과 문화적 맥락에 공감부터 해야 한다. **조용히 듣고, 판단하지 말고, 더 잘 이해하기 위한 질문만 해야 한다.** 저자는 『카네기 인간관계론』[Car09]과 토머스 와이펠의 『Culture Clash』[Zwe13]를 추천한다.

| 제약 | 주체 | 유형 | 내용 |

### 8.2 품질 속성 시나리오와 우선순위

미니 품질 속성 워크숍에서 도출된 시나리오 중 일부를 우선순위와 함께 정리한 예시.

| 품질 속성 | 시나리오 | 우선순위 |
|---|---|---|
| 가용성 | RFP 데이터베이스가 반응하지 않으면 라이언하트는 실패 로그를 기록하고 3초 내로 이전 정보로 응답한다 | 높음 |
| 가용성 | 사용자가 RFP를 검색할 때 99% 이상 응답을 받아야 한다 | 높음 |
| 성능 | 초당 2회의 검색 부하를 가정할 때 사용자가 5초 내로 검색 결과를 볼 수 있어야 한다 | 높음 |

| 확장성 | 데이터는 매년 5%가량 증가할 예정. 최소한의 공수로 대응할 수 있어야 한다 | 낮음 |

### 8.3 이해관계자와 함께 다듬기

라이언하트 팀은 이해관계자와 미팅하면서 다음 문장들을 공유했다.

- 보고받은 질문이나 문제는 (아무리 소수의 사용자의 것이라도) **1영업일 안에** 답장해야 한다
- 릴리스는 한 달에 최소 한 번은 해야 한다. 바람직하게는 **코딩을 끝내자마자** 릴리스할 수 있어야 한다
- RFP 인덱싱이 제대로 됐는지 확인해야 하며, 이 작업은 **자동화**해야 한다
- 현재 개발팀의 계약이 만료되면 이 프로젝트만 담당하는 개발팀을 **신속하게** 새로 구성해야 한다

각 문장마다 "~성"으로 표현할 수 있는 품질 속성(반응성, 배포 편의성, 자동화 가능성, 인적 확장성)과 암시적인 반응·측정 기준이 숨어 있다.

### 8.4 워크숍 결과

- 요구사항 워크숍에서 **수십 가지 새로운 기능**을 발견
- 미니 품질 속성 워크숍에서 **약 20가지 품질 속성 시나리오**를 도출하고 우선순위 정함
- 참가자들과 협력해 **최우선 시나리오 7개**를 구체화

지금까지의 초점은 "문제를 제대로 이해하는 것"이었다. 팀은 이 단계에서 **탐색하기 마인드셋**으로 전환할 준비를 마쳤다.

## 요약

- **아키텍처 핵심 요구사항(ASR)**: 요구사항 전부가 아니라, 그중 **아키텍처의 방향을 결정짓는** 요구사항만이 ASR이다
- ASR의 네 가지 카테고리: **제약** / **품질 속성** / **영향력 있는 기능 요구사항** / **기타 영향 요인**
- **제약**은 협상 불가능하며 되돌릴 수 없다 — 받아들일 때 보수적이 되고, "정말 제약인지" 다시 물을 것
- **품질 속성 시나리오**의 6요소: 자극·자극원·산출물·주변 상황·반응·**반응 측정**. 특히 반응 측정은 **구체적·측정 가능·테스트 가능**해야 한다
- 원시 시나리오(초벌) → 6요소 정제 순으로 다듬는다. 6요소를 모두 채울 필요는 없고, 자극·자극원·반응·반응 측정 정도만 채워도 실용적이다
- **영향력 있는 기능 요구사항 = 아키텍처 킬러** — 이 기능 하나를 놓치면 아키텍처 전체를 다시 만들어야 한다. 계산기의 "기록 기능"처럼 단순해 보이는 기능이 아키텍처를 재구성시킬 수 있다
- **콘웨이 법칙**은 양방향 — 조직 커뮤니케이션 구조가 아키텍처를 정의하고, 아키텍처가 팀 구성을 정의한다
- ASR은 제품 백로그, 이해관계자 인터뷰, 가정 나열, 미니 품질 속성 워크숍, GQM, 인셉션 덱 등에서 이끌어낼 수 있다
- **ASR 워크북**은 살아 있는 문서 — 초반에는 자주 갱신되지만 나중에는 참고 자료 겸 변화의 역사책이 된다
- **능동적 듣기**는 아키텍트의 핵심 커뮤니케이션 기술 — 판단하지 말고, 이해를 위한 질문만 하라

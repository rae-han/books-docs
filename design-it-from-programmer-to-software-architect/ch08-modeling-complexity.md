# Chapter 8: Modeling Complexity (의미있는 모델로 복잡도 관리하기)

## 핵심 질문

아키텍처 모델은 왜 필요한가? 그저 상자와 선이 아니라 "쓸모 있는 모델"은 어떻게 만들어야 하는가? 메타모델·개념 개별화·이름 짓기는 무엇을 의미하며 코드까지 어떻게 반영해야 하는가? 모델과 코드 사이의 간극은 어떻게 좁힐 수 있는가?

---

## 1. 아키텍처 파악하기

머릿속에 방대한 정보를 담는 데는 한계가 있다. 소프트웨어 개발자들은 이 한계를 극복하려 **협업**과 **추상화** 두 가지 방법을 발전시켜 왔다.

추상화를 통해 집중할 사항만 신경 쓰고 나머지는 관심에서 멀어지도록 할 수 있다. 객체지향 프로그래밍의 인터페이스가 그러하듯, 어떤 인터페이스를 만들고 싶은지에만 집중하고 세부 구현은 잊는다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 누구나 상자와 선을 그릴 수 있다. 하지만 진정으로 쓸모 있는 **아키텍처 모델(*Architecture Model - 커뮤니케이션을 강화하고 시스템을 파악하는 데 쓰이는 아키텍처의 한 부분에 대한 정확한 설명*)** 을 만들려면 심각하게 고민을 거듭해야 한다. 모델은 스케치가 아니다.
:::

### 1.1 좋은 아키텍처 모델의 장점

| 장점 | 설명 |
|---|---|
| **디자인 용어 사전** | 요소의 이름에 의미·의도가 담겨 팀 회의·코딩·시스템 관점 형성의 기반이 된다 |
| **관심사를 자세한 수준으로 유도** | 어떤 부분은 가리고 더 중요한 사항에 집중하게 해준다 |
| **품질 속성 예측** | 구현 전에 설계 수준에서 테스트할 수 있어 실험 비용·시간을 줄인다 |
| **아키텍트의 의도 반영** | 구조 속에 의도를 담아 개발자들이 시스템을 더 잘 개선·유지할 수 있게 한다 |

> **모델링을 건너뛰고 바로 코딩하면 안 되나요?**  
> 『기업 통합 패턴』과 『37 Things an Architect Knows』의 저자 그레거 호페(Gregor Hohpe)는 **"한 시간의 아키텍처 설계로 코딩하는 시간 한 달을 아낄 수 있다"** 고 말한다. 설계도상의 아키텍처 문제는 수만 줄의 코드를 수정하는 작업보다 고치기 쉽고 빠르다. 다만 코딩도 가급적 빨리 시작해야 한다 — 모델은 실제 소프트웨어 사용 경험을 대체할 수 없기 때문이다.

## 2. 메타모델 설계하기

**아키텍처 메타모델(*Architectural Metamodel - 모델을 만드는 개념과 이 개념이 적용되는 규칙. 설계용 문법 역할*)** 은 우리의 생각을 통제하면서도 아키텍처를 말할 때 활용하는 용어집이다.

```
      메타모델
    ┌─────────┐
    │ 개념+규칙 │
    └────┬────┘
         │ 설계 전용 용어·문법 규정
         ↓
     ┌───────┐
     │ 모델   │  일관성 있고 정교함
     └───────┘
```

메타모델을 만들려면 우선 동원할 개념을 정의하고 → 개념이 아키텍처의 요소·관계가 되면 → 그다음 사용 규칙을 만든다.

### 2.1 개념 개별화

**개념 개별화(*Concept Individuation - 인식하는 어떤 개념을 다른 개념과 구분하는 인지적 과정*)** 는 태어날 때부터 자연스럽게 하는 일이다. 아이가 문을 처음 볼 때 벽과 다르다고 인식하고, 문고리를 돌려보고 잠긴 문과 열린 문을 구분하며 자신의 모델을 갱신한다.

**호기심 순환 고리(*Curiosity Cycle - 조너선 무건이 제시한 학습 과정. 질문 → 테스트 → 개념 개별화 → 모델 구축 → 다시 질문 반복*)** 를 따르면 개념 개별화가 가능하다.

```
     질문하기
        │
        ↓
    모델 테스팅
        │
        ↓
   개념 개별화
        │
        ↓
    모델 구축
        │
        └──→ (다시 질문하기로)
```

**실제 활용 예시**.

- **질문**: 어떤 컴포넌트가 비정상 상태일 때 가장 많은 비용을 내야 하는가?
- **테스트**: 현재 모델로는 답할 수 없다. 컴포넌트는 있지만 비용을 알 수 없다
- **개념 개별화**: "비용" 개념을 도입
- **새 모델 구축**: 메타모델에 색상을 정의하고, 컴포넌트별로 비용에 따라 색상 표현. 흰색은 손실 없음, 색이 짙을수록 큰 손실
- **재테스트**: 아하! `Foo` 모델이 가장 큰 손실을 낸다

```
[기존 모델]        [개선한 모델]
   Foo               Foo (짙은 색)
   / \               / \
 Bar  Zed         Bar   Zed (색상 = 비용)
```

### 2.2 패턴은 메타모델의 씨앗

아키텍처 패턴은 특정 문제에 대해 미리 갖추어놓은 메타모델을 담고 있다. 패턴은 마치 **아키텍처의 씨앗**과 같다. 적절한 패턴만 고르면 메타모델은 자연히 얻게 되는 셈이다.

7장에서 살펴본 각 패턴은 요소·관계·사용 규칙이 이미 일관성 있게 정의되어 있으므로, 여러 상황에 맞게 활용할 수 있다.

### 2.3 메타모델을 코드로 유지하기

패턴을 통합해갈수록 메타모델도 결합되고, 결합된 메타모델 안에서 불일치가 일어날 수 있다. 예를 들어 두 메타모델이 모두 `worker`라는 요소를 정의했지만 역할과 사용 규칙은 완전히 다를 수 있다.

**불일치 조정 방법**.

- 유사 개념 병합
- 이름은 같지만 다른 개념이라면 이름 변경
- 사용 규칙 조정

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 규칙은 아키텍처에 적용할 제약 조건을 설명한다. 개념적 제약을 만들면 특정 품질 속성을 촉진할 수 있다. 예를 들어 레이어 패턴의 "동일한 레이어 또는 바로 아래 레이어만 사용" 규칙은 **유지보수성**을 촉진한다.
:::

### 2.4 이름 짓기 — 알로 벨시의 7단계

이름 짓기는 어렵지만 대단히 중요하다. 알로 벨시(Arlo Belshee)는 이름 짓기를 다음 7단계로 정리했다.

| 단계 | 이름 | 설명 |
|---|---|---|
| 1 | **모름** | 시스템·맥락 파악 부족으로 이름을 짓지 못함 |
| 2 | **넌센스** | 아무 의미 없음 (관련 아이디어 몇 개만) |
| 3 | **정직** | 요소의 역할 중 적어도 하나는 제대로 설명 |
| 4 | **정직하고 완전함** | 모든 요소의 모든 역할을 제대로 설명 |
| 5 | **올바름** | 요소의 미래 발전 방향까지 암시 |
| 6 | **의도를 담음** | 역할뿐만 아니라 왜 존재해야 하는지 목적을 설명 |
| 7 | **도메인 추상화** | 개별 요소를 초월해 새로운 추상화를 만듦. 메타모델에 새 개념 탄생 |

**웹에서 데이터를 가져와 변형하는 요소의 예시**.

| 단계 | 이름 |
|---|---|
| 모름 | The thing that does the thing (뭔가 하는 것) |
| 넌센스 | Cranberry |
| 정직 | Job Starter Process |
| 정직하고 완전함 | Data Fetcher, Checker, Transformer, Job Starter |
| 올바름 | Data Transformation Job Runner |
| 의도를 담음 | Data Preparer |
| 도메인 추상화 | Data Preparation **Agent** |

마지막 단계의 `Agent`는 시스템의 다른 부분에서 비슷한 역할이 이미 `agent`라는 표현을 사용하고 있어 채택되었다. 이렇게 메타모델에 새 개념이 추가된다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 이름은 **시스템을 얼마나 잘 이해하고 있는지 판별하는 리트머스 종이**다. 이름이 넌센스나 정직 수준에 머물러 있다면 시스템의 개념을 좀 더 깊이 파봐야 한다.
:::

### 2.5 개념 개별화 연습 — 라이프 게임

콘웨이의 라이프 게임(*Game of Life - 존 호턴 콘웨이가 고안한 세포 자동자*)을 예로 아키텍처 플립북을 만들며 개념 개별화를 연습할 수 있다. 라이프 게임의 규칙.

1. 살아있는 세포에 인접한 세포가 2개 이하일 때 죽는다(인구 부족)
2. 인접 세포가 2~3개면 다음 세대로 살아남는다
3. 인접 세포가 3개 초과면 죽는다(인구 과잉)
4. 죽은 세포에 인접한 세포가 3개면 다시 살아난다(재생)

::: callout {icon="📝" color="gray_bg"}
**직접 해보기**: 라이프 게임을 설명하는 아키텍처 플립북을 만들어보자.
- 규칙을 설명하는 동사와 명사는?
- 지어낸 이름은 이름 짓기 7단계 중 어디에 있는가?
- 모델이 답변해야 할 질문은?
- 다이어그램 범례에는 어떤 요소·관계가 있는가?
:::

## 3. 코드로 모델 구현하기

모델로 아키텍처를 유추할 수 있지만, 함부로 추측하면 오해를 산다. 대부분의 모델은 아키텍처의 표현일 뿐 코드와 별로 상관없다. **모델이 코드로 옮겨질 때 완전히 다른 모습이 될 수 있다.**

아키텍처 모델을 코드로 구현하면 이점이 크다.

- 설계의 무결성 유지 용이
- 원하는 품질 속성 촉진이 쉬워짐
- 모델과 코드가 한 몸으로 이동해 아키텍처가 누더기가 될 가능성 감소
- 시스템 자체에 설계 의도 내장 → 문서화 업무 감소

조지 페어뱅크스(George Fairbanks)의 **아키텍처적으로 명확한 코딩 스타일(*Architecturally Evident Coding Style - 모델의 힌트·사용 규칙·설계 근거를 코드에 포함해 개념 모델을 코드에 드러내는 방식*)** 을 사용하면 모델과 코드의 간극을 좁힐 수 있다.

### 3.1 모델 어휘를 코드에 그대로 사용하기

추상적 아키텍처를 코드로 옮길 때 흔한 혼란은 **용어 불일치**다. 아키텍처는 레이어·서비스·필터를 얘기하지만 코드는 패키지·클래스·메서드다.

**모델을 코드에 담는 가장 간단한 방법은 아키텍처 어휘를 그대로 사용하는 것**이다.

```typescript
// ❌ 어휘 불일치
class UserManager { ... }
class DataHelper { ... }

// ✅ 파이프-필터 패턴을 채택했다면
class DataFetcherFilter { ... }
class NormalizerFilter { ... }
class Pipe { ... }

// ✅ 시스템을 조종사·네비게이터에 비유했다면
class Pilot { ... }
class Navigator { ... }
```

도메인 주도 설계(*DDD - Domain-Driven Design*)에서 강조하는 대로 **도메인 모델을 코드에 넣는 방법**도 유효하다. 이벤트 기반 패턴이나 반응형 패턴도 도메인 정의에서 이벤트 모델을 만들고 패턴화한 형태다.

### 3.2 패키지 구성으로 아키텍처 반영

자바 컴파일러는 여러 클래스를 한 파일에 넣든 나누든 동일하게 컴파일한다. 하지만 코드 구성이 아키텍처 구조에 영향을 준다.

**레이어 기준 vs 기능 기준 패키징**.

```
// 레이어 기준
com.foo.application/
  ├─ display/
  ├─ business/
  └─ dataaccess/

// 기능 기준
com.foo.application/
  ├─ account/     (안에 display/business/data 다 포함)
  ├─ project/
  └─ user/
```

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 화이트보드에 그려진 패턴만으로는 아무런 품질 속성도 촉진하지 않는다. **코드가 촉진한다.** 구현된 시스템에 패턴이 없다면 그 아키텍처는 아무런 품질 속성과도 관련이 없다.
:::

### 3.3 코드로 아키텍처 제어하기

규칙만 믿지 말고 **코드로 아키텍처를 제어하는 방법**을 생각해야 한다. 코드로 꽉 짜인 설계는 거스를 수 없다.

**구조 형태별 제어 방법**.

| 구조 | 제어 방법 |
|---|---|
| **모듈 구조** | 접근 제어자(private/public)로 사용 가능성 제한. 정적 분석 도구로 규칙 위반 확인. 타입 시스템 활용 |
| **컴포넌트-커넥터 구조** | 계약에 의한 설계(*Design by Contract - 사전조건·사후조건·불변조건을 코드에 명시하고 위반 시 예외 발생시키는 기법*)로 빠른 실패. 인증 없이는 데이터 접근 불가 등의 강제 |
| **할당 구조** | PaaS·도커 컨테이너·인프라 코드(IaC)·깃 워크플로로 배포 제어. 정적 분석과 자동화된 아키텍처 검사 |

**계약에 의한 설계 예시(TypeScript)**.

```typescript
class OrderProcessor {
  // 사전조건: order는 유효해야 하고 결제가 완료되어야 함
  process(order: Order): void {
    if (!order.isValid()) {
      throw new Error('ASSUMPTION_VIOLATED: Order must be valid');
    }
    if (!order.isPaid()) {
      throw new Error('ASSUMPTION_VIOLATED: Order must be paid');
    }
    // 사후조건: 처리 후 order는 반드시 SHIPPED 상태
    this.doProcess(order);
    if (order.status !== 'SHIPPED') {
      throw new Error('INVARIANT_VIOLATED: Order must be SHIPPED after process');
    }
  }
}
```

### 3.4 아키텍처를 문서화하는 코드

코드로는 모델만 표현하게 된다. 코드로 의사결정을 강요할 수는 있지만, **왜** 그렇게 결정했는지는 알 수 없다. 좋은 이름 짓기와 패턴 사용에 더해 **주석**과 **예외 메시지**를 활용한다.

```typescript
// ❌ 범용 에러
throw new Error('UNKNOWN');

// ✅ 설계 이유가 담긴 에러
throw new Error('ASSUMPTION_VIOLATED: DocumentID required for validation');
```

### 3.5 코드에서 모델 생성하기

프로그래밍 언어·기술·패턴에 따라 다르지만, 생성된 모델로 규약을 점검하거나 설계상 변화를 자동 모니터링할 수 있다.

- 객체지향 언어 → UML 클래스·패키지 다이어그램 자동 생성
- 대부분 언어 → 종속성 분석 도구
- 컴포넌트-커넥터 구조 → 실행 중 관찰 가능하도록 계측 기능 추가

## 4. 라이언하트 프로젝트 사례

개발은 시작되었지만 험난했다. 팀은 실행 환경에 대해 전통적 계층 나눈 패턴을 사용했고 코드는 레이어 구조를 채택했다.

**문제**.

- 팀 구성원들이 아키텍처의 동일한 요소를 서로 다른 용어로 부름
- 설계 회의에서 모두 동의했다가도 각자 자리에 가면 다른 결론
- 코드는 이미 쓰레기 — 채택했다고 생각했던 패턴이 시스템 어디에도 없음

문제 파악 결과, **공통의 메타모델 추출**이 필요했다. 팀원들을 즉석 화이트보드 토론에 호출해 시스템을 정확하게 설명하도록 이끌었다. 결국 모델의 개념·요소·관계에 좋은 이름을 붙였고, 위키 문서에 메타모델을 기록해 공식화했다.

이후 페어 프로그래밍으로 코드 구조를 리팩터링했다. 시스템 구현 초기라 리팩터링은 간단했다. 하지만 이 과정에서 모든 사람이 현재 아키텍처에 동의하거나 이해하는 것은 아니라는 사실을 알게 되었고, **공동 워크숍**에서 아키텍처 대안을 살펴볼 필요가 있다고 판단했다.

::: callout {icon="📝" color="gray_bg"}
**직접 해보기**: 최근 참여한 프로젝트의 아키텍처를 팀원과 서로 다른 종이에 그려보자. 그림이 얼마나 다른가? 요소·관계 이름에 얼마나 차이가 있는가? 차이가 크다면 메타모델을 명시적으로 정리해야 한다는 신호다.
:::

## 요약

- 모델은 추상적 개념을 제공해 복잡한 사안을 다루는 데 도움을 준다. **누구나 상자와 선을 그릴 수 있지만 쓸모 있는 모델은 다르다**
- 좋은 모델은 **디자인 용어 사전, 관심사 유도, 품질 속성 예측, 아키텍트 의도 반영**의 네 가지 역할을 한다
- **아키텍처 메타모델**은 모델을 만드는 개념과 규칙 — 설계용 문법 같은 역할
- **개념 개별화**는 인식하는 개념을 다른 개념과 구분하는 인지적 과정. 호기심 순환 고리(질문→테스트→개별화→구축)를 따른다
- **패턴은 메타모델의 씨앗** — 적절한 패턴을 고르면 메타모델은 자연히 따라온다
- **이름 짓기 7단계**: 모름 → 넌센스 → 정직 → 정직하고 완전 → 올바름 → 의도를 담음 → 도메인 추상화. 이름은 시스템 이해도의 리트머스 종이
- 모델을 코드에 옮길 때는 **아키텍처 어휘를 그대로 사용**하고, **패키지 구성에 반영**하며, **접근 제어·계약·인프라 코드로 제어**한다
- 화이트보드 패턴이 아니라 **코드**가 품질 속성을 촉진한다
- 코드에서 UML·종속성 그래프 등 모델을 자동 생성해 아키텍처 준수 여부를 검증할 수 있다
- 팀 구성원이 같은 요소를 다른 용어로 부른다면 **공통 메타모델 추출**이 시급하다는 신호다

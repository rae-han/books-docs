# Chapter 7: Patterns (패턴으로 기초 만들기)

## 핵심 질문

아키텍처 패턴이란 무엇이며 왜 사용하는가? 흔히 쓰는 패턴들은 어떤 품질 속성을 촉진하고 어떤 대가를 요구하는가? 언제 어떤 패턴을 골라야 하며, 잘 알려진 패턴이 없을 때 새로운 패턴을 어떻게 발굴하는가?

---

## 1. 아키텍처 패턴이란 무엇인가

아키텍트가 접하는 기술적 문제 중 완전히 새로운 것은 거의 없다. 확장성, 유지보수성, 신뢰성, 고가용성, 테스트 용이성을 갖춘 소프트웨어 시스템은 수십 년간 만들어져 왔다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: **아키텍처 패턴(*Architecture Pattern - 특정 문제에 대해 재사용할 수 있는 해법. 알려진 구조의 조합으로 품질 속성을 끌어올리는 방법*)** 은 문제를 처음부터 디자인하는 수고를 덜어주고, 함정을 피하게 하며, 대화·설득에 사용할 수 있는 공용어를 제공한다.
:::

**디자인 패턴 vs 아키텍처 패턴**.

| 구분 | 디자인 패턴 (GoF 등) | 아키텍처 패턴 |
|---|---|---|
| 다루는 층위 | 클래스·객체 수준의 재사용성·유지보수성 | 시스템 전체의 품질 속성 시나리오 |
| 관심사 | 프로그래밍 언어 관용구, 객체 조합 | 설계 시간·실행 시간·추상적 컴포넌트 |
| 범위 | 좁고 구체적 | 넓고 추상적 |

실무에서는 두 패턴의 구분이 애매할 때도 있다. 누군가에겐 아키텍처지만 누군가에겐 프로그래밍 설계처럼 보인다.

## 2. 레이어 패턴

**레이어 패턴(*Layer Pattern - 관심사별로 코드를 수평으로 나누고, 상위 레이어는 하위 레이어만 사용하도록 제한하는 패턴*)** 은 가장 흔히 쓰이고 필요 이상으로 남용되는 패턴이다.

```
┌─────────────────┐
│      API        │
├─────────────────┤
│  비즈니스 로직   │
├─────────────────┤
│    공통 부분     │
├─────────────────┤
│  데이터 액세스   │
└─────────────────┘
   ↓ 사용 관계
```

| 항목 | 내용 |
|---|---|
| **카테고리** | 모듈 |
| **구성 요소** | 레이어(기능적으로 응집된 단위) |
| **관계** | 사용할 수 있음(allowed to use) |
| **사용 규칙** | 모듈은 하나의 레이어에만 존재. 상위 → 하위로만 사용. 순환 참조 금지 |
| **강점** | 유지보수성, 독립 구동 가능성, 재사용성, 테스트 가능성, 설계 시점 유연성 |
| **약점** | 추상화 작업으로 복잡도·성능 저하 가능. 너무 많은 레이어는 구현 어려움 |

## 3. 포트와 어댑터 패턴

**포트와 어댑터 패턴(*Ports and Adapters - 핵심 비즈니스 로직을 격리하고, 외부 시스템과의 연결점을 포트로 정의해 어댑터로 교체 가능하게 하는 패턴. 육각형 아키텍처라고도 함*)** 은 앨리스터 콕번(Alistair Cockburn)이 육각형 아키텍처(*Hexagonal Architecture - 포트와 어댑터를 육각형으로 시각화한 이름*)라는 이름으로 최초 제안했다.

```
    ┌── 어댑터(레이더 시뮬레이터) ──┐
    │                              │
[포트] → ┌──────────────────┐ ← [포트]
         │  핵심 비즈니스     │
[포트] → │      로직         │ ← [포트]
         └──────────────────┘
    │                              │
    └── 어댑터(mySQL)             ─┘
    └── 어댑터(이벤트 버스)       ─┘
```

| 항목 | 내용 |
|---|---|
| **카테고리** | 모듈, 컴포넌트-커넥터 |
| **구성 요소** | 레이어(도메인·비즈니스 로직), 포트(인터페이스), 어댑터(외부와 상호작용) |
| **관계** | 노출(레이어→포트), 구현(포트→어댑터), 주입(어댑터→레이어) |
| **사용 규칙** | 어댑터는 포트 제약 충족 필요. 포트를 통해서만 정보 주입 |
| **강점** | 테스트 가능성, 유지보수성, 변경 적응력. 팀별 병렬 개발 |
| **약점** | 실행 중 어댑터 선택 메커니즘 필요. 서드파티 어댑터는 보안·안정성 주의 |

레이더 장치를 시뮬레이터로 교체해도 핵심 비즈니스 로직은 그대로 유지된다. 로깅·통신 버스도 상황에 따라 교체 가능하다.

## 4. 파이프와 필터 패턴

**파이프와 필터 패턴(*Pipe and Filter - 단방향 데이터 변형만 수행하는 필터들을 파이프로 연결한 패턴*)** 은 데이터 분석·처리에서 자주 사용된다. 유닉스 명령어 파이프(`|`)를 써봤다면 이미 이 패턴을 맛본 셈이다.

```
[원본] → 불량 데이터 제거 → 점수 매기기 → 점수 정규화 → 저장 → [CSV 파일]
```

**배치 시퀀셜 패턴과의 차이**.

| 항목 | 파이프와 필터 | 배치 시퀀셜 |
|---|---|---|
| 병렬 처리 | 가능 | 불가능 (단계별 순차) |
| 데이터 저장 | 스트리밍 | 각 단계마다 디스크에 저장 |

| 항목 | 내용 |
|---|---|
| **카테고리** | 컴포넌트와 커넥터 |
| **구성 요소** | 필터(데이터 읽고 변형해 내보냄), 파이프(순서 유지하며 전달), 소스/싱크 |
| **관계** | 부착물(한 필터 출력 → 다른 필터 입력) |
| **사용 규칙** | 입출력 호환되는 필터끼리만 연결. 필터는 독립적이어야 함 |
| **강점** | 성능, 재사용성, 변경 대응력 |
| **약점** | 인터랙티브하지 않음. 사용자 인터페이스 갖기 어려움. 잘못 구성하면 성능 하락 |

## 5. 서비스 지향 아키텍처(SOA) 패턴

**SOA(*Service-Oriented Architecture - 특정 기능을 독립적 서비스로 구현하고 실행 중에 조합해 작업을 수행하는 패턴*)** 는 서비스가 어떻게 구현되었는지 몰라도 호출·실행할 수 있어야 한다.

전통적 SOA는 SOAP(*Simple Object Access Protocol - XML 기반 메시지 프로토콜*) 메시지 버스에 의존했지만, 최신 SOA는 **마이크로서비스**로 세분화하고 가벼운 HTTP로 통신한다.

```
       [웹 UI]
          │ HTTP/REST
          ↓
   ┌──[서비스 레지스트리 (Eureka)]──┐
   │              ↕                  │
   ↓                                 ↓
[서비스 A]                       [서비스 B]
   │                                 │
   ↓                                 ↓
[DB]                              [DB]
```

| 항목 | 내용 |
|---|---|
| **카테고리** | 컴포넌트와 커넥터 |
| **구성 요소** | 서비스, 서비스 레지스트리, 메시지 시스템(SOAP/REST/gRPC/비동기) |
| **사용 규칙** | 서비스는 자신이 어떻게 쓰이는지 모름. 외부 컴포넌트(레지스트리 등)를 통해서만 다른 서비스 조회 |
| **강점** | 통합성, 재사용성, 확장성 |
| **약점** | 분산 시스템의 모든 복잡성 보유. 버전 관리 어려움. 조립 어려움 |

---

## 6. 발행/구독 패턴

**발행/구독 패턴(*Publish-Subscribe - 발행자와 구독자가 서로 모르는 상태로 이벤트 버스를 통해 간접적으로 통신하는 패턴*)** 은 다수의 독립 컴포넌트가 동일 정보를 다룰 때 주로 사용한다.

```
[가스 센서] ─┐                        ┌── [알람]
[온도 센서] ─┼── [이벤트 버스] ──┼── [보일러]
[습도 센서] ─┘                        ├── [온도 조절 장치]
                                       └── [사용 이력]
```

| 항목 | 내용 |
|---|---|
| **카테고리** | 커넥터와 컴포넌트 |
| **구성 요소** | 발행자, 구독자, 이벤트 버스 |
| **관계** | 발행(→ 버스), 구독(버스에 등록) |
| **사용 규칙** | 모든 통신은 이벤트 버스에서. 컴포넌트는 이벤트 규약을 정의 |
| **강점** | 확장성, 재사용성, 테스트 가능성. 버스에 따라 가용성·신뢰성도 상승 |
| **약점** | 비동기 → 전체 성능 예측 어려움. 이벤트 버스에 안정성이 전적으로 종속 |

---

## 7. 공유 데이터 패턴

**공유 데이터 패턴(*Shared Data - 다수 컴포넌트가 하나의 데이터 저장소에 접근하는 패턴*)** 은 많은 컴포넌트가 많은 데이터를 필요로 할 때 유용하다.

```
[회원 검색] ─┐         ┌─ [프로필 보기]
[회원 등록] ─┼─ [저장소]─┼─ [권한 정보]
[프로비저닝]─┘         └─ [비밀번호 변경]
```

발행/구독은 메시지 전달로 통신하지만, 공유 데이터는 **데이터와 데이터 소스가 컴포넌트 간 인터랙션의 중간자** 역할을 한다는 점이 다르다.

| 항목 | 내용 |
|---|---|
| **카테고리** | 컴포넌트와 커넥터 |
| **구성 요소** | 데이터 저장소, 데이터 접근 컴포넌트 |
| **관계** | 읽기, 쓰기 |
| **사용 규칙** | 데이터 접근 컴포넌트만 데이터를 직접 다룸 |
| **강점** | 일관성·보안·정보보호 잘 관리 시 신뢰성 상승. 확장성·가용성도 튜닝 가능 |
| **약점** | 저장소 정지 시 전체 시스템 정지. 저장소 교체 시 모든 접근 컴포넌트 수정 필요. 남발 주의 |

## 8. 멀티계층 패턴

**멀티계층 패턴(*Multi-Tier - 실행 중인 소프트웨어를 논리적 단위(계층)로 나누고, 각 그룹을 특정 하드웨어에 배포하는 패턴*)** 은 레이어 패턴과 개념적으로 비슷하지만 **실행 환경(*Runtime Environment - 서버·클라우드 플랫폼 등 실제 실행되는 인프라*)** 관점이라는 점이 다르다.

| 구분 | 레이어(Layer) | 계층(Tier) |
|---|---|---|
| 구조 관점 | 모듈 구조 | 컴포넌트-커넥터 or 할당 |
| 시점 | 설계 시점 | 실행 시점 |
| 관심사 | 코드 응집·결합 | 배포·플랫폼 |

```
┌─ 애플리케이션 계층 (고객 서버) ─┐
│         API 계층                │
└─────────────┬───────────────────┘
              │ HTTP/REST
┌─ 프로덕트 서비스 계층 (공유 플랫폼) ─┐
│      공통 서비스 계층                │
└─────────────┬────────────────────────┘
              │ gRPC
┌─ 데이터 계층 (AWS 호스팅) ─┐
│      데이터베이스           │
└────────────────────────────┘
```

| 항목 | 내용 |
|---|---|
| **카테고리** | 컴포넌트와 커넥터 or 할당 |
| **구성 요소** | 계층(기능·계산·팀·통신·보안·데이터 접근권 단위로 분할) |
| **관계** | 소속됨, 통신함, 통신할 수 있음, 할당됨 |
| **사용 규칙** | 컴포넌트는 하나의 계층에만 속함. 계층 간 통신은 교차 지점에서만 |
| **강점** | 보안·성능·가용성·유지보수성·변경 가능성 향상. 비용과 배포 단위 판단 용이 |
| **약점** | 시스템이 커지면 계층이 애매해짐. 너무 많은 계층은 성능·유지보수성 저하 |

## 9. 숙련된 전문가 패턴

**숙련된 전문가 패턴(*Cone of Competence - 하나의 전문가 팀이 특정 영역(기술·유스케이스·패턴)의 표준·도구·교육을 제공하는 패턴*)** 은 시스템을 직접 개발하진 않지만 다른 팀의 개발 업무를 효율화하는 지원팀을 두는 방식이다.

| 항목 | 내용 |
|---|---|
| **카테고리** | 배치 |
| **구성 요소** | 숙련된 전문가 팀, 담당 영역(패턴·기술·유스케이스) |
| **관계** | 책임(전문가 팀 ↔ 담당 영역) |
| **강점** | 보안·확장성·성능·안정성·유지보수성 등 품질 속성 전반에 긍정적 영향 |
| **약점** | 전문가 이직 시 조직 붕괴 위험. 역량 부족 시 오히려 개발 속도 저하 |

**대기업 사례**.

| 숙련된 전문가 팀 | 담당 영역 |
|---|---|
| 업무 일정 관리·유스케이스 | 프레임워크·개발 도구 |
| 성능 | 성능·부하 테스트 도구·자원 |
| 데이터베이스 | 적합한 DB 선택·이관·교육 |
| 핵심 플랫폼 | 공통 컨테이너·도커 이미지·헬스체크 |

## 10. 오픈소스 공헌 패턴

**오픈소스 공헌 패턴(*Open Source Contribution - 팀이 특정 컴포넌트의 "자비로운 독재자" 역할을 하며 다른 팀의 코드 기여를 검토·수락/거부하는 패턴*)** 은 아키텍처가 너무 중앙집중적으로 몰리지 않게 하며, 전문가가 여러 팀에 넓게 퍼져 있거나 공통 의존 컴포넌트가 있을 때 유용하다.

| 항목 | 내용 |
|---|---|
| **카테고리** | 할당 |
| **구성 요소** | 팀, 저장소 |
| **관계** | 책임자(자비로운 독재자), 공헌자 |
| **강점** | 재사용성, 유지보수성, 개발 속도 향상 |
| **약점** | 컴포넌트가 잘 분리되어야만 성립. 학습 곡선이 가파르면 아무도 공헌 안 함 |

성공하려면 코드 스타일 가이드, 테스트 가능한 설계, 명확한 제약을 미리 정해두는 것이 좋다.

## 11. 큰 진흙공 패턴

**큰 진흙공 패턴(*Big Ball of Mud - 정의된 요소와 관계 없이 얽혀 있는 아키텍처 안티패턴*)** 은 패턴이라기보다 현실에서 흔히 볼 수 있는 상황이다. 정의된 요소·관계가 없고 어떤 품질 속성도 끌어올리지 않으며, 유지보수성·확장성이 크게 떨어진다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 모놀리식뿐만 아니라 **마이크로서비스도 분산 처리 단위가 큰 진흙공처럼 나쁘게 진화**할 수 있다. 아키텍처 원리의 이해가 결여되었거나 개발 문화가 성숙하지 않은 경우에 등장한다.
:::

전략적으로 쓰이기도 한다 — 기술 부채를 감당하고 초기 버전을 빠르게 릴리스해야 할 때. 하지만 대체로 첫 버전을 제때 없애지 못하고 기술 부채도 갚지 못해 위험한 전략이다.

## 13. 새로운 패턴 발굴하기

패턴은 경험에서 우러난다. 아키텍트가 새 패턴을 발굴하는 모습은 생물학자가 새 종을 발견하는 방식과 비슷하다 — 들판에서 시간을 보내고 관찰한다.

**발굴하는 두 가지 방법**.

1. **문제에 집중**: 공통 문제 발견 → 일반화된 해결법 생성 → 기존 해결법과 비교 → 패턴화
2. **해법에 집중**: 반복 적용된 해결법 발견 → 어떤 방법이었는지 분석 → 공통 분모 정의

패턴을 정립했으면 관련 경험자에게 피드백을 받고, 직접 구현해보며 개선한다.

## 14. 라이언하트 프로젝트 사례

팀원들과 콘퍼런스 룸에 모여 최근까지의 설계 탐색을 공유했다. 검색 기술은 팀원 중 한 명의 발표·데모로 쉽게 선택할 수 있었다. 하지만 아키텍처 선정은 만만치 않았다.

한 팀원이 "단순한 3계층 아키텍처로 하지요?"라고 화이트보드에 그림을 그렸다. 활동 19 "화이트보드 토론"이 즉흥적으로 이루어졌다. 다른 팀원이 반박했다. "마이크로서비스 책을 읽었는데 적용해도 좋을 것 같군요." 또 다른 팀원이 "간단한 웹 애플리케이션으로는 너무 작업량이 많아 보입니다"라고 거들었다.

화이트보드 토론이 몇 분간 지속되며 새 패턴도 제안되었다. 이제 개입할 때다. "점점 주제에서 벗어나고 있으니 결정을 내릴 때가 되었어요. 지금까지 거론한 아키텍처들은 이번 프로젝트의 **최상위 품질 속성**에 어떤 영향을 주나요?"

> **직접 해보기**: 진행 중인 프로젝트에서 사용하는 패턴을 3개 이상 열거해보자. 각 패턴이 촉진하는 품질 속성과 대가로 치르는 품질 속성을 표로 정리하고, 팀원과 함께 검토해보자. 큰 진흙공 패턴에 빠져 있는 부분은 없는가?

## 요약

- **아키텍처 패턴**은 특정 문제의 재사용 가능한 해법이며 알려진 구조의 조합으로 품질 속성을 끌어올린다
- 디자인 패턴이 클래스·객체 수준을 다룬다면, 아키텍처 패턴은 **품질 속성 시나리오** 수준을 다룬다
- **레이어**는 유지보수성을, **포트-어댑터**는 테스트 가능성과 변경 적응력을, **파이프-필터**는 성능·재사용성을 촉진
- **SOA**는 통합성·확장성을 주지만 분산 복잡성을, **발행/구독**은 유연성을 주지만 순서 보장을 어렵게 함
- **공유 데이터**는 신뢰성을 주지만 저장소가 SPOF, **멀티계층**은 실행 환경 관점의 그룹화(레이어와 다름)
- **숙련된 전문가 패턴**과 **오픈소스 공헌 패턴**은 배치·할당 관점의 조직 구조 패턴
- **큰 진흙공**은 패턴이라기보다 안티패턴 — 마이크로서비스도 이 함정에 빠질 수 있음
- 어떤 패턴도 완벽하지 않다. **품질 속성 요구사항에 맞춰 선택**하고, 필요하면 새 패턴을 발굴한다
- 새 패턴은 **문제 중심** 또는 **해법 중심** 관점에서 발굴할 수 있으며, 반드시 피드백과 실제 구현으로 검증한다

# Chapter 10: Visualizing Design (설계 시각화하기)

## 핵심 질문

아이디어를 손에 잡히지 않는 상태로 두면 왜 아무도 이해하지 못하는가? 하나의 다이어그램에 아키텍처를 모두 담을 수 없다면 어떤 관점으로 나누어 그려야 하는가? 요소-역할 뷰, 구체화 뷰, 품질 속성 뷰, 매핑 뷰, 아키텍처 카툰은 각각 언제 쓰이는가? 훌륭한 다이어그램이 되기 위한 조건은 무엇이며, C4 모델은 왜 도움이 되는가?

---

## 1. 왜 그림으로 그려야 하는가

아이디어를 공유하는 최선의 방법은 아이디어를 손에 잡힐 듯하게 만드는 것이다. 아키텍처를 개념적으로만 말하면 대부분 이해하지 못한다. 하지만 그림을 보여주면 상대는 자신의 이해력에 맞추어 스스로 탐색하고 이해한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아직 구체적이지 않은 아이디어를 이야기할 때는 화이트보드 앞에 서서 스케치하면서 이야기하는 게 좋다. **아이디어를 그리기 시작하면 참석자들의 상상 눈높이가 맞춰진다.** 아키텍처 다이어그램은 예쁘게 그릴 필요가 없다. 효과적으로 공유할 수 있으면 된다.
:::

앞서 8장에서 모델을 정확하게 규정할수록 아키텍처로 품질 속성을 끌어올릴 수 있다고 배웠다. 이 장에서는 그 모델을 다이어그램으로 옮겨 개발자들과 대화하는 방법을 알아본다.

---

## 2. 다양한 관점으로 아키텍처 표현하기

그림 한 장에 아키텍처의 모든 내용을 담을 수는 없다. 다이어그램 하나에 모든 것을 담기보다는, 다양한 **관점(*viewpoint - 특정 이해관계자 또는 이와 긴밀히 연관된 고려사항의 시각*)**에 기초해 그린 다이어그램 여러 개에 나눠 담는 게 좋다.

> 구글 지도로 여행 계획을 짠다고 생각해보자. 도시를 확대해 거리를 보고, 축소해 주요 고속도로를 살펴본다. 날씨나 실시간 교통 정보를 켜기도 하고, 위성지도·지형지도로 바꿔보기도 한다. 다양한 관점으로 여행 계획이 완성된다. 아키텍처도 이렇게 여러 관점으로 그려서 이해관계자들의 질문에 답할 수 있어야 한다.

소프트웨어 아키텍처를 바라보는 여러 관점을 **뷰(*view - 특정 관점에서 본 아키텍처의 표현*)**라고 부른다. 아키텍처 뷰는 미리 정해진 게 없지만, 대부분의 시스템에서 유용하게 쓰이는 몇 가지가 있다.

### 2.1 요소-역할 뷰

선과 상자는 아키텍트가 가장 많이 쓰는 도구다. 선과 상자로 이루어진 다이어그램은 복잡한 구조를 보기 쉽게 표현한다. 하지만 그것만으로는 부족하다.

**요소-역할 뷰(*element-responsibility view - 각 요소가 어떤 책임을 지는지 나열한 뷰*)**는 그림에 표현되지 않는 각 요소의 역할을 주석·표·글로 보완한다. 요소와 역할을 나열할 수만 있어도 실제 동작하는 시스템을 만들 수 있다.

### 2.2 구체화 뷰

범죄 영화에서 흐릿한 얼굴을 "해상도 향상!"이라고 외치며 확대하는 장면이 나오곤 한다. 소프트웨어의 멋진 점은 흐릿한 이미지와 달리 **한없이 세부적으로 확대할 수 있다는 것**이다.

**구체화 뷰(*refinement view - 한 모델을 점점 상세하게 확대하며 보여주는 뷰*)**는 요소의 내부 동작을 더 배율이 높은 렌즈로 들여다보는 것과 같다.

라이언하트 프로젝트의 예시를 보자. 처음에는 시스템 전체(웹 UI, 디스플레이 비즈니스, 검색 서비스, 즐겨찾기 서비스, 알림 등)를 상자로 그린다. 여기서 "디스플레이 비즈니스" 요소만 확대하면 그 안이 다시 레이어 패턴(디스플레이·비즈니스·서비스 액세스·모델 및 유틸리티)으로 구성됨을 알 수 있다. 한 번 더 확대하면 각 레이어 안의 패키지(RFP, 즐겨찾기 서비스, 검색 서비스, 회계 등)까지 드러난다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 2.1절 "모호함을 유지하라"에서 배운 아키텍처 미니멀리즘 원리에 따라, 모델은 **특정 품질 속성을 충족하고 우선순위가 높은 위험을 줄일 수 있음을 입증할 수준까지만** 구체화한다. 다이어그램 하나에 모든 관계를 표현하면 오히려 이해가 어려워진다.
:::

### 2.3 품질 속성 뷰

**품질 속성 뷰(*quality attribute view - 아키텍처가 특정 품질 속성을 어떻게 충족하는지 보여주는 뷰*)**는 현재 논의 중인 사안과 관련 없으면 가리고, 관련 있는 요소만 강조한다.

예를 들어 라이언하트 프로젝트의 가용성 시나리오가 "사용자가 시스템에서 보내는 시간의 99%는 열람 가능한 RFP 목록을 받고 RFP를 찾아보는 일"이라면, 이를 만족시키기 위해 중복 패턴을 적용한 다이어그램을 그린다.

- DNS/로드 밸런서 뒤에 검색 서비스 여러 인스턴스 배치
- 검색 색인은 1차·2차로 이중화
- 상태를 보존하지 않는 마이크로서비스는 쿠버네티스로 오토 스케일 적용
- 상태를 보존하는 검색 색인은 로드 밸런서와 DNS로 라우팅

이 다이어그램으로 "검색 색인 하나가 실패하면 로드 밸런서가 감지해 보조 색인으로 라우팅한다"는 시나리오를 눈으로 확인한다. 다만 다이어그램만으로는 "핑을 얼마나 자주 보내는가?", "로드 밸런서가 다운되면 어떻게 되는가?" 등에 답할 수 없으므로, 반드시 **글로 설명을 보완**해야 한다.

### 2.4 매핑 뷰

여러 뷰를 맞붙여서 하나의 뷰로 만들면 요소끼리 어떤 관계가 있는지 새롭게 보여줄 수 있다. 이를 **매핑 뷰(*mapping view - 두 개 이상의 뷰를 결합해 새로운 관점을 제시하는 뷰*)**라고 한다.

가장 유용한 두 조합은 **업무 할당 뷰**와 **배포 뷰**의 결합이다. 다음은 라이언하트 프로젝트의 업무 할당 뷰 예시다.

| 컴포넌트 | 할당된 팀 | 비고 |
|---|---|---|
| 웹 UI | 벌꿀오소리 | 웹 프런트엔드 전문가 집단 |
| 디스플레이 비즈니스 | 벌꿀오소리 | 웹 UI와 강하게 결합 |
| 검색 서비스 | 빨간 티셔츠 | — |
| 검색 색인 | 빨간 티셔츠 | Solr 개발자 집단 |
| 즐겨찾기 서비스 | 벌꿀오소리 | 사용자 경험에 직접 영향 |
| 알림 | 미정 | 가능한 팀이 우선 담당 |
| 크롤러 | 빨간 티셔츠 | 팀 전문성 있음 |
| DNS/로드 밸런서 | 트론 | 인프라 전문가 |
| 사용자 메타데이터/계약 DB | 스프링필드시 | 시에서 DB 보유 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 뷰는 다이어그램이 아니다. **목표는 의미 있는 설계 의사결정이 이루어지도록 하는 것**이다. 때로는 표만 만들어도 목표를 달성할 수 있다.
:::

매핑 뷰는 이해관계자 간의 서로 다른 관점을 반영한다. 업무 할당 뷰는 프로젝트 매니저에게, 배포 뷰는 IT 부서에게 완벽히 부합한다. 아키텍처 컴포넌트와 기능별 부가가치를 나란히 매핑하면 제품 관리자와 개발자 모두 소프트웨어의 어떤 부분이 어떤 기능을 지원하는지 파악할 수 있어 공감대가 형성된다.

### 2.5 아키텍처 카툰

지금까지 본 뷰는 대체로 정교하다. 정교함은 곧 비용이다. **아키텍처 카툰(*architecture cartoon - 분석보다 소통을 위해 빠르게 그린 비정교한 스케치*)**은 정교함은 떨어지지만 빠르게 그릴 수 있어 비공식적 소통, 탐색 워크숍, 즉석 토론에 잘 어울린다.

카툰은 말을 해부학적으로 정교하게 그린 그림과, 다리 네 개·꼬리·갈기만 있는 카툰 그림에 비유할 수 있다. 근육이나 해부학 정보가 필요하면 정교한 그림이 필요하지만, "말의 생김새"만 표현하려는 목적이라면 카툰도 충분하다.

라이언하트 프로젝트의 아키텍처 카툰은 웹 UI, 즐겨찾기 서비스, 검색 서비스(Apache Solr), 알림 프로세스, 크롤러 프로세스 등의 핵심 요소를 간단한 상자로만 표현한다. 아이디어를 다루는 동안에는 정확하지 않은 표기법을 써도 괜찮다. **설계 결정을 내려야 할 즈음에 더 정확한 모델을 만들면 된다.**

### 2.6 사용자 지정 뷰

특정한 목적에 맞게 창의적인 뷰를 만들 수 있다. 뷰는 여러 변수의 조합이다. 예를 들어 시스템 성능의 병목 현상을 보여줘야 한다면, 컴포넌트 다이어그램에 정보의 흐름과 실행 시간에 따른 색상을 추가해 성능 뷰를 즉시 만들 수 있다.

단 유의할 점은 **모든 뷰(심지어 커스텀 뷰)도 8.2절 "메타모델 설계하기"에서 설명한 메타모델로 관리되어야 한다**는 것이다.

## 3. 멋진 다이어그램 그리기

훌륭한 다이어그램은 그저 아름다운 그림이 아니다. **아키텍처의 개념적 토대를 반영한 정확한 모델**이다.

| 이렇게 하세요 | 이렇게 하지 마세요 |
|---|---|
| 범례를 만들어 다이어그램의 메타모델을 요약 | 다이어그램에서 쓰는 용어를 상대가 이미 안다고 가정 (UML조차 그렇게 단정 금지) |
| 구체적 제목을 붙이고 어떤 구조를 설명하는지 명시 | 다이어그램 하나에 모든 내용을 담기 |
| 주석으로 명료하게 설명 (흑백 인쇄 시 색이 사라짐에 주의) | 화려한 치장이나 지나치게 다양한 선·도형 사용 |
| 모든 다이어그램에 일관된 용어 사용 | 패턴 설명 생략 |
| 패턴을 시각적으로 드러내기 | — |

### 3.1 범례를 만들어라

범례가 없는 다이어그램은 상자·화살표·원통이 무슨 의미인지 알기 어렵다. 동일한 다이어그램에 범례를 추가하면 "이는 마이크로서비스", "A → B는 JDBC로 접속", "A ⇒ B는 Apache Thrift로 호출" 같은 정보가 명확해진다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 범례는 아키텍처의 **개념적 메타모델**을 알려준다. 범례를 먼저 그리면 다이어그램을 현실적이고 일관성 있게 유지할 수 있으며, 이를 분석 도구로도 쓸 수 있다. UML을 쓰든 직접 만든 표기법을 쓰든 **모든 다이어그램에는 범례가 있어야 한다**.
:::

### 3.2 패턴을 눈에 띄게 하라

앞의 다이어그램에는 비밀이 하나 있다. 컴포넌트의 위치만 조금 바꿔봐라. 서비스들을 재배치하면 **멀티 계층 패턴**이라는 사실이 드러난다. 이 재배치가 사소해 보여도, 패턴을 보이게 하면 아키텍처에 숨겨진 의도를 상세 구현자들에게 전달할 수 있다. 예컨대 REST API가 멀티 계층의 최상위라면, 이 API는 데이터 계층의 DB와 직접 통신하지 않을 것임을 예상할 수 있다.

### 3.3 C4 모델

> 단순하고 완결성 있게 하세요.
> — 사이먼 브라운, 『Software Architecture for Developers』 저자

사이먼 브라운의 **C4 모델**은 다양한 추상화 수준을 보여주는 다이어그램 모음이다.

| 단계 | 다이어그램 | 내용 |
|---|---|---|
| C1 | 시스템 콘텍스트 다이어그램 | 시스템이 사용자·주변 환경과 어떻게 연결되는가 |
| C2 | 컨테이너 다이어그램 | 시스템 안 애플리케이션·데이터 저장소 단위 구성 |
| C3 | 컴포넌트 다이어그램 | 특정 컨테이너 내부의 컴포넌트 |
| C4 | UML 클래스 다이어그램 | 특정 컴포넌트의 코드 수준 표현 |

UML은 여전히 유용하지만 저자는 소프트웨어 아키텍처를 설명할 때 간단한 상자와 선으로 그리는 편이다. 혼란을 줄이려면 가능한 한 단어 자체가 설명을 내포하도록 용어를 설정하고, 필요하면 이름과 범례를 추가하자.

### 3.4 색·모양·글꼴 신중히 사용하기

다이어그램에서는 점 하나도 의미가 있다. 색상·모양·방향·글꼴·위치는 모두 의미를 가진다. 다른 색상이나 모양을 쓰고 싶다면 **다이어그램을 예뻐 보이게 하는 수준이 아니라 다른 아이디어를 강조하기 위해서** 사용해야 한다.

아이디어를 공유할 때는 일관성이 최고다. 개념적으로 동일한 메타모델을 가진 아이디어가 두 다이어그램에 있다면 동일한 모양과 색을 써야 한다. 다이어그램 독자는 색상과 글꼴의 변화에 어떤 의미가 숨겨져 있는지 찾으려 하기 때문이다.

### 3.5 그림 옆에 이야기를 곁들여라

그림은 설명과 함께 있을 때 가장 좋다. 글을 포함하면 뷰의 여러 요소가 어떻게 결합해 품질 속성을 끌어올리거나 억제하는지, 왜 이렇게 설계했는지 이야기로 풀어낼 수 있다. 때로는 다이어그램이 뷰에서 가장 흥미롭지 않기도 하다. 진짜 움직이는 부분을 설명하려면 **글을 써야 한다.**

---

## 4. 아키텍처 명세 언어(ADL)를 써야 할까

**ADL(*Architecture Description Language - 아키텍처를 수학적·형식적으로 기술하는 언어*)**은 특정 모델에 대한 용어를 강제하고 제한하여 자동으로 모델을 확인해준다. 어떤 ADL에는 리버스 엔지니어링으로 코드를 생성하는 기능도 있다.

하지만 실무에서 쓰기엔 편치 않다. ADL은 설계 표현 범위를 제한하고, 대부분 도구가 완성도 부족·학습 곤란이다.

> 저자의 조언: ADL은 진심으로 사용하고 싶을 때만 사용하라.

---

## 5. 라이언하트 프로젝트

소프트웨어 개발이 한창 순조롭게 진행 중이다. 아키텍처도 세부사항을 포함하며 성숙해지고 있다. 팀은 정기적으로 그림을 그리며 아이디어를 모으고 대안을 모색한다.

팀은 설계 토론 중에 동일한 다이어그램을 스케치한다. 휴대폰 카메라로 화이트보드 스케치를 찍고 간단한 기록과 함께 사진을 소스코드 저장소에 넣는다. 또한 **요소-역할 뷰**를 만들어 모든 요소가 어우러져 동작하는 방식을 쉽게 설명한다. 아키텍처에 새 요소를 추가할 때 특히 유용하다.

지금까지 아무런 형식 없이 설계 결정을 공유해왔기에 종종 놓치는 것이 있었다. 이제는 문서화에 좀 더 힘을 쏟아 흩어진 지식을 모아야 할 때다.

## 6. 직접 해보기

> **직접 해보기**: 최근에 만든 소프트웨어의 다이어그램을 하나 가져와라. 지금까지 배운 시각화 방법에 비추어 이 다이어그램을 어떻게 개선할지 생각해보자.
>
> - 다이어그램이 어떤 점을 유추하는 데 도움이 되는가?
> - 다이어그램의 핵심 패턴은 무엇인가? 패턴이 숨겨져 있는가?
> - 다이어그램이 내포하는 메타모델은 무엇인가? 다이어그램 없이도 그 자체로 이해할 수 있는가?
> - 다이어그램은 완성본인가? 더 단순화해도 여전히 이해할 수 있는가?

## 요약

- **추상적 아이디어를 공유할 때는 그림을 그려라** — 그리면 상상의 눈높이가 맞춰지고 새로운 통찰이 솟는다
- 그림 한 장으로 아키텍처 전체를 담을 수 없다. **여러 관점(뷰)으로 나눠 그린다**
- **요소-역할 뷰**: 요소가 무슨 일을 하는지 나열 / **구체화 뷰**: 확대하며 세부 표현 / **품질 속성 뷰**: 특정 품질 속성 관련 요소만 강조 / **매핑 뷰**: 두 뷰를 결합해 새 관점 창출 / **아키텍처 카툰**: 빠르고 비공식적인 스케치
- 훌륭한 다이어그램은 **범례**, **구체적 제목**, **주석**, **일관된 용어**, **패턴 시각화**를 갖춘다
- **C4 모델**은 시스템 콘텍스트 → 컨테이너 → 컴포넌트 → 코드의 4단계 다이어그램 모음
- 다이어그램만으로는 아무 말도 하지 못한다. **이야기는 여러분에게 달렸다**
- ADL은 필요할 때만 사용. 대부분은 상자·선·범례로 충분하다

# Chapter 12: Evaluating Architecture (아키텍처 평가하기)

## 핵심 질문

아키텍처는 왜 "설계 끝나는 시점에 한 번만" 평가하면 안 되는가? 아키텍처 평가에는 무엇이 필요한가(산출물·루브릭·통찰력)? 좋은 루브릭은 어떻게 만드는가? 평가 워크숍은 어떻게 준비·진행·분석하는가? ATAM은 무엇이며 언제 쓰는가? 아키텍처 이슈의 "무지개"란 무엇인가? 격식 있는 평가와 격식 없는 평가를 어떻게 조합하는가?

---

## 1. 평가를 통해 배우기

학교에서 성적표는 학생·학부모·선생님 모두에게 중요한 피드백이다. 한 해가 끝날 때까지 결과를 기다리기보다 중간고사·기말고사로 정기적으로 성적을 받으면 너무 늦기 전에 보강할 부분을 알 수 있다. 아키텍처도 마찬가지다.

**아키텍처 평가(*architecture evaluation - 아키텍처가 목적에 얼마나 부합하는지 배우는 과정*)**의 흔한 오해는 "설계 끝나는 시점에 한 번만" 검사한다는 관념이다. 이때 아키텍처가 전체적으로 정확하지 않으면 실패로 선언하며 구현에 들어가지도 않는다. **이 생각은 완전히 틀렸다.**

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍처는 절대 **전적으로 좋거나 전적으로 나쁘지 않다**. 전체 아키텍처를 하나의 뷰로 볼 수 없듯이 한 번에 평가할 수도 없다. 한 컴포넌트가 잘 설계되어 있어도 다른 컴포넌트가 위험으로 가득할 수 있다. 구현 전에 모든 설계를 완벽히 할 수는 없다.
:::

**사인오프(*sign-off - 특정 기준을 충족했음을 확인하고 통과 신호를 주는 승인 절차*)**는 통합 지점이 많거나 중요 하드웨어 요구사항이 있는 시스템에서 재작업을 줄인다. 하지만 사인오프를 "통과만 하면 되는 검문소"로만 활용하면 좋은 평가 기회를 놓친다.

두 가지 질문에 답하는 것이 목표다. **"아키텍처가 얼마나 좋은가?"** 그리고 **"어떤 이유로 이 아키텍처가 좋은가?"** 답변에는 **아키텍처 핵심 요구사항(ASR)**을 활용한다.

> **이페크 오즈카야, 카네기멜런대 SEI 시니어 멤버의 조언**:
>
> - **유스케이스 시퀀스 다이어그램은 아키텍처가 아니다** — 유용하지만 클래스 추상화가 부족해 시스템 동작 추론에 부족하다
> - **코드 리뷰는 아키텍처 리뷰를 대체할 수 없다** — 코드 리뷰만으로는 아키텍처 요소를 일관된 관점으로 다루지 못한다
> - **아키텍처 요소에 네모만 있는 건 아니다** — 네모보다 이를 잇는 **선**이 아키텍처 의사결정의 중요한 맥락을 전달한다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 성능 향상을 목표로 하면 요소 간 통신의 빈도·용량에 중점을 둔다. 자주 변경되는 요소면 상호작용을 제한한다. 보안을 최적화하려면 요소 간 관계를 보호한다. **이 모든 의도는 선으로 표시된다.** 아키텍처 설계에서는 선을 최우선으로 챙기자.
:::

## 2. 설계 테스트하기

"일찍 테스트하고 자주 테스트하라"는 지침은 아키텍처에도 적용된다. 아키텍처 평가에 필요한 세 가지는 다음과 같다.

1. **산출물(artifact)**: 손에 잡힐 듯 표현된 설계 결과물
2. **루브릭(rubric)**: 이해관계자 관점에서 좋고 나쁨을 판별할 척도
3. **통찰력(insight)**: 리뷰어가 아키텍처의 우수성을 정리할 수 있도록 지원하는 계획

### 2.1 산출물 마련하기

평가 대상은 아이디어가 아니라 실존하는 무언가여야 한다.

- 작성한 코드 조각
- 화이트보드 스케치, 다이어그램 툴로 그린 모델
- 아키텍처를 여러 뷰로 그린 프레젠테이션 슬라이드
- 발표·화이트보드 실험 결과 요약
- 전통적 SAD, 아키텍처 하이쿠
- 어떤 컴포넌트가 어떤 품질 속성에 영향받는지 표현한 트리

특정 품질 속성에 초점을 맞추려면 산출물에 해당 뷰를 포함하자. 아키텍처가 미숙해 일반적 피드백을 원한다면 유동적임을 알리는 스케치를 쓰고, 위험·비용이 높다면 격식 있게 정교하게 구성하자.

### 2.2 디자인 루브릭 만들기

**디자인 루브릭(*design rubric - 리뷰어가 아키텍처 적합성을 판단할 때 사용할 기준과 척도*)**은 두 부분으로 구성된다.

- **기준(criteria)**: 설계 산출물을 평가하는 데 쓸 특성
- **등급(rating)**: 특성을 해석하는 데 쓸 척도

**ASR을 그대로 기준으로 삼기 좋다.** 라이언하트 프로젝트의 예:

| 품질 속성 시나리오 | 등급 (1~4) |
|---|---|
| 가용성: 색인이 동작 안 해도 라이언하트는 응답해야 한다 | — |
| 가용성: 유지보수 시간에도 항상 응답할 수 있어야 한다 | — |
| 성능: 평균 부하에서 5초 이내 응답 | — |
| 확장성: 향후 7년간 데이터 연평균 5% 증가 가정하에 확장 가능 | — |

등급 척도:

- **1점**: 시나리오를 충족하지 못하거나 평가할 수 없음
- **2점**: 부분적으로 충족하지만 위험·기술 부채·비용이 높음
- **3점**: 충족하며 감당할 만한 위험·부채·비용
- **4점**: 충족하며 위험·부채·비용이 거의 없음

**좋은 루브릭의 조건**:

| 조건 | 내용 |
|---|---|
| 중요하고 필수적 | ASR 관점에서 아키텍처의 좋은 점을 정의. 사소한 세부사항 지양 |
| 분명한 구분 | 하나의 기준이 다른 기준과 겹치지 않게. 이상적으로 독립 채점 가능 |
| 관찰·측정 가능 | 리뷰어가 평가·채점할 수 있어야 함. 산출물이 기준을 가시화 |
| 정교하고 명확 | 어떤 리뷰어라도 기준을 동일하게 이해 |

**등급 척도 선택**:

| 척도 개수 | 예시 | 언제 사용 |
|---|---|---|
| 2가지 | 예/아니오, 충족/불충족 | 모두 통과하거나 전혀 통과 못 하는 조건. 소수 리뷰어 대상 |
| 3가지 | 실패-통과-좋음, 낮음-허용-높음 | 최솟값 역치 또는 기대 수준 정의 시. 여러 리뷰어 |
| 4가지 | 실패-괜찮음-통과-초과 | 자세한 피드백. 정성적·복합적 요인 관여 |
| 5가지 이상 | 1~10 | 가급적 사용 안 함. 척도가 너무 많으면 일관되지 않은 결과 |

여러 리뷰어가 매긴 점수의 **평균이 허용치 안이더라도 1점이 있으면 표시해뒀다가 추가 논의**를 진행한다.

### 2.3 통찰력 만들어내기

루브릭이 답변 가능한 질문이라면, 통찰력을 만드는 활동은 답변을 이끌어내는 방법이다.

| 루브릭 기준 | 통찰력을 만드는 활동 |
|---|---|
| 리스크의 규모 | 리스크 스토밍(활동 35), 위험 유도 워크숍으로 개수·심각도 조사 |
| 불확실성 | 질문-코멘트-우려사항(활동 34)으로 질문 개수·난이도 평가 |
| 리뷰어 협의 | 투표, 설문, 추천/비추천, 등급 매기기 |
| 설계 완결성 | 컴포넌트와 그 현재 설계 상태 나열 |
| 문제와 정합성 | 시나리오 훑어보기(활동 37) — 민감도·문제 영역·위험·질문 파악 |
| 기술 부채 | 현재 아키텍처로 구현 불가능한 가치 있는 유스케이스 나열 |
| 품질 | 컴포넌트별 결정 수 카운트, 고품질·저품질 임곗값 정의 |

> **직접 해보기 (7개 질문)**: 정답을 모르는 최근 프로젝트 아키텍처에 대한 질문을 **7개 이상** 적어보자. 뻔한 질문을 넘어 흥미로운 질문을 찾으려면 많이 끄집어내야 한다.
>
> - 구체적으로 말하기 — 일반적 질문에서는 일반적 통찰력만 얻는다
> - 아키텍처의 **관계(선)**에 대해 무엇을 알고 있는가(모르는가)?
> - 모듈·컴포넌트-커넥터·할당 구조 뷰가 하나 이상 있는가?
> - 걱정하는 게 있는가? "만약에..." 사고는 지루하지만 실제 엔지니어링 문제는 여기서 자란다

## 3. 평가 워크숍 꾸리기

평가 워크숍은 다음 5단계를 따른다.

1. **준비**: 산출물 확보·제작, 루브릭 정의, 데이터 수집·리뷰어 초대 방법 결정
2. **리뷰어 준비**: 산출물·루브릭 소개, 평가 목표 설명, 질문 응답
3. **평가**: 리뷰어를 이끌어 산출물 탐색·루브릭 채점 활동 진행
4. **분석**: 리뷰어 제출 데이터 분석, 결과 요약·성향 조사
5. **팔로업**: 배운 점을 토대로 다음 단계 결정

### 3.1 준비

평가 목표에 따라 필요한 산출물이 달라진다.

| 필요한 평가 | 필요 산출물 |
|---|---|
| 특정 품질 속성 달성도 | 테스트 결과, 유스케이스, 품질 속성 시나리오, 관련 뷰 |
| 기술·패턴 | 기술 설명서, 실험 개요, 실험 결과, 품질 속성 시나리오 |
| 비용·일정 달성 가능성 | 컴포넌트 개요·측정치, 기술 의존성, 팀 역량 설명서 |
| 설계 변경 | 현재·미래 설계 개요, 작업 단계 표 |
| 아키텍처 설명의 완결성 | 아키텍처 명세서, 질문/답변 목록, 품질 체크리스트 |
| 보안 | 남용·오용·위협 모델, 데이터 저장소, 공격 가능 부분 |
| 릴리스 준비 | 품질 속성 시나리오, 관련 뷰, 릴리스 체크리스트, 테스트 결과 |

리뷰어는 이해관계자 외에 **비이해관계자**도 포함하며 디테일에 강하고 시스템에 관심 있는 전문가를 선정한다. 2명으로도 가능하지만 수십 명이 참여하기도 한다.

### 3.2 리뷰어 준비

리뷰어에게 시스템 맥락·ASR·검토 산출물 등 배경 지식을 미리 알려주자. 팀 내부 평가라면 워크숍 시기에 맞춰 함께 산출물을 만들고 검토도 함께 하자.

### 3.3 평가 진행 — 통찰력을 만드는 활동

- **시나리오 훑어보기(활동 37)**: 대부분 평가에서 가장 기본적이고 신뢰할 수 있는 방법
- **질문-코멘트-우려사항(활동 34)**: 시각적 브레인스토밍. 아키텍처의 사실과 의문점 신속 파악
- **리스크 스토밍(활동 35)**: 시각적 브레인스토밍. 특정 관점의 위험 집중
- **스케치하고 비교하기(활동 38)**: 대안과의 비교
- **ADR 재검토**: 결정을 원점에서 다시 고려해 여전히 유효한지 판단

3~4명 소규모 팀에 60분 질문-코멘트-우려사항 활동만 해도 흥미로운 통찰력을 얻을 수 있다.

### 3.4 분석

리뷰어의 다양한 관점이 데이터가 된다. 데이터를 공유하고 앞으로 어떤 모습이 그려질지 묻자. **이미 답을 아는 질문이라도 누군가가 질문했다는 건 커뮤니케이션 개선의 여지가 있다는 의미다.**

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 평가 결론은 단순 "통과/실패"가 아니어야 한다. **왜 목적에 부합하는지, 왜 부합하지 않는지 이해하는 것이 중요하다.** 훌륭한 설계는 항상 개선하는 설계이며, 부실한 설계라도 몇 가지는 올바르게 개선할 수 있다.
:::

### 3.5 팔로업

평가에서 확인한 문제·위험·질문을 모두 해결할 필요는 없다. 우선순위를 정하고 항목마다 작업할 사람을 지정한다. 소규모 워크숍은 이메일 목록으로도 충분하지만, 대규모라면 원본 노트 링크가 포함된 요약문이 좋다.

### 3.6 ATAM — 절대적 표준

**ATAM(*Architecture Tradeoff Analysis Method - SEI에서 개발한 아키텍처 평가의 대표 방법론*)**은 지금까지 나온 아키텍처 평가 방법 중 **최초이자 가장 포괄적으로 정의된 방법 중 하나**다. 교육받은 진행요원과 함께 수일~수주간 진행하는 워크숍이다.

미사일 유도 시스템, 자율주행차 시스템 같은 고도로 복잡한 시스템 설계에 매우 적절하다. 그 외 경우에는 축소된 과정을 따른다. ATAM은 엄격하고 잘 정의되어 있으니, 평가 경험이 없다면 참고하기 좋다.

## 4. 빠르게·자주·지속해서 평가하기

만일 아키텍처 평가를 안 했다면 그것이 첫 번째 실수, 했더라도 너무 늦게 시작했다면 두 번째 실수다. **개발 루틴에 평가 과정을 정기적으로 넣는 것이 가장 좋다.**

### 4.1 평가 피라미드

**테스트 피라미드**의 전제와 유사하다. 대부분 아키텍처 평가는 빠르고 저렴해야 하며, 빠른 검사가 놓치는 부분은 심층 평가로 잡는다.

| 평가 유형 | 빈도 | 대상 | 예시 |
|---|---|---|---|
| 심층 평가 | 1~3회 | 전체 시스템·ASR 간 연계 | ATAM |
| 표적 평가 | 수십 회 | 단일 결정·컴포넌트·ASR | 리스크 스토밍, 질문-코멘트-우려사항, 아키텍처 브리핑 |
| 빠른 검사 | 셀 수 없이 자주 | 설계 단위 결정 시 | 코드 리뷰, 스토리텔링, 화이트보드 토론, 온전성 검사 |

심층 평가는 시스템 수명의 주요 마일스톤, 표적 평가는 2~4주마다, 빠른 검사는 매일(혹은 매일 여러 번) 개발 워크플로에 내재화한다.

### 4.2 아키텍처 이슈의 무지개

편식하지 말라고 아이에게 "무지개를 먹어라"라고 말하듯, 건강한 아키텍처를 위해 다양한 종류의 이슈를 찾아야 한다.

| 색 | 이슈 유형 | 설명 |
|---|---|---|
| 빨간색 | **위험(Risk)** | 나쁜 일이 일어날 수도 있는 무언가. 조건과 결과로 구성. 완화하거나 수용 |
| 주황색 | **무지(Ignorance)** | ASR 충족 여부를 판단할 정보가 없음. "무지"를 "미지"로 바꾸는 것이 목표 (모른다는 사실을 아는 것) |
| 노란색 | **문제(Problem)** | 이미 겪은 나쁜 일. 결정이 원하는 방식으로 동작하지 않음. 수정하거나 받아들임 |
| 초록색 | **이해 격차(Understanding gap)** | 이해관계자가 생각한 설계와 실제가 다를 때. 교육으로 해결 |
| 파란색 | **아키텍처 침식(Erosion)** | 설계와 결과물 사이 미세한 차이가 쌓임. "표류/부패"라고도 함. 정기적으로 제거 |
| 보라색 | **개념 표류(Concept drift)** | 결정 이후 비즈니스·맥락이 바뀜. 비즈니스 목표·ASR·이해관계자 인식을 지속 점검 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 새 아키텍처에서 흔한 실수는 **항상 같은 종류의 이슈만 찾는 것**이다. 평가를 더 높은 수준으로 끌어올리는 방법은 **이전에 묻지 않았던 질문을 하는 것**이다. 이슈는 너무 다양해서 놀랄 뿐 몰라서 놀라는 게 아니다.
:::

### 4.3 격식 있는 방법과 격식 없는 방법

격식 있는 방법은 반복하기 쉽고 결과가 일관되지만 비용이 높다. 격식 없는 방법은 빠르고 저렴하지만 범위가 좁고 결과가 일관되지 않을 수 있다.

**팀이 아키텍처 평가에 익숙하지 않다면 ATAM 같은 방법으로 시작하면 팀원들이 불쾌해할 수 있다.** 격식 없는 방법으로 시작하자.

> **격식 없는 평가 예시**: 화이트보드 토론 후 그룹을 해산하기 전에 이렇게 말한다. "시작은 좋아 보입니다. 우리가 '품질 속성 A'를 끌어올리는 데 어떤 이슈가 있을까요?" 화이트보드 옆에 팀원 발언을 적고, 결과를 요약하고 다음 단계를 결정하도록 돕는다. **10분 이내에 즉석 아키텍처 평가 완료.**

팀이 격식 없는 방식에 익숙해지면 전략적으로 표적 평가를 도입하고, 궁극적으로는 심층 평가를 도입하자.

## 5. 라이언하트 프로젝트

프로젝트 시작 2개월 후 백로그를 채우고 개발 사이클을 여러 번 돌았다. 지속적 통합이 진행 중이며, 첫 번째 내부 릴리스도 만들었다. 지금까지의 설계를 재검토하기 좋은 시기다.

다음 개발 사이클에 **질문-코멘트-우려사항 워크숍**을 진행하기로 일정을 잡았다. 상위 5개 품질 속성 시나리오를 검토하며 시작한다. 팀원들이 관련 뷰를 몇 개 그리고, 각 시나리오를 살펴보며 다이어그램에 포스트잇으로 질문·의견·우려사항을 붙인다. 50분 후 온갖 문제점 목록이 나온다.

워크숍 종료 10분 전, 주요 이슈를 해결할 실행 계획을 수립한다. 설계와 최대한 일치하도록 **코드 리팩터링**, 무중단 배포를 위한 **배포 시스템 실험** 등이 포함된다.

다음 평가는 두 번의 스프린트 후로 예약했고, 매일 설계 이슈를 논의하며 이 훈련을 계속하기로 다짐했다.

## 요약

- **아키텍처는 한 번에 평가할 수 없다** — 절대 전적으로 좋거나 나쁘지 않다
- 평가에 필요한 3가지: **산출물, 루브릭, 통찰력**
- **디자인 루브릭 = 기준(criteria) + 등급(rating)**. ASR을 기준으로 삼기 좋다
- 좋은 루브릭 조건: 중요·필수적, 분명한 구분, 관찰·측정 가능, 정교·명확
- **선(관계)이 아키텍처 의사결정의 중요한 맥락을 전달**한다 — 네모보다 선을 챙기자
- 평가 워크숍 5단계: 준비 → 리뷰어 준비 → 평가 → 분석 → 팔로업
- **ATAM**은 아키텍처 평가의 절대적 표준. 고복잡도 시스템에 적합
- **평가 피라미드**: 심층(1~3회) / 표적(수십 회) / 빠른 검사(셀 수 없이 자주)
- **아키텍처 이슈의 무지개**: 위험·무지·문제·이해 격차·아키텍처 침식·개념 표류
- **격식 없는 방법부터 시작**해 점차 표적·심층 평가로 확장. 팀이 아키텍처 사고력을 갖추도록
- 평가 결론은 **"통과/실패"가 아니라 "왜 부합하는가·왜 부합하지 않는가"**

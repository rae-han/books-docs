# Chapter 11: Documenting Architecture (아키텍처 문서화하기)

## 핵심 질문

왜 소프트웨어 아키텍처 명세서(SAD)는 아무도 읽지 않고 무시받는가? 어떻게 하면 팀이 사랑하는 문서가 될 수 있는가? 원시부족적·공동체적·형식적 서술 방법은 각각 언제 쓰는가? ADR(아키텍처 의사결정 기록)과 아키텍처 하이쿠는 어떤 역할을 하는가? 이해관계자의 관점에 맞게 뷰를 어떻게 구성하고, 설계 결정의 논리적 근거는 왜 반드시 남겨야 하는가?

---

## 1. 왜 아키텍처를 문서화해야 하는가

소프트웨어 아키텍처 명세서(*SAD - Software Architecture Description, 아키텍처의 뷰·설계 결정·논리적 근거를 담은 종합 문서*)는 대체로 좋은 평을 받지 못한다. 코딩보다 한참 전에 작성되고, 독자적인 파일 형식이라 수정하기도 어렵다. 결과적으로 아무도 읽지 않는다. 그래서 "SAD"는 "아무도 읽지 않아 슬프다"라는 농담으로도 통한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 사람들이 알고 싶어 하는 정보를 정확하게 알려주면서 **인간미 있고 읽기 쉬운 형식**이라면 아키텍처 문서도 충분히 "사랑스러울" 수 있다. 좋은 문서는 선명한 비전을 보여주고 업무 계획의 근간이 된다.
:::

문서화의 가치는 다음과 같다.

### 1.1 조직 구성
소프트웨어 개발은 기술만큼 사람을 다루는 일이다. 명세서는 이 모두를 아울러 보여준다. 사람들은 시스템이 어떻게 동작하는지 알게 되면 그에 맞춰 움직인다.

### 1.2 기술·비즈니스 이해관계자의 어휘 통일
모델이 설계에 쓸 용어를 정의한다면 명세서는 그 모델을 이해관계자가 이해할 수 있는 개념으로 해석해준다. **이것이 아키텍처 명세서의 가장 중요한 존재 이유다.**

### 1.3 품질 속성 강조
품질 속성은 관심을 안 가지면 그만큼 멀어진다. 문서에서 강조해야만 모두가 품질 속성을 중요하게 생각한다.

### 1.4 생각의 명료화
아무것도 기록하지 않은 상태에선 모든 요소가 다 명확하다고 믿게 된다. **하지만 펜을 들고 종이에 적는 순간 얼마나 생각이 모호했는지 깨닫는다.** 아이디어를 더 단단하게 만들고 무엇을 아는지·모르는지 파악할 수 있다.

### 1.5 평가 대상 파악
평가 전까지 모든 설계를 코드로 구현하며 마냥 기다릴 수는 없다. 명세서로도 설계를 분석할 수 있고, 코드로 옮기기 전에 수정할 시간적 여유를 준다. **3시간 동안 모호한 아이디어를 교정하는 게 한 달 동안 구현만 하는 시간보다 의미 있다.**

### 1.6 보여주기
소프트웨어 아키텍처는 그 자체로도 멋지다. 좋은 명세서는 프로젝트 고객·윗사람들에게 목적·계획·비전을 선명하게 보여주며 확신을 준다.

## 2. 상황에 맞는 서술 방법

아키텍처 문서화에 절대적인 방법은 없다. 두 가지 질문을 던져보자.

- 설계가 변경될 가능성은 얼마나 되는가?
- 결정된 설계는 어디까지 공유해야 하는가?

이 답에 따라 서술 방법은 **원시부족적·공동체적·형식적·시간소모적** 네 유형으로 나뉜다.

| 서술 방법 | 공유 용이성 | 변경 용이성 | 언제 사용 |
|---|---|---|---|
| 원시부족적(tribal) | 낮음 | 높음 | 이 방법부터 시작 |
| 공동체적(community) | 높음 | 높음 | 원시부족에서 발전 |
| 형식적(formal) | 높음 | 낮음 | 필요할 때만 |
| 시간소모적(costly) | 낮음 | 낮음 | 이런 건 하지 말자 |

### 2.1 원시부족적 서술 방법

**원시부족적 서술(*tribal - 구전과 유물에 의존하는 방법*)**은 스토리텔링·은유·비공식 스케치가 대표적이다. 언제나 이 방식으로 시작한다. 진도가 빠른 초기 아키텍처 설계에 적합하다.

단점은 공유가 어렵다는 것. "활동 29 시스템 메타포"에서 말하는 시스템 메타포는 팀 전체가 대화에 귀 기울일 때만 문제없지만, 구전의 특성상 누군가 말할 때만 이야기의 생명력이 있다. 6명 정도의 소규모 팀도 끊임없이 이야기하면 지친다.

### 2.2 공동체적 서술 방법

같은 이야기를 여러 명에게 반복해서 말한다면 이제 서술 방법의 범위를 넓힐 때다. **공동체적 서술(*community - 공동체 안에서 공유되는 방법*)**은 원시부족 방식보다 공유가 쉽다.

예시:
- **아키텍처 하이쿠(*architecture haiku - 아키텍처를 간결하게 요약한 한 페이지 문서*)**
- **아키텍처적으로 명백한 코딩 스타일** (8.3절)
- **아키텍처 의사결정 기록(ADR)**

대부분 팀은 아키텍처가 성숙해지면서 자연스럽게 원시부족적에서 공동체적 서술로 전환한다.

### 2.3 형식적 서술 방법

**형식적 서술(*formal - 전통적 SAD, 형식 모델을 포함하는 방식*)**은 규모가 크고 노력이 많이 든다. 고위험군 시스템이나 산업 특성상 공식 문서가 필요할 때 적합하다. 이 경우에도 원시부족적 → 공동체적 → 형식적 순으로 진전시키는 편이 수월하다.

전통적 SAD는 보통 10~20쪽 정도이며, 다음 요소를 필수적으로 담는다.

| 섹션 | 내용 |
|---|---|
| 기본 정보와 서문 | 표지, 변경사항, 서명, 목차, 그림 목록, 라이선스, 법적 고지 |
| SAD 개괄과 안내 | 문서 목적, 어떤 방법론으로 구성했는지 설명 |
| 이해관계자·비즈니스 목표·ASR 요약 | 이해관계자 목록, 주요 제약사항, 품질 속성 (ASR 워크북 참조 링크) |
| 맥락 | 소프트웨어 시스템이 어떤 환경에 적합한지 |
| 의미 있는 뷰 | 품질 속성·요구사항에 맞게 여러 뷰를 이해관계자별로 묶어 배치 |
| 위험 요소·질문·향후 작업 | 알려진 문제와 미해결 질문 기록 |
| 부록 | 용어집, 축약어 정의, 품질 속성 분류표 등 |

> **팁**: 팀이 협업해 SAD를 작성하되, 한 명을 **SAD 마스터**로 지정하자. 템플릿을 만들고 섹션 담당자를 정하며 문서 전체의 일관성과 완성 책임을 진다. 추천 템플릿은 SEI의 "Views and Beyond"와 ISO/IEC/IEEE 42010 표준이다.

### 2.4 시간소모적 서술 방법 (피할 것)

수정도 어렵고 공유도 어려운 문서가 있다. 대표적으로 슬라이드로 만든 아키텍처 명세서다. **슬라이드는 발표자가 생략된 내용을 말할 때만 효과적**이며, 많은 애니메이션·상자·화살표가 겹겹이 쌓여 있으면 너무 아름다워서 고치기 주저하게 된다. 결국 현실은 변해도 명세서는 안 바뀐다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 훌륭한 명세서는 다음 4가지 특징을 지닌다.
1. 명세서를 읽을 **사람을 고려해** 작성한다
2. 아키텍처의 **다양한 뷰**를 보여준다
3. 각 요소와 그 **역할**을 분명히 정의한다
4. 설계 의사결정마다 **논리적 근거**를 설명한다
:::

## 3. 명세서의 독자 고려하기

아키텍처 명세서의 독자는 여러분을 이해하고 싶어 한다. 공유할 대상이 누구인지 알면 그들에게 정확히 필요한 정보를 제공할 수 있다.

### 3.1 여러 단계로 설명하기

> — 조지 페어뱅크스, 구글 소프트웨어 엔지니어, 『Just Enough Software Architecture』 저자

갑자기 프로젝트에 투입되어 처음 보는 코드를 이해하려 애쓴 적이 있을 것이다. 오픈소스 리포지터리에 폴더 하나에 수백 개 소스가 있으면 유추가 어렵다.

코드도 **여러 단계(레이어)로 이야기를 구성**해야 한다. 저녁 식사를 예로 들어보자. 저녁 → 코스 요리 → 각 코스의 요리 → 요리 재료. 확대·축소가 가능한 지도처럼. **저녁 시간이 얼마나 걸릴지 궁금하면 코스 개수를, 알레르기가 궁금하면 재료를 보면 된다.**

### 3.2 공감 지도 만들기

이해관계자를 떠올리고 그들이 무엇을 소중히 여기는지 생각하자. 정보를 어떻게 처리하고, 어떻게 사용하는가? **공감 지도(*empathy map - 특정 이해관계자가 무엇을 말하고, 생각하고, 만들고, 실행하는지 정리한 도구*)**를 만들면 도움이 된다.

예를 들어 개발자용 공감 지도에서 "딱히 부정하진 않지만 좀 더 토론이 필요합니다"라는 발언은 이 개발자가 **명확한 근거를 가진 문서**를 필요로 한다는 판단으로 이어진다.

## 4. 이해도가 중요하다

아키텍처 명세서에서 커뮤니케이션이 제일 중요하다.

| 이렇게 하세요 | 이렇게 하지 마세요 |
|---|---|
| 아키텍처의 핵심 개념을 처음 정의한 대로 사용 | 새로운 개념을 굳이 소개 |
| 해당 도메인의 언어로 말하기 | 다이어그램의 표기법을 모두가 안다고 단정 |
| 다이어그램에 범례 넣기 | 전문용어 사용 |
| 공통 템플릿이 있다면 우선 사용 | — |

예를 들어 이해관계자가 자재를 "마스터 넘버"라 부르면 굳이 "상품 ID"라고 하지 말고 그대로 따르자. 이해관계자가 "비기능적 요구사항"이라고 말하면 "품질 속성"으로 고치지 말고 그들의 용어를 쓰자.

**표기법도 이해도의 문제다.** UML에 통달했더라도 독자는 UML을 한 번도 공부한 적 없을 수 있다. **모든 다이어그램에 범례를 넣어 표기법과 메타모델을 정의**해야 한다.

## 5. 이해관계자의 관심사에 맞추어 뷰 구성하기

이해관계자마다 소프트웨어를 보는 관점이 다르다. 개발자는 코드 구성·배포·모듈 관계를, 테스터는 인터페이스·프로토콜을, 프로덕트 매니저는 진행 상황·기술 의존성을 알고 싶어 한다.

### 5.1 관점(viewpoint) 정의하기

**관점(*viewpoint - 어떤 뷰를 누구에게 어떤 표기법·용어로 보여줄지 정한 규격. ISO/IEC/IEEE 42010:2011 표준의 일부*)**은 어떤 뷰를 누구에게 보여줄지, 표기법·단어 사용법까지 정의한다.

라이언하트 프로젝트의 **배포 관점** 예시:

| 컴포넌트 | 배포 위치 |
|---|---|
| 웹 UI | 사용자 브라우저 실행. 디스플레이 비즈니스에서 제공 (로드 밸런서로 접근) |
| 디스플레이 비즈니스 | 리눅스 VM 위 Tomcat. 클라우드 호스팅 |
| 즐겨찾기 서비스 | 리눅스 VM 위 Tomcat, 클라우드 호스팅. 디스플레이 비즈니스와 독립 배포 |
| 검색 서비스 | 즐겨찾기 서비스의 VM에서 함께 동작 |
| 검색 색인 | 클라우드 서비스가 제공하는 Solr 사용 |
| 알림 프로세스 | 클라우드 스케줄러가 시작. 독립 컨테이너에서 동작 |
| 크롤러 프로세스 | 시 IT 부서가 직접 관리하는 서버에서 실행 |
| 연락처·사용자 메타데이터 DB | 클라우드가 제공하는 Postgres 사용 |

여기에 요소 간 의존성·플랫폼 의존성·서드파티 요구사항·위험 요소·비용·네트워크 위상 관계 등을 덧붙일 수 있다.

### 5.2 참고할 만한 관점 프레임워크

- **뷰와 그 너머(*Views and Beyond - SEI에서 제안한 관점 프레임워크. 모듈·컴포넌트-커넥터·할당 3가지 스타일*)**
- 필립 크루첸의 **4+1 구조적 뷰 모델**
- 로잔스키·우즈의 **시점과 관점(Viewpoint and Perspective)**
- 사이먼 브라운의 **C4 모델**

품질 속성 기준으로 관점을 구성하기도 한다.

- **확장성·보안·유지보수성 관점**: 아키텍처가 특정 품질 속성 시나리오를 충족하는지 보여준다
- **규제 관점**: 특정 이해관계자 그룹이 고려하는 규제사항을 검토한다
- **교육 용이성 관점**: 새 팀원이 하루 만에 아키텍처와 개발 환경을 익히고 커밋할 수 있는지
- **비즈니스 영향 관점**: 아키텍처의 여러 부분이 비즈니스에 어떻게 가치를 주는지 서술

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 관점은 전통적 명세서에서는 필수적이다. 원시부족적·공동체적 방법에서는 **가볍게 유지**하는 편이 낫다. 예를 들어 코드에 ADR이 몇 개 보인다면 관점 페이지 하나를 만들고 의사결정 기록과 코드 링크만 남겨도 충분하다. 아키텍처 명세서는 장황할 필요가 없다.
:::

## 6. 결정에 대한 논리적 근거 설명하기

모든 결정에는 대안이 있고 장단점이 있다. 세부 설계자가 결정의 근거를 잘 이해할수록 설계 의도를 더 잘 반영한다. **다른 사람이 여러분의 의도를 잘 이해할수록, 시스템이 발전해도 아키텍처 무결성이 유지될 가능성이 높다.**

### 6.1 가지 않은 길 (거절한 대안) 기록하기

하나의 결정을 이해시키는 좋은 방법 하나는 **거부했던 모든 선택지를 열거**하는 것이다. 라이언하트 프로젝트에서 선택하지 않은 방법들:

| 가지 않은 길 | 논의 사항 |
|---|---|
| 하나의 커다란 웹 애플리케이션 | 부하가 몰릴 때 적극적으로 스케줄링할 수 없다 |
| 디스플레이 비즈니스에서 Java 채택 | 팀원이 Node.js를 아는 만큼 유지보수성을 높이려 클라이언트·서버 모두 JavaScript 채택 |
| SQL DB에 RFP 색인 생성, 검색 서비스가 DB 직독 | 이해관계자가 요구하는 복잡한 쿼리 작성 불가 |
| MongoDB 사용 | 좋아 보이지만 팀에 전문가 없음 |
| 서비스마다 컨테이너 분리 | 컨테이너 운영 기술에 미숙. 일단 모든 서비스를 같은 VM에 올릴 예정. 추후 배포 전략에 따라 변경 가능 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 이런 지식이 없는 건 영화를 마지막 5분만 봤을 때와 같다. 등장인물의 전후 사정을 모른 채로는 행동에 아무 맥락이 없어 보인다. 배경 지식이 있으면 〈스타워즈 6 - 제다이의 귀환〉에서 다스 베이더의 결정도 깊은 의미로 다가온다. **거절된 아이디어가 유지된 아이디어보다 언제나 더 많다.** 팀에서 논의한 모든 사항을 공식적으로 기록하자.
:::

### 6.2 ADR (아키텍처 의사결정 기록)

**ADR(*Architecture Decision Record - 하나의 결정, 그 맥락, 대안, 결과를 짧게 정리한 문서*)**은 결정을 내릴 때마다 하나씩 남긴다. 결정을 내렸다면 그 결정에 동원한 내용을 아키텍처 하이쿠로 모으고, 코드가 모델을 온전히 반영하도록 지속적으로 리팩터링한다.

## 7. 라이언하트 프로젝트

한 공간에 함께 있을 수 있는 작은 팀이기에 화이트보드 스케치와 시스템 메타포를 계속 사용하기로 했다. 다만 설계할 때 결정한 여러 사항은 **ADR로 기록**하도록 권했다. 메타포와 화이트보드는 빠르고 손쉬운 방법이며, ADR은 다음 개발자에게 알려주기에도 좋다.

공식 아키텍처 명세서 작성은 일단 미루기로 했다. **공식 문서는 프로젝트 후반에 써야 편하다.** 그때쯤 대부분 설계가 코드로 고정되므로 내용이 비교적 적게 바뀌기 때문이다. 첫 번째 독자는 다음에 업무를 이관받을 팀이 될 것이다. 학습 용이성·배포·확장성 관점이 중요하겠다는 판단이 든다.

## 8. 직접 해보기

::: callout {icon="📝" color="gray_bg"}
**직접 해보기**: 최근에 진행한 프로젝트를 생각해보자. 팀과 협의했던 아키텍처 외에 대안으로 어떤 아키텍처가 있었는지, 그리고 왜 그 대안을 거절했는지 적어보자.

- 의사결정 과정에서 팀원들과 심도 있게 회의했는가?
- 팀에서 결정을 내리기 어려워했던 사항은 무엇인가?
- 여건이 불확실한 상황에서 어떤 결정이 내려졌는가?
- 돌이킬 수 없는 이유로 특정 방법을 선택할 수밖에 없지는 않았는가?
:::

## 요약

- 명세서(SAD)는 조직 구성·어휘 통일·품질 속성 강조·생각 명료화·평가·자랑을 위해 필요하다
- 서술 방법 4유형: **원시부족적**(스토리텔링) → **공동체적**(ADR·하이쿠) → **형식적**(전통 SAD) / **시간소모적**은 피할 것
- 훌륭한 명세서의 4가지 특징: **독자 고려**, **다양한 뷰**, **역할 명시**, **논리적 근거**
- **여러 단계로 이야기 구성**(저녁 → 코스 → 요리 → 재료)이 명료한 사고를 낳는다
- **공감 지도**로 이해관계자의 니즈를 파악하고 언어·표기법을 맞춘다
- **모든 다이어그램에 범례**를 넣어 표기법·메타모델을 정의한다
- **관점(viewpoint)**으로 뷰를 조직화. Views and Beyond, 4+1, C4 등의 프레임워크 활용
- **가지 않은 길**과 **ADR**로 결정의 논리적 근거를 남긴다. 거절된 대안이 유지된 결정보다 언제나 많다

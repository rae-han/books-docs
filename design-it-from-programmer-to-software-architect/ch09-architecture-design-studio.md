# Chapter 9: Architecture Design Studio (아키텍처 디자인 스튜디오 운영하기)

## 핵심 질문

아키텍처 디자인 스튜디오란 무엇이며 왜 필요한가? 워크숍을 어떻게 계획하고 진행해야 참가자들의 아이디어와 경험을 효과적으로 모을 수 있는가? 누구를 초대하고, 시간을 어떻게 나누며, 집단사고 같은 함정은 어떻게 피할 수 있는가? 원격으로 진행할 때는 어떤 점이 달라지는가?

---

## 1. 디자인 스튜디오란 무엇인가

19세기 프랑스 건축학과 교수들은 학생들의 프로젝트를 나무 수레로 걷어갔다. 학생들은 수레를 따라다니거나 올라타며 다리·건물을 완성했고, 다음 프로젝트가 수레에 실릴 때까지 나무 조각과 철사를 붙였다. 여럿이 뭉쳐 집중하는 이 모습을 **샤레트(*Charrette - 시간 제한 아래 여럿이 몰입해 진행하는 창의 작업. 프랑스어 "작은 수레"에서 유래*)** 라고 불렀다.

21세기에 와서 샤레트는 "시간을 더 쏟는다고 더 나은 설계가 나오지는 않는다"는 통찰의 기반이 되어, UX 커뮤니티가 이를 **디자인 스튜디오(*Design Studio - 그룹 협업을 장려하고 짧은 시간에 광범위한 아이디어를 도출하는 워크숍*)** 라는 이름으로 대중화했다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 디자인 스튜디오는 **3F 원칙**으로 운영된다 — **Fast(빠르고), Effective(효과적이며), Fun(재미있게).** 재미는 참여를 도모해 탐색 과정을 효과적·빠르게 만든다.
:::

### 1.1 디자인 스튜디오에서 얻는 세 가지 아이디어

| 유형 | 설명 |
|---|---|
| **구체화할 아이디어** | 유망해 보이는 것. 모델·프로토타입으로 세부화 |
| **연구가 더 필요한 아이디어** | 광범위한 가정이나 빠진 정보가 있어 실험·조사 필요 |
| **질문을 여는 아이디어** | 문제 해결 중 새로운 질문만 얻음. **디자인 스튜디오 워크숍의 최고 결과** |

## 2. 디자인 스튜디오 실행 단계

```
[준비] → [시작] → [만들기] ─┐
                    ↑       ↓
                    │    [공유하기]
                    │       ↓
                    └─── [비평하기]
                         (3회 이상 반복)
                            ↓
                         [대응하기]
```

| 단계 | 활동 |
|---|---|
| **1. 준비** | 탐색할 문제 파악·조사 |
| **2. 시작** | 워크숍 목표와 해결할 문제 그룹에 공유 |
| **3. 만들기** | 모델·스케치·프로토타입 제작. 시간 제한 |
| **4. 공유하기** | 작업물을 그룹에 발표. 설계가 어떻게 목표를 달성하는지 설명 |
| **5. 비평하기** | 그룹원의 피드백. 목표 충족 여부에 초점 |
| **6. 반복하기** | 3~5단계 반복. **적어도 3회** |
| **7. 대응하기** | 가장 적절한 아이디어·위험·질문으로 다음 단계 결정 |

### 2.1 워크숍 준비

워크숍을 시작하기 전에 목표를 정하고 누가 참여할지 결정해야 한다. **준비는 생각보다 오래 걸린다.** 이해관계자와 대화하고, 연구하고, 비즈니스 목표를 정의하고, 품질 속성을 포함한 아키텍처 핵심 요구사항(ASR)을 세부적으로 구분해야 한다.

**워크숍 목표 예시**.

| 탐구할 목표 | 예시 문장 |
|---|---|
| 품질 속성 | "확장성·신뢰성이 가장 중요한 품질 속성입니다. 어떻게 둘 다 끌어올릴 수 있을까요?" |
| 컴포넌트 간 인터페이스 | "API 개발에 REST를 채택했습니다. 이를 위해 시스템에서 어떤 부분을 살펴야 하는지 봅시다" |
| 도메인 모델 | "이해관계자 비즈니스를 파악해보니 공통 추상화할 수 있는 부분이 있어 보입니다" |
| 곤란한 상황 탈출 | "10년 전 설계할 때 상상 못 한 데이터가 쌓이고 있습니다. 극복 방법을 최대한 많이 찾아봅시다" |
| 할당 구조 | "시스템을 구분해 최대한 병렬 개발합시다" |
| 패턴 선택 | "A에서 얻은 데이터를 B로 보낼 때 어떤 방법이 있는지 조사해봅시다. 각각의 장단점을 비교합시다" |
| 폭넓은 아이디어 | "오늘은 최대한 다양한 아이디어를 얻고 싶습니다. 적어도 각자 5개 이상 개진해주세요" |

### 2.2 만들기 - 공유하기 - 비평하기

**만들기 단계**.

- 혼자 또는 소그룹으로 스케치·모델링
- **아날로그 도구(펜, 보드마커)** 를 사용해 아이디어에 집중
- 정교함보다 아이디어 우선
- **소그룹 워크숍은 5~7분**으로 제한. 5분 미만은 아키텍처 다루기에 부족

**공유하기 단계** (피치대라고도 함).

- 소그룹별 **3~5분**씩
- 무엇을 만들었고, 어떻게 목표를 충족하는지 설명
- **핵심만 말하며 모든 내용을 브리핑하지 않기**
- 이 단계에서 참가자들은 듣기만 하고 질문하지 않음

**비평하기 단계**.

| 바람직한 비평 | 피해야 할 비평 |
|---|---|
| 설계 목표와 해결 방식에 집중 | 비평받을 때 방어적 자세 |
| 구체적으로 사실에 근거 | 개인적 취향 |
| 문제 명확화를 위한 질문 | 문제 해결의 주변만 짚기 |
| 위험 요소 지적, 새 문제 예고 | 악랄하게 굶 |
| 설계상 이점 언급 | 단점만 파헤침 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 나쁜 설계 속에도 좋은 아이디어가 숨어 있고, 최고의 설계에도 발전할 구석이 있다. 어떤 설계든 **긍정적 피드백과 건설적 비평**을 함께 받게 해야 한다.
:::

### 2.3 반복하기

- 반복을 통해 아이디어는 **분기**(만들기)와 **융합**(비평)을 반복
- 새 회차마다 그룹 구성원을 **섞어야** 폭넓게 탐색되고 소속감도 커짐
- 홀로 일하는 사람은 짝을 만들어주거나 소규모 그룹으로
- **적어도 3번**의 온전한 반복

### 2.4 워크숍 마무리

- 새 주제를 나누고 소화 시간 부여
- **명확한 실행 아이템** 정하기
- 모든 자료를 사진으로 촬영해 공유 저장소에 업로드
- 실행 아이템을 개인별로 추적 확인

## 3. 적절한 설계 활동 선택하기

### 3.1 라운드로빈 중심 워크숍 예시

| 활동 | 소요 시간 | 목적 |
|---|---|---|
| 전체 맥락·목표 소개 | 15분 | 참가자 모두 능동적 참여 |
| **활동 18. 라운드로빈 설계** | 30분 | 지식을 빠르게 융합·분기 |
| **활동 17. 그룹 포스터** | 30분 | 알아낸 사항을 포스터로 공유. 공감대 형성 |
| 포스터 발표·비평 | 15분 | 포스터 하나당 3분, 스티커 투표 |
| 리뷰·다음 계획 | 10분 | 워크숍의 10% 정도 할당 |

### 3.2 스케치 연습

- 얼마나 다양한 선을 그릴 수 있는가? 화살표는 몇 가지?
- 아키텍처 다이어그램을 최대한 자세하게 그린 뒤, 동일한 아키텍처를 최대한 단순하게 다시 그리기
- 여러 형태의 사람 그려보기
- 종이 한 장을 도형·화살표·사람·낙서로 가득 채우기

## 4. 적절한 참가자 초대하기

### 4.1 규모

효율적 협업에는 **7명 이하** 그룹이 적당하다. 규모가 너무 크면 관리 어렵고 스케줄 조정 거의 불가능하다. 한 명의 진행자(facilitator)는 **서너 개의 하위 그룹**만 관리할 수 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 디자인 스튜디오 크기는 **10명 정도가 적당**하며, 아이디어 도출 작업은 **3~5명 소그룹**이 이상적이다. 짝이 필요하다면 오직 한 사람과 짝을 하는 것이 좋다.
:::

### 4.2 다양성

이상적인 디자인 스튜디오에는 **반대 의견을 제시하거나 다른 관점을 제시하는 사람이 한 명 이상** 있다. 다양성이 "유레카!" 순간의 확률을 높인다.

**초대 순서**.

1. 주요 이해관계자 먼저
2. 특정 문제에 대해 거의 알지 못하지만 다른 관점을 가진 사람
3. 팀 대부분이 개발자라면 → 테스터·프로덕트 매니저
4. 팀원이 전부 백엔드 개발자라면 → 프런트엔드 개발자

### 4.3 집단사고 경계하기

**집단사고(*Groupthink - 그룹이 개성을 잃고 워크숍 목표보다 화합과 공감대를 더 중요시하는 현상*)** 에 빠지면 결정은 차선책에 머물 뿐 아니라 해롭기까지 하다.

| 그룹 상황 | 현재 상태 |
|---|---|
| 의미를 명확히 하는 질문, 정중한 반박, 아이디어 구현 토론 | 협력을 잘하고 있다 |
| 지배적 아이디어에 쏠림, 비평 없음, 생각을 나누지 않음 | 충돌을 꺼리거나 협력에 확신이 없다 |
| 도돌이표처럼 처음 논의로 돌아옴 | 다양한 생각을 나누지 못하고 있다 |
| 한 사람만 말한다 | 주도적 성향이 조용한 다수를 누르고 있다 |

**의견에 반박하지 않는 토론은 긍정적으로 보이지만 사실 대부분 그렇지 않다.** 의견 충돌 지점에서 곧 탐색 영역의 범위가 결정된다.

## 5. 그룹 관리하기

### 5.1 시간 관리

시간이 부족하거나 활동을 서두르면 목표가 퇴색되고 참가자들이 결과를 신뢰하지 않는다.

| 상황 | 권장 워크숍 길이 |
|---|---|
| 한두 가지 목표만 빠른 탐색 | 한두 시간 |
| 탐색할 목표가 많음 | 하루~이틀 |
| 문제 파악 자체가 잘 안 됨 | 몇 주 소규모 세션 반복 |

### 5.2 대원칙(Ground Rules)

워크숍 시작 시 대원칙을 정한다.

| 대원칙 |
|---|
| 모두가 참여한다 |
| 시간이 종료되면 다음 순서로 넘어간다 |
| 옳은 답, 틀린 답은 없다 |
| 도움이 필요하면 질문한다 |
| 시간을 확인한다 |
| 즐겁게 임한다 |

### 5.3 Tell-Show-Tell

새 활동을 소개할 때 항상.

1. **Tell**: 무엇을 할 것인지 말한다
2. **Show**: 예시를 보여준다 (이전 워크숍 결과, 사진, 예제 등)
3. **Tell**: 참가자들에게 지침을 검토하게 한다

대부분의 사람은 새로운 것을 처음 볼 때 중요한 세부사항을 놓친다.

### 5.4 참가자 도우기

"자, 시작합시다!" 하면 대부분 얼어붙는다. 빈 종이만 덩그러니 보고 있는 그룹·참가자를 찾아 시작을 도와줘야 한다.

### 5.5 활동 시간 엄수와 즉석 교육

- 모든 활동에는 제한 시간이 있다. 진행자가 "끝!" 하면 손을 뗀다
- 협업 경험이 없는 참가자에겐 **즉석·적시**로 소프트웨어 아키텍처와 설계 콘셉트를 가르쳐야 한다
- 필요 시 품질 속성 시나리오에 대한 기초 강의 진행

### 5.6 흘려 넘긴 주제 처리

디자인 스튜디오 중 흥미로운 아이디어가 나오지만 시간이 여의치 않을 수 있다. **워크숍 이후에 마저 토론할 수 있도록 관련 주제를 기록해두고, 별도 토론 세션을 만든다.**

## 6. 원격으로 협업하기

한 장소에 사람이 모이기 어려울 때도 있다. 원격 워크숍 진행 팁.

| 팁 | 설명 |
|---|---|
| **온라인 협업 도구 사용** | 화면 공유, 공동 문서, 실시간 캔버스, 브레인스토밍, 그룹/음성 채팅 등 |
| **시간 넉넉히 정하기** | 협업 환경 구성 시 문제 발생 대비 |
| **더 많은 이야기 나누기** | 음성은 한 번에 한 사람만 → 채팅 등 병행 |
| **집중시키기** | 프레젠테이션 자료 배포, 공동 문서 작업 |
| **얼굴 보며 진행** | 화상 채팅 사용 |
| **비동기 진행 허용** | 라운드로빈은 이메일만으로도 가능 |

**원격 워크숍 시나리오 예시**.

진행자 마리는 워크숍 시작 전 원격 회의에 접속해 화면 공유를 시작한다. 모두 들어오자 프레젠테이션으로 목표·안건을 공유하고 화이트보드에 메모한다. 라운드로빈 첫 회차에 참가자들은 아키텍처를 스케치해 휴대폰으로 촬영하고 슬랙에 업로드한다. 두 번째 회차에서 스케치마다 이미지 편집 프로그램으로 의견을 붙이고 공유 폴더에 업로드한다. 이후 소그룹으로 나눠 화상 채팅과 공유 폴더 자료로 발표 자료를 만든다. 발표 후 비평 시간에는 실시간 공유 문서로 함께 메모한다. **모여 진행 시 90분 일정이지만 원격은 2시간 정도** 걸린다.

## 7. 라이언하트 프로젝트 사례

팀은 갈림길에 섰다. 팀원 중 누군가는 서비스 지향 아키텍처(마이크로서비스)를, 다른 팀원은 안전하고 검증된 멀티계층 패턴을 주장했다. 두 방법 모두 잘 동작하겠지만, 이번엔 디자인 스튜디오를 열어 어떤 선택을 하든 팀의 공감대를 얻고자 한다.

**워크숍 진행**.

1. 작은 그룹으로 나눠 아이디어를 냈다
2. 마이크로서비스와 멀티계층뿐만 아니라 다른 패턴도 살펴보며 새 아이디어를 도출
3. 첫 회차 발표·비평 후 그룹 포스터를 만들었다
4. 의외로 마이크로서비스를 주장한 팀원이 다른 아이디어를 탐색하고는 마이크로서비스가 적합하지 않다고 판단
5. **2시간 동안 6가지 디자인 대안을 탐색**해 훌륭한 솔루션을 찾았다

가장 큰 소득은 **최종 결정에 모든 사람이 발언권을 가졌고, 워크숍 전엔 존재하지 않았던 프로젝트에 대한 주인의식이 팀원들에게 만들어졌다**는 점이다.

> **직접 해보기**: 팀에서 결정이 지연되고 있는 아키텍처 이슈 하나를 고르고 90분짜리 디자인 스튜디오를 계획해보자. (1) 워크숍 목표 1~2개 (2) 참가자 5명 이내 (3) 라운드로빈 30분 + 포스터 30분 + 발표·비평 15분 + 리뷰 15분으로 구성. Tell-Show-Tell을 적용하고 대원칙을 명시할 것.

## 요약

- **디자인 스튜디오**는 시간을 엄격히 제한해 그룹의 아이디어와 경험을 효과적으로 모으는 협업 워크숍이다
- **3F 원칙**: Fast, Effective, Fun
- 7단계 실행: **준비 → 시작 → 만들기 → 공유 → 비평 → 반복 → 대응**. 만들기-공유-비평은 **적어도 3회 반복**
- 만들기는 **5~7분**, 공유는 **3~5분** 시간 제한. 아날로그 도구 사용. 정교함보다 아이디어 우선
- 비평은 **설계 목표와 해결 방식에 집중, 사실 근거, 위험 지적, 이점 언급**
- 참가자는 **3~5명 소그룹**이 이상적, 최대 **10명 이하**. 반드시 **다른 관점을 가진 사람**을 초대해 집단사고를 방지
- 집단사고 징후: 지배 의견 쏠림, 비평 없음, 도돌이표, 한 사람만 말함. **의견 충돌 지점에서 탐색 영역이 결정된다**
- 그룹 관리 도구: **대원칙 공유, Tell-Show-Tell, 활동 시간 엄수, 즉석 교육, 흘려 넘긴 주제 별도 세션**
- **원격 워크숍**은 시간 여유 확보, 다양한 소통 채널, 화면 공유·공동 문서 필수. 90분 일정은 2시간 정도로 계획
- 때로는 결정보다 **결정에 이르는 과정**이 더 중요하다. 함께 만든 주인의식이 코드에서 아키텍처까지 스며든다

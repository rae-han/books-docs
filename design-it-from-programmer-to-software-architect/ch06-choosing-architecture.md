# Chapter 6: Choosing Architecture (아키텍처 선택하기)

## 핵심 질문

어떤 소프트웨어 시스템에든 아키텍처는 존재한다. 하지만 아키텍트가 의사결정을 주도하지 않고 운명에 맡기면 어떤 결과가 나올지 알 수 없다. 그렇다면 아키텍처를 선택한다는 것은 정확히 무엇이며, 어떤 절차로 진행해야 하는가? 제약과 품질 속성 사이에서 어떻게 균형을 잡아야 하는가? 결정은 언제, 어떻게 내려야 하는가?

---

## 1. 대안을 위한 분기, 결정을 위한 융합

무언가를 "결정한다"는 말은 곧 여러 대안이 있다는 뜻이다. 옵션이 하나뿐이라면 결정할 필요가 없다. 아키텍처 설계도 마찬가지다. 대안을 만들어보지 않으면 설계 공간(*Design Space - 문제를 해결할 수 있는 가능한 설계 대안들의 집합*)을 제대로 탐색할 수 없다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 설계는 **분기(divergence)와 융합(convergence)의 반복**이다. 문제를 파악하면 여러 갈래로 대안을 뻗어보고, 대안이 어느 정도 나오면 맞지 않는 것을 지우면서 공감대를 만들어간다.
:::

인간의 뇌는 선택을 갈망한다. 여러 대안을 놓고 비교할 때 결정에 대한 신뢰도가 높아진다. 다만 모든 가능성을 다 고려할 시간은 없으므로, 아키텍트는 **품질 속성·구조·이에 영향을 미치는 설계**에 집중해야 한다.

### 1.1 아키텍트가 파악해야 할 것들

그래디 부치(Grady Booch)는 "모든 아키텍처는 설계지만, 모든 설계가 아키텍처는 아니다"라고 말했다. 아키텍트는 통상적인 소프트웨어 시스템 설계에서 다음을 파악해야 한다.

| 파악 대상 | 설명 |
|---|---|
| **구성요소와 역할** | 아키텍처는 요소들의 조합이다. 잘 설계된 아키텍처에서는 각 요소마다 명확한 역할이 있으며, 역할이 없는 요소는 제거되어야 한다 |
| **관계와 인터페이스** | 통신 메커니즘(HTTP, TCP, 공유 메모리)과 통신 규칙(API, 응답 개체 등)이 인터페이스를 정의한다 |
| **도메인** | 문제가 다루는 세계관과 고유 용어. 도메인 개념·개체·이벤트가 아키텍처 어딘가에 반영되어야 한다 |
| **기술과 프레임워크** | 프레임워크·미들웨어·라이브러리는 특정 아키텍처를 미리 가정한다. 독선적인 기술은 아키텍처를 특정 형태로 강요한다 |
| **설치와 배포** | 지속적 배포(*CD - Continuous Deployment*), 병렬 개발, 특정 테스트 전략 등이 필요하면 아키텍처가 이를 지원해야 한다 |
| **과거 설계의 관점** | 하늘 아래 새로운 설계는 없다. 대부분의 아키텍처 탐색은 기존 설계를 살펴보면서 시작한다 |

> 다수의 인스턴스에서 새 버전을 배포할 때 가장 간과하기 쉬운 것 중 하나가 배포다. 레드/블랙 배포와 롤링 업그레이드 어느 쪽을 쓰든 서비스 A→B→C 체인에서 인터페이스가 바뀌면 불일치가 일어날 수 있다. 구버전 호환성 유지, 기능 토글, 무시 정책 등으로 대응하되, 다수 인스턴스를 운용하면 언제든 불일치가 일어날 수 있다고 가정해야 한다.  
> — 렌 배스(Len Bass), 『소프트웨어 아키텍처 이론과 실제』 공저자

## 2. 제약 수용하기

**제약(*Constraint - 설계할 때 이미 결정되어 바꿀 수 없는 사항*)** 은 크게 두 가지다.

- **기술적 제약**: 기술 선택 시 주어짐 (예: ".NET 기반으로 작성해야 한다")
- **비즈니스적 제약**: 사람·프로세스·비용·일정에서 주어짐 (예: "7월 말 박람회 론칭")

기술적 제약은 대체로 받아들이고 아키텍처에 통합할 수밖에 없다. 반면 비즈니스적 제약은 좀 더 유연하다. 예를 들어 "7월 말 론칭"이라는 제약은 다음과 같이 대응할 수 있다.

- 동시 개발에 적합한 패턴 선택
- 증분 배포(*Incremental Deployment - 조금씩 점진적으로 배포*)에 적합한 패턴 선택
- 팀원들에게 익숙한 기술 선택
- 자동화·개발에 적합한 기술 선택
- 계획 없이 개발하고 기술 부채를 감당
- 위의 모든 방법을 조합

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 제약은 아키텍처에 심대한 영향을 미치지만, **아키텍처의 의사결정은 여전히 품질 속성을 올리는 데 초점**을 맞춰야 한다. 초기 설계 결정은 제약이 아니라 "버팀목"이다 — 옮기기 힘들지만 옮길 수는 있다.
:::

> 몇 년 전 제가 일했던 스타트업이 위험을 싫어하는 회사에 인수된 후, 법무팀이 특정 라이선스의 오픈소스를 사용하지 말라는 정책을 내렸습니다. 이 제약 때문에 거의 1년 동안 추가 개발을 해야 했습니다. 이해관계자는 제약이 아키텍처에 어떤 영향을 미칠 수 있는지 잘 이해하지 못하기 때문에, 제약이 너무 잔혹하면 그 영향에 대해 상의해봐야 합니다.

## 3. 품질 속성 끌어올리기

아키텍처 구조를 고르는 일은 스무디용 믹서기를 고르는 것과 비슷하다. 훌륭한 스무디를 만들려면 적당한 믹서기를 골라야 한다.

| 항목 | 일반 믹서기 | 무선 믹서기 | 초강력 믹서기 |
|---|---|---|---|
| 세척 편의성 | 중간 | 좋음 | 중간 |
| 선반 보관 | 좋음 | 나쁨 | 매우 나쁨 |
| 소음 | 중간 | 좋음 | 매우 나쁨 |
| 힘 | 중간 | 나쁨 | 매우 좋음 |
| 휴대성 | 매우 나쁨 | 좋음 | 매우 좋음 |
| 안전성 | 중간 | 중간 | 나쁨 |

믹서기 설계자와 마찬가지로, 아키텍트는 소프트웨어 시스템에서 원하는 품질 속성을 촉진할 수 있는 구조를 선택해야 한다. 구조를 선택하는 가장 일반적인 방법은 **패턴을 탐색하는 것**이다.

### 3.1 예시 — 데이터 기반 웹 애플리케이션의 아키텍처 선택

웹 브라우저용 데이터 기반 애플리케이션을 만든다면 다음 세 가지가 유력한 후보다.

- **옵션 A. 3계층 패턴**: 디스플레이·비즈니스·데이터베이스 계층으로 분리. 테스트·배포·설명이 쉬우나 확장성·가용성엔 취약
- **옵션 B. 발행/구독 패턴**: 이벤트 버스로 컴포넌트가 통신. 유연성이 크지만 메시지 순서 보장이 어려움
- **옵션 C. 서비스 지향 패턴**: 서비스 레지스트리를 중심으로 서비스 간 상호작용. 확장성·가용성이 뛰어나지만 인프라 복잡도 최고

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 어떤 패턴을 쓰든 요구사항은 구현할 수 있다. **품질 속성은 기능 구현보다 의사결정을 이끌어낸다.** 품질 속성으로 디자인의 맞고 틀림은 판단할 수 없지만, 다른 디자인보다 더 낫거나 나쁨은 판단할 수 있다.
:::

### 3.2 의사결정 매트릭스

**의사결정 매트릭스(*Decision Matrix - 여러 대안을 품질 속성 기준으로 비교하는 표*)** 는 절충안을 분석할 때 쓰는 간단한 방법이다.

**라이언하트 프로젝트 의사결정 매트릭스 예시**:

| 품질 속성 시나리오 | 3계층 | 발행/구독 | 서비스 지향 |
|---|---|---|---|
| 가용성 (데이터베이스) | ↑ | → | ++ |
| 가용성 (운용시간) | → | → | + |
| 성능 (5초 이내 응답) | ↑ | → | + |
| 보안 | ↑ | → | → |
| 확장성 (연간 5% 성장) | ↓ | + | ++ |
| 유지보수성 (팀 숙련도) | ++ | ↓ | ↓ |
| 빌드 편의성 (구현 위험) | ++ | → | → |

**범례**: `++` 매우 높임 / `+` 높임 / `→` 중간 / `↓` 낮춤 / `↓↓` 매우 낮춤

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: **매트릭스를 만드는 과정이 매트릭스 자체보다 더 중요하다.** 이해관계자와 토론하며 요구사항을 발굴하고 요약하는 데 좋은 방법이다. 절대 숫자로 점수를 매기지 말 것 — 숫자는 그릇된 자신감을 주고 부정확한 분석을 초래한다.
:::

## 4. 구성요소에 기능별 역할 할당하기

아키텍처에 맞는 구조를 선택한 뒤에는 요소별로 고유 기능을 할당해 요구사항을 충족시켜야 한다.

라이언하트 프로젝트의 기능 요구사항 예시.

- 과거·현재 계약서 검색
- 페이지별 결과 조회
- 회사 기본정보 조회
- 계약서 기본정보 조회
- 계약서 갱신 알림 수신

이 안에는 시스템 컴포넌트가 가져야 할 역할도 암시된다.

- 검색 → 색인(*Index - 빠른 검색을 위해 원본 데이터의 요약본을 특정 순서로 정리해 놓은 자료구조*)이 필요
- 조회 → 어딘가에 저장이 필요
- 알림 → 이메일 주소 저장 필요
- 갱신 알림 → 갱신 감지 방법이 필요

### 4.1 라이언하트 컴포넌트별 역할

| 구성요소 | 역할 |
|---|---|
| 웹 UI | 사용자 인터페이스 표시, 입력 수신 |
| 디스플레이 비즈니스 | 인증·권한 관리, 백엔드 서비스 프록시, 비즈니스 로직 평가 |
| 검색 서비스 | 쿼리 파싱, 검색, 페이징, 필터링 |
| 즐겨찾기 서비스 | 태그 일반화, 즐겨찾기 저장 |
| 알림 서비스 | 갱신 사항 발견, 구독자 이메일 발송 |
| 크롤러 | 계약서 DB에서 읽어 색인용 데이터로 가공 후 업로드 |
| 사용자 메타데이터 DB | 사용자 정보·부가정보 저장 |
| 검색 색인 | 계약서 검색·정렬 최적화된 데이터 저장 |
| 계약서 DB | RFP와 계약서 정보 저장 |
| HTTP 관계 | 표준 HTTP, RESTful API |
| DB 드라이버 관계 | 데이터베이스 네이티브 드라이버 |

이 목록은 **하나의 역할에 단 하나의 구성요소만** 담당하도록 정리했고, **모든 구성요소는 적어도 하나의 역할을 가지도록** 했다. 목적 없는 컴포넌트는 없다.

## 5. 변화에 대응하는 디자인

단단한 아키텍처를 구축하려면 중요한 설계마다 좋은 결정을 내려야 하지만, 소프트웨어는 언제나 뜻밖의 변화를 마주한다. 위대한 아키텍처는 **피할 수 없는 변화에 꾸준히 대응한 결과**다.

### 5.1 결정은 미룰 수 있을 때까지 미룬다

메리·톰 포펜디크(Mary/Tom Poppendieck)의 *Lean Software Development*에서 "책임감 있는 최후의 순간(*Last Responsible Moment - 다른 대안을 선택할 기회가 사라지는 때*)"이라는 표현이 나온다. 저자는 이를 "가장 책임감이 큰 순간"으로 재정의한다 — 어떤 의사결정이 소프트웨어에 가장 크고 긍정적인 영향을 미치는 때.

**현재가 결정하기 좋은 시기인지 파악하는 질문**.

- 결정을 못 해서 더 나아갈 수 없는가?
- 당장 결정해야만 해결되는 문제인가?
- 의사결정이 더 많은 선택지나 기회를 만드는가?
- 결정을 미루면 위험이 매우 커지는가?
- 결정할 때의 영향력을 충분히 숙지하고 있는가?
- 지금 결정해야 하는 논리적 근거가 명확한가?
- 잘못되었을 때 되돌릴 시간이 있는가?

### 5.2 아키텍처에 영향을 주지 않도록 결정하기

언제라도 쉽게 바꿀 수 있는 결정이라면 이미 아키텍처와는 별 상관이 없다. 자주 바뀔 수 있는 사항은 상세 설계 쪽으로 미뤄둔다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: **SOLID 원칙**은 아키텍처에서도 유효하다. 단일 책임(SRP)으로 요소당 하나의 역할만 갖게 하면 변경이 잦은 부분을 쉽게 분리할 수 있다. 추상화 원칙에 따라 깨끗한 인터페이스를 만들면 구조가 유연해진다.
:::

추가 가능한 아키텍처, 환경 변수 설정, 자기 기술적(*self-describing - 데이터 안에 스키마·구조가 함께 담긴*) 데이터, 동적 탐색 등은 아키텍처를 훼손하지 않으면서 시스템의 형태·동작을 바꿀 수 있는 방법이다.

## 6. 라이언하트 프로젝트 사례

미니 품질 속성 워크숍을 마친 뒤, 팀은 여러 아키텍처 선택지를 놓고 토론했다. 데이터 기반 웹 애플리케이션이라는 사실은 파악했지만 그 이상은 결정되지 않은 상태였다.

팀원 대부분이 MySQL 사용 경험이 있어서 데이터베이스는 **MySQL로 결정**했다. 검색 엔진은 조사·선정 작업을 백로그에 추가했다.

코딩 전 단계에서 코드 관리·컴포넌트 구성 방향을 결정하기 위해 **생각-실행-확인** 단계로 계획을 세웠다.

1. **생각(계획)**: 화이트보드에 고민거리와 위험 요소를 나열
2. **실행**: 팀을 나눠 각자 답을 찾음(활동 15 "나눠서 정복하기"). 절반은 품질 속성을 달성할 패턴, 나머지 절반은 기술 관련 질문에 답을 찾음
3. **확인**: 일주일 뒤 소집해 각자 찾은 답으로 결정

::: callout {icon="📝" color="gray_bg"}
**직접 해보기**: 진행 중인 프로젝트에서 세 가지 이상의 아키텍처 대안을 상상해보고, 6~8개 품질 속성 시나리오에 대해 각 대안이 어떤 영향을 주는지 매트릭스로 그려보자. 숫자를 쓰지 말고 `++`, `+`, `→`, `↓`, `↓↓` 다섯 수준으로만 표기할 것.
:::

## 요약

- 소프트웨어 아키텍처 설계는 **불확실성 속에서 의사결정을 내리는 일**이다. 절충과 타협이 있으며, 나쁜 것을 피하고 좋은 것을 포기하는 결정도 필요하다
- 설계는 **분기(대안 확장) → 융합(대안 좁힘)** 의 반복이다. 대안이 없으면 결정할 필요도 없다
- 아키텍트가 파악해야 할 것: 구성요소와 역할, 관계와 인터페이스, 도메인, 기술·프레임워크, 설치·배포, 과거 설계
- **제약**은 기술적 제약(주로 수용)과 비즈니스적 제약(더 유연)으로 나뉘며, 아키텍처의 의사결정은 여전히 품질 속성에 초점을 맞춰야 한다
- **품질 속성이 구조 선택을 이끈다** — 3계층, 발행/구독, SOA는 서로 다른 품질 속성을 촉진한다
- **의사결정 매트릭스**로 대안을 품질 속성 기준으로 비교하되, 절대 숫자로 점수 매기지 말 것. 매트릭스를 만드는 과정이 매트릭스 자체보다 중요하다
- 구성요소에 역할을 할당할 때는 **하나의 역할 = 하나의 구성요소**, **모든 구성요소는 적어도 하나의 역할**을 원칙으로 한다
- 결정은 **가장 책임감이 큰 순간(Last Responsible Moment)** 까지 미루되, SOLID 원칙과 추상화로 아키텍처에 영향을 주지 않으면서 자주 바뀌는 부분을 분리하라
- 위대한 아키텍처는 피할 수 없는 변화에 꾸준히 대응한 결과다

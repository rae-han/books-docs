# Chapter 32: Frameworks Are Details (프레임워크는 세부사항이다)

## 핵심 질문

프레임워크는 아키텍처가 될 수 있는가? 프레임워크 제작자와 사용자 사이의 관계는 왜 비대칭적이며, 프레임워크와의 결합에는 어떤 위험이 따르는가? 프레임워크를 사용하면서도 아키텍처를 보호할 수 있는 방법은 무엇인가?

---

## 1. 프레임워크의 가치와 한계

프레임워크는 상당한 인기를 끌고 있다. 일반적으로 말하자면 좋은 현상이다. 무료인 데다 강력하며 유용한 프레임워크가 많다.

**하지만 아무리 해도 프레임워크는 아키텍처가 될 수 없다.**

---

## 2. 프레임워크 제작자의 동기

대다수의 프레임워크 제작자는 커뮤니티에 도움이 되기를 바라는 마음에 자신의 작업물을 무료로 제공한다. 칭찬 받을 일이다.

하지만 고상한 동기와 관계없이, 이들은 당신이 풀어야 할 특별한 관심사를 염두에 두지 않는다.

| 프레임워크 제작자의 관점 | 당신의 관점 |
|--------------------------|------------|
| 자신의 고유한 문제를 해결 | 당신의 고유한 문제를 해결해야 함 |
| 동료와 친구들의 문제를 해결 | 프레임워크가 풀려는 문제와 겹치는 부분만 혜택 |
| **당신의 문제를 알지 못한다** | 겹치는 영역이 클수록 유용 |

---

## 3. 혼인 관계의 비대칭성

당신과 프레임워크 제작자 사이의 관계는 놀라울 정도로 비대칭적이다. **당신은 프레임워크를 위해 대단히 큰 헌신을 해야 하지만, 프레임워크 제작자는 당신을 위해 아무런 헌신도 하지 않는다.**

### 3.1 프레임워크 제작자가 원하는 것

프레임워크를 사용해야 할 경우 우리는 프레임워크 제작자가 제공하는 문서를 꼼꼼히 읽는다. 대개의 경우 문서는 다음과 같이 조언한다:

- 프레임워크를 **중심에 두고** 당신의 아키텍처는 그 바깥을 감싸야 한다
- 프레임워크의 **기반 클래스에서 직접 파생**하라
- 프레임워크의 기능들을 **업무 객체에 바로 임포트**해서 사용하라

프레임워크 제작자는 당신의 애플리케이션이 가능하면 프레임워크에 공고하게 결합될 것을 강하게 역설한다.

### 3.2 왜 결합을 원하는가?

프레임워크 제작자 입장에서는 프레임워크와의 결합이 위험 요소가 되지 않는다. 제작자는 그 프레임워크에 대해 절대적인 제어권을 쥐고 있기 때문이다.

한술 더 떠서 제작자는 **당신도 자신의 프레임워크에 결합되기를 바란다.** 한번 결합하면 그 관계를 깨기가 매우 어렵기 때문이다. 프레임워크의 기반 클래스로부터 기꺼이 파생하는 방식으로, 프레임워크와 절대로 분리할 수 없는 방식으로 쓰는 사용자가 많아지는 일은 자신의 프레임워크가 효과적임을 검증하는 최고의 방법이다.

> **핵심 통찰**: 사실상 프레임워크 제작자는 당신에게 프레임워크와 **혼인**하기를 요구하는 것이다. 장기간에 걸친 막대한 헌신을 요청하면서도, 그에 상응하는 헌신을 당신에게 하지는 않을 것이다. 이 혼인 관계는 일방적이다. **모든 위험과 부담은 오롯이 당신이 감수할 뿐, 제작자가 감수하는 건 아무것도 없다.**

---

## 4. 위험 요인

프레임워크와의 결합에서 고려해야 할 위험 요인은 다음과 같다.

### 4.1 의존성 규칙 위반

프레임워크의 아키텍처는 그다지 깔끔하지 않은 경우가 많다. 프레임워크는 **의존성 규칙을 위반하는 경향**이 있다. 업무 객체를 만들 때, 프레임워크 제작자는 자신의 코드를 상속할 것을 요구한다. 당신만의 고유한 엔티티를 만들 때 말이다! 프레임워크가 한번 안쪽 원으로 들어가버리면 **다시는 원 밖으로 나오지 않을 것이다.** 결혼반지는 이미 당신의 손가락에 끼워졌고, 다시는 빼지 못할 것이다.

### 4.2 성장 과정에서의 마찰

프레임워크는 애플리케이션의 초기 기능을 만드는 데는 도움이 될 것이다. 하지만 제품이 성숙해지면서 프레임워크가 제공하는 기능과 틀을 벗어나게 될 것이다. 결혼반지를 꼈다면, 시간이 지나면서 **프레임워크와 계속 싸우고 있는 자신을 발견**하게 될 것이다.

### 4.3 진화 방향의 불일치

프레임워크는 당신에게 도움되지 않는 방향으로 진화할 수도 있다. 도움도 되지 않는 신규 버전으로 업그레이드하느라 다른 일을 못할 수도 있다. 심지어 사용 중이던 기능이 사라지거나 반영하기 힘든 형태로 변경될 수도 있다.

### 4.4 더 나은 대안의 등장

새롭고 더 나은 프레임워크가 등장해서 갈아타고 싶을 수도 있다. 하지만 기존 프레임워크에 깊이 결합되어 있다면 전환 비용이 막대해진다.

---

## 5. 해결책: 프레임워크와 결혼하지 말라

프레임워크를 사용할 수는 있다. 다만 **프레임워크와 결합해서는 안 된다.** 적당히 거리를 두자.

### 5.1 핵심 원칙

| 원칙 | 실천 방법 |
|------|-----------|
| 프레임워크를 세부사항으로 취급하라 | 아키텍처의 바깥쪽 원에 배치 |
| 안쪽 원으로 들어오지 못하게 하라 | 업무 객체가 프레임워크를 모르게 유지 |
| 기반 클래스 상속을 거절하라 | 프락시(proxy)를 만들어 업무 규칙에 플러그인 |
| 의존성 규칙을 준수하라 | 핵심 코드에 플러그인할 수 있는 컴포넌트에 프레임워크를 통합 |

### 5.2 스프링(Spring) 예시

스프링은 훌륭한 의존성 주입 프레임워크다. 의존성을 연결할 때 스프링의 오토-와이어링(auto-wiring) 기능을 사용하는 것은 괜찮다.

하지만 `@autowired` 어노테이션이 **업무 객체 도처에 산재해서는 안 된다.** 업무 객체는 절대로 스프링에 대해 알아서는 안 된다.

```
나쁜 예: 업무 객체에 프레임워크 어노테이션 산재
--------------------------------------------------
@Service
public class OrderService {
    @Autowired
    private OrderRepository repository;  // 업무 객체가 스프링을 안다!
}

좋은 예: 메인 컴포넌트에서만 스프링 사용
--------------------------------------------------
// 업무 객체 (스프링에 대해 전혀 모른다)
public class OrderService {
    private final OrderRepository repository;
    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }
}

// 메인(Main) 컴포넌트에서 의존성 주입 설정
@Configuration
public class AppConfig {
    @Bean
    public OrderService orderService(OrderRepository repo) {
        return new OrderService(repo);
    }
}
```

업무 객체보다는 **메인(Main) 컴포넌트**에서 스프링을 사용해서 의존성을 주입하는 편이 낫다. 메인은 아키텍처 내에서 가장 지저분한, 최저 수준의 컴포넌트이기 때문에 스프링을 알아도 상관 없다.

## 6. 이제 선언합니다: 정말로 결혼해야 하는 경우

정말로 결혼해야만 하는 프레임워크도 존재한다.

| 언어 | 결혼 대상 | 이유 |
|------|-----------|------|
| C++ | STL | 피하기 힘든 관계 |
| Java | 표준 라이브러리 | 반드시 결혼해야 함 |

이러한 관계는 정상이다. 하지만 **선택적**이어야 한다. 애플리케이션이 프레임워크와 결혼하고자 한다면 애플리케이션의 남은 생애 동안 그 프레임워크와 항상 함께 해야 한다는 사실을 반드시 명심해야 한다.

- 좋을 때나 궂을 때나
- 아플 때나 건강할 때나
- 부유할 때나 가난할 때나
- 다른 모든 것을 저버릴 때도

프레임워크를 사용해야 할 것이다. **결코 가볍게 시작할 수 있는 관계가 아니다.**

---

## 7. 결론

프레임워크와의 첫만남부터 바로 결혼하려 들지 말라. 결혼 서약에 앞서 잠시 동안 연애를 할 수 있는 방법이 있는지 확인하라. 가급적이면 프레임워크를 가능한 한 오랫동안 아키텍처 경계 너머에 두자.

---

## 요약

- 프레임워크는 아키텍처가 **될 수 없다** -- 유용하지만 세부사항이다
- 프레임워크 제작자와의 관계는 **비대칭적**이다 -- 모든 위험은 사용자가 감수한다
- 프레임워크는 의존성 규칙을 위반하는 경향이 있으며, 성장 과정에서 마찰을 일으킬 수 있다
- 해결책: 프레임워크와 **결혼하지 말라** -- 적당한 거리를 유지하라
- 프레임워크는 아키텍처의 **바깥쪽 원**에 속하는 세부사항으로 취급하라
- 업무 객체가 프레임워크를 알아서는 안 되며, **프락시**를 통해 간접적으로 사용하라
- 언어의 표준 라이브러리처럼 정말로 결혼해야 할 때만 결혼하되, 그 결정의 무게를 충분히 인식하라

## 다른 챕터와의 관계

- **Chapter 22 (클린 아키텍처)**: 프레임워크는 클린 아키텍처에서 가장 바깥쪽 원(프레임워크와 드라이버)에 위치한다. 의존성 규칙에 따라 프레임워크가 안쪽 원으로 침투해서는 안 된다.
- **Chapter 26 (메인 컴포넌트)**: 프레임워크 설정과 의존성 주입은 메인 컴포넌트에서 처리해야 한다. 메인은 가장 지저분한 컴포넌트이므로 프레임워크를 알아도 된다.
- **Chapter 30 (데이터베이스는 세부사항이다)**: 데이터베이스 프레임워크(ORM 등)도 세부사항이다. 이들이 업무 객체에 침투하면 안 된다.
- **Chapter 31 (웹은 세부사항이다)**: 웹 프레임워크도 같은 원칙이 적용된다 -- 세부사항으로 취급하고 바깥쪽 원에 위치시켜야 한다.
- **Chapter 11 (DIP, 의존성 역전 원칙)**: 프레임워크에 대한 의존성을 역전시켜 업무 규칙이 프레임워크에 의존하지 않도록 하는 것이 핵심이다.
- **Chapter 8 (OCP, 개방-폐쇄 원칙)**: 프레임워크를 교체 가능하게 만드는 것은 OCP의 실천이기도 하다.

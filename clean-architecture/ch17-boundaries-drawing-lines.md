# Chapter 17: Boundaries: Drawing Lines (경계: 선 긋기)

## 핵심 질문

소프트웨어 아키텍처에서 "선을 긋는다"는 것은 무엇을 의미하며, 왜 너무 이른 결정이 위험한가? 경계선은 어디에 어떻게 그어야 하며, 업무 규칙과 세부사항(데이터베이스, GUI 등) 사이의 관계는 어떤 방향이어야 하는가? 플러그인 아키텍처란 무엇인가?

---

## 1. 경계란 무엇인가

소프트웨어 아키텍처는 **선을 긋는 기술**이며, 이 선을 경계(boundary)라고 부른다.

경계의 역할:
- 소프트웨어 요소를 서로 **분리**한다
- 경계 한편에 있는 요소가 반대편에 있는 요소를 **알지 못하도록** 막는다

경계선의 시점:
- 일부는 프로젝트 수명 중 **아주 초기에**, 심지어 코드가 작성되기도 전에 그어진다
- 어떤 선은 **매우 나중에** 그어진다
- 초기에 그어지는 선들은 가능한 한 오랫동안 결정을 연기시키기 위한 목적으로 쓰인다

### 1.1 인적 자원 효율을 떨어뜨리는 요인: 결합

아키텍트의 목표는 필요한 시스템을 만들고 유지하는 데 드는 **인적 자원을 최소화**하는 것이다. 인적 자원의 효율을 떨어뜨리는 요인은 **결합(coupling)**이다. 특히 **너무 일찍 내려진 결정에 따른 결합**이다.

### 1.2 이른 결정이란?

시스템의 업무 요구사항, 즉 유스케이스와 **아무런 관련이 없는 결정**이다:

- 프레임워크에 대한 결정
- 데이터베이스에 대한 결정
- 웹 서버에 대한 결정
- 유틸리티 라이브러리에 대한 결정
- 의존성 주입에 대한 결정

> **핵심 통찰**: 좋은 시스템 아키텍처란 이러한 결정이 부수적이며, 결정을 연기할 수 있는 아키텍처다. 좋은 시스템 아키텍처는 이러한 결정을 가능한 한 최후의 순간에 내릴 수 있게 해주며, 결정에 따른 영향이 크지 않게 만든다.

## 2. 두 가지 슬픈 이야기

### 2.1 P사: 3-티어의 비극

1980년대에 간단한 모노리틱 데스크톱 애플리케이션으로 성공한 P사는 1990년대 후반 웹 시대에 맞춰 변환 프로젝트에 착수했다.

**잘못된 결정**: 자바 프로그래머들은 서버 팜이 춤추는 이상을 꿈꾸며, 3-티어(GUI 서버 / 미들웨어 서버 / 데이터베이스 서버)로 구성된 "아키텍처"(*3-티어는 아키텍처가 아니라 망 구성방식(topology)이다. 좋은 아키텍처라면 반드시 미뤄야 하는 종류의 결정이다.*)를 채택했다. 모든 도메인 객체가 **세 가지 인스턴스**를 가져야 한다고 결정했다.

결과:
- 기존 레코드에 새 필드를 추가하는 **간단한 기능**을 구현할 때마다: 세 티어의 클래스 모두에 추가, 네 개의 메시지 프로토콜 설계, 여덟 개의 프로토콜 핸들러 작성, 세 개의 실행 파일 빌드가 필요했다
- 개발 중에 서버 팜을 확보할 수 없어 세 실행 파일을 **단일 장비에서 실행**했지만, 모든 직렬화, 마샬링/언마샬링, 소켓 통신 등의 오버헤드를 감수했다
- **역설적이게도 P사는 서버 팜을 필요로 하는 시스템을 한 번도 판매하지 못했다.** 모든 배포는 단일 서버였다

### 2.2 W사: SOA의 지옥

차량 관리 지역 기업 W사는 "아키텍트"를 고용하여 엔터프라이즈급 서비스 지향 '아키텍처'를 도입했다.

문제점:
- 판매 레코드에 연락 담당자를 추가하려면: ServiceRegistry에서 서비스 ID 요청 → CreateContact 메시지 전송 (수십 개의 필드를 가짜 데이터로 채움) → 새 ID를 판매 레코드에 삽입 → UpdateContact 메시지를 SaleRecordService로 전송
- 테스트하려면 모든 서비스, 메시지 버스, BPel 서버 등을 구동시켜야 하고, 전달 지연(propagation delay)이 발생
- 새 기능 추가 시 엄청난 양의 WSDL 변경과 재배포가 필요

**W사의 실수는 SOA 자체가 아니라, 거대한 도메인 객체 서비스를 너무 이른 시기에 채택한 것이다.**

## 3. 성공 사례: FitNesse

### 3.1 배경

Robert C. Martin과 아들 Micah는 2001년에 FIT 도구를 기반으로 하는 간단한 위키 페이지 FitNesse를 만들기 시작했다. '다운로드 후 바로 실행(Download and Go)' 원칙을 준수했다.

### 3.2 핵심 결정들

**결정 1 — 웹 서버 직접 작성**: 오픈 소스 웹 서버를 사용하지 않고 직접 작성했다. 기본 뼈대만 갖춘 웹 서버는 구현이 간단했고, **어떤 웹 프레임워크를 사용할지에 대한 결정을 훨씬 나중으로 연기**할 수 있었다.

**결정 2 — 데이터베이스 결정 연기**: MySQL을 염두에 두고 있었지만, 모든 데이터 접근 영역과 데이터 저장소 영역 사이에 **인터페이스를 추가**하는 설계 방식을 사용했다. 데이터 접근 메서드들을 `WikiPage`라는 인터페이스에 두었다.

### 3.3 발전 과정

```
단계 1 (석 달):  위키 텍스트 → HTML 변환 작업. 데이터 저장소 불필요.
                MockWikiPage 클래스의 스텁 메서드 사용.

단계 2:         실제 데이터 접근 필요. InMemoryPage 파생 클래스 작성.
                해시 테이블을 RAM에 유지.

단계 3 (1년):   기능들을 차례대로 작성. 완전하게 동작하는 최초 버전 출시.
                저장 기능만 미구현.

단계 4:         영속성 구현 시점. MySQL 대신 플랫 파일 선택.
                FileSystemWikiPage 구현.

단계 5 (석 달 후): 플랫 파일이 적합하다는 결론. MySQL 아이디어 폐기.

단계 6:         고객 한 명이 MySQL을 원함.
                하루 만에 MySqlWikiPage 파생 클래스 작성으로 전체 시스템 동작.
                (결국 아무도 사용하지 않아 선택사항에서 제거)
```

### 3.4 교훈

- **18개월간** 데이터베이스 없이 운영 → 스키마, 쿼리, 서버, 패스워드 문제가 모두 없었다
- 데이터베이스가 없으니 **테스트를 더 빨리** 돌릴 수 있었다
- 경계선을 긋는 행위는 결정을 늦추고 연기하는 데 도움이 되었고, 궁극적으로는 **시간을 엄청나게 절약**해주었다

## 4. 경계를 어디에, 어떻게 그을 것인가

### 4.1 기본 원칙: 관련이 있는 것과 없는 것 사이에 선을 긋는다

| 관계 | 경계 필요 여부 |
|------|-------------|
| GUI ↔ 업무 규칙 | 관련 없음 → **경계 필요** |
| 데이터베이스 ↔ GUI | 관련 없음 → **경계 필요** |
| 데이터베이스 ↔ 업무 규칙 | 관련 없음 → **경계 필요** |

### 4.2 업무 규칙과 데이터베이스 사이의 경계

업무 규칙이 데이터베이스와 떼어놓을 수 없다고 믿는 사람이 많지만, 이는 잘못된 생각이다. 데이터베이스는 업무 규칙이 간접적으로 사용할 수 있는 **도구**다. 업무 규칙이 알아야 할 것은 **데이터를 가져오고 저장할 때 사용할 수 있는 함수 집합이 있다는 사실이 전부다.**

클래스 수준의 구조 (그림 17.1, 17.2):

```
┌──────────────┐      ┌─────────────────┐
│  Business    │ ───→ │  <<Interface>>   │
│  Rules       │      │  Database        │
│              │      │  Interface       │
└──────────────┘      └─────────────────┘
                              △
                              │ (구현)
                      ┌─────────────────┐      ┌──────────┐
                      │  Database       │ ───→ │ Database │
                      │  Access         │      │          │
                      └─────────────────┘      └──────────┘

경계선은 DatabaseInterface 바로 아래, 상속 관계를 횡단하며 그어진다.
```

**핵심 관찰**: DatabaseAccess에서 출발하는 두 화살표는 모두 바깥쪽으로 향한다. 즉, DatabaseAccess가 존재한다는 사실을 알고 있는 클래스는 없다.

컴포넌트 수준의 구조 (그림 17.3):

```
┌─────────────────┐         ┌──────────────┐
│  Business       │ ◀────── │  Database    │
│  Rules          │         │              │
└─────────────────┘         └──────────────┘

화살표 방향: Database → BusinessRules
```

**이 선의 방향이 중요하다.** BusinessRules에게 있어 Database는 문제가 되지 않지만, Database는 BusinessRules 없이는 존재할 수 없다. 이를 통해 BusinessRules에서는 **어떤 종류의 데이터베이스도 사용할 수 있다** — 오라클, MySQL, 카우치, 플랫 파일 등. 업무 규칙은 전혀 개의치 않는다.

---

## 5. 입력과 출력은 중요하지 않다

개발자와 고객은 종종 GUI를 보고 GUI가 시스템이라고 생각한다. 하지만 매우 중요한 원칙이 있다:

> **입력과 출력은 중요하지 않다.**

비디오 게임을 예로 들면, 화면/마우스/버튼/음향이라는 인터페이스 뒤에는 정교한 모델(데이터 구조와 함수의 집합)이 존재한다. **모델은 인터페이스를 전혀 필요로 하지 않는다.** 게임이 화면에 전혀 출력되지 않더라도 모델은 주어진 역할을 충실하게 수행한다.

GUI와 BusinessRules 컴포넌트 사이의 경계도 동일한 패턴이다:

```
┌─────────────────┐         ┌──────────────┐
│  Business       │ ◀────── │  GUI         │
│  Rules          │         │              │
└─────────────────┘         └──────────────┘

화살표 방향: GUI → BusinessRules
```

GUI는 다른 종류의 인터페이스로 얼마든지 교체할 수 있으며, BusinessRules는 전혀 개의치 않는다.

## 6. 플러그인 아키텍처

데이터베이스와 GUI에 대해 내린 두 가지 결정을 하나로 합쳐서 보면 **플러그인 아키텍처** 패턴이 만들어진다. 이 패턴은 시스템에서 서드파티 플러그인을 사용할 수 있게 한 바로 그 패턴과 동일하다.

```
                ┌───────────┐
                │    DB     │
                └─────┬─────┘
                      │
                      ▼
┌───────────┐   ┌───────────┐
│   GUI     │──▶│ Business  │
│           │   │  Rules    │
└───────────┘   └───────────┘
```

핵심적인 업무 규칙은 선택적이거나 다양한 형태로 구현될 수 있는 나머지 컴포넌트로부터 분리되어 있고, 독립적이다:

- **사용자 인터페이스**: 웹 기반, 클라이언트/서버 기반, SOA, 콘솔 기반 등 어떤 기술이라도 플러그인 가능
- **데이터베이스**: SQL, NoSQL, 파일 시스템 기반 등 어떤 종류의 데이터베이스 기술로도 대체 가능

### 6.1 ReSharper와 Visual Studio의 비유

ReSharper(젯브레인즈, 러시아)는 Visual Studio(마이크로소프트, 레드몬드)에 플러그인된다. 의존성 구조를 보면:

```
┌──────────────┐
│     VS       │
└──────┬───────┘
       △
       │ (의존)
┌──────────────┐
│  ReSharper   │
└──────────────┘
```

- ReSharper 팀은 절대로 VS 팀을 건드릴 수 없다
- 하지만 VS 팀은 원한다면 언제든지 ReSharper 팀을 완전히 무력화할 수 있다

이 **비대칭적 관계**가 바로 우리가 시스템에서 갖추고자 하는 관계다. 누군가가 웹 페이지의 포맷을 변경하거나 데이터베이스 스키마를 변경하더라도 업무 규칙은 깨지지 않기를 바란다.

### 6.2 경계와 변경의 축

경계는 **변경의 축(axis of change)**이 있는 지점에 그어진다:
- GUI는 업무 규칙과는 다른 시점에 다른 속도로 변경된다 → 경계 필요
- 업무 규칙은 의존성 주입 프레임워크와는 다른 시점에 다른 이유로 변경된다 → 경계 필요

이것은 순전히 **단일 책임 원칙**에 해당한다. 단일 책임 원칙은 어디에 경계를 그어야 할지를 알려준다.

---

## 요약

- 소프트웨어 아키텍처는 **선을 긋는 기술**이다 — 경계는 소프트웨어 요소를 분리하고 서로 알지 못하게 막는다
- 너무 이른 결정에 따른 결합은 인적 자원의 효율을 떨어뜨린다 — P사와 W사의 실패 사례가 이를 증명한다
- FitNesse 사례처럼 **데이터베이스 결정을 연기**하고 인터페이스 뒤로 숨기면 18개월간 데이터베이스 없이도 개발이 가능하다
- 경계선은 **관련이 있는 것과 없는 것 사이에** 긋는다
- **의존성의 방향이 핵심**이다 — 화살표는 항상 저수준 세부사항에서 고수준 업무 규칙을 향해야 한다
- **플러그인 아키텍처**가 궁극적인 목표다 — 업무 규칙이 핵심이고, GUI와 데이터베이스는 플러그인이다
- 이 모든 것은 **의존성 역전 원칙**과 **안정된 추상화 원칙**을 응용한 것이다

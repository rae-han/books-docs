# Chapter 30: The Database Is a Detail (데이터베이스는 세부사항이다)

## 핵심 질문

데이터베이스는 아키텍처의 핵심 요소인가, 아니면 교체 가능한 세부사항에 불과한가? 데이터 모델과 데이터베이스 기술의 차이는 무엇이며, 왜 이 구분이 중요한가?

---

## 1. 데이터베이스는 엔티티가 아니다

아키텍처 관점에서 볼 때 데이터베이스는 엔티티가 아니다. 데이터베이스는 **세부사항**이며, 아키텍처의 구성요소 수준으로 끌어올릴 수 없다. 소프트웨어 시스템의 아키텍처와 데이터베이스의 관계를 건물로 비유하면, 건물의 아키텍처와 **문 손잡이**의 관계와 같다.

### 1.1 데이터 모델 vs 데이터베이스

여기서 중요한 구분이 있다.

| 구분 | 아키텍처적 중요성 | 설명 |
|------|-------------------|------|
| **데이터 모델** | 대단히 중요하다 | 애플리케이션 내부 데이터의 구조 |
| **데이터베이스** | 중요하지 않다 | 데이터에 접근할 방법을 제공하는 유틸리티 소프트웨어 |

데이터베이스는 데이터 모델이 아니다. 데이터베이스는 일개 소프트웨어일 뿐이며, 데이터에 접근할 방법을 제공하는 유틸리티다. 아키텍처 관점에서 보면 이러한 유틸리티는 저수준의 세부사항(메커니즘)일 뿐이라서 아키텍처와는 관련이 없다.

> **핵심 통찰**: 뛰어난 아키텍트라면 저수준의 메커니즘이 시스템 아키텍처를 오염시키는 일을 용납하지 않는다.

---

## 2. 관계형 데이터베이스

에드거 커드(Edgar Codd)는 1970년에 관계형 데이터베이스의 원칙을 정의했다. 관계형 모델은 계속 성장했고, 1980년대 중반이 되자 데이터 저장소의 지배적인 형태가 되었다. 관계형 데이터베이스가 인기를 끈 데는 타당한 이유가 있었다 -- 이 모델은 우아했고, 절제되었으며, 강건했다.

하지만 관계형 데이터베이스의 기술이 얼마나 뛰어나든, 유용하든, 수학적으로 견고하든, **결국은 그저 기술일 뿐이다.** 그리고 이는 관계형 데이터베이스가 세부사항임을 뜻한다.

### 2.1 테이블 구조의 한계

관계형 테이블은 특정한 형식의 데이터에 접근하는 경우에는 편리하지만, 데이터를 테이블에 행 단위로 배치한다는 자체는 아키텍처적으로 전혀 중요하지 않다. 애플리케이션의 유스케이스는 이러한 방식을 알아서는 안 되며, 관여해서도 안 된다. 데이터가 테이블 구조를 가진다는 사실은 오직 아키텍처의 외부 원에 위치한 **최하위 수준의 유틸리티 함수만** 알아야 한다.

많은 데이터 접근 프레임워크가 테이블과 행이 객체 형태로 시스템 여기저기에서 돌아다니게 허용하는데, 이는 아키텍처적으로 잘못된 설계다. 이렇게 하면 유스케이스, 업무 규칙, 심지어는 UI조차도 관계형 데이터 구조에 결합되어 버린다.

---

## 3. 데이터베이스 시스템은 왜 이렇게 널리 사용되는가?

어떻게 데이터베이스 시스템이 소프트웨어 시스템과 소프트웨어 기업을 장악할 수 있었을까? 오라클, MySQL, SQL 서버가 우위를 차지할 수 있던 이유는 무엇일까?

**한마디로 답하자면, 바로 '디스크' 때문이었다.**

### 3.1 회전식 자기 디스크의 지배

지난 반세기 동안 회전식 자기 디스크(magnetic disk)는 데이터 저장소의 중심에 있었다. 여러 세대의 프로그래머가 다른 형태의 데이터 저장소도 있다는 사실조차 모를 정도였다. 디스크 기술은 과거에 지름 48인치 플래터를 겹겹이 쌓아 수천 파운드나 되는 디스크가 겨우 20MB를 담을 수 있었던 것에서, 지금은 지름 3인치의 얇은 단일 원판으로 1TB 이상을 저장할 수 있는 수준까지 발전했다.

그리고 이렇게 질주하는 내내 프로그래머는 디스크 기술이 지닌 치명적인 한 가지 특성으로 인해 괴롭힘을 당했다. 바로 **느리다**는 특성이었다.

### 3.2 디스크의 접근 방식과 속도 문제

디스크에서 특정 바이트를 읽으려면 다음 과정을 거쳐야 한다:

1. **헤드 이동** -- 헤드(head)를 적절한 트랙(track)으로 옮긴다
2. **회전 대기** -- 디스크가 돌면서 헤드 위치에 적절한 섹터(sector)가 올 때까지 기다린다
3. **섹터 읽기** -- 해당 섹터에서 4K 모두를 RAM으로 읽어 들인다
4. **색인 탐색** -- RAM 버퍼의 색인을 찾아서 필요한 바이트를 가져온다

이 모든 작업에는 **밀리초** 단위의 시간이 걸렸다. 밀리초가 그리 길지 않다고 느낄 수도 있지만, 대다수의 프로세서에서 한 명령어를 처리하는 주기와 비교해 보면 **백만 배**나 오래 걸렸다.

### 3.3 데이터 접근 시스템의 분화

이처럼 디스크의 시간 지연을 완화하기 위해 색인, 캐시, 쿼리 계획 최적화가 필요해졌다. 시간이 지나면서 데이터 접근 및 관리 시스템은 두 가지 유형으로 분리되었다.

| 유형 | 특징 | 강점 | 약점 |
|------|------|------|------|
| **파일 시스템** | 문서(document) 기반 | 이름 기준 저장/조회에 탁월 | 내용 기준 검색에 부적합 |
| **RDBMS** | 내용(content) 기반 | 내용 기반 레코드 검색에 탁월 | 정형화되지 않은 문서 저장에 부적합 |

이 두 시스템 모두 데이터를 디스크에 체계화해서, 각 시스템에 특화된 방식으로 가능한 한 효율적으로 데이터를 저장하고 검색할 수 있도록 한다. 덧붙이자면, 데이터를 빠르게 조작할 수 있도록 결국에는 관련 있는 데이터를 RAM으로 가져온다.

---

## 4. 디스크가 없다면 어떻게 될까?

한때는 디스크가 성행했지만, 이제는 소멸 중인 부품이다. 머지않아 디스크는 테이프 드라이브, 플로피 드라이브, CD가 지나온 길을 따를 것이다. **디스크는 RAM으로 대체되고 있다.**

스스로에게 물어보라. 디스크가 모두 사라진다면, 그래서 모든 데이터가 RAM에 저장된다면 데이터를 어떻게 체계화할 것인가?

- 데이터를 테이블 구조로 만들어 SQL을 이용해 접근할 것인가?
- 파일 구조로 만들어 디렉터리를 통해 접근할 것인가?

**당연히 아니다.** 이 데이터들을 연결 리스트, 트리, 해시 테이블, 스택, 큐 혹은 여타 무수히 많은 데이터 구조로 체계화할 것이며, 데이터에 접근할 때는 포인터나 참조를 사용할 것이다. 이것이 프로그래머가 하는 일이기 때문이다.

> **핵심 통찰**: 데이터가 데이터베이스나 파일 시스템에 있더라도, RAM으로 읽은 후에는 다루기 편리한 형태로 그 구조를 변경한다 -- 리스트, 집합, 스택, 큐, 트리 등. 데이터를 파일이나 테이블 형태로 그대로 두는 경우는 거의 없다.

---

## 5. 데이터베이스는 세부사항이다

데이터베이스가 세부사항이라고 말하는 이유는 바로 이러한 현실 때문이다. 데이터베이스는 그저 메커니즘에 불과하며, 디스크 표면과 RAM 사이에서 데이터를 이리저리 옮길 때 사용할 뿐이다. 실제로 데이터베이스는 비트를 담는 거대한 그릇이며, 데이터를 장기적으로 저장하는 공간에 지나지 않는다.

따라서 아키텍처 관점에서 본다면 회전식 자기 디스크에 데이터가 있기만 한다면, 데이터가 어떤 형태인지는 절대로 신경 써서는 안 된다. 정말로 우리는 **디스크 자체가 존재한다는 사실조차도 인식해서는 안 된다.**

---

## 6. 하지만 성능은?

성능은 아키텍처와 관련된 관심사가 아닌가? 당연히 아키텍처적인 관심사다.

하지만 데이터 저장소의 측면에서 성능은 **완전히 캡슐화하여 업무 규칙과는 분리할 수 있는 관심사**다. 데이터 저장소에서 데이터를 빠르게 넣고 뺄 수 있어야 하는 것은 맞지만, 이는 저수준의 관심사다. 이 관심사는 저수준의 데이터 접근 메커니즘 단에서 다룰 수 있다.

| 관심사 | 수준 | 아키텍처적 중요성 |
|--------|------|-------------------|
| 업무 규칙 | 고수준 | 핵심 |
| 데이터 모델 | 고수준 | 핵심 |
| 데이터 저장소 성능 | 저수준 | 캡슐화 가능 |
| 데이터베이스 기술 선택 | 저수준 | 세부사항 |

**성능은 시스템의 전반적인 아키텍처와는 아무런 관련이 없다.**

---

## 7. 개인적인 일화: RDBMS 전쟁

Uncle Bob은 1980년대 후반 스타트업에서의 경험을 소개한다. 이 회사는 T1 원격 통신 회선의 데이터 무결성을 측정하는 네트워크 관리 시스템을 구축하고 있었다.

### 7.1 기술적 배경

- **플랫폼**: 유닉스
- **저장 방식**: 간단한 랜덤 액세스 파일에 트리나 연결 리스트로 저장
- **이유**: 데이터 사이에 내용 기반 관계가 없었으므로 관계형 데이터베이스가 불필요했다
- **데이터 조작**: RAM에서 이루어졌고, RAM에 로드하기 가장 편리한 형태로 데이터를 저장했다

### 7.2 갈등의 시작

마케팅 담당자가 **마케팅적인 이유**로 관계형 데이터베이스를 도입해야 한다고 주장했다. 하드웨어 엔지니어도 기술적 이유를 들어 RDBMS가 필요하다며 경영진 앞에서 "막대기 위에 집을 짓지 마십시오"라는 비유로 설득했다.

Uncle Bob은 결사적으로 싸웠다. 연결 리스트와 트리를 대량의 행과 테이블로 재배치해서 SQL로 접근해야 할 이유가 없었고, 거대한 RDBMS 도입에 따르는 성능 오버헤드와 경비를 감당할 필요가 없었기 때문이다.

### 7.3 결과와 교훈

결국 하드웨어 엔지니어가 소프트웨어 관리자로 승진했고, RDBMS가 시스템에 도입되었다. 그리고 **그들이 전적으로 옳았으며, Uncle Bob은 틀렸다.**

하지만 공학적인 이유 때문은 아니었다. 공학적으로는 Uncle Bob이 옳았다. **틀렸던 점은 고객이 관계형 데이터베이스를 기대했다는 사실이었다.** 고객은 RDBMS로 무엇을 할 수 있는지 알지 못했고, 관계형 데이터를 사용할 현실적인 방안도 없었다. 하지만 그런 것들은 상관이 없었다 -- 데이터베이스는 모든 소프트웨어 구매자가 리스트에 올려놓은, 반드시 체크해야 할 항목이 되어 있었다.

> **핵심 통찰**: 이러한 인식은 당시 데이터베이스 업체에서 실시한 대단히 효과적인 마케팅 캠페인에서 비롯되었다. 기업의 '데이터 자산'은 보호할 필요가 있으며, 데이터베이스 시스템은 이를 약속하는 이상적인 도구라고 고위 경영진을 납득시킨 것이다. 오늘날에도 '엔터프라이즈'라는 단어와 '서비스-지향 아키텍처'라는 개념에서 같은 종류의 마케팅 캠페인을 볼 수 있다.

### 7.4 되돌아본다면

옛날 이 시절로 돌아간다면 어떻게 행동하는 게 옳았을까?

- 시스템의 한쪽에 RDBMS를 붙이고
- RDBMS에 대한 제한적이며 안정적인 몇 가지 데이터 접근 채널을 제공하는 동시에
- **시스템의 중심부에는 랜덤 액세스 파일을 그대로 유지**하면 좋았을 것이다

실제로는? 회사를 관두고 컨설턴트가 되었다.

---

## 요약

- 체계화된 **데이터 구조와 데이터 모델**은 아키텍처적으로 중요하다
- 반면, 그저 데이터를 회전식 자기 디스크 표면에서 이리저리 옮길 뿐인 **기술과 시스템**은 아키텍처적으로 중요치 않다
- 데이터를 테이블 구조로 만들고 SQL로만 접근하도록 하는 관계형 데이터베이스 시스템은 전자보다는 후자와 훨씬 관련이 깊다
- **데이터는 중요하다. 데이터베이스는 세부사항이다.**
- 데이터베이스 기술은 아키텍처의 외부 원, 최저 수준에 위치해야 한다
- 성능 관련 관심사는 저수준의 데이터 접근 메커니즘 단에서 캡슐화할 수 있다
- 마케팅 압력이나 비기술적 요구에도 아키텍처의 원칙을 지키되, 현실적인 타협점을 찾아야 한다

## 다른 챕터와의 관계

- **Chapter 22 (클린 아키텍처)**: 데이터베이스는 클린 아키텍처의 가장 바깥쪽 원(프레임워크와 드라이버)에 위치하는 세부사항이다. 이 장은 그 원칙을 데이터베이스라는 구체적 사례에 적용한다.
- **Chapter 8 (OCP, 개방-폐쇄 원칙)**: 데이터 접근 메커니즘을 인터페이스 뒤에 숨기면, 데이터베이스 기술을 교체하더라도 업무 규칙에 영향이 없다.
- **Chapter 31 (웹은 세부사항이다)**: 데이터베이스와 마찬가지로 웹도 세부사항이다. 둘 다 아키텍처의 핵심이 아닌 교체 가능한 메커니즘이다.
- **Chapter 32 (프레임워크는 세부사항이다)**: 데이터 접근 프레임워크(ORM 등)가 테이블과 행을 시스템 전체에 퍼뜨리는 문제와 프레임워크의 침투 문제는 같은 맥락이다.
- **Chapter 26 (메인 컴포넌트)**: 데이터베이스 설정과 같은 세부사항은 메인 컴포넌트에서 처리해야 한다.

# Chapter 9: LSP: The Liskov Substitution Principle (리스코프 치환 원칙)

## 핵심 질문

상위 타입을 하위 타입으로 치환할 수 있다는 것이 왜 소프트웨어 설계에서 중요한가? LSP는 단순한 상속 가이드를 넘어 아키텍처 수준에서 어떤 의미를 가지며, 이를 위반하면 시스템에 무슨 일이 벌어지는가?

---

## 1. LSP의 정의

1988년 바바라 리스코프(Barbara Liskov)는 하위 타입(subtype)을 아래와 같이 정의했다(*바바라 리스코프, "Data Abstraction and Hierarchy," SIGPLAN Notices 23, 5 (1988년 3월).*):

> 여기에서 필요한 것은 다음과 같은 치환(substitution) 원칙이다. S 타입의 객체 o1 각각에 대응하는 T 타입 객체 o2가 있고, T 타입을 이용해서 정의한 모든 프로그램 P에서 o2의 자리에 o1을 치환하더라도 P의 행위가 변하지 않는다면, **S는 T의 하위 타입**이다.

쉽게 말해, **상위 타입이 사용되는 모든 곳에서 하위 타입으로 교체해도 프로그램의 정확성이 깨지지 않아야 한다**는 원칙이다.

---

## 2. 상속을 사용하도록 가이드하기

LSP를 준수하는 전형적인 예제를 살펴보자. `License`라는 클래스가 있고, 이 클래스는 `calcFee()`라는 메서드를 가진다. `Billing` 애플리케이션에서 이 메서드를 호출한다. `License`에는 `PersonalLicense`와 `BusinessLicense`라는 두 가지 하위 타입이 존재한다.

```
                      ┌─────────────────┐
                      │   <<I>>         │
┌──────────┐         │    License      │
│ Billing  │────────▶├─────────────────┤
└──────────┘         │ + calcFee()     │
                      └────────┬────────┘
                               △
                      ┌────────┴────────┐
                      │                 │
              ┌───────┴───────┐ ┌───────┴───────┐
              │   Personal    │ │   Business    │
              │   License     │ │   License     │
              └───────────────┘ ├───────────────┤
                                │ - users       │
                                └───────────────┘
```
*그림 9.1 — License와 파생 클래스는 LSP를 준수한다.*

이 설계는 LSP를 준수한다. `Billing` 애플리케이션의 행위가 `License` 하위 타입 중 무엇을 사용하는지에 **전혀 의존하지 않기** 때문이다. 두 하위 타입은 모두 `License` 타입을 치환할 수 있다.

## 3. 정사각형/직사각형 문제

LSP를 위반하는 전형적인 문제로는 유명한 (보는 관점에 따라서는 악명 높은) **정사각형/직사각형(square/rectangle) 문제**가 있다.

```
                    ┌───────────────────┐
┌──────────┐       │    Rectangle      │
│   User   │──────▶├───────────────────┤
└──────────┘       │ + setH, + setW    │
                    └────────┬──────────┘
                             △
                             │
                    ┌────────┴──────────┐
                    │      Square       │
                    ├───────────────────┤
                    │ + setSide         │
                    └───────────────────┘
```
*그림 9.2 — 악명 높은 정사각형/직사각형 문제*

이 예제에서 `Square`는 `Rectangle`의 하위 타입으로는 적합하지 않다. 그 이유는 다음과 같다:

| 타입 | 높이와 너비의 관계 |
|------|------------------|
| `Rectangle` | 높이와 너비가 **서로 독립적으로** 변경될 수 있다 |
| `Square` | 높이와 너비가 **반드시 함께** 변경되어야 한다 |

`User`는 대화하고 있는 상대가 `Rectangle`이라고 생각하므로 혼동이 생길 수 있다. 아래의 코드를 보면 혼동의 이유가 분명해진다:

```java
Rectangle r = ...
r.setW(5);
r.setH(2);
assert(r.area() == 10);
```

만약 `...` 코드에서 `Square`를 생성한다면 `assert`문은 실패하게 된다. `Square`의 `setH(2)` 호출이 너비도 함께 2로 변경하기 때문에, `area()`는 10이 아닌 4를 반환한다.

이런 형태의 LSP 위반을 막기 위한 유일한 방법은 (`if`문 등을 이용해서) `Rectangle`이 실제로는 `Square`인지를 검사하는 메커니즘을 `User`에 추가하는 것이다. 하지만 이렇게 하면 `User`의 행위가 사용하는 타입에 의존하게 되므로, 결국 **타입을 서로 치환할 수 없게 된다.**

> **핵심 통찰**: 수학적으로 정사각형은 직사각형의 특수한 경우(is-a 관계)지만, 소프트웨어 설계에서는 반드시 그렇지 않다. 상속 관계는 "is-a"가 아니라 "행위의 치환 가능성"으로 판단해야 한다.

## 4. LSP와 아키텍처

### 4.1 상속을 넘어선 확장

객체 지향이 혁명처럼 등장한 초창기에는 LSP는 상속을 사용하도록 가이드하는 방법 정도로 간주되었다. 하지만 시간이 지나면서 LSP는 **인터페이스와 구현체에도 적용되는 더 광범위한 소프트웨어 설계 원칙**으로 변모해 왔다.

여기에서 말하는 인터페이스는 다양한 형태로 나타난다:

| 언어/환경 | 인터페이스의 형태 |
|----------|-----------------|
| Java 등 정적 타입 언어 | 인터페이스 하나와 이를 구현하는 여러 개의 클래스 |
| Ruby 등 동적 타입 언어 | 동일한 메서드 시그니처를 공유하는 여러 개의 클래스 |
| 서비스 아키텍처 | 동일한 REST 인터페이스에 응답하는 서비스 집단 |

이상의 모든 상황에서 LSP를 적용할 수 있다. 잘 정의된 인터페이스와 그 인터페이스의 구현체끼리의 상호 치환 가능성에 기대는 사용자들이 존재하기 때문이다.

### 4.2 아키텍처 관점

아키텍처 관점에서 LSP를 이해하는 최선의 방법은 **이 원칙을 어겼을 때 시스템 아키텍처에서 무슨 일이 일어나는지 관찰하는 것**이다.

---

## 5. LSP 위배 사례: 택시 파견 서비스

### 5.1 시나리오

다양한 택시 파견(taxi dispatch) 서비스를 통합하는 애플리케이션을 만들고 있다고 해보자. 고객은 어느 택시업체인지는 신경쓰지 않고 자신의 상황에 가장 적합한 택시를 찾는다. 고객이 이용할 택시를 결정하면, 시스템은 REST 서비스를 통해 선택된 택시를 고객 위치로 파견한다.

택시 파견 REST 서비스의 URI가 운전기사 데이터베이스에 저장되어 있다고 가정하자. 시스템이 고객에게 알맞은 기사를 선택하면, 해당 기사의 레코드로부터 URI 정보를 얻은 다음, 그 URI를 이용하여 파견 요청을 보낸다.

예를 들어 택시기사 밥(Bob)의 택시 파견 URI는 다음과 같다:

```
purplecab.com/driver/Bob
```

시스템은 이 URI에 파견에 필요한 정보를 덧붙인 후, PUT 방식으로 호출한다:

```
purplecab.com/driver/Bob
  /pickupAddress/24 Maple St.
  /pickupTime/153
  /destination/ORD
```

### 5.2 LSP 위반의 발생

분명한 점은 파견 서비스를 만들 때 다양한 택시업체에서 **동일한 REST 인터페이스를 반드시 준수**하도록 만들어야 한다는 사실이다. 서로 다른 택시업체가 `pickupAddress`, `pickupTime`, `destination` 필드를 모두 동일한 방식으로 처리해야 한다.

이제 택시업체 **애크미(ACME)**(*미국의 애니메이션 루니 툰(Looney Tunes)에 등장하는 가공의 기업. 이 애니메이션에서 애크미 사의 제품들은 하나같이 처절하게 오작동한다.*)에서 프로그래머를 몇 명 고용했는데, 이들이 서비스 사양서를 그다지 신중하게 읽지 않았다고 해보자. 그래서 `destination` 필드를 `dest`로 축약해서 사용했다.

### 5.3 아키텍처에 미치는 영향

이 예외 사항을 처리하는 가장 간단한 방법은 파견 명령어를 구성하는 모듈에 `if`문을 추가하는 것이다:

```java
if (driver.getDispatchUri().startsWith("acme.com"))...
```

하지만 실력 있는 아키텍트라면 당연히 시스템을 이런 식으로 구성하는 것을 용납하지 않는다. `"acme"`라는 단어를 코드 자체에 추가하면, **끔찍할 뿐만 아니라 이해할 수도 없는 온갖 종류의 에러가 발생할 여지**를 만들게 된다. 보안이 침해되는 것은 말할 것도 없다.

예를 들어 애크미가 더 성장해서 다른 택시업체인 퍼플(Purple)을 인수한다면? 합병된 퍼플이 브랜드와 웹 사이트는 독립적으로 유지하되 회사 시스템은 모두 통합한다면? `"purple"`을 처리하는 또 다른 `if`문을 추가해야만 하는가?

### 5.4 올바른 해결책

아키텍트는 이 같은 버그로부터 시스템을 격리해야 한다. 이때 파견 URI를 키(key)로 사용하는 **설정용 데이터베이스(configuration database)**를 이용하는 파견 명령 생성 모듈을 만들어야 할 수도 있다. 설정 정보는 대체로 아래와 같을 것이다:

| URI | Dispatch Format |
|-----|----------------|
| `Acme.com` | `/pickupAddress/%s/pickupTime/%s/dest/%s` |
| `*.*` | `/pickupAddress/%s/pickupTime/%s/destination/%s` |

또한 아키텍트는 REST 서비스들의 인터페이스가 서로 치환 가능하지 않다는 사실을 처리하는 **중요하고 복잡한 매커니즘**을 추가해야 한다.

---

## 6. 결론

LSP는 아키텍처 수준까지 확장할 수 있고, **반드시 확장해야만 한다.** 치환 가능성을 조금이라도 위배하면 시스템 아키텍처가 오염되어 상당량의 별도 메커니즘을 추가해야 할 수 있기 때문이다.

---

## 요약

- **LSP란** 상위 타입이 사용되는 곳에서 하위 타입으로 치환해도 프로그램의 정확성이 깨지지 않아야 한다는 원칙이다.
- **정사각형/직사각형 문제**: 수학적 "is-a" 관계와 소프트웨어의 "행위 치환 가능성"은 다르다. 정사각형은 직사각형의 하위 타입이 될 수 없다.
- **LSP는 상속을 넘어선다**: 인터페이스와 구현체, REST 서비스 등 모든 형태의 인터페이스에 적용된다.
- **LSP 위반의 대가**: 택시 파견 서비스 사례에서 보듯, 치환 가능성이 깨지면 `if`문이 코드 전체에 퍼지고, 설정 데이터베이스 같은 복잡한 메커니즘이 필요해진다.
- **아키텍처를 오염시킨다**: 사소한 치환 가능성 위반이 시스템 전체에 파급 효과를 미칠 수 있다.

## 다른 챕터와의 관계

- **Chapter 7 (SRP)**: SRP가 액터별로 코드를 분리하라고 말한다면, LSP는 분리된 인터페이스의 구현체들이 서로 치환 가능해야 한다고 말한다. 두 원칙은 상호 보완적이다.
- **Chapter 8 (OCP)**: OCP는 확장에 열려 있고 변경에 닫혀 있어야 한다고 말한다. LSP를 준수해야만 인터페이스를 통한 확장이 기존 코드를 변경하지 않고 가능하다.
- **Chapter 10 (ISP)**: ISP는 사용하지 않는 인터페이스에 의존하지 말라고 한다. LSP 위반 시 사용자가 특정 타입을 감지하는 로직을 추가해야 하는데, 이는 ISP 위반과도 연결된다.
- **Chapter 11 (DIP)**: DIP는 구체적인 것이 아닌 추상적인 것에 의존하라고 한다. LSP를 준수하는 추상 인터페이스가 있어야 DIP가 의미를 가진다.
- **Chapter 22 (클린 아키텍처)**: 클린 아키텍처의 각 계층 경계에서 인터페이스를 통해 통신하는데, 이때 LSP 준수가 전제 조건이다.

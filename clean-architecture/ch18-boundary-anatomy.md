# Chapter 18: Boundary Anatomy (경계 해부학)

## 핵심 질문

아키텍처 경계는 물리적으로 어떤 형태를 띠는가? 단일체(모노리스), 배포형 컴포넌트, 로컬 프로세스, 서비스라는 네 가지 경계 형태는 각각 어떤 특성을 가지며, 경계를 횡단할 때 통신 비용은 어떻게 달라지는가?

---

## 1. 시스템 아키텍처와 경계

시스템 아키텍처는 일련의 소프트웨어 컴포넌트와 그 컴포넌트들을 분리하는 **경계**에 의해 정의된다. 이러한 경계는 다양한 형태로 나타난다.

---

## 2. 경계 횡단하기

'런타임에 경계를 횡단한다' 함은 그저 경계 한쪽에 있는 기능에서 반대편 기능을 호출하여 **데이터를 전달하는 일**에 불과하다.

적절한 위치에서 경계를 횡단하게 하는 비결은 **소스 코드 의존성 관리**에 있다. 소스 코드 모듈 하나가 변경되면, 이에 의존하는 다른 소스 코드 모듈도 변경하거나 다시 컴파일해서 새로 배포해야 할 수도 있기 때문이다. 경계는 이러한 **변경이 전파되는 것을 막는 방화벽**을 구축하고 관리하는 수단으로서 존재한다.

---

## 3. 두려운 단일체 (Monolith)

### 3.1 특성

아키텍처 경계 중에서 **가장 단순하며 가장 흔한** 형태다. 물리적으로 엄격하게 구분되지 않으며, 함수와 데이터가 단일 프로세서에서 같은 주소 공간을 공유하고 그저 나름의 규칙에 따라 분리되어 있을 뿐이다. 16장에서 이를 **소스 수준 분리 모드**라고 불렀다.

배포 관점에서는 정적으로 링크된 C/C++ 프로젝트, 실행 가능한 jar 파일로 묶인 자바 클래스, 단일 .EXE 파일로 묶인 .NET 바이너리 등으로 나타난다.

### 3.2 경계의 의미

배포 관점에서 단일체는 경계가 드러나지 않는다.(*단일체는 컴포넌트 수준으로 분리되지 않으므로 배포 시 개별 컴포넌트를 배포하는 대신 하나의 파일을 배포한다. 따라서 경계가 드러나지 않는다.*) 하지만 그렇다고 경계가 존재하지 않거나 무의미하다는 뜻은 아니다. 최종적으로 정적으로 링크된 단일 실행 파일을 만들더라도, 그 안에 포함된 다양한 컴포넌트를 **독립적으로 개발하고 바이너리로 만드는 과정**을 수행할 수 있게 하는 일은 대단히 가치 있다.

### 3.3 동적 다형성과 의존성 관리

이러한 아키텍처는 거의 모든 경우에 **동적 다형성**(*모노리틱 시스템에서는 정적 다형성(제네릭, 템플릿 등)을 사용하여 의존성을 관리하기도 한다. 그러나 동적 다형성은 재컴파일이나 재배포가 필요한 경우를 대체로 방지할 수 있다.*)에 의존하여 내부 의존성을 관리한다. 이것이 바로 최근 수십 년 동안 객체 지향 개발이 아주 중요한 패러다임이 될 수 있었던 이유다. 객체 지향이 없었다면, 아키텍트는 결합도를 적절히 분리하기 위해 **함수를 가리키는 포인터**라는 위험한 관행에 기대야만 했을 것이다.

### 3.4 두 가지 경계 횡단 패턴

**패턴 1 — 저수준 → 고수준 (동일 방향)**

```
          경계
            ┃
 Client ───┃──→ Service [+f]
    │       ┃       │
    │       ┃       ▼
    └───────┃──→ <<DS>> Data
            ┃
```

제어흐름은 왼쪽에서 오른쪽으로 경계를 횡단한다. Client는 Service의 함수 f()를 호출하며 Data 인스턴스를 전달한다. 런타임 의존성과 컴파일타임 의존성은 **모두 같은 방향**(저수준 → 고수준)으로 향한다. Data에 대한 정의가 **호출되는 쪽**에 위치한다.

**패턴 2 — 고수준 → 저수준 (의존성 역전)**

```
                    경계
                      ┃
 Client ──→ <<I>>     ┃    Service
            Service ◀─┃─── Impl
            [+f]      ┃    [+f]
    │                 ┃
    ▼                 ┃
 <<DS>> Data ◀────────┃────┘
                      ┃
```

제어흐름은 이전과 마찬가지로 왼쪽에서 오른쪽으로 경계를 횡단한다. 고수준 Client는 Service 인터페이스를 통해 저수준인 ServiceImpl의 함수 f()를 호출한다. 하지만 **의존성은 모두 오른쪽에서 왼쪽으로**, 즉 고수준 컴포넌트를 향한다. 데이터 구조의 정의가 **호출하는 쪽**에 위치한다.

> **핵심 통찰**: 정적 링크된 모노리틱 구조의 실행 파일이라도 이처럼 규칙적인 방식으로 구조를 분리하면 프로젝트를 개발, 테스트, 배포하는 작업에 큰 도움이 된다. 팀들은 서로의 영역에 침범하지 않은 채 자신만의 컴포넌트를 독립적으로 작업할 수 있다.

### 3.5 통신 비용

단일체에서 컴포넌트 간 통신은 **매우 빠르고 값싸다.** 통신은 전형적인 함수 호출에 지나지 않기 때문이다. 소스 수준에서 결합이 분리되면 경계를 가로지르는 통신은 상당히 빈번할 수 있다.

---

## 4. 배포형 컴포넌트 (Deployment Components)

### 4.1 특성

아키텍처의 경계가 물리적으로 드러날 수 있는 가장 단순한 형태는 **동적 링크 라이브러리**다:

| 플랫폼 | 형태 |
|--------|------|
| .NET | DLL |
| Java | jar 파일 |
| Ruby | Gem |
| Unix | 공유 라이브러리 (.so) |

컴포넌트를 이 형태로 배포하면 따로 컴파일하지 않고 곧바로 사용할 수 있다. 이는 **배포 수준 결합 분리 모드**에 해당한다.

### 4.2 단일체와의 차이

배포 과정에서만 차이가 날 뿐, 배포 수준의 컴포넌트는 단일체와 동일하다:
- 일반적으로 모든 함수가 동일한 프로세서와 주소 공간에 위치한다
- 컴포넌트를 분리하거나 의존성을 관리하는 전략도 단일체와 동일하다
- 경계를 가로지르는 통신은 순전히 함수 호출에 지나지 않으므로 **매우 값싸다**

단, 배포형 컴포넌트에서는 **정적 다형성을 사용할 수 없다.**

---

## 5. 스레드 (Threads)

스레드는 아키텍처 경계도 아니며 배포 단위도 아니다. 스레드는 **실행 계획과 순서를 체계화하는 방법**에 가깝다:

- 모든 스레드가 단 하나의 컴포넌트에 포함될 수도 있다
- 많은 컴포넌트에 걸쳐 분산될 수도 있다

단일체와 배포형 컴포넌트 모두 스레드를 활용할 수 있다.

---

## 6. 로컬 프로세스 (Local Processes)

### 6.1 특성

훨씬 강한 물리적 형태를 띠는 아키텍처 경계다:
- 주로 명령행이나 그와 유사한 시스템 호출을 통해 생성된다
- 동일한 프로세서 또는 하나의 멀티코어 시스템에 속한 여러 프로세서에서 실행되지만, 각각 **독립된 주소 공간**에서 실행된다
- 일반적으로 메모리 보호를 통해 프로세스들이 메모리를 공유하지 못하게 한다

### 6.2 통신

로컬 프로세스는 소켓(socket), 메일박스(mailbox), 메시지 큐(message queue) 등 운영체제에서 제공하는 통신 기능을 이용하여 서로 통신한다.

### 6.3 내부 구성

각 로컬 프로세스는 다음 중 하나로 구성될 수 있다:

| 형태 | 특성 |
|------|------|
| 정적으로 링크된 단일체 | 여러 모노리틱 프로세스가 같은 컴포넌트를 포함할 수 있다 (물리적으로 복사됨) |
| 동적으로 링크된 여러 컴포넌트 | 배포형 컴포넌트들을 서로 공유할 수 있다 |

로컬 프로세스는 일종의 **최상위 컴포넌트**라고 생각하면 된다. 내부적으로는 동적 다형성을 통해 의존성을 관리하는 저수준 컴포넌트로 구성된다.

### 6.4 분리 전략

소스 코드 의존성의 화살표는 단일체나 바이너리 컴포넌트와 동일한 방향으로 경계를 횡단한다 — **항상 고수준 컴포넌트를 향한다.** 따라서:
- 고수준 프로세스의 소스 코드가 저수준 프로세스의 이름, 물리 주소, 레지스트리 조회 키를 절대로 포함해서는 안 된다
- **저수준 프로세스가 고수준 프로세스의 플러그인**이 되도록 만드는 것이 아키텍처 관점의 목표다

### 6.5 통신 비용

로컬 프로세스 경계를 지나는 통신에는 운영체제 호출, 데이터 마샬링 및 언마샬링, 프로세스 간 문맥 교환 등이 포함되며 **제법 비싼 작업**에 속한다. 따라서 통신이 너무 빈번하게 이뤄지지 않도록 신중하게 제한해야 한다.

---

## 7. 서비스 (Services)

### 7.1 특성

물리적인 형태를 띠는 **가장 강력한** 경계다:
- 서비스는 프로세스로, 일반적으로 명령행 또는 그와 동등한 시스템 호출을 통해 구동된다
- **자신의 물리적 위치에 구애받지 않는다** — 동일한 프로세서에서 동작할 수도, 다른 머신에서 동작할 수도 있다
- 모든 통신이 **네트워크를 통해** 이뤄진다고 가정한다

### 7.2 통신 비용

서비스 경계를 지나는 통신은 함수 호출에 비해 **매우 느리다.** 전체 소요 시간이 수십 밀리초에서 수 초까지 걸릴 수 있다. 따라서:
- 가능하다면 빈번하게 통신하는 일을 피해야 한다
- **지연(latency)**에 따른 문제를 고수준에서 처리할 수 있어야 한다

### 7.3 분리 전략

로컬 프로세스에 적용한 규칙이 서비스에도 그대로 적용된다:
- 저수준 서비스는 반드시 고수준 서비스에 **'플러그인'**되어야 한다
- 고수준 서비스의 소스 코드에는 저수준 서비스를 특정 짓는 어떤 물리적인 정보(예: URI)도 절대 포함해서는 안 된다

## 8. 경계 형태 비교

| 경계 형태 | 물리적 분리 | 통신 방식 | 통신 비용 | 배포 |
|-----------|-----------|----------|----------|------|
| **단일체** | 없음 (같은 주소 공간) | 함수 호출 | 매우 낮음 | 단일 실행 파일 |
| **배포형 컴포넌트** | 동적 링크 | 함수 호출 | 매우 낮음 | DLL, jar 등 |
| **스레드** | 없음 (경계가 아님) | — | — | — |
| **로컬 프로세스** | 독립된 주소 공간 | 소켓, 메시지 큐 등 | 보통 | 별도 프로세스 |
| **서비스** | 네트워크 분리 가능 | 네트워크 패킷 | 높음 | 독립적 배포 |

---

## 9. 한 시스템 안의 경계 혼합

단일체를 제외한 대다수의 시스템은 **한 가지 이상의 경계 전략**을 사용한다:

- 서비스 경계를 활용하는 시스템이라면 로컬 프로세스 경계도 일부 포함하고 있을 수 있다
- 서비스는 상호작용하는 일련의 로컬 프로세스 **퍼사드(Facade)**에 불과할 때가 많다
- 개별 서비스 또는 로컬 프로세스는 거의 언제나 소스 코드 컴포넌트로 구성된 단일체이거나, 동적으로 링크된 배포형 컴포넌트의 집합이다

> **핵심 통찰**: 한 시스템 안에서도 통신이 빈번한 로컬 경계와 지연을 중요하게 고려해야 하는 경계가 혼합되어 있다. 아키텍트는 각 경계의 비용과 특성을 이해하고 적절한 형태를 선택해야 한다.

---

## 요약

- 아키텍처 경계는 **다양한 물리적 형태**로 나타난다 — 단일체, 배포형 컴포넌트, 로컬 프로세스, 서비스
- 경계 횡단의 핵심은 **소스 코드 의존성 관리**에 있다
- 단일체에서도 다형성을 활용한 경계는 **개발·테스트·배포에 큰 도움**이 된다
- 경계를 가로지르는 **통신 비용**은 형태에 따라 크게 달라진다 — 함수 호출 (거의 무료) vs 네트워크 통신 (매우 비쌈)
- 스레드는 아키텍처 경계가 **아니다** — 실행 계획을 체계화하는 방법일 뿐이다
- 모든 형태의 경계에서 의존성 방향은 동일하다 — **항상 고수준 컴포넌트를 향한다**
- 대다수의 실제 시스템은 **한 가지 이상의 경계 전략**을 혼합하여 사용한다

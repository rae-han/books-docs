# Chapter 29: Clean Embedded Architecture (클린 임베디드 아키텍처)

*제임스 그레닝(James Grenning) 기고*

## 핵심 질문

소프트웨어와 펌웨어의 차이는 무엇이며, 왜 임베디드 코드가 펌웨어로 전락하는가? 임베디드 시스템에서 클린 아키텍처 원칙을 어떻게 적용하여 타깃-하드웨어 병목현상을 줄이고, 소프트웨어의 유효 수명을 늘릴 수 있는가? HAL, PAL, OSAL이란 무엇이며, 이들은 어떤 역할을 하는가?

## 1. 소프트웨어와 펌웨어

더그 슈미트(Doug Schmidt)는 자신의 블로그에서 다음과 같이 주장했다:

> "소프트웨어는 닳지 않지만, 펌웨어와 하드웨어는 낡아 가므로 결국 소프트웨어도 수정해야 한다."

이 문장에서 두 가지 핵심 용어의 차이가 명확해진다:

| 용어 | 정의 | 특성 |
|------|------|------|
| **소프트웨어(Software)** | 긴 시간 유용하게 쓸 수 있는 것 | 닳지 않는다 |
| **펌웨어(Firmware)** | 하드웨어가 발전할수록 낡아 갈 것 | 하드웨어에 의존하여 수명이 짧다 |

그레닝은 더그의 주장에 다음과 같이 덧붙인다:

> **핵심 통찰**: 소프트웨어는 닳지 않지만, 펌웨어와 하드웨어에 대한 의존성을 관리하지 않으면 **안으로부터 파괴될 수 있다.** 잠재적으로 오래 살아남을 수 있던 임베디드 소프트웨어가 하드웨어 의존성에 오염되는 바람에 짧게 삶을 마감하는 일은 드물지 않다.

### 1.1 펌웨어에 대한 새로운 정의

통상적으로 받아들여지던 펌웨어 정의들은 다음과 같다:

- "ROM, EPROM 혹은 플래시 메모리 같은 비휘발성 메모리에 유지된다"
- "하드웨어 장치에 프로그래밍된 소프트웨어 프로그램 혹은 명령어 집합이다"
- "개별 하드웨어에 내장되는 소프트웨어다"
- "읽기 전용 메모리(ROM)에 쓰여진 소프트웨어다"

하지만 더그의 표현은 이러한 전통적 정의가 틀렸거나, 최소한 한물갔다는 사실을 깨닫게 해준다. **ROM에 상주하는 코드만이 펌웨어는 아니다.** 저장되는 위치가 펌웨어를 정의하지는 않는다. 이보다는 **무엇에 의존하는지**, 그리고 **하드웨어 발전에 맞춰 수정하기가 얼마나 어려운지**에 따라 정의된다.

### 1.2 펌웨어는 임베디드만의 문제가 아니다

임베디드 엔지니어가 아닌 엔지니어들 또한 펌웨어를 작성한다:

- 코드에 **SQL을 심어 놓는** 경우
- 개발하는 코드 전반에 **플랫폼 의존성을 퍼뜨려 놓는** 경우
- 안드로이드 앱 개발자가 **업무 로직을 안드로이드 API로부터 분리하지 않는** 경우

이 모든 경우에 본질적으로 펌웨어를 작성하는 셈이다. 엔지니어와 프로그래머에게 전하는 메시지는 분명하다: **펌웨어를 수없이 양산하는 일을 멈추고, 코드에게 유효 수명을 길게 늘릴 수 있는 기회를 주어라.**

### 1.3 실무 사례: TDM에서 VoIP로의 전환

1990년대 후반에 통신 관련 서브시스템을 재설계하는 작업을 하고 있었다. 시분할 다중화(TDM) 방식을 음성 인터넷 프로토콜(VoIP) 방식으로 전환하는 시스템이었다. 특정 상황에서의 호(call)(*call — 전화교환망에서 개개의 통신이 통신설비를 일시적으로 점유하는 것*)를 어떻게 처리해야 하는지를 시스템 엔지니어에게 물을 때마다, 엔지니어는 "현재 제품의 코드에서" 답변을 얻었다. 복잡하게 뒤얽힌 레거시 코드가 새 제품의 명세서였던 것이다.

기존 구현에서는 TDM과 전화를 거는 업무 로직이 **전혀 분리되어 있지 않았다.** 전체 제품은 가장 윗단부터 밑바닥까지 모두 하드웨어/기술 의존적이었고, 엉킨 것을 풀 수 없었다. **전체 제품이 본질적으로 펌웨어가 되어버린 것이다.**

### 1.4 실무 사례: 메시지 처리기와 UART

또 다른 예로, 명령어 메시지가 직렬 포트를 거쳐 시스템에 도착했다. 메시지 처리기/발송기가 메시지의 포맷을 알고 있었고, 파싱할 수 있었으며, 해당 요청을 처리하는 코드로 메시지를 발송할 수 있었다. 여기까지는 놀랄 일이 전혀 없었다.

하지만 메시지 처리기/발송기가 **UART(*UART — 직렬 포트를 제어하는 하드웨어 장비*)** 하드웨어와 상호작용하는 코드와 동일한 파일에 있다는 사실이 문제였다. 메시지 처리기는 UART의 세부사항으로 인해 오염되었다. 메시지 처리기는 오랜 기간 유용할 수 있는 잠재력이 있었지만, 이를 포기하고 펌웨어가 되었다.

## 2. 앱-티튜드 테스트 (App-titude Test)

### 2.1 왜 소프트웨어가 펌웨어로 변하는가

왜 잠재적인 임베디드 소프트웨어는 그렇게도 많이 펌웨어로 변하는가? 임베디드 코드가 **동작하게 만드는 데 대부분의 노력을 집중**하고, **오랫동안 유용하게 남도록 구조화하는 데는 그리 신경 쓰지 않기** 때문으로 보인다.

켄트 벡(Kent Beck)은 소프트웨어를 구축하는 세 가지 활동을 다음과 같이 기술했다:

| 단계 | 설명 | 그레닝의 해설 |
|------|------|-------------|
| 1. **먼저 동작하게 만들어라** | 소프트웨어가 동작하지 않는다면 사업은 망한다 | 기본 요건 |
| 2. **그리고 올바르게 만들어라** | 코드를 리팩터링해서 이해할 수 있게 만들고, 요구 변경 시 개선할 수 있게 만들어라 | 구조화 |
| 3. **그리고 빠르게 만들어라** | 코드를 리팩터링해서 '요구되는' 성능을 만족시켜라 | 최적화 |

현장에서 지켜본 수많은 임베디드 시스템 소프트웨어는 **"동작하게 하라"는 활동만을 염두에 두고 작성된 것**처럼 보인다. 또는 "빠르게 만들어라"라는 목표에도 집착하는데, 이는 아주 세세한 최적화를 틈만 나면 수행해야 달성되는 목표다.

프레드 브룩스(Fred Brooks)는 "버리기 위한 계획을 세우라"고 제안했다. 켄트와 프레드는 사실상 똑같은 충고를 하고 있다. **동작하는 것을 배워라. 그러고 나서 더 나은 해결책을 만들어라.**

### 2.2 앱-티튜드 테스트의 실패 사례

앱이 동작하도록 만드는 것을 **앱-티튜드 테스트(App-titude test)**라고 부른다. 프로그래머가 오직 앱이 동작하도록 만드는 일만 신경 쓴다면 자신의 제품과 고용주에게 해를 끼치는 것이다.

다음은 작은 임베디드 시스템의 한 파일에서 발견한 함수 목록을 분류한 것이다:

**도메인 로직을 포함하는 함수들:**
```c
float calc_RPM(void) { ... }
void Do_Average(void) { ... }
void GetNext_Measurement(void) { ... }
void ZeroSensor_1(void) { ... }
void ZeroSensor_2(void) { ... }
```

**하드웨어 플랫폼을 설정하는 함수들:**
```c
ISR(TIMER1_vect) { ... }
ISR(INT2_vect) { ... }
void uC_Sleep(void) { ... }
```

**전원 버튼에 반응하는 함수들:**
```c
void btn_Handler(void) { ... }
void Dev_Control(char Activation) { ... }
```

**하드웨어로부터 A/D 입력을 읽어 들이는 함수:**
```c
static char Read_RawData(void) { ... }
```

**영구 저장소에 값을 저장하는 함수들:**
```c
char LoadFLASH_Setup(void) { ... }
void SaveFLASH_Setup(void) { ... }
void Store_DataSet(void) { ... }
float bytes2float(char bytes[4]) { ... }
void Recall_DataSet(void) { ... }
```

이 모든 함수가 **하나의 파일**에 뒤섞여 있었다. 코드를 이해하는 데 방해가 되는 장애물이 많았고, 특정 임베디드 장치에서만 테스트할 수 있는 파일 구조를 포함하고 있었다. 코드의 거의 모든 부분이 특수한 마이크로프로세서 아키텍처에서 동작한다는 사실을 알고 있었다. 이 애플리케이션은 동작하지만, **클린 임베디드 아키텍처를 가진다고 말하기는 어렵다.**

## 3. 타깃-하드웨어 병목현상 (Target-Hardware Bottleneck)

임베디드 개발자들은 임베디드가 아니었다면 다루지 않아도 될 특수한 관심사를 많이 가지고 있다:

- 제한된 메모리 공간
- 실시간성 제약과 처리완료 시간
- 제한된 입출력
- 특이한 사용자 인터페이스
- 여러 센서와 실제 세상과의 상호작용

거의 모든 경우에 하드웨어, 소프트웨어, 펌웨어가 동시에 만들어진다. 개발 시점에 코드를 실행할 대상조차 없을 수 있고, 하드웨어가 준비되더라도 하드웨어 자체에 결함이 있을 가능성이 높다.

임베디드 코드가 클린 아키텍처 원칙과 실천법을 따르지 않고 작성된다면, 대개의 경우 **코드를 테스트할 수 있는 환경이 해당 특정 타깃으로 국한**될 것이다. 그리고 그 타깃이 테스트가 가능한 유일한 장소라면 **타깃-하드웨어 병목현상**이 발생하여 진척이 느려질 것이다.

> **핵심 통찰**: 클린 임베디드 아키텍처는 테스트하기 쉬운 임베디드 아키텍처다.

## 4. 계층 (Layers)

### 4.1 세 개의 기본 계층

먼저 세 개의 계층부터 시작하자:

```
┌──────────────┐
│   소프트웨어   │
├──────────────┤
│    펌웨어     │
├──────────────┤
│    하드웨어    │
└──────────────┘
```

*그림 29.1 세 개의 계층*

맨 하단에는 하드웨어가 있다. 하드웨어는 기술의 발전과 무어의 법칙에 따라 변할 것이다. 부품은 낡게 되고, 새로운 부품은 적은 전력을 사용하면서도 더 나은 성능을 제공하며, 더 저렴할 것이다.

### 4.2 하드웨어와 펌웨어의 분리

최소한 하드웨어가 정의된 이후라면 하드웨어와 나머지 시스템 사이의 분리는 주어진다:

```
┌──────────────┐
│  펌웨어 (오염됨) │  ← 하드웨어 관련 정보가 코드 전체를 오염
├──────────────┤
│    하드웨어    │
└──────────────┘
```

*그림 29.2 하드웨어는 시스템의 나머지 부분으로부터 반드시 분리되어야 한다*

이 상태에서 앱-티튜드 테스트를 해보면 대체로 문제가 발생한다. 하드웨어 관련 정보가 코드 전체를 오염시키지 못하게 막을 방법이 전혀 없다.

**소프트웨어와 펌웨어가 서로 섞이는 일은 안티 패턴(anti-pattern)이다.** 이 안티 패턴을 보이는 코드는:

- 변화에 저항하게 된다
- 변경하기 어려울 뿐 아니라 변경하는 일 자체가 위험을 수반한다
- 가벼운 변경에도 시스템 전체를 대상으로 회귀 테스트를 전부 실행해야 한다
- 기기를 외부적으로 테스트할 수 있게 구성하지 않았다면 지루한 수동 테스트를 비켜갈 방법이 없다

## 5. 하드웨어는 세부사항이다

소프트웨어와 펌웨어 사이의 경계는 코드와 하드웨어 사이의 경계와는 달리 잘 정의하기가 대체로 힘들다:

```
┌──────────────┐
│   소프트웨어   │  ← 경계가 흐릿함
├┄┄┄┄┄┄┄┄┄┄┄┄┄┄┤
│    펌웨어     │
├──────────────┤
│    하드웨어    │
└──────────────┘
```

*그림 29.3 소프트웨어와 펌웨어 사이의 경계는 코드와 하드웨어 사이의 경계보다는 조금 흐릿하다*

### 5.1 하드웨어 추상화 계층 (HAL)

임베디드 소프트웨어 개발자가 해야 할 일 하나는 이 경계를 분명하게 만드는 것이다. 소프트웨어와 펌웨어 사이의 경계는 **하드웨어 추상화 계층(Hardware Abstraction Layer, HAL)**이라고 부른다:

```
        ┌──────────────┐
 HAL →  │   소프트웨어   │
        ├──────────────┤
        │    펌웨어     │
        ├──────────────┤
        │    하드웨어    │
        └──────────────┘
```

*그림 29.4 하드웨어 추상화 계층*

HAL은 자신보다 위에 있는 소프트웨어를 위해 존재하므로, **HAL의 API는 소프트웨어의 필요에 맞게** 만들어져야 한다.

**플래시 메모리 예시:**

| 계층 | 제공하는 API | 추상화 수준 |
|------|------------|-----------|
| 펌웨어 | 바이트 또는 바이트의 배열을 플래시 메모리에 저장 | 저수준 (하드웨어 관점) |
| HAL | 이름/값 쌍을 영속성 장치에 저장하거나 읽기 | 고수준 (소프트웨어 관점) |

소프트웨어는 이름/값 쌍이 플래시 메모리에 저장되는지, 하드디스크에 저장되는지, 클라우드에 저장되는지를 전혀 개의치 않는다. 플래시로 구현하느냐는 소프트웨어로부터 반드시 숨겨야 하는 세부사항이다.

**LED 예시:**

| 계층 | 제공하는 API | 추상화 수준 |
|------|------------|-----------|
| 펌웨어/보드 지원 패키지 | `Led_TurnOn(5)` | 저수준 (하드웨어 관점) |
| HAL | `Indicate_LowBattery()` | 고수준 (소프트웨어/제품 관점) |

HAL은 애플리케이션에서 필요로 하는 서비스를 표현한다. 계층은 또 다른 계층을 포함할 수도 있어서, 계층의 수가 정해진 상태로 구성되기보다는 **프랙털(fractal) 패턴**에 더 가깝다. GPIO를 설정하는 일은 소프트웨어로부터 반드시 숨겨야 하는 세부사항이다.

### 5.2 HAL 사용자에게 하드웨어 세부사항을 드러내지 말라

클린 임베디드 아키텍처로 설계된 소프트웨어는 **타깃 하드웨어에 관계없이 테스트가 가능**하다. HAL을 제대로 만들었다면, HAL은 타깃에 상관없이 테스트할 수 있는 경계층 또는 일련의 대체 지점을 제공한다.

## 6. 프로세서는 세부사항이다

### 6.1 프로세서 종속 코드의 문제

임베디드 애플리케이션이 특수한 툴 체인을 사용한다면, 이러한 툴 체인은 "도움이 되는" 헤더 파일을 제공할 때가 많다. 흔히 이런 툴 체인의 컴파일러는 C 언어를 제멋대로 변경하여 해당 업체의 프로세서 기능에 접근할 수 있는 새로운 키워드를 제공하곤 한다. 그렇게 되면 이 코드는 **C처럼 보이겠지만, 더 이상 C가 아니다.**

프로세서 제작 업체가 제공하는 C 컴파일러는 종종 전역 변수처럼 보이는 것들을 제공하여, 프로세서 레지스터, 입출력 포트, 클럭 타이머, 인터럽트 컨트롤러 등에 직접 접근하도록 해준다. 이러한 대상들에 쉽게 접근하는 데는 도움이 되겠지만, **다른 프로세서에서는 컴파일되지 않을 것이고, 심지어 동일한 프로세서라도 다른 컴파일러로는 컴파일하지 못할 수도 있다.**

### 6.2 acmetypes.h 대신 stdint.h를 사용하라

다음은 애크미(ACME) 사의 DSP 제품군용으로 설계된 헤더 파일이다:

```c
#ifndef _ACME_STD_TYPES_
#define _ACME_STD_TYPES_

#if defined(_ACME_X42)
    typedef unsigned int   Uint_32;
    typedef unsigned short Uint_16;
    typedef unsigned char  Uint_8;
    typedef int            Int_32;
    typedef short          Int_16;
    typedef char           Int_8;
#elif defined(_ACME_A42)
    typedef unsigned long  Uint_32;
    typedef unsigned int   Uint_16;
    typedef unsigned char  Uint_8;
    typedef long           Int_32;
    typedef int            Int_16;
    typedef char           Int_8;
#else
    #error <acmetypes.h> is not supported for this environment
#endif

#endif
```

이 `acmetypes.h` 헤더 파일은 **절대로 직접 사용해서는 안 된다.** 직접 사용하면 코드가 애크미의 DSP 중 하나에 종속되어 버린다. 이식성을 고려하지 않고 애크미의 헤더를 사용하는 파일을 소수로 제한하지 않는다면 이식 작업이 훨씬 어려워진다.

`acmetypes.h`를 사용하기보다는 더 표준화된 방법인 **`stdint.h`**를 사용해야 한다. 타깃 컴파일러에서 `stdint.h`를 제공하지 않는다면 직접 작성할 수 있다:

```c
#ifndef _STDINT_H_
#define _STDINT_H_

#include <acmetypes.h>
typedef Uint_32 uint32_t;
typedef Uint_16 uint16_t;
typedef Uint_8  uint8_t;
typedef Int_32  int32_t;
typedef Int_16  int16_t;
typedef Int_8   int8_t;

#endif
```

### 6.3 프로세서 직접 접근의 위험

다음 함수는 직렬 출력 포트에 "hi"를 출력하는 실제 코드 기반의 예제다:

```c
void say_hi()
{
    IE = 0b11000000;
    SBUF0 = (0x68);
    while(TI_0 == 0);
    TI_0 = 0;
    SBUF0 = (0x69);
    while(TI_0 == 0);
    TI_0 = 0;
    SBUF0 = (0x0a);
    while(TI_0 == 0);
    TI_0 = 0;
    SBUF0 = (0x0d);
    while(TI_0 == 0);
    TI_0 = 0;
    IE = 0b11010000;
}
```

이 작은 함수에 수많은 문제가 있다:

| 문제 요소 | 설명 |
|----------|------|
| `0b11000000` | 이진 표기법으로, C 표준에서는 사용할 수 없다 |
| `IE` | 인터럽트 활성화 비트 — C 확장 기능으로 프로세서 주변장치에 직접 접근 |
| `SBUF0` | 직렬 출력 버퍼 — 마이크로 컨트롤러 내장 주변장치에 직접 접근 |
| `TI_0` | "직렬 송신 버퍼가 비어 있음" 인터럽트 — 하드웨어 종속 |

클린 임베디드 아키텍처라면 이들 장치 접근 레지스터를 직접 사용하는 코드는 **소수의, 순전히 펌웨어로만 한정**시켜야 한다. 이들 레지스터를 알고 있는 것은 모두 펌웨어가 되어야 하며, 따라서 실리콘 칩에 종속된다.

### 6.4 프로세서 추상화 계층 (PAL)

이러한 마이크로 컨트롤러를 사용할 때, 펌웨어가 저수준 함수들을 **프로세서 추상화 계층(Processor Abstraction Layer, PAL)**의 형태로 격리시켜줄 수 있다. PAL 상위에 위치하는 펌웨어는 타깃-하드웨어에 관계없이 테스트할 수 있게 되어, 펌웨어 자체도 덜 딱딱해(firm)질 수 있다.

## 7. 운영체제는 세부사항이다

HAL은 필수적이다. 하지만 HAL만으로 충분한가? 베어메탈(bare-metal)(*bare-metal — 운영체제가 없는 임베디드 시스템을 뜻한다.*) 임베디드 시스템이라면 HAL만으로도 충분할 것이다. 하지만 RTOS(Real-Time Operating System)를 사용하거나, 임베디드 버전의 리눅스/윈도우를 사용한다면?

```
┌──────────────┐
│   소프트웨어   │
├──────────────┤
│      OS      │
├──────────────┤
│    펌웨어     │
├──────────────┤
│    하드웨어    │
└──────────────┘
```

*그림 29.5 운영체제를 추가*

작성한 코드의 수명을 늘리려면, 무조건 **운영체제를 세부사항으로 취급하고 운영체제에 의존하는 일을 막아야** 한다.

### 7.1 OS 직접 의존의 위험

소프트웨어는 운영체제를 통해 운영 환경이 제공하는 서비스에 접근한다. OS는 소프트웨어를 펌웨어로부터 분리하는 계층이다. 하지만 OS를 직접 사용하면 다음과 같은 위험이 생긴다:

- RTOS 제공업체를 다른 기업에서 인수하여 **OS 사용료가 오르거나 OS 품질이 낮아지는** 경우
- 요구사항이 달라졌는데, 당장 필요한 기능을 **현재 사용 중인 RTOS에서 제공하지 않는** 경우

이 경우 수많은 코드를 변경해야 할 것이다. 새로운 OS의 API에 맞게 간단히 구문만 고치는 정도로 그치지 않고, **새로운 OS가 제공하는 기능과 시스템 명령어에 맞게 의미 자체를 조정**해야 할 가능성이 높다.

### 7.2 운영체제 추상화 계층 (OSAL)

클린 임베디드 아키텍처는 **운영체제 추상화 계층(Operating System Abstraction Layer, OSAL)**을 통해 소프트웨어를 운영체제로부터 격리시킨다:

```
        ┌──────────────┐
 OSAL → │   소프트웨어   │
        ├──────────────┤
 HAL  → │      OS      │
        ├──────────────┤
        │    펌웨어     │
        ├──────────────┤
        │    하드웨어    │
        └──────────────┘
```

*그림 29.6 운영체제 추상화 계층*

OSAL 구현은 **함수 이름을 바꿔주는 정도로 단순할 수도 있고**, 여러 함수를 하나로 묶어야 할 수도 있다.

소프트웨어를 특정 RTOS에서 다른 RTOS로 이식할 때, 소프트웨어가 OS에 직접적으로 의존하는 대신 OSAL에 의존한다면 **이식 작업의 대부분은 기존 OSAL과 호환되도록 새로운 OSAL을 작성하는 데 소요**될 것이다. 기존의 복잡한 코드 덩어리를 수정하는 일과, 정의된 인터페이스와 행위에 맞게 새로운 코드를 작성하는 일 중 어떤 일을 하고 싶은가?

OSAL은 OS를 사용하는 데 따른 수많은 중복이 격리되어 있는 장소다. OSAL을 정의하는 일은 결국 애플리케이션에서도 공통 구조를 가지도록 힘쓰는 일이기도 하다. 예를 들어 메시지 전달 메커니즘을 만들어 두어, 모든 스레드가 동시성 모델을 일일이 처리하지 않도록 할 수도 있다.

OSAL은 **테스트 지점을 만드는 데 도움**이 되며, 그 덕분에 소프트웨어 계층의 귀중한 애플리케이션 코드를 타깃이나 OS에 관계없이 테스트할 수 있게 된다.

## 8. 인터페이스를 통하고 대체 가능성을 높이는 방향으로 프로그래밍하라

HAL을 추가하거나 때로는 OSAL을 추가해야 할 뿐만 아니라, 모든 주요 계층(소프트웨어, OS, 펌웨어, 하드웨어) 내부에는 이 책에서 설명한 원칙들을 적용할 수 있다(아니, 해야 한다). 이들 원칙은 관심사를 분리시키고, 인터페이스를 활용하며, 대체 가능성을 높이는 방향으로 프로그래밍하도록 유도한다.

**계층형 아키텍처(layered architecture)**는 인터페이스를 통해 프로그래밍하자는 발상을 기반으로 한다. 모듈들이 서로 인터페이스를 통해 상호작용한다면 특정 서비스 제공자를 다른 제공자로 대체할 수 있다.

경험법칙에 따르면 인터페이스 정의는 헤더 파일에 해야 한다. 하지만 이 경우 헤더 파일에 무엇을 포함시켜야 할지는 신중하게 정해야 한다:

| 헤더 파일에 포함해야 할 것 | 헤더 파일에 포함하면 안 되는 것 |
|------------------------|--------------------------|
| 함수 선언 | 구현체에서만 필요한 데이터 구조 |
| 함수에서 사용하는 상수 | 구현체에서만 필요한 상수 |
| 구조체 이름 | 구현체에서만 필요한 타입 정의 |

오직 구현체에서만 필요한 세부사항으로 인터페이스 헤더 파일을 어지럽히지 말라. 이는 단순히 어수선해지는 문제로 끝나지 않고, 결국 **원치 않는 의존성을 만들어낼** 것이다. 구현 세부사항의 가시성을 제한하라. 세부사항을 알고 있는 부분이 적을수록 추적하고 변경해야 할 코드도 적어진다.

클린 임베디드 아키텍처에서는 모듈들이 인터페이스를 통해 상호작용하기 때문에 **각각의 계층 내부에서 테스트가 가능**하다.

## 9. DRY 원칙: 조건부 컴파일 지시자를 반복하지 말라

자주 간과되지만, 대체성을 활용하는 사례 하나를 들어보겠다. 임베디드 C/C++ 프로그램이 다양한 타깃과 운영체제를 다루는 방식과 관련 있다. 이런 경우는 주로 **조건부 컴파일**을 사용해서 특정 코드 블록을 활성화하거나 비활성화한다.

특히 기억에 남는 문제 사례는 통신 관련 애플리케이션이었는데, **`#ifdef BOARD_V2` 구문이 수천 번이나 사용**되었다.

이처럼 코드를 반복하는 일은 **반복하지 말라(Don't Repeat Yourself, DRY)(*Don't Repeat Yourself — 《실용주의 프로그래머》(인사이트, 2014)에서 제시된 원칙*)** 원칙을 위배한다. `#ifdef BOARD_V2`가 한 번만 나타난다면 문제가 되지 않는다. 하지만 **6,000번이라면 엄청난 문제**다.

만약 **하드웨어 추상화 계층**이 있다면 어떨까? 하드웨어 유형은 HAL 뒤에 가려진 세부사항이 될 것이다. 만약 이 HAL이 조건부 컴파일 대신 사용할 수 있는 일련의 인터페이스를 제공한다면, 우리는 **링커 또는 어떤 형태의 실시간 바인딩**을 사용해서 소프트웨어를 하드웨어와 연결할 수 있다.

## 요약

- **소프트웨어는 닳지 않지만, 펌웨어는 하드웨어와 함께 낡아 간다.** 하드웨어에 대한 의존성을 관리하지 않으면 소프트웨어도 안으로부터 파괴될 수 있다.
- **펌웨어의 정의는 저장 위치가 아니라 의존성으로 결정된다.** ROM에 상주하는 코드만이 펌웨어가 아니라, 하드웨어에 의존하여 수정하기 어려운 모든 코드가 펌웨어다.
- **임베디드가 아니어도 펌웨어를 작성할 수 있다.** SQL을 코드에 심어 놓거나 플랫폼 의존성을 퍼뜨리면 본질적으로 펌웨어를 작성하는 셈이다.
- **앱-티튜드 테스트만으로는 부족하다.** "동작하게 만들어라"는 첫 번째 단계일 뿐이다. "올바르게 만들어라", "빠르게 만들어라"까지 나아가야 한다.
- **타깃-하드웨어 병목현상을 피하라.** 코드를 테스트할 수 있는 환경이 특정 타깃으로 국한되면 개발 진척이 느려진다.
- **HAL(하드웨어 추상화 계층)**: 소프트웨어와 펌웨어 사이의 경계를 분명하게 만든다. API는 하드웨어 관점이 아니라 소프트웨어/제품 관점으로 설계한다.
- **PAL(프로세서 추상화 계층)**: 프로세서 종속적인 저수준 함수들을 격리하여, 상위 펌웨어를 타깃에 관계없이 테스트 가능하게 만든다.
- **OSAL(운영체제 추상화 계층)**: 소프트웨어를 운영체제로부터 격리시켜, OS 변경 시 이식 작업을 최소화한다.
- **인터페이스를 통해 프로그래밍하고 대체 가능성을 높여라.** 헤더 파일에는 인터페이스만, 구현 세부사항은 숨겨라.
- **조건부 컴파일 지시자를 반복하지 말라.** `#ifdef`의 반복은 DRY 원칙 위반이며, HAL이 이 문제를 해결한다.

## 다른 챕터와의 관계

- **Chapter 22 (클린 아키텍처)**: 클린 임베디드 아키텍처는 클린 아키텍처의 원칙을 임베디드 도메인에 적용한 것이다. 의존성 규칙, 경계, 추상화 계층의 개념이 HAL/OSAL/PAL로 구체화된다.
- **Chapter 8 (OCP)**: HAL과 OSAL은 개방 폐쇄 원칙의 임베디드 버전이다. 하드웨어나 OS가 변경되더라도 소프트웨어를 수정하지 않고 새로운 구현체를 대체할 수 있다.
- **Chapter 9 (LSP)**: HAL의 구현체를 교체할 수 있으려면 리스코프 치환 원칙을 따라야 한다. 새로운 HAL 구현이 기존 계약을 준수해야 소프트웨어가 정상 동작한다.
- **Chapter 28 (테스트 경계)**: 클린 임베디드 아키텍처의 궁극적 목표는 테스트 가능성이다. HAL, PAL, OSAL이 제공하는 경계층은 타깃에 관계없이 테스트할 수 있는 대체 지점을 제공한다.
- **Chapter 26 (메인 컴포넌트)**: 임베디드 시스템에서도 메인 컴포넌트는 하드웨어 초기화, 의존성 주입, 제어권 위임이라는 동일한 역할을 수행한다. 메인이 펌웨어 영역에 속하면서 소프트웨어에 대한 플러그인 역할을 한다.
- **Chapter 15 (아키텍처란?)**: "소프트웨어는 닳지 않지만 펌웨어와 하드웨어는 낡아 간다"는 이 장의 핵심 메시지는, 아키텍처가 선택사항을 열어 두어야 한다는 15장의 원칙과 직결된다.

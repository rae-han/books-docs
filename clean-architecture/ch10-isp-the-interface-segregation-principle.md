# Chapter 10: ISP: The Interface Segregation Principle (인터페이스 분리 원칙)

## 핵심 질문

왜 사용하지 않는 기능에 의존하면 문제가 발생하는가? "필요 이상으로 많은 것을 포함하는 모듈에 의존하지 말라"는 원칙은 소스 코드 수준을 넘어 아키텍처 수준에서도 적용되는가?

---

## 1. ISP의 기원: 뚱뚱한 클래스 문제

인터페이스 분리 원칙(ISP)은 다음과 같은 상황에서 그 이름이 유래했다.

### 1.1 문제 상황

다수의 사용자가 `OPS` 클래스의 오퍼레이션을 사용한다. `User1`은 오직 `op1`을, `User2`는 `op2`만을, `User3`는 `op3`만을 사용한다고 가정해 보자.

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│  User1  │  │  User2  │  │  User3  │
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     └────────────┼────────────┘
                  ▼
          ┌──────────────┐
          │     OPS      │
          ├──────────────┤
          │ + op1        │
          │ + op2        │
          │ + op3        │
          └──────────────┘
```
*그림 10.1 — 인터페이스 분리 원칙. 세 사용자가 모두 OPS 클래스에 의존한다.*

`OPS`가 정적 타입 언어로 작성된 클래스라고 해보자. 이 경우 `User1`에서는 `op2`와 `op3`를 전혀 사용하지 않음에도 `User1`의 소스 코드는 이 두 메서드에 의존하게 된다.

이러한 의존성으로 인해 **OPS 클래스에서 `op2`의 소스 코드가 변경되면 `User1`도 다시 컴파일한 후 새로 배포해야 한다.** 사실 `User1`과 관련된 코드는 전혀 변경되지 않았음에도 말이다.

### 1.2 해결: 오퍼레이션을 인터페이스 단위로 분리

이러한 문제는 오퍼레이션을 인터페이스 단위로 분리하여 해결할 수 있다.

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│  User1  │    │  User2  │    │  User3  │
└────┬────┘    └────┬────┘    └────┬────┘
     │              │              │
     ▼              ▼              ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│  <<I>>  │  │  <<I>>  │  │  <<I>>  │
│ U1Ops   │  │ U2Ops   │  │ U3Ops   │
├─────────┤  ├─────────┤  ├─────────┤
│ + op1   │  │ + op2   │  │ + op3   │
└────┬────┘  └────┬────┘  └────┬────┘
     △            △            △
     └────────────┼────────────┘
                  │
          ┌───────┴──────┐
          │     OPS      │
          ├──────────────┤
          │ + op1        │
          │ + op2        │
          │ + op3        │
          └──────────────┘
```
*그림 10.2 — 분리된 오퍼레이션. 각 사용자는 자신이 사용하는 오퍼레이션만 포함하는 인터페이스에 의존한다.*

정적 타입 언어로 이 다이어그램을 구현했다고 가정하면, `User1`의 소스 코드는 `U1Ops`와 `op1`에는 의존하지만 `OPS`에는 의존하지 않게 된다. 따라서 **OPS에서 발생한 변경이 `User1`과는 전혀 관계없는 변경이라면, `User1`을 다시 컴파일하고 새로 배포하는 상황은 초래되지 않는다.**

## 2. ISP와 언어

### 2.1 정적 타입 언어 vs 동적 타입 언어

앞의 예제에서 본 사례는 **언어 타입에 의존**한다.

| 언어 유형 | 의존성 선언 | 영향 |
|----------|-----------|------|
| 정적 타입 언어 (Java, C# 등) | `import`, `use`, `include` 등의 타입 선언문 강제 | 소스 코드 의존성 발생 → 재컴파일/재배포 강제 |
| 동적 타입 언어 (Ruby, Python 등) | 소스 코드에 선언문 없음. 런타임에 추론 | 소스 코드 의존성 없음 → 재컴파일/재배포 불필요 |

동적 타입 언어를 사용하면 정적 타입 언어를 사용할 때보다 **유연하며 결합도가 낮은 시스템**을 만들 수 있는 이유가 바로 이 때문이다(*이 부분은 깊이 들어가면 상황이 좀 복잡해진다. 예를 들어 자바는 정적 타입 언어지만, 그림 10.1에서 `op2` 메서드의 시그니처가 변경되면 이를 사용하는 `User2`만 재컴파일하면 된다. 시그니처는 그대로인 채 구현 코드만 변경됐다면 그마저도 다시 컴파일할 필요가 없다. 자바는 비-final, 비-private 인스턴스 변수에 대해 호출할 정확한 메서드를 런타임에 결정하는 늦은 바인딩(late binding)을 수행하기 때문이다.*).

이러한 사실로 인해 ISP를 아키텍처가 아니라, 언어와 관련된 문제라고 결론 내릴 여지가 있다.

## 3. ISP와 아키텍처

### 3.1 근본적인 동기

한 걸음 물러서서 ISP를 사용하는 근본적인 동기를 살펴보면, 잠재되어 있는 더 깊은 우려사항을 볼 수 있다.

> **일반적으로, 필요 이상으로 많은 걸 포함하는 모듈에 의존하는 것은 해로운 일이다.**

소스 코드 의존성의 경우 이는 분명한 사실인데, 불필요한 재컴파일과 재배포를 강제하기 때문이다. 하지만 **더 고수준인 아키텍처 수준에서도 마찬가지 상황이 발생한다.**

### 3.2 아키텍처 수준의 예제

예를 들어 S 시스템 구축에 참여하고 있는 아키텍트가 있다고 해보자. 아키텍트는 F라는 프레임워크를 시스템에 도입하기를 원한다. 그리고 F 프레임워크 개발자는 특정한 D 데이터베이스를 반드시 사용하도록 만들었다고 가정해 보자.

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  System S    │───▶│ Framework F  │───▶│ Database D   │
└──────────────┘    └──────────────┘    └──────────────┘
```
*그림 10.3 — 문제가 있는 아키텍처. S는 F에 의존하고, F는 D에 의존한다.*

F에서는 불필요한 기능, 따라서 S와는 전혀 관계없는 기능이 D에 포함된다고 가정하자. 그 기능 때문에 D 내부가 변경되면, **F를 재배포해야 할 수도 있고, 따라서 S까지 재배포해야 할지 모른다.**

더 심각한 문제는 **D 내부의 기능 중 F와 S에서 불필요한 그 기능에 문제가 발생해도 F와 S에 영향을 준다**는 사실이다.

> **핵심 통찰**: 불필요한 짐을 실은 무언가에 의존하면 예상치도 못한 문제에 빠진다. 이 원칙은 소스 코드 수준뿐 아니라 프레임워크, 데이터베이스, 외부 서비스 등 아키텍처 수준의 모든 의존성에 적용된다.

---

## 4. 결론

여기에서 배울 수 있는 교훈은 **불필요한 짐을 실은 무언가에 의존하면 예상치도 못한 문제에 빠진다**는 사실이다.

이 아이디어는 Chapter 13 "컴포넌트 응집도"에서 **공통 재사용 원칙(Common Reuse Principle, CRP)**을 논할 때 더 자세히 다루어진다.

---

## 요약

- **ISP란** 사용자가 자신이 사용하지 않는 메서드에 의존하도록 강제해서는 안 된다는 원칙이다.
- **인터페이스를 분리하라**: 뚱뚱한 인터페이스 하나보다 사용자별로 분리된 작은 인터페이스 여러 개가 낫다.
- **언어에 따라 영향이 다르다**: 정적 타입 언어에서는 재컴파일/재배포 문제가 심각하지만, 동적 타입 언어에서는 상대적으로 유연하다.
- **아키텍처 수준에서도 적용된다**: 불필요한 기능을 포함하는 프레임워크나 데이터베이스에 의존하면, 그 불필요한 부분의 변경이나 문제가 시스템 전체에 파급된다.
- **핵심 교훈**: 필요 이상으로 많은 것을 포함하는 모듈에 의존하지 말라.

## 다른 챕터와의 관계

- **Chapter 7 (SRP)**: SRP가 "하나의 액터에 대해서만 책임져라"라면, ISP는 "사용하지 않는 것에 의존하지 말라"이다. SRP는 모듈 제공자의 관점, ISP는 모듈 사용자의 관점에서 같은 문제를 바라본다.
- **Chapter 8 (OCP)**: OCP의 정보 은닉 부분에서 "자신이 직접 사용하지 않는 요소에는 절대로 의존해서는 안 된다"는 원칙이 언급되었는데, 이것이 바로 ISP다.
- **Chapter 9 (LSP)**: LSP 위반 시 사용자가 특정 타입을 구분하는 로직을 추가해야 하는데, 이는 ISP 관점에서 사용하지 않는 세부사항에 의존하는 것과 같다.
- **Chapter 13 (컴포넌트 응집도)**: ISP의 컴포넌트 수준 확장인 공통 재사용 원칙(CRP)을 상세히 다룬다. "컴포넌트 사용자를 불필요한 것에 의존하게 만들지 말라"는 원칙이다.
- **Chapter 11 (DIP)**: DIP는 추상에 의존하라고 말하는데, ISP로 인터페이스를 적절히 분리해야 DIP를 효과적으로 적용할 수 있다.

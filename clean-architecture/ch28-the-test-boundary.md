# Chapter 28: The Test Boundary (테스트 경계)

## 핵심 질문

테스트는 시스템의 일부인가, 아닌가? 아키텍처 관점에서 테스트는 어떤 위치를 차지하며, 왜 테스트를 "설계의 범위 밖"으로 여기는 것이 치명적인가? 깨지기 쉬운 테스트 문제(Fragile Tests Problem)는 왜 발생하며, 이를 해결하는 테스트 API란 무엇인가?

---

## 1. 시스템 컴포넌트인 테스트

테스트는 시스템의 일부이며, 아키텍처에도 관여한다. 시스템의 나머지 요소가 아키텍처에 관여하는 것과 동등하게 말이다. 어떤 면에서는 정말 평범하게 관여하고, 또 다른 면에서는 상당히 독특하게 관여한다.

테스트에 관련하여 상당한 혼동이 있다. 테스트는 시스템의 일부인가? 아니면 별개인가? 단위 테스트와 통합 테스트는 서로 다른가? 인수 테스트, 기능 테스트, Cucumber 테스트, TDD 테스트, BDD 테스트, 컴포넌트 테스트 등은 어떤가?

> **핵심 통찰**: 아키텍처 관점에서는 모든 테스트가 동일하다. TDD로 생성한 아주 작은 테스트이든, 대규모의 FitNesse, Cucumber, SpecFlow, JBehave 테스트이든, 이들 테스트는 아키텍처적으로 모두 동등하다.

### 1.1 테스트와 의존성 규칙

테스트는 태생적으로 의존성 규칙을 따른다:

| 특성 | 설명 |
|------|------|
| **의존성 방향** | 테스트는 세부적이며 구체적인 것으로, 의존성은 항상 테스트 대상이 되는 코드를 향한다 |
| **아키텍처 위치** | 테스트는 아키텍처에서 가장 바깥쪽 원으로 생각할 수 있다 |
| **역방향 의존성 없음** | 시스템 내부의 어떤 것도 테스트에는 의존하지 않는다 |
| **배포 독립성** | 테스트는 독립적으로 배포 가능하며, 대다수의 경우 테스트 시스템에만 배포한다 |

### 1.2 테스트의 독특한 위치

테스트는 시스템 컴포넌트 중에서 **가장 고립**되어 있다:

- 테스트가 시스템 운영에 꼭 필요하지는 않다
- 어떤 사용자도 테스트에 의존하지 않는다
- 테스트의 역할은 **운영이 아니라 개발을 지원**하는 데 있다

그렇다고 해서 테스트가 시스템 컴포넌트가 아니라는 뜻은 아니다. 사실 많은 면에서 테스트는 **다른 모든 시스템 컴포넌트가 반드시 지켜야 하는 모델**을 표현해 준다.

---

## 2. 테스트를 고려한 설계

테스트가 지닌 극단적인 고립성이 "테스트는 대체로 배포하지 않는다"는 사실과 어우러지며, 개발자는 종종 테스트가 시스템의 설계 범위 밖에 있다고 여긴다. **이 관점은 치명적이다.**

테스트가 시스템의 설계와 잘 통합되지 않으면:
- 테스트는 **깨지기 쉬워지고**
- 시스템은 **뻣뻣해져서** 변경하기가 어려워진다

### 2.1 깨지기 쉬운 테스트 문제 (Fragile Tests Problem)

시스템에 강하게 결합된 테스트라면 시스템이 변경될 때 함께 변경되어야만 한다. 시스템 컴포넌트에서 생긴 아주 사소한 변경도, 이와 결합된 수많은 테스트를 망가뜨릴 수 있다. 더 심각한 경우, 시스템의 공통 컴포넌트가 변경되면 **수백, 심지어 수천 개의 테스트가 망가진다**. 이 문제가 바로 **깨지기 쉬운 테스트 문제(Fragile Tests Problem)**다.

**예시**: GUI를 사용하여 업무 규칙을 검증하는 테스트 스위트가 있다고 해보자.

```
[로그인 화면] → [페이지 탐색] → [특정 업무 규칙 검사]
```

이러한 테스트들은 로그인 화면에서 시작해서 페이지 구조를 탐색해 나가면서 특정 업무 규칙을 검사한다. 따라서 로그인 페이지나 탐색 구조 어딘가가 달라지면 **엄청난 수의 테스트가 망가진다**.

### 2.2 깨지기 쉬운 테스트의 부작용

깨지기 쉬운 테스트는 **시스템을 뻣뻣하게 만든다**는 부작용을 낳을 때가 많다. 시스템에 가한 간단한 변경이 대량의 테스트 실패로 이어진다는 사실을 알게 되면, 개발자는 그러한 변경을 하지 않으려 들 것이다. 예를 들어 마케팅팀에서 페이지 탐색 구조를 살짝 손봐달라고 요청했는데, 이로 인해 망가지는 테스트가 1,000개라면 두 팀 사이의 대화가 어떻게 될지 상상해 보라.

### 2.3 설계의 첫 번째 규칙

이 문제를 해결하려면 테스트를 고려해서 설계해야 한다. 소프트웨어 설계의 첫 번째 규칙은 언제나 같다:

> **핵심 통찰**: 변동성이 있는 것에 의존하지 말라. GUI는 변동성이 크다. GUI로 시스템을 조작하는 테스트 스위트는 분명 깨지기 쉽다. 따라서 시스템과 테스트를 설계할 때, **GUI를 사용하지 않고 업무 규칙을 테스트할 수 있게 해야 한다.**

## 3. 테스트 API

### 3.1 테스트 API의 목적

이 목표를 달성하려면 테스트가 모든 업무 규칙을 검증하는 데 사용할 수 있도록 **특화된 API**를 만들면 된다. 이러한 API가 갖추어야 할 특성은 다음과 같다:

| 특성 | 설명 |
|------|------|
| **보안 제약 무시** | 테스트 목적으로 보안 제약사항을 건너뛸 수 있다 |
| **비용이 큰 자원 건너뛰기** | 데이터베이스와 같은 값비싼 자원을 우회할 수 있다 |
| **상태 강제** | 시스템을 테스트 가능한 특정 상태로 강제하는 강력한 힘을 지닌다 |
| **상위 집합** | 사용자 인터페이스가 사용하는 인터랙터(Interactor)와 인터페이스 어댑터(Interface Adaptor)들의 상위 집합이 된다 |

테스트 API는 테스트를 애플리케이션으로부터 분리할 목적으로 사용한다. 단순히 테스트를 UI에서 분리하는 것만이 아닌, **테스트 구조를 애플리케이션 구조로부터 결합을 분리**하는 게 목표다.

### 3.2 구조적 결합

구조적 결합은 테스트 결합 중에서 **가장 강하며, 가장 은밀하게 퍼져 나가는** 유형이다.

다음과 같은 테스트 스위트가 있다고 가정해 보자:

| 상용 코드 | 테스트 코드 |
|----------|-----------|
| 모든 상용 클래스 | 각각에 대응하는 테스트 클래스 |
| 모든 상용 메서드 | 각각에 대응하는 테스트 메서드 집합 |

이러한 테스트 스위트는 애플리케이션 구조에 강하게 결합되어 있다. 상용 클래스나 메서드 중 하나라도 변경되면 딸려 있는 다수의 테스트가 변경되어야 한다. 결과적으로 테스트는 깨지기 쉬워지고, 이로 인해 **상용 코드를 뻣뻣하게** 만든다.

**테스트 API의 역할은 애플리케이션의 구조를 테스트로부터 숨기는 데 있다.** 이렇게 만들면:

- 상용 코드를 리팩터링하거나 진화시키더라도 **테스트에는 전혀 영향을 주지 않는다**
- 테스트를 리팩터링하거나 진화시킬 때도 **상용 코드에는 전혀 영향을 주지 않는다**

이처럼 따로따로 진화할 수 있다는 점은 필수적이다. 시간이 지날수록:

```
테스트  →  더 구체적이고 더 특화된 형태
상용 코드  →  더 추상적이고 더 범용적인 형태
```

구조적 결합이 강하면 이러한 필수적인 진화 과정을 방해할(또는 지연시킬) 뿐만 아니라, 상용 코드의 범용성과 유연성이 충분히 좋아지지 못하게 막는다.

### 3.3 보안

테스트 API가 지닌 강력한 힘을 운영 시스템에 배포하면 위험에 처할 수 있다. 위험을 피하고 싶다면, 테스트 API 자체와 테스트 API 중 위험한 부분의 구현부는 **독립적으로 배포할 수 있는 컴포넌트로 분리**해야 한다.

## 요약

- **테스트는 시스템의 일부다.** 시스템 외부에 있지 않으며, 아키텍처에도 관여한다. 아키텍처 관점에서 모든 종류의 테스트는 동등하다.
- **테스트는 가장 바깥쪽 원이다.** 의존성 규칙을 따르며, 시스템 내부의 어떤 것도 테스트에 의존하지 않고, 테스트는 항상 안쪽(상용 코드)을 향해 의존한다.
- **테스트를 설계 범위 밖으로 여기면 치명적이다.** 테스트가 시스템 설계와 잘 통합되지 않으면 깨지기 쉬워지고, 시스템은 뻣뻣해진다.
- **깨지기 쉬운 테스트 문제(Fragile Tests Problem)**: 시스템에 강하게 결합된 테스트는 사소한 변경에도 대량으로 망가지며, 이는 개발자가 변경을 꺼리게 만들어 시스템을 뻣뻣하게 만든다.
- **변동성이 있는 것에 의존하지 말라.** GUI는 변동성이 크므로, GUI를 거치지 않고 업무 규칙을 테스트할 수 있어야 한다.
- **테스트 API를 만들라.** 애플리케이션의 구조를 테스트로부터 숨기는 특화된 API를 통해, 상용 코드와 테스트가 독립적으로 진화할 수 있게 하라.
- **구조적 결합을 피하라.** 상용 클래스마다 테스트 클래스가 1:1로 대응하는 구조는 가장 강하고 은밀한 결합이다.
- **테스트 API의 보안**: 강력한 테스트 API를 운영 시스템에 배포하지 않도록, 독립적으로 배포 가능한 컴포넌트로 분리하라.

## 다른 챕터와의 관계

- **Chapter 22 (클린 아키텍처)**: 테스트는 클린 아키텍처 다이어그램에서 가장 바깥쪽 원에 위치한다. 인터랙터와 인터페이스 어댑터를 통해 업무 규칙에 접근하는 구조가 바로 테스트 API의 기반이 된다.
- **Chapter 26 (메인 컴포넌트)**: 테스트와 메인은 모두 아키텍처의 가장 바깥쪽 원에 위치한다는 공통점이 있다. 시스템 내부의 어떤 것도 이들에 의존하지 않으며, 둘 다 독립적으로 배포 가능하다.
- **Chapter 8 (OCP)**: 테스트 API를 통해 상용 코드와 테스트가 독립적으로 진화할 수 있다는 것은 개방 폐쇄 원칙의 적용이다. 구조적 결합을 줄이면 상용 코드의 변경이 테스트에 영향을 주지 않는다.
- **Chapter 7 (SRP)**: 테스트와 상용 코드는 서로 다른 이유로 변경된다. 테스트는 더 구체적으로, 상용 코드는 더 추상적으로 진화한다. 이는 단일 책임 원칙의 직접적인 반영이다.
- **Chapter 14 (컴포넌트 결합)**: 테스트 API를 독립 배포 가능한 컴포넌트로 분리하는 전략은 컴포넌트 결합 원칙과 직결된다.

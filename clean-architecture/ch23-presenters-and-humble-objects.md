# Chapter 23: Presenters and Humble Objects (프레젠터와 험블 객체)

## 핵심 질문

아키텍처 경계를 식별하고 보호하는 데 도움이 되는 "험블 객체(Humble Object) 패턴"이란 무엇이며, 이 패턴은 프레젠터/뷰, 데이터베이스 게이트웨이, ORM, 서비스 리스너 등 시스템의 어디에서 발견할 수 있는가?

---

## 1. 험블 객체 패턴

22장에서 프레젠터(Presenter)의 개념을 소개했다. 프레젠터는 **험블 객체(Humble Object)(*험블 객체 패턴 — 제라드 메스자로스(Gerard Meszaros)가 집필한 《xUnit 테스트 패턴(xUnit Test Patterns)》(에이콘, 2010)에 나오는 패턴으로, "대강 만든 객체"로 번역되기도 한다.*) 패턴**을 따른 형태로, 아키텍처 경계를 식별하고 보호하는 데 도움이 된다. 실제로 이전 장 "클린 아키텍처"는 험블 객체 구현체들로 가득 차 있었다.

험블 객체 패턴은 디자인 패턴으로, **테스트하기 어려운 행위와 테스트하기 쉬운 행위를 단위 테스트 작성자가 분리하기 쉽게 하는 방법**으로 고안되었다.

아이디어는 매우 단순하다:

1. 행위들을 **두 개의 모듈 또는 클래스**로 나눈다
2. 이들 모듈 중 하나가 **험블(humble)**이다 — 가장 기본적인 본질은 남기고, **테스트하기 어려운 행위를 모두** 험블 객체로 옮긴다
3. 나머지 모듈에는 험블 객체에 속하지 않은, **테스트하기 쉬운 행위를 모두** 옮긴다

예를 들어 GUI의 경우 단위 테스트가 어려운데, 화면을 보면서 각 요소가 필요한 위치에 적절히 표시되었는지 검사하는 테스트는 작성하기 매우 어렵기 때문이다. 하지만 GUI에서 수행하는 행위의 대다수는 쉽게 테스트할 수 있다. 험블 객체 패턴을 사용하면 두 부류의 행위를 분리하여 **프레젠터**와 **뷰**라는 서로 다른 클래스로 만들 수 있다.

| 구분 | 테스트 용이성 | 역할 | 포함하는 행위 |
|------|-------------|------|------------|
| **프레젠터** | 테스트하기 **쉬움** | 데이터를 화면에 표현할 수 있는 포맷으로 변환 | 비즈니스 로직에 가까운 포맷 변환 |
| **뷰** | 테스트하기 **어려움** (험블 객체) | 데이터를 GUI로 이동시킴 | 최소한의 화면 출력만 수행 |

## 2. 프레젠터와 뷰

### 2.1 뷰 (험블 객체)

뷰는 **험블 객체**이고 테스트하기 어렵다. 이 객체에 포함된 코드는 가능한 한 간단하게 유지한다. 뷰는 데이터를 GUI로 이동시키지만, **데이터를 직접 처리하지는 않는다.**

### 2.2 프레젠터 (테스트하기 쉬운 객체)

프레젠터는 테스트하기 쉬운 객체다. 프레젠터의 역할은 애플리케이션으로부터 데이터를 받아 **화면에 표현할 수 있는 포맷**으로 만드는 것이다. 이를 통해 뷰는 데이터를 화면으로 전달하는 간단한 일만 처리하도록 만든다.

프레젠터가 수행하는 구체적인 변환 작업:

| 입력 데이터 | 프레젠터의 변환 | 뷰 모델 내 형태 |
|-----------|--------------|---------------|
| Date 객체 | 적절한 포맷의 문자열로 변환 | 문자열 |
| Currency 객체 | 소수점과 통화 표시가 된 포맷으로 변환 | 문자열 |
| 금액이 음수일 때 | 빨간색 표시 여부 결정 | boolean 플래그 |
| 버튼 이름 | 뷰 모델에 로드 | 문자열 |
| 버튼 비활성화 여부 | 적절한 값 설정 | boolean 플래그 |
| 메뉴 아이템 이름 | 뷰 모델에 로드 | 문자열 |
| 라디오 버튼, 체크 박스, 텍스트 필드 | 적절한 값 설정 | 문자열 + boolean 플래그 |
| 수치 테이블 | 적절한 형식의 문자열로 변환 | 테이블 형태의 문자열 |

### 2.3 뷰 모델

프레젠터가 변환한 데이터는 **뷰 모델(view model)**이라고 부르는 간단한 데이터 구조에 담긴다. 화면에 표시되고 애플리케이션에서 어느 정도 제어할 수 있는 요소라면 무조건 뷰 모델 내부에 **문자열, boolean, 또는 열거형(enum)** 형태로 표현한다.

뷰는 뷰 모델의 데이터를 화면으로 로드할 뿐이며, 이 외에 뷰가 맡은 역할은 전혀 없다. 따라서 뷰는 보잘것없다(humble).

> **핵심 통찰**: 프레젠터는 모든 포맷 변환과 비즈니스 판단을 담당하고, 뷰는 오직 뷰 모델의 데이터를 화면에 옮기는 일만 한다. 이렇게 분리하면 프레젠터는 테스트하기 쉽고, 뷰는 너무 단순해서 테스트할 필요가 거의 없다.

## 3. 테스트와 아키텍처

테스트 용이성은 좋은 아키텍처가 지녀야 할 속성으로 오랫동안 알려져 왔다. 험블 객체 패턴이 좋은 예인데, **행위를 테스트하기 쉬운 부분과 테스트하기 어려운 부분으로 분리하면 아키텍처 경계가 정의**되기 때문이다.

프레젠터와 뷰 사이의 경계는 이러한 경계 중 하나이며, 이 밖에도 수많은 경계가 존재한다.

## 4. 데이터베이스 게이트웨이

유스케이스 인터랙터와 데이터베이스 사이에는 **데이터베이스 게이트웨이(Database Gateway)**가 위치한다. 이 게이트웨이는 **다형적 인터페이스**로, 애플리케이션이 데이터베이스에 수행하는 생성, 조회, 갱신, 삭제 작업과 관련된 모든 메서드를 포함한다.

예를 들어 애플리케이션에서 어제 로그인한 모든 사용자의 성(last name)을 알 수 있어야 한다면, UserGateway 인터페이스는 `getLastNamesOfUsersWhoLoggedInAfter`라는 메서드를 제공할 것이고, 이 메서드는 날짜를 인자로 받아서 사용자 성들을 담은 목록을 반환할 것이다.

```
[유스케이스 인터랙터] --호출--> [게이트웨이 인터페이스] <--구현-- [데이터베이스 계층 구현체]
     (테스트 쉬움)              (유스케이스 계층에 위치)          (험블 객체)
```

유스케이스 계층은 SQL을 허용하지 않는다. 따라서:

- 유스케이스 계층은 필요한 메서드를 제공하는 **게이트웨이 인터페이스를 호출**한다
- 인터페이스의 **구현체는 데이터베이스 계층**에 위치한다
- 이 구현체는 **험블 객체**다 — 직접 SQL을 사용하거나 데이터베이스에 대한 임의의 인터페이스를 통해 게이트웨이의 메서드에서 필요한 데이터에 접근한다
- 인터랙터는 애플리케이션에 특화된 업무 규칙을 캡슐화하기 때문에 **험블 객체가 아니다** — 따라서 테스트하기 쉬운데, 게이트웨이는 스텁(stub)이나 테스트 더블(test-double)(*test-double — 테스트 대역이라고 번역하기도 한다. 테스트를 용이하게 하기 위해 진짜 객체를 대체하는 객체로, 스텁(stub), 모의(mock), 가짜(fake) 객체를 묶어서 지칭할 때 사용한다.*)로 적당히 교체할 수 있기 때문이다

## 5. 데이터 매퍼

하이버네이트(Hibernate) 같은 ORM은 어느 계층에 속한다고 보는가?

먼저 분명히 해야 할 점이 있다. **객체 관계 매퍼(Object Relational Mapper, ORM) 같은 건 사실 존재하지 않는다.** 이유는 간단하다 — 객체는 데이터 구조가 아니기 때문이다.

| 구분 | 특성 | 외부에서 보이는 것 |
|------|------|-----------------|
| **객체** | 데이터는 private, 행위(메서드)를 공개 | public 메서드(오퍼레이션의 집합) |
| **데이터 구조** | 함축된 행위 없는 public 데이터 변수의 집합 | public 데이터 |

최소한 객체를 사용하는 사람 관점에서 객체는 데이터 구조가 아니다. 데이터는 모두 private으로 선언되므로 객체의 사용자는 데이터를 볼 수 없다. 사용자 관점에서 볼 때 객체는 단순히 오퍼레이션의 집합이다.

ORM보다는 차라리 **'데이터 매퍼(Data Mapper)'**라고 부르는 편이 낫다. 관계형 데이터베이스 테이블로부터 가져온 데이터를 데이터 구조에 맞게 담아주기 때문이다.

이러한 ORM 시스템은 어디에 위치해야 하는가? 물론 **데이터베이스 계층**이다. 실제로 ORM은 게이트웨이 인터페이스와 데이터베이스 사이에서 일종의 또 다른 **험블 객체 경계**를 형성한다.

## 6. 서비스 리스너

애플리케이션이 다른 서비스와 반드시 통신해야 한다면, 또는 애플리케이션에서 일련의 서비스를 제공해야 한다면, 여기에서도 서비스 경계를 생성하는 **험블 객체 패턴**을 발견할 수 있다.

**데이터 송신 방향:**
1. 애플리케이션은 데이터를 간단한 데이터 구조 형태로 로드한다
2. 이 데이터 구조를 경계를 가로질러서 특정 모듈로 전달한다
3. 해당 모듈은 데이터를 적절한 포맷으로 만들어서 외부 서비스로 전송한다

**데이터 수신 방향:**
1. 서비스 리스너(service listener)가 서비스 인터페이스로부터 데이터를 수신한다
2. 데이터를 애플리케이션에서 사용할 수 있게 간단한 데이터 구조로 포맷을 변경한다
3. 이 데이터 구조는 서비스 경계를 가로질러서 내부로 전달된다

## 7. 아키텍처 경계에서의 험블 객체 패턴 총정리

각 아키텍처 경계마다 경계 가까이 숨어 있는 험블 객체 패턴을 발견할 수 있다.

| 아키텍처 경계 | 험블 객체 (테스트 어려움) | 테스트하기 쉬운 객체 |
|-------------|----------------------|-------------------|
| **UI 경계** | 뷰 (View) | 프레젠터 (Presenter) |
| **데이터베이스 경계** | 게이트웨이 구현체 | 유스케이스 인터랙터 |
| **ORM 경계** | ORM / 데이터 매퍼 | 게이트웨이 인터페이스 |
| **서비스 경계** | 서비스 리스너 / 송신 모듈 | 애플리케이션 내부 로직 |

경계를 넘나드는 통신은 거의 모두 간단한 데이터 구조를 수반할 때가 많고, 대개 그 경계는 **테스트하기 어려운 무언가**와 **테스트하기 쉬운 무언가**로 분리될 것이다.

> **핵심 통찰**: 아키텍처 경계에서 험블 객체 패턴을 사용하면 전체 시스템의 **테스트 용이성을 크게 높일 수 있다.**

---

## 요약

- **험블 객체 패턴은 테스트하기 어려운 행위와 쉬운 행위를 분리하는 디자인 패턴이다.** 테스트하기 어려운 행위를 험블 객체에, 쉬운 행위를 나머지 모듈에 옮긴다.
- **프레젠터는 테스트하기 쉬운 객체, 뷰는 험블 객체다.** 프레젠터가 모든 포맷 변환을 담당하고, 뷰는 뷰 모델의 데이터를 화면에 옮기기만 한다.
- **뷰 모델은 문자열, boolean, 열거형으로 구성된다.** 화면에 표시될 모든 데이터가 이미 최종 형태로 변환되어 담겨 있다.
- **데이터베이스 게이트웨이 구현체는 험블 객체다.** 유스케이스 인터랙터는 게이트웨이 인터페이스를 호출하고, SQL을 다루는 구현체는 데이터베이스 계층에 위치한다.
- **ORM은 '데이터 매퍼'로 부르는 것이 더 정확하다.** 객체는 데이터 구조가 아니며, ORM은 데이터베이스 계층에서 험블 객체 경계를 형성한다.
- **서비스 경계에서도 험블 객체 패턴을 발견할 수 있다.** 외부 서비스와의 통신을 담당하는 모듈이 험블 객체 역할을 한다.
- **각 아키텍처 경계마다 험블 객체 패턴이 숨어 있다.** 이 패턴을 활용하면 전체 시스템의 테스트 용이성을 크게 높일 수 있다.

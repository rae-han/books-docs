# Chapter 31: The Web Is a Detail (웹은 세부사항이다)

## 핵심 질문

웹은 소프트웨어 아키텍처를 근본적으로 변화시켰는가? IT 역사에서 반복되는 연산 능력 배치의 진동 패턴은 무엇이며, 아키텍트는 이러한 진동으로부터 시스템을 어떻게 보호해야 하는가?

---

## 1. 웹은 모든 것을 바꾸었는가?

1990년대에 웹이 등장했을 때, 개발자들은 웹이 모든 것을 바꾸어 버렸다고 믿었다. 눈부시게 새로운 기술을 앞에 놓고 오래된 클라이언트-서버 아키텍처를 경멸하듯 쳐다봤다.

**사실 웹이 바꾼 것은 아무것도 없었다.** 아니, 적어도 웹은 그렇게 해서는 안 됐었다. 1960년대 이래로 우리 업계는 일련의 반복되는 진동을 겪어왔고, 현재 웹은 그저 이러한 진동의 맨 끝에 있을 뿐이다. 이 진동은 모든 연산 능력을 **중앙 서버에 두는 방식**과 모든 연산 능력을 **단말에 두는 방식** 사이에서 끊임없이 움직여 왔다.

### 1.1 웹 시대의 진동

웹이 유명세를 탄 이래 지난 십여 년 사이에도 이러한 진동이 수차례 목격되었다.

| 시기 | 방향 | 연산 능력 위치 |
|------|------|---------------|
| 초기 웹 | 서버 집중 | 서버 팜(server farm), 브라우저는 단순 표시 장치 |
| 애플릿(applet) 시대 | 단말로 이동 | 브라우저에 애플릿 추가 |
| 서버 사이드 렌더링 | 서버로 회귀 | 동적 내용을 다시 서버에서 생성 |
| Web 2.0 / Ajax | 단말로 이동 | Ajax와 자바스크립트로 처리를 브라우저에서 수행 |
| SPA 시대 | 단말에 집중 | 거대한 애플리케이션 전부를 브라우저에서 실행 |
| Node.js | 서버로 회귀 | 자바스크립트를 다시 서버로 이동 |

---

## 2. 끝없이 반복하는 추

이 반복되는 진동은 웹으로부터 시작된 것이 아니다. 웹 이전에도 같은 패턴이 계속 반복되어 왔다.

| 시대 | 기술 | 연산 능력 배치 |
|------|------|---------------|
| 천공카드 시대 | 컴퓨터실과 천공카드 | 완전 중앙 집중 |
| 메인프레임 시대 | 그린 스크린(*green-screen — 과거 IBM 메인프레임과 통신하기 위한 단말기. 검정 배경에 녹색 텍스트를 사용해서 비공식적으로 '그린 스크린'이라 칭해졌다.*) 단말기 + 메인프레임 | 단말에 일부 지능 부여 |
| 미니컴퓨터 시대 | 멍청한 단말기 + 중앙집중식 미니컴퓨터(*minicomputer — 메인프레임과 비슷한 역할을 하되 그 크기와 성능을 간소화한 컴퓨터. 거대한 단일 메인프레임이 시분할 처리를 하는 반면, 미니컴퓨터는 개별 부서나 사업 단위에 맞춤화된 분산 컴퓨팅을 가능하게 했다.*) | 다시 중앙 집중 |
| 클라이언트-서버 | 데스크톱 + 서버 | 단말에 연산 능력 분산 |
| 웹 시대 | 브라우저 + 웹 서버 | 다시 중앙 집중에서 시작 |

앞으로도 우리는 연산 능력을 어디에 둘지 알 수 없을 것이다. 연산 능력을 중앙에 집중하는 방식과 분산하는 방식 사이에서 끊임없이 움직인다. 이러한 진동은 한동안 계속될 것이다.

> **핵심 통찰**: IT 역사 전체로 시야를 넓히면 웹은 아무것도 바꾸지 않았다. 웹은 우리가 발버둥치면서 생기는 여느 수많은 진동 중 하나에 불과하다. 이 진동은 우리가 태어나기 전에도 있어 왔고, 우리가 은퇴한 뒤에도 지속될 것이다.

---

## 3. 아키텍트의 자세: 멀리 내다보기

아키텍트로서 우리는 멀리 내다봐야 한다. 이 진동은 그저 **핵심 업무 규칙의 중심에서 밀어내고 싶은 단기적인 문제**일 뿐이다.

### 3.1 Q 회사 사례: 개인용 재무 시스템

Q사는 상당히 인기를 끈 개인용 재무 시스템을 개발했다. 데스크톱 앱으로 상당히 유용한 GUI를 제공했다.

어느 날 웹이 등장했다. 이어서 출시된 버전에서 Q사는 앱의 GUI가 마치 **브라우저처럼 보이고 동작하도록** 변경했다. 데스크톱에서 실행되는 개인용 재무 소프트웨어가 웹브라우저와 같은 룩앤필(look and feel)을 제공해야 한다고 결정한 것이다.

결과적으로 다른 사람들도 마찬가지로 새로운 인터페이스를 싫어했다. 몇 차례의 출시를 거치면서 Q사는 브라우저스러운 느낌을 조금씩 없애고 **다시 일반적인 데스크톱 GUI로 되돌렸다.**

**교훈**: Q사의 소프트웨어 아키텍트가 업무 규칙을 UI로부터 분리했어야 했다. 마케팅 천재가 다음에 무슨 일을 할지 우리는 전혀 모르기 때문이다.

### 3.2 A 회사 사례: 스마트폰 운영체제

매력적인 스마트폰을 만드는 A사는 자사 '운영체제'의 업그레이드 버전을 출시하면서 모든 애플리케이션의 룩앤필을 완전히 변경했다. 짐작컨대 마케팅 귀재가 그렇게 하라고 말했을 것이다.

**교훈**: A사의 아키텍트나 앱의 아키텍트가 UI와 업무 규칙을 서로 격리했기 바란다. 마케팅 귀재는 어디나 존재하며, 당신이 사소한 결합을 만드는 순간 물고 늘어지려 항상 대기하고 있기 때문이다.

## 4. GUI는 세부사항이다

요약하면 다음과 같은 삼단논법이 성립한다:

1. **GUI는 세부사항이다**
2. **웹은 GUI다**
3. **따라서 웹은 세부사항이다**

아키텍트라면 이러한 세부사항을 핵심 업무 로직에서 분리된 경계 바깥에 두어야 한다.

### 4.1 웹은 입출력 장치다

1960년대에 우리는 애플리케이션을 **장치 독립적**으로 만들면 어떤 가치가 있는지 배웠다. 장치 독립적으로 만들어야 했던 동기 자체는 지금도 유효하다. 웹도 이 규칙에서 예외가 될 수 없다.

### 4.2 완전한 추상화는 어려울 수 있다

자바스크립트의 유효성 검증, 드래그-앤-드롭 방식의 AJAX 호출, 그리고 웹 페이지에 넣을 수 있는 다른 무수한 위젯과 가젯으로 인한 복잡함을 생각해 볼 때, 웹에서 장치 독립성은 비현실적이라고 주장하기 쉽다.

이 주장이 어느 정도는 옳다. 애플리케이션과 GUI의 상호작용은 '빈번하며', 또한 사용 중인 GUI 종류에 따라 차이가 매우 크다. 브라우저와 웹 애플리케이션이 함께 추는 춤은 데스크톱 GUI와 데스크톱 애플리케이션이 함께 추는 춤과는 차이가 난다.

### 4.3 유스케이스 수준의 추상화

하지만 UI와 애플리케이션 사이에는 추상화가 가능한 또 다른 경계가 존재한다.

- 업무 로직은 다수의 **유스케이스**로 구성된다
- 각 유스케이스는 사용자를 대신해서 일부 함수를 수행하는 것으로 볼 수 있다
- 각 유스케이스는 **입력 데이터**, **수행할 처리 과정**, **출력 데이터**를 기반으로 기술할 수 있다

UI와 애플리케이션이 함께 춤추는 동안, 어떤 시점이 되면 입력 데이터가 완전히 구성될 것이고, 그러면 유스케이스를 실행할 수 있게 된다. 유스케이스가 종료되면 해당 입력 데이터에 따른 결과 데이터는 UI와 애플리케이션이 함께 추는 춤으로 다시 되돌려줄 수 있다.

```
[UI 계층]                      [업무 규칙 계층]

사용자 입력 수집     ──>    입력 데이터 구조
                           유스케이스 실행
결과 화면 표시       <──    출력 데이터 구조
```

완전한 입력 데이터와 그에 따른 출력 데이터는 데이터 구조로 만들어서 유스케이스를 실행하는 처리 과정의 입력 값과 출력 값으로 사용할 수 있다. 이 방식을 따르면 각 유스케이스가 **장치 독립적인 방식으로 UI라는 입출력 장치를 동작시킨다**고 간주할 수 있다.

---

## 5. 결론

이러한 종류의 추상화는 만들기 쉽지 않고, 제대로 만들려면 수차례의 반복 과정을 거쳐야 할 것이다. 하지만 가능하다. 그리고 세상은 마케팅 귀재로 가득하기 때문에 이러한 추상화가 꼭 필요할 때가 많다고 주장하기는 어렵지 않다.

---

## 요약

- 웹은 IT 역사에서 반복되는 **연산 능력 배치 진동**의 하나일 뿐이며, 아키텍처를 근본적으로 바꾸지 않았다
- 중앙 집중과 분산 사이의 진동은 1960년대부터 계속되어 왔고, 앞으로도 계속될 것이다
- **GUI는 세부사항이고, 웹은 GUI이므로, 웹은 세부사항이다**
- 아키텍트는 업무 규칙을 UI로부터 분리하여, 마케팅 주도의 UI 변경으로부터 시스템을 보호해야 한다
- 완전한 장치 독립성은 어렵더라도, **유스케이스 수준의 입출력 데이터 구조**를 통한 추상화는 가능하다
- 웹도 하나의 입출력 장치로 취급해야 하며, 1960년대에 배운 장치 독립성의 원칙은 여전히 유효하다

# Chapter 7: SRP: The Single Responsibility Principle (단일 책임 원칙)

## 핵심 질문

"모든 모듈이 단 하나의 일만 해야 한다"는 것이 정말 SRP의 의미인가? SRP에서 말하는 "책임"이란 정확히 무엇이며, 이 원칙을 위반하면 어떤 실질적인 문제가 발생하는가?

---

## 1. SRP에 대한 흔한 오해

SOLID 원칙 중에서 그 의미가 가장 잘 전달되지 못한 원칙이 바로 단일 책임 원칙(SRP)이다. 아마도 현저히 부적절한 이름 때문이기도 할 것이다. 프로그래머가 이 원칙의 이름을 듣는다면 **모든 모듈이 단 하나의 일만 해야 한다**는 의미로 받아들이기 쉽다.

하지만 헷갈리지 말아야 한다. "단 하나의 일만 해야 한다"는 원칙은 사실 따로 있다. 그것은 **함수는 반드시 하나의, 단 하나의 일만 해야 한다**는 원칙이다. 이 원칙은 커다란 함수를 작은 함수들로 리팩터링하는 더 저수준에서 사용된다. 하지만 이 원칙은 SOLID 원칙이 아니며, SRP도 아니다.

---

## 2. SRP의 정의: 변경의 이유와 액터

### 2.1 역사적 정의

역사적으로 SRP는 아래와 같이 기술되어 왔다:

> 단일 모듈은 변경의 이유가 하나, 오직 하나뿐이어야 한다.

소프트웨어 시스템은 사용자와 이해관계자를 만족시키기 위해 변경된다. SRP가 말하는 "변경의 이유"란 바로 이들 사용자와 이해관계자를 가리킨다. 따라서 이 원칙은 아래와 같이 바꿔 말할 수 있다:

> 하나의 모듈은 하나의, 오직 하나의 사용자 또는 이해관계자에 대해서만 책임져야 한다.

### 2.2 액터(Actor)의 등장

안타깝게도 "사용자"와 "이해관계자"라는 단어를 여기에 쓰는 것은 올바르지 않다. 시스템이 동일한 방식으로 변경되기를 원하는 사용자나 이해관계자가 두 명 이상일 수도 있기 때문이다. 여기에서는 해당 변경을 요청하는 한 명 이상의 사람들로 구성된 집단을 가리키며, 이러한 집단을 **액터(actor)**라고 부른다.

### 2.3 SRP의 최종 버전

> **하나의 모듈은 하나의, 오직 하나의 액터에 대해서만 책임져야 한다.**

그렇다면 "모듈"이란 무엇인가? 가장 단순한 정의는 **소스 파일**이다. 대부분의 경우 이 정의는 잘 들어맞는다. 하지만 일부 언어와 개발 환경에서는 코드를 소스 파일에 저장하지 않는다. 이러한 경우 모듈은 단순히 **함수와 데이터 구조로 구성된 응집된 집합**이다.

"응집된(cohesive)"이라는 단어가 SRP를 암시한다. 단일 액터를 책임지는 코드를 함께 묶어주는 힘이 바로 **응집성(cohesion)**이다.

> **핵심 통찰**: SRP는 "하나의 일만 하라"가 아니라 "하나의 액터에 대해서만 책임져라"이다. 변경의 이유가 되는 주체(액터)에 초점을 맞추는 것이 이 원칙의 핵심이다.

## 3. 징후 1: 우발적 중복

이 원칙을 이해하는 가장 좋은 방법은 이 원칙을 위반하는 징후들을 살펴보는 것이다.

### 3.1 Employee 클래스 예제

급여 애플리케이션의 `Employee` 클래스를 생각해보자. 이 클래스는 세 가지 메서드 `calculatePay()`, `reportHours()`, `save()`를 가진다.

```
┌──────────────────┐
│     Employee     │
├──────────────────┤
│ + calculatePay   │◄── CFO (회계팀)
│ + reportHours    │◄── COO (인사팀)
│ + save           │◄── CTO (DBA)
└──────────────────┘
```
*그림 7.1 — Employee 클래스. 세 가지 메서드가 서로 다른 세 명의 액터를 책임진다.*

이 클래스는 SRP를 위반한다. 이들 세 가지 메서드가 서로 매우 다른 세 명의 액터를 책임지기 때문이다.

| 메서드 | 기능 정의 주체 | 보고 대상 |
|--------|-------------|----------|
| `calculatePay()` | 회계팀 | CFO |
| `reportHours()` | 인사팀 | COO |
| `save()` | 데이터베이스 관리자(DBA) | CTO |

### 3.2 문제 시나리오

개발자가 이 세 메서드를 `Employee`라는 단일 클래스에 배치하여 세 액터가 서로 결합되어 버렸다. 이 결합으로 인해 **CFO 팀에서 결정한 조치가 COO 팀이 의존하는 무언가에 영향을 줄 수 있다.**

예를 들어 `calculatePay()` 메서드와 `reportHours()` 메서드가 초과 근무를 제외한 업무 시간을 계산하는 알고리즘을 공유한다고 해보자. 개발자는 코드 중복을 피하기 위해 이 알고리즘을 `regularHours()`라는 메서드에 넣었다.

```
┌────────────────┐    ┌────────────────┐
│ calculatePay   │    │  reportHours   │
└───────┬────────┘    └───────┬────────┘
        │                     │
        └──────┐   ┌──────────┘
               ▼   ▼
        ┌──────────────┐
        │ regularHours │
        └──────────────┘
```
*그림 7.2 — 공유된 알고리즘. 두 메서드가 regularHours()를 공유한다.*

이제 CFO 팀에서 초과 근무를 제외한 업무 시간 계산 방식을 약간 수정하기로 결정했다고 하자. 반면 COO 팀에서는 이와 다른 목적으로 이 값을 사용하기 때문에 변경을 원하지 않는다.

변경을 담당한 개발자는 `calculatePay()`가 `regularHours()`를 호출한다는 사실을 발견한다. 하지만 안타깝게도 이 함수가 `reportHours()`에서도 호출된다는 사실은 눈치채지 못한다.

개발자는 요청된 변경사항을 적용하고 신중하게 테스트한다. CFO 팀은 검증하고, 시스템은 배포된다. 물론 COO 팀에서는 이러한 일이 벌어지고 있다는 사실을 알지 못한다. COO 팀 직원은 `reportHours()`가 생성한 보고서를 여전히 이용하지만, 이제 이 보고서에 포함된 수치들은 엉터리다. 마침내 문제가 발견되고, COO는 격노한다. **잘못된 데이터로 인해 수백만 달러의 예산이 지출되었기 때문이다.**

이러한 문제는 **서로 다른 액터가 의존하는 코드를 너무 가까이 배치했기** 때문에 발생한다. SRP는 서로 다른 액터가 의존하는 코드를 서로 분리하라고 말한다.

---

## 4. 징후 2: 병합(Merge)

소스 파일에 다양하고 많은 메서드를 포함하면 **병합(merge)**이 자주 발생하리라고 짐작하기는 어렵지 않다. 특히 이들 메서드가 서로 다른 액터를 책임진다면 병합이 발생할 가능성은 확실히 더 높다.

### 4.1 병합 시나리오

예를 들어 CTO 팀(DBA 소속)에서 데이터베이스의 `Employee` 테이블 스키마를 약간 수정하기로 결정했다고 해보자. 이와 동시에 COO 팀에서는 `reportHours()` 메서드의 보고서 포맷을 변경하기로 결정했다.

두 명의 서로 다른 개발자가, 그리고 아마도 서로 다른 팀에 속한 두 개발자가 `Employee` 클래스를 체크아웃받은 후 변경사항을 적용하기 시작한다. 안타깝게도 이들 변경사항은 서로 충돌한다. 결과적으로 병합이 발생한다.

최근 도구는 굉장히 뛰어나지만, **어떤 도구도 병합이 발생하는 모든 경우를 해결할 수는 없다.** 결국 병합에는 항상 위험이 뒤따르게 된다.

이 문제를 벗어나는 방법은 **서로 다른 액터를 뒷받침하는 코드를 서로 분리하는 것**이다.

---

## 5. 해결책

이 문제의 해결책은 다양한데, 그 모두가 메서드를 각기 다른 클래스로 이동시키는 방식이다.

### 5.1 해결책 1: 데이터와 메서드 분리

가장 확실한 해결책은 데이터와 메서드를 분리하는 방식이다. 아무런 메서드가 없는 간단한 데이터 구조인 `EmployeeData` 클래스를 만들어, 세 개의 클래스가 공유하도록 한다.

```
┌──────────────────┐
│  PayCalculator   │──┐
├──────────────────┤  │
│ + calculatePay   │  │    ┌──────────────┐
└──────────────────┘  ├───▶│  Employee     │
┌──────────────────┐  │    │  Data         │
│  HourReporter    │──┤    └──────────────┘
├──────────────────┤  │
│ + reportHours    │  │
└──────────────────┘  │
┌──────────────────┐  │
│  EmployeeSaver   │──┘
├──────────────────┤
│ + saveEmployee   │
└──────────────────┘
```
*그림 7.3 — 세 클래스는 서로의 존재를 알지 못한다. 각 클래스는 자신의 메서드에 반드시 필요한 소스 코드만을 포함한다.*

각 클래스는 자신의 메서드에 반드시 필요한 소스 코드만을 포함한다. 세 클래스는 서로의 존재를 몰라야 한다. 따라서 "우연한 중복"을 피할 수 있다.

하지만 이 해결책은 **개발자가 세 가지 클래스를 인스턴스화하고 추적해야 한다**는 단점이 있다.

### 5.2 해결책 2: 퍼사드(Facade) 패턴

세 클래스를 직접 다루는 번거로움을 줄이기 위해 **퍼사드(Facade) 패턴**을 사용할 수 있다.

```
┌──────────────────┐    ┌──────────────────┐
│  Employee        │───▶│  PayCalculator   │
│  Facade          │    ├──────────────────┤
├──────────────────┤    │ + calculatePay   │
│ + calculatePay   │    └──────────────────┘    ┌──────────────┐
│ + reportHours    │───▶┌──────────────────┐───▶│  Employee     │
│ + save           │    │  HourReporter    │    │  Data         │
└──────────────────┘    ├──────────────────┤    └──────────────┘
        │               │ + reportHours    │
        │               └──────────────────┘
        │               ┌──────────────────┐
        └──────────────▶│  EmployeeSaver   │
                        ├──────────────────┤
                        │ + saveEmployee   │
                        └──────────────────┘
```
*그림 7.4 — 퍼사드(Facade) 패턴. EmployeeFacade는 세 클래스의 객체를 생성하고, 요청된 메서드를 가지는 객체로 위임하는 일을 책임진다.*

`EmployeeFacade`에 코드는 거의 없다. 이 클래스는 세 클래스의 객체를 생성하고, 요청된 메서드를 가지는 객체로 위임하는 일을 책임진다.

### 5.3 해결책 3: 가장 중요한 메서드를 원래 클래스에 유지

어떤 개발자는 가장 중요한 업무 규칙을 데이터와 가깝게 배치하는 방식을 선호한다. 이 경우 가장 중요한 메서드는 기존의 `Employee` 클래스에 그대로 유지하되, `Employee` 클래스를 덜 중요한 나머지 메서드들에 대한 퍼사드로 사용할 수 있다.

```
┌──────────────────────┐    ┌──────────────────┐
│     Employee         │───▶│  HourReporter    │
├──────────────────────┤    ├──────────────────┤
│ - employeeData       │    │ + reportHours    │
├──────────────────────┤    └──────────────────┘
│ + calculatePay       │
│ + reportHours        │    ┌──────────────────┐
│ + save               │───▶│  EmployeeSaver   │
└──────────────────────┘    ├──────────────────┤
                            │ + saveEmployee   │
                            └──────────────────┘
```
*그림 7.5 — 가장 중요한 메서드(calculatePay)는 Employee 클래스에 그대로 유지하되, 덜 중요한 나머지 메서드들에 대한 퍼사드로 사용한다.*

### 5.4 클래스당 메서드 수에 대한 반론

모든 클래스는 반드시 단 하나의 메서드를 가져야 한다는 주장에 근거하여 앞의 해결책에 반대할 수도 있다. 하지만 이 주장은 현실과는 전혀 다르다. 각 클래스에서 지불, 보고서 생성, 데이터 저장 기능을 구현하는 데 필요한 메서드의 개수는 실제로 훨씬 더 많을 것이다. 이들 클래스는 모두 다수의 `private` 메서드를 포함할 것이다.

이처럼 여러 메서드가 하나의 가족을 이루고, 메서드의 가족을 포함하는 각 클래스는 하나의 유효범위가 된다. 해당 유효범위 바깥에서는 이 가족에게 감춰진 식구(`private` 멤버)가 있는지를 전혀 알 수 없다.

---

## 6. 결론

단일 책임 원칙은 메서드와 클래스 수준의 원칙이다. 하지만 이보다 상위의 두 수준에서도 다른 형태로 다시 등장한다.

| 수준 | 원칙 |
|------|------|
| 메서드/클래스 | 단일 책임 원칙(SRP) |
| 컴포넌트 | 공통 폐쇄 원칙(Common Closure Principle, CCP) |
| 아키텍처 | 아키텍처 경계(Architectural Boundary)의 생성을 책임지는 변경의 축(Axis of Change) |

---

## 요약

- **SRP는 "하나의 일만 하라"가 아니다.** "하나의 액터에 대해서만 책임져라"가 올바른 해석이다.
- **액터(actor)란** 해당 변경을 요청하는 한 명 이상의 사람들로 구성된 집단이다.
- **우발적 중복(징후 1)**: 서로 다른 액터가 의존하는 코드가 공유 메서드를 통해 결합되면, 한쪽의 변경이 다른 쪽에 의도치 않은 영향을 미친다.
- **병합(징후 2)**: 서로 다른 액터를 위한 코드가 같은 소스 파일에 있으면, 서로 다른 팀의 변경이 충돌하여 병합 위험이 증가한다.
- **해결책은 분리다**: 데이터와 메서드를 분리하거나, 퍼사드 패턴을 사용하거나, 가장 중요한 메서드를 원래 클래스에 유지하되 나머지를 위임하는 방식이 있다.
- **SRP는 상위 수준에서도 반복된다**: 컴포넌트 수준에서는 공통 폐쇄 원칙(CCP), 아키텍처 수준에서는 변경의 축(Axis of Change)으로 나타난다.

# Chapter 19: Policy and Level (정책과 수준)

## 핵심 질문

소프트웨어 시스템을 구성하는 "정책"이란 무엇이며, 정책의 "수준"은 어떻게 결정되는가? 소스 코드 의존성은 데이터 흐름이 아닌 무엇에 따라 결합되어야 하며, 왜 저수준 컴포넌트가 고수준 컴포넌트에 의존해야 하는가?

---

## 1. 소프트웨어 시스템은 정책을 기술한 것이다

컴퓨터 프로그램의 핵심부는 이것이 전부다 — 각 입력을 출력으로 변환하는 **정책을 상세하게 기술한 설명서**다.

대다수의 주요 시스템에서 하나의 정책은 이 정책을 서술하는 여러 개의 조그만 정책들로 쪼갤 수 있다:

| 정책의 종류 | 예시 |
|------------|------|
| 업무 규칙 처리 방식 | 집계와 관련된 업무 규칙 |
| 보고서 포맷 | 특정 보고서를 어떤 포맷으로 만들지 서술 |
| 입력 데이터 검증 | 입력 데이터를 어떻게 검증할지 서술 |

---

## 2. 정책의 분리와 재편성

소프트웨어 아키텍처를 개발하는 기술에는 정책을 **신중하게 분리**하고, 정책이 변경되는 양상에 따라 정책을 **재편성**하는 일도 포함된다.

### 2.1 분리 기준

| 기준 | 결과 |
|------|------|
| 동일한 이유로 동일한 시점에 변경되는 정책 | 동일한 수준에 위치하며, **동일한 컴포넌트**에 속해야 한다 |
| 서로 다른 이유로 또는 다른 시점에 변경되는 정책 | 다른 수준에 위치하며, 반드시 **다른 컴포넌트**로 분리해야 한다 |

### 2.2 비순환 방향 그래프

흔히 아키텍처 개발은 재편성된 컴포넌트들을 **비순환 방향 그래프(Directed Acyclic Graph, DAG)**로 구성하는 기술을 포함한다:

- **정점(node)**: 동일한 수준의 정책을 포함하는 컴포넌트
- **간선(edge)**: 컴포넌트 사이의 의존성 (다른 수준의 컴포넌트를 연결)

이러한 의존성은 소스 코드, 컴파일타임의 의존성이다. 자바의 import, C#의 using, 루비의 require 등에 해당한다.

> **핵심 통찰**: 좋은 아키텍처라면 각 컴포넌트를 연결할 때 의존성의 방향이 컴포넌트의 수준을 기반으로 연결되도록 만들어야 한다. 즉, **저수준 컴포넌트가 고수준 컴포넌트에 의존**하도록 설계되어야 한다.

---

## 3. 수준(Level)의 정의

'수준'을 엄밀하게 정의하자면 **'입력과 출력까지의 거리'**다:

- 시스템의 입력과 출력 **모두로부터 멀리** 위치할수록 정책의 수준은 **높아진다**
- 입력과 출력을 **다루는** 정책이라면 시스템에서 **최하위 수준**에 위치한다

---

## 4. 암호화 프로그램 예제

### 4.1 데이터 흐름도

간단한 암호화 프로그램을 생각해보자. 입력 장치에서 문자를 읽어서, 테이블을 참조하여 번역한 후, 번역된 문자를 출력 장치로 기록한다.

```
                  ┌──────────┐
                  │  테이블   │
                  └────┬─────┘
                       │
    문자              ▼              문자
  ─────→  ┌──────────────┐  ─────→
          │    번역      │
  ◀─────  └──────────────┘  ◀─────
          ↗              ↖
   ┌──────────┐    ┌──────────┐
   │ 문자 읽기 │    │ 문자 쓰기 │
   └──────────┘    └──────────┘

  실선 화살표: 데이터 흐름
  점선 화살표: 소스 코드 의존성 (번역을 향한다)
```

**번역** 컴포넌트는 이 시스템에서 **최고 수준**의 컴포넌트다 — 입력과 출력에서부터 가장 멀리 떨어져 있기 때문이다.(*밀러 페이지-존스(Meilir Page-Jones)는 이 컴포넌트를 '중앙 변환(Central Transform)'이라고 불렀다.*)

### 4.2 핵심: 데이터 흐름 ≠ 의존성 방향

주목할 점은 **데이터 흐름과 소스 코드 의존성이 항상 같은 방향을 가리키지는 않는다**는 사실이다. 소스 코드 의존성은 그 **수준에 따라** 결합되어야 하며, **데이터 흐름을 기준으로** 결합되어서는 안 된다. 이것이 바로 소프트웨어 아키텍처가 가진 예술 중 하나다.

### 4.3 잘못된 아키텍처

```javascript
function encrypt() {
    while(true)
        writeChar(translate(readChar()));
}
```

이 코드는 **잘못된 아키텍처**다. 고수준인 encrypt 함수가 저수준인 readChar와 writeChar 함수에 **직접 의존**하기 때문이다.

### 4.4 개선된 아키텍처

클래스 다이어그램으로 개선해보면:

```
         ╭─────────────────────────────────────╮
         │                                     │
         │         ┌──────────┐               │
         │         │ Encrypt  │               │
         │         └──┬───┬───┘               │
         │            │   │                    │
         │            ▼   ▼                    │
         │   ┌────────────┐ ┌────────────┐    │
         │   │  <<I>>     │ │  <<I>>     │    │
         │   │ CharReader │ │ CharWriter │    │
         │   └─────△──────┘ └─────△──────┘    │
         │         │              │            │
         ╰─────────│──────────────│────────────╯
                   │              │
            ┌──────────┐  ┌──────────┐
            │ Console  │  │ Console  │
            │ Reader   │  │ Writer   │
            └──────────┘  └──────────┘

점선 경계: 이 경계를 횡단하는 의존성은 모두 경계 안쪽으로 향한다.
이 경계로 묶인 영역이 시스템에서 최고 수준의 구성요소다.
```

- **Encrypt**: 고수준 정책 (암호화 알고리즘)
- **CharReader, CharWriter**: 인터페이스 (경계 안쪽에 위치)
- **ConsoleReader, ConsoleWriter**: 저수준 구현 클래스 (입출력에 가까움)

이 구조에서 고수준의 암호화 정책을 저수준의 입력/출력 정책으로부터 분리시킨 덕분에, 암호화 정책을 **더 넓은 맥락에서 사용**할 수 있다. 입력과 출력에 변화가 생기더라도 암호화 정책은 거의 영향을 받지 않기 때문이다.

## 5. 정책 변경의 특성

정책을 컴포넌트로 묶는 기준은 정책이 변경되는 방식에 달려있다. **단일 책임 원칙(SRP)**과 **공통 폐쇄 원칙(CCP)**에 따르면 동일한 이유로 동일한 시점에 변경되는 정책은 함께 묶인다.

| 수준 | 변경 빈도 | 변경 이유 | 중요도 |
|------|----------|----------|--------|
| **고수준** (입출력에서 먼 정책) | 덜 빈번 | 보다 중요한 이유 | 높음 |
| **저수준** (입출력에 가까운 정책) | 더 빈번 | 보다 긴급하지만 덜 중요한 이유 | 낮음 |

예를 들어 암호화 프로그램에서:
- 입출력 장치가 변경될 가능성은 암호화 알고리즘이 변경될 가능성보다 **훨씬 높다**
- 암호화 알고리즘을 변경해야 한다면, 입출력 장치 변경보다 **더 실질적인 이유**로 변경될 가능성이 높다

### 5.1 변경의 영향도 줄이기

모든 소스 코드 의존성의 방향이 고수준 정책을 향할 수 있도록 정책을 분리했다면, **변경의 영향도를 줄일 수 있다.** 시스템의 최저 수준에서 중요하지 않지만 긴급한 변경이 발생하더라도, 보다 높은 위치의 중요한 수준에 미치는 영향은 거의 없게 된다.

---

## 6. 저수준은 고수준에 플러그인된다

이 논의는 **저수준 컴포넌트가 고수준 컴포넌트에 플러그인되어야 한다**는 관점으로 바라볼 수 있다:

```
┌──────────────────┐         ┌──────────────────┐
│                  │         │                  │
│   Encryption     │ ◀────── │   IO Devices     │
│                  │         │                  │
└──────────────────┘         └──────────────────┘
```

- **Encryption** 컴포넌트는 IO Devices 컴포넌트를 전혀 알지 못한다
- **IO Devices**는 Encryption 컴포넌트에 의존적이다

이를 통해 암호화 정책은 입출력 장치와 완전히 독립적이 되며, 어떤 입출력 장치와도 조합하여 사용할 수 있게 된다.

---

## 요약

- 소프트웨어 시스템이란 **정책을 기술한 것**이다 — 각 입력을 출력으로 변환하는 규칙의 집합
- 정책의 **수준(level)**은 입력과 출력까지의 거리로 결정된다 — 멀수록 고수준
- **데이터 흐름과 소스 코드 의존성은 같은 방향을 가리키지 않는다** — 의존성은 수준에 따라, 데이터 흐름과는 독립적으로 결합되어야 한다
- 고수준 정책은 덜 빈번하게, 보다 중요한 이유로 변경되고, 저수준 정책은 더 빈번하게, 덜 중요한 이유로 변경된다
- **저수준 컴포넌트는 고수준 컴포넌트에 플러그인**되어야 한다 — 이를 통해 변경의 영향도를 최소화할 수 있다
- 이 장의 논의는 SRP, OCP, CCP, DIP, SDP, SAP를 **모두 포함**한다

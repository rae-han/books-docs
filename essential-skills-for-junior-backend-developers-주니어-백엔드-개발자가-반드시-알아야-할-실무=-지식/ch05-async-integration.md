## 1. 동기 연동과 비동기 연동

### 1.1 동기 연동의 직관성과 한계

기능을 개발할 때 가장 자연스럽게 떠올리는 코드는 동기(*Synchronous - 한 작업이 끝날 때까지 다음 작업이 진행되지 않는 방식*) 방식이다. 예를 들어 로그인에 성공하면 포인트를 지급하는 기능을 개발할 때, 다음과 같은 코드를 떠올린다.

```java
public boolean login(String id, String password) {
    Optional<User> opt = findUser(id);           // 1. User 조회
    if (opt.isEmpty()) {                          // 2. User가 없으면 false 리턴
        return false;
    }
    User u = opt.get();
    if (!u.matchPassword(password)) {             // 3. 암호가 일치하지 않으면 false 리턴
        return false;
    }
    PointResult result = pointClient.giveLoginPoint(id); // 4. 포인트 지급 서비스 호출
    if (result.isFailed()) {                      // 5. 포인트 지급에 실패하면 익셉션 발생
        throw new PointException();
    }
    appendLoginHistory(id);                       // 6. 로그인 내역 추가
    return true;
}
```

이 코드의 실행 순서는 코드 순서와 동일하다. User 정보를 찾고, 암호를 비교하고, 포인트를 지급하고, 로그인 내역을 추가한다. 포인트를 지급하기 전에 로그인 내역을 추가하는 일은 벌어지지 않는다.

```
클라이언트        로그인 서비스        DB        포인트 서비스
    |                  |              |              |
    |--- 로그인 요청 -->|              |              |
    |                  |-- 회원 조회 ->|              |
    |                  |<- 조회 결과 --|              |
    |                  |-- 포인트 지급 요청 --------->|
    |                  |<- 지급 결과 응답 ------------|
    |                  |-- 내역 기록 ->|              |
    |<-- 로그인 응답 ---|              |              |
```

동기 방식은 프로그램의 흐름을 직관적으로 이해할 수 있고, 디버깅도 용이하다. 하지만 외부 연동을 만나면 고려할 것이 생긴다.

**외부 연동 실패가 전체 기능의 실패인가?**

앞선 예에서는 포인트 지급 서비스 연동에 실패하면 로그인에도 실패한다고 가정했다. 포인트 지급 서비스에 장애가 발생하면 그 시간 동안 로그인도 못 해서 전체 기능을 사용할 수 없게 된다. 이게 맞는 것일까?

아마도 그렇지 않을 것이다. 포인트 지급에 실패하더라도 로그인 자체는 정상적으로 동작해야 하고, 나머지 기능을 사용할 수 있기를 원할 것이다. 이를 위해 포인트 지급에 실패하면 나중에 후처리할 수 있도록 실패 내역을 따로 남기는 방식으로 코드를 작성할 수 있다.

```java
public boolean login(String id, String password) {
    Optional<User> opt = findUser(id);
    if (opt.isEmpty()) {
        return false;
    }
    User u = opt.get();
    if (!u.matchPassword(password)) {
        return false;
    }
    PointResult result = pointClient.giveLoginPoint(id); // 포인트 지급 서비스 호출
    if (result.isFailed()) {                             // 실패하면 후처리 위해 내역 남김
        recordPointFailure(id, result);
    }
    appendLoginHistory(id);
    return true;
}
```

**외부 서비스의 응답 시간 문제**

연동 서비스의 응답 시간이 길어질수록 전체 응답 시간이 느려지게 된다. 심한 경우 외부 연동 서비스로 인해 전체 서비스가 먹통이 되기도 한다(4장에서 언급한 바 있다).

> **핵심 통찰**: 다음 작업을 진행하기 위해 반드시 외부 연동 결과가 필요한 게 아니라면, 동기 방식 대신 비동기 방식으로 연동하는 것을 고민해 볼 필요가 있다.

### 1.2 비동기 연동의 이점

비동기(*Asynchronous - 한 작업이 끝날 때까지 기다리지 않고 바로 다음 작업을 처리하는 방식*) 방식을 사용하면 외부 연동이 끝날 때까지 기다리지 않고 바로 다음 작업을 진행할 수 있다. 로그인 예시에서는 포인트 지급이 끝날 때까지 기다리지 않고 바로 로그인 성공을 사용자에게 응답할 수 있다. 포인트 지급 처리에 소요되는 연동 시간만큼 사용자는 더 빠르게 응답을 받을 수 있게 된다.

```
클라이언트        로그인 서비스        DB        포인트 서비스
    |                  |              |              |
    |--- 로그인 요청 -->|              |              |
    |                  |-- 회원 조회 ->|              |
    |                  |<- 조회 결과 --|              |
    |                  |   [별도 스레드]               |
    |                  |-- 내역 기록 ->|              |
    |<-- 로그인 응답 ---|              |              |
    |                  |       (별도 스레드에서)       |
    |                  |-- 포인트 지급 요청 --------->|
    |                  |<- 지급 결과 응답 ------------|
```

사용자가 로그인 응답을 받은 시점에 포인트는 아직 지급 중이다. 로그인에 성공한 시점에 바로 포인트가 쌓이지 않으면 문제가 될 것 같지만, 사실 많은 경우 그렇지 않다. 로그인에 성공하고 수초 이내에 포인트가 쌓이면 문제가 되지 않을 때가 많다. 설사 수분 뒤에 포인트가 적립되더라도 크게 문제되지 않는다.

포인트 서비스에 일시적으로 문제가 생겨 포인트 지급 처리에 시간이 오래 걸려도 로그인 서비스의 응답 시간은 증가하지 않는다. 사용자는 평소처럼 로그인에 성공하고 서비스를 사용할 수 있다. 단지 포인트가 평소보다 조금 늦게 지급될 뿐이다.

### 1.3 비동기 연동이 적합한 경우

생각보다 많은 연동에서 비동기 방식을 사용해도 된다. 다음은 비동기 방식으로 연동해도 크게 문제가 되지 않는 몇 가지 예다.

- 쇼핑몰에서 주문이 들어오면 판매자에게 푸시 보내기(푸시 서비스 연동)
- 학습을 완료하면 학생에게 포인트 지급(포인트 서비스 연동)
- 컨텐츠를 등록할 때 검색 서비스에도 등록(검색 서비스 연동)
- 인증번호를 요청하면 SMS로 인증 메시지 발송(SMS 발송 서비스 연동)

이 예시들에는 몇 가지 공통적인 특징이 있다.

| 특징 | 설명 | 예시 |
|------|------|------|
| **시차 허용** | 연동에 약간의 시차가 생겨도 문제가 되지 않는다 | 주문 완료 후 1분 뒤에 판매자에게 푸시가 나가도 판매에 지장 없음 |
| **재시도 가능** | 실패했을 때 재시도가 가능하다 | 포인트 지급 실패 시 몇 초 뒤 다시 시도하여 성공하면 문제없음 |
| **수동 처리 가능** | 실패했을 때 나중에 수동으로 처리할 수 있다 | 검색 서비스 연동 실패 시 관리 툴로 수동 연동 가능 |
| **무시 가능** | 실패해도 전체 서비스에 치명적이지 않다 | 주문 알림 푸시가 발송되지 않아도 주문 확인이 늦어질 뿐 판매에 문제 없음 |

> 필자는 외부 연동이 4가지 특징 중 일부에 해당하면 비동기로 처리할 수 있는지 검토한다. 실제로 포인트 지급이나 결제 결과 반영 등 여러 기능에 비동기 연동을 적용하여 성능을 개선했다. 또한 연동 서비스에 장애가 발생했을 때 후처리를 용이하게 할 수 있었다.

> **실무 사례**: 저자가 실제로 비동기로 외부 연동을 처리한 사례 중 일부다.
> - **포인트 지급**: 사용자가 미션을 달성하면 포인트를 지급하는 데, 포인트 서비스와의 연동을 비동기로 처리했다.
> - **주문 정보 동기화**: 주문 시스템에 생성된 주문 정보를 회원 관리 시스템에 반영할 때 비동기로 동기화했다.
> - **택배사에 집하 요청**: 회원이 쇼핑몰에서 물건을 주문하면 택배사에 집하 요청을 하는 데, 비동기로 집하 요청 데이터를 전송했다.

### 1.4 비동기 연동의 5가지 방식

비동기 연동은 다양한 방식으로 구현할 수 있다. 이 장에서는 다음의 5가지 방식을 살펴본다.

1. 별도 스레드로 실행하기
2. 메시징 시스템 이용하기
3. 트랜잭션 아웃박스 패턴 사용하기
4. 배치로 연동하기
5. CDC 이용하기

## 2. 별도 스레드로 실행하기

비동기 연동을 하는 가장 쉬운 방법은 별도 스레드로 실행하는 것이다.

### 2.1 새 스레드 생성

푸시를 비동기로 연동하고 싶다면 새로운 스레드를 생성하여 연동 코드를 실행할 수 있다.

```java
public OrderResult placeOrder(OrderRequest req) {
    ... // 주문 생성 처리
    // 별도 스레드를 이용해서 푸시를 비동기로 발송
    new Thread(() -> pushClient.sendPush(pushData)).start();
    return successResult(...); // 푸시 발송을 기다리지 않고 리턴
}
```

### 2.2 스레드 풀 사용

매번 스레드를 생성하는 대신 스레드 풀을 사용하는 방법도 있다.

```java
ExecutorService executor = Executors.newFixedThreadPool(50);

public OrderResult placeOrder(OrderRequest req) {
    ... // 주문 생성 처리
    // 스레드 풀을 이용해서 푸시를 비동기로 발송
    executor.submit(() -> pushClient.sendPush(pushData));
    return successResult(...); // 푸시 발송을 기다리지 않고 리턴
}
```

### 2.3 프레임워크의 비동기 기능 활용

스프링 프레임워크는 `@Async` 애노테이션을 이용한 비동기 실행 기능을 제공한다.

```java
public class PushService {
    @Async
    public void sendPushAsync(PushData pushData) {
        pushClient.sendPush(pushData);
        // ... 기타 코드
    }
}
```

> **실무 팁**: `@Async` 애노테이션을 사용할 때는 메서드 이름에 비동기 실행과 관련된 단어를 추가하는 것이 좋다. 예를 들어 `sendPush()` 대신 `sendPushAsync()`로 이름을 짓는다. 그렇지 않으면 코드를 읽는 사람이 해당 메서드가 비동기로 실행된다는 사실을 알아채기 힘들다.

`sendPush()`라는 이름의 메서드에 `@Async`가 붙어 있다면, 호출하는 쪽에서는 비동기 실행 여부를 알 수 없다. 비동기로 실행된다는 사실을 모른 채 `try-catch`로 익셉션 처리 코드를 추가할 수도 있다. 그러나 비동기로 실행되기 때문에 익셉션이 발생해도 `catch` 블록은 실행되지 않는다.

```java
public OrderResult placeOrder(OrderRequest req) {
    // 주문 생성 처리
    try {
        pushService.sendPush(pushData);
    } catch (Exception ex) {
        // sendPush()가 비동기로 실행되므로 catch 블록은 동작하지 않는다.
        // 에러 처리 코드
    }
    return successResult(...);
}
```

> **실무 사례**: 익셉션이 발생하는데도 이상하게 롤백이 안 된다고 도와달라는 요청을 받은 적이 있었다. 코드에서는 `@Transactional`을 이용해 트랜잭션 처리를 했고, 코드 중간에 실행되는 메서드에서 익셉션이 발생했지만 트랜잭션이 롤백되지 않고 커밋된 문제였다. 코드를 살펴본 결과, 코드 중간에 실행한 메서드가 비동기로 실행되어 익셉션이 전파되지 않았던 것이다. 이로 인해 해당 코드는 정상적으로 실행되었고 트랜잭션이 커밋된 것이다. 트랜잭션 범위 안에서 비동기 코드를 실행할 때는 트랜잭션 연동 여부에 주의해야 한다.

### 2.4 별도 스레드의 오류 처리

별도 스레드로 실행하면 연동 과정에서 발생한 오류 처리에 더 신경 써야 한다. 익셉션을 전파해도 소용없기 때문이다. 별도 스레드로 실행되는 코드는 내부에서 연동 과정에서 발생한 오류를 직접 처리해야 한다.

```java
@Async
public void sendPushAsync(PushData pushData) {
    try {
        pushClient.sendPush(pushData);
    } catch (Exception e) {
        try {
            Thread.sleep(500);
        } catch (Exception ex) {}
        try {
            pushClient.sendPush(pushData); // 재시도를 하거나
        } catch (Exception e1) {
            // 실패를 로그로 남기거나
            ...
        }
    }
}
```

### 알아두기: 스레드와 메모리

스레드는 자체적으로 일정 크기의 메모리를 사용한다. 예를 들어 10만 개의 푸시 메시지를 비동기로 발송하기 위해 순간적으로 10만 개의 스레드를 생성한다고 하자. 스레드 1개가 사용하는 메모리가 256KB라면, 10만 개 스레드를 생성하는 데만 약 24GB의 메모리가 필요하다. 게다가 스레드 스케줄링에 많은 CPU 시간을 사용해서 실행 시간이 매우 느려질 수 있다.

| 방식 | 장점 | 단점 |
|------|------|------|
| **매번 스레드 생성** | 구현이 단순하다 | 메모리 사용량 예측 불가, 대량 생성 시 성능 저하 |
| **스레드 풀** | 메모리 사용량 일정, 스레드 생성 시간 단축 | 풀 크기 초과 시 대기 발생 |
| **가상 스레드/고루틴** | 적은 메모리 사용, 대량 생성 가능 | 네트워크 IO 작업에 적합, CPU 집약적 작업에는 부적합 |

비동기로 실행할 코드가 외부 API 호출이나 DB 연동과 같은 네트워크 IO 작업이라면 자바 언어의 가상 스레드(*Virtual Thread - 런타임에서 관리하는 경량 스레드로, 실제 네이티브 OS 스레드가 아닌 적은 메모리를 사용하는 스레드*)나 Go 언어의 고루틴을 사용하는 것도 방법이다.

## 3. 메시징

서로 다른 시스템 간에 비동기로 연동할 때 주로 사용하는 방식은 메시징 시스템을 사용하는 것이다.

### 3.1 메시징 시스템의 구조

메시징 시스템은 데이터 연동이 필요한 두 시스템 사이에 위치한다.

```
시스템 A ──(메시지 생성/전송)──▶ 메시징 시스템 ──(메시지 전달)──▶ 시스템 B
```

시스템 A가 시스템 B에 데이터를 전달하고 싶을 때, 시스템 A는 전달할 데이터를 가진 메시지를 생성해서 메시징 시스템에 전송한다. 메시징 시스템은 메시지를 다시 시스템 B에 전달하고, 시스템 B는 전달받은 데이터를 이용해서 필요한 작업을 처리한다.

### 3.2 메시징 시스템의 이점

시스템 A에서 시스템 B를 직접 호출하는 것과 비교하면 구조가 더 복잡해졌지만, 대신 다른 이점을 얻을 수 있다.

**첫째, 두 시스템이 서로 영향을 주지 않는다.** 시스템 A의 트래픽이 갑자기 증가하면서 전달할 데이터가 시스템 B의 처리량을 초과하는 상황을 생각해 보자. 직접 연동했다면 시스템 B에 성능 저하가 발생하고, 그 성능 저하는 다시 시스템 A에까지 영향을 미친다. 그러나 메시징 시스템을 사용하면 시스템 B가 느려지더라도 시스템 A는 영향을 받지 않는다. 메시징 시스템은 중간에서 메시지를 보관하는 버퍼 역할을 한다. 시스템 A의 트래픽이 급증하더라도 시스템 B는 자신의 용량에 맞게 메시지를 처리할 수 있다.

**둘째, 확장이 용이하다.** 시스템 A가 시스템 C에도 데이터를 전송해야 한다고 가정하자. 직접 전송했다면 시스템 A에 새로운 코드를 추가해야 한다. 그러나 메시징 시스템을 사용하면 시스템 C를 메시징 시스템에 연결하기만 하면 된다. 시스템 A의 코드를 수정할 필요가 없다.

```
                            ┌──▶ 시스템 B
시스템 A ──▶ 메시징 시스템 ──┤
                            └──▶ 시스템 C
```

### 알아두기: 생산자/소비자, 게시자/구독자

메시징 시스템에서 사용하는 용어로 생산자(*Producer*)와 소비자(*Consumer*)가 있다. 메시지를 생성해서 메시징 시스템에 보내는 측을 생산자라고 한다. 반대로 메시징 시스템으로부터 메시지를 받아 처리하는 측을 소비자라고 한다.

메시징 시스템을 사용하는 구조는 게시/구독(*Pub/Sub*) 구조라고도 표현된다. 이 구조에서 메시지를 생성하고 보내는 쪽을 게시자(*Publisher*)라고 부르며, 메시지를 수신하여 사용하는 쪽을 구독자(*Subscriber*)라고 부른다.

### 3.3 주요 메시징 시스템 비교

이 책을 쓰는 시점을 기준으로 메시징 시스템 용도로 많이 사용되는 기술은 카프카, 래빗MQ, 레디스 pub/sub 등이 있다.

| 특성 | 카프카 | 래빗MQ | 레디스 Pub/Sub |
|------|--------|--------|----------------|
| **처리량** | 초당 백만 개 이상 | 클러스터로 확장 가능하나 카프카보다 더 많은 자원 필요 | 메모리 기반으로 지연 시간이 짧고 래빗MQ 대비 처리량 높음 |
| **확장** | 수평 확장 용이(브로커, 파티션, 소비자 추가) | 클러스터를 통해 확장 | 모델이 단순해서 사용하기 쉬움 |
| **메시지 보관** | 파일에 보관, 유실되지 않음 | 메모리만 사용 설정 시 장애 시 유실 가능 | 기본적으로 영구 메시지 미지원, 구독자 없으면 유실 |
| **순서 보장** | 파티션 단위로 순서 보장, 토픽 수준에서는 보장 불가 | 큐에 등록된 순서대로 소비자에 전송 | - |
| **전달 모델** | 풀(*Pull*) 모델: 소비자가 브로커에서 읽어감 | 푸시(*Push*) 모델: 브로커가 소비자에 전송 | Pub/Sub 모델 |
| **기타** | 소비자가 메시지를 언제든지 재처리 가능 | AMQP, STOMP 등 여러 프로토콜 지원, 우선순위 지정 가능 | 모델이 단순해서 사용하기 쉬움 |

**선택 기준 정리**:
- 메시지가 유실되어도 상관없다면 **레디스 pub/sub**을 고려한다. 카프카나 래빗MQ에 비해 사용법이 간단하고, 적은 장비로 높은 성능을 낼 수 있다.
- 대량 트래픽(초당 수십만에서 수백만 이상의 메시지)이 발생한다면 **카프카**를 고려한다.
- 트래픽 규모가 크지 않고 메시지를 정확하게 순서대로 소비자에 전달해야 하거나, AMQP/STOMP 프로토콜로 연동해야 한다면 **래빗MQ**를 고려한다.

이 외에 NATS, 액티브MQ, 로켓MQ, 펄사 등을 메시징 시스템으로 사용할 수 있다. 클라우드가 제공하는 메시징 시스템(예: AWS SQS)도 좋은 선택이 될 수 있다.

### 알아두기: 경험과 유행

메시징 시스템을 고를 때 사용할 수 있는 또 다른 기준은 조직의 경험이다. 이미 레디스를 운영한 경험이 있는 조직에서 유실이 허용되는 메시징 시스템이 필요하다면 레디스를 사용하면 된다. 카프카 운영 경험이 있다면 초당 수백 개의 메시지를 처리할 때에도 카프카를 선택할 수 있다.

> **핵심 통찰**: 기술을 선택할 때는 개인의 호기심이나 취향보다는 조직의 역량과 지속 가능성을 고려해야 한다. 타당한 이유 없이 단순히 유행한다는 이유로 특정 기술을 선택했다가 두고두고 유지보수에 어려움을 겪는 일이 발생할 수 있다.

### 3.4 메시지 생성 측 고려 사항

메시지를 생성할 때 고려할 점은 **메시지 유실**에 대한 것이다. 타임아웃 등 네트워크 연결이 불안정하면 메시지 전송 실패가 언제든지 발생할 수 있다. 이때 오류 처리를 위해 선택할 수 있는 방법은 3가지다.

| 방법 | 설명 | 적합한 상황 |
|------|------|-------------|
| **무시** | 오류를 무시하고 메시지를 유실시킨다 | 단순 로그 메시지, 로그인 실패 알림 등 유실이 허용되는 경우 |
| **재시도** | 일시적 오류는 재시도로 해결 가능하나, 중복 메시지 전송 주의 | 일시적 네트워크 불안정이 원인인 경우 |
| **실패 로그 기록** | DB나 파일에 실패 로그를 남겨 후처리에 사용 | 나중에 수동 재처리가 필요한 경우 |

재시도 시 주의할 점은 **중복 메시지**가 전송될 수 있다는 것이다. 실제로는 전송에 성공했는데 일시적인 네트워크 오류로 실패한 것으로 인지하고 재시도할 수 있기 때문이다. 메시징 시스템이 중복 수신을 방지하는 기능을 제공하지 않으면, 메시지 소비자가 중복 메시지를 알맞게 처리해야 한다. 메시지마다 고유 식별자를 사용하면 소비자가 중복 메시지 여부를 판단하는 데 도움이 된다.

**DB 트랜잭션과의 연동 문제**

메시지 생산자는 DB 트랜잭션과의 연동도 고려해야 한다. DB 트랜잭션에 실패했는데 메시지가 발송되면 잘못된 데이터가 전달될 수 있다.

```
고객        주문 서비스        DB        메시징 시스템        알림 서비스
 |              |              |              |                  |
 |-- 주문 요청 ->|              |              |                  |
 |              |-- 트랜잭션 시작->|            |                  |
 |              |-- DB 변경 ---->|            |                  |
 |              |-- 메시지 전송 --|----------->|                  |
 |              |-- DB 변경 ---->|            |                  |
 |              |   변경 실패!    |            |-- 메시지 전파 --->|
 |              |-- 트랜잭션 롤백->|           |                  |-- 푸시 발송
 |<-- 에러 응답 -|              |              |                  |
```

위 시나리오에서 주문 서비스는 DB 변경에 실패해서 트랜잭션을 롤백했지만, 메시지는 트랜잭션 롤백 전에 이미 메시징 시스템으로 전송됐다. 고객은 주문에 실패했다는 에러 화면을 봤지만, 잠시 뒤에 주문이 완료됐다는 푸시를 받게 된다.

> **핵심 통찰**: 잘못된 메시지 전송을 방지하려면 트랜잭션이 끝난 뒤에(커밋 또는 롤백 후) 메시지를 전송해야 한다. 이렇게 하면 메시지가 유효한 데이터를 포함할 수 있게 된다.

### 알아두기: 글로벌 트랜잭션과 메시지 연동

여러 DB를 하나의 트랜잭션으로 묶어서 처리하는 것을 글로벌 트랜잭션(*Global Transaction*)이라고 한다. 글로벌 트랜잭션을 구현하는 알고리즘으로 2단계 커밋(*2-Phase Commit*)을 사용하는데, 그래서 글로벌 트랜잭션을 2PC라고 표현하기도 한다.

글로벌 트랜잭션을 지원하는 메시징 시스템(예: 액티브MQ)을 사용하면 DB 수정과 메시지 전송/처리를 한 트랜잭션으로 묶을 수 있다. 하지만 모든 메시징 시스템이 글로벌 트랜잭션을 지원하는 것은 아니며, 2단계 커밋 과정이 추가되면서 처리 속도가 느려지고 동시에 처리할 수 있는 요청이 줄어든다.

> **실무 팁**: 글로벌 트랜잭션이 반드시 필요한 상황이 아니라면 DB 처리와 메시지 연동을 묶지 말자. DB에 데이터를 반영한 뒤에 메시지를 최대한 유실 없이 보내고 싶다면 트랜잭션 아웃박스 패턴을 검토해 보자.

### 3.5 메시지 소비 측 고려 사항

메시지 소비자는 다음 2가지 이유로 동일 메시지를 중복해서 처리할 수 있다.

1. 메시지 생산자가 같은 데이터를 가진 메시지를 메시징 시스템에 두 번 전송
2. 소비자가 메시지를 처리하는 과정에서 오류가 발생해서 메시지 재수신

**고유 메시지 ID를 활용한 중복 처리 방지**

```java
// 고유의 메시지 ID를 사용했을 때의 카프카 소비자 코드 구조 예
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        Message m = messageConverter.convert(record.value());
        if (checkAlreadyHandled(m.getId())) {  // 이미 처리했는지 확인
            continue;                           // 처리하지 않고 무시함
        }
        handle(m);
        recordHandledLog(m.getId());            // 처리 여부 기록
    }
}
```

메시지를 처리했는지 여부는 DB 테이블에 기록하거나 메모리에 집합으로 관리하면 된다. 메모리로 관리할 때는 메모리 부족 에러가 발생하는 것을 막기 위해 일정 개수의 메시지 ID만 유지한다.

**멱등성을 활용한 중복 처리 대응**

메시지 재수신에 따른 중복 처리를 대응하는 또 다른 방법은 4장에서 언급한 멱등성(*Idempotency*)을 갖도록 API를 구현하는 것이다. API가 멱등성을 가지면 동일 요청을 여러 번 해도 결과가 바뀌지 않는다.

**모니터링의 중요성**

메시지 소비자의 처리 속도가 갑자기 느려지면 큐에 메시지가 계속 쌓이게 된다. 메시징 시스템에 따라 큐가 가득 차면 생산자가 메시지를 큐에 넣지 못하게 막는 상황도 발생할 수 있다. 소비자의 성능 저하가 생산자까지 영향을 줄 수 있는 것이다. 또한 메시지 처리가 늦어지면서 사용자는 원하는 기능이 동작하지 않아 불편을 겪게 된다.

### 3.6 메시지 종류: 이벤트와 커맨드

메시지에는 크게 2가지 종류가 있다. 하나는 이벤트(*Event*)이고, 다른 하나는 커맨드(*Command*)이다.

| 이벤트 | 커맨드 |
|--------|--------|
| 주문함 | 포인트 지급하기 |
| 로그인에 실패함 | 로그인 차단하기 |
| 상품 정보를 조회함 | 배송 완료 문자 발송하기 |
| 배송을 완료함 | |

**이벤트**는 어떤 일이 발생했음을 알려주는 메시지다. '주문함' 이벤트는 주문이 생성됐다는 사실을 의미하고, '배송을 완료함' 이벤트는 물건이 고객에게 전달됐다는 사실을 의미한다. 이벤트는 상태 변경(데이터 변경)과 관련된 것도 있고, '로그인에 실패함'이나 '상품 정보를 조회함'처럼 어떤 활동이 일어났다는 사실을 나타내는 경우도 있다.

**커맨드**는 무언가를 요청하는 메시지다. 커맨드 메시지를 수신하는 소비자는 메시지로 요구한 기능을 실행한다. 커맨드 메시지는 수신할 측의 기능 실행에 초점이 맞춰져 있다. 즉 수신자가 정해져 있다.

```
[커맨드: 수신자가 정해져 있다]
학습 서비스 ──(포인트 지급 요청)──▶ 메시징 시스템 ──(포인트 지급 요청)──▶ 포인트 서비스

[이벤트: 수신자가 정해져 있지 않다]
                                                    ┌──▶ 알림 서비스
배송 서비스 ──(배송 완료함)──▶ 메시징 시스템 ──────┤
                                                    └──▶ 주문 서비스
```

이벤트 메시지는 정해진 수신자가 없다. 발생한 사건에 관심이 있는 소비자가 메시지를 수신하는 방식이다. 이벤트 메시지는 소비자 확장에 적합하다. 예를 들어 '배송을 완료함' 이벤트를 주문 서비스가 수신하도록 구성하면, 주문 서비스는 해당 주문의 상태를 변경할 수 있다. 분석 서비스를 추가하고 싶다면 이 서비스 역시 '배송 완료함' 메시지를 수신해서 배송 완료 시점을 기록하고 분석에 사용할 수 있다.

### 알아두기: 궁극적 일관성(Eventual Consistency)

비동기 연동을 설명할 때 자주 등장하는 용어 중 하나가 궁극적 일관성(*Eventual Consistency - 일관성 모델 중 하나로, 즉시가 아닌 일정 시간이 지난 후에야 일관성이 맞춰지는 특성*)이다. 문서에 따라 최종적 일관성이나 결과적 일관성이라고 표현하기도 한다.

비동기 메시징 방식도 이 궁극적 일관성과 유사한 특성을 갖는다. 예를 들어 배송 서비스에서 배송 상태를 '완료'로 변경하더라도, 해당 변경 내용이 메시지를 통해 주문 서비스로 전달되기 전까지는 주문 서비스의 상태는 여전히 '배송 중'일 수 있다. 배송 완료 메시지가 전파되기 전까지는 두 시스템 간의 상태가 서로 불일치할 수 있다.

## 4. 트랜잭션 아웃박스 패턴

메시지 생성 시 고려 사항에서, 잘못된 메시지 발송을 막기 위해 DB 트랜잭션이 완료된 후 메시지를 전송하자는 내용을 언급했었다. 그런데 이렇게 해도 완벽하지는 않다. 재시도나 예외 처리를 넣더라도 메시징 시스템 연동이 실패할 수 있기 때문이다.

### 4.1 핵심 원리

메시지 데이터 자체가 유실되지 않도록 보장하는 방법은 먼저 해당 데이터를 DB에 안전하게 저장해 두는 것이다. 그 뒤, 저장된 메시지를 읽어 메시징 시스템에 전송하면 된다. 이처럼 메시지 데이터를 DB에 보관하는 방식이 바로 트랜잭션 아웃박스 패턴(*Transactional Outbox Pattern*)의 핵심이다.

트랜잭션 아웃박스 패턴은 하나의 DB 트랜잭션 내에서 다음 2가지 작업을 수행한다.

1. 실제 업무 로직에 필요한 DB 변경 작업을 수행한다.
2. 메시지 데이터를 아웃박스 테이블에 추가한다.

아웃박스 테이블에 쌓인 메시지 데이터는 별도의 메시지 중계 프로세스가 주기적으로 읽어서 메시징 시스템에 전송한다.

```
시스템 A              DB                메시지 중계         메시징 시스템
   |                  |                     |                   |
   |-- 1. 트랜잭션 시작 ->|                  |                   |
   |-- 2. 데이터 변경 --->|                  |                   |
   |-- 3. 아웃박스에    ->|                  |                   |
   |   메시지 데이터 저장  |                  |                   |
   |-- 4. 트랜잭션 커밋 ->|                  |                   |
   |                  |                     |                   |
   |                  |<- 5. 메시지 조회 ----|                   |
   |                  |                     |-- 6. 메시지 전송 ->|
   |                  |<- 7. 전송 완료 표기 -|                   |
```

DB 트랜잭션 범위에서 아웃박스 테이블에 메시지 데이터를 추가하므로 메시지 데이터가 유실되지 않는다. 트랜잭션을 롤백하면 메시지 데이터도 함께 롤백되므로 잘못된 메시지 데이터가 전송될 일도 없다.

### 4.2 메시지 중계 서비스 코드

메시지 중계 서비스는 발송하지 않은 메시지 데이터를 조회해서 메시징 시스템에 전송하고, 전송에 성공하면 전송 완료 처리를 하는 과정을 반복한다.

```java
// 이 메서드를 주기적으로 호출해서 메시지를 전송
public void processMessages() {
    // 아웃박스 테이블에서 대기 메시지 데이터를 순서대로 조회함
    List<MessageData> waitingMessages = selectWaitingMessages();
    for (MessageData m : waitingMessages) {
        try {
            sendMessage(m);            // 메시지를 전송함
            markDone(m.getId());       // 발송 완료 표시함
        } catch (Exception ex) {
            handleError(ex);           // 메시지 발송에 실패한 경우 후속 처리함
            break;                     // 에러가 났을 때 멈춤. 이유는 순서대로 발송하기 위함
        }
    }
}
```

특정 메시지를 전송하는 데 실패하면 루프를 멈춘다. 루프를 멈추는 이유는 메시지를 순서대로 발송하기 위함이다. 대기 메시지를 10개 읽어왔는데 중간에 5번째 메시지를 전송할 때 에러가 발생했다고 하자. 이때 중단하지 않고 6번째 이후 메시지를 발송하면 메시지 생성 순서와 다르게 메시지가 발송된다.

### 4.3 발송 완료 표기 방법

발송 완료를 표시하는 방법은 2가지가 있다.

**방법 1: 상태 칼럼 사용**

아웃박스 테이블에 발송 상태 칼럼을 두는 것이다. 발송 대기 상태를 갖는 데이터를 조회하고 발송에 성공하면 발송 완료로 변경하는 식이다.

```sql
-- 발송 대기 목록을 조회
SELECT * FROM outbox
WHERE status = 'WAITING'
ORDER BY id ASC
LIMIT 100;

-- 발송에 성공하면 상태를 완료로 변경
UPDATE outbox
SET status = 'DONE'
WHERE id = ?;
```

**방법 2: 마지막 메시지 ID 기록**

메시지 중계 서비스가 성공적으로 전송한 마지막 메시지 ID를 별도로 기록하는 방식이다. 다음번 대기 메시지를 조회할 때 이 ID 이후의 메시지만 선택하는 것이다.

> 필자는 개인적으로 아웃박스 테이블에 발송 상태 칼럼을 두는 방식을 선호한다. 이 방식은 어디까지 메시지가 처리됐는지 쉽게 모니터링할 수 있기 때문이다. 하지만 2개 이상의 메시지 중계 서비스가 하나의 아웃박스 테이블을 함께 사용하는 환경이라면, 마지막 메시지 ID를 기록하는 방식이 더 적합할 수 있다.

### 4.4 아웃박스 테이블 구조

| 칼럼 | 타입 | 설명 |
|------|------|------|
| `id` | `bigint` | 단순 증가값(PK). 저장된 순서대로 증가하는 값을 사용한다 |
| `messageId` | `varchar` | 메시지 고유 ID(고유 키) |
| `messageType` | `varchar` | 메시지 타입. 'LoginFailed', 'OrderPlaced' 같은 이름을 사용한다 |
| `payload` | `clob` | 메시지 데이터. JSON이나 XML 같은 형식으로 데이터를 담는다 |
| `status` | `varchar` | 이벤트 처리 상태: `WAITING`(대기), `DONE`(완료), `FAILED`(실패함) |
| `failCount` | `int` | 실패 횟수 |
| `occurredAt` | `timestamp` | 메시지 발생 시간 |
| `processedAt` | `timestamp` | 메시지 처리 시간 |
| `failedAt` | `timestamp` | 마지막 실패 시간 |

`status` 칼럼은 '대기', '완료', '실패함'의 3가지 상태를 갖는다. 대기 상태인 메시지 데이터만 조회하기 때문에, 어떤 조건에서 '실패함' 상태로 바꿀지 결정해야 한다.

- **자동 변경**: 메시지 발송에 N회 실패하면 '실패함' 상태로 바꾸고 다음 메시지를 처리할 수 있다
- **수동 변경**: 실패 횟수가 임계치를 넘어가면 모니터링 시스템을 통해 운영팀에 알리고, 운영팀이 수동으로 상태를 변경한다

'실패함' 상태로 바뀐 메시지는 알맞은 후속 조치를 해야 한다. 후속 조치를 하지 않으면 시스템 간 데이터 일관성이 깨질 수 있다. `memo`나 `remark` 같은 칼럼을 추가해서 실패 이유를 기록하면 원인 파악에 도움이 된다.

`status`의 값으로 '제외함(`EXCLUDED`)'을 추가할 수도 있다. 이는 실패가 아니라 수동으로 특정 메시지를 전송하고 싶지 않을 때 사용할 수 있다.

> **실무 팁**: 메시지 발송에 실패했다고 해서 바로 '실패함' 상태로 바꾸지는 말자. 일시적으로 문제가 발생한 것이라면 한두 차례 재시도를 통해 성공적으로 메시지를 발송할 수 있기 때문이다.

## 5. 배치 전송

배치 전송은 데이터를 비동기로 연동하는 가장 전통적인 방법이라고 할 수 있다. 메시징 시스템이 거의 실시간으로 데이터를 연동한다면, 배치는 일정 간격으로 데이터를 전송한다. 예를 들어 결제 승인 데이터를 모아서 다음 날 보내거나, 택배 발송 요청 데이터를 1시간 간격으로 보내는 식이다.

### 5.1 배치 전송의 전형적인 실행 과정

1. DB에서 전송할 데이터를 조회한다.
2. 조회한 결과를 파일로 기록한다.
3. 파일을 연동 시스템에 전송한다.

파일 전송은 FTP/SFTP 같은 파일 전송 프로토콜 혹은 SCP와 같은 명령어를 이용해 수행한다.

### 5.2 파일 형식

주로 사용하는 파일 형식은 다음과 같다.

| 형식 | 예시 | 장점 | 단점 |
|------|------|------|------|
| **구분자 기반** | `값1^값2^값3^값4` | 구현 간단, 파싱 속도 빠름 | 값의 위치에 따라 의미가 결정됨 |
| **이름=값 쌍** | `이름1=값1 이름2=값2` | 위치에 관계없이 어떤 값인지 알 수 있음 | 이름까지 포함되므로 데이터 크기 증가 |
| **JSON** | `{"name":"값1","id":"값2"}` | 대부분의 언어에서 쉽게 변환 가능 | 프로퍼티 이름, 따옴표 등으로 데이터 크기 증가 |

### 5.3 파일 전송 시 협의 사항

파일로 데이터를 주고받는 시스템은 형식 외에도 다음 항목들을 함께 정해야 한다.

**1. 송수신 주체**

파일 생산자와 파일 소비자 중 누가 전송을 담당할지 미리 협의해야 한다.

```
[업로드 방식]                    [다운로드 방식]
파일 생성 시스템 ──업로드──▶      파일 생성 시스템
                 파일 소비 시스템                ◀──다운로드── 파일 소비 시스템
```

> 공격적으로 확장하는 서비스는 소비자에 맞춰서 업로드해 주는 경우가 많다. 예를 들어 결제 서비스 업체는 익일에 결제 내역 데이터를 파일로 생성해서 고객 시스템에 업로드해 준다. 반대로 보안상의 이유로 다운로드 방식을 선호하는 소비자도 있다. 생산자가 파일을 업로드하려면 소비자 시스템을 외부에 노출해야 하기 때문이다.

**2. 시간**

소비자 시스템은 특정 시점에 데이터를 필요로 한다. 예를 들어 매월 5일까지 정산을 마쳐야 하는 조직이라면, 그 전에 월 단위 데이터를 받아야 한다. 배치 파일은 데이터 누락 등 오류에 대응할 수 있는 시간을 벌기 위해 근무가 시작되는 오전 시간대에 전송을 처리할 때가 많다.

**3. 경로**

한 시스템이 여러 서비스로부터 파일을 받을 수 있는데, 이 경우 파일이 저장될 경로나 이름이 충돌하지 않도록 규칙을 정한다.

### 5.4 소비자의 파일 처리 과정

생산자가 소비자로 파일을 업로드할 경우, 소비자는 다음과 같은 방식으로 동작한다.

1. 지정한 경로에 파일이 존재하는지 확인한다.
2. 파일이 존재하면 파일로부터 데이터를 읽어 온다.
3. 파일이 없으면 알맞게 후처리한다.
4. 읽어 온 데이터를 시스템에 반영한다.
5. 처리를 완료한 파일은 다른 폴더로 옮긴다.

처리가 끝나면 파일을 다른 폴더로 이동시킨다. 이렇게 해야 같은 파일이 중복 처리되는 것을 막을 수 있다. 바로 삭제하지 않고 다른 폴더로 백업해 두면 재처리가 필요할 때 활용할 수 있다.

> **실무 사례**: 고객사로부터 파일이 누락됐다는 연락을 받은 적이 있다. 확인해 보니 배치는 정상적으로 실행됐다. 문제는 전송할 데이터가 없으면 파일을 생성하지 않도록 구현되어 있었던 것이다. 고객사 입장에서는 데이터가 없어서 파일이 없는 건지, 오류 때문에 파일이 없는 건지 알 수 없었다. 결국 데이터가 없는 경우에도 빈 파일을 전송하도록 수정했다.

### 5.5 API를 이용한 배치 전송

파일 대신 API를 이용해서 데이터를 일괄로 전송할 때도 있다. API를 사용하면 파일 생성, 파일 전송, 파일 처리 과정이 없으므로 구현이 더 단순해지는 장점이 있다. 데이터 크기가 작거나 처리 항목이 적을 때 API를 이용한 데이터 전송 방식을 고려할 수 있다.

### 5.6 DB를 이용한 배치 전송

배치 전송의 또 다른 방식은 읽기 전용으로 DB를 열어주는 것이다. 같은 조직 내에서 데이터를 전달할 때 이 방식을 사용할 수 있다. 개발 시간이 부족할 때 선택할 수 있는 방법이다.

> 한 10년 전 일이다. 쇼핑 서비스와 택배 시스템을 연동하는 업무를 담당한 적이 있다. API 같은 문서를 기대했는데, DB IP, 계정, 테이블 명세서를 담은 문서가 왔다. 배송 요청 데이터를 해당 테이블에 추가하는 방식이었다. DB로 연동하는 게 어색할 수 있지만, DB 테이블을 API 명세라고 생각하면 된다. 연동 구현 방식이 다를 뿐이다.

### 5.7 재처리 기능 만들기

파일을 지정한 시간에 전송하지 못할 때가 있다. 파일을 생성하는 과정에서 오류가 발생하거나 네트워크 상태가 좋지 않아 전송하지 못하는 경우도 있다.

> **실무 팁**: 어떤 이유에서든 전송에 실패하면 일정 시간 후에 재전송하는 기능을 구현해 두자. 예를 들어 7시에 배치가 실행되고 평균 실행 시간이 20분 정도 걸린다면, 7시 40분쯤 배치가 성공했는지 확인하고 실패했다면 재실행하는 방식이다. 한두 번 정도만 재시도해도 수작업으로 재처리하는 번거로움을 상당히 줄일 수 있다. 재시도를 했음에도 실패하는 경우를 대비해서 수동으로 배치를 쉽게 실행할 수 있도록 명령어나 API를 만들어 두자.

## 6. CDC(Change Data Capture)

마지막으로 살펴볼 비동기 연동 방법은 CDC(*Change Data Capture - 변경된 데이터를 추적하고 판별해서, 변경된 데이터로 작업을 수행할 수 있도록 하는 소프트웨어 설계 패턴*)다.

### 6.1 CDC의 동작 원리

오라클이나 MySQL 같은 DBMS는 데이터가 변경되면 그 변경 내용을 통지하는 기능을 제공한다. CDC는 이 기능을 활용해서 구현한다. CDC 패턴에서는 3개의 구성 요소가 등장한다.

```
      1. 데이터 변경          2. 변경분 통지          3. 변경분 전파
DB ──────────────▶ CDC 처리기 ──────────────▶ 대상 시스템
```

INSERT, UPDATE, DELETE 쿼리를 실행하면 DB의 데이터가 변경된다. DB는 변경된 데이터를 CDC 처리기에 전송한다. DB는 **커밋된 데이터만** 변경된 순서에 맞게 전달한다. 롤백된 데이터가 전달되지 않으며, 잘못된 순서로 데이터가 전달되는 일도 없다.

변경 데이터는 레코드 단위로 전달된다. 1개 레코드를 추가하고 2개 레코드를 수정한 다음 3개 레코드를 삭제했다면 총 6개 레코드에 대한 변경분이 CDC 처리기에 전달된다. 변경분 데이터는 추가인지 수정인지 삭제인지를 구분할 수 있는 플래그를 갖는다. 수정인 경우에는 변경된 칼럼의 이전 값과 이후 값이 포함되어 어떤 칼럼이 어떻게 변경됐는지 확인할 수 있다.

### 6.2 CDC 처리기의 2가지 전파 방식

CDC 처리기는 전달받은 변경 데이터를 확인하고 가공한 뒤에 대상 시스템에 전파한다. 크게 2가지 형태가 있다.

**1. 변경 데이터를 그대로 대상 시스템에 전파**

두 시스템의 데이터가 (거의) 1대 1의 관계를 가질 때 적합하다. 예를 들어 회원 시스템의 회원 데이터를 컨텐츠 시스템의 회원 테이블에 동기화하는 상황에서, 회원 시스템에서 `member` 테이블의 `name` 칼럼을 변경하면 컨텐츠 시스템의 `member` 테이블에도 `name` 칼럼이 동일하게 변경되는 식이다.

**2. 변경 데이터를 가공/변환해서 대상 시스템에 전파**

주문 시스템과 통지 시스템을 연동하는 경우, CDC 처리기는 새로운 주문 데이터가 생성되거나 주문 상태가 '배송 중'으로 변경되면 주문 데이터를 모두 전송할 필요 없이 주문 ID와 변경된 주문 상태만 보내면 된다.

### 6.3 CDC의 전파 대상

목적에 따라 CDC 처리기는 DB, 메시징 시스템, API 등 다양한 대상에 데이터를 전파할 수 있다.

```
[DB 간 데이터 복제]
DB ──(변경분 통지)──▶ CDC 처리기 ──(변경분 전파)──▶ DB

[메시징 시스템을 통한 확장]
                                          ┌──▶ 소비자 1
DB ──▶ CDC 처리기 ──▶ 메시징 시스템 ──────┤
                                          └──▶ 소비자 2
```

메시징 시스템에 데이터를 전파하면 여러 시스템에 변경된 데이터를 전달할 수 있어 확장에 유리하다.

### 알아두기: CDC와 이벤트

CDC는 데이터의 변경분을 전달한다. 이는 어떤 일이 발생했음을 알려주는 이벤트 메시지에 가깝다. 하지만 이벤트처럼 정확하게 의미를 전달하지는 못한다. 데이터의 변경분을 통해서 의미를 도출할 수 있을 뿐이다.

예를 들어 '회원 암호 초기화함' 이벤트는 암호를 초기화했다는 것을 분명하게 나타내지만, '회원 테이블 데이터 변경(이전 암호, 이후 암호)'는 암호가 초기화됐다는 사실을 드러내지 않는다. 단지 암호 칼럼 값이 변경됐다는 것만 알 수 있다.

### 6.4 CDC와 데이터 위치

CDC 처리기는 변경 데이터를 어디까지 처리했는지 기록해야 한다. 예를 들어 MySQL은 바이너리 로그를 이용해서 CDC를 구현하는데, 각 로그 항목은 변경된 데이터와 로그 파일에서의 위치(포지션) 값을 갖는다. 이 위치를 기록해야 CDC 처리기를 재시작할 때 마지막으로 조회한 로그부터 읽어 올 수 있다. 이 위치를 기록하지 않으면 CDC 처리기를 재시작하는 시간 동안 발생한 변경 데이터를 놓치게 된다.

### 6.5 CDC가 유용할 때

> **실무 사례**: 시스템이 복잡해서 연동 코드를 넣기 부담스러울 때 CDC를 유용하게 활용할 수 있었다. 예전에 I사에서 근무할 때 주문 시스템을 새로 구축한 적이 있다. 신규 주문 시스템에서 발생한 주문 데이터를 기존 주문 시스템에 반영해야 하는 요구가 있었다. 하지만 신규 주문 시스템 개발 조직은 연동 코드를 추가하는 것에 난색을 표했다. 일정에 여유가 없기도 했지만, 이미 코드가 복잡해져서 연동 코드를 추가하기가 부담됐기 때문이다.
>
> 이때 선택한 방법이 CDC를 활용하는 것이었다. 신규 주문 시스템의 코드를 수정하지 않고도 CDC를 사용해 타 시스템에 관련 데이터를 전파할 수 있었다. 연동 대상 시스템이 2개였고, 서로 데이터를 처리하는 속도가 달라 중간에 메시징 시스템으로 카프카를 두었다. CDC로 연동 기능을 구현한 덕분에 신규 주문 시스템 개발 일정에 주는 영향을 최소화할 수 있었다.

```
신 주문 시스템 ──(주문 데이터 변경)──▶ DB ──▶ CDC 처리기 ──▶ 카프카 ──┬──▶ 소비자 1 ──▶ 구 주문 DB
                                                                      └──▶ 소비자 2 ──▶ 컨텐츠 서비스
```

## 자주 하는 실수

| 실수 | 문제점 | 해결 방법 |
|------|--------|-----------|
| 비동기 메서드에 `@Async` 같은 비동기 힌트를 이름에 넣지 않음 | 호출하는 쪽에서 비동기 실행 여부를 모르고 잘못된 예외 처리나 트랜잭션 코드를 추가 | 메서드 이름에 `Async` 등 비동기 관련 단어 추가 |
| 트랜잭션 완료 전에 메시지 전송 | DB 롤백되었는데 잘못된 메시지가 이미 전송됨 | 트랜잭션 커밋/롤백 후에 메시지 전송, 또는 아웃박스 패턴 사용 |
| 메시지 중복 처리를 고려하지 않음 | 동일 작업이 두 번 이상 수행되어 데이터 불일치 | 고유 메시지 ID 사용, 멱등성 있는 API 구현 |
| 아웃박스 패턴에서 실패 즉시 `FAILED` 처리 | 일시적 네트워크 오류인데 재시도 기회를 잃음 | 재시도 횟수 설정 후 임계치 초과 시에만 FAILED 처리 |
| 배치 전송 시 데이터 없으면 파일 미생성 | 소비자가 오류인지 데이터 없음인지 구분 불가 | 데이터가 없어도 빈 파일 전송 |
| 배치 전송 재처리 기능 미구현 | 전송 실패 시 매번 수동 재처리 필요 | 자동 재시도 + 수동 재실행 명령어/API 제공 |
| CDC 처리기에서 위치(포지션) 미기록 | 재시작 시 처리 중 발생한 변경 데이터 유실 | 처리 위치를 반드시 기록 |

## 요약

- **동기 연동**은 직관적이고 디버깅이 용이하지만, 외부 연동의 실패나 응답 지연이 전체 서비스에 영향을 미칠 수 있다.
- 외부 연동 결과가 다음 작업에 반드시 필요하지 않다면 **비동기 연동**을 고려한다. 시차 허용, 재시도 가능, 수동 처리 가능, 무시 가능 중 하나에 해당하면 비동기 후보다.
- **별도 스레드**는 가장 쉬운 비동기 방법이지만, 오류를 직접 처리해야 하고 메모리/CPU 자원 관리에 주의해야 한다.
- **메시징 시스템**은 두 시스템을 분리(버퍼 역할)하고 확장을 용이하게 한다. 카프카(대용량), 래빗MQ(정확한 순서/다양한 프로토콜), 레디스 pub/sub(유실 허용/간단함) 등 목적에 맞는 기술을 선택한다.
- 메시지 생산자는 유실, 중복, DB 트랜잭션과의 연동을 고려하고, 소비자는 중복 처리 방지(고유 ID, 멱등성)와 모니터링에 신경 써야 한다.
- 메시지는 **이벤트**(발생 사실 알림, 수신자 미지정)와 **커맨드**(기능 실행 요청, 수신자 지정)로 구분되며, 이벤트가 소비자 확장에 더 적합하다.
- **트랜잭션 아웃박스 패턴**은 DB 트랜잭션 내에서 메시지 데이터를 아웃박스 테이블에 저장하고, 별도 중계 프로세스가 이를 메시징 시스템에 전송하여 메시지 유실과 잘못된 메시지 전송을 방지한다.
- **배치 전송**은 가장 전통적인 비동기 연동 방식으로, 파일(구분자/이름=값/JSON)이나 API를 통해 일정 간격으로 데이터를 전달한다. 재처리 기능을 반드시 구현해 둔다.
- **CDC**는 DB의 변경 데이터를 추적하여 대상 시스템에 전파하는 패턴으로, 기존 시스템의 코드를 수정하지 않고도 연동할 수 있어 복잡한 시스템에서 유용하다.
- 모든 연동을 비동기로 구현할 필요는 없지만, 비동기로 구현했을 때 발생하는 복잡도 증가보다 얻는 이점이 더 크다면 비동기 연동을 고려해 보자.

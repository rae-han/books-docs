# Chapter 10: Network Fundamentals (모르면 답답해지는 네트워크 기초)

## 핵심 질문

서버 개발자가 알아야 할 네트워크 기초 지식은 무엇인가? IP 주소, 도메인, NAT, VPN, 프로토콜(TCP/UDP/QUIC)의 동작 원리를 이해하면 네트워크 관련 문제를 어떻게 더 효율적으로 해결할 수 있는가?

## 1. 네트워크 기초를 모르면

> 몇 년 전 사례다. 팀원이 잘 안 되는 문제가 있다면서 같이 봐달라고 했다. 서버에서 외부 업체가 제공한 API에 연결이 안 되는 문제였다. 실제로 같이 확인해 보니 연결이 안 됐다. 그런데 API 제공 업체에 문의하면 잘 되니 다시 확인해 보라는 답변만 온다면서 하소연을 했다.
>
> 같이 원인을 찾아봤다. 코드에 적혀 있는 API 주소가 유효한지 확인했다. 방화벽의 아웃바운드 설정이 잘못됐는지도 확인했다. API 제공자 측의 인바운드 허용 정책도 재차 확인했다. 그런데 모두 이상 없었다.
>
> 이렇게 한참을 헤매다가 원인을 알아냈다. 동료가 접속 허용을 요청한 IP를 잘못 기재한 것이었다. 외부 API를 호출할 서버는 2개의 공인 IP와 연결되어 있었다. 하나는 외부에서 서버에 연결할 때 사용하는 IP고 다른 하나는 서버에서 외부로 나갈 때 사용하는 IP다. 이 둘 중에 후자의 IP를 외부 업체에 공유해야 했는데 전자의 IP를 알려 줘서 연결이 안 됐던 것이다.

서버 개발자가 네트워크 엔지니어만큼 네트워크 관련 지식을 깊이 있게 알 필요는 없다. 하지만 네트워크 지식이 전무하면 네트워크 관련 문제가 생겼을 때 쉽게 해결할 수 있는 문제도 오랜 시간 동안 처리하지 못하게 된다. 기초적인 네트워크 지식 정도는 알아야 한다.

> **핵심 통찰**: 위 사례에서 인바운드 IP와 아웃바운드 IP가 다를 수 있다는 기본적인 네트워크 지식만 있었다면 훨씬 빠르게 문제를 해결할 수 있었다. 네트워크 기초 지식은 문제 해결 속도에 직접적인 영향을 준다.

## 2. 노드, 네트워크, 라우터

### 2.1 노드와 네트워크

데이터를 송수신하는 모든 장치를 노드(*Node*)라고 표현한다. 우리가 사용하는 휴대폰, 노트북, 서버 장비 등이 노드에 해당한다. 이런 각 노드가 서로 데이터를 주고받기 위해 연결된 시스템을 네트워크(*Network*)라고 한다.

```
┌─────────────────────────┐
│       네트워크            │
│                         │
│  [노드] ── [노드]       │
│    │                    │
│  [노드]                 │
└─────────────────────────┘
```

네트워크의 예로 가정에서 사용하는 공유기에 연결된 장치들을 들 수 있다. 공유기에 연결된 휴대폰이나 컴퓨터가 하나의 네트워크를 구성한다. 비슷하게 사무실에서 사용하는 컴퓨터도 하나의 네트워크를 구성한다.

### 알아두기: 패킷

노드가 네트워크를 통해 전송하는 데이터의 단위를 패킷(*Packet*)이라고 한다. 패킷은 헤더와 페이로드로 구성된다. 헤더는 패킷의 발신자와 수신자 정보를 포함하며, 페이로드에는 실제 데이터가 포함된다. 데이터는 일정 크기를 가진 여러 패킷으로 나뉘어 전송된다.

### 2.2 라우터

내가 사용하는 휴대폰과 앱에서 연결하는 서버는 서로 다른 네트워크에 포함된다. 서로 다른 네트워크에 속한 노드는 직접 연결해서 패킷을 송수신할 수 없다. 이때 사용되는 것이 라우터(*Router*)다. 라우터는 네트워크 간에 패킷을 전송하는 역할을 한다. 우리가 휴대폰 앱에서 서버에 연결하면 여러 라우터를 거쳐 최종적으로 패킷이 전송된다.

```
[휴대폰] ──▶ [라우터] ──▶ [라우터] ──▶ [라우터] ──▶ [서버]
  네트워크 A                                       네트워크 B
```

서울에서 부산까지 가는 고속도로 경로가 다양한 것처럼, 두 노드 간에는 다양한 라우터 이동 경로가 존재한다. 각 라우터는 목적지 노드의 주소에 따라 알맞은 다음 라우터를 선택해 데이터를 전송한다.

## 3. IP 주소와 도메인

### 3.1 IP 주소

택배사에서 물건을 배달하기 위해 주소를 사용하는 것처럼, 노드에 패킷을 전송할 때에도 주소를 사용한다. 네트워크에서 각 노드를 구분하기 위해 사용하는 주소를 IP 주소라고 한다.

현재 일반적으로 사용하는 IP 주소는 IPv4 주소이다. IPv4 주소는 `223.130.192.248`과 같이 1바이트(8비트) 4개의 숫자 블록으로 구성되어 있다. 각 숫자 블록은 0부터 255까지의 값을 갖는다.

### 알아두기: IPv6 주소

IPv4 주소는 32비트로 구성되므로 약 43억 개의 고유한 주소가 존재한다. 인터넷이 대중화되기 이전인 1990년대 초부터 IP 주소가 고갈될 것으로 예측했다. 이 당시 세계 인구는 이미 50억이 넘었기에 43억 개가 전부인 IPv4 주소가 부족할 것이라 예상했다. 그래서 더 많은 노드에 고유한 주소를 부여할 수 있는 IPv6(버전 6) 주소를 만들었다.

| 구분 | IPv4 | IPv6 |
|------|------|------|
| **비트 수** | 32비트 | 128비트 |
| **주소 개수** | 약 43억 개 | 사실상 무한 |
| **표기** | `223.130.192.248` | `fe80::1` (16진수) |
| **현재 상태** | 여전히 주류 | 점진적 전환 중 |

2000년대 초반에는 멀지 않아 IPv6 주소를 사용할 것이라고 전망하는 기사도 나오곤 했다. 하지만 지금도 여전히 IPv4 주소를 사용하고 있다. IP 주소가 고갈되지 않은 이유에는 사설 IP와 NAT가 있다.

### 3.2 도메인 이름과 DNS

각 노드의 IP 주소를 기억하기는 쉽지 않다. 그래서 IP 주소에 기억하기 쉬운 이름을 붙이는데, 그 이름이 도메인 이름(*Domain Name*)이다. 도메인 이름과 IP 주소를 변환하는 체계가 DNS(*Domain Name System*)다. DNS를 일종의 인터넷 전화번호부라고 생각하면 된다.

도메인 이름은 계층 구조를 갖는다. 도메인 이름의 각 계층은 점(`.`)으로 구분되며 오른쪽이 상위 계층이고 왼쪽이 하위 계층이다.

```
www  .  naver  .  com
 │       │       │
3차    2차     최상위(TLD)
도메인  도메인   도메인
```

**일반 최상위 도메인** (`com`, `org`, `net`, `gov`, `app`, `biz`, `tech` 등):
- 두 번째 계층인 2차 도메인이 도메인의 주요 이름이 된다
- 주로 회사 이름이나 브랜드 이름을 2차 도메인으로 사용한다
- `naver.com`이나 `google.com`에서 `naver`와 `google`이 2차 도메인에 해당한다
- 3차 이후는 `cafe.naver.com`, `www.google.com`과 같이 용도에 맞는 이름을 사용한다

**국가 최상위 도메인** (`kr`, `jp`, `au` 등):
- 두 번째 계층까지 미리 정의되어 있다
- 예를 들어 `ac.kr`은 대학 같은 교육 기관 용도, `co.kr`은 기업 용도로 사용한다
- 세 번째 계층이 도메인의 주요 이름이 된다 (예: `gasapp.co.kr`)

### 3.3 DNS 서버를 통한 IP 주소 확인

도메인 이름에 해당하는 IP 주소는 DNS 서버를 통해 확인한다.

```
[웹 브라우저]                    [DNS 서버]
     │                              │
     │──(1) www.naver.com IP 문의──▶│
     │                              │
     │◀──(2) 223.130.192.248 응답──│
     │
     │──(3) 데이터 전송──▶ [서버: 223.130.192.248]
```

브라우저에 `www.naver.com`을 입력하면, 브라우저는 DNS 서버에 해당 도메인의 IP 주소를 물어본다. DNS 서버는 응답으로 대응하는 IP 주소를 알려준다. 브라우저는 응답으로 받은 IP 주소에 패킷을 전송한다.

### 알아두기: localhost와 루프백 주소

로컬에서 서버를 띄우고 연결할 때 사용하는 `localhost`는 특별한 주소다. 컴퓨터에서 `localhost`의 IP 주소는 `127.0.0.1`이다. `127.0.0.1`은 루프백(*Loopback*) 주소로서 자기 자신을 참조할 때 사용하는 IP 주소이다.

### 알아두기: hosts 파일

각 컴퓨터는 hosts 파일을 갖고 있다. 리눅스에서는 `/etc` 디렉토리에, 윈도우에서는 `C:\Windows\System32\drivers\etc` 디렉토리에 위치한다. 이 파일은 호스트 이름과 IP 주소에 대한 매핑을 정의한 파일이다. DNS 서버보다 hosts 파일에 매핑된 설정이 우선이다. 참고로 `localhost`에 대한 IP 매핑도 이 파일에 정의되어 있다.

### 알아두기: 도메인 이름과 IP 주소 개수

도메인 이름에 매핑되는 IP 주소는 여러 개일 수 있다. `nslookup` 명령어를 사용하면 도메인의 IP 주소를 확인할 수 있다.

```bash
$ nslookup www.naver.com
...생략
www.naver.com canonical name = www.naver.com.nheos.com.
Name: www.naver.com.nheos.com
Address: 223.130.192.247
Name: www.naver.com.nheos.com
Address: 223.130.192.248
Name: www.naver.com.nheos.com
Address: 223.130.200.219
Name: www.naver.com.nheos.com
Address: 223.130.200.236
```

한 도메인 이름에 IP 주소를 여러 개 매핑하는 이유 중 하나는 **부하 분산**에 있다. 브라우저에서 도메인 이름의 IP 주소를 요청하면 DNS 서버는 등록된 IP를 번갈아 제공한다. 이를 통해 여러 서버에 분산해서 요청을 보낸다.

### 3.4 고정 IP와 동적 IP

동일 네트워크상에서 각 노드는 서로 다른 IP 주소를 가져야 한다. IP 주소를 사용해서 패킷을 전송할 노드를 선택하기 때문에, 같은 IP 주소를 가진 노드가 존재하면 IP 충돌이 발생한다.

| 구분 | 고정 IP | 동적 IP |
|------|---------|---------|
| **특징** | 노드가 고정된 IP를 갖는다 | 네트워크에 연결할 때마다 IP를 할당받는다 |
| **할당 방식** | IP 주소를 직접 지정 | DHCP(*Dynamic Host Configuration Protocol*) 서버가 제공 |
| **사용 예** | 서버 (바뀌지 않는 IP 필요) | 가정 공유기에 연결된 기기 |
| **제공 정보** | 직접 설정 | IP 주소, 게이트웨이 주소, 서브넷 마스크, DNS 서버 주소 모두 DHCP가 제공 |

> **실무 팁**: 동적 IP를 사용한다고 해서 매번 IP가 바뀌는 것은 아니다. DHCP 서버에 따라 일정 시간 동안 동일한 IP를 할당할 때도 있다. 보안이 중요한 곳은 IP로 사용자를 식별하기 위해 노드마다 IP를 고정해서 할당하기도 한다.

### 3.5 공인 IP와 사설 IP

웹 브라우저에 `www.google.com` 같은 도메인 이름을 입력하면 DNS 서버로부터 IP 주소를 받아와 해당 IP 주소에 접속한다. 이때 IP 주소는 인터넷에서 접근 가능한 IP 주소로서 공인(*Public*) IP 주소라고 부른다. 공인 IP 주소는 인터넷에 접속하는 모든 네트워크에 적용되는 주소이며, 공인 IP 주소를 가진 노드는 (방화벽으로 막지 않았다면) 누구나 연결할 수 있다.

모든 네트워크에 적용되는 공인 IP 주소와 달리, 네트워크 내부에만 적용되는 사설(*Private*) IP 주소도 있다. 사설 IP 주소는 특정 네트워크에 속한 노드에 할당하는 주소로서, 네트워크 외부에서 접근할 수 없다. 공유기에 연결된 휴대폰, 태블릿, 컴퓨터가 사설 IP 주소를 갖는다.

| 구분 | 공인 IP | 사설 IP |
|------|---------|---------|
| **범위** | 인터넷 전체에서 고유 | 네트워크 내부에서만 고유 |
| **접근** | 외부에서 접근 가능 | 네트워크 외부에서 접근 불가 |
| **중복** | 서로 같은 주소 불가 | 네트워크가 다르면 같은 주소 가능 |
| **사용 예** | 웹 서버, 공개 API | 가정/사무실 내부 기기 |

**사설 IP 주소 범위:**

사설 IP로 사용할 수 있는 주소 범위는 다음의 세 대역으로 제한되어 있다:

- `192.168.x.x`
- `10.x.x.x`
- `172.16.x.x` ~ `172.31.x.x`

우리가 흔히 사용하는 무선 공유기는 `192.168` 대역을 주로 사용한다. `ifconfig`(리눅스/macOS)나 `ipconfig`(윈도우) 명령어를 실행하면 현재 사용 중인 IP 주소를 확인할 수 있다.

> **핵심 통찰**: 인터넷 초창기에는 회사의 각 PC마다 공인 IP를 할당받아 사용하기도 했다. 만약 지금까지 PC와 휴대폰 등 모든 기기가 공인 IP를 사용했다면 IPv4 주소는 한참 전에 고갈됐을 것이다. 사설 IP 덕에 아직까지 IPv4 주소를 사용할 수 있다. 서버 네트워크, 회사, 가정 같은 환경에서는 사설 IP를 사용해서 필요한 공인 IPv4 주소 개수가 적다. 홈페이지나 공개 API처럼 인터넷에서 서비스를 제공해야 하는 대상에 한해 공인 IP 주소를 사용한다.

## 4. NAT

NAT(*Network Address Translation - 네트워크 주소 변환*)는 네트워크 주소를 변환하는 기술이다. 인터넷에 연결하려면 내부에서 사용하는 사설 IP와 인터넷에서 사용하는 공인 IP 주소 간의 변환이 필요한데, NAT가 이 변환을 담당한다.

NAT는 주로 인터넷에 연결된 라우터 같은 네트워크 장비가 담당한다.

### 4.1 SNAT (Source NAT)

라우터는 내부 네트워크에서 나가는 패킷의 사설 IP를 공인 IP로 변환한다. 소스 IP의 주소를 변환한다고 해서 이를 SNAT(*Source NAT*)라고 한다.

```
[192.168.1.10] ──▶ [라우터/NAT] ──▶ 인터넷
 (사설 IP)         218.39.xx.xx로
                   소스 IP 변환
                      (SNAT)
```

### 4.2 DNAT (Destination NAT)

공인 IP로 들어온 패킷의 목적지를 사설 IP로 변환하기도 하는데, 목적지 IP의 주소를 변환한다고 해서 이를 DNAT(*Destination NAT*)라고 한다.

```
                              ┌─ [서버 1: 10.1.1.101]
[58.221.xx.xx] ──▶ [네트워크  ──┤
 (클라이언트)       장비/DNAT]  └─ [서버 2: 10.1.1.102]
                   10.1.1.101로
                   목적지 IP 변환
```

DNAT는 서버를 구성할 때 사용된다. 일반적으로 네트워크를 구성할 때 보안, 이중화 등을 고려해 서버 노드는 공인 IP가 아닌 사설 IP 주소를 갖는다. 공인 IP 주소는 네트워크 연결을 관리하는 장비(라우터, 방화벽 등)에 할당한다. 해당 장비는 DNAT를 이용해서 공인 IP로 들어온 패킷을 사설 IP를 가진 서버 노드에 전송한다.

| 구분 | SNAT | DNAT |
|------|------|------|
| **변환 대상** | 소스(출발지) IP | 목적지(도착지) IP |
| **방향** | 내부 → 외부 | 외부 → 내부 |
| **변환** | 사설 IP → 공인 IP | 공인 IP → 사설 IP |
| **사용 예** | 내부 PC가 인터넷 접속 | 외부에서 서버에 접속 |

### 알아두기: 공인 IP 알아내기

SNAT는 사설 IP를 공인 IP로 변환한다고 했다. 이렇게 변환된 공인 IP 주소는 어떻게 알 수 있을까? 간단한 방법은 공인 IP 주소를 알려주는 `https://ifconfig.me` 같은 사이트에 접속해서 확인하는 것이다. 리눅스에서도 `curl ifconfig.me` 명령어로 공인 IP 주소를 확인할 수 있다.

> **실무 팁**: 이 장의 도입부 사례처럼, 외부 API를 호출할 때는 서버의 아웃바운드 공인 IP(SNAT로 변환되는 IP)를 확인해야 한다. 인바운드용 공인 IP와 아웃바운드용 공인 IP가 다를 수 있으므로, `curl ifconfig.me`와 같은 방법으로 실제 아웃바운드 IP를 확인하고 외부 업체에 공유하자.

## 5. VPN

백엔드 서버를 개발하고 운영하려면 서버 노드에 접속하거나 DB에 접속해야 할 일이 생긴다. 서버 노드에 SSH로 접속해서 프로세스를 확인하거나 OS 설정을 변경해야 할 때가 있고, DB에 접속해서 SQL을 실행해야 할 때도 있다.

### 5.1 서버 네트워크 접근의 문제

서버 네트워크에 존재하는 노드는 사설 IP를 사용한다. 외부 네트워크에서는 사설 IP에 직접 접근할 수 없다. NAT를 사용하면 공인 IP를 통해 사설 IP를 가진 서버에 접근할 수는 있지만, 노드 개수가 많은 경우 모든 사설 IP마다 공인 IP를 매핑할 수 없기도 하다. 설사 가능하다고 하더라도 보안에 문제가 생길 수 있다. 서버 네트워크의 모든 노드가 공인 IP로 노출되기 때문이다.

```
[일반 사용자] ──(ssh 연결)──▶ ┌─ [서버 1: 10.1.1.101]
                              │
[개발자]                      ├─ [서버 2: 10.1.1.102]    서버
 사무실 네트워크  ──(DB 연결)──▶│                         네트워크
                              └─ [DB: 10.1.2.10]
```

서버를 운영하려면 서버 네트워크에 구성된 노드에 안전하게 접근할 수 있어야 한다.

### 5.2 VPN의 동작 원리

안전한 접근 방법이 바로 VPN(*Virtual Private Network - 가상 사설 네트워크*)이다. VPN은 인터넷과 같은 공용 네트워크에서 서로 다른 네트워크 간에 암호화된 연결을 제공한다. 두 네트워크는 마치 하나의 사설 네트워크에 존재하는 것처럼 연결될 수 있다. 이를 통해 개발자는 안전하게 서버 네트워크에 접근할 수 있다.

```
[개발자]         [VPN]  ~~~암호화 터널~~~  [VPN]         [서버 1: 10.1.1.101]
 사무실      ──▶ 장비                     장비  ──▶     [서버 2: 10.1.1.102]
 네트워크                                              [DB: 10.1.2.10]
                                                        서버 네트워크
```

개발자는 전용 VPN 클라이언트를 통해 VPN에 접근할 수 있다. 집이나 카페 같은 외부 공간에서 서버 네트워크에 접근해야 할 때 이 방식을 사용한다.

```
[VPN 클라이언트] ~~~암호화 터널~~~ [VPN 장비] ──▶ 서버 네트워크
   (외부 환경)
```

> **핵심 통찰**: VPN은 물리적으로 떨어진 두 네트워크를 암호화된 터널로 연결해, 마치 같은 사설 네트워크에 있는 것처럼 통신할 수 있게 해준다. 서버 노드를 공인 IP로 직접 노출하지 않으면서도 안전하게 접근할 수 있는 핵심 기술이다.

## 6. 프로토콜과 TCP, UDP, QUIC

네트워크상에서 두 노드가 데이터를 주고받기 위해 정의한 규칙을 프로토콜(*Protocol*)이라고 한다. 네트워크는 여러 계층으로 구성되며, 각 계층마다 사용하는 프로토콜이 존재한다.

### 6.1 TCP/IP 모델

TCP/IP 모델에는 다음과 같은 계층이 있으며, 각 계층별로 사용되는 주요 프로토콜이 있다:

| 계층 | 주요 프로토콜 |
|------|-------------|
| **애플리케이션**(*Application*) 계층 | HTTP, FTP, SMTP |
| **전송**(*Transport*) 계층 | TCP, UDP |
| **네트워크**(*Network*) 계층 | IP |
| **데이터 링크**(*Data Link*) 계층 | Ethernet 등 |
| **물리**(*Physical*) 계층 | 전기 신호, 광 신호 등 |

> **실무 팁**: TCP/IP 모델은 애플리케이션, 전송, 인터넷, 네트워크 인터페이스의 4개 계층으로도 표현한다. 5개 계층은 네트워크 인터페이스 계층을 데이터 링크 계층과 물리 계층으로 나누어 표현한 것이다. 개발자는 주로 전송 계층과 애플리케이션 계층의 프로토콜을 사용한다.

### 6.2 TCP (Transmission Control Protocol)

TCP는 연결 기반 프로토콜이다. 통화를 하기 위해 한 명은 전화를 걸고 한 사람은 전화를 받은 뒤에 대화를 주고받는 것처럼, TCP도 두 노드 간에 먼저 연결을 맺은 뒤에 데이터를 주고받는다.

**3-Way Handshake:**

TCP에서 두 노드가 연결을 맺기 위한 과정을 3-Way Handshake라고 부른다.

```
[클라이언트]                    [서버]
     │                           │
     │──────── SYN ─────────────▶│  (1) 연결 요청
     │                           │
     │◀─────── SYN-ACK ─────────│  (2) 요청 수락 + 연결 요청
     │                           │
     │──────── ACK ─────────────▶│  (3) 수락 확인
     │                           │
     │    ◀═══ 연결 완료 ═══▶    │
```

클라이언트와 서버는 이 과정을 거친 뒤에 연결이 완료되며, 이후 서로 데이터를 주고받을 수 있다.

**TCP의 장점 - 신뢰성:**

TCP는 패킷 순서를 보장하고 패킷이 유실될 경우 재전송하는 기능을 제공해 안정적으로 데이터를 전송할 수 있다. 이런 이유로 애플리케이션 계층의 HTTP, SMTP와 같은 많은 프로토콜이 TCP를 기반으로 동작한다.

**TCP의 단점 - 속도:**

TCP가 전송을 보장하기 위해 사용하는 시퀀스 번호, 확인 응답, 재전송 등이 추가되면서 전송 속도는 (UDP 대비) 상대적으로 느려진다. 예를 들어 일부 패킷이 유실되면 해당 패킷이 도착할 때까지 이후 패킷을 제대로 처리하지 못하는 HOL 블로킹(*Head-of-Line Blocking*) 문제가 있는데, 이는 전체적인 전송 속도를 느리게 만든다.

### 6.3 UDP (User Datagram Protocol)

TCP와 달리 UDP는 연결 과정 없이 바로 데이터를 전송한다. UDP는 데이터가 정상적으로 전송됐는지 알 수 없고, 순서를 보장하지도 않는다. UDP를 사용하는 애플리케이션은 데이터 유실과 같은 상황이 발생할 수 있다고 가정하고 개발해야 한다.

UDP는 응답 확인이나 패킷 정렬과 같은 과정을 거치지 않기 때문에 전송 속도는 TCP 대비 빠르다. 이런 이유로 UDP는 속도가 중요하거나 일부 데이터가 유실되더라도 문제되지 않는 통신에 주로 사용된다. UDP의 사용 예로는 DNS, VoIP, 게임 등이 있다.

| 구분 | TCP | UDP |
|------|-----|-----|
| **연결** | 연결 기반 (3-Way Handshake) | 비연결 (바로 전송) |
| **신뢰성** | 패킷 순서 보장, 유실 시 재전송 | 순서 미보장, 유실 감지 안 함 |
| **속도** | 상대적으로 느림 | 빠름 |
| **HOL 블로킹** | 발생 가능 | 없음 |
| **사용 예** | HTTP, SMTP, FTP | DNS, VoIP, 게임 |
| **핵심 가치** | 안정성 | 속도 |

### 6.4 QUIC

TCP는 신뢰성이 있지만 느리다. 반면에 UDP는 빠르지만 신뢰성이 없다. 이 둘을 합쳐 빠르면서도 신뢰성이 있는 프로토콜이 바로 QUIC이다.

**QUIC의 핵심 특징:**

1. **UDP 기반**: QUIC은 UDP를 기반으로 하면서, TCP의 연결 관리 기능을 QUIC 프로토콜 수준에서 제공한다. 데이터에 연결 ID(*Connection ID*)를 포함시켜 두 노드 간의 연결을 유지한다. 혼잡 제어나 패킷 유실 복구와 같은 기능도 QUIC 프로토콜 수준에서 제어한다.

2. **TLS 통합**: QUIC은 TLS(*Transport Layer Security*)를 통합했다. QUIC 패킷은 기본적으로 암호화되어 전송된다. TCP 기준의 HTTPS는 TCP 연결을 위한 3-Way Handshake와 TLS 연결을 위한 TLS Handshake를 진행해서 여러 차례 통신을 한 뒤에 연결이 수립되는데, QUIC은 TLS를 통합함으로써 이 과정을 단축했다.

3. **멀티플렉싱**: QUIC은 한 연결에서 여러 스트림을 동시에 처리할 수 있다. 1개 스트림에서 HOL 블로킹이 발생해도 다른 스트림에는 영향을 주지 않는다. HTTP/2도 멀티플렉싱을 지원하지만 TCP 자체에서 발생하는 HOL 블로킹 문제를 피할 수는 없다.

**프로토콜 스택 비교:**

```
[HTTP/1.1]        [HTTP/2]          [HTTP/3]
     │                │                 │
   [TLS]           [TLS]          [QUIC (TLS 통합)]
     │                │                 │
   [TCP]           [TCP]            [UDP]
     │                │                 │
   [IP]             [IP]             [IP]
```

크롬, 에지, 사파리 등 주요 브라우저는 이미 HTTP/3을 지원하고 있다. 구글, 페이스북 등 주요 기업도 HTTP/3을 지원하고 있으며, 아카마이, 클라우드플레어, AWS 클라우드프론트 등 주요 CDN 서비스도 HTTP/3을 지원하고 있다. 브라우저 개발자 도구의 네트워크 탭에서 Protocol 값이 `h3`으로 표시되면 HTTP/3을 사용한 연결이다.

### 알아두기: TCP 연결은 65,535개가 한계인가?

포트 번호는 부호 없는 16비트 정수를 사용한다. 그래서 65,535가 포트 번호의 최대값이다. 포트 번호의 최대값이 65,535이다 보니 OS에서 생성할 수 있는 TCP 연결 개수가 65,535개라고 오해하는 경우가 있다. 하지만 실제로 한 장비에서 생성할 수 있는 TCP 연결 개수는 이론적으로 훨씬 많다.

TCP의 각 연결은 `(로컬 IP, 로컬 포트, 원격 IP, 원격 포트)`로 구분된다. IP는 32비트이고 포트는 16비트이다. 따라서 로컬 IP가 고정되어 있을 때 한 장비가 생성할 수 있는 연결 개수는 이론적으로 다음과 같다:

```
2^16 (로컬 포트) × 2^32 (원격 IP) × 2^16 (원격 포트) = 2^64 (약 1,844경)
```

하나의 로컬 IP에서 **특정 원격 IP의 1개 포트에 연결할 수 있는 TCP 연결 개수가 65,535**인 것이다. 이론적으로 생성할 수 있는 TCP 연결 개수는 매우 크지만, 실제로 생성할 수 있는 연결 개수는 OS 설정(파일 디스크립터 개수, 포트 범위 설정 등)에 따라 제약을 받는다.

## 요약

- **노드**는 데이터를 송수신하는 모든 장치이며, 노드들이 연결된 시스템이 **네트워크**다. 서로 다른 네트워크 간 패킷 전송은 **라우터**가 담당한다
- **IP 주소**는 네트워크에서 노드를 구분하는 주소이며, 현재 주로 사용하는 IPv4는 32비트(약 43억 개)로 구성된다
- **도메인 이름**은 IP 주소에 붙이는 기억하기 쉬운 이름이며, **DNS**가 도메인 이름과 IP 주소 간의 변환을 담당한다
- IP 주소는 **고정 IP**(서버 등)와 **동적 IP**(가정 공유기 등)로 나뉘며, **공인 IP**(인터넷 전체에서 고유)와 **사설 IP**(네트워크 내부에서만 유효)로도 구분된다
- **NAT**는 사설 IP와 공인 IP 간의 변환을 담당한다. 내부에서 외부로 나갈 때는 **SNAT**, 외부에서 내부로 들어올 때는 **DNAT**가 사용된다
- **VPN**은 공용 네트워크에서 암호화된 터널을 통해 서로 다른 네트워크를 안전하게 연결한다. 서버 네트워크에 접근할 때 필수적이다
- **TCP**는 3-Way Handshake를 통해 연결을 수립하고, 패킷 순서 보장과 유실 재전송으로 신뢰성을 제공하지만, HOL 블로킹 등으로 속도가 느리다
- **UDP**는 연결 과정 없이 바로 전송하여 속도가 빠르지만, 순서 보장이나 유실 복구를 제공하지 않는다. DNS, VoIP, 게임 등에 사용된다
- **QUIC**은 UDP 기반으로 TCP의 신뢰성과 UDP의 속도를 결합한 프로토콜이다. TLS를 통합하여 연결 수립 과정을 단축하고, 멀티플렉싱을 지원하여 스트림 간 HOL 블로킹 문제를 해결한다
- **HTTP/3**은 QUIC을 기반으로 하며, 주요 브라우저와 CDN 서비스에서 이미 지원하고 있다

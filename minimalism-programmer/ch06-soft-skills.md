# Chapter 6: Soft Skills (소프트스킬)

## 핵심 질문

코드와 도구를 잘 단순화해도 *사람 사이의 마찰*이 남으면 단순함은 무너진다. 의견 대립·공감·소통이라는 소프트스킬을 어떻게 *단순함의 도구*로 활용할 것인가?

> 이 장은 약간은 속죄하는 마음으로 씁니다. 이 책의 모든 내용 중에서 소프트스킬은 제가 가장 많이 개선해야 할 분야거든요.

저자가 직접 *못한다*고 고백하며 시작하는 챕터다. 그래서 더 신뢰가 간다.

## 1. 왜 소프트스킬인가

예전에는 한 명의 슈퍼 개발자가 모든 것을 머릿속에 담을 수 있었다. 오늘날 소프트웨어는 *팀 단위*로 작성되며, 팀은 의견과 감정과 두려움의 영향을 받는 사람들로 이루어져 있다.

```
[ 마찰 ] = [ 좌절감 ] + [ 시간 손실 ] + [ 오해 ] + [ 단순화 불가 ]
```

> **핵심 통찰**: 신뢰와 명확한 소통이 없으면 표현하기 쉬워야 할 것들이 훨씬 더 복잡해진다. 시스템이나 설계를 올바르게 이해했다고 확신할 수 없다면, 이를 단순화하기는 어렵다.

## 2. PRACTICE 16 — 의견 대립은 제로섬 게임이 아닙니다

### 변증법적 추론

> **아이디어 38**: 의견 대립을 제로섬 게임으로 생각하면 누구도 이길 수 없습니다.

대부분의 일은 흑백으로 나뉘지 않는다. *변증법적 추론(Dialectical Reasoning)* — 한쪽 주장에도, 다른 쪽 주장에도 진실의 요소가 있다고 보고, 그 둘을 종합해 더 큰 무언가를 만든다.

```
[ A의 주장 ]  +  [ B의 주장 ]  ─▶  [ 새로운 합의/통찰 ]
     ↑                               ↑
  진실 일부                      두 입장 모두를 넘어선
```

대립이 될 뻔한 상황을 *흥미롭고 생산적인 대화*로 바꾼다.

### 변증법적 사고 훈련

| 단계 | 행동 |
|------|------|
| 1 | 상대를 "바보"로 단정하지 않는다 |
| 2 | 상대가 틀렸음을 보이려 하지 말고, *내가 틀렸을지 모른다*고 가정 |
| 3 | "왜 그런 입장이 됐는가?"를 고민 |
| 4 | 각 입장의 *타당한 부분*을 찾는다 |
| 5 | 더 나은 새 해법으로 나아간다 |

실전 예문:

> "말씀 이해합니다. 다만 제 경험상으론 좀 다른 것 같아서 XYZ에 대해 더 자세히 알고 싶네요. 어떤 근거로 그렇게 생각하시게 되었는지 말씀해 주실 수 있습니까?"

> 반박하려 들지 마세요. 이기려는 게 아니니까요. 상대의 관점에서 바라보는 연습입니다. 덤으로 새로운 걸 배우게 되고요.

### 칼 놓고 논쟁하기 게임

```
점심 자리에서 주제가 갈렸을 때:

찬성: 칼끝이 테이블 중앙을 향하게 ↑
반대: 칼이 테이블 가장자리와 평행하게  ─

몇 분마다 두 명씩 칼 방향을 바꿈
이전에 주장하던 사람이 이제 반대 의견을 옹호
```

게임의 목표는 승자나 결론이 아니다. **자신이 굳게 믿어 온 생각을 다시 돌아보게** 만드는 것.

### 절대주의의 비효율

> **아이디어 39**: 절대적인 태도는 단순함의 적입니다.

100% 믿는 신념을 반대 증거 앞에서 유지하려면 얼마나 많은 노력이 드는가? 일부만 사실인 정보로 결정을 내리느라 얼마나 시간을 낭비하는가? **절대적 원칙으로 움직이는 팀은 효율적일 수 없다.**

## 3. PRACTICE 17 — 공감 능력 기르기

### 공감 = 사회적 윤활제

> 소프트스킬 중에서 하나만 개선할 수 있다면, 공감 능력에 투자하는 게 가장 좋습니다.

**공감(Empathy)**: 다른 사람이 느끼는 바를 이해하고 그 점을 고려해 상대를 대하는 능력. 동의가 아니라 *관점 존중*.

```
공감 있는 팀:
   - 비언어적 신호로 작은 결정 다수
   - 즉각적 피드백 / 미세한 수정
   - 정서적 안전망 → 자유롭게 실험

공감 없는 팀:
   - 자기 보호에 급급
   - 사일로화된 업무
   - 위계적 / 경직된 의사결정
   - 남 탓 → 학습 없음
```

### 사람들에게 시간을 내어주기

저자의 약점:

> 저는 일과 관련된 상호작용을 대개 *거래*처럼, 곧 어떤 정보를 주고받는 과정으로만 봅니다. 가능한 한 효율적으로 끝내고, 곧바로 다음 일로 넘어가려고 하죠. 이건 잘못된 태도입니다. 이럴 때마다 상대와의 관계는 조금씩 약해집니다. 그 사람을 하나의 *인적 자원*으로 대할 뿐, 한 명의 *인간*으로 대하지 않으니까요.

> **아이디어 40**: 스몰톡은 말하기가 아니라 경청이 중요합니다.

요령: 상대가 말을 풀어낼 수 있도록 *질문 몇 가지를 준비*. 결국 필요한 건 상대의 말을 기울여 들어 줄 시간을 내는 것뿐.

### 다른 사람에게 나는 어떻게 보일까

저자의 자조: "방탄 소재 상자에 마음을 숨기고 침대 아래 금고에 보관하는 영국인."

> **아이디어 41**: 사람들이 여러분의 감정을 안다고 가정하지 말고 직접 알려주세요.

```
나쁜 패턴: 짧고 퉁명스럽게 답변 → 상대 기분 상함 → 나도 실망

좋은 패턴: "죄송합니다. 그 표현이 다소 퉁명스럽게 들렸겠네요. 
          치통이 심해서 집중하기가 어렵네요."
```

### 상대의 기분 알아내는 법

기분이 나쁘지 않은 사람도 "기분 상했냐"는 질문을 반복하면 곧 나빠진다. 대신 자연스럽게 묻는 법:

```
"안녕하세요, Jan! 오늘은 좀 어때요?"
"리포트 포매터의 버그를 찾느라 오전이 다 갔네요."
"저런! 정말 답답했겠네요."
"사실 꽤 즐거웠어요."
"정말요? 왜 그렇죠?"
```

### 영향력 확인하기

개발자는 회사 내 누구보다 비즈니스를 잘 안다. 이 영향력을 휘둘러 *남 위에 군림*할 수 있다. 그러면 상대는 어깨를 으쓱하며 "프로그래머들이 다 그렇잖아"라고 넘긴다.

```
나쁜 게임: 신과 같은 지식을 뽐내 점수를 따기

좋은 게임: 상대방이 내 요점을 자기 말로 다시 설명할 때마다 점수
```

### 황금률 (Golden Rule)

존 스튜어트 밀: "자신에게 해 주길 바라는 대로 남에게 행동하라."

한 달 실험: *부탁받지 않은 친절한 행동* — 커피를 가져다주기, PR 리뷰가 잘 작성됐다고 알려 주기, 동료가 일찍 퇴근해야 할 때 테스트를 마쳐 주기.

### 공감은 미리 받는 피드백

> 저는 공감이 복잡한 것을 단순하게 만드는 데 핵심적인 역할을 한다고 믿습니다. 공감은 우리 자신에 대해 본질적인 피드백을 *수집하는 방법*입니다. 다른 사람의 관점에서 상황을 상상한 뒤, 그 관점에서 얻은 통찰을 바탕으로 자신의 행동을 조정합니다.

```
공감 없음:                       공감 있음:
무심코 행동 → 반응 처리 → 잘못 판단 → 다 되돌림    실시간 피드백 → 미세한 수정 반복
```

### 조하리의 창 (Johari Window)

| | 내가 아는 것 | 내가 모르는 것 |
|---|---|---|
| **남이 아는 것** | 열린 영역 (Open) | 보이지 않는 영역 (Blind) |
| **남이 모르는 것** | 내가 숨긴 영역 (Hidden) | 발견될 영역 (Unknown) |

자기 평가와 타인의 평가 사이의 차이를 드러내고, 신뢰를 쌓기 위한 도구. 균형을 위해 *참여하지 않는 사람*이 진행하는 것이 좋다.

## 4. PRACTICE 18 — 사물에도 공감하기

### 힘으로 밀어붙이는 친척

> 누구나 "힘으로 밀어붙이면 다 된다"는 식으로 물건을 다루는 친척이 한 명쯤 있습니다. 열쇠가 안 돌아가면 부러질 때까지 더 세게 돌리고, 자동차 시동이 안 걸리면 배터리를 방전시키고, 병뚜껑이 안 열리면 칼끝이 휘어질 때까지 비틀어 엽니다.

> **아이디어 42**: 환경과 싸우지 말고 이해하세요.

### 요트 스키퍼의 비유

> 친구와 요트를 공동소유했을 때 우리는 키를 *힘으로* 밀어 돌렸습니다. 하지만 숙련된 스키퍼는 손가락 하나로 키를 눌러 살짝 오므렸다 폈다 하는 정도로 부드럽게 배를 조종했습니다. 그는 배를 *굴복시키려* 들지 않았습니다. 대신 배의 움직임을 느끼고, 배와 *합의*를 이루려 했습니다.

> **핵심 통찰**: 그는 사물에 대한 공감을 통해 더 적은 힘으로 더 빠르고 더 부드럽게 항해했습니다. 이렇게 항해를 단순하게 만들었습니다.

### 코드에 귀 기울이기

```
함수를 작성할 때 불안함을 느낀다면 → 코드가 반발하고 있다는 신호
"이건 평소보다 더 어려운 것 아닌가?" → 코드가 단순화를 요구하고 있다
```

디버거의 비결: 버그를 *실수*로 보지 않고, 시스템이 *시키는 대로 움직인 결과*로 본다.

> 그들은 "어느 코드가 잘못됐지?"라고 찾지 않습니다. 대신 시스템 전체를 바라보며 "도대체 뭐가 이런 반응을 만들지?"를 찾습니다.

### 회피 신호 — 무의식의 경고

| 증상 | 의미 |
|------|------|
| 평소보다 자주 음료를 마시러 일어남 | 코드가 거부 반응 |
| 계속 이름을 바꾸고 있다 | 추상화가 안 맞는다 |
| 타이핑이 힘들게 느껴진다 | 잘못된 길 |
| 그만하고 다른 일을 하고 싶다 | 회피 행동 |

### 두려움의 선물

개빈 드 베커(*Gavin de Becker*)의 *『The Gift of Fear』*: 강도를 당하기 전 사람들 대부분이 *불안감*을 느꼈지만 무시했다.

> 불안한 감정은 잠재의식이 여러분이 인식하지 못하는 패턴을 인식할 때 나타납니다. 이런 반사 작용은 더 원시적인 시대에 우리의 생존을 도왔습니다. 지금의 반사 작용을 무시하면 *수천 년의 집단 경험*과 수십 년의 개인 경험을 포기하는 셈입니다.

### 잠재의식의 외침에 귀 기울이기

샤워나 산책 중 최고의 아이디어가 떠오르는 이유 — *의식적 뇌가 휴식 상태*일 때 잠재의식이 뚫고 나올 공간이 생긴다. 이상함을 느끼면 산책·낙서·정리를 시작하라.

## 5. PRACTICE 19 — 이야기 엮기

### 엘리베이터의 상사

```
상사: "왜 필요한 용량의 4배나 되는 저장 공간을 결제하는 거죠?"
나:   "아, 그건 인덱스에 필요한 여분의 공간입니다."
상사: "인덱스가 뭔데요?"

[ 1차 시도 — 면접식 답변 ]
"인덱스는 하나 이상의 열에 있는 값들과 그 값들을 포함하는 행 사이의 매핑 정보입니다. 
인덱스는 해시 방식이거나 정렬되어 데이터 접근을 빠르게 합니다..."

→ 상사는 얼굴을 찡그리며 내려 버린다
```

면접 답변은 사실에 근거하지만 *상대는 이해하지 못했다*. 이 챕터의 답은 **비유**다.

### 두 번째 시도 — 비유로

```
"현재 시스템은 수십억 개의 정보에서 특정한 정보를 신속하게 찾아내야 합니다. 
그래서 인덱스를 생성합니다. 책의 색인과 비슷한 건데요. 색인을 보고 원하는 페이지로
바로 넘기듯, 인덱스가 있으면 원하는 내용을 빠르게 찾을 수 있죠. 그런데 사람마다 
정보를 찾는 방식이 다양해서, 여러 종류의 인덱스를 사용합니다. 책에서 색인이 일정한 
페이지를 차지하듯이, 데이터베이스의 인덱스 역시 저장 공간을 차지하죠."

→ 상사는 미소를 짓는다. "당신의 상사도 그렇게 명확하게 설명해 줬으면 좋았을 텐데요."
```

### 비유 체크리스트

| 질문 | 적용 (인덱스 예시) |
|------|---------------------|
| 어떤 정보가 맥락에서 중요한가? | 인덱스가 저장 공간을 차지한다는 점 |
| 어떤 용어가 적절한가? | 책의 색인 (해시·B-tree 같은 기술 용어 회피) |
| 어떤 비유가 가능한가? | 책의 색인 |
| 비유가 누락하는 정보는? | 인덱스의 동적 재구성 등 — 본 대화에서 *중요하지 않음* |

> **아이디어 43**: 비유는 유용한 허구입니다.

### 비유의 본질

좋은 비유는 *구성요소들의 상호작용*까지 대응시킨다.

| 책의 색인 | 데이터베이스 인덱스 |
|-----------|---------------------|
| 특정 정보를 빠르게 찾기 | 데이터 위치를 빠르게 조회 |
| 책 내에서 공간을 차지 | 저장 공간을 차지 |
| 색인이 많을수록 조회 방법 다양 | 인덱스가 많을수록 조회 방법 다양 |
| 색인이 많을수록 공간 더 필요 | 인덱스가 많을수록 공간 더 필요 |

> **아이디어 44**: 스토리에서 등장하는 개념들은 실제 세계의 대응 개념처럼 상호작용해야 합니다.

### 두 번째 사례 — 데이터베이스 락 + 공항 비유

티켓 판매 앱에서 락 충돌. 관리자: "서버 12대 더 추가하면 안 되나요?"

```
공항 비유:
- 셀프 체크인 기계 = 들어오는 요청을 처리하는 서버
- 보안 검색대 = 데이터베이스 업데이트 (한 번에 하나만)

체크인 기계를 더 추가해도 → 보안 검색대에서 여전히 대기
∴ 데이터베이스 병목을 봐야 함
```

### 도메인 선택 — 피해야 할 것

| 피해야 할 도메인 | 이유 |
|------------------|------|
| 한 사람만의 전문 영역 | 야구를 모르면 인필드 플라이 비유는 무용 |
| 정치·종교·사회 이슈 | 당연 |
| 구축에 오래 걸리는 환경 | 15분 동안 비유 설명 들을 시간이 없다 |
| 최근 SNS 화제 | 모두가 같은 해시태그를 따르지 않는다 |

선호: 명작 영화, 실생활 경험, 사무실 네트워크 상황, 날씨 등.

### 비유 만들기 — 이름에서 시작

데이터베이스 *인덱스(Index)* → 책의 *색인(Index)*. 단어에서 시작해 자연스럽게 이어진다.

### 즉흥 연기 수업

저자가 컨퍼런스에서 즉흥 연기 코미디언과 보낸 오후. 참가자들이 점차 자신감을 얻고, 잠시 생각할 시간을 버는 *대화 기법(예: "예, 그리고…")*을 배웠다.

## 6. 소프트스킬 단순화 체크리스트

| 항목 | 자기 점검 |
|------|------------|
| 의견 대립 | 마지막에 "내가 틀렸을 수도"라고 가정한 적 있는가? |
| 절대주의 | 100% 믿는 신념을 다시 검토한 적 있는가? |
| 스몰톡 | 동료의 일상 한 가지를 기억하고 있는가? |
| 감정 표현 | 퉁명스러웠을 때 *이유*를 알려 주었는가? |
| 코드 듣기 | 회피 행동을 알아챈 적 있는가? |
| 비유 | 마지막 기술 설명을 비유로 옮겨 본 적 있는가? |

## 요약

- 마찰은 손실. 공감과 명확한 소통이 *단순함의 기반*이다.
- 의견 대립은 *제로섬*이 아니다. **변증법적 추론**으로 더 큰 답을 만들 수 있다.
- 절대주의는 단순함의 적. 일부만 사실인 정보로 결정을 내리느라 시간을 잃는다.
- 공감은 *사회적 윤활제*이자 *미리 받는 피드백*. 거래가 아닌 *인간*으로 사람을 본다.
- 사람뿐 아니라 *사물에도 공감*하라. 코드의 회피 신호, 직감의 경고는 무시하지 말 것.
- 좋은 설명은 *비유*다. 비유는 유용한 허구이며, 구성요소의 상호작용까지 대응시켜야 한다.
- 도메인 선택이 핵심 — 양쪽이 공유하는 배경에서 골라라.

## 다른 챕터와의 관계

이 챕터는 *Part 3(상호작용의 단순화)*의 핵심이다. 3장(프로젝트 최적화)이 *프로세스* 차원의 마찰을 줄였다면, 이 챕터는 *대인관계*의 마찰을 줄인다.

3장의 *DWP*가 효과적이려면 본 챕터의 공감과 변증법적 추론이 필수적이다. 6장 없는 3장은 절차만 있는 빈 껍데기다.

이어지는 *Part 4(코드의 단순화)*는 다시 코드로 돌아간다. 그러나 7장(데이터 주도 개발)의 핵심 명제 *"데이터 > 코드"* 는 이 챕터의 비유 사용 — *문제를 다른 도메인으로 옮겨 사고하기* — 의 코드 버전이다.

5장의 *야누스의 두 얼굴*과도 짝을 이룬다. 5장이 *시간*에 대한 공감(과거의 검증된 도구로 일하고, 미래에선 놀이)이었다면, 6장은 *공간*에 대한 공감 — 사람과 사물의 관점을 받아들이는 일이다.

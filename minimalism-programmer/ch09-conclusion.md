# Chapter 9: Conclusion (마치며)

## 핵심 질문

저자가 마지막까지 미뤄 둔 가장 어려운 질문에 답할 시간이다. **단순함이란 정확히 무엇인가?** 어떤 방식이 다른 방식보다 *더 단순*하다는 것을 어떻게 판단할 수 있는가?

> 드디어 이 책의 마지막 장이군요. 저는 가장 어려운 질문을 조심스럽게 피해 왔습니다. 단순함이란 뭘까요? 즉, 어떤 방식이 다른 방식보다 더 단순하다는 걸 어떻게 판단할 수 있을까요?

## 1. 단순함의 평가 기준

저자의 답은 *상황에 따라 다르다*이다. 그러나 평가의 *일반적 기준*은 제시할 수 있다.

> **핵심 통찰**: 다음 조건이 모두 충족된다면 *단순함(S)*은 *복합적복잡성(C)*보다 더 단순하다고 볼 수 있다.

```
┌──────────────────────────────────────────────────────────────┐
│ 1. S는 C보다 이해하기 더 쉬운가?                                  │
├──────────────────────────────────────────────────────────────┤
│ 2. S가 어떤 일을 어떻게 수행하는지 C보다 더 빠르게 파악할 수 있는가?     │
├──────────────────────────────────────────────────────────────┤
│ 3. S가 수행하는 작업을 C보다 더 간단한 문장으로 다른 사람에게         │
│    설명할 수 있는가?                                              │
├──────────────────────────────────────────────────────────────┤
│ 4. 복잡한 코드를 분석하거나 잘 알려지지 않은 라이브러리를 사용해야      │
│    할 때, S를 사용할 때가 C를 사용할 때보다 멈추고 고민하는 시간이      │
│    더 적은가?                                                     │
├──────────────────────────────────────────────────────────────┤
│ 5. 나는 S를 변경할 수 있다는 자신감이 C를 변경할 수 있다는 자신감보다  │
│    더 높다고 느끼는가?                                              │
└──────────────────────────────────────────────────────────────┘
```

### 평가 기준의 본질

| 차원 | 핵심 |
|------|------|
| **이해** | 1~3번 — 코드가 *읽는 사람의 머릿속*에서 어떤 형태로 자리 잡는가 |
| **속도** | 4번 — *멈춤 없이* 진행할 수 있는가 |
| **자신감** | 5번 — 변경했을 때 *깨지지 않을 것*이라는 신뢰 |

이 다섯 가지 모두 *주관적*이다. 절대적 메트릭이 아니라, 코드를 보는 사람의 *경험*을 기준으로 한다. 그래서 같은 코드도 사람·맥락·시점에 따라 다르게 평가된다.

> **핵심 통찰**: 단순함은 일을 처리하는 *방법*이 아니라 일을 대하는 *마음가짐*이다 (1장의 메시지가 9장에서 다시 닫힌다).

## 3. 단순함이란 무엇이 *아닌가*

| 오해 | 진실 |
|------|------|
| 코드를 짧게 줄이는 기교 | 프로젝트의 군더더기를 걷어내고 핵심 가치에 집중하는 *태도* |
| 본질을 희석하는 것 | 변경 가능성과 이해 가능성을 *높이는 것* |
| 정해진 규칙의 적용 | 상황과 맥락에 따른 *판단* |
| 만능 해결책 | 저자의 한 여정의 *기록* |

> **핵심 통찰**: 미니멀리즘 프로그래머에게 단순함이란 코드를 짧게 줄이는 기교가 아니다. 프로젝트의 군더더기를 걷어내고 *핵심 가치에 집중하는 태도*이자, *변화를 받아들이는 용기*다.

## 5. 잃어버린 즐거움의 회복

> 복잡한 코드의 홍수 속에서 잃어버린 *프로그래밍의 즐거움*을 되찾는 방법을 소개합니다.

이 책의 가장 깊은 약속은 *효율성*도 *생산성*도 아니다 — **즐거움**이다. 단순함은 코드를 더 빠르게 짜기 위함도, 더 적게 일하기 위함도 아니다. 우리가 처음 프로그래밍을 시작했을 때 느꼈던 *마법사 같은 기쁨*을 되찾기 위해서다.

> 우리는 개발자이고, 마치 마법사와 같습니다. 상상에서 무언가를 불러내어 이전에는 불가능했던 것들을 현실로 만듭니다. 그리고, 아시잖아요? 재미있습니다! (Ch 5 인용)

## 요약

- *단순함*의 평가는 5가지 주관적 기준으로 가능하다 — *이해, 설명, 파악 속도, 진행 속도, 변경 자신감*.
- 단순함은 *기교*가 아니라 *태도*다. 군더더기를 걷어내고 핵심 가치에 집중하는 마음가짐이다.
- 단순함은 *방법*이 아니라 *마음가짐*이며 *나침반*이다.
- 4부의 핵심: **비워라 / 단순한 무기로 / 군더더기 없는 소통 / 코드를 넘어 데이터로**.
- 책은 1장의 *현황파악-실행-학습* 사이클로 시작해 9장의 *평가 기준*으로 닫힌다 — 사이클의 *학습* 단계가 평가다.
- 가장 깊은 약속: 복잡함의 홍수 속에서 *프로그래밍의 즐거움*을 되찾는 것.

## 다른 챕터와의 관계

이 챕터는 책 전체의 *닫힘*이다. 1장에서 시작한 모든 질문이 여기에서 다시 만난다.

- 1장의 *복합적*과 *복잡함*의 구분 → 9장의 *S vs C* 평가
- 1장의 *현황파악-실행-학습* 사이클 → 9장의 5가지 학습 질문
- 1장의 *이름 없는 품질* → 9장의 *나침반*

각 부의 챕터들이 9장에서 통합된다.

- *Part 1 (Ch 2~3)*: 채우지 말고 비워라
- *Part 2 (Ch 4~5)*: 복잡한 도구를 단순한 무기로
- *Part 3 (Ch 6)*: 군더더기 없는 소통
- *Part 4 (Ch 7~8)*: 코드를 넘어 데이터로

책 전체가 한 권의 *원형 구조*다. 1장에서 던진 "이게 정말 맞나?"라는 질문이 9장에서 독자의 손에 다시 쥐어진다. 답은 책에 없다 — 독자의 *다음 한 줄의 코드*에 있다.

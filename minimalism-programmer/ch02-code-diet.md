# Chapter 2: Code Diet (코드 다이어트)

## 핵심 질문

비대해진 의존성과 프레임워크, 그리고 "있으면 좋은" 기능들 속에서 코드를 어떻게 가볍게 유지할 것인가? 의존성을 추가할 때 우리는 무엇을 잃고 있는가? 더 적은 기능을 더 빨리 전달하는 방법은 무엇인가?

> 거의 100년 전의 연구에 따르면, 도로망을 x% 확장할 경우 교통량도 적어도 x%만큼 따라서 증가한다고 합니다. 경제학자들은 이를 *유발수요(Induced Demand - 자원이나 시설이 더 많아지면 사람들이 그것을 더 많이 사용한다는 개념)*라고 부릅니다.

저자는 이 챕터에서 무어의 법칙이 가져온 *유발수요*가 소프트웨어를 비대하게 만들었다고 진단한다. 1969년에 4KB RAM 컴퓨터로 사람을 달에 보냈는데, 오늘날엔 비어 있는 React 프로젝트조차 350MB 코드를 다운로드한다.

## 1. 비대해진 코드의 현재

빈 프로젝트가 얼마나 비대한지 보자.

| 명령어 | 설치되는 의존성 | 용량 |
|--------|-----------------|------|
| `mix phx.new my_project` | 43개 | 15 MB |
| `rails new my_app` | 83개 (각각 수백 개 더) | — |
| `npx create-react-app my-app` | 874개 모듈 | 약 350 MB |

> **핵심 통찰**: 위 용량은 "빈" 프로젝트의 용량이다. 여기에 코드를 작성하기 시작하면 의존성을 추가하고, 그 의존성도 또 다른 의존성을 가지며 끝없이 이어진다. 모두가 짐을 한가득 짊어지고 있다.

## 2. PRACTICE 01 — 건강하지 않은 의존성 줄이기

### 의존성이 위험한 이유

> **아이디어 1**: 의존성을 추가하면 애플리케이션의 통제권 일부를 포기한다는 뜻입니다.

리액트의 기본 874개 모듈에는 수만 명에 달하는 사람이 기여했을 것이다. 이 중 한 명만이라도 나쁜 마음을 먹으면, 또는 *실수로* 보안 취약점을 끼워 넣기만 해도, 우리 앱은 망가질 수 있다.

#### 사례 1: left-pad 사건 (2016)

한 자바스크립트 라이브러리 개발자가 NPM 저장소에서 자신의 모든 라이브러리를 갑자기 삭제했다. 삭제된 모듈 중 하나인 `left-pad`는 수천 개 라이브러리에서 사용 중이었고, 그 라이브러리들은 다시 Babel과 React 같은 주요 도구에 포함되어 있었다. **개발자 한 명의 결정이 자바스크립트 애플리케이션의 상당 부분을 빌드 불가능하게 만들었다.**

#### 사례 2: XZ 압축 라이브러리 백도어 (2024)

XZ 압축 라이브러리에 악성 코드가 심어진 사실이 발견되어 *CVE-2024-3094* 경고가 발령되었다. OpenSSH 데몬에서 사용하던 라이브러리였기 때문에, 발견되지 않았다면 안전하다고 믿어 온 SSH 통신 경로가 무단 접속·데이터 탈취 통로로 악용될 뻔했다.

### 11줄의 left-pad

문제의 left-pad는 단 11줄이었다.

```javascript
module.exports = leftpad;
function leftpad(str, len, ch) {
  str = String(str);
  var i = -1;
  if (!ch && ch !== 0) ch = " ";
  len = len - str.length;
  while (++i < len) {
    str = ch + str;
  }
  return str;
}
```

오늘날에는 자바스크립트가 `padStart()`를 내장하고 있고, 직접 작성해도 3줄이면 충분하다.

```javascript
function leftpad(str, len, ch = " ") {
  str = String(str);
  len = len - str.length;
  return (len > 0) ? ch.repeat(len) + str : str;
}
```

> **아이디어 2**: 11줄짜리 코드가 의존성이 되어선 안 됩니다.

> 다른 사람의 문제라는 함정에 빠졌을 수도 있습니다. 코드의 일부를 라이브러리 작성자에게 위임하면서, 그로 인해 발생하는 문제에 대한 책임도 넘겼다고 착각하는 것입니다.

### 의존성 결정 체크리스트

저자가 의존성을 추가하기 전 자문하는 항목들이다.

| 질문 | 핵심 |
|------|------|
| 이 기능이 꼭 필요한가? | "재미있어 보여서"는 이유가 못 된다. 적을수록 더 낫다 |
| 직접 코딩하는 편이 더 나은가? | 함수 하나라면 직접 구현. 어렵다면 오픈소스 출처 명시 후 일부만 발췌 |
| 정글까지 얻는 건 아닐까? | 바나나 하나 필요한데 고릴라와 정글 전체가 딸려오는지 |
| 프로젝트에 필수적인가? | 그렇다면 로컬 사본을 확보 — 404를 받았을 때의 충격을 피한다 |
| 특정 버전에 고정되어 있는가? | 의존성 관리자(`asdf` 등)로 잠금. 단, 한 의존성을 업데이트하면 다른 5개가 깨지는 도미노가 시작된다 |
| 안전한가? | `npm audit` 결과를 무시하지 말 것. GitHub의 보안 경보를 활용 |

> *Joe Armstrong*: "바나나 하나만 필요했을 뿐인데, 바나나를 든 고릴라와 정글 전체가 딸려옵니다."

> **아이디어 3**: 각 의존성은 심사숙고해서 선택합시다.

### 의존성 격리 — 래퍼 함수 패턴

API 변경에 대비하기 위한 가장 효과적인 패턴이다.

```typescript
// ❌ 의존성을 앱 곳곳에서 직접 사용
import { fetchUser } from "third-party-lib";
function showProfile(id: string) {
  return fetchUser(id);
}

// ✅ 래퍼 함수로 격리
import { fetchUser as _fetchUser } from "third-party-lib";

function getUser(id: string) {
  return _fetchUser(id);
}

function showProfile(id: string) {
  return getUser(id);
}
```

API가 변경돼도 래퍼 한 곳만 수정하면 된다.

### 저자의 의존성 지옥 경험담

> 저는 2023년에 7년의 공백 끝에 *Pragmatic Bookshelf*로 복귀했습니다. 사업의 중추인 소프트웨어가 사실상 유지보수되지 않고 있었고, 여전히 Rails 3을 쓰고 있었습니다. 별다른 검토 없이 업데이트를 시도했더니, Rails는 릴리스 사이에 호환성을 깨는 수정이 너무 많아 애플리케이션을 시작조차 할 수 없었습니다. 결국 8년 된 리눅스 버전을 실행하는 도커 컨테이너를 만들어야 했고, 제 평생 가장 복잡하면서도 취약한 개발 환경을 갖추게 됐습니다.

## 3. PRACTICE 02 — 프레임워크: 성분표 꼼꼼히 확인하기

### 한 번 들어오면 빠져나가기 어려운 덫

프레임워크는 애플리케이션 코드를 바깥에서 감싸고 전체를 주도한다. 일단 어떤 프레임워크를 쓰기로 마음먹으면 코드를 떼어 내기가 매우 어렵다. **프레임워크는 API를 쥐고 있을 뿐 아니라 애플리케이션 아키텍처까지 규정해 버리기 때문이다.**

### 인플레이션의 소용돌이

프레임워크는 끊임없이 진화한다. 경쟁자가 새 기능을 내놓으면 따라붙고, 자기 것도 더 좋다고 주장한다. 사용하지 않는 옛 기능들은 버려진다.

> **아이디어 4**: 프레임워크에는 계속 새로운 기능이 붙습니다.

### 그린필드 프로젝트의 함정

새 풀스택 자바스크립트 프로젝트를 시작하면 무수한 옵션 미로에 빠진다. 동기/비동기, 콜백/이벤트/상태머신, 메시징 인프라 4종, SSL 인증서, 루트 인증서… *Hello, World*에 도달하기까지 산을 넘어야 한다.

> **아이디어 5**: 프레임워크가 여러 옵션을 제공한다면, 사용하지 않을 요소가 있다는 의미입니다.

이 옵션들에는 대가가 따른다 — 복합적복잡성, 의존성, 불필요한 용량. 그리고 사용하지 않고 방치한 기능은 **내일의 보안 취약점**이다.

### 브라운필드 프로젝트의 함정

V0.8 시절부터 써 온 익숙한 프레임워크라도, 빨라지는 변경 속도와 잦은 기능 지원 중단이 발목을 잡는다. 보안 업데이트를 적용하려면 새 릴리스를 따라가야 하는데 호환성이 깨진다.

> **아이디어 6**: 코드보다 프레임워크 유지보수에 더 많은 시간을 쓴다면, 그 프레임워크는 별 도움이 안 되는 걸지도 모릅니다.

### 가벼운 프레임워크의 추천

| 언어 | 풀스택 (무거움) | 미니멀 (저자 추천) |
|------|-------------------|---------------------|
| Ruby | Rails | Sinatra |
| Node.js | (Express + 풀스택 부가) | Express, Nest, Adonis |
| Python | Django | Flask, FastAPI |

> **핵심 통찰**: 환경을 잘 이해할수록 다루기 쉬워진다. 환경이 작을수록 이해와 유지관리도 쉽고, 새로 합류한 개발자가 적응하기도 수월하다.

### 프레임워크 팽창 지수

자신의 프레임워크에서 기능 목록을 찾아보고, 그중 몇 퍼센트나 실제 쓰는지 세어 보자. 저자가 여러 팀에 물었을 때 가장 많이 나온 답은 **10~20%**였다. 빙산의 물 위에 드러난 부분만 쓰는 셈이다.

## 4. PRACTICE 03 — 기능은 적을수록 좋다

### 기능은 미래의 부채

> **아이디어 7**: '기능'은 '미래의 부채'를 그럴싸하게 포장한 마케팅 용어입니다.

소프트웨어에 추가하는 모든 기능은 미래의 누군가가 이해하고, 지원하고, 유지보수하고, 확장해야 한다.

### 필요주도개발 (Necessity-Driven Development)

소프트웨어의 단순함을 유지하는 기본 규칙은 **누군가 필요로 하지 않는 한 코드를 쓰지 않는 것**이다. "이런 기능이 있으면 멋지지 않을까", "추가하기 정말 쉬울 텐데"는 *필요*가 아니다.

> **아이디어 8**: 아무도 요청하지 않은 코드는 작성하지 마세요.

### 가치 중심으로 질문 바꾸기

저자는 접근 방식을 뒤집었다.

| 전통적 질문 | 저자의 질문 |
|-------------|-------------|
| 어떤 기능을 만들어 줄까요? | 이 프로젝트에서 어떤 *가치*를 기대하세요? 그 가치들의 우선순위는? |

질문을 바꾼 뒤, 팀은 각 가치의 노력을 *쉬움 / 보통 / 어려움 / 알 수 없음* 으로 거칠게 분류한다. 그리고 고객에게 *전통적 한 번 배포* vs *점진적 부분 배포* 두 옵션을 제시한다.

> **아이디어 9**: 기능이 아니라 점진적인 가치를 전달하세요.

### 점진적 전달의 효과

저자가 직접 경험한 사례:

> 한 고객이 다섯 가지 요청 기능 가운데 처음 두 기능만 완성된 버전을 받고는, 그 시스템이 자신에게 필요한 일을 다 해 준다고 말했습니다. 나머지 세 기능은 보험 삼아 넣어 둔 거라 필수적이지 않았던 거죠.

| 측면 | 점진적 전달의 이점 |
|------|---------------------|
| 고객 측 | 가치를 더 빨리 회수, 직접 써 보고 새 통찰을 얻음 |
| 개발자 측 | 빠른 피드백, 신뢰 형성, 잠재 문제 조기 발견 |
| 시스템 측 | 만들지 않아도 되는 기능 발견 |

> **아이디어 10**: 복합적복잡성은 기능 수보다 더 빠르게 증가합니다.

## 5. 단순함 점검 — Code Diet 체크리스트

| 항목 | 행동 |
|------|------|
| 직접 의존성 | 각각이 무슨 일을 하는지 알고 있는가? |
| 전이 의존성 | `.lock` 파일을 열어 전체 그래프를 본 적 있는가? |
| 프레임워크 사용률 | 공식 문서의 기능 중 실제로 몇 %를 쓰는가? |
| 한 번도 사용되지 않는 코드 | 코드베이스에서 데드 코드의 비율은? |
| 기능 요청 | "왜 필요한가"를 5번 물었는가, 아니면 "어떻게 만들지"부터 시작했는가? |

## 요약

- 무어의 법칙이 만든 *유발수요*는 소프트웨어를 비대하게 만든다. "기가는 항상 더 많아진다"는 말은 함정이다.
- **의존성을 추가하는 것은 통제권을 양보하는 것**이다. left-pad와 XZ 사건이 증명한다.
- 11줄짜리 코드는 의존성이 되어선 안 된다. 함수 하나 때문에 라이브러리를 추가하지 마라.
- 의존성을 광범위하게 쓴다면 *래퍼 함수*로 격리해라 — API 변경 시 한 곳만 수정하면 된다.
- 프레임워크는 한 번 들이면 빠져나오기 어렵다. 가장 화려한 것보다 안정성이 입증된 가벼운 것을 골라라.
- 대부분의 팀은 프레임워크 기능의 **10~20%**만 쓴다. 사용하지 않는 기능은 내일의 보안 취약점이다.
- 기능은 미래의 부채다. *필요주도개발*: 아무도 요청하지 않은 코드는 작성하지 마라.
- "어떤 기능을 만들지" 대신 "어떤 *가치*를 원하는지" 물어라. 다섯 기능 중 두 개만으로 충분한 경우가 흔하다.

## 다른 챕터와의 관계

이 챕터(Part 1의 시작)는 **무엇을 만드는지**의 단순화다. 다음 챕터(Ch 3)는 **어떻게 만드는지**, 즉 팀과 프로세스의 단순화로 이어진다. 더 적은 의존성과 더 적은 기능이 출발점이라면, 더 적은 회의와 더 낮은 결합도가 다음 단계다.

7장(데이터 주도 개발)에서 등장하는 *데이터 > 코드*의 원칙은 결국 본 챕터의 "코드는 적을수록 좋다"는 정신의 코드 레벨 적용이다.

# Chapter 4: Task Automation (업무 자동화)

## 핵심 질문

기억해야 할 것을 어떻게 줄일 수 있는가? 데스크톱·터미널·에디터·배포 같은 일상 환경을 어떻게 자동화해 *인지 부담*을 덜 것인가? 컴퓨터가 망가져도 5분 안에 동일한 환경을 복구할 수 있는가?

> 제 뇌에는 거미막 낭종이 있습니다. 왼쪽 측두엽에 6cm 크기의 빈 공간이 있어, 청각 변별력과 일부 기억을 담당하는 부위가 비어 있습니다. 군중 속에서 사람들의 말을 잘 알아듣지 못하고, 기억하고 싶은 것들을 기억하지 못합니다.

저자가 자신의 인지적 한계를 자동화로 보완해 온 경험에서 출발한다. **자동화를 더 많이 할수록 기억해야 할 것이 줄어든다.** 버튼 하나로 배포하면 체크리스트를 찾아다닐 필요가 없다. 인지 부담이 줄면 스트레스도 줄고, 더 행복하고 더 나은 소프트웨어가 만들어진다.

## 1. PRACTICE 09 — 데스크톱 환경 최적화하기

### 작업공간으로 컨텍스트 분리

산만함의 가장 큰 적은 *겹친 168개의 창*이다. 저자의 해결책은 **작업공간(Workspaces)** — 각 프로젝트마다 고유한 창과 앱을 갖춘 가상 데스크톱.

```
[작업공간 1: 이메일/브라우저]   [작업공간 2: 책 집필]   [작업공간 3: 코딩 프로젝트 A]
       ↑               ↑                ↑
       └───── 키보드 단축키로 즉시 전환 ─────┘
```

> **아이디어 25**: 데스크톱을 시간 관리하듯 정리하세요.

| 도구 | 환경 |
|------|------|
| Spaces | macOS 기본 (저자: 키보드 제어 부족 → 비추천) |
| 타일형 윈도우 매니저 | 저자의 선택. 창이 겹치지 않음, 전부 키보드 |

타일형 윈도우 매니저는 수평/수직 스택으로 창을 배치한다. 처음에는 답답하지만, 마우스 없이 키보드만으로 창을 이동·배치할 수 있어 산만함이 줄어든다.

> **핵심 통찰**: 자리를 비웠다 돌아와도 모든 게 그대로다. 168개 창이 겹쳐 있는 화면을 다시 만나지 않는다.

### 시도 가이드

| 단계 | 행동 |
|------|------|
| 관찰 | `Cmd+Tab`을 얼마나 자주 누르는가? 화면 정리에 시간을 쓰는가? |
| 일주일 실험 | 윈도우 매니저나 작업공간을 일주일 써 본다 |
| 습관화 | 작업을 작업공간 단위로 분할하는 습관 |

## 2. PRACTICE 10 — 터미널 활용 극대화하기

### 왜 터미널인가

> **아이디어 26**: 효과적인 개발자는 터미널을 능숙하게 다룹니다.

GUI는 *"보이는 것이 전부다(WYSIAYG, What You See Is All You Get)"* 와 같다. 앱에 내장된 기능에 갇힌다. 셸은 운영체제 전체에 접근하게 해 주며, 명령을 조합·파이프할 수 있다.

| 측면 | GUI | 터미널 |
|------|-----|--------|
| 기능 한계 | 앱이 제공하는 것까지 | 운영체제 전체 |
| 조합 | 어렵다 | 파이프 / 리디렉션 / 인자 전달 |
| 자동화 | 매크로·스크립트 한정 | 셸 스크립트로 자유 |

### 일상 명령어 예시

```bash
# src/의 모든 .ts 파일을 .bak으로 변경
$ mv src/*.{ts,bak}

# 원고의 단어 수 계산
$ wc -w **/*.md

# 파일이 변경될 때마다 책을 빌드
$ ls **/*.md | entr ./rake clean screen

# 'colour' 영국식 철자(대소문자 무시) 검색
$ rg -i colour

# 포켓몬 뮤츠가 배울 수 있는 기술 출력
$ curl -s 'https://pokeapi.co/api/v2/pokemon/mewtwo' | jq -r '.moves[] | .move.name'
```

### 터미널 평가 기준

| 항목 | 기준 |
|------|------|
| **속도** | GPU 렌더링. 16,000줄을 45ms 안에 스크롤 |
| **스크롤백** | 최소 2,000줄 이상 |
| **테마** | 두통이 안 오는 색 |
| **키 매핑** | 에디터와 같은 키로 창 전환 |
| **다중 패널** | 필수 — 로그를 옆 패널에 띄워 두기 |

> **아이디어 27**: 터미널을 정보를 표시하는 대시보드로 사용하세요.

### 다중 패널 — tmux보다 단순하게

저자는 *tmux*의 화려한 세션 관리를 쓰지 않는다. 이유: **익혀야 할 키 시퀀스가 하나 더 늘어나기 때문**. 대신 키보드만으로 분할 영역을 만드는 터미널을 쓰고, 같은 키로 에디터 버퍼와 터미널 창 사이를 이동하도록 매핑한다.

### 셸의 자동완성

```bash
# 명령 자동완성
$ ed-<Tab>     # ed로 시작하는 명령 제안
$ intro-<Tab>  # intro로 시작하는 파일 제안
```

문맥 기반 자동완성은 *작업 디렉터리를 기억*해, 작업공간이 바뀌면 그 디렉터리 명령을 우선 제안한다. 서로 다른 작업공간이 분리된 느낌을 준다.

> **아이디어 28**: 셸은 강력한 제어 능력을 제공합니다.

### 저자가 매일 쓰는 도구

| 환경 | 도구 |
|------|------|
| macOS 터미널 | Kitty |
| Linux 셸 | Fish |

## 3. PRACTICE 11 — 그 외 모든 것 자동화하기

### 자동화는 곧 신뢰성

> **아이디어 29**: 자동화는 곧 신뢰성입니다.

옛날에는 컴파일러를 실행하고 오브젝트를 링크해 이미지를 만들고 *수동으로* 다른 컴퓨터에 복사했다. 결과는 늘 두려움. 오늘날은 빌드·배포·업데이트가 모두 도구로 자동화된다.

### 자동화 먼저, 코드는 나중에

저자는 새 프로젝트를 시작할 때 첫 한 시간을 환경 설정에 쓴다. **애플리케이션 코드 한 줄 작성하기 전에** 이 5단계를 마쳐야 한다.

```
1. 로컬 복사본 삭제 → 버전 관리에서 다시 복원
2. 모든 의존성을 단일 명령어로 설치
3. 단일 명령어로 빌드 (처음엔 "Hello, World")
4. 테스트를 빌드 일부로 자동 실행
5. 프로젝트를 타겟 환경에 배포
```

> 코드가 전혀 없는 상태에서 왜 먼저 이 작업을 해야 할까요? 이런 관리 업무는 안 하면 계속 미루게 된다는 걸 (쓴 경험으로) 배웠기 때문입니다.

### 0일차에 배포하기

0일차에 배포하면 단순함을 유지할 수 있다 — 설정할 환경도, 초기화할 DB도, 메일러도, 메시지 큐도 없다. 단지 다른 기기에서 *Hello, World*를 실행시킬 뿐. 그 후 기능을 추가하면서 *계속* 배포한다.

```
[ 0일차 배포 ]                       [ 마지막 배포 ]
- Hello, World만 동작                - 모든 구성요소 완벽 동작 필요
- 단순함이 유지됨                       - DB? 환경변수? 메시지 큐 URL? 무엇이 문제인지 모름
- 배포 가능 자체를 검증                  - 한꺼번에 모든 게 깨질 수 있음
```

### 단일 명령 배포의 황금률

> **아이디어 30**: 베어 머신에서 단일 명령으로 애플리케이션을 실행할 수 있어야 합니다.

전원이 켜지고 SSH 키만 프로비저닝된 새 시스템에서 시작해, 단 하나의 배포 명령으로 OS 구성·의존성 설치·서비스 가동·앱 배포·DB 마이그레이션까지 모두 가능해야 한다. 저자는 *Ansible*을 사용한다.

저자의 실험 사이클:

| 빈도 | 행동 |
|------|------|
| 매일 여러 번 | 클라우드에 저렴한 서버 띄우고 배포 |
| 주 1회 | 원격 머신을 OS+SSH 키만 남긴 기본 상태로 초기화 후 배포 명령 검증 |

> 새벽 세 시 사이렌이 울릴 때, 미리 그렇게 해 둔 자신에게 감사하게 됩니다.

## 4. PRACTICE 12 — 에디터 완전히 장악하기

### 에디터는 가장 짧은 지름길

> **아이디어 31**: 에디터는 머리에 있는 정보를 코드로 옮기는 가장 짧은 지름길입니다.

저자의 에디터 여정:

```
20년 Emacs (수천 줄 Lisp 커스텀)
    ↓ "설정 만지작거리는 시간이 일하는 시간보다 많아짐"
TextMate / VS Code (간소화)
    ↓
Neovim (현재) — 기본 상태로 시작 → "할 수 없는 일" 목록을 만들고 70%가 이미 내장돼 있었음을 발견
```

### 미니멀 Neovim 확장 목록

저자가 약 3년간 사용 중인 확장:

| 확장 | 역할 |
|------|------|
| 패키지 관리자 | 확장 설치 자동화 |
| 다크 테마 2개 | 시간대별 전환 |
| LSP + Tree-sitter | 언어 인식 편집 |

| 상태 표시줄 (LSP/Git) | 컨텍스트 가시화 |
| 다중 라인 정렬 | 8장의 정렬 프랙티스에 활용 |
| 다중 커서 | 동시 편집 |
| 블록 중첩 표시 | 들여쓰기 가시화 |
| Tree-sitter 구문 탐색 | 의미 단위 탐색 |
| 마크다운 미리보기 | 글쓰기 |

### 마우스 사용 줄이기

```
편집 시 마우스를 쓰던 3가지 → 키보드로 대체

1. 버퍼 안 빠른 이동      → 키 명령 학습
2. 텍스트 선택           → 빔의 텍스트 오브젝트 (vif=, vapgq)
3. 긴 목록에서 항목 선택  → 퍼지 파인더 ("A/Ed" → "Automate/Editor.md")
```

`vif=` = visually select inside the current function + reformat
`vapgq` = visually around paragraph + reformat lines

> 재미있는 건 하루가 끝날 무렵 오른쪽 손목이 더 이상 아프지 않다는 겁니다.

### 도전 과제

현재 에디터에서 다음을 마우스 없이 해 보자:

- 문장의 본문, 현재 함수, 전체 파일을 재정렬
- 줄바꿈 열을 75로 설정 후 주석 다시 줄바꿈
- 커서를 현재 함수의 시작/끝으로 이동
- 닫는 중괄호에서 대응하는 여는 중괄호로 이동
- 문자열의 홑따옴표를 겹따옴표로 바꾸기
- 마크다운 문서의 단어 수 알아내기

## 5. PRACTICE 13 — 개발 장비 세팅 자동화하기

### 사건과 복구

> 지난주에 천둥번개가 치면서 번개 몇 줄기가 집 마당에 떨어졌습니다. UPS로 보호되고 있었지만, 차폐가 안 된 이더넷 케이블이 유도 전압을 받아 컴퓨터와 스위치가 손상됐습니다.

복구는 두 부분이다 — *작업 파일* + *도구·구성*.

| 부분 | 시간 |
|------|------|
| 노트북에 동기화 (이미 자동화됨) | 5분 |
| 수리된 컴퓨터 OS 설정 | 40분 |
| 기본 소프트웨어 설치 | 20분 |
| 설정/개발 파일 복구 | 1~2분 |

> **아이디어 32**: 개발용 머신도 프로덕션 서버처럼 프로비저닝하세요.

### 저자의 도구 스택

| 분류 | 도구 | 역할 |
|------|------|------|
| **언어/컴파일러 버전** | asdf | 프로젝트별 다중 버전 |
| **일반 패키지** | Homebrew | macOS 패키지 |
| **구성 파일 관리** | yadm | dotfile 버전 관리 |

### 도구 추적 — Brewfile과 .tool-versions

> **아이디어 33**: 설치하는 동안 사용하는 도구를 기록하세요.

```bash
# Homebrew의 현재 설정을 Brewfile로 덤프
$ brew bundle dump

# 새 머신에서 한 줄로 복원
$ brew bundle

# asdf의 글로벌/로컬 도구는 .tool-versions 파일에 자동 저장
$ asdf install   # 글로벌 + 로컬 모두 설치
```

저자의 자동화: `brew bundle dump`와 yadm 업데이트는 **하루 두 번 실행되는 cron job**으로 관리된다.

### yadm 사용 흐름

```
[ Mac mini (망가진 컴퓨터) ]
        │ yadm push
        ▼
[ GitLab/GitHub (dotfile repo) ]
        │ yadm clone
        ▼
[ 노트북 (백업) ]
        │
        ▼  5분 안에 동일 환경 복구 완료
```

| 명령 | 행동 |
|------|------|
| `yadm add` | 추적할 파일 추가 (스파스 체크아웃으로 일부만) |
| `yadm commit` | 변경 커밋 |
| `yadm push` | 원격 저장 |
| `yadm clone` | 새 머신에 복원 |
| `yadm status` | 변경된 파일 확인 |

### 정리 작업의 보상

> 솔직히 말하면 제 컴퓨터가 완전히 고장 났을 때 저는 크게 실망하지 않았습니다. 처음부터 새로 시작할 수 있다는 게 오히려 좋았습니다. 시간이 지나면 제 드라이브는 목적을 잃은 잡다한 항목들로 가득 차거든요. 정돈된 기계는 조금 더 빨라지고 오류도 덜 발생하는 것처럼 느껴집니다.

## 6. 자동화 점검 — 환경 회복 체크리스트

| 측정 항목 | 목표 |
|-----------|------|
| 새 머신을 받아 *생산적인 작업 시작*까지 | 1시간 이내 |
| 단일 명령 배포 가능 | 베어 머신에서 |
| OS+SSH 키만 남긴 머신 → 풀 배포 | 1주일에 1회 검증 |
| 마우스 없이 편집 가능 시간 | 하루 종일 |

## 요약

- 자동화의 본질은 *기억해야 할 것을 줄이는 것*. 인지 부담이 신뢰성으로 직결된다.
- 작업공간 + 타일형 윈도우 매니저로 *컨텍스트 분리*. 마우스 없이 키보드로.
- 터미널은 운영체제 전체에 접근하는 초능력이다. 명령은 *조합·파이프·자동완성*으로 강해진다.
- 0일차부터 *Hello, World*를 배포하라. 자동화는 *코드보다 먼저*.
- 황금률: 베어 머신에서 단일 명령으로 앱을 실행할 수 있어야 한다.
- 에디터는 *가장 짧은 지름길*. 기본부터 시작해 필요할 때만 확장. Emacs 수천 줄 Lisp의 함정에 빠지지 말 것.
- 개발 머신도 프로덕션 서버처럼 프로비저닝한다. **asdf + Homebrew + yadm** 트리오로 5분 안에 복구.
- 정돈된 기계는 더 빠르고 오류가 적다.

## 다른 챕터와의 관계

이 챕터는 *Part 2(환경의 단순화)*의 첫걸음이다. Part 1이 *코드와 팀*을 줄였다면, 이 챕터는 *도구와 절차*를 줄인다.

다음 챕터(Ch 5)는 *학습과 변화*로 옮겨 간다 — 자동화로 비워 낸 시간을 어떻게 활용할 것인가에 답한다.

*8장(가독성 높은 코드)*의 정렬·다중 커서 같은 실천법은 본 챕터의 에디터 자동화 위에서만 효율적으로 작동한다.

1장의 *현황파악-실행-학습* 사이클은 자동화에도 적용된다. 매주 머신을 초기화하고 배포 명령을 검증하는 것이 *학습*에 해당하는 회고다.

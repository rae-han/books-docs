# Chapter 5: Embracing Change (변화의 수용)

## 핵심 질문

급변하는 소프트웨어 세계에서 어떻게 도태되지 않고 지속적으로 성장할 것인가? *탐색(exploration)*은 정당한 업무인가, 사치인가? 새 기술과 검증된 기술 사이의 균형은 어떻게 잡는가?

> 우리는 나이가 들어서 놀이를 멈추는 것이 아니다. 놀이를 멈추기 때문에 나이가 드는 것이다.
> — *조지 버나드 쇼(George Bernard Shaw)*

이 챕터는 코드가 아니라 *개발자 자신*의 단순화를 다룬다. 제프리 토머스가 펀치카드에 BASIC을 천공하던 일을 지금까지 하고 있다면, 그는 실업자였을 것이다.

> **아이디어 34**: 소프트웨어 개발은 관리되는 변화입니다.

켄트 벡의 *익스트림 프로그래밍* 부제는 "Embrace Change"다. 변화하는 환경·요구사항·기술에 적응하는 팀이 더 성공한다는 철학이다. 저자는 이를 *사람*에게도 적용한다.

## 1. 정찰대 비유

작은 걸음으로 나아가라는 조언은 *앞에 무엇이 있는지 모를 때*를 가정한다. 만약 정찰대를 보낼 수 있다면?

| 도메인 | 정찰대의 이름 |
|--------|---------------|
| 소프트웨어 개발 | 프로토타입 |
| 개인 개발 | **탐색(Exploration)** |

> **핵심 통찰**: 프로토타입과 리서치 모두 우리의 결정을 더 확신 있고 정확하게 만든다.

## 2. 왜 탐색해야 하는가

### 실용적인 이유

> 모든 개발자가 잘 알고 있듯이, 지금까지 만들어진 모든 산업 중에서 소프트웨어 개발 분야가 아마도 가장 변동성이 큽니다. 적응하고 받아들이거나, 도태되거나 둘 중 하나입니다.

### 본질적인 이유

> 우리는 개발자이고, 마치 마법사와 같습니다. 상상에서 무언가를 불러내어 이전에는 불가능했던 것들을 현실로 만듭니다. 그리고 우리는 대체로 모두의 이익을 위해 그렇게 합니다. 그리고 — 아시잖아요? 재미있습니다!

저자의 가족은 탐색을 *놀이*라고 부른다. 배워야 해서 배우는 것은 골칫거리지만, **원해서 배우는 것은 기쁨**이다.

## 3. PRACTICE 14 — 실용성과 기발함 섞기

### 두 가지 목표

저자는 탐색에서 두 가지 목표를 세운다.

| 목표 | 핵심 질문 |
|------|-----------|
| **실용성 (경력 도움)** | 경력 개발에 도움이 되는 기술 분야는 무엇인가? |
| **개인적 보람 (놀이)** | 업무 맥락을 벗어나 개인적으로 흥미로운 기술 분야는 무엇인가? |

> 저는 후자를 더 우선시합니다.

### 저자의 2025년 탐색 영역 (예시)

| 분류 | 항목 |
|------|------|
| **실무적** | Headless CMS, 프론트엔드 프레임워크, 엣지 서비스 |
| **실무적** | 다운타임 최소화 DB 마이그레이션, 단순 배포 옵션 |
| **실무적** | Ruby on Rails / Phoenix 최신 동향 |
| **개인 흥미** | 마이크로컨트롤러 (ESP32, 센서, 디스플레이) |
| **개인 흥미** | 실용적인 함수형 프로그래밍 |
| **개인 흥미** | 대수적 효과 (모나드의 대안적 해법) |
| **개인 흥미** | Unison 언어 — 이뮤터블 코드, 512비트 체크섬으로 배포 |

> **아이디어 35**: 탐색은 어른들의 놀이입니다.

### 탐색은 정당한 업무다

> **아이디어 36**: 일만 하고 놀지 않으면 저도 꽤 심술궂게 변합니다.

탐색을 *여가에 하는 일*로 여기면 두 가지 함정에 빠진다:

1. 여유 시간이 없어 결국 못 한다.
2. 죄책감 때문에 제대로 즐기지 못한다.

또는 한동안 손을 놓으면 압박감이 쌓여, *해야만 하는 지경*이 되었을 때는 항목 목록이 너무 많아 어느 것도 깊게 파고들지 못한다.

### 정기적 학습 시간 확보

| 패턴 | 예시 |
|------|------|
| 매일 짧게 | 하루 20분 |
| 가끔 길게 | 어느 저녁 3시간 |
| 저자 패턴 | 아침: 이메일 후 30분 독서. 밤: 침대에서 독서 |

### 세션 목표와 일지

```
세션 목표 예시:
  ✦ "Unison으로 분산 프로그램 작성하기"
  ✦ "서버사이드 렌더링이 배포에 어떤 영향을 미칠까?"
  ✦ "José Valim은 지금 무엇을 하고 있을까?"
```

일지를 펼쳐 놓고 메모한다. 흥미로운 *다른 영역*을 발견하면 동그라미로 표시해 두고, 이번 세션 목표가 빗나가지 않게 한다.

### 약간의 놀이 시간

저자의 사례:

| 프로젝트 | 도구 | 배운 것 |
|----------|------|---------|
| Fisher-Price 풀-A-튠 실로폰 → 터치 MIDI 컨트롤러 | Raspberry Pi | 사운드 처리, 저수준 네트워킹, 플라스틱 다루기 |
| Teddy Ruxpin 곰인형이 디지털 녹음에 반응해 움직이게 | 서보모터 | 동일 |

> **아이디어 37**: 무엇이든 사용하지 않으면 무뎌집니다.

배운 뒤에는 빠른 프로토타입을 만들어 본다. 엉성한 코드라도 괜찮다 — 끝나면 삭제할 테니까. 일지에 배운 것을 적는다.

### 두 개의 목록 만들기

```
[ 해야 할 일 (필수) ]      |   [ 하면 재미있는 일 (호기심) ]
                          |
1. Headless CMS           |   1. ESP32 IoT
2. DB 마이그레이션 기법       |   2. 함수형 언어
3. 프론트엔드 프레임워크      |   3. 대수적 효과
                          |
       ↑ 번갈아 가며 한 항목씩 진행 ↑
```

처음에는 번갈아 가며: 필수 1개 → 호기심 1개 → 필수 1개. 진행 중 새 주제가 떠오르면 맞는 목록에 추가.

## 4. PRACTICE 15 — 미래에서 놀고, 과거에서 일하기

> 과거와 미래를 모두 바라본 로마의 신, 야누스(Janus).

### 야누스의 두 얼굴

새롭고 흥미로운 것을 접하면 다음 프로젝트에 적용해 보고 싶은 충동이 든다. 신선한 아이디어를 실제로 사용해 보며 경험을 쌓고 싶다.

| 두 얼굴 | 의미 |
|---------|------|
| **미래(놀이)** | 새 기술로 자유롭게 실험. 실패해도 잃을 게 없다 |
| **과거(일)** | 검증된 기술로 실제 업무를 처리. 안정성 우선 |

이 챕터의 핵심은 **새로움을 추구하는 영역과 안정을 추구하는 영역을 분리**하라는 것이다. 흥미로운 것을 발견했다고 해서 곧장 프로덕션에 들이면 안 된다. 먼저 *놀이의 공간*에서 충분히 다뤄 보고, 검증되었을 때만 *일의 공간*으로 옮긴다.

> **핵심 통찰**: 새로운 것은 미래에 두고 놀이로 즐기되, 일은 과거의 검증된 도구로 한다. 그래야 단순함을 잃지 않고도 성장할 수 있다.

## 5. 단순한 학습 사이클

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 목표 설정 │ ─▶│ 시간 확보 │ ─▶│ 탐색 실행 │ ─▶│ 일지 기록 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ▲                                              │
      └──────────────  다음 사이클로 반영  ──────────┘
```

이 사이클은 1장의 *현황파악-실행-학습*과 정확히 같은 구조다. 코드뿐 아니라 자기 자신에게도 적용된다.

## 6. 변화 수용 체크리스트

| 항목 | 자기 점검 |
|------|------------|
| 정기적 학습 시간 | 매주 정해진 시간이 있는가? |
| 두 목록 | 필수 / 호기심 목록이 분리되어 있는가? |
| 세션 목표 | 매 세션마다 적어 둔 한 줄짜리 목표가 있는가? |
| 일지 | 배운 것을 흘려보내지 않고 적어 두는가? |
| 야누스의 얼굴 | 미래의 놀이 공간과 과거의 일 공간이 분리되어 있는가? |
| 프로토타입 | 배운 것을 코드로 검증해 보았는가? |

## 요약

- 소프트웨어 개발은 *관리되는 변화*. 적응하지 못하면 도태된다.
- 탐색은 어른들의 놀이다. *원해서 배우는 것*만이 기쁨이고, 그것이 가장 깊이 남는다.
- 두 가지 목표를 모두 가져라: **실용성**(경력)과 **개인적 보람**(놀이).
- 탐색은 *정당한 업무*다. 여가에 죄책감 속에 하지 말 것.
- 정기적 시간을 확보하라 — 하루 20분이든 한 저녁 3시간이든. 일지에 적어라.
- *야누스의 두 얼굴*: 미래의 새 기술은 놀이 공간에, 과거의 검증된 기술은 일 공간에.
- 사용하지 않으면 무뎌진다. 배운 것은 프로토타입으로 검증하라.

## 다른 챕터와의 관계

이 챕터는 *Part 2(환경의 단순화)*를 닫는다. Ch 4가 *외부 환경(도구·자동화)*의 단순화였다면, Ch 5는 *내부 환경(학습·태도)*의 단순화다. 자동화로 비워 낸 시간을 어떻게 채울 것인가에 대한 답이기도 하다.

다음 *Part 3(상호작용의 단순화)*은 시선을 다시 외부로 돌린다 — Ch 6 소프트스킬에서 동료와의 협업을 다룬다. 자기 자신을 단순화한 사람이 동료와의 관계를 단순화할 수 있다.

1장의 *현황파악-실행-학습* 사이클이 이 챕터에서 가장 직접적으로 등장한다. 학습 자체가 그 사이클이기 때문이다.

3장의 "보유한 기술 널리 퍼뜨리기(Practice 07)"와 짝을 이룬다 — 본 챕터가 *개인 학습*이라면, Practice 07은 *팀 차원의 지식 분산*이다. 둘 다 정보 사일로를 막는 같은 동전의 양면이다.

# Chapter 3: Project Optimization (프로젝트 최적화)

## 핵심 질문

코드는 잘 만들었는데 왜 팀은 여전히 느린가? 회의·결합·지식 사일로처럼 *조직의 복잡함*은 어떻게 단순화할 수 있는가? 그리고 이 모든 것의 출발점인 **콘웨이의 법칙**을 우리는 어떻게 역이용할 것인가?

> 어떤 조직이 설계하는 시스템은 해당 조직의 의사소통 구조를 반영한다.
> — *멜빈 콘웨이(Melvin Conway, 1968)*

저자는 2023년 말 약 50개 팀과 대화하며 *성공적인 팀이 공통으로 가진 세 가지*를 발견했다.

| 요소 | 의미 |
|------|------|
| **의사소통** | 정보가 빠르고 정확하게 흐른다. 의견 차이는 인정한다 |
| **보완적인 기술** | 사람마다 맡은 작업이 다르며, 어느 작업도 더 가치 있다고 여기지 않는다 |
| **아낌없는 공유** | 자신의 지식과 경험을 팀원에게 전하는 것을 책임으로 여긴다 |

이 챕터의 5개 프랙티스는 위 세 요소를 단순함의 관점에서 풀어낸다.

## 1. PRACTICE 04 — 팀의 결합도 낮추기

### 코드의 직관을 팀에 적용하기

개발자는 코드에 대해 이런 직관을 갖고 있다.

- 결합도가 낮은 코드는 변경하기 쉽다.
- API와 모듈의 경계는 코드를 이해하기 쉽게 한다.
- 작은 단위는 큰 덩어리보다 작성하기 쉽다.
- 추상화 계층은 복잡성을 다루는 데 도움이 된다.

**이 직관을 팀에도 적용할 수 있는가?** 저자의 답은 *그렇다*.

### 시간적 결합 (Temporal Coupling)

> 시간적 결합이란, 팀 구성원이 다른 팀 구성원을 *기다리게 만드는 모든 것*이다.

전형적인 시간적 결합의 예:

```
[ Hassan ] ──의존──▶ [ Jan의 코드 ]
                          │
                          └─ Jan은 휴가중
                          
결과: Hassan은 일을 진행하지 못함
```

| 결합 패턴 | 손실 |
|-----------|------|
| 한 번에 하나의 기능만 — 다음 단계 진행은 검토 후에야 | 직선 대기 |
| 막판 긴급 회의 소집 | 회의 시간 + 맥락 회복 30분 |
| 2주 단위 동기화된 스프린트 | 새 작업이 다음 스프린트까지 묶임 |

> **아이디어 11**: 결합은 팀과 코드 모두에 영향을 미칩니다.

### 비동기적인 삶

> 저는 *유지보수 모드*라고 부르면서 실제로는 거의 무기한 그 상태에 머물렀던 팀에서 일한 적이 있습니다. 이런 방식은 팀의 사기를 꺾고 우선순위를 정하는 능력을 무너뜨리며 항상 런닝머신 위에 있는 듯한 느낌을 줍니다. — *Noel Rappin*

**이벤트 주도 방식은 애자일 계획의 안주 모드(fallback mode)**다. 진짜 애자일 프랙티스보다 작업량은 적지만 효과는 훨씬 떨어진다.

비동기 협업의 도구들:

| 도구 | 효과 |
|------|------|
| 이메일 | 보내고 다음 일을 진행 |
| 공유 작업 목록 / 칸반 보드 | 카드를 옮기면 상대가 알아챔 |
| 공유 문서 | 검토 대기 없이 코멘트로 비동기 토론 |

저자의 사례: *실용주의 프로그래머* 2판 집필 시 GitLab 칸반 보드로 원고를 비동기로 주고받았다. 한 사람이 끝내면 카드를 옮기고 담당자를 바꾼다.

> **아이디어 12**: 비동기적으로 상호작용할 방법을 찾아보세요.

### 실험: 대기 시간 측정

| 단계 | 행동 |
|------|------|
| 1주차 | 대기 상황을 기록하고 잃어버린 시간을 적는다 |
| 분류 | 회의 대기 / 코드 리뷰 대기 / 다른 사람 코드 완료 대기 / 질문 답변 대기 |
| 실험 | 각 분류에서 대기를 없앨 방법을 찾고 적용 |
| 비교 | 1~2개월 후 합계를 비교 |

## 2. PRACTICE 05 — 지긋지긋한 회의 줄이기

### 회의의 (이론적인) 목적과 현실

도널드 럼스펠드의 3가지 지식 분류:

| 분류 | 의미 |
|------|------|
| Known knowns | 우리가 안다는 사실을 아는 것 |
| Known unknowns | 우리가 모른다는 사실을 아는 것 |
| Unknown unknowns | 우리가 모른다는 사실조차 모르는 것 |

회의의 이상은 1번 지식을 가진 사람이 나머지를 깨우치는 것이다. 그러나 현실은 1번 그룹과 3번 그룹이 각자의 현실을 놓고 논쟁하는 동안 2번 그룹이 소셜미디어를 본다.

> **아이디어 13**: 회의는 원래 애자일하지 않습니다.

### 회의의 4가지 죄악

| 죄악 | 근거 |
|------|------|
| **비효율** | 12명 회의 5분 = 1시간 손실. 늦게 시작·발표자 노트북·잡담·준비 부족 |
| **불공평** | 직급/공격성/큰 목소리에 지배됨. 조용한 사람의 의견은 묻힘 |
| **방해** | 시작 10분 전부터 새 작업 시작 못 함. 종료 후 맥락 복원 15~30분 |
| **비싸다** | 스크럼 2주: 13.5시간 × 10명 × $200/시간 = **$27,000** |

> **아이디어 14**: 회의의 부정적인 영향은 회의실 밖 멀리까지 퍼집니다.

> **아이디어 15**: 회의는 비효율적이고 불공평하며 업무를 방해하고 비용도 많이 듭니다.

### 포트폴리오 없는 개발자 (DWP, Developer Without Portfolio)

저자가 만난 성공적 팀들이 만들어 낸 새 역할이다.

> **아이디어 16**: 포트폴리오가 없는 개발자는 팀 내 마찰을 줄입니다.

```
        ┌──────────────────────┐
        │  팀 = 엔진             │
        │  DWP = 윤활유          │
        └──────────────────────┘
              │
              ▼
   각 팀원이 무엇을 하는지, 어떤 고충/오해가 있는지 파악
              │
              ▼
        장애물 제거 / 사람 연결
```

DWP의 효과로 사라지는 회의들:

| 회의 종류 | DWP 도입 후 |
|-----------|-------------|
| 데일리 스탠드업 | DWP가 상시 파악 후 갱신·공유 |
| 리뷰 회의 | DWP가 관찰을 사전에 공유 → 해결책에 집중 |
| 설계 회의 | 3~4명으로 제한, 6번 프랙티스의 지침 따름 |

회의 시간을 절반만 줄여도 개발자 한 명 근무 시간의 80%가 절감된다. **DWP 도입은 1.5명 추가 투입과 같다.**

## 3. PRACTICE 06 — 굳이 회의를 해야 한다면 생산적으로 하자

### 좋은 회의의 7가지 요소

| 요소 | 핵심 |
|------|------|
| 명확한 목표 | "X에 대해 논의" 같은 모호함 금지. 측정 가능해야 함 |
| 필요성 확인 | 일을 하는 것보다 회의가 더 쉬운 것뿐 아닌가? |
| 준비 | 거꾸로 수업(flipped classroom) 모델 |
| 참석자 선택 | 결과 도출에 실질적으로 필요한 사람만 |
| 일정 | 자연스러운 구간(아침 첫 일정 등). 시간대 배려 |
| 진행 | 정시 시작, 토큰(세마포어), 진행자가 모범 |
| 정리·후속 | 마무리 시간 확보, 메모 발송 |

> **아이디어 17**: 좋은 회의는 오직 잘 준비된 회의뿐입니다.

### 회의 뒤집기 — Flipped Meeting

기존 회의 대부분은 학교 수업과 닮았다. 한 사람이 길게 설명하고, 나머지는 자리로 돌아가 일을 분배받는다. 이를 뒤집어라.

> **아이디어 18**: 회의 뒤집기 — 사전에 공유하고 진행 중에 검토하세요.

| 전통 회의 | 회의 뒤집기 |
|-----------|-------------|
| 자리에서 정보 전달 | 사전 자료로 미리 공부 |
| 회의 끝나고 행동 결정 | 회의 자리에서 곧장 논의·결정 |

숨은 이점: 자료를 준비하다 보면 *문제를 오해했음을 깨닫고 회의 자체가 사라지는* 경우가 많다.

### 측정 가능한 회의 목표 — 좋은 예와 나쁜 예

| 좋은 목표 (측정 가능) | 나쁜 목표 (모호) |
|----------------------|------------------|
| 영업 모듈을 레드팀 → 블루팀 인수인계 *계획 수립* | 다음 릴리스의 배포에 대해 *논의* |
| 연말 휴가 일정 *조율* (캘린더 공백 없음 확인) | 프로젝트 추정치를 *개선* |
| DB 스키마 마이그레이션 *자원·권한 확보 확인* | (행동 후 결과를 정의할 수 없는 것) |

### 탐색적 회의는 다르다

저자가 *애자일 소프트웨어 개발 선언* 워크숍에 참여했을 때 사용한 방식은 "We value X over Y" 형식의 문장을 만들어 내는 것이었다.

> **아이디어 19**: 탐색적 회의에는 다른 규칙이 적용됩니다.

탐색적 회의의 특징:

- 의자를 타원형으로 배치
- 인덱스 카드에 각자 한 항목씩 적기
- *스티커 투표*로 우선순위 결정 (각자 3~4표)
- 진행자(가능하면 외부)가 필수

> **아이디어 20**: 서로 배려하며 협력하세요.

## 4. PRACTICE 07 — 보유한 기술 널리 퍼뜨리기

> 배움 속에서 가르치고, 가르침을 통해 배운다.
> — *필 콜린스(Phil Collins), Son of Man 가사 중*

### 가르치며 배우는 역설

저자가 2000년대 중반 컨퍼런스에서 페어 프로그래밍 부스를 운영하며 깨달은 것:

> 그 몇 시간 동안 저는 지난 2년 동안 배운 것보다 더 많은 것을 배웠습니다. 학생들은 단 한 단어로 제 모든 기반을 무너뜨렸습니다. *"왜죠?"*

> **아이디어 21**: 배우기 위해 가르치세요.

기술 공유는 사탕 나눔과 다르다. **모두가 승자가 되는 게임(positive-sum game)**이다.

### 동질성은 목표가 아니다

다양성이 부족한 팀은 *마르모크렙스 민물가재*나 *치타*처럼 환경 변화에 취약하다. 자바와 오라클만 아는 팀은 세상이 PostgreSQL과 자바스크립트로 옮기면 쓸모없어진다.

```
경험을 시각화하면 (5명 팀, 9개 기술 영역):

다양한 팀 (커버리지 양호)        |   동질적 팀 (커버리지 빈약)
-------------------------------- | --------------------------------
DB1 ●●●  Lang1 ○    Env1 ●●●●  |  DB1 ●●●●●  Lang1 ●●●●●  Env1 ●●●●●
DB2 ●●    Lang2 ●●●  Env2 ●●○○  |  DB2 ○○○○○  Lang2 ○○○○○  Env2 ○○○○○
DB3 ●●●●  Lang3 ●●●  Env3 ●●●●  |  DB3 ○○○○○  Lang3 ○○○○○  Env3 ○○○○○
```

> **아이디어 22**: 모든 경험은 유용합니다. 다양한 경험이 더 많은 것을 가르쳐줍니다.

### 가르치는 자세

- 누군가를 바로잡고 싶을 때 **행동 전에 멈추고** 왜 그런 실수를 했는지 생각한다.
- 답을 주는 대신 함께 원인을 탐색하고 *스스로 해결책을 찾게* 한다.
- 누군가 내 작업의 문제를 발견하면 *방어적*이지 않게, 그 말을 곱씹어 본다.

## 5. PRACTICE 08 — 정보를 자유롭게 흐르게 하기

### 정보 사일로의 형성

작은 팀에 애착이 생기면 자체 용어가 생긴다. 다른 팀과 대화하기 어려워지고, 결국 따로 일하게 된다. 정보가 *사일로(silo)*에 갇힌다.

> **아이디어 23**: 사적으로 축적한 지식은 썩고 맙니다.

지식 두 가지를 결합했을 때의 가치는 *단순히 더한 것*보다 더 크다. 새 아이디어가 기존 모든 아이디어와 상호작용해 새로운 직관을 만든다.

### 지식 분산을 위한 9가지 활동

| 활동 | 핵심 |
|------|------|
| **페어 프로그래밍** | 짝을 자주 교체. 모든 구성원이 서로 협업하도록 |
| **DWP** | 질문이 있는 사람을 답이 있는 사람과 연결 |
| **점심 도시락 모임** | 북클럽이나 흥미로운 발견 공유 |
| **개발자 셔플** | 몇 달에 한 번씩 팀 간 한 명씩 이동 — 정체된 프로젝트의 새 시각 |
| **코드 리뷰** | 팀 외부 사람에게 리뷰 요청 |
| **몹 프로그래밍** | 일반 문제 하나를 여러 팀이 하루 동안 함께 해결 |
| **기술 역량 섞기** | 상호보완적 기술의 구성원으로 팀 구성 |
| **다양성 경험** | 경험 수준이 다른 구성원 — 주니어의 "왜?"가 시니어를 깨운다 |
| **금요일의 놀이 시간** | 주당 반나절을 비업무 프로젝트에. ESP32 실험, 새 언어 학습 등 |

> **아이디어 24**: 지식을 공유하면 그 지식은 배가됩니다.

### 경영진의 동의가 필요한 이유

> 대규모 조직에서 이런 실천법을 제대로 적용하려면 경영진의 확실한 동의와 지원이 정말로 필요합니다. 이러한 활동들은 대개 관리자의 인센티브에 역행하는데, 관리자 인센티브는 보통 단기 성과에 집중되어 있기 때문입니다. — *Noel Rappin*

## 요약

- 50개 팀의 성공 요인은 **의사소통, 보완적 기술, 아낌없는 공유**다.
- 팀의 *시간적 결합*은 코드의 결합과 똑같이 해롭다 — 비동기 도구(칸반, 공유 문서)로 풀자.
- 회의는 비효율적·불공평·방해·비싸다. 10명 팀의 2주 회의 비용 = $27,000.
- *DWP(포트폴리오 없는 개발자)*: 데일리 스탠드업과 리뷰 회의를 대부분 흡수하는 윤활유 역할.
- 회의를 꼭 해야 한다면 *측정 가능한 목표*와 *사전 자료*로 시작하라. 회의 뒤집기는 강력하다.
- 가르치면서 배운다. *왜?*라는 질문이 시니어의 자동화된 직관을 다시 깨운다.
- 다양성은 회복력이다. 동질적인 팀은 환경 변화에 취약하다.
- 정보 사일로는 썩는다. 지식을 모은 사람의 머릿속에서만 살게 두지 말 것.

## 다른 챕터와의 관계

이 챕터는 *Part 1*을 닫는다. Part 1은 "하는 일과 일하는 방식의 단순화"를 다룬다 — 2장이 *코드와 의존성*을, 3장이 *팀과 프로세스*를 줄였다.

이어지는 *Part 2(Ch 4~5)*는 환경의 단순화로 옮겨간다. 도구·자동화·학습 환경의 차원이다. 그러나 모든 챕터의 출발점은 1장의 사이클(현황파악 → 실행 → 학습)이다. DWP의 책임 중 하나도 이 사이클을 팀에 정착시키는 것이다.

6장(소프트스킬)에서 등장하는 *공감과 의견 대립의 원칙*은 본 챕터의 회의·협업 조언이 작동하기 위한 토대가 된다.

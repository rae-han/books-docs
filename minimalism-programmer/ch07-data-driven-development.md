# Chapter 7: Data-Driven Development (데이터 주도 개발)

## 핵심 질문

코드와 데이터 중 누가 운전대를 잡아야 하는가? 반복적인 코드를 *데이터 구조*로 바꾸면 왜 더 단순해지는가? 테스트와 복잡한 분기 로직을 어떻게 데이터로 대체할 수 있는가?

> 한때 저는 프로그래머의 역할이 *데이터를 조작하는 코드*를 짜는 것이라고 생각했습니다. 입력값을 출력값으로 바꾸고 정보를 만들어 내는 주체가 코드라고 믿었죠. 물론, 제 판단이 잘못되었습니다.

저자의 깨달음은 **코드는 주도적인 역할을 하지 않는다**는 것이다. 코드는 가치를 전달할 때 사용하는 여러 도구 중 하나일 뿐이다. 효과적인 프로그래밍은 데이터와 코드가 *서로 제어권을 주고받는* 구조다.

> **아이디어 45**: 데이터는 코드보다 단순합니다.

## 1. 왜 데이터가 더 단순한가

| 측면 | 코드 | 데이터 |
|------|------|--------|
| 변경 용이성 | 컴파일·테스트·배포 필요 | 값만 바꾸면 됨 |
| 논리 이해 | 흐름 추적 필요 | 표를 보면 한눈에 |
| 외부화 | 어렵다 | 쉽다 (설정 파일, DB) |
| 재사용 | 함수 단위 | 데이터 단위 |

이 챕터에서 다루는 두 가지 방법:

```
1. 테이블로 코드 실행 순서를 제어
2. 상태 머신으로 복잡한 로직을 단순화
```

## 2. PRACTICE 20 — 데이터에 운전대 맡기기

> 코드를 데이터로 바꿔치기 하는 그 뿌듯함이야말로 소프트웨어 개발의 기쁨입니다.

### 도형 그리기 예제 — 코드의 반복

```typescript
// 첫 번째 도형 추가
shape = add_shape(Triangle, { sides: 3 });
shapes_by_sides[3].push(shape);

// 두 번째 도형 추가
shape = add_shape(Square, { sides: 4 });
shapes_by_sides[4].push(shape);

// 세 번째 도형 추가
shape = add_shape(Rhombus, { sides: 4 });
shapes_by_sides[4].push(shape);
```

세 번째 도형쯤에서 *변경되는 데이터*와 *변하지 않는 코드*를 분리하고 싶어진다.

### 데이터 주도 변환 후

```typescript
[
  [Triangle, 3],
  [Square,   4],
  [Rhombus,  4],
].forEach(([cls, sides]) => {
  const shape = add_shape(cls, { sides });
  shapes_by_sides[sides].push(shape);
});
```

원본보다 약간 길지만 *더 단순*하다. 두 가지 이유:

1. **도형 목록이 명확** — 한눈에 확인 가능, 유지보수 수월
2. **두 개념이 분리됨** — 도형의 *목록*과 시스템에 *추가하는 코드*가 분리

새로운 도형을 추가하려면 표에 한 줄만 더 입력하면 된다.

### 오류 처리가 포함된 메서드 체인

```typescript
// ❌ 중첩된 오류 처리 — 평탄화되지 않음
let square = new Square(10);
let [status, square1] = scale(square, 0.5);
if (status === "ok") {
  let [status2, square2] = rotate(square1, -5);
  if (status2 === "ok") {
    let [status3, square3] = skew(square2, 0.1);
    if (status3 === "ok") {
      display(square3);
    }
  }
}
```

데이터 주도 변환:

```typescript
// ✅ 데이터 주도 — 평탄화됨
const transform = [
  (shape) => scale(shape, 0.5),
  (shape) => rotate(shape, -5),
  (shape) => skew(shape, 0.1),
];

let shape = new Square(10);
for (const cmd of transform) {
  const [status, newShape] = cmd(shape);
  if (status !== "ok") {
    shape = undefined;
    break;
  }
  shape = newShape;
}

if (shape) {
  display(shape);
}
```

| 이점 | 설명 |
|------|------|
| 명세와 실행 분리 | 변환 순서의 *명세*와 *실행 코드*가 분리됨 |
| 데이터 재사용 | 변환 목록을 다양한 곳에서 활용 가능 |
| 평탄화된 흐름 | 중첩 대신 반복문 (또는 reduce) |
| 단위 테스트 용이 | 각 변환을 개별로 테스트 |

> **아이디어 46**: 데이터는 소스 코드보다 변경하기가 더 쉽습니다.

> **아이디어 47**: 데이터는 *무엇에 해당하며*, 코드는 *어떻게 구현하는지*를 의미합니다.

### 데이터 주도 코드는 인터프리터다

> 제가 인터프리터라고 언급할 때, 많은 분들은 자바·루비·파이썬·엘릭서 같은 거대하고 복잡한 환경을 떠올립니다. 제가 말하는 인터프리터는 꼭 그렇게 복잡한 것은 아니어도 됩니다.

```
인터프리터 = 데이터를 기반으로 명령을 받아 동작하는 코드

도형 그리기 예제:
  - 데이터: 도형 이름과 변의 수가 저장된 테이블
  - 인터프리터: 테이블을 순회하며 각 도형을 호출하는 반복문
```

> **핵심 통찰**: 단순한 인터프리터는 소프트웨어를 유연하게 변경할 수 있도록 하는 핵심 전략이다.

### 외부 데이터에 의한 구동

데이터는 코드 내부에 하드코딩될 필요가 없다. *동적*일 수도 있고, 사용자 상호작용에 따라 *실시간 생성*될 수도 있다. 도형 그리기 앱에서 사용자가 *변환 절차*에 이름을 붙여 저장하고 다른 도형에 적용할 수도 있다.

## 3. PRACTICE 21 — 테이블로 테스트를 단순화하기

### PDP-11 에뮬레이터 사례

저자가 1970년대 미니컴퓨터 PDP-11의 어셈블러와 에뮬레이터를 만들면서 겪은 사례. 16비트 워드에 연산 코드, 두 가지 주소 지정 방식, 두 개의 레지스터까지 담겨야 하는 복잡한 명령어 형식.

### 명령어 디코드 — 반복적 코드

```javascript
test("decode mov", () => {
  expect(emulator.decode(0o010000)).toBe("mov");
});

test("decode movb", () => {
  expect(emulator.decode(0o110000)).toBe("movb");
});

test("decode cmp", () => {
  expect(emulator.decode(0o020000)).toBe("cmp");
});

// ... 모든 명령에 대해 반복 (수백 개)
```

> 보일러플레이트 코드가 반복되다 보니 실제로 테스트하려는 *대상*이 무엇인지 한눈에 파악하기 어렵습니다.

### 테이블 주도 방식

```javascript
[
  [0o010000, "mov"],
  [0o110000, "movb"],
  [0o020000, "cmp"],
  // ...
  [0o160000, "sub"],
].forEach(([instruction, op]) => {
  test(`decode ${op}`, () => {
    expect(emulator.decode(instruction)).toBe(op);
  });
});
```

각 행이 *고유한 테스트*를 생성하며, 테스트 이름에 연산 코드 이름이 포함되어 어떤 동작인지 한눈에 알 수 있다.

> **아이디어 48**: 반복적인 테스트에는 테이블을 활용하세요.

### 데이터를 활용한 다단계 테스트 — Shopify 사례

쇼피파이 GraphQL API로 주문 데이터를 수수료 분배 시스템에 입력하는 검증.

문제: API는 주문의 *현재 상태*만 알려주지만 *과정*은 알려주지 않는다.

```
시간 t1: 고객 구매 (결제 트랜잭션 없음)
시간 t2: 결제 완료 → 결제 트랜잭션 추가, 일부 합계 수정
시간 t3: 고객이 쿠폰 적용 요청 → 할인 정보, 할인 적용, 환불 거래 추가
시간 t4: 부분 환불 → 또 새 레코드들
```

각 테스트는 여러 단계로 구성되어야 한다.

```ruby
[
  ["three items then refund one", [
    [...step1...],
    [...step2...],
    [...step3...],
  ]]
]
```

각 단계는 다음과 같은 중첩된 구조:

```ruby
"#2288-a-start-with-one-item",
expect(2288, TEST_USER_FOX_HOLLOW, TEST_USER_FOX_HOLLOW) do
  tax_total: BigDecimal(2.06),
  shipping_total: BigDecimal(0),
  discount_total: BigDecimal(0),
  grand_total: BigDecimal(27.01),
  product_total: BigDecimal(24.95),
  ["tvm-elixir-p-00", 1, BigDecimal(24.95)],
  royalty_lines: [],
end
```

| 측정 | 코드 직접 작성 | 테이블 주도 |
|------|----------------|-------------|
| 단일 응답 테스트 | 135줄 | 데이터 한 행 |
| 두 응답 처리 테스트 | 200줄+ | 데이터 두 행 |
| 다양한 시나리오 검증 | 인내심 한계 | 무한 |

### 테스트는 본질적으로 데이터 주도다

여러 테스트가 동일한 패턴을 따르면서 입력과 출력만 다르다면, 로직을 *함수로 추출*하고 *데이터 테이블*로 매개변수화한다. 테스트 실패 시 *실패를 유발한 데이터*를 반드시 기록해야 한다.

## 4. PRACTICE 22 — 상태 머신으로 로직 단순화하기

### 상태 머신에 대한 두려움

> 상태 머신에 대한 *FUD(Fear, Uncertainty, Doubt)*가 여전히 많이 존재합니다. 이런 이유로 많은 개발자가 이 방법을 기피하죠. **그건 바보 같은 짓입니다.**

상태 머신은 테이블 기반 코드에 *컨텍스트*를 추가하고, 주어진 상황에 따라 실행할 코드를 *동적으로 결정*하는 능력을 더한 한 단계 발전된 개념이다.

> **아이디어 49**: 상태 머신을 구현하는 데 반드시 라이브러리나 디자인 패턴이 필요한 것은 아닙니다.

### 상태 머신을 쓸 때

| 사례 | 효과 |
|------|------|
| 파싱 (CSV 변형 등) | 이스케이프 문자 처리 |
| Shopify 주문 정보 추출 | 복잡한 규칙을 정확한 순서로 |
| 주문 이행 | 배송·환불·알림 생성 |
| 로그 패턴 감지 | 5초 내 같은 IP·URL 3회 접근 |

여러 이벤트가 시간차로 분산 발생하고, 각 이벤트가 다음 이벤트의 처리 방식에 영향을 줄 때 상태 머신이 빛을 발한다.

### 가장 단순한 상태 머신 — 노크식 볼펜

```
       누름
   ┌──────────┐
[ 심 들어감 ]      [ 심 나옴 ]
   └──────────┘
       누름
```

| 구성 요소 | 내용 |
|-----------|------|
| 이벤트 | 버튼 눌림 (한 종류) |
| 상태 | 심 나옴 / 심 들어감 (두 가지) |
| 전이 | 나옴↔들어감 (두 가지) |

구현은 한 줄:

```typescript
let penRetracted = true;
eventStream.on("click", () => {
  penRetracted = !penRetracted;
});
```

### 더 복잡한 예 — 키패드 잠금장치 (1-5-9)

```
요구사항:
  - 1, 5, 9를 순서대로 누르면 열림
  - 시퀀스 시작 전 아무 키나 자유롭게 누름 가능
  - 잘못된 숫자 입력 시 처음부터 다시
  - 중간에 1을 입력하면 시퀀스 초기화
```

상태 다이어그램:

```
                ┌──── 다른 숫자 ────┐
                ▼                    │
[wait_first] ──── 1 ────▶ [wait_second]
                              │
                ┌── 다른 숫자 ─┘
                ▼              │
              [wait_first]     │ 5
                               ▼
                          [wait_third] ── 9 ──▶ [unlocked]
                               │
                               └── 1 ──▶ [wait_second]
                               └── 다른 ─▶ [wait_first]
```

핵심 통찰: **유효한 시퀀스(예: 1-5)로 시작했어도 중간에 1이 입력되면 두 번째 숫자를 기다리는 상태로 돌아가야 한다.** 그래서 `15159`도 통과한다.

### 룩업 테이블로 구현

```ruby
TRANSITIONS = {
  "wait_for_first_digit" => {
    "1"     => "wait_for_second_digit",
    "other" => "wait_for_first_digit",
  },
  "wait_for_second_digit" => {
    "5"     => "wait_for_third_digit",
    "1"     => "wait_for_second_digit",
    "other" => "wait_for_first_digit",
  },
  "wait_for_third_digit" => {
    "9"     => "unlocked",
    "1"     => "wait_for_second_digit",
    "other" => "wait_for_first_digit",
  },
  "unlocked" => nil,
}
```

상태 머신을 구동하는 코드:

```ruby
state = "wait_for_first_digit"
while TRANSITIONS[state] && key = gets
  key = key.strip
  state = TRANSITIONS[state][key] || TRANSITIONS[state]["other"]
end
```

> 실제로 상태 머신 구현은 *코드 한 줄*로 이루어져 있습니다.

| 구성 | 데이터로 표현 | 코드로 표현 |
|------|----------------|--------------|
| 상태 | 딕셔너리 키 | (없음 — 변수 하나) |
| 전이 | 중첩 맵의 값 | (없음) |
| 실행 | (없음 — 데이터 룩업) | 한 줄의 룩업 |

### 더 나아갈 수 있는 길

| 확장 | 효과 |
|------|------|
| 전이마다 액션 | 표에 *동작*도 기록 |
| 장시간 상태 유지 | DB에 상태 저장 — 워크플로우 |
| 코드 동작을 이벤트로 | 일반 코드도 상태 머신으로 처리 |

## 5. 데이터 주도 개발의 균형

> 코드가 지나치게 많아지면 복합적복잡성이 높아지고, 소프트웨어의 유연성이 떨어질 수 있습니다. 데이터가 지나치게 많으면 코드가 실제로 어떤 동작을 하는지 파악하기 어려울 수 있습니다.

```
[ 코드만 많음 ]   ←──  복합적복잡성   ←──→   가독성 저하  ──→  [ 데이터만 많음 ]
                                      │
                                  [ 적절한 균형 ]
                                      │
                            데이터가 코드의 흐름을 주도하고
                            코드는 데이터/다른 코드와 분리됨
```

> **핵심 통찰**: 데이터가 코드의 흐름을 주도하고, 코드가 데이터와 다른 코드로부터 효과적으로 분리되어 있는 순간이야말로 소프트웨어의 가장 이상적인 상태다.

## 6. 데이터 주도 점검 체크리스트

| 항목 | 자기 점검 |
|------|------------|
| 반복 코드 | 같은 패턴이 3번 등장한 부분이 있는가? |
| 변하는 부분 | 변하는 데이터와 변하지 않는 코드를 분리할 수 있는가? |
| 테스트 보일러플레이트 | 테스트 본문이 거의 똑같고 입력만 다른가? |
| 중첩 if | `if-if-if` 깊이 3 이상의 분기가 있는가? |
| 시퀀스 처리 | 이벤트 시간차 처리에 임시 변수가 흩어져 있는가? |
| 외부화 | 데이터를 설정 파일이나 DB로 옮길 수 있는가? |

## 요약

- 코드는 주도적 역할이 아니다. **코드와 데이터가 제어권을 주고받는 구조**가 가장 단순하다.
- 데이터는 코드보다 *변경하기 쉽다*. 표 한 줄을 추가하는 것이 새 함수를 짜는 것보다 단순하다.
- 반복적 호출은 데이터 주도 인터프리터로 평탄화된다 — 도형 그리기, 변환 체인 등.
- 테스트의 보일러플레이트는 테이블로 옮긴다. PDP-11 사례처럼 *수백 개*가 *한 표*로.
- 다단계 테스트도 데이터로 표현 가능 — Shopify 사례처럼 *250줄 메서드*가 *한 행*으로.
- 상태 머신을 두려워 말 것. **룩업 테이블 + 한 줄 코드**로 충분하다.
- 데이터와 코드의 *균형*이 핵심. 너무 많은 데이터는 흐름을 가리고, 너무 많은 코드는 유연성을 죽인다.

## 다른 챕터와의 관계

이 챕터는 *Part 4(코드의 단순화)*의 첫 챕터로, 책 전체에서 가장 *코드 레벨*에 가까운 조언을 담고 있다.

2장의 *기능은 적을수록 좋다(Practice 03)*가 **무엇을 만들지 줄이라**는 거시적 조언이라면, 본 챕터는 **만든 것을 어떻게 표현할지** 줄이는 미시적 조언이다.

다음 8장(가독성 높은 코드)은 본 챕터의 데이터 구조를 *읽기 쉽게 표현*하는 방법을 다룬다 — 정렬, 마지막 쉼표, 테이블 형식 등 8장의 모든 프랙티스가 본 챕터의 데이터 테이블 가독성에 직접 적용된다.

6장의 *비유(Practice 19)*는 본 챕터의 *데이터로 코드를 대체*하는 발상의 인지적 사촌이다. 비유가 *낯선 도메인을 익숙한 도메인으로* 옮기듯, 데이터 주도 개발은 *복잡한 흐름을 단순한 표*로 옮긴다.

1장의 *현황파악-실행-학습* 사이클은 본 챕터에서 *반복 코드 식별 → 데이터로 추출 → 결과 검증*으로 나타난다.

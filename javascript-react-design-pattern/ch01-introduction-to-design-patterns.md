# Chapter 01: Introduction to Design Patterns (디자인 패턴 소개)

## 핵심 질문

> 디자인 패턴이란 무엇이며, 왜 현대 자바스크립트 개발에서 여전히 중요한가?

---

## 1. 디자인 패턴의 역사

디자인 패턴의 기원은 소프트웨어 공학이 아닌 **건축학**에서 시작되었다. 건축가 크리스토퍼 알렉산더(*Christopher Alexander*)는 건물과 도시 설계에서 반복적으로 나타나는 문제를 해결한 경험을 문서화했다. 그는 **특정 디자인 구조를 반복해서 사용하면 최적의 효과를 얻을 수 있다**는 사실을 깨달았다.

알렉산더는 사라 이시카와(*Sara Ishikawa*)와 머레이 실버스타인(*Murray Silverstein*)과 협력하여 **패턴 언어**(*Pattern Language*)를 탄생시켰다. 이 패턴 언어는 규모에 상관없이 설계와 구현을 원하는 모두에게 도움이 될 발명품이었으며, 1977년에 「A Pattern Language」라는 논문으로 발표되어 나중에 책으로 출간되었다.

### 소프트웨어 공학으로의 전파

1990년 무렵, 소프트웨어 엔지니어들은 초보자를 위한 디자인 패턴 문서의 첫 장에 알렉산더의 원칙들을 담기 시작했다. 디자인 패턴에 대한 개념은 아직 체계화된 형태를 갖추지 못했지만, 개발 업계의 초창기부터 사용되어 왔다.

1995년, 에리히 감마(*Erich Gamma*), 리차드 헬름(*Richard Helm*), 랄프 존슨(*Ralph Johnson*), 존 블리시드(*John Vlissides*)가 저술한 **『GoF의 디자인 패턴』**이 출간되었다. 이 4명은 현재까지도 **GoF**(*Gang of Four*)라고 불리며, 이들의 저서는 소프트웨어 엔지니어링 분야에서 가장 상징적인 저서로 꼽힌다.

GoF의 저서는 여러 개발 기법과 유의점을 설명하며, 오늘날 전 세계적으로 자주 사용되는 **23가지 핵심 객체 지향 디자인 패턴**을 소개한다. 이 패턴들은 이 책의 6장에서 카테고리별로 분류하고, 7~9장에서 자바스크립트로 구현한다.

> **핵심 통찰**: 디자인 패턴은 특정 언어나 프레임워크에 종속되지 않는 **범용적 설계 원칙**이다. 건축학에서 시작되어 소프트웨어 공학으로 전파된 만큼, 그 본질은 "반복되는 문제에 대한 검증된 해결책"에 있다.

## 2. 패턴이란 무엇인가

패턴(*Pattern*)이란 소프트웨어 설계에서 **반복되는 문제와 주제에 적용할 수 있는 재사용 가능한 템플릿**을 뜻한다. 자바스크립트로 웹 애플리케이션을 개발할 때 여러 상황에서 유용하게 사용할 수 있다.

### 디자인 패턴의 핵심 특성

| 특성 | 설명 |
|------|------|
| **검증됨** | 앞서간 개발자들의 경험과 통찰의 산물이며, 오랜 시간 동안 검증된 효과적인 접근 방식이다 |
| **재사용 가능** | 독창적인 솔루션을 제공하며 사용자의 요구에 맞춰 적용할 수 있다 |
| **알아보기 쉬움** | 정해진 구조와 공통 표현(*vocabulary*)을 사용하여 광범위한 문제에 정교한 솔루션을 제공한다 |

### 디자인 패턴의 부가적 이점

- **큰 문제 방지**: 정해진 패턴을 사용하면 코드 구조가 잘못될 염려를 덜고 개발에 집중할 수 있다. 구조적이고 체계적인 코드를 자연스럽게 작성하여 리팩터링 상황을 줄인다.
- **종합적 해결책**: 애플리케이션의 형태나 언어에 상관없이 적용해 코드 구조를 개선할 수 있다.
- **코드 양 감소**: 일반화된 함수로 비슷한 작업을 수행하는 여러 함수를 줄여 DRY(*Don't Repeat Yourself*) 원칙을 실현한다.
- **의사소통 원활**: 팀원과의 소통, 커뮤니티 토론, 유지보수 시 공통된 어휘로 이야기할 수 있다.
- **커뮤니티 선순환**: 기존 패턴이 개선되거나 새로운 패턴이 만들어지는 선순환 구조를 통해 패턴 기반 솔루션이 점점 더 견고해진다.

> **Osmani의 조언**: 패턴은 완벽한 해결책이 아니다. 패턴은 단지 체계화된 방법을 제시할 뿐이며, 설계에 관한 모든 문제를 해결해주지 않는다. 좋은 패턴을 선택하려면 여전히 좋은 설계자가 필요하다.

## 3. 현대 자바스크립트에서 디자인 패턴의 위치

자바스크립트는 서버와 클라이언트 모든 곳에서 활용되는 최신 웹 개발의 중심이다. 지난 수년간 자바스크립트는 기능과 문법 면에서 눈부신 발전을 보여주었다.

- **모듈**(*module*), **클래스**(*class*), **화살표 함수**(*arrow function*), **템플릿 리터럴**(*template literal*) 등 이전에 없던 기능 지원
- 리액트, Vue.js, 앵귤러 같은 라이브러리와 프레임워크의 등장

디자인 패턴은 어딘가에 얽매이거나 어느 한 언어에 국한되지 않는다. 사용하기 적합한 상황이라고 해서 반드시 적용할 필요는 없다. 최신 프레임워크나 라이브러리가 이미 잘 설계되어 디자인 패턴이 필요하지 않을 수도 있고, 반대로 어떤 프레임워크에서는 특정 패턴을 사용해야만 할 수도 있다.

> **핵심 통찰**: 패턴의 가치는 "외워서 적용하는 것"이 아니라 **무의식적으로 작성하던 코드 패턴에 이름을 붙이고, 그 특징을 명확히 이해하는 것**에 있다. "이러한 패턴을 사용했습니다"라는 한 마디가 코드의 가독성과 유지보수 의도를 간결하게 전달한다.

## 4. 디자인 패턴의 일상 활용 사례

### 공급자 패턴 (Provider Pattern)

리액트를 사용해본 적이 있다면 **공급자 패턴**(*Provider Pattern*)을 접해보았을 것이다. 웹 애플리케이션의 컴포넌트 트리는 사용자 정보나 접근 권한 같은 인증 관련 데이터를 공유해야 하는 경우가 많다.

**문제 — 프롭 드릴링**:

기존 자바스크립트에서는 루트 컴포넌트에 데이터를 저장한 뒤 상위에서 하위 컴포넌트로 전달했다. 컴포넌트 계층 구조가 깊어지면 데이터는 안쪽 끝까지 들어가게 되어 **프롭 드릴링**(*Prop Drilling — 부모 컴포넌트로부터 자식 컴포넌트에 이르기까지 여러 계층을 통해 prop을 전달하는 과정*)을 초래한다.

```tsx
// 프롭 드릴링의 문제
interface User {
  name: string;
  role: string;
}

function App() {
  const [user, setUser] = useState<User | null>(null);
  return <Dashboard user={user} />;
}

function Dashboard({ user }: { user: User | null }) {
  return <Settings user={user} />;
}

function Settings({ user }: { user: User | null }) {
  return <Profile user={user} />;
}

function Profile({ user }: { user: User | null }) {
  return <span>{user?.name}</span>;
}
```

**해결 — Context API**:

리액트의 Context API를 사용하면 **컨텍스트 공급자**(*Context Provider*)를 통해 상태나 데이터를 여러 컴포넌트로 한 번에 전달할 수 있다. 공유되는 데이터가 필요한 하위 컴포넌트는 `useContext` Hook을 사용하여 공급자의 데이터에 접근한다.

```tsx
// 공급자 패턴으로 해결
interface User {
  name: string;
  role: string;
}

const UserContext = createContext<User | null>(null);

function App() {
  const [user, setUser] = useState<User | null>(null);
  return (
    <UserContext.Provider value={user}>
      <Dashboard />
    </UserContext.Provider>
  );
}

function Profile() {
  const user = useContext(UserContext);
  return <span>{user?.name}</span>;
}
```

공급자 패턴은 흔히 볼 수 있는 문제를 해결하는 디자인 패턴의 훌륭한 예시다.

## 5. 이 책의 구성과 범위

이 책은 15개의 장으로 구성되어 있으며, 순서대로 자바스크립트의 최신 패턴, 추가된 기능, 리액트 특화 패턴을 다룬다. 각 장은 이전 장을 기반으로 하고 있으므로 지식을 점차 확장해 나가는 효율적인 구성을 따르고 있다.

| 파트 | 범위 | 챕터 |
|------|------|------|
| **기초** | 패턴 이론, 구조, 안티패턴 | Ch1 ~ Ch4 |
| **언어** | 최신 JS 문법, 패턴 분류 | Ch5 ~ Ch6 |
| **전통 패턴** | 생성/구조/행위 패턴 20가지+ | Ch7 |
| **아키텍처** | MV* 패턴, 비동기, 모듈 | Ch8 ~ Ch11 |
| **리액트** | 리액트 패턴, 렌더링, 구조 | Ch12 ~ Ch14 |

> 이 책은 단순히 패턴을 소개하는 것에 그치지 않고, 코드의 구성, 성능, 렌더링처럼 좋은 웹 애플리케이션을 개발할 때 필요한 주제를 함께 다룬다.

## 최신 업데이트 (2026)

원서 출판(2023) 이후 자바스크립트와 프론트엔드 생태계에 몇 가지 중요한 변화가 있었다.

| 영역 | 변화 |
|------|------|
| **React 19** | `use()` Hook, Server Actions, React Compiler, `useOptimistic`, `useFormStatus` 등 |
| **Next.js 15** | App Router 안정화, Partial Prerendering, Server Actions 기본 통합 |
| **ECMAScript** | Decorators (Stage 3→표준화), `Promise.withResolvers()`, Explicit Resource Management (`using`) |
| **빌드 도구** | Vite 6, Turbopack 안정화, Rspack 등장 |
| **상태 관리** | Zustand, Jotai 등 경량 라이브러리가 Redux를 대체하는 추세 |
| **렌더링** | React Server Components 안정화, Islands Architecture 확산, View Transitions API |

이 책의 노트에서는 원서의 내용을 기반으로 하되, 위의 최신 변화를 반영하여 보충 설명을 추가한다.

## 요약

- 디자인 패턴은 **건축학의 크리스토퍼 알렉산더**에서 시작되어 **GoF의 23가지 패턴**으로 소프트웨어 공학에 체계화되었다
- 패턴은 **반복되는 문제에 대한 검증된, 재사용 가능한 해결책**이다
- 패턴의 가치는 코드 품질 향상뿐 아니라 **개발자 간 의사소통 도구**로서도 중요하다
- 현대 자바스크립트(ES2022+)와 프레임워크(React 19, Next.js 15) 환경에서도 디자인 패턴의 원칙은 여전히 유효하다
- 패턴은 만능이 아니다. 적합한 상황에서 적절히 사용해야 하며, 프레임워크가 이미 제공하는 솔루션과 중복될 수 있다

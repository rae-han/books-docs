# Section 3.4: Concurrency: Time Is of the Essence (동시성: 시간이 본질이다)

## 핵심 질문

대입이 도입한 "시간"의 문제는 여러 프로세스가 **동시에** 실행될 때 어떻게 폭발하며, 이를 어떻게 제어하는가?

---

## 핵심 개념

| 개념 | 정의 |
|------|------|
| **동시성(Concurrency)** | 여러 프로세스가 동시에 실행되는 상황 |
| **경쟁 조건(Race condition)** | 실행 순서에 따라 결과가 달라지는 버그 |
| **직렬화(Serialization)** | 특정 함수들이 동시에 실행되지 않도록 보장하는 기법 |
| **뮤텍스(Mutex)** | 상호 배제를 구현하는 동기화 원시 수단 |
| **교착 상태(Deadlock)** | 두 프로세스가 서로의 자원을 기다리며 영원히 멈추는 상태 |

---

## 1. 동시성과 공유 상태의 문제

### 1.1 은행 계좌의 동시 접근

두 프로세스가 동시에 같은 은행 계좌에서 출금한다:

```javascript
// Peter: withdraw(W, 25)   (잔고 100에서 25 출금)
// Paul:  withdraw(W, 10)   (잔고 100에서 10 출금)
```

순차 실행이면 결과는 항상 65. 하지만 동시 실행에서 `balance`의 읽기-수정-쓰기가 인터리빙(*Interleaving — 두 프로세스의 연산이 시간적으로 교차하여 실행되는 것*)되면:

| Peter | Paul | balance |
|-------|------|---------|
| 읽기: 100 | | 100 |
| | 읽기: 100 | 100 |
| 쓰기: 75 | | 75 |
| | 쓰기: 90 | **90** (오류!) |

정확한 답은 65여야 하지만 90이 될 수 있다. Peter의 출금이 "사라졌다".

### 1.2 문제의 본질

대입이 있는 시스템에서 동시성은 **비결정적(nondeterministic)** 동작을 만든다. 같은 프로그램이 실행할 때마다 다른 결과를 낼 수 있다. 버그가 재현 불가능할 수 있다.

---

## 2. 직렬화 (Serialization)

### 2.1 직렬화기 (Serializer)

직렬화기는 함수 집합 중 하나만 동시에 실행되도록 보장한다:

```javascript
const s = make_serializer();
const protected_withdraw = s(withdraw);
const protected_deposit = s(deposit);
```

`s`로 보호된 함수들은 절대 동시에 실행되지 않는다. 한 함수가 완료될 때까지 다른 함수는 대기한다.

### 2.2 구현: 뮤텍스

직렬화기는 **뮤텍스(mutex)** 로 구현된다:

```javascript
function make_serializer() {
    const mutex = make_mutex();
    return f => {
               return (...args) => {
                          mutex("acquire");
                          const result = f(...args);
                          mutex("release");
                          return result;
                      };
           };
}
```

뮤텍스는 `acquire`(잠금)과 `release`(해제) 두 연산을 제공하며, 이미 잠긴 뮤텍스를 `acquire`하려는 프로세스는 해제될 때까지 대기한다.

---

## 3. 동시성의 난제

### 3.1 교착 상태 (Deadlock)

Peter가 계좌 A에서 B로, Paul이 계좌 B에서 A로 동시에 이체:

```
Peter: A 잠금 → B 잠금 시도 (대기)
Paul:  B 잠금 → A 잠금 시도 (대기)
→ 영원히 대기 — 교착 상태!
```

교착 상태를 피하는 고전적 방법: 자원에 순서를 부여하고, 항상 낮은 번호부터 잠금.

### 3.2 근본적 어려움

> **핵심 통찰**: SICP는 동시성 문제가 대입의 필연적 귀결이라고 강조한다. 대입이 "시간"을 도입하고, 동시성이 "시간의 비결정성"을 도입한다. 함수형 프로그래밍이 대입을 피하는 이유 중 하나가 바로 동시성에서의 안전성이다. Section 3.5의 스트림은 이 문제에 대한 우아한(하지만 완전하지는 않은) 대안을 제시한다.

---

## 요약

- **동시성 + 공유 변이 데이터**는 경쟁 조건을 일으킨다. 실행 순서에 따라 결과가 달라진다.
- **직렬화**는 특정 함수들이 동시에 실행되지 않도록 보장하여 경쟁 조건을 방지한다.
- **뮤텍스**는 직렬화의 기본 구현 수단이다.
- **교착 상태**는 직렬화의 부작용으로, 자원 순서 지정 등으로 방지한다.
- 동시성 문제는 대입의 필연적 귀결이며, 함수형 프로그래밍이 이를 피하는 동기 중 하나다.

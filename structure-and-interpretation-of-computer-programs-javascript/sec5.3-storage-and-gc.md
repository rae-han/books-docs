# Section 5.3: Storage Allocation and Garbage Collection (저장소 할당과 가비지 컬렉션)

## 핵심 질문

리스트, 쌍, 환경 같은 추상적 데이터 구조를 **유한한 메모리 위에서** 어떻게 구현하며, 더 이상 사용되지 않는 메모리는 어떻게 자동으로 회수하는가?

---

## 핵심 개념

| 개념 | 정의 |
|------|------|
| **벡터(Vector)** | 인덱스로 접근 가능한 연속 메모리 공간 |
| **가비지 컬렉션(Garbage collection, GC)** | 더 이상 접근 불가능한 메모리를 자동으로 회수하는 기법 |
| **stop-and-copy GC** | 메모리를 반으로 나누어 살아있는 객체를 복사하는 GC 알고리즘 |
| **루트 집합(Root set)** | GC의 시작점 — 레지스터와 스택에서 직접 참조하는 객체들 |

---

## 1. 메모리를 벡터로 표현

### 1.1 쌍의 구현

`pair(1, 2)`를 두 개의 벡터(the_heads, the_tails)에 저장:

```
인덱스:     0    1    2    3    ...
the_heads: [1]  [...]  ...
the_tails: [2]  [...]  ...
```

`pair`는 인덱스(포인터)를 반환하고, `head`와 `tail`은 벡터에서 값을 읽는다.

### 1.2 리스트의 메모리 표현

`list(1, 2, 3, 4)` = `pair(1, pair(2, pair(3, pair(4, null))))`

```
인덱스:     0    1    2    3
the_heads: [1]  [2]  [3]  [4]
the_tails: [p1] [p2] [p3] [null]
```

각 `tail`이 다음 쌍의 인덱스를 가리킨다.

---

## 2. 가비지 컬렉션 (Garbage Collection)

### 2.1 필요성

`pair`가 호출될 때마다 메모리를 소비한다. 무한히 `pair`를 만들 수는 없으므로, 더 이상 접근 불가능한 쌍의 메모리를 회수해야 한다.

### 2.2 Stop-and-Copy 알고리즘

메모리를 두 반으로 나눈다: **작업 메모리(working memory)** 와 **여유 메모리(free memory)**.

1. 작업 메모리가 가득 차면 **중지(stop)**
2. 루트 집합(레지스터, 스택)에서 도달 가능한 모든 객체를 **여유 메모리로 복사(copy)**
3. 두 메모리의 역할을 **교환(flip)**
4. 도달 불가능한 객체는 복사되지 않으므로 자동으로 회수됨

```
[작업 메모리: A B C D E F]  →  Stop!
         ↓  도달 가능: A, C, E만
[여유 메모리: A C E _ _ _]  →  이것이 새 작업 메모리
```

### 2.3 루트 집합과 도달 가능성

"도달 가능(reachable)"의 정의: 레지스터나 스택에서 포인터 체인을 따라갈 수 있는 객체. 도달 불가능한 객체는 프로그램이 절대 접근할 수 없으므로 안전하게 회수된다.

> **핵심 통찰**: 가비지 컬렉션은 "더 이상 사용되지 않는 메모리"를 정확히 판별하는 문제다. 이론적으로는 불가능(halting problem과 동치)하지만, "도달 불가능하면 사용되지 않는다"는 보수적 근사는 안전하고 실용적이다. 현대 언어(JavaScript, Java, Python 등)의 메모리 관리가 이 원리에 기반한다.

---

## 요약

- 리스트와 쌍은 **벡터(배열)** 로 구현된다. 포인터(인덱스)를 통해 쌍을 연결한다.
- **가비지 컬렉션**은 도달 불가능한 메모리를 자동으로 회수한다.
- **Stop-and-copy** 알고리즘은 살아있는 객체만 복사하여 메모리를 정리한다.
- GC는 현대 프로그래밍 언어의 메모리 관리 기초이며, 프로그래머를 수동 메모리 관리에서 해방한다.

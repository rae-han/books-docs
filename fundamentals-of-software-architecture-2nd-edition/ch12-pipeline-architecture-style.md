# Chapter 12: Pipeline Architecture Style (파이프라인 아키텍처 스타일)

## 핵심 질문

'파이프와 필터'란 무엇이며, 유닉스 셸이 왜 대표적인 예인가? 필터의 네 유형은 무엇인가? 왜 파이프는 단방향이어야 하고 필터에 책임을 몰면 안 되는가(거버넌스)? 언제 이 스타일을 고르고, 언제 이벤트 주도로 가야 하는가?

---

## 1. 개요

파이프라인(파이프와 필터, pipe and filter) 아키텍처는 기능성을 여러 부분으로 나누는 기본 스타일이다. **유닉스 터미널 셸**(bash·zsh의 `|`), **MapReduce**, **ETL** 도구가 대표적이다.

---

## 2. 토폴로지 — 파이프와 필터

- **필터(filter)**: 자기완결적 기능 조각. 보통 **무상태**이며 **한 가지 작업**만 한다(복합 작업은 여러 필터로).
- **파이프(pipe)**: 필터 사이의 **단방향** 통신 경로(점대점, 소량 데이터).

```
[생산자] ─파이프→ [변환기] ─파이프→ [테스터] ─파이프→ [소비자]
```

**필터의 네 유형**:
- **생산자(producer)**: 시작점(소스). 데이터를 내보내기만 한다.
- **변환기(transformer)**: 입력을 변환해 출력(함수형의 `map`).
- **테스터(tester)**: 기준에 따라 검사해 선택적으로 출력(함수형의 `reduce` 유사; "5달러 미만이면 넘기지 않음").
- **소비자(consumer)**: 종료점. DB 저장·화면 출력.

> **핵심 통찰**: 단방향·조합 구조 덕에 **조합적 재사용(compositional reuse)**이 강력하다. 커누스가 "텍스트에서 가장 빈번한 단어 K개" 문제를 **10쪽 넘는 파스칼 코드**로 푼 반면, 매킬로이는 `tr | tr | sort | uniq -c | sort -rn | sed`라는 **한 줄 셸 파이프라인**으로 우아하게 풀었다.

---

## 3. 데이터 · 클라우드 · 거버넌스

- **데이터 토폴로지**: 단일 DB부터 필터당 하나까지 다양.
- **클라우드**: 필터를 **분산 함수**로 배포할 수 있다 — 예: AWS Step Functions에서 각 필터를 개별 람다로(스탠더드/익스프레스 작업흐름).
- **거버넌스**: 적합성 함수로 필터 유형을 자동 검증하긴 어려우니, **태그(자바 애너테이션·C# 속성)**로 각 필터의 유형(`PRODUCER`·`TESTER`…)과 진입점을 표시해 개발자가 책임을 넘기지 않게 돕는다.

> **핵심 통찰 — 흔한 위험**: (1) 필터에 **책임 과다** → 비구조적 모놀리스로 변질, (2) **양방향 통신** → 파이프는 단방향이어야 한다(필요하면 스타일이 안 맞거나 필터가 너무 복잡하다는 신호), (3) 파이프라인 도중 **오류 조건** 처리·복구, (4) 필터 간 **계약 관리**.

---

## 4. 왜 · 언제 사용하고, 언제 피하는가

- **적합**: 처리 단계가 명확히 분리되고 **고정된 순서**가 있으며 **단방향**인 **결정론적 작업흐름**(EDI 변환, ETL, Apache Camel 같은 중재자). 단순해서 시간·예산이 빠듯할 때도 좋다.
- **피함**: 높은 확장성·탄력성·내결함성이 필요할 때(모놀리스). 필터 간 **양방향 통신**이나 **비결정론적 작업흐름**이 필요하면 → **이벤트 주도 아키텍처(제15장)**가 낫다. (운영 특성은 비동기 분산으로 구현하면 개선되지만 비용·단순성을 대가로 치른다.)

---

## 5. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 기술적 |
| 퀀텀 수 | 1 |
| 전반적인 비용 | ★★★★★ |
| 단순성 | ★★★★★ |
| 모듈성 | ★★★★☆ |
| 배포성 | ★★★☆☆ |
| 테스트성 | ★★★☆☆ |
| 유지보수성 | ★★★☆☆ |
| 진화성 | ★★★☆☆ |
| 반응성 | ★★★☆☆ |
| 확장성 | ★☆☆☆☆ |
| 탄력성 | ★☆☆☆☆ |
| 내결함성 | ★☆☆☆☆ |

- **강점은 비용·단순성·모듈성** — 모놀리스라 분산 복잡성이 없고, 필터 유형별 관심사 분리로 어떤 필터든 다른 필터에 영향 없이 교체 가능.
- **배포성·테스트성은 평균**(계층형보다 약간 높음 — 필터의 높은 모듈성 덕).
- **탄력성·확장성·내결함성 ★1** — 모놀리스 배포 때문. (비동기 분산으로 개선 가능하나 비용·단순성 하락.)

> **참고**: 별점은 OCR 격자표 훼손으로 본문 서술(강점 3종, 배포성·테스트성 "평균, 계층형보다 약간↑", 탄력성·확장성·내결함성 "별 1개")에 근거해 재구성했다.

---

## 요약

- **파이프(단방향)와 필터(무상태·단일 작업)** — 유닉스 셸·MapReduce·ETL이 예. 조합적 재사용이 강력(매킬로이의 한 줄 파이프라인)
- 필터 네 유형: **생산자·변환기(map)·테스터(reduce)·소비자**
- 위험: 필터 책임 과다·양방향 통신·오류 조건·계약 — **태그**로 필터 유형을 거버넌스
- 강점 **비용·단순성·모듈성**, 약점 **확장성·탄력성·내결함성**(모놀리스) — 결정론적 단방향 흐름에 적합하고, 비결정론적/양방향이면 이벤트 주도로

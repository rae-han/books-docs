# Chapter 3: Modularity (모듈성)

## 핵심 질문

모듈성과 세분도(granularity)는 어떻게 다르며, 왜 세분도가 안티패턴의 원인인가? 응집·결합을 어떻게 측정하는가? 추상도·불안정도로부터 나오는 '주 시퀀스 거리'는 무엇을 말해주는가? **동변성(connascence)**이란 무엇이고, 왜 결합을 서술하는 더 정확한 언어인가?

---

## 1. 모듈성 vs 세분도

> [소프트웨어 아키텍처] 문헌의 95%는 '모듈성'의 장점을 찬양하기만 할 뿐, 실제로 그것을 어떻게 달성할지에 관해서는 거의 언급하지 않는다.<br>— 글렌포드 J. 마이어스(Glenford J. Myers)

모듈성은 하나의 **조직화 원칙**이다. 물리학의 복잡계처럼 소프트웨어도 방치하면 엔트로피가 높아지므로(무질서를 향함), 구조적 건전성을 유지하려면 **아키텍트가 지속적으로 에너지를 투입**해야 한다. 모듈성은 요구사항에 명시되는 일이 드문 **암묵적(implicit) 아키텍처 특성**이지만, 지속 가능한 코드베이스에 꼭 필요하다.

- **모듈성**: 시스템을 더 작은 조각으로 **나누는 것**.
- **세분도(granularity)**: 그 조각들의 **크기**.

> **핵심 통찰**: 문제는 대개 **세분도**에서 생긴다. 세분도가 서비스·컴포넌트를 결합하게 만들어 스파게티 아키텍처·분산 모놀리스·**분산 진흙잡탕(Distributed Big Ball of Mud)** 같은 안티패턴을 낳는다. "**모듈성을 받아들이되 세분도를 조심하라**"(마크 리처즈) — 세분도와 결합 수준에 늘 주의를 기울여야 한다.

---

## 2. 모듈성이란

이 책은 모듈성을 **서로 연관된 코드를 논리적으로 묶는 활동**으로 정의한다(객체 지향이면 클래스 그룹, 구조적/함수형이면 함수 그룹). 이는 **물리적 분리가 아니라 논리적 분리**를 뜻한다 — 모놀리스에서 클래스를 느슨하게 묶어 두면 편리하지만, 나중에 재구성할 때 그 느슨함이 조장한 결합도 때문에 분해가 어려워진다. 그래서 물리적 분리와 별개로 모듈성을 논하는 것이 유용하다.

---

## 3. 응집 (Cohesion)

응집은 **하나의 모듈을 구성하는 요소들이 정말로 그 안에 있어야 하는가**를 측정한다. 유형을 **최선 → 최악** 순으로:

1. **기능적 응집(functional)**: 모듈의 모든 요소가 관련되고, 기능에 필요한 모든 것이 들어 있다. (최선)
2. **순차적 응집(sequential)**: 한 모듈의 출력이 다른 모듈의 입력이 된다.
3. **통신적 응집(communicational)**: 두 모듈이 통신 체인을 이뤄 각자 처리·기여한다.
4. **절차적 응집(procedural)**: 두 모듈이 반드시 특정 순서로 실행된다.
5. **시간적 응집(temporal)**: 시간적 의존(예: 시작 시점 초기화)으로만 연관된다.
6. **논리적 응집(logical)**: 논리적으로는 관련되나 기능적으로는 다르다(예: `StringUtils` — String을 다룬다는 것 외엔 무관한 정적 메서드 모음).
7. **우발적 응집(coincidental)**: 그저 같은 소스 파일에 있을 뿐이다. (최악)

> **핵심 통찰**: "응집력 있는 모듈을 나누려고 해봤자 결합도가 증가하고 가독성이 떨어질 뿐이다"(래리 콘스탄틴). 응집의 적정선은 트레이드오프 분석의 문제다 — "이 연산이 이 모듈에 꼭 필요한가?"의 답은 늘 "**상황에 따라 다르다**". 구조적 지표 **LCOM(메서드 응집 결여도)**은 우발적으로 결합된 클래스를 찾아주지만, **구조적** 응집 결여만 잡을 뿐 **논리적** 적합성은 알려주지 못한다(→ 제2법칙: '왜'가 중요).

## 4. 결합 (Coupling)

호출 그래프를 그래프 이론으로 분석한 두 핵심 지표:
- **구심 결합도(afferent, 들어오는)**: 해당 요소로 **들어오는** 연결 수.
- **원심 결합도(efferent, 나가는)**: 다른 요소로 **나가는** 연결 수.

여기서 로버트 C. 마틴의 파생 지표가 나온다.
- **추상도(abstractness)** = 추상 요소 / (추상 + 구체 요소). 코드가 얼마나 추상적인지(0=거대한 `main()` 하나, 1=추상 투성이).
- **불안정도(instability)** = 원심 / (원심 + 구심). 코드베이스의 변동성 — 높으면 결합이 많아 변경 시 깨지기 쉽다.

### 4.1 주 시퀀스로부터의 거리

**주 시퀀스 거리** `D = |추상도 + 불안정도 − 1|`는 클래스가 추상성·불안정성의 **이상적 균형선**에서 얼마나 떨어졌는지를 나타낸다.

```
추상도 1 ┌─────────────────┐
        │ ＼   무용 지역    │  ← 너무 추상적 → 쓰기 어려워 쓸모없음
        │    ＼            │
        │   주 ＼시퀀스     │  ← 이상적 균형선 (D=0)
        │        ＼        │
        │ 고통 지역  ＼     │  ← 너무 구체적 → 취약·유지보수 고통
        └─────────────────┘ 불안정도 1
```

- **무용 지역(Zone of Uselessness)**: 추상화 과다 → 사용하기 어려워 쓸모없다.
- **고통 지역(Zone of Pain)**: 구현 과다·추상화 부족 → 취약하고 유지보수가 고통스럽다.

> **참고**: 이런 코드 수준 지표는 다른 공학 분야에 비하면 무디고 **해석이 필요**하다. 예컨대 순환 복잡도는 본질적 복잡성과 우발적 복잡성을 구별하지 못한다. 그래도 기준선을 세워 두면 코드베이스를 평가하는 데 유용하다.

## 5. 동변성 (Connascence)

메일리어 페이지존스가 제시한 **동변성**은 결합의 지표가 아니라, **다양한 유형의 결합을 더 정확하게 서술하는 언어**다. 두 컴포넌트가 **동변적(connascent)**이라는 것은 — 한쪽이 변하면 전체 정확성을 위해 다른 쪽도 변해야 한다는 뜻이다.

**정적 동변성(static, 소스 코드 수준)** — 약함 → 강함:
- **이름 동변성**: 엔터티 **이름**에 합의(가장 흔하고 바람직 — 리팩터링 도구로 쉽게 변경).
- **타입 동변성**: 엔터티 **타입**에 합의.
- **의미 동변성**: 값의 **의미**에 합의(규약 동변성). 예: 하드코딩된 `1`/`0`.
- **위치 동변성**: 값들의 **순서**에 합의. 예: `updateSeat(name, seat)`에 인수를 뒤바꿔 전달.
- **알고리즘 동변성**: 특정 **알고리즘**에 합의. 예: 서버·클라이언트의 해싱 알고리즘.

**동적 동변성(dynamic, 실행 시점)** — 정적보다 강함(나쁨):
- **실행 동변성**: 실행 **순서**가 중요(예: 이메일 발송 전에 수신자 설정).
- **타이밍 동변성**: 실행 **타이밍**이 중요(예: 경합 조건).
- **값 동변성**: 여러 값이 **함께** 변해야 함(예: 사각형 꼭짓점, 분산 트랜잭션).
- **신원 동변성**: 동일한 **엔티티**를 참조해야 함(예: 공유 분산 대기열).

### 5.1 동변성의 세 속성

- **강도(strength)**: 리팩터링하기 쉬운 정도. **정적 동변성을 동적보다 선호**하라 — 예: 마법의 값을 상수로 바꿔 **의미 동변성 → 이름 동변성**으로 개선.
- **지역성(locality)**: 모듈들이 얼마나 가까운지. 같은 모듈 안의 강한 동변성은 멀리 떨어진 것보다 덜 해롭다(DDD의 **경계 컨텍스트**와 통하는 개념).
- **정도(degree)**: 변경이 영향을 주는 클래스 수. 코드베이스가 커질수록 작았던 문제가 커진다.

> **핵심 통찰**: 짐 웨이리치의 두 규칙 — (1) **정도의 규칙**: 강한 동변성을 더 약한 동변성으로 변환하라. (2) **지역성의 규칙**: 요소 사이 거리가 멀수록 더 약한 동변성을 적용하라. 동변성은 설계 패턴처럼 **공통 개념에 이름을 붙여** 소통을 정밀하게 한다 — "상수로 추출하세요" 대신 "의미 동변성을 이름 동변성으로 바꾸세요".

## 요약

- **모듈성**(쪼개기)과 **세분도**(크기)는 다르다 — 안티패턴은 대개 세분도에서 온다("모듈성은 받아들이되 세분도를 조심하라")
- 모듈성은 **논리적 분리** — 물리적 분리와 별개로, 지속 가능성을 위한 암묵적 특성
- **응집**은 최선(기능적)~최악(우발적) 7단계 — LCOM은 구조적 결여만 잡고 논리적 적합성은 못 본다(제2법칙)
- **결합**: 구심(들어오는)·원심(나가는) → 추상도·불안정도 → **주 시퀀스 거리**(무용 지역=과추상, 고통 지역=과구현)
- **동변성**은 결합을 서술하는 언어 — 정적(이름·타입·의미·위치·알고리즘)이 동적(실행·타이밍·값·신원)보다 약하다(좋다). **강한→약한, 멀수록 약하게**(웨이리치 규칙)

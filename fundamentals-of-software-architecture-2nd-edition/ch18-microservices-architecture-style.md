# Chapter 18: Microservices Architecture Style (마이크로서비스 아키텍처 스타일)

## 핵심 질문

왜 마이크로서비스는 '무공유(share nothing)' 아키텍처라 불리며 재사용보다 **중복**을 택하는가? 경계 컨텍스트가 왜 이 스타일의 심장인가? 서비스 세분도를 잘못 잡으면(모래알·분산된 진흙잡탕) 무슨 일이 생기는가? 서비스 간 트랜잭션은 왜 피해야 하고, 불가피하면 사가로 어떻게 다루는가? 사이드카·서비스 메시는 '무공유'와 어떻게 공존하는가?

---

## 1. 개요 — DDD 경계 컨텍스트의 물리적 구현

마이크로서비스는 마틴 파울러·제임스 루이스의 2014년 블로그로 이름이 굳어진, **DDD의 경계 컨텍스트(bounded context)**에서 영감을 받은 스타일이다. 경계 컨텍스트는 **분리(decoupling) 스타일** — 한 컨텍스트 내부(코드·스키마)는 결합해도 되지만, **다른 컨텍스트의 외부 요소와는 절대 결합하지 않는다**. 그래서 '무공유' 아키텍처라 불린다.

> **핵심 통찰**: **재사용은 결합을 통해 구현된다**(제1법칙). 고도로 분리된 시스템이 목표인 마이크로서비스는 **재사용보다 중복(duplication)을 선호**한다 — 모놀리스라면 공유했을 `Address` 같은 공통 클래스도 각 경계 컨텍스트 안에 **중복**해 가둔다. 마이크로서비스는 **DDD의 논리적 개념을 물리적으로 구현**한 것이다.

## 2. 토폴로지

```
 [클라이언트]  [클라이언트]  [클라이언트]
        └──────────┼──────────┘
                [API 계층]         ← 라우팅·보안 등 횡단 관심사만 (중재자 X)
     ┌──────┬──────┼──────┬──────┐
  [서비스] [서비스] [서비스] [서비스] [서비스]  ← 단일 목적, 독립 배포
    │        │       │       │       │
   [DB]     [DB]    [DB]    [DB]    [DB]   ← 서비스당 데이터베이스
```

각 서비스는 **독립 작동에 필요한 모든 것(DB·의존요소 포함)**을 담고 **VM/컨테이너의 개별 프로세스**로 실행된다. 다른 분산 스타일(SOA·EDA·서비스 기반)보다 **훨씬 작다**(단일 목적). 분산이라 **성능**(네트워크 호출·종단점 보안 검사)이 단점이며, **서비스 경계를 넘는 트랜잭션을 쓰지 말라**는 것이 숙련 아키텍트의 조언이다 — 세분도 결정이 성패를 가른다.

---

## 3. 경계 컨텍스트 · 세분도 · 데이터 격리

- **경계 컨텍스트**: 각 서비스는 하나의 기능·하위 도메인·작업흐름을 모델링하고, 그에 필요한 모든 것(컴포넌트·스키마·DB)을 담는다.
- **세분도(granularity)** — 가장 어려운 문제:

> **핵심 통찰**: "**마이크로서비스는 이름표(label)이지 설명(description)이 아니다**"(마틴 파울러). '마이크로'를 글자 그대로 받아들여 너무 잘게 쪼개면, 유용한 일을 하려 서비스 간 통신을 다시 잇게 되고 결국 **분산된 진흙잡탕(Big Ball of Distributed Mud)**이 된다. 세분도는 세 지침으로 잡는다 — **목적**(기능적 응집, 하나의 중요한 행동방식), **트랜잭션**(협력 엔티티가 경계를 암시하되 트랜잭션은 피하도록 설계), **코레오그래피**(통신량이 과하면 더 큰 서비스로 다시 묶어라). 좋은 세분도는 **반복(iteration)**으로만 찾는다.

- **데이터 격리**: 공유 스키마·DB를 통합 지점으로 쓰는 것도 결합이므로 금지. **엔티티 함정**(서비스를 DB 엔티티처럼 모델링)을 경계하고, 진실 공급원 도메인을 두거나 복제·캐싱으로 분산한다. 대신 팀마다 **가장 적합한 DB 기술을 자유롭게** 고르고 바꿀 수 있다(다른 팀이 구현 세부에 결합 못 하므로).

## 4. API 계층 · 운영 재사용 · 프런트엔드

- **API 계층(API 게이트웨이)**: 소비자와 서비스 사이의 계층. **라우팅·보안·모니터링·로깅 등 횡단 관심사만** 담는다.

> **핵심 통찰**: API 계층을 **중재자·오케스트레이터로 쓰면 안 된다** — 모든 흥미로운 비즈니스 로직은 경계 컨텍스트 안에 있어야 한다. 중재자는 기술 분할 아키텍처의 것이고, 마이크로서비스는 **확고히 도메인 분할**이다.

- **운영 재사용 — 사이드카 패턴**: 모니터링·로깅·서킷 브레이커처럼 **결합이 이득인 운영 관심사**는 각 서비스에 붙는 **사이드카(Sidecar) 컴포넌트**로 뽑는다(공유 인프라 팀이 사이드카만 업그레이드하면 전 서비스에 반영). 사이드카들을 **서비스 평면(service plane)**으로 연결하면 **서비스 메시(service mesh)**(Istio 등)가 되어 운영 관심사를 전역 제어한다. **서비스 발견(service discovery)**도 여기 넣어 탄력성을 확보한다.
- **프런트엔드**: **모놀리스 프런트엔드**(단일 UI) 또는 **마이크로프런트엔드**(UI를 백엔드와 같은 세분도의 컴포넌트로 분할).

## 5. 통신 · 코레오그래피 vs 오케스트레이션 · 사가

- **통신**: 동기/비동기. **프로토콜 인식 이기종 상호운용성** — 중앙 허브가 없어 각 서비스가 호출 프로토콜을 스스로 알아야 하고(**프로토콜 인식**), 서비스마다 다른 기술 스택이 가능하며(**이기종·폴리글랏**), 서로 네트워크로 호출한다(**상호운용성**). 한 아키텍트는 실수로 코드를 공유하지 못하게 **팀마다 다른 스택을 강제**하기도 했다(좁은 문제에 올바른 규모의 기술).
- **코레오그래피 vs 오케스트레이션**: 중앙 중재자가 없는 것이 기본(**코레오그래피** — 경계 컨텍스트 철학에 부합). 여러 서비스 조정이 필요하면 **지역 중재자(오케스트레이션 서비스)**를 둔다. 코레오그래피에서 한 서비스가 사실상 중재자 역할을 겸하면 **프런트 컨트롤러(Front Controller)** 패턴(복잡성 가중). 트레이드오프는 **분리·성능(코레오) vs 조정·오류 처리(오케)**.

> **핵심 통찰 — 트랜잭션과 사가**: 서비스 경계를 넘는 트랜잭션은 핵심 분리 원칙을 위반하고 최악의 **값 동변성**을 유발한다 — 조언은 "**하지 마세요!**"이며, 필요를 느끼면 **세분도가 너무 잘다는 신호**다. 불가피하면 **사가(Saga) 패턴** — 중재자가 각 부분을 호출·기록하다 하나가 실패하면 나머지에 **취소(undo)를 지시**(보상 트랜잭션 프레임워크). 단 보상 트랜잭션은 조정 트래픽을 많이 유발한다(『Software Architecture: The Hard Parts』에 8가지 사가 패턴).

## 6. 데이터 토폴로지 — 서비스당 데이터베이스

**데이터 분리가 필수인 유일한 스타일**이다. 60개 서비스가 한 DB를 공유하면 (1) 스키마 변경이 60개 서비스 배포 조율을 강제하고 (2) 확장성·연결 풀·가용성이 무너지며 (3) **경계 컨텍스트 개념 자체가 붕괴**한다.

> **핵심 통찰**: 표준은 **서비스당 데이터베이스(Database-per-Service)** — 각 서비스가 자기 데이터만 소유하고, 남의 데이터는 **계약을 통해 요청**한다(내부 구조와 결합 안 됨). 변경 영향이 소유자 서비스에만 미쳐 확장성·탄력성·가용성·내결함성이 모두 우수하다. 성능 등의 이유로 **소수(5~6개 이하)**가 DB를 공유할 수는 있으나(경계 컨텍스트를 더 넓게 잡은 것), 그 이상이면 모놀리스 DB의 문제가 다시 나타난다.

## 7. 위험 · 거버넌스 · 팀

- **위험**: (1) **모래알(Grains of Sand)** — 서비스를 너무 잘게. (2) **과도한 서비스 간 통신**(→ 더 성긴 서비스로 통합). (3) **데이터 공유 남용**. (4) **코드 재사용**(공유 라이브러리가 여러 경계 컨텍스트에 퍼지면 '무공유' 위반).
- **거버넌스**: **구조적 붕괴(structural decay)** 방지가 핵심 — **정적 결합**(공유 라이브러리·계약, SBOM·의존성 도구로 파악)과 **동적 결합**(호출 로그·레지스트리 → 적합성 함수로 서비스 간 호출 맵)을 모니터링. 정적 결합은 적을수록 좋다.
- **팀**: **도메인 정렬 교차 기능 팀**이 최적(기술 분할 팀은 부적합). 스트림 정렬 팀(도메인 정렬 시 잘 맞음)·활성화 팀·난해한 하위시스템 팀·**플랫폼 팀**(사이드카·서비스 메시의 횡단 운영 기능 담당)이 높은 모듈성 덕에 독립적으로 협업한다.

## 8. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 도메인 |
| 퀀텀 수 | 1 이상 |
| 배포성 | ★★★★★ |
| 테스트성 | ★★★★★ |
| 모듈성 | ★★★★★ |
| 확장성 | ★★★★★ |
| 탄력성 | ★★★★★ |
| 진화성 | ★★★★★ |
| 내결함성 | ★★★★★ |
| 유지보수성 | ★★★★☆ |
| 반응성 | ★★☆☆☆ |
| 전반적인 비용 | ★☆☆☆☆ |
| 단순성 | ★☆☆☆☆ |

- **여러 특성이 극과 극**이다. **배포성·테스트성**(★5)은 데브옵스 혁명 없이는 존재할 수 없었던 이 스타일의 정체성. 단일 목적·독립·세분화가 **내결함성·확장성·탄력성·진화성**(★5)을 준다. **현대 아키텍처 중 퀀텀 구분이 가장 뚜렷**(도메인 분할, 퀀텀 1 이상).
- **약점은 성능·단순성·비용**: 분산이라 네트워크 호출·보안 검사·데이터 지연이 반응성을 떨어뜨리고(그래서 캐싱·복제·코레오그래피로 완화), 가장 복잡하고 비싸다.

> **참고**: 별점은 OCR 격자표 훼손으로 본문 서술(배포성·테스트성·내결함성·확장성·탄력성·진화성이 강점, 성능·단순성·비용이 약점)에 근거해 재구성했다.

**예시**: **환자 활력 징후 모니터링 시스템** — 심박수·혈압·산소 등 각 징후를 독립 마이크로서비스(+전용 DB)로, 알림은 `Alert Staff` 공유 서비스로. 한 징후 서비스가 죽어도 나머지는 정상(**내결함성**), 혈압 서비스만 작게 테스트(**테스트성**), 새 징후를 영향 없이 추가(**진화성**).

## 요약

- 마이크로서비스는 **DDD 경계 컨텍스트의 물리적 구현** — '무공유'로 **재사용보다 중복**을 택한다
- 세분도가 성패 — 너무 잘면 **분산된 진흙잡탕**("이름표지 설명이 아니다"). 목적·트랜잭션·코레오그래피로 반복해 잡는다
- **API 계층은 횡단 관심사만**(중재자 금지), 운영 관심사는 **사이드카·서비스 메시**로 결합, 통신은 **코레오그래피 기본**
- **서비스 경계 넘는 트랜잭션은 피하고**, 불가피하면 **사가(보상 트랜잭션)**. 데이터는 **서비스당 DB**가 표준
- 강점 **배포성·테스트성·내결함성·확장성·탄력성·진화성**(★5 다수), 약점 **성능·단순성·비용** — 도메인 분할, 클라우드 네이티브

# Chapter 11: Modular Monolith Architecture Style (모듈형 모놀리스 아키텍처 스타일)

## 핵심 질문

모듈형 모놀리스는 왜 초판(2020) 이후 인기를 끌었고, 계층형 아키텍처와 무엇이 다른가? "모놀리스 구조"와 "모듈형 구조" 두 옵션의 트레이드오프는? 모듈 간 통신은 어떻게 하며 왜 최소화해야 하는가? 잘 설계된 모듈형 모놀리스가 '진흙잡탕'으로 전락하지 않게 하는 거버넌스는? 언제 이 스타일을 고르고 언제 피하는가?

---

## 1. 개요 — 도메인으로 나눈 모놀리스

모듈형 모놀리스(modular monolith)는 초판(2020) 이후 **도메인 주도 설계(DDD)의 확산과 도메인 분할에 대한 관심** 덕분에 크게 인기를 끌어, 2판에 새로 추가된 스타일이다. 이름대로 기본적으로 **모놀리스**여서 WAR·EAR 파일이나 .NET 단일 어셈블리 같은 **단일 배포 단위**로 배포된다. 단, 컴포넌트를 기술 기능이 아니라 **비즈니스 도메인**으로 묶는 **도메인 분할 아키텍처**다.

> **핵심 통찰**: 계층형(제10장)과의 차이는 **이름공간의 세 번째 노드**에서 드러난다. 계층형에서 고객 프로필 표현 로직은 `com.app.presentation.customerprofile`(세 번째 노드 = **기술** 관심사)이지만, 모듈형 모놀리스에서는 `com.app.customerprofile`(세 번째 노드 = **도메인** 관심사)이다. 기술이 아니라 도메인이 조직화의 1차 기준이다.

---

## 2. 토폴로지

기능성이 **도메인 영역별 모듈**로 그룹화되어 하나의 배포 단위 안에 담긴다. 여기서 도메인(때로는 서브도메인)을 **모듈(module)**이라 부른다.

```
┌──────────────── 단일 배포 단위 (모놀리스) ────────────────┐
│  ┌────────┐  ┌────────┐  ┌────────┐                      │
│  │ 주문접수 │  │ 결제처리 │  │  배송   │  ← 도메인 모듈     │
│  └────────┘  └────────┘  └────────┘                      │
│  ┌────────┐  ┌────────┐  ┌────────┐                      │
│  │ 재고관리 │  │ 주문이행 │  │ 고객알림 │                   │
│  └────────┘  └────────┘  └────────┘                      │
└───────────────────────────┬───────────────────────────────┘
                      ┌──────┴──────┐
                      │  데이터베이스 │
                      └─────────────┘
```

- 하나의 모듈은 **하나 이상의 논리적 컴포넌트**로 이루어진다(모듈과 컴포넌트는 1대다).
- 예: `com.orderentry.orderplacement`, `com.orderentry.inventorymanagement`, `com.orderentry.paymentprocessing` … 각 이름공간이 하나의 도메인 모듈에 대응.

---

## 3. 두 가지 구조 옵션 — 모놀리스 구조 vs 모듈형 구조

모듈을 조직화하는 방법은 크게 두 가지이며, 무엇을 고를지는 (늘 그렇듯) 트레이드오프에 달렸다.

| 구분 | 모놀리스 구조 | 모듈형 구조 |
|---|---|---|
| 형태 | 모든 모듈이 **하나의 소스 코드 저장소**에, 이름공간·디렉터리로 구분 | 각 모듈이 **독립 요소(JAR·DLL)** — 배포 시에만 단일 단위로 결합 |
| 장점 | 소스가 한곳에 있어 유지보수·테스트·배포가 용이(가장 간단) | 모듈이 완결적 → **팀별로 개별 모듈 작업**(전용 저장소 가능), 경계·관심사 분리가 명확 |
| 약점 | 개발자가 다른 모듈 클래스를 너무 쉽게 인스턴스화 → 경계 붕괴 위험 | 서로 의존하는 모듈이 통신해야 하면 **컴파일 시점 의존성**이 생겨 효과 저하 |
| 적합 | 작고 단순한 시스템 | 모듈이 대체로 독립적인, 크고 복잡한 시스템 |

> **핵심 통찰**: 모놀리스 구조는 간단하지만, 개발자가 **모듈 간에 코드를 과도하게 재사용하거나 통신을 남발**하면 잘 설계된 모듈형 모놀리스가 **구조화되지 않은 '진흙잡탕(big ball of mud)'**으로 전락한다. 그래서 이 스타일은 **엄격한 거버넌스**를 전제로만 성립한다(6절).

---

## 4. 모듈 간 통신 — 동급 간 vs 중재자

이 스타일에서 모듈 간 통신은 **바람직하지 않지만**(모듈은 독립·완결적이어야 한다), 복잡한 작업흐름에서는 필요할 때가 많다(예: 주문접수 모듈이 재고 조정을 위해 재고관리 모듈과 통신). 방식은 두 가지다.

- **동급 간(peer-to-peer)**: 한 모듈이 다른 모듈의 클래스를 직접 인스턴스화·호출. 가장 간단하지만 결합이 강하고, 모듈형 구조에서는 인터페이스 클래스를 공유하지 않으면 **JAR/DLL 지옥 안티패턴**을 부른다.
- **중재자(mediator)**: 중재자 컴포넌트가 요청을 받아 적절한 모듈로 전달하는 오케스트레이터 역할. 모듈 간 추상화 계층으로 결합도를 낮춘다.

```
          ┌───────────┐
          │   중재자    │  ← 요청을 받아 적절한 모듈로 전달
          └─────┬─────┘
      ┌─────────┼─────────┐
  ┌───┴───┐ ┌───┴───┐ ┌───┴───┐
  │주문접수 │ │결제처리 │ │재고관리 │
  └───────┘ └───────┘ └───────┘
```

> **핵심 통찰**: 중재자는 모듈 **사이의** 결합은 없애지만, 각 모듈이 이제 **중재자와 결합**한다 — 모든 결합·의존성을 없애지는 못한다(제1법칙: 트레이드오프). 그래도 아키텍처를 단순화하고 모듈 독립성을 지키는 데 도움이 된다. 인터페이스/API를 필요로 하는 쪽은 의존하는 모듈들이 아니라 **중재자**라는 점이 특징이다.

## 5. 데이터 토폴로지

단일 배포 단위이므로 보통 **모놀리스 데이터베이스**에 의존한다. 단일 DB로 모듈들이 데이터를 공유하면 모듈 간 통신이 줄어든다. 반대로 모듈들이 서로 독립적이고 특화된 기능을 한다면, 아키텍처는 모놀리스여도 **모듈마다 자체 컨텍스트 데이터베이스**를 둘 수 있다.

## 6. 위험과 거버넌스

**주요 위험**은 세 가지다.
1. **너무 커짐**: 다른 모놀리스처럼 커지면 유지보수·테스트·배포가 어려워진다(경고 신호: 변경에 오래 걸림, 한 곳 변경이 다른 곳을 깨뜨림, 팀원 간 작업 충돌, 시동 시간 지연).
2. **과도한 코드 재사용**: 모듈 경계가 모호해져 **비정형 모놀리스(unstructured monolith)** — 상호 의존이 너무 높아 분석·분해 불가한 코드 — 로 빠진다.
3. **과도한 모듈 간 통신**: 상호 통신이 너무 많다면 애초에 **도메인이 잘못 정의됐다는 신호**다. 도메인을 다시 정의해야 한다.

이 위험들 때문에 **자동화된 거버넌스**가 핵심이다. ArchUnit(자바)·ArchUnitNet/NetArchTest(.NET)·PyTestArch(파이썬)·TSArch(TS/JS) 같은 도구로 다음을 검사한다:
- 모든 소스가 **정의된 모듈 이름공간 안에** 있는가.
- 특정 모듈의 **총 의존요소 개수가 상한(예: 5개)을 넘지 않는가**(통신량 제한).
- 통신하면 안 되는 모듈 쌍(예: 주문접수 → 배송)의 **의존성이 금지되는가**.

> ```
> // ArchUnit 예 — 주문접수가 배송에 접근하지 못하게 강제
> noClasses().that().resideInAPackage("..com.orderentry.orderplacement..")
>   .should().accessClassesThat().resideInAPackage("..com.orderentry.shipping..")
>   .check(myClasses);
> ```

> **참고 — 팀 토폴로지**: 도메인 분할이므로 **도메인 정렬 팀(교차 기능 팀, CFT)** 과 가장 잘 맞는다. 스트림 정렬 팀·활성화 팀·난해한 하위시스템 팀·플랫폼 팀 모두와 정렬되기 쉽다. 반대로 UI 팀·백엔드 팀처럼 **기술별로 조직된 팀**과는 잘 맞지 않는다(도메인 요구사항 하나에 여러 기술 팀의 조율이 필요).

## 7. 왜 · 언제 사용하고, 언제 피하는가

**적합한 경우**:
- **예산·시간이 빠듯할 때, 새 시스템을 시작할 때** — 단순하고 비용이 낮다.
- 아키텍처 방향이 아직 불분명할 때: 분산 아키텍처로 바로 뛰어들기보다 **모듈형 모놀리스로 시작해 나중에 서비스 기반(제14장)·마이크로서비스(제18장)로 전환**하는 편이 효과적.
- 변경 대부분이 **도메인 기반**이고, **DDD를 쓰는 도메인 중심 팀**일 때.

**피해야 할 경우**:
- **확장성·탄력성·가용성·내결함성·반응성·성능 등 운영 특성**이 높아야 할 때(대부분의 모놀리스처럼 역부족).
- 변경 대부분이 **기술 지향**일 때(예: UI·DB 기술을 계속 교체) — 도메인 분할이라 모든 모듈에 영향이 가고 조율 비용이 크다. 이럴 땐 **계층형(제10장)**이 낫다.

## 8. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 도메인 |
| 퀀텀 수 | 1 |
| 전반적인 비용 | ★★★★☆ |
| 단순성 | ★★★★☆ |
| 모듈성 | ★★★★☆ |
| 유지보수성 | ★★★☆☆ |
| 진화성 | ★★★☆☆ |
| 배포성 | ★★☆☆☆ |
| 테스트성 | ★★☆☆☆ |
| 반응성 | ★★☆☆☆ |
| 확장성 | ★☆☆☆☆ |
| 탄력성 | ★☆☆☆☆ |
| 내결함성 | ★☆☆☆☆ |

- **강점은 전반적인 비용·단순성·모듈성**: 본질적으로 모놀리스라 분산 아키텍처의 복잡성이 없어 단순하고 저렴하다. 모듈성은 도메인·서브도메인별 관심사 분리로 확보되어 **계층형보다 높다**.
- **배포성·테스트성은 별 2개**: 모놀리스 계열이라 의례(ceremony)·위험·배포 빈도 면에서 낮지만, 높은 모듈성 덕에 계층형보다는 약간 높다.
- **탄력성·확장성은 별 1개**: 모놀리스 배포라 일부 기능만 확장하려면 이 스타일에 안 맞는 복잡한 기법(다중 스레딩·내부 메시징 등)이 필요하다.
- **내결함성은 별 1개**: 한 부분의 메모리 부족이 애플리케이션 전체를 죽인다. MTTR이 길어 가용성이 낮고 시동은 분 단위다.

> **참고**: 원문 OCR의 별점 격자표가 훼손되어, 위 등급은 본문 서술(강점 3종, 배포성·테스트성 "별 2개", 탄력성·확장성·내결함성 "별 1개")에 근거해 재구성했다. 유지보수성·진화성·반응성은 서술 맥락에서 추정한 값이다.

## 요약

- 모듈형 모놀리스는 **DDD 인기로 2판에 추가**된 스타일 — 단일 배포 단위지만 컴포넌트를 **기술이 아니라 도메인**으로 묶는다(이름공간 세 번째 노드 = 도메인).
- 구조 옵션은 **모놀리스 구조**(한 저장소, 간단)와 **모듈형 구조**(독립 JAR/DLL, 팀별 작업·명확한 경계) — 트레이드오프로 선택.
- 모듈 간 통신은 최소화하되, 필요하면 **동급 간**(간단·강결합) 또는 **중재자**(결합↓, 단 중재자와 결합). 남발은 '진흙잡탕'·비정형 모놀리스 신호.
- **자동화된 거버넌스**(ArchUnit 등)로 모듈 이름공간·의존 개수·금지된 통신을 강제해야 유지된다. 도메인 정렬 팀(CFT)과 궁합이 좋다.
- 강점은 **비용·단순성·모듈성**, 약점은 **확장성·탄력성·내결함성**. 예산이 빠듯하거나 새 시스템 시작·도메인 중심 변경에 적합하고, 운영 특성이 중요하거나 기술 지향 변경이 잦으면 계층형이 낫다.

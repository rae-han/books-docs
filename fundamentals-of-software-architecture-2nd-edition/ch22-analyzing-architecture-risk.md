# Chapter 22: Analyzing Architecture Risk (아키텍처 위험 분석)

## 핵심 질문

주관적인 위험 평가를 어떻게 객관화하는가(위험 평가 행렬)? 위험 평가표는 무엇을 보여주며, 왜 '방향'이 '스냅숏'보다 중요한가? 리스크스토밍의 세 페이즈는 왜 개별 식별로 시작하는가? 생소한 기술에는 왜 자동으로 최고 위험을 매기는가?

---

## 1. 개요

모든 아키텍처에는 **운영상 위험**(가용성·확장성·데이터 무결성)과 **구조적 위험**(정적 결합 등)이 따른다. 위험을 **정량화·평가·식별**하는 것은 아키텍트의 가장 중요한 활동 중 하나로, 결함과 구조적 부패를 시정하게 해준다.

---

## 2. 위험 평가 행렬

위험을 두 차원 — **영향(impact)**과 **발생 가능성(likelihood)** — 으로 정량화한다. 각 차원을 낮음(1)·중간(2)·높음(3)으로 평가하고 **둘을 곱한 값**이 위험 점수다. **1~2 = 낮음(녹색), 3~4 = 중간(노랑), 6~9 = 높음(빨강)**(색맹을 위해 회색조 병기 권장).

> **핵심 통찰**: **영향을 먼저, 가능성을 나중에** 평가하라. **가능성이 확실하지 않으면 확실해질 때까지 일단 높음(3)으로** 둔다. (예: 중앙 DB 가용성 — 영향 높음(3)이지만 고가용성 클러스터라 가능성 낮음(1) → 3×1 = 중간 위험.)

---

## 3. 위험 평가표

**위험 기준(왼쪽) × 맥락(위쪽)**의 스프레드시트로, 각 칸이 위험 점수다.

> **핵심 통찰**: **아키텍처가 지원해야 할 핵심 아키텍처 특성이 훌륭한 위험 평가 기준**이다(확장성·탄력성·데이터 무결성이 핵심이면 성능 위험에 시간 쓸 이유가 없다). 맥락은 **도메인·하위 도메인** 단위가 적합하다(서비스 단위는 세분도가 너무 높아 서비스 간 통신 위험을 놓친다).

- **필터링**: 이해관계자에게 보여줄 땐 저·중 위험(잡음)을 빼고 **고위험(신호)만** 강조해 신호 대 잡음비를 높인다.
- **방향(direction)**: 표는 특정 시점의 스냅숏일 뿐 개선/악화를 못 보여준다.

> **핵심 통찰**: 위험의 **방향**(증가 ▲ · 감소 ▽ · 변화 없음 ●)이 스냅숏보다 중요하다 — **적합성 함수로 지속 모니터링**해 추세를 파악한다.

---

## 4. 리스크스토밍(Risk-storming)

아키텍트 혼자서는 (일부 위험을 놓치고, 시스템 전부를 알 수 없어) 전반적 위험을 판단할 수 없다. **리스크스토밍**은 아키텍트 + **수석 개발자·기술 리더**가 함께 위험 수준을 결정하는 협업 활동으로, 세 페이즈로 진행된다.

- **페이즈 1 — 식별(개별)**: 각자 **독립적으로** 위험 평가 행렬로 평가하고 색 스티커에 점수를 적는다.

> **핵심 통찰**: 식별 페이즈를 **반드시 개별로** 하는 이유는 참가자들이 서로 영향을 주거나 주의를 분산시키지 않도록 **편견 없이** 평가하게 하기 위해서다. 가능하면 활동을 **하나의 기준 또는 하나의 맥락**으로 제한한다.

- **페이즈 2 — 합의(협업)**: 스티커를 도식에 붙이고 논의해 합의한다.

> **핵심 통찰**: **입증되지 않거나 생소한 기술에는 자동으로 최고 위험(9)을 매긴다** — 위험 평가 행렬을 적용할 수 없기 때문이다("Redis가 뭔가요?"라고 답한 개발자의 사례). 참가자가 특정 기술을 모른다는 사실 자체가 귀중한 정보다(교체하거나 교육에 투자).

- **페이즈 3 — 완화(협업)**: 위험을 줄이도록 아키텍처를 변경한다. **완화는 비용을 유발하므로 비즈니스 이해관계자를 참여**시켜 비용 vs 위험을 협상한다(5만 달러 클러스터화 대신 1.6만 달러 DB 분리로 타협한 사례).

또한 리스크스토밍은 **사용자 스토리 위험 분석**(스토리 그루밍에서 반복 완료 위험 평가)에도 쓰인다.

---

## 5. 예시 — 간호 핫라인 진단 시스템

핵심 특성이 **가용성·탄력성·보안**인 콜센터 시스템에 세 번의 리스크스토밍을 진행한다.

- **가용성**: 중앙 DB(6)·외부 진단 엔진(9). 완화 → DB를 **간호사 프로필(클러스터)과 사례 노트로 분리**, 외부 시스템은 **SLA 조사**(99.99%·99.90%)로 위험 제거.
- **탄력성**: 진단 엔진(초당 500건)이 병목(9). 완화 → API 게이트웨이-엔진 간 **비동기 대기열(배압)** + **구급차(Ambulance) 패턴**(간호사·셀프서비스 우선순위 채널 분리) + **유행병 질문 캐싱**.
- **보안**: HIPAA — 간호사만 의료 기록 접근. API 게이트웨이 보안(6). 완화 → **사용자 유형별(관리·셀프서비스·간호사) 별도 API 게이트웨이**로 비간호사 호출이 의료 기록에 도달하지 못하게.

원래 아키텍처 대비 리스크스토밍 후 아키텍처는 세 특성 우려가 해결돼 성공 가능성이 크게 높아졌다.

---

## 요약

- 위험 = **영향 × 가능성**(위험 평가 행렬). 영향 먼저, 가능성 불확실하면 높음(3)
- **위험 평가표**는 핵심 **아키텍처 특성(기준) × 도메인(맥락)** — 고위험만 필터링, **방향(추세)**을 적합성 함수로 추적
- **리스크스토밍**: 식별(개별·편견 방지) → 합의(생소한 기술 = 자동 9) → 완화(비즈니스 이해관계자와 비용 협상)
- 예시: 간호 핫라인 — 가용성(DB 분리·SLA)·탄력성(대기열·구급차 패턴·캐시)·보안(유형별 게이트웨이)
- 리스크스토밍은 **일회성이 아니다** — 주요 기능 추가 후·반복 종료마다 반복한다

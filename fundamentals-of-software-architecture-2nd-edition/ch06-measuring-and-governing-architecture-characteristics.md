# Chapter 6: Measuring and Governing Architecture Characteristics (아키텍처 특성의 측정과 거버넌스)

## 핵심 질문

모호한 아키텍처 특성을 어떻게 객관적으로 측정하는가(운영·구조·개발 프로세스)? 순환 복잡도는 무엇을 재고 얼마가 적당한가? '적합성 함수(fitness function)'란 무엇이며, 중요하지만 시급하지 않은 원칙을 어떻게 자동으로 지키게 하는가? 넷플릭스는 프로덕션에서 거버넌스를 어떻게 자동화했는가?

---

## 1. 아키텍처 특성의 측정

특성 측정이 어려운 이유는 — (1) 의미가 모호하고("터무니없이 빠른 성능"?), (2) 정의가 부서마다 다르며, (3) 너무 복합적(민첩성=모듈성+배포성+테스트성)이기 때문이다. **복합 특성을 하위 특성으로 분해해 객관적으로 정의**하면 세 문제가 함께 풀리고, 조직의 **보편 언어**가 만들어진다.

### 1.1 운영 특성

성능·확장성은 직접 측정할 수 있지만 해석이 필요하다 — **평균 응답 시간**만 보면 1%의 이상치를 놓치므로 **최대 응답 시간**도 함께 본다.

> **핵심 통찰**: 성숙한 팀은 임의의 수치 목표 대신 **통계 모델**을 쓴다 — 일정 기간 데이터로 예측 모델을 만들고, 실시간 지표가 모델을 벗어나면 경보를 울린다(모델이 틀렸거나 시스템에 문제가 생겼거나, 둘 다 알고 싶은 정보다). 요즘은 **최초 콘텐츠 렌더링(first contentful paint)** 같은 성능 예산과 **K-가중치 예산**(페이지 라이브러리 크기 제한)에 집중한다.

### 1.2 구조적 특성 — 순환 복잡도

모듈성 같은 내부 특성은 측정이 어렵지만, **순환 복잡도(Cyclomatic Complexity, CC)**로 코드 복잡도를 잴 수 있다. `CC = E − N + 2`(E=간선/분기 수, N=노드/코드 수) — 조건문이 없으면 1, 하나면 2다.

> **핵심 통찰**: 업계 임계값은 흔히 **10**이지만, 저자는 응집력 있는 코드라면 **5 미만**이어야 한다고 본다. 단, CC는 복잡성이 **본질적**(문제 자체가 복잡)인지 **우발적**(설계가 나쁨)인지 구별하지 못하니 해석이 필요하다. **TDD**는 작은 단위로 개발하게 해 자연스럽게 CC가 낮은 메서드를 만든다.

### 1.3 개발 프로세스 특성

**테스트성**은 코드 커버리지로(단, 단언문이 부실하면 커버리지 100%도 무의미), **배포성**은 배포 성공률·소요 시간·이슈 수로 측정한다. 이 특성들의 우선순위를 높이면 아키텍트는 **모듈성·격리**를 중시하는 구조적 결정을 내리게 된다 — 특성이 구조로 이어지는 사례다.

## 2. 거버넌스와 적합성 함수

거버넌스는 아키텍트의 핵심 책임이다. 문제는 "**시급함이 모든 것을 지배**"하는 상황 — 모듈성은 중요하지만 긴급하진 않다. 그래서 자동화된 거버넌스 도구가 필요하다.

**적합성 함수(fitness function)**는 (『Building Evolutionary Architectures』에서 온 개념으로) **하나 또는 여러 아키텍처 특성의 무결성을 객관적으로 평가하는 모든 메커니즘**이다. 새로운 도구가 아니라 기존 도구(단위 테스트·지표·모니터링·카오스 엔지니어링)를 보는 새로운 시각이다.

예시:
- **순환 의존성 검출**: IDE의 자동 import가 무분별한 순환 의존(진흙잡탕)을 부른다. JDepend로 패키지 순환을 검사하는 테스트를 지속적 빌드에 넣는다.
- **주 시퀀스 거리**: 각 패키지의 거리가 허용 오차를 넘으면 실패.
- **계층 규칙(ArchUnit / NetArchTest)**: "표현 계층은 영속성을 직접 참조 못 함" 같은 규칙을 코드로 강제.

> **핵심 통찰**: 적합성 함수는 **체크리스트**(아툴 가완디)처럼 생각하라 — 무거운 거버넌스 도구가 아니라, 중요한 원칙을 명시하고 자동 검증하는 장치다. 보안이 중요함은 개발자도 알지만 수백 가지 현안과 우선순위를 다툰다. 적합성 함수는 그 규칙을 **아키텍처의 토대에 자연스럽게 녹여** 넣는다. (단, 개발자가 지표만 충족시키려 악용할 수 있으니 — 예: 단언문 없는 테스트로 커버리지만 올리기 — ArchUnit으로 "모든 테스트에 단언문 1개 이상" 같은 규칙도 건다.)

> **넷플릭스 시미언 아미(Simian Army)**: 온프레미스→AWS 이전 후 운영을 직접 통제할 수 없게 되자 **카오스 엔지니어링**을 창안했다. **카오스 몽키**(무작위 장애 주입), **컨포미티 몽키**(거버넌스 준수 감시), **시큐리티 몽키**(보안 결함 점검), **재니터 몽키**(고아 인스턴스 청소)가 프로덕션에서 적합성 함수처럼 동작한다. 교훈: **장애는 기정사실이니, 고장 났을 때 잘 대처하게 만들어라.**

## 요약

- 특성 측정이 어려운 것은 모호·부서별 정의 차이·복합성 때문 — **하위 특성으로 분해**해 객관적으로 정의하면 보편 언어가 된다
- 운영은 **통계 모델**(평균+최대), 구조는 **순환 복잡도**(권장 <5, 본질적/우발적 복잡성 구별은 못 함), 개발 프로세스는 커버리지·배포 지표로 측정
- **적합성 함수**는 특성의 무결성을 객관적으로 평가하는 모든 메커니즘 — 순환 의존성·주 시퀀스 거리·계층 규칙(ArchUnit)을 지속적 빌드로 자동 검증
- 적합성 함수는 **체크리스트**처럼, 중요하지만 시급하지 않은 원칙을 아키텍처의 토대에 녹인다 — 넷플릭스 **시미언 아미**가 프로덕션 자동 거버넌스의 예

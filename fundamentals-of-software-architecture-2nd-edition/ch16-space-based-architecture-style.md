# Chapter 16: Space-Based Architecture Style (공간 기반 아키텍처 스타일)

## 핵심 질문

전통적 웹 토폴로지의 확장성 한계는 왜 결국 데이터베이스에서 막히는가? 공간 기반 아키텍처는 중앙 DB 병목을 어떻게 제거하는가? 처리 단위·가상화 미들웨어·데이터 펌프는 각각 무엇을 하는가? 복제 캐싱과 분산 캐싱은 어떻게 다른가? 이 복잡한 스타일은 언제 쓰는가?

---

## 1. 개요 — 확장성 문제를 위한 설계

전통적 웹 토폴로지(웹 서버 → 애플리케이션 서버 → DB)는 부하가 커지면 병목이 **웹 → 앱 → DB 순으로 옮겨가며**, 결국 **데이터베이스**가 동시 처리량의 상한을 정한다(가장 확장하기 어렵고 비싸다). 공간 기반 아키텍처는 **높은 확장성·탄력성·동시성**을 위해 설계된 스타일로, 특히 **동시 사용자가 가변적·예측 불가능**한 애플리케이션에 유용하다.

> **핵심 통찰**: 이름은 **튜플 공간(tuple space)**(공유 메모리로 병렬 처리)에서 왔다. 중앙 DB를 동기적 제약으로 두는 대신, **데이터를 메모리 안에서 복제**하고 갱신은 **데이터 펌프로 DB에 비동기 전송**한다. 표준 트랜잭션 처리에 DB가 관여하지 않으므로 **DB 병목이 사라져 거의 무한에 가까운 확장성**을 얻는다.

---

## 2. 토폴로지

```
   [처리 단위]   [처리 단위]   [처리 단위]   ← 앱 로직 + 메모리 내 데이터 그리드
        \           |           /
   ┌──────────── 가상화된 미들웨어 ────────────┐
   │ 메시징 그리드 · 데이터 그리드 · 처리 그리드 · 배포 관리자 │
   └───────────────────┬────────────────────┘
                  [데이터 펌프] → [데이터 기록기] → [데이터베이스]
```

- **처리 단위(processing unit)**: 앱 기능성 + **메모리 내 데이터 그리드**(헤이즐캐스트·Ignite·Coherence) + 복제 엔진. 부하에 따라 인스턴스가 동적으로 증감한다.
- **가상화된 미들웨어**(처리 단위 관리·조정): **메시징 그리드**(요청·세션 상태 → 부하 분산), **데이터 그리드**(캐시 동기화·복제), **처리 그리드**(여러 처리 단위 오케스트레이션, 선택), **배포 관리자**(부하에 따라 인스턴스 시작·종료 = 탄력성).
- **데이터 펌프 → 데이터 기록기**: 갱신을 **비동기·최종 일관성**으로 DB에 반영(처리 단위는 DB를 직접 읽고 쓰지 않는다). **데이터 판독기**는 처리 단위 시동 시 DB에서 데이터를 읽어 온다.

---

## 3. 캐싱 — 복제 vs 분산

트랜잭션 처리가 **캐싱에 의존**하는 것이 이 스타일의 핵심이다(DB 직접 접근이 없어 확장성·성능↑).

| 기준 | **복제 캐싱(replicated)** | **분산 캐싱(distributed)** |
|---|---|---|
| 구조 | 처리 단위마다 캐시, 서로 동기화 | 중앙 캐시 서버, 처리 단위는 원격 접근 |
| 최적화 | 성능·내결함성(**SPOF 없음**) | 데이터 일관성 |
| 캐시 크기 | 작음(<100MB) | 큼(>500MB) |
| 갱신 빈도 | 낮음 | 높음 |

> **핵심 통찰**: **복제 캐싱이 표준**이다 — 중앙 서버가 없어 SPOF가 없고 빠르다. 단 대용량 데이터나 잦은 갱신에는 **데이터 충돌**(갱신율이 복제 속도를 초과해 일관성 붕괴)이 생겨 분산 캐싱을 쓴다. 둘 중 하나로 통일할 필요 없이 **처리 단위마다** 다르게 고른다 — 재고(일관성) = 분산, 고객 프로필(성능·내결함성) = 복제. (전면+후면을 섞는 **근접 캐시**는 전면 캐시들이 서로 동기화되지 않아 권장하지 않는다.)

---

## 4. 데이터 펌프 · 데이터 기록기

처리 단위가 캐시를 갱신하면, 그 갱신을 **데이터 펌프**(보통 메시징 — 비동기·전달 보장·순서 보장)로 보내 **데이터 기록기**가 DB를 갱신한다. 메시징 덕에 데이터 기록기가 죽어도 처리 단위는 멈추지 않는다. 데이터 기록기는 **도메인 기반**(한 도메인의 여러 펌프를 하나가 처리) 또는 **처리 단위별 전용**(정렬이 좋아 확장성·민첩성↑)으로 둔다.

---

## 5. 왜 · 언제 사용하고, 언제 피하는가

- **적합**: **극단적·가변적·예측 불가능한 부하**(콘서트 티켓 예매, 온라인 경매, 플래시 세일 등) — 처음부터 이 문제에 특화된 아키텍처를 쓰는 편이 낫다.
- **피함**: 일반적인 비즈니스 애플리케이션 — **가장 복잡하고 비싼** 스타일이라 과한 경우가 대부분이다. 데이터 일관성이 최우선이거나 부하가 예측 가능하면 다른 스타일이 낫다.

## 6. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 기술적 |
| 퀀텀 수 | 1 이상 |
| 성능 | ★★★★★ |
| 확장성 | ★★★★★ |
| 탄력성 | ★★★★★ |
| 반응성 | ★★★★★ |
| 내결함성 | ★★★★☆ |
| 모듈성 | ★★★☆☆ |
| 배포성 | ★★★☆☆ |
| 진화성 | ★★★☆☆ |
| 유지보수성 | ★★★☆☆ |
| 전반적인 비용 | ★☆☆☆☆ |
| 단순성 | ★☆☆☆☆ |
| 테스트성 | ★☆☆☆☆ |

- **강점은 성능·확장성·탄력성·반응성**(별 5개) — 메모리 캐싱·복제와 동적 인스턴스 관리로 극한 부하를 견딘다. 복제 캐싱은 SPOF가 없어 **내결함성**도 좋다.
- **약점은 단순성·테스트성·비용**(별 1개) — 구성요소가 많아 가장 복잡하고, 극한 부하를 테스트하기 어려우며, 인프라 비용이 크다.

> **참고**: 별점은 OCR 격자표 훼손으로 이 스타일의 알려진 특성(성능·확장성·탄력성이 최고 강점, 단순성·테스트성·비용이 약점)에 근거해 재구성했다.

---

## 요약

- 공간 기반 아키텍처는 **중앙 DB 병목을 제거**해 극한 확장성을 얻는 스타일 — 데이터를 **메모리에서 복제**하고 DB는 **데이터 펌프로 비동기 갱신**
- 구성: **처리 단위**(앱+데이터 그리드) + **가상화 미들웨어**(메시징·데이터·처리 그리드·배포 관리자) + **데이터 펌프/기록기/판독기**
- 캐싱은 **복제**(성능·내결함성, SPOF 없음)와 **분산**(일관성)을 처리 단위마다 골라 쓴다
- 강점 **성능·확장성·탄력성·반응성**(최고), 약점 **단순성·테스트성·비용** — **가변적·예측 불가 고부하**(티켓·경매)에 특화된, 가장 복잡한 스타일

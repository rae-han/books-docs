# Chapter 13: Microkernel Architecture Style (마이크로커널 아키텍처 스타일)

## 핵심 질문

마이크로커널(플러그인) 아키텍처는 왜 제품·커스텀화 도메인에 잘 맞는가? 코어 시스템과 플러그인의 역할은? 왜 코어는 안정적이어야 하고 플러그인은 서로 의존하면 안 되는가? 이 스타일이 도메인·기술 분할을 **둘 다** 지원하는 유일한 스타일인 이유는?

---

## 1. 개요

마이크로커널(플러그인) 아키텍처는 **코어 시스템 + 플러그인 컴포넌트**로 이루어진 단순한 모놀리스다. **제품 기반 애플리케이션**(이클립스처럼 설치형)과 **커스텀화가 필요한 도메인**(미국 보험사의 주별 규칙, 국제 배송의 법적 변형)에 자연스럽게 들어맞는다.

---

## 2. 토폴로지 — 코어 시스템 + 플러그인

```
   [플러그인]  [플러그인]  [플러그인]
        \        |        /
          ┌──────────────┐
          │  코어 시스템   │  ← 최소 기능 / 해피 패스
          └──────────────┘
        /        |        \
   [플러그인]  [플러그인]  [플러그인]
```

**코어 시스템**은 (1) 시스템 실행에 필요한 **최소 기능**(이클립스 코어 = 기초 텍스트 에디터)이거나 (2) **해피 패스(정상 경로)** — 맞춤 처리가 거의 없는 일반 흐름이다. 코어는 크기에 따라 계층형 또는 모듈형으로 구현할 수 있다.

> **핵심 통찰**: 마이크로커널의 핵심은 **순환 복잡성(맞춤 로직)을 코어에서 빼내 플러그인으로** 넣는 것이다. 클라이언트별 `if-else` 체인 대신, 레지스트리에서 플러그인을 찾아 호출하면 확장 능력·유지보수성·테스트성이 좋아진다.

```java
// before: 커스텀화가 코어에 쌓임 (순환 복잡성↑)
if (deviceID.equals("iPhone6s")) { assessiPhone6s(); }
else if (deviceID.equals("iPad1")) { assessiPad1(); }
else if (deviceID.equals("Galaxy5")) { assessGalaxy5(); } // ...

// after: 기기별 로직을 플러그인으로 분리, 코어는 일반적으로 실행
String plugin = pluginRegistry.get(deviceID);
DevicePlugin devicePlugin = /* 레지스트리로 플러그인 인스턴스화 */;
devicePlugin.assess();
```

## 3. 플러그인 컴포넌트

플러그인은 코어를 확장하는 **독립형** 구성요소로, **변동성 높은 코드를 격리**한다(이상적으로 플러그인 간 의존성 없음). 코어와 **점대점**(메서드 호출)으로 통신한다.

- **연결 방식**: **실행 시점 플러그인**(OSGi·Jigsaw·Prism 등 — 재배포 없이 추가/제거) vs **컴파일 기반 플러그인**(간단하지만 전체 재배포 필요).
- **구현**: 공유 라이브러리(JAR·DLL·gem), 패키지/이름공간(`app.plugin.<도메인>.<컨텍스트>`), 또는 **원격(REST/메시징)**으로 개별 서비스화(단, 코어가 모놀리스라 **여전히 퀀텀 1** — 모든 요청이 코어를 거친다).
- **마이크로커널성 스펙트럼**: 코어에 기능이 적을수록 '순수'하다 — 이클립스·린터(코어는 파싱만, 플러그인 없이 거의 무용)는 순수, **웹 브라우저**(플러그인 없이도 완전 기능)는 반대쪽.
- **레지스트리**: 코어가 가용 플러그인과 접근법을 아는 수단(간단한 `Map`부터 주키퍼·Consul까지).
- **계약**: 보통 도메인 전반에 표준화. 서드파티 플러그인의 **맞춤형 계약**은 표준 계약과의 사이에 **어댑터**를 둔다.

---

## 4. 데이터 · 클라우드 · 거버넌스 · 위험

- **데이터**: 단일 DB. 플러그인은 보통 **코어를 통해** 데이터를 받는다(결합↓ — DB가 바뀌어도 코어만 영향). 단, 플러그인이 자체 데이터 저장소를 가질 수 있다.
- **클라우드**: 모놀리스라 선택지가 제한적(전체 배포 / 데이터만 클라우드 / 코어는 온프레미스·플러그인은 클라우드 — 단 마지막은 반응성 문제).
- **거버넌스**: **코어의 변동성**을 버전 관리 변경 빈도로 확인하는 적합성 함수, **계약 테스트**.

> **핵심 통찰 — 두 가지 위험**: (1) **변동성 높은 코어** — 코어는 초기 개발 후 안정적이어야 한다. 변화를 플러그인으로 격리하는 게 이 스타일의 철학인데, 코어를 계속 고치는 건 위배다. (2) **플러그인 의존성** — 플러그인은 코어하고만 통신해야 한다("의존성 없는 플러그인"). 플러그인끼리 의존하면 **전이적 의존성 충돌**(같은 라이브러리의 다른 버전) 같은 복잡성이 폭발한다.

## 5. 왜 · 언제 사용하고, 언제 피하는가

- **적합**: **커스텀화·확장 능력**이 중요한 제품/애플리케이션(이클립스·Jira 같은 IDE·협업 도구), 장소·클라이언트마다 설정이 달라야 하는 문제(**강한 도메인-아키텍처 동형성**). 세무 소프트(1040=코어, 각 양식=플러그인 → 세법 변경 격리), 보험 청구(주별 규칙=플러그인).
- **피함**: 높은 확장성·탄력성·내결함성이 필요한 대규모 시스템(모놀리스 한계).

## 6. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 기술적 / 도메인 |
| 퀀텀 수 | 1 |
| 전반적인 비용 | ★★★★★ |
| 단순성 | ★★★★☆ |
| 테스트성 | ★★★☆☆ |
| 배포성 | ★★★☆☆ |
| 모듈성 | ★★★☆☆ |
| 진화성 | ★★★☆☆ |
| 반응성 | ★★★☆☆ |
| 유지보수성 | ★★★☆☆ |
| 확장성 | ★☆☆☆☆ |
| 탄력성 | ★☆☆☆☆ |
| 내결함성 | ★☆☆☆☆ |

- **분할 방식이 독특**: 마이크로커널은 **도메인 분할과 기술 분할을 모두 지원하는 유일한 스타일**이다(대개 기술 분할이지만 강한 도메인-아키텍처 동형성으로 도메인 분할도).
- **강점은 단순성·비용**(계층형과 유사). 퀀텀은 항상 1(요청이 코어를 거쳐야 플러그인 도달).
- **테스트성·배포성·모듈성·진화성·반응성 ★3** — 기능을 독립 플러그인으로 격리하는 덕(세법 바뀌면 플러그인만 추가/제거). 반응성은 앱이 작고 싱크홀 고통이 덜해 평균 이상.
- **확장성·탄력성·내결함성 ★1** — 모놀리스 배포 때문.

> **참고**: 별점은 OCR 격자표 훼손으로 본문 서술(단순성·비용 강점, 테스트성·배포성·모듈성·진화성·반응성 "별 3개", 확장성·탄력성·내결함성 "별 1개")에 근거해 재구성했다.

## 요약

- 마이크로커널 = **코어 시스템(최소 기능/해피 패스) + 플러그인(변동성 격리)** — 제품·커스텀화 도메인에 자연스럽다
- **순환 복잡성을 코어에서 플러그인으로** 빼낸다(레지스트리로 찾아 호출). 플러그인은 실행 시점/컴파일 기반, 점대점/원격으로 연결
- 위험: **변동성 높은 코어**(안정적이어야)·**플러그인 간 의존성**(전이 충돌 → 피하라)
- 강점 **단순성·비용**, 약점 **확장성·탄력성·내결함성** — **도메인·기술 분할을 모두 지원하는 유일한 스타일**. 세무·보험처럼 커스텀화가 핵심인 도메인에 최적

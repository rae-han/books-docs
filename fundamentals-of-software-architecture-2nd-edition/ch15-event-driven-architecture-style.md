# Chapter 15: Event-Driven Architecture Style (이벤트 주도 아키텍처 스타일)

## 핵심 질문

요청 기반 모델과 이벤트 기반 모델은 어떻게 다른가? '이벤트'와 '메시지'의 차이는? 비동기 통신이 왜 반응성을 높이고 '동적 퀀텀 얽힘'을 푸는가? 이벤트 페이로드를 데이터 기반과 키 기반 중 어떻게 고르는가? 코레오그래피와 중재자 토폴로지의 트레이드오프는? EDA에 흔한 안티패턴은?

---

## 1. 요청 기반 vs 이벤트 기반 · 토폴로지

- **요청 기반 모델**: 요청을 **오케스트레이터**(보통 UI)가 요청 처리기에 **결정론적·동기적**으로 전달한다("지난 6개월 주문 내역 조회").
- **이벤트 기반 모델**: 시스템이 **이벤트에 반응**한다("입찰이 제출됨" → 최고 입찰자 결정).

EDA는 **비동기 '발사 후 망각(fire-and-forget)'** 통신을 쓴다. 네 컴포넌트: **개시 이벤트(initiating event)** → **이벤트 브로커**(토픽·스트림, 발행-구독) → **이벤트 처리기(서비스)** → **파생 이벤트(derived event)**. 처리기는 개시 이벤트를 받아 처리한 뒤 파생 이벤트로 자신이 한 일을 방송하고, 다른 처리기들이 그에 반응하는 과정이 모두 유휴 상태가 될 때까지 이어진다(계주 경기의 바통처럼).

> **핵심 통찰 — 독성 이벤트(poison event)**: 서비스들이 파생 이벤트를 발생·반응하는 과정이 **무한 반복**되는 것(재고 갱신 → 재고 조정 → 재고 갱신 …). EDA에서 자주 일어나므로 주의해서 피해야 한다.

---

## 2. 이벤트 vs 메시지

| | **이벤트** | **메시지** |
|---|---|---|
| 의미 | 이미 일어난 일 방송("주문 접수됨") | 해야 할 일 명령/쿼리("결제하라") |
| 응답 | 대체로 불요 | 대체로 요구 |
| 대상 | 여러 처리기(발행-구독, 일대다) | 하나(점대점, 일대일) |
| 채널 | 토픽·스트림 | 대기열 |

이 차이가 EDA의 **분리(decoupling)** 성질을 만든다. EDA는 주로 이벤트를 쓰지만, 요청-응답이나 중재자 토폴로지에서는 메시지를 쓴다.

---

## 3. 비동기의 힘 — 반응성과 동적 분리

> **핵심 통찰 — 반응성 vs 성능**: 리뷰 게시가 3,000ms 걸릴 때, **동기**는 사용자가 3,100ms를 기다리지만 **비동기**는 25ms만에 "접수됨"을 응답한다(실제 처리는 배후에서). **반응성**은 "받았고 곧 처리됨"을 빨리 알리는 것, **성능**은 종단 간 처리를 빠르게 하는 것 — 둘은 다르다. 단, 비동기는 **오류 처리가 어렵다**(사용자에게 알릴 방법이 없을 수 있다).

> **핵심 통찰 — 동적 퀀텀 얽힘**: 두 시스템이 **동기 호출**로 묶이면 가용성·반응성·확장성이 **두 시스템 사이에 존재**해 사실상 하나의 퀀텀이 된다(한쪽이 느리면 다른 쪽도 느리다). **비동기 통신**은 이 동적 의존성을 끊어 각자 독립 퀀텀이 되게 한다.

이벤트를 방송할 때 누가 반응하는지 신경 쓸 필요가 없다는 **브로드캐스팅** 특성이 **의미적 분리(semantic decoupling)**를 만든다 — 한 처리기가 다른 처리기의 동작을 전혀 모른다.

---

## 4. 이벤트 페이로드와 안티패턴

- **데이터 기반 페이로드**: 처리에 필요한 모든 데이터를 담아 보냄 → DB 조회 불필요(성능·확장성↑). 단 **스탬프 결합**(안 쓰는 필드까지 공유 → 변경 시 무관한 처리기도 영향, 대역폭 낭비)·계약/버전 관리·데이터 일관성 문제.
- **키 기반 페이로드**: 맥락 키(주문 ID)만 담음 → 각 처리기가 DB 조회(단일 기록 시스템, 계약 간단). 단 DB 부하로 성능·확장성↓.

| 기준 | 데이터 기반 | 키 기반 |
|---|---|---|
| 성능·확장성 | 좋음 | 나쁨 |
| 계약 관리·대역폭 | 나쁨 | 좋음 |
| 데이터 일관성 | 나쁨 | 좋음 |

> **핵심 통찰 — 세 가지 안티패턴**: (1) **빈혈성 이벤트(anemic event)**: 페이로드에 처리에 필요한 맥락(예: 변경된 필드·이전 값)이 부족. (2) **하루살이 떼(Swarm of Gnats)**: 너무 세밀한 파생 이벤트를 너무 많이 발생 → 이벤트 흐름을 아무도 이해 못 함(개별 필드 갱신을 하나의 '프로필 갱신됨'으로 묶어라). (3) **독성 이벤트**(무한 반복). 페이로드 세분도는 두 극단 사이 **스펙트럼**에서 고른다.

## 5. 코레오그래피 vs 중재자 토폴로지

- **코레오그래피형(choreographed)**: 지금까지의 브로드캐스팅 방식 — 중앙 제어 없이 처리기들이 이벤트를 주고받으며 협업.
- **중재자형(mediator)**: **이벤트 중재자**가 개시 이벤트를 받아 작업흐름의 단계들을 알고 **파생 메시지**를 점대점으로 지시. 처리기는 결과를 중재자에게만 보고(방송 안 함).

| | 코레오그래피 | 중재자 |
|---|---|---|
| 제어 | 분산(중앙 없음) | 중재자가 작업흐름 제어 |
| 통신 | 이벤트(방송) | 메시지(점대점) |
| 강점 | **높은 성능·확장성·분리** | **작업흐름 제어·오류 처리·재시작** |
| 약점 | 오류 처리·상태 관리 어려움 | 성능↓, 분리↓, **중재자 SPOF·병목** |

> **핵심 통찰**: 트레이드오프는 **성능·확장성(코레오그래피) vs 작업흐름 제어·오류 처리(중재자)**로 귀결된다. 중재자는 이벤트를 **단순·어려움·복잡**으로 분류해 Apache Camel(단순)·BPEL(복잡·동적)·BPM(사람 개입·장기 실행)에 위임하고, SPOF를 피하려 도메인별로 여러 중재자를 둔다. 흔히 일반 흐름은 중재자, 예외적·동적 흐름은 코레오그래피로 처리하는 **혼합**을 쓴다.

---

## 6. 오류 처리 · 데이터 손실 · 요청-응답

- **작업흐름 이벤트 패턴**: 오류 발생 시 이벤트 소비자가 즉시 **작업흐름 처리기**에 위임하고 다음 메시지로 넘어간다(반응성 유지). 작업흐름 처리기가 프로그래밍적으로 복구·재제출하거나, 못 하면 사람 대시보드로 보낸다.
- **데이터 손실 방지**: 영속적 대기열 + 동기적 전송(생산자→브로커), 클라이언트 승인 모드(대기열→소비자), ACID 트랜잭션 + LPS(소비자→DB).
- **요청-응답(의사동기)**: 요청 대기열 + 응답 대기열. **상관관계 ID(CID)**(권장) 또는 임시 대기열로 응답을 짝짓는다.

---

## 7. 데이터 토폴로지

- **모놀리스 DB**: 모든 처리기가 중앙 DB 직접 조회(서비스 간 통신 불필요) — 단 내결함성·확장성·변경 관리 나쁨, 단일 퀀텀.
- **도메인 DB**: 도메인별 DB — 내결함성·확장성·변경 제어 우월. 단 다른 도메인 데이터가 필요하면 **동기 호출**이 생긴다.
- **전용 DB(서비스당 DB)**: 처리기마다 DB — 세 중 **최고의 내결함성·확장성·변경 제어**. 단 비용·**동기 동적 결합** 위험이 가장 크다.

> 동기 결합이 늘수록 도메인/전용 DB의 장점이 사라진다 — 동기 통신이 많으면 도메인 경계를 재평가하거나 모놀리스 DB로 되돌린다.

---

## 8. 위험 · 거버넌스 · 팀

- **위험**: **비결정론적** 이벤트 흐름(무슨 일이 벌어질지 예측 어려움), **정적 결합**(페이로드 계약)으로 인한 취약성, 과도한 **동기 통신**, **상태 관리**(개시 이벤트가 완료됐는지 알기 어려움).
- **거버넌스**: 대부분 비구조적 → **관측성(로그)**이 핵심. 계약 변경률·스탬프 결합·동기 호출을 적합성 함수로 추적.
- **팀**: 주로 기술 분할이나 도메인 정렬 CFT·**난해한 하위시스템 팀**과 잘 맞고(복잡 처리를 처리기로 격리), **활성화 팀**과는 잘 안 맞는다(이벤트 흐름 이해를 방해).

---

## 9. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 기술적 |
| 퀀텀 수 | 1 이상 |
| 진화성 | ★★★★★ |
| 반응성 | ★★★★★ |
| 확장성 | ★★★★★ |
| 탄력성 | ★★★★★ |
| 내결함성 | ★★★★★ |
| 모듈성 | ★★★★☆ |
| 배포성 | ★★★★☆ |
| 유지보수성 | ★★★☆☆ |
| 전반적인 비용 | ★★☆☆☆ |
| 테스트성 | ★★☆☆☆ |
| 단순성 | ★☆☆☆☆ |

- **강점은 성능·확장성·탄력성·내결함성·진화성** — 비동기·분리 덕에 고성능·고확장 애플리케이션에 최적이고, 확장 가능 파생 이벤트로 **확장 능력**이 뛰어나다.
- **약점은 단순성·테스트성·비용** — 비결정론적 흐름은 이해·테스트가 어렵고, 분산이라 비싸다.

> **참고**: 별점은 OCR 격자표 훼손으로 EDA의 알려진 특성(성능·확장성·탄력성·내결함성·진화성이 강점, 단순성·테스트성이 약점)에 근거해 재구성했다.

## 요약

- EDA는 **비동기 이벤트**에 반응하는 분산 스타일 — 개시/파생 이벤트, 브로커, 처리기. 고성능·고확장의 대표 스타일
- **이벤트**(이미 일어남·방송·응답 불요) ≠ **메시지**(명령·점대점·응답 요구)
- 비동기는 **반응성**을 높이고(≠성능) **동적 퀀텀 얽힘**을 푼다
- 페이로드는 **데이터 기반**(성능·확장성 ↔ 스탬프 결합)과 **키 기반**(계약 간단 ↔ DB 부하)의 트레이드오프. 안티패턴: 독성·빈혈성·하루살이 떼
- **코레오그래피**(성능·분리) vs **중재자**(작업흐름 제어·오류 처리·SPOF) — 흔히 혼합. 데이터는 모놀리스/도메인/전용 DB
- 강점 **성능·확장성·탄력성·내결함성·진화성**, 약점 **단순성·테스트성·비용**

# Chapter 14: Service-Based Architecture Style (서비스 기반 아키텍처 스타일)

## 핵심 질문

서비스 기반 아키텍처는 왜 "가장 실용적인 분산 아키텍처"로 불리는가? 성긴(coarse-grained) 도메인 서비스는 마이크로서비스와 세분도·트랜잭션에서 어떻게 다른가? 데이터베이스 변경의 파급을 어떻게 관리하는가? "모든 것이 마이크로서비스일 필요는 없다"는 무슨 뜻인가?

---

## 1. 개요 · 토폴로지

서비스 기반 아키텍처는 **마이크로서비스의 혼합 변형**으로, 분산이지만 마이크로서비스·이벤트 주도보다 **덜 복잡하고 저렴**해 비즈니스 애플리케이션에 널리 쓰인다. 기본 토폴로지는 **분리 배포된 UI + 성긴 도메인 서비스들 + (생략 가능한) 모놀리스 DB**의 분산 거시 계층 구조다.

```
        [사용자 인터페이스]
     ┌───────┼───────┬───────┐
 [도메인]  [도메인]  [도메인]  [도메인]  ← 성긴 서비스(보통 12개 이하)
   서비스    서비스    서비스    서비스
     └───────┴───┬───┴───────┘
            [데이터베이스]
```

- 각 도메인 서비스는 시스템의 도메인/하위 도메인을 대표하며(주문 이행·배송 등 **성긴** 단위), 독립적으로 배포된다. 컨테이너화는 필수가 아니다.
- UI는 REST·서비스 로케이터 등으로 서비스를 원격 호출한다. 단일 모놀리스 DB면 SQL 조인이 자유롭다.

---

## 2. 스타일 세부 사항 — 세분도와 트랜잭션

각 도메인 서비스는 **API 퍼사드 + 비즈니스 + 영속성** 계층형으로(또는 도메인 컴포넌트로) 설계한다. **API 퍼사드**가 UI의 비즈니스 요청 흐름을 조율한다.

> **핵심 통찰**: 서비스 기반과 마이크로서비스의 핵심 차이는 **세분도**다. 주문 처리를 서비스 기반은 **한 서비스 안에서 클래스 수준으로 내부 조율**(주문 등록·결제·재고를 한 서비스가)하는 반면, 마이크로서비스는 **외부 서비스들이 상호 조율**한다. 그래서 서비스 기반은 단일 서비스 안에서 **ACID 트랜잭션**(커밋/롤백)으로 무결성을 보장하지만, 마이크로서비스는 **BASE(최종 일관성)와 보상 갱신**에 의존한다.

단점: 도메인 서비스에 기능이 많으니, 주문 처리를 고치려면 **결제까지 함께 테스트·재배포**해야 하고, 변경이 다른 기능에 영향을 줄 위험도 크다.

**유연성**: UI 변형(단일 모놀리스 / 도메인별 / 서비스별), **API 게이트웨이·역방향 프록시** 계층으로 횡단 관심사(보안·지표·서비스 발견) 집약.

---

## 3. 데이터 토폴로지 — 변경 관리

분산이면서 **모놀리스 DB를 효과적으로 지원**하는 독특한 스타일이다. 단일 DB부터 도메인별 DB까지 가능하다.

> **핵심 통찰 — 엔티티 객체 안티패턴**: DB 테이블을 나타내는 엔티티 객체를 **하나의 공유 라이브러리**에 다 담으면, 스키마 변경 시 그 테이블을 안 쓰는 서비스까지 **전부 재배포**해야 한다(안티패턴). 해결책은 DB를 **논리적으로 분할**하고 분할마다 **별도 공유 라이브러리**(customer·invoicing·order…)를 두는 것 — 그러면 변경이 해당 라이브러리를 쓰는 서비스에만 영향을 준다. 모든 서비스가 쓰는 `common` 엔티티는 DB 팀만 변경하도록 잠근다.

---

## 4. 클라우드 · 위험 · 거버넌스 · 팀

- **클라우드**: 도메인 서비스는 범위가 넓어 서버리스보다 **컨테이너화**가 일반적.
- **위험**: (1) **서비스 간 통신**은 피한다 — 결합은 DB 수준에서만. 통신이 많으면 도메인 분할이 잘못됐거나 스타일이 안 맞는 신호. (2) 도메인 서비스를 **너무 많이(12개 초과) 만들지 않는다**.
- **거버넌스**: 변경이 여러 서비스에 걸치지 않게 하고, **오케스트레이션은 UI/API 게이트웨이 레벨**에서.
- **팀**: 도메인 정렬 CFT와 잘 맞는다(기술별 팀과는 안 맞음).

## 5. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 방식 | 도메인 |
| 퀀텀 수 | 1 이상 |
| 전반적인 비용 | ★★★★☆ |
| 단순성 | ★★★★☆ |
| 모듈성 | ★★★★☆ |
| 유지보수성 | ★★★★☆ |
| 테스트성 | ★★★★☆ |
| 배포성 | ★★★★☆ |
| 진화성 | ★★★★☆ |
| 반응성 | ★★★★☆ |
| 내결함성 | ★★★★☆ |
| 확장성 | ★★★☆☆ |
| 탄력성 | ★★☆☆☆ |

- **별 5개는 없지만 여러 핵심 특성이 별 4개**라 전반적으로 우수하고 **실용적**이다. 도메인 분할·독립 배포로 **민첩성·테스트성·배포성**이 좋고, 서비스가 자기완결적이라(DB·코드 공유로 서비스 간 통신 불필요) **내결함성·가용성**도 별 4개 — 한 서비스가 죽어도 나머지는 정상.
- **확장성 ★3, 탄력성 ★2**: 세분도가 낮아 기능 중복이 많아 자원 효율이 떨어진다(트래픽 많은 서비스만 다중 인스턴스로).
- **단순성·비용이 강점** — 마이크로서비스·이벤트 주도보다 저렴하다("페라리를 사서 막히는 도로를 달리는" 낭비를 피함). **DDD·ACID와 잘 맞는다**.

> **핵심 통찰**: "**애플리케이션의 모든 부분이 마이크로서비스일 필요는 없다**"(마크 리처즈). 서비스 기반은 마이크로서비스로 가는 **디딤돌**로 좋다 — 도메인을 분석하면 정말 마이크로서비스가 필요한 부분(고잉 그린의 Assessment)과 그대로 둘 부분(Recycling·Accounting)이 드러난다.

---

## 요약

- 서비스 기반 = **마이크로서비스의 실용적 혼합 변형** — 분리 UI + 성긴 도메인 서비스(≤12) + (선택) 모놀리스 DB
- 세분도가 낮아 **단일 서비스 안 ACID 트랜잭션**이 가능(마이크로서비스는 BASE·보상 갱신)
- 데이터: 엔티티 객체를 **하나로 묶으면 안티패턴** → 논리적 분할 + 분할별 공유 라이브러리
- 강점 **비용·단순성·민첩성·내결함성**(별 4개 다수), 약점 **확장성·탄력성**(세분도 낮음)
- **"모든 게 마이크로서비스일 필요 없다"** — 분산 아키텍처로 가는 좋은 디딤돌

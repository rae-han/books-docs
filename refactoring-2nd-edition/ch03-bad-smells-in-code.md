# Chapter 3: Bad Smells in Code (코드에서 나는 악취)

## 핵심 질문

리팩터링을 **언제** 시작하고 멈춰야 하는가? "리팩터링이 필요하다"는 신호를 어떻게 알아채는가? 각 코드 냄새(bad smell)는 무엇이며, 어떤 리팩터링으로 치유하는가?

---

## 0. 이 장에서 다루는 것

리팩터링의 **방법**을 아는 것과 **제때** 적용할 줄 아는 것은 다르다. 켄트 벡과 파울러는 "리팩터링이 필요한 시점"을 **냄새(smell)**라는 비유로 포착했다 — 정확한 종료 기준 대신, **리팩터링하면 해결되는 문제의 징후**를 제시한다.

> 냄새 나면 당장 갈아라.<br>— 켄트 벡 할머니의 육아 원칙

이 장은 24가지 냄새를 소개하고, 각각에 대응하는 리팩터링(6~12장)을 가리킨다. 어떤 기법을 쓸지 모를 때는 이 장(또는 **부록 B**)에서 냄새를 찾아 처방을 따라가면 된다. 냄새는 **정량적 규칙이 아니라 직관을 돕는 징후**임을 유의하라(인스턴스 변수 몇 개, 함수 몇 줄 같은 기준은 경험으로 감을 키운다).

> 이 장은 **코드가 아니라 냄새의 목록**이라 코드 예제가 없다. 각 냄새는 **징후 + 처방(리팩터링, 절 번호)** 형태로 정리한다. 원서는 24개를 나열하지만, 이 노트는 기억을 돕기 위해 8개 주제로 묶었다.

---

## 1. 이름과 주석

- **기이한 이름 (Mysterious Name, 3.1)**: 이름만 보고 무슨 일을 하는지 모른다. 코드 명료성의 최우선 요소가 이름이다. → **함수 선언 바꾸기(6.5)**·**변수 이름 바꾸기(6.7)**·**필드 이름 바꾸기(9.2)**. (마땅한 이름이 안 떠오르면 더 깊은 설계 문제의 신호)
- **주석 (Comments, 3.24)**: 주석 자체는 향기지만, **탈취제처럼** 나쁜 코드를 덮는 데 쓰면 냄새다. → 주석이 설명하려는 코드 블록을 **함수 추출하기(6.1)**, 이름으로 **함수 선언 바꾸기(6.5)**, 선행조건이면 **어서션 추가하기(10.6)**. "왜 이렇게 짰는지"를 남기는 주석은 좋다.

---

## 2. 중복과 크기

- **중복 코드 (Duplicated Code, 3.2)**: 같은 구조가 여러 곳에 반복돼 수정 시 모두 손봐야 한다. → **함수 추출하기(6.1)**; 비슷하지만 다르면 **문장 슬라이드하기(8.6)**로 모은 뒤 추출; 서브클래스 중복은 **메서드 올리기(12.1)**.
- **긴 함수 (Long Function, 3.3)**: 오래 사는 프로그램은 짧은 함수로 이뤄진다. 핵심은 길이가 아니라 **목적(의도)과 구현의 괴리**다. → **함수 추출하기(6.1)**(99%); 임시 변수 많으면 **임시 변수를 질의 함수로 바꾸기(7.4)**, 매개변수 많으면 **매개변수 객체 만들기(6.8)**·**객체 통째로 넘기기(11.4)**; 조건문은 **조건문 분해하기(10.1)**, 반복문은 **반복문 쪼개기(8.7)**.
- **긴 매개변수 목록 (Long Parameter List, 3.4)**: 매개변수가 많으면 이해하기 어렵다. → **매개변수를 질의 함수로 바꾸기(11.5)**, **객체 통째로 넘기기(11.4)**, **매개변수 객체 만들기(6.8)**, 플래그 인수는 **플래그 인수 제거하기(11.3)**.
- **거대한 클래스 (Large Class, 3.20)**: 필드·코드가 너무 많아 중복을 부른다. → **클래스 추출하기(7.5)**·**슈퍼클래스 추출하기(12.8)**·**타입 코드를 서브클래스로 바꾸기(12.6)**. (접두어/접미어가 같은 필드가 추출 후보)

---

## 3. 데이터의 문제

- **전역 데이터 (Global Data, 3.5)**: 어디서든 건드릴 수 있고 누가 바꿨는지 추적 불가("유령 같은 원격작용"). 가장 지독한 축. → **변수 캡슐화하기(6.6)**로 접근을 함수로 감싸고 접근 범위를 최소화.
- **가변 데이터 (Mutable Data, 3.6)**: 변경이 예상 밖 버그를 부른다. → **변수 캡슐화하기(6.6)**, **변수 쪼개기(9.1)**, 갱신 로직 분리(**함수 추출하기**·**문장 슬라이드하기**), **질의 함수와 변경 함수 분리하기(11.1)**, **세터 제거하기(11.7)**, **파생 변수를 질의 함수로 바꾸기(9.3)**.
- **데이터 뭉치 (Data Clumps, 3.10)**: 항상 함께 몰려다니는 데이터 서너 개. (판별법: 값 하나를 빼도 나머지가 의미 있는가?) → 필드 뭉치는 **클래스 추출하기(7.5)**, 매개변수 뭉치는 **매개변수 객체 만들기(6.8)**·**객체 통째로 넘기기(11.4)**.
- **기본형 집착 (Primitive Obsession, 3.11)**: 화폐·좌표·구간을 전용 타입 없이 숫자·문자열("문자열화된")로 표현. → **기본형을 객체로 바꾸기(7.3)**; 타입 코드면 **타입 코드를 서브클래스로 바꾸기(12.6)** + **조건부 로직을 다형성으로 바꾸기(10.4)**.
- **임시 필드 (Temporary Field, 3.16)**: 특정 상황에서만 값이 차는 필드. → **클래스 추출하기(7.5)** + **함수 옮기기(8.1)**로 관련 코드를 몰아넣고, **특이 케이스 추가하기(10.5)**.
- **데이터 클래스 (Data Class, 3.22)**: 필드 + 게터/세터뿐인 클래스. → public 필드는 **레코드 캡슐화하기(7.1)**, 불변 필드는 **세터 제거하기(11.7)**; 데이터를 다루는 동작을 **함수 옮기기(8.1)**로 이 클래스로 이동. (단, 단계 쪼개기의 중간 데이터 구조 같은 불변 결과 객체는 예외 — 동작을 넣을 필요 없음)

---

## 4. 변경의 확산

같은 원인(맥락 구분 실패)에서 나온 정반대 두 냄새다.

- **뒤엉킨 변경 (Divergent Change, 3.7)**: **한 모듈이 여러 이유로** 자주 바뀐다(단일 책임 원칙 위반 — 한 코드에 여러 맥락이 섞임). → 순차적 맥락은 **단계 쪼개기(6.11)**, 흩어진 함수는 **함수 옮기기(8.1)**로 맥락별 모듈에 모으고 필요하면 **함수 추출하기(6.1)**·**클래스 추출하기(7.5)**.
- **산탄총 수술 (Shotgun Surgery, 3.8)**: **하나를 바꾸려면 여러 모듈을** 자잘하게 고쳐야 한다(한 맥락이 여러 코드에 흩뿌려짐). → **함수 옮기기(8.1)**·**필드 옮기기(8.2)**로 한 모듈에 모으고, **여러 함수를 클래스로 묶기(6.9)**·**여러 함수를 변환 함수로 묶기(6.10)**·**단계 쪼개기(6.11)**. (때로 **함수/클래스 인라인하기**로 일단 합친 뒤 다시 쪼갠다)

---

## 5. 잘못된 위치와 결합

- **기능 편애 (Feature Envy, 3.9)**: 함수가 자기 모듈보다 **다른 모듈의 데이터**와 더 많이 상호작용한다. → 데이터 근처로 **함수 옮기기(8.1)**; 일부만 편애하면 **함수 추출하기(6.1)** 후 이동. (가장 많은 데이터를 가진 모듈로)
- **메시지 체인 (Message Chains, 3.17)**: `a.b().c().d()`처럼 객체를 연쇄로 타고 들어가 내비게이션 구조에 종속된다. → **위임 숨기기(7.7)**; 최종 결과를 쓰는 코드를 **함수 추출하기(6.1)** 후 **함수 옮기기(8.1)**.
- **중개자 (Middle Man, 3.18)**: 클래스 메서드의 절반이 다른 클래스로 **위임만** 한다. → **중개자 제거하기(7.8)**로 실제 객체와 직접 소통. (위임이 캡슐화의 수단이지만 지나치면 냄새)
- **내부자 거래 (Insider Trading, 3.19)**: 모듈끼리 데이터를 은밀히 많이 주고받아 결합도가 높다. → **함수 옮기기(8.1)**·**필드 옮기기(8.2)**로 분리, 공통은 제3의 모듈이나 **위임 숨기기(7.7)**; 상속 결탁은 **서브클래스/슈퍼클래스를 위임으로 바꾸기(12.10·12.11)**.

---

## 6. 조건문과 반복문

- **반복되는 switch문 (Repeated Switches, 3.12)**: 같은 조건 분기(`switch`/긴 `if-else`)가 **여러 곳에 반복**돼, 분기 하나 추가에 모두 손봐야 한다. → **조건부 로직을 다형성으로 바꾸기(10.4)**. (단순히 switch를 썼다고 냄새는 아님 — 반복이 문제)
- **반복문 (Loops, 3.13)**: 일급 함수를 지원하는 언어라면 낡은 요소. → **반복문을 파이프라인으로 바꾸기(8.8)**(`filter`·`map` 등)로 원소 처리 흐름을 드러낸다.

---

## 7. 군더더기

- **성의 없는 요소 (Lazy Element, 3.14)**: 본문을 그대로 쓰는 것과 다름없는 함수, 메서드 하나뿐인 클래스 등 **값을 못 하는 구조**. → **함수 인라인하기(6.2)**·**클래스 인라인하기(7.6)**; 상속이면 **계층 합치기(12.9)**.
- **추측성 일반화 (Speculative Generality, 3.15)**: "나중에 필요할 거야"로 만든, 지금은 안 쓰는 후킹·특이 케이스. → 안 쓰는 추상 클래스는 **계층 합치기(12.9)**, 쓸데없는 위임은 **인라인**, 안 쓰는 매개변수는 **함수 선언 바꾸기(6.5)**, 테스트만 쓰는 코드는 테스트 삭제 후 **죽은 코드 제거하기(8.9)**.

---

## 8. 상속의 문제

- **서로 다른 인터페이스의 대안 클래스들 (Alternative Classes with Different Interfaces, 3.21)**: 교체 가능해야 할 클래스들의 인터페이스가 다르다. → **함수 선언 바꾸기(6.5)**로 시그니처를 맞추고, **함수 옮기기(8.1)**로 동작을 밀어넣고, 중복이 생기면 **슈퍼클래스 추출하기(12.8)**.
- **상속 포기 (Refused Bequest, 3.23)**: 서브클래스가 부모의 유산(메서드·데이터)을 원치 않는다. → (드물게) 새 서브클래스로 **메서드 내리기(12.4)**·**필드 내리기(12.5)**; **인터페이스까지 거부**하면 심각 — **서브클래스/슈퍼클래스를 위임으로 바꾸기(12.10·12.11)**. (구현만 재활용하는 상속은 대개 참을 만하다)

## 요약

- 리팩터링의 시점은 규칙이 아니라 **냄새(직관적 징후)**로 판단한다 — 이 장과 **부록 B**가 냄새→리팩터링을 안내한다.
- **이름·주석**: 기이한 이름은 이름 바꾸기로, 주석은 탈취제로 쓰지 말고 함수 추출·이름 개선으로.
- **중복·크기**: 중복 코드·긴 함수·긴 매개변수·거대한 클래스는 추출 계열로 쪼갠다.
- **데이터**: 전역·가변 데이터는 캡슐화, 데이터 뭉치·기본형 집착은 전용 타입/객체로.
- **변경 확산**: 뒤엉킨 변경(한 모듈 여러 이유) vs 산탄총 수술(한 변경 여러 모듈) — 맥락을 분리하거나 모은다.
- **결합**: 기능 편애·메시지 체인·중개자·내부자 거래는 함수/필드 옮기기와 위임 숨기기로.
- **조건·반복·군더더기·상속**: 반복 switch는 다형성, 반복문은 파이프라인, 값 못 하는 요소는 인라인, 잘못된 상속은 위임으로.

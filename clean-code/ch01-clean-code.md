# Chapter 1: Clean Code (깨끗한 코드)

## 핵심 질문

깨끗한 코드란 무엇이며, 왜 깨끗한 코드를 작성하는 것이 프로그래머의 생존과 직결되는가?

---

## 1. 코드가 존재하리라

코드의 종말을 예견하는 사람들이 있다. 언젠가 코드가 사라지고 모델이나 요구사항에서 프로그램이 자동 생성되리라 예측한다. 하지만 이는 불가능하다. 코드는 요구사항을 상세히 표현하는 수단이다. 어느 수준에서든 요구사항의 정밀한 표현이 필요하며, 그 표현이 곧 코드다. 기계가 실행할 수 있을 정도로 상세하게 요구사항을 명시하는 작업, 이것이 프로그래밍이고 그 결과물이 코드다.

> **핵심 통찰**: 코드는 요구사항을 정밀하게 표현하는 유일한 수단이다. 추상화 수준이 높아지고 도메인 특화 언어가 늘어나더라도, 코드는 사라지지 않는다.

---

## 2. 나쁜 코드

누구나 나쁜 코드에 발목 잡혀 고생한 경험이 있다. 이를 '고행(wading)'이라 부른다 — 엉킨 덩굴과 숨겨진 함정으로 가득한 늪지를 힘겹게 헤쳐나가는 것이다.

한때 인기 있던 킬러 앱을 만들었던 회사가 있었다. 하지만 출시에 바빠 코드를 마구 짰고, 기능을 추가할수록 코드는 엉망이 되어갔다. 결국 감당이 불가능한 수준에 이르러 회사는 망했다. **회사가 망한 원인은 바로 나쁜 코드 탓이었다.**

나쁜 코드를 짠 이유는 다양하다:
- 시간에 쫓겨서
- 상사에게 욕 먹을까봐
- 지겨워서 빨리 끝내려고
- 밀린 업무가 너무 많아서

우리 모두는 "나중에 정리하겠다"고 다짐한 경험이 있다. 하지만 **르블랑의 법칙(LeBlanc's Law)** 을 기억하라:

> **나중은 결코 오지 않는다.**

---

## 3. 나쁜 코드로 치르는 대가

나쁜 코드는 개발 속도를 크게 떨어뜨린다. 프로젝트 초반에는 번개처럼 나가다가 1~2년 만에 굼뱅이처럼 기어간다.

나쁜 코드가 쌓이는 과정:
1. 코드를 고칠 때마다 엉뚱한 곳에서 문제 발생
2. 간단한 변경은 없음 — 매번 얽히고설킨 코드를 '해독'해서 더 얽힌 코드를 추가
3. 쓰레기 더미는 점점 높아지고 깊어지고 커짐
4. 팀 생산성은 떨어져 마침내 **0에 근접**
5. 관리층이 인력을 추가 투입하지만, 새 인력은 설계 의도를 모른 채 나쁜 코드를 더 양산
6. 생산성은 더더욱 떨어짐

### 원대한 재설계의 꿈

결국 팀이 반기를 들고 재설계를 요구한다. 새로운 **타이거 팀**이 구성되어 기존 시스템의 모든 기능을 제공하는 새 시스템을 만들기 시작한다.

하지만 이 경주는 **10년 이상** 걸리기도 한다. 새 시스템이 기존 시스템을 따라잡을 즈음이면 초창기 팀원들은 모두 떠나고, 새 팀원들이 또다시 재설계를 요구한다. 현재 시스템이 너무 엉망이라서.

> **핵심 통찰**: 시간을 들여 깨끗한 코드를 만드는 노력은 비용을 절감하는 방법일 뿐만 아니라, 전문가로서 살아남는 길이다.

---

## 4. 태도

코드가 엉망이 된 이유를 우리는 온갖 핑계로 돌린다 — 요구사항 변경, 촉박한 일정, 멍청한 관리자, 쓸모없는 마케팅 부서. **하지만 잘못은 전적으로 프로그래머에게 있다.**

관리자와 마케팅은 약속과 공약을 내걸며 우리에게 정보를 구한다. 일정과 요구사항을 강력하게 밀어붙이는 것이 그들의 책임이라면, **좋은 코드를 사수하는 것은 프로그래머의 책임**이다.

> **Uncle Bob의 경험**: 의사에 비유하면 이해하기 쉽다. 환자가 수술 전에 손을 씻지 말라고 요구한다 — 시간이 너무 걸리니까. 하지만 의사는 단호하게 거부한다. 질병과 감염의 위험은 환자보다 의사가 더 잘 아니까. 프로그래머도 마찬가지다. 나쁜 코드의 위험을 이해하지 못하는 관리자 말을 그대로 따르는 행동은 전문가답지 못하다.

---

## 5. 원초적 난제

프로그래머가 직면하는 근본적 난제가 있다:

- **사실**: 나쁜 코드가 업무 속도를 늦춘다
- **착각**: 기한을 맞추려면 나쁜 코드를 양산할 수밖에 없다

진짜 전문가는 두 번째 부분이 **틀렸다**는 사실을 안다. 나쁜 코드를 양산하면 속도가 곧바로 늦어져 결국 기한을 놓친다.

> **핵심 통찰**: 기한을 맞추는 유일한 방법은, 빨리 가는 유일한 방법은, 언제나 코드를 최대한 깨끗하게 유지하는 습관이다.

---

## 6. 깨끗한 코드라는 예술?

깨끗한 코드를 구현하는 행위는 그림을 그리는 행위와 비슷하다. 잘 그린 그림을 구분하는 능력이 그림을 잘 그리는 능력은 아니다. 마찬가지로, 깨끗한 코드와 나쁜 코드를 구분할 줄 안다고 깨끗한 코드를 작성할 줄 안다는 뜻은 아니다.

깨끗한 코드를 작성하려면 **'코드 감각(code-sense)'** 이 필요하다:

| 코드 감각이 없는 프로그래머 | 코드 감각이 있는 프로그래머 |
|---|---|
| 나쁜 모듈을 알아본다 | 나쁜 모듈을 보면 좋은 모듈로 개선할 방안을 떠올린다 |
| 그것으로 끝이다 | 최고 방안을 선택하고 이동 경로를 계획한다 |

---

## 7. 깨끗한 코드란?

저자는 소프트웨어 분야의 유명 프로그래머들에게 깨끗한 코드의 정의를 물었다.

### 7.1 비야네 스트롭스트룹 (C++ 창시자)

> *"나는 우아하고 효율적인 코드를 좋아한다. 논리가 간단해야 버그가 숨어들지 못한다. 의존성을 최대한 줄여야 유지보수가 쉬워진다. 오류는 명백한 전략에 의거해 철저히 처리한다. 성능을 최적으로 유지해야 사람들이 원칙 없는 최적화로 코드를 망치려는 유혹에 빠지지 않는다. **깨끗한 코드는 한 가지를 제대로 한다.**"*

핵심 키워드 분석:

| 키워드 | 의미 |
|---|---|
| **우아한(elegant)** | 보기에 즐거운 코드. 잘 만든 오르골이나 잘 디자인된 차처럼 |
| **효율적** | CPU 자원 낭비 없이. 단순히 속도만이 아닌 전반적 효율 |
| **유혹** | 나쁜 코드는 나쁜 코드를 '유혹'한다 — 깨진 창문 이론(*Broken Window — 창문이 깨진 건물은 누구도 상관하지 않는다는 인상을 풍기고, 사람들도 관심을 끊어 쇠퇴가 시작된다는 이론. Dave Thomas와 Andy Hunt가 『실용주의 프로그래머』에서 소개했다.*) |
| **한 가지** | 각 함수, 클래스, 모듈은 주변 상황에 오염되지 않은 채 한길만 걷는다 |

### 7.2 그래디 부치 (OOAD 저자)

> *"깨끗한 코드는 단순하고 직접적이다. 깨끗한 코드는 잘 쓴 문장처럼 읽힌다. 깨끗한 코드는 결코 설계자의 의도를 숨기지 않는다. 오히려 **명쾌한 추상화**와 단순한 제어문으로 가득하다."*

그래디는 **가독성**을 강조한다. 코드는 추측이 아니라 사실에 기반해야 하며, 반드시 필요한 내용만 담아야 한다.

### 7.3 '큰' 데이브 토마스 (OTI 창립자, 이클립스 전략의 대부)

> *"깨끗한 코드는 작성자가 아닌 사람도 읽기 쉽고 고치기 쉽다. 단위 테스트 케이스와 인수 테스트 케이스가 존재한다. 깨끗한 코드에는 의미 있는 이름이 붙는다. 특정 목적을 달성하는 방법은 (여러 가지가 아니라) 하나만 제공한다."*

데이브의 핵심 반전: 읽기 쉬운 코드와 **고치기 쉬운 코드**는 엄연히 다르다. 그리고 **테스트 케이스가 없는 코드는 깨끗한 코드가 아니다** — 아무리 우아하고 가독성이 높아도.

### 7.4 마이클 페더스 (Working Effectively with Legacy Code 저자)

> *"깨끗한 코드의 특징은 많지만 그 중에서도 모두를 아우르는 특징이 하나 있다. 깨끗한 코드는 언제나 **누군가 주의 깊게 짰다는 느낌**을 준다. 고치려고 살펴봐도 딱히 손 댈 곳이 없다."*

한 마디로 요약하면 **'주의(care)'**. 깨끗한 코드는 누군가 시간을 들여 깔끔하고 단정하게 정리한, 세세한 사항까지 꼼꼼하게 신경 쓴 코드다.

### 7.5 론 제프리스 (Extreme Programming 저자)

론은 켄트 벡의 **단순한 코드 규칙**으로 구현을 시작하고 거의 끝낸다:

1. **모든 테스트를 통과한다**
2. **중복이 없다**
3. **시스템 내 모든 설계 아이디어를 표현한다**
4. **클래스, 메서드, 함수 등을 최대한 줄인다**

론의 깨끗한 코드 비결 세 가지:
- **중복 줄이기** — 같은 작업을 여러 차례 반복하면 아이디어를 제대로 표현하지 못한다는 증거
- **표현력 높이기** — 의미 있는 이름, 기능별 객체/메서드 분리
- **초반부터 간단한 추상화 고려하기** — 집합에서 항목을 찾는 등의 공통 패턴을 추상화

### 7.6 워드 커닝햄 (Wiki 창시자, XP 공동 창시자)

> *"코드를 읽으면서 짐작했던 기능을 각 루틴이 그대로 수행한다면 깨끗한 코드라 불러도 되겠다. 코드가 그 문제를 풀기 위한 **언어처럼 보인다면** 아름다운 코드라 불러도 되겠다."*

워드의 정의에서 핵심:
- 깨끗한 코드는 **읽으면서 놀랄 일이 없다** — 짐작한 대로 돌아간다
- 각 모듈은 다음 무대를 준비한다 — 다음에 벌어질 상황이 보인다
- 언어를 단순하게 보이도록 만드는 열쇠는 **프로그래머**다

### 대가들의 정의 종합

| 대가 | 핵심 키워드 | 강조점 |
|---|---|---|
| 비야네 스트롭스트룹 | 우아함, 효율, 한 가지 | 논리의 단순함, 철저한 오류 처리 |
| 그래디 부치 | 가독성, 명쾌한 추상화 | 잘 쓴 문장처럼 읽히는 코드 |
| 데이브 토마스 | 테스트, 최소 의존성 | 고치기 쉬움, 테스트 필수 |
| 마이클 페더스 | 주의(care) | 누군가 꼼꼼하게 신경 쓴 느낌 |
| 론 제프리스 | 중복 제거, 표현력 | 단순한 코드 규칙 |
| 워드 커닝햄 | 놀라움 없음, 언어처럼 | 짐작한 대로 동작하는 코드 |

## 8. 우리는 저자다

`@author` 필드가 말해주듯 우리는 **저자**다. 저자에게는 독자와 잘 소통할 책임이 있다.

코드를 읽는 시간 대 코드를 짜는 시간 비율은 **10:1을 훌쩍 넘는다**. 편집 세션을 재생해 보면, 대부분이 화면을 스크롤하거나 다른 모듈을 찾아보는 동작이다:

```
밥이 모듈을 입력한다.
변경할 함수로 스크롤해 내려간다.
잠시 멈추고 생각한다.
이런! 모듈 상단으로 스크롤해 변수 초기화를 확인한다.
다시 내려와 입력하기 시작한다.
이런, 입력을 지운다!
다시 입력한다. 다시 지운다!
지금 바꾸려는 함수를 호출하는 함수로 스크롤한 후 호출 방식을 살펴본다.
다시 돌아와 방금 지운 코드를 입력한다.
...
```

> **핵심 통찰**: 새 코드를 짜면서 우리는 끊임없이 기존 코드를 읽는다. 읽기 쉽게 만들면 사실은 짜기도 쉬워진다. 급하다면, 서둘러 끝내려면, 쉽게 짜려면 — **읽기 쉽게 만들면 된다.**

## 9. 보이스카우트 규칙

잘 짠 코드가 전부는 아니다. 시간이 지나도 언제나 깨끗하게 유지해야 한다. 미국 보이스카우트의 규칙이 우리에게도 유용하다:

> **캠프장은 처음 왔을 때보다 더 깨끗하게 해놓고 떠나라.**

체크아웃할 때보다 좀 더 깨끗한 코드를 체크인한다면 코드는 절대 나빠지지 않는다. 한꺼번에 많은 시간과 노력을 투자할 필요 없다:

- 변수 이름 하나를 개선하고
- 조금 긴 함수 하나를 분할하고
- 약간의 중복을 제거하고
- 복잡한 if 문 하나를 정리하면 충분하다

> **핵심 통찰**: 지속적인 개선이야말로 전문가 정신의 본질이다. 시간이 지날수록 코드가 좋아지는 프로젝트에서 작업한다고 상상해보라!

## 10. 프리퀄과 원칙

이 책은 Robert C. Martin의 *Agile Software Development: Principles, Patterns, and Practices(PPP)*의 프리퀄이다. PPP는 객체 지향 설계의 원칙을 설명하고 전문 개발자들이 사용하는 실무 기법을 소개한다.

이 책에서 산발적으로 거론하는 설계 원칙:

| 원칙 | 설명 |
|---|---|
| **SRP** (단일 책임 원칙) | 클래스에는 한 가지, 단 한 가지 변경 이유만 존재해야 한다 |
| **OCP** (개방-폐쇄 원칙) | 클래스는 확장에 열려 있어야 하며 변경에 닫혀 있어야 한다 |
| **LSP** (리스코프 치환 원칙) | 상속받은 클래스는 기초 클래스를 대체할 수 있어야 한다 |
| **DIP** (의존성 역전 원칙) | 추상화에 의존해야 하며, 구체화에 의존하면 안 된다 |
| **ISP** (인터페이스 분리 원칙) | 클라이언트에 밀접하게 작게 쪼개진 인터페이스를 유지한다 |

## 요약

- **코드는 사라지지 않는다** — 요구사항을 정밀하게 표현하는 유일한 수단이다
- **나쁜 코드는 회사를 망하게 한다** — 르블랑의 법칙: 나중은 결코 오지 않는다
- **나쁜 코드의 대가**: 생산성 0 수렴 → 인력 추가 → 더 나쁜 코드 → 재설계의 악순환
- **좋은 코드를 사수하는 것은 프로그래머의 책임**이다 — 관리자 탓이 아니다
- **빨리 가는 유일한 방법은 깨끗한 코드를 유지하는 것**이다
- 깨끗한 코드의 공통 속성: **우아함, 가독성, 테스트, 주의, 중복 없음, 표현력, 최소화**
- **코드 읽기:쓰기 = 10:1** — 읽기 쉬운 코드가 짜기도 쉽다
- **보이스카우트 규칙**: 체크아웃할 때보다 더 깨끗하게 체크인하라

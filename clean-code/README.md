# Clean Code

**저자**: Robert C. Martin (Uncle Bob)
**출판**: 2008, Prentice Hall / 한국어판 2013, 인사이트 (옮긴이: 박재호, 이해영)

---

## 책 소개

"Clean Code: A Handbook of Agile Software Craftsmanship"은 깨끗하고 읽기 좋은 코드를 작성하는 방법을 다루는 소프트웨어 개발의 고전이다. Robert C. Martin(Uncle Bob)이 수십 년간의 경험에서 추출한 코드 품질의 원칙, 패턴, 실천법을 코드 예제와 함께 제시한다.

이 책의 핵심 전제는 단순하다:

> **코드를 읽는 시간 대 코드를 짜는 시간 비율은 10:1을 훌쩍 넘는다. 읽기 쉽게 만들면 사실은 짜기도 쉬워진다.**

### 이 책의 특징

- **실전 코드 중심**: 추상적인 원칙 나열이 아니라, 실제 코드를 개선하는 과정을 보여준다. 나쁜 코드와 좋은 코드를 대비하며 "왜" 나쁘고 "어떻게" 고치는지를 구체적으로 설명한다.
- **3부 구조**: Part I에서 원칙을 배우고, Part II에서 실제 코드를 리팩터링하는 사례 연구를 거치고, Part III에서 냄새와 휴리스틱 체크리스트를 정리한다.
- **보이스카우트 규칙**: "캠프장은 처음 왔을 때보다 더 깨끗하게 해놓고 떠나라" — 코드를 체크아웃할 때보다 더 깨끗하게 체크인하는 지속적 개선의 철학을 관통한다.
- **대가들의 지혜**: 비야네 스트롭스트룹, 그래디 부치, 워드 커닝햄 등 소프트웨어 분야 거장들의 깨끗한 코드 정의를 종합한다.

### Clean Code vs Clean Coder vs Clean Architecture

| 구분 | Clean Code (2008) | The Clean Coder (2011) | Clean Architecture (2017) |
|------|-------------------|------------------------|---------------------------|
| 핵심 주제 | 깨끗한 **코드** 작성법 | 프로 **개발자**의 행동 규범 | 깨끗한 **시스템 구조** 설계법 |
| 다루는 범위 | 이름, 함수, 클래스, 테스트, 리팩터링 | 거절하기, 추정, 시간 관리, 협업 | 컴포넌트, 경계, 의존성, 배포 |
| 대상 독자 | 코드 품질을 높이고 싶은 개발자 | 프로로서의 태도를 갖추고 싶은 개발자 | 시스템 설계를 이해하고 싶은 개발자/아키텍트 |
| 핵심 질문 | "좋은 코드란 무엇인가?" | "프로 개발자란 어떤 사람인가?" | "좋은 아키텍처란 무엇인가?" |
| 형식 | 코드 예제와 리팩터링 사례 중심 | 일화(anecdote)와 대화 시나리오 중심 | 원칙과 다이어그램 중심 |

## 이 저장소의 목적

이 저장소는 "Clean Code"의 내용을 챕터별로 상세하게 정리한 학습 자료이다. 각 챕터의 핵심 원칙, 코드 예시(나쁜 예 vs 좋은 예), 휴리스틱을 책을 대체할 수 있을 정도로 충실하게 담고 있으며, 다음과 같은 용도로 활용할 수 있다:

- 깨끗한 코드 작성의 원칙을 체계적으로 학습하는 교재
- 코드 리뷰 시 참조하는 체크리스트
- 팀 내 코딩 컨벤션과 품질 기준을 논의하는 토론 자료

## 학습 가이드

### 추천 읽기 순서

이 책은 세 파트로 구성되어 있으며, Part I → Part II → Part III 순서로 읽는 것이 자연스럽다. 하지만 특정 주제에 관심이 있다면 아래 가이드를 참고할 수 있다.

**1단계: 마인드셋 — 왜 깨끗한 코드인가 (Chapter 1)**

모든 원칙의 기반이 되는 "깨끗한 코드"의 정의를 이해한다. 대가들의 정의, 보이스카우트 규칙, 르블랑의 법칙을 먼저 파악한다. 이후 모든 챕터가 이 철학 위에 쌓이므로 반드시 먼저 읽는다.

**2단계: 코드의 기초 단위 — 이름, 함수, 주석, 형식 (Chapter 2~5)**

코드를 구성하는 가장 작은 단위인 변수명, 함수 설계, 주석의 올바른 사용법, 코드 포맷팅을 다룬다. 일상적인 코딩에서 즉시 적용할 수 있는 실천법을 배운다.

**3단계: 설계 수준 — 객체, 오류, 경계, 테스트 (Chapter 6~9)**

코드의 구조적 설계를 다룬다. 객체와 자료 구조의 차이, 예외 처리 전략, 외부 코드와의 경계 관리, 단위 테스트의 원칙을 배운다.

**4단계: 시스템 수준 — 클래스, 시스템, 창발성, 동시성 (Chapter 10~13)**

더 큰 단위의 설계를 다룬다. 클래스 설계, 시스템 수준의 관심사 분리, 단순한 설계 규칙, 동시성 프로그래밍의 원칙을 배운다.

**5단계: 실전 리팩터링 — 사례 연구 (Chapter 14~16)**

실제 코드를 점진적으로 개선하는 과정을 따라간다. 명령행 인수 파서(Args), JUnit 내부 코드, SerialDate 클래스를 리팩터링하며, Part I에서 배운 원칙이 어떻게 적용되는지 체험한다.

**6단계: 체크리스트 — 냄새와 휴리스틱 (Chapter 17)**

책 전체에서 도출된 코드 냄새와 휴리스틱을 7개 카테고리(주석, 환경, 함수, 일반, 자바, 이름, 테스트)로 종합 정리한 챕터다. 코드 리뷰 시 참조하기에 적합하다.

## 핵심 개념 요약

이 책의 모든 내용은 하나의 핵심 가치로 수렴한다:

> **깨끗한 코드는 한 가지를 제대로 한다. 누군가 주의 깊게 짰다는 느낌을 주며, 읽으면서 짐작했던 기능을 그대로 수행한다.**

### 깨끗한 코드의 5대 원칙

| 원칙 | 설명 | 관련 챕터 |
|------|------|-----------|
| **의도를 드러내라** | 이름, 함수, 클래스가 스스로 자신의 목적을 설명해야 한다. 주석이 필요하다면 코드가 충분하지 않다는 뜻이다. | Ch 2, 4, 17(N) |
| **한 가지만 하라** | 함수는 한 가지 작업만 수행하고, 클래스는 한 가지 책임만 진다. 작게 만들어 이해하기 쉽게 하라. | Ch 3, 10, 17(G30) |
| **중복을 제거하라** | DRY(Don't Repeat Yourself) — 중복은 추상화의 기회다. 같은 코드가 반복되면 함수, 클래스, 패턴으로 추출하라. | Ch 12, 17(G5) |
| **테스트로 보호하라** | 테스트 없는 코드는 깨끗한 코드가 아니다. 모든 변경은 테스트가 뒷받침해야 한다. | Ch 9, 17(T) |
| **지속적으로 개선하라** | 보이스카우트 규칙 — 코드를 체크아웃할 때보다 더 깨끗하게 체크인하라. 완벽한 코드를 처음부터 짜는 것이 아니라, 끊임없이 리팩터링하라. | Ch 1, 14~16 |

### 코드 냄새 카테고리 (Chapter 17)

| 카테고리 | 코드 범위 | 핵심 질문 |
|----------|----------|-----------|
| **주석 (C1~C5)** | 주석의 품질과 필요성 | 이 주석은 코드가 대신할 수 없는 정보를 제공하는가? |
| **환경 (E1~E2)** | 빌드와 테스트 환경 | 한 명령으로 빌드하고 테스트할 수 있는가? |
| **함수 (F1~F4)** | 함수의 설계 | 인수는 적은가? 한 가지만 하는가? |
| **일반 (G1~G36)** | 전반적인 코드 품질 | 중복은 없는가? 추상화 수준은 올바른가? |
| **자바 (J1~J3)** | 자바 특화 관행 | enum과 static import를 활용하는가? |
| **이름 (N1~N7)** | 명명 규칙 | 이름만으로 의도가 드러나는가? |
| **테스트 (T1~T9)** | 테스트 품질 | 깨질 만한 부분을 모두 테스트하는가? |

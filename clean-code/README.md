# 클린 코드 (Clean Code: A Handbook of Agile Software Craftsmanship)

> *Clean Code: A Handbook of Agile Software Craftsmanship* (Robert C. Martin, Prentice Hall, 2008)
> 한국어판: 『Clean Code 클린 코드 — 애자일 소프트웨어 장인 정신』 (박재호·이해영 옮김, 인사이트, 2013)

깨끗하고 읽기 좋은 코드를 작성하는 방법을 다루는 소프트웨어 개발의 고전. 핵심 전제는 단순하다 — **코드를 읽는 시간 대 짜는 시간 비율은 10:1을 훌쩍 넘는다. 읽기 쉽게 만들면 사실은 짜기도 쉬워진다.**

## 개요

추상적인 원칙 나열이 아니라 실제 코드를 개선하는 과정을 보여준다 — 나쁜 코드와 좋은 코드를 대비하며 "왜" 나쁘고 "어떻게" 고치는지를 구체적으로 설명한다. Part I에서 원칙을 배우고, Part II에서 실제 코드 리팩터링 사례 연구를 거치고, Part III에서 냄새와 휴리스틱 체크리스트로 종합한다.

전체를 관통하는 철학은 **보이스카우트 규칙** — "캠프장은 처음 왔을 때보다 더 깨끗하게 해놓고 떠나라." 『클린 코드』(코드) → 『클린 코더』(태도) → 『클린 아키텍처』(구조) 시리즈의 첫 책이다.

## 학습 가이드

1. **1단계 (Ch1)**: 깨끗한 코드의 정의·보이스카우트 규칙·르블랑의 법칙 — 모든 챕터의 철학적 기반
2. **2단계 (Ch2~5)**: 코드의 기초 단위 — 이름·함수·주석·형식. 일상 코딩에 즉시 적용
3. **3단계 (Ch6~9)**: 설계 수준 — 객체/자료 구조 구분, 오류 처리, 외부 경계, 단위 테스트
4. **4단계 (Ch10~13)**: 시스템 수준 — 클래스·시스템·창발적 설계·동시성
5. **5단계 (Ch14~16)**: 사례 연구 — 원칙이 실제 리팩터링에 적용되는 과정 체험
6. **6단계 (Ch17)**: 냄새와 휴리스틱 — 코드 리뷰용 체크리스트로 상시 참조

## 핵심 개념 맵

### 깨끗한 코드의 5대 원칙

| 원칙 | 설명 | 관련 챕터 |
|------|------|-----------|
| **의도를 드러내라** | 이름·함수·클래스가 스스로 목적을 설명해야 한다. 주석이 필요하다면 코드가 충분하지 않다는 뜻 | Ch 2, 4, 17(N) |
| **한 가지만 하라** | 함수는 한 가지 작업만, 클래스는 한 가지 책임만. 작게 만들어 이해하기 쉽게 | Ch 3, 10, 17(G30) |
| **중복을 제거하라** | DRY — 중복은 추상화의 기회다. 반복되면 함수·클래스·패턴으로 추출 | Ch 12, 17(G5) |
| **테스트로 보호하라** | 테스트 없는 코드는 깨끗한 코드가 아니다. 모든 변경은 테스트가 뒷받침 | Ch 9, 17(T) |
| **지속적으로 개선하라** | 보이스카우트 규칙 — 체크아웃할 때보다 깨끗하게 체크인. 끊임없는 리팩터링 | Ch 1, 14~16 |

### 코드 냄새 카테고리 (Ch 17)

| 카테고리 | 코드 범위 | 핵심 질문 |
|----------|----------|-----------|
| **주석 (C1~C5)** | 주석의 품질과 필요성 | 이 주석은 코드가 대신할 수 없는 정보를 제공하는가? |
| **환경 (E1~E2)** | 빌드와 테스트 환경 | 한 명령으로 빌드하고 테스트할 수 있는가? |

| **일반 (G1~G36)** | 전반적인 코드 품질 | 중복은 없는가? 추상화 수준은 올바른가? |
| **자바 (J1~J3)** | 자바 특화 관행 | enum과 static import를 활용하는가? |

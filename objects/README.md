# 오브젝트 (Objects: 코드로 이해하는 객체지향 설계)

> 『오브젝트 — 코드로 이해하는 객체지향 설계』 (조영호 지음, 위키북스, 2019)

객체지향이란 무엇인가라는 질문에 대한 조영호의 두 번째 대답. 전작 『객체지향의 사실과 오해』가 은유와 사고실험으로 역할·책임·협력을 풀었다면, 이 책은 **코드**로 객체지향 설계의 핵심 원칙을 증명한다.

## 책 정보

| 항목 | 내용 |
|------|------|
| **저자** | 조영호 |
| **출판** | 위키북스, 2019 |
| **예제 언어** | Java (노트에는 TypeScript 병기) |
| **대상 독자** | 객체지향 프로그래밍 경험이 있는 개발자 |

## 개요

티켓 판매 시스템, 영화 예매 시스템, 전화 요금 계산 시스템이라는 세 가지 예제를 반복적으로 발전시키며, 절차적 코드가 왜 문제인지, 캡슐화와 다형성이 어떻게 변경에 유연한 설계를 만드는지를 단계별로 보여준다. 추상적 원칙을 코드로 증명하고 나쁜 설계와 좋은 설계를 나란히 비교하며 "왜 그래야 하는가"의 직관을 길러준다.

《[객체지향의 사실과 오해](../the-essence-of-object-orientation-객체지향의-사실과-오해/README.md)》의 후속작이다. 전작이 객체지향의 첫 두 걸음(객체를 바라보는 것, 협력 공동체로 바라보는 것)을 다뤘다면, 이 책은 세 번째·네 번째 걸음(적절한 역할과 책임 부여, 프로그래밍 언어로 표현하는 기술)에 초점을 맞춘다. 두 책은 독립적으로 읽을 수 있다.

## 핵심 개념 맵

- **객체지향 = 협력하는 객체들의 공동체** — 중심은 클래스가 아니라 협력·책임·역할 (Ch1~3)
- **설계 품질의 기준은 '변경'** — 낮은 결합도·높은 응집도는 변경 비용으로 판단한다 (Ch4)
- **캡슐화 = 변하는 것을 숨기기** — 데이터가 아니라 변경 가능성을 숨긴다. 인터페이스에만 의존하게 (Ch4·6·8)
- **책임 주도 설계 → GRASP**: 책임을 수행할 정보를 가진 전문가에게 할당 (Ch3·5)
- **상속은 타입 계층(서브타이핑) 목적으로, 코드 재사용은 합성으로** (Ch10~13)
- **추상화에 의존하라** — OCP·DIP·DI·패턴·프레임워크 모두 같은 원리의 변주 (Ch9·15)
- **유연성에는 복잡성 비용이 따른다** — 변경이 필요할 때만 유연하게 (Ch4·9의 트레이드오프)

## 시그니처 요소와 표기 규칙

- `## 리팩터링 과정`: 나쁜 코드 → 좋은 코드 단계별 개선 추적 테이블
- `## 설계 원칙`: 챕터별 OOP 설계 원칙 정리 테이블
- `### 나쁜 설계` / `### 좋은 설계`: before/after 코드 대비
- **클래스 다이어그램**: ASCII/유니코드 박스 문자로 클래스 관계 시각화
- `> **핵심 통찰**:` 콜아웃 / 레이블 없는 `>` 인용문(저자 경험담·외부 인용)
- **코드 예제**: Java(원본)는 `<details>` 접기, TypeScript(변환)는 항상 펼침

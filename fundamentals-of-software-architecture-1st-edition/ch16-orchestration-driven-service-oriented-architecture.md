# Chapter 16: Orchestration-Driven Service-Oriented Architecture (오케스트레이션 기반 서비스 지향 아키텍처 스타일)

## 핵심 질문

1990년대 후반 어떤 외부 여건이 이 아키텍처를 낳았는가? "엔터프라이즈 레벨 재사용"이라는 중심 철학은 왜 논리적으로 보였지만 재앙이 되었는가? 서비스 택소노미(비즈니스·엔터프라이즈·애플리케이션·인프라)와 오케스트레이션 엔진은 무엇인가? 이 스타일의 실패가 왜 마이크로서비스를 낳았는가?

---

## 1. 역사와 철학 — 시대의 맥락

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍처 스타일은 예술처럼 **그것이 진화한 시대의 맥락**에서 이해해야 한다. 이 아키텍처는 외부 요인들이 조합되어, 얼핏 논리적으로 보이지만 궁극적으로는 재앙적인 조직 철학과 결부되어 탄생한 사례다.
:::

서비스 지향 아키텍처(SOA)는 1990년대 후반 등장했다. 당시 여건:
- 대기업이 인수합병으로 급성장 → 세련된 IT가 필요했지만 컴퓨팅 리소스가 부족하고 상용 제품뿐
- 운영체제·상용 DB 라이선스가 시스템 단위로 비쌌다

그 결과 아키텍트는 **"최대한 재사용"** 을 목표로 삼았고, **모든 형태의 재사용이 이 아키텍처의 중심 철학**이 되었다. 이 스타일은 **아키텍트가 기술 분할에 집착하면 어떻게 되는지** 잘 보여주는 사례다 — 취지는 좋았으나 결과는 좋지 않았다.

---

## 2. 토폴로지와 택소노미

아키텍처 내부에 **서비스 택소노미(taxonomy, 분류 체계)** 를 정립하여 레이어별로 책임을 지운다.

```
        [비즈니스 서비스]  ← 진입점, 도메인 행위, 코드 없음(입력/출력/스키마)
              ↓
      ┌─ 오케스트레이션 엔진 (ESB) ─┐
              ↓
        [엔터프라이즈 서비스]  ← 세분화된 공유 구현체, 원자적 행위
              ↓
        [애플리케이션 서비스]  ← 한 번만 쓰는 단일 구현체
              ↓
        [인프라 서비스]  ← 모니터링·로깅·인증
```

| 서비스 유형 | 역할 |
|---|---|
| **비즈니스 서비스(business service)** | 최상단 진입점. `ExecuteTrade`·`PlaceOrder` 등 도메인 행위. 코드 없이 입력·출력·스키마만. 리트머스 시험: "우리가 …비즈니스를 하고 있나요?"에 확답 가능한지 |
| **엔터프라이즈 서비스(enterprise service)** | 세분화된 공유 구현체. `CreateCustomer`·`CalculateQuote` 같은 원자적 행위. **재사용 자산**의 핵심 |
| **애플리케이션 서비스(application service)** | 한 번만 쓰는 단일 구현체. 재사용 가치가 없는 지리공간 정보 등. 한 팀이 소유 |
| **인프라 서비스(infrastructure service)** | 모니터링·로깅·인증/인가 등 운영 관심사 |

### 2.1 오케스트레이션 엔진

**오케스트레이션 엔진(orchestration engine, ESB)** 은 이 분산 아키텍처의 요체다. 비즈니스 서비스를 엮고 **트랜잭션 조정·메시지 변환**을 수행한다. 서비스마다 DB가 있는 게 아니라 단일 관계형 DB를 쓰며, 트랜잭셔널 로직은 DB가 아닌 **엔진에서 선언적으로** 처리된다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 콘웨이의 법칙에 따르면, 엔진을 담당한 **통합 아키텍트 팀이 조직에서 정치 세력화되고 결국 관료적 병목(bureaucratic bottleneck)이 된다.** 트랜잭셔널 로직을 오케스트레이션 도구에 위임한다는 아이디어는 그럴듯했으나, 트랜잭션을 어느 레벨까지 세분화할지 알아내기가 더욱 어려워져 아키텍처가 한층 복잡해졌다.
:::

모든 요청은 엔진을 흘러간다 — 내부 호출조차 엔진을 경유한다.

## 3. 재사용... 그리고 커플링

주된 목표는 서비스 레벨 재사용이다. 보험회사의 모든 부서에 `Customer` 개념이 있으면, 이를 하나의 재사용 가능한 `Customer` 서비스로 빼내 모두가 참조하게 한다. 재사용 목표는 확실히 달성했지만 어두운 그림자가 드러난다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 재사용 위주로 구축하면 **컴포넌트 간 커플링이 심해진다.** `Customer` 서비스를 함부로 바꾸면 이를 참조하는 모든 서비스에 영향을 준다. 사소한 변경 하나도 여파가 커서 아키텍트가 점진적 변화를 추구하기 어렵고, 전체 테스트·배포 조정이 필요해진다.
:::

또 다른 부작용:
- **로직 통합의 대가**: 하나의 `Customer` 서비스에 모든 상세 정보를 담아야 한다. 자동차 보험용 운전면허증 세부 정보까지 넣으면, 장애 보험 팀도 관심 없는 그 복잡성을 떠안는다
- **도메인이 너무 얇게 퍼짐**: `CatalogCheckout` 같은 개념이 아키텍처 전체에 얇게 흩어져, "주소 라인 추가" 같은 단순 작업조차 여러 티어의 수십 개 서비스 + 단일 DB 스키마 변경을 유발한다

| 모듈성 | ★★★☆☆ |
| 전체 비용 | ★☆☆☆☆ |

| 신뢰성 | ★★★☆☆ |
| 확장성 | ★★★☆☆ |
| 단순성 | ★☆☆☆☆ |
| 시험성 | ★☆☆☆☆ |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: SOA는 **지금까지 시도된 아키텍처 중 가장 기술적으로 분할된 범용 아키텍처**일 것이다. 이 구조의 단점에 대한 반발로 **마이크로서비스**가 탄생했다. 분산 아키텍처지만 (1) 단일 DB 커플링, (2) 모든 것을 조정하는 오케스트레이션 엔진이라는 거대한 커플링 지점 때문에 **퀀텀은 1** — 모놀리식과 분산 아키텍처 양쪽의 단점을 모두 갖는다.
:::

- **배포성·시험성·단순성·비용·진화성 1개**: 태동 시대엔 이런 현대적 목표가 중요하지 않았다. 커플링 탓에 점진적 변화가 불가능
- **성능 1개**: 각 비즈니스 요청이 아키텍처 여러 부분에 흩어져 있고 모든 호출이 엔진을 경유해 정말 형편없다
- **탄력성·확장성·내고장성·모듈성·신뢰성 3개**: 벤더가 세션 복제 등으로 확장성을 일부 구현했으나 구현이 까다롭다

## 요약

- SOA는 **1990년대 후반 리소스 부족 + "최대한 재사용" 철학**의 산물 — 시대 맥락에서 이해해야 한다
- 서비스 택소노미: **비즈니스**(진입점) → **엔터프라이즈**(재사용 원자 행위) → **애플리케이션**(일회성) → **인프라**(운영). **오케스트레이션 엔진(ESB)** 이 모든 것을 조정
- **재사용 집착 → 극심한 커플링**: `Customer` 서비스 변경이 전체에 파급, 도메인이 너무 얇게 퍼져 단순 변경도 수십 서비스에 영향
- 거의 모든 현대적 특성에서 최하점. **모놀리식과 분산의 단점을 모두** 가진 퀀텀 1 아키텍처
- 이 실패가 **분산 트랜잭션의 어려움과 기술 분할의 한계**를 가르쳐 마이크로서비스를 낳은 중요한 이정표

# Chapter 15: Space-Based Architecture Style (공간 기반 아키텍처 스타일)

## 핵심 질문

웹 애플리케이션은 왜 사용자가 늘면 결국 데이터베이스가 병목이 되는가? 공간 기반 아키텍처는 중앙 데이터베이스를 어떻게 제거하고 "거의 무한한 확장성"을 얻는가? 처리 장치·가상 미들웨어·데이터 펌프는 각각 무엇을 하는가? 왜 탄력성·확장성·성능은 최고점인데 단순성·시험성·비용은 최악인가?

---

## 1. 개요 — 삼각형 토폴로지의 한계

웹 애플리케이션은 브라우저 → 웹 서버 → 애플리케이션 서버 → 데이터베이스 순서로 요청이 흐른다. 사용자가 늘면 병목이 웹 서버 → 애플리케이션 서버 → **데이터베이스**로 이동한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 확장이 가장 쉬운 웹 서버가 가장 긴 변, 확장이 가장 어려운 데이터베이스가 가장 짧은 변인 **삼각형 토폴로지(triangle-shaped topology)** 가 그려진다. 대용량 애플리케이션은 결국 **데이터베이스의 동시 트랜잭션 처리 한계**가 최종 제약이 된다.
:::

공간 기반 아키텍처(*Space-Based Architecture - 튜플 공간에서 유래. 중앙 DB를 제거하고 인메모리 데이터 그리드로 확장하는 아키텍처*)는 이 문제를 **아키텍처적으로** 해결한다. 동시 사용자 수가 매우 가변적이고 예측조차 곤란한 애플리케이션에 유용하다.

---

## 2. 토폴로지

명칭은 **튜플 공간(tuple space)** — 공유 메모리로 통신하는 다중 병렬 프로세서 기술 — 에서 유래했다. 동기 제약인 **중앙 데이터베이스를 없애고**, 복제된 **인메모리 데이터 그리드(in-memory data grid)** 를 활용한다.

```
   [처리 장치]  [처리 장치]  [처리 장치]   ← 인메모리 그리드 + 복제 엔진
        \          |          /
   ┌──────────────────────────────────┐
   │           가상 미들웨어            │
   │ 메시징 · 데이터 · 처리 그리드 · 배포관리자 │
   └──────────────────────────────────┘
        (데이터 펌프 ↓ 비동기)
   [데이터 라이터] → [데이터베이스] ← [데이터 리더]
```

- 데이터는 메모리에 두고 모든 활성 처리 장치가 복제한다. 업데이트는 **퍼시스턴트 큐**로 DB에 **비동기 전송**한다
- 중앙 DB가 표준 트랜잭션 처리에 관여하지 않으므로 병목이 사라지고 **거의 무한에 가까운 확장성**이 보장된다

### 2.1 다섯 컴포넌트

1. **처리 장치(processing unit)**: 애플리케이션 로직 + 인메모리 데이터 그리드/복제 엔진(헤이즐캐스트·아파치 이그나이트·오라클 코히어런스). 부하에 따라 동적으로 시작/종료
2. **가상 미들웨어(virtualized middleware)**: 데이터 동기화·요청 처리를 제어하는 인프라
3. **데이터 펌프(data pump)**: 업데이트된 데이터를 DB에 비동기 전송
4. **데이터 라이터(data writer)**: 데이터 펌프에서 받아 DB에 씀
5. **데이터 리더(data reader)**: 처리 장치 시작 시 DB에서 데이터를 읽어 그리드에 전달

### 2.2 가상 미들웨어의 4그리드

| 그리드 | 역할 |
|---|---|
| **메시징 그리드** | 입력 요청·세션 상태 관리, 처리 장치로 요청 분배 |
| **데이터 그리드** | 처리 장치 간 데이터 복제 관리 — 이 스타일에서 **가장 중요한 컴포넌트** |
| **처리 그리드**(선택) | 여러 처리 장치 간 요청 오케스트레이션 |
| **배포 관리자** | 부하에 따라 처리 장치 인스턴스를 동적으로 시작/종료 (탄력성의 핵심) |

## 3. 데이터 충돌과 캐시 전략

복제 캐시는 처리 장치 간 업데이트를 전파하는데, 인스턴스 수와 업데이트 빈도가 높으면 **데이터 충돌(data collision)** 이 발생할 수 있다. 충돌 발생 확률은 복제 지연·인스턴스 수·업데이트 비율에 비례한다.

| | 복제 캐시(replicated) | 분산 캐시(distributed) |
|---|---|---|
| 캐시 위치 | 각 처리 장치 내부 | 전용 외부 캐시 서버 |
| 성능 | 매우 빠름(로컬) | 느림(원격 호출) |
| 내고장성 | 높음 | 캐시 서버 장애에 취약 |
| 데이터 일관성 | 충돌 위험 | 단일 소스라 일관성 우수 |

- **니어 캐시(near cache)** 는 분산 캐시(풀 백킹 캐시) + 각 처리 장치의 로컬 프론트 캐시 하이브리드지만, 저자는 **권장하지 않는다** (동기화 복잡도가 이점을 상쇄)

## 4. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 도메인 + 기술 |
| 퀀텀 수 | 하나 또는 여러 개 |
| 배포성 | ★★★☆☆ |
| 탄력성 | ★★★★★ |
| 진화성 | ★★★☆☆ |
| 내고장성 | ★★★☆☆ |
| 모듈성 | ★★★☆☆ |
| 전체 비용 | ★☆☆☆☆ |
| 성능 | ★★★★★ |
| 신뢰성 | ★★★★☆ |
| 확장성 | ★★★★★ |
| 단순성 | ★☆☆☆☆ |
| 시험성 | ★☆☆☆☆ |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 공간 기반 아키텍처는 **탄력성·확장성·성능의 끝판왕(모두 5점)** 이다. 인메모리 캐시를 활용하고 제약이던 데이터베이스를 없앴기에 수백만 동시 사용자도 거뜬히 처리한다. 하지만 그 대가로 **단순성·시험성·비용이 최악**이다.
:::

- **단순성·시험성 1개**: 캐시 + 최종 일관성으로 구조가 매우 복잡하다. 최고 부하에서 수십만 동시 사용자를 테스트하는 것은 지극히 어렵고 비싸며, 대부분 프로덕션에서 테스트해 리스크가 크다
- **전체 비용 1개**: 높은 확장성·탄력성을 위해 상용 캐시 제품 라이선스 비용 + 높은 리소스 사용률
- **분할 유형 도메인 + 기술**: 처리 장치를 도메인 서비스처럼 쓸 수 있어(도메인) 동시에 데이터 펌프로 트랜잭션 처리를 캐시와 분리하므로(기술) 둘 다 해당
- **퀀텀**: 처리 장치가 DB와 동기 통신을 하지 않으므로 DB는 퀀텀 방정식에서 제외된다. 동기 통신(또는 처리 그리드 오케스트레이션)하는 처리 장치는 동일 퀀텀에 속한다

---

## 5. 구현 예시

- **콘서트 티켓 판매 시스템**: 티켓 오픈 순간 폭발적 동시 접속. 예측 불가능한 극단적 부하에 공간 기반이 제격
- **온라인 경매 시스템**: 마감 직전 입찰 폭주. 순간 확장성이 결정적

## 요약

- 웹 애플리케이션의 궁극적 병목은 **데이터베이스의 동시 트랜잭션 한계**(삼각형 토폴로지). 공간 기반은 중앙 DB를 없애 이를 해결
- **처리 장치**(인메모리 그리드 + 로직) + **가상 미들웨어**(메시징·데이터·처리 그리드 + 배포 관리자) + **데이터 펌프·라이터·리더**로 구성
- DB 업데이트는 퍼시스턴트 큐로 **비동기** 전송 — 최종 일관성 기반
- **복제 캐시**(빠름·충돌 위험) vs **분산 캐시**(일관성·느림), 니어 캐시는 비권장
- **탄력성·확장성·성능 5점(끝판왕)**, **단순성·시험성·비용은 최악** — 극단적·가변적 확장이 필요한 콘서트 티켓·경매 같은 시스템에 적합

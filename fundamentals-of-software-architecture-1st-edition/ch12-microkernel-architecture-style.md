# Chapter 12: Microkernel Architecture Style (마이크로커널 아키텍처 스타일)

## 핵심 질문

마이크로커널(플러그인) 아키텍처는 왜 이클립스·크롬 같은 제품부터 보험금 청구·세무 신고 같은 대규모 비즈니스 애플리케이션까지 폭넓게 쓰이는가? 코어 시스템과 플러그인 컴포넌트는 각각 무엇을 책임지는가? 레지스트리와 계약은 어떻게 코어와 플러그인을 이어주는가? 이 스타일은 왜 도메인·기술 분할이 모두 가능한 유일한 스타일인가?

---

## 1. 개요

수십 년 전에 만들어진 마이크로커널 아키텍처(*Microkernel Architecture - 플러그인 아키텍처라고도 함*)는 오늘날에도 널리 쓰인다. 단일 모놀리식 배포 단위로 패키징되며, 고객 사이트에 서드파티 제품으로 설치되는 **제품 기반(product-based) 애플리케이션**에 적합하고, 일반 비즈니스 애플리케이션에서도 많이 쓴다.

---

## 2. 토폴로지

**코어 시스템(core system)** 과 **플러그인 컴포넌트(plug-in component)** 라는 두 요소로 구성된 비교적 단순한 모놀리식 아키텍처다.

```
      [플러그인]  [플러그인]  [플러그인]
           \        |        /
            ┌───────────────┐
            │   코어 시스템   │
            └───────────────┘
           /        |        \
      [플러그인]  [플러그인]  [플러그인]
```

### 2.1 코어 시스템

코어 시스템은 **시스템을 실행시키는 데 필요한 최소한의 기능**으로 정의한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 이클립스 IDE가 좋은 예다. 이클립스 코어는 파일을 열고 고치고 저장하는 기본 텍스트 에디터에 불과하며, 플러그인을 추가해야 쓸 만한 제품이 된다. 코어 시스템은 커스텀 처리가 거의 필요 없는 **정상 경로(happy path)**, 즉 일반적인 처리 흐름이라고 정의할 수 있다.
:::

코어 시스템의 순환 복잡도를 없애고 별도 플러그인으로 분리하면 확장성·유지보수성·시험성이 좋아진다. 전자제품 재활용 애플리케이션 예:

```java
// 나쁜 예 — 코어 시스템에 클라이언트 종속 코드가 몰림 (높은 순환 복잡도)
public void assessDevice(String deviceID) {
    if (deviceID.equals("iPhone6s")) {
        assessiPhone6s();
    } else if (deviceID.equals("iPad1")) {
        assessiPad1();
    } else if (deviceID.equals("Galaxy5")) {
        assessGalaxy5();
    } else ...
}
```

```java
// 좋은 예 — 코어는 레지스트리에서 플러그인을 찾아 호출만 한다
public void assessDevice(String deviceID) {
    String plugin = pluginRegistry.get(deviceID);
    Class<?> theClass = Class.forName(plugin);
    Constructor<?> constructor = theClass.getConstructor();
    DevicePlugin devicePlugin = (DevicePlugin)constructor.newInstance();
    devicePlugin.assess();
}
```

감정할 제품이 추가되면 새 플러그인을 만들고 레지스트리만 업데이트하면 된다. 코어 시스템은 규모에 따라 레이어드 아키텍처, 모듈러 모놀리스, 혹은 도메인 서비스로 구현할 수 있다. 어느 경우든 전체 모놀리스는 보통 **하나의 데이터베이스를 공유**한다.

### 2.2 플러그인 컴포넌트

플러그인 컴포넌트는 특수 처리 로직·부가 기능·커스텀 코드를 담은 스탠드얼론 컴포넌트다. **변동성이 큰 코드를 분리**하여 유지보수성·시험성을 높인다. 이상적인 플러그인은 상호 독립적이며 의존성이 없다.

| 구현 방식 | 특징 |
|---|---|
| **컴파일 기반** | 관리는 편하지만 변경·추가·삭제 시 전체 모놀리스를 재배포해야 함 |
| **런타임 기반** | 코어/다른 플러그인 재배포 없이 추가·삭제 가능. OSGi·펜로즈·직소(자바)·프리즘(닷넷) 등 프레임워크로 관리 |

점대점 플러그인은 JAR·DLL·gem 같은 공유 라이브러리나 패키지명(자바)/네임스페이스(C#)로 구현한다. 네임스페이스는 `app.plugin.<도메인>.<콘텍스트>` 포맷(예: `app.plugin.assessment.iphone6s`)으로 명명하는 게 좋다.

### 2.3 원격 플러그인 — 점대점 vs REST

플러그인을 스탠드얼론 서비스(REST·메시징)로 만들 수도 있다. 다만:

- **장점**: 커플링이 낮아져 확장성·처리량 개선, 특수 프레임워크 없이 런타임 변경 가능, 비동기 통신으로 반응성 향상
- **단점**: 모놀리식이 아닌 **분산 아키텍처로 바뀌어야** 하며 복잡도·비용이 높아진다. 서드파티 온프렘 제품은 이렇게 배포하기 어렵고, 플러그인 무응답 시 요청이 완료될 수 없다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 원격 플러그인이라도 **코어 시스템이 모놀리식이면 여전히 단일 아키텍처 퀀텀**이다. 모든 요청이 무조건 코어를 거쳐 각 플러그인 서비스로 흘러가기 때문이다.
:::

### 2.4 데이터

플러그인이 중앙 공유 DB에 직접 접속할 일은 거의 없다. **코어 시스템이 데이터를 가져와 각 플러그인에 전달**한다 — 이유는 디커플링이다. DB 변경이 코어에만 영향을 주고 플러그인에는 영향이 없으며, 플러그인은 자기만 액세스하는 데이터 저장소(제품별 감정 규칙 DB·규칙 엔진)를 따로 가질 수 있다.

## 3. 레지스트리

코어 시스템은 어떤 플러그인이 있고 어떻게 가져오는지 알아야 한다. 가장 일반적인 방법은 **플러그인 레지스트리(registry)** 다. 레지스트리에는 플러그인 명칭·데이터 계약·원격 액세스 프로토콜 정보가 담긴다.

```java
Map<String, String> registry = new HashMap<String, String>();
static {
    // 점대점 액세스
    registry.put("iPhone6s", "iPhone6sPlugin");
    // 메시징
    registry.put("iPhone6s", "iphone6s.queue");
    // REST
    registry.put("iPhone6s", "https://atlas:443/assess/iphone6s");
}
```

레지스트리는 코어가 소유한 단순 맵일 수도, 아파치 주키퍼·콘술 같은 디스커버리 도구를 쓰는 복잡한 형태일 수도 있다.

## 4. 계약

플러그인과 코어 간 계약은 보통 플러그인 도메인 단위로 표준화된다. 서드파티 플러그인처럼 계약을 마음대로 바꿀 수 없을 때는 **커스텀 계약 + 어댑터**를 만든다.

```java
public interface AssessmentPlugin {
    public AssessmentOutput assess();
    public String register();
    public String deregister();
}

public class AssessmentOutput {
    public String assessmentReport;
    public Boolean resell;      // 되팔 수 있는가
    public Double value;        // 제품 가치
    public Double resellPrice;  // 권장 가격
}
```

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 역할·책임 모델을 잘 보라. 코어 시스템은 감정 리포트를 **출력하거나 유저에게 보여주는 일만** 할 뿐, 리포트의 세부 내용을 포맷팅하고 이해하는 것은 코어의 책임이 아니다.
:::

## 5. 실제 용례

- **제품/도구**: 이클립스 IDE, PMD, 지라, 젠킨스, 크롬·파이어폭스(뷰어·플러그인)
- **대규모 비즈니스**: 보험금 청구 처리(관할구역별 규칙을 플러그인으로), 세무 신고 소프트웨어(미국 1040 폼 = 코어, 각 부가 양식·워크시트 = 플러그인). 세법 개정 시 해당 플러그인만 영향받아 변경이 수월하다

## 6. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 도메인 및 기술 |
| 퀀텀 수 | 1 |
| 배포성 | ★★★☆☆ |
| 탄력성 | ★☆☆☆☆ |
| 진화성 | ★★★☆☆ |
| 내고장성 | ★☆☆☆☆ |
| 모듈성 | ★★★☆☆ |
| 전체 비용 | ★★★★★ |
| 성능 | ★★★☆☆ |
| 신뢰성 | ★★★☆☆ |
| 확장성 | ★☆☆☆☆ |
| 단순성 | ★★★★☆ |
| 시험성 | ★★★☆☆ |

- **단순성·전체 비용이 주요 강점** (레이어드와 공통). 모놀리식 배포 탓에 탄력성·내고장성·확장성이 약함. 모든 요청이 코어를 거치므로 퀀텀은 언제나 1
- **도메인·기술 분할이 모두 가능한 유일한 스타일**: 대부분 기술 분할이지만, 강력한 도메인 대 아키텍처 동형성(*domain-to-architecture isomorphism - 도메인 구조와 아키텍처 구조가 일치하는 성질*)으로 도메인 분할도 실현된다. 위치·클라이언트별 설정, 커스터마이징·기능 신장성 중심 제품에 제격
- **배포성·시험성·신뢰성 3개(평균 이상)**: 기능을 독립 플러그인으로 분리할 수 있어(특히 런타임 배포 시) 변경분에 대한 테스트 범위·배포 리스크가 줄어든다
- **모듈성 3개(평균 이상)**: 자기 완비형 플러그인으로 기능을 추가·삭제·변경할 수 있어 변경 대응이 빠르다 (세법 개정 시 세무 양식 플러그인만 교체)
- **성능 3개**: 대부분 규모가 작고, 싱크홀 안티패턴이 없으며, 불필요한 기능을 해제(unplug)해 처리 흐름을 간소화할 수 있다 (와일드플라이 예)

## 요약

- 마이크로커널은 **코어 시스템(최소 기능·정상 경로) + 플러그인 컴포넌트(변동성 큰 커스텀 코드)** 로 구성된 단순 모놀리식
- 클라이언트 종속 로직을 플러그인으로 빼내 코어의 순환 복잡도를 낮춘다. 새 기능은 **플러그인 추가 + 레지스트리 업데이트**
- **레지스트리**로 플러그인을 찾고, **계약**(인터페이스·JSON·XML)으로 코어-플러그인 역할을 분리
- 플러그인은 점대점(모놀리식)이 기본. 원격(REST) 전환은 확장성을 주지만 분산 복잡도를 부른다
- **도메인·기술 분할이 모두 가능한 유일한 스타일**. 강점은 **단순성·저비용·모듈성**, 약점은 **탄력성·확장성·내고장성**
- 이클립스·크롬 같은 제품과 보험·세무 같은 규칙 다변형 비즈니스에 적합

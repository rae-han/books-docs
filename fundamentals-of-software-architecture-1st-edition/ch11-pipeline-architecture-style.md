# Chapter 11: Pipeline Architecture Style (파이프라인 아키텍처 스타일)

## 핵심 질문

파이프라인 아키텍처는 왜 유닉스 쉘부터 함수형 프로그래밍, 맵리듀스까지 소프트웨어 곳곳에서 끊임없이 등장하는가? 파이프와 필터란 무엇이며, 네 가지 필터 타입은 어떻게 조합되는가? 커누스의 10쪽짜리 파스칼 프로그램을 매킬로이가 한 줄 쉘 스크립트로 이긴 일화는 무엇을 말해주는가?

---

## 1. 개요

파이프라인 아키텍처(*Pipeline Architecture - 파이프와 필터(Pipes and Filters) 아키텍처라고도 함*)는 소프트웨어에서 끊임없이 등장하는 기본 스타일이다. 개발자가 기능을 개별 파트로 분리하기로 결정하는 순간부터 이 패턴이 수반된다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 파이프라인은 Bash·Zsh 같은 유닉스 터미널 쉘의 기초 원리이며, 함수형 언어의 언어 구조와도 유사하다. **맵리듀스(MapReduce) 프로그래밍 모델**을 응용한 많은 도구가 이 토폴로지를 따른다. 저수준에서 구현할 수도, 고수준 비즈니스 애플리케이션에도 적용할 수 있다.
:::

---

## 2. 토폴로지

파이프라인 아키텍처는 다수의 **파이프**와 **필터**로 구성된다. 필터 간 단방향 통신은 보통 점대점(point-to-point) 방식이다.

```
[필터] ──파이프──▶ [필터] ──파이프──▶ [필터] ──파이프──▶ [필터]
```

### 2.1 파이프

**파이프(pipe)** 는 한 소스에서 입력을 받아 다른 소스로 출력을 내는 **필터 간 통신 채널**이다.

- 성능상 이유로 보통 **단방향·점대점** 방식
- 페이로드 데이터는 어떤 포맷이든 가능하지만, 아키텍트는 고성능에 유리한 적은 양의 데이터를 선호

### 2.2 필터 — 네 가지 타입

**필터(filter)** 는 자기 완비형(self-contained)이고 다른 필터와 독립적이며 일반적으로 무상태(stateless)다. 한 가지 태스크만 수행하므로 복합 태스크는 여러 필터를 이어붙여 처리한다.

| 필터 타입 | 역할 | 함수형 대응 |
|---|---|---|
| **프로듀서(producer)** | 프로세스의 시작점. 아웃바운드만 있어 **소스(source)** 라고도 함 | — |
| **변환기(transformer)** | 입력을 받아 데이터를 변환 후 아웃바운드 파이프로 전달 | **map** |
| **테스터(tester)** | 입력을 받아 하나 이상의 기준에 대해 테스트하고 필요시 결과 생산 | **reduce** |
| **컨슈머(consumer)** | 파이프라인의 종착역. 결과를 DB에 저장하거나 UI에 표시 | **sink** |

### 2.3 커누스 vs 매킬로이 일화

> 도널드 커누스(Donald Knuth)는 텍스트 파일에서 출현 빈도 상위 n개 단어를 빈도순으로 정렬해 출력하는 프로그램을 의뢰받고, 새 알고리즘을 설계해 10쪽이 넘는 파스칼 프로그램을 짰다. 반면 더그 매킬로이(Doug McIlroy)는 트위터 한 포스트에 들어갈 분량의 쉘 스크립트로 같은 문제를 우아하게 해결했다.

```bash
tr -cs A-Za-z '\n' |
tr A-Z a-z |
sort |
uniq -c |
sort -rn |
sed ${1}q
```

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 각 파이프와 필터는 단방향이고 워낙 단순해서 얼마든지 조합할 수 있다. 단순하지만 강력하고 복합적으로 추상화된 이 조합력이 파이프라인 아키텍처의 본질이다.
:::

## 3. 예제 — 카프카 스트리밍 파이프라인

파이프라인 패턴은 EDI(*Electronic Data Interchange - 전자 데이터 교환*) 도구, ETL(*Extract-Transform-Load - 추출·변환·적재*) 도구, 아파치 카멜 같은 오케스트레이터/중재자에서 흔히 쓰인다.

다양한 서비스의 텔레메트리(*telemetry - 원격 계측*) 정보를 아파치 카프카에 스트리밍하는 예:

```
[서비스 정보 캡처]          (프로듀서) — 카프카 토픽 구독
       ↓
[지속시간 필터]             (테스터) — 지속시간 관련? 
       ├─ 예 → [지속시간 계산기] (변환기)
       └─ 아니오 ↓
[가동시간 필터]             (테스터) — 가동시간 관련?
       ├─ 아니오 → 종료 (쓸모없는 데이터)
       └─ 예 → [가동시간 계산기] (변환기)
                    ↓
              [DB 출력]     (컨슈머) — MongoDB 저장
```

- **필터마다 관심사가 분리**된다. 서비스 정보 필터는 카프카 연결만, 지속시간 필터는 검증 후 다음 파이프 전달만 신경 쓴다
- **확장성**을 잘 보여준다 — DB 접속 대기시간 같은 새 메트릭을 전달하려면 가동시간 필터 뒤에 테스트 필터를 하나 더 추가하면 된다

## 4. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 기술적 |
| 퀀텀 수 | 1 |
| 배포성 | ★★★☆☆ |
| 탄력성 | ★☆☆☆☆ |
| 진화성 | ★★★☆☆ |
| 내고장성 | ★☆☆☆☆ |
| 모듈성 | ★★★☆☆ |
| 전체 비용 | ★★★★★ |
| 성능 | ★★☆☆☆ |
| 신뢰성 | ★★★☆☆ |
| 확장성 | ★☆☆☆☆ |
| 단순성 | ★★★★★ |
| 시험성 | ★★★☆☆ |

- **기술 분할** 아키텍처(필터 타입별로 로직 분리)이고 보통 **모놀리식**으로 배포하므로 퀀텀은 언제나 1
- **주요 강점은 모듈성과 결부된 전체 비용·단순성**: 모놀리식에 가까워 분산 복잡도가 없고, 필터 간 관심사 분리로 모듈성을 확보한다. 필터를 교체·수정해도 다른 필터에 영향이 없다
- **배포성·시험성(3개)**: 필터를 통한 모듈성이 우수해 레이어드(각 2)보다 조금 낫지만, 모놀리스라서 절차·리스크·배포 빈도의 영향을 받는다
- **신뢰성 3개**: 분산 아키텍처의 네트워크·대역폭·레이턴시 문제가 덜하지만 모놀리식 배포의 고질적 문제 탓에 그 이상은 어렵다
- **탄력성·확장성 1개**: 모놀리식 단일 퀀텀이라 특정 지점까지만 확장 가능
- **내고장성 1개**: 한 파트의 OOM이 애플리케이션 전체 충돌로 이어진다

## 요약

- 파이프라인은 유닉스 쉘·함수형·맵리듀스 곳곳에 등장하는 **가장 기본적인 조합형 스타일**
- **파이프**(단방향 점대점 통신 채널) + **필터**(자기 완비·무상태·단일 태스크) 4종: **프로듀서·변환기(map)·테스터(reduce)·컨슈머(sink)**
- 커누스 10쪽 파스칼 vs 매킬로이 한 줄 쉘 — **단순한 필터의 조합력**이 강력함을 보여준다
- ETL·EDI·카프카 스트리밍 등 **단방향 처리 태스크**에 적합. 필터 추가로 쉽게 확장
- 강점은 **단순성·저비용·모듈성**, 약점은 **탄력성·확장성·내고장성** — 모놀리식 단일 퀀텀의 한계

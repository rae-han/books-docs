# Chapter 6: Measuring and Governing Architecture Characteristics (아키텍처 특성의 측정 및 거버넌스)

## 핵심 질문

"민첩성을 위한 설계"처럼 모호한 아키텍처 특성을 어떻게 객관적으로 측정할 수 있는가? 성능·순환 복잡도·시험성 같은 특성은 각각 무엇으로 재는가? 아키텍트가 정한 특성과 원칙을 개발자들이 계속 지키도록 어떻게 자동으로 강제할 수 있는가? 피트니스 함수란 무엇이며 왜 "새로운 프레임워크가 아니라 새로운 시각"인가?

---

## 1. 아키텍처 특성을 측정하기 어려운 세 가지 이유

아키텍트는 소프트웨어 프로젝트의 거의 전 부문에 걸쳐 엄청나게 다양한 아키텍처 특성을 다룬다. 성능·탄력성·확장성 같은 운영 특성이 모듈성·배포성 같은 구조적 관심사와 한데 뒤섞인다. 그런데 이런 특성을 정의하려 할 때 흔히 세 가지 문제가 발생한다.

1. **물리학이 아니다**: 아키텍처 특성은 대부분 의미가 모호하다. "민첩성이나 배포성을 위한 설계"란 대체 어떤 것인가? 동일한 용어도 업계마다 바라보는 시각이 제각각이고, 법적 상황에 따라 좌지우지되거나 우발적으로 의미가 정해지는 경우도 있다
2. **정의가 너무 다양하다**: 성능 같은 중요한 특성조차 같은 조직 안에서 부서마다 정의가 일치하지 않는다. 개발자·아키텍트·운영자가 정의를 통일하기 전까지는 원활한 소통이 어렵다
3. **너무 복합적이다**: 바람직한 아키텍처 특성은 대부분 더 작은 여러 특성들로 구성된다. 예를 들어 민첩성은 모듈성·배포성·시험성 등으로 세분화할 수 있다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 이 세 가지 문제는 아키텍처 특성을 **객관적으로 정의**하면 모두 해결된다. 명확하게 정의하고 조직 전체가 이에 동의하면 팀은 **공통의 아키텍처 언어**를 확립할 수 있다. 복합적인 특성도 더 잘게 나누어 분석하면 객관적으로 정의할 수 있는, 측정 가능한 특성을 밝혀낼 수 있다.
:::

## 2. 운영적 측정

성능·확장성처럼 비교적 정확하게 측정할 수 있는 특성도 많지만, 팀 목표에 따라 해석이 미묘하게 갈릴 때가 많다.

예를 들어 특정 요청의 **평균 응답 시간**을 측정한다고 하자. 어떤 경계 조건 때문에 1%의 요청이 다른 요청보다 처리 시간이 10배나 오래 걸린다면 어떻게 해야 할까? 사내 네트워크 리소스가 충분하다면 이런 특이점(*Outlier - 다른 관측치와 크게 다른 데이터 포인트*)은 평균에 묻혀 드러나지 않을 수 있다. 따라서 **최대 응답 시간**도 함께 측정해야 특이점까지 잡아낼 수 있다.

### 2.1 성능의 여러 가지 맛

많은 아키텍처 특성은 다수의 미묘한 의미를 갖는다. 성능만 봐도 그렇다. 대부분의 프로젝트는 (웹 애플리케이션의 요청/응답 시간을 재는 것처럼) 일반적인 성능을 살펴보지만, 아키텍트와 데브옵스 엔지니어는 **성능 예산**을 책정하는 데 (즉 애플리케이션의 어떤 부분은 얼마만큼의 예산을 쓸지 결정하는 데) 엄청난 작업을 수행해왔다.

- 유저 행동 패턴을 분석한 결과, 첫 페이지의 렌더링 시간(브라우저·모바일 기기에서 웹 페이지가 뜨기 시작하는 가시적 결과가 처음 나타나는 시간)은 **500ms, 즉 0.5초**가 최적이라는 결론을 얻기도 한다
- 선도적인 회사들은 페이지 다운로드에 **K-가중치 예산(*K-Weight budget - 특정 페이지에 허용된 라이브러리와 프레임워크의 최대 바이트 수*)**을 설정한다. 이는 "한 번에 네트워크를 통해 이동할 수 있는 바이트 수는 제한적"이라는 물리학의 제약에서 파생된 사상으로, 레이턴시가 큰 모바일 기기라면 더욱 중요하다
- 요즘은 **최초 콘텐츠 렌더링(*First Contentful Paint - 브라우저가 DOM에서 콘텐츠의 첫 비트를 렌더링하는 시점*)**, **최초 CPU 유휴(*First CPU Idle - 페이지가 최소한의 상호작용을 하기에 충분할 정도로 조용해지는 시점*)** 같은 메트릭에 성능 예산을 집중하기도 한다

수준 높은 팀은 달성하기 어려운 성능 수치를 무작정 정하는 대신, **통계 분석 결과로 얻은 나름의 정의**에 기반한다. 예를 들어 확장성을 모니터링하는 비디오 스트리밍 업체의 엔지니어는 아무 수치나 대충 목표로 삼는 게 아니라, 시간에 따른 추이를 측정하고 통계 모델을 수립한 뒤, 실시간 수집한 메트릭이 예측 모델에서 벗어나면 알림을 보낸다.

> 만약 이 과정이 수포로 돌아간다면 원인은 모델 자체가 부정확했거나(팀이 알고 싶어 하는 것), 뭔가가 잘못되었거나(역시 팀이 알고 싶어 하는 것) 두 가지 중 하나일 것이다.

## 3. 구조적 측정 — 순환 복잡도

성능처럼 목표치가 확실하지 않은 메트릭도 있다. 잘 정의된 모듈성처럼 내부 구조에 관한 특성이 그렇다. 아직 내부 코드 품질에 대한 종합적인 메트릭은 없지만, 아키텍트는 여러 도구를 이용해 코드 구조의 중요한 부분을 들여다볼 수 있다. 그중 코드의 복잡도는 **순환 복잡도(*Cyclomatic Complexity, CC - 코드 복잡도를 객관적으로 나타내는 지표*)**라는 메트릭으로 명쾌하게 측정할 수 있다.

### 3.1 순환 복잡도의 계산

순환 복잡도(CC)는 1976년 토마스 맥케이브(Thomas McCabe)가 개발한 코드 레벨 메트릭으로, 함수/메서드·클래스·애플리케이션 레벨에서 코드 복잡도를 나타낸다. CC는 코드에 **그래프 이론**을 적용해 계산하는데, 구체적으로는 상이한 실행 경로(*Execution Path*)를 유발하는 **결정점(*Decision Point - if 문 같은 조건 분기*)**을 이용한다.

- 함수에 결정문이 하나도 없으면 CC = 1
- 조건 분기가 하나 있으면 실행 경로가 두 갈래로 갈라지므로 CC = 2

하나의 함수/메서드에서 CC를 구하는 공식은 `CC = E - N + 2`다. 여기서 N은 노드(*Node - 코드 라인*), E는 간선(*Edge - 가능한 결정*)이다.

```java
public void decision(int c1, int c2) {
    if (c1 < 100)
        return 0;
    else if (c1 + c2 > 500)
        return -1;
    else
        return 1;
}
```

위 코드의 순환 복잡도를 구하면 `5 - 4 + 2 = 3`이다. 공식 끝의 `2`는 단일 함수/메서드를 단순화한 값이다. 다른 메서드도 호출하는 경우(그래프 이론에서는 "연결된 컴포넌트(*Connected Component*)"라고 함)까지 고려한 일반 공식은 `CC = E - N + 2P`이며, 여기서 P는 연결된 컴포넌트 수다.

### 3.2 순환 복잡도는 어느 정도가 적당한가

너무 복잡한 코드는 곧 코드 스멜(*Code Smell - 더 심오한 문제를 일으킬 가능성이 있는 소스 코드의 특징*)이며, 모듈성·시험성·배포성 등 거의 모든 바람직한 코드베이스 특성을 저해한다. "CC 값은 어느 정도가 적당한가?"라는 질문의 답은 소프트웨어 아키텍처의 모든 답이 그렇듯 **경우에 따라 다르다**. 알고리즘이 복잡한 문제는 솔루션에도 복잡한 함수가 많이 등장한다. 그러나 함수가 복잡한 이유가 문제 영역 때문인지, 코딩 품질이 낮아서인지, 코드 분할이 제대로 안 돼서인지 아키텍트는 CC를 모니터링하며 잘 살펴야 한다.

| 기준 | CC 값 | 평가 |
|---|---|---|
| 업계 일반 기준 | 10 이하 | 도메인 복잡도를 고려하지 않은 통상적 허용치 |
| 저자 권장 기준 | 5 이하 | 응집도가 괜찮은, 짜임새 있는 코드 |
| 위험 신호 | 50 이상 | 코드 커버리지를 아무리 높여도 구제 불능 |

> 닐(Neal)은 하나의 C 함수가 상용 소프트웨어 패키지의 심장부 역할을 하는 끔찍한 경우도 보았는데, CC가 무려 800이 넘었다고 한다! 4,000라인이 넘는 코드가 함수 하나에 몰려 있었고 (불가능할 정도로 깊게 중첩된 루프에서 탈출하고자) GOTO 문이 여기저기 도배되어 있었다.

자바 진영에는 **Crap4j** 같은 도구가 있어 CC와 코드 커버리지를 함께 측정해 코드가 얼마나 불량한지 파악할 수 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 테스트 주도 개발(TDD) 같은 엔지니어링 프랙티스는 대체로 **더 작고 덜 복잡한 메서드**를 생성하는 부수적(그러나 긍정적) 효과를 가져온다. TDD를 실천하는 개발자는 먼저 간단한 테스트를 작성한 뒤 그것을 통과시키는 최소한의 코드를 작성하려 한다. 구체적인 동작과 명확한 테스트 경계에 집중하면 짜임새 있고 고도로 응집된 메서드가 나오고, 그 결과 CC 값도 낮게 나온다.
:::

## 4. 프로세스 측정

소프트웨어 개발 프로세스와 교차하는 아키텍처 특성도 있다. 예를 들어 **민첩성**은 바람직한 특성으로 보일 때가 많은데, 이는 시험성·배포성 등으로 나눌 수 있는 복합적 특성이다.

- **시험성**: 거의 모든 플랫폼에서 **코드 커버리지 도구**로 측정할 수 있다. 단, 코드 커버리지가 100%여도 어설션(*Assertion - 코드의 정확성을 검증하는 단언*)이 형편없으면 소용없으므로, 시험성은 사고(thinking)와 판단(commonsense)을 대체할 수는 없다. 그래도 시험성은 분명 객관적으로 측정 가능한 특성이다
- **배포성**: 실패 대비 배포 성공률(%), 배포 소요 시간, 배포 시 발생한 이슈/버그 수 등 다양한 메트릭으로 측정한다. 어떤 측정 세트가 유용한지는 각 팀이 알아서 준비해야 하며, 이렇게 측정한 메트릭들은 대개 그 팀의 우선순위와 목표가 된다

민첩성 관련 특성은 소프트웨어 개발 프로세스와 연관되지만, 이 프로세스는 다시 **아키텍처 구조에도 영향을 미친다**. 예를 들어 배포 용이성과 시험성이 최우선이라면 아키텍트는 아키텍처 수준에서 모듈성·격리성을 높이는 데 주력하게 된다. 이것이 바로 **아키텍처 특성이 구조를 주도하는** 좋은 예다.

## 5. 거버넌스와 아키텍처 특성 관리

아키텍트가 아키텍처 특성을 확정하고 우선순위를 정했다면, 개발자들이 이 우선순위를 잘 지킬 거라고 어떻게 확신할 수 있을까? 예를 들어 모듈성은 아키텍처 관점에서는 중요하지만 긴급하지는 않은 특성이다. 소프트웨어 프로젝트에서 긴급성을 무시할 수는 없지만, 그래도 아키텍트는 거버넌스 메커니즘을 강구해야 한다.

거버넌스(*Governance - "이끌다"를 뜻하는 그리스어 kubernan에서 유래*)는 아키텍트가 담당하는 중요한 업무로, 이름에서 느껴지듯 아키텍트가 영향력을 행사하려는 모든 소프트웨어 개발 프로세스를 포괄한다. 예컨대 소프트웨어 품질 보장 업무도 아키텍처 범주에 속하므로 거버넌스 항목이며, 이를 소홀히 하면 끔찍한 품질 문제가 불거진다.

다행히 요즘은 아키텍트의 수고를 덜어주는 솔루션이 많이 등장했다. 익스트림 프로그래밍(XP)에서 비롯된 자동화 움직임은 지속적 통합(CI)으로 발전했고, 운영까지 자동화하는 데브옵스 체계에 이르렀으며, 지금도 아키텍처 거버넌스를 통해 계속 발전하고 있다. 『Building Evolutionary Architectures』(O'Reilly, 2017)에는 아키텍처 거버넌스의 여러 부문을 자동화하는 **피트니스 함수**라는 기법이 소개되어 있다.

## 6. 피트니스 함수

『Building Evolutionary Architectures』 제목의 "진화적(evolutionary)"은 생물학보다 **진화적 컴퓨팅(*Evolutionary Computing*)**에서 더 많이 나오는 단어다. 저자 중 한 사람인 레베카 파슨스(Rebecca Parsons) 박사는 유전자 알고리즘 등 진화적 컴퓨팅 분야를 오래 연구했다. 유전자 알고리즘을 실행해 유익한 결과를 얻으려면 결과의 품질을 객관적으로 측정하면서 알고리즘을 통제할 수 있어야 하는데, 이처럼 **결과가 목표에 얼마나 근접했는지 나타내는 목표 함수가 바로 피트니스 함수(*Fitness Function*)**다.

예를 들어 외판원 문제(한 외판원이 방문할 도시 목록과 도시 간 거리가 주어질 때 최적 이동 경로를 찾는 문제)를 유전자 알고리즘으로 풀 때, 이동 경로의 거리를 계산해 가장 짧은 경로를 표시하는 피트니스 함수, 전체 비용을 최소화하는 피트니스 함수, 전체 여행 시간을 줄이는 피트니스 함수 등을 생각해볼 수 있다. 『Building Evolutionary Architectures』는 이 개념을 차용해 **아키텍처 피트니스 함수**를 정의했다.

> **아키텍처 피트니스 함수**<br>어떤 아키텍처 특성(또는 그런 특성들의 조합)의 객관적인 무결성을 평가하는 모든 메커니즘

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 피트니스 함수는 아키텍트가 그냥 다운로드받아 쓰는 **새로운 프레임워크가 아니라, 수많은 기존 도구를 바라보는 새로운 시각**이다. 정의에서 "모든 메커니즘"이라는 어구가 핵심이다. 피트니스 함수는 사용법에 따라 메트릭·모니터·단위 테스트 라이브러리·카오스 엔지니어링 등 기존의 많은 검증 메커니즘과 중첩된다.
:::

### 6.1 순환 의존성 감지

모듈성은 대부분의 아키텍트가 관심을 기울이는 암묵적 특성이다. 그런데 자바나 닷넷 IDE에서 아직 임포트하지 않은 클래스를 참조하면 자동 임포트하라는 대화창이 뜨고, 이런 일이 자주 발생하면 개발자는 반사작용처럼 아무 클래스나 마구 임포트하게 된다. 그 결과 컴포넌트끼리 서로를 참조하는 **순환 의존성(*Cyclic Dependency*)**이 생기면, 한 컴포넌트를 재사용하려 해도 딸린 컴포넌트를 전부 가져와야 하므로 모듈성이 크게 떨어지고 아키텍처는 진흙잡탕 안티패턴에 빠져든다.

코드 리뷰는 도움은 되지만 개발 주기상 시기가 너무 늦어 비효율적이다. 이 문제는 피트니스 함수로 순환 참조 여부를 자동 감지해 해결할 수 있다.

```java
public class CycleTest {
    private JDepend jdepend;

    @BeforeEach
    void init() {
        jdepend = new JDepend();
        jdepend.addDirectory("/path/to/project/persistence/classes");
        jdepend.addDirectory("/path/to/project/web/classes");
        jdepend.addDirectory("/path/to/project/thirdpartyjars");
    }

    @Test
    void testAllPackages() {
        Collection packages = jdepend.analyze();
        assertEquals("Cycles exist", false, jdepend.containsCycles());
    }
}
```

아키텍트는 **JDepend**라는 메트릭 도구로 패키지 간 의존성을 체크한다. 이 도구는 자바 패키지 구조를 알고 있어 순환 참조가 하나라도 존재하면 테스트가 실패한다. 이 테스트를 프로젝트의 지속적 빌드에 장치하면, 개발자 때문에 순환 참조가 발생하지 않을까 하는 염려를 덜 수 있다.

### 6.2 메인 시퀀스로부터의 거리

'메인 시퀀스로부터의 거리'라는 다소 난해한 메트릭도 피트니스 함수로 확인할 수 있다.

```java
@Test
void testAllPackages() {
    double ideal = 0.0;
    double tolerance = 0.5; // 프로젝트마다 값이 다름
    Collection packages = jdepend.analyze();
    Iterator iter = packages.iterator();
    while (iter.hasNext()) {
        JavaPackage p = (JavaPackage) iter.next();
        assertEquals("Distance exceeded: " + p.getName(),
            ideal, p.distance(), tolerance);
    }
}
```

JDepend로 수용 가능한 임계치(`tolerance`)를 설정하고, 클래스가 이 범위를 벗어나면 테스트를 실패 처리하는 코드다. 두 예제 모두 피트니스 함수의 설계·구현 시 **개발자와 아키텍트의 상호 협력**이 얼마나 중요한지 보여준다. 아키텍트가 상아탑에 갇혀 개발자가 이해할 수 없는 난해한 피트니스 함수를 만들라는 뜻이 아니다.

> **TIP** 아키텍트는 개발자에게 피트니스 함수 사용을 권하기 전에 그들이 정확한 목적을 이해할 수 있도록 충분히 설명해야 한다.

### 6.3 레이어 의존성 검증 — ArchUnit과 NetArchTest

최근 수년간 피트니스 함수 도구는 점점 정교해졌다. **ArchUnit**은 JUnit의 영향을 받아 탄생한 자바 테스트 프레임워크로, 단위 테스트로 코드화한 사전 정의 거버넌스 규칙을 풍성하게 제공한다.

레이어드 아키텍처를 설계할 때 아키텍트는 정당한 사유로 레이어를 정의하지만, 개발자들이 이를 잘 지키리라 확신하기 어렵다. 패턴의 중요성을 모르는 개발자도 있고, "허락을 받느니 차라리 용서를 구하는 게 낫다"는 자세로 성능 같은 로컬 관심사를 위해 레이어를 건너뛰는 개발자도 있다. 이 문제는 ArchUnit 피트니스 함수로 해결할 수 있다.

```java
layeredArchitecture()
    .layer("Controller").definedBy("..controller..")
    .layer("Service").definedBy("..service..")
    .layer("Persistence").definedBy("..persistence..")
    .whereLayer("Controller").mayNotBeAccessedByAnyLayer()
    .whereLayer("Service").mayOnlyBeAccessedByLayers("Controller")
    .whereLayer("Persistence").mayOnlyBeAccessedByLayers("Service")
```

플랫폼만 다를 뿐 닷넷 진영에도 사용법이 비슷한 **NetArchTest**가 있다.

```csharp
// 프레젠테이션 레이어에 있는 클래스는 리포지터리를 직접 참조해선 안 됨
var result = Types.InCurrentDomain()
    .That()
    .ResideInNamespace("NetArchTest.SampleLibrary.Presentation")
    .ShouldNot()
    .HaveDependencyOn("NetArchTest.SampleLibrary.Data")
    .GetResult()
    .IsSuccessful;
```

### 6.4 카오스 엔지니어링 — 넷플릭스의 시미안 아미

넷플릭스의 **카오스 몽키(Chaos Monkey)**와 **시미안 아미(Simian Army)**도 피트니스 함수의 응용 사례다. 그중 몇 가지 몽키는 피트니스 함수의 접근 방식을 잘 나타낸다.

- **적합성 몽키(Conformity Monkey)**: 프로덕션에서 강제할 거버넌스 규칙을 정의한다. 예컨대 "모든 서비스가 모든 REST 동사에 유효한 응답을 해야 한다"고 결정했다면 해당 체크 로직을 구축한다
- **보안 몽키(Security Monkey)**: 열려 있으면 안 되는 포트, 각종 설정 오류 등 서비스에 잘 알려진 보안 취약점이 없는지 체크한다
- **문지기 몽키(Janitor Monkey)**: 더 이상 트래픽이 들어오지 않는 고아 서비스(*Orphaned Service - 아무도 호출하지 않는 서비스*)를 찾아 프로덕션에서 제거한다. 클라우드에서 실행되는 서비스는 과금 문제가 있기 때문이다

> **시미안 아미의 기원** — 넷플릭스가 운영 인프라를 아마존 클라우드로 이전하기로 결정했을 때, 아키텍트들은 운영 제어권을 잃게 될 것을 걱정했다. 그래서 카오스 엔지니어링의 카오스 몽키라는 프랙티스를 정착시켰고, 이것이 시미안 아미로 발전했다. 카오스 몽키는 일반적인 카오스를 프로덕션 환경에서 시뮬레이션하여 시스템이 얼마나 견딜 수 있는지 알아본다. 당시 일부 AWS 인스턴스에서 레이턴시 문제가 있었기에 나중에 레이턴시 몽키도 만들었고, **카오스 콩(Chaos Kong)**은 전체 아마존 데이터센터의 장애를 시뮬레이션한다.

카오스 엔지니어링은 아키텍처에 신선한 관점을 제공한다. 결국 **무엇이 잘못될 것인지가 아니라 언제 그렇게 잘못될 것인지가 문제**다. 이런 장애와 테스트를 미리 대비하면 시스템이 더욱 견고해진다.

> 아툴 가완디(Atul Gawande)의 『체크! 체크리스트』(21세기북스, 2010)에는 조종사와 외과의사 같은 전문직이 체크리스트를 활용하는 방법이 나온다. 이들이 일을 모르거나 자주 잊어서가 아니라, 세부적인 일을 반복하다 보면 간혹 빠뜨리는 것이 있기 때문이다. 피트니스 함수도 마찬가지다. 무거운 거버넌스 메커니즘이 아니라, 아키텍트가 중요한 원칙을 표현하고 자동으로 검증할 수 있는 메커니즘을 제공한다.

## 요약

- 아키텍처 특성은 **모호하고·정의가 다양하고·복합적**이라 측정이 어렵다. 해결책은 **객관적으로 정의**하여 공통의 아키텍처 언어를 확립하는 것이다
- **운영적 측정**: 평균 응답 시간만 보면 특이점을 놓치므로 최대 응답 시간도 봐야 한다. 성능은 렌더링 시간(500ms)·K-가중치 예산·최초 콘텐츠 렌더링 등 "여러 맛"이 있고, 수준 높은 팀은 통계 모델에 기반해 목표를 정한다
- **구조적 측정**: 순환 복잡도(CC)는 `CC = E - N + 2`로 계산한다. 업계는 10 이하를 허용하나 저자는 5 이하를 권장하며, CC 50 이상은 구제 불능이다. TDD는 CC를 낮추는 부수 효과가 있다
- **프로세스 측정**: 민첩성은 시험성(코드 커버리지)·배포성(성공률·소요 시간)으로 나눠 측정한다. 특성이 다시 구조를 주도한다
- **피트니스 함수**는 새 프레임워크가 아니라 기존 도구를 보는 **새로운 시각**이다. "어떤 아키텍처 특성의 객관적 무결성을 평가하는 모든 메커니즘"으로, JDepend(순환 의존성)·ArchUnit/NetArchTest(레이어 검증)·넷플릭스 시미안 아미(카오스 엔지니어링) 등으로 구현한다
- 거버넌스의 핵심은 무겁게 감시하는 것이 아니라, 중요한 원칙을 **자동으로 검증**되게 만들어 개발 주기 안에서 지켜지도록 하는 것이다

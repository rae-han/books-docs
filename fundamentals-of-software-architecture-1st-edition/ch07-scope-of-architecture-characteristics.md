# Chapter 7: Scope of Architecture Characteristics (아키텍처 특성 범위)

## 핵심 질문

"확장성"이나 "가용성"을 말할 때, 그 범위는 시스템 전체인가 아니면 시스템의 일부인가? 왜 마이크로서비스 시대에는 시스템 레벨 범위라는 전통적 전제가 무너졌는가? 컴포넌트가 서로 얼마나 강하게 "연결되어" 있는지를 어떻게 측정하는가? 아키텍처 특성의 새로운 범위를 정의하는 "아키텍처 퀀텀"이란 무엇인가?

---

## 1. 왜 시스템 레벨 범위로는 부족한가

소프트웨어 아키텍처 세계에서는 전통적으로 아키텍처 특성의 범위를 **시스템 레벨**에 두는 것을 당연시했다. 아키텍트가 확장성을 논하면 으레 전체 시스템의 확장성을 가리켰다. 10년 전까지만 해도 거의 모든 시스템이 모놀리식이었으니 안전한 전제였다. 하지만 마이크로서비스 등의 아키텍처 스타일이 등장하면서 아키텍처 특성의 범위는 상당히 좁혀졌고, 기존에 당연시했던 공식도 서서히 퇴물이 되고 있다.

6장에서 소개한 코드 레벨 메트릭들은 모두 코드에 관한 저수준 세부만 밝힐 뿐, (특히 운영과 관련된) 아키텍처 특성에 영향을 미치는 **코드베이스 외부의 의존성**을 지닌 컴포넌트는 평가하지 못한다. 아키텍트가 성능 좋고 탄력적인 코드베이스를 아무리 잘 설계해도, 그런 특성에 부합하지 않는 데이터베이스를 사용하면 애플리케이션은 성공할 수 없다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍트는 운영 아키텍처 특성을 따져볼 때 **코드베이스 외부의 컴포넌트(데이터베이스 등)까지** 살펴봐야 한다. 이런 종류의 의존성을 측정하기 위해 『Building Evolutionary Architectures』의 저자들은 **아키텍처 퀀텀(architecture quantum)**이라는 용어를 정의했다. 아키텍처 퀀텀을 이해하려면 먼저 **커네이선스**라는 핵심 메트릭을 알아야 한다.
:::

## 2. 커플링과 커네이선스

구심적/원심적 커플링 같은 코드 레벨 커플링 메트릭은 아키텍처 분석용으로는 세분도(*Granularity - 얼마나 잘게 나누었는가*)가 너무 높다. 1996년 밀러 페이지-존스(Meilir Page-Jones)는 『What Every Programmer Should Know About Object-Oriented Design』(Dorset House, 1996)에서 **커네이선스(connascence)**라는 새로운 커플링 메트릭을 소개했다.

> **커네이선스**<br>두 컴포넌트 중 한쪽이 변경될 경우 다른 쪽도 변경해야 전체 시스템의 정합성이 맞는다면, 이들은 커네이선스를 갖고 있는 것이다.

페이지-존스는 커네이선스를 두 종류로 분류했다.

- **정적 커네이선스(*Static Connascence*)**: 정적 코드 분석으로 발견할 수 있는 커네이선스. 예를 들어 마이크로서비스의 두 서비스가 `Address`라는 동일한 클래스를 공유한다면, 이 클래스를 변경할 때 두 서비스를 모두 변경해야 하므로 정적 커네이선스를 가진다
- **동적 커네이선스(*Dynamic Connascence*)**: 런타임 동작에 관한 커네이선스. 다시 동기·비동기 두 종류로 나뉜다
  - **동기**: 분산 서비스끼리 동기 호출을 하면 호출부는 피호출부(*Callee - 호출당하는 쪽*)의 응답을 기다려야 한다
  - **비동기**: 이벤트 기반 아키텍처의 비동기 호출은 파이어 앤드 포겟(*Fire-and-forget - 응답을 기다리지 않고 호출 후 잊어버리는 방식*)이므로 두 서비스를 개별적으로 작동시킬 수 있다

아키텍처 퀀텀을 정의하려면 컴포넌트가 서로 어떻게 "연결되어" 있는지 측정할 방법이 필요한데, 이것이 바로 커네이선스라는 개념이다.

## 3. 아키텍처 퀀텀

소프트웨어를 서로 엮는 것은 컴포넌트 레벨의 커플링만이 아니다. 많은 비즈니스 개념이 의미상 여러 시스템 파트를 한데 엮어 **기능적으로 응집(*Functional Cohesion*)**되어 있다. 소프트웨어를 성공적으로 설계·분석·발전시키려면 개발자는 문제가 될 만한 커플링 지점을 모두 살펴봐야 한다.

물리학에서 퀀텀(양자)은 상호작용을 하는 모든 물리 입자 가운데 가장 작은 양으로, "얼마나 많은(how many)", "얼마큼(how much)"이라는 뜻의 라틴어 quantum에서 유래한다. 저자들은 이 개념에서 아키텍처 퀀텀을 도입했다.

> **아키텍처 퀀텀**<br>높은 기능 응집도(functional cohesion)와 동기적 커네이선스(synchronous connascence)를 가진, 독립적으로 배포 가능한(independently deployable) 아티팩트

이 한 문장에는 세 가지 개념이 담겨 있다.

### 3.1 독립적으로 배포 가능

아키텍처 퀀텀은 아키텍처의 다른 파트와 독립적으로 작동되는 **모든 필수 컴포넌트**를 포함한다. 데이터베이스를 사용하는 애플리케이션은 데이터베이스 없이는 작동하지 않으므로, 데이터베이스는 퀀텀의 일부다.

- 단일 데이터베이스를 사용해 배포된 거의 모든 레거시 시스템은 아키텍처 퀀텀이 **1**이다
- 마이크로서비스 아키텍처는 서비스마다 데이터베이스가 따로 있으니 퀀텀이 **여러 개**다 (서비스별 데이터베이스는 마이크로서비스의 중심 철학인 경계 콘텍스트의 일부)

### 3.2 높은 기능 응집도

응집도는 컴포넌트 설계에 따라 구현된 코드가 얼마나 목적에 맞게 통합되어 있는지를 나타낸다.

- `Customer` 컴포넌트는 고객 엔티티와 관련된 모든 프로퍼티와 메서드를 갖고 있으므로 **응집도가 높다**
- `Utility` 컴포넌트는 잡다한 메서드를 무작위로 섞어놓아 **응집도가 낮다**

기능 응집도가 높다는 건 아키텍처 퀀텀이 목적에 맞는 뭔가를 하고 있다는 뜻이다. 단일 데이터베이스를 쓰는 모놀리식에서는 이런 구분이 별 의미가 없지만, 마이크로서비스는 각 서비스를 하나의 워크플로(경계 콘텍스트)에 맞게 설계하므로 기능 응집도가 높게 나타난다.

### 3.3 동기적 커네이선스

동기적 커네이선스는 아키텍처 퀀텀을 형성하는 애플리케이션 콘텍스트 내부 또는 분산 서비스 간의 **동기 호출**을 의미한다. 마이크로서비스에서 어떤 서비스가 다른 서비스를 동기 호출할 때, 운영 아키텍처 특성 측면에서 두 서비스는 두드러진 차이를 보일 수 없다. 만약 호출부가 피호출부보다 확장성이 훨씬 좋다면 타임아웃과 신뢰성 문제가 일어난다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 동기 호출은 호출의 길이에 대해 동적 커네이선스를 만들어낸다. 즉, 한쪽이 다른 쪽을 기다린다면 **호출하는 도중에는 양쪽의 운영 아키텍처 특성이 동일해야 한다.**
:::

예를 들어 `Payment` 서비스와 `Auction` 서비스가 있는데, `Auction`은 경매가 끝나면 `Payment`로 결제 정보를 전달하고, `Payment`는 매 500밀리초마다 하나의 결제만 처리할 수 있다고 하자. 엄청나게 많은 경매가 일시에 종료되면, 아키텍처를 제대로 설계하지 않았을 경우 첫 호출만 통과하고 나머지는 모조리 타임아웃이 발생한다. `Payment`와 `Auction` 사이에 **비동기 통신 링크**를 만들고 메시지 큐가 일시적 버퍼 역할을 수행하도록 설계하면, 비동기적 커네이선스 덕분에 아키텍처가 한층 유연해진다.

## 4. 도메인 주도 설계의 경계 콘텍스트

에릭 에반스(Eric Evans)의 『도메인 주도 설계』(위키북스, 2011)는 현대 아키텍처 사고방식에 지대한 영향을 미친 책이다. 도메인 주도 설계(DDD)는 복잡한 문제 도메인을 체계적으로 분해하는 모델링 기법으로, **경계 콘텍스트(*Bounded Context - 도메인에 관한 모든 것이 내부에서는 보이지만 외부에서는 보이지 않는 경계*)**라는 개념을 정의한다.

DDD 이전에는 사내 공통 엔티티를 개발자가 아무렇게나 가져가 재사용했는데, 이렇게 아티팩트를 함께 쓰다 보니 커플링이 심해져 팀 간 조율이 힘들고 복잡도가 증가하는 등 문제가 많았다. 경계 콘텍스트 개념은 조직 전체가 함께 쓰는 통합 `Customer` 클래스를 만드는 대신, **각 문제 도메인에 자체 클래스를 두고 차이점은 그것들을 통합하는 지점에서 조정**하는 것이다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍처 퀀텀은 아키텍처 특성의 **새로운 범위**를 제시한다. 현대 시스템에서 아키텍트는 시스템 레벨보다는 **퀀텀 레벨**의 아키텍처 특성을 정의한다. 중요한 운영 관심사에 대해 범위를 좁혀보면 중요한 아키텍처 문제를 조기에 발견하여 **하이브리드 아키텍처**를 설계할 수도 있다.
:::

## 5. 사례 연구 — GGG 아키텍처 카타

이 절에서는 **GGG(Going, Going, Gone)**라는 온라인 경매 회사의 아키텍처 카타를 살펴본다. 다음은 이 카타의 명세다.

- **설명**: 온라인 경매 시장을 전국적으로 확장하려 한다. 고객이 참여할 경매를 선택하고 기다렸다가, 경매가 시작되면 마치 경매인과 같은 공간에 있었던 것처럼 입찰을 시작한다
- **유저**: 경매당 참여자는 수백 명, 어쩌면 수천 명까지 늘어날 수 있고 많은 동시 경매가 이루어진다
- **요구사항**: 경매는 가급적 실시간 처리가 가능해야 하고 / 입찰자는 낙찰 시 자동 결제할 신용카드를 등록하며 / 입찰자는 평판 지수로 추적되고 / 입찰자는 라이브 동영상 스트림을 시청하면서 발생하는 모든 입찰을 볼 수 있어야 하며 / 온라인·라이브 입찰 모두 입찰이 일어난 순서대로 접수되어야 한다
- **부가 콘텍스트**: 공격적으로 경쟁사를 인수합병하고 있고 / 예산 제한은 없으며 / 이 회사는 위조 혐의로 고발된 소송에서 이제 막 벗어났다

### 5.1 요구사항에서 아키텍처 특성 도출

아키텍트는 각 요구사항을 꼼꼼히 검토해 아키텍처 특성을 찾아낸다.

| 요구사항 | 도출된 특성 | 판단 근거 |
|---|---|---|
| 전국적 확장, 수천 명 참여자, 실시간 처리 | 확장성·탄력성·성능 | 확장성은 명시적, 탄력성은 도메인상 암묵적(경매 막판에 입찰이 폭주) |
| 자동 결제 신용카드, 과거 위조 소송 | 보안 | 대부분 앱의 암묵적 특성. 과거 보안 문제가 있었으니 더 꼼꼼히 짚어야 함 |
| 평판 지수로 추적 | 감사성·로그성 | "추적"은 감사성을 의미. 단, "평판 지수" 자체는 도메인 지식이 필요 |
| 경쟁사 인수합병 | (트레이드오프 결정 팩터) | 설계에 직접 영향은 없으나 통신 프로토콜 선정 시 결정 팩터 |
| 예산 제한 없음 | (설계 자유도) | 더 정교하거나 목적 특화된 아키텍처 선정 가능 |
| 라이브 스트림 시청, 입찰 순서 보장 | 가용성·신뢰성 | 가용성이 파트마다 다름을 보여주는 흥미로운 난제 |

암묵적인 아키텍처 특성을 집어내려면 도메인 지식이 매우 중요하다. "모든 유저가 입찰 중에 점잖게 가격을 부를까, 아니면 끝으로 갈수록 미쳐 날뛰게 될까?" 같은 도메인 통찰이 탄력성이라는 암묵적 특성을 드러낸다.

> **리트머스 시험지** — 도메인이냐 아키텍처 특성이냐 하는 결정의 기로에서 아키텍트에게 유용한 시험지가 있다. "구현을 하려면 **도메인 지식**이 필요한가, 아니면 **추상적인 아키텍처 특성**인가?" "평판 지수"라는 어구를 본 아키텍트는 비즈니스 분석가에게 정확한 의미를 물어봐야 한다. 즉 평판 지수는 일반적인 아키텍처 특성의 표준 정의가 아니다. 반면 탄력성(폭주하는 트래픽을 처리하는 능력)은 애플리케이션 종류와 무관하게 순수하게 추상화하여 이야기할 수 있다.

### 5.2 퀀텀 레벨에서 특성 범위 지정

GGG 카타의 마지막 요구사항(라이브 스트림 시청, 입찰 순서 보장)은 시스템 레벨보다 더 세분화된 아키텍처 범위의 필요성을 잘 보여준다. "한 입찰자의 가용성이 수백 명 중 한 사람의 가용성보다 더 중요할까? 하지만 경매인이 접속할 수 없다면 어느 누구도 온라인 입찰이 불가능하다." 이처럼 파트마다 필요한 특성 수준이 다르다.

아키텍트는 아키텍처 퀀텀을 **배포·커플링·데이터를 둘 장소·아키텍처 내부 통신 스타일을 생각하는 수단**으로 활용하며, 퀀텀마다 상이한 특성을 분석해 초기 단계부터 하이브리드 아키텍처를 설계할 수 있다. GGG에서 발견한 퀀텀과 그 특성은 다음과 같다.

- **입찰자 피드백**(입찰 스트림 + 입찰 동영상 스트림): 가용성 · 확장성 · 성능
- **경매인**(라이브 경매인): 가용성 · 신뢰성 · 확장성 · 탄력성 · 성능 · 보안
- **입찰자**(온라인 입찰자와 입찰): 신뢰성 · 가용성 · 확장성 · 탄력성

세 파트는 분명히 아키텍처 특성이 다르므로, 각 파트를 별도의 퀀텀으로 보고 범위를 지정하는 것이 시스템 전체로 특성을 뭉뚱그리는 것보다 훨씬 정확하다.

## 요약

- 전통적으로 아키텍처 특성의 범위는 **시스템 레벨**이었지만, 마이크로서비스 시대에는 훨씬 더 좁게 잡아야 한다. 코드베이스 외부 의존성(데이터베이스 등)까지 특성에 영향을 주기 때문이다
- **커네이선스**(밀러 페이지-존스)는 "한쪽을 바꾸면 다른 쪽도 바꿔야 정합성이 맞는" 관계다. **정적**(공유 클래스 등)과 **동적**(런타임 동기/비동기 호출)으로 나뉜다
- **아키텍처 퀀텀** = 높은 기능 응집도 + 동기적 커네이선스 + 독립적으로 배포 가능한 아티팩트. 레거시 단일 DB 시스템은 퀀텀 1, 마이크로서비스는 퀀텀 여러 개
- **동기 호출**은 호출 도중 양쪽의 운영 특성이 동일해야 한다는 제약을 만든다. **비동기 링크**(메시지 큐 버퍼)로 이 커플링을 풀 수 있다
- **경계 콘텍스트**(DDD)는 공통 엔티티를 함께 쓰지 않고 도메인마다 자체 클래스를 두는 개념으로, 퀀텀 사고의 뿌리다
- 아키텍트는 시스템 레벨이 아닌 **퀀텀 레벨**에서 특성 범위를 지정하며, 리트머스 시험지("도메인 지식이 필요한가, 추상적 특성인가")로 특성 여부를 판단한다. GGG 카타에서는 입찰자 피드백·경매인·입찰자 세 퀀텀이 서로 다른 특성을 가져 하이브리드 아키텍처로 이어진다

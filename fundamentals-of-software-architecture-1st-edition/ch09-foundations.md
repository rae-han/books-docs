## 1. 아키텍처 스타일이란

PART II에서는 아키텍처 스타일들을 본격적으로 다룬다. 먼저 스타일과 패턴을 구분해야 한다.

- **아키텍처 스타일(*Architecture Style*)**: (모놀리식 배포든 별도 배포된 서비스든) 유저 인터페이스와 백엔드 소스 코드를 구성하는 방법, 그리고 소스 코드가 데이터 저장소와 상호작용하는 방법에 관한 **결정적으로 중요한 구조**
- **아키텍처 패턴(*Architecture Pattern*)**: 아키텍처 스타일 안에서 (일련의 작업이나 서비스 간의 고확장성·고성능을 실현하는 방법처럼) 특정한 솔루션을 마련하는 데 유용한 **저수준의 설계 구조**

아키텍처 스타일은 종종 아키텍처 패턴이라고도 부르며, 다양한 아키텍처 특성을 다루는 컴포넌트의 **명명된 관계(*Named Relationship*)**를 기술한다. 스타일의 명칭은 (설계 패턴이 그렇듯) 숙련된 아키텍트들 사이에서 간명하게 지칭할 수 있도록 붙여진 것이다. 한 아키텍트가 "레이어드 모놀리스"를 이야기하면, 상대는 그 구조적 측면과 어울리는(또는 문제를 일으킬) 아키텍처 특성, 통상적인 배포 모델과 데이터 전략 등을 한꺼번에 이해한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍처 스타일은 각 명칭마다 상당히 많은 세부 내용이 함축되어 있으며, 토폴로지와 기본 전제된 아키텍처 특성을 **이로운 것과 해로운 것 모두** 기술한다. 이 책 2부는 현대 아키텍처 패턴들을 두루 다루지만, 아키텍트는 규모가 더 큰 패턴에서 나타나는 몇 가지 **근본적인 패턴들**을 꿰고 있어야 한다.
:::

## 2. 기초 패턴

소프트웨어 아키텍처 역사를 통틀어 끊임없이 나타나는 패턴들이 있다. 예를 들어 "레이어"는 기능에 따라 서로 다른 관심사를 분리하는 개념으로, 소프트웨어 자체만큼이나 오래됐고 지금도 현대적 변형으로 계속 나타난다.

### 2.1 진흙잡탕

뭐 하나 뚜렷한 아키텍처 구조가 전무한 상태를 **진흙잡탕(*Big Ball of Mud*)**이라고 한다. 브라이언 푸트(Brian Foote)와 조지프 요더(Joseph Yoder)가 1997년 발표한 논문의 안티패턴 이름에서 유래했다.

> 진흙잡탕은 대충 되는 대로, 아무렇게나 막, 지저분하게, 테이프를 덕지덕지 붙여놓은 스파게티 코드 정글이다. 이런 시스템은 거의 반드시 무질서한 성장과 반복적인 땜질식 수리의 징후를 보인다. 정보는 멀찍이 떨어져 있는 시스템 파트끼리 난잡하게 공유되고, 거의 모든 중요한 정보는 전역에 흩어져 있거나 중복된 경우도 많다.<br>— 브라이언 푸트와 조지프 요더

진흙잡탕은 요즘은 실제 내부 구조라 할 만한 것이 하나도 없는, **데이터베이스를 직접 호출하는 이벤트 핸들러를 가진 단순한 스크립팅 애플리케이션**을 가리킨다. 별 대수롭지 않게 시작한 애플리케이션이 규모가 커지면서 처치 곤란한 상태가 되는 것이다. 구조가 없으면 변경·배포·테스트·확장·성능이 모두 고통스러워지므로 아키텍트는 무슨 수를 써서라도 이를 피하려 한다.

> 일부러 진흙잡탕을 만드는 아키텍트는 없겠지만, 많은 프로젝트가 **코드 품질 및 구조에 관한 거버넌스가 결여된 탓에** 본의 아니게 그렇게 된다. 실제로 닐은 클래스를 원 둘레의 점으로, 클래스 간 연결을 선으로 시각화했을 때 선이 굵고 빽빽하게 얽혀 사소한 변경조차 파급효과를 예측하기 어려운 고객사 코드베이스를 본 적이 있다.

### 2.2 유니터리 아키텍처

소프트웨어 태동기에는 단 1대의 컴퓨터에서 소프트웨어가 돌아갔다. 이후 하드웨어·소프트웨어는 진화하면서 처음엔 단일 엔티티로 시작했다가, 정교한 기능이 필요해지면서 나뉘었다. **유니터리(*Unitary - 단일, 통일*) 아키텍처**는 이제 임베디드 시스템과 극소수 제약 환경을 제외하면 거의 쓰이지 않는다. 소프트웨어는 시간이 지날수록 기능이 늘어나므로, 성능·확장 등의 운영 특성을 유지하려면 관심사를 분리할 필요가 있기 때문이다.

### 2.3 클라이언트/서버

단일 시스템에서 여러 기능을 분리할 필요성이 대두되었고, 그 분리 방법은 많은 아키텍처 스타일의 기초가 되었다. 프런트엔드와 백엔드로 기능을 분리한 **2티어(*Two-tier*) 또는 클라이언트/서버 아키텍처**가 대표적인 기본 스타일로, 시대와 컴퓨팅 파워에 따라 여러 형태로 존재해왔다.

1. **데스크톱 + 데이터베이스 서버**: 초창기 PC 아키텍처는 윈도우 같은 UI 기반의 리치 데스크톱 애플리케이션을 개발하고 데이터는 별도의 데이터베이스 서버로 분리했다. 프레젠테이션 로직은 데스크톱에, 계산량이 많은 액션은 사양이 탄탄한 데이터베이스 서버에서 실행했다
2. **브라우저 + 웹 서버**: 현대 웹 개발 시대가 오면서 웹 브라우저가 웹 서버에 접속하고, 웹 서버가 다시 데이터베이스 서버에 접속하는 형태가 일반화됐다. 클라이언트가 데스크톱보다 훨씬 가벼운 브라우저로 대체되었다
3. **3티어**: 1990년대 후반에 인기를 끌었다. 고성능 데이터베이스 티어, 애플리케이션 서버가 관리하는 애플리케이션 티어, 자바스크립트로 가득 찬 프런트엔드 티어로 완성됐다. 분산 아키텍처에 적합한 CORBA·DCOM 같은 네트워크 수준 프로토콜과 잘 맞았다

> **3티어, 언어 설계, 그리고 장기적 전망** — 자바가 처음 설계된 시절엔 3티어가 대세였고 "미래의 모든 시스템은 3티어가 될 것"이라는 전망이 지배적이었다. 그래서 자바 설계자는 네트워크를 통해 객체를 옮기기 위한 **직렬화(serialization)** 메커니즘을 언어의 코어 기능으로 넣었다(`Serializable` 인터페이스). 3티어가 불멸의 스타일이 될 거라 믿었기 때문이다. 이 스타일은 잠깐 떴다 졌지만 그 잔해는 아직 자바에 남아, 지금은 아무도 쓰지 않는 직렬화를 하위 호환성 때문에 계속 지원해야 하는 처지가 됐다. **소프트웨어에서 설계 결정의 장기적 영향을 내다보기란 항상 어렵고, "단순하게 설계하라"는 조언도 결국 미래의 결과에 대한 방어책이다.**

## 3. 모놀리식 대 분산 아키텍처

아키텍처 스타일은 크게 두 종류로 나뉜다.

- **모놀리식**: 전체 코드를 **단일 단위로 배포**하는 아키텍처
- **분산형**: 원격 액세스 프로토콜을 통해 **여러 단위로 배포**하는 아키텍처

완벽한 분류 체계는 없지만, 분산 아키텍처는 모놀리식에서는 찾아볼 수 없는 공통 난제를 갖고 있으므로 스타일을 분류하기에 좋은 기준이다. 이 책에서 다루는 스타일은 다음과 같다.

| 분류 | 아키텍처 스타일 | 다루는 장 |
|---|---|---|
| 모놀리식 | 레이어드 아키텍처 | 10장 |
| 모놀리식 | 파이프라인 아키텍처 | 11장 |
| 모놀리식 | 마이크로커널 아키텍처 | 12장 |
| 분산형 | 서비스 기반 아키텍처 | 13장 |
| 분산형 | 이벤트 기반 아키텍처 | 14장 |
| 분산형 | 공간 기반 아키텍처 | 15장 |
| 분산형 | 서비스 지향 아키텍처(SOA) | 16장 |
| 분산형 | 마이크로서비스 아키텍처 | 17장 |

분산 아키텍처는 성능·확장성·가용성 측면에서 훨씬 강력하지만, 이런 파워에는 **결코 무시할 수 없는 트레이드오프**가 수반된다.

## 4. 분산 컴퓨팅의 여덟 가지 오류

모든 분산 아키텍처에서 처음 맞닥뜨리는 이슈들은 1994년 L. 피터 도이치(L. Peter Deutsch)와 선 마이크로시스템즈의 동료들이 「분산 컴퓨팅의 오류(The Fallacies of Distributed Computing)」라는 글에서 최초로 거론했다. 여기서 오류(*Fallacy - 옳다고 믿거나 가정하지만 실은 틀린 것*)는 오늘날의 분산 아키텍처에도 그대로 적용된다.

| # | 오류 (틀린 가정) | 실제 진실과 대응 |
|---|---|---|
| 1 | 네트워크는 믿을 수 있다 | 네트워크는 여전히 미덥지 못하다. 서비스 B가 정상이어도 네트워크 문제로 A가 B에 닿지 못할 수 있다 → **타임아웃**, **회로 차단기(서킷 브레이커)** |
| 2 | 레이턴시는 0이다 | 원격 호출 시간(t_remote)은 항상 로컬 호출(t_local)보다 크다. **평균**보다 **95~99 백분위수**가 더 중요하다 |
| 3 | 대역폭은 무한하다 | 서비스가 잘게 쪼개지면 통신이 대역폭을 잠식한다. **스탬프 커플링**이 주범 |
| 4 | 네트워크는 안전하다 | 분산되면 공격 표면이 넓어진다. 모든 엔드포인트·서비스 간 통신에 보안이 필요해 성능이 떨어진다 |
| 5 | 토폴로지는 절대 안 바뀐다 | 라우터·허브·스위치·방화벽은 늘 바뀐다. 새벽의 "소소한" 네트워크 업그레이드가 타임아웃의 주범일 수 있다 |
| 6 | 관리자는 한 사람뿐이다 | 대기업 네트워크 관리자는 수십 명. 조율에 상당한 노력이 든다 |
| 7 | 운송비는 0이다 | 레이턴시가 아니라 **실제 비용**. 서버·게이트웨이·방화벽·프록시 등 인프라가 더 든다 |
| 8 | 네트워크는 균일하다 | 여러 업체 하드웨어(주니퍼·시스코 등)가 얽혀 있어 100% 완벽히 호환되지 않는다 |

### 4.1 오류 #2: 레이턴시는 0이다

로컬 인메모리 메서드 호출은 나노초~밀리초 단위지만, 같은 호출을 REST·메시징·RPC 같은 원격 프로토콜로 하면 밀리초 단위로 측정된다. 마이크로서비스는 서비스가 잘게 나뉘어 통신량이 많은데, 요청당 평균 레이턴시가 100밀리초이고 특정 기능에 10번 연달아 서비스 호출을 해야 하면 총 소요 시간은 1,000밀리초를 넘는다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 평균 레이턴시도 중요하지만 **95~99번째 백분위수**를 이해하는 것이 더 중요하다. 평균이 60밀리초(괜찮은 수치)여도 95번째 백분위수는 400밀리초일 수 있다. 보통 이런 **긴 꼬리(long tail)** 레이턴시가 분산 아키텍처의 성능을 저해하는 주범이다.
:::

### 4.2 오류 #3: 대역폭은 무한하다 — 스탬프 커플링

**스탬프 커플링(*Stamp Coupling - 필요 이상으로 큰 데이터를 통째로 주고받는 커플링*)**은 대역폭을 잠식하는 대표적 원인이다. 위시리스트 서비스(A)가 고객 이름만 필요한데 고객 프로필 서비스(B)가 45개 속성 500KB를 통째로 반환하는 경우를 보자. 필요한 데이터는 고객 이름 200byte뿐이다.

- 위시리스트 요청이 초당 2,000번 발생하면, 1회 서비스 간 호출에 소모되는 대역폭이 무려 **1GB**에 달한다
- B가 꼭 필요한 200byte만 돌려주면 대역폭은 **400KB** 정도면 충분하다

스탬프 커플링은 다음 방법으로 해결할 수 있다.

1. 프라이빗 REST API 엔드포인트를 둔다
2. 계약에 필드 셀렉터(field selector)를 사용한다
3. GraphQL로 계약을 분리한다
4. 컨슈머 주도 계약(CDC)과 값 주도 계약(VDC)을 병용한다
5. 내부 메시징 엔드포인트를 사용한다

어떤 기법이든 **서비스·시스템 간에 최소한의 데이터만 주고받도록** 하는 것이 이 오류를 바로잡는 최선의 길이다.

## 5. 그 밖의 분산 아키텍처 고려사항

여덟 가지 오류 외에도 분산 아키텍처에는 모놀리식에 없는 난제들이 있다.

- **분산 로깅**: 모놀리식은 로그가 하나뿐이라 요청을 추적하기 쉽지만, 분산 아키텍처는 로그 종류가 수백 가지에 위치·포맷도 제각각이라 근본 원인 파악이 어렵다. 스플렁크(Splunk) 같은 통합 도구도 완벽한 해결책은 아니다
- **분산 트랜잭션**: 모놀리식은 퍼시스턴스 프레임워크가 **ACID**(원자성·일관성·격리성·지속성) 트랜잭션으로 데이터 무결성을 강제한다. 분산 아키텍처는 **최종 일관성(*Eventual Consistency - 미리 알 수 없는 어느 시점에 데이터를 일관된 상태로 동기화*)**을 바탕으로 하며, 확장성·성능·가용성을 얻는 대가로 데이터 일관성을 희생한다
- **계약 관리 및 버저닝**: 계약(클라이언트와 서비스가 합의한 행위와 데이터)은 분리된 서비스를 각기 다른 팀이 소유하기에 유지보수가 특히 어렵고, 버전 구식화(deprecation)에 필요한 통신 모델은 더욱 복잡하다

분산 트랜잭션을 관리하는 방법으로 **트랜잭셔널 사가(Saga)**가 있으며, 보상을 위해 이벤트를 소싱하거나 유한 상태 기계로 트랜잭션 상태를 관리한다. 사가 외에 **BASE 트랜잭션**도 쓰는데, BASE는 기본 가용성(Basic Availability)·소프트 상태(Soft state)·최종 일관성(Eventual consistency)의 약자로, 소프트웨어 조각이 아니라 일종의 기법이다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 분산 아키텍처를 이제 막 시작하려는 아키텍트는 용량·대역폭·레이턴시·보안 구역 측면에서 **현재의 서버와 네트워크 토폴로지를 철저히 분석**하여 "착각의 늪"에 빠지지 않아야 한다. 네트워크가 빠르다고 주장하며 레이턴시를 외면하는 아키텍트가 많지만, 운영 환경의 평균 왕복 레이턴시를 아는 것이 분산 아키텍처의 실현 가능성을 판단하는 유일한 방법이다.
:::

## 요약

- **아키텍처 스타일**은 코드 구성과 데이터 상호작용에 관한 결정적 구조이고, **아키텍처 패턴**은 스타일 안에서 특정 문제를 푸는 저수준 설계 구조다. 스타일 명칭에는 토폴로지와 특성이 (이로운 것·해로운 것 모두) 함축되어 있다
- 기초 패턴: **진흙잡탕**(구조 없는 스파게티, 거버넌스 결여의 산물), **유니터리**(단일 컴퓨터, 이제 거의 안 쓰임), **클라이언트/서버**(데스크톱+DB → 브라우저+웹서버 → 3티어)
- 자바 직렬화 사례는 "설계 결정의 장기적 영향을 내다보기 어렵다"는 교훈을 준다 — 단순하게 설계하라는 조언은 미래에 대한 방어책이다
- 아키텍처 스타일은 배포 방식에 따라 **모놀리식**(레이어드·파이프라인·마이크로커널)과 **분산형**(서비스 기반·이벤트 기반·공간 기반·SOA·마이크로서비스)으로 분류된다
- 분산은 성능·확장성·가용성이 강력하지만 **여덟 가지 오류**(네트워크 신뢰·레이턴시 0·무한 대역폭·안전·불변 토폴로지·단일 관리자·운송비 0·균일성)라는 트레이드오프가 있다
- 특히 **레이턴시**는 95~99 백분위수(긴 꼬리)가 중요하고, **대역폭**은 스탬프 커플링이 주범이며 최소 데이터 전송으로 해결한다
- 그 밖에 분산 로깅, 분산 트랜잭션(ACID → 최종 일관성/BASE/사가), 계약 관리·버저닝이라는 난제가 있다. 시작 전 네트워크 토폴로지를 철저히 분석해야 한다

# Chapter 17: Microservices Architecture (마이크로서비스 아키텍처 스타일)

## 핵심 질문

마이크로서비스는 왜 "재사용보다 중복"을 택하며, 경계 콘텍스트가 어떻게 결정적 영향을 미쳤는가? 서비스 세분도는 어떻게 정하고 데이터는 왜 격리하는가? 사이드카·서비스 메시로 운영 관심사를 어떻게 재사용하는가? 오케스트레이션과 코레오그래피, 사가는 무엇인가? 왜 확장성·진화성 최고점과 성능·단순성 최하점이 공존하는가?

---

## 1. 역사와 철학 — 재사용보다 중복

마이크로서비스는 2014년 3월 마틴 파울러와 제임스 루이스의 블로그 「Microservices」로 널리 퍼졌다. **도메인 주도 설계(DDD)**, 특히 **경계 콘텍스트(bounded context)** 개념이 결정적 영향을 미쳤다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 경계 콘텍스트 안에서는 코드·데이터 스키마가 함께 연결되지만, 콘텍스트 **밖의 것들과는 전혀 커플링되지 않는다.** 재사용은 유익하지만 커플링이 문제다(제1법칙). **고도의 디커플링이 목표라면 재사용보다 중복을 우선한다** — 이것이 마이크로서비스의 핵심 철학이다.
:::

SOA의 재사용 집착이 낳은 커플링 재앙에 대한 반발로 탄생했다. 경계 콘텍스트의 논리적 개념을 **물리적으로 모델링**하는 극도의 디커플링을 추구한다.

---

## 2. 토폴로지

각 서비스가 **단일 목적**만 가지므로 SOA보다 서비스 규모가 훨씬 작다. 각 서비스는 데이터베이스 등 의존 컴포넌트를 포함해 **독립적으로 작동**한다.

```
      [클라이언트]  [클라이언트]  [클라이언트]
              \        |        /
            ┌─────── API 레이어 ───────┐
           /      |       |       \
      [서비스]  [서비스]  [서비스]  [서비스]
        |         |         |         |
       [DB]      [DB]      [DB]      [DB]   ← 각 서비스 전용 데이터
```

- **데이터 격리**: SOA의 단일 DB와 정반대. 각 서비스가 자기 데이터를 소유해 커플링을 끊는다. "재사용보다 중복"의 실천
- **API 레이어**(선택): 클라이언트와 서비스 사이의 프록시. 단, 오케스트레이션 도구나 미들웨어가 아니라 얇은 접근 계층이어야 한다

---

## 3. 세분도 — 얼마나 작아야 하는가

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: "마이크로"라는 이름이 함정이다. **너무 작게 나누면** 서비스 간 통신·오케스트레이션·트랜잭션 문제가 폭증한다. 목표는 "최대한 작게"가 아니라 **적절한 크기(purpose-driven)** 다.
:::

세분도 판단 가이드라인:
- **목적**: 서비스가 하나의 응집된 기능을 수행하는가
- **트랜잭션**: 여러 서비스가 ACID 트랜잭션을 공유해야 한다면 너무 잘게 나눈 것
- **코레오그래피**: 서비스 간 통신이 과하면 경계를 잘못 그은 것

---

## 4. 운영 관심사의 재사용 — 사이드카와 서비스 메시

도메인은 중복해도, 모니터링·로깅·인증 같은 **운영 관심사**는 재사용이 필요하다. **사이드카 패턴(sidecar pattern)** 으로 해결한다.

- 각 서비스 옆에 공통 운영 기능을 담은 **사이드카** 컴포넌트를 붙인다 (서비스 코드와 분리)
- 모든 사이드카를 연결하면 **서비스 메시(service mesh)** 가 되어 일관된 운영·통신 계층을 형성
- **서비스 디스커버리(service discovery)** 로 서비스를 동적으로 찾아 확장·리던던시를 실현

---

## 5. 통신 — 오케스트레이션 vs 코레오그래피

| 방식 | 특징 |
|---|---|
| **오케스트레이션(orchestration)** | 중재자 서비스가 워크플로를 조정 (지휘자). 복잡한 워크플로 제어에 유리하나 결합·병목 발생 |
| **코레오그래피(choreography)** | 중재자 없이 각 서비스가 서로 소통 (댄서들). 커플링이 적어 통신 속도·병목에 유리 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 마이크로서비스는 커플링이 적을수록 통신 속도가 빨라지고 병목이 줄기 때문에 **오케스트레이션보다 코레오그래피를 선호**한다. 다만 복잡한 비즈니스 프로세스에는 오케스트레이션이 필요할 수 있다.
:::

### 5.1 트랜잭션과 사가

서비스마다 DB가 분리되어 있어 **분산 트랜잭션**이 문제다. 널리 쓰이는 패턴이 **사가 패턴(saga pattern)**:

- 중재자 서비스가 여러 서비스 호출에 걸친 트랜잭션을 조정
- 트랜잭션 첫 파트는 성공했는데 두 번째가 실패하면, 중재자가 **보상 트랜잭션(compensating transaction)** 을 발행해 이전 작업을 되돌린다

> 트랜잭션이 여러 서비스에 걸쳐 있다면 세분도를 잘못 잡았을 가능성이 높다. 사가는 필요악이지 목표가 아니다.

## 6. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 도메인 |
| 퀀텀 수 | 하나 또는 여러 개 |
| 배포성 | ★★★★★ |
| 탄력성 | ★★★★★ |
| 진화성 | ★★★★★ |
| 내고장성 | ★★★★★ |
| 모듈성 | ★★★★★ |
| 전체 비용 | ★☆☆☆☆ |
| 성능 | ★★☆☆☆ |
| 신뢰성 | ★★★★☆ |
| 확장성 | ★★★★★ |
| 단순성 | ★☆☆☆☆ |
| 시험성 | ★★★★★ |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: **데브옵스 혁명과 운영 자동화의 진보가 없었다면 마이크로서비스는 존재하지 못했을 것이다.** 자동화 배포·시험성 등 현대 엔지니어링 프랙티스를 매우 훌륭하게 지원한다.
:::

- **주안점은 확장성·탄력성·진화성(모두 5점)**: 가장 확장성 우수한 시스템 다수가 이 스타일. 자동화·운영 통합으로 탄력성을 주고, 고도의 디커플링으로 아주 작은 배포 단위를 만들어 점진적 변화를 훌륭히 지원
- **배포성·시험성·모듈성 5점**: 작고 독립적인 서비스라 자동화 배포·테스트가 쉽다
- **내고장성 5점, 신뢰성 4점**: 독립적 단일 목적 서비스는 내고장성이 높지만, 서비스 간 통신을 과용하면 신뢰성이 위협받는다 (서비스 디스커버리·리던던시로 완화)
- **성능 2점**: 분산 특성상 잦은 네트워크 호출 + 엔드포인트마다 보안 체크로 오버헤드가 크다. 데이터 캐시·복제, 코레오그래피 선호로 개선
- **단순성 1점·비용 1점**: 극도의 디커플링은 "수많은 이들의 두통을 유발"하고 운영 비용이 높다
- **가장 독보적인 퀀텀**: 각 서비스 경계 = 도메인 = 퀀텀. 퀀텀 측정의 의미를 가장 잘 보여주는 사례

---

## 요약

- 마이크로서비스는 SOA의 커플링 재앙에 대한 반발로 탄생 — **경계 콘텍스트**를 물리적으로 모델링, **재사용보다 중복**으로 극도의 디커플링 추구
- 각 서비스가 **자기 데이터를 소유(데이터 격리)**, 단일 목적, API 레이어는 얇게
- 세분도의 핵심은 "최대한 작게"가 아니라 **적절한 크기** — 과분할은 통신·트랜잭션 문제를 부른다
- 운영 관심사는 **사이드카·서비스 메시**로 재사용, **서비스 디스커버리**로 확장·리던던시
- 통신은 **코레오그래피 선호**, 분산 트랜잭션은 **사가 + 보상 트랜잭션**
- **배포성·탄력성·진화성·내고장성·모듈성·시험성·확장성 5점**, **성능·단순성·비용은 최하점** — 데브옵스 자동화가 있어야 성립하는 현대적 아키텍처

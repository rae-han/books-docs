# Chapter 13: Service-Based Architecture Style (서비스 기반 아키텍처 스타일)

## 핵심 질문

서비스 기반 아키텍처는 왜 "가장 실용적인 분산 아키텍처"로 불리는가? 마이크로서비스와 무엇이 다르며, 왜 4~12개의 굵직한 도메인 서비스와 단일 공유 데이터베이스를 쓰는가? 데이터베이스 변경이 모든 서비스에 영향을 주는 문제는 어떻게 완화하는가? 세분도(granularity)의 차이가 왜 마이크로서비스와의 핵심 분기점인가?

---

## 1. 개요

서비스 기반 아키텍처(*Service-Based Architecture - 마이크로서비스의 일종이지만 훨씬 유연한 하이브리드 분산 아키텍처*)는 분산 아키텍처이면서도 **비교적 덜 복잡하고 비용이 적게 들어** 많은 비즈니스 애플리케이션에 널리 채택된다.

---

## 2. 토폴로지

기본 토폴로지는 따로 배포된 **유저 인터페이스 + 원격 도메인 서비스 + 단일 모놀리식 데이터베이스**로 이루어진 대규모 분산 레이어 구조다.

```
        [유저 인터페이스]
        /      |      \
  [도메인   [도메인   [도메인
   서비스]   서비스]   서비스]   ← 보통 4~12개, 평균 7개
        \      |      /
        [단일 공유 데이터베이스]
```

- **서비스(도메인 서비스)** 는 큼지막한 단위로 분리해 별도 배포한다. EAR·WAR·어셈블리처럼 배포하며 컨테이너화는 필수가 아니다
- 여러 서비스가 **단일 모놀리식 데이터베이스를 공유**하므로 서비스는 4~12개 (평균 7개) 정도
- 원격 액세스는 보통 REST(메시징·RPC·SOAP도 가능). 확장·내고장성 요구에 따라 서비스 인스턴스를 여럿 두고 부하 분산 가능

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: **중앙 공유 데이터베이스**를 쓴다는 점이 중요하다. 덕분에 기존 모놀리식 레이어드처럼 SQL 조인을 그대로 쓸 수 있고, 서비스가 적어 DB 커넥션도 문제가 안 된다. 다만 **데이터베이스 자체의 변경**은 모든 서비스에 영향을 줄 수 있는 이슈다.
:::

### 2.1 토폴로지 변형

유연성 덕분에 변형이 다양하다:
- **유저 인터페이스**: 단일 모놀리식 → 도메인 기반 → 서비스 기반으로 나눌 수 있다
- **데이터베이스**: 단일 → 도메인별 개별 DB로 분리 가능 (단, 한 도메인 데이터를 다른 도메인 서비스가 필요로 하지 않게 설계해 **서비스 간 통신을 방지**)
- **API 레이어**: 리버스 프록시/게이트웨이를 UI와 서비스 사이에 추가해 공통 관심사(메트릭·보안·감사·서비스 디스커버리)를 통합

---

## 3. 서비스 설계와 세분도

도메인 서비스는 단위가 커서 보통 내부를 **API 퍼사드 + 비즈니스 + 퍼시스턴스** 레이어드로 설계한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 서비스 기반과 마이크로서비스의 핵심 차이는 세분도다. 주문 처리 시 서비스 기반은 **내부 클래스 수준의 오케스트레이션**(하나의 OrderService 안에서 주문 생성·결제·재고 갱신)을 하고, 마이크로서비스는 **외부 서비스 수준의 오케스트레이션**(별도 배포된 여러 서비스 조율)을 한다.
:::

이 차이가 트랜잭션에 결정적 영향을 준다:

- **서비스 기반**: 굵은 단위라 단일 서비스 안에서 **ACID(*원자성·일관성·격리성·지속성*)** 트랜잭션(커밋/롤백)을 그대로 쓴다. 만료 신용카드 결제 실패 시 같은 서비스 내 원자적 트랜잭션이라 그냥 롤백하면 된다
- **마이크로서비스**: 잘게 나뉘어 **BASE(*기본적 가용성·소프트 상태·최종 일관성*)** 분산 트랜잭션에 의존한다. 주문은 삽입됐는데 결제가 실패하면 재고를 어떻게 할지 등 신경 쓸 문제가 많아진다

트레이드오프: 서비스 기반은 주문 처리를 변경하면 **전체 서비스를 테스트**해야 하지만, 마이크로서비스는 작은 서비스 하나만 영향받는다.

---

## 4. 데이터베이스 분할 — 엔티티 객체 문제

단일 공유 DB의 테이블 스키마 변경은 모든 서비스에 악영향을 줄 수 있다.

- **비효율적 구현**: DB 테이블을 나타내는 **엔티티 객체(entity object, 공유 클래스 파일)** 를 전체 서비스가 함께 쓰는 **단일 공유 라이브러리**(JAR/DLL)에 둔다. 테이블을 조금만 바꿔도 전체 서비스를 재배포해야 한다
- **개선책 — 연합 공유 라이브러리(federated shared library)**: DB를 논리적으로 분할(공통·고객·청구·주문·추적)하고, 도메인 서비스마다 해당 논리 분할을 바라보는 **전용 공유 라이브러리**를 둔다. 청구 테이블을 바꾸면 청구 서비스만 영향받는다

> **TIP**: 서비스 기반 아키텍처에서는 데이터베이스 변경을 통제하기 위해 데이터 도메인을 올바르게 정의·관리하고, 데이터베이스는 가급적 논리적으로 잘게 나누는 것이 좋다.

공통 엔티티 객체는 버전 관리 시스템에서 락킹하고 수정 권한을 DB 팀에만 부여해 변경을 통제한다.

## 5. 아키텍처 예시 — 전자제품 재활용 시스템

중고 전자제품 재활용 프로세스: 견적 → 수취 → 감정 → 회계 → (언제든) 제품 상태 조회 → 재활용 → 리포팅. 각 도메인을 개별 배포되는 도메인 서비스로 구현한다.

- **부분 확장**: 처리량이 높은 견적·제품 상태 서비스만 확장하고, 나머지는 단일 인스턴스로 충분
- **퀀텀 2개**: 고객 대면 부문(고객 UI + DB + 견적/상태 서비스)에 퀀텀 하나, 내부 작업 부문(수취·감정·재활용)에 퀀텀 하나 — 방화벽으로 단방향 접근해 데이터 보안 확보
- 감정 서비스는 신제품 출시 시 규칙이 추가되어 자주 변경되는데, 이를 단일 도메인 서비스로 격리하면 민첩성·시험성·배포성이 높아진다

## 6. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 도메인 |
| 퀀텀 수 | 하나 또는 여러 개 |
| 배포성 | ★★★★☆ |
| 탄력성 | ★★☆☆☆ |
| 진화성 | ★★★☆☆ |
| 내고장성 | ★★★★☆ |
| 모듈성 | ★★★★☆ |
| 전체 비용 | ★★★★☆ |
| 성능 | ★★★☆☆ |
| 신뢰성 | ★★★★☆ |
| 확장성 | ★★★☆☆ |
| 단순성 | ★★★☆☆ |
| 시험성 | ★★★★☆ |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 서비스 기반은 **별점 5개짜리 특성이 하나도 없지만 4개를 받은 영역이 많다.** "가장 구현하기 쉽고 비용 효율적인 분산 아키텍처"라는 매력이 있지만, 4개를 받은 모든 특성에 **비용 절감·단순함의 트레이드오프**가 깔려 있다 — 비용을 더 들이고 복잡해지면 등급이 올라간다.
:::

- **도메인 분할** — 도메인 변경 시 해당 서비스·UI만 영향. 분산이라 퀀텀은 1 이상 (DB·UI를 공유하면 1, 연합하면 여러 개)
- **배포성·시험성·진화성**: 도메인별로 나뉘어 테스트 커버리지가 좋고, 덜 위험하게 자주 배포 가능 → 시장 출시 단축
- **내고장성·신뢰성 4개**: 자기 완비형이고 서비스 간 통신이 거의 없어, 한 서비스가 실패해도 다른 서비스는 영향 없다. 대규모 서비스라 네트워크 트래픽·분산 트랜잭션이 적어 신뢰성 우수
- **확장성 3개·탄력성 2개**: 굵은 단위라 마이크로서비스만큼 세밀하게 확장하지 못하고, 중복 기능이 많아 효율이 떨어진다
- **단순성·전체 비용 4개**: 마이크로서비스·이벤트 기반·공간 기반 등 복잡한 분산 아키텍처와 차별화되는 강점

## 7. 언제 이 아키텍처 스타일을 사용하는가

> 출퇴근 시간에 시속 50km밖에 낼 수 없는 도로에서 페라리처럼 강력한 차가 있은들 무슨 소용일까? 겉모습만 멋지지 결국 리소스와 돈 낭비일 뿐이다.

- **가장 실용적인 분산 아키텍처** — 더 강력한 분산 스타일도 있지만 그 강력함은 가파른 비용 상승을 동반한다
- **도메인 주도 설계와 궁합이 좋다** — 서비스를 큰 단위로 나누고 범위를 도메인으로 한정
- **트랜잭션 관리가 수월** — 대부분 트랜잭션이 특정 도메인 서비스에 한정되어 전통적 커밋/롤백이 그대로 가능 (UI/게이트웨이가 여러 서비스를 오케스트레이션할 때만 사가·BASE 필요)
- 세분도의 함정에 빠지지 않고 **적당한 모듈성**을 달성할 수 있다

---

## 요약

- 서비스 기반은 **UI + 4~12개 굵직한 도메인 서비스 + 단일 공유 DB** 의 하이브리드 분산 아키텍처
- 마이크로서비스와의 핵심 차이는 **세분도** — 내부 클래스 오케스트레이션(ACID 유지) vs 외부 서비스 오케스트레이션(BASE 필요)
- 공유 DB의 변경 영향을 줄이려면 DB를 논리 분할하고 **연합 공유 라이브러리**로 엔티티 객체를 나눈다
- **별점 5개는 없지만 4개가 많다** — 배포성·시험성·내고장성·신뢰성·모듈성·전체비용이 강점, 약점은 탄력성·확장성
- "가장 구현하기 쉽고 비용 효율적인 분산 아키텍처". DDD 궁합이 좋고 ACID 트랜잭션이 잘 보존된다

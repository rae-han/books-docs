# Chapter 18: Choosing the Appropriate Architecture Style (최적의 아키텍처 스타일 선정)

## 핵심 질문

최적의 아키텍처 스타일은 무엇인가? 아키텍처 '유행'은 왜 끊임없이 변하는가? 스타일을 고를 때 어떤 결정 기준을 종합해야 하는가? 모놀리스냐 분산이냐, 데이터를 어디에 둘 것이냐, 통신을 동기로 할 것이냐 비동기로 할 것이냐 — 이 결정들을 어떤 순서로 내리는가?

---

## 1. "상황에 따라 다르다"

그렇다면 최적의 아키텍처 스타일은 무엇일까? 거듭 말하지만 **상황에 따라 다르다!** 어느 것을 쓰라고 자신 있게 권하고 싶어도 그것은 불가능하다. 조직 내부의 수많은 팩터와 어떤 소프트웨어를 구축하는지에 따라 맥락이 달라지기 때문이다.

아키텍처 스타일 선택은 **아키텍처 특성·도메인 고려사항·전략적 목표, 그밖의 다양한 것들의 트레이드오프를 고려하고 분석한 마지막 결과**다.

---

## 2. 아키텍처 '유행'은 계속 변한다

사람들이 좋아하는 아키텍처 스타일은 여러 팩터로 그때그때 계속 변한다.

| 팩터 | 내용 |
|---|---|
| **과거를 돌아보다** | 새 스타일은 과거의 쓰라린 경험을 반영한다. 코드 재사용의 단점을 겪은 아키텍트는 재사용의 함의를 진지하게 고민한다 |
| **생태계의 변화** | 끊임없는 변화가 소프트웨어 생태계의 특성이다. 수년 전 아무도 몰랐던 쿠버네티스가 지금은 대세지만, 이조차 몇 년 후 대체될지 모른다 |
| **새로운 기능** | 새 기능은 도구 교체를 넘어 완전히 새로운 패러다임 전환을 부른다. 도커 같은 컨테이너의 등장이 소프트웨어 개발 전반을 뒤흔든 것처럼 |
| **가속** | 생태계는 변할 뿐 아니라 변화 속도도 계속 빨라진다. 아키텍트는 일정한 흐름(flux) 속에 산다 |
| **도메인 변경** | 비즈니스가 진화하고 합병되면서 개발 도메인도 가만히 있지 않는다 |
| **기술 변화** | 조직은 최소한 분명하고 중요한 이익을 주는 기술 변화의 흐름은 따라잡으려 한다 |
| **외부 팩터** | 완벽히 만족하는 도구라도 라이선스 비용이 너무 비싸면 다른 도구를 알아봐야 하는 경우가 있다 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 조직이 유행 관점에서 어느 위치에 있든, 아키텍트는 **업계 동향을 잘 파악해 유행을 따를 때와 예외를 둘 때를 현명하게 결정**해야 한다. 새 도구뿐 아니라 새 패러다임까지 예의 주시하되, 모든 신기능이 게임 체인저는 아니라는 점도 안다.
:::

## 3. 결정 기준

아키텍트는 스타일을 고를 때 도메인 설계 구조의 원인이 될 만한 모든 요소를 종합해야 한다.

- **도메인**: 어느 부분이 운영 아키텍처 특성에 영향을 미치는지 파악한다. 전문가일 필요는 없어도 중요한 파트의 최소한의 지식은 있어야 한다
- **구조에 영향을 미치는 아키텍처 특성**: 도메인·외부 요소를 지원하는 데 필요한 특성을 정확히 밝혀낸다
- **데이터 아키텍처**: DBA와 협업해 신규 시스템이 기존 데이터 아키텍처와 상호작용할 때 데이터 설계가 아키텍처 설계에 미치는 영향을 파악한다
- **조직 문제**: 클라우드 이용료, 인수합병 계획 등 외부 팩터가 설계에 영향을 준다
- **프로세스·팀·운영 문제에 관한 지식**: 개발 프로세스·QA·운영팀 소통 등이 설계에 영향을 준다. 예: 애자일 성숙도가 낮은 조직에 애자일 기반 프랙티스에 의존하는 스타일을 도입하면 시작부터 난관에 부딪힌다

### 3.1 도메인/아키텍처 동형성

아키텍처의 토폴로지와 잘 맞는 문제 영역이 있다. 이 궁합을 **도메인/아키텍처 동형성(*Domain/Architecture Isomorphism*)**이라 한다.

- **잘 맞는 예**: 플러그인 커스터마이징이 필요한 시스템 ↔ 마이크로커널 / 무수히 많은 개별 연산이 필요한 게놈 분석 시스템 ↔ 공간 기반 아키텍처
- **불협화음 예**: 확장성이 우수한 시스템을 큰 모놀리식으로 설계하기 어렵다. 의미적으로 커플링된 문제 영역(예: 각 페이지가 이전 페이지 콘텍스트에 의존하는 멀티페이지 보험 웹앱)을 고도로 분리된 마이크로서비스로 모델링하면 갖가지 난제에 부딪힌다 — 이런 경우 커플링이 낮은 서비스 기반 아키텍처가 적합하다

## 4. 핵심 결정 세 가지

아키텍트는 위 사항을 종합해 다음 결정을 순서대로 내린다.

### 4.1 모놀리스냐, 분산이냐

3장의 **아키텍처 퀀텀(*Architecture Quantum*)** 개념을 이용해 판단한다.

- **단일 아키텍처 특성만으로 충분** → **모놀리스**가 적합 (다른 팩터로 결국 분산이 될 수는 있음)
- **시스템 파트별로 상이한 아키텍처 특성이 필요** → **분산 아키텍처**가 정답에 가깝다

### 4.2 데이터를 어디에 둘 것인가

- 모놀리식은 보통 하나 또는 소수의 관계형 DB를 전제한다
- 분산 아키텍처에서는 **어느 서비스가 어떤 데이터를 저장할지** 결정해야 한다. 워크플로를 구축하기 전에 전체 데이터 흐름을 충분히 숙고하고, 구조와 행위를 모두 고려해 최적 조합을 찾기 위해 **설계를 무한 반복**하는 것을 두려워하면 안 된다

### 4.3 서비스 간 통신은 동기냐, 비동기냐

- **동기 통신**: 대부분 더 간편하지만 확장성·안정성 등에 부정적일 수 있다
- **비동기 통신**: 성능·확장성이 우월하나 데이터 동기화, 데드락(교착 상태), 경합 조건, 디버깅 등 골칫거리가 많다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: **동기 통신을 기본으로 사용하고 필요할 때만 비동기 통신을 사용하라.** 동기 통신이 설계·구현·디버깅 문제가 비교적 적기 때문이다.
:::

이 과정의 결과물은 **선택한 스타일(및 하이브리드화)이 반영된 아키텍처 토폴로지 + 아키텍처 결정 기록(ADR) + 중요한 원칙과 운영 특성을 지키는 피트니스 함수**로 나타난다.

## 5. 모놀리스 사례 연구: 실리콘 샌드위치

아키텍처 특성을 조사한 결과 실리콘 샌드위치 카타는 **단일 퀀텀**으로 충분하고, 적은 예산으로 구축 가능한 단순 애플리케이션이라 모놀리스의 단순함이 어울린다고 결정했다. 두 가지 설계를 비교한다.

### 5.1 모듈러 모놀리스

단일 퀀텀으로 배포된 단일 DB 기반의 도메인 중심 시스템이다.

```
   ┌───────────────── 웹/모바일 UI ─────────────────┐
   │ [U컴포넌트][U컴포넌트][U컴포넌트]...             │
   └────────────────────┬───────────────────────────┘
                        ▼   (오버라이드 엔드포인트)
   [레시피][재고][구매][주문결제][프로모션][배송][주소]...
                        │        각 도메인 = 컴포넌트
                        ▼
                 [단일 관계형 DB]
```

- 총비용을 줄이려 단일 웹 UI(모바일 반영)와 단일 관계형 DB로 구현한다
- 식별된 각 도메인이 컴포넌트로 표시된다. 시간·리소스가 넉넉하면 도메인 단위로 테이블을 분리해 **나중에 분산 아키텍처로 쉽게 마이그레이션**되도록 설계한다
- 아키텍처가 커스터마이징을 지원하지 않으므로, 개발자가 커스터마이징 결과를 업로드하는 **오버라이드 엔드포인트**를 도메인 설계에 녹여 넣고, 각 도메인이 오버라이드 컴포넌트를 참조하게 해 피트니스 함수를 만든다

### 5.2 마이크로커널

식별된 특성 중 하나가 **맞춤성**이었으므로, 도메인/아키텍처 동형성을 검토해 마이크로커널로도 구현할 수 있다.

```
        [iOS BFF][안드로이드 BFF][웹 BFF]   ← JSON 어댑터
                        │
   ┌────────── 코어 시스템 (도메인 컴포넌트) ──────────┐
   │ [상품][레시피][주문][재고][결제][프로모션][위치]... │
   └────────────────────┬───────────────────────────┘
                        ▼
                 [단일 관계형 DB]
   + 커스터마이징 = 플러그인 (공통 플러그인 / 로컬 플러그인, 각자 데이터 보유)
```

- 코어 시스템은 도메인 컴포넌트와 단일 관계형 DB로 구성된다. 도메인·데이터를 신중히 동기화하면 향후 코어를 분산 아키텍처로 마이그레이션할 수 있다
- 커스터마이징은 플러그인으로 나타나며, 어느 플러그인도 서로 커플링되지 않아 자체 데이터를 유지할 수 있다
- **프런트엔드를 위한 백엔드(*BFF - Backends for Frontend*)** 패턴으로 API 레이어를 마이크로커널 어댑터로 쓸 수 있다. iOS용 BFF는 백엔드의 일반 출력을 데이터 포맷·페이지네이션·레이턴시 등 iOS 네이티브 사양에 맞게 변환한다

두 설계 모두 실리콘 샌드위치는 엄청난 성능·탄력성을 요하지 않고 오래 걸릴 작업도 없으므로 **동기 통신으로 충분**하다.

## 6. 분산 아키텍처 사례 연구: GGG

GGG(온라인 경매) 카타는 컴포넌트 분석 결과 **아키텍처 파트마다 상이한 특성**을 요구한다. 예를 들어 가용성·확장성이 입찰자와 경매인 역할마다 다르다. 또 야심 찬 규모·탄력성·성능이 요구된다.

대부분의 특성을 충족하는 후보는 저수준 이벤트 기반 또는 마이크로서비스인데, 이 중 **마이크로서비스가 다양한 운영 특성을 더 잘 지원**한다(순수 이벤트 기반은 통신 스타일(오케스트레이션/코레오그래피)로 구분될 뿐 운영 특성으로 나누지 않는다).

```
                          [API 레이어]
   ┌──────┬──────────┬──────────┬──────────┬──────────┐
   ▼      ▼          ▼          ▼          ▼          ▼
[Auction  [Auctioneer [Bid      [Bid       [Video     [Payment]
 Session]  Capture]    Capture]  Streamer]  Streamer]
                          │(비동기)   │(읽기전용)  ...
                          ▼
                     [Bid Tracker] ← 두 입찰 스트림을 실시간 정렬
                     (두 인바운드 커넥션 모두 비동기, 메시지 큐 버퍼)
```

주요 서비스:

- **BidCapture**: 온라인 입찰을 캡처해 BidTracker로 비동기 전송(관문 역할, 퍼시스턴스 없음)
- **BidStreamer**: 입찰 정보를 고성능 읽기 전용 스트림으로 스트리밍
- **BidTracker**: `AuctioneerCapture`·`BidCapture` 양쪽 입찰을 실시간에 가깝게 정렬. 두 인바운드가 모두 비동기라 메시지 큐를 버퍼로 써 다른 속도로 처리
- **AuctioneerCapture**: 경매인용 입찰 캡처. 특성이 완전히 달라 `BidCapture`와 분리
- **AuctionSession** / **Payment**(서드파티) / **VideoCapture** / **VideoStreamer**

아키텍트는 동기·비동기 스타일을 신중히 식별했다. 비동기를 택한 것은 주로 서비스마다 특성이 달라서다. 예를 들어 결제 서비스가 500밀리초마다 하나씩 처리 가능하고 많은 경매가 동시 종료되는 상황에서 동기 통신을 하면 타임아웃·안정성 문제가 생긴다. 이럴 때 **메시지 큐로 취약한 파트에 안정성을 확보**한다.

최종적으로 이 설계는 **5개 퀀텀**(Payment, Auctioneer, Bidder, Bidder Streams, BidTracker)으로 마무리됐다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 이것이 GGG의 정답도, 유일한 설계도, 심지어 최선의 설계(*best possible design*)도 아니다. 하지만 **트레이드오프가 가장 나쁜 것 중에서 제일 나은 설계**인 것 같다. 소프트웨어 아키텍처 제1법칙대로 완벽한 정답은 없고 각각 일장일단이 있다. 아키텍트가 할 일은 컴포넌트 설계 단계부터 퀀텀 분석을 활용해 서비스·데이터·통신 경계를 식별하는 것이다.
:::

## 요약

- 최적의 아키텍처 스타일은 없다. 스타일 선택은 아키텍처 특성·도메인·전략적 목표의 **트레이드오프를 분석한 마지막 결과**다(제1법칙: 모든 것은 트레이드오프)
- 아키텍처 '유행'은 과거 회고·생태계 변화·새 기능·가속·도메인/기술 변화·외부 팩터로 끊임없이 변한다. 아키텍트는 유행을 따를 때와 예외를 둘 때를 현명하게 결정한다
- 결정 기준은 **도메인·아키텍처 특성·데이터 아키텍처·조직 문제·프로세스/팀/운영 지식·도메인/아키텍처 동형성**을 종합한다
- 핵심 결정은 세 가지 순서다: (1) **모놀리스냐 분산이냐**(단일 특성이면 모놀리스, 파트별 상이한 특성이면 분산), (2) **데이터를 어디에 둘 것인가**, (3) **동기냐 비동기냐**(동기를 기본, 필요 시 비동기)
- **모놀리스 사례(실리콘 샌드위치)**: 단일 퀀텀·단순·저예산 → 모듈러 모놀리스 또는 마이크로커널(맞춤성). 동기 통신으로 충분
- **분산 사례(GGG)**: 파트별 상이한 특성 → 마이크로서비스, 특성 차이 때문에 비동기 채택, 5개 퀀텀. 정답이 아니라 "가장 나쁜 것 중 제일 나은" 트레이드오프
- 결과물은 **토폴로지 + 아키텍처 결정 기록(ADR) + 피트니스 함수**로 나타난다

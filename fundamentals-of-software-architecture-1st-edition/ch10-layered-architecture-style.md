# Chapter 10: Layered Architecture Style (레이어드 아키텍처 스타일)

## 핵심 질문

레이어드 아키텍처는 왜 가장 흔하고 자연스러운 아키텍처 스타일이 되었는가? 레이어를 "폐쇄"하거나 "개방"한다는 것은 무엇이며, 레이어 격리는 왜 중요한가? 아키텍처 싱크홀 안티패턴이란 무엇이고 언제 문제가 되는가? 언제 이 스타일을 선택하고 언제 벗어나야 하는가?

---

## 1. 개요 — 사실상의 표준

레이어드 아키텍처(*Layered Architecture - n티어 아키텍처라고도 함*)는 가장 흔한 아키텍처 스타일이다. 단순하고 대중적이며 비용도 적게 들어 **사실상의 표준(de facto standard)** 이다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 콘웨이의 법칙(*Conway's Law - 시스템 구조는 그것을 설계하는 조직의 소통 구조를 그대로 복제한다*)을 떠올려보면, 레이어드 아키텍처가 왜 자연스러운지 알 수 있다. UI 개발자·백엔드 개발자·DBA는 어느 회사에나 있고, 이 조직 계층 구조가 전통적인 레이어드 아키텍처와 잘 맞아떨어지기 때문이다.
:::

다만 이 스타일은 아키텍처 스타일을 확신하지 못한 채 "일단 코딩부터" 시작할 때 흘러드는 경우가 많아 **묵시적 아키텍처(architecture by implication)**, **우발적 아키텍처(accidental architecture)** 안티패턴에 속하기도 한다.

---

## 2. 토폴로지

내부 컴포넌트는 논리적으로 수평한 레이어들로 구성되며, 각 레이어가 애플리케이션에서 주어진 역할(프레젠테이션 로직, 비즈니스 로직 등)을 수행한다. 일반적으로 4개 표준 레이어로 구성한다.

```
┌─────────────────────────┐
│   프레젠테이션 레이어    │  ← UI, 브라우저 통신 로직
├─────────────────────────┤
│     비즈니스 레이어      │  ← 요청을 받아 비즈니스 규칙 실행
├─────────────────────────┤
│    퍼시스턴스 레이어     │  ← 데이터 접근 (SQL/HSQL)
├─────────────────────────┤
│    데이터베이스 레이어   │  ← 데이터 저장
└─────────────────────────┘
```

- 레이어 개수·유형에 특별한 제한은 없다. 퍼시스턴스 로직이 비즈니스 레이어에 내장되면 퍼시스턴스 레이어를 비즈니스 레이어에 병합한다
- 소규모 애플리케이션은 3개, 크고 복잡한 애플리케이션은 5개 이상의 레이어로 구성

### 2.1 물리적 배포 변형

배포 관점에서 여러 토폴로지 변형이 가능하다:

1. **프레젠테이션+비즈니스+퍼시스턴스를 단일 배포 단위**로 합치고 데이터베이스만 외부 분리
2. **프레젠테이션을 별도 배포 단위**로 떼고, 비즈니스+퍼시스턴스를 두 번째 단위로 합침 (DB는 외부)
3. **4개 레이어 전부를 단일 배포 단위**로 뭉침 — 인메모리 DB나 내장 DB를 쓰는 소규모 애플리케이션, 온프레미스(*on-premises - 클라우드가 아닌 자체 전산센터에 서버를 직접 설치·운영*) 제품에 적합

### 2.2 관심사의 분리

각 레이어는 주어진 비즈니스 요청을 충족하는 데 필요한 업무 위주로 추상화되어 있다. 프레젠테이션 레이어는 고객 데이터를 **어떻게 조회하는지** 알 필요가 없고, 비즈니스 레이어는 데이터를 **어디서 가져올지·어떻게 보여줄지** 관여하지 않는다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 관심사의 분리(*Separation of Concerns - 각 레이어가 자신의 역할에만 집중하도록 나누는 것*) 덕분에 개발자는 기술 역량을 특정 도메인(프레젠테이션 로직, 퍼시스턴스 로직)에 집중할 수 있다. 하지만 그 대가로 **전체적인 민첩성(변화에 신속히 반응하는 능력)이 떨어지는 트레이드오프**가 있다.
:::

레이어드 아키텍처는 **기술 분할된(technically partitioned)** 아키텍처다. 컴포넌트를 도메인(고객 등) 단위가 아니라 기술 역할(프레젠테이션·비즈니스 등)에 따라 묶는다. 그래서 '고객' 도메인이 모든 레이어에 분산되어 있어 도메인 변경이 어렵고, **도메인 주도 설계(DDD)와는 잘 맞지 않는다.**

## 3. 레이어 격리 — 폐쇄 vs 개방

각 레이어는 **폐쇄(closed)** 또는 **개방(open)** 상태다.

- **폐쇄 레이어(closed layer)**: 요청이 상위에서 하위로 이동하며 중간 레이어를 건너뛸 수 없다. 반드시 현재 레이어를 거쳐야 다음 레이어로 나아간다

```
요청 →  프레젠테이션 레이어  (폐쇄)
           ↓  건너뛸 수 없음
        비즈니스 레이어      (폐쇄)
           ↓
        퍼시스턴스 레이어    (폐쇄)
           ↓
        데이터베이스 레이어  (폐쇄)
```

단순 조회라면 레이어를 건너뛰고 프레젠테이션이 DB를 직접 액세스하는 편이 빠를 텐데(2000년대 초에는 이를 **추월 차선 리더 패턴(fast-lane reader pattern)** 이라 불렀다), 왜 폐쇄가 기본일까? 답은 **레이어 격리(layers of isolation)** 에 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 레이어 격리란 **어느 레이어에서 변경이 일어나도 다른 레이어의 컴포넌트에 영향을 주지 않는다**는 개념이다. 요청의 메인 흐름에 관한 레이어가 폐쇄되어 있어야 격리가 성립한다. 프레젠테이션이 퍼시스턴스를 직접 액세스하면, 퍼시스턴스 변경 시 비즈니스·프레젠테이션 모두 영향을 받아 단단히 커플링된 취약한 아키텍처가 된다.
:::

레이어를 격리하면 (잘 정의된 계약과 비즈니스 위임 패턴을 전제로) 어느 레이어든 다른 레이어에 영향을 주지 않고 교체할 수 있다. 예: 오래된 JSF(*JavaServer Faces*) 프레젠테이션 레이어를 React JS로 교체해도 다른 레이어에 영향이 없다.

## 4. 레이어 추가 — 서비스 레이어의 예

어떤 레이어는 개방하는 것이 합리적이다. 비즈니스 레이어에 공통 기능(날짜·문자열 유틸리티, 감사, 로깅)을 담은 공유 객체를 두고, **프레젠테이션이 이 공유 객체를 직접 쓰지 못하게** 강제하고 싶다고 하자.

이를 아키텍처적으로 강제하려면 공유 객체를 모은 **새 서비스 레이어**를 추가한다:

```
프레젠테이션 레이어   (폐쇄)
     ↓
비즈니스 레이어       (폐쇄)
     ↓ ↘ (건너뛰기 가능)
서비스 레이어         (개방)  ← 공유 컴포넌트
     ↓
퍼시스턴스 레이어     (폐쇄)
     ↓
데이터베이스 레이어   (폐쇄)
```

- 비즈니스 레이어가 폐쇄이므로 프레젠테이션은 공유 객체를 직접 액세스할 수 없다
- 서비스 레이어를 **개방**으로 두면, 비즈니스 레이어는 이 레이어를 액세스하거나(공유 객체 사용) 지나쳐 퍼시스턴스로 갈 수 있다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 어느 레이어가 개방/폐쇄인지 **(그리고 왜 그런지) 정확히 문서화하여 소통하지 않으면**, 테스트·유지보수·배포가 힘든 단단히 커플링된 아키텍처가 되어버린다.
:::

## 5. 아키텍처 싱크홀 안티패턴

**아키텍처 싱크홀 안티패턴(architecture sinkhole anti-pattern)** 은 요청이 레이어를 통과하면서 각 레이어가 **아무 비즈니스 로직도 처리하지 않고 그냥 통과시키는** 것을 말한다.

예: 기본 고객 데이터(이름·주소) 조회 요청이 프레젠테이션 → 비즈니스 → 규칙 → 퍼시스턴스로 내려가는데, 각 레이어가 아무 일도 안 하고 넘기기만 하다가, 퍼시스턴스가 단순 조회 SQL만 실행하고 취합·계산·규칙 적용 없이 왔던 길을 그대로 돌아간다. 불필요한 객체 초기화로 메모리를 낭비하고 성능에 악영향을 준다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 싱크홀 시나리오가 전무한 레이어드 아키텍처는 거의 없다. **80대 20 법칙**을 적용해 전체 요청의 20%가 싱크홀인 정도면 괜찮지만, 80%가 싱크홀이라면 이 문제 도메인에 레이어드 아키텍처는 부적합하다는 증거다.
:::

해결책으로 모든 레이어를 개방할 수도 있지만, 그러면 **변경 관리(레이어 격리)의 어려움이 가중되는 트레이드오프**가 있다.

## 6. 왜 이 아키텍처 스타일을 사용하는가

- 작고 단순한 애플리케이션이나 웹사이트에 알맞다
- **처음 구축을 시작할 때, 예산·일정이 빠듯할 때** 좋은 출발점 — 개발자·아키텍트 모두 익숙하고 복잡하지 않으며 가장 저렴한 축
- 아직 비즈니스 니즈를 분석 중이고 어떤 스타일이 최선인지 불확실할 때도 나쁘지 않다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 레이어드 아키텍처 기반 애플리케이션은 **규모가 커질수록 유지보수성·민첩성·시험성·배포성이 점점 나빠진다.** 따라서 대규모 시스템은 더 모듈러한 아키텍처 스타일이 잘 맞는다. 나중에 다른 스타일로 갈아탈 것을 대비해 재사용은 최소한으로, 객체 계층은 가볍게 유지하라.
:::

## 7. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 기술적 |
| 퀀텀 수 | 1 |
| 배포성 | ★☆☆☆☆ |
| 탄력성 | ★☆☆☆☆ |
| 진화성 | ★☆☆☆☆ |
| 내고장성 | ★☆☆☆☆ |
| 모듈성 | ★☆☆☆☆ |
| 전체 비용 | ★★★★★ |
| 성능 | ★★☆☆☆ |
| 신뢰성 | ★★★☆☆ |
| 확장성 | ★☆☆☆☆ |
| 단순성 | ★★★★★ |
| 시험성 | ★★☆☆☆ |

- **전체 비용·단순성이 주요 강점**: 모놀리식에 가까워 분산 복잡도가 낮고, 구조가 단순해 알기 쉽고 구축·유지보수 비용이 적다. 단, 규모가 커지면 이 장점도 빛이 바랜다
- **배포성·시험성이 매우 낮음**: 3줄만 고쳐도 전체를 재배포해야 하고 배포 리스크가 높다. 시험성에 별점 2개를 준 것은 컴포넌트를 모킹/스터빙할 수 있어 테스트 공수를 덜 수 있기 때문
- **신뢰성 3개**: 분산 아키텍처의 네트워크·대역폭·레이턴시 문제가 덜하지만, 모놀리식 배포·시험성·배포 리스크 탓에 그 이상 주기 어렵다
- **탄력성·확장성·내고장성·모듈성 1개**: 모놀리식 단일 퀀텀이라 특정 지점까지만 확장 가능하고, 한 부분의 OOM(Out-of-Memory)이 애플리케이션 전체 충돌로 이어진다

## 요약

- 레이어드 아키텍처는 콘웨이의 법칙 덕분에 **가장 자연스럽고 흔한** 스타일이지만, "일단 코딩부터" 흘러드는 묵시적/우발적 아키텍처가 되기 쉽다
- 표준 4레이어(프레젠테이션·비즈니스·퍼시스턴스·데이터베이스)로 **관심사를 분리**하되, 기술 분할이라 도메인 변경은 어렵다 (DDD와 안 맞음)
- **폐쇄 레이어**가 기본 — **레이어 격리**로 한 레이어 변경이 다른 레이어에 영향을 주지 않게 한다. 필요하면 서비스 레이어를 **개방**으로 추가
- **아키텍처 싱크홀 안티패턴**(로직 없이 통과만) 비율이 80%면 부적합 신호. 80대 20 법칙으로 판단
- 강점은 **단순성·저비용**, 약점은 **배포성·확장성·탄력성·내고장성**. 소규모·초기 구축·불확실할 때의 좋은 출발점이며, 커지면 더 모듈러한 스타일로 이동

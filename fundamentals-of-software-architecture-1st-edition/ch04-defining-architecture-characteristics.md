# Chapter 4: Defining Architecture Characteristics (아키텍처 특성 정의)

## 핵심 질문

아키텍처 특성이란 무엇이며 왜 '비기능 요구사항'이라 부르면 안 되는가? 어떤 요소가 아키텍처 특성이 되려면 어떤 기준을 충족해야 하는가? 아키텍처 특성은 어떻게 분류되는가(운영·구조·공통)? 왜 모든 아키텍처 특성을 다 지원하는 시스템은 실패하는가?

---

## 1. 아키텍처 특성이란 무엇인가

어떤 문제를 소프트웨어로 해결하려면 아키텍트는 시스템 요구사항을 취합하고 필요한 기술을 정리한다. 그런데 아키텍트는 이 밖에도 많은 부분을 고려해야 한다. 아키텍트는 개발팀과 함께 도메인/비즈니스 요구사항을 정의하기도 하지만, 주로 **도메인 기능과 직접적인 관련이 없는 모든 것**, 즉 아키텍처 특성(*Architecture Characteristic - 도메인 기능과 직교하는, 시스템의 성공 기준*)을 정의·발견·분석하는 일을 수행한다.

회사에서는 이런 특성을 흔히 비기능 요구사항(*Nonfunctional Requirement - 기능 요구사항과 구분하려 만든 용어*)이라 부르지만, 저자들은 이 용어를 좋아하지 않는다. '비기능적'이라는 말은 언어 자체가 부정적인 인상을 주기 때문이다. 다른 팀에서 '비기능적인' 뭔가에 관심을 가질 리 없다. 품질 속성(*Quality Attribute*)이라는 말도 설계보다는 사후 품질 평가라는 느낌이 있어 불만이다. 그래서 저자들은 시스템 성공에 중차대한 관심사임을 나타내고 그 중요성을 폄하하지 않는 **아키텍처 특성**이라는 용어를 선호한다.

### 1.1 아키텍처 특성의 세 가지 기준

아키텍처 특성은 다음 세 가지 기준을 충족한다.

1. **비도메인 설계 고려사항을 명시한다**: 요구사항이 '무엇을' 해야 하는지를 정의한다면, 아키텍처 특성은 그것을 '어떻게' 구현하고 왜 그런 선택을 했는지와 관련된 운영/설계 기준을 명시한다. 예를 들어 성능은 중요한 특성이지만 요구사항 정의서에 적혀 있지 않은 경우가 허다하고, '기술 부채를 방지한다'는 문구는 어디에도 없지만 당연한 설계 고려사항이다
2. **설계의 구조적 측면에 영향을 미친다**: '이 특성은 어떤 특별한 구조적 요소를 고려해야 하는가?'가 핵심이다. 예를 들어 보안은 모든 프로젝트의 기본 관심사이지만, 아키텍트가 뭔가 특별한 것을 설계해야 할 때 비로소 아키텍처 특성 수준으로 격상된다
3. **애플리케이션 성공에 (절대적으로) 중요하다**: 지원하는 특성을 하나만 늘려도 설계 복잡도가 그만큼 가중된다. 따라서 **가급적 아키텍처 특성을 적게 선정하는 일**도 아키텍트의 중요한 책무다

두 번째 기준을 결제 시스템 사례로 보자.

- **서드파티 결제 프로세서**: 결제 처리를 한 곳에서 처리한다면 아키텍트가 특별히 구조에 신경 쓸 일은 없다. 암호화·해시 등 표준 보안 장치를 설계에 반영하면 된다
- **애플리케이션 내부 결제 처리**: 애플리케이션이 직접 결제를 처리한다면 아키텍트는 보안 문제를 구조적으로 분리하기 위해 특정 모듈·컴포넌트·서비스를 설계해야 한다. 이 경우 보안은 아키텍처와 설계 모두에 영향을 미친다

### 1.2 명시적 특성 vs 암묵적 특성

- **명시적(explicit) 아키텍처 특성**: 요구사항 정의서나 지침서에 기재된다
- **암묵적(implicit) 아키텍처 특성**: 요구사항 정의서에는 거의 안 나오지만 프로젝트 성공에 꼭 필요한 특성이다. 가용성·신뢰성·보안은 거의 모든 애플리케이션의 근간이지만 설계 문서에는 좀처럼 등장하지 않는다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍트는 분석 단계에서 문제 영역에 대해 습득한 지식을 최대한 활용해 **암묵적 아키텍처 특성**을 밝혀내야 한다. 예를 들어 거래가 매우 잦은 무역 회사의 요구사항 정의서에 '낮은 레이턴시'가 명시되지 않아도, 아키텍트는 이 도메인에서 낮은 레이턴시가 얼마나 중요한지 간파해야 한다.
:::

## 2. 아키텍처 특성 목록

아키텍처 특성은 저수준 코드 특성부터 복잡한 운영 문제까지 넓은 범위에 존재한다. 아직 보편적 표준은 없고 조직마다 자체적으로 정의하지만, 아키텍트는 보통 특성을 넓은 범주로 나눈다.

### 2.1 운영 아키텍처 특성

운영 아키텍처 특성(*Operational Architecture Characteristic*)은 성능·확장성·탄력성·가용성·신뢰성 등의 능력을 말하며, 운영 및 데브옵스와 많은 부분에서 중첩된다.

| 용어 | 정의 |
|---|---|
| **가용성(availability)** | 시스템이 얼마나 오랫동안 사용 가능해야 하나? (24/7이면 장애 시 신속하게 재가동시키는 절차가 준비돼야 함) |
| **연속성(continuity)** | 재해 복구 능력 |
| **성능(performance)** | 스트레스 테스트, 피크 분석, 사용 빈도 분석, 필요 용량, 응답 시간. 충분한지 확인하려면 직접 돌려봐야 하며 수개월이 걸린다 |
| **복구성(recoverability)** | 비즈니스 연속성 요구사항. 백업 전략과 하드웨어 다중화 요건에 영향을 미친다 |
| **신뢰성/안전(reliability/safety)** | 시스템에 페일 세이프(*Fail-Safe - 고장 시 안전한 방향으로 흘러가게 하는 설계*)가 필요한가? 실패 시 회사에 거액의 손실이 발생하는가? |
| **견고성(robustness)** | 실행 중 인터넷 접속 끊김, 정전, 하드웨어 실패 등 에러 및 경계 조건을 감당하는 능력 |
| **확장성(scalability)** | 유저 수·요청 수가 늘어나도 시스템이 그에 맞는 성능을 발휘하는 능력 |

### 2.2 구조 아키텍처 특성

아키텍트는 코드 구조에도 심혈을 기울여야 하며, 우수한 모듈성·컴포넌트 커플링 제어·가독성 등 코드 품질 문제를 전담(또는 공동 담당)한다.

| 용어 | 정의 |
|---|---|
| **설정성(configurability)** | 최종 유저가 (쉬운 인터페이스로) 소프트웨어 설정을 쉽게 바꿀 수 있는가? |
| **신장성(extensibility)** | 새로운 기능을 삽입하는 일의 중요성 |
| **설치성(installability)** | 필요한 모든 플랫폼에 시스템을 얼마나 손쉽게 설치할 수 있나? |
| **활용성/재사용(leverageability/reuse)** | 공통 컴포넌트를 여러 제품에서 활용할 수 있나? |
| **지역성(localization)** | 다국어가 지원되는가? 멀티바이트 문자(*Multibyte Character - 한중일처럼 2바이트 이상 필요한 문자*), 측정·화폐 단위 요구사항 |
| **유지보수성(maintainability)** | 시스템을 얼마나 쉽게 변경/개선할 수 있나? |
| **이식성(portability)** | 하나 이상의 플랫폼에서 시스템을 실행할 수 있나? |
| **지원성(supportability)** | 어느 정도의 기술 지원이 필요한가? 에러 디버깅을 위한 로깅 수준은? |
| **업그레이드성(upgradeability)** | 구버전을 새 버전으로 쉽고 빠르게 업그레이드할 수 있는가? |

### 2.3 아키텍처 공통 특성

쉽게 분류하기 어렵지만 중요한 설계 제약조건과 고려사항들이다.

| 용어 | 정의 |
|---|---|
| **접근성(accessibility)** | 색맹·청각장애인 등 모든 유저가 불편함 없이 접근할 수 있나? |
| **보관성(archivability)** | 데이터를 아카이빙해야 하나, 일정 시간 경과 후 삭제해야 하나? |
| **인증(authentication)** | 유저가 본인임을 증명하기 위해 필요한 보안 요구사항 |
| **인가(authorization)** | 유저가 애플리케이션에서 정해진 기능만 사용하도록 강제하는 보안 요구사항 |
| **합법성(legal)** | 법적 제약조건이 있는가? (SOX, GDPR 등) 회사는 어떤 권리를 유보해야 하나? |
| **프라이버시(privacy)** | 회사 내부 임직원의 트랜잭션을 외부에 드러내지 않는 기능 (암호화 트랜잭션은 DBA·네트워크 아키텍트도 해독 불가) |
| **보안(security)** | 데이터를 암호화해 저장해야 하나? 내부 통신도 암호화해야 하나? 원격 액세스에 어떤 인증이 필요한가? |
| **사용성/성취성(usability/achievability)** | 유저가 원하는 목적을 달성하기 위해 필요한 교육/훈련 수준 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 아키텍처 특성은 어떻게 나열해도 **불완전한 목록**이 될 수밖에 없다. 게다가 정의가 미묘하거나 중복된다. 예를 들어 상호운용성(interoperability)은 문서화된 API를 통한 통합 용이성을, 호환성(compatibility)은 산업·도메인 표준과의 연관성을 가리켜 미묘하게 다르다. 가용성과 신뢰성은 거의 모든 경우에 중복된다.
:::

### 2.4 ISO 정의

국제표준기구(ISO)도 기능별 목록을 발표했지만 역시 중복되고 불완전하다. 다음은 ISO 정의에서 일부 발췌한 것이다.

| ISO 특성 | 하위 특성 |
|---|---|
| **성능효율(performance efficiency)** | 시간 특징(응답·처리 시간, 처리율), 리소스 사용, 능력 |
| **호환성(compatibility)** | 공존, 상호운용성 |
| **사용성(usability)** | 적합성 인지도, 학습성, 유저 에러 방지, 접근성 |
| **신뢰성(reliability)** | 성숙도, 가용성, 내고장성, 복구성 |
| **보안(security)** | 기밀성, 무결성, 부인방지, 책임소재, 진위 |
| **유지보수성(maintainability)** | 모듈성, 재사용성, 분석성, 수정성, 시험성 |
| **이식성(portability)** | 적응성, 설치성, 교체성 |

ISO 목록의 마지막 항목인 **기능적합성(functional suitability)**은 기능 완전성·기능 정확성·기능 타당성 등 소프트웨어의 기능적 측면을 다루므로, 저자들은 아키텍처 특성 목록에 속하지 않는다고 본다.

> 소프트웨어 아키텍처 자체의 활동을 비롯해 대부분의 중요한 내용이 명확히 정의되어 있지 않다는 사실에 많은 아키텍트가 끊임없는 좌절감에 빠지곤 한다! 그래서 저자들은 도메인 주도 설계(DDD)의 조언을 따라, 동료들끼리 오해하지 않도록 서로 보편적인 언어(*Ubiquitous Language - 팀이 공유하는 통일된 도메인 용어*)를 정해서 사용하길 권장한다.

## 3. 트레이드오프 및 나쁜 것 중에서 제일 나은 아키텍처

지금까지 열거한 아키텍처 특성은 여러 이유로 일부만 지원 가능하다. 첫째, 특성마다 설계 노력이 필요하고 구조적으로 지원돼야 한다. 둘째, 더 큰 문제로 **각 특성이 다른 특성에 영향을 미치는 경우가 많다.** 예를 들어 보안을 강화하면 십중팔구 성능에 부정적인 영향을 미친다. 실시간 암호화나 시크릿을 숨기는 간접적 기능을 수행하는 과정에서 성능이 떨어지기 때문이다.

> 헬리콥터 조종사는 양손·양발로 조종을 하는데, 손과 발을 독립적으로 움직이기가 쉽지 않아 배우는 사람들이 애를 먹는다. 헬리콥터를 띄우는 것은 균형을 잡는 연습인데, 이는 아키텍처 특성을 선택할 때 트레이드오프가 발생하는 상황과 비슷하다.

시스템을 설계하며 모든 아키텍처 특성을 빠짐없이 최상으로 반영하기란 불가능에 가깝다. 아키텍처 특성을 너무 욕심내면 모든 비즈니스 문제를 해결하려는 일반적인 솔루션이 되어버리고, 그런 아키텍처는 설계하기가 대단히 까다로워 실현 가능성이 낮다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 최고(best)의 아키텍처를 고집하지 말고 **나쁜 것 중에서 제일 나은(least worst) 아키텍처**를 선택하라. 아키텍처를 쉽게 변경할 수 있다면 첫술에 배불러야 한다는 강박관념도 사라진다. 반복(iteration)의 가치는 애자일 소프트웨어 개발에서 가장 중요한 교훈 중 하나로, 아키텍처를 비롯한 모든 레벨의 소프트웨어 개발에 적용된다.
:::

## 요약

- 아키텍처 특성은 **도메인 기능과 직접 관련 없는 모든 것**으로, '비기능 요구사항'이나 '품질 속성'보다 폄하되지 않는 용어다
- 아키텍처 특성의 세 가지 기준은 **① 비도메인 설계 고려사항 명시 ② 설계의 구조적 측면에 영향 ③ 애플리케이션 성공에 절대적으로 중요**이다
- 요구사항 정의서에 기재되면 **명시적 특성**, 성공에 필요하지만 문서에 없으면 **암묵적 특성**이다
- 아키텍처 특성은 **운영(성능·가용성·확장성 등), 구조(설정성·유지보수성·이식성 등), 공통(보안·인증·프라이버시 등)** 으로 분류되지만, 어떤 목록도 불완전하고 정의가 서로 중복된다
- 특성은 서로 영향을 미치므로 모두를 최상으로 만들 수 없다. 아키텍트는 **최고가 아니라 '나쁜 것 중에서 제일 나은' 아키텍처**를 반복적으로 찾아야 한다

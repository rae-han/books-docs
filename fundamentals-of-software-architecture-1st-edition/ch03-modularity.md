# Chapter 3: Modularity (모듈성)

## 핵심 질문

모듈성이란 무엇이며 왜 정의하기 어려운가? 모듈성을 정량적으로 측정하는 도구에는 어떤 것이 있는가? 응집·커플링·커네이선스는 각각 무엇을 측정하며 어떻게 다른가? 구심/원심 커플링, 추상도·불안정도·메인 시퀀스로부터의 거리는 어떻게 계산하는가? 왜 정적 커네이선스가 동적 커네이선스보다 바람직한가?

---

## 1. 모듈성이란 무엇인가

플랫폼마다 코드 재사용 메커니즘은 제각각이지만, 연관된 코드를 '모듈(*Module - 복잡한 구조를 만드는 데 쓰이는 표준화된 부품이나 독립적 단위*)'로 묶는 방법은 모두 지원한다. 그런데 모듈성은 사실 제대로 정의하기가 쉽지 않은 개념이다.

> 소프트웨어 아키텍처 용어의 95%는 '모듈성'의 이로움을 찬양하는 데 사용되고 있지만, 정작 모듈성을 어떻게 달성할지에 대해서는 별다른 얘기가 없다.<br>— 글렌포드 J. 마이어스(Glenford J. Myers), 1978년

이 책에서는 객체지향 언어의 클래스나 함수형 언어의 함수처럼, 서로 연관된 코드를 논리적으로 묶은 덩어리를 **모듈성**이라는 일반 용어로 나타낸다. 모듈성은 일종의 구성 원리(*Organizing Principle - 시스템을 조직하는 원칙*)다. 물리학에 비유하자면 소프트웨어 시스템은 엔트로피(*Entropy - 무질서*)가 증가하는 방향으로 움직이는 복잡한 시스템을 모델링한다. 질서를 유지하려면 물리적 시스템에 에너지를 투입해야 하듯, 아키텍트도 끊임없이 에너지를 소비해 시스템을 구조적으로 탄탄하게 유지해야 한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 모듈성은 대표적인 **암묵적(implicit) 아키텍처 특성**이다. 모듈이 확실히 구분되고 모듈 간 통신이 잘 되어야 한다고 공식적으로 요구하는 프로젝트는 드물지만, 코드베이스를 잘 발전시키려면 암묵적으로 질서와 일관성이 필요하다.
:::

### 1.1 논리적 구분 vs 물리적 구분

모듈성은 **논리적인 구분**이지 물리적인 구분은 아니며, 이 차이가 굉장히 중요할 때가 있다. 예를 들어 모놀리식 애플리케이션은 편의상 꽤 많은 클래스를 한 덩이로 묶어도 크게 상관없지만, 아키텍처를 재구축할 때는 이렇게 커플링된 구조가 모놀리스를 나누는 데 걸림돌이 된다.

역사적으로 모듈의 개념은 진화해왔다. 1968년 에츠허르 데이크스트라(Edsger Dijkstra)의 「GOTO 문의 해로움」 논문은 구조적 프로그래밍 언어(파스칼, C)의 시대를 열었고, 이후 모듈러 언어(모듈라(Modula), 아다(Ada))가 잠시 번성했다. 객체지향 언어가 인기를 끌면서 모듈러 언어는 사라졌지만, 언어 설계자들은 모듈의 유용함을 깨닫고 패키지(자바)·네임스페이스(닷넷) 같은 형태로 모듈을 존속시켰다.

## 2. 모듈성 측정 — 세 가지 핵심 개념

모듈성을 이해하는 데 도움이 되는 언어 독립적 메트릭들이 있다. 여기서는 세 가지 핵심 개념 — **응집, 커플링, 커네이선스** — 를 집중적으로 살펴본다.

## 3. 응집(Cohesion)

응집(*Cohesion - 한 모듈의 파트(구성 요소)가 얼마나 서로 연관되어 동일한 모듈 안에 포함되어 있는지*)은 모듈을 구성하는 파트가 서로 얼마나 연관되어 있는가를 나타낸다. 이상적으로 응집된 모듈이라면 모든 파트가 함께 패키징되어 있다.

> 응집된 모듈을 나누려고 해봐야 더 커플링되고 가독성은 떨어진다.<br>— 래리 콘스탄틴(Larry Constantine)

### 3.1 응집도의 종류

컴퓨터과학자들은 응집도의 측정 범위를 정의했다. 다음은 가장 좋은 것부터 나쁜 것 순서다.

| 응집도 (좋음 → 나쁨) | 정의 |
|---|---|
| **기능적 응집(functional)** | 각 파트가 다른 파트와 연관되어 있고, 기능상 꼭 필요한 모든 것이 모듈에 들어 있다 |
| **순차적 응집(sequential)** | 한쪽이 데이터를 출력하면 다른 한쪽이 그것을 입력받는 형태로 상호작용한다 |
| **소통적 응집(communication)** | 통신 체인을 형성한다 (예: DB에 레코드를 추가하면 그 정보로 이메일을 만든다) |
| **절차적 응집(procedural)** | 정해진 순서대로 실행되어야 한다 |
| **일시적 응집(temporal)** | 시점 의존성으로 연관된다 (예: 시동 시 관련 없어 보이는 것들을 죽 초기화) |
| **논리적 응집(logical)** | 논리적으로만 연관될 뿐 하는 일은 전혀 다르다 (예: 텍스트·직렬화 객체·스트림을 변환하는 `StringUtils`) |
| **동시적 응집(coincidental)** | 같은 소스 파일에 있을 뿐 아무 연관성도 없다. **가장 나쁜 형태** |

응집은 커플링보다 덜 정확한 메트릭이므로 아키텍트 재량에 따라 측정값이 달라진다. 예를 들어 `고객관리` 모듈에 '고객 추가/수정/조회/알림'과 함께 '고객 주문 조회/취소'가 있다면, 이 마지막 두 항목을 `주문관리` 모듈로 분리해야 할까? 답은 "경우에 따라 다르다." 주문 관리 작업이 앞으로 확장될지, 두 모듈이 작동하려면 고객 정보를 얼마나 알아야 하는지에 따라 달라진다. 이런 질문이 바로 아키텍트 업무의 핵심인 **트레이드오프 분석**이다.

### 3.2 LCOM 메트릭

카이댐버와 케메러(Chidamber and Kemerer)는 객체지향 메트릭 스위트를 만들었고, 그중 **LCOM(*Lack of Cohesion in Methods - 메서드의 응집 결여도*)**은 모듈(보통 컴포넌트)의 구조적 응집도를 나타낸다. LCOM은 다음과 같이 정의하면 이해하기 쉽다.

`LCOM = 공유 필드를 통해 공유되지 않는 메서드의 총 개수`

예를 들어 프라이빗 필드 `a`, `b`가 있는 클래스에서 `a`만 액세스하는 메서드와 `b`만 액세스하는 메서드가 따로 있다면, 공유 필드를 통해 공유되지 않는 메서드가 많으므로 LCOM 점수가 높다. 즉, 응집 결여도가 높은 것이다. LCOM 점수가 낮을수록 구조적 응집이 우수하다.

LCOM 메트릭은 아키텍처 스타일을 전환하기 위해 코드베이스를 분석하는 아키텍트에게 유용하다. 특히 공유 유틸리티 클래스처럼 **어쩔 수 없이 커플링된 클래스, 처음부터 한 클래스가 아니었던 클래스**를 발견하는 데 도움이 된다. 단, LCOM이 찾아내는 것은 '구조적 응집 결여도'일 뿐, 어떤 조각들이 서로 논리적으로 잘 맞는지는 판단하지 못한다. '어떻게'보다 '왜'가 더 중요한 소프트웨어 아키텍처 제2법칙이 적용되는 대목이다.

## 4. 커플링(Coupling)

코드베이스의 커플링은 그래프 이론에 기반한 좋은 분석 도구들이 많다. 메서드의 호출과 반환은 호출 그래프를 형성하므로 수학적 분석이 가능하다. 1979년 에드워드 요던(Edward Yourdon)과 래리 콘스탄틴이 쓴 『Structured Design』에는 구심 커플링과 원심 커플링을 비롯한 중요한 개념들이 등장한다.

| 유형 | 정의 | 니모닉 |
|---|---|---|
| **구심 커플링(afferent, Ca)** | 코드 아티팩트로 **유입되는(incoming)** 접속 수 | `a`는 `e`보다 먼저 → 유입이 유출보다 먼저 |
| **원심 커플링(efferent, Ce)** | 다른 코드 아티팩트로 **유출되는(outgoing)** 접속 수 | `e`는 `exit`(출구) → 밖으로 빠져나감 |

두 용어는 시작하는 모음(`a`, `e`)만 다르고 발음도 비슷하다. 요던과 콘스탄틴은 뚜렷한 의미보다 수학적 대칭성을 중시해 이런 이름을 붙였으므로, 위 니모닉(*Mnemonic - 첫 글자로 외우기 쉽게 연상하는 방법*)으로 기억하면 좋다.

## 5. 추상도, 불안정도, 메인 시퀀스로부터의 거리

컴포넌트 커플링에서 파생된 메트릭들도 있다. 이들은 로버트 마틴(Robert Martin)이 자신의 C++ 책에서 처음 사용했지만 다른 객체지향 언어에도 널리 적용된다.

### 5.1 추상도(Abstractness)

추상도는 (추상 클래스·인터페이스 등의) 추상 아티팩트와 구상 아티팩트(구현체)의 비율, 즉 구현 대비 추상화 정도를 나타낸다.

`추상도(A) = Σmᵃ / Σmᶜ`

여기서 `mᵃ`는 모듈의 추상 요소(인터페이스 또는 추상 클래스), `mᶜ`는 구상 요소(비추상 클래스)다. 예를 들어 5,000라인을 전부 `main()` 메서드에 구현한 애플리케이션은 분자가 1, 분모가 5,000이므로 추상도가 0에 가깝다.

### 5.2 불안정도(Instability)

여기서 파생된 불안정도는 원심 커플링과 (구심 커플링 + 원심 커플링)의 비율이다.

`불안정도(I) = Ce / (Ce + Ca)`

여기서 `Ce`는 원심(나가는) 커플링, `Ca`는 구심(들어오는) 커플링이다. 불안정도는 코드베이스의 변동성(volatility)을 의미하므로, 불안정도가 높은 코드베이스는 변경 시 커플링이 높아 더 깨지기 쉽다. 여러 다른 클래스를 호출해 작업을 위임하는 클래스는, 호출되는 메서드 중 하나만 변경돼도 잘못될 공산이 크다.

### 5.3 메인 시퀀스로부터의 거리

메인 시퀀스로부터의 거리(*Distance from the Main Sequence - 아키텍처 구조를 평가하는 전체적 메트릭*)는 불안정도와 추상도를 이용해 계산한다.

`거리(D) = |A + I - 1|`

추상도와 불안정도는 비율이므로 항상 0과 1 사이의 값이다. 이 둘의 이상적인 관계를 나타내는 대각선을 **메인 시퀀스**라 하고, 이 선에 가까운 클래스는 두 경쟁적 메트릭의 건전한 조합이라고 볼 수 있다. 클래스를 그래프로 그려 이상적인 선에서 얼마나 떨어져 있는지 잰다.

| 구역 | 위치 | 의미 |
|---|---|---|
| **쓸모없는 구역(zone of uselessness)** | 오른쪽 위 | 너무 추상화를 많이 해 사용하기 어려운 코드 |
| **고통스런 구역(zone of pain)** | 왼쪽 아래 | 추상화를 거의 안 하고 구현 코드만 잔뜩 넣어 취약하고 관리하기 힘든 코드 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 코드 레벨 메트릭이 있긴 하나 다른 엔지니어링 분야에 비하면 아직 부족하다. 순환 복잡도(*Cyclomatic Complexity - 코드 복잡도를 측정하는 메트릭*)조차 (하부 문제가 복잡해 생기는) 근본적 복잡도(essential complexity)와 (코드가 쓸데없이 복잡해 생기는) 우발적 복잡도(accidental complexity)를 구별하지 못한다. 코드 레벨 메트릭은 거의 다 주관적이므로, 아키텍트가 평가할 수 있도록 **베이스라인**을 분명히 정해두는 게 좋다.
:::

## 6. 커네이선스(Connascence)

밀러 페이지-존스(Meilir Page-Jones)는 1996년 저서에서 구심/원심 커플링을 더욱 발전시킨 커네이선스 개념을 객체지향 언어의 화두로 던졌다.

> 두 컴포넌트 중 한쪽이 변경될 경우 다른 쪽도 변경해야 전체 시스템의 정합성이 맞는다면, 이들은 커네이선스(*Connascence - 변화 종속성. 어느 한쪽을 바꾸면 다른 쪽도 함께 바꿔야 정합성이 유지되는 상태*)를 갖고 있는 것이다.<br>— 밀러 페이지-존스

### 6.1 정적 커네이선스

정적 커네이선스(*Static Connascence - 실행 시간 커플링과 정반대인 소스 코드 레벨의 커플링*)는 『Structured Design』의 구심/원심 커플링을 발전시킨 개념이다. 아키텍트는 구심적이든 원심적이든 다음 종류의 정적 커네이선스를 '뭔가에 커플링된 정도'로 본다.

| 종류 | 정의 | 예시 |
|---|---|---|
| **명칭(CoN)** | 여러 컴포넌트의 엔티티명이 일치해야 함 | 메서드명. 가장 일반적이면서 바람직한 커플링 방법 |
| **타입(CoT)** | 여러 컴포넌트의 엔티티 타입이 일치해야 함 | 정적 타입 언어에서 변수·매개변수를 특정 타입으로 제한 |
| **의미(CoM) 또는 관례(CoC)** | 여러 컴포넌트에 걸쳐 어떤 값의 의미가 일치해야 함 | 상수 대신 `int TRUE = 1; int FALSE = 0`처럼 하드코딩 |
| **위치(CoP)** | 여러 컴포넌트는 값의 순서가 일치해야 함 | `updateSeat(String name, String seatLocation)`에 인자 순서를 바꿔 전달 |
| **알고리즘(CoA)** | 여러 컴포넌트는 특정 알고리즘이 일치해야 함 | 서버/클라이언트가 동일 결과를 내야 하는 보안 해시 알고리즘 |

### 6.2 동적 커네이선스

동적 커네이선스(*Dynamic Connascence - 런타임 호출을 분석하는 유형*)는 다음과 같이 종류가 여러 가지다.

| 종류 | 정의 | 예시 |
|---|---|---|
| **실행(CoE)** | 여러 컴포넌트의 실행 순서가 중요함 | 이메일 객체에서 `send()` 후 `setSubject()`를 호출하면 안 됨 |
| **시점(CoT)** | 여러 컴포넌트의 실행 시점이 중요함 | 동시 실행 중인 두 스레드로 인한 경합 조건(race condition) |
| **값(CoV)** | 상호 연관된 다수의 값을 함께 변경할 때 발생 | 분산 시스템의 트랜잭션 (전부 바꾸거나 아무것도 안 바꾸거나) |
| **식별(CoI)** | 여러 컴포넌트가 동일한 엔티티를 참조할 때 발생 | 독립된 두 컴포넌트가 분산 큐를 공유해 업데이트 |

런타임 호출은 호출 그래프에 비해 효과적인 분석 도구가 많지 않아, 아키텍트는 동적 커네이선스를 파악하기가 쉽지 않다.

### 6.3 커네이선스의 세 가지 속성

커네이선스는 다음 세 속성을 잘 활용하면 개발자에게 큰 도움이 된다.

1. **강도(strength)**: 개발자가 얼마나 쉽게 리팩터링할 수 있는지에 따라 결정된다. **정적 커네이선스가 동적 커네이선스보다 바람직하다** — 소스 코드 분석이나 최신 도구로 쉽게 개선할 수 있기 때문이다. 예를 들어 의미 커네이선스(매직 밸류)를 명칭 커네이선스(기명 상수)로 리팩터링하면 개선된다
2. **지역성(locality)**: 코드베이스의 모듈들이 서로 얼마나 가까이 있는가이다. 동일 모듈의 근접한 코드는 더 높은(강한) 형태의 커네이선스를 가져도 별개 모듈로 흩어진 경우보다 덜 해롭다
3. **정도(degree)**: 커네이선스가 미치는 영향의 규모다 (소수 vs 다수 클래스). 이 값이 작을수록 바람직하다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 개발자는 강도와 지역성을 함께 고민해야 한다. 동일한 모듈에서 강한 커네이선스가 발견된다면, 같은 커네이선스가 코드 전반에 널리 흩어져 있는 것보다는 코드 스멜(*Code Smell - 더 심오한 문제를 일으킬 가능성이 있는 코드의 특징*)이 덜하다는 증거다.
:::

### 6.4 짐 웨이리치의 두 가지 규칙

전설적인 소프트웨어 아키텍처 혁신가 짐 웨이리치(Jim Weirich)는 커네이선스 개념을 다시 대중화하며 두 가지 조언을 남겼다.

1. **정도의 규칙(Rule of Degree)**: 강한 형태의 커네이선스를 보다 약한 형태의 커네이선스로 전환하라
2. **지역성의 규칙(Rule of Locality)**: 소프트웨어 엘리먼트 간의 거리가 멀어질수록 보다 약한 형태의 커네이선스를 사용하라

또한 페이지-존스는 커네이선스로 시스템 모듈성을 개선하는 세 가지 방법을 제시했다.

1. 시스템을 캡슐화한 요소들로 잘게 나누어 전체 커네이선스를 최소화한다
2. 캡슐화 경계를 벗어나는 나머지 커네이선스를 모조리 최소화한다
3. 캡슐화 경계 내부에서 커네이선스를 최대화한다

### 6.5 커플링과 커네이선스의 통합

시기와 목표가 상이한 커플링과 커네이선스는 아키텍트 관점에서 서로 중첩된다. 페이지-존스가 정적 커네이선스라고 밝힌 것들은 유출/유입 커플링 정도를 나타낸다. 구조적 프로그래밍은 들어오고 나가는 것에만 관심이 있는 반면, 커네이선스는 여러 요소가 서로 **어떻게** 커플링되는지에 주목한다.

이 메트릭들에는 한계도 있다. 첫째, 아키텍처 구조보다는 저수준 코드의 세부(코드 품질·정리 상태)를 관찰한다. 아키텍트는 커플링 정도보다 모듈이 **어떻게** 커플링되어 있는지(동기 통신이냐 비동기 통신이냐)가 더 궁금하다. 둘째, 커네이선스 자체는 마이크로서비스 같은 분산 아키텍처에서 아키텍트가 내려야 할 근본적 결정은 다루지 않는다.

> **소프트웨어 아키텍처 제1법칙**<br>소프트웨어 아키텍처의 모든 것은 다 트레이드오프다.

커네이선스를 바라보는 새로운 사고방식은 7장에서 아키텍처 특성의 범위를 다루면서 소개된다.

## 요약

- 모듈성은 연관된 코드를 논리적으로 묶는 **구성 원리**이자 대표적인 **암묵적 아키텍처 특성**이다. 논리적 구분과 물리적 구분을 구별하는 것이 중요하다
- 모듈성 측정의 세 핵심 개념은 **응집·커플링·커네이선스**다
- **응집**은 기능적(최선) → 순차적 → 소통적 → 절차적 → 일시적 → 논리적 → 동시적(최악) 순이며, LCOM은 '공유 필드를 통해 공유되지 않는 메서드의 총 개수'다
- **커플링**은 유입되는 구심(afferent, Ca)과 유출되는 원심(efferent, Ce)으로 나뉜다. 여기서 추상도 `A = Σmᵃ / Σmᶜ`, 불안정도 `I = Ce / (Ce + Ca)`, 메인 시퀀스로부터의 거리 `D = |A + I - 1|` 가 파생된다. 극단은 쓸모없는 구역과 고통스런 구역이다
- **커네이선스**는 한쪽 변경 시 다른 쪽도 변경해야 정합성이 맞는 관계다. 정적(명칭·타입·의미·위치·알고리즘)과 동적(실행·시점·값·식별)으로 나뉘며, 강도·지역성·정도의 속성을 가진다
- **정적 커네이선스가 동적 커네이선스보다 바람직**하며, 강한 커네이선스는 약한 커네이선스로(정도의 규칙), 거리가 멀수록 약한 커네이선스로(지역성의 규칙) 전환한다

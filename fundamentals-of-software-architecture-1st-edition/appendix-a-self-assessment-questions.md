# Appendix A: Self-Assessment Questions (자율평가문제)

각 장에서 다룬 핵심 개념을 스스로 점검하기 위한 질문 모음이다. 원서에는 정답이 실려 있지 않으므로, 여기서는 장별 질문 목록으로 재구성했다. 해당 장을 학습한 뒤 아래 질문에 스스로 답해 보며 이해도를 확인하면 좋다.

---

## 1장 서론

1. 소프트웨어 아키텍처를 정의하는 네 차원은 무엇인가요?
2. 아키텍처 결정과 설계 원칙은 어떤 차이점이 있나요?
3. 소프트웨어 아키텍트의 8대 핵심 기대치를 나열하세요.
4. 소프트웨어 아키텍처 제1법칙은 무엇인가요?

## 2장 아키텍처 사고

1. 아키텍처 대 개발의 기존 접근 방식을 설명하고, 이제는 더 이상 기존 접근 방식이 통하지 않는 이유를 설명하세요.
2. 지식 피라미드의 세 레벨의 지식을 열거하고 각각의 예를 들어 보세요.
3. 아키텍트에게는 왜 기술의 깊이보다 폭에 집중하는 것이 더 중요한가요?
4. 아키텍트로서 기술 깊이를 유지하고 실무 능력을 잃지 않으려면 어떻게 해야 하나요?

## 3장 모듈성

1. 커네이선스라는 용어는 무엇을 의미하나요?
2. 정적 커네이선스와 동적 커네이선스의 차이점은 무엇인가요?
3. 타입 커네이선스는 무엇인가요? 이것은 정적 커네이선스인가요, 아니면 동적 커네이선스인가요?
4. 가장 강력한 형태의 커네이선스는 무엇인가요?
5. 가장 약한 형태의 커네이선스는 무엇인가요?
6. 정적 커네이선스와 동적 커네이선스 중 코드베이스에서는 어느 쪽이 더 바람직한가요?

## 4장 아키텍처 특성 정의

1. 어떤 속성이 아키텍처 특성이 되려면 충족해야 할 세 기준은 무엇인가요?
2. 암묵적 특성과 명시적 특성의 차이점은 무엇인가요? 각각의 예를 들어 보세요.
3. 운영 특성(*operational characteristic*)의 예를 들어 보세요.
4. 구조 특성(*structural characteristic*)의 예를 들어 보세요.
5. 공통 특성(*cross-cutting characteristic*)의 예를 들어 보세요.
6. 가용성과 성능 중 어느 아키텍처 특성을 추구하는 것이 더 중요한가요?

## 5장 아키텍처 특성 식별

1. 아키텍처가 지원해야 할 아키텍처 특성의 가짓수를 제한하는 것이 왜 중요한가요?
2. 대부분의 아키텍처 특성은 비즈니스 요구사항과 유저 스토리에서 비롯됩니다. 맞는 말인가요, 틀린 말인가요?
3. 비즈니스 이해관계자는 시장 출시 시기(즉, 새로운 기능과 버그 조치 등의 서비스를 가능한 한 신속하게 고객에게 제공하는 것)를 가장 중요한 비즈니스 관심사라고 말합니다. 이런 상황이라면 어떤 아키텍처 특성이 가장 시급하게 지원되어야 하나요?
4. 확장성과 탄력성의 차이점은 무엇인가요?
5. 고객사가 고객층을 크게 넓히고자 몇 가지 중요한 기업 인수를 추진할 계획이라고 합니다. 여러분은 어떤 아키텍처 특성을 가장 신경 써야 하나요?

## 6장 아키텍처 특성의 측정 및 거버넌스

1. 아키텍처를 분석할 때 순환 복잡도가 중요한 메트릭인 이유는 무엇인가요?
2. 아키텍처 피트니스 함수는 무엇인가요? 아키텍처를 분석하는 데 이 함수를 어떻게 사용하나요?
3. 아키텍처 피트니스 함수를 이용해서 아키텍처 확장성을 측정하는 예를 들어 보세요.
4. 아키텍트·개발자가 피트니스 함수를 생성할 수 있게 하는, 아키텍처 특성의 가장 중요한 기준은 무엇인가요?

## 7장 아키텍처 특성 범위

1. 아키텍처 퀀텀은 무엇인가요? 이것은 왜 아키텍처에 중요한가요?
2. 독립적으로 배포된 4개 서비스가 있고 각 서비스마다 고유한 데이터베이스가 별도로 포함되어 있으며, 단일 유저 인터페이스로 구성된 시스템이 있습니다. 이 시스템의 아키텍처 퀀텀은 1개일까요, 아니면 4개일까요? 그 이유는 무엇인가요?
3. 정적 참조 데이터(예: 제품 카탈로그 및 창고 정보)를 관리하는 관리 부문과 고객 주문을 관리하는 고객 대면 부문으로 구성된 시스템이 있습니다. 이 시스템의 아키텍처 퀀텀은 몇 개인가요? 또 그 이유는 무엇인가요? 여러분이 다수의 퀀텀을 생각하고 있다면 관리 퀀텀과 고객 대면 퀀텀이 데이터베이스를 공유할 수 있을까요? 만약 그렇다면 데이터베이스는 어느 퀀텀에 두어야 할까요?

## 8장 컴포넌트 기반 사고

1. 컴포넌트라는 용어는 애플리케이션의 빌딩 블록, 즉 애플리케이션이 하는 일이라고 정의할 수 있습니다. 컴포넌트는 보통 클래스 또는 여러 소스 파일들로 구성됩니다. 컴포넌트는 대부분 애플리케이션이나 서비스에서 어떻게 등장하나요?
2. 기술 분할과 도메인 분할의 차이점은 무엇인가요? 각각의 예를 하나씩 들어 보세요.
3. 도메인 분할의 장점은 무엇인가요?
4. 어떤 경우에 기술 분할이 도메인 분할보다 유리한가요?
5. 엔티티 함정이란 무엇인가요? 왜 이것은 컴포넌트 식별에 좋지 않은 접근 방식일까요?
6. 코어 컴포넌트를 식별할 때 어떤 경우에 액터/액션 접근법이 아닌, 워크플로 접근법을 선택하나요?

## 9장 기초

1. 분산 컴퓨팅의 8가지 오류를 열거하세요.
2. 모놀리식 아키텍처에는 없고 분산 아키텍처에만 존재하는 세 가지 어려움을 설명하세요.
3. 스탬프 커플링이란 무엇인가요?
4. 스탬프 커플링을 해소하려면 어떻게 해야 하나요?

## 10장 레이어드 아키텍처 스타일

1. 개방 레이어와 폐쇄 레이어의 차이점은 무엇인가요?
2. 레이어 격리 개념은 무엇인가요? 이 개념이 어떤 점에서 이로운지 설명하세요.
3. 아키텍처 싱크홀 안티패턴이란 무엇인가요?
4. 레이어드 아키텍처를 결국 사용할 수밖에 없도록 만드는 주요 아키텍처 특성은 무엇인가요?
5. 레이어드 아키텍처에서 시험성(*testability*)이 잘 지원되지 않는 이유는 무엇인가요?
6. 레이어드 아키텍처에서 민첩성(*agility*)이 잘 지원되지 않는 이유는 무엇인가요?

## 11장 파이프라인 아키텍처 스타일

1. 파이프라인 아키텍처에서 파이프는 양방향 통신이 가능한가요?

3. 필터는 여러 파이프를 통해 데이터를 보낼 수 있나요?
4. 파이프라인 아키텍처는 기술 분할인가요, 아니면 도메인 분할인가요?
5. 파이프라인 아키텍처는 어떤 방식으로 모듈성을 지원하나요?
6. 파이프라인 아키텍처의 예를 두 가지 만들어 보세요.

## 12장 마이크로커널 아키텍처 스타일

1. 마이크로커널 아키텍처 스타일의 별칭은 무엇인가요?
2. 플러그인 컴포넌트가 다른 플러그인 컴포넌트에 의존적인 것이 문제가 되지 않는 것은 어떤 경우일까요?
3. 플러그인 관리 용도로 쓸 만한 도구와 프레임워크는 어떤 것들이 있나요?
4. 코어 시스템의 표준 플러그인 계약에 맞지 않는 서드파티 플러그인이 있다면 여러분은 어떻게 할 건가요?
5. 마이크로커널 아키텍처 스타일의 예를 두 가지 들어 보세요.
6. 마이크로커널 아키텍처 스타일은 기술 분할인가요, 아니면 도메인 분할인가요?
7. 마이크로커널 아키텍처는 왜 항상 단일 아키텍처 퀀텀일까요?
8. 도메인/아키텍처 동형성(*isomorphism*)이란 무엇인가요?

## 13장 서비스 기반 아키텍처 스타일

4. 서비스 기반 아키텍처에서 데이터베이스 변경을 관리하는 기법은 어떤 것들이 있나요?
5. 도메인 서비스는 (도커 같은) 컨테이너를 실행해야 하나요?

8. 서비스 기반 아키텍처의 아키텍처 퀀텀은 어떻게 해야 늘릴 수 있나요?

## 14장 이벤트 기반 아키텍처 스타일

1. 브로커 토폴로지와 중재자 토폴로지의 가장 중요한 차이점은 무엇인가요?
2. 워크플로를 더 잘 제어하려면 중재자 토폴로지를 사용해야 하나요, 아니면 브로커 토폴로지를 사용해야 하나요?
3. 브로커 토폴로지는 일반적으로 토픽 기반의 발행/구독 모델인가요, 아니면 큐 기반의 점대점 모델인가요?
4. 비동기 통신의 중요한 장점을 두 가지 설명하세요.
5. 요청 기반 모델의 전형적인 요청을 하나만 예시하세요.
6. 이벤트 기반 모델의 전형적인 요청을 하나만 예시하세요.
7. 이벤트 기반 아키텍처에서 초기 이벤트와 처리 이벤트의 차이점은 무엇인가요?
8. 큐에(서) 메시지를 보내고 받을 때 데이터 소실을 방지하려면 어떻게 해야 하나요?
9. 이벤트 기반 아키텍처를 움직이는 세 가지 중요한 아키텍처 특성은 무엇인가요?
10. 이벤트 기반 아키텍처에서 잘 지원되지 않는 아키텍처 특성으로는 어떤 것들이 있나요?

## 15장 공간 기반 아키텍처 스타일

1. 공간 기반 아키텍처라는 명칭은 어디에서 유래됐나요?
2. 공간 기반 아키텍처가 다른 아키텍처 스타일과 차별화되는 주요한 특징은 무엇인가요?
3. 공간 기반 아키텍처에서 가상화 미들웨어를 구성하는 컴포넌트 네 가지를 설명하세요.
4. 메시징 그리드의 역할은 무엇인가요?
5. 공간 기반 아키텍처에서 데이터 라이터의 역할은 무엇인가요?
6. 서비스는 어떤 조건에서 데이터 리더를 통해 데이터를 액세스해야 하나요?
7. 캐시 크기를 줄이면 데이터 충돌 가능성이 높아지나요, 아니면 낮아지나요?
8. 복제 캐시와 분산 캐시의 차이점은 무엇인가요? 공간 기반 아키텍처는 보통 어느 캐시를 많이 사용하나요?
9. 공간 기반 아키텍처에서 가장 강력하게 지원되는 세 가지 아키텍처 특성을 열거하세요.
10. 공간 기반 아키텍처에서 시험성 등급이 제일 낮은 이유는 무엇인가요?

## 16장 오케스트레이션 기반 서비스 지향 아키텍처 스타일

3. 서비스 지향 아키텍처가 몰락하게 된 요인을 열거하세요.

5. SOA는 도메인 재사용 문제를 어떻게 해결했나요? 또 운영 재사용 문제는 어떻게 해결했나요?

## 17장 마이크로서비스 아키텍처 스타일

1. 마이크로서비스 아키텍처에서는 경계 콘텍스트 개념이 왜 그렇게 중요한 건가요?
2. 마이크로서비스의 세분도(*granularity*) 수준이 적정한지 판단하는 세 가지 방법을 설명하세요.
3. 사이드카에는 어떤 기능이 내장되어 있나요?
4. 오케스트레이션과 코레오그래피의 차이점은 무엇인가요? 마이크로서비스는 둘 중 어느 쪽을 지원하나요? 마이크로서비스에서 둘 중 어느 하나가 다른 것보다 통신 스타일이 용이한가요?
5. 마이크로서비스에서 사가란 무엇인가요?
6. 마이크로서비스에서 민첩성, 시험성, 배포성이 잘 지원되는 이유는 무엇인가요?
7. 마이크로서비스에서 성능이 이슈가 되는 두 가지 이유는 무엇인가요?
8. 마이크로서비스는 도메인 분할 아키텍처인가요, 아니면 기술 분할 아키텍처인가요?
9. 마이크로서비스 체계가 오직 단일 퀀텀인 토폴로지를 기술하세요.
10. 마이크로서비스는 도메인 재사용 문제를 어떻게 해결했나요? 또 운영 재사용 문제는 어떻게 해결했나요?

## 18장 최적의 아키텍처 스타일 선정

1. 데이터 아키텍처(논리/물리 데이터 모델의 구조)는 어떤 식으로 아키텍처 스타일에 영향을 미치나요?
2. 데이터 아키텍처는 여러분이 선택한 아키텍처 스타일에 어떻게 영향을 미치나요?
3. 아키텍트가 아키텍처 스타일, 데이터 파티셔닝, 통신 스타일을 결정하기 위해 사용하는 절차를 자세히 설명하세요.
4. 아키텍트가 분산 아키텍처에 끌릴 수밖에 없는 요인은 무엇인가요?

## 19장 아키텍처 결정

1. '네 패를 먼저 보여주지 마' 안티패턴은 무엇인가요?
2. 이메일 기반 아키텍처 안티패턴을 방지하려면 어떻게 해야 하나요?
3. 마이클 나이가드는 아키텍처적으로 중요한 뭔가를 식별하는 데 필요한 다섯 가지 팩터를 어떻게 정의했나요?
4. 아키텍처 결정 기록(ADR)의 5대 기본 섹션은 무엇인가요?
5. 아키텍처 결정의 정당화는 보통 ADR 어느 섹션에 추가하나요?
6. 대안(*Alternatives*) 섹션이 따로 필요 없는 경우, 여러분이 제안한 솔루션의 대안은 ADR의 어느 섹션에 열거하나요?
7. ADR 상태를 제안됨(*Proposed*)으로 표시하는 세 가지 기본적인 기준은 무엇인가요?

## 20장 아키텍처 리스크 분석

1. 리스크 평가 매트릭스의 두 차원은 무엇인가요?
2. 리스크 평가에서 특정 리스크의 방향을 나타내려면 어떻게 해야 하나요? 리스크가 개선되는지, 악화되는지 다른 방법으로 알 수는 없을까요?
3. 리스크 스토밍이 협력 활동이 되어야 하는 이유는 무엇인가요?
4. 리스크 스토밍의 식별 활동이 협력 활동이 아닌, 개별 활동인 이유는 무엇인가요?
5. 아키텍처의 특정 영역에 대해 세 참가자는 리스크 높음(6)으로 식별했지만 다른 참가자는 리스크 중간(3)으로 식별했습니다. 이런 상황에서 아키텍트인 여러분은 어떻게 할 건가요?
6. 검증되지 않거나 알려지지 않은 기술에 대해 리스크 등급(1~9)은 어떻게 매겨야 하나요?

## 21장 아키텍처 도식화 및 프레젠테이션

1. 무리한 아티팩트 집착(*irrational artifact attachment*)이란 무엇인가요? 이것이 아키텍처를 문서화/도식화하는 문제에 있어서 중요한 이유는 무엇인가요?
2. C4 모델링 기법에서 C4의 네 가지는 각각 무엇을 가리키나요?
3. 아키텍처를 도식화할 때 컴포넌트 사이의 점선은 무슨 의미인가요?
4. 총알 투성이 시체(*bullet-riddled corpse*) 안티패턴은 무엇인가요? 이 안티패턴을 방지하려면 프레젠테이션을 만들 때 어떻게 해야 하나요?
5. 연사가 프레젠테이션을 할 때 사용하는 두 가지 주요 정보 채널은 무엇인가요?

## 22장 개발팀을 효율적으로

1. 세 가지 아키텍트 유형은 무엇인가요? 각 유형별 아키텍트는 어떤 종류의 경계를 형성하나요?
2. 아키텍트가 팀을 제어하는 정도를 결정짓는 5개 팩터는 무엇인가요?
3. 팀 규모가 너무 커지고 있음을 깨닫게 하는 세 가지 징후는 무엇인가요?
4. 개발팀에 이로운 세 가지 기본 체크리스트를 열거하세요.

## 23장 협상과 리더십 스킬

1. 왜 아키텍트에게 협상 능력이 중요할까요?
2. 비즈니스 이해관계자는 파이브 나인스 가용성을 주장하지만, 실제로 쓰리 나인스 가용성이면 충분합니다. 이럴 때 활용 가능한 협상 기법을 설명하세요.
3. 비즈니스 이해관계자가 "어제 필요했습니다"라고 말했다면 어떤 사실을 추론할 수 있을까요?
4. 협상에서 시간과 비용 문제를 가장 마지막까지 아껴두어야 하는 이유는 무엇인가요?
5. 분할 및 정복 규칙(*divide-and-conquer rule*)이란 무엇인가요? 비즈니스 이해관계자와 아키텍처 특성에 대해 협상할 때 이 규칙은 어떻게 적용할 수 있을까요? 예를 들어 설명해 보세요.
6. 아키텍처 4C를 열거하세요.
7. 아키텍트가 실용적이면서 동시에 비전을 가지는 것이 중요한 이유는 무엇일까요?
8. 아키텍트가 자신을 초대한 회의 수를 관리하고 줄이려면 어떻게 해야 하나요?

## 24장 커리어 패스 개발

1. 20분 규칙이란 무엇인가요? 이 규칙이 가장 잘 적용되는 시점은 언제인가요?
2. 쏘우트웍스 기술 레이더의 4개 원은 무엇이고 각각의 의미는 무엇인가요? 이 4대 원은 여러분의 레이더에 어떻게 적용시킬 수 있나요?
3. 소프트웨어 아키텍트에게 해당되는 기술의 깊이와 폭의 차이점을 설명하세요. 아키텍트는 둘 중 어느 것을 극대화하기 위해 노력해야 하나요?

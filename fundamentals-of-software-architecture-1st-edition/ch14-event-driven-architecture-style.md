# Chapter 14: Event-Driven Architecture Style (이벤트 기반 아키텍처 스타일)

## 핵심 질문

요청 기반 모델과 이벤트 기반 모델은 무엇이 다른가? 브로커 토폴로지와 중재자 토폴로지는 각각 언제 쓰며 어떻게 트레이드오프가 갈리는가? 비동기 통신에서 오류 처리와 데이터 손실은 어떻게 막는가? 이벤트 기반 아키텍처는 왜 성능·확장성·내고장성에 최고점을 받으면서도 단순성·시험성은 최하점인가?

---

## 1. 요청 기반 vs 이벤트 기반

대부분의 애플리케이션은 **요청 기반 모델(request-based model)** 을 따른다. 요청 오케스트레이터(보통 UI)가 요청을 받아 요청 프로세서에 **확정적(deterministic)·동기적(synchronous)** 으로 전달한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: "지난 반년치 주문 이력 조회"는 특정 콘텍스트에서 데이터를 달라고 전달된 **확정적 요청**이지, 시스템이 반응해야 할 이벤트가 아니다. 반면 온라인 경매의 입찰은 "현재가 발표 직후 발생하는 **이벤트**"에 가깝고, 시스템은 이에 반응해 최고가 입찰자를 결정한다.
:::

이벤트 기반 아키텍처(*Event-Driven Architecture - 이벤트를 비동기로 수신·처리하는 분산 아키텍처*)는 확장성이 뛰어난 고성능 애플리케이션에 널리 쓰이며, 스탠드얼론으로도, 다른 스타일(이벤트 기반 마이크로서비스)에 내장해서도 쓸 수 있다.

---

## 2. 브로커 토폴로지

중앙에 이벤트 중재자가 **없다.** 메시지는 경량 브로커(RabbitMQ·ActiveMQ 등)를 통해 브로드캐스팅되어 이벤트 프로세서에 분산된다. 이벤트 흐름이 단순하고 중앙 조정이 필요 없을 때 유용하다.

네 가지 컴포넌트: **시작 이벤트 → 이벤트 브로커 → 이벤트 프로세서 → 처리 이벤트**

```
[시작 이벤트] → 이벤트 채널 → [이벤트 프로세서A]
                                    ↓ 처리 이벤트 발행
                                이벤트 채널 → [이벤트 프로세서B]
                                                  ↓ 처리 이벤트 발행
                                              이벤트 채널 → [이벤트 프로세서C]
```

- 각 이벤트 프로세서가 작업 후 **처리 이벤트(processing event)** 를 발행해 "자신이 한 일"을 비동기로 알린다 (파이어 앤 포겟, 발행-구독)
- 이 과정은 아무도 관심 갖지 않을 때까지 반복된다

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 브로커 토폴로지에서는 **다른 프로세서의 관심 여부와 무관하게 각 프로세서가 자신이 한 일을 모두에게 알리는 게 항상 바람직하다.** 그래야 나중에 기능 추가가 필요해도 새 프로세서가 기존 이벤트를 구독하기만 하면 되어 아키텍처를 쉽게 확장할 수 있다. (Notification 프로세서가 발행한 이벤트를 아무도 안 들으면 그냥 사라질 뿐, 손해는 없다.)
:::

**단점**: 워크플로 제어·에러 처리가 없고, 비즈니스 트랜잭션의 재시작이나 진행 상황 파악이 어렵다.

---

## 3. 중재자 토폴로지

브로커 토폴로지의 단점을 보완한다. 여러 프로세서 간 조정이 필요한 시작 이벤트의 워크플로를 관리·제어하는 **이벤트 중재자(event mediator)** 가 핵심이다.

5개 컴포넌트: **시작 이벤트 → 이벤트 큐 → 이벤트 중재자 → 이벤트 채널 → 이벤트 프로세서**

```
[시작 이벤트] → 이벤트 큐 → [이벤트 중재자]
                              ↓ 점대점 (큐)
              ┌───────────┬───────────┬───────────┐
         이벤트 채널  이벤트 채널  이벤트 채널  이벤트 채널
              ↓           ↓           ↓           ↓
          [프로세서]   [프로세서]   [프로세서]   [프로세서]
                          (중재자에게 완료 응답)
```

- 중재자는 **처리 단계 정보만** 갖고 점대점(큐)으로 각 프로세서에 처리 이벤트를 보낸다
- 프로세서는 처리 후 **중재자에게 완료 응답**한다 (프로세서끼리는 서로 알리지 않는다 — 브로커와 다른 점)
- 도메인별 중재자(고객 중재자·주문 중재자)를 여럿 두어 단일 장애점(SPOF)을 줄인다

중재자 구현체는 이벤트 복잡도에 따라 선택한다:

| 중재자 | 적합한 경우 |
|---|---|
| 아파치 카멜·Mule ESB·스프링 인티그레이션 | 간단한 에러 처리·오케스트레이션 (코드로 제어) |
| 아파치 ODE·오라클 BPEL | 조건부·동적 경로가 많은 복잡한 워크플로 (BPEL 기반) |
| jBPM 같은 BPM 엔진 | 사람 개입이 필요한 장기 실행 트랜잭션 (예: 거래 한도 초과 시 승인 대기) |

**트레이드오프**: 중재자는 워크플로 제어·에러 처리·복구가 뛰어나지만, 이벤트 프로세서의 완전한 디커플링이 어렵고 처리 흐름 병목·확장성 제약이 브로커보다 크다.

---

## 4. 비동기 능력 — 응답성 vs 성능

이벤트 기반 아키텍처는 **파이어 앤 포겟(fire-and-forget)** 비동기 통신으로 요청-응답 동기 통신에는 없는 장점을 준다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 비동기의 핵심은 **응답성(responsiveness)과 성능(performance)의 차이**다. 댓글에 욕설 필터·문법 검사를 하는 사이트에서, 동기 방식은 3000ms(응답성 나쁨)를 사용자가 기다리지만, 비동기는 "댓글이 게시될 것"이라 즉시 알리고(응답성 25ms) 백그라운드로 처리한다. **성능(총 처리 시간)은 그대로여도 응답성은 극적으로 좋아진다.**
:::

---

## 5. 오류 처리 — 워크플로 이벤트 패턴

비동기 워크플로의 에러 처리 난제를 **워크플로 이벤트 패턴(workflow event pattern)** 이라는 리액티브 아키텍처 패턴으로 해결한다. 탄력성과 응답성을 동시에 겨냥한다.

- 이벤트 프로세서가 처리 못 하는 메시지를 **워크플로 프로세서**에 위임
- 워크플로 프로세서가 오류를 분석·수정해 다시 원래 큐에 넣거나(재제출), 고칠 수 없으면 사람이 볼 **대시보드 큐**로 보낸다
- 이벤트 소비자는 에러 처리에 시간을 뺏기지 않아 응답성이 유지된다

---

## 6. 데이터 손실 방지

비동기에서 데이터 손실은 흔한 우려다. 메시지가 사라질 수 있는 세 지점과 해법:

1. **프로듀서 → 큐 전송 중 유실**: **퍼시스턴트 큐(persistent queue)** + **동기 전송(sync send)** 으로 브로커가 메시지를 물리적으로 저장할 때까지 블로킹
2. **큐 → 컨슈머, 처리 전 컨슈머 다운**: **클라이언트 확인 모드(client acknowledge mode)** 로 메시지를 큐에 남겨두고, 다른 컨슈머가 집을 수 있게 함
3. **컨슈머 → DB 저장 실패**: **마지막 참가자 지원(Last Participant Support, LPS)** 으로 처리 완료를 확인 후에만 큐에서 메시지 제거

---

## 7. 하이브리드 이벤트 기반 아키텍처

이벤트 기반은 다른 스타일에 내장되는 경우가 많다 — **이벤트 기반 마이크로서비스**, 이벤트 기반 마이크로커널, 이벤트 기반 파이프라인 등. 다른 스타일에 **확장성·응답성**을 더하는 데 효과적이다.

---

## 8. 아키텍처 특성 등급

| 아키텍처 특성 | 등급 |
|---|---|
| 분할 유형 | 기술적 |
| 퀀텀 수 | 하나 또는 여러 개 |
| 배포성 | ★★★☆☆ |
| 탄력성 | ★★★★★ |
| 진화성 | ★★★★★ |
| 내고장성 | ★★★★★ |
| 모듈성 | ★★★★☆ |
| 전체 비용 | ★★★☆☆ |
| 성능 | ★★★★★ |
| 신뢰성 | ★★★☆☆ |
| 확장성 | ★★★★★ |
| 단순성 | ★☆☆☆☆ |
| 시험성 | ★★☆☆☆ |

- **기술 분할**: 한 도메인이 여러 이벤트 프로세서·중재자·메시징 아티팩트에 흩어져 있어 도메인 분할이 아니다
- **퀀텀 1 이상**: 통신은 비동기여도, 여러 프로세서가 단일 DB를 공유하거나 요청-응답으로 동기적으로 묶이면 동일 퀀텀에 속한다
- **성능·확장성·내고장성 5개 (주요 강점)**: 경쟁 컨슈머(competing consumer)로 프로그래밍 방식 로드 밸런싱이 가능하고, 부하 증가 시 프로세서를 추가하면 된다. 고도로 분리된 비동기 프로세서 + 프라미스/퓨처로 내고장성을 실현
- **진화성 5개**: 새 프로세서가 기존 이벤트를 구독하기만 하면 되어 (특히 브로커 토폴로지) 인프라·기존 프로세서 변경 없이 기능 추가 가능
- **단순성·시험성 최하점**: 비결정적·동적 이벤트 흐름 때문에 "이벤트 트리 다이어그램"이 매우 복잡해지고, 시나리오가 수백~수천 개라 관리·테스트가 매우 어렵다

---

## 요약

- **요청 기반**(확정적·동기)과 달리 **이벤트 기반**은 이벤트에 비동기로 반응
- **브로커 토폴로지**(중재자 없음, 발행-구독, 확장 쉬움, 제어 어려움) vs **중재자 토폴로지**(워크플로 제어·에러 처리 우수, 결합·병목 존재)
- **응답성 ≠ 성능** — 비동기는 총 처리 시간을 줄이지 않아도 응답성을 극적으로 높인다
- 에러는 **워크플로 이벤트 패턴**, 데이터 손실은 **퍼시스턴트 큐·클라이언트 확인 모드·LPS** 로 방어
- 강점은 **성능·확장성·내고장성·진화성(모두 5점)**, 약점은 **단순성·시험성** — 비결정적 흐름의 대가
- 다른 스타일에 내장되는 **하이브리드**로도 자주 쓰여 확장성·응답성을 더한다

# Chapter 2: Implementation Skills and Learning (구현 기술과 학습)

## 핵심 질문

개발자가 구현 기술을 어떻게 선택하고, 어떤 전략으로 학습하며, 실무에 적용할 때 무엇을 고려해야 하는가?

---

## 1. 구현 기술

구현 기술은 소프트웨어를 만드는 데 꼭 필요한 도구다. 개발자가 얼마나 구현 기술을 잘 활용하느냐에 따라 결과물이 달라진다. 개발자가 익혀야 할 기술은 다양하다:

| 분류 | 예시 | 설명 |
|------|------|------|
| **프로그래밍 언어** | Java, JavaScript, TypeScript, Kotlin, Python | 주력 언어 외에도 상황에 따라 여러 언어를 다뤄야 한다 |
| **프레임워크/라이브러리** | Spring Boot, Hibernate, MyBatis, React | 언어보다 더 많이 익혀야 하는 대상이 될 수 있다 |
| **별도 프로그램** | MySQL, Kafka, Redis | 주요 설정과 구성 요소의 동작을 이해해야 한다 |
| **클라우드 플랫폼** | AWS | 대표적인 서비스 요소와 기본적인 조작·설정 방법을 알아두어야 한다 |
| **운영체제** | Linux | 부팅 시 자동 시작, 권한 설정, 스케줄링, 상태 보기 등 서비스 운영에 필요한 설정과 명령어를 알아야 한다 |

개발자가 어떤 역할(직군)을 하는지에 따라 선택해야 하는 기술이 달라진다. 자바를 주력으로 하는 백엔드 개발자는 스프링, MyBatis, JPA, 톰캣을, 모바일 게임 개발자는 C#과 유니티를 선택할 것이다.

## 2. 학습 대상

익혀야 할 구현 기술이 너무 많으므로 학습할 대상을 잘 선택해야 한다. 유행한다고 닥치는 대로 학습하면 안 된다. 학습 대상을 정할 때 적용할 **2가지 기준**:

### 2.1 현재 사용 중인 기술

지금 당장 잘 활용하는 것이 중요하다. 현재 사용 중인 기술이 본인의 취향과 맞지 않더라도 코드를 이해하고 수정할 수 있을 정도의 지식은 있어야 한다.

> D사에서 전사 공통 서비스 개발에 참여했을 때 스트럿츠, 스프링, 아이바티스, 벨로시티를 사용했다. 스프링을 제외한 나머지 기술은 경험이 없었기 때문에 레퍼런스 문서와 관련 책을 읽어가며 개발했다. 또한 개인적으로 선호하지 않는 OO 플랫폼을 써야 하는 프로젝트에서도 기존 코드와 공식 문서를 보면서 기술을 익혔다.

처음부터 완벽하게 모든 것을 익히려 하지 않아도 된다. 보통 기술의 일부 요소만 사용하므로 필요한 부분만 알고 있으면 된다. 예를 들어 스프링 API 서버에서는 컨트롤러, JSON 변환 처리, 데이터베이스 연동과 트랜잭션 처리 정도의 지식만 있으면 기존 코드를 분석하고 수정할 수 있다. 부족한 부분은 핸즈온(*Hands-on - 직접 따라 하며 배우는 실습*)과 검색을 병행하며 채우면 된다.

### 2.2 문제를 해결하기 위한 기술

문제는 2가지 유형으로 나뉜다:

**당장 해결해야 하는 문제**: 트래픽 급증으로 API 서버 응답 시간이 늘어난 상황이라면 레디스나 카페인(*Caffeine - JVM 기반 로컬 캐시 라이브러리*) 같은 캐시 기술을 빠르게 찾아 익히고 적용해야 한다.

**가까운 미래에 해결해야 할 문제**: 예를 들어 모놀리식 시스템에서 다음과 같은 위험 신호가 관찰될 때:

- 여러 기능이 혼재되어 여러 팀이 동시에 코드를 수정한다
- 운영에 적용되면 안 되는 코드가 배포된 적이 있다
- 특정 기능 이상이 전체 시스템 응답 시간에 영향을 준다
- 시스템이 커지면서 배포 시간이 점점 증가한다

이런 위험 신호를 방치하면 큰 장애가 발생할 가능성이 높다. 서비스와 데이터베이스 분리를 검토한다면 연동 기술, 데이터 일관성 기술 등을 조사하고 학습해야 하며, 필요에 따라 PoC(*Proof of Concept - 새로운 기술 도입 전에 실현 가능성과 효과를 검증하는 과정*)로 적합성을 증명해야 한다.

> **핵심 통찰**: 문제를 해결하기 위한 기술을 선택할 때는 **기술의 유명세에 휘둘리지 말아야 한다**. 유행하는 기술이라고 해서 카프카를 사용하고 쿠버네티스로 클러스터를 구축하면, 서비스 분리 목적과 다르게 새로운 기술만 적용한 상황이 된다.

### 딥 다이브: 액티브X 개발 경험

> 웹 애플리케이션에서 원하는 형식으로 문서를 출력하는 기능이 필요했는데, 당시 웹 브라우저에는 마땅한 수단이 없어 액티브X 컴포넌트를 개발해야 했다. 자바만 다뤄본 상태에서 비주얼 베이식을 선택해 책과 온라인 문서를 보며 개발했다. 이때 학습 범위를 **출력 기능 구현과 액티브X 컴포넌트 생성**에 집중했고, GUI 프로그래밍까지 넓히지 않았다. 만약 범위를 넓혔다면 원하는 시점에 기능을 완성하지 못했을 것이다.

평소에 구현 기술을 탐색하고 학습해두면, 눈앞에 문제가 닥쳤을 때 적합한 기술을 빠르게 선택할 수 있다.

## 3. 기술 파기

구현 기술 습득은 설계나 업무 관리에 비해 상대적으로 쉬운 영역에 속한다. 공식 문서나 핸즈온, 검색과 LLM(*Large Language Model - ChatGPT 같은 대규모 언어 모델*)을 활용하면 빠르게 학습을 시작할 수 있다. 하지만 기술의 **모든 부분을 제대로 이해하고 다루기는 쉽지 않다**. 학습해야 할 양이 방대하기 때문이다.

현실적으로 모든 기술을 깊이 학습하기는 어렵다. 기술에 대한 이해 부족이나 사용 미숙으로 문제가 발생하기도 한다.

### 딥 다이브: 기술 이해 부족으로 발생한 문제 1 — 템플릿 엔진 캐시

> D사에서 점심시간에 서비스가 먹통이라는 연락을 받았다. 메모리 부족으로 Full GC가 계속 발생하고 있었다. 힙 덤프를 분석해보니 범인은 템플릿 엔진에서 사용하는 객체였다. **누군가 템플릿 엔진의 캐시를 비활성화해서** 요청마다 템플릿 처리를 위한 객체가 생성되고 있었다. 많은 트래픽을 처리하는 서비스에서 이로 인해 메모리가 빠르게 쌓였고, Full GC가 빈번하게 발생하며 서비스가 먹통이 된 것이다. 캐시 설정을 활성화하는 것만으로 문제를 해결할 수 있었다.

### 딥 다이브: 기술 이해 부족으로 발생한 문제 2 — @Transactional과 프록시

> 동료 개발자가 `@Transactional` 애너테이션을 사용해서 트랜잭션을 처리하고 있는데 유독 한 코드만 롤백이 안 된다고 했다. 문제 코드를 보니 당연히 안 될 수밖에 없었다:

```java
public void someMethod() {
    this.internalMethod();
}

@Transactional
private void internalMethod() {
    // ...
}
```

이 코드를 만든 개발자는 `@Transactional`을 사용하면 된다는 것은 알고 있었지만, 스프링 AOP와 프록시에 대한 이해가 부족했다. `private` 메서드에 `@Transactional`을 적용해도 프록시가 가로챌 수 없기 때문에 트랜잭션이 동작하지 않는다.

**기술 학습의 효과적인 접근법:**

1. **처음 접하는 기술** → 핸즈온이나 튜토리얼로 빠르게 감을 잡는다
2. **일상 업무 수준** → 자연스럽게 손에 익을 정도로 자주 사용하거나 따로 연습한다
3. **더 깊게** → 필요할 때마다 조금씩 더 깊게 학습한다. 사용하는 모든 기술의 내부 동작 원리까지 처음부터 알 필요는 없다

> **핵심 통찰**: 더 나은 방법을 찾으려고 노력해야 한다. 방법을 잘 모르거나 구현하기 어려운 문제가 생겼을 때 해결을 위한 더 나은 방법이 있는지 찾아봐야 한다. 더 나은 방법을 찾지 않고 기존 방법만 답습하면 기술 역량을 높일 기회는 찾아오지 않는다.

## 4. 학습 전략

기술 생태계는 끊임없이 변화한다. 한때 유행했던 기술이 순식간에 사라지기도 한다:

| 과거 | 현재 | 변화 |
|------|------|------|
| 서블릿 직접 작성 | 스프링/스프링 부트 | 프레임워크가 대체 |
| EJB | 사라짐 | 스프링에 밀림 |
| 스트럿츠 | 사라짐 | 스프링에 밀림 |
| DWR | 사라짐 | 반짝 유행 후 버림받음 |
| jQuery | 사용 감소 | 리액트 등에 자리 내줌 |
| JSP | 사용 감소 | 프런트엔드/백엔드 분리 추세 |
| 자바 | **여전히 주력** | C 언어와 함께 건재 |
| 스프링 | **여전히 주력** | 가장 인기 있는 자바 서버 프레임워크 |
| MyBatis, JPA | **여전히 주력** | 오랜 기간 데이터베이스 연동 기술로 사용 |

유행하는 기술의 흐름이 자주 바뀌므로 **전략적으로 학습해야 한다**:

### 4.1 주력 기술 집중 학습

주력 기술이란 **당장 또는 가까운 미래에 경제적 이익을 얻는 데 필요한 기술**을 말한다. 시장 점유율이 높은 기술을 익혀야 유리하다. 단, 모든 대세 기술을 알 필요는 없다. 본인의 주된 업무에 필요한 기술을 집중적으로 익히고, 나머지는 그 다음이다.

### 4.2 기술 탐색

지금 당장 사용하지 않더라도 주기적으로 어떤 기술이 주목받고 있는지 조사하고 핸즈온을 해봐야 한다. 탐색 방법:

- **스터디 참여**: 카프카, 스파크, 도커, 쿠버네티스, 코틀린, 고언어 같은 기술을 탐색하는 좋은 방법
- **뉴스레터/블로그 구독**: 매주 여러 소식을 알려주므로 제목만 봐도 기술 트렌드를 파악할 수 있다
- **SNS 팔로우**: 기술 트렌드에 민감한 인물의 SNS를 팔로우한다

> **핵심 통찰**: 「실용주의 프로그래머」의 팁 — **"지식 포트폴리오에 주기적으로 투자하라."** 좋은 시니어 개발자가 되고 싶다면 구현 기술을 주기적으로 탐색하고 학습해야 한다.

### 딥 다이브: 과거에 머물지 말고 과감하게 포기하기

스프링이 대세가 된 상태에서 스트럿츠 2(웹워크)를 계속 사용하는 것은 스스로 기술적 고립을 자초하는 일이었다. 많은 시간을 들여 익힌 구현 기술을 포기하는 것은 쉽지 않지만, **시장에서 선택받지 못한다면 기술을 포기하는 과감함도 필요하다**.

> 오랜 기간 JSP 책을 써온 만큼 JSP에 대한 애착이 크지만, 신규 프로젝트에서는 더 이상 JSP를 사용하지 않는다. 대신 타임리프와 같은 다른 템플릿 엔진을 사용하고 있다.

## 5. 유행에 상관없는 구현 기술

유행하는 기술만 좇다 보면 **기초 지식을 놓치기 쉽다**.

> 예전 직장에서 대형 사고가 난 적이 있다. 본인 계정으로만 볼 수 있는 문서가 다른 사용자 계정에서도 보이는 문제가 무작위로 발생했다. 핵심 서비스였기에 회사에 비상이 걸렸고 언론에도 나올 만큼 파장이 컸다. 원인은 **다중 스레드 환경에서 사용자 아이디를 필드에 저장하고, 다른 메서드가 그 필드 값을 사용하는 코드** 때문이었다. 여러 스레드가 같은 필드를 사용하다 보니 다른 스레드에 의해 값이 변경되어 다른 사용자의 문서가 표시되는 오류가 발생했다. 비슷한 문제로 결제 시스템에서 A 사용자의 결제 내역이 B 사용자에 반영되거나 결제 내역이 사라지는 사고도 겪었다.

이런 사례는 **동시성 처리 기초 지식이 얼마나 중요한지** 보여준다. 기초 지식이 필요한 영역:

- **동시성 처리**: 다중 스레드 환경에서의 안전한 코드 작성
- **네트워크**: 네트워크는 불안정하다는 전제 하에 타임아웃 설정 등 적절한 대응
- **데이터베이스**: 데이터 규모나 접근 패턴을 고려한 인덱스 설계
- **HTTP 프로토콜**: 웹 서비스의 기반이 되는 프로토콜 이해
- **프로그래밍 언어 기초**: 언어 자체에 대한 깊은 이해

> **핵심 통찰**: 주니어 개발자라면 유행과 상관없는 기초 지식을 **1년에 1개 이상** 학습하자. 이러한 기초 지식을 꾸준히 학습해야 기본기가 튼튼한 개발자로 성장할 수 있다.

## 6. 구현 기술 적용

공부한 구현 기술을 업무에 적용하는 것은 즐거운 일이다. 하지만 상황과 조건에 맞게 사용해야 한다. 새로운 기술을 도입할 때 고려해야 할 **5가지 원칙**:

### 6.1 신뢰 구축

동료에게 먼저 신뢰받아야 한다. 입사하자마자 기존 시스템의 부족함을 지적하면서 새 기술 도입을 주장하면 안 된다.

> D사에서 팀을 옮겼을 때, 오래전부터 시스템을 유지보수한 시니어 개발자 S가 개발 리더였다. 처음 두 달 정도는 내 의견이 잘 통과되지 않았다. S가 나를 신뢰하지 않았기 때문이다. 기존 틀 안에서 맡겨진 일을 잘 하기 위해 노력했고, **6개월 정도 지나서야** 비로소 내 의견이 받아들여지기 시작했다. 동료에게 신뢰를 얻기까지 최소 3개월에서 1년 정도 걸린다.

### 6.2 함께할 동료

새로 적용하고자 하는 기술에 대해 함께 논의하고 공감대를 형성할 동료가 꼭 필요하다. **혼자서는 아무것도 할 수 없다. 아니, 하면 안 된다.** 한 명이 하는 주장보다 둘이 하는 주장에 힘이 더 실린다. 1~2개월 내 승부를 보겠다는 마음은 버려야 한다.

### 6.3 타당성

익숙하지 않은 기술을 적용해야 하는 **이유가 타당해야 한다**. 단순히 유행하는 기술이라거나, 너무 사용해보고 싶다는 마음은 이유가 될 수 없다. 구현 기술로 해결하고자 하는 문제가 분명해야 한다.

> 스프링에 웹플럭스가 추가되었을 때, 많은 개발자가 API 서버 구현 기술로 웹플럭스를 검토했다. 하지만 왜 도입하려고 하는지 물어보면 "성능이 좋다고 들어서"라는 답이 대부분이었다. 어떤 성능을 높이기 위해 도입하려 하는지 재차 물어보면 명확하게 대답하는 사람이 별로 없었다.

### 딥 다이브: 답이 아닌 질문 따라하기

개발자이자 연사인 제시카 커(*Jessica Kerr*)는 **"답이 아닌 질문을 따라하라(Copy the question, not the answer)"**라고 말했다. 스포티파이의 스쿼드 모델이 유명해지자 여러 스타트업이 외형적인 모습만 따라했을 뿐, 정작 스쿼드 모델로 이루고자 하는 목표는 따르지 않았다. 구현 기술 적용도 마찬가지다. 유명한 기업이 도입한 기술이 우리 회사에는 적합하지 않을 수 있다. 먼저 **왜** 그 기술을 도입했는지 이유를 찾아보고, 우리 회사가 도입해야 하는 목적과 목표를 세워야 한다.

### 6.4 점진적 적용

한 번에 다 바꾸겠다는 생각은 위험하다. 중요하지 않은 기능과 시스템에 먼저 적용해서 안정성을 검증해야 한다. 크게 일을 벌여놨는데 성과를 내지 못하면 그동안 쌓은 신뢰를 한순간에 잃게 된다.

### 6.5 시장 상황

도입하려는 기술에 능숙한 인력을 원활히 채용할 수 있어야 한다. 그래야 서비스를 지속할 수 있다.

> 한 콘퍼런스에서 스칼라와 아카 도입 경험 발표를 들으면서 앞으로 유지보수하기 힘들겠다는 생각이 먼저 떠올랐다. 흥미로운 기술이지만 국내 시장에서 경험 있는 개발자 수가 적어 구인이 쉽지 않다. 실제로 그 기술을 도입한 개발자가 이직한 후, 남은 인력 중 능숙한 사람이 없어 유지보수에 애를 먹었다고 한다.

| 원칙 | 핵심 |
|------|------|
| 신뢰 구축 | 먼저 기존 시스템을 이해하고 주어진 일을 잘 수행해야 동료의 신뢰를 얻을 수 있다 |
| 함께할 동료 | 혼자가 아닌 공감하는 동료와 함께 변화를 추진해야 한다 |
| 타당성 | 해결하고자 하는 문제가 분명해야 한다. 유행이나 개인의 욕구는 이유가 될 수 없다 |
| 점진적 적용 | 비핵심 영역에서 먼저 검증한 뒤 핵심 영역에 적용한다 |
| 시장 상황 | 채용 가능성과 동료의 이력까지 고려해야 한다 |

### 딥 다이브: 구현 기술과 직업 윤리

구현 기술을 선택할 때는 조직 상황도 함께 고려해야 한다. 조직의 구성원이 새로운 기술을 경험할 수 있도록 시간을 주고 참여시켜야 한다. 그래야 유지보수할 때 심각한 문제가 생기지 않는다. 이것은 일종의 **직업 윤리**와 연결된다.

## 7. 주의할 점

- **기술 맹종 금지**: 특정 기술을 사용한다고 우월해지는 것은 아니다. 오래된 기술을 사용한다고 열등한 것도 아니다. 구현 기술은 단지 도구일 뿐이다
- **유연한 적용**: 모든 문제에 하나의 기술을 사용하려 하면 안 된다. 카프카 비동기 연동의 장점을 경험했다고 해서 모든 비동기 연동에 카프카를 사용할 필요는 없다
- **영웅주의 경계**: 한 명의 개발자가 모든 것을 잘 할 수는 없다. 특정 기술을 잘 사용하자는 의지가 모든 기술을 능숙하게 다루겠다는 마음보다 낫다
- **기술의 유통기한**: 잠깐 사용한 기술은 시간이 지나면 금세 잊힌다. 과거에 경험했어도 오랜 기간 사용하지 않으면 더 이상 내가 보유한 기술이 아니다
- **구현 기술이 전부가 아니다**: 개발의 영역은 코딩보다 훨씬 넓다. 구현 기술 외에 키워야 할 역량이 많다

## 요약

- 구현 기술은 **현재 사용 중인 기술**과 **문제를 해결하기 위한 기술**을 우선 학습한다
- 처음 접하는 기술은 핸즈온으로 빠르게 감을 잡고, 필요할 때마다 깊게 파면 된다. 모든 기술을 처음부터 깊게 학습할 필요는 없다
- **주력 기술을 집중적으로** 학습하되, 미래 경쟁력을 위해 **기술을 주기적으로 탐색**해야 한다
- 시장에서 선택받지 못한 기술은 과감하게 포기할 줄도 알아야 한다
- 유행과 상관없는 기초 지식(동시성, 네트워크, HTTP 등)을 **1년에 1개 이상** 꾸준히 학습한다
- 새로운 기술을 도입할 때는 **신뢰 구축 → 동료 확보 → 타당성 확인 → 점진적 적용 → 시장 상황 고려**의 순서로 접근한다
- 기술을 맹종하거나 영웅주의에 빠지지 말고, 구현 기술은 도구일 뿐임을 잊지 말자

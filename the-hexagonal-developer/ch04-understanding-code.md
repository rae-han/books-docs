# Chapter 4: Understanding Code (코드 이해)

## 핵심 질문

코드를 이해하는 시간을 줄이기 위한 도구와 기법은 무엇이며, 이해하기 좋은 코드를 작성하려면 어떤 원칙을 따라야 하는가?

---

## 1. 코드 변경

새로운 기능을 추가하든 기존 기능을 수정하든 결국 변경이 이루어지는 곳은 코드다. 멋진 기획이나 디자인이 나와도 코드를 작성하지 않으면 없는 것과 마찬가지다. 코드는 중요한 산출물 중 하나라고 할 수 있다.

개발자가 코드를 변경할 때 다음 **2단계 과정**을 거친다:

1. **코드 이해**: 코드를 읽고 논리 구조를 파악하며, 실행 흐름(*workflow*)에 따라 상태 변경을 추적한다
2. **코드 수정**: 코드가 어떻게 동작하는지 머릿속에 어느 정도 그림이 그려질 때 비로소 변경 범위가 정해진다

개발자가 코드를 변경하기 위해 코드를 읽는 시간은 얼마나 될까? 조사마다 다르지만 일반적으로 코드를 이해하는 데 **개발 시간의 60%**를 쓴다고 한다. 버그 수정 같은 작업은 코드를 이해하는 데 **90% 이상**의 시간을 쏟기도 한다. 코드를 작성하는 시간보다 코드를 이해하는 시간이 더 소요되기 때문에, 기존 코드를 이해하는 작업은 유지보수에서 매우 중요하다.

개발 시간을 줄이고 싶다면 코드를 작성하는 시간 못지않게 **코드를 이해하는 시간을 줄여야 한다.** 이를 위해서는 2가지 역량이 요구된다:

| 역량 | 효과 |
|---|---|
| **코드를 제대로 이해하는 역량** | 코드 분석에 소요되는 시간을 줄일 수 있다 |
| **이해하기 쉬운 코드를 작성하는 역량** | 품질이 낮은 스파게티 코드는 품질이 좋은 코드보다 40% 더 많은 이해 시간이 소요된다 |

> **핵심 통찰**: 코드를 이해하는 시간이 개발 시간의 대부분을 차지한다. 따라서 이해하기 좋은 코드를 만들면 개발 시간도 줄어든다.

## 2. 코드 이해 도구

규모가 크지 않은 코드는 분석하는 데 긴 시간이 걸리지 않는다. 전체 길이가 수십 줄에 불과한 코드를 분석하기는 쉬운 일이다. 변수도 적고 데이터 변경이나 분기 처리도 많지 않기 때문이다. 반면에 코드가 수천 줄 이상 넘어가면 분석하는 게 쉽지 않다. 코드가 커지면 많은 변수가 출현하고 데이터 변경도 복잡해지며 분기 처리도 많아진다. 탐색해야 할 코드 파일도 늘어나서 자연스럽게 분석 시간이 길어진다.

복잡한 미로에서 탈출하려면 출구가 표시된 지도가 필요하듯이, 규모가 크고 복잡한 코드를 분석할 때도 지도와 같은 도구가 필요하다. 코드 분석에 도움이 되는 다양한 수단 중 주로 사용하는 것은 **시각화, 코드 출력, 스크래치 리팩터링** 등이다.

### 2.1 코드 시각화

코드를 분석하고 그 결과를 그림으로 그리면 수정할 위치나 수정 영향을 받는 범위를 찾을 때 큰 도움이 된다. 의미만 통하면 어떤 형식을 사용해도 되지만 가급적 UML(*Unified Modeling Language - 시각적으로 모델을 만들기 위한 표기법*)을 사용하는 것이 좋다. 자신만의 표기법을 사용하면 다른 사람과의 의사소통에서 서로 다르게 해석될 수 있지만, UML은 도형마다 정해진 의미가 있어서 서로 다르게 해석할 여지를 줄여준다.

#### 액티비티 다이어그램

UML 다이어그램 중 액티비티 다이어그램은 **코드의 실행 흐름**을 이해하는 데 도움이 된다. `dto.gubun` 값이 8이나 9일 때의 동작과 2일 때 기능이 어떻게 흘러가는지 쉽게 볼 수 있다. 코드를 직접 보며 실행 흐름을 따라가려면 코드 편집기에서 계속 스크롤하면서 다른 소스 파일로 이동해야 하는데, 액티비티 다이어그램으로 한눈에 전반적인 실행 흐름을 파악할 수 있다.

액티비티 다이어그램을 그릴 때 코드 한 줄 한 줄마다 액션을 하나씩 그릴 필요는 없다. 한 줄마다 액션을 하나씩 그리면 다이어그램만 복잡해지고 길어진다. **복잡한 코드를 쉽게 이해하기 위해** 코드 시각화를 하는 것인데 시각화한 결과물이 복잡하면 애초 목적이 퇴색된다. 여러 코드가 하나의 목적으로 사용된다면 논리적인 단위로 묶어서 표시한다.

#### 시퀀스 다이어그램

시퀀스 다이어그램은 런타임에 **구성 요소가 어떻게 상호 작용하는지**를 이해하는 데 도움을 준다. 객체 간 상호 작용뿐 아니라 프로세스 간 상호 작용도 시간 흐름에 따라 정리할 수 있어 유용하다. 문제 발생 시점이나 동시성 문제 원인을 찾을 때 시퀀스 다이어그램을 주로 사용한다.

#### 클래스 다이어그램

클래스 다이어그램은 코드의 **정적인 구조**를 이해할 때 도움이 된다. 어떤 클래스가 존재하고 각 클래스가 서로 어떻게 연결되어 있는지 파악할 때 사용한다. 도메인 모델을 분석할 때 유용하게 사용할 수 있으며, 클래스가 몇 개 안 되더라도 다이어그램으로 정리된 구조를 보는 게 매번 클래스 코드를 탐색하는 것보다 효율적이고 기억하기도 쉽다.

단, 레거시 코드에서는 한 클래스에 많은 코드가 들어 있고 클래스 구조는 단순할 때가 많아 클래스 다이어그램의 효용이 떨어질 수 있다. 예를 들어 회원 관련 코드가 MemberController, MemberService, MemberDto, MemberDao 4개의 클래스에 전부 들어 있다면 구성 요소는 4개밖에 안 되어 구조 분석에 큰 도움이 되지 않는다.

#### 상태 다이어그램

업무 로직은 상태의 영향을 받는다. 같은 기능이라도 상태에 따라 다르게 동작한다. 예를 들어 주문 취소 기능은 주문이 결제 완료 상태인지 배송 완료 상태인지에 따라 다르게 동작한다. 상태가 많고 상태를 변경하는 기능이 많아질수록 시스템은 복잡해진다.

보통 상태를 변경하는 로직은 여러 메서드나 클래스에 흩어져 있다. 이렇게 상태와 상태 변경 로직은 여러 코드에 흩어져 있기 때문에 전체적인 상태 변화 로직을 이해하기 어렵다. 이럴 때 **상태 다이어그램**을 그리면 상태 변화를 이해하는 데 도움이 된다.

#### 의존·영향 관계 그래프

구성 요소 간 데이터 흐름이 복잡한 레거시 코드를 분석할 때 데이터 간의 영향 관계를 그래프로 표시하면 파악하기 쉬워진다. 참고할 만한 2가지 그래프가 있다:

- **의존 그래프**: 「프로그래머의 뇌」(제이펍, 2022)에서 소개
- **이펙트 스케치**: 「레거시 코드 활용 전략」(에이콘출판사, 2018)에서 소개

두 그래프를 그리는 목적은 같다. 의존 관계에 있는 코드를 시각적으로 표시해서 **변경이 어디까지 영향을 주는지** 파악하기 위한 것이다. 기존 코드를 변경할 때 그 여파가 어디까지 미칠지 모르는 것만큼 두려운 일은 없다. 하지만 데이터나 함수 간 의존 관계를 시각적으로 정리하면 두려움을 떨쳐낼 수 있다.

### 딥 다이브: 다이어그램을 그릴 때 유의사항

코드를 다이어그램으로 표현할 때 상세한 모든 내용을 표시하면 안 된다. 다이어그램을 그리는 의도에 맞게 필요한 요소만 표시하고 불필요한 요소는 생략해야 한다. 시퀀스 다이어그램을 그릴 때 모든 분기문을 다 표시한다든가, 클래스 다이어그램을 그릴 때 모든 메서드를 다 표시하면 다이어그램만 복잡해지고 의도한 정보를 제공하는 데 방해가 될 뿐이다. **어떤 정보를 제공하기 위해 다이어그램을 그리는지 분명히 하고 목적에 맞는 요소만 포함해야 한다.**

### 딥 다이브: 시각화의 힘

> W사에서 근무할 때 서로 다른 부서에 속한 모바일 개발자, 서버 개발자, 기획자가 모여서 회의를 한 적이 있다. 모바일 앱과 서버가 어떻게 연결되는지, 플랫폼 서비스와 연동은 어떻게 되는지 등을 정리하기 위한 회의였다. 내용은 간단했지만 시간이 지날수록 각자 다른 소리를 하기 시작했다. 같은 용어를 사용했지만 서로 상상한 구조가 달랐다.

같은 용어를 사용하더라도 각자 생각하는 모습이 다를 수 있다. 대화가 제대로 되고 있지 않다는 것을 알아채고 곧바로 화이트보드에 그림을 그리기 시작했다. 지금까지 언급된 요소를 그림으로 표시하고 각 요소를 선으로 연결했다. 확실하게 결정된 내용은 실선으로, 애매하거나 결정이 필요한 내용은 점선으로 연결했다. 그림이 완성되자 더 이상 다른 소리를 하지 않았다. **그림으로 상호 간의 차이를 없애고 정확하게 의사소통할 수 있었기 때문이다.** 이때부터 회의는 공회전 없이 빠르게 진행됐고 신속히 결론 낼 수 있었다.

### 2.2 코드 출력

예전보다 모니터가 커지고 해상도도 높아지면서 한 화면에 많은 코드를 볼 수 있게 됐지만 여전히 편집기를 스크롤해야 한다. API가 어떻게 시작해서 어떻게 끝나는지 모든 과정을 이해하려면 서로 다른 클래스와 메서드를 탐색해야 한다. 이 파일 저 파일 코드를 탐색하다 보면 머릿속에서 코드가 꼬인다.

이럴 때 코드를 출력해서 보면 편리하다. A3 용지 같이 큰 종이에 출력하면 더 좋다. 출력한 코드의 활용법:

- **형광펜으로 관심 있는 코드를 칠한다**: 값이 변하는 변수, 데이터베이스 연동 코드 등
- **2가지 이상의 색상을 사용한다**: 색상마다 의미를 부여한다 (예: DB 연동 코드는 노란색, 주요 변수·파라미터는 주황색)
- **서로 영향을 주는 코드는 선으로 긋는다**: 의존 그래프와 같은 방식
- **이해한 내용을 적당한 위치에 메모한다**: 출력물을 다시 볼 때 기억을 되살리는 데 도움

종이를 위아래로 훑는 것과 편집기에서 스크롤하면서 코드를 보는 것은 차이가 크다. 회의실의 넓은 책상을 활용하면 더 좋다.

### 2.3 스크래치 리팩터링

리팩터링이 외부에 드러나는 기능 동작 변경 없이 코드 내부 구조를 재구성하는 기법이라면, 스크래치 리팩터링(*scratch refactoring*)은 **코드 분석을 목적으로 하는 리팩터링**이다. 리팩터링을 하려면 코드 구조나 의미를 이해하는 과정을 거쳐야 하는데, 스크래치 리팩터링은 바로 여기에 집중한다. 리팩터링을 하면 변수 이름 변경, 메서드 추출, 클래스 추출, 파라미터 타입 변경과 같은 수정을 하게 되는데 이런 작업을 진행하면서 코드 동작을 깊이 이해할 수 있게 된다.

스크래치 리팩터링은 코드 개선이 아닌 **코드 이해가 목적**이므로 리팩터링한 결과는 반영하지 않고 다시 원래대로 되돌린다.

### 2.4 함께 모여 보기

코드를 이해하려는 방법의 하나로 코드 함께 보기가 있다. 코드를 함께 보면:

- 코드에 대한 지식이 **여러 명에게 쌓인다**
- 서로의 지식을 주고받으며 **부족한 부분을 채울 수 있다**
- 잘못 이해하는 것을 **줄일 수 있다**

짝 코딩처럼 하나의 코드를 보고 대화를 나누다 보면 자연스럽게 각자 머릿속에 코드에 대한 지식이 쌓이게 된다. 서로 알고 있는 지식이 다르므로 코드를 분석하는 과정에서 상호 보완도 된다. 혼자서 복잡한 코드를 분석하면 코드 동작을 다르게 추론할 수 있는데, 같이 코드를 분석하면 서로 잘못 이해한 부분을 알려주고 고칠 수 있다.

함께 모여서 코드를 볼 때는 **2~4명 정도**가 적당하다. 참여 인원이 늘어나면 대화가 분산되면서 개개인의 집중력과 참여 적극성이 떨어질 수 있기 때문이다.

## 3. 이해하기 좋은 코드

다양한 도구를 사용해서 코드를 이해하는 역량을 높이는 것은 코드 분석에 도움이 된다. 하지만 이보다 더 중요한 것은 **코드 자체를 이해하기 쉽게 만드는 것**이다. 이해하기 좋은 코드를 만들 때 가장 염두에 두어야 하는 기본은 **가독성**이다.

코드를 분석하려면 코드를 읽어야 한다. 어색한 번역투나 맞지 않은 단어를 사용한 글을 읽으면 내용을 이해하기 힘든 것처럼 가독성이 떨어지는 코드를 읽으면 동작을 이해하기 힘들다. 이해하기 좋은 코드를 작성하는 방법을 알려주는 추천 도서:

- 「켄트 벡의 구현 패턴」(에이콘출판사, 2008) — 읽기 쉬운 코드를 작성하는 다양한 기법, 두껍지 않아 부담 적음
- 「클린 코드」(인사이트, 2013) — 좋은 코드 작성에 도움되는 내용이 많으나, 책 내용을 그대로 적용하면 안 되고 **왜 이런 규칙을 만들었는지** 곱씹어보고 상황에 맞춰 유연하게 적용해야 한다

## 4. 이해를 돕는 코드 작성법

### 4.1 이름

이름은 코드 가독성에 큰 영향을 준다. 가독성이 좋은 코드를 작성하기 위한 출발점은 올바른 이름을 선택하는 것부터 시작된다. 클래스, 메서드, 변수, 함수, 필드 등이 모두 이름을 가지고 있기 때문에 코드를 읽을 때 가장 많이 보이는 게 이름이다.

**의미나 의도 또는 목적을 유추할 수 있는 이름을 사용하자.** 비영어권인 국내 환경에서 알맞은 영어 단어를 찾기가 쉽지만은 않다. 예를 들어 '고객', '사용자' 같은 단어는 비교적 쉽게 영어로 변환할 수 있지만 '거주사실확인서' 같은 용어에는 알맞은 영어 이름을 붙이기가 쉽지 않다. 영어 단어와 우리말 간의 미묘한 차이도 작명에 어려움을 준다. 예를 들어 'status'와 'state'는 둘 다 '상태'를 뜻하지만 표현하고자 하는 개념에 따라 더 적합한 영어 단어를 사용해야 한다.

이런 어려움이 있더라도 시간을 들여 신중하게 이름을 지어야 한다. **표기한 이름이 실제 역할과 다르다면 코드를 분석할 때 많은 혼란을 준다:**

```java
List<Input> inputs = selectInput(inputList);
```

이 메서드의 이름만 보면 무언가를 조회할 것 같은 느낌을 준다. 그런데 이 메서드가 실제로 하는 일이 inputList를 데이터베이스에 **저장**하고 저장된 데이터를 리턴하는 것이라면? `selectInput()` 메서드를 분석하기 전까지는 어떤 것을 조회하는 코드 동작이라고 잘못 유추할 수 있다.

**이름이 사용되는 범위에 따라 이름 길이도 달라진다:**

- **짧은 범위**: for 블록 안에서만 사용되는 변수는 이름이 짧아도 된다
- **넓은 범위**: 클래스 이름은 서술적이어야 한다. 같은 이름의 클래스가 여러 패키지에 존재하면 혼란을 준다 (예: `Job` → `VisitJob`, `AsyncJob`으로 구분)
- **클래스 필드**: 클래스 이름이 맥락을 제공하므로 짧아도 의미가 전달된다 (예: Member 클래스의 필드는 `memberName`이 아닌 `name`으로 충분)

### 4.2 중첩 if 최소화

코드 분석을 어렵게 만드는 요인 중 하나가 복잡한 if-else 문이다. if 문이 중첩될수록 복잡도는 배가 된다. 복잡한 if 문을 분석하려면 분기 조건과 같은 맥락을 기억해야 한다:

```java
if (조건 A) {
    ... 코드 1
    if (조건 B) {
        ... 코드 2
        if (조건 C) {
            ... 코드 3  // 조건 A, B, C를 추적해야 함
        }
    } else {
        ... 코드 4  // 조건 B 아닐 때 실행
    }
} else {
    ... 코드 5  // 조건 A 아닐 때 실행
}
```

**if 조건을 역으로 바꿔 코드 구조를 단순화**할 수 있다. 긴 if 블록 코드가 있고 조건이 참이 아닐 때 아무것도 수행하지 않는다면:

```java
// Before: 스크롤해야만 else 없음을 알 수 있다
if (조건) {
    ... 스크롤이 필요한 꽤 긴 코드
}

// After: 조건 불충족 시 바로 리턴함을 알 수 있다
if (!조건) {
    return;
}
... 스크롤이 필요한 꽤 긴 코드
```

중첩된 if 문도 같은 방식으로 단순화할 수 있다:

```java
// Before: 중첩 구조
if (A) {
    B 조건 구함
    if (B) {
        E 실행
    }
}

// After: 평탄한 구조
if (!A) return;
B 조건 구함
if (!B) return;
E 실행
```

if 조건을 역으로 바꾸면 **else가 없다는 사실을 빨리 알 수 있고**, 조건을 기억할 범위도 좁아진다. 코드 들여쓰기(깊이)가 줄어들기 때문에 코드 복잡도가 낮아져 코드 분석을 쉽게 할 수 있다.

### 4.3 변수 줄이기

일반적으로 변수가 사용되는 범위가 넓거나 변수 개수가 많을 때, 변수 변경이 빈번하거나 변수 용도가 중간에 바뀔 때 코드 분석 과정에서 기억해야 할 내용이 많아져 **인지 부하** 증가로 이어진다. 변수로 인한 인지 부하를 낮추려면 변수 자체를 줄여야 한다.

```java
// 불필요한 변수 — 한 번만 사용하고 값 변경 없이 그대로 할당
String name = any.getName();
String id = any.getId();
Summary ret = Summary.builder().name(name).id(id).build();

// 변수를 없애고 직접 값을 사용
Summary ret = Summary.builder().name(any.getName()).id(any.getId()).build();
```

반면, 의미를 부여하거나 새로운 타입으로 변환할 때 변수를 사용하면 코드 분석에 도움이 된다:

```java
// 변수 없이 계산식을 전달 — 전달하는 값의 의미를 유추하기 어려움
checkLimit(thisYear - mem.getBirthYear());

// 계산한 결과를 age 변수에 담아 전달 — 의미가 명확함
int age = thisYear - mem.getBirthYear();
checkLimit(age);
```

**변수 사용 범위는 최대한 좁게 한정하자:**

- 사용되기 직전에 정의
- 짧은 루프 블록 안으로 한정
- 짧은 if-else 블록 안으로 한정
- 짧은 메서드로 한정

```java
// Bad: for 블록 안에서만 사용되는 uname을 블록 밖에서 선언
String uname;
for (String m : members) {
    uname = masking(m.getName()) + "(" + m.getCompany() + ")";
    ... uname 사용 코드
}

// Good: for 블록 안에서 선언
for (String m : members) {
    String uname = masking(m.getName()) + "(" + m.getCompany() + ")";
    ... uname 사용 코드
}
```

같은 변수를 서로 다른 용도로 사용하면 안 된다. 단순히 타입이 같다는 이유 때문에 1개의 변수를 여러 의미로 사용하지 말아야 한다.

### 4.4 값 변경 최소화하기

변숫값이 중간에 계속해서 바뀌면 코드를 분석할 때 해당 변수에 담긴 값을 추적해야 한다는 부담이 생긴다. 특히 if 문 조건절에서 사용되는 변수가 중간에 변경된다면 코드 추적이 어려워진다.

변수가 담고 있는 값뿐만 아니라 변수가 참조하는 객체·구조체의 값이 바뀌어도 코드 분석 부담이 증가한다:

```java
StringBuilder sb = new StringBuilder("hello");
String s1 = sb.append(" world").toString();  // "hello world"
String s2 = sb.append(" world").toString();  // "hello world world"
```

코드를 주의 깊게 보지 않으면 s1과 s2 둘 다 "hello world"라고 생각할 수 있다. 하지만 `StringBuilder.append()`는 내부 데이터를 변경하기 때문에 s2는 "hello world world"가 된다.

**변수 변경은 가능한 한 최소화해야 한다.** 언어가 제공하는 요소의 특성을 잘 활용하면 데이터가 바뀌지 않는다는 것을 명확하게 표현할 수 있다:

- `const`나 `final` 같은 키워드로 변수가 바뀌지 않음을 보장
- 자바 레코드(*record*) 타입이나 코틀린의 데이터 클래스로 불변(*Immutable - 한 번 생성되고 나서 값이 변하지 않고 수정할 수 없는 상태*) 객체를 쉽게 생성

파라미터 값 변경도 최소화하자. 파라미터로 전달받은 값을 변경하면 함수를 호출한 코드를 분석할 때 어려움을 겪는다.

### 딥 다이브: 불변 객체와 성능

불변 객체를 사용하면 데이터 변경이 필요할 때 매번 새로운 객체를 생성한다. 예를 들어 자바의 BigInteger는 불변이다. `add()` 메서드는 기존 값을 변경하지 않고 새로운 객체를 생성한다. 이렇다 보니 "성능에 문제가 생기지 않을까"라고 걱정할 수 있다.

하지만 미리 걱정하지는 말자. 높은 성능을 요구하는 시스템이 아닌 일반적인 시스템에서 불변 객체를 사용한다고 해서 성능에 큰 문제가 생기지 않는다. GC도 개선되어 응답 지연 시간이 줄어들고 있다. 성능이 걱정된다면 목표를 설정하고 부하 테스트를 수행해서 측정하자. 목표 성능이 나오지 않으면 그때 문제 지점을 찾아서 수정하면 된다.

### 4.5 알맞은 파라미터 타입 사용하기

여러 메서드의 파라미터 타입으로 하나의 클래스를 공유해서 사용하는 방식은 초기에는 빠르게 개발이 진행된다고 느낄 수 있다. 하지만 개발과 유지보수가 진행됨에 따라 코드 분석을 어렵게 만든다:

```java
public class MemberService {
    public void save(MemberDto m) { ... }
    public void update(MemberDto m) { ... }
    public void setPassword(MemberDto m) { ... }
}
```

세 메서드는 각각의 기능을 수행하기 위해 서로 다른 값이 필요하다. 따라서 MemberDto 클래스는 세 메서드에서 필요로 하는 **모든 값**을 갖게 된다. `setPassword()` 메서드에서 사용하는 필드가 어떤 것인지 알아내려면 메서드 코드를 하나씩 확인해야 한다. 데이터베이스 연동 코드가 있다면 관련 코드도 전부 확인해야 한다.

또한 내부 구현 과정에서 필요한 데이터도 파라미터 타입에 추가하게 된다:

```java
public void save(MemberDto m) {
    int cnt = memberDao.count(m);     // m을 전달
    if (cnt == 1) {
        throw new DupException();
    }
    String otherId = memberDao.selectSameAddress(m);
    m.setFamilyId(otherId);           // 내부 구현 과정에서 필요한 값
    memberDao.insert(m);
}
```

이곳저곳에 데이터를 전달하기 위해 파라미터로 전달받은 타입을 공유해서 사용하면 코드를 분석하기가 힘들어진다. 불필요해 보이는 필드를 삭제할 때도 해당 타입을 사용하는 **모든 코드**를 확인해야 한다.

**이해하기 좋은 코드를 만들려면 메서드를 실행하는 데 필요한 값만 파라미터로 전달해야 한다:**

```java
// 암호 변경에 필요한 값만 담은 전용 파라미터 타입
public class ChangePasswordRequest {
    private String id;
    private String currentPassword;
    private String newPassword;
    private String newPasswordConfirm;
}
```

내부에서 다른 메서드를 호출할 때도 호출하는 메서드가 필요로 하는 값만 전달해야 한다. 타입이 늘어나면 코드 복잡도도 올라가고 귀찮기도 하지만, **하나의 타입을 여러 메서드의 파라미터 타입으로 사용하면서 늘어나는 코드 분석 시간이 더 큰 문제**다.

### 딥 다이브: 개발 생산성과 Map

> H사에서 웹 서비스 개발에 필요한 기반 코드를 작성해 달라는 요청을 받았다. H사는 Map을 사용할 것을 요구했고 컨트롤러, 서비스, 매퍼에서 Map을 사용하겠다고 했다. Map을 사용하면 처음에는 빠르게 기능이 만들어진다고 느낄 수 있다. 작성해야 하는 코드가 줄어들기 때문이다. 메서드마다 알맞은 파라미터 타입을 만들 필요가 없고, 한 타입에서 다른 타입으로 변환하는 코드를 만들지 않아도 된다.

하지만 어디까지나 개발 초기에만 이러한 이점을 누릴 수 있다. 코드 수정을 하려면 먼저 코드를 읽어야 하는데, Map이 사용된 코드라면 어떤 속성이 사용되는지 찾기 위해 프런트 코드부터 SQL 쿼리까지 탐색해야 한다. Map에 담겨 있는 값의 키 이름을 알 수 없기 때문이다. 코드 자동완성 같은 도움도 받기 힘들고, 컴파일러의 도움도 받을 수 없다. 키 이름을 잘못 썼다고 해도 컴파일은 되기 때문에 실제 실행해봐야 비로소 잘못된 이름을 사용한 사실을 알 수 있다.

**Map을 사용하면 코드 양이 줄긴 하지만 개발 생산성이 높아지는 것은 아니다.** 코드 양이 증가하고 시스템이 복잡해질수록 Map을 사용한 코드를 분석하고 버그를 찾고 수정하는 시간은 더 늘어난다.

## 5. 길지 않은 코드와 메서드 추출

암호문 같은 코드를 제외하면 긴 코드보다는 짧은 코드가 분석하기 쉽다. 클래스, 메서드, 함수가 길면 머리에 담아야 할 내용이 많아지고 코드 분석은 힘들어진다. 메서드나 함수를 길지 않게 만들려면 모든 구현을 한 메서드에 넣으면 안 된다. 대신 구현의 일부를 별도 메서드나 객체로 분리해야 한다.

```java
// Before: 모든 구현이 한 메서드에
public void save(SaveRequest req) {
    if (!StringUtils.hasText(req.getId()))
        throw new IllegalArgumentException("id는 필수");
    if (!StringUtils.hasText(req.getName()))
        throw new IllegalArgumentException("name은 필수");
    if (!StringUtils.hasText(req.getEmail()))
        throw new IllegalArgumentException("email은 필수");
    Long seq = repository.createSeq();
    Member member = Member.builder()
        .seq(seq)
        .id(req.getId()).name(req.getName()).email(req.getEmail())
        .createdAt(LocalDateTime.now())
        .build();
    repository.save(member);
}

// After: 검증과 생성을 별도 메서드로 분리
public void save(SaveRequest req) {
    validate(req);
    Member member = create(req);
    repository.save(member);
}

private void validate(SaveRequest req) {
    if (!StringUtils.hasText(req.getId()))
        throw new IllegalArgumentException("id는 필수");
    // ... 생략
}

private Member create(SaveRequest req) {
    Long seq = repository.createSeq();
    return Member.builder()
        .seq(seq)
        .id(req.getId()).name(req.getName()).email(req.getEmail())
        .createdAt(LocalDateTime.now())
        .build();
}
```

`save()` 메서드는 코드가 길지 않기 때문에 전반적인 코드 흐름을 빨리 이해할 수 있다. 상세한 구현은 알 수 없지만 대략 요청 데이터를 검증한 다음 요청 데이터에서 Member 객체를 생성하며 저장하고 있다는 것을 볼 수 있다. 자세한 구현이 궁금하면 각 메서드로 이동하면 된다.

## 6. 추상화 수준 맞추기

추상화 수준(*코드를 읽으면서 파악할 수 있는 정보의 수준*)이란, 한 메서드 내에서 코드가 표현하는 상세함의 정도를 말한다.

한 메서드에 모든 로직을 구현하는 개발자도 있다. 코드 편집기에서 이리저리 여러 탭을 왔다 갔다 하면 집중력이 떨어지기 때문이다. 하지만 코드가 어느 수준 이상으로 길어지면 추적해야 할 변수와 로직이 많아 전반적인 로직을 이해하는 데 어려움을 겪게 된다.

코드 길이를 길지 않게 유지하면서 코드 분석을 원활하게 하려면 **개념적인 단위로 코드를 묶어서 표현**해야 한다:

```java
// 구현 수준의 코드
public void register(RegisterRequest req) {
    int sameIdCount = userRepository.countById(req.getId());
    if (sameIdCount > 0) throw new DupIdException();
    int sameEmailCount = userRepository.countByEmail(req.getEmail());
    if (sameEmailCount > 0) throw new DupEmailException();
    User user = User.builder()
        .id(req.getId()).email(req.getEmail())
        .status(PENDING)
        .build();
    userRepository.save(user);
    UserHistory history = UserHistory.builder()
        .id(user.getId())
        .status(user.getStatus())
        .timestamp(now())
        .oldStatus(null)
        .build();
    historyDao.insert(history);
}

// 상위 수준으로 표현한 코드
public void register(RegisterRequest req) {
    checkSameIdExists(req.getId());
    checkSameEmailExists(req.getEmail());
    User user = createPendingUser(req);
    saveUser(user);
    appendNewUserCreatedHistory(user);
}
```

상세한 구현을 메서드로 분리하면 `register()` 메서드의 전체 흐름을 더 쉽게 이해할 수 있다.

**추상화 수준이 들쑥날쑥하면 코드를 분석할 때 개발자가 해석하는 과정을 거쳐야 한다.** 예를 들어 `checkSameIdExists`를 `checkSameIdCountIsOverZero`로 바꾸면, 다른 메서드들은 "무엇을 하는지"를 표현하는 수준인데 이 메서드만 "어떻게 하는지"를 표현하는 수준이 된다. 코드를 보면 처음에는 '같은 ID 개수가 0보다 큰지 검사한다'로 이해하고, 그러고 나서야 '같은 ID가 존재하는지 검사한다'로 해석할 수 있게 된다. 추상화 수준이 들쑥날쑥하면 머릿속으로 다시 수준을 맞추고 코드 분석을 해야 하므로 당연하게도 코드 분석 시간이 증가한다.

> **핵심 통찰**: 이해하기 좋은 코드를 만들려면 최대한 한 메서드 내에서 코드가 표현하는 추상화 수준을 맞추려고 노력해야 한다.

## 요약

- 개발 시간의 60%는 코드를 이해하는 데 소요된다. 버그 수정은 90% 이상을 코드 이해에 쓰기도 한다
- 코드를 이해하는 역량과 이해하기 쉬운 코드를 작성하는 역량, 두 가지 모두 필요하다
- **코드 이해 도구**: UML 다이어그램(액티비티, 시퀀스, 클래스, 상태), 의존·영향 관계 그래프, 코드 출력, 스크래치 리팩터링, 함께 모여 보기
- **이해하기 좋은 코드 작성법**:
  - 의미와 의도를 유추할 수 있는 이름을 사용하고, 범위에 따라 이름 길이를 조절한다
  - 중첩 if를 최소화하고 조건을 역으로 바꿔 구조를 단순화한다
  - 불필요한 변수를 줄이고 변수 사용 범위를 좁게 한정한다
  - 값 변경을 최소화하고 불변을 활용한다
  - 메서드에 알맞은 파라미터 타입을 사용하고, 하나의 타입을 여러 메서드에서 공유하지 않는다
- 메서드 추출을 통해 길이를 줄이고, 한 메서드 내에서 추상화 수준을 맞춰야 한다

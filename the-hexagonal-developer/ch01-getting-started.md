# Chapter 1: Getting Started (들어가며)

## 핵심 질문

개발이란 무엇이며, 좋은 시니어 개발자가 되기 위해서는 구현 기술 외에 어떤 역량이 필요한가?

---

## 1. 실패

### 1.1 물류 시스템 개발 실패

> 첫 직장에서 몇 달간 유지보수 위주 업무를 하다가, 모 업체와 물류 경매 시스템을 공동 개발하는 프로젝트에 참여하게 되었다. 화주와 차주를 온라인으로 연결해주는 웹 서비스였다. 나와 J씨는 경력 1년밖에 되지 않은 풋내기였지만 열심히 코딩했다. 리플렉션(*Reflection - 런타임에 클래스 정보를 조사하고 조작하는 기법*)을 사용해 SQL 쿼리를 생성하는 매퍼를 만들기도 했고, 자바 커넥션을 확장한 코드도 작성했다. 간단한 웹 MVC 프레임워크까지 직접 만들어 사용했다. 기술적으로 흥미로운 경험이었고 실력이 늘었다고 생각했다. **하지만 서비스는 출시조차 하지 못했다.**

이 실패에는 여러 원인이 있었다:

- **개발 리더 부재**: 신입 2명만 개발에 참여했고, 이끌어줄 사람이 없었다
- **요구사항 분석 부재**: 요구사항 이해를 위한 회의는 초기 한두 번이 전부였고, 경력 개발자와 기획자는 참석하지도 않았다
- **일정 관리 부재**: 얼마나 진척됐는지 아무도 모른 채 "둘이 알아서 하라"는 식이었다
- **샛길에 빠짐**: 기능 구현을 완성하기보다 기술적 호기심에 이끌려 샛길로 빠지기 일쑤였다

비슷한 실패는 이전에도 있었다. 3개 회사가 모여 기능을 연동하는 프로젝트에서 신입이었던 저자가 2개 회사와 연동을 협의하고 설계하며 구현해야 했다. H사 담당자도 주니어 개발자여서 구현에 어려움을 겪었고, 각 회사의 관리자는 프로젝트를 제대로 관리하지 않았다. 결국 이 프로젝트도 어영부영 진행되다 중단되었다. 두 프로젝트의 공통점은 명확하다 — **코딩 외의 활동(요구사항 분석, 일정 관리, 리더십)이 부재하면 아무리 열심히 코딩해도 프로젝트는 실패한다.**

> **핵심 통찰**: 리플렉션 같은 고급 기술을 사용하고 열심히 코딩했지만, 제품을 만들기 위해 필요한 코딩 외의 활동에 소홀하면 프로젝트는 실패한다. 두 번의 실패 모두 기술력이 아닌 관리와 리더십의 부재가 원인이었다.

## 2. 3년 차 — 성장의 정체

입사 후 3년이 지나자 유지보수에 능숙해지고 혼자서 새 기능을 만들 수 있는 개발자가 되었다. 하지만 이때부터 **실력이 늘지 않기 시작했다**:

- 같은 구현 기술을 사용하고 늘 비슷한 코드를 작성하는 반복
- 동일 패턴의 SQL 쿼리, 유사한 JSP 코드 작성의 연속
- 회사 업무에 대한 흥미 저하

> 하루가 멀다 하고 새로운 도구가 쏟아져 나오는 시대에 현재 사용하는 기술에만 멈춰 있으면 뒤처진다고 생각했고, 새로운 기술만 쫓게 되었다. 또한 뜬금없이 중간 관리자 역할을 맡게 되었는데, 관리자가 된 선배를 보니 개발 경력이 7~8년도 안 된 상태에서 개발에서 아예 손을 떼고 있었다. 적어도 어릴 때는 개발만 하고 싶었고, 전반적인 프로젝트 관리는 최대한 경력을 쌓은 뒤에 해야 한다고 생각했다.

### 딥 다이브: 새로운 구현 기술 사용 ≠ 성장

신입으로 채용한 직원이 "자기도 신규 프로젝트에 참여해서 여러 기술을 써보고 싶다"고 면담을 요청한 적이 있다. 또 한 번은 6년 경력의 직원이 퇴사 이유로 "새로운 기술을 사용해보고 싶어서"라고 했다. 늘 같은 일을 하며 같은 기술만 쓰다 보니 스스로 정체되어 성장하지 못했다고 느낀 것이다.

서비스 기업에서는 유지보수가 큰 비중을 차지하므로 최신 기술을 적용하기 쉽지 않다. 하지만 **언젠가 기회는 온다**. 오래된 서비스라도 일부 또는 전체를 새로 만들어야 하는 상황이 발생한다. 또한 구현 기술 외적으로도 성장시켜야 할 역량이 많으니, 어떤 구현 기술을 사용하고 싶다는 욕망에 너무 사로잡히지 말아야 한다.

## 3. 개발이란

사회 초년생 시절에는 **구현 기술을 사용해서 코드를 작성하는 것이 개발**이라고 생각했다. 이 외의 다른 활동은 개발이라 여기지 않았다. 하지만 경험이 쌓이면서 개발에 포함되는 영역이 코딩 이상이라는 사실을 알게 되었다.

개발이란:

- **고객에게 기능을 제공하는 일**: 사용자에게 도움을 주거나, 사용자가 원하는 매력적인 기능을 만드는 활동
- **수익과 직결되는 과정**: 직간접적으로 회사의 수익과 투자에 영향을 주며, 서비스와 회사의 지속 가능성을 높이는 활동
- **코딩 이상의 영역**: 요구사항 파악, 일정 관리, 설계, 아키텍처 등 다양한 활동을 포함

> **핵심 통찰**: 코딩과 구현 기술은 개발의 일부이지 개발의 전부는 아니다. 어떤 기술을 사용하기 위해 또는 특정 개인의 이력 관리를 위해 시작한 프로젝트는 겉으로는 그럴듯해 보여도 실제로는 별다른 결과 없이 사라지곤 한다.

> 그나마 다행인 점은 개발을 좁게만 생각한 것과 성장이 같지 않다는 사실을 빨리 깨우쳤다는 것이다. 서점에서 스티브 맥코넬의 「소프트웨어 프로젝트 생존전략」을 읽으면서 프로젝트 일정, 요구사항, 위험 관리 등 여러 내용을 접하며 내가 생각한 개발은 전체에서 아주 일부라는 사실을 비로소 깨달았다.

## 4. 개발에 필요한 것

좋은 개발을 하려면 여러 역량이 필요하다. 주니어 개발자가 시니어 개발자로 성장하기 위해 익혀야 할 역량은 다음과 같다:

| 역량 | 설명 |
|------|------|
| **구현 기술** | 코드를 작성하여 동작하는 소프트웨어를 만드는 능력. 초창기 몇 년만 학습하고 이후 새로운 기술을 등한시하면 좋은 시니어 개발자가 될 수 없다 |
| **설계 역량** | 서비스 경쟁력에 직결되는 능력. 확실한 정답이 없고 상황에 따라 좋고 나쁨이 달라진다. 다양한 경험과 꾸준한 학습이 필요하며 단기간에 늘지 않는다 |
| **업무 관리** | 경력과 무관하게 많은 개발자가 중요하지 않게 여기지만, 관리자만 하는 일이 아니다 |
| **요구 분석** | 고객이 원하는 것을 정확히 파악하는 능력 |
| **공유** | 글쓰기와 발표를 통한 소통 역량 |
| **리드 & 팔로우** | 협업을 위한 리더와 팔로워의 역할 이해 |

### 딥 다이브: 물류 시스템 개발은 왜 망했을까?

출시도 못 하고 실패한 물류 시스템의 가장 큰 문제는 **개발 리더 부재**였다:

- 신입 2명만 참여했고, 정말 열심히 코딩했지만 제품을 만들기 위해 필요한 다른 활동에 소홀했다
- 정확하게 말하면 **코딩 외에 어떤 일을 해야 하는지 몰랐다**
- 요구사항 분석이라는 첫 단추부터 틀어졌고, 일정 관리도 엉망이었다
- "잘 되고 있지?"라는 물음에 "네"라고 답했지만, 실제로 잘 진행되고 있다는 것은 착각이었다

## 요약

- 구현 기술만으로는 좋은 소프트웨어를 만들 수 없다. 요구사항 분석, 일정 관리, 설계, 리더십 등 다양한 역량이 필요하다
- **새로운 기술 사용 ≠ 성장**이다. 유지보수 업무에서도 구현 기술 외적인 역량을 키울 수 있다
- 개발은 코딩이 아니라 **고객에게 가치 있는 기능을 제공하는 전 과정**이다
- 시니어 개발자로 성장하려면 구현 기술, 설계, 업무 관리, 공유, 리더십/팔로워십 등 다양한 역량을 균형 있게 키워야 한다
- 코딩에만 몰두하고 나머지를 소홀히 하면, 프로젝트는 실패하고 개발자로서의 성장도 정체된다

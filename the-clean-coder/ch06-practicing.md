# Chapter 6: Practicing (연습)

## 핵심 질문

프로 소프트웨어 개발자는 어떻게 자신의 기술을 유지하고 향상시키는가? 업무 시간의 코딩이 충분한 연습이 아니라면, 개발자는 무엇을 어떻게 연습해야 하는가?

---

## 1. 연습과 공연의 구분

### 1.1 업무는 연습이 아니다

많은 개발자가 "나는 매일 코딩하니까 별도의 연습이 필요 없다"고 생각한다. 하지만 Uncle Bob은 이것이 근본적인 착각이라고 지적한다.

**업무 시간의 코딩은 공연(performance)이지 연습(practice)이 아니다.**

음악가를 생각해보자. 콘서트에서 연주하는 것은 "공연"이다. 하지만 음악가는 공연만으로 실력이 향상되지 않는다는 것을 안다. 공연에서는 실수를 최소화해야 하고, 이미 아는 곡을 완벽하게 재현해야 한다. **새로운 기술을 시도하거나 한계를 밀어붙이는 것은 공연장이 아니라 연습실에서 해야 한다.**

소프트웨어 개발도 마찬가지다:

| 구분 | 업무(공연) | 연습 |
|------|-----------|------|
| **목적** | 고객에게 가치를 전달한다 | 자신의 기술을 향상시킨다 |
| **실수 비용** | 높다 (프로덕션 장애, 일정 지연) | 낮다 (아무도 영향받지 않는다) |
| **새로운 시도** | 제한적이다 (검증된 방법을 써야 한다) | 자유롭다 (실패해도 괜찮다) |
| **시간 압박** | 있다 (데드라인, 스프린트) | 없다 (자기 페이스) |
| **피드백** | 느리다 (릴리스 후에야 안다) | 즉각적이다 (바로 확인할 수 있다) |

> **핵심 통찰**: 프로 음악가가 무대에서 처음 보는 악보를 연주하지 않듯이, 프로 개발자는 업무 시간에 처음 접하는 기법을 실험하지 않는다. 새로운 기술은 개인 연습 시간에 먼저 익히고, 충분히 숙달된 후에 업무에 적용하는 것이 프로의 방식이다.

### 1.2 다른 전문직의 연습

Uncle Bob은 소프트웨어 개발자가 연습에 소홀한 것이 다른 전문직과 비교했을 때 얼마나 이례적인지를 강조한다:

- **음악가**: 하루에 5시간 이상 연습한다. 공연 시간보다 연습 시간이 압도적으로 많다. 기본 스케일을 수십 년 경력의 연주자도 매일 연습한다.
- **무술가**: 카타(형)를 수백, 수천 번 반복한다. 검은 띠 고수도 기본 동작을 매일 연습한다.
- **운동선수**: 실전 경기 시간보다 훈련 시간이 몇 배 더 많다. 농구 선수는 자유투를 수천 번 반복 연습한다.
- **외과의사**: 수술 시뮬레이션을 통해 새로운 기법을 연습한다. 환자에게 바로 적용하지 않는다.
- **프로그래머**: ???

대부분의 프로그래머는 **별도의 연습을 전혀 하지 않는다**. 업무 시간의 코딩이 유일한 코딩이다. Uncle Bob은 이것이 **비프로적(unprofessional)**이라고 단언한다.

## 2. 코딩 카타(Coding Kata)

### 2.1 카타란 무엇인가

"카타(Kata)"는 일본 무술에서 유래한 용어로, **정해진 형(型)을 반복적으로 연습하는 것**을 말한다. 가라테에서 카타는 일련의 공격과 방어 동작을 정해진 순서대로 수행하는 것이다. 검도에서는 정해진 형태의 베기와 찌르기를 반복한다.

코딩 카타는 이와 동일한 개념을 프로그래밍에 적용한 것이다:

**이미 해답을 알고 있는 간단한 프로그래밍 문제를 반복적으로 풀면서, 손가락과 마음의 움직임을 훈련하는 것이다.**

> **핵심 통찰**: 카타의 목적은 문제를 "해결"하는 것이 아니다. 이미 해결 방법을 알고 있다. 카타의 목적은 해결 과정에서의 **동작 — 타이핑, 단축키 사용, 리팩토링 순서, TDD 사이클의 리듬 — 을 몸에 익히는 것**이다. 음악가가 이미 알고 있는 스케일을 매일 연습하는 것과 같은 이유이다.

### 2.2 대표적인 코딩 카타

Uncle Bob이 자주 수행하는 카타와 소프트웨어 개발 커뮤니티에서 널리 알려진 카타 몇 가지를 소개한다:

#### 볼링 게임(Bowling Game) 카타

볼링 점수 계산 프로그램을 TDD로 작성하는 카타이다. 스트라이크, 스페어, 거터(*Gutter — 볼링에서 공이 핀에 도달하지 못하고 양쪽 홈에 빠지는 것. 점수는 0이다.*) 등의 규칙을 테스트 주도로 구현한다.

- **난이도**: 초급~중급
- **소요 시간**: 숙달 후 15~30분
- **연습 포인트**: TDD의 Red-Green-Refactor 사이클, 점진적 설계

```java
// Red: 실패하는 테스트 작성
@Test
public void gutterGame() {
    Game g = new Game();
    for (int i = 0; i < 20; i++) {
        g.roll(0);
    }
    assertEquals(0, g.score());
}

// Green: 최소한의 코드로 통과
public class Game {
    public void roll(int pins) {}
    public int score() { return 0; }
}

// 다음 테스트로 진행 → 스페어 처리 → 스트라이크 처리 → 리팩토링
```

#### 소인수 분해(Prime Factors) 카타

주어진 정수를 소인수의 리스트로 분해하는 프로그램을 TDD로 작성한다.

- **난이도**: 초급
- **소요 시간**: 숙달 후 10~15분
- **연습 포인트**: 알고리즘의 점진적 출현, 테스트 순서의 중요성

```java
// 1 → []
// 2 → [2]
// 3 → [3]
// 4 → [2, 2]
// 6 → [2, 3]
// 8 → [2, 2, 2]
// 9 → [3, 3]

public static List<Integer> generate(int n) {
    List<Integer> primes = new ArrayList<>();
    for (int divisor = 2; n > 1; divisor++) {
        while (n % divisor == 0) {
            primes.add(divisor);
            n /= divisor;
        }
    }
    return primes;
}
```

이 카타에서 흥미로운 점은 테스트를 하나씩 추가하면서 구현하면, 알고리즘이 **자연스럽게 출현**한다는 것이다. 처음부터 완성된 알고리즘을 설계하지 않아도, TDD 사이클을 따르면 올바른 코드에 도달한다.

#### 단어 줄바꿈(Word Wrap) 카타

주어진 문자열을 지정된 열 너비에서 줄바꿈하는 프로그램을 작성한다.

- **난이도**: 중급
- **소요 시간**: 숙달 후 15~20분
- **연습 포인트**: 경계 조건 처리, 문자열 조작

### 2.3 카타 수행의 핵심 원칙

카타를 효과적으로 수행하기 위한 핵심 원칙을 정리한다:

| 원칙 | 설명 |
|------|------|
| **반복** | 같은 카타를 수십, 수백 번 반복한다. 한 번 풀고 끝내는 것은 카타가 아니다 |
| **속도 향상** | 반복할수록 더 빠르고 정확하게 수행하는 것을 목표로 한다 |
| **자동화된 테스트** | 반드시 TDD로 수행한다. 카타의 핵심 목적 중 하나가 TDD 근육 기억을 기르는 것이다 |
| **도구 숙달** | 에디터 단축키, 리팩토링 도구, 디버거 등을 의식적으로 활용한다 |
| **변형** | 충분히 숙달된 후에는 같은 카타를 다른 언어, 다른 접근법으로 시도한다 |

> **Uncle Bob의 경험**: Uncle Bob은 매일 아침 업무를 시작하기 전에 코딩 카타를 수행한다. 마치 음악가가 공연 전에 손가락을 풀기 위해 스케일을 연주하듯이, 카타는 프로그래밍 "워밍업"의 역할을 한다. 이 습관을 시작한 후 코딩 속도와 자신감이 눈에 띄게 향상되었다고 말한다. 특히 TDD 사이클의 리듬이 몸에 배면서, 업무 시간에도 자연스럽게 테스트 우선 개발이 이루어지게 되었다.

## 3. 코딩 와사(Coding Wasa)

### 3.1 와사란 무엇인가

"와사(Wasa)"는 무술에서 **두 사람이 짝을 이루어 정해진 형을 수행하는 연습**을 말한다. 한 사람이 공격하면 다른 사람이 방어하고, 역할을 바꿔가며 반복한다.

코딩 와사는 이 개념을 프로그래밍에 적용한 것으로, **핑퐁 TDD(Ping-Pong TDD)**라고도 부른다:

1. **개발자 A**: 실패하는 테스트를 작성한다 (Red)
2. **개발자 B**: 테스트를 통과시키는 최소한의 코드를 작성한다 (Green)
3. **개발자 B**: 새로운 실패하는 테스트를 작성한다 (Red)
4. **개발자 A**: 테스트를 통과시키는 최소한의 코드를 작성한다 (Green)
5. 역할을 계속 교대하며 반복한다

### 3.2 와사의 가치

와사는 단순한 카타에 **사회적 차원**을 추가한다:

- **다른 사람의 사고 과정을 관찰한다**: 상대방이 어떤 테스트를 작성하는지, 어떤 순서로 코드를 작성하는지를 볼 수 있다
- **의사소통 능력을 기른다**: 코드로 의도를 전달하는 능력이 향상된다
- **다양한 접근법을 배운다**: 같은 문제에 대해 다른 사람이 전혀 다른 접근법을 취하는 것을 경험할 수 있다
- **페어 프로그래밍(*Pair Programming — 두 명의 개발자가 하나의 워크스테이션에서 함께 코딩하는 실천법. 한 명이 코드를 작성하고(드라이버), 다른 한 명이 검토하며 방향을 제시한다(내비게이터). XP의 핵심 실천법 중 하나이다.*)의 준비 운동이 된다**: 와사를 통해 두 사람이 하나의 코드베이스에서 협업하는 리듬을 익힌다

### 대화: 와사 세션

> **개발자 A**: "좋아, 내가 첫 테스트를 쓸게. 빈 문자열을 입력하면 빈 리스트를 반환하는 테스트."
>
> **개발자 B**: "테스트 통과시켰어. 그냥 빈 리스트 반환하면 되잖아. 이제 내 차례 — 'a'를 입력하면 ['a']를 반환하는 테스트."
>
> **개발자 A**: "흠, 너는 거기서 바로 split을 쓸 줄 알았는데, 그냥 하드코딩했네? 재밌다. 그럼 이 테스트는 — 'a,b'를 입력하면 ['a', 'b']를 반환..."
>
> **개발자 B**: "아, 이제 진짜 split이 필요하겠다."

이 대화에서 볼 수 있듯이, 와사는 두 사람이 **테스트와 코드를 주고받으며 점진적으로 솔루션을 완성하는 과정**이다. 서로의 접근법에서 배우고, 코드로 대화하는 능력을 기른다.

## 4. 란도리(Randori)

### 4.1 란도리란 무엇인가

"란도리(Randori)"는 일본 무술에서 **자유 대련**을 뜻한다. 정해진 형이 없이, 여러 사람이 자유롭게 공격과 방어를 주고받는 연습 형태이다.

코딩 란도리는 **그룹 코딩 연습**이다:

1. 화면을 프로젝터에 띄운다
2. 한 사람이 실패하는 테스트를 작성한다
3. 다른 사람이 앞으로 나와서 테스트를 통과시킨다
4. 테스트를 통과시킨 사람이 새로운 테스트를 작성한다
5. 또 다른 사람이 나와서 통과시킨다
6. 이 과정을 계속 반복한다

### 4.2 란도리의 독특한 가치

란도리는 카타나 와사와 다른 독특한 가치를 제공한다:

| 특성 | 카타 | 와사 | 란도리 |
|------|------|------|--------|
| **인원** | 1명 | 2명 | 그룹 (5명 이상) |
| **형식** | 정해진 문제 반복 | 핑퐁 TDD | 자유 형식 |
| **사회적 요소** | 없음 | 중간 | 높음 |
| **학습 방향** | 개인 기술 연마 | 상호 학습 | 다양한 관점 노출 |
| **재미** | 명상적 | 대화적 | 에너지 넘침 |

란도리의 핵심 가치:

- **팀 빌딩**: 함께 코딩하면서 팀의 유대감이 형성된다
- **교육**: 경험 많은 개발자의 코딩 과정을 실시간으로 볼 수 있다
- **용기**: 다른 사람 앞에서 코딩하는 것에 대한 두려움을 극복한다
- **다양한 스타일 노출**: 같은 문제에 대해 각자 다른 접근법을 취하는 것을 관찰한다

> **핵심 통찰**: 란도리는 기술적 연습인 동시에 **사회적 활동**이다. 코딩은 흔히 혼자 하는 활동이라고 생각되지만, 란도리는 코딩을 팀 스포츠로 만든다. 서로의 코드를 보고, 배우고, 토론하며, 함께 성장한다.

### 4.3 란도리를 팀에 도입하는 방법

란도리를 팀에 도입하려면 다음 단계를 따르는 것이 효과적이다:

1. **간단한 문제로 시작한다**: FizzBuzz나 볼링 게임 같은 잘 알려진 카타를 사용한다
2. **심리적 안전감을 보장한다**: "틀려도 괜찮다"는 분위기를 먼저 조성한다
3. **시간을 제한한다**: 처음에는 30~45분으로 시작한다
4. **회고한다**: 란도리 후에 무엇을 배웠는지, 어떤 접근법이 인상적이었는지를 공유한다
5. **정기적으로 한다**: 일회성이 아니라 주 1회 또는 격주 1회로 정착시킨다

## 5. 경험의 폭 넓히기

### 5.1 오픈 소스 기여

Uncle Bob은 개인 연습을 넘어 **경험의 폭을 넓히라**고 권장한다. 가장 효과적인 방법 중 하나가 오픈 소스 프로젝트에 기여하는 것이다.

오픈 소스 기여의 학습 가치:

- **다른 사람의 코드를 읽는다**: 실력 향상의 가장 빠른 방법 중 하나는 잘 작성된 코드를 읽는 것이다
- **코드 리뷰를 받는다**: 오픈 소스 메인테이너(*Maintainer — 오픈 소스 프로젝트의 주요 관리자. 코드 리뷰, 릴리스 관리, 프로젝트 방향 설정 등의 책임을 진다.*)들의 엄격한 코드 리뷰는 값비싼 교육이다
- **다양한 코딩 표준에 노출된다**: 각 프로젝트마다 다른 컨벤션과 관행을 익힐 수 있다
- **실제 사용자가 있는 소프트웨어에 기여한다**: 개인 프로젝트와 달리, 실제 사용자의 피드백을 받을 수 있다

### 5.2 다른 언어로 연습하기

Uncle Bob은 업무에서 사용하는 언어 외에 **다른 프로그래밍 언어로도 연습하라**고 권장한다.

다른 언어를 배우는 것이 현재 사용하는 언어의 실력도 향상시키는 이유:

| 언어 패러다임 | 배울 수 있는 것 | 예시 언어 |
|-------------|---------------|----------|
| **함수형** | 불변성, 고차 함수, 사이드 이펙트 최소화 | Haskell, Clojure, Erlang |
| **동적 타입** | 유연한 설계, 덕 타이핑(*Duck Typing — "오리처럼 걷고 오리처럼 꽥꽥거리면, 그것은 오리다"에서 유래한 개념. 객체의 타입보다 객체가 가진 메서드와 속성이 중요하다는 프로그래밍 스타일.*), 메타프로그래밍 | Ruby, Python, JavaScript |
| **정적 타입** | 타입 안전성, 컴파일 타임 검증, 도메인 모델링 | Haskell, Rust, Scala |
| **저수준** | 메모리 관리, 성능 최적화, 하드웨어 이해 | C, Rust, Assembly |
| **논리형** | 선언적 사고, 제약 기반 프로그래밍 | Prolog |

> **핵심 통찰**: 프로그래밍 언어는 단순한 도구가 아니라 **사고의 틀**이다. 새로운 언어를 배우면 새로운 방식으로 문제를 바라보게 된다. Java만 아는 개발자와 Java, Haskell, Clojure를 아는 개발자는 같은 문제를 보더라도 **전혀 다른 해결책을 떠올린다**.

### 5.3 다양한 도메인 경험

같은 기술 스택이라도 다양한 도메인을 경험하면 개발자로서의 폭이 넓어진다:

- 금융 시스템의 정확성 요구사항
- 게임 개발의 성능 최적화
- 웹 서비스의 확장성 설계
- 임베디드 시스템의 자원 제약

각 도메인은 고유한 문제와 해결 패턴을 가지고 있으며, 이를 경험하면 다른 도메인에서도 더 나은 해결책을 떠올릴 수 있다.

## 6. 연습의 윤리: 누구의 시간인가

### 6.1 고용주의 시간 vs 자신의 시간

Uncle Bob은 연습에 대해 **논쟁적이지만 단호한 입장**을 취한다:

**연습은 자기 시간에 하라. 고용주는 당신의 연습 시간을 지불할 의무가 없다.**

이 주장의 근거:

- 고용주는 당신이 이미 전문가라는 전제로 급여를 지불한다
- 기술 유지와 향상은 **개인의 책임**이다
- 의사가 새로운 수술 기법을 환자에게 직접 연습하지 않듯이, 개발자도 고객의 프로젝트에서 연습하지 않아야 한다

### 6.2 이 주장에 대한 반론과 균형

Uncle Bob의 이 입장에 대해 커뮤니티에서 다양한 반론이 존재한다:

| 반론 | Uncle Bob의 재반박 |
|------|-------------------|
| "고용주도 직원 교육에 투자해야 한다" | 맞다. 하지만 기본적인 기술 유지는 개인 책임이다. 고용주의 교육 투자는 보너스이지 기본이 아니다 |
| "주 40시간 외에 20시간 더 투자하면 번아웃된다" | 20시간은 권장 상한이다. 중요한 것은 시간의 양이 아니라 **의도적으로 연습한다는 사실** 자체이다 |
| "업무 중에도 새로운 것을 배울 수 있다" | 물론이다. 하지만 업무 중 학습은 프로젝트 요구사항에 종속된다. 체계적인 기술 향상과는 다르다 |

> **Uncle Bob의 경험**: "나는 개발자들에게 주 60시간 일하라고 말하는 것이 아니다. 주 40시간은 고용주를 위해 일하고, 그 외에 매일 1~2시간, 출퇴근 시간이든 아침 시간이든, **의도적으로 자기 기술을 연마하는 시간을 가져라**는 것이다. 프로 음악가에게 '공연 외에도 연습을 하세요'라고 말하면 아무도 이상하게 생각하지 않는다. 프로 개발자에게 같은 말을 하면 왜 논쟁이 되는가?"

### 6.3 현실적인 연습 시간 확보 방법

Uncle Bob의 원칙에 동의하더라도 실행이 어려울 수 있다. 현실적인 방법을 정리한다:

1. **아침 루틴에 카타를 포함한다**: 출근 전 15~30분, 커피 한 잔과 함께 카타 하나를 수행한다
2. **출퇴근 시간을 활용한다**: 기술 팟캐스트를 듣거나, 기술 서적을 읽는다
3. **점심시간 활용**: 짧은 코딩 카타나 기술 블로그 읽기에 활용한다
4. **주말에 개인 프로젝트**: 관심 있는 오픈 소스에 기여하거나, 새로운 언어로 작은 프로젝트를 만든다
5. **팀 활동으로 전환**: 란도리나 코딩 도장을 팀 활동으로 만들면, 업무 시간에 연습의 요소를 포함시킬 수 있다

## 실전 시나리오

### 시나리오 1: 새로운 프레임워크 도입

**상황**: 팀에서 새로운 프레임워크를 도입하기로 결정했다. PM이 "다음 스프린트에서 새 프레임워크로 전환하자"고 한다.

**비프로 대응**:
> "좋아요, 바로 적용해보겠습니다. 해보면서 배우면 되죠."

**프로 대응**:
> "새 프레임워크의 핵심 기능을 먼저 개인 시간에 연습하겠습니다. 이번 주말에 간단한 프로토타입을 만들어보고, 다음 주에 팀에 공유하겠습니다. 충분히 숙달된 후 스프린트에서 적용하겠습니다."

**원칙**: 프로는 고객의 프로젝트를 연습장으로 사용하지 않는다. 새로운 기술은 개인 시간에 먼저 익힌다.

### 시나리오 2: 기술적 정체

**상황**: 5년차 개발자가 최근 1년간 업무에서 새로 배우는 것이 없다고 느낀다. 매일 같은 패턴, 같은 기술 스택, 같은 유형의 문제만 반복하고 있다.

**비프로 대응**:
> "뭐, 회사에서 쓰는 기술에는 능숙하니까 괜찮지. 업무 외에 따로 공부할 시간도 없고."

**프로 대응**:
> 매일 아침 15분 코딩 카타를 시작한다. 매주 업무 외 언어(예: Rust, Clojure)로 같은 카타를 풀어본다. 월 1회 오픈 소스 프로젝트에 기여한다. 매달 기술 서적 1권을 읽는다.

**원칙**: 기술적 성장이 멈추었다면 그것은 환경의 문제가 아니라 개인의 태도 문제이다. 프로는 성장의 책임을 환경에 전가하지 않는다.

## 프로의 원칙

| 원칙 | 설명 |
|------|------|
| **업무는 공연이고, 연습은 별도로 한다** | 프로는 업무 시간을 연습 시간과 구분한다. 새로운 기술은 개인 시간에 먼저 익힌다 |
| **카타를 반복한다** | 이미 아는 문제를 반복적으로 풀며 손과 마음의 속도를 높인다 |
| **다른 사람과 함께 연습한다** | 와사와 란도리를 통해 사회적 차원의 학습을 한다 |
| **경험의 폭을 넓힌다** | 다른 언어, 다른 도메인, 오픈 소스를 통해 시야를 확장한다 |
| **연습은 자기 책임이다** | 기술 유지와 향상은 고용주의 의무가 아니라 프로 자신의 의무이다 |

## 요약

- **업무 시간의 코딩은 공연이지 연습이 아니다.** 음악가가 콘서트에서 새 기법을 시험하지 않듯, 프로 개발자는 업무에서 새 기술을 처음 시도하지 않는다.
- **코딩 카타(Kata)는 이미 아는 문제를 반복적으로 푸는 것이다.** 목적은 문제 해결이 아니라 TDD 리듬, 타이핑 속도, 도구 숙달 등 동작의 자동화이다.
- **코딩 와사(Wasa)는 두 사람이 핑퐁 TDD를 하는 것이다.** 테스트 작성자와 구현자의 역할을 교대하며 서로의 사고 과정을 배운다.
- **란도리(Randori)는 그룹 코딩 연습이다.** 팀 빌딩과 다양한 코딩 스타일 노출의 효과가 있다.
- **경험의 폭을 넓혀라.** 오픈 소스 기여, 다른 언어 학습, 다양한 도메인 경험이 개발자의 역량을 근본적으로 확장한다.
- **연습은 자기 시간에 한다.** 기술 유지와 향상은 프로 자신의 책임이다. 고용주에게 연습 시간을 요구하는 것은 프로답지 못하다.
- **다른 전문직과 비교해 보라.** 음악가, 무술가, 운동선수, 의사 — 모두 공연/실전 외에 별도의 연습을 한다. 프로그래머만이 예외일 수 없다.

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: 프로의 직업 윤리에서 "자기 시간에 투자하라"는 원칙을 제시했다. 이 챕터는 그 원칙의 **구체적인 실천 방법** — 카타, 와사, 란도리 — 을 제공한다.
- **Chapter 4 (Coding)**: 코딩 시 갖춰야 할 마음가짐과 규율을 다루었다. 연습은 이러한 규율을 **몸에 익히는 방법**이다. 연습을 통해 TDD, 리팩토링, 집중력 유지 등이 자동화되면 업무 시간의 코딩 품질이 향상된다.
- **Chapter 5 (Test Driven Development)**: TDD의 원칙과 규율을 소개했다. 카타는 TDD 사이클을 **근육 기억으로 만드는** 가장 효과적인 방법이다. TDD를 이해하는 것과 TDD가 몸에 배는 것은 다르며, 그 차이를 만드는 것이 반복 연습이다.
- **Chapter 7 (Acceptance Testing)**: 인수 테스트를 작성하고 자동화하는 능력도 연습이 필요하다. 테스트 작성 능력은 타고나는 것이 아니라 카타를 통해 훈련되는 것이다.
- **Chapter 12 (Collaboration)**: 와사와 란도리는 코딩 연습인 동시에 **협업 연습**이다. 다른 사람과 함께 코딩하는 것에 익숙해지면 페어 프로그래밍, 코드 리뷰 등 실무 협업도 자연스러워진다.
- **Chapter 14 (Mentoring, Apprenticeship, and Craftsmanship)**: 란도리와 와사는 **멘토링의 도구**가 될 수 있다. 경험 많은 개발자와 함께 카타를 수행하며 배우는 것은 도제 모델의 현대적 구현이다.

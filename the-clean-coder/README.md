# The Clean Coder

**저자**: Robert C. Martin (Uncle Bob)
**출판**: 2011, Prentice Hall

---

## 책 소개

"The Clean Coder: A Code of Conduct for Professional Programmers"는 소프트웨어 개발자가 **프로페셔널**이 된다는 것이 무엇을 의미하는지를 다루는 책이다. 같은 저자의 "Clean Code"가 **코드의 품질**에 집중한다면, 이 책은 코드를 작성하는 **사람의 태도와 행동**에 집중한다.

Robert C. Martin(Uncle Bob)은 40년 이상의 소프트웨어 개발 경험에서 얻은 실패와 성공의 교훈을 솔직하게 공유하며, 프로 개발자가 갖춰야 할 규율, 기술, 태도를 구체적인 행동 강령으로 제시한다.

### Clean Code vs Clean Coder

| 구분 | Clean Code (2008) | The Clean Coder (2011) |
|------|-------------------|------------------------|
| 핵심 주제 | 깨끗한 **코드** 작성법 | 프로 **개발자**의 행동 규범 |
| 다루는 범위 | 변수명, 함수, 클래스, 테스트 | 거절하기, 추정, 시간 관리, 협업 |
| 대상 독자 | 코드 품질을 높이고 싶은 개발자 | 프로로서의 태도를 갖추고 싶은 개발자 |
| 핵심 질문 | "좋은 코드란 무엇인가?" | "프로 개발자란 어떤 사람인가?" |
| 형식 | 코드 예제 중심 | 일화(anecdote)(*anecdote — 저자의 실제 경험에서 가져온 짧은 이야기. Uncle Bob은 1970년대부터의 자신의 실패담을 교훈의 도구로 활용한다.*)와 대화 시나리오 중심 |

### 이 책의 특징

- **솔직한 자기 고백**: Uncle Bob은 자신이 해고당한 경험, 고객에게 거짓말한 경험, 일정을 지키지 못한 경험 등을 숨기지 않는다. 이러한 실패담이 각 원칙의 설득력 있는 근거가 된다.
- **구체적인 행동 강령**: "프로가 되어라"라는 모호한 조언 대신 "No라고 말하는 방법", "추정하는 방법", "압박 속에서 행동하는 방법" 등 실천 가능한 지침을 제공한다.
- **대화 시나리오**: 각 챕터에 관리자-개발자 간 대화 시나리오를 포함하여, 이상적인 대응과 잘못된 대응을 대비시킨다.
- **논쟁적 입장**: TDD 필수론, 야근 반대론, QA에게 버그를 넘기지 말라는 주장 등 일부 강한 입장은 개발 커뮤니티에서 활발한 토론을 촉발했다.

## 이 저장소의 목적

이 저장소는 "The Clean Coder"의 내용을 챕터별로 상세하게 정리한 학습 자료이다. 각 챕터의 핵심 원칙, Uncle Bob의 경험담, 실전 시나리오를 책을 대체할 수 있을 정도로 충실하게 담고 있으며, 다음과 같은 용도로 활용할 수 있다:

- 프로 개발자의 태도와 규율을 체계적으로 학습하는 교재
- 팀 내 개발 문화와 협업 규범을 논의하는 토론 자료
- 커리어 초기 개발자가 업계의 기대치를 이해하는 가이드

## 학습 가이드

### 추천 읽기 순서

이 책은 앞에서 뒤로 순서대로 읽는 것이 자연스럽다. 하지만 특정 주제에 관심이 있는 개발자라면 아래 가이드를 참고할 수 있다.

**1단계: 마인드셋 — 프로란 무엇인가 (Introduction, Chapter 1)**

모든 원칙의 기반이 되는 "프로페셔널리즘"의 정의를 이해한다. Uncle Bob이 왜 이 책을 썼는지, 프로 개발자에게 기대되는 것이 무엇인지를 먼저 파악한다.

**2단계: 커뮤니케이션 — 말하는 법 (Chapter 2~3)**

개발자가 가장 어려워하는 영역인 "거절하기"와 "약속하기"를 다룬다. 관리자, PM, 동료와의 실전 대화 시나리오를 통해 프로다운 커뮤니케이션 방법을 배운다.

**3단계: 핵심 기술 — 코딩, 테스트, 연습 (Chapter 4~6)**

프로 개발자의 기술적 규율을 다룬다. 코딩할 때의 마음가짐, TDD의 필요성, 그리고 지속적인 연습(카타, 와사)의 중요성을 배운다.

**4단계: 팀 운영 — 테스트, 시간, 추정, 압박 (Chapter 7~11)**

팀 환경에서 프로답게 일하는 방법을 다룬다. 인수 테스트로 소통하기, 시간 관리, 정직한 추정, 압박 속에서도 규율을 유지하는 방법을 배운다.

**5단계: 조직과 성장 — 협업, 팀, 장인정신 (Chapter 12~14)**

프로 개발자가 조직에서 어떻게 협업하고, 후배를 어떻게 멘토링하며, 장인정신을 어떻게 전파하는지를 다룬다.

## 핵심 개념 요약: 프로의 5가지 핵심 속성

이 책의 모든 내용은 프로 소프트웨어 개발자가 갖춰야 할 다섯 가지 핵심 속성으로 수렴한다:

| 속성 | 설명 | 관련 챕터 |
|------|------|-----------|
| **책임감(Responsibility)** | 자신의 코드와 결정에 대해 책임을 진다. QA가 버그를 찾도록 내버려두지 않는다. | Ch 1, 5, 8 |
| **규율(Discipline)** | 압박 속에서도 TDD, 리팩토링 등 자신의 규율을 유지한다. 위기 상황이 규율의 진짜 시험이다. | Ch 4, 5, 11 |
| **정직(Honesty)** | "시도해 보겠다"가 아니라 솔직하게 가능/불가능을 말한다. 추정을 약속으로 포장하지 않는다. | Ch 2, 3, 10 |
| **협업(Collaboration)** | 혼자 일하는 것을 선호하더라도 팀과 비즈니스의 목표를 위해 적극적으로 소통한다. | Ch 7, 12, 13 |
| **지속 학습(Continuous Learning)** | 주 40시간은 고용주를 위한 시간이고, 그 외에 자기 개발 시간을 따로 투자한다. | Ch 1, 6, 14 |

## 프로의 원칙 종합

각 챕터에서 도출된 프로의 핵심 원칙들을 종합한 테이블이다. 개별 챕터의 "프로의 원칙" 섹션에서 더 상세한 설명을 확인할 수 있다.

| 챕터 | 원칙 | 한 줄 요약 |
|------|------|-----------|
| Ch 1 | 해를 끼치지 마라 | QA가 아무것도 찾지 못하게 하라 |
| Ch 2 | "아니오"라고 말할 줄 알아야 한다 | 대립은 프로의 의무다 |
| Ch 3 | 약속의 언어를 사용하라 | "할 것이다(I will)"와 "시도해 보겠다(I'll try)"를 구분하라 |
| Ch 4 | 준비된 상태에서 코딩하라 | 피곤하거나 불안할 때는 코딩하지 마라 |
| Ch 5 | TDD는 프로의 규율이다 | TDD를 선택이 아닌 기본 실천으로 삼아라 |
| Ch 6 | 자기 시간에 연습하라 | 카타와 와사로 기술을 유지하라 |
| Ch 7 | 인수 테스트로 소통하라 | "완료"의 정의를 자동화된 테스트로 합의하라 |
| Ch 8 | QA는 아무것도 찾지 못해야 한다 | 개발자가 테스트 책임을 져라 |
| Ch 9 | 회의를 관리하라 | 불필요한 회의는 거절하고, 참여한 회의에는 집중하라 |
| Ch 10 | 추정은 약속이 아니다 | 추정에 불확실성 범위를 포함하라 |
| Ch 11 | 압박에서도 규율을 유지하라 | 위기 상황의 행동이 진짜 규율을 드러낸다 |
| Ch 12 | 프로그래밍은 혼자 하는 일이 아니다 | 비즈니스를 이해하고 팀과 협업하라 |
| Ch 13 | 프로젝트에 팀을 할당하라 | 팀에 프로젝트를 할당하지 말고, 응집된 팀에 프로젝트를 맡겨라 |
| Ch 14 | 장인정신은 전파된다 | 멘토링과 도제 모델로 다음 세대를 키워라 |

# Chapter 8: Testing Strategies (테스트 전략)

## 핵심 질문

프로 개발팀은 소프트웨어의 품질을 보장하기 위해 어떤 수준의 테스트를 어떤 비율로 구성해야 하며, QA의 진정한 역할은 무엇인가?

---

## 1. QA는 아무것도 찾지 못해야 한다

### 1.1 원칙의 재확인

Chapter 1에서 처음 등장한 이 원칙이 이 챕터에서 더 구체적인 맥락을 갖는다. Uncle Bob은 단호하게 말한다 — 개발팀이 QA(*Quality Assurance — 소프트웨어 품질 보증. 개발팀과 별도로 소프트웨어의 결함을 찾고 품질을 검증하는 역할 또는 팀을 말한다.*)에게 코드를 넘길 때, **QA가 결함을 하나도 찾지 못하는 것**이 올바른 상태이다.

이것이 비현실적으로 들릴 수 있다. 하지만 여기서 핵심은 **목표(goal)**와 **태도(attitude)**의 문제이다. QA가 결함을 찾는 것을 "프로세스의 정상적인 일부"로 여기는 팀과, QA가 결함을 찾을 때마다 "개발팀의 실패"로 인식하는 팀은 완전히 다른 품질의 소프트웨어를 만들어낸다.

> **핵심 통찰**: QA가 결함을 찾지 못한다는 것이 결함이 0이라는 뜻은 아니다. 그것은 개발팀이 **자신의 코드에 대해 최대한의 확신을 가지고 넘겼다**는 뜻이다. 테스트를 충분히 작성했고, 수동으로도 검증했으며, 알려진 결함이 없는 상태에서 넘겼다는 뜻이다.

### 1.2 QA가 결함을 찾으면 어떻게 해야 하는가

그럼에도 QA가 결함을 찾는 상황은 현실적으로 발생한다. 이때 프로 개발자의 반응은 다음과 같아야 한다:

1. **놀라라** — QA가 결함을 찾았다는 사실에 진심으로 당황해야 한다. "그럴 수 있지"가 아니라 "어떻게 이걸 놓쳤지?"라고 반응해야 한다.
2. **원인을 분석하라** — 단순히 버그를 수정하는 것이 아니라, 왜 이 버그가 테스트에서 걸리지 않았는지를 분석한다.
3. **프로세스를 개선하라** — 같은 유형의 버그가 다시 빠져나가지 않도록 테스트 전략을 보강한다.

> **Uncle Bob의 경험**: Uncle Bob은 한 프로젝트에서 QA 팀이 매 릴리스마다 수십 개의 버그를 찾아내는 것이 "정상"으로 간주되던 상황을 회상한다. 개발팀은 QA를 "안전망"으로 사용하고 있었다. 결과적으로 개발팀은 테스트에 투자하지 않았고, QA는 점점 더 많은 버그를 찾았으며, 릴리스 주기는 계속 늘어났다. 악순환이었다. 개발팀이 "QA는 아무것도 찾지 못해야 한다"는 목표를 세우고 테스트에 진지하게 투자하기 시작하자, 6개월 후 QA가 찾는 결함 수가 극적으로 줄어들었다.

## 2. QA의 진정한 역할

### 2.1 QA는 버그를 잡는 팀이 아니다

많은 조직에서 QA의 역할을 "개발팀이 만든 결함을 찾아내는 것"으로 정의한다. Uncle Bob은 이 정의가 근본적으로 잘못되었다고 주장한다. QA의 진정한 역할은 버그를 잡는 것이 아니라, **소프트웨어의 품질을 정의하고 특성화하는 것**이다.

Uncle Bob은 QA의 역할을 크게 두 가지로 나눈다:

### 2.2 명세 작성자(Specifier)로서의 QA

QA의 첫 번째 역할은 비즈니스와 함께 **인수 테스트 명세를 작성하는 것**이다. 이것은 Chapter 7(Acceptance Testing)에서 다룬 주제와 직접 연결된다.

QA는 비즈니스 이해관계자와 협력하여 시스템이 **어떻게 동작해야 하는지**를 자동화된 테스트의 형태로 정의한다. 이 테스트들은 개발이 시작되기 전에 작성되며, 개발팀이 "완료"의 기준으로 사용한다.

| 전통적 QA 역할 | Uncle Bob이 제안하는 QA 역할 |
|---------------|---------------------------|
| 개발 완료 후 테스트 | 개발 시작 전 명세 작성 |
| 결함 발견에 집중 | 요구사항 명확화에 집중 |
| 개발팀과 대립적 관계 | 개발팀과 협력적 관계 |
| "이거 안 돼요" 보고 | "이렇게 동작해야 해요" 정의 |

### 2.3 특성화 담당자(Characterizer)로서의 QA

QA의 두 번째 역할은 시스템의 **실제 행동을 특성화**하여 비즈니스에 보고하는 것이다. 이것은 명세 작성과는 다른 활동이다.

명세 작성이 "시스템이 어떻게 동작**해야 하는가**"를 정의하는 것이라면, 특성화는 "시스템이 실제로 어떻게 동작**하는가**"를 파악하는 것이다. QA는 탐색적 테스트(*Exploratory Testing — 사전에 정의된 스크립트 없이 테스터의 경험과 직관에 기반하여 시스템을 탐색하며 테스트하는 기법. 이 챕터의 테스트 자동화 피라미드 최상단에 위치한다.*)와 같은 기법을 사용하여 시스템의 실제 행동을 탐색하고, 그 결과를 비즈니스에 전달한다.

이 역할은 특히 다음 상황에서 중요하다:

- 레거시 시스템의 실제 행동을 문서화할 때
- 명세에 정의되지 않은 엣지 케이스의 동작을 파악할 때
- 비기능적 특성(성능, 사용성 등)을 평가할 때

## 3. 테스트 자동화 피라미드

### 3.1 피라미드의 개요

Uncle Bob은 전문적인 개발 조직이 갖추어야 할 테스트 전략을 **피라미드 형태**로 제시한다. 이 피라미드는 Mike Cohn이 처음 제안한 개념을 Uncle Bob이 자신만의 관점으로 재구성한 것이다.

피라미드의 핵심 원칙은 간단하다:

- 아래층으로 갈수록 **테스트 수가 많고**, **실행 속도가 빠르며**, **범위가 좁다**
- 위층으로 갈수록 **테스트 수가 적고**, **실행 속도가 느리며**, **범위가 넓다**
- 각 층은 **서로 다른 것을 테스트**한다 — 층 간에 테스트를 중복하지 않는다

```
        /\
       /  \     탐색적 테스트 (수동, 창의적)
      /    \
     /------\   시스템 테스트 (~10%)
    /--------\  통합 테스트 (~20%)
   /----------\ 컴포넌트 테스트 (~50%)
  /------------\단위 테스트 (~100%)
 /--------------\
```

### 3.2 1층: 단위 테스트(Unit Tests)

단위 테스트는 피라미드의 **기반(base)**이다. 전체 테스트 전략의 토대를 형성하며, 가장 많은 수의 테스트가 이 층에 존재한다.

| 속성 | 설명 |
|------|------|
| **작성자** | 개발자 (프로그래머) |
| **커버리지 목표** | 해피 패스(*Happy Path — 모든 입력이 정상이고 예외 상황이 발생하지 않는 이상적인 실행 경로. 오류 경로(error path)나 엣지 케이스와 대비되는 개념이다.*) 기준 ~100% |
| **실행 속도** | 매우 빠름 (수천 개가 수 분 내에 실행) |
| **범위** | 개별 함수, 메서드, 클래스 |
| **목적** | 코드가 개발자의 의도대로 동작하는지 저수준에서 검증 |

단위 테스트의 특징:

- **TDD(Test-Driven Development)**로 작성된다 — Chapter 5에서 다룬 것처럼 테스트를 먼저 작성하고, 그 테스트를 통과시키는 코드를 작성한다
- **시스템의 나머지 부분으로부터 격리**되어 실행된다 — 외부 의존성은 목(mock)이나 스텁(stub)으로 대체한다
- **수천 개**가 존재하며, 빌드 시마다 실행된다
- **행복한 경로(happy path)의 거의 100%**를 커버해야 한다. Uncle Bob은 100%가 비현실적이지 않다고 주장한다 — TDD를 올바르게 실천하면 자연스럽게 높은 커버리지에 도달하기 때문이다

> **핵심 통찰**: 단위 테스트는 시스템의 명세(specification)이다. 누군가가 "이 클래스가 뭘 하는 거죠?"라고 물으면, 단위 테스트가 그 답을 제공해야 한다. 단위 테스트는 문서이자 명세이자 검증 도구이다.

### 3.3 2층: 컴포넌트 테스트(Component Tests)

컴포넌트 테스트는 피라미드의 두 번째 층이다. 시스템의 개별 컴포넌트를 비즈니스 규칙 수준에서 테스트한다.

| 속성 | 설명 |
|------|------|
| **작성자** | QA와 개발자가 협력하여 작성 |
| **커버리지 목표** | 시스템의 ~50% |
| **실행 속도** | 단위 테스트보다 느림 |
| **범위** | 개별 컴포넌트 (하나의 비즈니스 규칙을 구현하는 묶음) |
| **목적** | 비즈니스 규칙이 올바르게 구현되었는지 검증 |

컴포넌트 테스트의 특징:

- **인수 테스트의 일종**이다 — Chapter 7에서 다룬 인수 테스트 중 상당수가 이 층에 해당한다
- 하나의 **컴포넌트를 격리하여** 테스트한다 — 시스템의 다른 컴포넌트는 목(mock)으로 대체한다
- 비즈니스 규칙의 **입력과 출력**을 검증한다 — "이 입력이 들어오면 이 출력이 나와야 한다"는 수준의 테스트
- FitNesse(*FitNesse — Uncle Bob이 만든 인수 테스트 프레임워크. 위키 기반으로 비즈니스 이해관계자가 읽을 수 있는 형태의 테스트를 작성할 수 있다.*), Cucumber, SpecFlow 같은 도구로 비즈니스 이해관계자가 읽을 수 있는 형태로 작성될 수 있다

### 대화: 컴포넌트 테스트의 범위

> **QA**: "주문 취소 기능을 테스트하려면 결제 시스템과 재고 시스템도 연동해야 하지 않나요?"

> **개발자**: "아니요, 컴포넌트 테스트에서는 주문 취소 컴포넌트만 격리해서 테스트합니다. 결제 시스템과 재고 시스템은 목(mock)으로 대체해요. '결제 환불이 성공했다고 가정하고, 주문 상태가 올바르게 변경되는가'를 테스트하는 겁니다."

> **QA**: "그러면 결제 시스템과의 실제 연동은 어디서 테스트하나요?"

> **개발자**: "그건 한 단계 위인 통합 테스트에서 합니다. 각 층은 서로 다른 것을 테스트해요."

### 3.4 3층: 통합 테스트(Integration Tests)

통합 테스트는 피라미드의 세 번째 층이다. **여러 컴포넌트가 함께 동작하는 방식**을 테스트한다.

| 속성 | 설명 |
|------|------|
| **작성자** | 시스템 아키텍트 또는 기술 리드 |
| **커버리지 목표** | 시스템의 ~20% |
| **실행 속도** | 컴포넌트 테스트보다 느림 |
| **범위** | 컴포넌트 간의 상호작용 |
| **목적** | 컴포넌트들이 올바르게 조립(choreography)되는지 검증 |

> **핵심 통찰**: 통합 테스트는 **비즈니스 규칙을 테스트하지 않는다**. 비즈니스 규칙은 이미 단위 테스트와 컴포넌트 테스트에서 검증되었다. 통합 테스트가 검증하는 것은 **"이 컴포넌트들이 서로 제대로 대화하는가?"**이다.

Uncle Bob은 이것을 **안무 테스트(choreography test)**라고 부른다. 무용가 한 명 한 명의 동작은 이미 개별 연습에서 검증되었고(단위/컴포넌트 테스트), 이제는 여러 무용가가 함께 **안무(choreography)에 맞춰 움직이는지**를 확인하는 것이다.

통합 테스트의 예시:

- 주문 컴포넌트가 결제 컴포넌트에 올바른 메시지를 보내는가?
- 인증 서비스가 인가 서비스와 올바르게 연동되는가?
- 알림 서비스가 사용자 서비스에서 올바른 이메일 주소를 가져오는가?

### 3.5 4층: 시스템 테스트(System Tests)

시스템 테스트는 피라미드의 네 번째 층이다. **전체 시스템을 끝에서 끝까지(end-to-end)** 테스트한다.

| 속성 | 설명 |
|------|------|
| **작성자** | 시스템 아키텍트, 기술 리드 |
| **커버리지 목표** | 시스템의 ~10% |
| **실행 속도** | 느림 |
| **범위** | 전체 시스템 (UI에서 DB까지) |
| **목적** | 시스템이 전체적으로 올바르게 동작하는지 검증 |

시스템 테스트의 특징:

- **자동화된** 테스트이다 — 수동이 아니다
- **성능 테스트**와 **처리량(throughput) 테스트**가 이 층에 포함된다
- 비즈니스 규칙이 아니라 **시스템 전체의 구성(construction)**이 올바른지 확인한다
- 실행 시간이 길기 때문에 매 커밋마다 실행하기 어려울 수 있다 — 주기적으로(예: 야간 빌드) 실행한다

### 3.6 꼭대기: 탐색적 테스트(Exploratory Tests)

탐색적 테스트는 피라미드의 **꼭대기**이며, 피라미드에서 유일하게 **자동화되지 않는** 테스트이다.

| 속성 | 설명 |
|------|------|
| **수행자** | 인간 (QA, 개발자, 또는 누구든) |
| **커버리지 목표** | 정의할 수 없음 |
| **실행 속도** | 해당 없음 (수동) |
| **범위** | 시스템 전체 |
| **목적** | 예상하지 못한 문제를 인간의 창의성으로 발견 |

탐색적 테스트의 핵심 특징:

- **스크립트가 없다** — 사전에 정의된 테스트 케이스가 아니다
- **자동화하지 않는다** — 자동화의 목적이 아니라 인간의 직관과 창의성을 활용하는 것이 목적이다
- **목표는 "시스템을 깨는 것"이다** — 자동화된 테스트가 놓치는 것들을 인간의 눈으로 찾는다
- **재현 가능한 결함을 발견하면 즉시 자동화된 테스트로 전환한다** — 탐색적 테스트에서 발견된 문제는 적절한 피라미드 층의 자동화 테스트로 추가한다

> **Uncle Bob의 경험**: Uncle Bob은 탐색적 테스트의 가치를 높이 평가한다. 아무리 완벽한 자동화 테스트 스위트를 갖추더라도, 인간이 시스템을 "가지고 놀면서" 찾아내는 문제들이 있다. 예를 들어 "이 버튼을 연타하면 어떻게 되지?", "로그인한 상태에서 URL을 직접 수정하면?", "네트워크를 끊은 상태에서 저장을 누르면?"과 같은 시나리오는 인간의 창의적 호기심에서 비롯된다.

---

## 4. 테스트 층간의 관계

### 4.1 중복을 피하라

피라미드의 각 층은 **서로 다른 것을 테스트**해야 한다. 같은 비즈니스 규칙을 단위 테스트에서도, 컴포넌트 테스트에서도, 통합 테스트에서도 반복 검증하는 것은 낭비이다.

| 층 | 검증하는 것 | 검증하지 않는 것 |
|----|------------|-----------------|
| 단위 테스트 | 개별 코드 단위의 로직 | 컴포넌트 간 상호작용 |
| 컴포넌트 테스트 | 비즈니스 규칙 | 컴포넌트 간 연동 |
| 통합 테스트 | 컴포넌트 간 안무(choreography) | 비즈니스 규칙 (이미 검증됨) |
| 시스템 테스트 | 전체 시스템 구성의 정확성 | 개별 비즈니스 규칙 (이미 검증됨) |
| 탐색적 테스트 | 예상치 못한 문제 | 이미 자동화된 시나리오 |

### 4.2 비율의 의미

피라미드의 비율(100%, 50%, 20%, 10%)은 엄격한 규칙이 아니라 **방향성**이다. 핵심 메시지는 이것이다:

- **단위 테스트가 가장 많아야 한다** — 가장 빠르고, 가장 저렴하며, 가장 넓은 커버리지를 제공한다
- **위로 갈수록 테스트 수가 줄어야 한다** — 상위 테스트는 느리고, 비용이 높으며, 유지보수가 어렵다
- **역피라미드(아이스크림 콘)는 안티패턴이다** — 수동 테스트나 E2E(*End-to-End — 시스템의 시작점(예: UI)에서 끝점(예: 데이터베이스)까지 전체 흐름을 테스트하는 방식.*) 테스트가 많고 단위 테스트가 적은 구조는 느리고, 불안정하며, 비용이 높다

---

## 5. 지속적 통합(Continuous Integration)

### 5.1 모든 자동화 테스트는 매 커밋마다 실행되어야 한다

Uncle Bob은 테스트 자동화 피라미드의 효과를 극대화하기 위한 필수 조건으로 **지속적 통합(CI)**을 제시한다. 모든 자동화된 테스트가 매 커밋마다 실행되어야 한다.

CI의 핵심 규칙:

- 코드를 커밋하면 **모든 단위 테스트와 컴포넌트 테스트가 자동으로 실행**된다
- 통합 테스트와 시스템 테스트는 빈도를 조절할 수 있지만, 최소한 **하루에 한 번**은 실행되어야 한다
- **실패한 테스트는 즉시 수정**해야 한다 — "나중에 고치겠다"는 프로답지 못한 태도이다
- 테스트가 실패하면 **다른 모든 작업을 멈추고** 테스트를 통과시키는 것이 최우선이다

### 5.2 깨진 빌드의 비용

테스트가 실패한 채로 방치되면, 팀 전체의 테스트에 대한 신뢰가 무너진다:

1. 테스트가 가끔 실패한다 → "그냥 그런 테스트야"라고 무시하기 시작한다
2. 무시하는 습관이 퍼진다 → 진짜 중요한 실패도 놓치기 시작한다
3. 테스트를 아무도 믿지 않는다 → 테스트를 작성하는 동기가 사라진다
4. 테스트를 작성하지 않는다 → 버그가 증가하고 QA 의존도가 높아진다

> **핵심 통찰**: 테스트 자동화 피라미드는 **지속적 통합 없이는 무의미**하다. 아무리 잘 구성된 테스트 스위트도 정기적으로 실행되지 않으면 빠르게 부패한다. 테스트는 살아있는 코드와 함께 호흡해야 한다.

## 6. 전체 테스트 전략의 조감도

Uncle Bob이 제안하는 전체 테스트 전략을 하나의 테이블로 요약하면 다음과 같다:

| 층 | 커버리지 | 작성자 | 실행 빈도 | 핵심 목적 | 도구 예시 |
|----|---------|--------|-----------|-----------|-----------|
| **단위 테스트** | ~100% | 개발자 | 매 커밋 | 코드 단위의 의도 검증 | JUnit, NUnit, pytest |
| **컴포넌트 테스트** | ~50% | QA + 개발자 | 매 커밋 | 비즈니스 규칙 검증 | FitNesse, Cucumber |
| **통합 테스트** | ~20% | 아키텍트/리드 | 주기적 | 컴포넌트 간 안무 검증 | 통합 테스트 프레임워크 |
| **시스템 테스트** | ~10% | 아키텍트/리드 | 야간/주기적 | 전체 시스템 검증 | Selenium, Cypress |
| **탐색적 테스트** | N/A | 인간 | 수시 | 예상치 못한 문제 발견 | 인간의 창의성 |

> **핵심 통찰**: "목표는 종합적이고 자동화된 테스트 스위트를 갖추어, 시스템이 명세대로 동작한다는 거의 100%에 가까운 확신을 제공하는 것이다."

## 실전 시나리오

### 시나리오 1: 테스트 전략 없는 팀

**상황**: 팀에 합류했는데, 자동화 테스트가 거의 없다. QA 팀이 매 릴리스마다 수동으로 전체 시스템을 테스트하며, 릴리스 주기가 3개월이다.

**비프로 대응**:
> "기존 팀의 방식이니 따르겠습니다. 제 담당 기능이 동작하는 것만 수동으로 확인하고 QA에 넘기겠습니다."

**프로 대응**:
> "지금 당장 전체 피라미드를 구축할 수는 없지만, 오늘부터 제가 작성하는 모든 새 코드에는 단위 테스트를 포함하겠습니다. 그리고 팀에 테스트 전략 수립을 제안하겠습니다. 아래에서 위로, 단위 테스트부터 시작하는 것이 현실적입니다."

**원칙**: 피라미드는 하루 만에 완성되지 않는다. 하지만 "테스트가 없다"는 현실이 "테스트를 작성하지 않아도 된다"는 허가가 되어서는 안 된다. 오늘부터 시작하라.

### 시나리오 2: 역피라미드(아이스크림 콘)

**상황**: 팀의 테스트가 대부분 E2E 테스트이다. 단위 테스트는 거의 없고, Selenium 기반의 UI 테스트가 수백 개 있다. CI 실행 시간이 2시간이며, 테스트의 30%가 간헐적으로 실패한다(flaky test).

**비프로 대응**:
> "테스트가 많으니까 괜찮지 않나요? 실패하는 건 다시 돌리면 통과하니까요."

**프로 대응**:
> "이건 역피라미드입니다. E2E 테스트에 의존하고 있어서 느리고, 불안정하고, 유지보수 비용이 높습니다. 간헐적으로 실패하는 테스트는 테스트에 대한 신뢰를 파괴합니다. 단위 테스트 커버리지를 높이면서 불안정한 E2E 테스트를 점진적으로 교체하는 계획을 세워야 합니다."

**원칙**: 테스트의 수가 아니라 **구조와 비율**이 중요하다. 피라미드가 뒤집히면, 느리고, 비싸고, 불안정한 테스트에 시간을 빼앗기게 된다.

---

## 프로의 원칙

| 원칙 | 설명 |
|------|------|
| **QA는 아무것도 찾지 못해야 한다** | 개발팀이 QA에 넘기기 전에 최대한의 확신을 갖는 것을 목표로 한다 |
| **QA는 명세 작성자이다** | QA의 역할은 버그를 잡는 것이 아니라 비즈니스와 함께 인수 테스트 명세를 작성하는 것이다 |
| **피라미드를 따르라** | 단위 테스트가 기반이고, 위로 갈수록 테스트 수가 줄어드는 피라미드 구조를 유지한다 |
| **각 층은 다른 것을 테스트한다** | 테스트를 층 간에 중복하지 않는다 — 단위 테스트는 로직을, 통합 테스트는 안무를 검증한다 |
| **탐색적 테스트는 인간의 영역이다** | 자동화로 대체할 수 없는 인간의 창의적 테스트를 수행한다 |
| **지속적 통합은 필수다** | 자동화 테스트는 매 커밋마다 실행되어야 의미가 있다 |

---

## 요약

- **QA가 아무것도 찾지 못하는 것이 목표다.** QA가 결함을 찾는 것을 "정상"이 아닌 "개발팀의 실패"로 인식해야 한다.
- **QA의 진정한 역할은 명세 작성과 시스템 특성화다.** QA는 버그를 잡는 팀이 아니라, 비즈니스와 협력하여 인수 테스트를 정의하고 시스템의 실제 행동을 특성화하는 팀이다.
- **테스트 자동화 피라미드를 따르라.** 단위 테스트(~100%), 컴포넌트 테스트(~50%), 통합 테스트(~20%), 시스템 테스트(~10%), 탐색적 테스트(수동)로 구성한다.
- **각 층은 서로 다른 것을 테스트한다.** 비즈니스 규칙은 아래쪽에서 검증하고, 위쪽에서는 컴포넌트 간 연동과 전체 시스템 구성을 검증한다.
- **역피라미드(아이스크림 콘)는 안티패턴이다.** E2E 테스트가 많고 단위 테스트가 적은 구조는 느리고, 비싸고, 불안정하다.
- **탐색적 테스트는 자동화의 대안이 아니라 보완이다.** 인간의 창의성으로만 발견할 수 있는 문제가 있다.
- **지속적 통합 없이는 테스트 전략이 무의미하다.** 모든 자동화 테스트는 정기적으로 실행되어야 한다.

## 자기 점검

- [ ] 우리 팀은 QA가 결함을 찾았을 때 "당연한 과정"이 아닌 "개발팀의 실패"로 인식하는가?
- [ ] QA가 인수 테스트 명세 작성에 참여하고 있는가, 아니면 개발 완료 후 테스트만 수행하고 있는가?
- [ ] 우리의 테스트 구조가 피라미드 형태인가, 역피라미드(아이스크림 콘) 형태인가?
- [ ] 단위 테스트가 해피 패스의 거의 100%를 커버하고 있는가?
- [ ] 통합 테스트에서 비즈니스 규칙을 중복 검증하고 있지는 않은가?
- [ ] 모든 자동화 테스트가 CI에서 정기적으로 실행되고 있는가?
- [ ] 간헐적으로 실패하는(flaky) 테스트를 방치하고 있지는 않은가?
- [ ] 정기적으로 탐색적 테스트를 수행하고 있는가?

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: "QA는 아무것도 찾지 못해야 한다"는 원칙이 처음 등장한 곳이다. 이 챕터에서는 그 원칙을 실현하기 위한 구체적인 테스트 전략을 제시한다.
- **Chapter 5 (Test Driven Development)**: 테스트 자동화 피라미드의 기반인 단위 테스트를 작성하는 규율이다. TDD 없이는 피라미드의 기반이 부실해진다.
- **Chapter 7 (Acceptance Testing)**: 피라미드의 컴포넌트 테스트 층과 직접 연결된다. 인수 테스트의 정의와 작성 방법을 다룬 챕터이다.
- **Chapter 9 (Time Management)**: 테스트 작성에 시간을 투자하는 것이 장기적으로 시간을 절약한다는 관점에서 연결된다. 테스트 없이 빠르게 가는 것은 착각이다.
- **Chapter 10 (Estimation)**: 테스트 작성 시간을 추정에 포함시켜야 한다. 테스트 없는 추정은 불완전한 추정이다.
- **Chapter 11 (Pressure)**: 압박 속에서 테스트를 생략하고 싶은 유혹을 다룬다. 프로는 위기 상황에서도 테스트 전략을 유지한다.

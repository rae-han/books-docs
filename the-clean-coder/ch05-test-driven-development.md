# Chapter 5: Test Driven Development (테스트 주도 개발)

## 핵심 질문

TDD는 단지 유용한 기법 중 하나인가, 아니면 프로 개발자라면 마땅히 따라야 할 **규율**인가? TDD의 세 가지 법칙은 무엇이며, 이를 따를 때 어떤 이점이 발생하고, TDD가 적용되지 않는 상황은 어떤 것인가?

---

## 1. Uncle Bob의 TDD 여정

### 1.1 회의에서 확신으로

Uncle Bob은 TDD를 처음부터 받아들인 것이 아니다. 그의 여정은 **회의(skepticism)**에서 시작하여 **경험을 통한 확신(conviction)**으로 끝났다.

> **Uncle Bob의 경험**: 1999년경 Kent Beck으로부터 TDD에 대해 처음 들었을 때, Robert Martin은 회의적이었다. "프로덕션 코드를 작성하기 전에 테스트를 먼저 작성하라고? 그건 시간 낭비 아닌가?" 하지만 호기심에 직접 시도해 보기로 했다. 처음 며칠은 어색하고 느렸다. 그러나 일주일이 지나자, 무언가 달라지기 시작했다. 코드의 구조가 자연스럽게 깔끔해졌고, 디버거를 켜는 횟수가 급격히 줄었다. 한 달이 지나자, TDD 없이 코딩하는 것이 **안전벨트 없이 운전하는 것**처럼 불안하게 느껴졌다.

이 경험이 Uncle Bob을 TDD의 강력한 옹호자로 만들었다. 그는 이후 TDD를 단순한 기법이 아니라 **프로의 규율**로 주장하게 된다.

### 1.2 왜 이렇게 강하게 주장하는가

Uncle Bob이 TDD에 대해 이토록 강한 입장을 취하는 이유는 **경험적 증거**에 기반한다. 그는 TDD를 적용한 후:

- 버그 리포트가 극적으로 줄었다
- 리팩토링에 대한 두려움이 사라졌다
- 코드 설계가 자연스럽게 개선되었다
- 디버깅에 소비하는 시간이 현저히 줄었다

이러한 경험이 축적되면서, Uncle Bob은 TDD를 "좋은 기법" 수준이 아니라 "프로라면 마땅히 따라야 할 규율"로 승격시킨 것이다.

## 2. TDD의 세 가지 법칙

### 2.1 세 가지 법칙의 정의

TDD의 핵심은 세 가지 간단한 법칙으로 표현된다:

| 법칙 | 내용 |
|------|------|
| **제1법칙** | 실패하는 단위 테스트를 작성하기 전에는 프로덕션 코드를 작성할 수 없다. |
| **제2법칙** | 실패하기에 충분한 정도 이상으로 단위 테스트를 작성할 수 없다. (컴파일 실패도 실패에 해당한다.) |
| **제3법칙** | 현재 실패하는 단위 테스트를 통과시키기에 충분한 정도 이상으로 프로덕션 코드를 작성할 수 없다. |

### 2.2 30초 사이클

이 세 가지 법칙은 개발자를 **약 30초 단위의 극히 짧은 사이클**에 가둔다:

```
[Red]    → 실패하는 테스트를 한두 줄 작성한다 (몇 초)
[Green]  → 그 테스트를 통과시키는 프로덕션 코드를 작성한다 (몇 초)
[Refactor] → 코드를 정리한다 (몇 초)
[반복]
```

이 사이클의 핵심은 **극단적으로 짧은 피드백 루프**에 있다. 30초마다 코드가 동작하는지 확인하므로, 무언가 잘못되면 30초 이내의 변경에서 원인을 찾을 수 있다.

> **핵심 통찰**: 세 가지 법칙을 충실히 따르면, 프로덕션 코드의 모든 줄에는 그것을 테스트하는 테스트가 존재하게 된다. 이것은 코드베이스 전체에 대한 **안전망(safety net)**을 구축하는 것이다.

### 2.3 법칙을 따르는 것이 왜 어려운가

이 법칙들은 단순하지만, 실천은 쉽지 않다:

1. **습관의 변경**: 대부분의 개발자는 "먼저 코드를 작성하고 나중에 테스트한다"는 습관이 몸에 배어 있다. 이 순서를 뒤집는 것은 사고방식의 전환을 요구한다.
2. **인내심**: 테스트를 먼저 작성하면 처음에는 느리게 느껴진다. 하지만 이 "느림"은 나중에 디버깅과 리팩토링에서 시간을 절약함으로써 상쇄된다.
3. **설계 사고**: 테스트를 먼저 작성하려면 코드의 인터페이스를 먼저 생각해야 한다. 이것은 설계를 의식적으로 하게 만들지만, 처음에는 부담스럽다.

## 3. TDD의 이점

### 3.1 확신(Certainty)과 용기(Courage)

TDD의 가장 중요한 이점은 **변경에 대한 용기**를 제공한다는 것이다.

포괄적인 테스트 스위트(*Test Suite — 특정 소프트웨어에 대한 테스트 케이스의 집합. 전체 테스트 스위트를 실행하면 시스템의 핵심 기능이 모두 정상적으로 동작하는지 확인할 수 있다.*)가 있으면, 코드를 변경한 후 테스트를 실행하여 아무것도 깨지지 않았음을 확인할 수 있다. 이 확인이 **리팩토링의 용기**를 준다.

테스트가 없는 코드베이스에서는 다음과 같은 악순환이 발생한다:

```
코드 변경이 무섭다
→ 변경을 피한다
→ 코드가 점점 부패한다
→ 코드 변경이 더 무서워진다
→ (반복)
```

TDD를 적용하면 이 악순환이 선순환으로 바뀐다:

```
테스트가 있으니 변경이 안전하다
→ 자주 리팩토링한다
→ 코드가 점점 깨끗해진다
→ 변경이 더 쉬워진다
→ (반복)
```

> **핵심 통찰**: 테스트 스위트는 코드베이스에 대한 **보험**이다. 보험이 있으면 위험을 감수할 수 있고, 위험을 감수할 수 있으면 코드를 지속적으로 개선할 수 있다. TDD가 주는 가장 큰 선물은 **두려움 없이 코드를 변경할 수 있는 용기**다.

### 3.2 결함 주입률(Defect Injection Rate) 감소

여러 연구와 현장 보고가 TDD를 적용하면 버그 발생률이 유의미하게 감소한다는 것을 보여준다. Uncle Bob은 구체적 수치를 제시한다:

| 지표 | TDD 미적용 | TDD 적용 |
|------|-----------|---------|
| 결함 밀도(*Defect Density — 코드의 크기(보통 KLOC, 즉 1,000줄 단위) 대비 발견된 결함의 수. 결함 밀도가 낮을수록 코드 품질이 높다.*) | 높음 | 2~10배 감소 |
| 디버깅 시간 | 전체 개발 시간의 30~50% | 현저히 감소 |
| 릴리스 후 버그 | 빈번 | 드묾 |

왜 이런 결과가 나오는가? 세 가지 법칙에 따라 코딩하면, **모든 프로덕션 코드가 작성되는 순간에 테스트된다.** 버그가 주입되는 순간과 발견되는 순간 사이의 시간 간격이 극도로 짧기 때문에(30초 이내), 원인 파악과 수정이 거의 즉시 이루어진다.

반면 테스트 없이 코딩하면, 버그는 작성 후 며칠 또는 몇 주 후에 발견되고, 그때쯤이면 맥락이 기억에서 사라져 있어 원인 파악에 막대한 시간이 소요된다.

### 3.3 문서화(Documentation)

테스트 코드는 **실행 가능한 문서(executable documentation)**다.

전통적인 문서(설계 문서, API 문서 등)는 코드가 변경되면 쉽게 낡아진다. 하지만 테스트는 코드와 함께 실행되므로, 코드가 변경되면 테스트도 함께 업데이트되어야 한다. 따라서 테스트는 **항상 최신 상태의 문서**다.

잘 작성된 단위 테스트는 다음과 같은 정보를 제공한다:

- **이 함수를 어떻게 호출하는가?** — 테스트가 함수의 사용법을 보여준다
- **어떤 입력에 어떤 출력이 나오는가?** — 테스트가 기대 동작을 명시한다
- **경계 조건은 무엇인가?** — 에지 케이스에 대한 테스트가 제약 조건을 보여준다
- **예외 상황에서 어떻게 동작하는가?** — 예외 테스트가 에러 처리 방식을 보여준다

```python
# 문서 대신 테스트가 말해준다
class TestUserRegistration:
    def test_valid_email_creates_user(self):
        """유효한 이메일로 사용자를 생성할 수 있다"""
        user = register_user("alice@example.com", "SecureP@ss1")
        assert user.email == "alice@example.com"
        assert user.is_active is True

    def test_duplicate_email_raises_error(self):
        """중복 이메일은 에러를 발생시킨다"""
        register_user("bob@example.com", "SecureP@ss1")
        with pytest.raises(DuplicateEmailError):
            register_user("bob@example.com", "AnotherP@ss2")

    def test_weak_password_rejected(self):
        """약한 비밀번호는 거부된다"""
        with pytest.raises(WeakPasswordError):
            register_user("carol@example.com", "123")
```

이 테스트를 읽으면 API 문서 없이도 `register_user` 함수의 동작을 정확히 이해할 수 있다.

### 3.4 설계(Design)

TDD는 **설계를 강제**한다. 테스트를 먼저 작성하려면 코드의 인터페이스를 먼저 결정해야 하기 때문이다.

더 중요한 것은, **테스트하기 어려운 코드는 나쁜 설계의 신호**라는 점이다.

| 테스트가 어려운 코드의 특징 | 설계 문제 |
|---------------------------|----------|
| 다른 모듈과 강하게 결합(*Coupling — 두 모듈 사이의 의존 관계의 정도. 결합도가 높으면 한 모듈의 변경이 다른 모듈에 파급 효과를 일으킨다. 좋은 설계는 낮은 결합도와 높은 응집도를 추구한다.*)되어 있다 | 높은 결합도(coupling) |
| 전역 상태에 의존한다 | 숨겨진 의존성(hidden dependency) |
| 하나의 함수가 너무 많은 일을 한다 | 단일 책임 원칙(SRP) 위반 |
| 생성자에서 복잡한 초기화를 수행한다 | 의존성 주입(DI)의 부재 |

TDD를 따르면, 테스트하기 쉬운 코드를 작성하도록 자연스럽게 유도되고, 테스트하기 쉬운 코드는 곧 **잘 설계된 코드**다.

> **핵심 통찰**: TDD는 테스트 기법인 동시에 **설계 기법**이다. "이 코드를 어떻게 테스트할 것인가?"라는 질문은 "이 코드를 어떻게 설계할 것인가?"라는 질문과 본질적으로 같다. 테스트가 어렵다면, 그것은 테스트의 문제가 아니라 **설계의 문제**다.

## 4. 프로의 선택(The Professional Option)

### 4.1 복식부기와의 비유

Uncle Bob은 TDD를 **회계사의 복식부기(*Double-Entry Bookkeeping — 모든 회계 거래를 차변(debit)과 대변(credit) 두 곳에 동시에 기록하는 회계 방식. 15세기 이탈리아에서 체계화되었으며, 현대 회계의 기본 원칙이다. 차변 합계와 대변 합계가 일치해야 하므로, 오류를 자동으로 감지할 수 있다.*)**에 비유한다. 이것은 TDD의 성격을 이해하는 데 매우 유용한 비유다.

복식부기는 회계사에게 선택 사항이 아니다. 모든 거래를 차변과 대변에 이중으로 기록하는 것은 회계의 기본 규율이다. 복식부기를 하지 않는 회계사는 프로가 아니다.

Uncle Bob은 TDD도 마찬가지라고 주장한다:

| 복식부기 (회계) | TDD (소프트웨어) |
|----------------|-----------------|
| 모든 거래를 차변과 대변에 이중 기록 | 모든 기능을 테스트와 프로덕션 코드에 이중 기록 |
| 차변 합계 = 대변 합계로 검증 | 테스트 실행으로 검증 |
| "너무 번거롭다"는 이유로 생략하지 않음 | "너무 느리다"는 이유로 생략하지 않아야 함 |
| 프로 회계사의 기본 규율 | 프로 개발자의 기본 규율이 되어야 함 |

이 비유는 의도적으로 도발적이다. Uncle Bob은 TDD를 하지 않는 개발자가 복식부기를 하지 않는 회계사만큼이나 무책임하다고 암시하는 것이다.

### 4.2 책임있는 선택

프로 개발자는 자신이 작성한 코드에 대해 책임을 진다 (Chapter 1 참조). TDD는 이 책임을 이행하는 구체적인 방법이다:

- **코드가 동작하는지 확인할 책임** → 테스트가 이를 검증한다
- **버그를 최소화할 책임** → TDD가 결함 주입률을 낮춘다
- **코드를 변경 가능하게 유지할 책임** → 테스트 스위트가 안전한 리팩토링을 가능하게 한다
- **다음 개발자에게 코드를 설명할 책임** → 테스트가 실행 가능한 문서 역할을 한다

## 5. TDD가 아닌 것(What TDD Is NOT)

### 5.1 종교가 아니다

Uncle Bob은 TDD를 강력히 옹호하면서도, 그것이 **종교가 아님**을 명확히 한다. TDD는 맹목적 신앙이 아니라 **경험적 증거에 기반한 규율**이다.

TDD가 잘 적용되지 않는 경우도 있다:

| 상황 | 설명 |
|------|------|
| **GUI 코드** | 화면 레이아웃, 색상 등의 시각적 요소는 자동화된 테스트로 검증하기 어렵다. 비즈니스 로직을 GUI에서 분리하여 로직은 TDD로, GUI는 별도로 테스트하는 것이 현실적이다. |
| **일부 레거시 코드** | 테스트가 전혀 없는 거대한 레거시 시스템에 TDD를 적용하려면, 먼저 테스트 가능한 구조로 리팩토링해야 한다. 이 과정이 쉽지 않을 수 있다. |
| **탐색적 프로토타이핑** | 무엇을 만들어야 할지 자체가 불확실한 탐색 단계에서는 테스트를 먼저 작성하는 것이 비효율적일 수 있다. 단, 프로토타입을 프로덕션 코드로 전환할 때는 TDD를 적용해야 한다. |

### 5.2 좋은 설계를 보장하지 않는다

TDD는 설계를 **돕지만**, **보장하지는 않는다.**

TDD를 따르더라도 개발자가 설계에 대해 생각하지 않으면, 테스트가 있는 나쁜 코드가 만들어질 수 있다. TDD는 도구이지, 사고의 대체물이 아니다.

```python
# TDD를 했지만 나쁜 설계의 예
class UserManager:
    def do_everything(self, action, user_id, data=None):
        """하나의 메서드가 모든 것을 처리한다 — SRP 위반"""
        if action == "create":
            # 100줄의 생성 로직
            pass
        elif action == "update":
            # 100줄의 수정 로직
            pass
        elif action == "delete":
            # 50줄의 삭제 로직
            pass
        elif action == "notify":
            # 80줄의 알림 로직
            pass
```

이 코드는 테스트가 있더라도 나쁜 설계다. TDD는 설계를 의식적으로 할 수 있도록 유도하지만, 설계 역량 자체를 대체하지는 못한다.

### 5.3 100% 커버리지는 점근선이다

Uncle Bob은 100% 테스트 커버리지(*Test Coverage — 테스트가 실행하는 코드의 비율. 라인 커버리지, 브랜치 커버리지, 경로 커버리지 등 여러 측정 방법이 있다. 높은 커버리지가 반드시 좋은 테스트를 의미하지는 않지만, 낮은 커버리지는 거의 확실하게 불충분한 테스트를 의미한다.*)가 **이상적 목표이지 현실적 요구는 아니라**고 인정한다.

어떤 코드는 테스트하기 어렵거나, 테스트 비용 대비 효과가 미미하다. 중요한 것은 특정 숫자(80%, 90%, 100%)에 집착하는 것이 아니라, **비즈니스에 중요한 로직이 충분히 테스트되고 있는가**를 판단하는 것이다.

그러나 이것이 낮은 커버리지를 정당화하는 핑계가 되어서는 안 된다. "100%는 불가능하니까 60%로 충분하다"는 논리는 "완벽한 건강은 불가능하니까 운동을 안 해도 된다"는 논리와 같다.

## 6. TDD 논쟁: 찬반 양론

### 6.1 반대 의견과 Uncle Bob의 반박

TDD는 소프트웨어 업계에서 가장 논쟁적인 주제 중 하나다. Uncle Bob은 반대 의견을 인정하면서도 각각에 대해 반박한다:

| 반대 의견 | Uncle Bob의 반박 |
|----------|-----------------|
| "TDD는 느리다" | 단기적으로 코딩 속도가 느려질 수 있다. 하지만 디버깅, 수정, 리팩토링에 드는 시간을 포함하면 **총 개발 시간은 줄어든다.** |
| "모든 코드를 테스트할 수 없다" | 맞다. 하지만 "전부 테스트할 수 없으니 아무것도 테스트하지 않겠다"는 논리는 성립하지 않는다. |
| "테스트 코드도 유지보수해야 한다" | 맞다. 하지만 테스트가 없을 때의 유지보수 비용(디버깅, 회귀 버그, 변경 두려움)이 훨씬 크다. |
| "설계를 미리 생각하지 않게 된다" | TDD가 설계 사고를 대체하는 것이 아니라 **보완**한다. TDD를 하면서 동시에 설계를 의식적으로 해야 한다. |
| "경험 많은 개발자에게는 불필요하다" | 경험이 많을수록 코드의 복잡성도 높은 프로젝트를 다룬다. 복잡한 코드일수록 안전망의 가치는 커진다. |

### 6.2 Uncle Bob의 입장 요약

Uncle Bob의 입장을 한 문장으로 요약하면:

> **핵심 통찰**: "TDD가 완벽한가? 아니다. TDD가 모든 상황에 적용 가능한가? 아니다. 하지만 TDD가 프로 소프트웨어 개발자에게 가장 **책임감 있는 선택**인가? 그렇다." — TDD는 완벽하지 않지만, 현재 우리가 가진 가장 효과적인 품질 보증 규율이다.

## 7. TDD의 실천: 구체적 가이드

### 7.1 시작하는 방법

TDD를 처음 시작하는 개발자를 위한 Uncle Bob의 조언:

1. **작은 것부터 시작하라**: 전체 프로젝트에 한꺼번에 TDD를 적용하려 하지 말고, 작은 유틸리티 함수 하나부터 시작하라.
2. **카타를 연습하라**: 프로덕션 코드가 아닌 연습 문제(카타)로 TDD 사이클에 익숙해지라. (Chapter 6 참조)
3. **페어 프로그래밍으로 배워라**: TDD를 이미 실천하는 개발자와 페어링하면 학습 속도가 훨씬 빠르다.
4. **인내하라**: 처음 2~3주는 불편하다. 하지만 그 고비를 넘기면 TDD 없이는 코딩하기 불안해지는 단계에 도달한다.

### 7.2 Red-Green-Refactor 사이클

TDD 사이클의 각 단계를 구체적인 예시로 살펴보자.

**예시: 스택(Stack) 구현**

```python
# --- [Red] 실패하는 테스트 작성 ---
def test_new_stack_is_empty():
    stack = Stack()
    assert stack.is_empty() is True
# 실행 → 실패 (Stack 클래스가 없으므로)

# --- [Green] 테스트를 통과시키는 최소한의 코드 ---
class Stack:
    def is_empty(self):
        return True
# 실행 → 통과

# --- [Red] 다음 실패하는 테스트 ---
def test_stack_after_push_is_not_empty():
    stack = Stack()
    stack.push(42)
    assert stack.is_empty() is False
# 실행 → 실패 (push 메서드가 없고, is_empty가 항상 True를 반환)

# --- [Green] 테스트를 통과시키는 코드 ---
class Stack:
    def __init__(self):
        self._items = []

    def push(self, item):
        self._items.append(item)

    def is_empty(self):
        return len(self._items) == 0
# 실행 → 통과

# --- [Red] 다음 테스트 ---
def test_pop_returns_pushed_item():
    stack = Stack()
    stack.push(42)
    assert stack.pop() == 42
# 실행 → 실패

# --- [Green] ---
class Stack:
    def __init__(self):
        self._items = []

    def push(self, item):
        self._items.append(item)

    def pop(self):
        return self._items.pop()

    def is_empty(self):
        return len(self._items) == 0
# 실행 → 통과

# --- [Refactor] 필요하면 정리 ---
# 현재는 충분히 깔끔하므로 다음 테스트로 진행
```

이 예시에서 볼 수 있듯이, 각 단계는 **극히 작은 변경**으로 이루어진다. 한 번에 큰 기능을 구현하는 것이 아니라, 테스트 하나 → 코드 조금 → 테스트 하나 → 코드 조금의 사이클을 반복한다.

### 7.3 효과적인 TDD를 위한 팁

| 팁 | 설명 |
|----|------|
| **테스트 이름을 명확하게 지어라** | `test_1`, `test_func`가 아니라 `test_empty_stack_raises_on_pop`처럼 동작을 설명하는 이름을 사용한다 |
| **하나의 테스트에 하나의 assertion** | 가능한 한 하나의 테스트가 하나의 동작만 검증하도록 한다 |
| **테스트가 빨라야 한다** | 전체 테스트 스위트가 몇 초 내에 실행되어야 한다. 느린 테스트는 실행 빈도를 떨어뜨린다 |
| **테스트 간 독립성을 유지하라** | 테스트 A의 결과가 테스트 B에 영향을 미치면 안 된다 |
| **프로덕션 코드의 일부가 되지 않게 하라** | 테스트 코드가 프로덕션 배포에 포함되어서는 안 된다 |

## 실전 시나리오

### 시나리오 1: "TDD 할 시간이 없다"

**상황**: 마감이 촉박하다. PM이 "테스트 작성할 시간이 없으니 기능 구현에만 집중하라"고 요청한다.

**비프로 대응**:
> "알겠습니다. 테스트는 나중에 작성하죠." — 그리고 "나중"은 오지 않는다. 테스트 없는 코드가 프로덕션에 배포되고, 몇 주 후 회귀 버그가 연쇄적으로 발생한다.

**프로 대응**:
> "TDD를 빼면 단기적으로 빨라 보이지만, 디버깅과 회귀 버그 수정을 포함한 총 시간은 오히려 늘어납니다. 만약 일정이 정말 촉박하다면, 범위를 줄이는 것이 더 나은 선택입니다. 품질을 희생하는 것은 미래의 일정을 더 악화시킵니다."

**원칙**: TDD를 빼는 것은 시간을 절약하는 것이 아니라 **시간을 빌리는 것**이다. 그것도 높은 이자를 붙여서.

### 시나리오 2: "이 코드는 테스트할 수 없다"

**상황**: 레거시 시스템의 코드가 너무 얽혀 있어서 단위 테스트를 작성할 수 없다.

**비프로 대응**:
> "이 코드는 테스트가 불가능하니 그냥 수동으로 확인합시다." — 그리고 변경할 때마다 무엇이 깨졌는지 알 수 없게 된다.

**프로 대응**:
> "현재 구조에서는 단위 테스트가 어렵습니다. 먼저 의존성을 분리하는 최소한의 리팩토링을 수행하고, 변경하려는 부분 주변에 특성 테스트(characterization test)(*Characterization Test — Michael Feathers가 "Working Effectively with Legacy Code"에서 제안한 기법. 기존 코드의 현재 동작을 있는 그대로 기록하는 테스트. 코드의 의도가 아니라 실제 동작을 포착하여, 리팩토링 후에도 동작이 변하지 않았음을 확인하는 데 사용한다.*)를 추가하겠습니다. 모든 것을 한 번에 테스트할 수는 없지만, 변경하는 부분만큼은 테스트 커버리지 아래에 놓겠습니다."

**원칙**: "테스트할 수 없다"는 것은 대부분 "설계가 나쁘다"는 뜻이다. 테스트 가능한 구조로 점진적으로 개선하라.

### 시나리오 3: 팀 내 TDD 도입

### 대화: TDD 도입 논의

> **개발자 A**: TDD를 팀에 도입하고 싶은데, 다들 반대할 것 같아요.
>
> **개발자 B (프로)**: 강요하지 마세요. 먼저 자기 코드에 TDD를 적용하고, 결과를 보여주세요. 당신의 코드에서 버그가 줄고, 리팩토링이 쉬워지고, 변경 속도가 빨라지면, 동료들이 관심을 갖기 시작합니다. 그때 페어 프로그래밍을 제안하세요. 말로 설득하지 말고, **결과로 설득하세요.**

---

## 프로의 원칙

| 원칙 | 설명 |
|------|------|
| **TDD는 프로의 규율이다** | TDD는 선택적 기법이 아니라, 프로 개발자가 코드 품질에 대한 책임을 이행하는 핵심 규율이다. |
| **세 가지 법칙을 따르라** | 실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다. 이것이 30초 사이클의 기반이다. |
| **테스트는 용기를 준다** | 포괄적인 테스트 스위트가 있으면, 두려움 없이 코드를 변경하고 리팩토링할 수 있다. |
| **테스트하기 어려움 = 설계 문제** | 테스트가 어려운 코드는 설계가 나쁜 코드다. TDD는 테스트 기법인 동시에 설계 기법이다. |
| **TDD에도 한계가 있다** | GUI, 일부 레거시 코드, 탐색적 프로토타이핑에서는 TDD의 적용이 제한될 수 있다. 맹목적 신앙이 아닌 실용적 판단이 필요하다. |
| **결과로 설득하라** | TDD를 팀에 도입하려면 말이 아니라 자신의 코드로 먼저 증명하라. |

## 요약

- **TDD는 프로 개발자의 규율이다.** Uncle Bob은 TDD를 회계사의 복식부기에 비유하며, 선택이 아닌 기본 실천으로 주장한다.
- **세 가지 법칙이 30초 사이클을 만든다.** 실패하는 테스트 → 통과시키는 코드 → 리팩토링의 사이클을 극히 짧은 주기로 반복한다.
- **TDD는 변경의 용기를 준다.** 포괄적인 테스트 스위트는 리팩토링의 안전망이 되어, 코드를 지속적으로 개선할 수 있게 한다.
- **TDD는 결함 주입률을 줄인다.** 버그가 주입되는 순간과 발견되는 순간 사이의 시간이 극히 짧아지므로, 버그 수정 비용이 극적으로 줄어든다.
- **테스트는 실행 가능한 문서다.** 잘 작성된 테스트는 코드의 사용법, 기대 동작, 제약 조건을 보여주는 항상 최신 상태의 문서다.
- **테스트하기 어려운 코드는 나쁜 설계의 신호다.** TDD는 테스트 기법인 동시에 설계 기법이다.
- **TDD는 종교가 아니다.** GUI, 레거시 코드, 프로토타이핑 등 TDD가 적합하지 않은 상황도 있다. 실용적 판단이 필요하다.
- **100% 커버리지는 점근선이다.** 특정 숫자에 집착하기보다 비즈니스에 중요한 로직이 충분히 테스트되는가를 판단하라.
- **팀에 TDD를 도입하려면 결과로 설득하라.** 강요가 아니라 자신의 코드로 먼저 증명하라.

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: "QA가 아무것도 찾지 못해야 한다"는 원칙의 실천 방법으로 TDD를 제시한다. TDD는 코드 품질에 대한 책임감의 구체적 표현이다.
- **Chapter 3 (Saying Yes)**: TDD를 빼면 시간을 절약할 수 있다는 제안에 "아니오"라고 말하는 것은 Chapter 2~3의 커뮤니케이션 원칙과 연결된다.
- **Chapter 4 (Coding)**: "디버깅은 낭비다"라는 원칙의 실천 방법이 TDD다. TDD의 30초 사이클이 Zone을 피하는 도구이기도 하다.
- **Chapter 6 (Practicing)**: TDD 기술을 유지하고 향상시키기 위한 카타, 와사 등의 연습 방법을 다룬다. TDD는 프로덕션에서 적용하기 전에 연습에서 먼저 익숙해져야 한다.
- **Chapter 7 (Acceptance Testing)**: 단위 테스트(TDD)와 인수 테스트는 다른 수준의 테스트다. TDD가 코드 수준의 품질을 보장한다면, 인수 테스트는 비즈니스 요구사항 수준의 품질을 보장한다.
- **Chapter 8 (Testing Strategies)**: TDD는 전체 테스트 전략의 일부다. 단위 테스트, 통합 테스트, 인수 테스트, 탐색적 테스트가 어떻게 조합되는지를 Chapter 8에서 다룬다.
- **Chapter 11 (Pressure)**: 압박 속에서 TDD를 포기하는 것이 가장 큰 유혹이다. "위기 상황에서의 행동이 진짜 규율을 드러낸다"는 Chapter 11의 핵심 메시지와 직접 연결된다.
- **Chapter 14 (Mentoring, Apprenticeship, and Craftsmanship)**: TDD는 멘토링을 통해 전수되어야 할 핵심 규율 중 하나다. 장인이 도제에게 가르쳐야 할 가장 중요한 실천법이다.

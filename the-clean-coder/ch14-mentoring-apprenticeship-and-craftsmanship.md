# Chapter 14: Mentoring, Apprenticeship, and Craftsmanship (멘토링, 도제, 장인정신)

## 핵심 질문

소프트웨어 산업은 어떻게 차세대 전문가를 체계적으로 양성할 수 있으며, 장인정신(craftsmanship)은 그 과정에서 어떤 역할을 하는가?

---

## 1. 컴퓨터 과학 교육의 실패

### 1.1 대학이 가르치는 것과 가르치지 않는 것

대학의 컴퓨터 과학(CS) 교육은 이론적 기반을 제공하는 데는 탁월하다. 알고리즘, 자료구조, 운영체제, 컴파일러, 수학적 기초 등은 대학에서 잘 다루는 영역이다. 그러나 **전문적 소프트웨어 개발 실무**에 대해서는 거의 가르치지 않는다.

대학 졸업생이 일반적으로 **알고 있는 것**:

| 영역 | 구체적 내용 |
|------|------------|
| 이론 | 알고리즘 분석, 계산 복잡도, 오토마타 이론 |
| 자료구조 | 트리, 그래프, 해시 테이블, 정렬 알고리즘 |
| 언어 | 1~2개 프로그래밍 언어의 문법과 기본 사용법 |
| 수학 | 이산수학, 선형대수, 확률/통계 기초 |

대학 졸업생이 일반적으로 **모르는 것**:

| 영역 | 구체적 내용 |
|------|------------|
| 팀 협업 | 코드 리뷰, 페어 프로그래밍, 팀 커뮤니케이션 |
| 테스트 | TDD, 단위 테스트 작성, 인수 테스트, 테스트 전략 |
| 설계 | 실무 수준의 소프트웨어 설계, 리팩터링, 클린 코드 |
| 프로젝트 관리 | 추정, 일정 관리, 이해관계자와의 소통 |
| 도구 | 버전 관리, CI/CD, 빌드 시스템, 배포 |
| 직업 윤리 | 프로로서의 책임, 거절하는 법, 약속 관리 |

### 1.2 학문적 지식과 전문적 역량 사이의 간극

이론적 지식과 전문적 역량 사이의 간극은 거대하다. 의학에 비유하면, 대학은 해부학과 생리학을 가르치지만, 실제로 수술하는 법은 가르치지 않는 것과 같다. 의학에는 이 간극을 메우기 위한 **레지던시 제도**가 있지만, 소프트웨어 산업에는 그에 상응하는 체계가 없다.

> **핵심 통찰**: CS 학위는 소프트웨어 전문가가 되기 위한 **시작점**일 뿐, 완성이 아니다. 대학 졸업은 면허가 아니라 학습 허가증에 가깝다.

## 2. 실패의 양상

### 2.1 산업 현장의 현실

Uncle Bob은 소프트웨어 산업의 인력 양성 방식이 근본적으로 잘못되어 있다고 진단한다. 너무 많은 신입 개발자가 적절한 멘토링 없이 프로젝트에 투입된다. 그들은 시행착오를 통해 배우는데, 대부분은 **오류를 통한 학습**이다.

이 과정에서 발생하는 문제는 다음과 같다:

- **나쁜 습관의 고착화**: 멘토 없이 배운 개발자는 잘못된 관행을 "정상"으로 받아들인다
- **같은 실수의 반복**: 선배 개발자가 이미 겪은 실수를 후배가 똑같이 반복한다
- **품질 의식의 부재**: 좋은 코드가 어떤 것인지 본 적이 없으면, 좋은 코드를 작성할 수 없다
- **성장 속도의 저하**: 체계적 학습 없이는 전문가로 성장하는 데 불필요하게 긴 시간이 걸린다

> **Uncle Bob의 경험**: Uncle Bob 자신도 커리어 초기에 제대로 된 멘토 없이 일했다. 그는 모든 것을 스스로 깨우쳐야 했고, 누군가가 자신을 멘토링해 주었더라면 훨씬 빨리, 훨씬 적은 고통으로 전문가가 될 수 있었을 것이라고 회고한다. 그의 초기 경력에서의 실수와 시행착오는 적절한 지도만 있었어도 피할 수 있는 것이었다.

### 2.2 체계적 훈련 시스템의 부재

소프트웨어 산업에는 의학, 법률, 항공 등 다른 전문 분야에 존재하는 **체계적 훈련 과정**이 없다. 의사는 레지던시를 거치고, 변호사는 로클럭(*law clerk — 판사 밑에서 실무를 배우는 법률 수습 과정*)을 거치며, 파일럿은 수백 시간의 비행 훈련을 받는다. 그러나 소프트웨어 개발자는? 대학을 졸업하면 곧바로 프로덕션 코드를 작성하는 위치에 놓인다.

## 3. 멘토링

### 3.1 멘토링의 의무

시니어 개발자에게 멘토링은 **선택이 아니라 의무**다. 전문가(professional)라는 지위에는 후배를 양성할 책임이 수반된다. 이것은 이타적 행위가 아니라, 전문 직업인으로서의 본질적 의무다.

멘토링이 포함하는 활동은 다음과 같다:

| 활동 | 설명 |
|------|------|
| 페어 프로그래밍 | 함께 코드를 작성하며 사고 과정을 공유한다 |
| 코드 리뷰 | 코드를 검토하며 개선점과 원리를 설명한다 |
| 실천법 전수 | TDD, 리팩터링, 클린 코드 등의 규율을 가르친다 |
| 커리어 가이드 | 기술적 성장 방향과 경력 개발을 안내한다 |
| 실수 허용 | 안전한 환경에서 실수하고 배울 기회를 제공한다 |

### 3.2 멘토링의 양방향성

멘토링은 일방적 지식 전달이 아니다. 가장 좋은 학습 방법은 누군가에게 배우는 것이고, 가장 좋은 지식 강화 방법은 누군가를 가르치는 것이다. 멘토는 가르치는 과정에서 자신의 이해를 더 깊게 하고, 무의식적으로 알고 있던 것을 명시적으로 정리하게 된다.

> **핵심 통찰**: 멘토링의 최대 수혜자는 종종 멘티가 아니라 멘토 자신이다. 가르침은 배움의 가장 강력한 형태다.

### 3.3 효과적인 멘토링 방법

효과적인 멘토링은 단순히 "옆에 앉아서 지켜보라"는 것이 아니다. 구체적인 실천 방법은 다음과 같다:

1. **함께 일하기**: 멘토와 멘티는 같은 코드를 함께 작성한다. 페어 프로그래밍이 가장 효과적인 멘토링 도구다.
2. **의사결정 과정 공유**: "왜 이렇게 하는지"를 설명한다. 결과물이 아니라 사고 과정을 전수한다.
3. **점진적 자율성 부여**: 처음에는 밀착 지도하고, 역량이 성장함에 따라 점차 독립성을 부여한다.
4. **피드백 루프**: 정기적으로 코드와 행동에 대한 피드백을 주고받는다.
5. **실전 과제 부여**: 이론이 아닌 실제 프로젝트 맥락에서 학습하도록 한다.

## 4. 도제 모델 (Apprenticeship Model)

### 4.1 Uncle Bob의 제안

Uncle Bob은 소프트웨어 산업에 **도제 제도**를 도입할 것을 제안한다. 이 모델은 중세 길드 제도와 현대 의학의 레지던시 과정에서 영감을 받은 것이다. 핵심 아이디어는 개발자의 성장 단계를 명확히 구분하고, 각 단계에 적합한 책임과 감독 수준을 부여하는 것이다.

### 4.2 세 가지 단계

#### 마스터 (Master)

| 항목 | 내용 |
|------|------|
| 경력 | 10년 이상의 실무 경험 |
| 리더십 | 복수의 프로젝트를 이끈 경험 |
| 기술 역량 | 깊은 기술적 전문성 |
| 관리 능력 | 기술적 의사결정과 팀 관리 능력 |
| 멘토링 | 저니맨과 어프렌티스를 지도하는 역할 |

마스터는 단순히 오래 일한 사람이 아니다. 기술적 탁월함과 함께 다른 사람을 이끌고 가르칠 수 있는 능력을 갖춘 사람이다. 마스터는 조직의 기술 문화를 정의하고, 품질 기준을 설정하며, 후배 개발자의 성장을 책임진다.

#### 저니맨 (Journeyman)

| 항목 | 내용 |
|------|------|
| 경력 | 5년 이상의 실무 경험 |
| 역량 | 유능하고 자율적으로 일할 수 있음 |
| 학습 | 여전히 마스터로부터 배우는 중 |
| 멘토링 | 어프렌티스를 직접 지도함 |
| 이동성 | 여러 프로젝트와 팀을 경험하며 시야를 넓힘 |

저니맨은 "여행하는 기술자"라는 원래 의미처럼(*journeyman — 중세 길드에서 도제 과정을 마치고 독립 장인이 되기 전, 여러 공방을 돌며 기술을 연마하던 기술자. "journey"는 여행이 아니라 하루(journée)의 의미에서 유래했다.*), 다양한 프로젝트와 환경을 경험하며 역량을 넓히는 단계다. 독립적으로 업무를 수행할 수 있지만, 마스터의 감독과 지도를 통해 계속 성장한다.

#### 어프렌티스/인턴 (Apprentice/Intern)

| 항목 | 내용 |
|------|------|
| 대상 | 신규 졸업생 또는 입문 개발자 |
| 감독 | 저니맨과 밀착하여 함께 작업 |
| 자율성 | 독립적 작업 없음, 항상 페어로 작업 |
| 학습 초점 | TDD, 리팩터링, 클린 코드, 설계 원칙 등 전문적 규율 |
| 기간 | 약 1년간의 집중 수련 |

### 4.3 어프렌티스십의 구체적 모습

어프렌티스 기간 동안의 핵심 원칙은 다음과 같다:

- **자율적 작업 금지**: 어프렌티스는 혼자 코드를 작성하지 않는다. 항상 저니맨이나 마스터와 페어로 작업한다.
- **규율의 체화**: TDD, 리팩터링, 지속적 통합 등 전문적 실천법을 일상으로 익힌다. 이것은 이론이 아니라 몸에 배는 습관이어야 한다.
- **점진적 책임 확대**: 약 1년의 수련 기간을 거친 후, 저니맨으로 승급하여 자율적으로 일할 수 있게 된다.
- **평가와 피드백**: 정기적으로 역량을 평가받고, 부족한 부분에 대한 구체적 피드백을 받는다.

```
[도제 모델의 성장 경로]

어프렌티스 (1년) → 저니맨 (5년+) → 마스터 (10년+)
     ↑                    ↑                ↑
  밀착 감독          자율적 업무       기술 리더십
  페어 작업          멘토링 시작       문화 정의
  규율 학습          다양한 경험       후진 양성
```

### 4.4 현실적 적용

Uncle Bob의 도제 모델은 이상적으로 보일 수 있지만, 현실에서 완전히 구현하기는 어렵다. 그러나 핵심 원리를 부분적으로 적용하는 것은 가능하다:

- 신입 개발자를 일정 기간 시니어와 페어로 배치하는 **온보딩 프로그램**
- 코드 리뷰를 단순한 게이트키핑이 아닌 **학습의 기회**로 활용
- 멘토-멘티 관계를 공식적으로 지정하는 **멘토링 프로그램**
- 경력 단계별 기대 역량을 명시하는 **역량 프레임워크**

## 5. 장인정신 (Craftsmanship)

### 5.1 소프트웨어 장인정신 운동

소프트웨어 장인정신(Software Craftsmanship) 운동은 소프트웨어 개발을 단순한 공학적 행위가 아닌 **장인의 기예**로 바라보는 관점이다. Uncle Bob은 이 운동의 중심 인물 중 한 명이다.

이 운동은 애자일 선언문(Agile Manifesto, 2001)의 정신을 계승하되, 기술적 탁월함에 더 강한 초점을 맞춘다. 애자일이 "프로세스"에 대한 선언이었다면, 장인정신은 "실천"에 대한 선언이다.

### 5.2 소프트웨어 장인정신 선언문 (2009)

2009년에 발표된 소프트웨어 장인정신 선언문(Manifesto for Software Craftsmanship)(*소프트웨어 장인정신 선언문 — manifesto.softwarecraftsmanship.org에서 원문을 볼 수 있다. Uncle Bob을 포함한 여러 소프트웨어 전문가들이 서명했다.*)은 다음과 같은 가치를 천명한다:

| 애자일 선언문 | 장인정신 선언문 (추가) |
|-------------|---------------------|
| 동작하는 소프트웨어 | **잘 만들어진 소프트웨어** (well-crafted software) |
| 변화에 대응 | **꾸준히 가치를 더하기** (steadily adding value) |
| 개인과 상호작용 | **전문가 공동체** (a community of professionals) |
| 고객 협력 | **생산적 파트너십** (productive partnerships) |

> **Uncle Bob의 경험**: 소프트웨어 장인정신 선언문이 탄생한 과정 자체가 장인정신의 한 사례였다. 소프트웨어 업계의 여러 뜻있는 전문가들이 모여 "동작하는 소프트웨어"만으로는 충분하지 않다는 공감대를 형성했고, 기술적 탁월함을 향한 더 높은 기준을 세우고자 이 선언문을 작성했다.

### 5.3 장인정신의 핵심 가치

#### 잘 만들어진 소프트웨어 (Well-Crafted Software)

단순히 "동작하는" 소프트웨어로는 충분하지 않다. 전문가는 **잘 설계되고, 잘 테스트되고, 깨끗하고, 유지보수 가능한** 소프트웨어를 만든다. 코드는 작성되는 순간보다 읽히는 순간이 훨씬 더 많다. 따라서 코드의 가독성, 명확성, 구조적 건전성은 "동작 여부" 못지않게 중요하다.

#### 꾸준히 가치를 더하기 (Steadily Adding Value)

소프트웨어는 시간이 지남에 따라 가치가 떨어지는 것이 아니라 **계속 가치가 증가**해야 한다. 이것은 기술 부채를 관리하고, 코드를 꾸준히 개선하며, 새로운 기능을 안정적으로 추가할 수 있는 구조를 유지한다는 뜻이다. 매번 코드를 건드릴 때마다 조금씩 더 나아지게 만드는 보이스카우트 규칙(*Boy Scout Rule — "캠핑장을 발견했을 때보다 더 깨끗하게 떠나라." 코드를 수정할 때 기존보다 조금이라도 더 깨끗하게 만들라는 원칙이다.*)이 이 가치의 실천이다.

#### 전문가 공동체 (A Community of Professionals)

장인정신은 개인의 탁월함만이 아니라 **공동체**를 강조한다. 전문가들은 서로 배우고, 가르치고, 영감을 주고받는다. 사용자 그룹, 컨퍼런스, 오픈소스 프로젝트, 스터디 그룹 등을 통해 지식을 공유하고 표준을 높인다.

#### 생산적 파트너십 (Productive Partnerships)

개발자와 고객/비즈니스 사이의 관계는 종속이 아니라 **파트너십**이다. 전문가는 고객의 요구를 맹목적으로 따르지 않고, 전문적 판단에 기반하여 최선의 방안을 제안하고 협력한다.

### 5.4 장인정신은 전염된다

장인정신의 가장 강력한 특성은 **전염성**이다. 한 명의 장인이 팀에 합류하면, 그 팀의 기준이 올라간다. 높은 수준의 코드를 보면 다른 팀원들도 자연스럽게 더 나은 코드를 작성하려고 노력하게 된다.

이것은 강제가 아니라 **감화**를 통해 일어난다:

- 장인이 TDD로 코드를 작성하는 것을 본 동료가 TDD를 배우기 시작한다
- 깨끗한 코드를 리뷰하면서 "이런 코드도 가능하구나"라는 깨달음을 얻는다
- 페어 프로그래밍을 통해 좋은 습관이 자연스럽게 전파된다
- 한 팀의 문화가 다른 팀으로 퍼져나간다

> **핵심 통찰**: 장인정신은 엘리트주의가 아니다. 특별한 재능을 가진 소수만을 위한 것이 아니라, 모든 개발자가 더 높은 기준을 향해 성장하도록 돕는 것이다. 장인정신은 문을 닫는 것이 아니라 문을 여는 것이다.

### 5.5 장인정신의 실천

장인정신은 추상적 이상이 아니라 구체적 실천으로 구현된다:

| 실천법 | 설명 |
|--------|------|
| TDD | 테스트를 먼저 작성하여 설계와 품질을 동시에 확보한다 |
| 리팩터링 | 코드를 지속적으로 개선하여 부패를 방지한다 |
| 페어 프로그래밍 | 함께 작업하며 지식을 공유하고 품질을 높인다 |
| 지속적 학습 | 끊임없이 새로운 기술과 원리를 학습한다 |
| 코드 리뷰 | 동료의 코드를 검토하며 상호 학습한다 |
| 클린 코드 | 읽기 쉽고 유지보수 가능한 코드를 작성한다 |

## 실전 시나리오

### 상황

당신은 시니어 개발자다. 팀에 대학을 갓 졸업한 신입 개발자가 합류했다. 프로젝트는 마감이 빠듯하다. 신입에게 독립적인 모듈을 맡기면 일정에 도움이 될 것 같다.

### 비전문가의 대응

"시간이 없으니 이 모듈을 혼자 맡아서 처리해. 모르는 건 검색해보고, 막히면 그때 물어봐."

신입은 방치되고, 나쁜 습관을 형성하며, 결국 작성한 코드는 품질 문제로 인해 나중에 재작업이 필요하게 된다. 단기적으로 시간을 아끼려다 장기적으로 더 큰 비용을 치르는 결과를 초래한다.

### 전문가의 대응

"처음 2주는 나와 페어로 작업하자. 우리 팀의 코딩 표준, TDD 방식, 코드 리뷰 프로세스를 함께 익히고 나서 점차 독립적인 작업을 시작하자."

단기적으로 시니어의 생산성은 떨어지지만, 신입은 올바른 습관을 빨리 형성하고, 몇 주 후부터 팀에 실질적 기여를 시작한다. 장기적으로 팀 전체의 역량이 향상된다.

### 원칙

**멘토링은 투자이지 비용이 아니다.** 신입을 방치하는 것이 진짜 비용이다.

## 프로의 원칙

| 원칙 | 설명 |
|------|------|
| 멘토링은 의무다 | 시니어는 후배를 가르칠 전문적 책임이 있다 |
| 배움은 가르침으로 완성된다 | 지식을 전수하는 과정에서 자신의 이해가 깊어진다 |
| 체계적 양성이 필요하다 | 시행착오 학습은 비효율적이고 위험하다 |
| 단계별 성장을 존중하라 | 어프렌티스 → 저니맨 → 마스터, 각 단계에 맞는 기대치가 있다 |

| 공동체에 기여하라 | 지식 공유와 상호 학습을 통해 업계 전체의 수준을 높인다 |
| 겸손하게 배우라 | 장인정신은 엘리트주의가 아니라 지속적 성장의 태도다 |

## 요약

- 대학의 CS 교육은 이론적 기초를 제공하지만, 전문적 소프트웨어 개발 실무는 거의 가르치지 않는다. 학위와 역량 사이의 간극은 크다.
- 소프트웨어 산업은 체계적인 인력 양성 시스템이 부재하다. 너무 많은 신입이 멘토 없이 현장에 투입되어 시행착오를 통해 배운다.
- 멘토링은 시니어 개발자의 **선택이 아니라 의무**다. 페어 프로그래밍, 코드 리뷰, 실천법 전수, 커리어 가이드가 핵심 활동이다.
- Uncle Bob은 중세 길드와 의학 레지던시를 모델로 한 **도제 제도**를 제안한다: 어프렌티스(1년, 밀착 감독) → 저니맨(5년+, 자율적) → 마스터(10년+, 기술 리더).
- 어프렌티스 기간에는 독립적 작업 없이 항상 페어로 작업하며, TDD, 리팩터링 등 전문적 규율을 체화한다.
- **소프트웨어 장인정신 선언문**(2009)은 잘 만들어진 소프트웨어, 꾸준한 가치 추가, 전문가 공동체, 생산적 파트너십을 핵심 가치로 천명한다.
- 장인정신은 엘리트주의가 아니라, 모든 개발자의 기준을 높이는 운동이다. 전염성이 있어 한 명의 장인이 팀 전체의 수준을 끌어올릴 수 있다.
- 멘토링은 비용이 아니라 투자다. 단기적 생산성 저하를 감수하더라도 장기적으로 팀과 산업 전체에 훨씬 큰 가치를 만들어낸다.

## 자기 점검

- [ ] 나는 후배 개발자를 적극적으로 멘토링하고 있는가?
- [ ] 나의 멘토링은 구체적이고 실질적인가, 아니면 "모르면 물어봐" 수준에 머물러 있는가?
- [ ] 나는 페어 프로그래밍이나 코드 리뷰를 통해 지식을 전수하고 있는가?
- [ ] 나의 팀에 신입 개발자를 위한 체계적 온보딩 프로세스가 있는가?
- [ ] 나는 "잘 만들어진" 소프트웨어를 위해 "동작하는" 것 이상을 추구하고 있는가?
- [ ] 나는 전문가 공동체(스터디, 컨퍼런스, 오픈소스 등)에 참여하고 기여하고 있는가?
- [ ] 나의 기술적 기준이 팀에 긍정적 영향을 미치고 있는가?
- [ ] 나 자신은 여전히 배우고 성장하고 있는가, 아니면 현재 수준에 안주하고 있는가?

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: 1장에서 정의한 "프로"의 개념이 14장에서 완성된다. 프로는 자기 자신만의 탁월함이 아니라, 후배를 양성하고 업계 전체의 수준을 높이는 데까지 책임을 진다.
- **Chapter 3 (Saying Yes)**: 멘토링에 "예"라고 말하는 것은 전문가의 중요한 약속이다. 시간이 없다는 이유로 멘토링을 거부하는 것은 프로답지 못한 태도다.
- **Chapter 4 (Coding)**: 코딩 규율(TDD, 클린 코드)은 멘토링을 통해 전수되어야 할 핵심 내용이다. 혼자서는 배우기 어려운 것들이다.
- **Chapter 6 (Practicing)**: 연습(kata, 도장 등)은 어프렌티스가 규율을 체화하는 중요한 방법이다. 멘토는 효과적인 연습 방법을 가르쳐야 한다.
- **Chapter 9 (Time Management)**: 멘토링에 시간을 할애하는 것은 시간 낭비가 아니라 가장 높은 ROI를 가진 시간 투자다.
- **Chapter 13 (Teams and Projects)**: 팀의 기술 문화는 멘토링과 장인정신을 통해 형성되고 유지된다. 좋은 팀은 좋은 멘토링 문화를 가지고 있다.
- **Appendix A (Tooling)**: 도구의 효과적 사용법 역시 멘토링을 통해 전수되는 중요한 지식이다.

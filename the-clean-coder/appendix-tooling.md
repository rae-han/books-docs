# Appendix A: Tooling (도구)

## 핵심 질문

프로 개발자가 갖춰야 할 도구(tool)의 본질은 무엇이며, 끊임없이 변화하는 도구 생태계 속에서 변하지 않는 원칙은 무엇인가?

---

## 1. 장인의 도구

### 1.1 도구와 전문가의 관계

모든 전문 직종에는 고유한 도구가 있다. 목수에게 망치와 끌이 있고, 외과의에게 메스가 있듯이, 소프트웨어 개발자에게도 전문적 도구가 있다. 전문가는 자신의 도구를 깊이 이해하고, 능숙하게 다루며, 적절한 도구를 선택할 줄 안다.

> **Uncle Bob의 경험**: Uncle Bob은 40년이 넘는 커리어 동안 수많은 도구의 흥망성쇠를 목격했다. 펀치 카드에서 터미널로, vi에서 Emacs로, Emacs에서 Eclipse로, Eclipse에서 IntelliJ로. 도구는 끊임없이 변했지만, 도구를 사용하는 **원칙**은 놀라울 만큼 일관되게 유지되었다. 이 경험을 통해 그가 얻은 교훈은 분명하다: 특정 도구에 집착하지 말고, 도구 뒤에 있는 원칙을 이해하라.

### 1.2 도구 선택의 원칙

> **핵심 통찰**: 도구 자체보다 도구를 선택하고 사용하는 **원칙**이 중요하다. 특정 도구는 10년 안에 사라질 수 있지만, 좋은 도구를 고르는 눈과 도구를 마스터하는 습관은 커리어 전체를 관통한다.

도구를 선택할 때 고려해야 할 기준은 다음과 같다:

| 기준 | 설명 |
|------|------|
| 생산성 향상 | 도구가 실제로 작업 속도와 품질을 높이는가 |
| 학습 비용 | 도구를 익히는 데 드는 시간이 합리적인가 |
| 팀 호환성 | 팀 전체가 함께 사용할 수 있는가 |
| 단순성 | 불필요한 복잡성을 추가하지 않는가 |
| 지속 가능성 | 활발히 유지보수되고 커뮤니티가 있는가 |

## 2. 소스 제어 (Source Control)

### 2.1 소스 제어는 선택이 아니다

모든 전문 개발자는 소스 제어 시스템을 사용한다. **예외는 없다.** 소스 제어 없이 소프트웨어를 개발하는 것은, 안전벨트 없이 운전하는 것과 같다. 기술적으로 가능하지만 전문가라면 절대 하지 않는 행위다.

Uncle Bob의 커리어에서 소스 제어의 변천사는 다음과 같다:

| 시기 | 도구 | 특징 |
|------|------|------|
| 초기 | SCCS, RCS | 파일 단위 잠금(lock) 방식 |
| 1990년대 | CVS | 동시 편집 허용, 병합(merge) 도입 |
| 2000년대 | SVN (Subversion) | CVS의 개선판, 중앙 집중식 |
| 2010년대~ | Git | 분산형, 브랜치가 가벼움, 사실상 표준 |

### 2.2 현대의 소스 제어: Git

이 부록이 쓰인 2011년에도 Git은 이미 강력한 도구였지만, 현재는 소스 제어의 **사실상 표준(de facto standard)**이 되었다. Git을 사용하지 않는 것은 현대 소프트웨어 개발에서 점점 더 비정상적인 선택이 되고 있다.

### 2.3 소스 제어의 핵심 원칙

특정 도구에 관계없이 지켜야 할 소스 제어 원칙이 있다:

1. **자주 커밋하라**: 작은 단위로 자주 커밋하면 변경 이력을 추적하기 쉽고, 문제 발생 시 원인을 빠르게 파악할 수 있다.
2. **의미 있는 커밋 메시지를 작성하라**: "수정", "버그 픽스" 같은 모호한 메시지가 아니라, **왜** 변경했는지를 설명하는 메시지를 작성한다.
3. **브랜치를 효과적으로 사용하라**: 기능 개발, 버그 수정, 실험은 별도 브랜치에서 진행하고, 안정된 코드만 메인 브랜치에 병합한다.
4. **모든 것을 버전 관리하라**: 소스 코드뿐만 아니라 설정 파일, 빌드 스크립트, 데이터베이스 스키마 등 프로젝트에 필요한 모든 것을 관리한다.

> **핵심 통찰**: 소스 제어 시스템의 가장 중요한 기능은 "되돌리기"가 아니라 **"기록하기"**다. 프로젝트의 모든 변경 이유와 맥락을 보존하는 것이 소스 제어의 본질이다.

## 3. IDE vs 에디터

### 3.1 도구 마스터리의 중요성

Uncle Bob은 경력 동안 다양한 편집 도구를 사용했다: vi, Emacs, Eclipse, IntelliJ IDEA 등. 그의 결론은 **어떤 도구를 쓰느냐보다 그 도구를 얼마나 잘 다루느냐가 중요하다**는 것이다.

IDE(*Integrated Development Environment — 코드 편집, 컴파일, 디버깅, 버전 관리 등을 하나의 환경에서 제공하는 통합 개발 환경*)는 현대 소프트웨어 개발에서 강력한 생산성 도구다. 하지만 IDE의 진정한 가치는 기능의 수에 있지 않다. 개발자가 그 기능을 **얼마나 유창하게** 사용하느냐에 달려 있다.

### 3.2 도구 유창성 (Tool Fluency)

IDE를 "사용하는" 것과 IDE에 "유창한" 것은 다르다. 유창함이란 도구가 생각과 행동 사이의 **장벽이 되지 않는** 수준을 의미한다.

| 수준 | 특징 |
|------|------|
| 초급 | 메뉴를 탐색하며 기능을 찾는다 |
| 중급 | 주요 단축키를 알고 기본적인 리팩터링 도구를 사용한다 |
| 고급 | 도구가 손의 연장처럼 자연스럽다. 의식하지 않고 사용한다 |
| 마스터 | 도구를 자신의 워크플로우에 맞게 커스터마이즈하고, 플러그인을 활용한다 |

전문가가 반드시 익혀야 할 IDE 기능은 다음과 같다:

- **키보드 단축키**: 마우스에 손을 옮기는 것은 흐름을 끊는다. 핵심 작업은 모두 키보드로 수행할 수 있어야 한다.
- **리팩터링 도구**: 이름 변경(Rename), 메서드 추출(Extract Method), 변수 추출(Extract Variable) 등 자동화된 리팩터링은 현대 IDE의 핵심 기능이다.
- **코드 네비게이션**: 정의로 이동, 사용처 찾기, 파일 검색 등을 즉시 수행할 수 있어야 한다.
- **디버거**: 브레이크포인트, 스텝 실행, 변수 검사 등 디버깅 도구에 익숙해야 한다.
- **버전 관리 통합**: IDE 내에서 커밋, 디프, 블레임 등을 바로 확인할 수 있어야 한다.

> **핵심 통찰**: IDE는 마음의 연장이어야 한다. 생각을 코드로 변환하는 과정에서 도구가 걸림돌이 되어서는 안 된다. 도구에 투자하는 시간은 절대 낭비가 아니다.

### 3.3 도구 교체에 대한 유연성

도구 마스터리가 중요하다고 해서 하나의 도구에 영원히 묶여야 한다는 뜻은 아니다. 더 나은 도구가 등장하면 전환할 줄 아는 유연성도 전문가의 덕목이다. 다만, 새 도구로 전환할 때는 충분한 시간을 투자하여 이전 도구와 동등한 수준의 유창성을 확보해야 한다.

## 4. 이슈 트래킹 (Issue Tracking)

### 4.1 이슈 트래커의 필요성

모든 팀에는 버그와 작업을 추적하는 시스템이 필요하다. 사람의 기억은 신뢰할 수 없으며, 구두 커뮤니케이션은 사라진다. 이슈 트래커는 팀의 **공유 기억**이다.

### 4.2 단순함의 가치

Uncle Bob은 이슈 트래킹 시스템에 대해 한 가지 강력한 조언을 한다: **단순하게 유지하라.** 지나치게 복잡한 이슈 트래킹 시스템은 그 자체가 짐이 된다. 이슈를 등록하는 데 10분이 걸린다면, 사람들은 이슈를 등록하지 않게 된다.

좋은 이슈 트래커의 특성:

| 특성 | 설명 |
|------|------|
| 쉬운 등록 | 이슈를 만드는 데 30초 이상 걸리지 않아야 한다 |
| 명확한 상태 관리 | 열림/진행중/완료 정도의 단순한 상태면 충분하다 |
| 검색과 필터 | 원하는 이슈를 빠르게 찾을 수 있어야 한다 |
| 투명성 | 모든 팀원이 모든 이슈를 볼 수 있어야 한다 |

### 4.3 이슈 트래커는 소통 도구다

이슈 트래커의 본질은 관료주의적 통제가 아니라 **소통**이다. "이 버그를 누가 알고 있는가?", "이 기능의 진행 상태는 어떤가?", "다음에 무엇을 해야 하는가?"라는 질문에 답하는 것이 이슈 트래커의 목적이다.

프로세스를 위한 프로세스는 해롭다. 이슈 트래커에 수십 개의 필수 입력 필드(*우선순위, 심각도, 카테고리, 서브카테고리, 영향 범위, 담당 부서, 검토자, 승인자 등 과도한 메타데이터 필드를 요구하는 이슈 트래커를 경계해야 한다.*)와 복잡한 워크플로우를 설정하면, 도구는 소통을 돕는 것이 아니라 방해하게 된다.

## 5. 지속적 빌드 / 지속적 통합 (CI)

### 5.1 CI의 본질

지속적 통합(Continuous Integration)(*Continuous Integration — 개발자가 코드를 공유 저장소에 자주 통합하고, 매 통합마다 자동화된 빌드와 테스트를 실행하는 실천법*)은 전문 팀의 필수 실천법이다. 모든 커밋이 자동으로 빌드와 테스트를 트리거하고, 그 결과를 팀 전체에 즉시 공유한다. CI 서버는 프로젝트의 **심장 박동**과 같다.

### 5.2 CI의 핵심 원칙

1. **모든 커밋이 빌드를 트리거한다**: 코드를 저장소에 푸시하면, CI 시스템이 자동으로 빌드하고 테스트한다. 수동 빌드에 의존하면 안 된다.
2. **빌드 깨짐은 최우선 사항이다**: CI 빌드가 실패하면, 그것은 팀의 **최우선 과제**가 된다. 깨진 빌드를 방치한 채 다른 작업을 계속하는 것은 비전문적이다.
3. **빠른 피드백**: CI 빌드는 가능한 한 빨리 완료되어야 한다. 빌드에 1시간이 걸린다면, 피드백이 너무 느려 CI의 가치가 크게 감소한다.
4. **팀 전체의 책임**: CI는 특정 담당자의 업무가 아니라 팀 전체의 규율이다.

### 5.3 CI에서 CD로

Uncle Bob이 이 부록을 쓴 시점에는 CI가 중심이었지만, 현대에는 지속적 배포(Continuous Deployment)(*Continuous Deployment — CI를 확장하여 테스트를 통과한 코드를 자동으로 프로덕션 환경에 배포하는 실천법. Continuous Delivery와 혼용되기도 하지만, Continuous Delivery는 배포 가능한 상태를 유지하는 것이고, Continuous Deployment는 실제로 자동 배포하는 것이다.*)까지 확장된 개념이 일반적이다. 원칙은 동일하다: **자동화할 수 있는 것은 자동화하라.** 수동 프로세스는 실수의 온상이다.

> **핵심 통찰**: CI 서버가 "초록"(통과) 상태인 것은 팀의 건강 신호다. "빨강"(실패) 상태를 5분 이상 방치하는 것은 전문가 팀에서 용납되지 않는다.

## 6. 테스트 도구 (Testing Tools)

### 6.1 테스트 프레임워크

전문 개발자는 다양한 수준의 테스트 도구를 알고 활용한다:

| 테스트 유형 | 대표적 도구 (2011년 기준) | 현대적 도구 (참고) | 목적 |
|------------|------------------------|-------------------|------|
| 단위 테스트 | JUnit, NUnit | Jest, pytest, Go test | 개별 함수/클래스의 정확성 검증 |
| 인수 테스트 | FitNesse(*FitNesse — Uncle Bob이 직접 개발에 참여한 인수 테스트 프레임워크. 위키 기반으로 비즈니스 전문가도 테스트를 작성할 수 있도록 설계되었다.*), Cucumber | Cucumber, Playwright | 비즈니스 요구사항 충족 여부 검증 |
| 통합 테스트 | 다양한 프레임워크 | Testcontainers 등 | 컴포넌트 간 상호작용 검증 |
| 시스템 테스트 | Selenium | Cypress, Playwright | 전체 시스템의 동작 검증 |

### 6.2 도구보다 규율

테스트 도구의 세계에서 Uncle Bob이 가장 강조하는 점은, **도구보다 그것을 사용하는 규율이 중요하다**는 것이다. 최고급 테스트 프레임워크를 갖추고 있어도 테스트를 작성하지 않으면 무의미하다. 반대로, 단순한 assert 문만으로도 규율 있게 테스트하면 충분히 효과적이다.

테스트 규율의 핵심:

- **모든 코드에 테스트를 작성한다**: 테스트 없는 코드는 레거시 코드다.
- **테스트를 먼저 작성한다(TDD)**: 테스트가 설계를 이끈다.
- **테스트를 유지보수한다**: 깨진 테스트를 삭제하거나 무시하지 않는다.
- **테스트도 깨끗하게 유지한다**: 테스트 코드도 프로덕션 코드와 동일한 품질 기준을 적용한다.

## 7. UML / 모델링 도구

### 7.1 Uncle Bob의 UML 여정

Uncle Bob은 한때 UML(Unified Modeling Language)(*Unified Modeling Language — 소프트웨어 시스템의 구조와 동작을 시각적으로 표현하기 위한 표준 모델링 언어. 클래스 다이어그램, 시퀀스 다이어그램 등이 포함된다.*)의 열렬한 옹호자였다. 그는 UML 관련 책을 쓰고, UML을 활용한 설계 방법론을 가르쳤다. 그러나 수년간의 실무 경험을 거치면서 그의 입장은 크게 변했다.

### 7.2 다이어그램의 올바른 사용

Uncle Bob이 현재 권장하는 다이어그램 사용 원칙은 다음과 같다:

1. **다이어그램은 소통 도구다**: 다이어그램의 목적은 사람들 사이의 소통이지, 시스템의 공식 청사진이 아니다.
2. **화이트보드에 그려라**: CASE 도구(*Computer-Aided Software Engineering — 소프트웨어 개발의 다양한 단계를 자동화하거나 지원하는 도구. UML 모델링 도구(Rational Rose, Enterprise Architect 등)가 대표적이다.*)에 정교한 다이어그램을 그리는 데 시간을 쏟지 말라. 화이트보드나 종이에 빠르게 스케치하는 것으로 충분하다.
3. **30분 규칙**: 다이어그램에 30분 이상을 투자하기 전에 코딩을 시작하라. 다이어그램은 코드의 대체재가 아니다.
4. **일시적이어도 괜찮다**: 모든 다이어그램을 영구적으로 보존할 필요는 없다. 소통의 목적을 달성했으면 버려도 된다.

### 7.3 설계 vs 청사진

소프트웨어 설계에서 다이어그램을 "청사진"처럼 사용하려는 시도는 대부분 실패한다. 건축에서 청사진은 최종 산물(건물)과 1:1 대응하지만, 소프트웨어에서 UML 다이어그램은 코드와 1:1 대응하지 않는다. 다이어그램은 **근사치**이며, **코드가 진짜 설계**다.

| 접근 방식 | 설명 | 결과 |
|----------|------|------|
| 다이어그램 = 청사진 | 코딩 전에 모든 것을 다이어그램으로 설계 | 다이어그램과 코드가 괴리. 유지보수 불가능 |
| 다이어그램 = 소통 도구 | 아이디어를 빠르게 스케치하고 논의 | 효과적인 팀 소통. 코드에 집중 |

> **핵심 통찰**: 코드가 설계이고, 설계가 코드다. 다이어그램은 설계를 **논의하기 위한** 도구이지, 설계 그 **자체**가 아니다.

## 8. 변하지 않는 원칙

### 8.1 도구는 변하고 원칙은 남는다

이 부록에서 언급된 구체적 도구 중 상당수는 이미 시대에 뒤처졌거나 대체되었다. 그러나 도구 뒤에 있는 원칙은 여전히 유효하다:

| 변한 것 (도구) | 변하지 않은 것 (원칙) |
|---------------|---------------------|
| CVS → SVN → Git | 모든 코드를 버전 관리하라 |
| Eclipse → IntelliJ → VS Code | 도구에 유창해져라 |
| Bugzilla → Jira → Linear | 이슈를 추적하되 단순하게 유지하라 |
| Hudson → Jenkins → GitHub Actions | 모든 커밋을 자동으로 검증하라 |
| JUnit 3 → JUnit 5 → 다양한 프레임워크 | 테스트를 작성하는 규율을 가져라 |
| Rational Rose → draw.io | 다이어그램은 소통 도구로만 사용하라 |

### 8.2 도구에 대한 전문가의 태도

전문가는 도구에 대해 다음과 같은 태도를 취한다:

- **실용주의**: 유행이 아니라 실질적 가치에 기반하여 도구를 선택한다.
- **마스터리**: 선택한 도구를 피상적으로 사용하지 않고 깊이 익힌다.
- **유연성**: 더 나은 도구가 등장하면 전환할 준비가 되어 있다.
- **원칙 중심**: 도구 자체가 아니라 도구가 구현하는 원칙을 이해한다.
- **절제**: 도구를 위한 도구, 프로세스를 위한 프로세스를 경계한다.

## 요약

- **소스 제어**는 모든 전문 개발자의 기본이다. Git이 현재 사실상 표준이지만, 핵심은 자주 커밋하고, 의미 있는 메시지를 작성하며, 브랜치를 효과적으로 활용하는 것이다.
- **IDE/에디터** 선택보다 중요한 것은 선택한 도구에 대한 **유창성**이다. 도구는 생각과 코드 사이의 장벽이 아니라 다리가 되어야 한다.
- **이슈 트래커**는 소통 도구이지 관료주의 도구가 아니다. 단순하게 유지하여 팀이 실제로 사용하도록 한다.
- **CI/CD**는 프로젝트의 심장 박동이다. 모든 커밋을 자동 검증하고, 빌드 깨짐은 최우선으로 해결한다.
- **테스트 도구**에서 도구보다 중요한 것은 테스트를 작성하는 **규율**이다. 최고의 도구도 사용하지 않으면 무의미하다.
- **UML/다이어그램**은 소통 도구로 가볍게 사용해야 한다. 코드가 진짜 설계이며, 다이어그램은 30분 이상 투자하기 전에 코딩을 시작해야 한다.
- **도구는 변하지만 원칙은 남는다.** 특정 도구에 집착하지 말고, 도구 뒤의 원칙을 이해하는 것이 전문가의 태도다.

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: 전문 도구를 능숙하게 다루는 것은 프로의 기본 소양이다. 도구에 대한 투자는 전문성에 대한 투자다.
- **Chapter 4 (Coding)**: 코딩의 효율성과 품질은 도구 활용 능력에 크게 좌우된다. IDE의 리팩터링 기능, 코드 네비게이션 등은 깨끗한 코드를 작성하는 데 직접적으로 기여한다.
- **Chapter 5 (TDD)**: TDD는 테스트 프레임워크라는 도구 위에서 실천된다. 도구의 기능을 잘 아는 것이 TDD의 흐름을 원활하게 한다.
- **Chapter 7 (Acceptance Testing)**: 인수 테스트 도구(FitNesse, Cucumber 등)는 비즈니스와 개발 사이의 소통을 자동화하는 도구다.
- **Chapter 9 (Time Management)**: 도구에 대한 유창성은 곧 시간 절약이다. 단축키 하나를 익히는 것이 매일 수 분을 절약한다.
- **Chapter 14 (Mentoring, Apprenticeship, and Craftsmanship)**: 도구의 효과적 사용법은 멘토링을 통해 전수되는 중요한 지식이다. 장인은 자신의 도구를 사랑하고 깊이 이해한다.

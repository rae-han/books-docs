# Chapter 13: Teams and Projects (팀과 프로젝트)

## 핵심 질문

소프트웨어 프로젝트를 성공시키려면 프로젝트에 맞춰 팀을 구성해야 하는가, 아니면 이미 존재하는 팀에 프로젝트를 맡겨야 하는가? 응집된 팀(gelled team)의 가치는 무엇이며, 왜 그것을 보존해야 하는가?

---

## 1. 응집된 팀(The Gelled Team)

### 1.1 응집된 팀이란 무엇인가

Uncle Bob이 말하는 "응집된 팀(gelled team)"(*Gelled Team — "gel"은 액체가 굳어 하나의 덩어리가 되는 것을 의미한다. 응집된 팀은 개별 구성원이 하나의 통합된 유닛으로 작동하는 상태를 가리킨다. Tom DeMarco와 Timothy Lister의 "Peopleware"에서 처음 대중화된 개념이다.*)은 단순히 오래 함께 일한 팀이 아니다. 그것은 팀원들이 서로의 강점과 약점을 알고, 서로를 보완하며, 하나의 유기체처럼 움직이는 팀이다.

응집된 팀의 특징:

| 특징 | 설명 |
|------|------|
| **상호 이해** | 팀원들이 서로의 작업 스타일, 강점, 약점을 안다 |
| **암묵적 조율** | 세부 조율 없이도 각자 무엇을 해야 하는지 안다 |
| **상호 보완** | 한 사람의 약점을 다른 사람의 강점으로 자연스럽게 메운다 |
| **공유된 맥락** | 코드베이스, 비즈니스 도메인, 기술 스택에 대한 공유된 이해가 있다 |
| **심리적 안전감** | 실수를 인정하고, 도움을 요청하고, 반대 의견을 말할 수 있다 |
| **공동 소유** | 코드, 설계, 결정에 대해 팀 전체가 책임을 진다 |

### 1.2 응집의 과정: 시간이 필요하다

팀이 응집되기까지는 **상당한 시간**이 걸린다. Uncle Bob은 일반적으로 **6개월에서 1년** 정도의 시간이 필요하다고 본다.

응집의 단계:

```
형성(Forming)  →  갈등(Storming)  →  규범(Norming)  →  성과(Performing)
   1~2개월          2~4개월           2~3개월          지속
```

**형성(Forming)**: 팀원들이 서로를 알아가는 시기. 예의 바르지만 조심스럽다. 서로의 역량과 성향을 파악한다.

**갈등(Storming)**: 작업 방식, 코딩 스타일, 의사결정 방법에 대한 갈등이 표면화된다. 이 단계를 잘 넘기는 것이 중요하다. 많은 팀이 이 단계에서 좌절하거나 해체된다.

**규범(Norming)**: 갈등을 거쳐 팀만의 규범이 확립된다. 코딩 표준, 회의 문화, 의사결정 프로세스 등이 자리잡는다. 팀원들이 서로를 존중하고 신뢰하기 시작한다.

**성과(Performing)**: 팀이 최고의 생산성을 발휘하는 단계. 팀원들이 하나의 유기체처럼 움직인다. 이 단계에 도달한 팀은 **그 합보다 더 큰 가치**를 만들어낸다.

> **핵심 통찰**: 팀이 응집되기까지 투입되는 시간과 에너지는 **비용이 아니라 투자**이다. 응집된 팀이 생산해내는 가치는 투자의 몇 배를 상회한다. 이 투자를 허물어뜨리는 것 — 즉 응집된 팀을 해체하는 것 — 은 조직의 자산을 파괴하는 것이다.

### 1.3 응집된 팀의 생산성

응집된 팀은 단순히 "일을 잘하는 팀"이 아니다. 그것은 **개인 생산성의 합을 넘어서는 팀**이다. Uncle Bob은 응집된 팀이 비응집 팀 대비 **2~5배의 생산성**을 보인다고 주장한다.

이 초과 생산성은 어디서 오는가:

| 원천 | 설명 |
|------|------|
| **의사소통 비용 감소** | 서로를 알기 때문에 설명과 조율에 드는 시간이 최소화된다 |
| **결함 조기 발견** | 팀원들이 서로의 작업을 자연스럽게 검토하여 결함이 빨리 발견된다 |
| **재작업 감소** | 공유된 맥락 덕분에 오해와 잘못된 가정이 줄어든다 |
| **위험 분산** | 한 사람이 빠져도 다른 팀원이 즉시 대체할 수 있다 |
| **자기 조직화** | 관리자의 개입 없이도 팀이 스스로 작업을 분배하고 조율한다 |
| **사기(morale) 향상** | 좋은 팀에서 일하는 즐거움이 동기부여가 되어 더 열심히 일한다 |

> **Uncle Bob의 경험**: Uncle Bob은 자신이 경험한 최고의 팀을 회상한다. 그 팀은 약 8명의 개발자로 구성되어 있었고, 1년 이상 함께 일하며 완전히 응집된 상태였다. "우리는 말하지 않아도 서로 무엇을 해야 하는지 알았다. 누군가가 막히면 물어보기도 전에 다른 사람이 와서 도왔다. 코드 리뷰는 형식적 절차가 아니라 자연스러운 대화였다. 그 팀에서의 1년은 내 커리어의 다른 어떤 3년보다 생산적이었다."

## 2. 프로젝트 할당의 잘못된 방식(The Wrong Approach)

### 2.1 프로젝트 중심 사고(Project-Centric Thinking)

대부분의 조직이 따르는 전통적 접근법은 **프로젝트 중심 사고**이다:

```
새 프로젝트 등장
    ↓
프로젝트에 필요한 기술과 인원 파악
    ↓
여러 팀에서 적합한 사람을 뽑아 새 팀 구성
    ↓
프로젝트 수행
    ↓
프로젝트 완료
    ↓
팀 해체, 각자 원래 팀으로 복귀 또는 다른 프로젝트에 배정
```

이 접근법은 직관적이고 합리적으로 보인다. 프로젝트의 필요에 맞게 최적의 인력을 배치하는 것이니까. 하지만 Uncle Bob은 이것이 **근본적으로 잘못된 접근**이라고 주장한다.

### 2.2 프로젝트 중심 사고의 문제점

| 문제 | 상세 설명 |
|------|----------|
| **응집 파괴** | 프로젝트가 끝날 때마다 응집된 팀이 해체된다. 6개월~1년간의 투자가 사라진다 |
| **반복되는 형성 비용** | 새 프로젝트마다 팀이 형성-갈등-규범 단계를 다시 거쳐야 한다 |
| **예측 불가능한 속도** | 새 팀은 자신의 속도(velocity)(*Velocity — 애자일 개발에서 팀이 일정 기간(보통 1~2주의 스프린트) 동안 완료하는 작업의 양을 나타내는 지표. 팀이 오래 함께 일할수록 속도가 안정적이 되어 예측이 정확해진다. Chapter 10(Estimation)과 관련된다.*)를 모른다. 추정이 불확실해진다 |
| **지식 유실** | 팀이 해체되면 팀이 축적한 암묵적 지식(도메인 이해, 기술적 맥락)이 흩어진다 |
| **관계 리셋** | 팀원 간의 신뢰와 이해가 매번 처음부터 다시 시작된다 |
| **소속감 부재** | "내 팀"이라는 느낌이 없으므로 팀에 대한 헌신이 약하다 |

### 대화: 프로젝트 중심 사고의 결과

> **관리자**: "Project Alpha가 끝났으니, 김 개발자는 Project Beta로, 이 개발자는 Project Gamma로 가세요."
>
> **개발자 (속마음)**: *(다시 새로운 사람들과 처음부터... 또 3개월은 서로 적응하느라 비효율적이겠지. Alpha 팀에서 쌓은 호흡이 아까운데.)*
>
> **개발자 (프로)**: "Alpha 팀이 지금 정말 잘 돌아가고 있는데, 이 팀을 유지한 채로 Beta나 Gamma를 맡기는 건 어떨까요? 새 팀을 만들면 최소 3개월은 적응 기간이 필요하고, 그 동안 생산성이 크게 떨어질 것입니다."

## 3. 올바른 접근: 팀 중심 사고(Team-Centric Thinking)

### 3.1 팀에 프로젝트를 할당하라

Uncle Bob이 주장하는 접근법은 정반대이다: **팀을 프로젝트에 맞추지 말고, 프로젝트를 팀에 맞춰라.**

```
응집된 팀이 존재한다
    ↓
새 프로젝트가 등장한다
    ↓
기존 응집된 팀에 프로젝트를 할당한다
    ↓
프로젝트 완료
    ↓
팀은 유지되고, 다음 프로젝트를 받는다
```

이 접근법의 핵심 차이:

| 프로젝트 중심 | 팀 중심 |
|-------------|---------|
| 프로젝트가 일시적이고, 팀도 일시적이다 | 팀은 영속적이고, 프로젝트가 일시적이다 |
| 프로젝트 시작 시 팀을 구성한다 | 이미 존재하는 팀에 프로젝트를 할당한다 |
| 프로젝트 종료 시 팀을 해체한다 | 프로젝트가 끝나도 팀은 유지된다 |
| 매번 새로운 팀 형성 비용을 치른다 | 응집 비용을 한 번만 치르고 지속적으로 이득을 얻는다 |
| 매번 불확실한 속도로 시작한다 | 예측 가능한 속도로 즉시 시작한다 |

### 3.2 팀 중심 모델의 이점

**1. 즉시 생산적(Immediately Productive)**

응집된 팀은 새 프로젝트를 받으면 거의 즉시 생산적으로 일을 시작할 수 있다. 팀 형성 단계가 필요 없기 때문이다. 물론 도메인 학습 시간은 필요하지만, **팀 자체의 워밍업**은 이미 완료되어 있다.

**2. 예측 가능한 속도(Predictable Velocity)**

응집된 팀은 자신의 속도를 알고 있다. "우리 팀은 스프린트당 평균 40 포인트를 처리한다"와 같은 데이터가 있으면 프로젝트 기간 추정이 훨씬 정확해진다. 새로 구성된 팀은 이 데이터가 없으므로 추정이 불확실하다.

**3. 축적된 지식(Accumulated Knowledge)**

팀은 프로젝트를 수행하면서 기술적 지식과 도메인 지식을 축적한다. 이 지식은 다음 프로젝트에도 자산이 된다. 팀을 해체하면 이 지식 자산이 흩어진다.

**4. 높은 사기(High Morale)**

좋은 팀에서 일하는 것은 개발자에게 큰 동기부여가 된다. 팀이 해체될 걱정 없이 장기적으로 팀에 헌신할 수 있으면 더 깊은 유대감과 책임감이 형성된다.

### 3.3 하나의 팀, 여러 프로젝트

Uncle Bob은 응집된 팀이 **동시에 여러 프로젝트를 수행**할 수 있다고 주장한다. 이것은 프로젝트 중심 사고에서는 이상하게 들릴 수 있지만, 팀 중심 사고에서는 자연스러운 귀결이다.

응집된 팀은 작업을 분배하고 조율하는 능력이 이미 갖춰져 있으므로, 여러 프로젝트의 작업을 효과적으로 관리할 수 있다. 마치 한 팀이 여러 기능을 동시에 개발하는 것과 같은 원리이다.

물론 한계는 있다. 팀의 크기와 프로젝트의 규모에 따라 동시에 처리할 수 있는 프로젝트의 수가 결정된다 (이것은 Section 4에서 다룬다).

## 4. 이상적인 팀 구성(Ideal Team Composition)

### 4.1 Uncle Bob의 제안

Uncle Bob은 이상적인 팀의 구성을 구체적으로 제안한다:

| 역할 | 인원 | 책임 |
|------|------|------|
| **프로그래머** | ~7명 | 코드 설계, 구현, 단위 테스트, 리팩토링 |
| **테스터** | ~2명 | 인수 테스트 작성, 탐색적 테스트, 품질 보증 |
| **분석가(*Analyst — 비즈니스 요구사항을 분석하여 개발팀이 이해할 수 있는 형태로 변환하는 역할. 비즈니스 분석가(BA), 제품 소유자(PO), 기획자와 유사한 역할이다.*)** | ~2명 | 요구사항 분석, 사용자 스토리 작성, 비즈니스 규칙 정의 |
| **프로젝트 매니저** | 1명 | 일정 관리, 이해관계자 소통, 장애물 제거 |
| **합계** | ~12명 | |

### 4.2 역할 간의 상호작용

이 팀 구성에서 각 역할은 다음과 같이 상호작용한다:

```
          [분석가 2명]
          요구사항 정의
               ↓
   [테스터 2명] ←→ [프로그래머 7명]
   인수 테스트 작성    코드 구현
               ↑
      [PM 1명]
   조율 및 장애물 제거
```

- **분석가**는 비즈니스 요구사항을 사용자 스토리로 변환한다. 프로그래머 및 테스터와 긴밀하게 협력하여 요구사항을 정교화한다.
- **테스터**는 분석가와 함께 인수 테스트를 작성한다 (Chapter 7). 프로그래머가 코드를 작성하기 **전에** 인수 테스트가 정의되어야 한다.
- **프로그래머**는 인수 테스트를 통과시키는 코드를 작성한다. TDD(Chapter 5)를 따르며, 페어 프로그래밍(Chapter 12)을 실천한다.
- **PM**은 이 모든 과정을 조율하고, 외부 이해관계자와의 소통을 담당하며, 팀의 장애물을 제거한다.

### 4.3 이 비율의 근거

Uncle Bob이 제안하는 비율(7:2:2:1)의 근거:

| 비율 | 근거 |
|------|------|
| **프로그래머 7명** | 핵심 생산 인력. 페어 프로그래밍을 하면 3.5개의 페어가 동시에 작업할 수 있어 충분한 처리량이 나온다 |
| **테스터 2명** | 인수 테스트 작성과 탐색적 테스트를 병행하기에 적절한 인원. 프로그래머 3~4명당 테스터 1명 |
| **분석가 2명** | 요구사항 분석과 정교화에 집중. 프로그래머가 구현하는 속도에 맞춰 요구사항을 준비한다 |
| **PM 1명** | 12명 규모의 팀을 조율하기에 적절. 더 큰 팀이면 PM이 추가로 필요할 수 있다 |

### 4.4 동시 처리 가능한 프로젝트 수

Uncle Bob은 이 규모(12명)의 응집된 팀이 프로젝트 규모에 따라 **1~3개의 프로젝트를 동시에 처리**할 수 있다고 본다:

| 프로젝트 규모 | 동시 처리 가능 수 | 설명 |
|-------------|-----------------|------|
| 대규모 | 1개 | 팀 전체가 하나의 프로젝트에 집중한다 |
| 중규모 | 2개 | 프로그래머를 3~4명씩 나누어 각 프로젝트에 배정한다 |
| 소규모 | 3개 | 프로그래머를 2~3명씩 나누어 각 프로젝트에 배정한다 |

중요한 것은 **팀이 쪼개지는 것이 아니라 팀 안에서 작업이 분배되는 것**이라는 점이다. 팀은 하나로 유지되고, 팀의 규율, 문화, 소통 체계도 하나로 유지된다.

## 5. 응집된 팀의 속도(Velocity)

### 5.1 속도가 중요한 이유

소프트웨어 프로젝트에서 가장 어려운 질문 중 하나는 **"언제 끝나는가?"**이다. Chapter 10(Estimation)에서 다뤘듯이, 추정은 본질적으로 불확실하다. 하지만 응집된 팀은 이 불확실성을 **크게 줄일 수 있다.**

그 핵심이 바로 **속도(velocity)**이다.

### 5.2 속도의 안정화

새로 구성된 팀의 속도는 불안정하다:

```
새 팀의 속도 추이:
스프린트 1: 15 포인트
스프린트 2: 35 포인트
스프린트 3: 20 포인트
스프린트 4: 40 포인트
스프린트 5: 25 포인트
→ 편차가 크고, 예측이 어렵다
```

반면, 응집된 팀의 속도는 안정적이다:

```
응집된 팀의 속도 추이:
스프린트 1: 38 포인트
스프린트 2: 42 포인트
스프린트 3: 40 포인트
스프린트 4: 37 포인트
스프린트 5: 41 포인트
→ 평균 ~40, 편차가 작고, 예측이 정확하다
```

### 5.3 속도를 통한 예측

안정적인 속도는 **신뢰할 수 있는 예측**을 가능하게 한다:

- 총 작업량이 200 포인트로 추정된 프로젝트
- 팀의 속도가 스프린트당 40 포인트
- 예상 소요: 약 5 스프린트 (불확실성 범위: 4~6 스프린트)

이 수준의 예측 정확도는 새 팀에서는 불가능하다. 새 팀은 자신의 속도를 모르기 때문이다.

> **핵심 통찰**: 프로젝트의 일정 추정이 부정확한 이유 중 하나는 **추정하는 팀이 아직 형성 단계에 있기 때문**이다. 응집된 팀에게 프로젝트를 할당하면 추정의 정확도가 크게 향상된다. 이것만으로도 팀 중심 사고를 도입할 충분한 이유가 된다.

## 6. 응집된 팀을 보존하라

### 6.1 관리자에 대한 호소

Uncle Bob은 관리자들에게 강력하게 호소한다: **응집된 팀을 해체하지 마라.**

> **Uncle Bob의 경험**: Uncle Bob은 자신이 속했던 최고의 팀이 경영진의 결정으로 해체된 경험을 이야기한다. 프로젝트가 성공적으로 완료된 후, 경영진은 각 팀원을 다른 프로젝트에 배정했다. 논리적으로는 합리적으로 보였다 — 각 프로젝트에 가장 적합한 인력을 배치하려는 것이었다. 하지만 결과는 참담했다. "우리 팀은 함께 일하면서 놀라운 생산성을 보여줬다. 하지만 해체된 후, 각자가 배정된 새 팀에서는 평범한 개발자에 불과했다. 우리의 생산성은 개인의 능력이 아니라 **팀의 화학작용(chemistry)**에서 나온 것이었다. 경영진은 인력을 재배치했다고 생각했지만, 실제로는 가장 귀중한 자산을 파괴한 것이다."

### 6.2 왜 관리자들은 팀을 해체하는가

| 이유 | Uncle Bob의 반론 |
|------|-----------------|
| "프로젝트가 끝났으니 팀도 해체해야 한다" | 팀은 프로젝트보다 오래 살아야 한다. 다음 프로젝트를 팀에 할당하라 |
| "새 프로젝트에 특정 전문가가 필요하다" | 전문가 한 명을 뽑아가면, 그 전문가가 속한 팀의 응집이 깨진다. 전문가를 컨설턴트로 초빙하되 팀을 해체하지 마라 |
| "자원을 최적으로 배분해야 한다" | 개인을 자원으로 보지 마라. 팀을 자원으로 보라. 팀 단위로 프로젝트를 할당하라 |
| "모든 프로젝트에 최고 인력이 필요하다" | 최고의 생산성은 최고의 개인이 아니라 최고의 팀에서 나온다 |

### 6.3 팀원 교체의 비용

응집된 팀에서 한 명이 빠지더라도 상당한 영향이 있다. 새 팀원이 합류하면:

- **학습 기간**: 새 팀원이 코드베이스, 도메인, 팀 규범을 배우는 데 1~3개월이 소요된다
- **속도 저하**: 팀의 전체 속도가 일시적으로 떨어진다. 새 팀원을 돕느라 기존 팀원의 시간이 빠진다
- **동적 변화**: 팀의 관계 동학(dynamics)이 변한다. 새 팀원과의 새로운 균형을 찾아야 한다

한두 명의 교체는 팀이 흡수할 수 있다. 하지만 팀의 절반 이상이 바뀌면 사실상 새 팀이 되어 형성 단계부터 다시 시작해야 한다.

## 7. 프로의 책임: 팀에 기여하기

### 7.1 개인으로서의 책임

프로 개발자는 자신이 속한 팀의 응집에 기여할 책임이 있다. 이것은 관리자만의 책임이 아니다. 각 개인이 할 수 있는 것:

1. **관계를 구축하라** — 팀원과의 관계에 시간과 에너지를 투자하라. 코드만 짜지 말고, 사람과도 일하라.
2. **팀의 응집을 도우라** — 갈등(Storming) 단계에서 건설적으로 참여하라. 갈등을 피하지 말고, 해결하라.
3. **지식을 공유하라** — Chapter 12에서 다뤘듯이, 혼자만 아는 것은 팀에 해를 끼친다. 페어 프로그래밍, 코드 리뷰, 문서화를 통해 지식을 퍼뜨려라.
4. **팀 규범을 따르라** — 팀이 합의한 코딩 표준, 회의 문화, 프로세스를 존중하고 따르라. 자기만의 방식을 고집하지 마라.
5. **팀의 유지를 옹호하라** — 경영진이 팀을 해체하려 할 때, 팀의 가치를 설명하고 유지를 주장하라.

### 7.2 좋은 협업자가 되는 것의 의미

Chapter 12(Collaboration)에서 다룬 협업 원칙이 여기서 팀 수준으로 확장된다:

| 개인 수준 협업 (Chapter 12) | 팀 수준 협업 (Chapter 13) |
|---------------------------|-------------------------|
| 코드를 공유한다 | 팀의 공동 코드 소유를 실천한다 |
| 비즈니스를 이해한다 | 팀 전체가 도메인 지식을 공유한다 |
| 페어 프로그래밍을 한다 | 팀 내에서 자유롭게 페어를 구성한다 |
| 도움을 주고받는다 | 팀이 자기 조직화하여 문제를 해결한다 |

### 대화: 팀 해체에 대한 옹호

> **관리자**: "Project Beta에 경험 많은 개발자가 필요합니다. Alpha 팀에서 두 명을 차출하겠습니다."
>
> **개발자 (비프로)**: "네, 알겠습니다." *(불만이 있지만 말하지 않는다)*
>
> **개발자 (프로)**: "Alpha 팀이 지금 응집 상태에 도달해서 매우 높은 생산성을 보이고 있습니다. 두 명을 차출하면 Alpha 팀의 생산성이 30% 이상 떨어질 수 있고, Beta 팀도 새 팀원과 적응하느라 처음 2~3개월은 기대한 생산성이 나오지 않을 것입니다. 대신 Alpha 팀 전체에 Beta 프로젝트를 추가로 할당하는 건 어떨까요? 또는 Beta 팀에 컨설팅 형태로 도움을 주는 방법도 있습니다."

## 8. 현실적 제약과 타협

### 8.1 이상 vs 현실

Uncle Bob의 팀 중심 사고는 이상적이지만, 현실의 조직에서는 여러 제약이 있다:

- **예산**: 프로젝트 단위로 예산이 배정되는 조직에서 팀을 영속적으로 유지하기 어려울 수 있다
- **인력 풀 크기**: 소규모 회사에서는 하나의 응집된 팀을 유지할 인원이 부족할 수 있다
- **전문성 요구**: 특정 프로젝트에 팀이 보유하지 않은 전문 기술이 필요할 수 있다
- **조직 문화**: 프로젝트 중심 사고가 깊이 뿌리박힌 조직에서 패러다임을 바꾸기 어렵다

### 8.2 점진적 전환

완전한 팀 중심 모델로의 전환이 어렵다면, 다음과 같은 점진적 접근을 시도할 수 있다:

1. **핵심 팀 유지**: 팀의 전체가 아니더라도, 핵심 멤버(3~5명)는 프로젝트 간에 유지한다
2. **팀 간 이동 최소화**: 부득이하게 인원을 이동시킬 때는 팀당 1명 이하로 제한한다
3. **충분한 전환 기간**: 팀원을 이동시킬 때는 최소 2주의 인수인계(*Handoff — 한 팀원이 다른 팀원에게 업무, 지식, 맥락을 전달하는 과정. 문서화만으로는 부족하고 페어 프로그래밍이나 셰도잉(*Shadowing — 새 팀원이 기존 팀원의 작업을 옆에서 관찰하며 배우는 방식. 멘토링의 한 형태이다.*)을 통해 해야 효과적이다.*) 기간을 둔다
4. **팀 성과 측정**: 팀 단위의 성과(속도, 품질, 고객 만족도)를 추적하여 응집의 가치를 데이터로 보여준다

## 실전 시나리오

### 시나리오 1: 성공적 프로젝트 후 팀 해체 위기

**상황**: 8개월간 함께 일하며 완전히 응집된 6명의 팀이 프로젝트를 성공적으로 완료했다. 관리자가 각 멤버를 다른 프로젝트에 배정하려 한다.

**비프로 대응**:
> "네, 알겠습니다." *(각자 새 프로젝트로 흩어진다. 3개월 후 각자의 새 팀에서 적응하느라 고생한다. 이전 팀에서의 놀라운 생산성은 다시는 재현되지 않는다.)*

**프로 대응**:
> "이 팀이 지난 8개월간 달성한 성과를 보여드리겠습니다. [데이터 제시] 이 생산성은 개인의 능력이 아니라 팀의 응집에서 나온 것입니다. 팀을 해체하면 이 자산을 잃습니다. 대신 다음 프로젝트를 이 팀에 할당하는 것을 제안합니다. 팀의 도메인이 다르더라도, 응집된 팀의 적응 속도는 새 팀의 형성 속도보다 훨씬 빠릅니다."

**원칙**: 응집된 팀은 조직의 자산이다. 프로는 이 자산을 보존하기 위해 적극적으로 옹호한다.

### 시나리오 2: 새 프로젝트, 새 팀 구성 압박

**상황**: 대규모 신규 프로젝트가 시작된다. 관리자가 여러 팀에서 "최고의 인력"을 뽑아 드림팀을 구성하려 한다.

**비프로 대응**:
> "드림팀에 선발되다니 영광입니다!" *(3개월 후, "드림팀"은 여전히 형성-갈등 단계에 있고, 각자의 기존 팀은 핵심 인력을 잃어 고전하고 있다.)*

**프로 대응**:
> "각 팀의 최고 인력을 뽑아오면, 새 팀이 응집되기까지 최소 6개월이 필요합니다. 그 사이에 기존 팀들도 약화됩니다. 대신 현재 가장 응집된 팀에 이 프로젝트를 통째로 맡기는 건 어떨까요? 부족한 전문성은 외부 컨설턴트나 팀 간 협력으로 보완할 수 있습니다."

**원칙**: "드림팀"은 환상이다. 진짜 최고의 팀은 함께 오래 일한 팀이다.

### 시나리오 3: 팀의 속도를 무시하는 일정

**상황**: 관리자가 팀의 과거 속도 데이터를 무시하고 경영진의 기대에 맞춘 일정을 제시한다. "지난번보다 50% 빠르게 해야 합니다."

**비프로 대응**:
> "네, 노력해 보겠습니다." *(결과: 마감 미달, 품질 저하, 팀 사기 저하)*

**프로 대응**:
> "우리 팀의 지난 6개월 속도 데이터를 보면, 스프린트당 평균 40 포인트입니다. 이 프로젝트는 약 200 포인트로 추정되므로 5 스프린트가 필요합니다. 50% 빠르게 하려면 범위를 줄이거나 인원을 추가해야 하는데, 인원 추가는 오히려 초기에 속도를 떨어뜨릴 수 있습니다(브룩스의 법칙(*Brooks's Law — "지연되고 있는 소프트웨어 프로젝트에 인력을 추가하면 더 늦어진다." Fred Brooks가 "The Mythical Man-Month"(1975)에서 제시한 원칙. 새 인원의 학습 비용과 의사소통 오버헤드가 추가 인력의 이득을 상쇄하기 때문이다.*)). 가장 현실적인 방안은 핵심 기능부터 우선 배포하는 것입니다."

## 프로의 원칙

| 원칙 | 설명 |
|------|------|
| **응집된 팀은 자산이다** | 팀이 응집되기까지 6개월~1년의 투자가 필요하다. 이 투자를 파괴하지 마라 |
| **팀에 프로젝트를 할당하라** | 프로젝트에 팀을 맞추지 말고, 기존의 응집된 팀에 프로젝트를 맡겨라 |
| **팀의 속도를 신뢰하라** | 응집된 팀의 속도 데이터는 가장 신뢰할 수 있는 추정의 기반이다 |
| **"드림팀"은 환상이다** | 개인 능력의 합보다 팀의 화학작용이 중요하다. 응집된 팀을 해체하고 "최고 인력"을 재조합하는 것은 가치를 파괴하는 행위이다 |
| **팀 보존을 옹호하라** | 프로는 관리자에게 팀의 가치를 설명하고, 팀의 유지를 적극적으로 주장한다 |
| **팀의 일원이 되어라** | 개인으로서 팀의 응집에 기여할 책임이 있다. 관계를 구축하고, 지식을 공유하고, 팀 규범을 따르라 |

## 요약

- **응집된 팀(gelled team)**은 서로의 강점과 약점을 알고, 하나의 유기체처럼 움직이는 팀이다. 이 상태에 도달하기까지 6개월~1년이 걸린다.
- **응집된 팀의 생산성은 개인의 합을 넘어선다.** 의사소통 비용 감소, 결함 조기 발견, 재작업 감소, 위험 분산 등이 초과 생산성의 원천이다.
- **전통적인 프로젝트 중심 사고 — 프로젝트마다 팀을 새로 구성하는 것 — 는 잘못되었다.** 매번 응집 비용을 새로 치르고, 예측 불가능한 속도로 시작하며, 축적된 지식을 잃는다.
- **올바른 접근은 팀 중심 사고 — 응집된 팀에 프로젝트를 할당하는 것 — 이다.** 팀은 영속적이고, 프로젝트가 일시적이어야 한다.
- **이상적인 팀은 약 12명**: 프로그래머 7명, 테스터 2명, 분석가 2명, PM 1명으로 구성된다. 프로젝트 규모에 따라 1~3개 프로젝트를 동시에 수행할 수 있다.
- **응집된 팀의 속도(velocity)는 안정적이고 예측 가능하다.** 이것은 신뢰할 수 있는 추정의 기반이 된다.
- **응집된 팀을 해체하는 것은 조직의 자산을 파괴하는 것이다.** 프로는 팀의 보존을 적극적으로 옹호한다.
- **프로는 팀의 응집에 기여할 개인적 책임이 있다.** 관계를 구축하고, 지식을 공유하고, 좋은 협업자가 되어야 한다.

## 자기 점검

- [ ] 나는 현재 팀에서 응집된 상태에 가까운가? 팀원들과 서로의 강점, 약점을 알고 있는가?
- [ ] 우리 조직은 프로젝트마다 팀을 새로 구성하는가, 아니면 기존 팀에 프로젝트를 할당하는가?
- [ ] 응집된 팀이 프로젝트 완료 후 해체된 경험이 있는가? 그 결과는 어떠했는가?
- [ ] 우리 팀의 속도(velocity)를 알고 있는가? 이것을 추정에 활용하고 있는가?
- [ ] 팀의 유지를 위해 관리자에게 적극적으로 의견을 제시한 적이 있는가?
- [ ] 나는 팀의 응집에 기여하고 있는가 — 관계 구축, 지식 공유, 팀 규범 준수 — 아니면 방해하고 있는가?
- [ ] "드림팀"의 유혹에 빠진 적이 있는가? 최고의 개인을 모으면 최고의 팀이 된다는 환상을 갖고 있었는가?

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: 프로의 정의에는 팀에 대한 책임이 포함된다. 팀의 응집에 기여하고, 팀의 보존을 옹호하는 것이 프로의 의무이다.
- **Chapter 2 (Saying No)**: 팀 해체나 비현실적 일정에 대해 "아니오"라고 말하는 용기가 필요하다.
- **Chapter 3 (Saying Yes)**: 팀의 속도 데이터에 기반하여 약속을 하면 "예"라고 말할 때의 확신이 커진다.
- **Chapter 10 (Estimation)**: 응집된 팀의 안정적 속도는 추정의 정확도를 크게 높인다. 팀 중심 모델과 추정은 상호 보완적이다.
- **Chapter 11 (Pressure)**: 응집된 팀은 압박 상황에서도 더 효과적으로 대처한다. 서로를 알기 때문에 도움을 주고받기가 쉽다.
- **Chapter 12 (Collaboration)**: 이 챕터의 개인 수준 협업 원칙이 팀 수준으로 확장된다. 집단적 코드 소유, 페어 프로그래밍, 비즈니스 이해가 팀의 응집을 지탱한다.
- **Chapter 14 (Mentoring, Apprenticeship, and Craftsmanship)**: 응집된 팀은 자연스러운 멘토링 환경을 제공한다. 경험 있는 멤버가 새 멤버를 팀 안에서 양성한다.

# Chapter 12: Collaboration (협업)

## 핵심 질문

프로그래머는 본질적으로 혼자 코드를 작성하는 사람인가, 아니면 팀과 비즈니스를 위해 함께 일하는 사람인가? 코드 소유권은 개인에게 있어야 하는가, 팀에 있어야 하는가? 그리고 페어 프로그래밍은 정말로 시간 낭비인가?

---

## 1. 프로그래머 vs 사람(Programmers vs People)

### 1.1 혼자 일하고 싶은 본능

많은 프로그래머가 이 직업을 선택한 이유 중 하나는 **기계와의 대화**를 좋아하기 때문이다. 컴퓨터는 예측 가능하고, 논리적이며, 감정적이지 않다. 사람은 그렇지 않다. 그래서 많은 프로그래머가 사람보다 코드를 상대하는 것을 선호한다.

이 본능 자체는 잘못이 아니다. 하지만 이 본능에 굴복하여 **협업을 회피하는 것**은 프로답지 못하다.

> **Uncle Bob의 경험**: Uncle Bob은 자신이 커리어 초기에 전형적인 "혼자 일하는 프로그래머"였다고 고백한다. 헤드폰을 끼고 코드에 몰두하며, 동료의 질문을 방해로 느꼈다. 회의를 시간 낭비로 여겼고, 다른 사람의 코드를 보는 것을 불쾌하게 느꼈다. 하지만 경험이 쌓이면서 깨달았다 — 자신이 혼자 만든 코드보다 팀과 함께 만든 코드가 언제나 더 나았다는 것을.

### 1.2 프로그래밍은 팀 활동이다

소프트웨어 개발은 혼자 하는 예술이 아니다. 상업적 소프트웨어 개발은 **팀 활동**이다. 아무리 뛰어난 개발자라도 혼자서는 의미 있는 규모의 소프트웨어를 만들 수 없다. 그리고 설령 만들 수 있다 해도, 그것을 유지보수하고, 발전시키고, 운영하는 것은 팀이 해야 한다.

| 혼자 일하는 프로그래머 | 팀과 함께하는 프로그래머 |
|---------------------|---------------------|
| 자기 코드에만 집중한다 | 전체 시스템을 이해한다 |
| 자기 방식만 사용한다 | 팀의 표준을 따른다 |
| 방해받기를 싫어한다 | 소통을 업무의 일부로 받아들인다 |
| 혼자 문제를 해결한다 | 어려운 문제를 함께 해결한다 |
| 지식이 개인에게 묶인다 | 지식이 팀 전체에 퍼진다 |

> **핵심 통찰**: 혼자 일하는 것을 선호하는 것은 성격의 문제이다. 하지만 **혼자 일하기를 고집하는 것은 프로페셔널리즘의 문제이다.** 프로는 자신의 선호와 팀의 필요 사이에서 팀의 필요를 우선시한다.

## 2. 프로그래머와 고용주(Programmers and Employers)

### 2.1 고용주에 대한 의무

프로 개발자는 고용주에게 다음을 빚지고 있다:

- **최선의 노력(Best Effort)**: 근무 시간 동안 최고의 생산성을 발휘한다
- **시간(Time)**: 약속한 시간을 성실하게 채운다 (소셜 미디어를 보며 시간을 보내는 것은 도둑질이다)
- **창의성(Creativity)**: 문제를 해결하기 위한 창의적 사고를 제공한다
- **비즈니스 이해(Business Understanding)**: 기술적 결정이 비즈니스에 미치는 영향을 이해한다

### 2.2 비즈니스를 이해하라

Uncle Bob이 특히 강조하는 것은 **비즈니스 도메인에 대한 이해**이다. 많은 프로그래머가 기술적 문제에만 관심을 가지고, 자신이 만드는 소프트웨어가 **어떤 비즈니스 문제를 해결하는지**에는 무관심하다.

이것은 프로답지 못하다. 프로 개발자는:

- **자신이 일하는 도메인을 이해한다** — 금융 소프트웨어를 만들면 금융의 기본을 안다, 의료 소프트웨어를 만들면 의료 워크플로우를 이해한다
- **비즈니스 용어를 사용할 수 있다** — 도메인 전문가와 같은 언어로 대화할 수 있다
- **기술적 결정의 비즈니스 영향을 설명할 수 있다** — "이 리팩토링이 향후 기능 추가 속도를 30% 높입니다"처럼 비즈니스 가치로 번역할 수 있다
- **요구사항에 의문을 제기할 수 있다** — "이 기능이 사용자에게 정말 필요한가요?"라고 물을 수 있다

### 대화: 비즈니스 무관심의 문제

> **PM**: "고객이 이번 주문에서 할인 쿠폰을 이중 적용했는데, 시스템이 허용해버렸어요."
>
> **개발자 (비프로)**: "쿠폰 중복 적용 제한이 요구사항에 없었어요. 명세에 적어주시면 구현하겠습니다."
>
> **개발자 (프로)**: "죄송합니다. 할인 정책에 대해 더 깊이 이해했어야 합니다. 쿠폰 중복 적용은 비즈니스 관점에서 당연히 막아야 하는 것이었는데, 제가 도메인을 충분히 파악하지 못했습니다. 지금 바로 수정하겠습니다. 추가로, 전체 할인 정책을 한번 정리해서 다른 빈틈이 없는지 확인하고 싶습니다."

> **핵심 통찰**: 프로 개발자는 **코드를 작성하는 사람이 아니라 문제를 해결하는 사람**이다. 코드는 문제 해결의 도구일 뿐이다. 문제를 이해하지 못하는 사람이 올바른 해결책을 만들 수 없다.

### 2.3 "내 일이 아니다" 증후군

"그건 요구사항에 없었어요", "그건 PM의 책임이에요", "그건 QA가 잡았어야 해요" — 이런 말은 프로에게서 나오지 않는다.

프로는 **팀의 성공이 곧 자신의 성공**이라는 것을 안다. 요구사항 누락이 PM의 실수라 하더라도, 그 누락을 발견할 수 있었던 개발자가 침묵한 것은 프로답지 못하다. 프로는 자기 역할의 경계를 넘어 **팀 전체의 성공에 기여**한다.

## 3. 프로그래머와 프로그래머(Programmers and Programmers)

### 3.1 코드 소유권의 독(Owned Code)

많은 조직에서 **코드 소유권(code ownership)**(*Code Ownership — 특정 코드 영역에 대해 한 개인이 변경 권한과 책임을 독점적으로 갖는 관행. Extreme Programming(XP)에서는 이를 반대하고 집단적 코드 소유(Collective Code Ownership)를 주장한다.*) 모델을 사용한다. 각 개발자가 특정 모듈이나 영역을 "소유"하고, 다른 사람은 그 코드를 함부로 변경할 수 없다.

Uncle Bob은 이 모델이 **독(poison)**이라고 주장한다. 개인 코드 소유권의 문제점:

| 문제 | 결과 |
|------|------|
| **지식 사일로** | 한 사람만 해당 코드를 이해한다. 그 사람이 떠나면 큰 위험이 발생한다 |
| **병목 현상** | 해당 영역을 변경하려면 반드시 소유자를 거쳐야 한다. 소유자가 바쁘면 전체 팀이 기다린다 |
| **코드 성벽** | 소유자가 자신의 영역을 "방어"하며, 다른 팀원의 개선 제안을 거부한다 |
| **비일관성** | 각 소유자가 자기 스타일로 코딩하여 코드베이스 전체의 일관성이 떨어진다 |
| **학습 차단** | 다른 영역의 코드를 만질 기회가 없어 팀원의 기술 성장이 제한된다 |

### 3.2 집단적 코드 소유(Collective Code Ownership)

Uncle Bob이 주장하는 대안은 **집단적 코드 소유**이다. 코드는 팀 전체의 것이며, 누구나 어떤 부분이든 변경할 수 있다.

집단적 코드 소유가 동작하기 위한 전제 조건:

1. **공유된 코딩 표준** — 팀 전체가 합의한 코딩 스타일과 관례를 따른다
2. **포괄적인 테스트** — 테스트가 안전망 역할을 하여 누구나 자신 있게 코드를 변경할 수 있다
3. **지속적 통합** — 변경이 즉시 통합되어 충돌이 조기에 발견된다
4. **코드 리뷰** — 변경 사항을 다른 팀원이 검토하여 품질을 유지한다
5. **신뢰** — 팀원의 능력과 의도를 신뢰한다

### 대화: 코드 소유권 갈등

> **개발자 A**: "제 모듈에 왜 커밋을 했나요? 변경이 필요하면 저한테 먼저 말해야죠."
>
> **개발자 B (비프로)**: "죄송합니다. 다음부터는 먼저 말하겠습니다." *(비효율적인 병목이 유지된다)*
>
> **개발자 B (프로)**: "급한 버그 수정이 필요했고, 당신이 휴가 중이어서 직접 수정했습니다. 코드 리뷰를 올려뒀으니 확인해주세요. 그리고 우리 팀의 코드 소유 정책에 대해 논의해보면 어떨까요? 지금처럼 개인 소유 모델은 병목이 발생하는 것 같습니다."

> **핵심 통찰**: **코드는 개인의 작품이 아니라 팀의 자산이다.** 코드에 자부심을 느끼는 것은 좋지만, 코드를 자기 영역으로 독점하는 것은 팀에 해를 끼친다.

### 3.3 페어 프로그래밍(Pair Programming)

Uncle Bob은 **페어 프로그래밍**(*Pair Programming — 두 명의 프로그래머가 하나의 워크스테이션에서 함께 작업하는 소프트웨어 개발 기법. 한 명(드라이버)이 코드를 작성하고, 다른 한 명(내비게이터)이 코드를 검토하며 전략적 방향을 제시한다. XP의 핵심 실천법 중 하나이다.*)을 강력하게 옹호한다. 페어 프로그래밍은 두 명의 개발자가 한 대의 컴퓨터 앞에 앉아 함께 코드를 작성하는 기법이다.

많은 개발자가 페어 프로그래밍에 대해 회의적이다. 흔한 반대 의견과 Uncle Bob의 반론:

| 반대 의견 | Uncle Bob의 반론 |
|-----------|-----------------|
| "두 명이 한 일을 하면 생산성이 반으로 줄어든다" | 실제 연구에 따르면 페어 프로그래밍의 생산성 저하는 15% 수준이며, 결함 감소와 지식 공유를 고려하면 순이익이다 |
| "혼자일 때 더 집중할 수 있다" | 혼자일 때 집중하는 것처럼 보이지만, 실제로는 산만해지는 시간이 많다. 페어일 때 진짜 집중이 유지된다 |
| "함께 일하면 불편하다" | 불편함은 초기에만 존재한다. 익숙해지면 오히려 혼자 코딩하는 것이 비효율적으로 느껴진다 |
| "나는 혼자가 더 빠르다" | 자기 평가는 종종 부정확하다. 페어로 만든 코드의 결함률을 비교해 보라 |

#### 페어 프로그래밍의 이점

1. **지식 공유(Knowledge Sharing)**: 시스템에 대한 지식이 팀 전체에 퍼진다. "버스 팩터"(*Bus Factor — "팀원 중 몇 명이 버스에 치여야 프로젝트가 멈추는가"라는 질문에서 유래한 지표. 버스 팩터가 1이면 한 명이 빠져도 프로젝트가 위험하다는 뜻이다. 높을수록 팀이 건강하다.*)가 높아진다.
2. **실시간 코드 리뷰(Real-time Code Review)**: 코드가 작성되는 즉시 리뷰가 이루어진다. 사후 코드 리뷰보다 훨씬 효과적이다.
3. **결함 감소(Defect Reduction)**: 두 쌍의 눈이 보면 결함이 줄어든다. 한 사람이 놓치는 것을 다른 사람이 잡아낸다.
4. **집중력 유지(Focus)**: 파트너가 옆에 있으면 소셜 미디어를 확인하거나 다른 데 정신이 팔리기 어렵다.
5. **작가의 블록 해소(Writer's Block)**: 코딩이 막혀서 진행이 안 될 때, 파트너와 대화하면 새로운 관점이 열린다.
6. **더 나은 설계(Better Design)**: 두 사람이 함께 고민하면 더 나은 설계 결정이 나온다.

### 3.4 소뇌 문제(The Cerebellum Problem)

Uncle Bob이 제시하는 독특한 관찰이 있다. 프로그래머가 혼자 코딩에 깊이 몰두하면, 의식(cerebrum)이 코드의 세부 사항에 집중하는 동안 **큰 그림을 놓치기 쉽다**는 것이다.

혼자 코딩할 때 흔히 발생하는 문제:

- 지역적으로 최적이지만 전체적으로 비최적인 설계를 만든다
- 자기 코드에 빠져서 더 좋은 접근법을 보지 못한다
- 해결책이 아닌 기술적 도전 자체에 매료된다 ("이 알고리즘 재미있다!" vs "이것이 문제를 해결하는 최선의 방법인가?")
- 사용자의 관점을 잊는다

페어 프로그래밍은 이 문제를 해결한다. 파트너가 "잠깐, 왜 이렇게 하는 거야?"라고 물어주면, 세부 사항에 매몰되었던 자신을 큰 그림으로 되돌릴 수 있다.

> **핵심 통찰**: 프로그래밍은 **세부 사항에 몰입하는 활동**이다. 그렇기 때문에 **큰 그림을 잃지 않도록 도와주는 파트너**가 필요하다. 페어 프로그래밍은 단순히 "두 명이 함께 코딩하는 것"이 아니라, 서로의 시야를 넓혀주는 장치이다.

## 4. 협업의 범위: 개발자만의 문제가 아니다

### 4.1 비즈니스와의 협업

프로그래머 간의 협업만이 협업의 전부가 아니다. Uncle Bob은 **비즈니스 측면과의 협업**도 동등하게 중요하다고 강조한다.

프로 개발자가 비즈니스와 협업하는 방법:

| 협업 대상 | 협업 방법 | 기대 결과 |
|----------|---------|----------|
| **PM/기획자** | 요구사항을 함께 분석하고, 기술적 관점에서 피드백을 제공한다 | 실현 가능하고 효율적인 요구사항 정의 |
| **QA** | 테스트 전략을 함께 수립하고, 인수 테스트를 공동 작성한다 | "완료"의 정의에 대한 합의 (Chapter 7) |
| **운영팀** | 배포 파이프라인과 모니터링을 함께 설계한다 | 안정적인 프로덕션 운영 |
| **경영진** | 기술적 결정의 비즈니스 영향을 투명하게 설명한다 | 정보에 기반한 의사결정 |
| **사용자** | 사용자 피드백을 직접 듣고, 사용자의 실제 워크플로우를 이해한다 | 사용자가 진짜로 필요한 기능 개발 |

### 4.2 "우리 vs 그들" 사고의 위험

많은 조직에서 개발팀과 비즈니스 팀 사이에 보이지 않는 벽이 존재한다. 개발자는 "비즈니스 사람들은 기술을 몰라"라고 생각하고, 비즈니스 측은 "개발자들은 비즈니스를 몰라"라고 생각한다.

이 벽을 허무는 것이 프로의 역할이다. "우리(개발) vs 그들(비즈니스)"의 사고방식은 다음과 같은 결과를 낳는다:

- 요구사항이 "벽 너머로 던져진다" → 오해와 재작업
- 기술적 결정이 비즈니스 맥락 없이 이루어진다 → 쓸모없는 기술적 우아함
- 비즈니스 결정이 기술적 현실 없이 이루어진다 → 비현실적 일정과 약속

### 대화: 벽을 허무는 소통

> **PM**: "이 기능 얼마나 걸려요?"
>
> **개발자 (비프로)**: "3주요." *(대화 끝. 비즈니스 맥락도 모르고, PM도 기술적 현실을 모른다.)*
>
> **개발자 (프로)**: "이 기능의 비즈니스 목표가 뭔가요? 어떤 사용자 문제를 해결하려는 건가요? (...대화...) 그러면 전체를 한 번에 만드는 것보다, 핵심 워크플로우부터 1주 만에 출시하고, 사용자 반응을 보면서 나머지를 결정하는 건 어떨까요? 비즈니스 가치를 더 빨리 전달할 수 있습니다."

## 5. 협업을 가로막는 장벽과 극복법

### 5.1 흔한 장벽들

| 장벽 | 증상 | 극복법 |
|------|------|--------|
| **내향성** | 대화를 피하고 혼자 일하려 한다 | 페어 프로그래밍을 정기적으로 실천한다. 처음에는 불편하지만 점차 익숙해진다 |
| **자존심** | 자기 코드가 최고라고 생각하고 다른 사람의 수정을 거부한다 | 집단적 코드 소유를 도입하고, 코드 리뷰를 의무화한다 |
| **두려움** | 자기 코드를 보여주거나 다른 사람의 코드를 수정하는 것을 두려워한다 | 안전한 피드백 문화를 만든다. 코드 리뷰를 비판이 아닌 학습의 기회로 정의한다 |
| **소유욕** | "내 코드"에 대한 집착 | 팀의 목표가 개인의 영역보다 중요하다는 것을 인식한다 |
| **의사소통 부재** | 팀원이 무엇을 하는지 모른다 | 일일 스탠드업(*Stand-up — 매일 15분 이내로 진행하는 짧은 팀 회의. 각자 어제 한 일, 오늘 할 일, 장애물을 공유한다. 서서 진행하여 짧게 끝내도록 한다. Chapter 9(Time Management)에서 회의 관리의 일부로 다룬다.*), 짧은 싱크 미팅을 통해 정보를 공유한다 |

### 5.2 협업의 습관 형성

협업은 기술이 아니라 **습관**이다. 다음과 같은 작은 습관들이 협업 문화를 만든다:

1. **아침에 팀원과 인사하라** — 사소하지만 관계의 시작이다
2. **점심을 함께 먹어라** — 비공식적 대화에서 중요한 정보가 공유된다
3. **매주 한 번 이상 페어 프로그래밍을 하라** — 반드시 모든 시간을 페어로 하지 않아도 된다. 주 1~2회만으로도 큰 차이를 만든다
4. **다른 사람의 코드를 읽어라** — 리뷰 요청이 없어도, 동료의 코드를 읽으면 시스템 이해도가 높아진다
5. **도움을 요청하라, 도움을 제안하라** — "도움이 필요하면 말해"가 아니라 "지금 힘들어 보이는데, 같이 볼까?"

## 6. 개인 소유 코드 vs 집단 소유 코드: 전환 전략

### 6.1 현실적 전환 단계

이미 개인 코드 소유 모델로 운영되고 있는 팀이 집단 소유 모델로 전환하는 것은 쉽지 않다. Uncle Bob의 조언에 기반한 현실적 전환 전략:

**1단계: 코드 리뷰 의무화**
- 모든 코드 변경에 최소 한 명의 리뷰어를 배정한다
- 이것만으로도 지식이 퍼지기 시작한다

**2단계: 교차 작업(Cross-work) 시작**
- 때때로 자기 영역이 아닌 코드를 수정하도록 의도적으로 배정한다
- 반드시 해당 영역의 기존 소유자가 리뷰한다

**3단계: 페어 프로그래밍 도입**
- 교차 작업 시 기존 소유자와 페어 프로그래밍으로 진행한다
- 자연스럽게 지식이 이전된다

**4단계: 공유 표준 확립**
- 팀 전체의 코딩 표준, 네이밍 관례, 아키텍처 가이드라인을 합의한다
- 개인 스타일의 차이로 인한 갈등을 줄인다

**5단계: 소유권 공식 해제**
- 코드 영역별 소유자를 폐지한다
- 누구나 어떤 코드든 변경할 수 있다
- 코드 리뷰와 테스트가 품질을 보장한다

### 6.2 집단 소유 모델의 전제 조건

집단적 코드 소유가 혼란이 아닌 생산성으로 이어지려면 반드시 다음이 갖춰져야 한다:

| 전제 조건 | 없으면 발생하는 문제 |
|----------|-------------------|
| **포괄적 테스트 스위트** | 누군가의 변경이 다른 부분을 깨뜨려도 알 수 없다 |
| **지속적 통합(CI)** | 통합 문제가 늦게 발견되어 대규모 충돌이 발생한다 |
| **합의된 코딩 표준** | 코드 스타일이 뒤섞여 가독성이 떨어진다 |
| **팀 간 신뢰** | "누가 내 코드를 망쳐놓았나"라는 비난이 발생한다 |
| **효과적인 코드 리뷰** | 품질 저하가 감지되지 않는다 |

## 실전 시나리오

### 시나리오 1: "내 모듈 건드리지 마세요"

**상황**: 긴급 버그 수정이 필요한데, 해당 코드의 "소유자"가 휴가 중이다.

**비프로 대응**:
> "그건 김 선임의 코드니까, 복귀할 때까지 기다려야 합니다." *(고객은 1주일간 버그의 영향을 받는다.)*

**프로 대응**:
> 다른 팀원이 코드를 분석하고, 테스트를 추가한 후, 수정을 적용한다. 코드 리뷰를 다른 팀원에게 요청하고, 김 선임이 복귀하면 변경 내용을 공유한다. "코드는 팀의 것이다. 누가 작성했든, 팀 전체가 유지보수할 수 있어야 한다."

**원칙**: 코드 소유권이 고객에게 피해를 주어서는 안 된다. 코드는 팀의 자산이다.

### 시나리오 2: "혼자가 더 빨라요"

**상황**: 복잡한 기능을 구현해야 하는데, 팀장이 페어 프로그래밍을 제안했다. 당신은 혼자 하는 게 더 빠르다고 생각한다.

**비프로 대응**:
> "저 혼자 하면 3일이면 되는데, 페어로 하면 오히려 느려질 것 같습니다. 혼자 하겠습니다." *(3일 후, 미묘한 버그가 여러 개 포함된 코드를 제출한다. 디버깅에 2일이 더 소요된다.)*

**프로 대응**:
> "솔직히 혼자가 더 편하긴 합니다. 하지만 이 기능은 복잡도가 높아서 페어로 하면 설계 품질이 더 나을 것 같습니다. 그리고 제가 이 영역만 아는 것보다 다른 팀원도 알게 되면 팀 전체에 도움이 됩니다. 페어로 진행하겠습니다."

**원칙**: 단기적 개인 생산성보다 장기적 팀 생산성이 중요하다. 페어 프로그래밍은 지식 공유와 품질 향상의 도구이다.

### 시나리오 3: 비즈니스를 모르는 개발자

**상황**: 전자상거래 플랫폼 개발자가 재고 관리 기능을 구현하면서, 비즈니스 로직을 이해하지 않고 순전히 기술적 명세만으로 구현했다.

**비프로 대응**:
> "명세에 있는 대로 정확히 구현했습니다. 비즈니스 로직은 PM이 책임지는 거 아닌가요?" *(결과: 재고가 음수로 내려가는 버그, 동시 주문 시 재고 초과 판매 등 비즈니스적으로 치명적인 결함 발생)*

**프로 대응**:
> 구현 전에 비즈니스 측과 재고 관리의 워크플로우를 논의한다. "재고가 0일 때 어떻게 해야 하나요? 동시 주문은 어떻게 처리하나요? 예약 재고 개념이 필요한가요?" 이런 질문을 통해 명세에 빠진 중요한 비즈니스 규칙을 발견한다.

**원칙**: 프로 개발자는 비즈니스 도메인을 이해한다. 기술적 명세만 따르는 것은 프로가 아니라 타이피스트이다.

---

## 프로의 원칙

| 원칙 | 설명 |
|------|------|
| **프로그래밍은 팀 활동이다** | 혼자 일하기를 선호하더라도, 팀과 비즈니스를 위해 적극적으로 협업한다 |
| **비즈니스를 이해하라** | 코드를 작성하는 사람이 아니라 문제를 해결하는 사람이 되어라. 도메인을 알아야 올바른 해결책을 만든다 |
| **코드는 팀의 것이다** | 개인 코드 소유권은 독이다. 집단적 코드 소유를 통해 지식을 공유하고 병목을 제거하라 |
| **페어 프로그래밍을 실천하라** | 지식 공유, 실시간 리뷰, 결함 감소, 집중력 유지를 위해 정기적으로 페어링하라 |
| **"내 일이 아니다"를 버려라** | 프로는 자기 역할의 경계를 넘어 팀 전체의 성공에 기여한다 |
| **벽을 허물어라** | "개발 vs 비즈니스"의 사고방식을 버리고, 하나의 팀으로 일하라 |

---

## 요약

- **많은 프로그래머가 혼자 일하기를 선호한다.** 하지만 프로그래밍은 팀 활동이며, 협업을 회피하는 것은 프로답지 못하다.
- **고용주에 대한 의무에는 비즈니스 이해가 포함된다.** 기술만 아는 개발자는 반쪽짜리 프로이다. 도메인을 이해해야 올바른 해결책을 만들 수 있다.
- **"내 일이 아니다" 증후군은 독이다.** 프로는 자기 역할의 경계를 넘어 팀 전체의 성공에 기여한다.
- **개인 코드 소유권은 지식 사일로, 병목, 비일관성을 만든다.** 집단적 코드 소유가 더 건강한 모델이다.
- **페어 프로그래밍은 시간 낭비가 아니다.** 지식 공유, 실시간 리뷰, 결함 감소, 집중력 유지의 이점이 비용을 상쇄한다.
- **소뇌 문제 — 혼자 코딩하면 세부 사항에 매몰되어 큰 그림을 놓친다.** 파트너가 시야를 넓혀준다.
- **비즈니스와의 협업도 개발자 간 협업만큼 중요하다.** "개발 vs 비즈니스" 사고방식을 버려야 한다.

## 다른 챕터와의 관계

- **Chapter 1 (Professionalism)**: 프로의 정의에 "팀을 위해 협업한다"는 요소가 포함된다. 혼자 일하기를 고집하는 것은 프로답지 못한 행동이다.
- **Chapter 2 (Saying No)**: 비즈니스를 이해하면 "아니오"라고 말할 때 더 설득력 있는 대안을 제시할 수 있다.
- **Chapter 5 (Test Driven Development)**: TDD는 집단적 코드 소유의 전제 조건이다. 테스트가 안전망을 제공하므로 누구나 코드를 변경할 수 있다.
- **Chapter 7 (Acceptance Testing)**: 인수 테스트는 개발자와 비즈니스 사이의 협업 도구이다. "완료"의 정의를 함께 합의한다.
- **Chapter 11 (Pressure)**: 압박 속에서 도움을 요청하고, 페어 프로그래밍을 활용하는 것은 이 챕터의 협업 원칙의 적용이다.
- **Chapter 13 (Teams and Projects)**: 이 챕터의 개인 수준 협업 원칙이 팀 수준으로 확장된다. 응집된 팀(gelled team)은 협업의 결실이다.
- **Chapter 14 (Mentoring, Apprenticeship, and Craftsmanship)**: 멘토링은 협업의 확장이다. 지식을 공유하는 것은 프로의 의무이다.

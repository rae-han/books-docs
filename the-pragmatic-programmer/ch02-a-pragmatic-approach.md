# Chapter 2: A Pragmatic Approach (실용주의 접근법)

## 핵심 질문

소프트웨어 개발의 모든 차원에 적용되는 보편적 접근법은 무엇인가? 좋은 설계의 단 하나의 뿌리(ETC)와 그로부터 파생되는 DRY·직교성·가역성은 어떻게 연결되는가? 예광탄과 프로토타입은 어떻게 다른가? 도메인 언어와 추정은 왜 실용주의의 핵심 기술인가?

> 개발 중에 이런 기본 원리들을 유념하라. 더 좋은, 더 빠른, 더 강력한 코드를 작성하게 될 것이다.

---

## Topic 8: 좋은 설계의 핵심 (The Essence of Good Design)

> **Tip 14**: 좋은 설계는 나쁜 설계보다 바꾸기 쉽다.

잘 설계된 것은 그것을 사용하는 사람에게 **적응하여 맞춰진다**. 코드에 적용하면, 잘 설계된 코드는 바뀜으로써 사용자에게 맞춰져야 한다 — 그래서 **ETC 원칙**을 따른다: **바꾸기 더 쉽게(Easier To Change)**.

> **핵심 통찰**: **세상의 모든 설계 원칙은 ETC의 특수한 경우**다. 왜 결합도를 줄이면 좋은가? 관심사가 분리돼 각각 바꾸기 쉬워지니까(ETC). 왜 단일 책임 원칙이 유용한가? 요구가 바뀌어도 모듈 하나만 바꾸면 되니까(ETC). 왜 이름 짓기가 중요한가? 코드를 바꾸려면 읽어야 하고, 좋은 이름은 읽기 쉬우니까(ETC).

- **ETC는 규칙이 아니라 가치(value)**다 — 갈림길에서 "내가 방금 한 일이 시스템을 바꾸기 쉽게 만들었나, 어렵게 만들었나?"를 파일 저장·테스트·버그 수정마다 물어라. 어떤 길이 더 나은지 모를 땐 **교체하기 쉽게**(결합도↓·응집도↑) 만들고, 엔지니어링 일지에 선택과 추측을 남겨 직관을 발전시켜라.

**도전 과제**
- 일상적으로 쓰는 설계 원칙이 무언가를 **바꾸기 쉽게** 만들려는 것인가?
- 에디터에 파일 저장 시 "ETC?" 팝업을 띄우도록 설정하고, 방금 작성한 코드가 바꾸기 쉬운지 생각해 보라.

## Topic 9: DRY: 중복의 해악 (The Evils of Duplication)

프로그래머는 늘 **유지 보수 모드**에 있다 — 우리의 이해는 날마다 바뀐다. 유지 보수를 하려면 지식의 표현을 찾아 바꿔야 하는데, 지식이 여러 곳에 중복되면 악몽이 시작된다.

> **Tip 15**: DRY — 반복하지 말라(Don't Repeat Yourself).

> **핵심 통찰**: DRY의 정의는 "**모든 지식은 시스템 내에서 단 한 번만, 애매하지 않게, 권위 있게 표현되어야 한다**"이다. DRY는 **복사-붙여넣기 금지가 아니라 지식·의도의 중복** 금지다(그것은 작고 자명한 일부일 뿐). 리트머스 시험지: 한 측면을 바꿀 때 여러 곳(코드+문서+DB 스키마)을 바꾸는가? 그렇다면 DRY하지 않다.

- **모든 코드 중복이 지식 중복은 아니다**: `validate_age`와 `validate_quantity`의 코드가 같아도 **표현하는 지식이 다르면**(우연히 규칙만 같음) 중복이 아니다.
- **DRY의 여러 형태**: 코드 중복(함수 추출), **문서화 중복**(모든 함수에 주석 다는 미신 — 주석과 코드가 어긋난다), **데이터 중복**(계산 가능한 필드는 저장하지 말고 계산하라), **표현상 중복**(API·스키마 — 코드 생성·명세로 완화), **개발자 간 중복**(같은 일을 하는 코드가 중복 — 소통으로 해결).

```java
// 나쁜 예: length가 start·end와 중복 (한쪽이 바뀌면 어긋남)
class Line { Point start; Point end; double length; }

// 좋은 예: length를 계산 필드로 (DRY)
class Line {
  Point start; Point end;
  double length() { return start.distanceTo(end); }
}
```

> 성능상 어쩔 수 없이 캐싱으로 DRY를 위배해야 한다면, **중복의 영향을 국소화**하라 — 바깥세상엔 위배를 노출하지 말고 클래스 내부 메서드가 고생하게. (버트런드 마이어의 **단일 접근 원칙** — 저장이든 계산이든 일관된 표기법으로 접근.)

> **Tip 16**: 재사용하기 쉽게 만들어라.

**직접 만드는 것보다 찾아 재사용하기 쉬운 환경**을 조성하라 — 쉽지 않으면 사람들은 안 하고, 지식 중복 위험을 감수하게 된다.

## Topic 10: 직교성 (Orthogonality)

직교성은 기하학에서 온 말로, 두 직선이 직각으로 만나면 **독립적(independent)**이다. 컴퓨터 과학에선 **결합도 줄이기(decoupling)** — 하나가 바뀌어도 다른 것에 영향이 없으면 직교한다.

> 점심으로 생선을 먹은 헬리콥터 조종사가 기절했다. 위키백과로 배운 네 조종간을 만져 보니, **하나를 움직이면 나머지 모두에 파생 효과**가 생겨 손발이 끊임없이 균형을 잡느라 바쁘다. 헬리콥터 조종은 단연코 직교적이지 않다 — 비직교 시스템은 변경·조정이 본질적으로 복잡하다.

> **Tip 17**: 관련 없는 것들 간에 서로 영향이 없도록 하라.

> **핵심 통찰**: 직교 시스템의 두 이점은 **생산성 향상**(변화 국소화, 재사용 촉진, 직교 컴포넌트 결합 시 M×N 기능)과 **리스크 감소**(감염된 코드 격리, 잘 안 깨짐, 테스트 쉬움, 벤더 종속↓)다. 설계가 직교적인지 확인하는 법: "한 기능의 요구가 대폭 바뀌면 **몇 개 모듈이 영향받는가?**" 직교적이면 답은 **'하나'**여야 한다.

- **기법**: '부끄럼쟁이(shy)' 코드(불필요한 것을 노출·의존하지 않음, 데메테르 법칙), **전역 데이터 회피**(싱글턴 남용 주의 — 불필요한 결합), 유사 함수 회피(전략 패턴), **제어할 수 없는 외부 식별자에 의존하지 말라**(전화번호·주민번호를 고객 식별자로 쓰면 체계가 바뀔 때 곤란). 문서에도 적용(내용 vs 표현 축 분리).

> 직교성(컴포넌트 간 의존도↓)은 DRY(중복↓)와 짝을 이룬다 — 하나 바꿀 때 네 가지가 이상해진다면 **헬리콥터의 악몽**이니, 리팩터링할 시간이다.

**도전 과제**
- GUI 프로그램과 조합 가능한 명령 줄 유틸리티의 차이를 생각해 보라 — 어느 쪽이 더 직교적이고 왜?
- 다중 상속(C++)·다중 인터페이스(자바)·믹스인(루비)이 직교성에 끼치는 영향은? 위임 vs 상속의 차이는?

## Topic 11: 가역성 (Reversibility)

> 당신이 가진 생각이 딱 하나밖에 없다면, 그것만큼 위험한 것은 없다.<br>— 에밀 오귀스트 샤르티에(Émile-Auguste Chartier)

"이건 이 방법으로만 해결할 수 있어"라는 근시안으로 프로젝트에 임하면, 중요한 결정은 쉽게 되돌릴 수 없어 큰 비용을 치른다. DRY·결합도 줄이기·외부 설정을 따르면 **되돌릴 수 없는 결정의 수**를 줄일 수 있다(초기에 늘 최선의 결정을 내리진 못하므로).

> **Tip 18**: 최종 결정이란 없다.

> **핵심 통찰**: 결정을 **돌에 새긴 것이 아니라 바닷가 모래 위에 쓴 글씨**라 생각하라 — 큰 파도가 언제든 지운다. DB를 하나의 서비스로 추상화하면 달리는 중에 말을 갈아탈 수 있고, 웹 앱이 모바일 앱으로 바뀌어도 HTML 렌더링만 API로 대체하면 된다. 코드뿐 아니라 **아키텍처·배포·외부 통합도 유연하게** — 외부 API를 추상화 계층 뒤로 숨기고 코드를 컴포넌트로 쪼개라(결국 한 서버에 배포하더라도 쪼개는 게 훨씬 쉽다).

> **Tip 19**: 유행을 좇지 말라.

**도전 과제**
- **슈뢰딩거의 고양이**: 각 결정은 다른 버전의 미래를 낳는다. 당신의 코드는 몇 가지 미래를 지원할 수 있는가? 어떤 미래가 일어날 가능성이 크고, 그때 지원이 얼마나 어려울까?

## Topic 12: 예광탄 (Tracer Bullets)

기관총 사수는 일반 탄환 사이에 낀 **예광탄(tracer bullet)**의 빛 궤적으로 **실시간 피드백**을 받아 조준을 재조정한다. 전에 없던 새 시스템을 만들 때(막연한 요구, 낯선 기술, 변하는 환경), 시스템을 극도로 명세화하고 "발사 후 맞기를 비는" 대신 **소프트웨어 판 예광탄**을 쓴다.

> **Tip 20**: 목표물을 찾기 위해 예광탄을 써라.

> **핵심 통찰**: 요구 사항에서 최종 시스템의 일부까지 **빨리·눈에 보이게·반복적으로** 도달하게 해 줄 것을 찾아라 — 가장 위험하거나 불확실한 부분의 코드를 먼저 작성한다. 첫 예광탄은 "프로젝트 만들고 'hello world' 컴파일·실행"이고, 그다음 불확실한 곳에 뼈대를 추가한다. 장점: 사용자가 일찍 작동하는 것을 봄, 개발자가 일할 구조를 얻음, 매일 통합, 늘 보여줄 것이 있음, 진행 상황을 정확히 앎.

> **핵심 통찰 — 예광탄 vs 프로토타입**: **프로토타입은 나중에 버리는 코드**(어떤 개념을 실험하고 교훈을 얻어 새로 작성)이고, **예광탄은 기능은 별로 없지만 완결된 코드**(최종 시스템 골격의 일부, 오류 검사·문서화 갖춤)다. 프로토타입은 예광탄을 쏘기 전의 정찰·정보 수집이다.

## Topic 13: 프로토타입과 포스트잇 (Prototypes and Post-it Notes)

프로토타입은 **위험을 분석·노출한 뒤 저렴하게 바로잡을 기회**를 준다(자동차 회사의 점토 모델처럼). 반드시 코드일 필요는 없다 — 포스트잇·화이트보드로 작업 흐름·UI를 프로토타이핑할 수 있다.

> **Tip 21**: 프로토타이핑으로 학습하라.

> **핵심 통찰**: 프로토타이핑의 가치는 **생산한 코드가 아니라 배우는 교훈**에 있다. 정확성·완전성·안정성·스타일 같은 세부는 무시해도 된다(더미 데이터·오류 검사 생략 OK). 대상은 **위험을 수반하는 모든 것**(아키텍처·새 기능·외부 데이터/도구·성능·UI). ⚠️ 시작 전 모두에게 "**폐기할 코드**"임을 분명히 주지시켜라 — 안 그러면 관리자가 배포하자고 한다. 오해 가능성이 크면 예광탄 방식이 낫다.

**도전 과제**
- 마케팅팀이 클릭 가능한 이미지 맵 웹 페이지 프로토타입을 15분 안에 원한다. 어떤 도구를 쓰겠는가? (→ 화이트보드에 대충 그리고 포스트잇을 클릭 위치에 붙이는 **로우 테크**로 충분)

## Topic 14: 도메인 언어 (Domain Languages)

> 언어의 한계가 곧 자기 세계의 한계다.<br>— 루트비히 비트겐슈타인(Ludwig Wittgenstein)

컴퓨터 언어는 사고 방식에 영향을 준다. 실용주의 프로그래머는 한 걸음 더 나아가 **문제 도메인의 어휘·문법·의미론(=도메인 언어)으로 프로그래밍**하기도 한다.

> **Tip 22**: 문제 도메인에 가깝게 프로그래밍하라.

> **핵심 통찰 — 내부 vs 외부 도메인 언어**: **내부(internal)**(RSpec·피닉스 라우터 — 호스트 언어 안에서 구현, 그 기능을 공짜로 쓰지만 호스트 문법에 매임)와 **외부(external)**(큐컴버·앤서블 — 자체 언어, 파서가 필요해 자유롭지만 만들기 어려움). 제안: **절약하는 것보다 더 많은 시간을 쓰지 말라** — 가능하면 YAML·JSON·CSV 같은 널리 통용되는 외부 언어를, 아니면 내부 언어를 쓰고, 외부 언어는 사용자가 직접 코드를 쓸 때만 추천.

> 큐컴버 같은 자연어 테스트도 사업 부서 사람들은 잘 안 읽는다 — 요구 사항을 문서로 결재받는 것보다 **동작하는 코드를 보여 줘라**. 그러면 진짜 필요한 것이 드러난다.

## Topic 15: 추정 (Estimating)

> **Tip 23**: 추정으로 놀람을 피하라.

추정 능력을 기르면 무언가의 규모를 직관적으로 짚고 가능성을 가늠할 수 있다. 핵심 단계: **얼마나 정확해야 하는지 파악**(단위가 해석을 좌우 — "130일"은 "6개월"보다 정확해 보인다) → **이미 해본 사람에게 물어보기** → **무엇을 묻는지 이해**(조건scope 파악) → **모델을 만들고 컴포넌트로 나눠 매개변수에 값 할당**(곱·나눗셈 매개변수가 결과에 큰 영향) → **답 계산**(범위로) → **추정 실력 기록**(틀리면 왜 틀렸는지).

**기간 추정 단위**: 1~15일 → '일', 3~6주 → '주', 8~20주 → '달', 20주 이상 → 다시 생각하라.

- **프로젝트 일정 추정**: **PERT**(낙관·최빈·비관 추정치를 범위로) — 부풀리기를 막지만, 수식을 썼다고 정확하다 믿는 함정 주의. **코끼리 먹기**(한 번에 한입) — 작은 단위로 요구 확인→위험 분석→설계·구현·통합→검증을 반복하며 일정을 다듬는다.

> **Tip 24**: 코드와 함께 일정도 반복하며 조정하라.

> **핵심 통찰 — 추정 요청에 대한 단 하나의 올바른 답**: **"나중에 연락드릴게요."** 잠시 멈추고 이 항목의 단계를 밟으면 대부분 더 좋은 추정치를 얻는다 — 커피 머신 앞에서 허투루 말한 추정치는 커피처럼 해를 끼친다.

**도전 과제**
- 계산한 추정치와 실제 정확도를 기록하라. 오차가 50% 이상이면 원인을 찾아보라.

## 연습 문제 (해답 예시)

각 문제는 부록 '연습 문제 해답 예시'와 연결된다. 아래는 문제와 해답 요지다.

1. **(Topic 10)** `Split1`(파일 열기+줄 읽기+필드 반환) vs `Split2`(줄 쪼개기만) 중 어느 쪽이 더 직교적인가? → **Split2** — 줄 쪼개기에만 집중하고 데이터 출처를 무시하므로 파일·다른 루틴·환경 변수 문자열을 모두 처리할 수 있다.
2. **(Topic 10)** OO 언어와 함수형 언어의 직교성 차이는? → 어떤 언어로든 직교적 코드는 가능하나 각 언어엔 결합을 높이는 유혹이 있다. OO는 **다중 상속·예외·연산자 오버로딩·부모 메서드 오버라이딩**이 불투명한 결합을, 함수형은 **함수 조합 시 데이터 포맷 변경이 후속 변환을 망가뜨리는** 결합을 만든다(좋은 타입 체계로 예방).
3. **(Topic 13)** 15분 안에 웹 페이지 프로토타입? → **로우 테크** — 화이트보드에 자동차·전화기·집을 대충 그리고, 이동 페이지 내용을 적은 포스트잇을 클릭 위치에 붙인다.
4. **(Topic 14)** 터틀 그래픽 외부 DSL 파서를 구현하라. → **표 기반 파서**(명령 글자·인자 필요 여부·호출 루틴을 표에 담아 순차 검색) — 새 명령어 추가가 쉽다.
5. **(Topic 14)** 같은 것을 내부 DSL로 재구현하라. → 사실 문제 4의 외부 언어 인터프리터 안에 이미 내부 언어 인터프리터(`doXxx` 함수들)가 들어 있다.
6. **(Topic 14)** 시간 표현 파싱 BNF 문법을 설계하라(`4pm`, `7:38pm`, `23:42` …). → `<time> ::= <hour> <ampm> | <hour> : <minute> <ampm> | <hour> : <minute>` 식으로 정의하고, 시(00~23)·분(00~59) 범위까지 반영하면 더 좋다.
7. **(Topic 14)** BNF를 PEG 파서 생성기로 구현하라(자정 이후 분 반환). → PEG.js 예시로 각 규칙을 표현식으로 옮긴다.
8. **(Topic 14)** 시간 파서를 스크립트 언어 + 정규 표현식으로 구현하라. → 루비 정규식에서 **이름 붙인 하위 패턴을 앞에 정의**하고 본식에서 조합한다.
9. **(Topic 15)** "1Gbps 네트워크 vs 1TB 저장 장치를 들고 걷는 사람" 대역폭 비교에 필요한 제약 조건은? → 정보가 저장 장치에 있음, 걷는 속도·컴퓨터 간 거리를 앎, 저장 입출력 시간은 무시 등 **가정**이 필요하다.
10. **(Topic 15)** 그래서 어느 쪽이 더 넓은가? → 1TB=2⁴³비트를 1Gbps로 보내면 약 9,000초(2.5시간). 시속 4km로 걸으면 **두 컴퓨터가 10km 이상 떨어져야 네트워크 승**, 아니면 사람 승.

## 요약

- **좋은 설계 = ETC(바꾸기 더 쉽게)** — 모든 설계 원칙이 ETC의 특수한 경우다(규칙이 아니라 가치)
- **DRY** — 지식·의도의 중복을 없애라(복사-붙여넣기 금지가 아님). **재사용하기 쉽게** 만들어라
- **직교성** — 관련 없는 것끼리 영향이 없게(한 기능 변경 = 한 모듈). DRY와 짝을 이룬다
- **가역성** — 최종 결정이란 없다. 결정을 모래 위 글씨로 여기고 아키텍처·배포도 유연하게. **유행을 좇지 말라**
- **예광탄**(완결된 골격, 계속 사용) vs **프로토타입**(버리는 코드, 학습이 목적)
- **도메인 언어** — 문제 도메인에 가깝게. 내부(호스트 언어) vs 외부(자체 파서)
- **추정** — 놀람을 피하고, 코드와 함께 일정을 반복 조정하라. 요청받으면 "나중에 연락드릴게요"

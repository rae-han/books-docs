# Chapter 5: Bend, or Break (구부러지거나 부러지거나)

## 핵심 질문

빠른 변화 속에서 코드가 부러지지 않고 구부러지게 하려면? 결합도를 어떻게 줄이는가(열차 사고·전역 데이터·상속)? 이벤트에 반응하는 네 전략은? 변환 프로그래밍·상속 대안·외부 설정은 어떻게 유연성을 주는가?

> 가역성(Ch4)이 되돌릴 수 없는 결정의 위험을 말했다면, 이 장은 **되돌릴 수 있는 결정을 내리는 구체적 방법**을 다룬다 — 다리는 단단해야 하지만 소프트웨어는 유연해야 한다.

## Topic 28: 결합도 줄이기 (Decoupling)

높은 결합도는 변경의 적이다(결합은 **추이적** — A-B, B-M이면 A-M). 증상: 관계없는 모듈의 희한한 의존, '간단한' 수정이 전역으로 퍼짐, 수정을 두려워함.

> **Tip 44**: 결합도가 낮은 코드가 바꾸기 쉽다.

- **열차 사고(train wreck)**: `customer.orders.find(id).getTotals().grandTotal = ...` 같은 연쇄 호출은 다섯 단계의 구현 세부를 다 알아야 하고 다 결합된다.

> **Tip 45**: 묻지 말고 말하라(Tell, Don't Ask, TDA).

> **핵심 통찰**: 다른 객체의 내부 상태를 **묻고 판단해 갱신하지 마라** — 할인 처리를 `totals` 객체에 **위임**하면 캡슐화가 산다. 단 TDA는 자연법칙이 아니라 패턴이다(고객·주문 같은 최상위 개념 노출은 실용적 판단으로 OK).

- **데메테르 법칙(LoD)**: 원칙은 유효하되 원문은 쓰기 어려워, 저자들은 **"메서드 호출을 엮지 말라"**로 재표현. 단 **절대 안 바뀔 것**(언어 내장 라이브러리)이면 `people.sort_by{}.first(10).map{}`처럼 엮어도 된다.

> **Tip 46**: 메서드 호출을 엮지 말라.

- **전역 데이터의 해악**: 전역 데이터 하나는 모든 메서드에 매개변수가 몰래 추가된 것과 같다. **싱글턴도 전역 데이터**(인스턴스 변수를 노출하면 이름만 길어진 전역), **수정 가능한 외부 리소스(DB·파일)도 전역 데이터**다.

> **Tip 47**: 전역 데이터를 피하라.
> **Tip 48**: 전역적이어야 할 만큼 중요하다면 API로 감싸라.

## Topic 29: 실세계를 갖고 저글링하기 (Juggling the Real World)

**이벤트**는 정보가 있다는 것을 뜻한다. 애플리케이션을 이벤트에 **반응(responsive)**하게 만드는 네 전략:

- **유한 상태 기계(FSM)**: 이벤트 처리 명세 — 상태·현재 상태·이벤트별 다음 상태. **오직 데이터(상태 이행 표)만으로** 표현할 수 있어 저평가돼 있다(몇 줄이면 난장판을 막는다). 상태 이행에 **행동(action)**을 붙여 강화할 수 있다.
- **감시자 패턴(observer)**: 감시 대상에 감시자가 콜백을 등록. 오래 검증됐지만 **모든 감시자가 등록해야 해 결합**이 생기고, 동기 호출이라 **성능 병목**이 될 수 있다.
- **게시-구독(pub-sub)**: 게시자·구독자를 **채널**로 연결(코드 밖, 대개 비동기). 감시자의 결합·성능 문제를 해결하나, 많이 쓰면 **무슨 일이 벌어지는지 파악하기 어렵다**.
- **반응형 프로그래밍과 스트림**: 스프레드시트처럼 값이 바뀌면 반응. **스트림**은 이벤트를 리스트처럼 다루게 해(동기/비동기 통합, 병렬 처리) 이벤트를 **비동기 컬렉션**으로 만든다(RxJS 등).

> **핵심 통찰**: 이벤트는 어디에나 있다 — 이벤트 중심으로 만든 코드는 일직선 코드보다 **더 잘 반응하고 결합도가 낮다**.

## Topic 30: 변환 프로그래밍 (Transforming Programming)

모든 프로그램은 **입력을 출력으로 바꾼다**. 유닉스 프로그래머가 `find . -type f | xargs wc -l | sort -n | tail -5`로 "가장 긴 파일 5개"를 **연속된 변환**으로 풀듯, 코드를 조립 라인처럼 생각하라.

> **Tip 49**: 프로그래밍은 코드에 관한 것이지만, 프로그램은 데이터에 관한 것이다.

요구 사항에서 입출력을 찾아 전체 함수를 정하고, 이를 **더 작은 변환들**로 나눈다(애너그램 탐색기 예). 파이프 연산자(`|>`, `->>`, `%>%`)는 데이터가 변환에서 변환으로 **흐르는 것을 보게** 한다(없는 언어는 임시 변수로).

> **Tip 50**: 상태를 쌓아 놓지 말고 전달하라.

> **핵심 통찰**: 데이터를 작은 웅덩이(객체)에 흩어 놓고 서로 상태를 바꾸는 대신, **거대한 강(흐름)**으로 생각하라 — 데이터가 기능과 동등해지고, 매개변수만 맞으면 어떤 함수든 재사용돼 **결합이 대폭 준다**. 오류 처리는 값을 날것으로 넘기지 말고 **래퍼(하스켈 `Maybe`, F#·스칼라 `Option`, 엘릭서 `{:ok,값}`/`{:error,원인}`)**로 싸서, 파이프라인 어디서 오류가 나면 이후 단계를 건너뛰게 한다.

## Topic 31: 상속세 (Inheritance Tax)

> 당신이 원한 것은 바나나 하나였지만, 당신이 받은 것은 바나나를 들고 있는 고릴라와 정글 전체다.<br>— 조 암스트롱(Joe Armstrong)

상속은 두 이유로 쓰인다 — **코드 공유**(타입 싫어서)와 **타입 정의**(타입 좋아서). 둘 다 문제다.

> **핵심 통찰**: **상속은 결합**이다. 코드 공유용 상속은 자식이 부모의 모든 조상과 얽혀, 부모가 API·변수명을 바꾸면 자식을 쓰는 코드까지 망가진다. 타입 정의용 상속은 클래스 계층도가 벽을 덮는 괴물로 자라고, 다중 상속 없이는 도메인을 정확히 모델링할 수 없다.

> **Tip 51**: 상속세를 내지 말라.

세 가지 더 나은 대안:
- **인터페이스와 프로토콜**: 코드 없이 "이 동작을 구현하라"고 지시 — 상속 없이 다형성을 준다.

> **Tip 52**: 다형성은 인터페이스로 표현하는 것이 좋다.

- **위임(delegation)**: 프레임워크 기반 클래스를 상속하는 대신 **감싸서** API 노출을 막는다("왜 `Account`가 영속성을 알아야 하나" → `AccountRecord`로 분리).

> **Tip 53**: 서비스에 위임하라. Has-A가 Is-A보다 낫다.

- **믹스인·트레이트**: 상속 없이 기능 집합을 합친다(`AccountForCustomer extends Account with AccountValidations, ...`) — 상황별 전문화 클래스를 만든다.

> **Tip 54**: 믹스인으로 기능을 공유하라.

**도전 과제**
- 다음 상속을 쓸 때 인터페이스·위임·믹스인(또는 조합)으로 결합을 줄일 수 없는지 검토하라.

## Topic 32: 설정 (Configuration)

출시 후 바뀔 수 있는 값, 환경·고객마다 다른 값은 **코드 밖에서** 관리하라 — 코드가 환경에 적응한다.

> **Tip 55**: 외부 설정으로 애플리케이션을 조정할 수 있게 하라.

- 설정 대상: 외부 서비스 인증 정보·로그 레벨·포트/IP·검증 매개변수·배송비·서식·라이선스 키.
- **정적 설정**(YAML·JSON·DB 테이블) vs **서비스형 설정(Configuration-As-A-Service)**(API 뒤에서 — 여러 앱 공유, 일괄 변경, **동적 변경 시 알림**). 어느 쪽이든 설정을 **얇은 API 뒤로 숨겨** 코드와 분리하고, 설정 변경에 코드 빌드가 필요해선 안 된다.

> **핵심 통찰 — 지나치게 하지는 말라**: (1판 조언의 보정) 모든 필드를 설정 가능하게 한 고객은 4만여 개 설정 변수의 '코딩 지옥'에 빠졌다. **게으름 때문에 결정을 미루고 설정으로 사용자에게 떠넘기지 마라** — 진짜 토론이 벌어졌다면 한 방법으로 해 보고 피드백을 구하라.

22. **(Topic 30)** 문자열을 검증해 18~150 정수로 바꾸는 "검증 및 변환"의 개별 변환을 작성하라. → 문자열 → (정수 변환) → (범위 검사 18~150) → `{:ok, 값}` 또는 `{:error, 원인}`. 각 단계가 오류 튜플을 전달한다.
23. **(Topic 30)** 임시 변수 방식 vs 메서드 체이닝(`content_of(f).find_matching_lines(p).truncate_lines()`)의 차이와 저자의 선호는? → 메서드 체이닝은 각 객체가 **다음 메서드를 제공해야 해 결합**을 낳는다(열차 사고). 저자는 독립적 변환 함수를 잇는 **파이프라인/임시 변수** 방식을 선호한다(숨은 구현에 의존하지 않으므로).

## 요약

- **결합도 줄이기**: 열차 사고를 피하고(**묻지 말고 말하라**·메서드 엮지 말기), **전역 데이터**(싱글턴·외부 리소스 포함)를 피하거나 API로 감싸라
- **실세계 저글링**: 이벤트에 **FSM·감시자·게시-구독·반응형 스트림**으로 반응하라
- **변환 프로그래밍**: 코드를 **입력→출력 변환의 파이프라인**으로 생각하고 **상태를 전달**하라(래퍼로 오류 처리)
- **상속세를 내지 마라**: 상속은 결합 — **인터페이스·위임(Has-A)·믹스인**이 대개 더 낫다
- **외부 설정**으로 앱을 조정하되, **지나치게 하지는 마라**(결정을 설정으로 떠넘기지 말 것)

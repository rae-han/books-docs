# Chapter 2: Checklists (체크리스트)

## 핵심 질문

프로그래머에서 소프트웨어 엔지니어로 거듭나려면 어떻게 해야 할까? 항공기 조종사와 외과의사가 사용하는 체크리스트를 우리도 사용할 수 있을까? "실력을 향상시키지 않고도 결과를 개선할 수 있다"는 말은 정말 가능한가?

---

## 1. 기억을 돕는 보조 수단

소프트웨어 개발의 근본적인 문제 중 하나는 **아주 많은 일이 동시에 벌어진다**는 점이다. 우리 두뇌는 동시에 여러 가지를 추적하지 못하며, 당장 중요해 보이지 않는 일은 건너뛰는 경향도 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 문제는 어떻게 일해야 하는지를 모르는 게 아니라, 해야 할 일 자체를 잊는다는 점이다.
:::

이 문제는 프로그래밍에만 국한된 것이 아니다. 항공기 조종사들도 같은 문제로 골머리를 앓아왔으며, 결국 간단하고 효과적인 해결 방법을 찾았다. 바로 **체크리스트(*Checklist - 반드시 수행해야 할 항목을 나열한 짧은 목록*)** 다.

### 1.1 B-17 폭격기 사례

아툴 가완디(Atul Gawande)에 따르면 1935년 B-17 폭격기에 처음 체크리스트가 도입되었다. B-17은 이전 비행기보다 훨씬 복잡했고, 무기 구매자를 위한 시연 비행에서 추락하여 조종사를 포함한 승무원 2명이 사망할 정도였다.

조사 결과 원인은 "조종사 실수"로 결론이 났다. 하지만 사고를 낸 조종사는 육군 항공단(Army Air Corps, AAC)에서 가장 경험 많은 시범 조종사였다. 단순히 훈련 부족으로 치부할 수는 없었다. 당시 신문의 표현처럼 "한 사람이 조종하기에는 너무 복잡한 비행기"였던 것이다.

시범 조종사 그룹은 해결책으로 이륙과 착륙 과정에서 수행해야 할 간단한 작업들을 나열한 체크리스트를 고안했다. 이 간단한 체크리스트는 숙련된 전문가에게 큰 힘이 되었다.

### 1.2 체크리스트의 본질

체크리스트는 사소한 일에 신경 쓰지 않고 작업의 어려운 부분에 집중하도록 도와준다. 사소한 것까지 모두 기억하려고 노력할 필요 없이, 중간중간 체크리스트를 확인하는 것만 기억하면 된다.

> 외과의사의 수술 체크리스트를 본 적이 있습니다. "반드시 손을 씻는다", "팀원 전부에게 이야기해야 한다" 등의 목록은 기술을 향상시키지 않고도 좋은 수술 결과를 이끌어냅니다. 이것이 바로 우리가 체크리스트를 사용하면서 얻길 바라는 것입니다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 체크리스트는 실무자를 감시하거나 감독하려는 것이 아니라, 실무자를 지원하고 작업을 편리하게 진행하도록 돕기 위해 만들어졌다. 조종사와 외과의사가 체크리스트를 따른다면, 우리도 못 할 이유가 없다.
:::

가장 강력한 목록은 따로 체크하지 않아도 되는(즉 흔적을 남기지 않아도 되는) 목록일 것이다. 포스트잇, 클립보드, 링바인더 같은 곳에 짧게 적어 두어도 충분하다. **체크리스트는 사용자에게 제약을 걸기 위해서가 아니라 결과를 개선하기 위한 것**이다.

## 2. 새로운 코드베이스를 위한 체크리스트

저자가 제안하는 새로운 코드베이스 시작 시 필요한 체크리스트는 딱 세 가지다.

- 깃(Git)을 사용할 것
- 빌드를 자동화할 것
- 모든 오류 메시지를 켜 둘 것

너무 간단해 보이지만 의도적으로 그렇게 만든 것이다. 체크리스트는 해야 할 일을 자세하게 적은 복잡한 순서도가 아니라, **몇 분 안에 처리할 수 있는 간단한 항목**으로 만든 목록이다.

체크리스트의 형태는 두 가지다.

| 형태 | 방식 |
|---|---|
| **읽기-실행(*Read-Do*)** | 각 항목을 읽고, 다음 항목으로 넘어가기 전에 즉시 작업을 실행 |
| **실행-확인(*Do-Confirm*)** | 모든 작업을 실행한 후, 빠짐없이 완료했는지 마지막에 확인 |

위 목록은 명령형 문장이므로 "읽기-실행" 형태다. "실행-확인" 형태로 바꿀 수도 있지만, 이 경우 다른 사람과 함께 확인해야 실효성이 있다. 조종사가 한 명은 읽고 다른 한 명이 확인해 나가는 이유가 여기 있다. 혼자 진행하면 한 단계를 빼먹기 쉬우므로 보조 조종사가 같이 확인하는 것이다.

### 2.1 깃을 사용할 것

깃은 사실상 표준 소스 제어 시스템이 되었다. 반드시 사용해야 한다. CVS나 서브버전(Subversion) 같은 **중앙 집중식 소스 제어 시스템(*Centralised Source Control System - 중앙 서버가 소스 이력을 관리하는 방식*)** 과 비교하면, **분산 소스 제어 시스템(*Distributed Source Control System - 각 개발자 로컬에도 전체 이력이 저장되는 방식*)** 은 엄청난 장점을 제공한다.

> 깃이 가장 사용자 친화적인 기술은 아니지만, 여러분은 프로그래머다. 프로그래밍 언어를 적어도 하나는 배웠다는 이야기이므로 깃의 기초를 배우는 건 아주 쉽다. 하루 이틀 정도 투자해서 배워보라. GUI 사용법이 아니라 실제로 어떻게 동작하는지를 배우는 게 중요하다.

새로운 코드베이스에서 가장 먼저 해야 할 일은 로컬 저장소를 만드는 것이다.

```bash
$ git init
```

친구인 엔리코 캄피돌리오(Enrico Campidoglio)의 조언에 따르면, 아무것도 없이도 커밋할 수 있다.

```bash
$ git commit --allow-empty -m "Initial commit"
```

저자는 온라인 깃 서비스에 저장소를 올리기 전에 위 명령을 자주 사용한다. 깃은 저장소의 이력을 다시 변경할 수 있기 때문이다. 하지만 보통 이렇게까지 할 필요는 없다.

### 2.2 빌드를 자동화할 것

코드가 별로 없을 때는 컴파일·테스트·배포를 자동화하기 쉽다. 하지만 기존 코드베이스에 지속적 배포를 다시 적용하는 것은 만만치 않은 작업이다. 그렇기 때문에 **지금 당장 해야 한다**.

수직 슬라이스보다 한 단계 앞서 있는 개념이 **작동하는 골격(*Walking Skeleton - 실제 기능을 가진 가장 간단한 구현. 자동 빌드·배포·테스트가 모두 가능한 최소 애플리케이션*)** 이다. 이 책에서는 C# ASP.NET Core로 온라인 레스토랑 예약 시스템을 만들 것이므로, 일단 HTTP 요청을 처리할 웹 서비스가 필요하다.

비주얼 스튜디오로 만든 최소 웹 서비스는 다음과 같다.

```csharp
public class Program
{
    public static void Main(string[] args)
    {
        CreateHostBuilder(args).Build().Run();
    }

    public static IHostBuilder CreateHostBuilder(string[] args) =>
        Host.CreateDefaultBuilder(args)
            .ConfigureWebHostDefaults(webBuilder =>
                webBuilder.UseStartup<Startup>());
}
```

이 상태에서 웹사이트를 실행하면 `Hello World!`만 반환한다. 지금은 이 정도로 충분하다. 코드를 깃에 커밋하고, IDE 밖에서도 빌드가 되도록 스크립트를 만든다.

```bash
#!/usr/bin/env bash
dotnet build --configuration Release
```

> **실무 팁**: 자동화된 빌드는 최종 운영 환경을 반영해야 한다. 빌드 단계가 추가될 때마다 이 스크립트에도 반드시 추가해야 하며, 스크립트는 개발자 자신의 컴퓨터에서 아무런 이질감 없이 실행되어야 한다.

다음 단계는 **배포 파이프라인(*Deployment Pipeline - 마스터에 커밋하면 자동으로 운영 환경에 배포되거나, 최소한 배포 가능한 상태로 만들어주는 프로세스*)** 을 만드는 것이다. 구체적인 사항은 사용하는 서비스(애저 데브옵스, 젠킨스, 팀시티 등)에 따라 다르므로 책에서 자세히 다루지는 않는다.

### 2.3 모든 오류 메시지를 켜 둘 것

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 컴파일러 경고를 비롯한 여러 가지 자동화 도구들은 코드의 문제를 감지할 수 있다. 반드시 사용하라. 깃을 사용하는 것과 더불어 가장 손쉽게 좋은 결과를 기대할 수 있는 부분이다.
:::

저자가 예전에 다른 프로그래머에게 유닛 테스트 추가하는 방법을 가르쳤을 때, 코드가 컴파일은 됐지만 제대로 동작하지 않는 문제가 발생했다.

> "컴파일러 경고가 있는지 확인해볼까요?"
>
> 그 프로그래머는 이렇게 대답했다.
>
> "그건 소용없어요. 이 코드베이스에는 컴파일러 경고가 수백 개쯤 있으니까요."

진짜였다. 하지만 저자는 경고 목록을 살펴봐야 한다는 주장을 굽히지 않았고, 예상대로 문제에 대한 경고가 있었으며, 그 경고는 문제를 정확하게 나타내고 있었다.

이미 경고가 124개나 떠 있는 경우 새로 뜨는 경고를 발견하기는 어렵다. 따라서 경고에 예외를 허용하지 말아야 한다. **항상 경고가 0개**여야 한다. 다시 말해, **경고를 오류로 처리**해야 한다.

이전부터 누적된 수백 개의 경고를 한꺼번에 해결하는 건 어려운 작업이다. 경고가 표시될 때마다 하나씩 처리하는 것이 훨씬 쉽다. 따라서 새 코드베이스에서는 처음 시작할 때부터 `warnings-as-errors` 옵션을 켜야 한다.

C#에는 컴파일러 경고 외에도 **분석기(*Analyser - .NET Roslyn 기반 정적 코드 분석기. 자바스크립트/하스켈 등의 린터와 유사한 도구*)** 라는 유틸리티가 있다. 최신 비주얼 스튜디오에서는 간단히 추가할 수 있다. 분석기는 명명 규칙 위반, 잠재적 보안 문제, 알려진 라이브러리 API의 잘못된 사용, 성능 문제 등을 찾아낸다.

앞서 만든 최소 코드에 기본 규칙을 적용해 코드 분석기를 켜면 최소 7개의 경고가 발생한다. 이제 경고가 오류로 처리되므로 컴파일되지 않는다. 언뜻 방해되는 것처럼 보이지만, **정말 방해가 되는 건 코드에 대해 신중하게 고민하지 않아도 유지보수가 가능할 것이라는 착각**이다.

경고 7개는 대부분 코드 일부를 삭제해서 고칠 수 있는 문제다. `Program` 클래스에서는 한 가지만 바꾸면 된다.

```csharp
public static class Program  // ← static 추가
{
    public static void Main(string[] args)
    {
        CreateHostBuilder(args).Build().Run();
    }
    // ...
}
```

클래스에 공유 멤버만 있는 경우 인스턴스화를 지원할 이유가 없으므로 `static` 키워드를 붙였다. `Startup` 클래스에는 `sealed`를 붙이고, `ConfigureServices` 메서드가 아무 것도 하지 않으면 삭제하며, `ConfigureAwait(false)` 호출을 추가한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 정적 코드 분석은 자동화된 코드 리뷰와 같다. 저자는 C# 코드 리뷰 요청이 오면 먼저 분석기를 실행하라고 이야기해준다. 이를 통해 몇 시간 분의 수수료를 절약할 수 있다.
:::

#### 널 가능 참조 형식

C# 8에는 **널 가능 참조 형식(*Nullable Reference Types - 정적 형식 시스템에서 객체가 널이 될 수 있는지를 선언할 수 있는 기능. 실제로는 non-nullable 참조 형식이라 부르는 게 정확함*)** 이 도입되었다. 널이 될 수 없는 객체와 널이 될 수 있는 객체를 구별할 수 있으면 방어 코드의 양이 줄어들고, 런타임 결함 수가 줄어든다. 코드베이스를 새로 만들 때부터 켜 두는 것이 좋다.

린터·분석기는 컴파일러와 달리 **오탐지(*False Positive - 실제로는 문제가 없는데 경고를 띄우는 경우*)** 를 낼 수 있다. 하지만 대부분의 도구는 오탐지를 줄이는 다양한 옵션이 있으므로, "오탐지 때문에 못 쓰겠다"는 이야기는 말이 안 된다.

## 3. 기존 코드베이스를 위한 체크리스트

실무에서는 새로운 코드베이스부터 시작할 기회가 별로 없다. 회사에서 개발하는 소프트웨어는 대부분 기존 코드를 가지고 있다. 새로운 코드베이스에서 경고를 오류로 처리하는 것은 덜 까다롭지만, 기존 코드베이스에서도 불가능한 것은 아니다.

### 3.1 점진적 개선

핵심은 **점진적으로 하나씩 처리해 나가는 것**이다. 기존 코드베이스는 대부분 여러 라이브러리를 포함하므로, 한 번에 라이브러리 하나씩 추가 확인 옵션을 켜 나가면 된다.

한 번에 한 가지 유형의 경고만 켤 수도 있다. 기존 코드베이스에는 컴파일러가 이미 수백 개의 경고를 띄우고 있을 수 있다. 이런 경우 다음 절차를 따른다.

1. 경고 목록을 추출하고 유형별로 묶는다
2. 대략 열 개 정도의 경고를 띄우는 유형을 하나 선택한다
3. 해당 유형의 경고를 모두 제거한다
4. 해당 규칙을 오류로 승격한다
5. 다시 다른 규칙 선택

컴파일러 경고를 오류로 바꾸지 않고 경고 상태로 유지하면 코드를 계속 사용하는 데는 문제가 없다. 개선할 때마다 깃에 변경사항을 커밋하고 마스터에 병합해 나가면 된다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 모든 경우의 핵심은 **보이스카우트 규칙**을 따르는 것이다. 코드를 손대기 전보다 더 좋은 상태로 만들어라.
:::

### 3.2 조직 문화를 바꾸는 전략

컨퍼런스 발표 후 사람들이 다가와 "관리자들이 내부 품질(*Internal Quality - 사용자에게는 직접 보이지 않지만 유지보수 속도를 좌우하는 코드 품질*)에 집중하도록 놔두지 않을 것 같다"고 이야기하는 경우가 많다.

경고를 오류로 처리할 때 얻을 수 있는 장점은 **제도적으로 품질이 떨어지는 것을 막을 수 있다**는 점이다. "정석대로 할 시간이 없다", "그냥 배포하라"는 압력을 받았을 때 다음과 같이 대답할 수 있다.

> "죄송하지만 그렇게 하면 코드가 컴파일되지 않습니다."

이 대답은 개발 원칙을 무시하려는 이해관계자들의 고집을 막아준다. 자동 검사를 우회할 수도 있지만, 이걸 모든 사람에게 알려줄 필요는 없다. 즉 **이전에는 인간이 결정하던 부분을 기계가 강제하는 규칙으로 바꿔버리는 전략**이다.

이 방법이 도덕적으로 적절한지는 여러분의 판단에 맡긴다. 건전한 조직에서 가장 좋은 전략은 공개적이고 정직하게 작업하는 것이다. 하지만 건전하지 않은 조직(예: **허슬 문화(*Hustle Culture - 과도한 근무·성과 압박이 만연한 문화*)** 가 만연한 곳)에서는 반대 전략을 취하는 것이 좋을 수도 있다. 자동화된 품질 관리 도구를 이용해 조직 문화를 약간 바꿀 수 있으며, 궁극적인 목표는 우수한 소프트웨어 공학이 진행되도록 지원하는 것이다.

## 4. 자주 하는 실수

| 안티패턴 | 해결법 |
|---|---|
| 경고를 무시하고 쌓아둠 | 즉시 `warnings-as-errors` 옵션 활성화 |
| 프로젝트 초기에는 자동화를 나중으로 미룸 | 코드가 없을 때부터 배포 파이프라인 구축 |
| 체크리스트를 규제 도구로 사용 | 실무자를 지원하는 보조 도구로만 사용 |
| 한꺼번에 수백 개 경고 처리하려다 포기 | 한 가지 규칙씩 점진적으로 활성화 |
| GUI만 익히고 깃 원리를 무시 | 명령줄 인터페이스로 실제 동작 학습 |

## 5. 결론

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 공학이란 광범위하게 이야기하면 궁극적인 성공의 기회를 높이기 위해 모든 휴리스틱 방식, 결정론적 방법, 장치들을 사용하는 것이다. 이러한 도구는 **자동화된 체크리스트**라고 볼 수 있다.
:::

체크리스트는 기술의 향상 없이도 결과를 개선할 수 있다. 반드시 사용하기 바란다. 이 장의 체크리스트에는 세 항목만 있었다.

1. **깃을 사용할 것**: 가장 간단한 항목. 이 작은 노력이 나중에 수많은 이득으로 돌아온다
2. **빌드를 자동화할 것**: 빌드 스크립트를 만들고 배포 파이프라인에 통합
3. **모든 오류 메시지를 켜 둘 것**: 컴파일러 경고를 오류로, 린터/분석기도 활성화

공학이란 체크리스트를 따르거나 자동화할 수 있는 것을 자동화하는 것 이상의 일이다. 하지만 이런 조치를 했다는 것 자체가 올바른 방향으로 나아가는 첫걸음이며, 오늘 당장 할 수 있는 작은 개선사항이다.

## 요약

- 소프트웨어 개발의 근본 문제는 아주 많은 일이 동시에 벌어져 두뇌가 다 추적하지 못한다는 것. 문제는 **일하는 방법을 몰라서가 아니라 해야 할 일을 잊어서** 발생한다
- 항공기 조종사와 외과의사처럼 우리도 **체크리스트**를 쓸 수 있다. 체크리스트는 실무자를 지원하는 도구이지 제약하는 도구가 아니다
- 저자가 제안하는 새 코드베이스 체크리스트 3항목: **깃 사용 · 빌드 자동화 · 모든 오류 메시지 켜기**
- 컴파일러 경고와 정적 분석기(린터·.NET 분석기) 경고를 **오류로 처리(warnings-as-errors)** 하여 누적을 원천 차단
- 기존 코드베이스에서는 **점진적으로** 라이브러리 하나씩, 규칙 하나씩 켜가며 개선. 보이스카우트 규칙 적용
- 자동 검사는 "그냥 배포하라"는 압력을 막는 **제도적 안전장치**가 된다. 인간의 결정을 기계의 강제로 바꾸는 전략
- 공학이란 결국 성공 확률을 높이기 위해 휴리스틱·결정론적 방법·자동화 장치를 총동원하는 것. 체크리스트는 그 가장 간단한 시작점

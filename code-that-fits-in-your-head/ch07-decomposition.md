# Chapter 7: Decomposition (분해하기)

## 핵심 질문

메서드는 왜 그렇게 커지고 마는가? 아무도 의도적으로 레거시 코드를 작성하지 않는데, 왜 코드베이스는 결국 레거시가 되어버리는가? 코드를 **머리에 들어오는 크기의 덩어리**로 쪼개려면 어떤 휴리스틱과 임계값이 필요한가?

---

## 1. 코드의 부패 — 서서히 끓는 물 속의 개구리

> 그 누구도 의도적으로 레거시 코드를 작성하지는 않습니다. 다만 코드가 점점 악화된 결과일 뿐입니다.

한 파일에 코드가 수천 줄이라면 좋지 않다는 것, 메서드가 수백 줄이면 사용하기 어렵다는 것은 모두 알고 있다. 그런데도 코드베이스는 그런 상태로 굴러간다. 이유는 단순하다. 코드에서 일어나는 하나하나의 변화가 사소해 보이기 때문이다. **하루아침에 붕괴되는 것이 아니라 아주 조금씩 천천히 무너진다.**

처음에는 메서드의 복잡도가 낮다. 결함을 수정하고 기능을 추가할 때마다 조금씩 복잡도가 증가한다. 주의를 기울이지 않으면 아무도 눈치채지 못한 상태에서 **순환 복잡도(*Cyclomatic Complexity - 메서드를 통과하는 독립적인 실행 경로의 수*)** 가 7을 넘어가고, 10을 넘어가고, 15와 20을 넘어간다. 문제를 발견하는 시점은 지표가 나빠졌을 때가 아니라, 누구나 알아차릴 정도로 코드가 이미 손을 쓰기에 늦었을 때다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 코드의 부패는 서서히 끓는 물 속의 개구리와 같다. 변화 하나하나는 사소해 보이지만, 축적된 결과는 되돌릴 수 없다. 그래서 **선제적으로 부패를 감지할 임계값이 필요**하다.
:::

### 1.1 임계값 — 자동으로 붕괴를 막는 장치

임계값을 정해두면 코드가 썩는 것을 막는 데 도움이 된다. 예를 들어 순환 복잡도를 계속 확인해서 7을 넘어가는 경우 자동으로 수정 사항을 거부하도록 설정할 수 있다.

코드 품질을 향상시키는 것은 **7이라는 특정 숫자 자체가 아니라, 임계값 기반 규칙이 자동으로 동작한다는 사실**이다. 임계값으로 10을 사용한다 해도 이전과는 다른 결과를 이끌어낼 수 있다. 그럼에도 저자가 7을 선호하는 이유는, 7이 인간 두뇌의 **작업 기억(*Working Memory - 단기적으로 정보를 유지하고 조작하는 기억 시스템*)** 용량을 상징하는 값이기 때문이다.

물론 규칙을 엄격하게 지키다 보면 규칙 자체가 방해가 되는 순간도 온다. 임계값을 넘겨야만 하는 상황이라면 넘긴다. 다만 그 뒤에 반드시 **문제의 코드를 다시 정리할 방법**을 찾아야 한다. 임계값 초과 알림을 아예 꺼버리면 코드는 점진적으로 붕괴한다.

이 과정은 자동화할 수 있다. **지속적 통합(*Continuous Integration - 코드 변경을 지속적으로 통합하고 검증하는 개발 방식, CI*)** 빌드 과정에 순환 복잡도 분석을 포함시켜서 임계값을 넘어가는 변경 사항을 거부하도록 만든다. 다만 **의도하지 않은 결과의 법칙(*Law of Unintended Consequences - 어떤 규칙이든 예상 밖의 부작용을 낳을 수 있다는 원칙*)** 은 항상 조심해야 한다.

> 임계값 규칙을 도입해서 많은 사람이 임계값 규칙을 인지하게 만드는 데 성공했습니다. 이 방법은 기술 리더가 개선하고자 하는 방향을 품질에 집중하도록 도와줄 수 있습니다. 팀의 사고방식이 바뀌면 이런 규칙 자체가 불필요해질 것입니다.

### 1.2 순환 복잡도 계산하기

순환 복잡도는 실무에서 유용하게 사용할 수 있는 몇 안 되는 코드 품질 측정 방법이다. **코드 조각을 통과하는 경로의 수를 측정하는 척도**로 이해하면 된다.

계산은 단순하다. 1에서 시작해서, 분기(`if`, `else if`)와 루프(`for`, `foreach`, `while`, `do`), `switch`의 `case`, 단락 평가 연산자(`&&`, `||`), 그리고 널 병합(`??`) 같은 키워드를 만날 때마다 1씩 증가시킨다. 가장 단순한 코드도 경로가 하나는 있으므로 최소 순환 복잡도는 1이다.

레스토랑 예약 시스템의 6장 마지막 `Post` 메서드를 예로 들어보자. 이 메서드의 순환 복잡도는 얼마일까?

계산해 보면 답은 **7** 이다. 5개의 `if` 문에서 5를 얻고, `Name ?? ""` 의 널 병합 연산자에서 하나를 더 얻어 총 6이 나온다. 여기에 시작값 1을 더하면 7이다. 널 병합 연산자는 놓치기 쉽다. `dto.Name`이 `null`인 경로와 `null`이 아닌 경로, 두 가지 분기를 나타내기 때문이다.

임계값을 7로 잡으면 이 `Post` 메서드는 이미 한계에 도달했다. 지금은 그대로 둬도 되지만, 나중에 8번째 분기를 추가해야 한다면 그때 리팩터링부터 해야 한다. 그때는 시간이 부족할 수 있으니, 지금 시간이 있을 때 미리 리팩터링해 두는 편이 낫다.

### 1.3 80/24 규칙 — 크리스토퍼 알렉산더의 유산

코드 줄 수 역시 복잡도의 실용적인 예측 지표다. 코드 줄이 많아질수록 코드베이스는 나빠진다. **코드 줄은 오직 "삭제한 줄 수"를 측정할 때만 생산성 지표로 쓸모가 있다.**

그렇다면 메서드는 얼마나 작아야 하는가? 이 질문에 보편적인 답은 없고 언어에 따라 다르다. 저자는 C#으로 작성할 때 메서드가 20줄에 가까워지면 불편함을 느끼기 시작하고, 대략 30줄이 한계라고 말한다. 임의로 정한 숫자이므로 편의상 **24줄**을 최대치로 삼는다. 한 줄의 길이는 업계의 오랜 표준인 **80자**를 최대치로 삼는다.

왜 80×24인가? 그 이유는 **크리스토퍼 알렉산더(*Christopher Alexander - 『패턴 랭귀지』로 유명한 건축학자. 소프트웨어 패턴 운동의 원류*)** 이전 시대의 유물, 즉 유명한 **VT100 터미널의 화면 크기가 80자 × 24줄** 이었다는 점에서 비롯한다. 오늘날의 큰 모니터에서는 여러 개의 작은 메서드를 한 번에 볼 수 있다. 예를 들어 화면 분할로 단위 테스트와 대상 코드를 나란히 놓을 수 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 구체적인 80과 24라는 값은 임의의 숫자다. 하지만 **과거와 연속성을 유지하는 것이 근본적으로 옳다**. 중요한 것은 특정 값이 아니라, "메서드는 인간이 한눈에 파악할 수 있는 창(window) 안에 들어와야 한다" 는 원칙이다.
:::

C계열 언어는 `;`로 여러 문장을 한 줄에 이어붙일 수 있지만, 그렇게 24줄 규칙을 피해 가면 애초에 왜 작은 메서드를 지향하는지 그 목적에 어긋난다. **머리에 잘 들어오는 코드를 만들려는 것이 목적이므로 작을수록 좋다.**

6장의 `Post` 메서드는 순환 복잡도가 7이고 정확히 24줄이다. 두 가지 임계값에 모두 도달했다는 것 자체가 리팩터링해야 할 신호다.

## 2. 머리에 잘 들어오는 코드 — 육각꽃 다이어그램

뇌는 동시에 7개 항목을 추적할 수 있다. **하나의 코드 조각 안에서 7가지를 넘는 일이 벌어져서는 안 된다.**

저자는 이 규칙을 시각화하기 위해 **육각꽃 다이어그램(*Hex Flower Diagram - 7개의 육각형을 꽃 모양으로 배치해 단기 기억의 슬롯을 표현한 도표*)** 을 도입한다. 원 7개를 서로 접하도록 눌러붙이면 자연스럽게 육각형 6개가 중심의 육각형 하나를 둘러싼 꽃 모양이 나온다. 각 육각형은 하나의 "메모리 슬롯" 또는 "레지스터"에 해당한다.

6장 마지막 `Post` 메서드에는 정확히 7개의 경로가 있으므로, 육각꽃의 슬롯이 이미 꽉 찬 상태다. 여기에 "이전 예약을 거절한다", "다양한 크기의 테이블을 지원한다" 같은 규칙을 더 추가하려면 반드시 메서드를 분해해야 한다.

### 2.1 응집 — 어디서부터 쪼갤 것인가

`Post` 메서드는 이미 빈 줄로 대략 네 영역이 구분되어 있다. 어느 영역부터 손을 대야 할까?

첫 번째 영역은 데이터 유효성을 확인하는 **가드 절(*Guard Clause - 함수 초입에서 유효하지 않은 입력을 걸러내는 조건 검사*)** 이다. 두 번째와 세 번째는 `Repository` 프로퍼티를 사용하고, 네 번째는 반환문 하나뿐이다.

첫 번째 영역이 가장 눈에 띄는 이유는 이 영역만 `ReservationsController` 클래스의 인스턴스 멤버를 전혀 사용하지 않기 때문이다. 이 관찰은 객체지향 언어의 핵심 개념인 **응집(*Cohesion - 하나의 모듈이 얼마나 관련된 책임에 집중되어 있는지를 나타내는 척도*)** 과 관련이 있다.

> 같은 속도로 변화하는 것들은 함께 있어야 하며, 다른 속도로 변화하는 것들은 서로 떨어져 있어야 합니다.
> — 켄트 벡(Kent Beck)

모든 메서드가 클래스의 모든 필드를 사용하면 최대 응집력이고, 반대의 경우가 최소 응집력이다. **클래스 필드를 전혀 사용하지 않는 코드 블록**은 매우 수상하다. 다른 곳에 있어야 할 코드가 그 자리에 있다는 신호일 가능성이 크다.

첫 번째 시도로 가드 절을 헬퍼 메서드로 뽑아본다.

```csharp
private static bool IsValid(ReservationDto dto)
{
    return DateTime.TryParse(dto.At, out _)
        && !(dto.Email is null)
        && 0 < dto.Quantity;
}
```

6줄, 순환 복잡도 3. 지표만 보면 매우 좋다. 다만 이 메서드가 `static`이라는 사실을 기억해 두자. **인스턴스 멤버를 하나도 쓰지 않는다는 뜻**이며, 코드 분석기가 이를 감지한다. 이는 잠재적 코드 악취이므로 곧 다시 다룬다.

호출 측 `Post` 메서드는 다음과 같이 개선된다.

```csharp
public async Task<ActionResult> Post(ReservationDto dto)
{
    if (dto is null)
        throw new ArgumentNullException(nameof(dto));
    if (!IsValid(dto))
        return new BadRequestResult();

    var d = DateTime.Parse(dto.At!, CultureInfo.InvariantCulture);
    var reservations =
        await Repository.ReadReservations(d).ConfigureAwait(false);
    int reservedSeats = reservations.Sum(r => r.Quantity);
    if (10 < reservedSeats + dto.Quantity)
        return new StatusCodeResult(
            StatusCodes.Status500InternalServerError);

    var r =
        new Reservation(d, dto.Email!, dto.Name ?? "", dto.Quantity);
    await Repository.Create(r).ConfigureAwait(false);
    return new NoContentResult();
}
```

줄 수는 22줄로 줄었고, 순환 복잡도는 5로 떨어졌다. `IsValid`의 복잡도 3이 `Post`에 더해져야 하지 않느냐고 반문할 수 있다. **하지만 그런 식으로 복잡도를 측정하지 않는다.** 호출된 메서드까지 매번 파고들어 계산하면 얻는 것이 별로 없고, 오히려 하나의 동작으로 취급하는 편이 육각꽃 다이어그램에서 복잡도를 더 잘 나타낸다.

> 단기 기억은 덩어리(chunk) 단위로 측정되는데, 이 덩어리는 장기 기억에 있는 더 큰 정보 구조를 가리키는 레이블로 사용될 수 있기 때문입니다.

작은 조각 3개가 조금 더 큰 조각 하나로 대체된다. 로버트 마틴(Robert C. Martin)의 정의처럼 **"추상화는 무관한 것을 제거하고 본질적인 것을 강조하는 것"** 이다. `IsValid`라는 이름은 "DTO의 유효성을 검사한다"는 본질을 강조하고, 검사의 세부는 감춘다.

### 2.2 기능 편애 — 데이터가 있는 곳으로 메서드를 옮기다

응집도 리팩터링이 복잡도를 낮췄지만 새 문제가 생겼다. `IsValid` 메서드가 `static`인 채로 `ReservationDto` 매개변수만 사용한다는 사실이다. 이것은 **기능 편애(*Feature Envy - 어떤 메서드가 자기 클래스보다 다른 클래스의 데이터에 더 관심을 두는 코드 악취*)** 의 전형이다.

『리팩터링』 책의 처방대로, 참조하는 데이터가 많은 객체 쪽으로 메서드를 이동시킨다. `IsValid`는 `ReservationDto` 안으로 들어가고, 매개변수 없이도 되므로 프로퍼티로 바뀐다.

```csharp
internal bool IsValid
{
    get
    {
        return DateTime.TryParse(At, out _)
            && !(Email is null)
            && 0 < Quantity;
    }
}
```

이 동작은 전제 조건이 없고 예외를 던지지 않는, .NET의 프로퍼티 획득자 프레임워크 가이드라인에도 잘 맞는 단순한 계산이다. 호출 측은 이렇게 바뀐다.

```csharp
if (!dto.IsValid)
    return new BadRequestResult();
```

모든 테스트가 통과한다. 커밋하고 배포 파이프라인으로 흘려보낸다.

### 2.3 바꾸는 과정에서 잃는 것들

리팩터링은 만능이 아니다. 한 문제를 해결하면 다른 문제가 튀어나올 수 있다. `IsValid` 프로퍼티를 도입한 뒤 발견되는 문제는 두 가지다.

첫째, `Post` 메서드에서 `At`과 `Email`이 `null`이 아니라는 사실을 C# 컴파일러가 알 수 없게 되었다. 그래서 널 허용 연산자 `!`를 붙여야 하고, 결국 컴파일러의 **널 가능 참조 형식(*Nullable Reference Types - C# 8에서 도입된 null 안전성 기능*)** 정적 흐름 분석을 우회한다.

둘째, `At` 프로퍼티를 `IsValid` 안에서 한 번, `Post`에서 다시 한 번, 실질적으로 두 번 파싱한다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: `IsValid`처럼 부울 플래그만 반환하는 유효성 검사기는 **객체지향적 유효성 검사의 전형적인 문제**를 갖는다. 검사기는 답만 던져줄 뿐, 검사 과정에서 얻은 값(파싱된 날짜 등)을 호출자에게 넘겨주지 않는다. 결과적으로 호출자는 같은 검사를 반복하게 된다.
:::

즉 `IsValid` 추상화 과정에서 너무 많은 부분이 제거되었고, 강조된 부분은 오히려 부족했다. 좋은 추상화가 아니다.

### 2.4 유효성 검사 대신 분석하라 — Parse, don't validate

부울을 반환하는 대신, **사전 조건이 충족되면 DTO를 도메인 객체로 변환**하도록 시그니처를 바꾼다.

```csharp
internal Reservation? Validate()
{
    if (!DateTime.TryParse(At, out var d))
        return null;
    if (Email is null)
        return null;
    if (Quantity < 1)
        return null;

    return new Reservation(d, Email, Name ?? "", Quantity);
}
```

`Validate` 메서드는 가드 절로 `Reservation` 클래스의 사전 조건을 확인한다. `At` 문자열을 적절한 `DateTime`으로 파싱하는 것도 여기 포함된다. 모든 사전 조건이 충족되어야 `Reservation` 객체를 반환하고, 그렇지 않으면 `null`을 반환한다.

시그니처의 `?`가 핵심이다. **호출 코드에게 "이 반환값은 `null`일 수 있다"는 사실을 강제로 알려준다.** C#의 널 가능 참조 형식 기능이 켜져 있는 상태라면 컴파일러가 `null` 처리 누락을 경고한다. 시그니처 하나에 본질이 다 담긴, 좋은 추상화다.

호출 측 `Post`는 이렇게 정리된다.

```csharp
public async Task<ActionResult> Post(ReservationDto dto)
{
    if (dto is null)
        throw new ArgumentNullException(nameof(dto));

    Reservation? r = dto.Validate();
    if (r is null)
        return new BadRequestResult();

    var reservations = await Repository
        .ReadReservations(r.At)
        .ConfigureAwait(false);
    int reservedSeats = reservations.Sum(r => r.Quantity);
    if (10 < reservedSeats + r.Quantity)
        return new StatusCodeResult(
            StatusCodes.Status500InternalServerError);

    await Repository.Create(r).ConfigureAwait(false);
    return new NoContentResult();
}
```

이제 정적 흐름 분석기 기능을 끌 필요가 없고, 날짜 구문 분석도 한 번뿐이다. `Post` 메서드의 순환 복잡도는 **4** 로 떨어졌다.

이 기법은 알렉시스 킹(Alexis King)이 정리한 **"유효성 검사 대신 분석하라(Parse, don't validate)"** 원칙이다.

> 생각해 보세요. 파서가 뭔가요? 실제로 파서는 덜 구조화된 입력을 받아서, 조금 더 구조화된 출력을 생성하는 기능일 뿐입니다. 파서는 본질적으로 부분 함수이며, 도메인의 일부 값은 적절한 범위에 해당하지 않으므로, 모든 파서는 어느 정도 실패에 대한 개념을 가져야 합니다.
> — 알렉시스 킹(Alexis King)

`Validate`는 덜 구조화된 `ReservationDto`를 입력받아 더 잘 구조화된 `Reservation`을 출력한다. **본질적으로 파서다.** 이름을 `Parse`로 지어도 되지만, "파싱은 텍스트를 다루는 것"이라는 통념 때문에 혼동을 피하려 `Validate`를 유지했다.

이는 5장의 **캡슐화(*Encapsulation - 객체가 자신의 사전·사후 조건, 즉 불변성을 스스로 보호하는 것*)** 관점과 이어진다. 올바르게 초기화된 `Reservation` 객체는 이미 유효한 상태다. 그렇지 않으면 생성자가 사전 조건을 확인하지 못해 캡슐화가 깨진다. **도메인을 모델링한 클래스는 변하지 않는 것들을 포함**해야 하고, 이는 데이터 전송 객체(DTO)와 대조된다.

## 3. 프랙탈 구조 — 모든 배율에서 머리에 들어오는 코드

### 3.1 덩어리 뒤에 감춰진 복잡도

`Post` 메서드의 육각꽃 다이어그램에는 이제 4개의 슬롯만 채워져 있다. 나머지 3개는 앞으로 새로운 규칙을 추가할 여유 공간이다.

물론 `Validate` 덩어리 뒤에는 복잡도 5가 그대로 숨겨져 있다. **덩어리 안을 확대해 보면 다시 육각꽃**이 나타난다. 축소하면 어떻게 될까? `Post`의 직접 호출자는 없지만, ASP.NET 프레임워크가 `Startup` 클래스의 설정을 기반으로 컨트롤러를 호출한다. `Startup` 클래스 전체의 순환 복잡도는 5 정도로 낮고, 이 역시 육각꽃 하나에 그려진다.

**확대 수준에 상관없이 복잡성의 구조는 비슷한 형태로 보인다.** 이 특성이 수학의 프랙탈과 닮았기 때문에 저자는 이를 **프랙탈 구조(*Fractal Architecture - 어느 배율에서 보아도 자기 유사성을 유지하며 각 층이 머리에 들어오도록 정리된 코드 구조*)** 라 부른다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 프랙탈 구조는 저절로 만들어지지 않는다. 작성하는 **각 코드 블록에서 복잡도를 명확하게 고려**해야 한다. 순환 복잡도를 계산하거나, 코드 줄을 세거나, 변수의 수를 세는 방법이 있다. 다만 **복잡도를 정확히 평가하는 것보다 낮게 유지하는 것이 더 중요**하다.
:::

수학적 프랙탈과 달리 코드베이스에서는 무제한으로 확대·축소를 계속할 수 없다. 결국 다른 코드를 호출하지 않는 리프 노드에 도달한다. 이런 형태를 트리로 그리면 몸통에서 7개의 가지가 뻗어나가고, 각 가지에서 다시 7개의 가지가 갈라지는 프랙탈 트리가 된다.

### 3.2 변수 세기 — 또 다른 관점

메서드에 등장하는 **변수의 수**를 세는 것도 복잡도의 또 다른 관점이다. 관련된 것을 모두 계산해야 한다 — 로컬 변수, 메서드 인수, 클래스 필드까지.

예제 `Post` 메서드에는 `dto`, `r`, `reservations`, `Repository`, `reservedSeats` 총 5개가 있다. 이 정도는 머리에 담을 만하다. 이 방법은 **메서드에 매개변수를 하나 더 추가해도 되는지 판단할 때** 유용하다. 매개변수 4개는 7개 한계 안이지만, 그것이 로컬 변수 5개, 클래스 필드 3개와 상호작용한다면 이미 너무 많은 일이 벌어지고 있는 셈이다. 이럴 때 **매개변수 객체(*Parameter Object - 여러 매개변수를 하나의 객체로 묶는 리팩터링 기법*)** 를 사용한다.

## 4. 결론

코드베이스는 처음부터 레거시로 태어나지 않는다. 시간이 지나면서 퇴화한다. 브라이언 푸트(Brian Foote)와 조셉 요더(Joseph Yoder)가 지적한 대로 **품질 높은 코드는 불안정한 균형 상태**다.

> 아이러니하게도, 이해 가능성이 높을수록 우리가 만들어낸 결과물을 이해할 수 있는 속도보다 빠르게 변화시킬 수 있게 되면서 보존에 불리하게 작용할 수 있습니다. 인터페이스가 분명하고 내부를 이해하기 어려운 결과물은 비교적 온전하게 남아 있을 수 있습니다.
> — 브라이언 푸트 & 조셉 요더

즉 좋은 코드일수록 손대기 쉬워서 오히려 부패에 노출된다. 그래서 **적극적으로 부패를 막아야** 한다. 이 장의 휴리스틱을 정리하면 다음과 같다.

| 휴리스틱 | 임계값/원칙 | 목적 |
|---|---|---|
| 순환 복잡도 | ≤ 7 | 두뇌 작업 기억 용량 안에 유지 |
| 메서드 줄 수 | ≤ 24 | VT100 창 안에 들어오는 크기 |
| 한 줄 너비 | ≤ 80자 | 좌우 스크롤 없이 읽히는 폭 |
| 응집 | 필드를 안 쓰는 코드는 다른 곳으로 | 같은 속도로 변화하는 것끼리 |
| 기능 편애 | 데이터가 있는 클래스로 이동 | 자기 데이터를 안 쓰면 남의 자리 |
| 유효성 검사 대신 분석 | 부울 대신 도메인 객체 반환 | 검사와 파싱을 한 번에, 캡슐화 유지 |
| 변수 세기 | 로컬+인수+필드의 합계 | 추적할 것이 몇 개인지 감각 |

지표들이 보편적이라는 환상을 품을 필요는 없다. 척도는 유용한 지침일 뿐, **궁극적으로는 자신이 판단**을 내려야 한다. 하지만 척도에 지속적으로 관심을 기울이면 코드 부패에 대한 경각심을 높일 수 있다.

**어디를 보든지 코드가 머리에 잘 들어오도록 코드베이스의 아키텍처를 만드는 것을 목표로 삼는다.** 높은 수준에서는 7개 이하의 일이 일어나야 하고, 하위 수준에서도 추적해야 할 것이 7개를 넘지 않아야 한다. 중간 수준도 마찬가지다. 이 자기 유사성이 프랙탈을 닮았으므로, 저자는 이를 프랙탈 구조라 부른다.

## 요약

- 코드는 서서히 부패한다 — 하루아침이 아니라 조금씩. 문제를 발견했을 때는 이미 손쓰기에 늦은 경우가 많다
- **임계값 기반 자동 규칙**이 부패를 막는 핵심 장치다. 특정 숫자보다 "자동으로 강제된다"는 사실이 중요
- **순환 복잡도**: 1에서 시작해 분기/루프/`&&`/`||`/`??`를 만날 때마다 +1. 7 이하로 유지
- **80/24 규칙**: 크리스토퍼 알렉산더 이전, VT100 터미널의 화면 크기에서 유래. 인간이 한눈에 파악하는 창의 크기
- **응집**은 켄트 벡의 정의를 빌리면 "같은 속도로 변화하는 것들은 함께" 있는 상태. 클래스 필드를 안 쓰는 코드 블록은 수상하다
- **기능 편애**: 다른 클래스의 데이터만 쓰는 메서드는 그 클래스로 옮긴다. 매개변수가 사라지면 시그니처가 단순해진다
- **유효성 검사 대신 분석하라(Parse, don't validate)**: 부울 플래그 대신 파싱된 도메인 객체를 반환. 캡슐화·널 안전성·중복 파싱 제거를 한 번에 해결
- **육각꽃 다이어그램**은 단기 기억 슬롯 7개를 시각화한다. 각 덩어리 뒤에는 또 다른 육각꽃이 감춰져 있다
- **프랙탈 구조**: 어느 배율에서 보든 코드가 머리에 잘 들어오는 아키텍처. 저절로 만들어지지 않으며 매 블록마다 의식적으로 복잡도를 관리해야 함
- **변수 세기**(로컬+인수+필드)는 매개변수 추가 여부를 판단할 때 유용한 보조 지표
- 지표는 지침일 뿐 절대적 진리가 아니다. 그러나 지표에 계속 관심을 기울이면 부패에 대한 경각심이 유지된다

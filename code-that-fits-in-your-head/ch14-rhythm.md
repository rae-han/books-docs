# Chapter 14: Rhythm (리듬)

## 핵심 질문

특정 개발 방법론(스크럼, XP, 칸반 등)을 따르는 것과 무관하게, 하루와 팀에는 어떤 리듬을 부여해야 지치지 않고 지속적으로 소프트웨어를 만들 수 있는가? 개인의 집중력과 팀의 협업 구조 사이에서 어떤 균형을 찾아야 하는가?

---

## 1. 왜 "리듬"인가

수많은 개발 조직을 만나 함께 일하다 보면 재미있는 사실을 발견하게 된다. **특정 프로세스를 따른다고 말하지만 실제로는 다르게 행동하는** 경우가 많다.

- 매일 스탠드업(*Stand-up - 짧게 서서 진행하는 데일리 미팅*)을 한다고 하지만 기분에 따라 이틀에 한 번 하는 팀
- 훌륭한 칸반 보드(*Kanban Board - 작업 흐름을 시각화하는 보드*)를 갖췄지만 대부분의 시간을 문제 해결에만 쓰는 팀
- 반면 별다른 프로세스는 없지만, 지속적 배포로 이해관계자가 소화하기 어려울 만큼 빠르게 기능을 내보내는 팀

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 이 책은 특정 방법론(스크럼, XP, PRINCE2)을 채택하라고 강요하지 않는다. 다만 **하루의 리듬이나 구조를 느슨하게 가져가는 것**이 개인과 팀 모두에게 도움이 된다는 사실을 저자는 오랜 경험으로 확인했다.
:::

리듬은 두 층위로 나뉜다.

1. **개인적 리듬**: 하루 일과를 어떻게 구성할 것인가
2. **팀의 리듬**: 팀 활동을 어떻게 정기화할 것인가

---

## 2. 개인적인 리듬

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 하루 단위로 빡빡하게 의무적인 규칙을 만들면 안 된다. 하루라도 못 지키면 스트레스가 쌓인다. 그러나 **느슨한 구조**는 시간 낭비를 최소화한다.
:::

### 2.1 타임박싱(*Timeboxing - 정해진 시간 안에 작업을 수행하는 기법*)

가장 유용한 도구 중 하나는 25분 단위 집중 + 5분 휴식 사이클이다. 이를 **포모도로 테크닉(*Pomodoro Technique - 25분 작업 + 5분 휴식을 반복하는 시간 관리 기법. 이탈리아어로 토마토를 뜻하는 pomodoro는 발명자 프란체스코 시릴로가 사용한 주방 타이머의 모양에서 유래*)** 이라 부르기도 하지만, 원조 포모도로에는 더 많은 부가 활동이 있어 이와는 조금 다르다.

25분 타임박스의 장점은 명백한 것과 감춰진 것으로 나뉜다.

| 구분 | 장점 |
|---|---|
| 명백한 장점 | 25분 동안 방해 없이 몰입 가능 |
| 명백한 장점 | "적어도 25분만 확인해보자"라고 스스로를 설득하여 시작 저항을 낮춤 |
| 명백한 장점 | 카운트다운 타이머를 보면 트위터/이메일 유혹을 이길 수 있음 |
| 숨은 장점 | 휴식이 강제되어 잘못된 방향의 코드를 조기에 알아챔 |
| 숨은 장점 | 15분 낭비가 3시간 낭비보다 훨씬 낫다 |

> 프로그래머는 몰입 상태(*flow*)에서 생산성이 극대화된다고 믿지만, 실제로 그렇다는 보장은 없다. 집중한다고 해서 깊게 생각할 수 있는 것은 아니다. 많은 코드를 작성하더라도 실제로 유용할 거라는 보장은 없다.

가장 흥미로운 부분은 이것이다. **몰입이 정말 유용한 상태라면 5분 정도 쉬는 것은 아무 문제가 되지 않는다.** 컴퓨터에서 몇 분 떨어져 있어도 다시 돌아오면 곧바로 몰입 상태로 복귀할 수 있다.

### 2.2 휴식 취하기

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 컴퓨터에서 떨어져 있을 때 오히려 생산성이 폭발한다. 유레카의 순간은 대부분 자리에서 일어나 있을 때 찾아온다.
:::

저자의 경험담이다.

> 오픈소스 프로젝트의 대규모 재설계를 해야 했을 때, 나는 대부분의 설계를 자전거로 출퇴근하는 30분 동안 해냈다. 규칙적으로 운동하거나, 뛰거나, 샤워를 하거나, 설거지를 할 때 많은 깨달음이 찾아왔다. 컴퓨터 앞에서 깨달음을 얻었던 순간은 단 한 번도 없다.

이는 **시스템 1(*System 1 - 대니얼 카너먼이 『생각에 관한 생각』에서 제시한 개념. 빠르고 직관적이며 무의식적인 사고 체계*)** 이 배후에서 문제를 계속 굴리기 때문이다. 다만 조건이 있다. 컴퓨터 앞에서 **먼저 충분히 문제와 씨름한 뒤**에만 이 효과가 나타난다. 소파에 누워 깨달음의 흐름이 스쳐지나가길 바랄 수는 없다.

권장: **20~30분간 컴퓨터에서 벗어나 신체 활동을 곁들여라.** 근처 마트에서 장을 봐도 좋다. 지적 노동은 육체 노동과 달라 시간으로 생산성을 측정할 수 없다. 오래 일할수록 오히려 생산성이 떨어진다.

### 2.3 시간을 계획적으로 사용하기

『실용주의 프로그래머』는 **매년 새로운 언어를 배워라**를 제안한다. 저자는 이에 완전히 동의하지는 않지만, 계속 배워야 한다는 취지에는 공감한다.

저자의 개인 루틴은 다음과 같다.

- 특별한 약속이 없는 한 **매일 아침 25분짜리 타임박스 두 개**를 학습에 할애
- 요즘은 책 읽고 연습문제 풀기
- 초기에는 스택 오버플로/유즈넷에 답변 달기 — 가르치면서 배운다
- **카타(*Kata - 반복 연습을 통해 코드 작성 기술을 익히는 훈련*)** 프로그래밍도 시도

또 다른 팁: **회의를 제한하라.** 회의가 많은 회사에 컨설팅했을 때, 대부분의 회의가 사실은 "정보 요청"에 불과했다. 회의 내용을 기록해두고 안건이 오면 문서로 대신 답변하니 회의를 대량으로 취소할 수 있었다. 문서는 많은 사람에게 배포 가능하지만, 회의는 규모가 제한된다.

### 2.4 자판 외우기

저자는 자녀들에게 하루 한 시간씩 타이핑 연습을 시킨 경험이 있다. **자판을 외우지 못하는 프로그래머와 일해보면 그 비효율이 상당하다.**

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 자판을 외워야 하는 이유는 타이핑 속도 때문이 아니라 **화면을 보지 못하기 때문**이다. 독수리 타법을 쓰면 IDE가 실시간으로 알려주는 경고, 자동 완성, 오타 감지 등의 신호를 놓친다.
:::

- 최신 IDE는 입력 과정에서 실시간으로 실수를 알려준다
- 자판을 보느라 이 힌트를 놓치면 컴파일할 때까지 오류를 발견하지 못한다
- 짝 프로그래밍을 하다 보면 뻔한 오타를 상대가 알아채는 데 수십 초씩 걸리는 고통을 겪는다

저자는 IDE를 **통합 개발 환경(Integrated Development Environment)** 이 아니라 **대화형 개발 환경(Interactive Development Environment)** 이라 부르는 편이 더 적절하다고 말한다. 자판을 외우지 못하면 이 상호작용을 놓치게 된다.

## 3. 팀의 리듬

팀에서 일할 때는 개인 리듬을 팀 리듬과 맞춰야 한다. 매일 스탠드업, 격주 회고, 정해진 점심 시간 — 이런 반복 활동이 있다. 특정 프로세스를 강요하진 않지만, **일정에 반드시 포함시켜야 할 몇 가지 활동**이 있다.

### 3.1 주기적으로 의존성 갱신하기

코드베이스는 의존성을 가진다. NuGet(.NET), NPM(JavaScript), RubyGems(Ruby) 같은 패키지 관리자를 통해 이 의존성이 공급된다. 패키지 배포자는 지속적 배포를 실현하고 있어, 새 기능이나 보안 패치를 자주 내보낸다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 최신 버전을 무조건 쫓아갈 필요는 없다. 하지만 **너무 뒤처지는 것은 위험하다**. 패키지 변경사항이 쌓이면 어느 순간 업데이트를 시도할 엄두조차 나지 않게 된다. 언어/플랫폼 버전이 오래되면 새 직원 채용도 어려워진다.
:::

역설적이게도 **정기적으로 업데이트하면 어려움이 없다**. 예제 코드베이스의 깃 로그를 보면 의존성 업데이트 커밋이 규칙적으로 섞여 있다.

```
0964099 (HEAD) Add a schedule link to each day
2295752 Rename test classes
fdf2a2f Update Microsoft.CodeAnalysis.FxCopAnalyzers NuGet
9e5d33a Update Microsoft.AspNetCore.Mvc.Testing NuGet pkg
f04e6eb Update coverlet.collector NuGet package
3bfc64f Update Microsoft.NET.Test.Sdk NuGet package
a2bebea Update System.Data.SqlClient NuGet package
34b818f Update xunit.runner.visualstudio NuGet package
ff5314f Add cache header on year calendar
```

**얼마나 자주?** — 이는 의존성 개수와 안정성에 따라 달라진다. 저자의 책 예제 코드베이스는 두 달에 한 번. 대규모 코드베이스라면 더 자주. 실험을 통해 적절한 빈도를 찾아야 한다.

권장 리듬: **2주짜리 스프린트라면 새 스프린트의 첫 작업으로** 의존성 업데이트를 배치.

### 3.2 다른 작업 일정 잡기

의존성 업데이트처럼 **깜빡하기 쉽고, 놓치면 늦어버리는** 작업들이 몇 가지 있다.

| 작업 | 잊었을 때의 결과 |
|---|---|
| TLS 인증서(*Certificate - HTTPS 등에 사용되는 디지털 인증서, 보통 몇 년 주기로 만료*) 갱신 | 인증서 만료 → 서비스 중단 |
| 도메인 이름 갱신 | 도메인 만료 → 사이트 접근 불가 |
| 데이터베이스 백업 검증 | 재해 시 복구 실패 |

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 자동 백업이 돌아가고 있다는 것과 **실제로 백업에서 복원이 되는 것**은 다른 문제다. 정기적으로 백업 복구를 연습하라. 실제 필요할 때 백업이 동작하지 않으면 절망적이다.
:::

이런 작업들은 반드시 **일정을 예약**해두어야 한다. 원 개발자가 회사에 남아있지 않을 시점에 문제가 터지기 쉽기 때문이다.

### 3.3 콘웨이의 법칙

저자는 1994년 첫 직장에서 개인 사무실을 받았다. 이후 다시는 개인 사무실을 가져본 적이 없다. 오픈 오피스가 비용상 유리하고, XP 같은 애자일도 팀 공동 착석을 권장하기 때문이다.

저자는 **오픈 오피스를 좋아하지 않는다.** 시끄럽고 산만하다. 하지만 얼굴을 맞대는 대화가 협력을 촉진한다는 점은 인정한다. 채팅 포럼에서 며칠씩 이어지는 논쟁도 15분 만남으로 해결되는 경우가 많다.

**그러나** 대면 대화에만 의존하면 **구전 문화**의 위험이 있다. 아무것도 기록되지 않고, 같은 토론이 반복되고, 사람이 떠나면 지식도 사라진다.

여기서 콘웨이의 법칙이 등장한다.

> 시스템을 설계하는 조직은 필연적으로 해당 조직의 커뮤니케이션 구조를 복제한 설계 구조를 만들어낼 수밖에 없다.
> — 멜빈 콘웨이(Melvin Conway)

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 모든 사람이 언제든 누구와도 대화할 수 있으면, **알아보기 어려운 프로그램 구조**가 만들어진다. 필요에 따라 이곳저곳 임시로 호출하고 통신하는 형태 — 곧 스파게티 코드다.
:::

저자가 선호하는 방식은 **오픈소스 스타일의 비동기 협업**이다.

- 풀 리퀘스트(*Pull Request - 변경사항을 병합해달라고 요청하는 협업 단위*)
- 리뷰
- 대부분을 글로 소통

이런 구조는 비동기 소프트웨어 개발을 가능케 하고, 자연스럽게 코드베이스도 잘 정돈된다. 이 방식이 유일한 정답은 아니지만, **팀 구성과 아키텍처가 연결되어 있음을 인식하는 것**이 요점이다. 선호하는 아키텍처와 소통 형식 모두를 촉진하는 방향으로 팀을 구성해야 한다.

---

## 4. 결론

개인 생산성에 대한 책은 이미 많으므로 여기서 자세히 다루지 않았다. 어떤 방식으로 일할 것인지는 개인적 결정이고, 팀 구성 방식도 매우 다양하다.

저자가 몇 년에 걸쳐 깨달은 최소한의 원칙 세 가지:

1. **휴식을 취하라.**
2. **휴식할 때는 컴퓨터에서 벗어나라.**
3. **다른 일을 하는 동안 가장 좋은 아이디어가 나온다.**

## 요약

- 방법론(스크럼/XP/PRINCE2 등)을 강요하지 않지만 **하루의 느슨한 구조**는 개인과 팀 모두에게 유용하다
- 개인 리듬: **타임박싱(25분 + 5분)** 으로 시작 저항을 낮추고 오판을 조기에 발견
- 몰입은 신화가 아니지만, **컴퓨터에서 벗어난 시간**이 최고의 아이디어를 만든다
- 지적 노동은 오래 할수록 생산성이 떨어진다 — 오래 일하지 말 것
- 매일 아침 학습 타임박스 확보, 회의는 문서로 대체
- **자판을 외워라** — 타이핑 속도가 아니라 IDE와의 상호작용을 위해
- 팀 리듬: **의존성 정기 업데이트**, 인증서/도메인/백업 검증 일정 예약
- **콘웨이의 법칙**: 조직의 커뮤니케이션 구조가 시스템 구조를 결정 → 팀 구성과 아키텍처는 함께 설계
- 대면과 문서화의 균형: 저자는 오픈소스식 비동기 협업(PR + 리뷰 + 글로 소통)을 선호

# Chapter 9: Teamwork (팀워크)

## 핵심 질문

소프트웨어 개발자는 거의 언제나 팀으로 일한다. 그런데 우리는 팀으로 일하기 위해 어떤 프로세스를 실제로 사용하고 있는가? 깃(Git)은 단순한 버전 관리 도구인가, 팀원 간 상호작용을 위한 도구인가? 커밋 메시지와 커밋 크기는 왜 중요한가? 코드의 공동 소유는 어떻게 달성하며, 짝 프로그래밍·몹 프로그래밍·코드 리뷰는 어느 지점에서 서로를 대체하고 어느 지점에서 서로를 보완하는가?

---

## 1. 프로세스와 결과 — 왜 헷갈리면 안 되는가

엔지니어가 된다는 것은 다양한 **프로세스(*Process - 결과를 얻기 위해 반복적으로 따르는 절차*)** 를 따르는 것을 의미한다. 프로세스를 따르면 성공 가능성이 올라간다. 그러나 프로세스와 결과를 혼동하면 안 된다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 프로세스는 결과에 도달하기 위한 **대리자(proxy)** 다. 어떤 프로세스가 유익한 이유를 이해해야, 그 프로세스를 따라야 할 때와 따르지 않아야 할 때를 판단할 수 있다.
:::

행동의 직접적인 결과는 대부분 측정하기 어렵다. 지금 당장은 긍정적으로 보여도 6개월 뒤에는 부정적인 영향을 남길 수 있다. **기술 부채(*Technical Debt - 단기 편의를 위한 낮은 품질의 코드가 축적되어 나중에 갚아야 할 상태*)** 가 시간이 지나면서 누적되는 것도 같은 맥락이다.

이 장에서 다룰 프로세스들은 완벽한 보장이 아니라, **결과에 도달할 확률을 높이는 실천법**이다. 따라야 할 때와 우회해야 할 때를 스스로 판단할 수 있어야 한다.

## 2. 깃(Git) — 단순한 버전 관리 도구가 아니다

대부분의 소프트웨어 조직은 CVS나 서브버전 같은 도구 대신 이미 깃을 사용한다. 분산 제어 도구임에도 실제로는 깃허브(GitHub), 애저 데브옵스 서비스(Azure DevOps Services), 스태시(Stash), 깃랩(GitLab) 같은 중앙 집중형 서비스와 함께 쓴다.

그런데 대부분의 개발자는 깃을 **"내 코드를 팀 코드베이스에 통합하는 수단"** 정도로만 여긴다. **팀원과 상호작용하기 위한 도구**로는 좀처럼 생각하지 않는다.

> 이렇게 사용한다면 굳이 깃을 쓸 필요가 없다. 기회를 낭비하는 셈이다.

깃을 조금 더 전략적으로 사용해야 한다.

### 2.1 커밋 메시지 — 50/72 규칙

커밋할 때는 커밋 메시지를 써야 한다. 대부분의 프로그래머에게 이건 "귀찮지만 어떻게든 넘기는 장애물"이다. 깃은 메시지가 비어 있으면 오류를 내지만, 아무 내용이나 넣기만 하면 통과한다.

사람들은 보통 커밋에 **'무엇'** 이 포함되는지만 적는다. "첫 번째 이름 추가됨", "빈 이야기 부분 제거", "CustomerUpdated 핸들러 추가" 같은 식이다. 하지만 이런 메시지는 생각만큼 쓸모가 없다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 커밋 메시지는 미래의 자신과 팀원에게 보내는 편지다. `git diff` 로 무엇이 바뀌었는지는 이미 볼 수 있으니, 커밋 메시지는 **"왜 바꿨는지"** 에 집중해야 한다.
:::

깃 커밋 메시지에는 팀 포프(Tim Pope)의 **50/72 규칙**이라는 사실상의 표준이 존재한다.

- 요약(subject)은 **50자 이내**, **명령형**으로 작성한다.
- 추가 내용이 필요하면 두 번째 줄은 **빈 줄**로 둔다.
- 이후 본문은 원하는 만큼 쓰되, 각 줄은 **72자를 넘지 않게** 한다.

이 규칙은 깃의 다양한 기능이 동작하는 방식에 뿌리를 둔다. 커밋 목록을 볼 때 `git log --oneline` 은 요약 부분만 보여준다.

```
$ git log --oneline
8fa3e47 (HEAD) Make /reservations URL segment lowercase
fbf74ae Return IDs from database in range query
033388a Return 404 Not Found for non-guid id
0f97b34 Return 404 Not Found for absent reservation
ee3c786 Read existing reservation
62f3a56 Introduce TimeOfDay struct
```

명령줄뿐 아니라 GUI 도구들도 대부분 깃 CLI를 감싸서 동작하므로, 50/72를 지키면 어디서 봐도 문제가 없다.

**요약은 명령형으로 쓰자.** 강력한 이유가 있는 것은 아니고 일종의 관습이다. 저자도 몇 년 동안 과거형("Introduced ...")으로 써왔지만 규칙을 알고 나서 명령형("Introduce ...")으로 바꿨다. 명령형이 과거형보다 대체로 짧다. `return`은 `returned`보다 짧으므로 50자 안에 요약을 담기가 더 쉽다.

커밋이 자명하다면 요약만 있어도 된다. 하지만 이유가 궁금해질 만한 변경이라면 본문에 **왜 그렇게 바꿨는지**를 남긴다. 예를 들어 다음과 같은 메시지는 두 가지 질문에 답한다.

```
Introduce TimeOfDay struct

MaitreD 생성자 매개변수의 의미를 명확히 하기 위해 도입.
TimeOfDay 타입 대부분은 Visual Studio 가 자동 생성한 코드다.
```

- TimeOfDay 타입을 도입한 이유
- 대부분의 코드가 테스트 주도로 작성되지 않은 이유

> 소프트웨어 개발에서 코드 이면에 있는 "왜"를 이해하는 것이 가장 중요한 문제일 수 있다. 그러므로 그 부분을 최대한 명확하게 남겨야 한다.

### 2.2 지속적 통합(Continuous Integration) — 서버가 아니라 실천

**지속적 통합(*Continuous Integration - 동료의 작업 코드와 자주 병합하여 통합 지점을 항상 살아있게 유지하는 실천*)** 은 대부분의 조직이 "이미 하고 있다"고 착각하는 실천이다. 실제로는 지속적 통합 **서버**를 사용하는 것과 혼동한다. 서버가 있다고 지속적 통합이 되는 것은 아니다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 지속적 통합은 도구가 아니라 **일하는 방법**이다. "동료가 작업 중인 코드와 지속적으로 코드를 통합해야 한다." 여기서 통합은 **병합(merge)** 을 뜻하며, 경험적으로 **최소 4시간에 한 번**은 공유되어야 한다.
:::

깃이 뛰어나다고 이야기하는 개발자들은 대부분 "병합 지옥 문제를 해결해준다"고 말한다. 아이러니하게도 깃 자체가 병합 지옥을 해결해주지는 않는다. 다만 중앙 집중식 도구와 다른 워크플로를 가능하게 만들 뿐이다.

**병합 지옥의 근본은 도구가 아니라 동시성(concurrency)** 이다. 데이터베이스 트랜잭션과 같은 문제를 다루고 있는 것이다. 여러 클라이언트가 공유 자원을 동시에 수정하려 하면 충돌이 생긴다.

| 잠금 방식 | 동작 | 한계 |
|---|---|---|
| **비관적 잠금(*Pessimistic Locking - 자원을 편집하기 전 잠금을 걸어 다른 사람의 편집을 막는 방식*)** | 파일을 체크아웃 → 잠금 → 체크인해야 잠금 해제 | 조직이 커지면 확장 불가. 잠근 채 퇴근하면 팀 전체 정지 |
| **낙관적 잠금(*Optimistic Locking - 편집 시작 시점의 스냅샷을 저장하고, 저장 시 현재 상태와 비교해 충돌 여부를 판단하는 방식*)** | 스냅샷 저장 → 편집 → 저장 시 현재 상태와 비교 | 충돌 자체는 완전히 막지 못하지만 확장성이 좋음 |

깃은 낙관적 잠금 계열이다. 같은 코드의 다른 줄을 서로 바꿨다면 자동 병합되지만, 같은 줄을 서로 바꿨다면 **병합 충돌(merge conflict)** 이 발생한다.

**충돌 확률을 낮추는 유일한 방법은 편집 창을 짧게 유지하는 것**이다.

지속적 통합이 "trunk 브랜치에서만 작업하는 것"이라고 오해하는 사람들이 있다. 어떤 이는 이를 문자 그대로 받아들여 브랜치를 하나도 안 만든다. 하지만 **문제는 브랜치 이름이 아니라 동시성**이다. 모든 동료와 몹 프로그래밍을 하고 있는 게 아니라면, 누군가가 나와 같은 줄을 편집할 위험은 늘 존재한다.

> 위험을 줄여라. 작게 변경하고, 가능한 한 자주 병합하라. 최소 4시간마다 통합하는 것을 권장한다. 반나절 이상 팀원들과 공유하지 않은 상태로 쌓아두면 로컬 저장소가 분기되면서 병합 지옥에 빠질 수 있다.

4시간 안에 기능을 완성할 수 없다면 **기능 플래그(*Feature Flag - 완성되지 않은 기능을 배포 코드에는 포함시키되 스위치로 감춰두는 기법*)** 뒤에 숨기고 어떻게든 통합한다.

### 2.3 작은 커밋 — 깃의 기동성

프로그래밍하는 동안 코드는 예측 불가능한 리듬으로 만들어진다. 어떤 4시간에는 많은 코드가 나오지만, 어떤 반나절엔 버그를 재현하느라 한 줄도 못 짤 수 있다. 익숙하지 않은 API를 이해하는 데 며칠이 걸리기도 하고, 몇 시간 짠 코드를 통째로 버리기도 한다.

깃의 중요한 장점은 **기동성(*Manoeuvrability - 위치 에너지를 운동 에너지로 얼마나 빨리 바꿀 수 있는지를 나타내는 전투기 용어. 소프트웨어에서는 방향을 얼마나 빨리 바꾸고 가속·감속할 수 있는지의 지표*)** 이다.

> 환경의 변동성이 클수록 예기치 못한 사건에 대처하는 능력이 더욱 중요해진다. 깃은 전술적 기동성을 제공한다.

어떤 코드를 실험해보고, 잘 동작하면 커밋하라. 안 되면 되돌려라. **이런 실험은 조금씩 커밋해나갈 때 가장 잘 동작한다.** 마지막으로 커밋한 코드의 양이 적을수록, 실제로 제거하려는 부분만 정확히 버릴 수 있다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 커밋 기록은 **동작하는 소프트웨어의 스냅샷 연속**이어야 한다. 동작하지 않는 코드는 커밋하지 말고, 코드가 성공적으로 빌드될 때마다 커밋하라.
:::

다음 상황들은 전부 커밋 대상이다.

- 심볼 이름을 바꾼 경우 → 커밋
- 메서드를 새로 만든 경우 → 커밋
- 메서드를 인라인화한 경우 → 커밋
- 테스트를 추가하고 통과시킨 경우 → 커밋
- 데이터 유효성 확인부를 추가한 경우 → 커밋
- 코드 포맷을 정리한 경우 → 커밋
- 주석을 추가한 경우 → 커밋
- 중복 코드를 제거한 경우 → 커밋
- 오타를 수정한 경우 → 커밋

띄엄띄엄 커밋하면 관련 없는 변경이 하나의 큰 커밋에 뭉쳐 있어, 나중에 일부만 되돌리기가 어렵다. 반대로 작은 커밋이 많을수록 **`git bisect`, `git revert`, 히스토리 재구성** 같은 기동이 쉬워진다.

> 로컬 브랜치를 푸시하기 전까지는 편집할 수 있다는 점이 좋다. 실수를 숨기거나 선견지명이 있는 척하기 위해서가 아니라, **편하게 실험하고 일관된 커밋 흔적을 남길 수 있기 때문**이다.

## 3. 코드의 공동 소유(Collective Code Ownership)

리나(Irina)가 코드베이스의 일부를 혼자 전담한다고 상상해보자. 그녀가 휴가를 가거나, 아프거나, 회사를 그만두면 어떻게 되는가?

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 코드베이스의 어떤 부분도 **한 사람만 접근 가능해서는 안 된다**. 코드 공유는 **버스 지수(bus factor)** 를 높인다.
:::

### 버스 지수(Bus Factor)

**버스 지수(*Bus Factor - 몇 명의 개발자가 버스에 치이면 프로젝트가 중단되는지를 나타내는 지표. 높을수록 안전*)** — "1"이라면 한 명이 자리를 비우는 순간 개발이 실패한다는 뜻이다. 어떤 사람은 이 용어를 병적으로 싫어해서 **복권 지수(lottery factor)** — "베라가 복권에 당첨돼 그만둬도 팀이 유지되는가?" — 라고 부르기도 한다. 표현만 다를 뿐 아이디어는 같다.

중요한 것은 **한 사람이라도 없으면 안 되는 상황이 발생하지 않도록 업무를 구성하는 것**이다.

한 팀에 개발자가 두 명 이상이면 자연스레 각자 특정 부분을 맡게 된다. 누구는 UI를, 누구는 백엔드를 선호하기 때문이다. 코드의 공동 소유는 이런 특화를 **금지하는 것이 아니라, 담당 영역을 겹치게 만드는 것**이다.

```
     Ann         Max         Sue
      │           │           │
      ▼           ▼           ▼
   ┌──────┐   ┌──────┐    ┌──────┐
   │ HTTP │   │Domain│    │Data  │
   │ API  │◀──▶Model ◀───▶│Access│
   └──────┘   └──────┘    └──────┘
              │      │
         Ann/Max   Max/Sue
```

Ann·Max·Sue 세 명이 한 코드베이스에서 일한다. Ann은 왼쪽·위쪽 모듈(HTTP API, 도메인 모델)을, Max는 위쪽·오른쪽 모듈을, Sue는 아래쪽 모듈을 좋아한다. **모두 다른 팀원과 코드베이스의 일부를 공유한다.**

코드의 공동 소유가 이뤄지고 있는지 다음 질문에 "예"라고 답할 수 있어야 한다.

> "팀 코드의 특정 부분을 작업하는 데 익숙한 사람이 **2명 이상** 있는가?"

이 상태를 만드는 공식적/비공식적 도구가 짝 프로그래밍, 몹 프로그래밍, 코드 리뷰다. **핵심은 모든 코드 변경에 두 명 이상이 관련되어야 한다는 것**이다.

### 3.1 짝 프로그래밍(Pair Programming)

**짝 프로그래밍(*Pair Programming - 두 개발자가 하나의 문제를 실시간으로 함께 푸는 방식*)** 은 두 사람이 같은 문제를 실시간으로 협력해 푸는 것이다. 다양한 스타일(driver-navigator 등)이 있지만 실시간 협업이라는 점은 동일하다.

이 과정 자체가 **끊이지 않는 코드 리뷰**다. 짝 프로그래밍으로 만든 코드는 세부 구현까지 이미 두 명에게 합의된 상태다. 결과적으로 커밋에는 최소 두 명에게 익숙한 코드가 들어간다.

다만 짝 프로그래밍만으로 공동 소유가 저절로 되지는 않는다. 지식이 특정 짝에 갇히지 않도록 **짝을 주기적으로 바꾸고, 담당 코드 부분도 순환**시켜야 한다.

> 저는 전형적인 내향형이라 남과 뭔가를 함께 하는 것이 피곤하다고 느낀다. 짝 프로그래밍 중에는 깊이 생각할 여유가 별로 없고, 서로 일정도 맞춰야 한다. 모든 팀이 짝 프로그래밍을 해야 하는 건 아니지만, 앞서 설명한 장점을 반박하기도 어렵다.

### 3.2 몹 프로그래밍(Mob Programming) / 앙상블 프로그래밍

두 명이 좋다면 세 명은? 네 명은? 여러 사람이 한 공간에서 하나의 코드를 함께 작성하는 방식이 **몹 프로그래밍(*Mob Programming - Woody Zuill이 주창. 팀 전체가 한 화면 앞에서 하나의 문제를 함께 푸는 실천*)** — 최근에는 **앙상블 프로그래밍**이라고도 부른다.

경영진에게 "짝 프로그래밍이 생산적입니다"라고 설득하기도 쉽지 않은데, "세 명 이상이 같은 문제를 풀어도 생산성이 떨어지지 않습니다"라고 설득하는 건 더 어렵다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 생산성은 **타이핑 속도가 아니다**. 다만 투입 대비 수익이 줄어드는 지점은 존재한다 — 50명이 하나의 몹을 이룬다면 대부분은 기여하지 못한다.
:::

저자는 프로그래밍 코치로서 몹 프로그래밍으로 큰 성공을 거둔 경험이 있다. 일주일에 2~3일씩 여러 프로그래머와 함께 프로덕션 코드베이스에 테스트 주도 개발을 적용하도록 도왔고, 몇 달 뒤 저자가 휴가를 다녀왔을 때도 팀은 계속 TDD를 사용하고 있었다. **몹 프로그래밍은 지식 전달에 아주 좋다.**

짝 프로그래밍과 마찬가지로, 몹 프로그래밍도 하나의 변경에 두 명 이상이 협력하는 방식이므로 별도의 검토·승인 절차가 사실상 필요 없다.

### 3.3 코드 리뷰 — 대기 시간을 줄여라

로랑 보사비트(Laurent Bossavit)의 지적처럼 소프트웨어 개발의 대부분의 '상식'은 실증되지 않았다. 그중 **효과가 실제로 문서화되어 입증된 몇 안 되는 실천이 코드 리뷰**다.

> 코드 리뷰는 코드의 결함을 찾아내는 **가장 효과적인 방법**이지만, 대부분의 조직에서는 사용하지 않는다. 개발 속도를 늦춘다고 여기기 때문이다.

이 인식은 잘못됐다. 확인되지 않은 버그를 안고 빠르게 개발하는 것이 결과적으로 더 늦다.

많은 조직에서 개발자는 하나의 기능을 담당하고, "완료"라고 보고하면 그걸로 끝난다. 어떤 조직은 이를 **"완전히 완료(done done)"** — 실제 시스템에 반영되어 사용 가능해야 진짜 완료 — 라 부르기도 한다. 그러나 '가치' 제공에만 근시안적으로 매달리면 엉성하고 잘못된 기능이 그대로 시스템에 들어간다.

문제는 **리뷰의 부재가 아니라 대기 시간**이다.

- **리뷰 없는 개발**: 기능 완료 → (한참 뒤) 버그 발견 → 계획 밖의 수정 작업 → 마감 초과 → 크런치 모드
- **일반적인 코드 리뷰**: 기능 완료 → (며칠 대기) 리뷰 → 개선 → 통합. 대기 시간이 며칠씩 늘어짐
- **바람직한 흐름**: 기능 완료 → (당일) 리뷰 → 개선 → 통합. 대기 시간 최대 4시간

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 코드 리뷰를 조직의 **매일 반복되는 루틴**으로 만들어라. 스크럼 미팅이 그렇듯이, 매일 정해진 시각에 하는 이벤트는 일을 안정적으로 앞으로 굴린다.
:::

예를 들어 **매일 아침 30분, 점심 이후 30분**을 코드 리뷰에 배정한다. 코드는 반나절 미만의 작은 단위로만 변경하는 것을 전제로, 모든 팀원이 하루에 두 번씩 작은 변경을 리뷰하면 **최대 대기 시간은 약 4시간**이 된다.

### 3.4 큰 변경사항 거부하기

저자는 어떤 조직이 고립된 개발에서 공동 소유 방식으로 전환하는 것을 돕던 중, 몇 주 동안 연락이 없던 원격 개발자로부터 **엄청난 크기의 풀 리퀘스트**를 받았다. 수천 줄, 50개 이상의 파일에 걸친 변경이었다.

> 내용이 너무 컸기 때문에 리뷰하지 않고 바로 거절한 다음, 전체 팀원과 협력해서 작은 변화를 만들어가는 방법을 알려주었다. 이후로는 다시 그 정도 크기의 풀 리퀘스트를 받지 않았다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 코드 리뷰가 단지 요식행위(rubber stamp)에 불과하다면 아무런 가치가 없다. **필요하다면 거부할 수 있어야 한다.**
:::

큰 변경사항을 거부하기 어려운 이유는 **매몰 비용 오류(*Sunk Cost Fallacy - 이미 투입한 시간·비용 때문에 손해가 커질 것을 알면서도 계속 밀고 가는 인지 편향*)** 다. 동료가 며칠(혹은 몇 주)을 쏟은 것은 사실이지만, 나쁜 설계를 유지보수하며 더 많은 시간을 낭비할 수 있다는 점을 생각하면 선택은 명확하다.

> 손실을 줄이세요. 동료가 낭비한 시간은 이미 지난 일이다. 잘못 구성된 코드로 더 이상 시간을 낭비하지 말라.

며칠 작업을 거부하는 건 마음이 아프지만 반나절 작업은 그나마 거부할 만하다. 이것이 **작은 변경사항**을 사용해야 하는 또 다른 이유다. 게다가 **1시간 이상 걸리는 코드 리뷰는 이미 비효율적**이다.

### 3.5 코드 리뷰에서 확인해야 할 것들

코드 리뷰가 답해야 하는 **근본적인 질문은 단 하나**다.

> "이 코드를 유지해도 괜찮은가?"

작성자는 당연히 유지할 만하다고 생각하고 올렸을 것이다. 그러므로 리뷰어인 내가 유지할 만하다고 판단해야, 코드가 **두 사람에게 공동 소유되는** 방향으로 옮겨간다.

가장 중요한 기준은 **가독성** — 즉 머릿속에 담기는지다. 문서는 낡고 주석은 오해를 부르기 쉽다. 궁극적으로 믿을 수 있는 유일한 산출물은 코드 그 자체이며, 유지보수 시점엔 작성자가 없을 수도 있다.

**작성자가 옆에 앉아 코드를 설명하는 리뷰는 바람직하지 않다.**

- 리뷰어가 코드 자체만으로 가독성을 판단할 수 없다.
- 작성자가 문제 있는 프랙티스를 어물쩡 넘어가도록 설득할 여지가 생긴다.

리뷰어의 임무는 **작성자에게 상처 주는 것도, 자기 잘남을 증명하는 것도 아니다.** 앞으로 어떻게 나아갈지 합의하는 것이다. 코드 포맷·변수명 같은 사소한 지적에 너무 매달리지 말고, **메서드가 너무 길거나 복잡하지 않은가**를 우선한다.

코리 하우스(Cory House)는 리뷰에서 확인할 항목을 다음과 같이 제시한다.

- 코드가 의도한 대로 동작하는가?
- 의도가 명확한가?
- 불필요한 중복이 있는가?
- 기존 코드로도 이 문제를 해결할 수 있었는가?
- 이보다 더 간단할 수 있는가?
- 테스트가 포괄적이고 명확한가?

코드 리뷰의 결과는 승인/거부라는 이분법이 아니라, **작성자와 검토자가 이야기할 제안 목록**이다. 작성자는 코드 읽기에는 개입하지 않지만, 도출된 개선사항에 대해서는 우호적으로 상호작용한다.

**모든 팀원이 작성자이자 리뷰어**여야 한다. 리뷰는 선택받은 소수의 특권도, 부담도 아니다. 이는 공동 소유를 촉진할 뿐 아니라, 모든 사람이 예의 바른 태도로 리뷰를 진행하게 만든다.

### 3.6 풀 리퀘스트(Pull Request)

깃허브, 애저 데브옵스 서비스 같은 온라인 깃 서비스는 **깃허브 플로우(GitHub Flow)** — 로컬에서 브랜치를 만들고 중앙 서비스에서 병합하는 경량 워크플로 — 를 지원한다. 브랜치를 master로 병합할 때 **풀 리퀘스트(pull request)** 를 요청한다.

풀 리퀘스트도 결국은 **코드 리뷰를 수행하는 다른 방식**일 뿐이다. 풀 리퀘스트를 만들 때 기억할 규칙은 다음과 같다.

| 규칙 | 이유 |
|---|---|
| 가능한 한 작게 만들기 — 생각보다 더 작게 | 큰 PR은 결국 대충 승인되거나 며칠씩 끌린다 |
| 한 가지만 처리하기 | 다양한 관심사가 섞이면 리뷰 초점이 흐려짐 |
| 코드 형태만 바꾸는 PR 피하기 | 실질 변경과 섞이면 diff 노이즈만 커짐 |
| 코드가 빌드 가능해야 함 | 리뷰 이전에 최소 조건 |
| 모든 테스트가 통과해야 함 | 위와 동일 |
| 새 동작에 대해 테스트 추가 | 회귀 방지 |
| 적절한 커밋 메시지 작성 | 50/72 규칙 적용 |

풀 리퀘스트 리뷰는 **비동기 작업**이므로, 코드가 이미 작성된 뒤에 리뷰가 이뤄진다. 글로만 이뤄지는 소통에서는 **의도와 어조가 쉽게 사라진다**. 상처 줄 의도가 없어도 상대는 상처받을 수 있으므로 예의를 갖추고, 필요하면 이모티콘도 활용해 친근한 톤을 남기는 것이 좋다.

리뷰어는 제대로 검토할 시간을 들여야 한다. 풀 리퀘스트가 너무 크면 **대충 보고 승인하는 것보다 그냥 거부하는 것이 낫다.**

리뷰하기로 결정했다면 작성자와 협력해 개선점을 찾자. **마음에 들지 않는 부분만 지적하지 말고 구체적인 대안**을 제시한다. 마음에 드는 부분이 있다면 응원하는 것도 잊지 말고, 필요하다면 코드를 로컬로 풀(pull)해서 직접 실행해본다.

## 4. 결론 — 공학에 가장 가까운 실천

팀원들은 각자 잘하는 기술을 몇 가지씩 가지고 있고, 자신에게 잘 맞는 코드베이스 영역에 자연스럽게 끌린다. 이를 **약한 코드 소유(*Weak Code Ownership - 특정 코드에 '자연스러운' 주 담당자는 있지만, 모든 사람이 변경 가능한 형태의 소유 방식*)** 라고 부르며, 이 정도는 괜찮다. 강한 개인 소유(누구도 남의 영역을 못 건드리는 상태)만 피하면 된다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 코드베이스에서 이뤄지는 **모든 변경사항에 두 명 이상**이 참여하게 만들어야 한다. 짝 프로그래밍·몹 프로그래밍으로 비공식적으로, 코드 리뷰·풀 리퀘스트로 공식적으로. **진행되는 모든 작업을 둘 이상의 눈으로 지켜보는 것은 소프트웨어 개발에서 사용할 수 있는 가장 공학에 가까운 방식 중 하나다.**
:::

이 장에서 반복적으로 나타난 휴리스틱은 결국 하나로 수렴한다.

- **작게 만든다** — 커밋도, 브랜치도, 풀 리퀘스트도, 리뷰 세션도.
- **자주 통합한다** — 최소 4시간에 한 번.
- **대기 시간을 줄인다** — 리뷰를 매일 반복되는 루틴으로.
- **두 명 이상이 본다** — 짝 프로그래밍, 몹 프로그래밍, 코드 리뷰 중 어느 방식이든.

큰 것들은 병합 지옥·대충 승인·매몰 비용 오류를 만든다. **크기를 줄이면 이 문제들이 대부분 사라진다.**

## 요약

- **프로세스와 결과를 혼동하지 말라.** 프로세스는 결과에 도달하기 위한 대리자이며, 이유를 이해해야 언제 따르고 언제 벗어날지 판단할 수 있다
- **깃은 팀원과 상호작용하는 도구**로 사용해야 한다 — 단순한 코드 통합 수단이 아니다
- **커밋 메시지는 50/72 규칙**을 따르라: 50자 이내 명령형 요약 + 빈 줄 + 72자 이내 본문. 본문은 "무엇"이 아니라 **"왜"** 를 설명한다
- **지속적 통합**은 서버가 아니라 실천이다. **최소 4시간에 한 번** 팀 코드와 병합하고, 완성 못 하면 기능 플래그 뒤에 숨기고라도 통합한다
- **병합 지옥의 근본은 브랜치 이름이 아니라 동시성**이다. 편집 창을 짧게 유지하는 것이 유일한 해결책
- **작은 커밋을 자주 만들어라** — 이름 변경, 메서드 추출, 포맷 정리, 오타 수정 하나하나가 커밋 대상. 그래야 revert·bisect 같은 깃의 기동성이 살아난다
- **코드의 공동 소유** — 코드베이스의 어떤 부분도 한 사람만 접근 가능해서는 안 된다. **버스 지수**를 높여야 한다
- **짝 프로그래밍**은 지속적인 실시간 코드 리뷰이며, 짝을 순환시키지 않으면 지식이 특정 짝에 갇힌다
- **몹 프로그래밍**은 지식 전달과 어려운 문제 해결에 강력하지만, 항상 기본값이 될 필요는 없다
- **코드 리뷰는 문서로 효과가 입증된 몇 안 되는 실천**이다. 문제는 리뷰의 부재가 아니라 **대기 시간** — 매일 정기 루틴으로 만들어 대기 시간을 4시간 이하로 낮춰라
- **큰 변경사항은 거부하라.** 매몰 비용 오류에 굴복하지 말고, 크기 자체가 리뷰 실패의 원인임을 팀에 학습시켜라
- **리뷰의 근본 질문은 "이 코드를 유지해도 괜찮은가?"** — 가독성이 최우선 기준이다
- **풀 리퀘스트도 결국 코드 리뷰**다. 작게, 하나만, 빌드·테스트 통과, 좋은 커밋 메시지, 예의 있는 톤을 지켜야 한다
- 모든 변경에 두 명 이상이 눈을 두는 것이 **소프트웨어 개발에서 가장 공학에 가까운 방식** 중 하나다

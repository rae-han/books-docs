# Chapter 11: Editing Unit Tests (유닛 테스트 편집하기)

## 핵심 질문

리팩터링의 전제 조건은 "확고한 테스트 스위트"라고들 한다. 프로덕션 코드는 테스트가 안전망이 되어주지만, **테스트 코드 자체를 편집할 때는 무엇이 안전망이 되어주는가**? 테스트를 잘못 건드리면 자기도 모르게 검증력이 약화된 채 계속 초록 불빛을 볼 수 있다. 그렇다면 유닛 테스트 코드를 어떻게 편집해야 안전할까?

---

## 1. 테스트 코드의 안전망 문제

『리팩터링』은 자동화된 테스트 스위트를 안전망 삼아 **프로덕션 코드**의 구조를 바꾸는 방법을 다룬다. 반면 『xUnit Test Patterns』의 부제는 "Refactoring Test Code"다. 그런데 테스트 코드를 편집한다는 것은 미묘한 문제다.

> 리팩터링을 위한 필수 선결 조건은 [...] 확고한 테스트다.

이 말이 사실이라면 **공식적으로는 유닛 테스트를 리팩터링할 수 없다**. 테스트를 검증할 상위 테스트가 없기 때문이다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 테스트 스위트는 리팩터링의 근간이지만, 테스트 코드는 편집할수록 신뢰할 수 없게 된다. 테스트를 편집할 때는 **조심스럽고, 신중하게 하나씩** 처리해야 한다.
:::

테스트 코드를 편집한다고 해서 모두 위험한 것은 아니다. 어떤 종류의 편집은 다른 것보다 훨씬 안전하다.

---

## 2. 안전한 편집: 새로운 것을 "추가"하기

테스트 코드를 가장 안전하게 편집하는 방법은 **새로 추가**하는 것이다. 기존 것을 건드리지 않으므로 기존 신뢰도가 유지된다.

### 2.1 새로운 테스트 클래스/메서드 추가

완전히 새로운 테스트 클래스를 추가하는 것이 가장 독립적이지만, 기존 클래스에 새 테스트 메서드를 추가하는 것도 무방하다. 각 테스트 메서드는 서로 독립적으로 실행되기 때문이다.

### 2.2 매개변수화된 테스트에 케이스 추가

기존 매개변수화된 테스트에 케이스를 한 줄 추가하는 것도 위험하지 않다.

```csharp
[Theory]
[InlineData(null, "j@example.net", "Jay Xerxes", 1)]
[InlineData("not a date", "w@example.edu", "Wk Hd", 8)]
[InlineData("2023-11-30 20:01", null, "Thora", 19)]
[InlineData("2022-01-02 12:10", "3@example.org", "3 Beard", 0)]
public async Task PostInvalidReservation(
```

한 줄 추가만으로 새로운 시나리오 하나를 커버한다.

### 2.3 어설션 추가 — 사후조건 강화

한 테스트에 어설션을 더 추가하는 것도 안전하다. 기존에 응답이 500인지만 확인하던 어설션에 응답 본문 검증을 얹은 예다.

```csharp
Assert.Equal(
    HttpStatusCode.InternalServerError,
    response.StatusCode);
Assert.NotNull(response.Content);
var content = await response.Content.ReadAsStringAsync();
Assert.Contains(
    "tables",
    content,
    StringComparison.OrdinalIgnoreCase);
```

"테스트는 어설션이 하나만 있어야 한다"는 **어설션 룰렛(*Assertion Roulette - 여러 어설션 중 어느 것이 실패했는지 알기 어려운 문제. 종종 "어설션은 한 개"라는 극단적 규칙으로 오해됨*)** 논리를 따르는 개발자를 만나곤 하지만, 이는 과도한 단순화다.

새 어설션은 **사후조건을 강화**한다. 강화된 사후조건은 위음성(*false negative — 실제 오류인데도 감지 못함*)을 줄인다.

이는 **리스코프 치환 원칙(*Liskov Substitution Principle - 하위 형식은 상위 형식이 있어야 할 곳에 항상 대체 가능해야 하며, 하위 형식에서 사전조건은 약화·사후조건은 강화만 허용되는 원칙*)** 을 떠올리게 한다. 시간을 하위형식 관계처럼 생각하면, **시간이 지나면서 시스템의 사후조건은 강화될 수 있지만 약화되어서는 안 된다**.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: 새로운 테스트나 어설션을 추가하는 것은 시스템에 대한 보증을 **강화**한다. 반대로 삭제는 보증을 **약화**시켜 회귀 버그의 통로가 된다. 삭제는 되도록 피하자.
:::

## 3. 테스트 리팩터링과 프로덕션 리팩터링은 분리한다

IDE에 내장된 리팩터링(변수 이름 바꾸기, 메서드 추출, 메서드 이동 등)은 프로덕션 코드는 물론 테스트 코드에서도 비교적 안전하다. 하지만 그 외의 변경은 위험하다.

문제는 이렇다. **프로덕션 코드**는 잘못 바꿔도 테스트가 알려준다. **테스트 코드**는? 실수로 사후조건을 약화시켰다면 그 실수 자체를 감지할 방법이 없다.

예를 들어 여러 곳에 흩어진 비슷한 어설션들을 도우미 메서드로 추출하는 리팩터링을 한다고 하자. "메서드 추출"은 안전한 리팩터링이지만, 이를 여러 곳에 적용할 때 **미묘하게 다른 어설션까지 같은 도우미로 대체**해버릴 수 있다. 만약 원래 어설션이 더 강한 사후조건을 갖고 있었다면, 이 리팩터링은 검증을 조용히 약화시킨다.

반면 실수로 **사후조건을 강화**한 경우는 테스트가 실패하면서 즉시 드러난다.

따라서 규칙은 이렇다.

> 테스트 코드를 리팩터링할 때는 프로덕션 코드를 건드리지 않는다.

그리고 그 반대도 마찬가지다. 프로덕션 코드를 리팩터링할 때는 테스트 코드를 건드리지 않는다. 만약 리팩터링 도중 테스트 코드도 손봐야 한다는 것을 깨달았다면 **git stash로 잠시 저장**한 뒤 테스트 코드만 독립적으로 편집하고 커밋한 다음 원래 작업을 이어간다.

### 3.1 SpyPostOffice 리팩터링 예제

레스토랑 예약 코드에 이메일 기능을 추가하고 있다. 처음에는 예약 생성 이벤트만 있었다.

```csharp
public interface IPostOffice
{
    Task EmailReservationCreated(Reservation reservation);
}
```

테스트에서는 `SpyPostOffice`가 `Collection<Reservation>`을 상속받아 예약을 그대로 담아둔다.

```csharp
public class SpyPostOffice : Collection<Reservation>, IPostOffice
{
    public Task EmailReservationCreated(Reservation reservation)
    {
        Add(reservation);
        return Task.CompletedTask;
    }
}
```

이후 예약 삭제 시에도 이메일을 보내는 기능을 추가하면서 `EmailReservationDeleted`를 인터페이스에 추가했다. 그러자 문제가 생겼다. 스파이에는 `Reservation`만 들어가므로, **어느 메서드로 들어온 예약인지 구분할 수 없다**. 스파이의 "민감도"를 높여야 했다.

이 시점에서 저자는 이미 프로덕션 코드에 여러 변경을 진행 중이었다. 여기서 git이 빛을 발한다. **작업 중이던 변경을 stash로 저장**하고 `SpyPostOffice`만 독립적으로 리팩터링했다.

```csharp
internal class SpyPostOffice :
    Collection<SpyPostOffice.Observation>, IPostOffice
{
    public Task EmailReservationCreated(Reservation reservation)
    {
        Add(new Observation(Event.Created, reservation));
        return Task.CompletedTask;
    }
    internal enum Event
    {
        Created = 0
    }
}
```

이벤트 종류를 담는 `Observation` 중첩 클래스를 도입하고, 기반 컬렉션을 `Observation` 컬렉션으로 바꿨다. 이 리팩터링만 커밋한 뒤, 저장해둔 변경을 다시 꺼내 이어서 작업했다.

::: callout {icon="💡" color="gray_bg"}
**핵심 통찰**: git의 stash는 개발 흐름을 바꾸는 **기동성**을 제공한다. 테스트 리팩터링이 필요해질 때마다 원래 작업을 잠시 옆으로 밀어두고 테스트 리팩터링만 독립 커밋으로 처리할 수 있다.
:::

## 4. 테스트 실패를 참조하기

테스트와 프로덕션을 도저히 분리할 수 없어 동시에 바꿔야 하는 경우도 있다. 그럴 때는 **일부러 테스트를 실패시켜 본다**.

**항상 성공하는 어설션(*Autological Assertion - 프로덕션 코드에 결함이 있어도 결코 실패하지 않는, 사실상 무의미한 어설션*)** 을 작성하는 것은 놀랍도록 쉽다. 실패해본 적 없는 테스트는 신뢰하지 말라.

절차는 이렇다.

1. 새 테스트를 작성하거나 기존 테스트를 편집한다
2. 커밋할 변경을 **stage**한다
3. 프로덕션 코드의 일부를 주석 처리하거나 하드코딩된 값을 반환하도록 임시로 방해공작을 넣는다
4. 테스트가 실패하는지 확인한다 — 실패해야 정상
5. 작업 디렉터리의 방해공작을 되돌린다 (`git checkout`)
6. staged된 변경만 커밋한다

이 절차는 테스트가 실제로 검증력을 갖고 있음을 확인시켜 준다.

## 5. 상대적 안전도 요약

| 편집 종류 | 안전도 | 설명 |
|---|---|---|
| 새 테스트 클래스 추가 | 매우 안전 | 기존 것과 완전 독립 |
| 새 테스트 메서드 추가 | 매우 안전 | 각 메서드는 독립 실행 |
| 매개변수 케이스 추가 | 안전 | 시나리오 하나 추가 |
| 어설션 추가 | 안전 | 사후조건 강화 |
| IDE의 자동 리네임/추출 | 대체로 안전 | 도구가 정확히 수행 |
| 어설션 삭제/변경 | 위험 | 사후조건 약화 가능성 |
| 사후조건 약화 | 매우 위험 | 감지 불가능한 실수 |
| 어설션 도우미 대량 치환 | 조건부 위험 | 미묘한 차이 놓칠 수 있음 |

## 요약

- 유닛 테스트 코드를 편집할 때는 **안전망이 없음**을 인식해야 한다
- 공식적으로 유닛 테스트는 리팩터링할 수 없지만, 실무에서는 편집이 필요한 경우가 많다
- **안전한 편집**은 "추가": 새 테스트/케이스/어설션 추가는 시스템의 보증을 강화하기만 한다
- 어설션 추가는 사후조건 강화다. "테스트당 어설션 하나" 규칙은 과도한 단순화
- 리스코프 치환 원칙의 렌즈로 보면, 시간이 지나면서 **사후조건은 강화만 되어야** 한다
- 프로덕션 코드 리팩터링과 테스트 코드 리팩터링은 **분리**한다. 각각을 별개 커밋으로 나눈다
- 리팩터링 중 테스트 편집이 필요해지면 원래 작업을 ``git stash``로 저장하고 테스트만 먼저 처리한다
- 테스트와 프로덕션을 동시에 바꿔야 한다면 **일부러 실패시켜본다**: 방해공작 → 실패 확인 → 원상복구 → staged 변경만 커밋
- 실패해본 적 없는 테스트는 신뢰하지 않는다. **항상 성공하는 어설션**은 놀랍도록 쓰기 쉽다

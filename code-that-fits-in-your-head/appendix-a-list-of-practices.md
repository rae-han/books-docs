## 2. 카테고리별 정리

### 2.1 코드 형식과 크기

**A.1 50/72 규칙** — 깃 커밋 메시지 스타일.

- 요약은 50자 이내, 명령형으로 작성
- 본문을 추가할 경우 두 번째 줄은 빈 줄
- 이후 각 줄은 72자 이내
- **"무엇"이 아니라 "왜"** 를 설명 (diff는 이미 무엇을 보여준다)

**A.2 80/24 규칙** — C 기반 언어(C#, Java, C++, JavaScript)에서 80자 × 24줄. 예전 터미널 크기에서 온 값이며, 파레토 원칙(80/20)과도 닮았다. 숫자를 문자 그대로 받아들이지 말고 **한계값을 정하고 일관성 있게 지키는 것**이 요점.

**A.7 변수 개수 세기** — 지역변수 + 매개변수 + 클래스 필드의 총합을 최소화. 이 숫자는 메서드가 머리에 얼마나 잘 들어오는지의 지표다.

**A.8 순환 복잡도** — 실제 유용한 몇 안 되는 코드 품질 지표. 저자의 경험상 **임계값 7**이 잘 동작한다. 이 수치는 메서드를 완전히 커버하는 최소 테스트 케이스 수도 알려준다.

### 2.2 테스트

**A.3 준비-행동-어설트(AAA)** — 각 부분이 끝나고 시작하는 것을 명확히 구분되도록 테스트 구성.

**A.10 악마의 변호인** — 테스트 스위트의 신뢰성을 평가하는 휴리스틱. 의도적으로 SUT를 잘못 구현해보고, 테스트가 그 결함을 잡아내는지 확인. 잘못 구현할 수 있는 게 많을수록 테스트 케이스가 부족한 것.

**A.17 빨강-초록-리팩터** — TDD 사이클 체크리스트.

1. 실패한 테스트를 작성
   - A. 테스트를 실행했나?
   - B. 실패했나?
   - C. 어설션 때문에 실패한 건가?
   - D. 가장 마지막 어설션 때문에 실패한 건가?
2. 가장 간단한 방법으로 통과
3. 코드를 개선 (모든 테스트 여전히 통과)
4. 반복

**A.19 결함을 테스트로 재현** — 버그를 자동화된 테스트로 잡아 회귀 방어.

**A.22 테스트/프로덕션 리팩터링 분리** — 테스트 코드에는 자동 테스트가 없으므로 리팩터링이 더 위험. **테스트와 프로덕션 코드를 동시에 리팩터링하지 말 것.**

**A.27 XX-주도 개발** — 정적 분석, 유닛 테스트, 리팩터링 도구 등을 개발 방법론의 **드라이버**로 활용.

### 2.3 설계 원칙

**A.6 명령과 쿼리 분리(CQS)** — 명령은 부수효과를 만드는 프로시저, 쿼리는 데이터 반환 함수. 하나의 메서드가 둘 다이면 안 됨.

**A.9 횡단 관심사 데코레이터** — 로깅/캐싱/내결함성을 비즈니스 로직에 주입하지 말고 데코레이터 패턴으로 분리.

**A.12 함수형 코어, 명령 셸** — 순수 함수의 참조 투명성이 궁극의 추상화. 순수 함수는 조합 가능성과 테스트 용이성이 뛰어남.

**A.15 유효성 검사 말고 구문 분석** — JSON/XML/CSV 등 외부 데이터를 잘 구조화된 도메인 타입으로 **가능한 한 빠르게 변환**. 이후로는 데이터 무결성을 신뢰할 수 있음.

**A.16 포스텔의 법칙(*Postel's Law - "관대함의 원칙", 로버트 포스텔의 인터넷 프로토콜 설계 지침*)** — "보내는 부분은 보수적으로, 받는 부분은 더 자유롭게." 입력은 최대한 받아들이되 반환은 신뢰할 수 있는 값으로.

**A.28 X로 이름 바꾸기** — 메서드 이름을 ``X`` 로 바꾸면 시그니처만으로 얼마나 정보를 주는지 알 수 있다. 실제 IDE에서 안 해도, 머릿속에서만 해봐도 된다. 정적 타입 언어에서 형식이 얼마나 많은 정보를 전달할 수 있는지 확인하는 기법.

### 2.4 정보 전달

**A.13 정보 전달 단계** — 우선순위 (위가 더 중요):

1. API에 특유의 데이터 형식 제공
2. 메서드에 유용한 이름 부여
3. 좋은 주석
4. 자동화된 테스트로 사용 예제 제공
5. 좋은 깃 커밋 메시지
6. 좋은 문서

**A.14 예외 규정 시 이유 설명** — 좋은 규칙도 방해가 되는 상황이 있다. 규칙에서 벗어나되 **정당한 이유를 반드시 문서화**. 결정 전 다른 사람의 의견을 구하기.

### 2.5 리팩터링과 변경 관리

**A.11 기능 플래그** — 반나절 이상 걸리는 변경은 플래그 뒤에 숨기고 다른 사람의 작업과 계속 통합.

**A.21 유의적 버전 관리** — Semantic Versioning으로 라이브러리 API 호환성 관리.

**A.23 슬라이스** — 수직 슬라이스(*Vertical Slice - UI부터 DB까지 관통하는 얇은 기능 단위*)로 시작해 기능을 늘려간다. 이 방식만으로 100% 진행되진 않음 — 버그 수정, 횡단 관심사 작업은 별도.

**A.24 스트랭글러(*Strangler - 마틴 파울러가 제안한 점진적 마이그레이션 패턴. 나무를 감아 죽이는 무화과 나무에서 이름을 따옴*)** — 반나절 이상 걸리는 리팩터링. 옛 방식과 새 방식을 나란히 두고 점진적으로 이관. 시스템은 항상 일관된 상태 유지.

**A.26 변환 우선순위 전제(*Transformation Priority Premise, TPP - Robert C. Martin이 제안한, 유효한 상태 사이 변환의 우선순위 목록*)** — 코드가 대부분 유효한 상태에 머물도록 작은 변환을 순서대로 적용.

### 2.6 협업과 리듬

**A.5 새 코드베이스 체크리스트** — 짧고 간단하게 유지.

- 깃 사용
- 빌드 자동화
- 모든 오류 메시지 켜기

**A.18 정기적인 의존성 업데이트** — 잊어버리기 쉬우나 너무 뒤처지면 따라잡기 어려움. 스프린트 첫 작업으로.

**A.20 코드 리뷰** — 알려진 가장 효과적인 QA 기법 중 하나. 짝/몹 프로그래밍이나 PR 리뷰. **실제로 거부 가능해야** 리뷰의 가치가 있음. 매일의 리듬으로.

### 2.7 문제 해결

**A.4 이분법(*Bisection - 문제 원인을 찾기 위해 코드/이력을 절반씩 줄여가는 기법*)** — 코드의 절반을 제거하고 문제가 재현되는지 확인. 반복하며 좁혀나감.

### 2.8 보안

**A.25 위협 모델** — 보안 전문가가 아니어도 STRIDE로 체계적 접근 가능.

- 스푸핑(Spoofing)
- 변조(Tampering)
- 거부(Repudiation)
- 정보 노출(Information Disclosure)
- 서비스 거부(Denial of Service)
- 권한 상승(Elevation of Privilege)

위협 완화는 보안 위협과 사업 관심사의 균형이 필요하므로 **IT 전문가와 이해관계자가 참여**.

## 요약

- 이 책은 28개의 실천 가능한 프랙티스로 요약된다
- 크게 여덟 개 범주로 나눌 수 있다: 형식/크기, 테스트, 설계, 정보 전달, 리팩터링, 협업, 문제 해결, 보안
- 각 프랙티스는 특정 챕터/절과 연결된다 — 상황에 맞게 되돌아가서 상세 내용 확인
- 프랙티스는 규칙이 아니라 **휴리스틱** — 상황에 맞게 조정하되 예외를 정당화할 것(A.14)
- 정보 전달 우선순위(A.13)와 순환 복잡도(A.8), 함수형 코어/명령 셸(A.12)은 이 책의 핵심 축
- 새 프로젝트에서는 A.5(새 코드베이스 체크리스트)부터 시작, 진행 중에는 A.20(코드 리뷰), A.18(의존성 업데이트)을 리듬으로
